07. HTTP 요청/응답 구조¶

이 장의 목표: HTTP 메시지의 구조를 한 줄씩 읽을 수 있다. Start Line, Header, Body 구분이 바로 된다.

1. HTTP 메시지 = 텍스트¶

1.1 핵심¶

HTTP 메시지는 사람이 읽을 수 있는 텍스트다.
바이너리(01001010...)가 아니라 영어 텍스트.

그래서:
  - 개발자 도구에서 바로 읽을 수 있다
  - curl로 직접 볼 수 있다
  - 디버깅이 쉽다

HTTP 메시지의 종류는 딱 2가지:
  1. 요청 (Request): 클라이언트 → 서버
  2. 응답 (Response): 서버 → 클라이언트

1.2 공통 구조¶

모든 HTTP 메시지는 이 구조를 따른다:

  ┌──────────────────────┐
  │ Start Line           │  ← 1줄 (요청/응답의 핵심 정보)
  ├──────────────────────┤
  │ Header               │  ← 여러 줄 (메타데이터)
  │ Header               │
  │ Header               │
  ├──────────────────────┤
  │ (빈 줄)              │  ← 반드시 있어야 함!
  ├──────────────────────┤
  │ Body                 │  ← 선택사항 (실제 데이터)
  └──────────────────────┘

이 구조를 머리에 박아둬.
요청이든 응답이든 이 4가지 파트로 나뉜다.

2. 요청(Request) 메시지 해부¶

2.1 실제 HTTP 요청¶

POST /api/users HTTP/1.1
Host: api.example.com
Content-Type: application/json
Authorization: Bearer abc123token
Content-Length: 42
Accept: application/json

{"name": "홍길동", "age": 25}

한 줄씩 뜯어보자.

2.2 Start Line (요청 라인)¶

POST /api/users HTTP/1.1
^^^^  ^^^^^^^^^^  ^^^^^^^^
 │        │          │
 │        │          └── HTTP 버전
 │        └── 요청 대상 (Request Target, path + query)
 └── HTTP 메서드

분해:
  메서드: POST → "생성해줘"
  요청 대상: /api/users → "이 경로에"
  HTTP 버전: HTTP/1.1 → "이 버전의 규칙으로"

합치면: "HTTP/1.1 규칙으로 /api/users에 생성 요청한다"

메서드별 Start Line 예시:

  GET /users/123 HTTP/1.1            → 123번 사용자 조회
  POST /users HTTP/1.1               → 사용자 생성
  PUT /users/123 HTTP/1.1            → 123번 사용자 전체 교체
  PATCH /users/123 HTTP/1.1          → 123번 사용자 부분 수정
  DELETE /users/123 HTTP/1.1         → 123번 사용자 삭제
  GET /search?q=HTTP&page=1 HTTP/1.1 → 검색 (query string 포함)

2.3 Headers (요청 헤더)¶

Host: api.example.com
Content-Type: application/json
Authorization: Bearer abc123token
Content-Length: 42
Accept: application/json

헤더 = key: value 형태의 메타데이터

각 헤더의 의미:

  Host: api.example.com
  → 요청을 보내는 서버의 주소
  → HTTP/1.1에서 필수!

  Content-Type: application/json
  → body의 데이터 형식이 JSON이라는 뜻
  → "이 body를 JSON으로 파싱해"

  Authorization: Bearer abc123token
  → 인증 정보
  → "나 로그인한 사람이야, 이 토큰이 증거야"

  Content-Length: 42
  → body의 크기가 42바이트라는 뜻
  → 서버가 body를 어디까지 읽어야 하는지 알려줌

  Accept: application/json
  → "응답을 JSON으로 보내줘"
  → 클라이언트가 원하는 응답 형식

자주 보는 요청 헤더

헤더 의미 예시
Host 서버 주소 (필수) api.example.com
Content-Type body 데이터 형식 application/json
Accept 원하는 응답 형식 application/json
Authorization 인증 정보 Bearer token123
Content-Length body 크기 (바이트) 42
User-Agent 클라이언트 정보 Mozilla/5.0...
Cookie 쿠키 전달 session=abc123

헤더에 대한 자세한 내용은 09장(HTTP 헤더 완전정복)에서 다룬다.

2.4 빈 줄 (Empty Line)¶

Content-Length: 42
                        ← 이 빈 줄!
{"name": "홍길동"}

빈 줄이 없으면 에러다

빈 줄 = 헤더와 body의 경계선

HTTP 파서가 메시지를 읽을 때: 1. Start Line 읽는다 2. 헤더를 한 줄씩 읽는다 3. 빈 줄을 만나면 "헤더 끝!"이라고 인식한다 4. 그 다음부터 body로 읽는다

빈 줄이 없으면? → 파서가 body를 헤더로 읽으려고 한다 → 파싱 에러 → 요청 실패

이건 규칙이다. 예외 없다.

2.5 Body (요청 본문)¶

{"name": "홍길동", "age": 25}

body = 실제 전송하는 데이터

규칙:
  - 선택사항이다 (GET, DELETE는 보통 body 없음)
  - POST, PUT, PATCH는 보통 body가 있다
  - Content-Type 헤더가 body의 형식을 알려준다
  - Content-Length 헤더가 body의 크기를 알려준다

자주 쓰는 body 형식:
  application/json        → {"key": "value"}
  application/x-www-form-urlencoded → key=value&key2=value2
  multipart/form-data     → 파일 업로드 시

body 형식에 대한 자세한 내용은 10장(Content-Type과 Body)에서 다룬다.

3. 응답(Response) 메시지 해부¶

3.1 실제 HTTP 응답¶

HTTP/1.1 200 OK
Content-Type: application/json
Set-Cookie: session=abc123; Path=/; HttpOnly
Content-Length: 52
Date: Wed, 18 Mar 2026 09:00:00 GMT
Cache-Control: no-cache

{"id": 1, "name": "홍길동", "age": 25}

3.2 Status Line (상태 라인)¶

HTTP/1.1 200 OK
^^^^^^^^ ^^^ ^^
   │      │   │
   │      │   └── 사유 구문 (Reason Phrase): 상태 코드의 설명
   │      └── 상태 코드 (Status Code): 요청 처리 결과
   └── HTTP 버전

분해:
  HTTP 버전: HTTP/1.1
  상태 코드: 200 → "성공"
  사유 구문: OK → "성공이야"

합치면: "HTTP/1.1 규칙으로 응답하는데, 결과는 200 성공이야"

자주 보는 Status Line:

  HTTP/1.1 200 OK                → 성공
  HTTP/1.1 201 Created           → 생성 성공
  HTTP/1.1 301 Moved Permanently → 영구 이동 (리다이렉트)
  HTTP/1.1 400 Bad Request       → 잘못된 요청
  HTTP/1.1 401 Unauthorized      → 인증 필요
  HTTP/1.1 403 Forbidden         → 권한 없음
  HTTP/1.1 404 Not Found         → 없는 리소스
  HTTP/1.1 500 Internal Server Error → 서버 에러

상태 코드에 대한 자세한 내용은 08장(상태코드 완전정복)에서 다룬다.

사유 구문(Reason Phrase)은 참고용

200 OK에서 OK 부분은 사람이 읽기 위한 것이다. 실제 처리에는 상태 코드 200만 쓰인다. 200 Success라고 써도 동작에는 문제없다. HTTP/2부터는 사유 구문이 아예 없다.

3.3 Headers (응답 헤더)¶

Content-Type: application/json
Set-Cookie: session=abc123; Path=/; HttpOnly
Content-Length: 52
Date: Wed, 18 Mar 2026 09:00:00 GMT
Cache-Control: no-cache

각 헤더의 의미:

  Content-Type: application/json
  → 응답 body가 JSON이라는 뜻
  → 클라이언트가 이걸 보고 JSON으로 파싱한다

  Set-Cookie: session=abc123; Path=/; HttpOnly
  → "이 쿠키 저장해" (16장에서 자세히)
  → 브라우저가 이 쿠키를 저장하고, 다음 요청에 자동으로 보낸다

  Content-Length: 52
  → 응답 body가 52바이트

  Date: Wed, 18 Mar 2026 09:00:00 GMT
  → 응답이 생성된 시간

  Cache-Control: no-cache
  → "이 응답 캐시하지 마" (캐시 관련은 09장에서)

자주 보는 응답 헤더

헤더 의미 예시
Content-Type 응답 body 형식 application/json
Content-Length 응답 body 크기 52
Set-Cookie 쿠키 설정 session=abc; HttpOnly
Cache-Control 캐시 정책 no-cache, max-age=3600
Location 리다이렉트 주소 https://example.com/new
Date 응답 생성 시간 Wed, 18 Mar 2026...

3.4 빈 줄 + Body¶

Cache-Control: no-cache
                                ← 빈 줄 (헤더 끝!)
{"id": 1, "name": "홍길동", "age": 25}

요청과 동일한 규칙이다. 빈 줄 다음이 body.

4. 요청 vs 응답 구조 비교¶

4.1 나란히 비교¶

요청 (Request):                    응답 (Response):
┌────────────────────────┐        ┌────────────────────────┐
│ POST /api/users HTTP/1.1│        │ HTTP/1.1 201 Created   │ ← Start Line
├────────────────────────┤        ├────────────────────────┤
│ Host: api.example.com  │        │ Content-Type: app/json │ ← Headers
│ Content-Type: app/json │        │ Content-Length: 52     │
│ Authorization: Bearer..│        │ Location: /users/1     │
├────────────────────────┤        ├────────────────────────┤
│ (빈 줄)                │        │ (빈 줄)                │ ← 빈 줄
├────────────────────────┤        ├────────────────────────┤
│ {"name": "홍길동"}     │        │ {"id": 1, "name":...}  │ ← Body
└────────────────────────┘        └────────────────────────┘

4.2 차이점 정리¶

구분 요청 (Request) 응답 (Response)
Start Line 메서드 + 경로 + HTTP 버전 HTTP 버전 + 상태 코드 + 사유 구문
Start Line 예시 POST /api/users HTTP/1.1 HTTP/1.1 200 OK
Headers 요청 메타데이터 응답 메타데이터
특징적 헤더 Host, Authorization, Accept Set-Cookie, Location, Cache-Control
Body 전송할 데이터 (생성/수정 시) 응답 데이터
빈 줄 헤더와 body 구분 (동일) 헤더와 body 구분 (동일)

5. 실전: HTTP 메시지 직접 보는 법¶

이론으로만 보면 감이 안 온다. 실제로 봐야 한다.

5.1 개발자 도구 (F12)¶

브라우저에서 F12 → Network 탭

1. 아무 웹사이트 열기
2. F12 누르기
3. Network 탭 클릭
4. 페이지 새로고침 (F5)
5. 요청 목록이 쫙 나온다
6. 아무 요청 하나 클릭

보이는 것:
  General:
    Request URL: https://example.com/api/data
    Request Method: GET
    Status Code: 200 OK

  Response Headers:
    content-type: application/json
    cache-control: max-age=3600

  Request Headers:
    accept: application/json
    user-agent: Mozilla/5.0...

  Response (또는 Preview):
    {"data": [...]}

F12 Network 탭 활용법

개발할 때 "요청이 안 돼요"라고 하기 전에 F12 → Network 탭부터 열어봐.

거기에 다 나온다:

  • 어떤 URL로 요청했는지
  • 어떤 메서드를 썼는지
  • 상태 코드가 뭔지
  • 요청/응답 헤더가 뭔지
  • body가 뭔지

이걸 안 보고 디버깅하는 건 눈 감고 운전하는 거다.

5.2 curl로 보기¶

# -v 옵션: verbose (상세 출력)
curl -v https://example.com

# 출력:
> GET / HTTP/1.1          ← 요청 Start Line
> Host: example.com       ← 요청 헤더
> User-Agent: curl/8.0    ← 요청 헤더
> Accept: */*             ← 요청 헤더
>                         ← 빈 줄
< HTTP/1.1 200 OK        ← 응답 Status Line
< Content-Type: text/html ← 응답 헤더
< Content-Length: 1256    ← 응답 헤더
<                         ← 빈 줄
<!doctype html>...        ← 응답 Body

# > 는 보내는 것 (요청)
# < 는 받는 것 (응답)

# POST 요청 보내기
curl -v -X POST https://api.example.com/users \
  -H "Content-Type: application/json" \
  -d '{"name": "홍길동"}'

# -X POST: 메서드 지정
# -H: 헤더 추가
# -d: body 데이터

5.3 curl 옵션 정리¶

옵션 의미 예시
-v 상세 출력 (요청/응답 전체) curl -v url
-X HTTP 메서드 지정 -X POST, -X PUT
-H 헤더 추가 -H "Content-Type: application/json"
-d body 데이터 -d '{"name": "홍길동"}'
-I HEAD 요청 (헤더만) curl -I url
-o 파일로 저장 -o output.html

6. 메시지 구조의 흐름¶

전체 요청-응답 과정을 Mermaid로 시각화하면 이렇다.

sequenceDiagram
    participant C as 클라이언트
    participant S as 서버

    Note over C: 요청 메시지 구성
    C->>C: Start Line: POST /api/users HTTP/1.1
    C->>C: Header: Content-Type, Authorization...
    C->>C: (빈 줄)
    C->>C: Body: {"name": "홍길동"}

    C->>S: HTTP 요청 전송

    Note over S: 요청 처리
    S->>S: Start Line 파싱 → POST 메서드, /api/users 경로
    S->>S: Header 파싱 → JSON 타입, 인증 확인
    S->>S: 빈 줄 → 헤더 끝 인식
    S->>S: Body 파싱 → 사용자 데이터 추출
    S->>S: 비즈니스 로직 실행

    Note over S: 응답 메시지 구성
    S->>S: Status Line: HTTP/1.1 201 Created
    S->>S: Header: Content-Type, Location...
    S->>S: (빈 줄)
    S->>S: Body: {"id": 1, "name": "홍길동"}

    S-->>C: HTTP 응답 전송

7. 자주 하는 실수¶

7.1 Content-Type 안 보내기¶

❌ 잘못된 요청:
  POST /api/users HTTP/1.1
  Host: api.example.com

  {"name": "홍길동"}

  → Content-Type 헤더가 없다
  → 서버가 body를 뭘로 파싱해야 할지 모른다
  → 400 Bad Request 또는 파싱 에러

✅ 올바른 요청:
  POST /api/users HTTP/1.1
  Host: api.example.com
  Content-Type: application/json

  {"name": "홍길동"}

  → Content-Type이 있으니 서버가 JSON으로 파싱한다

7.2 GET에 body 보내기¶

❌ 잘못된 요청:
  GET /users HTTP/1.1
  Host: api.example.com
  Content-Type: application/json

  {"filter": "active"}

  → GET에 body를 보내는 건 HTTP 스펙상 금지는 아니지만
  → 많은 서버/프록시가 GET의 body를 무시한다
  → 동작이 보장되지 않는다
  → 이런 짓 하지 마

✅ 올바른 요청:
  GET /users?filter=active HTTP/1.1
  Host: api.example.com

  → query string으로 전달한다

7.3 상태 코드 무시하기¶

❌ 잘못된 처리:
  응답: HTTP/1.1 401 Unauthorized
  프론트엔드: "요청 실패인 것 같은데... 왜지?"
  → 상태 코드를 안 본 거다. 401이면 인증 문제다.

✅ 올바른 처리:
  응답: HTTP/1.1 401 Unauthorized
  프론트엔드: "401이니까 토큰이 만료됐거나 없는 거네"
  → 재로그인 유도 또는 토큰 갱신

8. 전체 흐름 예시: 사용자 생성 API¶

하나의 완전한 요청-응답 과정을 처음부터 끝까지 보자.

8.1 요청¶

POST /api/v1/users HTTP/1.1
Host: api.example.com
Content-Type: application/json
Authorization: Bearer eyJhbGciOiJIUzI1NiJ9...
Accept: application/json
Content-Length: 62

{"name": "홍길동", "email": "test@example.com", "role": "user"}

해석:
  Start Line: POST /api/v1/users HTTP/1.1
  → POST 메서드로 /api/v1/users에 요청 (사용자 생성)

  Host: api.example.com
  → 서버 주소

  Content-Type: application/json
  → body가 JSON 형식

  Authorization: Bearer eyJhbG...
  → JWT 토큰으로 인증 (24장에서 다룸)

  Accept: application/json
  → 응답도 JSON으로 달라

  Content-Length: 62
  → body 크기 62바이트

  body: 생성할 사용자 정보

8.2 응답¶

HTTP/1.1 201 Created
Content-Type: application/json
Location: /api/v1/users/42
Content-Length: 85
Date: Wed, 18 Mar 2026 09:00:00 GMT

{"id": 42, "name": "홍길동", "email": "test@example.com", "role": "user", "createdAt": "2026-03-18"}

해석:
  Status Line: HTTP/1.1 201 Created
  → 201 = 생성 성공

  Content-Type: application/json
  → 응답 body가 JSON

  Location: /api/v1/users/42
  → 새로 생성된 리소스의 위치
  → "42번 사용자가 여기에 만들어졌어"

  body: 생성된 사용자의 전체 정보 (id 포함)

9. 핵심 정리¶

핵심 정리

HTTP 메시지 구조:

Start Line    ← 핵심 정보 1줄
Headers       ← 메타데이터 (key: value)
(빈 줄)       ← 헤더 끝 표시 (필수!)
Body          ← 실제 데이터 (선택)

요청의 Start Line: 메서드 경로 HTTP버전 예: POST /api/users HTTP/1.1

응답의 Start Line: HTTP버전 상태코드 사유구문 예: HTTP/1.1 200 OK

빈 줄: 헤더와 body의 경계. 이게 없으면 파싱 에러.

실전 도구:

  • 브라우저 F12 → Network 탭
  • curl -v URL

다음 장: 상태코드 완전정복 → 200, 301, 400, 404, 500... 각 상태 코드가 뭘 의미하는지 전부 다룬다.

10. 확인 문제¶

문제 1¶

HTTP 요청 메시지의 4가지 파트를 순서대로 말하고,
각각 한 줄로 설명해봐.
정답 보기
  1. Start Line (요청 라인): 메서드 + 요청 대상 + HTTP 버전. 예: GET /users HTTP/1.1
  2. Headers (헤더): key: value 형태의 메타데이터. 예: Content-Type: application/json
  3. 빈 줄 (Empty Line): 헤더의 끝을 알리는 구분선. 이게 없으면 파싱 에러.
  4. Body (본문): 실제 전송 데이터. 선택사항. GET은 보통 없고, POST/PUT은 있다.

문제 2¶

요청의 Start Line과 응답의 Start Line은 구조가 다르다.
각각의 구조를 써보고, 예시를 하나씩 들어봐.
정답 보기

요청 Start Line 구조: 메서드 요청대상 HTTP버전 예: POST /api/users HTTP/1.1

응답 Start Line 구조: HTTP버전 상태코드 사유구문 예: HTTP/1.1 201 Created

요청은 "뭘 해달라"는 정보(메서드 + 경로)가 핵심이고, 응답은 "처리 결과"(상태 코드)가 핵심이다.

문제 3¶

빈 줄(Empty Line)이 왜 필수인지 설명해봐.
빈 줄이 없으면 어떤 문제가 생겨?
정답 보기

빈 줄은 헤더의 끝을 알리는 구분자다.

HTTP 파서는 메시지를 한 줄씩 읽으면서: - Start Line 다음부터 헤더로 인식한다 - 빈 줄을 만나면 "헤더 끝, 다음부터 body"로 인식한다

빈 줄이 없으면 파서가 body를 헤더의 일부로 읽으려고 시도하고, 파싱 에러가 발생해서 요청/응답 처리가 실패한다.

문제 4¶

다음 HTTP 요청을 읽고, 각 줄이 어떤 파트인지 구분하고
이 요청이 뭘 하려는 건지 한 문장으로 말해봐.

DELETE /api/posts/99 HTTP/1.1
Host: blog.example.com
Authorization: Bearer token123
Accept: application/json
정답 보기
줄 파트 설명
DELETE /api/posts/99 HTTP/1.1 Start Line 메서드: DELETE, 경로: /api/posts/99
Host: blog.example.com Header 서버 주소
Authorization: Bearer token123 Header 인증 토큰
Accept: application/json Header 응답을 JSON으로 받겠다
(마지막 빈 줄) 빈 줄 헤더 끝 표시
(없음) Body DELETE라서 body 없음

이 요청의 의미: 인증된 사용자가 blog.example.com에서 99번 게시글을 삭제하려는 요청이다.
