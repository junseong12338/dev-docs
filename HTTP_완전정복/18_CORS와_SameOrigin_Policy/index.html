18. CORS와 Same-Origin Policy¶

이 챕터의 핵심

브라우저는 다른 출처(Origin)의 리소스 접근을 기본적으로 막는다. 이걸 Same-Origin Policy라고 해. 그리고 "이 Origin은 괜찮아, 허용해"라고 서버가 명시적으로 열어주는 게 CORS다. 프론트엔드와 백엔드를 분리해서 개발하면 100% 만나는 주제야. 이거 모르면 콘솔에 빨간 CORS 에러 뜰 때 멘붕 온다.

왜 이런 게 필요해?¶

상황을 하나 생각해봐.

  1. 너가 bank.com에 로그인했어. 세션 쿠키가 브라우저에 있지.
  2. 그 상태로 evil.com에 접속했어.
  3. evil.com의 JavaScript가 이런 코드를 실행해:
// evil.com의 악성 스크립트
fetch('https://bank.com/api/transfer', {
    method: 'POST',
    body: JSON.stringify({ to: '공격자계좌', amount: 1000000 }),
    credentials: 'include'  // 쿠키 포함!
});

브라우저에 bank.com의 세션 쿠키가 있으니까, 이 요청에 쿠키가 자동으로 붙어. 아무 제한 없으면? 송금이 실행된다. 너도 모르게.

이걸 막는 게 Same-Origin Policy야.

evil.com에서 bank.com으로 보내는 요청은 출처(Origin)가 다르니까 브라우저가 차단한다.

핵심을 기억해

Same-Origin Policy는 악성 사이트가 다른 사이트의 API를 함부로 호출하는 걸 막는 보안 정책이다. 이게 없으면 웹은 보안 지옥이 돼.

Origin(출처)이란¶

Origin의 구성¶

Origin은 scheme + host + port 조합이다.

https://api.example.com:443
  │         │            │
scheme    host          port

이 3가지가 전부 일치해야 Same-Origin(같은 출처)이야.

Same-Origin 판별 예시¶

기준 URL: https://www.example.com:443/page

비교 URL Same-Origin? 이유
https://www.example.com:443/other O scheme, host, port 전부 일치 (경로는 무관)
https://www.example.com/other O HTTPS 기본 포트가 443이라 동일
http://www.example.com:443/page X scheme이 다름 (https vs http)
https://api.example.com:443/page X host가 다름 (www vs api)
https://www.example.com:8080/page X port가 다름 (443 vs 8080)
https://www.example.com:443/page?q=1 O 쿼리스트링은 무관

서브도메인도 다른 Origin이야

www.example.com과 api.example.com은 다른 Origin이다. 같은 example.com 도메인이라도 서브도메인이 다르면 Cross-Origin이야. 이거 모르면 "같은 도메인인데 왜 CORS 에러나지?" 하고 삽질해.

Same-Origin Policy (동일 출처 정책)¶

정의¶

Same-Origin Policy는 브라우저가 강제하는 보안 정책이다.

다른 Origin의 리소스에 대한 JavaScript의 접근을 제한한다.

서버가 막는 게 아니야. 브라우저가 막는 거야.

이게 가장 많이 헷갈리는 부분이야.

  1. 브라우저가 evil.com에서 bank.com으로 요청을 보낸다
  2. 서버는 정상적으로 응답을 돌려준다 (서버는 Origin 신경 안 써도 됨)
  3. 브라우저가 응답을 받고, Origin을 확인한다
  4. "이 응답에 CORS 허용 헤더가 없네? 차단!"
  5. 브라우저가 JavaScript에게 응답을 안 넘겨준다

서버는 응답을 다 보냈어. 브라우저가 그 응답을 JS에게 안 주는 거야.

sequenceDiagram
    participant JS as JavaScript (evil.com)
    participant B as 브라우저
    participant S as 서버 (bank.com)

    JS->>B: fetch('bank.com/api/data')
    B->>S: GET /api/data
    S-->>B: 200 OK + 데이터
    Note over B: Origin 확인: evil.com ≠ bank.com
    Note over B: CORS 허용 헤더 없음
    B--xJS: 차단! 응답 안 넘겨줌
    Note over JS: CORS 에러 발생

제한되는 것과 제한 안 되는 것¶

Same-Origin Policy가 막는 건 JavaScript의 데이터 접근이야. 리소스 로드 자체가 전부 막히는 건 아니야.

동작 제한 여부 설명
<img src="다른Origin/image.png"> 허용 이미지 로드는 OK
<script src="다른Origin/app.js"> 허용 스크립트 로드는 OK
<link href="다른Origin/style.css"> 허용 CSS 로드는 OK
<form action="다른Origin/submit"> 허용 폼 전송은 OK
fetch('다른Origin/api/data') 제한 JS로 데이터 읽기는 차단
XMLHttpRequest to 다른 Origin 제한 AJAX 요청의 응답 읽기 차단

왜 이미지, 스크립트는 허용해?

웹이 원래 그렇게 설계됐어. CDN에서 jQuery 로드하고, 다른 도메인의 이미지를 보여주는 건 웹의 기본 기능이니까. 문제는 JavaScript가 프로그래밍적으로 데이터를 읽는 것이야. 그게 허용되면 악성 JS가 다른 사이트의 API를 마음대로 호출할 수 있으니까.

CORS (Cross-Origin Resource Sharing)¶

정의¶

CORS는 서버가 "이 Origin에서 오는 요청은 허용할게"라고 명시적으로 알려주는 메커니즘이다.

Same-Origin Policy는 기본적으로 다 막아. 하지만 실무에서는 다른 Origin에서 API를 호출해야 하는 경우가 많잖아.

  • 프론트엔드(localhost:3000)가 백엔드(localhost:8080) API 호출
  • www.example.com에서 api.example.com API 호출
  • 모바일 앱이 API 서버 호출

이런 정상적인 요청을 허용하려면 서버가 CORS 헤더를 보내야 한다.

CORS 관련 헤더¶

서버가 응답에 포함하는 헤더들:

Access-Control-Allow-Origin¶

Access-Control-Allow-Origin: https://www.example.com

어떤 Origin을 허용할지 지정한다.

값 의미
https://www.example.com 이 Origin만 허용
* 모든 Origin 허용 (주의!)

와일드카드(*)는 위험할 수 있어

Access-Control-Allow-Origin: *는 모든 사이트에서 접근 가능하다는 뜻이야. 공개 API면 괜찮지만, 인증이 필요한 API에 *를 쓰면 보안 구멍. 특히 credentials: 'include'(쿠키 포함)와 *는 동시에 쓸 수 없어. 브라우저가 거부한다.

Access-Control-Allow-Methods¶

Access-Control-Allow-Methods: GET, POST, PUT, DELETE

어떤 HTTP 메서드를 허용할지 지정한다.

Access-Control-Allow-Headers¶

Access-Control-Allow-Headers: Content-Type, Authorization

어떤 요청 헤더를 허용할지 지정한다. 기본 헤더(Accept, Content-Language 등) 외에 커스텀 헤더를 쓸 때 필요.

Access-Control-Allow-Credentials¶

Access-Control-Allow-Credentials: true

쿠키를 포함한 요청을 허용할지 지정한다.

이게 true면 Access-Control-Allow-Origin에 *를 못 써. 정확한 Origin을 명시해야 해.

Access-Control-Max-Age¶

Access-Control-Max-Age: 3600

Preflight 응답을 캐싱할 시간(초). 이 시간 동안은 같은 요청에 대해 Preflight를 다시 보내지 않는다.

CORS 동작 예시¶

sequenceDiagram
    participant JS as JavaScript (www.example.com)
    participant B as 브라우저
    participant S as API 서버 (api.example.com)

    JS->>B: fetch('api.example.com/users')
    B->>S: GET /users + Origin: https://www.example.com
    S-->>B: 200 OK + Access-Control-Allow-Origin: https://www.example.com
    Note over B: Origin 허용 확인
    B-->>JS: 응답 데이터 전달

서버가 Access-Control-Allow-Origin에 요청한 Origin을 포함시키면, 브라우저가 응답을 JavaScript에 전달해준다.

Preflight 요청¶

이게 뭐야¶

특정 조건의 요청은 브라우저가 본 요청을 보내기 전에 OPTIONS 요청을 먼저 보낸다. 이걸 Preflight 요청이라고 해.

6장에서 OPTIONS 메서드 배웠지? "이 서버가 어떤 메서드를 지원하는지 확인하는 메서드"라고 했어. CORS에서 바로 여기서 쓰인다.

"내가 이런 요청을 보내도 되겠니?" 하고 먼저 물어보는 거야.

Simple Request vs Preflight Request¶

모든 요청에 Preflight가 붙는 건 아니야. Simple Request 조건을 만족하면 Preflight 없이 바로 본 요청을 보낸다.

Simple Request 조건 (전부 만족해야 함):

조건 허용 값
메서드 GET, HEAD, POST만
Content-Type application/x-www-form-urlencoded, multipart/form-data, text/plain만
헤더 Accept, Accept-Language, Content-Language, Content-Type 등 기본 헤더만

이 조건을 하나라도 벗어나면 Preflight가 발생한다.

실무에서 Preflight가 발생하는 흔한 경우

  • Content-Type: application/json (Simple에서 허용하는 3가지에 안 들어감)
  • Authorization 헤더 사용 (기본 헤더가 아님)
  • PUT, DELETE 메서드 사용

REST API 만들면 거의 100% Preflight 발생해. JSON 보내고, 토큰 인증하면 Simple Request 조건을 벗어나니까.

Preflight 흐름¶

sequenceDiagram
    participant JS as JavaScript
    participant B as 브라우저
    participant S as API 서버

    JS->>B: fetch(url, {method: 'PUT', headers: {'Content-Type': 'application/json'}})

    Note over B: Simple Request 아님!<br/>Preflight 필요

    rect rgb(255, 240, 230)
        Note over B,S: Preflight 요청
        B->>S: OPTIONS /api/users<br/>Origin: https://www.example.com<br/>Access-Control-Request-Method: PUT<br/>Access-Control-Request-Headers: Content-Type
        S-->>B: 204 No Content<br/>Access-Control-Allow-Origin: https://www.example.com<br/>Access-Control-Allow-Methods: GET, PUT, DELETE<br/>Access-Control-Allow-Headers: Content-Type<br/>Access-Control-Max-Age: 3600
    end

    Note over B: Preflight 통과!

    rect rgb(230, 255, 230)
        Note over B,S: 본 요청
        B->>S: PUT /api/users<br/>Origin: https://www.example.com<br/>Content-Type: application/json<br/>{"name": "junsung"}
        S-->>B: 200 OK<br/>Access-Control-Allow-Origin: https://www.example.com<br/>{"result": "updated"}
    end

    B-->>JS: 응답 데이터 전달

단계별로:

  1. JavaScript가 fetch 호출 (PUT + JSON = Simple Request 아님)
  2. 브라우저가 Preflight(OPTIONS) 전송: "PUT 메서드에 Content-Type 헤더 써도 돼?"
  3. 서버가 Preflight 응답: "PUT 허용해, Content-Type도 OK"
  4. 브라우저가 본 요청 전송: PUT /api/users
  5. 서버가 본 응답 반환: 200 OK + 데이터
  6. 브라우저가 JavaScript에 응답 전달

Preflight가 실패하면

서버가 Preflight에 CORS 헤더를 안 보내면? 본 요청 자체가 안 나가. 브라우저가 막는다. 서버 로그에 본 요청이 안 찍히면 Preflight 단계에서 막힌 거야.

개발할 때 만나는 CORS 에러¶

에러 메시지¶

브라우저 콘솔에 이런 에러가 뜬다:

Access to fetch at 'http://localhost:8080/api/users' from origin
'http://localhost:3000' has been blocked by CORS policy: No
'Access-Control-Allow-Origin' header is present on the requested resource.

이 에러가 뜨면 서버가 CORS 헤더를 안 보내고 있다는 거야.

자주 하는 실수

"CORS 에러니까 프론트엔드를 고쳐야지" → 아니야. CORS는 서버가 해결하는 거야. 서버가 허용 헤더를 보내야 해. 프론트엔드에서 할 수 있는 건 거의 없어.

Spring에서 해결¶

방법 1: @CrossOrigin 어노테이션¶

@RestController
@CrossOrigin(origins = "http://localhost:3000")
public class UserController {

    @GetMapping("/api/users")
    public List<User> getUsers() {
        return userService.findAll();
    }
}

컨트롤러나 메서드 단위로 CORS를 허용한다. 간단하지만 컨트롤러마다 붙여야 해서 번거로울 수 있어.

방법 2: WebMvcConfigurer (전역 설정)¶

@Configuration
public class WebConfig implements WebMvcConfigurer {

    @Override
    public void addCorsMappings(CorsRegistry registry) {
        registry.addMapping("/api/**")          // /api/로 시작하는 모든 경로
                .allowedOrigins("http://localhost:3000", "https://www.example.com")
                .allowedMethods("GET", "POST", "PUT", "DELETE")
                .allowedHeaders("*")
                .allowCredentials(true)          // 쿠키 포함 허용
                .maxAge(3600);                   // Preflight 캐싱 1시간
    }
}

전역으로 CORS를 설정한다. 실무에서는 이 방법을 더 많이 써.

방법 3: Spring Security와 함께¶

Spring Security를 쓰면 Security 필터가 먼저 요청을 처리하기 때문에, Security 설정에서도 CORS를 열어줘야 해.

@Configuration
@EnableWebSecurity
public class SecurityConfig {

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .cors(cors -> cors.configurationSource(corsConfigurationSource()))
            // ... 다른 설정
        ;
        return http.build();
    }

    @Bean
    public CorsConfigurationSource corsConfigurationSource() {
        CorsConfiguration config = new CorsConfiguration();
        config.setAllowedOrigins(List.of("http://localhost:3000"));
        config.setAllowedMethods(List.of("GET", "POST", "PUT", "DELETE"));
        config.setAllowedHeaders(List.of("*"));
        config.setAllowCredentials(true);

        UrlBasedCorsConfigurationSource source = new UrlBasedCorsConfigurationSource();
        source.registerCorsConfiguration("/api/**", config);
        return source;
    }
}

Spring Security 쓰면서 WebMvcConfigurer만 설정하는 실수

Spring Security가 활성화되어 있으면 Security 필터가 먼저 CORS를 체크해. WebMvcConfigurer에만 CORS를 설정해놓으면 Security 필터에서 이미 막힌다. Security 설정에서 .cors() 활성화 필수.

Nginx에서 해결¶

location /api/ {
    # Preflight 요청 처리
    if ($request_method = 'OPTIONS') {
        add_header 'Access-Control-Allow-Origin' 'https://www.example.com';
        add_header 'Access-Control-Allow-Methods' 'GET, POST, PUT, DELETE, OPTIONS';
        add_header 'Access-Control-Allow-Headers' 'Content-Type, Authorization';
        add_header 'Access-Control-Max-Age' 3600;
        return 204;
    }

    add_header 'Access-Control-Allow-Origin' 'https://www.example.com';
    add_header 'Access-Control-Allow-Credentials' 'true';

    proxy_pass http://backend;
}

또 다른 방법: 프록시¶

개발 환경에서 CORS를 우회하는 방법도 있어. 프론트엔드 개발 서버가 프록시 역할을 하는 거야.

graph LR
    A[브라우저] -->|Same-Origin| B[프론트 개발서버<br/>localhost:3000]
    B -->|서버-서버 통신<br/>CORS 무관| C[백엔드 서버<br/>localhost:8080]

    style A fill:#9999ff
    style B fill:#99ff99
    style C fill:#ffcc99

브라우저는 localhost:3000에만 요청하니까 Same-Origin. 프론트 서버가 백엔드로 대신 요청을 보내준다. 서버 간 통신에는 CORS가 없어. CORS는 브라우저 정책이니까.

왜 CORS가 있어?¶

정리하자.

없으면 어떻게 되는데? 설명
악성 사이트가 API 호출 evil.com이 bank.com/api/transfer 호출 가능
쿠키 자동 첨부 브라우저가 bank.com 쿠키를 자동으로 보냄
사용자 모르게 실행 사용자는 evil.com 구경하는 줄 알지만 은행 송금이 실행됨

CORS(Same-Origin Policy)가 있으니까:

  • evil.com의 JavaScript가 bank.com API를 호출해도
  • bank.com 서버가 evil.com을 허용하지 않았으면
  • 브라우저가 응답을 차단한다

CORS는 귀찮지만 반드시 필요해

개발할 때 CORS 에러 만나면 짜증나지. 하지만 이게 없으면 웹은 보안 지옥이 돼. "왜 막아, 그냥 다 열어" 하면 안 돼. 필요한 Origin만 정확히 열어주는 게 맞다.

주의사항 / 함정¶

흔한 실수들

1. "CORS는 프론트엔드 문제다"

아니야. CORS 헤더는 서버가 보내는 것이야. 프론트엔드에서 할 수 있는 건 요청을 보내는 것뿐. 허용 여부는 서버가 결정하고, 차단 여부는 브라우저가 결정해.

2. "Postman에서는 되는데 브라우저에서는 안 돼요"

맞아. Postman은 브라우저가 아니니까 Same-Origin Policy가 없어. CORS는 브라우저 전용 보안 정책이야. Postman, curl, 서버 간 통신에서는 CORS가 적용 안 돼. 이게 바로 "서버가 막는 게 아니라 브라우저가 막는 것"의 증거야.

3. "Access-Control-Allow-Origin: * 하면 다 되잖아"

되긴 해. 하지만 credentials: 'include'(쿠키 포함)와 함께 쓸 수 없어. 그리고 인증이 필요한 API에 *를 쓰면 모든 사이트에서 접근 가능해져서 위험해. 허용할 Origin을 정확히 명시하는 게 맞다.

4. "OPTIONS 요청이 왜 자꾸 오지?"

Preflight야. Simple Request 조건을 만족하지 않아서 브라우저가 사전 확인 요청을 보내는 거야. Content-Type: application/json이나 Authorization 헤더를 쓰면 발생해. 서버에서 OPTIONS를 제대로 처리해줘야 한다.

5. "CORS만 설정하면 보안 끝?"

절대 아니야. CORS는 브라우저에서의 무단 접근만 막아. 서버에서 직접 호출(curl, Postman, 다른 서버)하면 CORS 안 걸려. 서버 자체의 인증/인가(JWT, 세션 등)는 별도로 반드시 해야 해.

정리¶

개념 한 줄 요약
Origin scheme + host + port 조합. 하나라도 다르면 Cross-Origin
Same-Origin Policy 브라우저가 다른 Origin의 리소스 접근을 제한하는 보안 정책
CORS 서버가 특정 Origin의 접근을 허용하는 메커니즘
Access-Control-Allow-Origin 허용할 Origin 지정
Preflight 본 요청 전에 OPTIONS로 허용 여부 확인
Simple Request GET/HEAD/POST + 기본 헤더만 쓰면 Preflight 없이 바로 요청
credentials 쿠키 포함 요청. * Origin과 함께 쓸 수 없음

한 문장 정리

Same-Origin Policy는 브라우저가 다른 Origin의 리소스 접근을 막는 보안 정책이고, CORS는 서버가 "이 Origin은 허용한다"고 명시적으로 알려주는 메커니즘이다. 서버가 막는 게 아니라 브라우저가 막는 거야. 이거 헷갈리면 안 돼.

확인 문제¶

문제 1¶

다음 중 https://www.example.com:443과 Same-Origin인 것은?

A. http://www.example.com:443 B. https://api.example.com:443 C. https://www.example.com:8080 D. https://www.example.com/api/users

정답 보기

D. https://www.example.com/api/users

Origin은 scheme + host + port로 결정돼. 경로(path)는 무관해. A는 scheme이 다르고(http vs https), B는 host가 다르고(www vs api), C는 port가 다르다(443 vs 8080). D는 경로만 다르고 scheme, host, port가 전부 일치하니까 Same-Origin이야.

문제 2¶

CORS에서 Preflight(OPTIONS) 요청이 발생하는 조건은?

A. 모든 Cross-Origin 요청 B. GET 요청에 Cookie가 포함된 경우 C. PUT 요청 또는 Content-Type: application/json 등 Simple Request 조건을 벗어난 경우 D. 서버가 CORS 헤더를 보내지 않는 경우

정답 보기

C. PUT 요청 또는 Content-Type: application/json 등 Simple Request 조건을 벗어난 경우

Simple Request(GET/HEAD/POST + 기본 Content-Type + 기본 헤더) 조건을 벗어나면 브라우저가 Preflight를 보내. JSON API 호출하면 거의 다 걸린다. Content-Type: application/json이 Simple의 허용 목록에 없으니까.

추가 보강: 실전 CORS 트러블슈팅 (NexCMS 사례)¶

왜 보강하나?

이론은 위에서 다 배웠어. 근데 실전에서는 "교과서에 없는 상황"이 터져. CDN + S3 + Spring이 동시에 CORS 헤더를 넣으면? 아래가 진짜 삽질 기록이야.

1. CORS 헤더 중복 문제 — 이게 진짜 지옥이야¶

위에서 배운 CORS는 "헤더가 있으면 통과, 없으면 차단"이었지? 근데 헤더가 2개면?

access-control-allow-origin: *                      ← CDN이 추가
access-control-allow-origin: http://localhost:8083   ← S3가 추가

브라우저 반응

"Access-Control-Allow-Origin 헤더에 여러 값이 포함됨. 하나만 허용됨." → 거부

왜 이런 일이 생기냐면:

sequenceDiagram
    participant B as 브라우저 (HLS.js)
    participant C as CDN
    participant S as S3 원본
    B->>C: GET playlist.m3u8 (Origin: http://localhost:8083)
    C->>S: 파일 요청 (Origin 전달)
    S-->>C: 응답 + ACAO: http://localhost:8083
    C-->>B: 응답 + ACAO: * (CDN 자체 추가) + ACAO: http://localhost:8083 (S3 통과)
    Note over B: 헤더 2개 → CORS 에러!

CDN이 S3 응답의 CORS 헤더를 그대로 통과시키면서 자기 것도 추가해서 2개가 되는 거야.

해결 원칙

Gateway + Backend 아키텍처에서는 한쪽에서만 CORS 설정해야 해. Nginx + Spring이든, CDN + S3이든 똑같은 원칙이야.

시도 결과
S3 CORS 삭제 (deleteBucketCors) iwinv 플랫폼 레벨 강제 → 안 먹힘
S3 CORS 덮어쓰기 (no-match origin) 마찬가지로 안 먹힘
서버 프록시 같은 출처로 요청 → CORS 자체 발생 안 함 → 해결

2. <img> vs XHR — 왜 썸네일은 되고 영상은 안 되나?¶

같은 CDN URL인데:

요청 방식 CORS 체크 결과
<img src="CDN URL"> 안 함 잘 됨
<script src="CDN URL"> 안 함 잘 됨
XMLHttpRequest (HLS.js) 함 CORS 에러
fetch() 함 CORS 에러

핵심

HTML 태그(<img>, <script>, <link>)는 cross-origin이어도 CORS를 안 따져. JavaScript API(XHR, fetch)는 반드시 CORS를 체크해. HLS.js는 XHR로 .m3u8과 .ts를 가져오니까 CORS 필수야.

3. Referer vs Origin — 헷갈리면 안 돼¶

헤더 목적 누가 보내나 제어 가능?
Referer 이전 페이지 URL 브라우저 자동 <meta name="referrer">
Origin 요청 출처 (CORS용) cross-origin XHR/fetch 시 자동 제어 불가 
<!-- Referer만 제거 (CDN 핫링크 방지 우회) -->
<meta name="referrer" content="no-referrer">

정리

  • CDN이 Referer 기반으로 403 차단? → no-referrer로 해결
  • CDN CORS 중복? → no-referrer로 해결 안 됨 (Origin은 그대로 감)
  • Referer 문제와 CORS 문제는 별개야

4. allowedOriginPatterns("*") + allowCredentials(true)¶

Spring WebConfig에서 이 조합 쓰면:

registry.addMapping("/**")
    .allowedOriginPatterns("*")   // 모든 Origin 허용
    .allowCredentials(true);       // 쿠키 전송 허용

*가 아니라 Origin 에코야

credentials: true일 때 Spring은 Access-Control-Allow-Origin: *를 안 보내. 대신 요청의 Origin 값을 그대로 반환해: Origin: http://localhost:8083 → ACAO: http://localhost:8083

5. 서버 프록시 우회 패턴¶

CORS는 브라우저 전용이라고 위에서 배웠지? 그걸 이용하는 거야.

sequenceDiagram
    participant B as 브라우저 (HLS.js)
    participant P as NexCMS 서버 (/cdn-proxy)
    participant C as CDN
    B->>P: GET /cdn-proxy/nexcms/.../playlist.m3u8 (같은 출처!)
    P->>C: GET https://cdn.../nexcms/.../playlist.m3u8 (서버-서버, CORS 없음)
    C-->>P: 응답 (CORS 헤더 있어도 상관없음)
    P-->>B: 응답 전달 (CORS 헤더 없이)
    Note over B: 같은 출처 → CORS 체크 안 함 → 성공!
항목 직접 CDN 접근 프록시 경유
CORS 중복 헤더 에러 없음 (같은 출처)
서버 부하 없음 영상 트래픽 경유
CDN 캐시 직접 활용 CDN은 활용하되 서버 경유

트레이드오프

프록시는 CORS를 완전히 해결하지만, 영상 트래픽이 서버를 거친다는 단점이 있어. CDN 인프라에서 CORS 중복이 수정되면 프록시를 제거하고 CDN 직접 접근으로 전환하는 게 맞아.

문제 3¶

Postman에서는 API가 정상 동작하는데 브라우저에서 CORS 에러가 나는 이유는?

A. Postman이 CORS 헤더를 자동으로 추가해서 B. Postman은 브라우저가 아니라 Same-Origin Policy가 적용되지 않아서 C. 서버가 Postman 요청만 허용해서 D. 브라우저의 캐시 문제

정답 보기

B. Postman은 브라우저가 아니라 Same-Origin Policy가 적용되지 않아서

CORS는 브라우저 전용 보안 정책이야. Postman, curl, 서버 간 통신에는 적용 안 돼. 서버는 누가 요청하든 동일하게 응답을 보내. 다만 브라우저는 응답을 받고 나서 CORS 헤더를 확인하고, 없으면 JS에게 안 넘겨주는 거야. 이게 "서버가 막는 게 아니라 브라우저가 막는 것"의 정확한 의미야.

문제 4¶

Access-Control-Allow-Origin: *와 Access-Control-Allow-Credentials: true를 동시에 쓰면?

A. 정상 동작 B. 서버 에러 발생 C. 브라우저가 거부 D. 쿠키가 평문으로 노출

정답 보기

C. 브라우저가 거부

*(모든 Origin 허용)과 credentials: true(쿠키 포함)는 동시에 쓸 수 없어. 모든 사이트에서 쿠키를 포함한 요청이 가능하면 보안 대참사니까. credentials: true를 쓰려면 정확한 Origin을 명시해야 해. 예: Access-Control-Allow-Origin: https://www.example.com

문제 5¶

CDN 응답에 Access-Control-Allow-Origin 헤더가 *와 http://localhost:8083 두 개 있다. 브라우저는 어떻게 반응해?

A. *를 우선 적용해서 통과 B. http://localhost:8083을 우선 적용해서 통과 C. 중복 헤더로 CORS 에러 발생 D. 서버에 재요청

정답 보기

C. 중복 헤더로 CORS 에러 발생

CORS 규격상 Access-Control-Allow-Origin은 정확히 1개만 허용돼. 2개 이상이면 "multiple values" 에러로 거부해. 값이 뭐든 상관없어.

문제 6¶

<img src='https://cdn.example.com/thumb.jpg'>는 CORS 에러 안 나는데 fetch('https://cdn.example.com/video.m3u8')는 CORS 에러가 난다. 왜?

A. 이미지 파일은 CORS가 적용 안 돼서 B. <img> 태그는 브라우저가 CORS 체크를 안 해서 C. m3u8 파일이 비공개라서 D. fetch는 POST 요청이라서

정답 보기

B. <img> 태그는 브라우저가 CORS 체크를 안 해서

HTML 태그(<img>, <script>, <link>)는 cross-origin이어도 CORS를 따지지 않아. JavaScript API(fetch, XMLHttpRequest)만 CORS를 체크해. 같은 URL이라도 요청 방식에 따라 결과가 달라지는 거야.

문제 7¶

<meta name='referrer' content='no-referrer'>를 설정하면 CORS 중복 헤더 문제가 해결될까?

A. 해결됨 — Referer와 Origin 둘 다 제거돼서 B. 해결 안 됨 — Origin 헤더는 여전히 전송돼서 C. 해결됨 — CDN이 Origin 대신 Referer를 체크하니까 D. 부분 해결 — <img>만 되고 XHR은 안 됨

정답 보기

B. 해결 안 됨 — Origin 헤더는 여전히 전송돼서

no-referrer는 Referer 헤더만 제거해. Origin 헤더는 브라우저가 cross-origin XHR/fetch 시 자동으로 추가하는 거라 제어 불가야. CDN Referer 차단(403)은 해결되지만, CORS 중복 헤더 문제는 Origin 기반이라 별개 문제야.
