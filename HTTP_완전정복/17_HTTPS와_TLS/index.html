17. HTTPS와 TLS¶

이 챕터의 핵심

HTTP는 평문이야. 데이터가 그대로 날아간다. 카페 와이파이에서 로그인하면? 옆자리 사람이 네 비밀번호를 볼 수 있어. HTTPS = HTTP + TLS. HTTP를 암호화해서 보내는 거다. 이거 모르면 보안 사고 터졌을 때 왜 터졌는지도 모른다.

HTTP는 평문이다¶

HTTP로 보내는 모든 데이터는 암호화 없이 그대로 네트워크를 타고 간다.

POST /login HTTP/1.1
Host: example.com
Content-Type: application/x-www-form-urlencoded

username=junsung&password=mySecret123

이게 네트워크 위를 그대로 날아간다고. password=mySecret123이 평문으로.

카페 시나리오

  1. 너가 카페 와이파이에 연결
  2. http://example.com/login에 로그인
  3. 같은 와이파이의 누군가가 패킷 캡처 도구(Wireshark 등) 실행
  4. 너의 아이디, 비밀번호가 그대로 보임

이게 HTTP의 현실이야. 평문 통신은 엽서랑 같아. 배달 과정에서 누구나 읽을 수 있다.

그래서 HTTPS가 필요하다.

HTTPS란¶

HTTPS = HTTP + TLS(Transport Layer Security)

HTTP 프로토콜 자체는 바뀌는 게 없어. 요청-응답 구조, 헤더, 메서드 전부 동일. 다만 HTTP 메시지를 보내기 전에 TLS라는 암호화 계층을 거쳐서 암호화한 후 전송하는 것이다.

graph LR
    subgraph HTTP
        A[요청/응답 메시지] -->|평문 그대로| B[네트워크]
    end
graph LR
    subgraph HTTPS
        C[요청/응답 메시지] -->|TLS 암호화| D[암호화된 데이터] -->|네트워크| E[TLS 복호화] --> F[요청/응답 메시지]
    end
구분 HTTP HTTPS
URL http:// https://
포트 80 443
암호화 없음 (평문) TLS로 암호화
속도 약간 빠름 TLS 핸드셰이크 오버헤드 있음 (거의 체감 안 됨)
브라우저 표시 "주의 요함" 경고 자물쇠 아이콘

SSL? TLS?

SSL(Secure Sockets Layer)은 TLS의 옛날 이름이야. SSL 3.0 → TLS 1.0 → TLS 1.1 → TLS 1.2 → TLS 1.3 (현재 표준) SSL은 보안 취약점 때문에 더 이상 안 써. 하지만 습관적으로 "SSL 인증서"라고 부르는 사람이 많아. 실체는 TLS다.

TLS가 보장하는 3가지¶

TLS는 통신에서 3가지를 보장한다. 이 3가지가 보안의 근간이야.

1. 기밀성 (Confidentiality)¶

제3자가 내용을 볼 수 없다.

데이터를 암호화해서 전송하니까, 중간에 패킷을 가로채도 의미 없는 데이터만 보인다.

평문:   username=junsung&password=mySecret123
암호화: 3a7f2b9c8d1e4f6a...b2c3d4e5f6 (해독 불가)

2. 무결성 (Integrity)¶

데이터가 중간에 변조되지 않았음을 보장한다.

메시지에 MAC(Message Authentication Code)을 붙여서, 받는 쪽이 "이 데이터가 변조됐는지" 확인할 수 있다.

공격자가 데이터를 가로채서 수정하면? MAC 검증 실패. 변조 감지.

3. 인증 (Authentication)¶

서버가 진짜인지 확인한다.

내가 naver.com에 접속했을 때, 이 서버가 진짜 네이버인지 어떻게 알아? 인증서(Certificate)로 확인한다. 가짜 서버는 진짜 인증서를 못 만들어.

보장 위협 설명
기밀성 도청(Eavesdropping) 암호화로 내용 못 봄
무결성 변조(Tampering) MAC으로 변조 감지
인증 위장(Spoofing) 인증서로 서버 신원 확인

3가지 다 필요한 이유

기밀성만 있으면? 가짜 서버와 암호화 통신할 수 있어. (인증 없으니까) 인증만 있으면? 진짜 서버지만 데이터가 평문이라 도청당해. (기밀성 없으니까) 3가지가 세트야. 하나라도 빠지면 구멍이 생긴다.

대칭키 vs 비대칭키 암호화¶

TLS 핸드셰이크를 이해하려면, 두 가지 암호화 방식을 먼저 알아야 해.

대칭키 (Symmetric Key)¶

하나의 키로 암호화하고, 같은 키로 복호화.

암호화: 평문 + 키A → 암호문
복호화: 암호문 + 키A → 평문
장점 단점
빠르다 키를 어떻게 안전하게 전달하지?

문제: 처음에 서버와 클라이언트가 "이 키를 쓰자"고 약속해야 하는데, 그 키를 네트워크로 보내면 누가 가로채잖아. 키를 안전하게 보내려면 암호화가 필요한데, 암호화하려면 키가 필요하고... 닭과 달걀 문제.

비대칭키 (Asymmetric Key)¶

공개키로 암호화하면, 개인키로만 복호화.

암호화: 평문 + 공개키 → 암호문
복호화: 암호문 + 개인키 → 평문
장점 단점
공개키는 누구나 알아도 됨. 개인키 없으면 복호화 못 해 느리다 (대칭키의 100~1000배)

공개키는 말 그대로 공개해도 괜찮아. 공개키로 암호화한 건 개인키 가진 사람만 풀 수 있으니까.

TLS의 해법: 둘 다 쓴다¶

TLS는 비대칭키로 대칭키를 안전하게 교환하고, 이후 통신은 대칭키로 한다.

graph TD
    A[비대칭키 암호화] -->|대칭키를 안전하게 교환| B[대칭키 확보]
    B -->|이후 모든 통신| C[대칭키로 암호화/복호화]

    style A fill:#ff9999
    style B fill:#99ff99
    style C fill:#9999ff

비대칭키의 안전성 + 대칭키의 속도. 둘의 장점을 합친 거야.

TLS 핸드셰이크¶

HTTPS 연결이 시작될 때, 실제 데이터를 주고받기 전에 TLS 핸드셰이크가 일어난다. 이 과정에서 암호화 방식을 합의하고, 인증서를 검증하고, 대칭키를 교환한다.

sequenceDiagram
    participant C as 클라이언트 (브라우저)
    participant S as 서버

    Note over C,S: TLS 핸드셰이크 시작

    C->>S: 1. Client Hello
    Note right of C: "나 이런 암호화 지원해"<br/>+ 클라이언트 랜덤값

    S-->>C: 2. Server Hello
    Note left of S: "그럼 이거 쓰자"<br/>+ 서버 랜덤값

    S-->>C: 3. 인증서 전송
    Note left of S: "내 신분증이야. 확인해봐"

    Note over C: 4. 인증서 검증
    Note over C: CA 체인으로 신뢰 확인

    C->>S: 5. 키 교환
    Note right of C: 비대칭키로 암호화해서<br/>대칭키 재료 전송

    Note over C,S: 6. 양쪽이 동일한 대칭키 생성

    Note over C,S: 핸드셰이크 완료. 이후 대칭키로 통신

    C->>S: 7. 암호화된 HTTP 요청
    S-->>C: 8. 암호화된 HTTP 응답

단계별로 뜯어보자.

1단계: Client Hello¶

클라이언트가 서버에 먼저 손을 내민다.

보내는 정보:

  • 지원하는 TLS 버전 (TLS 1.2, TLS 1.3)
  • 지원하는 암호화 방식 목록 (Cipher Suites)
  • 클라이언트 랜덤값 (나중에 대칭키 생성에 사용)

2단계: Server Hello¶

서버가 응답한다.

  • 사용할 TLS 버전 선택
  • 사용할 암호화 방식 선택
  • 서버 랜덤값

3단계: 인증서 전송¶

서버가 자기 인증서(Certificate)를 보낸다. "나 진짜 naver.com이야" 를 증명하는 신분증.

4단계: 인증서 검증¶

클라이언트(브라우저)가 인증서를 검증한다.

  • 이 인증서가 신뢰할 수 있는 CA(인증 기관)가 발급한 건지?
  • 인증서의 도메인과 접속하려는 도메인이 일치하는지?
  • 인증서가 만료되지 않았는지?

검증 실패하면 브라우저가 "이 연결은 안전하지 않습니다" 경고를 띄운다.

5단계: 키 교환¶

인증서 검증 통과하면, 클라이언트가 대칭키를 만들기 위한 재료를 서버에 보낸다. 이때 서버의 공개키(인증서에 포함)로 암호화해서 보내니까, 개인키를 가진 서버만 복호화할 수 있다.

6단계: 대칭키 생성¶

양쪽이 가진 정보(클라이언트 랜덤 + 서버 랜덤 + 교환한 키 재료)로 동일한 대칭키를 각자 생성한다.

7단계 이후: 암호화 통신¶

이제 대칭키로 HTTP 메시지를 암호화/복호화한다. 빠르고 안전하게.

TLS 1.3에서는 더 빨라졌다

TLS 1.2는 핸드셰이크에 2 왕복(Round Trip)이 필요한데, TLS 1.3은 1 왕복으로 줄였다. 심지어 이전에 연결한 적 있으면 0 왕복(0-RTT)도 가능해. 그래서 "HTTPS가 느리다"는 옛말이야.

인증서 (Certificate)¶

SSL 인증서란¶

인증서는 서버의 신분증이다.

인증서에 들어있는 정보:

항목 내용
도메인 이 인증서가 유효한 도메인 (예: *.naver.com)
공개키 서버의 공개키 (핸드셰이크에서 사용)
발급자 어떤 CA가 발급했는지
유효기간 언제부터 언제까지 유효한지
서명 CA의 디지털 서명 (위조 방지)

CA (Certificate Authority)¶

CA는 인증서를 발급하는 신뢰받는 기관이다.

아무나 인증서를 만들면 의미가 없잖아. 가짜 서버도 "나 진짜야"라고 인증서를 만들 수 있으니까.

그래서 신뢰받는 제3자(CA)가 "이 서버는 진짜 이 도메인의 소유자가 맞다"고 보증한다.

주요 CA:

  • Let's Encrypt: 무료. 자동화 지원. 개인/소규모 프로젝트에서 많이 씀
  • DigiCert: 유료. 기업용
  • Sectigo(구 Comodo): 유료. 기업용
  • GlobalSign: 유료. 기업용

인증서 체인 (Certificate Chain)¶

인증서 검증은 체인 구조로 동작한다.

graph TD
    A[루트 CA 인증서] -->|서명| B[중간 CA 인증서]
    B -->|서명| C[서버 인증서]

    style A fill:#ff9999
    style B fill:#ffcc99
    style C fill:#99ccff
인증서 역할 어디에 있어?
루트 CA 인증서 최상위 신뢰 기관 브라우저/OS에 미리 설치되어 있음
중간 CA 인증서 루트 CA가 권한을 위임한 기관 서버가 핸드셰이크 시 전송
서버 인증서 실제 서버의 인증서 서버가 핸드셰이크 시 전송

검증 과정:

  1. 서버 인증서를 중간 CA가 서명했나? (중간 CA의 공개키로 검증)
  2. 중간 CA 인증서를 루트 CA가 서명했나? (루트 CA의 공개키로 검증)
  3. 루트 CA를 브라우저가 신뢰하나? (미리 설치된 목록에 있나?)

전부 통과하면 브라우저 주소창에 자물쇠가 뜬다.

자물쇠 = 암호화일 뿐, 안전한 사이트라는 뜻은 아냐

피싱 사이트도 인증서를 받을 수 있어. Let's Encrypt는 도메인 소유 확인만 하니까. https://naver-login.evil.com 같은 피싱 사이트도 자물쇠가 뜰 수 있다. 자물쇠는 "이 서버와의 통신이 암호화되어 있다"는 뜻이지, "이 사이트가 안전하다"는 뜻이 아니야.

HTTP vs HTTPS 비교¶

항목 HTTP HTTPS
프로토콜 HTTP HTTP over TLS
URL http:// https://
기본 포트 80 443
암호화 없음 TLS 암호화
인증서 불필요 필요
SEO 불이익 Google이 가산점 줌
브라우저 "주의 요함" 표시 자물쇠 표시
성능 핸드셰이크 없음 TLS 핸드셰이크 오버헤드 (미미함)

2026년 현재, HTTP는 거의 안 써

주요 브라우저들은 HTTP 사이트에 경고를 띄우고, Google은 HTTPS를 SEO 랭킹 요소로 반영해. Let's Encrypt로 무료 인증서도 쉽게 받을 수 있으니까 새로 만드는 서비스에서 HTTP를 쓸 이유가 없다.

개발할 때 어떤 영향이 있어?¶

localhost 개발¶

개발할 때 http://localhost:8080으로 개발하는 경우가 많아. 이건 HTTPS가 아닌데 괜찮을까?

  • localhost는 브라우저가 안전한 컨텍스트(Secure Context)로 취급해 준다
  • 그래서 localhost에서는 HTTP여도 대부분의 API가 동작해
  • 하지만 일부 기능(Geolocation, Service Worker 등)은 HTTPS 필요

로컬에서 HTTPS가 필요하면

mkcert 같은 도구로 로컬 인증서를 생성할 수 있어. 

mkcert localhost
# localhost.pem, localhost-key.pem 생성됨

Mixed Content 문제¶

HTTPS 페이지에서 HTTP 리소스를 불러오면 Mixed Content 에러가 발생한다.

<!-- HTTPS 페이지인데... -->
<img src="http://example.com/image.png">  <!-- HTTP 이미지 = Mixed Content -->
<script src="http://example.com/app.js"></script>  <!-- HTTP 스크립트 = 차단됨 -->
타입 동작 예시
Active Mixed Content 차단 JS, CSS, iframe
Passive Mixed Content 경고 (일부 차단) 이미지, 비디오, 오디오

Mixed Content는 보안 구멍

HTTPS 페이지에서 HTTP 스크립트를 불러오면? 공격자가 그 HTTP 스크립트를 중간에서 바꿔치기할 수 있어. HTTPS로 아무리 잘 암호화해도 HTTP 리소스 하나 때문에 무너진다. HTTPS 사이트는 모든 리소스도 HTTPS로 불러와야 해.

Spring Boot에서 HTTPS 설정¶

# application.yml
server:
  ssl:
    key-store: classpath:keystore.p12
    key-store-password: changeit
    key-store-type: PKCS12
  port: 8443

실무에서는 보통 Spring Boot 앞에 Nginx나 로드밸런서가 HTTPS를 처리(SSL Termination)하고, Spring Boot는 HTTP로 받는 구조가 많다.

주의사항 / 함정¶

흔한 오해들

1. "HTTPS면 100% 안전하다"

아니야. HTTPS는 전송 구간을 암호화하는 거야. 서버에 SQL Injection 취약점이 있으면? HTTPS여도 해킹당해. 서버 내부 로직의 보안은 HTTPS와 별개야.

2. "HTTPS는 느리다"

2010년대 초반 얘기야. TLS 1.3은 핸드셰이크가 1 왕복이고, HTTP/2는 HTTPS를 기본으로 쓰는데 오히려 HTTP/1.1보다 빨라. 현대에서 HTTPS 성능 걱정은 안 해도 돼.

3. "자체 서명 인증서 쓰면 되지"

개발 환경에서는 OK. 프로덕션에서? 브라우저가 경고 띄운다. 사용자가 "이 사이트 위험해요" 보면 도망가. CA가 발급한 인증서를 써야 해.

4. "인증서 갱신 안 해도 되지?"

인증서에는 유효기간이 있어. Let's Encrypt는 90일. 만료되면 사이트 접속이 안 돼. 자동 갱신 설정 반드시 해놔야 해.

정리¶

개념 한 줄 요약
HTTPS HTTP + TLS. HTTP를 암호화해서 전송
TLS 기밀성 + 무결성 + 인증을 보장하는 암호화 프로토콜
대칭키 하나의 키로 암호화/복호화. 빠르지만 키 교환 문제
비대칭키 공개키/개인키 쌍. 안전하지만 느림
TLS 핸드셰이크 암호화 합의 + 인증서 검증 + 대칭키 교환 과정
인증서 서버의 신분증. CA가 발급
CA 인증서를 발급하는 신뢰받는 기관
인증서 체인 루트 CA -> 중간 CA -> 서버 인증서
Mixed Content HTTPS에서 HTTP 리소스 로드 시 보안 문제

한 문장 정리

HTTPS는 TLS로 HTTP를 암호화하여 도청, 변조, 위장으로부터 통신을 보호한다. TLS 핸드셰이크에서 인증서를 검증하고 대칭키를 교환한 후, 이후 모든 통신을 대칭키로 암호화한다.

확인 문제¶

문제 1¶

TLS가 보장하는 3가지가 아닌 것은?

A. 기밀성 (Confidentiality) B. 무결성 (Integrity) C. 인증 (Authentication) D. 가용성 (Availability)

정답 보기

D. 가용성 (Availability)

TLS는 기밀성, 무결성, 인증 3가지를 보장해. 가용성(서비스가 항상 동작하는 것)은 TLS의 영역이 아니야. DDoS 공격으로 서버를 마비시키는 건 HTTPS여도 막을 수 없다.

문제 2¶

TLS 핸드셰이크에서 비대칭키 암호화의 역할은?

A. 모든 HTTP 데이터를 암호화 B. 대칭키를 안전하게 교환 C. 인증서를 생성 D. DNS를 암호화

정답 보기

B. 대칭키를 안전하게 교환

비대칭키는 느리니까 전체 통신에 쓰면 성능이 안 나와. 핸드셰이크 때 대칭키(재료)를 안전하게 교환하는 데만 쓰고, 이후 실제 데이터 통신은 빠른 대칭키로 한다. 비대칭키의 안전성 + 대칭키의 속도를 합친 거야.

문제 3¶

HTTPS 페이지에서 HTTP로 JavaScript를 로드하면 어떻게 되나?

A. 정상 동작 B. 경고만 표시 C. 브라우저가 차단 D. 서버에서 거부

정답 보기

C. 브라우저가 차단

JavaScript, CSS 같은 Active Mixed Content는 브라우저가 차단한다. HTTP 스크립트를 중간에서 변조할 수 있으니까. HTTPS 사이트에서 HTTP 리소스를 쓰면 안 돼. 모든 리소스를 HTTPS로 통일해야 한다.

문제 4¶

브라우저 주소창의 자물쇠 아이콘이 의미하는 것은?

A. 이 사이트는 악성코드가 없다 B. 이 사이트는 합법적인 사이트다 C. 이 서버와의 통신이 암호화되어 있다 D. 이 사이트의 데이터가 안전하게 보관되어 있다

정답 보기

C. 이 서버와의 통신이 암호화되어 있다

자물쇠는 인증서가 유효하고, TLS로 암호화 통신하고 있다는 뜻이야. 피싱 사이트도 인증서를 받을 수 있기 때문에, 자물쇠가 있다고 "안전한 사이트"는 아니야. 자물쇠 = 암호화 통신이지, 사이트 신뢰도가 아니다.
