28. 전체 흐름 통합¶

한 줄 요약

01장부터 27장까지 배운 모든 개념이 하나의 시나리오로 연결된다. 이 챕터를 읽고 전부 이해되면, HTTP 완전정복 끝.

시나리오¶

"Zoom 미팅이 끝나면 NexClass에 출결이 자동으로 처리된다."

이 한 문장 안에 01~27장 전부가 들어있어. 하나씩 풀어보자.

전체 흐름 한눈에¶

sequenceDiagram
    participant U as 사용자 (브라우저)
    participant D as DNS 서버
    participant Z as Zoom 서버
    participant N as NexClass 서버

    rect rgb(230, 240, 255)
    Note over U,Z: Phase 1: 사용자가 Zoom 미팅 참여
    U->>D: "zoom.us의 IP가 뭐야?" (DNS 조회)
    D-->>U: "141.193.213.10" (IP 응답)
    U->>Z: TCP 3-way handshake (SYN → SYN-ACK → ACK)
    U->>Z: TLS 핸드셰이크 (인증서 검증 + 키 교환)
    U->>Z: HTTPS GET /j/123456 (미팅 참여)
    Z-->>U: 200 OK (미팅 페이지)
    end

    rect rgb(255, 245, 230)
    Note over Z: Phase 2: 미팅 진행 및 종료
    Note over Z: meeting.ended 이벤트 발생
    end

    rect rgb(230, 255, 230)
    Note over Z,N: Phase 3: Webhook 발송
    Z->>Z: message = "v0:" + timestamp + ":" + body
    Z->>Z: signature = HMAC-SHA256(secret, message)
    Z->>N: POST /webhook/zoom (서명 + body)
    end

    rect rgb(255, 230, 230)
    Note over N: Phase 4: NexClass 처리
    N->>N: 타임스탬프 검증 (5분 이내?)
    N->>N: 서명 검증 (HMAC 일치?)
    N->>N: 이벤트 처리 (출결 기록)
    N-->>Z: 200 OK
    end

Phase 1: 사용자가 Zoom 미팅에 참여한다¶

사용자가 브라우저에서 https://zoom.us/j/123456을 입력해. 이 순간부터 01~04장이 동작해.

Step 1: DNS 조회 (03장)¶

사용자: "zoom.us"가 어디야?
DNS:    "141.193.213.10이야"

브라우저는 도메인 이름으로 통신 못 해. IP 주소가 필요하니까 DNS에 물어봐.

flowchart LR
    A[zoom.us] -->|DNS 조회| B[141.193.213.10]
    B --> C[이 IP로 연결 시작]
개념 03장 연결
도메인 → IP 변환 DNS의 핵심 역할
DNS 캐시 매번 조회하면 느리니까 캐싱
DNS 서버 계층 로컬 → ISP → Root → TLD → Authoritative

Step 2: TCP 연결 (02장)¶

IP를 알았으니 연결해야지. TCP 3-way handshake.

사용자 → SYN → Zoom         "연결하자"
사용자 ← SYN-ACK ← Zoom     "좋아, 연결하자"
사용자 → ACK → Zoom         "오케이, 연결!"
개념 02장 연결
TCP 신뢰성 있는 통신. 순서 보장, 손실 복구
포트 HTTPS = 443번 포트
3-way handshake SYN → SYN-ACK → ACK

Step 3: TLS 핸드셰이크 (17장)¶

TCP 연결 위에 암호화 레이어를 까는 과정.

1. ClientHello: "나 이런 암호화 방식 지원해"
2. ServerHello: "이 방식으로 하자" + 인증서 전송
3. 인증서 검증: "이 인증서 진짜야?" (CA 확인)
4. 키 교환: 대칭키 생성 (이후 통신에 사용)
5. 암호화 통신 시작
개념 17장 연결
HTTPS HTTP + TLS. 암호화된 HTTP
인증서 서버의 신원 증명
대칭키 교환 비대칭키로 대칭키를 안전하게 교환 (21장)

Step 4: HTTP 요청 (04장, 06장, 07장)¶

암호화된 채널이 만들어졌으니 이제 HTTP 요청을 보내.

GET /j/123456 HTTP/1.1
Host: zoom.us
User-Agent: Mozilla/5.0 ...
Accept: text/html
Cookie: _zm_ssid=abc123
요소 관련 챕터 설명
GET 6장 HTTP 메서드. 데이터 요청
/j/123456 5장 URL 경로. 미팅 ID
Host: zoom.us 9장 필수 헤더
Cookie 16장 세션 유지

Zoom 서버가 200 OK로 미팅 페이지를 반환하고, 사용자는 미팅에 참여한다.

Phase 2: 미팅이 끝난다¶

미팅이 진행되다가 종료돼. 이 순간 Zoom 서버 내부에서:

이벤트 시스템: "meeting.ended 이벤트 발생!"
Webhook 엔진: "이 미팅에 구독된 Webhook이 있나?"
             → "있다! https://nexclass.com/webhook/zoom"
             → "POST 보내자"

이건 Zoom 내부 동작이야. 25장에서 배운 "이벤트 발생 → Webhook 발송" 흐름.

Phase 3: Zoom이 우리 서버에 Webhook을 보낸다¶

이것도 HTTP야¶

Zoom이 우리 서버에 보내는 것도 결국 HTTP 요청이야. Phase 1이랑 같은 과정을 거쳐.

Zoom → DNS 조회: nexclass.com → IP
Zoom → TCP 3-way handshake (02장)
Zoom → TLS 핸드셰이크 (17장)
Zoom → HTTP POST 요청 (아래)

Webhook HTTP 요청 분석¶

POST /webhook/zoom HTTP/1.1
Host: nexclass.com
Content-Type: application/json
x-zm-signature: v0=a1b2c3d4e5f6789...
x-zm-request-timestamp: 1742306400

{
  "event": "meeting.ended",
  "event_ts": 1742306400000,
  "payload": {
    "account_id": "ABCD1234",
    "object": {
      "id": "123456789",
      "topic": "HTTP 완전정복 스터디",
      "start_time": "2026-03-18T14:00:00Z",
      "end_time": "2026-03-18T15:00:00Z",
      "duration": 60,
      "host_id": "xyz789"
    }
  }
}

이 요청 하나에 얼마나 많은 개념이 들어있는지 보자:

요소 값 관련 챕터 설명
메서드 POST 6장 데이터를 body에 담아 전송
경로 /webhook/zoom 5장 Webhook 엔드포인트 URL
Content-Type application/json 10장 body가 JSON임을 명시
x-zm-signature v0=a1b2c3... 9장, 22장, 26장 커스텀 헤더. HMAC 서명
x-zm-request-timestamp 1742306400 9장, 26장 커스텀 헤더. Replay Attack 방어
Body JSON 객체 10장, 25장 이벤트 데이터
HTTPS 암호화 전송 17장 전송 구간 보호

서명은 어떻게 만들었어? (22장, 26장)¶

Zoom이 이 요청을 보내기 전에:

1. message = "v0:" + "1742306400" + ":" + body(JSON 문자열)
2. signature = "v0=" + HMAC-SHA256(Secret Token, message)
3. x-zm-signature 헤더에 서명 넣기
4. x-zm-request-timestamp 헤더에 타임스탬프 넣기
5. POST 전송

Phase 4: NexClass 서버가 처리한다¶

Step 1: 요청 수신 (27장)¶

Spring Boot의 @PostMapping("/webhook/zoom")이 이 요청을 받아.

@PostMapping("/zoom")
public ResponseEntity<?> handleZoomWebhook(
        @RequestBody String body,              // JSON body
        @RequestHeader("x-zm-signature") ...,  // 서명
        @RequestHeader("x-zm-request-timestamp") ...  // 타임스탬프
)

Step 2: 타임스탬프 검증 (26장)¶

요청 시간: 1742306400 (2026-03-18 15:00:00)
현재 시간: 1742306520 (2026-03-18 15:02:00)
차이: 120초 = 2분 → 5분 이내 → 통과!

Step 3: 서명 검증 (22장, 26장)¶

message = "v0:" + "1742306400" + ":" + body
expectedSignature = "v0=" + HMAC-SHA256(Secret Token, message)

expectedSignature == x-zm-signature?
→ 일치! Zoom이 보낸 진짜 요청이야.

Step 4: 이벤트 처리 (REST API, 14장)¶

event = "meeting.ended"
→ 미팅 ID: 123456789
→ 출결 처리 로직 실행
→ DB에 출석 기록 저장

Step 5: 응답 (08장)¶

HTTP/1.1 200 OK
Content-Length: 0

200 OK를 빠르게 돌려줘. Zoom은 이걸 받으면 "전달 성공"으로 기록해.

주의사항과 함정¶

함정 1: "이거 전부 순서대로 일어나는 거 아니야?"¶

Phase마다 순서가 있지만, 독립적이기도 해

Phase 1(사용자 미팅 참여)과 Phase 3(Webhook 발송)은 독립적이야. Phase 1은 사용자 브라우저 ↔ Zoom. Phase 3은 Zoom ↔ NexClass 서버. 두 개는 서로 다른 HTTP 연결이야.

함정 2: "Webhook도 브라우저가 보내는 거지?"¶

아니야

Webhook은 서버가 서버에게 보내는 거야. 브라우저 관여 없어. Zoom 서버가 NexClass 서버에 직접 HTTP POST. Server-to-Server 통신이야.

함정 3: "이 중에 하나만 잘 알면 되지 않아?"¶

전부 연결되어 있어

서명 검증(22장)을 이해하려면 해싱(20장)을 알아야 하고, Webhook(25장)을 이해하려면 HTTP POST(6장)를 알아야 하고, HTTPS(17장)를 이해하려면 TCP(2장)와 암호화(21장)를 알아야 해. 하나 빠지면 전체 그림이 안 그려져.

정리¶

Phase 핵심 관련 챕터
1. DNS 조회 도메인 → IP 03장
2. TCP 연결 3-way handshake 02장
3. TLS 핸드셰이크 암호화 채널 수립 17장, 21장
4. HTTP 요청 GET (페이지 요청) 04~10장
5. 미팅 종료 이벤트 발생 25장
6. Webhook 발송 HTTP POST + 서명 06장, 22장, 25장
7. 서명 검증 HMAC 비교 22장, 26장
8. 이벤트 처리 출결 기록 27장
9. 200 OK 응답 빠른 응답 08장, 25장

한 문장 정리

"Zoom 미팅이 끝나면 출결이 처리된다"는 한 문장 안에 DNS, TCP, TLS, HTTP 메서드, 헤더, body, 상태 코드, HMAC, 서명 검증, Webhook 전부가 들어있어. 이 흐름이 읽히면 01~27장 다 이해한 거야.

확인 문제¶

문제 1

사용자가 브라우저에서 https://zoom.us를 입력했을 때, 실제 HTTP 요청이 전송되기까지 거치는 단계를 순서대로 설명해봐. 최소 4단계.

정답 보기

  1. DNS 조회 (03장): zoom.us라는 도메인을 DNS 서버에 질의해서 IP 주소를 알아낸다. 캐시에 있으면 바로, 없으면 재귀적 조회.

  2. TCP 3-way handshake (02장): 알아낸 IP의 443번 포트로 TCP 연결을 수립한다. SYN → SYN-ACK → ACK.

  3. TLS 핸드셰이크 (17장): TCP 연결 위에 암호화 레이어를 만든다. ClientHello → ServerHello → 인증서 검증 → 키 교환.

  4. HTTP 요청 전송 (04장, 07장): 암호화된 채널을 통해 GET / HTTP/1.1 요청을 보낸다.

문제 2

Zoom이 NexClass에 Webhook을 보낼 때의 HTTP 요청에서, 각 구성 요소(메서드, 경로, 헤더, body)가 각각 어느 챕터의 어떤 개념인지 연결해봐.

정답 보기 

POST /webhook/zoom HTTP/1.1
- POST: 6장 HTTP 메서드. body에 데이터를 담아서 전송. - /webhook/zoom: 5장 URL 경로. Webhook 수신 엔드포인트.

Content-Type: application/json
- 10장 Content-Type. body가 JSON 형식임을 명시.

x-zm-signature: v0=a1b2c3...
x-zm-request-timestamp: 1742306400
- 9장 커스텀 헤더 (x- 접두사). - 22장 HMAC-SHA256 서명. - 26장 서명 검증, Replay Attack 방어용 타임스탬프.

{"event": "meeting.ended", ...}
- 10장 JSON body. 이벤트 데이터. - 25장 Webhook 이벤트 구조.

문제 3

HTTPS와 HMAC 서명 검증은 둘 다 보안 관련 기술이야. 이 시나리오에서 각각 어느 단계에서, 무엇을 보호하는지 구분해서 설명해봐.

정답 보기

HTTPS (TLS) - 17장: - 단계: TCP 연결 직후, HTTP 요청 전송 전. - 보호 대상: 전송 구간. 요청과 응답이 네트워크를 통해 이동하는 동안 도청/변조 방지. - 못 하는 것: "이 요청이 Zoom에서 왔는지" 확인 불가.

HMAC 서명 검증 - 22장, 26장: - 단계: NexClass 서버가 요청을 받은 후. - 보호 대상: 요청의 출처와 무결성. 이 요청이 Secret Token을 아는 Zoom이 보냈는지 확인. - 못 하는 것: 전송 중 도청 방지 불가 (그건 HTTPS 역할).

정리: HTTPS = 전송 보안 (파이프 보호), HMAC = 출처 인증 (발신자 확인). 역할이 다르다.

문제 4

만약 서명 검증을 구현하지 않으면 (26장 내용 무시), 이 시나리오에서 어떤 공격이 가능해져? 구체적인 공격 시나리오를 하나 말해봐.

정답 보기

가짜 Webhook 공격:

  1. 해커가 /webhook/zoom URL을 알아냄 (URL은 추측 가능)
  2. 해커가 가짜 meeting.ended 이벤트를 POST로 전송
  3. 서명 검증이 없으니까 NexClass 서버가 진짜로 믿어버림
  4. 가짜 미팅 ID, 가짜 참가자로 출결이 처리됨
  5. 수업에 안 들어온 학생이 출석 처리될 수 있음

서명 검증이 있으면? 해커는 Secret Token을 모르니까 유효한 서명을 만들 수 없어. 서명 불일치 → 401 Unauthorized → 공격 실패.

문제 5

이 전체 시나리오에서 "인코딩(19장)", "해싱(20장)", "암호화(21장)"가 각각 어디에서 사용되는지 찾아봐. 3개 모두 역할이 다르다는 걸 설명해.

정답 보기

인코딩 (19장) - 형식 변환 (누구나 복원 가능): - HMAC 결과 바이트를 Hex 인코딩해서 문자열로 변환 (27장 Hex.encodeHexString) - JSON body의 문자를 UTF-8 인코딩으로 바이트 변환 - 비밀도 아무것도 없어. 그냥 형식 바꾸기.

해싱 (20장) - 단방향 변환 (복원 불가): - HMAC-SHA256의 핵심 엔진이 SHA-256 해싱 - 입력(message + key)에서 고정 길이 해시값 생성 - 해시에서 원본을 역산할 수 없음

암호화 (21장) - 양방향 변환 (키가 있으면 복원 가능): - TLS 핸드셰이크에서 비대칭키(RSA)로 대칭키 교환 - 이후 통신은 대칭키(AES)로 암호화 - 키가 있으면 복호화 가능. 없으면 불가능.

정리:

인코딩 해싱 암호화
방향 양방향 단방향 양방향
키 필요 불필요 불필요(HMAC은 필요) 필요
목적 형식 변환 무결성/서명 기밀성
이 시나리오 Hex, UTF-8 HMAC-SHA256 TLS(AES, RSA)
