20. 해싱¶

한 줄 요약

해싱 = 입력을 고정 길이 값으로 변환. 되돌릴 수 없어. 인코딩과는 근본적으로 달라.

핵심 개념¶

해싱이 뭐야¶

해싱(Hashing)은 어떤 입력이든 고정된 길이의 값(해시값)으로 변환하는 거야. 그리고 가장 중요한 건: 되돌릴 수 없다.

입력: "password123"
SHA-256 해시: ef92b778bafe771e89245b89ecbc08a44a4e166c06659911881f383d4473e94f

입력: "password124" (1글자만 바꿈)
SHA-256 해시: 5765a0794e1b57dab06327aad4855792ad9b52d54e18e0e43c50fa5ac3fd7f27

완전히 다른 결과. 원본을 추측할 수도 없어.

비유하면 고기를 갈아서 다진 고기로 만드는 것. 다진 고기를 보고 원래 고기 모양을 복원할 수 있어? 불가능하지. 해싱도 마찬가지야.

해싱의 5가지 특성¶

이거 전부 이해해야 해싱을 제대로 아는 거야.

1. 단방향 (One-way)¶

해시값에서 원본을 복원할 수 없다.

"password123" → SHA-256 → ef92b778bafe... ✅ 가능
ef92b778bafe... → ??? → "password123"    ❌ 불가능

인코딩(19장)과의 결정적 차이:

인코딩 해싱
되돌리기 Base64Decode() → 원본 복원 불가능. 방법이 없음

2. 결정적 (Deterministic)¶

같은 입력 → 항상 같은 출력.

SHA-256("hello") = 항상 2cf24dba5fb0a30e26e83b2ac5b9e29e...
SHA-256("hello") = 항상 2cf24dba5fb0a30e26e83b2ac5b9e29e...

100번 돌려도 같은 값. 내일 돌려도 같은 값.

이 특성이 있어야 비교 검증이 가능하지. "비밀번호 맞는지 확인"이 이 원리야.

3. 고정 길이 (Fixed Length)¶

입력 크기가 뭐든 출력 크기는 항상 같다.

입력 크기 SHA-256 출력
"a" 1바이트 64자리 16진수 (256비트)
"hello world" 11바이트 64자리 16진수 (256비트)
영화 파일 (4GB) 4,000,000,000바이트 64자리 16진수 (256비트)

4GB 파일이든 1글자든 SHA-256 출력은 항상 256비트(64자리 16진수).

4. 눈사태 효과 (Avalanche Effect)¶

입력이 1비트만 바뀌어도 출력이 완전히 달라진다.

SHA-256("hello") = 2cf24dba5fb0a30e26e83b2ac5b9e29e1b161e5c1fa7425e73043362938b9824
SHA-256("hellp") = 7455a1fce06a43f7f3e4523bbaa2f5c0db28f1b6d0c24b9e4b9c1808ed0e2c32

'o' → 'p' 하나만 바꿨는데 결과가 완전히 다르다.

이 특성 덕분에 "비슷한 입력 → 비슷한 해시"로 원본을 추측하는 게 불가능해.

5. 충돌 저항성 (Collision Resistance)¶

서로 다른 두 입력이 같은 해시값을 만들기 어렵다.

이론적으로는 가능해. 입력은 무한한데 출력은 고정 길이니까.
하지만 실용적으로 찾는 게 거의 불가능해야 "충돌 저항성이 있다"고 한다.

SHA-256에서 충돌을 찾으려면?
→ 2^128번 연산이 필요 (우주가 멸망해도 못 끝남)

충돌이 발견되면?

그 해시 알고리즘은 폐기된다. MD5, SHA-1이 그래서 버려졌어. "충돌 찾았다" = "이 알고리즘 더 이상 신뢰 못 한다" = 끝.

주요 해시 알고리즘¶

MD5 - 퇴역한 베테랑¶

항목 값
출력 길이 128비트 (32자리 16진수)
속도 매우 빠름
보안 충돌 발견됨 (2004년)
현재 상태 보안 용도 금지 
MD5("hello") = 5d41402abc4b2a76b9719d911017c592

빠르니까 파일 체크섬(다운로드 검증) 정도에는 아직 쓰지만, 보안 목적으로는 절대 쓰지 마.

SHA-1 - 역시 퇴역¶

항목 값
출력 길이 160비트 (40자리 16진수)
보안 충돌 발견됨 (2017년, 구글이 증명)
현재 상태 보안 용도 폐기. Git에서 커밋 해시로 쓰이지만 보안과는 무관

SHA-256 - 현재 표준¶

항목 값
출력 길이 256비트 (64자리 16진수)
보안 충돌 발견 안 됨
현재 상태 현재 표준. 가장 많이 사용
사용처 비밀번호 해싱, 디지털 서명, HMAC, 블록체인 
SHA-256("hello") = 2cf24dba5fb0a30e26e83b2ac5b9e29e1b161e5c1fa7425e73043362938b9824

22장 연결

HMAC-SHA256에서 이 SHA-256이 사용돼. Zoom Webhook 서명 검증도 HMAC-SHA256이야. 22장에서 자세히 다룬다.

해싱 용도¶

1. 비밀번호 저장¶

절대 금지: 비밀번호 원문 저장

DB에 비밀번호를 그대로 저장하면? DB가 털리면 모든 사용자 비밀번호 노출. 해킹 사고의 대표적 원인이야.

올바른 방법: 해시값만 저장

sequenceDiagram
    participant U as 사용자
    participant S as 서버
    participant DB as DB

    Note over U,DB: 회원가입 시
    U->>S: 비밀번호: "mypass123"
    S->>S: SHA-256("mypass123") = a1b2c3d4...
    S->>DB: 해시값 저장: a1b2c3d4...

    Note over U,DB: 로그인 시
    U->>S: 비밀번호: "mypass123"
    S->>S: SHA-256("mypass123") = a1b2c3d4...
    S->>DB: 저장된 해시 조회
    DB-->>S: a1b2c3d4...
    S->>S: 계산한 해시 == 저장된 해시? → 일치!
    S-->>U: 로그인 성공

서버도 원본 비밀번호를 모른다. 해시값끼리 비교할 뿐이야.

2. 데이터 무결성 검증¶

파일 다운로드할 때 체크섬(checksum) 본 적 있지?

다운로드 파일: ubuntu-24.04.iso
SHA-256: a1b2c3d4e5f6... (공식 사이트에서 제공)

다운받은 후 SHA-256 계산 → 같으면: 파일 변조 안 됨 ✅
                          → 다르면: 파일이 변조됐거나 깨짐 ❌

3. 디지털 서명¶

문서에 서명할 때 문서 전체를 서명하는 게 아니라, 문서의 해시값에 서명해. 문서가 1MB든 1GB든 해시는 항상 256비트니까 서명이 빠르지.

이건 22장(HMAC과 디지털 서명)에서 자세히 다룬다.

Salt: 같은 비밀번호 문제 해결¶

문제: Rainbow Table 공격¶

해싱은 결정적이야. 같은 입력 → 항상 같은 출력. 그러면 해커가 이렇게 할 수 있어:

미리 계산해놓는 표 (Rainbow Table):

"password"    → 5e884898da28047151d0e56f8dc...
"123456"      → 8d969eef6ecad3c29a3a629280e...
"password123" → ef92b778bafe771e89245b89ecb...
"qwerty"      → 65e84be33532fb784c48129675f...
...수백만 개

DB에서 해시값 ef92b778bafe... 발견
→ 표에서 검색
→ "password123"이네!

해결: Salt (소금을 뿌려)¶

Salt = 비밀번호에 랜덤 값을 붙여서 해싱하는 것.

Salt 없이:
SHA-256("password123") = ef92b778bafe...  ← 항상 같은 값

Salt 있으면:
사용자 A의 Salt: "x7k9m2"
SHA-256("x7k9m2" + "password123") = 3f2a8b1c... ← 유니크한 값

사용자 B의 Salt: "p3q8w1"
SHA-256("p3q8w1" + "password123") = 9d4e7f2a... ← 완전히 다른 값

같은 비밀번호 "password123"인데 해시값이 다르다. Rainbow Table 무력화.

Salt 저장¶

저장 항목 비밀인가? 위치
Salt 아니야. 공개 가능 DB에 해시값 옆에 저장
해시값 공개 가능 (원본 복원 불가) DB
원본 비밀번호 저장 안 함 어디에도 없음

Salt는 왜 비밀이 아니야?

Salt의 목적은 "같은 비밀번호 → 다른 해시"를 만드는 거야. 공격자가 Salt를 알아도, 모든 Salt+비밀번호 조합을 다시 계산해야 하니까 Rainbow Table이 무력화되는 건 마찬가지야.

해싱은 암호화가 아니야¶

혼동 금지

해싱 암호화
방향 단방향 (원본 복원 불가) 양방향 (키로 복원 가능)
키 필요 없음 필요
목적 무결성 검증, 비밀번호 저장 데이터 기밀성 보호
비유 지문 채취 (지문으로 사람 복원 불가) 금고 (열쇠로 열 수 있음)

"SHA-256으로 암호화했어요" ← 이 문장은 틀렸어. "SHA-256으로 해싱했어요" ← 이게 맞는 표현이야.

주의사항 / 함정¶

삽질 포인트

1. MD5를 비밀번호 해싱에 쓰지 마

MD5는 너무 빠르고 충돌이 발견됐어. 보안 용도로는 죽은 알고리즘이야. 비밀번호 해싱에는 bcrypt, scrypt, Argon2 같은 전용 알고리즘을 써. 이것들은 의도적으로 느리게 만들어져서 무차별 대입 공격을 막아.

2. Salt 없이 해싱하지 마

Salt 없으면 Rainbow Table 한 방에 뚫려. 모든 비밀번호 해싱에는 사용자별 고유 Salt를 써야 해.

3. 해시 알고리즘 직접 구현하지 마

이미 검증된 라이브러리를 써. 직접 구현하면 100% 취약점 생겨. Java: MessageDigest, Spring Security의 PasswordEncoder 실무에서는 BCryptPasswordEncoder를 쓴다.

개념 핵심
해싱 입력을 고정 길이 값으로 변환. 단방향
5가지 특성 단방향, 결정적, 고정 길이, 눈사태 효과, 충돌 저항성

한 문장 정리

해싱은 되돌릴 수 없는 단방향 변환이야. 비밀번호 저장과 무결성 검증에 쓰이고, 22장의 HMAC-SHA256에서 핵심이 된다.

확인 문제¶

문제 1

해싱의 5가지 특성을 전부 말하고, 각각 한 문장으로 설명해봐.

정답 보기
  1. 단방향: 해시값에서 원본을 복원할 수 없다
  2. 결정적: 같은 입력은 항상 같은 해시값을 만든다
  3. 고정 길이: 입력 크기와 무관하게 출력 크기가 동일하다
  4. 눈사태 효과: 입력이 1비트만 바뀌어도 출력이 완전히 달라진다
  5. 충돌 저항성: 같은 해시를 만드는 서로 다른 입력을 찾기 어렵다

문제 2

Salt가 왜 필요한지 설명해봐. Salt가 없으면 어떤 공격이 가능해?

정답 보기

Salt가 없으면 Rainbow Table 공격이 가능해.

해싱은 결정적이라서 "password123"의 SHA-256 해시는 항상 같아. 공격자가 자주 쓰는 비밀번호 수백만 개의 해시를 미리 계산해놓은 표(Rainbow Table)를 가지고 있으면, DB에서 해시값을 보고 바로 원본 비밀번호를 찾을 수 있어.

Salt를 쓰면 같은 비밀번호라도 사용자마다 다른 해시값이 나오니까 Rainbow Table이 무력화된다.

문제 3

MD5와 SHA-256의 차이를 말하고, 왜 비밀번호 해싱에 MD5를 쓰면 안 되는지 설명해봐.

정답 보기
항목 MD5 SHA-256
출력 길이 128비트 256비트
충돌 발견됨 (2004년) 발견 안 됨
보안 보안 용도 금지 현재 표준

MD5를 비밀번호 해싱에 쓰면 안 되는 이유:

  1. 충돌이 발견됨: 서로 다른 입력이 같은 해시를 만들 수 있어. 신뢰 불가
  2. 너무 빠름: 초당 수십억 개의 해시를 계산할 수 있어서 무차별 대입 공격에 취약
  3. Rainbow Table이 이미 존재: MD5 해시에 대한 대규모 Rainbow Table이 공개되어 있음

문제 4

"해시값을 복호화했어"라는 말이 왜 틀린지 설명해봐.

정답 보기

"복호화(Decryption)"는 암호화의 반대야. 키를 사용해서 암호문을 원문으로 되돌리는 거지.

해싱은 암호화가 아니야. 단방향 변환이라 되돌리는 방법 자체가 없어. 복호화할 "키"도 없고, 복호화라는 개념 자체가 해싱에는 적용되지 않아.

올바른 표현: "해시값과 비교했어" (같은 입력을 해싱해서 결과가 같은지 비교) 틀린 표현: "해시값을 복호화했어" (불가능한 행위)
