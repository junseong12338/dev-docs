12. REST 아키텍처¶

한 줄 요약

REST는 기술이 아니라 아키텍처 스타일(제약조건 모음)이다. 6가지 제약을 지키면 RESTful, 안 지키면 그냥 HTTP API.

핵심 개념¶

REST가 뭐야¶

REST = Representational State Transfer

2000년, Roy Fielding이라는 사람이 박사 논문에서 제안한 웹 아키텍처 스타일이야.

Roy Fielding이 누구냐면

HTTP/1.0, HTTP/1.1 명세의 주요 저자. Apache 웹 서버의 공동 창시자. 즉, 웹의 아버지 중 한 명이야. 이 사람이 "웹을 잘 쓰려면 이렇게 해라" 한 게 REST.

단어를 뜯어보자:

단어 의미
Representational 표현 (리소스를 JSON, XML 등으로 표현)
State 상태 (리소스의 현재 상태)
Transfer 전송 (클라이언트와 서버 간 상태의 표현을 전송)

합치면: 리소스의 상태를 표현으로 전송하는 아키텍처 스타일.

이거 중요해

REST는 프로토콜이 아니야. 프레임워크도 아니야. 라이브러리도 아니야. "이렇게 설계하면 좋은 웹 시스템이 된다"는 제약조건 모음이야. 기술이 아니라 철학에 가깝다.

6가지 제약조건¶

REST를 정의하는 건 이 6가지 제약조건이야. 이걸 지키면 "RESTful하다"고 말할 수 있어.

mindmap
  root((REST 제약조건))
    1. Client-Server
    2. Stateless
    3. Cacheable
    4. Layered System
    5. Uniform Interface
      리소스 식별
      표현을 통한 조작
      자기 서술적 메시지
      HATEOAS
    6. Code on Demand

1. Client-Server (클라이언트-서버 분리)¶

클라이언트와 서버의 관심사를 분리한다.

클라이언트 서버
UI, 사용자 경험 데이터 저장, 비즈니스 로직
"어떻게 보여줄까" "어떻게 처리할까"

왜?

  • 독립적 진화: 서버 바꿔도 클라이언트 안 건드려도 됨. 반대도 마찬가지.
  • 확장성: 서버만 따로 스케일업 가능
  • 이식성: 같은 API로 웹, 앱, IoT 다 대응

실전에서

LMS 서버(Spring)를 리팩토링해도, API 명세만 안 바꾸면 프론트(JSP)는 영향 없어. 나중에 JSP를 React로 바꿔도 같은 API 그대로 쓸 수 있어.

2. Stateless (무상태)¶

서버가 클라이언트의 상태를 저장하지 않는다.

모든 요청에는 그 요청을 처리하는 데 필요한 모든 정보가 담겨 있어야 해.

sequenceDiagram
    participant C as 클라이언트
    participant S as 서버

    Note over C,S: Stateful (상태 유지) - 잘못된 예
    C->>S: "나 로그인할래" (1단계)
    Note right of S: 상태 저장: "로그인 진행중"
    C->>S: "아이디는 junseong" (2단계)
    Note right of S: 상태 저장: "아이디 받음"
    C->>S: "비번은 1234" (3단계)
    Note right of S: 이전 상태 참조해서 처리

    Note over C,S: Stateless (무상태) - 올바른 예
    C->>S: "로그인해줘. 아이디: junseong, 비번: 1234"
    Note right of S: 이 요청 하나에 모든 정보 있음
    S-->>C: "로그인 성공. 토큰: eyJhbG..."

왜?

  • 확장성: 서버가 상태 안 들고 있으니 아무 서버나 요청 처리 가능 (로드밸런싱 쉬움)
  • 신뢰성: 서버 하나 죽어도 다른 서버가 바로 대체
  • 단순성: 서버 로직이 간단해짐

그럼 로그인 상태는 어떻게?

서버가 상태를 안 가지니까, 클라이언트가 매 요청마다 인증 정보를 보내야 해. 이게 바로 토큰(JWT) 방식이 탄생한 이유야.

GET /api/users/me
Authorization: Bearer eyJhbGciOiJIUzI1NiJ9...

매 요청마다 토큰을 보내서 "나 인증된 사용자야"를 증명하는 거지.

3. Cacheable (캐시 가능)¶

응답에 캐시 가능 여부를 명시해야 한다.

HTTP/1.1 200 OK
Cache-Control: max-age=3600
ETag: "abc123"

{"id": 1, "name": "공지사항 목록", ...}

왜?

  • 성능: 같은 데이터 또 요청 안 해도 됨 → 네트워크 트래픽 감소
  • 응답 속도: 캐시에서 바로 반환 → 사용자 체감 속도 향상
  • 서버 부하: 불필요한 요청 줄어듦
캐시 적합 캐시 부적합
공지사항 목록 실시간 채팅 메시지
강의 정보 현재 접속자 수
사용자 프로필 이미지 결제 처리 결과
정적 리소스 (CSS, JS) 일회성 인증 코드

4. Layered System (계층화 시스템)¶

클라이언트는 직접 서버에 연결됐는지, 중간에 뭐가 있는지 모른다.

flowchart LR
    C[클라이언트] --> LB[로드밸런서]
    LB --> S1[서버 1]
    LB --> S2[서버 2]
    LB --> S3[서버 3]
    S1 --> DB[(DB)]
    S2 --> DB
    S3 --> DB

    C -.->|"클라이언트 입장에서는<br/>그냥 서버 하나"| LB

왜?

  • 보안: 프록시, 방화벽 같은 보안 계층 추가 가능
  • 확장성: 로드밸런서로 트래픽 분산
  • 유연성: CDN, 캐시 서버 등 중간 계층 자유롭게 추가/제거

실무에서

클라이언트 → Nginx(리버스 프록시) → Spring Boot(WAS) → DB 이 구조가 계층화 시스템의 전형적인 예시야. 클라이언트는 Nginx 뒤에 뭐가 있는지 전혀 모르고, 알 필요도 없어.

5. Uniform Interface (균일한 인터페이스)¶

REST의 핵심 차별점

다른 5개 제약조건은 다른 아키텍처에도 있어. Uniform Interface가 REST를 REST답게 만드는 핵심이야.

4가지 하위 제약으로 구성돼:

5-1. 리소스 식별 (Identification of Resources)¶

모든 리소스는 URI로 고유하게 식별된다.

/users/123        ← 123번 사용자라는 "리소스"
/courses/456      ← 456번 강좌라는 "리소스"
/lectures/789     ← 789번 강의라는 "리소스"

5-2. 표현을 통한 리소스 조작 (Manipulation through Representations)¶

리소스를 직접 주고받는 게 아니라, 리소스의 표현(representation)을 주고받아.

DB에 저장된 사용자 데이터 (리소스 그 자체)
    ↓ 표현으로 변환
JSON 형태 (리소스의 표현)
    ↓ 전송
클라이언트가 받음

같은 리소스라도 JSON으로 줄 수도 있고, XML로 줄 수도 있어. 표현이 다를 뿐 리소스는 같아.

5-3. 자기 서술적 메시지 (Self-descriptive Messages)¶

메시지만 보면 뭘 해야 하는지 다 알 수 있어야 해.

GET /users/123 HTTP/1.1
Host: api.example.com
Accept: application/json
Authorization: Bearer eyJhbG...

이 요청만 보면: "api.example.com의 123번 사용자를 JSON으로 달라, 인증 토큰은 이거" -- 다 알 수 있지?

5-4. HATEOAS (Hypermedia As The Engine Of Application State)¶

응답에 다음에 할 수 있는 행동(링크)이 포함돼야 해.

{
  "id": 123,
  "name": "이준성",
  "_links": {
    "self": { "href": "/users/123" },
    "courses": { "href": "/users/123/courses" },
    "update": { "href": "/users/123", "method": "PUT" },
    "delete": { "href": "/users/123", "method": "DELETE" }
  }
}

현실은...

HATEOAS를 제대로 구현하는 API는 거의 없어. 이론적으로는 중요하지만, 실무에서는 대부분 API 문서로 대체해. "이상적인 REST"와 "현실의 REST"는 차이가 있다는 거야.

6. Code on Demand (선택 사항)¶

서버가 클라이언트에 실행 가능한 코드를 전송할 수 있다.

예를 들어, 서버가 JavaScript 코드를 보내서 클라이언트(브라우저)가 실행하는 것.

이건 선택이야

6가지 제약 중 유일하게 선택 사항(optional)이야. 나머지 5개는 반드시 지켜야 RESTful이라고 할 수 있어.

"RESTful하다" = 이 6가지를 잘 지킨다¶

질문 RESTful 아닌 것
클라이언트-서버 분리? O X (서버에서 HTML 직접 렌더링)
무상태? O (매 요청에 인증 포함) X (서버에 세션 의존)
캐시 명시? O (Cache-Control 헤더) X (캐시 정보 없음)
계층화 가능? O (프록시 추가 가능) X (직접 연결만 가능)
균일한 인터페이스? O (URI + 메서드 + 자기서술) X (URL에 동사, 일관성 없음)

"REST API" vs "HTTP API"¶

불편한 진실

세상에 있는 대부분의 "REST API"는 진짜 RESTful하지 않아. 그냥 HTTP로 JSON 주고받는 API를 REST API라고 부르는 경우가 대부분이야.

Richardson Maturity Model¶

REST의 성숙도를 4단계로 나눈 모델이야:

레벨 이름 설명 현실
Level 0 The Swamp of POX HTTP를 RPC처럼 사용. 하나의 URL에 모든 요청 SOAP이 여기
Level 1 Resources 리소스별 URL 분리 (/users, /courses) 조금 나아짐
Level 2 HTTP Verbs HTTP 메서드 활용 (GET, POST, PUT, DELETE) 대부분 여기
Level 3 Hypermedia Controls HATEOAS 구현 거의 없음

현실적인 목표

Level 2를 잘 지키는 것이 현실적인 목표야. Level 3(HATEOAS)까지 완벽하게 구현하는 건 오버엔지니어링인 경우가 많아.

그래서 엄밀히 말하면 "REST API"보다 "HTTP API" 또는 "RESTish API"가 정확한 표현이야. 하지만 업계 관행상 Level 2만 지켜도 "REST API"라고 부르는 경우가 대부분이야.

왜 REST가 이겼나¶

2000년대 초중반, Web API의 대표 기술은 SOAP이었어.

항목 SOAP REST
프로토콜 독자 프로토콜 (XML 기반) HTTP 그대로 활용
메시지 형식 XML만 가능 JSON, XML 등 자유
복잡도 WSDL, 스키마, 엔벨로프... URL + 메서드 + JSON
학습 곡선 가파름 완만함
표준 엄격한 표준 가이드라인 수준
도구 의존성 전용 도구 필요 브라우저, curl로 가능

REST가 이긴 이유:

  1. 단순함: HTTP를 그대로 쓰니까 배우기 쉬워
  2. 가벼움: JSON은 XML보다 가볍고 읽기 쉬워
  3. 도구 독립: 별도 도구 없이 HTTP만 보내면 돼
  4. 웹 친화적: 웹의 기존 인프라(캐시, 프록시)를 그대로 활용

주의사항 / 함정¶

흔한 착각

착각 1: "JSON 쓰면 REST다"

아니야. JSON은 데이터 형식일 뿐이야. REST 제약을 지켜야 REST야. SOAP도 JSON 쓸 수 있고, REST도 XML 쓸 수 있어.

착각 2: "REST는 HTTP에서만 동작한다"

이론적으로는 다른 프로토콜에서도 가능해. 하지만 실질적으로 HTTP와 함께 쓰이는 게 99.9%야.

착각 3: "Stateless면 로그인 기능 못 만든다"

Stateless는 서버가 세션 상태를 안 가진다는 거야. 토큰(JWT)을 클라이언트가 매번 보내면 로그인 유지 가능해. 상태를 클라이언트가 들고 다니는 거지.

면접에서 자주 나오는 질문

"REST API의 특징을 설명하세요" → 6가지 제약조건을 말할 수 있어야 해. "RESTful하다는 게 뭔 뜻이에요?" → 6가지 제약을 지킨다는 뜻. "REST와 SOAP의 차이?" → 위 비교표 내용.

정리¶

제약조건 핵심 왜 필요해
Client-Server 관심사 분리 독립적 진화
Stateless 서버에 상태 없음 확장성, 신뢰성
Cacheable 캐시 가능 명시 성능 향상
Layered System 중간 계층 허용 보안, 확장성
Uniform Interface 일관된 인터페이스 REST의 핵심 차별점
Code on Demand 코드 전송 (선택) 클라이언트 확장성

기억법

C-S-C-L-U-C (씨-에스-씨-엘-유-씨) Client-Server / Stateless / Cacheable / Layered / Uniform / Code on Demand

확인 문제¶

문제 1

REST는 프로토콜인가, 아키텍처 스타일인가? 차이가 뭔지 설명해봐.

정답 보기

REST는 아키텍처 스타일이야.

  • 프로토콜: 엄격한 규칙. 지키지 않으면 통신 불가. (예: HTTP, TCP)
  • 아키텍처 스타일: 설계 가이드라인/제약조건 모음. 안 지켜도 동작은 해. 다만 좋은 설계가 아닐 뿐.

REST는 "이렇게 하면 좋은 웹 시스템이 된다"는 제약조건 모음이지, 안 지키면 통신이 안 되는 건 아니야.

문제 2

Stateless 제약의 의미와, 이게 로그인 기능과 어떻게 양립하는지 설명해봐.

정답 보기

Stateless: 서버가 클라이언트의 상태(세션)를 저장하지 않는다. 모든 요청에 처리에 필요한 모든 정보가 포함되어야 한다.

로그인과의 양립: 서버가 세션을 저장하는 대신, 클라이언트가 인증 토큰(JWT 등)을 매 요청마다 헤더에 담아 보낸다. 서버는 토큰을 검증만 하고, 별도의 세션 상태를 가지지 않아. 상태를 서버가 아닌 클라이언트가 관리하는 것.

문제 3

Uniform Interface의 4가지 하위 제약을 나열하고, 현실에서 가장 잘 안 지켜지는 건 뭔지 말해봐.

정답 보기
  1. 리소스 식별 (URI로 리소스 고유 식별)
  2. 표현을 통한 리소스 조작 (리소스의 표현(JSON 등)을 주고받음)
  3. 자기 서술적 메시지 (메시지만 보면 뭘 해야 하는지 알 수 있음)
  4. HATEOAS (응답에 다음 행동 링크 포함)

가장 안 지켜지는 건 HATEOAS. 대부분의 API가 응답에 하이퍼미디어 링크를 포함하지 않아. API 문서로 대체하는 게 현실이야.

문제 4

Richardson Maturity Model에서 Level 2와 Level 3의 차이가 뭐야? 대부분의 "REST API"는 어느 레벨이야?

정답 보기
  • Level 2: HTTP 메서드를 올바르게 활용 (GET=조회, POST=생성, PUT=수정, DELETE=삭제)
  • Level 3: Level 2 + HATEOAS 구현 (응답에 하이퍼미디어 링크 포함)

대부분의 "REST API"는 Level 2에 해당해. Level 3까지 구현하는 경우는 극히 드물어. 그래서 엄밀히 말하면 "RESTful API"가 아니라 "HTTP API"에 가까워.
