14. REST API 요청/응답 실전¶

한 줄 요약

REST API의 실제 요청/응답 형태를 전부 보여준다. 생성, 조회, 수정, 삭제, 페이징, 에러 처리까지 -- 실전에서 쓰는 그대로.

핵심 개념¶

이전 챕터에서 URL 설계 원칙을 배웠어. 이제 실제로 요청하면 뭐가 오는지 코드로 보자.

이 챕터의 목표

"이 API에 요청하면 뭐가 와?" 라는 질문에 바로 대답할 수 있게 되는 것. API 문서를 읽을 줄 알고, 직접 요청/응답을 설계할 수 있어야 해.

생성 (POST)¶

요청¶

POST /api/v1/users HTTP/1.1
Host: api.example.com
Content-Type: application/json
Authorization: Bearer eyJhbGciOi...

{
  "name": "홍길동",
  "email": "junseong@example.com",
  "role": "STUDENT"
}

각 줄이 뭔지 하나씩 보자:

줄 의미
POST /api/v1/users "users 리소스를 새로 만들어줘"
Content-Type: application/json "내가 보내는 건 JSON이야"
Authorization: Bearer ... "내 인증 토큰이야"
Body (JSON) 생성할 데이터

응답 (성공)¶

HTTP/1.1 201 Created
Location: /api/v1/users/123
Content-Type: application/json

{
  "id": 123,
  "name": "홍길동",
  "email": "junseong@example.com",
  "role": "STUDENT",
  "createdAt": "2026-03-18T10:00:00"
}
포인트 설명
201 Created 200이 아니라 201이야. "만들어졌다"는 뜻
Location 헤더 새로 만든 리소스의 URL을 알려줌
응답 Body 서버가 생성한 id, createdAt 등 포함

200 vs 201

생성 성공은 201 Created가 맞아. 200 OK로 응답하는 API도 많지만, HTTP 표준에 따르면 201이 정확해. "리소스가 새로 만들어졌다"는 의미를 명확히 전달하니까.

응답 (실패 - 유효성 검증)¶

HTTP/1.1 400 Bad Request
Content-Type: application/json

{
  "status": 400,
  "error": "Bad Request",
  "message": "유효성 검증 실패",
  "timestamp": "2026-03-18T10:00:00",
  "path": "/api/v1/users",
  "errors": [
    {
      "field": "email",
      "value": "not-an-email",
      "reason": "이메일 형식이 올바르지 않습니다"
    },
    {
      "field": "name",
      "value": "",
      "reason": "이름은 필수입니다"
    }
  ]
}

조회 (GET)¶

단건 조회¶

요청:

GET /api/v1/users/123 HTTP/1.1
Host: api.example.com
Accept: application/json
Authorization: Bearer eyJhbGciOi...

GET 요청의 특징

GET에는 Body가 없어. 필요한 정보는 URL 경로나 쿼리 스트링에 담아.

응답 (성공):

HTTP/1.1 200 OK
Content-Type: application/json

{
  "id": 123,
  "name": "홍길동",
  "email": "junseong@example.com",
  "role": "STUDENT",
  "createdAt": "2026-03-18T10:00:00",
  "updatedAt": "2026-03-18T10:00:00"
}

응답 (없는 리소스):

HTTP/1.1 404 Not Found
Content-Type: application/json

{
  "status": 404,
  "error": "Not Found",
  "message": "사용자를 찾을 수 없습니다 (id: 999)",
  "timestamp": "2026-03-18T10:00:00",
  "path": "/api/v1/users/999"
}

목록 조회 + 페이징¶

요청:

GET /api/v1/users?page=1&size=10&sort=createdAt,desc&role=STUDENT HTTP/1.1
Host: api.example.com
Accept: application/json
Authorization: Bearer eyJhbGciOi...
파라미터 의미
page=1 1번째 페이지 (보통 0부터 시작하는 곳도 있고 1부터도 있어)
size=10 한 페이지에 10개
sort=createdAt,desc 생성일 내림차순
role=STUDENT 학생만 필터

응답:

HTTP/1.1 200 OK
Content-Type: application/json

{
  "content": [
    {
      "id": 130,
      "name": "김철수",
      "email": "chulsoo@example.com",
      "role": "STUDENT",
      "createdAt": "2026-03-17T15:00:00"
    },
    {
      "id": 129,
      "name": "박영희",
      "email": "younghee@example.com",
      "role": "STUDENT",
      "createdAt": "2026-03-17T14:00:00"
    }
  ],
  "page": 1,
  "size": 10,
  "totalElements": 53,
  "totalPages": 6,
  "first": true,
  "last": false
}
필드 의미
content 실제 데이터 배열
page 현재 페이지 번호
size 한 페이지 크기
totalElements 전체 데이터 수
totalPages 전체 페이지 수
first / last 첫/마지막 페이지 여부

Spring Data의 Page 객체

Spring Data JPA를 쓰면 Page<User> 객체가 이 형태를 자동으로 만들어줘. 직접 페이징 메타데이터 만들 필요 없어.

수정 (PUT vs PATCH)¶

PUT - 전체 교체¶

요청:

PUT /api/v1/users/123 HTTP/1.1
Host: api.example.com
Content-Type: application/json
Authorization: Bearer eyJhbGciOi...

{
  "name": "홍길동",
  "email": "new-email@example.com",
  "role": "STUDENT",
  "phone": "010-1234-5678"
}

응답:

HTTP/1.1 200 OK
Content-Type: application/json

{
  "id": 123,
  "name": "홍길동",
  "email": "new-email@example.com",
  "role": "STUDENT",
  "phone": "010-1234-5678",
  "updatedAt": "2026-03-18T11:00:00"
}

PUT의 함정

위 요청에서 address 필드를 빠뜨렸어. 원래 "서울시 강남구"였는데. PUT은 전체 교체니까 address가 null이 된다.

이게 PUT이 위험한 이유야. 필드 하나 빠뜨리면 데이터 날아가.

PATCH - 부분 수정¶

요청:

PATCH /api/v1/users/123 HTTP/1.1
Host: api.example.com
Content-Type: application/json
Authorization: Bearer eyJhbGciOi...

{
  "email": "new-email@example.com"
}

email만 보냈으니 email만 바뀜. 나머지 필드는 안 건드려.

응답:

HTTP/1.1 200 OK
Content-Type: application/json

{
  "id": 123,
  "name": "홍길동",
  "email": "new-email@example.com",
  "role": "STUDENT",
  "phone": "010-1234-5678",
  "address": "서울시 강남구",
  "updatedAt": "2026-03-18T11:00:00"
}

address가 살아있지? PATCH는 보낸 것만 바꾸니까.

삭제 (DELETE)¶

요청¶

DELETE /api/v1/users/123 HTTP/1.1
Host: api.example.com
Authorization: Bearer eyJhbGciOi...

DELETE 요청에는 보통 Body가 없어. URL의 ID만으로 충분하니까.

응답: 200 vs 204¶

두 가지 선택지가 있어:

방식 1: 200 OK + 삭제된 데이터 반환

HTTP/1.1 200 OK
Content-Type: application/json

{
  "id": 123,
  "name": "홍길동",
  "deletedAt": "2026-03-18T12:00:00"
}

방식 2: 204 No Content (Body 없음)

HTTP/1.1 204 No Content
방식 장점 단점
200 + 데이터 삭제된 데이터 확인 가능 Body가 있어서 약간 더 무거움
204 No Content 깔끔, 가벼움 뭐가 삭제됐는지 확인 불가

실무에서

어떤 걸 선택할지는 팀 컨벤션에 달려있어. 중요한 건 프로젝트 전체에서 일관성 있게 쓰는 거야. 어떤 API는 200, 어떤 건 204 -- 이러면 안 돼.

소프트 삭제 vs 하드 삭제¶

방식 설명 DB 처리
소프트 삭제 삭제 표시만 (실제 데이터 유지) UPDATE SET deleted=true
하드 삭제 진짜 삭제 DELETE FROM table

실무에서는 소프트 삭제가 더 흔해. 데이터 복구가 가능하니까. API 응답은 동일하게 204나 200을 반환하되, 내부적으로는 deleted 플래그만 바꾸는 거야.

에러 응답 설계¶

에러 응답이 중요한 이유

클라이언트가 에러를 받았을 때 뭐가 잘못됐는지 즉시 파악할 수 있어야 해. "400 Bad Request"만 덜렁 오면? 뭐가 잘못인지 모르니까 디버깅 지옥이야.

일관된 에러 형태¶

모든 에러 응답은 같은 구조여야 해:

{
  "status": 400,
  "error": "Bad Request",
  "message": "이메일 형식이 올바르지 않습니다",
  "timestamp": "2026-03-18T10:00:00",
  "path": "/api/v1/users"
}
필드 설명 왜 필요해
status HTTP 상태 코드 프로그래밍적 분기 처리
error 에러 유형 사람이 읽는 분류
message 구체적 에러 메시지 뭐가 잘못됐는지 바로 알 수 있음
timestamp 발생 시각 로그 추적용
path 요청 경로 어떤 API에서 터졌는지

상황별 에러 응답¶

400 Bad Request - 잘못된 요청

{
  "status": 400,
  "error": "Bad Request",
  "message": "유효성 검증 실패",
  "timestamp": "2026-03-18T10:00:00",
  "path": "/api/v1/users",
  "errors": [
    { "field": "email", "value": "abc", "reason": "이메일 형식이 올바르지 않습니다" },
    { "field": "name", "value": null, "reason": "이름은 필수 항목입니다" }
  ]
}

401 Unauthorized - 인증 실패

{
  "status": 401,
  "error": "Unauthorized",
  "message": "인증 토큰이 만료되었습니다",
  "timestamp": "2026-03-18T10:00:00",
  "path": "/api/v1/users"
}

403 Forbidden - 권한 없음

{
  "status": 403,
  "error": "Forbidden",
  "message": "관리자만 접근 가능합니다",
  "timestamp": "2026-03-18T10:00:00",
  "path": "/api/v1/admin/users"
}

404 Not Found - 리소스 없음

{
  "status": 404,
  "error": "Not Found",
  "message": "강좌를 찾을 수 없습니다 (id: 999)",
  "timestamp": "2026-03-18T10:00:00",
  "path": "/api/v1/courses/999"
}

409 Conflict - 충돌

{
  "status": 409,
  "error": "Conflict",
  "message": "이미 등록된 이메일입니다",
  "timestamp": "2026-03-18T10:00:00",
  "path": "/api/v1/users"
}

500 Internal Server Error - 서버 오류

{
  "status": 500,
  "error": "Internal Server Error",
  "message": "서버 내부 오류가 발생했습니다",
  "timestamp": "2026-03-18T10:00:00",
  "path": "/api/v1/users"
}

500 에러의 message

500 에러에서는 구체적인 에러 내용을 클라이언트에 노출하면 안 돼. 스택 트레이스나 DB 에러 메시지가 나가면 보안 취약점이야. 일반적인 메시지만 주고, 상세 내용은 서버 로그에만 남겨.

Spring의 @RestControllerAdvice¶

Spring에서는 @RestControllerAdvice로 에러 처리를 통일해:

@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(UserNotFoundException.class)
    public ResponseEntity<ErrorResponse> handleUserNotFound(
            UserNotFoundException e, HttpServletRequest request) {
        ErrorResponse error = ErrorResponse.builder()
            .status(404)
            .error("Not Found")
            .message(e.getMessage())
            .timestamp(LocalDateTime.now())
            .path(request.getRequestURI())
            .build();
        return ResponseEntity.status(404).body(error);
    }

    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ResponseEntity<ErrorResponse> handleValidation(
            MethodArgumentNotValidException e, HttpServletRequest request) {
        // 유효성 검증 에러 처리
        // ...
    }
}

에러 처리 한 곳에 모아

각 컨트롤러에서 try-catch 하는 게 아니라, @RestControllerAdvice 하나에서 전역적으로 에러를 잡아. 일관성도 지키고, 중복 코드도 없어지고, 유지보수도 쉬워.

벌크 작업 (Bulk Operations)¶

여러 건을 한번에 처리해야 할 때:

벌크 생성¶

POST /api/v1/users/bulk HTTP/1.1
Content-Type: application/json

[
  { "name": "김철수", "email": "chulsoo@example.com" },
  { "name": "박영희", "email": "younghee@example.com" },
  { "name": "최민수", "email": "minsoo@example.com" }
]

HTTP/1.1 201 Created
Content-Type: application/json

{
  "created": 3,
  "results": [
    { "id": 124, "name": "김철수", "status": "SUCCESS" },
    { "id": 125, "name": "박영희", "status": "SUCCESS" },
    { "id": 126, "name": "최민수", "status": "SUCCESS" }
  ]
}

벌크 삭제¶

DELETE /api/v1/users/bulk HTTP/1.1
Content-Type: application/json

{
  "ids": [124, 125, 126]
}

벌크 작업 설계 시 고려사항

  • 부분 실패: 3건 중 2건 성공, 1건 실패면? 전체 롤백? 부분 성공?
  • 응답 형태: 각 건의 성공/실패 상태를 개별적으로 알려줘야 해
  • 크기 제한: 한번에 1000건 요청하면? 상한선을 정해야 해

응답 봉투 패턴 (Envelope Pattern)¶

API 응답을 어떻게 감쌀지 두 가지 방식이 있어:

방식 1: 봉투 패턴 (Envelope)¶

{
  "success": true,
  "data": {
    "id": 123,
    "name": "홍길동"
  },
  "message": null,
  "timestamp": "2026-03-18T10:00:00"
}

에러 시:

{
  "success": false,
  "data": null,
  "message": "사용자를 찾을 수 없습니다",
  "timestamp": "2026-03-18T10:00:00"
}

방식 2: 직접 반환 (No Envelope)¶

성공:

{
  "id": 123,
  "name": "홍길동"
}

에러:

{
  "status": 404,
  "error": "Not Found",
  "message": "사용자를 찾을 수 없습니다"
}

비교¶

항목 봉투 패턴 직접 반환
일관성 항상 같은 구조 성공/에러 구조 다름
응답 크기 약간 더 큼 더 가벼움
HTTP 활용 상태코드 외에 success 필드 중복 HTTP 상태코드 적극 활용
클라이언트 처리 response.data로 항상 접근 상태코드로 분기
RESTful 여부 덜 RESTful (HTTP 상태코드 무시 경향) 더 RESTful

뭘 써야 해?

정답은 없어. 팀에서 결정하고 일관성 있게 쓰면 돼.

다만 RESTful 관점에서는 직접 반환이 맞아. HTTP 상태 코드(200, 201, 400, 404...)가 이미 성공/실패를 알려주니까, success: true/false는 중복이야.

반면 봉투 패턴은 프론트엔드 처리가 편해. 항상 같은 구조니까 파싱 코드가 단순해져.

주의사항 / 함정¶

실전 함정들

함정 1: GET 요청에 Body 넣기

HTTP 스펙상 GET 요청에도 Body를 넣을 수는 있어. 하지만 하지 마라. 많은 서버, 프록시, CDN이 GET의 Body를 무시하거나 잘라내. 호환성 문제 생겨.

함정 2: 에러인데 200 반환

HTTP/1.1 200 OK

{ "success": false, "message": "사용자를 찾을 수 없습니다" }

이러면 HTTP 클라이언트가 "성공"으로 인식해. 에러 처리 로직을 못 타. 반드시 적절한 4xx/5xx 코드를 써.

함정 3: 서버 에러 상세 노출

{
  "message": "com.mysql.jdbc.exceptions.MySQLSyntaxErrorException: Table 'users' doesn't exist"
}

DB 구조, 테이블명, 쿼리 에러가 그대로 나가면 보안 사고야. 클라이언트에는 일반적인 메시지만 주고, 상세 내용은 서버 로그에만.

가장 흔한 실수

에러 응답 형태가 API마다 다른 것. 어떤 API는 message, 어떤 건 msg, 어떤 건 error_message... @RestControllerAdvice로 전역 에러 핸들링해서 반드시 통일해라.

정리¶

동작 메서드 성공 코드 요청 Body 응답 Body
생성 POST 201 Created O (생성 데이터) O (생성된 리소스)
단건 조회 GET 200 OK X O (리소스)
목록 조회 GET 200 OK X O (리소스 배열 + 페이징)
전체 수정 PUT 200 OK O (전체 데이터) O (수정된 리소스)
부분 수정 PATCH 200 OK O (변경 필드만) O (수정된 리소스)
삭제 DELETE 200 or 204 X (보통) 선택

확인 문제¶

문제 1

POST 요청이 성공했을 때 응답 코드로 200과 201 중 어떤 게 맞고, 왜 그런지 설명해봐. Location 헤더는 왜 넣어야 해?

정답 보기

201 Created가 맞아.

  • 200 OK는 "요청이 성공적으로 처리됐다"는 일반적 의미
  • 201 Created는 "새 리소스가 성공적으로 만들어졌다"는 구체적 의미
  • HTTP 상태 코드를 정확하게 사용하는 게 RESTful 설계의 기본이야

Location 헤더: 새로 만들어진 리소스에 접근할 수 있는 URL을 알려줘. 클라이언트가 "방금 만든 리소스를 바로 조회하고 싶으면?" 하면 Location 헤더의 URL로 GET 요청하면 돼.

문제 2

PUT으로 사용자 정보를 수정할 때 필드 하나를 빠뜨리면 어떻게 되는지, 그리고 PATCH로 했으면 어떻게 되는지 비교 설명해봐.

정답 보기

PUT (전체 교체): 빠뜨린 필드가 null로 초기화된다. PUT은 리소스 전체를 보낸 데이터로 교체하는 거니까, 보내지 않은 건 "없애겠다"는 의미.

PATCH (부분 수정): 빠뜨린 필드는 기존 값 유지. PATCH는 보낸 필드만 수정하고 나머지는 건드리지 않아.

실무에서는 PATCH가 더 안전해. PUT은 실수로 필드 빠뜨리면 데이터가 날아가니까.

문제 3

에러 응답에서 500 Internal Server Error의 message에 "com.mysql.jdbc.exceptions.MySQLSyntaxErrorException" 같은 내용을 넣으면 왜 안 되는지 설명해봐.

정답 보기

보안 취약점이야.

  1. DB 종류(MySQL)가 노출됨 → 특정 DB 취약점 공격 가능
  2. 테이블명, 컬럼명이 노출될 수 있음 → SQL 인젝션 공격에 활용
  3. 서버 내부 구조(패키지명, 클래스명) 노출 → 공격 표면 확대

클라이언트에는 "서버 내부 오류가 발생했습니다" 같은 일반적인 메시지만 주고, 구체적인 에러 내용은 서버 로그에만 기록해야 해.

문제 4

봉투 패턴(Envelope)과 직접 반환 방식의 장단점을 설명하고, RESTful 관점에서 어느 쪽이 더 적절한지 말해봐.

정답 보기

봉투 패턴: { "success": true, "data": {...} }

  • 장점: 응답 구조가 항상 동일해서 클라이언트 파싱이 단순
  • 단점: HTTP 상태 코드와 success 필드가 중복, 응답 크기 증가

직접 반환: 성공 시 데이터만, 에러 시 에러 객체만

  • 장점: HTTP 상태 코드를 적극 활용, 응답이 가벼움
  • 단점: 성공/에러의 응답 구조가 다름

RESTful 관점에서는 직접 반환이 더 적절해. HTTP 상태 코드(200, 201, 400, 404 등)가 이미 성공/실패를 구분하는 역할을 하니까, success: true/false는 의미 중복이야. HTTP의 기능을 활용하는 게 REST 철학에 맞아.
