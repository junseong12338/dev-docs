05. URL 해부¶

이 장의 목표: URL을 보고 각 파트가 뭔지 한 치의 망설임 없이 분해할 수 있다

1. URL이 뭐야¶

1.1 정의¶

URL = Uniform Resource Locator

풀어서 말하면:
  Uniform  → 통일된 형식
  Resource → 리소스 (웹 페이지, 이미지, API 데이터 등)
  Locator  → 위치를 알려주는 것

= 인터넷에서 리소스의 위치를 가리키는 주소

1.2 비유¶

URL = 택배 보내는 주소

  "서울특별시 강남구 테헤란로 123, 4층 401호"
  → 이 주소로 택배 보내면 원하는 곳에 도착한다

  "https://api.example.com:8080/users/123?page=1"
  → 이 URL로 요청 보내면 원하는 데이터가 온다

둘 다 "어디"에 있는지를 알려주는 것이다.

2. URL vs URI vs URN¶

면접에서 자주 나오고, 헷갈리기 쉬운 것. 표로 깔끔하게 정리하자.

2.1 개념 비교¶

구분 풀네임 의미 예시
URI Uniform Resource Identifier 리소스를 식별하는 문자열 (가장 큰 개념) https://example.com/users/123
URL Uniform Resource Locator 리소스의 위치를 알려주는 URI https://example.com/users/123
URN Uniform Resource Name 리소스의 이름으로 식별하는 URI urn:isbn:0451450523

2.2 관계¶

URI (Uniform Resource Identifier)
 ├── URL (위치로 식별) ← 우리가 주로 쓰는 것
 └── URN (이름으로 식별) ← 거의 안 씀

URI가 가장 큰 개념이다.
URL과 URN은 URI의 하위 개념이다.

실무에서는?
  → URL이라고 하면 거의 다 통한다
  → URI라고 해도 틀리지 않는다
  → URN은 만날 일이 거의 없다

실무에서의 사용

RFC 3986 스펙상 엄밀히 따지면 URI가 정확한 용어다. 하지만 실무에서 "URL 뭐야?"하면 다 알아듣고, "URI 뭐야?"하면 좀 더 정확한 표현이다.

이 문서에서는 일반적으로 통용되는 URL을 기준으로 설명한다.

3. URL 완전 분해¶

이제 진짜 중요한 거다. URL을 파트별로 쪼개보자.

3.1 전체 구조¶

https://api.example.com:8080/users/123?page=1&size=10#section1
│       │                │    │         │              │
scheme  host             port path      query string   fragment
(프로토콜) (서버 주소)    (포트) (경로)    (추가 정보)    (문서 내 위치)

3.2 scheme (프로토콜)¶

https://api.example.com:8080/users/123
^^^^^^
이 부분

scheme = 어떤 프로토콜로 통신할 건지

일반적인 값:
  http   → 암호화 없이 통신
  https  → TLS 암호화 적용 (17장에서 다룸)
  ftp    → 파일 전송 프로토콜
  mailto → 이메일

우리가 신경 쓸 것:
  http 또는 https
  요즘은 https가 사실상 필수다.

http vs https

  • http:// → 데이터가 평문으로 오간다. 누구나 볼 수 있다.
  • https:// → 데이터가 암호화되어 오간다. 안전하다.

운영 서비스에서 http를 쓰고 있으면? 그건 보안 사고를 기다리는 거다.

3.3 host (서버 주소)¶

https://api.example.com:8080/users/123
        ^^^^^^^^^^^^^^^
        이 부분

host = 서버가 어디 있는지

형태:
  도메인: api.example.com
  IP 주소: 192.168.1.100
  localhost: 127.0.0.1 (내 컴퓨터)

서브도메인 구조:
  api.example.com
  │   │       │
  │   │       └── TLD (Top Level Domain): .com
  │   └── 도메인: example
  └── 서브도메인: api

실전 예시:
  www.google.com    → www = 서브도메인
  mail.google.com   → mail = 서브도메인
  api.example.com   → api = 서브도메인
  localhost:8080    → 개발할 때 내 컴퓨터

03장(DNS)에서 배웠듯이, 도메인은 DNS를 통해 IP 주소로 변환된다.

3.4 port (포트)¶

https://api.example.com:8080/users/123
                        ^^^^
                        이 부분

port = 서버에서 어떤 프로그램(서비스)에 연결할 건지

규칙:
  - 범위: 0 ~ 65535
  - 생략 가능 (프로토콜별 기본 포트 사용)

기본 포트 (생략 시):
  http  → 80
  https → 443

그래서:
  https://example.com       = https://example.com:443
  http://example.com        = http://example.com:80
  http://localhost:8080     = 포트 8080 (개발 서버에서 자주 봄)

포트가 보이면

http://localhost:8080 → 개발 환경

https://example.com (포트 없음) → 운영 환경 (443이 기본)

URL에 포트 번호가 보이면 대부분 개발/테스트 환경이다. 운영 환경에서는 기본 포트(80, 443)를 쓰니까 생략한다.

3.5 path (경로)¶

https://api.example.com:8080/users/123
                             ^^^^^^^^^^
                             이 부분

path = 서버에서 어떤 리소스를 원하는지

구조:
  /users/123
  │     │
  │     └── 특정 사용자 (ID가 123인 사용자)
  └── 사용자 리소스

실전 예시:
  /                     → 루트 (메인 페이지)
  /users                → 사용자 목록
  /users/123            → 123번 사용자
  /users/123/orders     → 123번 사용자의 주문 목록
  /api/v1/products      → API v1의 상품 목록

경로 규칙:
  - 슬래시(/)로 계층을 구분한다
  - 소문자를 사용한다 (대소문자 구분됨)
  - 명사를 쓴다 (동사 아님!)
  - 복수형을 쓴다 (user가 아니라 users)

path는 대소문자를 구분한다

/Users/123과 /users/123은 서버에서 다른 경로다.

URL 경로는 소문자로 통일하는 게 관례다. 이거 무시하면 "왜 404 뜨지?" 하고 삽질한다.

3.6 query string (쿼리 스트링)¶

https://api.example.com:8080/users/123?page=1&size=10
                                       ^^^^^^^^^^^^^^^
                                       이 부분

query string = 추가 정보를 key=value 형태로 전달

구조:
  ? → 쿼리 스트링 시작
  key=value → 하나의 파라미터
  & → 파라미터 구분자

분해:
  ?page=1&size=10
   │       │
   │       └── size = 10 (한 페이지에 10개)
   └── page = 1 (1번째 페이지)

실전 예시:

  구글 검색:
  https://www.google.com/search?q=HTTP&hl=ko
  → q = HTTP (검색어)
  → hl = ko (언어: 한국어)

  게시판 목록:
  /board/list?page=2&category=notice&sort=date
  → page = 2
  → category = notice
  → sort = date

  API 필터링:
  /api/users?status=active&role=admin
  → status = active
  → role = admin

query string 특징

  • ?로 시작한다
  • key=value 쌍이다
  • 여러 개면 &로 연결한다
  • 순서 상관없다: ?a=1&b=2와 ?b=2&a=1은 같다
  • 서버에 전달된다 (fragment와 다름!)

3.7 fragment (프래그먼트)¶

https://api.example.com:8080/users/123?page=1&size=10#section1
                                                      ^^^^^^^^^
                                                      이 부분

fragment = 문서 내에서 특정 위치로 이동

규칙:
  - # 으로 시작한다
  - 서버에 전달되지 않는다!!! (클라이언트 전용)
  - 브라우저가 해당 위치로 스크롤한다

fragment는 서버에 안 간다

https://example.com/page#section1

서버가 받는 요청: GET /page #section1 부분은 서버에 전달되지 않는다. 브라우저가 응답 받은 후 로컬에서 section1 위치로 스크롤하는 것이다.

이거 모르면 "서버에서 fragment 값을 읽으려고" 삽질한다.

4. URL 인코딩¶

4.1 왜 필요해¶

URL에 쓸 수 있는 문자가 정해져 있다.

쓸 수 있는 문자 (안전한 문자):
  영문: A-Z, a-z
  숫자: 0-9
  특수: - _ . ~

쓸 수 없는 문자 (인코딩 필요):
  한글: 홍길동 → %EC%9D%B4%EC%A4%80%EC%84%B1
  공백: hello world → hello%20world
  특수문자: & = ? # 등 (URL 구조에 쓰이는 문자들)

4.2 인코딩 방식¶

URL 인코딩 = 퍼센트 인코딩 (Percent Encoding)

규칙:
  안전하지 않은 문자를 %XX 형태로 바꾼다
  XX = 해당 문자의 UTF-8 바이트를 16진수로 표현

예시:
  공백 → %20
  한글 "가" → %EA%B0%80
  & → %26
  = → %3D
  ? → %3F

실전에서

브라우저가 자동으로 인코딩해주니까 우리가 직접 할 일은 거의 없다.

하지만 API를 직접 호출하거나 URL을 프로그래밍으로 만들 때는 인코딩을 직접 해줘야 한다.

Java: URLEncoder.encode("홍길동", "UTF-8") JavaScript: encodeURIComponent("홍길동")

인코딩에 대한 자세한 내용은 19장(인코딩)에서 다룬다.

5. URL 설계 - 잘못된 예 vs 올바른 예¶

REST API에서 URL 설계가 왜 중요한지는 12~13장에서 자세히 다루지만, 기본 원칙은 여기서 잡아두자.

5.1 잘못된 URL 설계¶

❌ 동사를 쓴다
  /getUser?id=123
  /createUser
  /deleteUser?id=123
  /updateUserInfo

❌ 대소문자 섞는다
  /getUserList
  /User/Profile

❌ 의미 없는 경로
  /page1
  /data
  /info

❌ 확장자 붙인다
  /users/123.json
  /api/list.do

5.2 올바른 URL 설계¶

✅ 명사를 쓴다 (동사는 HTTP 메서드가 담당)
  GET    /users/123      → 123번 사용자 조회
  POST   /users          → 사용자 생성
  DELETE /users/123      → 123번 사용자 삭제
  PUT    /users/123      → 123번 사용자 수정

✅ 소문자 + 하이픈
  /user-profiles
  /order-items

✅ 계층 관계를 표현
  /users/123/orders       → 123번 사용자의 주문들
  /users/123/orders/456   → 123번 사용자의 456번 주문

✅ 복수형
  /users (O)
  /user  (X)

5.3 비교 정리¶

잘못된 설계 올바른 설계 이유
GET /getUser?id=123 GET /users/123 동사는 메서드가, 경로는 명사로
POST /createUser POST /users POST 자체가 "생성"의 의미
GET /getUserList GET /users 카멜케이스 대신 소문자
/user/profile /users/123/profile 복수형 + 리소스 ID 명시
/api/list.do /api/items 확장자 불필요

6. 실전에서 자주 보는 URL 패턴¶

6.1 웹 페이지¶

일반 웹사이트:
  https://www.example.com/                    → 메인 페이지
  https://www.example.com/about               → 소개 페이지
  https://www.example.com/products            → 상품 목록
  https://www.example.com/products/123        → 123번 상품 상세

게시판:
  /board/list?page=2&category=notice          → 게시판 2페이지
  /board/view?boardNo=456                     → 456번 게시글 상세

6.2 REST API¶

RESTful API:
  GET    /api/v1/users                        → 사용자 목록
  GET    /api/v1/users/123                    → 123번 사용자
  POST   /api/v1/users                        → 사용자 생성
  PUT    /api/v1/users/123                    → 123번 사용자 수정
  DELETE /api/v1/users/123                    → 123번 사용자 삭제

페이징 + 필터:
  GET /api/v1/users?page=1&size=20&sort=name  → 이름순 정렬, 20개씩
  GET /api/v1/users?status=active&role=admin  → 필터링

6.3 우리 프로젝트 스타일¶

우리 프로젝트 (eGov 기반):
  /lms/main.do                                → 메인 페이지
  /lms/board/boardList.do?bbsId=NOTICE        → 게시판 목록
  /lms/user/userView.do?userNo=123            → 사용자 상세

→ .do 확장자, 카멜케이스, 쿼리스트링 활용
→ RESTful 스타일은 아니지만, 레거시 프로젝트에서 흔한 패턴
→ 이게 왜 "올바르지 않은" 설계인지는 12~13장에서 다룬다

7. 핵심 정리¶

핵심 정리

URL = 인터넷에서 리소스의 위치를 나타내는 주소

URL 구조: 

scheme://host:port/path?query#fragment

파트 역할 서버 전달
scheme 프로토콜 (http, https) O
host 서버 주소 O
port 서비스 포트 (기본: 80/443) O
path 리소스 경로 O
query 추가 파라미터 (key=value) O
fragment 문서 내 위치 X (클라이언트 전용)

URI > URL > URN: URI가 가장 큰 개념

URL 인코딩: 안전하지 않은 문자를 %XX로 변환

URL 설계 원칙: 소문자, 명사, 복수형, 계층적 구조

다음 장: HTTP 메서드와 멱등성 → URL이 "어디"라면, 메서드는 "뭘 해달라고" 하는 건지.

8. 확인 문제¶

문제 1¶

다음 URL을 파트별로 분해해봐.

https://store.example.com:3000/api/products/42?color=red&size=L#reviews
정답 보기
파트 값
scheme https
host store.example.com
port 3000
path /api/products/42
query string color=red&size=L
fragment reviews

문제 2¶

fragment(#section1)는 서버에 전달될까, 안 될까?
이유도 설명해봐.
정답 보기

전달되지 않는다.

fragment는 클라이언트(브라우저) 전용이다. 서버에 요청이 갈 때 # 이후 부분은 잘려나간다. 브라우저가 응답을 받은 후 로컬에서 해당 위치로 스크롤하는 데 사용한다.

문제 3¶

다음 중 올바른 URL 설계는? 이유도 말해봐.

A) GET /getUsers
B) GET /users
C) GET /UserList
D) GET /api/get-all-users.do
정답 보기

B) GET /users

  • A: 동사(get)를 경로에 넣었다. 동사는 HTTP 메서드(GET)가 담당한다.
  • B: 올바름. 명사, 복수형, 소문자.
  • C: 카멜케이스. URL은 소문자가 관례다. 대소문자가 구분되므로 혼동을 일으킨다.
  • D: 동사(get-all), 확장자(.do). 둘 다 불필요.

문제 4¶

https://example.com 과 https://example.com:443 은
같은 주소야, 다른 주소야? 이유는?
정답 보기

같은 주소다.

https의 기본 포트가 443이기 때문이다. 포트를 생략하면 프로토콜의 기본 포트가 자동 적용된다.

  • https:// → 기본 포트 443
  • http:// → 기본 포트 80

따라서 https://example.com은 https://example.com:443과 동일하다.
