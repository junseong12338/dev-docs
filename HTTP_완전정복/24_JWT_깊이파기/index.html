24. JWT 깊이 파기¶

한 줄 요약

JWT = Header.Payload.Signature 3파트로 구성된 토큰. 서명으로 변조를 감지하고, Base64로 인코딩되어 있어서 누구나 내용을 읽을 수 있다.

핵심 개념¶

JWT가 뭐야¶

JWT (JSON Web Token) = JSON 형태의 정보를 자체적으로 담고 있는 토큰.

23장에서 Bearer Token을 배웠지? JWT는 그 Bearer Token의 대표적 구현이야.

일반 토큰:  "abc123xyz" → 의미 없는 문자열. 서버가 DB에서 찾아야 함.
JWT:        "eyJhbG..." → 토큰 자체에 사용자 정보가 들어 있음. DB 조회 불필요.

이게 왜 좋냐? 무상태(Stateless) 인증이 가능해지니까. 서버가 세션을 저장할 필요가 없어. 4장에서 배운 HTTP의 무상태 특성과 딱 맞아.

JWT 구조: xxxxx.yyyyy.zzzzz¶

JWT는 점(.)으로 구분된 3개의 파트로 구성돼.

eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiIxMjM0NTY3ODkwIiwibmFtZSI6IuydtOykgOyEsSIsImlhdCI6MTUxNjIzOTAyMn0.4VB0pL7kCDeYhVbORsxJjIOT3DzA0bEsY5GRQE_9mrc

│            Header           │              Payload              │         Signature          │

하나씩 뜯어보자.

1. Header (헤더)¶

토큰의 메타 정보. "이 토큰이 뭐고, 어떤 알고리즘으로 서명했는지".

{
  "alg": "HS256",
  "typ": "JWT"
}
필드 의미 값
alg 서명 알고리즘 HS256, RS256 등
typ 토큰 타입 JWT

이 JSON을 Base64URL 인코딩하면 Header 파트가 된다.

{"alg":"HS256","typ":"JWT"}
→ Base64URL
→ eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9

Base64URL이 뭐야?

Base64에서 + → -, / → _로 바꾸고, = 패딩을 제거한 버전. URL에 안전하게 포함할 수 있도록 변형한 거야.

2. Payload (페이로드)¶

토큰에 담긴 실제 정보(Claims). "이 토큰이 누구의 것이고, 언제 만료되는지".

{
  "sub": "1234567890",
  "name": "홍길동",
  "role": "ADMIN",
  "iat": 1516239022,
  "exp": 1516242622
}

Claims의 종류¶

종류 설명 예시
Registered Claims 표준으로 정해진 것 iss, sub, exp, iat, nbf
Public Claims 공개적으로 사용되는 것 name, email
Private Claims 우리끼리 정한 것 role, userId, departmentId

주요 Registered Claims¶

클레임 풀네임 의미
iss Issuer 토큰 발급자
sub Subject 토큰 대상 (보통 사용자 ID)
exp Expiration 만료 시간 (Unix timestamp)
iat Issued At 발급 시간
nbf Not Before 이 시간 이전에는 유효하지 않음
aud Audience 토큰 수신자
jti JWT ID 토큰 고유 식별자

Payload도 Base64URL 인코딩한다.

{"sub":"1234567890","name":"홍길동","iat":1516239022}
→ Base64URL
→ eyJzdWIiOiIxMjM0NTY3ODkwIiwibmFtZSI6IuydtOykgOyEsSIsImlhdCI6MTUxNjIzOTAyMn0

3. Signature (서명)¶

토큰이 변조되지 않았음을 증명하는 부분. 22장에서 배운 HMAC이 여기서 쓰인다.

HMAC-SHA256(
  base64UrlEncode(header) + "." + base64UrlEncode(payload),
  secret
)

풀어서 말하면:

  1. Header를 Base64URL 인코딩한다
  2. Payload를 Base64URL 인코딩한다
  3. 둘을 .으로 연결한다
  4. 비밀키(secret)와 함께 HMAC-SHA256으로 해싱한다
  5. 그 결과가 Signature
입력: "eyJhbGci...eyJzdWIi..."  (Header.Payload)
키:   "my_super_secret_key"

HMAC-SHA256(key, input) = 4VB0pL7kCDeYhVbORsxJjIOT3DzA0bEsY5GRQE_9mrc

22장 연결

22장에서 배운 HMAC-SHA256이 바로 여기서 쓰여. "메시지(Header.Payload) + 비밀키 → 해시값(Signature)" 비밀키를 모르면 이 서명을 만들 수 없어.

JWT 검증 과정¶

서버가 JWT를 받으면 이렇게 검증해:

sequenceDiagram
    participant C as 클라이언트
    participant S as 서버

    C->>S: Authorization: Bearer eyJhbG...eyJzdWI...4VB0pL7k

    Note over S: 1. 토큰을 . 으로 분리
    S->>S: Header = eyJhbG...
    S->>S: Payload = eyJzdWI...
    S->>S: Signature = 4VB0pL7k...

    Note over S: 2. Header, Payload를 Base64URL 디코딩
    S->>S: alg = "HS256", sub = "1234567890" 등 확인

    Note over S: 3. 서명 재생성
    S->>S: HMAC-SHA256(secret, Header.Payload) = expected_signature

    Note over S: 4. 서명 비교
    alt 일치
        S->>S: 변조 안 됨 + 우리가 발급한 거 맞음
        Note over S: 5. exp 만료 시간 확인
        alt 만료 안 됨
            S-->>C: 200 OK (인증 성공)
        else 만료됨
            S-->>C: 401 Unauthorized (토큰 만료)
        end
    else 불일치
        S-->>C: 401 Unauthorized (변조된 토큰)
    end

검증의 핵심¶

서명이 일치하면 두 가지가 보장돼:

  1. 변조 안 됨: Payload가 변경됐으면 서명이 달라졌을 거야
  2. 우리가 발급한 거 맞음: 비밀키를 모르면 이 서명을 만들 수 없으니까

이 모든 과정에서 DB 조회가 필요 없어. 토큰 자체에 정보가 있고, 서명만 검증하면 되니까. 이게 JWT의 최대 장점이야.

JWT의 함정¶

함정 1: JWT는 암호화가 아니야!

Header와 Payload는 Base64URL 인코딩일 뿐이야. 19장에서 배웠지? Base64는 누구나 디코딩 가능해.

Payload: eyJzdWIiOiIxMjM0NTY3ODkwIiwibmFtZSI6IuydtOykgOyEsSJ9

Base64URL 디코딩하면:
{"sub":"1234567890","name":"홍길동"}

누구나 볼 수 있어!

jwt.io 가서 JWT 토큰 붙여넣기하면 내용이 바로 보여. 민감한 정보(비밀번호, 주민번호 등)를 Payload에 넣으면 절대 안 돼.

함정 2: 발급 후 서버에서 강제 만료 불가

JWT는 무상태(Stateless)야. 서버에 토큰 정보를 저장하지 않아. 그래서 한 번 발급하면 만료 시간까지 유효해.

사용자가 "로그아웃" 했어도, 탈취된 토큰은 만료될 때까지 쓸 수 있어.

해결 방법: - Access Token 만료를 짧게 (15분~1시간) - Refresh Token으로 갱신 전략 - 블랙리스트 (서버에 무효화 목록 저장 -- 하지만 이러면 Stateless 장점이 줄어)

함정 3: 토큰 탈취

Bearer Token은 "소지자에게 권한을 준다"야. 토큰이 탈취되면 공격자가 마음대로 사용 가능.

대응: - HTTPS 필수 (전송 구간 암호화) - httpOnly Cookie에 저장 (XSS 방어) - 짧은 만료 시간 - Refresh Token Rotation (갱신할 때마다 Refresh Token도 새로 발급)

HS256 vs RS256¶

JWT 서명 알고리즘. 이 두 가지가 가장 많이 쓰여.

항목 HS256 RS256
정식 이름 HMAC-SHA256 RSA-SHA256
방식 대칭키 (하나의 비밀키) 비대칭키 (공개키 + 개인키)
서명 비밀키로 개인키로
검증 같은 비밀키로 공개키로
키 공유 서명자와 검증자가 같은 키 검증자는 공개키만 필요
속도 빠름 느림
용도 내부 시스템 (서버가 발급+검증) 외부 공개 API, MSA

언제 뭘 쓰나¶

flowchart TD
    A{누가 검증하나?} -->|발급자 = 검증자<br/>같은 서버| B[HS256]
    A -->|발급자 ≠ 검증자<br/>외부 서비스가 검증| C[RS256]

    B --> D["내부 API 인증<br/>모놀리식 서버"]
    C --> E["마이크로서비스<br/>외부 파트너 API<br/>공개 인증 서버"]

HS256: 우리 서버가 토큰을 만들고, 우리 서버가 검증한다. 비밀키를 공유할 필요 없으니 안전.

RS256: 인증 서버가 토큰을 만들고, 여러 서비스가 검증한다. 각 서비스에 비밀키를 나눠주는 건 위험하니까 공개키만 배포.

우리 프로젝트는?

NexClass는 단일 서버에서 토큰 발급+검증을 하니까 HS256이 적합해. 카카오, 구글 같은 인증 서버는 여러 서비스가 검증하니까 RS256을 써.

토큰 전체 예시¶

Header:
{
  "alg": "HS256",
  "typ": "JWT"
}
→ eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9

Payload:
{
  "sub": "user_001",
  "name": "홍길동",
  "role": "STUDENT",
  "iat": 1679000000,
  "exp": 1679003600
}
→ eyJzdWIiOiJ1c2VyXzAwMSIsIm5hbWUiOiLsnbTspIDshLEiLCJyb2xlIjoiU1RVREVOVCIsImlhdCI6MTY3OTAwMDAwMCwiZXhwIjoxNjc5MDAzNjAwfQ

  "eyJhbGci...eyJzdWIi...",

eyJhbGci...eyJzdWIi...XkZ4Hm7pQ9...

서버에서 검증하는 코드 (개념)¶

// 1. 토큰 분리
String[] parts = token.split("\\.");
String header = parts[0];
String payload = parts[1];
String signature = parts[2];

// 2. 서명 재계산
String expectedSignature = hmacSha256(secretKey, header + "." + payload);

// 3. 비교
if (!constantTimeEquals(expectedSignature, signature)) {
    throw new UnauthorizedException("토큰 변조됨");
}

// 4. Payload 디코딩
String payloadJson = base64UrlDecode(payload);
Claims claims = parseJson(payloadJson);

// 5. 만료 확인
if (claims.getExp() < currentTime()) {
    throw new UnauthorizedException("토큰 만료");
}

// 6. 인증 완료
User user = new User(claims.getSub(), claims.getName(), claims.getRole());

19~24장 총 연결¶

flowchart TD
    A["19장: 인코딩<br/>Base64URL"] -->|"Header, Payload 인코딩"| JWT
    B["20장: 해싱<br/>SHA-256"] -->|"서명에 사용"| HMAC
    C["21장: 암호화<br/>대칭키/비대칭키"] -->|"HS256(대칭) / RS256(비대칭)"| HMAC
    D["22장: HMAC<br/>해싱+비밀키"] -->|"Signature 생성"| HMAC
    E["23장: 인증 방식<br/>Bearer Token"] -->|"JWT를 Bearer로 전달"| JWT

    HMAC --> JWT["24장: JWT<br/>Header.Payload.Signature"]

    style JWT fill:#4CAF50,color:#fff,stroke-width:3px

19장~22장에서 배운 개념이 전부 JWT에서 합쳐져:

  • Base64URL 인코딩 (19장) → Header, Payload 인코딩
  • SHA-256 해싱 (20장) → 서명 알고리즘의 기반
  • 대칭키/비대칭키 (21장) → HS256(대칭) vs RS256(비대칭)
  • HMAC-SHA256 (22장) → Signature 생성
  • Bearer Token (23장) → JWT를 Authorization 헤더에 담아 전달

주의사항 / 함정¶

삽질 포인트

1. Payload에 민감 정보 넣지 마

JWT Payload는 Base64URL 디코딩하면 누구나 볼 수 있어. 비밀번호, 주민등록번호, 카드번호 같은 건 절대 넣지 마.

2. Secret Key를 안전하게 관리해 (HS256)

Secret Key가 노출되면 누구나 유효한 JWT를 만들 수 있어. 소스 코드에 하드코딩하지 말고, 환경변수나 키 관리 서비스에 저장해.

3. alg: "none" 공격 주의

JWT 라이브러리 중에 알고리즘을 "none"으로 보내면 서명 검증을 건너뛰는 버그가 있었어. 반드시 서버에서 허용할 알고리즘을 명시적으로 지정해야 해.

4. 토큰 크기 주의

Payload에 데이터를 많이 넣으면 토큰이 커져. 매 요청마다 전송되니까 불필요한 데이터는 빼. 필요한 최소한의 클레임만 넣어.

정리¶

개념 핵심
JWT JSON Web Token. 자체적으로 정보를 담은 토큰
Header 알고리즘 + 타입. Base64URL 인코딩
Payload 클레임(정보). Base64URL 인코딩. 누구나 볼 수 있음
Signature HMAC-SHA256(Header.Payload, secret). 변조 방지
HS256 대칭키 방식. 내부 시스템에 적합
RS256 비대칭키 방식. 외부/MSA에 적합
핵심 함정 JWT는 암호화가 아니야. Payload는 공개 데이터

한 문장 정리

JWT는 Header.Payload.Signature 구조로, 서명으로 변조를 감지하지만 내용 자체는 공개되어 있다. 19~22장의 인코딩, 해싱, 암호화, HMAC이 전부 여기서 합쳐져.

확인 문제¶

문제 1

JWT의 3개 파트를 말하고, 각각 뭐가 들어있는지 설명해봐.

정답 보기

  1. Header: 토큰 메타 정보. 서명 알고리즘(alg)과 타입(typ). Base64URL로 인코딩됨.

  2. Payload: 클레임(토큰에 담긴 정보). 사용자 ID, 이름, 만료 시간 등. Base64URL로 인코딩됨. 누구나 디코딩해서 볼 수 있음.

  3. Signature: Header와 Payload를 비밀키로 HMAC-SHA256 해싱한 값. 토큰이 변조되지 않았음을 증명.

문제 2

"JWT는 암호화되어 있으니까 안전하다"라는 말이 왜 틀린지 설명해봐.

정답 보기

JWT는 암호화가 아니야. Header와 Payload는 Base64URL 인코딩일 뿐이야. 19장에서 배웠듯이 Base64는 누구나 디코딩 가능해.

jwt.io에 토큰을 붙여넣으면 내용이 바로 보여. JWT가 보장하는 건 무결성(변조 감지)이지, 기밀성(내용 숨기기)이 아니야.

그래서 Payload에 비밀번호, 주민번호 같은 민감 정보를 넣으면 안 돼. 전송 구간의 기밀성은 HTTPS(TLS)가 담당한다.

문제 3

HS256과 RS256의 차이를 설명하고, 각각 어떤 상황에서 쓰이는지 말해봐.

정답 보기
항목 HS256 RS256
방식 대칭키 (비밀키 하나) 비대칭키 (공개키 + 개인키)
서명/검증 같은 키로 개인키로 서명, 공개키로 검증
속도 빠름 느림

HS256: 토큰을 발급하는 쪽과 검증하는 쪽이 같을 때. 예: 단일 서버에서 발급하고 같은 서버에서 검증 (우리 NexClass).

RS256: 발급자와 검증자가 다를 때. 예: 인증 서버가 발급하고, 여러 마이크로서비스가 검증. 비밀키(개인키)는 인증 서버만 갖고, 공개키만 배포하니까 안전.

문제 4

19장~24장에서 배운 개념이 JWT에서 어떻게 연결되는지 설명해봐. 최소 4개의 연결점을 말해.

정답 보기

  1. 19장 인코딩 (Base64URL): Header와 Payload를 Base64URL로 인코딩. 누구나 디코딩 가능하므로 JWT는 암호화가 아님.

  2. 20장 해싱 (SHA-256): Signature를 만들 때 SHA-256 해시 함수가 사용됨. 단방향이라 서명에서 비밀키를 역추적할 수 없음.

  3. 21장 암호화 (대칭키/비대칭키): HS256은 대칭키(21장), RS256은 비대칭키(21장) 방식. 용도에 따라 선택.

  4. 22장 HMAC: HS256의 Signature = HMAC-SHA256(비밀키, Header.Payload). 비밀키를 아는 사람만 올바른 서명을 만들 수 있어서 변조 감지 + 인증이 가능.

  5. 23장 Bearer Token: JWT는 Authorization: Bearer <token> 형태로 전달. Bearer Token의 가장 대표적인 구현이 JWT.
