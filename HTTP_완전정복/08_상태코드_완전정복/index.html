08. HTTP 상태 코드 완전정복¶

이것만 기억해

에러 나면 상태 코드부터 봐. 4xx면 네 잘못, 5xx면 서버 잘못. 이거 하나만 알아도 디버깅 속도가 2배 빨라져.

상태 코드란?¶

서버가 요청을 받고 처리한 결과를 3자리 숫자로 알려주는 거야.

HTTP/1.1 200 OK

여기서 200이 상태 코드, OK가 사유 문구(Reason Phrase)야.

  • 첫 번째 자리: 응답의 종류를 결정해
  • 나머지 두 자리: 세부 상황을 구분해
범위 의미 한마디
1xx 정보 "알겠어, 계속 보내"
2xx 성공 "잘 처리했어"
3xx 리다이렉트 "다른 데로 가"
4xx 클라이언트 에러 "네가 잘못 보냈어"
5xx 서버 에러 "내가 터졌어"

이 5가지 분류를 머릿속에 박아놔. 상태 코드 수십 개를 다 외울 필요는 없어. 첫 자리만 봐도 "아 이건 내 잘못이구나" 바로 판단 가능해.

1xx - 정보 응답¶

실무 빈도: 거의 안 봄

존재만 알면 돼. 직접 다룰 일은 거의 없어.

100 Continue¶

클라이언트가 큰 body를 보내기 전에 "이 요청 계속 보내도 돼?" 하고 물어볼 때 서버가 "ㅇㅇ 계속 보내" 하고 응답하는 거야.

동작 흐름:

sequenceDiagram
    participant C as 클라이언트
    participant S as 서버
    C->>S: POST /upload (헤더만, Expect: 100-continue)
    S->>C: 100 Continue
    C->>S: (실제 body 전송)
    S->>C: 200 OK

왜 이런 게 있냐면, 10GB짜리 파일 보내려는데 서버가 "그 Content-Type 안 받아" 하면? body 다 보내고 나서 거절당하면 시간 낭비잖아. 그래서 헤더만 먼저 보내서 확인받는 거야.

101 Switching Protocols¶

프로토콜을 바꿀 때 사용해. 대표적으로 WebSocket 연결할 때 나와.

GET /chat HTTP/1.1
Upgrade: websocket
Connection: Upgrade

HTTP/1.1 101 Switching Protocols
Upgrade: websocket
Connection: Upgrade

HTTP에서 WebSocket으로 업그레이드하는 과정에서 딱 한 번 나와. 이후로는 WebSocket 프로토콜로 통신해.

2xx - 성공¶

핵심

"요청 잘 처리했어"의 변주들. 200, 201, 204 이 세 개는 무조건 알아야 해.

200 OK¶

가장 기본적인 성공 응답. GET으로 데이터 조회했을 때, PUT으로 수정했을 때 주로 사용해.

GET /api/users/1 HTTP/1.1

HTTP/1.1 200 OK
Content-Type: application/json

{
  "id": 1,
  "name": "홍길동",
  "email": "test@example.com"
}

201 Created¶

새 리소스가 생성됐을 때. POST로 뭔가 만들면 201이 맞아.

POST /api/users HTTP/1.1
Content-Type: application/json

{
  "name": "홍길동",
  "email": "hong@example.com"
}

HTTP/1.1 201 Created
Location: /api/users/42
Content-Type: application/json

{
  "id": 42,
  "name": "홍길동",
  "email": "hong@example.com"
}

Location 헤더

201 응답할 때 Location 헤더에 새로 만든 리소스의 URL을 넣어주는 게 좋은 설계야. "여기에 만들어놨어" 하고 알려주는 거지.

204 No Content¶

성공했지만 돌려줄 body가 없을 때. DELETE에서 자주 사용해.

DELETE /api/users/42 HTTP/1.1

HTTP/1.1 204 No Content
(body 없음)

삭제했는데 뭘 돌려줘? 삭제된 데이터를? 그건 이상하잖아. 그래서 "잘 삭제했어, 근데 줄 건 없어" = 204.

200 vs 201 vs 204 비교¶

상황 상태 코드 body 유무
GET으로 데이터 조회 성공 200 O (조회된 데이터)
POST로 새 리소스 생성 201 O (생성된 데이터) + Location 헤더
PUT으로 기존 리소스 수정 200 O (수정된 데이터)
DELETE로 리소스 삭제 204 X
PUT으로 수정했지만 body 안 줄 때 204 X

흔한 실수

POST로 뭔가 만들어놓고 200 보내는 경우가 많아. 동작은 하지만 의미가 다른 거야. 200은 "잘 처리했어", 201은 "새로 만들었어". API 설계할 때 이 차이를 지켜야 소비자가 헷갈리지 않아.

202 Accepted (보너스)¶

"요청은 받았는데 아직 처리 안 끝났어." 비동기 작업에서 사용해. 이메일 발송, 대용량 파일 처리 같은 것.

3xx - 리다이렉트¶

핵심

"네가 찾는 거 여기 아니야, 저기로 가." 브라우저가 자동으로 따라가.

301 Moved Permanently¶

영구적으로 이동했어. "이 URL 더 이상 안 써, 저쪽으로 바뀌었어."

GET /old-page HTTP/1.1
Host: example.com

HTTP/1.1 301 Moved Permanently
Location: https://example.com/new-page
  • 브라우저가 Location 헤더 URL로 자동 이동
  • 검색 엔진(구글)이 새 URL로 SEO 점수를 이전해줌
  • 브라우저가 이 리다이렉트를 캐시함 (다음에 old-page 접근하면 서버에 안 물어보고 바로 new-page로 감)

302 Found¶

임시로 이동했어. "지금은 저기로 가, 근데 나중에 또 바뀔 수 있어."

GET /event HTTP/1.1

HTTP/1.1 302 Found
Location: https://example.com/event-2025
  • 브라우저가 Location으로 자동 이동
  • 검색 엔진이 기존 URL의 SEO 점수를 유지
  • 브라우저가 이 리다이렉트를 캐시하지 않음

301 vs 302 차이¶

구분 301 Moved Permanently 302 Found
의미 영구 이동 임시 이동
SEO 새 URL로 점수 이전 기존 URL에 점수 유지
브라우저 캐시 O (캐시함) X (매번 물어봄)
사용 예시 도메인 변경, URL 구조 변경 로그인 후 리다이렉트, 이벤트 페이지

함정

301을 잘못 쓰면 큰일나. 브라우저가 캐시하니까 "아 잘못 설정했다" 하고 서버에서 바꿔도 사용자 브라우저에는 이전 캐시가 남아있어. 확실하지 않으면 302 써.

304 Not Modified¶

이건 리다이렉트가 아니라 캐시 관련이야. "네가 가진 캐시 그대로 써도 돼."

sequenceDiagram
    participant C as 브라우저
    participant S as 서버
    C->>S: GET /style.css (If-None-Match: "abc123")
    S->>C: 304 Not Modified (body 없음)
    Note over C: 캐시에 저장된 style.css 사용

동작 원리:

  1. 처음 요청: 서버가 ETag: "abc123" 헤더와 함께 파일을 줌
  2. 두 번째 요청: 브라우저가 If-None-Match: "abc123" 보냄 ("이거 아직 유효해?")
  3. 서버가 확인하고 안 바뀌었으면 → 304 (body 없이, 네트워크 절약)
  4. 바뀌었으면 → 200 + 새 파일

304는 body를 안 보내니까 네트워크 트래픽을 확 줄여줘. 이미지, CSS, JS 같은 정적 파일에서 엄청 유용해.

4xx - 클라이언트 에러¶

핵심

"네가 잘못 보냈어." 프론트엔드 개발자가 가장 많이 보는 에러들. 4xx 만나면 내 요청부터 확인해.

400 Bad Request¶

요청 형식이 잘못됐어. 서버가 이해를 못 해.

POST /api/users HTTP/1.1
Content-Type: application/json

{ name: "홍길동" }  ← JSON 형식 오류! 키에 따옴표 없음

흔한 400 원인들:

  • JSON 형식 오류 (따옴표 빠짐, 쉼표 잘못됨)
  • 필수 파라미터 누락
  • 데이터 타입 불일치 (숫자 자리에 문자열)
  • 유효성 검증 실패 (이메일 형식 아님, 날짜 형식 틀림)

401 Unauthorized¶

인증 안 됨. "넌 누군지 모르겠어. 로그인해."

GET /api/my-profile HTTP/1.1
(Authorization 헤더 없음)

HTTP/1.1 401 Unauthorized
WWW-Authenticate: Bearer
  • 로그인 안 한 상태에서 인증 필요한 API 호출
  • 토큰이 만료됨
  • 토큰 형식이 잘못됨

403 Forbidden¶

권한 없음. "넌 누군지 알겠어. 근데 이건 못 해."

GET /api/admin/users HTTP/1.1
Authorization: Bearer user-token-123

HTTP/1.1 403 Forbidden
  • 일반 사용자가 관리자 API 호출
  • 다른 사람의 리소스에 접근 시도
  • IP 차단됨

401 vs 403: 이거 진짜 헷갈려¶

이거 모르면 삽질해

면접에서도 나오고, 실무에서도 잘못 쓰는 경우가 많아. 확실히 구분해.

구분 401 Unauthorized 403 Forbidden
의미 인증 실패 (Authentication) 인가 실패 (Authorization)
상황 "넌 누구야?" "넌 알겠는데, 안 돼"
해결법 로그인 / 토큰 갱신 권한 요청 / 관리자 문의
예시 토큰 없이 API 호출 일반 유저가 admin API 호출
재시도 의미 인증 정보 넣으면 성공 가능 같은 인증으로 재시도해도 실패 
flowchart TD
    A[API 요청] --> B{인증 정보 있어?}
    B -->|없음| C[401 Unauthorized]
    B -->|있음| D{인증 유효해?}
    D -->|무효| C
    D -->|유효| E{권한 있어?}
    E -->|없음| F[403 Forbidden]
    E -->|있음| G[200 OK]

쉽게 외우는 법:

  • 401 = 신분증 안 가져옴 (출입구에서 막힘)
  • 403 = 신분증 있는데 VIP 아님 (VIP 라운지에서 막힘)

404 Not Found¶

리소스 없음. 가장 유명한 에러.

GET /api/users/99999 HTTP/1.1

HTTP/1.1 404 Not Found
  • URL 잘못 입력
  • 존재하지 않는 리소스 ID
  • 삭제된 리소스

보안 목적의 404

가끔 403 대신 404를 일부러 보내는 경우가 있어. 왜? 403을 보내면 "이 URL에 뭔가 있긴 한데 권한이 없구나"라는 정보를 노출하잖아. 보안이 중요한 경우 "그런 거 없어"라고 404로 숨기기도 해. GitHub의 private 레포에 접근하면 404가 나오는 이유가 이거야.

405 Method Not Allowed¶

이 URL에 이 HTTP 메서드 안 돼.

DELETE /api/users HTTP/1.1

HTTP/1.1 405 Method Not Allowed
Allow: GET, POST

/api/users에는 GET(목록 조회)이랑 POST(생성)만 되는데 DELETE를 날린 거야. Allow 헤더로 어떤 메서드가 가능한지 알려줘.

409 Conflict¶

충돌. 현재 리소스 상태와 요청이 충돌할 때.

PUT /api/users/1 HTTP/1.1
If-Match: "abc123"  ← 이 버전 기준으로 수정할게

HTTP/1.1 409 Conflict  ← 이미 다른 사람이 수정함

흔한 상황: - 동시에 같은 문서를 수정할 때 (Optimistic Locking 실패) - 이미 존재하는 username으로 가입 시도 - 이미 삭제된 리소스를 수정하려 할 때

429 Too Many Requests¶

요청 너무 많아. Rate Limiting에 걸린 거야.

HTTP/1.1 429 Too Many Requests
Retry-After: 60
  • API 호출 제한 초과 (예: 분당 100회)
  • Retry-After 헤더로 몇 초 후에 다시 시도하라고 알려줌
  • 외부 API(GitHub, Zoom, Google) 쓸 때 자주 만남

429 만나면

무한 재시도하면 더 오래 차단당해. Retry-After 값을 보고 그만큼 기다렸다가 재시도해. 그리고 지수 백오프(Exponential Backoff) 패턴을 적용해: 1초 → 2초 → 4초 → 8초 이렇게 간격을 늘려가면서 재시도.

5xx - 서버 에러¶

핵심

"서버가 잘못했어." 백엔드 개발자가 가장 싫어하는 에러들. 5xx 나오면 서버 로그부터 봐.

500 Internal Server Error¶

서버 터짐. 가장 범용적인 서버 에러.

HTTP/1.1 500 Internal Server Error

서버 코드에서 처리 안 된 예외가 터진 거야. NullPointerException, ArrayIndexOutOfBoundsException 같은 거.

500 만나면 체크리스트:

  1. 서버 로그 확인 (스택 트레이스 찾기)
  2. 최근 배포/변경 사항 확인
  3. 요청 데이터에 예상 못 한 값 있는지 확인
  4. DB 연결 상태 확인

502 Bad Gateway¶

게이트웨이/프록시가 뒤쪽 서버에서 잘못된 응답을 받음.

sequenceDiagram
    participant C as 클라이언트
    participant N as Nginx (프록시)
    participant A as App Server (Tomcat)
    C->>N: GET /api/data
    N->>A: 요청 전달
    A->>N: 잘못된 응답 또는 연결 끊김
    N->>C: 502 Bad Gateway

흔한 원인: - 뒤쪽 앱 서버(Tomcat, Node)가 죽어있음 - 앱 서버가 비정상적인 응답을 보냄 - 앱 서버 프로세스가 크래시

503 Service Unavailable¶

서버가 일시적으로 처리할 수 없음. 과부하 또는 점검 중.

HTTP/1.1 503 Service Unavailable
Retry-After: 3600
  • 서버 점검 중 (배포 중)
  • 트래픽 폭주로 요청 처리 불가
  • 의도적으로 503 내보내는 경우 (점검 페이지)

504 Gateway Timeout¶

게이트웨이가 뒤쪽 서버 응답을 기다리다가 시간 초과.

sequenceDiagram
    participant C as 클라이언트
    participant N as Nginx (프록시)
    participant A as App Server
    C->>N: GET /api/heavy-query
    N->>A: 요청 전달
    Note over N,A: 30초 경과... 응답 없음
    N->>C: 504 Gateway Timeout

흔한 원인: - DB 쿼리가 너무 오래 걸림 - 외부 API 호출이 응답 안 함 - 서버 처리 로직이 무한 루프

502 vs 504: Nginx 쓸 때 자주 만나는 구분¶

구분 502 Bad Gateway 504 Gateway Timeout
의미 뒤쪽 서버가 이상한 응답 줌 뒤쪽 서버가 응답 안 함
원인 서버 죽음, 비정상 응답 서버 느림, 처리 시간 초과
해결 앱 서버 상태 확인, 재시작 쿼리 최적화, 타임아웃 설정 조정
Nginx 로그 connect() failed upstream timed out

실무 팁

Nginx에서 502/504 만나면 proxy_read_timeout이랑 proxy_connect_timeout 설정을 확인해. 기본값이 60초인데, 무거운 API는 이것보다 오래 걸릴 수 있어. 근데 타임아웃 늘리는 건 임시방편이야. 근본적으로는 느린 처리를 최적화해야 해.

실전에서 자주 만나는 상태 코드 TOP 10¶

순위 코드 이름 어디서 만나?
1 200 OK 거의 모든 성공 응답
2 404 Not Found URL 오타, 삭제된 페이지
3 500 Internal Server Error 서버 버그, 예외 처리 누락
4 401 Unauthorized 토큰 만료, 미인증 접근
5 403 Forbidden 권한 부족, IP 차단

정리¶

이 장에서 배운 것

  • 상태 코드는 3자리 숫자로, 첫 자리가 응답의 종류를 결정해
  • 2xx = 성공 (200, 201, 204 구분이 중요)
  • 3xx = 리다이렉트 (301 영구, 302 임시, 304 캐시)
  • 4xx = 클라이언트 잘못 (401 인증, 403 권한 구분 필수)
  • 5xx = 서버 잘못 (502 vs 504 구분하면 디버깅 빨라져)
  • 에러 나면 상태 코드 첫 자리부터 보는 습관을 들여

확인 문제¶

문제 1

POST로 새 사용자를 생성하는 API를 만들었어. 성공 응답으로 어떤 상태 코드를 보내야 할까? 200? 201? 204? 이유도 말해봐.

정답 보기

201 Created. POST로 새 리소스를 만들었으니까. 200은 "처리 성공"이고 201은 "새로 만들었어"라는 더 정확한 의미를 전달해. 그리고 Location 헤더에 생성된 리소스 URL도 같이 보내면 베스트. 204는 body 없을 때 쓰는 건데, 보통 생성 결과를 body로 돌려주니까 여기선 안 맞아.

문제 2

로그인한 일반 사용자가 관리자 전용 API를 호출했어. 401이야 403이야? 그 이유는?

정답 보기

403 Forbidden. 로그인은 했으니까 인증(Authentication)은 통과한 거야. 근데 관리자 권한이 없으니까 인가(Authorization)에서 막힌 거지. 401은 "넌 누군지 모르겠어"이고, 403은 "넌 알겠는데 권한 없어"야. 로그인 안 한 상태였으면 401이 맞아.

문제 3

Nginx + Tomcat 구조에서 504 에러가 계속 나와. 어디를 먼저 확인해야 해?

정답 보기

504는 게이트웨이(Nginx)가 뒤쪽 서버(Tomcat) 응답을 기다리다 시간 초과된 거야. 먼저 확인할 것: (1) Tomcat이 살아있는지, (2) Tomcat에서 처리 중인 요청이 오래 걸리는 건 아닌지 (느린 DB 쿼리 등), (3) Nginx의 proxy_read_timeout 설정값. 502는 Tomcat이 죽었거나 이상한 응답을 줄 때고, 504는 Tomcat이 살아있지만 너무 느릴 때야.

문제 4

브라우저가 CSS 파일을 요청했는데 304가 왔어. 이게 무슨 뜻이고, body는 있어 없어?

정답 보기

"네가 가진 캐시 그대로 써도 돼"라는 뜻이야. body는 없어. 브라우저가 If-None-Match 헤더로 "이 버전 아직 유효해?" 물어봤고, 서버가 확인해보니 안 바뀌었으니까 body 없이 304만 보낸 거야. 이러면 네트워크 트래픽을 절약할 수 있어. 파일이 바뀌었으면 200 + 새 파일을 보냈겠지.

문제 5

외부 API를 호출했더니 429 에러가 왔어. 바로 재시도하면 될까? 어떻게 해야 해?

정답 보기

절대 바로 재시도하면 안 돼. 바로 재시도하면 더 오래 차단당할 수 있어. 해야 할 것: (1) Retry-After 헤더를 확인해서 그만큼 기다려, (2) 지수 백오프(Exponential Backoff) 패턴을 적용해 - 1초, 2초, 4초, 8초 이렇게 간격을 늘려가면서 재시도, (3) 근본적으로는 요청 빈도를 줄이거나 캐싱을 적용해서 API 호출 횟수를 줄여야 해.
