30. 최종시험 - 정답 및 해설¶

해설 읽는 법

"정답 맞히는 건 시작이야. 왜 그게 정답인지 설명 못 하면 불합격."

이건 정답지가 아니야. 해설지야. 정답만 보고 넘어가면 Lv1에서 영원히 못 벗어나. 틀린 문제는 해당 챕터 다시 읽고 와.

PART 1: 객관식 정답 및 해설¶

Q1. HTTP의 핵심 특성이 아닌 것 (객관식, 2점)¶

정답: (B) 연결 지향 (Connection-Oriented)

해설:

HTTP/1.1은 Keep-Alive로 연결을 재사용하긴 하지만, HTTP 프로토콜 자체의 핵심 특성은 요청-응답, 무상태, 텍스트 기반 이 3가지야.

보기 왜 틀렸는가 (또는 왜 맞는가)
(A) 무상태 핵심 특성 맞음. 서버는 이전 요청을 기억하지 않는다.
(B) 연결 지향 HTTP는 연결 지향이 아니야. TCP가 연결 지향이지. HTTP는 TCP 위에서 동작할 뿐. HTTP 자체는 각 요청이 독립적.
(C) 요청-응답 핵심 특성 맞음. 항상 클라이언트가 먼저 요청, 서버가 응답.
(D) 텍스트 기반 핵심 특성 맞음. HTTP/1.1 기준 사람이 읽을 수 있는 텍스트.

"연결 지향"은 TCP의 특성이야. HTTP와 TCP를 혼동하면 안 돼. HTTP는 TCP 위에서 동작하는 응용 계층 프로토콜이지, TCP의 특성을 자기 것으로 가져가는 게 아니야.

Q5. TLS가 보장하지 않는 것 (객관식, 2점)¶

보기 왜 틀렸는가 (또는 왜 맞는가)

Q6. Base64는 암호화인가 (객관식 함정, 2점)¶

정답: (B) 틀리다. Base64는 암호화가 아니라 인코딩이다. 누구나 디코딩할 수 있다.

해설:

이건 면접에서도 나오는 클래식 함정이야. "Base64로 암호화했어요"라고 말하면 즉시 불합격.

보기 왜 틀렸는가 (또는 왜 맞는가)
(A) Base64에 키 개념이 없어. 키 없이 동작하니까 암호화가 아니야.
(B) 정답. Base64는 데이터 형태 변환(인코딩)일 뿐이야. 규칙이 공개되어 있고 누구나 디코딩 가능.
(C) Base64는 원본을 추측할 필요조차 없어. 그냥 디코딩하면 원본이 나와.
(D) Base64는 해싱이 아니야. 되돌릴 수 있으니까 단방향이 아님.

인코딩 vs 해싱 vs 암호화 구분:

  • 인코딩: 형태 변환. 키 없음. 누구나 되돌림 가능. (Base64, URL 인코딩)
  • 해싱: 단방향 변환. 키 없음. 되돌림 불가능. (SHA-256)
  • 암호화: 양방향 변환. 키 필요. 키 가진 사람만 되돌림 가능. (AES, RSA)

Q20. RESTful URL 안티패턴 수정 (코드 분석, 5점)¶

정답:

(1) GET /getUsers -> GET /users

  • 문제: URL에 동사(get)가 포함. 조회는 GET 메서드가 담당.

(2) POST /api/deleteUser?id=123 -> DELETE /api/users/123

  • 문제: 삭제인데 POST 사용 + URL에 동사(delete) + 단수형(User) + ID를 query string으로 전달.

(3) GET /api/User/123 -> GET /api/users/123

  • 문제: 대문자 사용(User) + 단수형. URL은 소문자 + 복수형이 원칙.

(4) PUT /api/update_user_profile/123 -> PUT /api/user-profiles/123 또는 PUT /api/users/123/profile

  • 문제: URL에 동사(update) + 스네이크 케이스(_). URL은 케밥 케이스(-) 사용.

(5) GET /api/courses/456/getLectures -> GET /api/courses/456/lectures

  • 문제: URL에 동사(get). 계층 구조는 맞지만 동사를 빼야 함.

해설:

REST URL 안티패턴 정리:

위반 원칙
URL에 동사 넣기 명사만 사용. 동작은 HTTP 메서드로
단수형 (/user) 복수형 (/users). 컬렉션이니까
대문자 (/User) 소문자 (/users)
스네이크 케이스 (user_profile) 케밥 케이스 (user-profiles)
ID를 query string으로 경로 파라미터 (/users/123)

Q22. JWT Payload 분석 (코드 분석, 5점)¶

정답:

(1) 소유자(subject): user123 (sub 클레임)

(2) iat = Issued At (토큰 발급 시간), exp = Expiration Time (토큰 만료 시간). 둘 다 Unix Timestamp(1970-01-01부터 초 단위).

(3) 유효 시간: exp - iat = 1700003600 - 1700000000 = 3600초 (60분, 1시간)

(4) JWT의 Payload는 Base64 인코딩이므로 암호화가 아니야. 누구나 디코딩해서 내용을 읽을 수 있다. 따라서 Payload에 비밀번호나 민감한 개인정보를 넣으면 안 된다.

(5) Signature는 Header + Payload가 변조되지 않았음을 검증하는 역할을 한다. 서버가 비밀키로 서명하고, 검증 시 같은 키로 다시 서명해서 비교한다. 값이 다르면 변조된 것.

해설:

JWT의 핵심을 정리하면:

  • Header, Payload = Base64 인코딩 (누구나 읽을 수 있음, 암호화 아님)
  • Signature = 비밀키로 만든 서명 (변조 감지용)
  • JWT는 기밀성이 아니라 무결성을 보장하는 거야

채점 기준: 각 항목 1점씩. 총 5점.

Q23. 상황별 상태 코드 (코드 분석, 5점)¶

정답:

(1) 201 Created -- POST로 새 리소스를 생성했으니 201. 200이 아니야. "잘 처리했어"가 아니라 "새로 만들었어"라는 더 정확한 의미.

(2) 204 No Content -- 삭제 성공했지만 돌려줄 body가 없을 때. "성공했어, 근데 줄 건 없어."

(3) 400 Bad Request -- 클라이언트가 잘못된 요청을 보낸 것. 필수 필드 누락은 요청 형식 오류.

(4) 429 Too Many Requests -- Rate Limiting에 걸린 것. Retry-After 헤더로 얼마 후 재시도하라고 안내.

(5) 504 Gateway Timeout -- Nginx(프록시)가 Tomcat(앱 서버) 응답을 기다리다 시간 초과. 502와 구분: 502는 앱 서버가 죽었거나 비정상 응답, 504는 앱 서버가 살아있지만 느린 것.

해설:

상황 코드 혼동하기 쉬운 코드 차이
POST 생성 성공 201 200 200은 범용 성공, 201은 "생성됨"
DELETE body 없음 204 200 200은 body 있을 때, 204는 body 없을 때
필수 필드 누락 400 422 400은 형식 오류, 422는 의미 오류
Rate Limiting 429 403 429는 "요청 너무 많아", 403은 "권한 없어"
프록시 타임아웃 504 502 504는 시간 초과, 502는 비정상 응답

채점 기준: 상태 코드 정확하면 0.5점 + 이유 설명 0.5점. 각 항목 1점씩. 총 5점.

Q25. Webhook 서명 검증 빈칸 채우기 (코드 작성, 5점)¶

정답:

(A): secret_key (서버와 미리 공유한 비밀키) (B): received_body (Webhook으로 받은 요청 body) (C): sha256 (해시 알고리즘)

추가 답변: 서명 검증 없이 Webhook을 받으면, 공격자가 가짜 요청을 보내도 구분할 수 없기 때문이다. 누구나 우리 서버의 Webhook URL로 POST 요청을 보낼 수 있으므로, 그 요청이 진짜 Zoom(또는 해당 서비스)에서 온 것인지 서명으로 검증해야 한다.

해설:

HMAC 서명 검증의 핵심 원리:

HMAC-SHA256(비밀키, 메시지) = 서명값

검증 과정:
1. 받은 body와 우리가 가진 비밀키로 HMAC-SHA256 계산
2. 계산한 값과 요청 헤더에 담겨 온 서명값 비교
3. 일치하면 → 진짜 (비밀키를 아는 사람만 만들 수 있으니까)
4. 불일치면 → 가짜 또는 변조됨

HMAC의 핵심: 비밀키를 모르면 같은 서명값을 만들 수 없다. 일반 SHA-256은 누구나 계산할 수 있지만, HMAC은 비밀키를 알아야만 정확한 값을 만들 수 있어.

채점 기준: (A) 1점, (B) 1점, (C) 1점, 추가 답변 2점. 총 5점.

Q26. 쿠키와 세션으로 로그인 유지 (서술형, 4점)¶

정답:

쿠키의 역할: 서버가 브라우저에 심는 작은 데이터 조각. 브라우저가 저장해두고 이후 요청마다 자동으로 서버에 돌려보낸다. "나 이 사람이야"라는 증거를 매번 들고 다니는 메모지 같은 것.

세션의 역할: 서버 측에서 사용자별 상태를 저장하는 공간. 세션 ID를 키로 사용해서 해당 사용자의 정보(로그인 여부, 사용자 정보 등)를 서버 메모리나 DB에 보관한다.

협력 과정:

  1. 사용자가 로그인 요청을 보냄 (POST /login, 아이디+비밀번호)
  2. 서버가 인증 성공하면 세션을 생성하고, 세션 ID를 발급
  3. 서버가 응답할 때 Set-Cookie: sessionId=abc123 헤더로 세션 ID를 브라우저에 전달
  4. 브라우저가 쿠키를 저장
  5. 이후 요청마다 브라우저가 자동으로 Cookie: sessionId=abc123 헤더를 붙여서 보냄
  6. 서버가 Cookie 헤더에서 세션 ID를 꺼내고, 해당 세션을 조회해서 "아, 홍길동이구나" 확인

HTTP 헤더: 서버 -> 브라우저는 Set-Cookie, 브라우저 -> 서버는 Cookie.

핵심: HTTP 자체는 무상태지만, 쿠키라는 메커니즘으로 매 요청마다 "나 로그인한 사람이야"라는 증거(세션 ID)를 전달해서 상태를 유지하는 것.

채점 기준:

항목 배점
쿠키와 세션 각각의 역할 구분 1점
로그인 -> Set-Cookie -> Cookie 전달 흐름 1.5점
관련 HTTP 헤더 (Set-Cookie, Cookie) 명시 0.5점
"HTTP는 무상태지만 쿠키로 보완한다"는 근본 이해 1점

Q27. TLS 핸드셰이크 과정 (서술형, 5점)¶

정답:

TLS 핸드셰이크 과정:

1단계 - Client Hello: 클라이언트가 서버에 먼저 손을 내민다. 지원하는 TLS 버전, 지원하는 암호화 방식 목록(Cipher Suites), 클라이언트 랜덤값을 보낸다.

2단계 - Server Hello: 서버가 사용할 TLS 버전과 암호화 방식을 선택하고, 서버 랜덤값을 보낸다.

3단계 - 인증서 전송: 서버가 자기 인증서(Certificate)를 보낸다. 이 인증서는 "나 진짜 이 도메인의 서버야"를 증명하는 신분증이다. 인증 기관(CA)이 발급한 것이며, 서버의 공개키가 포함되어 있다.

4단계 - 인증서 검증: 클라이언트(브라우저)가 인증서를 검증한다. 신뢰할 수 있는 CA가 발급한 건지, 도메인이 일치하는지, 만료되지 않았는지 확인.

5단계 - 키 교환: 클라이언트가 대칭키를 만들기 위한 재료를 서버의 공개키(비대칭키)로 암호화해서 보낸다. 개인키를 가진 서버만 복호화 가능.

6단계 - 대칭키 생성: 양쪽이 클라이언트 랜덤 + 서버 랜덤 + 키 교환 재료로 동일한 대칭키를 각자 생성한다. 이후 모든 통신은 이 대칭키로 암호화.

비대칭키와 대칭키를 둘 다 쓰는 이유:

  • 비대칭키: 안전하지만 느리다 (대칭키의 100~1000배). 대칭키를 안전하게 교환하는 데만 사용.
  • 대칭키: 빠르지만 키 전달이 문제. 비대칭키로 안전하게 교환한 후 사용.
  • 결론: 비대칭키의 안전성 + 대칭키의 속도, 둘의 장점을 합친 것.

핸드셰이크 완료 후 실제 통신: 대칭키로 HTTP 메시지를 암호화/복호화한다. 빠르고 안전하게.

채점 기준:

항목 배점
Client Hello 설명 (TLS 버전, Cipher Suite, 랜덤값) 1점
인증서 전송과 검증 설명 1점
비대칭키/대칭키 둘 다 쓰는 이유 (안전성 + 속도) 1.5점
핸드셰이크 후 대칭키로 통신 명시 0.5점
전체 흐름의 논리적 연결 1점

  • 세션 기반: 서버가 상태를 가지므로 HTTP의 무상태 원칙과 충돌. 서버 여러 대면 세션 공유(Redis 등) 필요.
  • JWT 기반: 토큰 자체에 정보가 있어서 DB 조회 불필요. 무상태. 하지만 탈취 시 대응이 어렵다는 치명적 단점.
  • API Key: 단순하지만 강력. 주로 서버 간 통신에서 사용. 사용자 인증보다는 클라이언트(앱) 인증에 적합.

Q29. Webhook 서명 검증 필요성과 과정 (서술형, 4점)¶

정답:

서명 검증 없이 Webhook을 받으면 생기는 위험:

Webhook URL은 공개된 HTTP 엔드포인트야. 누구나 그 URL로 POST 요청을 보낼 수 있어. 서명 검증이 없으면:

  • 공격자가 가짜 이벤트를 보내서 서버를 속일 수 있다 (예: 가짜 "결제 완료" 이벤트)
  • 요청이 중간에서 변조되어도 감지할 수 없다
  • 어떤 요청이 진짜 Zoom(서비스 제공자)에서 온 것인지 구분 불가

HMAC-SHA256과 일반 SHA-256의 차이:

  • SHA-256: 키 없이 동작. 누구나 같은 해시값을 만들 수 있음. 따라서 공격자가 변조된 메시지와 그 해시값을 같이 보내면 구분 불가.
  • HMAC-SHA256: 비밀키를 섞어서 해시를 만듦. 비밀키를 모르면 같은 HMAC 값을 만들 수 없음. 이게 핵심 차이. "누가 만들었는지" 증명 가능.

서명 검증 전체 과정:

  1. 사전 공유: 서비스 제공자(예: Zoom)와 우리 서버가 비밀키(Secret Token)를 미리 공유한다.
  2. 서명 생성 (발신측): Zoom이 Webhook을 보낼 때, 요청 body를 비밀키로 HMAC-SHA256 해싱하여 서명값을 만들고, x-zm-signature 같은 커스텀 헤더에 담아서 보낸다.
  3. 서명 검증 (수신측): 우리 서버가 요청을 받으면, (a) 받은 body와 우리가 가진 비밀키로 HMAC-SHA256을 직접 계산하고, (b) 계산한 값과 헤더에서 꺼낸 서명값을 비교한다.
  4. 판정: 일치하면 진짜 요청 (처리), 불일치하면 변조되었거나 가짜 요청 (거부).

채점 기준:

항목 배점
서명 검증 없을 때 위험성 설명 (가짜 요청, 변조 감지 불가) 1점
HMAC과 일반 해싱의 차이 (비밀키 유무) 1점
서명 검증 과정 3단계 이상 서술 1.5점
비밀키를 모르면 서명을 만들 수 없다는 근본 원리 언급 0.5점

Q30. 전체 흐름 통합 (서술형, 4점)¶

정답:

사용자가 https://lms.example.com/api/courses에 접속하여 강좌 목록을 조회하는 전체 과정:

1단계 - DNS 조회: 브라우저가 lms.example.com의 IP 주소를 알아내기 위해 DNS 서버에 질의한다. DNS 캐시(브라우저 -> OS -> 라우터 -> ISP DNS)를 순서대로 확인하고, 없으면 재귀적 DNS 조회를 통해 IP 주소(예: 203.0.113.50)를 얻는다.

2단계 - TCP 3-way handshake: 알아낸 IP 주소의 443번 포트(HTTPS 기본 포트)로 TCP 연결을 수립한다. 클라이언트가 SYN 보내고, 서버가 SYN-ACK 보내고, 클라이언트가 ACK 보낸다. 이제 신뢰성 있는 연결이 맺어졌다.

3단계 - TLS 핸드셰이크: HTTPS이므로 TCP 연결 위에서 TLS 핸드셰이크가 진행된다. Client Hello -> Server Hello -> 인증서 전송 및 검증 -> 키 교환 -> 대칭키 생성. 이후 모든 통신은 대칭키로 암호화된다.

4단계 - HTTP GET 요청: TLS로 암호화된 채널 위에서 HTTP 요청이 나간다.

GET /api/courses HTTP/1.1
Host: lms.example.com
Accept: application/json

5단계 - 서버 처리 및 응답: 서버가 요청을 처리하고 강좌 목록 데이터를 응답한다.

HTTP/1.1 200 OK
Content-Type: application/json

[{"id": 1, "title": "HTTP 완전정복"}, ...]

상태코드 200은 "요청 성공적으로 처리했어"를 의미하고, Content-Type: application/json은 응답 body가 JSON 형식임을 알려준다.

6단계 - 브라우저 렌더링: 브라우저가 응답 데이터를 받아서 화면에 표시한다.

전체 흐름: DNS -> TCP 3-way handshake -> TLS 핸드셰이크 -> HTTP GET 요청 -> 200 OK (Content-Type: application/json) -> 렌더링
