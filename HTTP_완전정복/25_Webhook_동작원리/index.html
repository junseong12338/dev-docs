25. Webhook 동작 원리¶

한 줄 요약

Webhook = 서버가 클라이언트에게 먼저 HTTP POST를 쏘는 것. 지금까지 배운 HTTP의 역전이야. 클라이언트가 요청하는 게 아니라 서버가 먼저 알려준다.

핵심 개념¶

HTTP는 원래 클라이언트가 먼저야¶

4장에서 배웠지? HTTP는 요청-응답 모델이야.

클라이언트 → "데이터 줘" → 서버
클라이언트 ← "여기 있어"  ← 서버

항상 클라이언트가 먼저 말을 걸어. 서버는 물어봐야 대답해. 서버가 먼저 "야, 새 소식 있어!" 이렇게 할 수 없어.

근데 실무에서 이런 상황이 있어:

  • Zoom 미팅이 끝나면 출결을 처리하고 싶다
  • GitHub에 커밋이 되면 자동으로 배포하고 싶다
  • 결제가 완료되면 주문 상태를 바꾸고 싶다

이걸 어떻게 해? 두 가지 방법이 있어.

Pull 방식 (Polling)¶

클라이언트가 계속 물어보는 방식¶

우리 서버: "미팅 끝났어?"
Zoom:      "아니."

(5초 후)
우리 서버: "미팅 끝났어?"
Zoom:      "아니."

(5초 후)
우리 서버: "미팅 끝났어?"
Zoom:      "아니."

(5초 후)
우리 서버: "미팅 끝났어?"
Zoom:      "응, 끝남."

우리 서버: "오케이, 출결 처리!"

이걸 Polling(폴링)이라고 해. 주기적으로 API를 호출해서 "변화 있어?" 확인하는 거야.

현실 비유: 택배 조회¶

택배 기다릴 때 조회 버튼 계속 누르는 거 있지?

  • 오전 10시: 조회 → "배송중"
  • 오전 11시: 조회 → "배송중"
  • 오후 12시: 조회 → "배송중"
  • 오후 1시: 조회 → "배달완료!"

10번 조회해서 9번은 의미 없는 요청이야. 리소스 낭비.

Polling의 문제점¶

문제 설명
리소스 낭비 변화 없어도 계속 요청 보냄. 서버 부담
실시간성 부족 폴링 간격(5초, 1분)만큼 지연 발생
API 호출 제한 대부분의 서비스에 Rate Limit이 있음. 너무 자주 물어보면 차단당해
비효율적 이벤트가 드물면 대부분의 요청이 "변화 없음"

Polling의 딜레마

간격을 짧게 하면? 리소스 낭비 + Rate Limit 위험. 간격을 길게 하면? 실시간성 포기. 어떻게 해도 최적이 아니야.

Push 방식 (Webhook)¶

서버가 먼저 알려주는 방식¶

우리 서버: "Zoom아, 미팅 끝나면 이 URL로 알려줘"
           (https://nexclass.com/webhook/zoom 등록)

(미팅 진행 중... 우리는 아무것도 안 함)

(미팅 종료!)
Zoom → POST https://nexclass.com/webhook/zoom
       Body: { "event": "meeting.ended", "meeting_id": "123" }

우리 서버: "오케이, 출결 처리!"

이게 Webhook이야. 이벤트가 발생하면 서버가 우리한테 HTTP POST를 쏴.

현실 비유: 택배 알림¶

스마트폰 택배 알림이랑 똑같아.

  • 조회 안 해도 됨
  • 배달 완료되면 알림이 와: "택배 도착했습니다"
  • 그때 가서 가져오면 됨

Webhook = "이벤트 발생하면 알림 보내줘" 를 HTTP로 구현한 거야.

Polling vs Webhook 비교¶

sequenceDiagram
    participant C as 우리 서버
    participant S as Zoom 서버

    rect rgb(255, 230, 230)
    Note over C,S: Polling 방식 (Pull)
    C->>S: GET /meeting/123/status
    S-->>C: {"status": "ongoing"}
    C->>S: GET /meeting/123/status
    S-->>C: {"status": "ongoing"}
    C->>S: GET /meeting/123/status
    S-->>C: {"status": "ongoing"}
    C->>S: GET /meeting/123/status
    S-->>C: {"status": "ended"}
    Note over C: 4번 요청, 3번은 의미 없음
    end

    rect rgb(230, 255, 230)
    Note over C,S: Webhook 방식 (Push)
    C->>S: POST /webhook (URL 등록, 1회)
    S-->>C: 200 OK (등록 완료)
    Note over C: 기다림... (리소스 0)
    S->>C: POST /webhook/zoom {"event":"meeting.ended"}
    C-->>S: 200 OK
    Note over C: 딱 1번, 필요할 때만
    end
항목 API 호출 (Polling) Webhook (Push)
방향 클라이언트 → 서버 서버 → 클라이언트
타이밍 내가 원할 때 (주기적) 이벤트 발생 시
실시간성 폴링 간격에 의존 거의 실시간
리소스 사용 폴링 비용 지속 발생 효율적 (필요할 때만)
신뢰성 내가 제어 가능 재시도 정책에 의존
복잡도 단순 (그냥 API 호출) 엔드포인트 구축 필요
적합한 상황 이벤트 빈도 높을 때 이벤트 빈도 낮을 때

Webhook 등록 과정¶

Webhook을 사용하려면 먼저 등록을 해야 해. 흐름은 이래:

sequenceDiagram
    participant N as 우리 (NexClass)
    participant Z as Zoom

    Note over N: 1. Callback URL 준비
    Note over N: https://nexclass.com/webhook/zoom

    N->>Z: "이 URL로 이벤트 알려줘" (Webhook 등록)
    Note over Z: 2. URL 검증 (CRC)
    Z->>N: "너 진짜 이 서버 주인 맞아?" (검증 요청)
    N-->>Z: "맞아, 증명할게" (검증 응답)
    Note over Z: 3. 검증 통과 → 구독 활성화

    Note over N,Z: (시간 경과... 미팅 진행...)

    Note over Z: 4. 이벤트 발생 (meeting.ended)
    Z->>N: POST /webhook/zoom (이벤트 데이터)
    N-->>Z: 200 OK (빠르게!)

각 단계 상세¶

1단계: Callback URL 준비

우리가 Zoom 이벤트를 받을 엔드포인트를 만들어. 공개 URL이어야 해.

https://nexclass.com/webhook/zoom

이 URL이 외부에서 접근 가능해야 Zoom이 요청을 보낼 수 있어.

2단계: URL 검증 (CRC)

Zoom이 "이 URL이 진짜 너 거 맞아?" 확인하는 과정이야. 26장에서 상세히 다룬다.

3단계: 구독 활성화

검증 통과하면 Zoom이 이벤트를 보내기 시작해.

4단계: 이벤트 수신

이벤트 발생하면 Zoom이 HTTP POST로 데이터를 보내. 이때 빠르게 200 OK를 돌려줘야 해.

타임아웃 주의

Zoom은 우리 응답을 3초 정도 기다려. 3초 안에 200 OK 안 보내면? Zoom은 "실패했나?" 하고 재시도해. 그래서 무거운 처리는 바로 하면 안 돼. 200 OK 먼저 보내고, 처리는 비동기로.

Webhook의 특징¶

1. HTTP POST를 사용한다¶

왜 POST야? GET이 아니라?

  • Webhook은 이벤트 데이터를 body에 담아서 보내
  • GET은 body가 없어 (6장에서 배웠지)
  • POST는 body에 데이터를 넣을 수 있으니까 POST가 적합
POST /webhook/zoom HTTP/1.1
Host: nexclass.com
Content-Type: application/json

{
  "event": "meeting.ended",
  "payload": {
    "meeting_id": "123456",
    "topic": "HTTP 스터디",
    "end_time": "2026-03-18T15:00:00Z"
  }
}

7장에서 배운 HTTP 요청 구조 그대로야. start line, header, body.

2. JSON body로 이벤트 데이터 전달¶

10장에서 배운 Content-Type: application/json 형태야. 이벤트 종류, 관련 데이터가 JSON으로 들어와.

3. 헤더에 메타데이터¶

서명, 타임스탬프 같은 메타 정보는 커스텀 헤더로 온다.

x-zm-signature: v0=abc123...
x-zm-request-timestamp: 1710765600

9장에서 배운 x- 접두사 커스텀 헤더야. 이게 보안의 핵심인데, 26장에서 자세히 다룬다.

4. 실패 시 재시도 (Retry)¶

우리가 200 OK를 안 보내면? Zoom은 다시 시도해.

1차 시도 → 타임아웃 → 실패
(잠시 후)
2차 시도 → 500 에러 → 실패
(더 기다린 후)
3차 시도 → 200 OK → 성공!

재시도 간격은 보통 지수 백오프(Exponential Backoff) 방식이야. 1초 → 2초 → 4초 → 8초... 점점 간격이 늘어나.

멱등성 다시 등장

재시도가 있으니까 같은 이벤트가 중복으로 올 수 있어. 6장에서 배운 멱등성 기억나? 같은 요청이 여러 번 와도 결과가 같아야 해. Webhook 핸들러도 멱등하게 만들어야 한다.

Webhook을 쓰는 서비스들¶

Webhook은 이제 업계 표준이야. 거의 모든 SaaS가 지원해.

서비스 Webhook 이벤트 예시 용도
Zoom meeting.ended, recording.completed 화상회의 이벤트
GitHub push, pull_request, issue CI/CD 트리거
Slack message, reaction_added 봇 연동
Stripe payment_intent.succeeded 결제 완료 알림
Notion page.created, page.updated 페이지 변경 감지

전부 같은 원리야:

  1. Callback URL 등록
  2. 이벤트 발생하면 HTTP POST
  3. JSON body로 데이터 전달
  4. 서명 헤더로 검증

주의사항과 함정¶

함정 1: "Webhook은 새로운 프로토콜이다"¶

틀렸어

Webhook은 새로운 기술이 아니야. 그냥 HTTP POST야. 지금까지 배운 것의 응용일 뿐. URL, 메서드, 헤더, body, 상태 코드. 전부 이미 배운 거야.

함정 2: "Webhook URL은 아무나 호출할 수 없다"¶

틀렸어

URL이 공개되어 있으면 누구나 POST를 보낼 수 있어. 그래서 서명 검증이 필수야. 26장에서 이걸 다룬다.

함정 3: "200 OK 보내기 전에 처리를 다 해야 한다"¶

틀렸어

타임아웃이 있어. 무거운 처리를 다 하고 200 보내면 타임아웃 돼. 200 OK를 먼저 보내고, 처리는 비동기(큐, 스레드)로 해.

함정 4: "Webhook은 한 번만 온다"¶

틀렸어

재시도가 있으니까 같은 이벤트가 여러 번 올 수 있어. 중복 처리 방지 로직이 필요해 (이벤트 ID로 구분).

정리¶

개념 핵심
Webhook 서버가 클라이언트에게 HTTP POST로 이벤트를 알려주는 방식
Polling 클라이언트가 주기적으로 API 호출. 비효율적
Webhook 등록 Callback URL 등록 → URL 검증 → 이벤트 수신
HTTP POST Webhook은 POST 사용. body에 이벤트 데이터(JSON)
서명 헤더 x-zm-signature 등 커스텀 헤더로 보안 검증
재시도 200 OK 안 보내면 재시도함. 멱등성 필요
빠른 응답 200 OK 먼저 보내고, 처리는 비동기로

한 문장 정리

Webhook은 새로운 게 아니야. HTTP POST + JSON + 커스텀 헤더. 지금까지 배운 것을 조합한 것이다. 다음 장에서 이 Webhook을 안전하게 받는 방법(서명 검증)을 배운다.

확인 문제¶

문제 1

Polling과 Webhook의 차이를 설명해봐. 각각 어떤 상황에서 적합한지도 말해.

정답 보기

Polling (Pull): 클라이언트가 주기적으로 서버에 "변화 있어?" 물어보는 방식. 이벤트 빈도가 높아서 거의 매번 데이터가 있을 때 적합. 단점: 리소스 낭비, 실시간성 부족, Rate Limit 위험.

Webhook (Push): 이벤트 발생 시 서버가 클라이언트에게 HTTP POST로 알려주는 방식. 이벤트 빈도가 낮고 실시간 반응이 필요할 때 적합. 단점: 엔드포인트 구축 필요, 보안 검증 필요, 재시도 정책에 의존.

문제 2

Webhook이 HTTP POST를 사용하는 이유가 뭐야? GET을 쓰면 안 되는 이유는?

정답 보기

Webhook은 이벤트 데이터를 body에 담아서 보내야 해. GET은 body가 없어 (6장). URL 쿼리 파라미터로 복잡한 JSON 데이터를 보내기엔 한계가 있고, 보안에도 취약해. POST는 body에 JSON을 넣을 수 있으니까 이벤트 데이터 전달에 적합하다.

문제 3

Webhook을 받을 때 "200 OK를 빨리 보내고 처리는 비동기로 해야 한다"고 했어. 왜 그래야 하는지, 안 그러면 어떻게 되는지 설명해봐.

정답 보기

Webhook 발신자(예: Zoom)는 응답을 일정 시간(보통 3~10초) 기다려. 그 안에 200 OK가 안 오면 타임아웃으로 실패 처리하고 재시도해.

출결 처리, DB 저장 같은 무거운 작업을 200 OK 보내기 전에 하면 처리 시간이 길어져서 타임아웃이 발생할 수 있어. 결과: 같은 이벤트가 재시도로 중복 수신됨.

올바른 방법: 200 OK를 즉시 응답하고, 비즈니스 로직은 메시지 큐나 비동기 스레드로 처리.

문제 4

Webhook 핸들러에 멱등성이 왜 필요한지, 6장에서 배운 멱등성 개념과 연결해서 설명해봐.

정답 보기

6장에서 배운 멱등성: 같은 요청을 여러 번 보내도 결과가 동일한 성질.

Webhook은 재시도 메커니즘이 있어서 같은 이벤트가 2번, 3번 올 수 있어. 예: meeting.ended 이벤트가 재시도로 2번 오면, 출결을 2번 처리하면 안 돼.

그래서 이벤트 ID를 기록해두고, 이미 처리한 이벤트가 다시 오면 무시해야 해. 이게 Webhook 핸들러의 멱등성이야.
