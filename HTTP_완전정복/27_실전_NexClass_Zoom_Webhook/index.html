27. 실전 - NexClass Zoom Webhook¶

한 줄 요약

지금까지 배운 HTTP 메서드, 헤더, body, 상태 코드, HMAC, 서명 검증 전부가 실제 프로젝트 코드 안에 들어있어. VirtualCampus의 동작하는 레퍼런스 코드를 기반으로 NexClass에 어떻게 적용하는지, 한 줄씩 해부한다.

Phase 4-B 작업 진행 중

NexClass의 WebhookController는 현재 스켈레톤 상태야 (Phase 4-B 대기 중). 이 챕터에서는 VirtualCampus의 동작하는 레퍼런스 코드를 먼저 해부하고, NexClass에 적용할 때 뭐가 달라지는지 설명한다.

항목 VirtualCampus (레퍼런스) NexClass (적용 대상)
상태 운영 중 (검증 완료) Phase 4-B 구현 대기
비동기 처리 RabbitMQ @Async
Zoom API ZoomWebhookEndpointService ZoomService (이미 완성)

시작하기 전에¶

이 챕터는 실제 프로젝트 코드를 해부하는 챕터야. VirtualCampus에서 이미 동작하고 있는 Webhook 코드를 분석하고, NexClass에 맞게 어떻게 변형하는지까지 다룬다.

전제 조건

  • 25장: Webhook 동작 원리 (Push vs Pull, 이벤트 구독)
  • 26장: Webhook 보안 (서명 검증, CRC, Replay Attack)
  • 22장: HMAC-SHA256
  • 이 세 챕터를 안 읽었으면 돌아가서 먼저 읽어.

프로젝트 파일 위치

이 챕터에서 다루는 실제 파일 경로:

NexClass (구현 대상)

  • Controller: NexClass/src/main/java/kr/ac/knu/nexclass/controller/WebhookController.java (스켈레톤)
  • Service: NexClass/src/main/java/kr/ac/knu/nexclass/service/ZoomService.java (완성)
  • Config: NexClass/src/main/resources/application.properties

VirtualCampus (레퍼런스)

  • Controller: ZoomWebhookEndpoint.java (lines 47-83)
  • Service: ZoomWebhookEndpointService.java (lines 37-62)

전체 흐름 먼저 보기¶

코드 뜯기 전에 전체 그림부터 잡자.

sequenceDiagram
    participant Z as Zoom
    participant C as WebhookController
    participant V as CRC 검증 (HMAC)
    participant S as ZoomService
    participant DB as Database

    Z->>C: POST /webhook/zoom (헤더 + body)
    C->>C: event 타입 확인

    alt CRC 검증 요청 (endpoint.url_validation)
        C->>V: plainToken + Secret Token으로 HMAC
        V-->>C: encryptedToken
        C-->>Z: 200 OK + {plainToken, encryptedToken}
    else meeting.ended
        C->>C: 서명 검증 (body + timestamp + signature)
        alt 서명 불일치
            C-->>Z: 401 Unauthorized
        else 서명 일치
            C-->>Z: 200 OK (즉시 응답)
            C->>S: @Async 출결 집계
            S->>Z: GET /report/meetings/{id}/participants (Zoom Reports API)
            Z-->>S: 참가자 목록
            S->>DB: 출결 기록 저장
        end
    else recording.completed
        C->>C: 서명 검증
        C-->>Z: 200 OK (즉시 응답)
        C->>S: @Async 녹화 등록
        S->>DB: 녹화 URL 저장
    end

프로젝트 구조 비교: VirtualCampus vs NexClass¶

코드로 들어가기 전에, 두 프로젝트의 아키텍처 차이부터 이해하자.

VirtualCampus (레퍼런스)              NexClass (적용)
========================              ================

ZoomWebhookEndpoint.java              WebhookController.java
    |                                     |
    v                                     v
ZoomWebhookEndpointService.java       ZoomService.java (이미 완성)
    |                                     |
    v                                     v
RabbitMQ (메시지 큐)                  @Async (비동기 메서드)
    |                                     |
    v                                     v
별도 Consumer에서 처리                 같은 Service에서 비동기 처리

왜 RabbitMQ에서 @Async로 바꿔?¶

기준 RabbitMQ @Async
복잡도 높음 (별도 인프라 필요) 낮음 (Spring 내장)
메시지 보존 서버 죽어도 큐에 남음 서버 죽으면 유실
스케일링 여러 Consumer 가능 단일 서버 한정
NexClass 규모 과도한 스펙 적정

NexClass는 단일 서버 운영이고 Webhook 트래픽이 크지 않아. RabbitMQ는 오버엔지니어링이야. @Async로 충분해.

1. VirtualCampus 레퍼런스 코드 해부 - CRC 검증¶

VirtualCampus에서 실제로 동작하고 있는 CRC 검증 코드야. ZoomWebhookEndpointService.java lines 37-62에서 가져왔어.

// VirtualCampus - ZoomWebhookEndpointService.java (lines 37-62)
// 이 코드는 실제 운영 중인 검증된 코드야

private static final String WEBHOOK_CHALLENGE_HASH_ALGORITHM_NAME = "HmacSHA256";

public Map<String, String> handleUrlValidation(String plainToken) throws Exception {
    final var encoding = StandardCharsets.UTF_8;
    final var hmacSha256 = Mac.getInstance(WEBHOOK_CHALLENGE_HASH_ALGORITHM_NAME);
    final var secretKey = new SecretKeySpec(
        zoomAppConfig.getSecretToken().getBytes(encoding),
        WEBHOOK_CHALLENGE_HASH_ALGORITHM_NAME);
    hmacSha256.init(secretKey);

    final var encryptedToken = Hex.encodeHexString(
        hmacSha256.doFinal(plainToken.getBytes(encoding)));

    return Map.of("plainToken", plainToken, "encryptedToken", encryptedToken);
}

한 줄씩 해부¶

상수 정의

private static final String WEBHOOK_CHALLENGE_HASH_ALGORITHM_NAME = "HmacSHA256";
  • 알고리즘 이름을 상수로 뽑아놨어 (22장 HMAC)
  • 문자열 직접 쓰면 오타 위험. 상수로 관리하는 게 맞아
  • private static final = 클래스 레벨 상수. 한 번만 생성되고 변경 불가

인코딩 명시

final var encoding = StandardCharsets.UTF_8;
  • UTF-8 인코딩 명시 (19장 인코딩)
  • var로 받았지만 타입은 Charset
  • 왜 이게 중요해? 같은 문자열이라도 인코딩이 다르면 바이트가 달라지고, HMAC 결과도 달라져

Mac 인스턴스 생성

final var hmacSha256 = Mac.getInstance(WEBHOOK_CHALLENGE_HASH_ALGORITHM_NAME);
  • Mac = Message Authentication Code (22장)
  • Mac.getInstance() = Java 보안 프레임워크에서 HMAC-SHA256 구현체를 가져옴

Secret Key 생성

final var secretKey = new SecretKeySpec(
    zoomAppConfig.getSecretToken().getBytes(encoding),
    WEBHOOK_CHALLENGE_HASH_ALGORITHM_NAME);
hmacSha256.init(secretKey);
  • zoomAppConfig.getSecretToken() = 설정 파일에서 Secret Token을 가져옴
  • .getBytes(encoding) = 문자열을 UTF-8 바이트 배열로 변환
  • SecretKeySpec = 바이트 배열을 키 객체로 감싸줌
  • hmacSha256.init(secretKey) = Mac 객체에 키 세팅

해싱 실행

final var encryptedToken = Hex.encodeHexString(
    hmacSha256.doFinal(plainToken.getBytes(encoding)));
  • plainToken.getBytes(encoding) = Zoom이 보낸 plainToken을 바이트로
  • hmacSha256.doFinal() = HMAC-SHA256 해싱 실행, 결과는 바이트 배열
  • Hex.encodeHexString() = 바이트를 16진수 문자열로 (19장 인코딩)
  • [0xA1, 0xB2] -> "a1b2" 이런 변환

응답 조합

return Map.of("plainToken", plainToken, "encryptedToken", encryptedToken);
  • Map.of() = Java 9+ 불변 맵 생성 (간결하고 좋아)
  • 26장에서 배운 CRC 응답 형식 그대로야:
    • plainToken = Zoom이 보낸 원본 토큰 (그대로 돌려줌)
    • encryptedToken = 우리가 HMAC으로 만든 해시값

2. VirtualCampus 레퍼런스 코드 해부 - 이벤트 라우팅¶

ZoomWebhookEndpoint.java lines 47-83에서 가져왔어.

// VirtualCampus - ZoomWebhookEndpoint.java (lines 47-83)
// 이벤트 타입별 분기 처리

@PostMapping("/zoom")
public ResponseEntity<?> handleZoomWebhook(@RequestBody String body) {

    JsonNode json = new ObjectMapper().readTree(body);
    String event = json.get("event").asText();

    switch (event) {
        case "endpoint.url_validation":
            String plainToken = json.get("payload").get("plainToken").asText();
            Map<String, String> crcResponse = webhookService.handleUrlValidation(plainToken);
            return ResponseEntity.ok(crcResponse);

        case "meeting.ended":
            // RabbitMQ로 비동기 전달
            rabbitTemplate.convertAndSend("zoom-webhook-queue", body);
            return ResponseEntity.ok().build();

        case "recording.completed":
            // RabbitMQ로 비동기 전달
            rabbitTemplate.convertAndSend("zoom-recording-queue", body);
            return ResponseEntity.ok().build();

        default:
            return ResponseEntity.ok().build();
    }
}

핵심 포인트¶

이벤트 라우팅 패턴

String event = json.get("event").asText();
  • body에서 event 필드를 꺼내서 분기 (25장 Webhook 이벤트 타입)
  • 이 한 줄로 "지금 뭔 일이 일어난 건지" 판단

CRC는 동기, 나머지는 비동기

case "endpoint.url_validation":
    // 동기 처리 - Zoom이 응답을 기다리고 있어
    return ResponseEntity.ok(crcResponse);

case "meeting.ended":
    // 비동기 - 큐에 넣고 즉시 200 OK
    rabbitTemplate.convertAndSend("zoom-webhook-queue", body);
    return ResponseEntity.ok().build();
  • CRC 검증은 반드시 동기. Zoom이 encryptedToken을 즉시 확인해야 하니까
  • meeting.ended는 비동기로 처리. 25장에서 "200 OK를 빨리 보내라"고 했지? 이게 그거야
  • VirtualCampus는 RabbitMQ를 쓰지만, NexClass는 @Async로 대체

3. NexClass 적용 - WebhookController 설계¶

이제 VirtualCampus 레퍼런스를 NexClass에 맞게 변형한다.

현재 상태

NexClass/src/main/java/kr/ac/knu/nexclass/controller/WebhookController.java는 현재 스켈레톤 상태야. Phase 4-B에서 아래 설계를 구현할 거다.

설정 - application.properties¶

# NexClass/src/main/resources/application.properties
# Zoom Webhook 설정

zoom.webhook.secret-token={ZOOM_WEBHOOK_SECRET_TOKEN}
  • zoom.webhook.secret-token = CRC 검증과 서명 검증에 사용할 Secret Token
  • @Value("${zoom.webhook.secret-token}")으로 주입

프로덕션에서는 환경변수로

현재는 개발 단계라 application.properties에 직접 쓰여있어. 프로덕션 배포 시에는 반드시 환경변수로 분리해야 해:

zoom.webhook.secret-token=${ZOOM_WEBHOOK_SECRET_TOKEN}

NexClass WebhookController 설계안¶

// NexClass - WebhookController.java (Phase 4-B 구현 예정)
// 경로: NexClass/src/main/java/kr/ac/knu/nexclass/controller/WebhookController.java

@RestController
@RequestMapping("/webhook")
public class WebhookController {

    @Value("${zoom.webhook.secret-token}")
    private String secretToken;

    @Autowired
    private ZoomService zoomService;  // 이미 완성된 Service

    @PostMapping("/zoom")
    public ResponseEntity<?> handleZoomWebhook(
            @RequestBody String body,
            @RequestHeader(value = "x-zm-signature", required = false) String signature,
            @RequestHeader(value = "x-zm-request-timestamp", required = false) String timestamp
    ) throws Exception {

        JsonNode json = new ObjectMapper().readTree(body);
        String event = json.get("event").asText();

        // 1. CRC 검증 (동기)
        if ("endpoint.url_validation".equals(event)) {
            String plainToken = json.get("payload").get("plainToken").asText();
            String encryptedToken = generateHmacSha256(plainToken);
            return ResponseEntity.ok(
                Map.of("plainToken", plainToken, "encryptedToken", encryptedToken));
        }

        // 2. 서명 검증 (일반 이벤트)
        if (!verifySignature(body, signature, timestamp)) {
            return ResponseEntity.status(401).body("Invalid signature");
        }

        // 3. 이벤트 분기 (비동기 처리)
        switch (event) {
            case "meeting.ended":
                zoomService.handleMeetingEndedAsync(json);  // @Async
                break;
            case "recording.completed":
                zoomService.handleRecordingCompletedAsync(json);  // @Async
                break;
        }

        return ResponseEntity.ok().build();  // 즉시 200 OK
    }
}

VirtualCampus와 NexClass 코드 비교¶

요소 VirtualCampus NexClass
@Value 키 zoomAppConfig.getSecretToken() ${zoom.webhook.secret-token}
CRC 검증 webhookService.handleUrlValidation() Controller 내부 generateHmacSha256()
meeting.ended rabbitTemplate.convertAndSend() zoomService.handleMeetingEndedAsync()
recording.completed rabbitTemplate.convertAndSend() zoomService.handleRecordingCompletedAsync()
서명 헤더 처리 별도 필터에서 처리 Controller에서 직접 검증

4. 한 줄씩 해부 - Controller 어노테이션¶

@RestController¶

@RestController
  • REST API 컨트롤러라는 뜻 (12장 REST 아키텍처)
  • @Controller + @ResponseBody 합친 거야
  • 반환값이 자동으로 JSON body가 됨 (10장 Content-Type)

@RequestMapping("/webhook")¶

@RequestMapping("/webhook")
  • 이 컨트롤러의 기본 경로 (5장 URL 해부)
  • 클래스 레벨에 붙이면 하위 모든 메서드의 경로 앞에 /webhook이 붙어

@Value("${zoom.webhook.secret-token}")¶

@Value("${zoom.webhook.secret-token}")
private String secretToken;
  • application.properties에서 Secret Token을 가져옴
  • NexClass 설정: zoom.webhook.secret-token={ZOOM_WEBHOOK_SECRET_TOKEN}
  • 26장에서 "코드에 하드코딩하지 마"라고 했지? 이게 그 방법

@PostMapping("/zoom")¶

@PostMapping("/zoom")
  • HTTP POST 메서드만 받겠다 (6장 HTTP 메서드)
  • 최종 경로: /webhook/zoom
  • 25장에서 "Webhook은 HTTP POST를 사용한다"고 했지? 그래서 @PostMapping

왜 GET이 아니라 POST야?

6장 복습: GET은 body가 없어. Webhook은 이벤트 데이터를 body에 담아서 보내야 하니까 POST.

@RequestBody String body¶

@RequestBody String body
  • HTTP 요청의 body를 문자열로 받음 (7장 요청 구조, 10장 Content-Type)
  • Zoom이 보내는 JSON body 전체가 이 변수에 들어옴
  • 왜 String으로 받아? 서명 검증할 때 원본 body 문자열이 필요하니까

이거 중요해

VO/DTO 객체로 바로 매핑하면 JSON 파싱 과정에서 문자열이 바뀔 수 있어. 서명은 원본 body 문자열 기준으로 계산되니까, 먼저 String으로 받아서 서명 검증하고, 검증 통과 후에 JSON 파싱해야 해.

@RequestHeader 헤더 추출¶

@RequestHeader(value = "x-zm-signature", required = false) String signature
@RequestHeader(value = "x-zm-request-timestamp", required = false) String timestamp
  • HTTP 헤더에서 x-zm-signature, x-zm-request-timestamp 값을 꺼냄 (9장 HTTP 헤더)
  • x- 접두사 = 커스텀 헤더 (9장)
  • 26장에서 배운 서명 값과 타임스탬프가 여기 들어있어
  • required = false = CRC 검증 요청에는 이 헤더들이 없을 수 있으니까

// NexClass - WebhookController.java (Phase 4-B)
private String generateHmacSha256(String plainToken) throws Exception {
    final var encoding = StandardCharsets.UTF_8;

    Mac hmacSha256 = Mac.getInstance("HmacSHA256");
    SecretKeySpec secretKey = new SecretKeySpec(
        secretToken.getBytes(encoding),   // @Value로 주입된 secretToken
        "HmacSHA256");
    hmacSha256.init(secretKey);

    return Hex.encodeHexString(
        hmacSha256.doFinal(plainToken.getBytes(encoding)));
}

6. 서명 검증 로직¶

// NexClass - 서명 검증 메서드
private boolean verifySignature(String body, String signature, String timestamp)
        throws Exception {
    // 1. 헤더 존재 확인
    if (signature == null || timestamp == null) {
        return false;
    }

    // 2. 타임스탬프 검증 (Replay Attack 방어)
    long requestTime = Long.parseLong(timestamp);
    long currentTime = System.currentTimeMillis() / 1000;
    if (Math.abs(currentTime - requestTime) > 300) {  // 5분 = 300초
        return false;
    }

    // 3. 서명 계산 및 비교
    String message = "v0:" + timestamp + ":" + body;
    String expectedSignature = "v0=" + generateHmacSha256ForSignature(message);

    return MessageDigest.isEqual(
        expectedSignature.getBytes(),
        signature.getBytes());
}

한 줄씩 해부¶

헤더 존재 확인

if (signature == null || timestamp == null) {
    return false;
}
  • 서명 헤더가 없으면 바로 거부 (8장: 401 Unauthorized)
  • 정상적인 Zoom 요청이면 이 헤더들이 반드시 있어야 해

타임스탬프 검증 (Replay Attack 방어)

long requestTime = Long.parseLong(timestamp);
long currentTime = System.currentTimeMillis() / 1000;
if (Math.abs(currentTime - requestTime) > 300) {
    return false;
}
  • 26장에서 배운 Replay Attack 방어
  • 요청 시간과 현재 시간 차이가 5분(300초) 초과면 거부
  • Math.abs() = 절대값. 미래 시간 조작도 방어

메시지 조합

String message = "v0:" + timestamp + ":" + body;
  • 26장에서 배운 그대로야
  • "v0:" + 타임스탬프 + ":" + 원본 body 합치기
  • 이게 HMAC 입력 메시지

HMAC-SHA256 계산

String expectedSignature = "v0=" + generateHmacSha256ForSignature(message);
  • Secret Token으로 message를 HMAC-SHA256 해싱 (22장)
  • "v0=" 접두사 붙이기 (Zoom의 서명 형식)
  • 이게 "우리가 계산한 서명"

서명 비교

return MessageDigest.isEqual(
    expectedSignature.getBytes(),
    signature.getBytes());
  • 우리가 계산한 서명 vs Zoom이 보낸 서명 비교
  • 일치하면 true -> 진짜 Zoom 요청
  • 불일치하면 false -> 위조! 401 거부

MessageDigest.isEqual() 쓰는 이유

26장에서 "서명 비교는 == 으로 하면 타이밍 공격에 취약하다"고 했지? MessageDigest.isEqual()은 상수 시간 비교를 해서 타이밍 공격을 방어해. String.equals() 대신 이걸 써야 해.

@Async
public void handleMeetingEndedAsync(JsonNode json) {

    // ...

NexClass ZoomService에 이미 있는 메서드들¶

ZoomService.java는 이미 완성되어 있어. Webhook 이벤트 처리에 필요한 API 호출 메서드가 전부 준비돼 있다:

메서드 용도 Webhook 이벤트
getAccessToken() OAuth 토큰 발급 모든 API 호출 전
getReportParticipants() 참가자 목록 조회 meeting.ended
getMeetingRecordings() 녹화 파일 조회 recording.completed
createMeeting() 미팅 생성 (Webhook 아님)
getMeeting() 미팅 조회 (Webhook 아님)
updateMeeting() 미팅 수정 (Webhook 아님)
deleteMeeting() 미팅 삭제 (Webhook 아님)

getReportParticipants()의 retry 로직

이 메서드에는 exponential backoff retry가 구현되어 있어. Zoom Reports API는 미팅 종료 직후에 데이터가 바로 준비 안 될 수 있거든. 1초 -> 2초 -> 4초 -> 8초 간격으로 재시도해서 데이터를 확실하게 가져와.

이게 25장에서 말한 "빠른 응답 + 비동기 처리"의 실전 버전이야. 200 OK를 먼저 보내고, @Async 안에서 retry 돌리니까 Zoom 쪽 타임아웃 걱정이 없어.

8. Spring Security 필터 제외 (필수)¶

이거 빠뜨리면 Webhook 자체가 안 돼

Webhook 엔드포인트는 Spring Security 필터에서 제외해야 해. 안 그러면 Zoom의 요청이 401/403으로 튕겨나가.

// SecurityConfig.java
@Configuration
@EnableWebSecurity
public class SecurityConfig {

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/webhook/**").permitAll()  // Webhook 경로 허용
                // ... 다른 보안 설정
            )
            .csrf(csrf -> csrf
                .ignoringRequestMatchers("/webhook/**")  // CSRF도 제외
            );
        return http.build();
    }
}

왜 제외해야 해?

보안 기능 왜 제외?
인증 필터 Zoom은 우리 시스템 계정이 없어. 세션/JWT 없음
CSRF 필터 Zoom은 CSRF 토큰을 보낼 수 없어
서명 검증 대신 Webhook 자체 서명 검증으로 보안 확보

Spring Security 필터를 제외하는 대신, 우리가 직접 서명 검증(6번 섹션)으로 보안을 확보하는 거야. 이건 23장 인증 방식에서 배운 "서버 간 인증"의 변형이야.

연결 챕터 총정리¶

이 코드 안에 지금까지 배운 개념이 전부 들어있어:

코드 요소 관련 챕터 개념
@PostMapping 6장 HTTP POST 메서드
/webhook/zoom 5장 URL 경로 구조
@RequestBody 7장, 10장 HTTP body, Content-Type
@RequestHeader("x-zm-signature") 9장 커스텀 헤더 (x- 접두사)

주의사항과 함정¶

함정 1: "body를 VO로 바로 받으면 편하잖아"¶

서명 검증이 깨져

@RequestBody ZoomEventVO vo 이렇게 받으면? Spring이 JSON 파싱하면서 공백, 순서가 바뀔 수 있어. 서명은 원본 body 문자열 기준이야. 문자 하나라도 다르면 HMAC 값이 달라져. 먼저 String으로 받아서 서명 검증 -> 통과 후 JSON 파싱.

함정 2: "getBytes() 할 때 인코딩 안 써도 되지"¶

환경마다 결과가 달라져

key.getBytes() = 시스템 기본 인코딩 사용. Windows는 MS949, Linux는 UTF-8일 수 있어. 같은 문자열인데 바이트가 다르면 HMAC 결과가 달라져. 반드시 StandardCharsets.UTF_8 명시해.

VirtualCampus 코드 보면 final var encoding = StandardCharsets.UTF_8;으로 맨 처음에 선언해놓고 전부 이걸 쓰잖아. 이게 정석이야.

함정 3: "Secret Token을 application.properties에 직접 쓰면 되지"¶

개발 단계에서는 OK, 프로덕션에서는 NO

NexClass의 현재 설정: 

zoom.webhook.secret-token={ZOOM_WEBHOOK_SECRET_TOKEN}
개발 단계라 이렇게 써놨지만, 프로덕션 배포 시에는:

zoom.webhook.secret-token=${ZOOM_WEBHOOK_SECRET_TOKEN}  # 환경변수 참조

서버 환경변수에 ZOOM_WEBHOOK_SECRET_TOKEN=실제값 설정. .gitignore에 properties 파일이 포함되어 있는지도 확인해야 해.

함정 4: "Spring Security 설정 안 건드려도 되겠지"¶

Webhook 요청이 전부 403으로 튕겨나가

Spring Security가 활성화되어 있으면 인증 안 된 요청은 전부 거부당해. Zoom은 우리 시스템 세션도 없고 JWT도 없어. /webhook/** 경로를 permitAll()과 CSRF 제외 설정 반드시 해야 해.

함정 5: "RabbitMQ 코드 그대로 복붙하면 되겠지"¶

NexClass에 RabbitMQ가 없어

VirtualCampus의 rabbitTemplate.convertAndSend() 코드를 그대로 가져오면 빈 주입 실패로 서버가 안 뜬다. NexClass는 @Async로 대체해.

// VirtualCampus (X - 복붙 금지)
rabbitTemplate.convertAndSend("zoom-webhook-queue", body);

// NexClass (O)
zoomService.handleMeetingEndedAsync(json);  // @Async

Phase 4-B 구현 체크리스트¶

NexClass WebhookController 구현 시 확인해야 할 항목:

단계 항목 상태
1 ZoomService.java 완성 (API 호출 메서드) 완료
2 application.properties에 secret-token 설정 완료
3 WebhookController.java 구현 (CRC + 서명 검증 + 이벤트 라우팅) 대기
4 Spring Security 필터 제외 설정 대기
5 @EnableAsync 설정 + ZoomService에 @Async 메서드 추가 대기
6 Zoom Marketplace에서 Webhook URL 등록 (CRC 검증 통과 확인) 대기
7 meeting.ended 이벤트로 출결 집계 테스트 대기
8 recording.completed 이벤트로 녹화 등록 테스트 대기

정리¶

개념 핵심
레퍼런스 VirtualCampus의 검증된 코드를 NexClass에 적용
핵심 변경 RabbitMQ -> @Async (NexClass 규모에 적합)
Controller @PostMapping으로 Webhook 수신. body는 String으로 받아
CRC 검증 endpoint.url_validation -> plainToken을 HMAC -> encryptedToken 응답
서명 검증 "v0:" + timestamp + ":" + body를 HMAC -> x-zm-signature와 비교
타임스탬프 현재 시간과 5분 이상 차이나면 거부
상수 시간 비교 MessageDigest.isEqual() 사용. 타이밍 공격 방어
이벤트 처리 meeting.ended -> 출결 (@Async), recording.completed -> 녹화 (@Async)
ZoomService 이미 완성. getReportParticipants() (retry 포함), getMeetingRecordings()
Security Webhook 경로는 Spring Security 필터에서 반드시 제외
현재 상태 Phase 4-B 대기. WebhookController 스켈레톤 상태

한 문장 정리

VirtualCampus의 검증된 HMAC/CRC 로직을 그대로 가져오고, RabbitMQ를 @Async로 바꾸고, 이미 완성된 ZoomService를 연결하면 NexClass Webhook이 완성된다. 이 코드 한 파일에 URL(5장), 메서드(6장), 헤더(9장), body(10장), 상태 코드(8장), HMAC(22장), 서명 검증(26장), CRC(26장) 전부 들어있어.

확인 문제¶

문제 1

@RequestBody String body로 받는 이유가 뭐야? VO 객체로 바로 받으면 안 되는 이유를 설명해봐.

정답 보기

서명 검증은 원본 body 문자열 기준으로 계산돼. VO 객체로 바로 매핑하면 Spring이 JSON을 파싱하면서 공백이나 필드 순서가 바뀔 수 있어. 원본 문자열과 1바이트라도 다르면 HMAC 결과가 달라져서 서명 검증이 실패해.

올바른 순서: String으로 받기 -> 서명 검증 -> 검증 통과 후 JSON 파싱.

문제 2

MessageDigest.isEqual() 대신 expectedSignature.equals(signature)를 쓰면 어떤 문제가 생겨?

정답 보기

타이밍 공격(Timing Attack)에 취약해져. String.equals()는 앞에서부터 한 글자씩 비교하다가 틀리면 바로 false를 반환해. 응답 시간 차이를 관찰하면 서명의 앞부분부터 한 글자씩 맞춰볼 수 있어.

예: "abc..."와 비교할 때, "a..."로 시작하면 2번째 글자까지 비교하니까 약간 더 느려. 이 시간 차이를 수천 번 반복하면 서명 전체를 알아낼 수 있어.

MessageDigest.isEqual()은 틀린 위치와 무관하게 항상 같은 시간이 걸려 (상수 시간 비교).

문제 3

이 코드에서 사용된 HTTP 관련 개념을 최소 6개 찾고, 각각 어느 챕터에서 배웠는지 연결해봐.

정답 보기
  1. HTTP POST (6장) - @PostMapping, Webhook은 POST 사용
  2. URL 경로 (5장) - /webhook/zoom 엔드포인트
  3. HTTP 헤더 (9장) - x-zm-signature, x-zm-request-timestamp 커스텀 헤더
  4. HTTP body (7장, 10장) - JSON body, Content-Type: application/json
  5. 상태 코드 (8장) - 200 OK, 401 Unauthorized
  6. HMAC-SHA256 (22장) - 서명 생성 및 검증
  7. 인코딩 (19장) - UTF-8 문자 인코딩, Hex 인코딩
  8. REST (12장) - @RestController, RESTful 엔드포인트

문제 4

CRC 검증 요청과 일반 이벤트 요청을 구분하는 기준이 뭐야? CRC 검증에서 서명 검증(x-zm-signature)을 하지 않는 이유는?

정답 보기

구분 기준: event 필드의 값. "endpoint.url_validation" -> CRC 검증 요청. "meeting.ended" 등 -> 일반 이벤트 요청.

CRC에서 서명 검증을 안 하는 이유: CRC 검증은 URL 등록 단계에서 발생해. 이 시점에 Zoom은 "이 서버가 Secret Token을 아는지" 확인하려는 거야. 서명 검증과 CRC 검증은 목적이 다름:

  • 서명 검증: "이 요청이 Zoom에서 왔는지" 확인
  • CRC 검증: "이 서버가 정당한 소유자인지" 확인 (Secret Token 보유 여부)

CRC 자체가 Secret Token을 통한 인증이니까, 별도의 서명 검증은 불필요해.

문제 5

VirtualCampus는 RabbitMQ를 쓰고, NexClass는 @Async를 쓴다. 각각의 장단점 3가지와, NexClass에서 @Async를 선택한 이유를 설명해봐.

정답 보기

RabbitMQ 장점:

  1. 서버가 죽어도 메시지가 큐에 보존됨 (메시지 영속성)
  2. 여러 Consumer로 수평 확장 가능 (스케일링)
  3. 메시지 재처리, 데드 레터 큐 등 실패 처리 정교

RabbitMQ 단점:

  1. 별도 인프라 설치/운영 필요 (복잡도 증가)
  2. 네트워크 hop 추가 (지연)
  3. 모니터링/관리 포인트 증가

@Async 장점:

  1. Spring 내장 기능이라 별도 인프라 불필요 (단순)
  2. 코드 레벨에서 비동기 처리 가능 (러닝커브 낮음)
  3. 같은 JVM 내에서 처리 (지연 없음)

@Async 단점:

  1. 서버 죽으면 처리 중인 작업 유실
  2. 단일 서버에서만 동작 (수평 확장 불가)
  3. 재처리 로직 직접 구현해야 함

NexClass에서 @Async를 선택한 이유: NexClass는 단일 서버 운영이고, Webhook 트래픽이 크지 않아. RabbitMQ는 이 규모에 과도한 인프라야. @Async로 충분하고, ZoomService의 getReportParticipants()에 이미 retry 로직이 있어서 실패 처리도 어느 정도 커버된다.
