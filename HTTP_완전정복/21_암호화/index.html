21. 암호화¶

한 줄 요약

암호화 = 데이터를 키로 잠그고 키로 여는 것. 키 없으면 못 열어. 이게 인코딩, 해싱과의 차이야.

핵심 개념¶

인코딩, 해싱과 뭐가 달라¶

19장(인코딩)과 20장(해싱)을 배웠으니, 이제 삼총사의 마지막 -- 암호화를 배울 차례야.

인코딩 (19장) 해싱 (20장) 암호화 (이번 장)
목적 형태 변환 무결성 검증 기밀성 보호
되돌리기 누구나 가능 불가능 키 가진 사람만 가능
키 불필요 불필요 필요
비유 번역 지문 채취 금고에 잠그기

암호화의 핵심: 허가된 사람(키 보유자)만 데이터를 읽을 수 있게 만드는 것.

대칭키 암호화 (AES)¶

하나의 키로 잠그고 여는 방식¶

대칭키(Symmetric Key) = 암호화할 때 쓰는 키와 복호화할 때 쓰는 키가 같다.

비유: 같은 열쇠로 잠그고 여는 금고.

sequenceDiagram
    participant A as 보내는 사람
    participant B as 받는 사람

    Note over A,B: 같은 비밀키를 공유하고 있음
    A->>A: 평문 + 비밀키 → 암호화
    A->>B: 암호문 전송
    B->>B: 암호문 + 같은 비밀키 → 복호화
    B->>B: 평문 획득

AES (Advanced Encryption Standard)¶

현재 가장 널리 쓰이는 대칭키 암호화 알고리즘.

항목 값
종류 AES-128, AES-192, AES-256
숫자 의미 키 길이 (비트). 클수록 강력
속도 매우 빠름
용도 파일 암호화, DB 암호화, HTTPS 데이터 전송 
AES-256 예시:

평문: "주민등록번호: 123456-7890123"
키:   "my_secret_key_32bytes_long_xxxxx"

암호화 결과: U2FsdGVkX1+3xK9mP7... (의미 없는 바이트열)

같은 키로 복호화: "주민등록번호: 123456-7890123"
다른 키로 복호화: 실패 (또는 쓰레기 데이터)

대칭키의 장점과 단점¶

장점 단점
빠르다. 비대칭키보다 수백~수천 배 키 배포 문제: 키를 어떻게 안전하게 전달하나?
대용량 데이터 암호화에 적합 키가 탈취되면 끝
구현이 상대적으로 단순 통신 상대마다 다른 키 필요 (N명이면 N개 키 관리)

핵심 문제: 키 배포

A가 B에게 암호화된 메시지를 보내려면, 먼저 비밀키를 공유해야 해. 근데 그 비밀키를 어떻게 안전하게 전달하지?

비밀키를 인터넷으로 보내면? 도청당하면 끝이야. 직접 만나서 전달? 전 세계 모든 서버와 만날 순 없잖아.

이 문제를 해결하는 게 비대칭키 암호화야.

비대칭키 암호화 (RSA)¶

두 개의 키: 공개키와 개인키¶

비대칭키(Asymmetric Key) = 암호화 키와 복호화 키가 다르다.

  • 공개키(Public Key): 누구에게나 공개. 암호화에 사용.
  • 개인키(Private Key): 나만 가지고 있음. 복호화에 사용.

비유: 우체통. 누구나 편지를 넣을 수 있지만(공개키로 암호화), 열쇠를 가진 사람만 꺼낼 수 있어(개인키로 복호화).

sequenceDiagram
    participant A as 보내는 사람
    participant B as 받는 사람

    Note over B: B가 공개키/개인키 쌍을 생성
    B->>A: 공개키 전달 (누구나 볼 수 있어도 OK)
    A->>A: 평문 + B의 공개키 → 암호화
    A->>B: 암호문 전송
    Note over B: B의 개인키만 복호화 가능
    B->>B: 암호문 + B의 개인키 → 복호화
    B->>B: 평문 획득

RSA¶

대표적인 비대칭키 암호화 알고리즘.

항목 값
키 길이 RSA-2048, RSA-4096
속도 느리다 (대칭키보다 수백~수천 배)
용도 키 교환, 디지털 서명, TLS 핸드셰이크

공개키로 암호화, 개인키로 복호화¶

B의 공개키: MIIBIjANBgkq... (누구나 볼 수 있음)
B의 개인키: MIIEvgIBADANBg... (B만 가지고 있음)

A가 보낼 메시지: "비밀 데이터"

A → B의 공개키로 암호화: kX7mP2bQ9v...
도청자: kX7mP2bQ9v... (가로챘지만 개인키 없어서 복호화 불가)
B → B의 개인키로 복호화: "비밀 데이터" ✅

개인키로 서명, 공개키로 검증¶

비대칭키는 반대 방향으로도 쓸 수 있어. 이게 디지털 서명이야.

A의 개인키: (A만 가지고 있음)
A의 공개키: (누구나 볼 수 있음)

A가 문서에 서명:
문서의 해시 + A의 개인키 → 서명값 생성

누구나 검증 가능:
서명값 + A의 공개키 → 해시 복원
문서를 다시 해싱 → 비교
일치하면: A가 서명한 거 맞음 + 문서 변조 안 됨

22장 연결

이 서명 원리가 디지털 서명과 HMAC의 기반이야. 22장에서 HMAC-SHA256과 디지털 서명의 차이를 자세히 다룬다.

비대칭키의 장점과 단점¶

장점 단점
키 배포 문제 해결: 공개키를 공개해도 안전 느리다: 대칭키보다 수백~수천 배
디지털 서명 가능 대용량 데이터 암호화에 부적합
부인 방지(Non-repudiation) 키 관리가 복잡

실전: 비대칭키로 대칭키를 교환한다¶

TLS가 이 방식이야 (17장 연결)¶

대칭키는 빠르지만 키 배포가 문제. 비대칭키는 키 배포는 해결했지만 느리다.

그래서 실전에서는 둘을 합쳐서 쓴다:

  1. 비대칭키로 대칭키를 안전하게 교환
  2. 이후 대칭키로 실제 데이터 통신

이게 바로 TLS(HTTPS)의 핵심 원리야.

sequenceDiagram
    participant C as 클라이언트 (브라우저)
    participant S as 서버

    Note over C,S: 1단계: 비대칭키로 대칭키 교환
    S->>C: 서버의 공개키 전달 (인증서에 포함)
    C->>C: 대칭키(세션키) 랜덤 생성
    C->>C: 대칭키를 서버 공개키로 암호화
    C->>S: 암호화된 대칭키 전송
    S->>S: 서버 개인키로 복호화 → 대칭키 획득

    Note over C,S: 2단계: 대칭키로 실제 통신 (빠름)
    C->>S: 대칭키로 암호화된 요청
    S-->>C: 대칭키로 암호화된 응답
    C->>S: 대칭키로 암호화된 요청
    S-->>C: 대칭키로 암호화된 응답

왜 이렇게 해?

  • 비대칭키만 쓰면? → 너무 느려. 매 요청마다 RSA 연산하면 서버 터져
  • 대칭키만 쓰면? → 키를 어떻게 전달해? 도청 당하면 끝
  • 둘 다 쓰면? → 키 교환은 안전하게(비대칭키), 통신은 빠르게(대칭키). 최적해.

17장(HTTPS와 TLS)에서 배운 TLS 핸드셰이크가 바로 이 과정이야.

인코딩 vs 해싱 vs 암호화 최종 비교¶

19장~21장의 핵심을 한 표로 정리. 이거 면접에서 물어본다.

항목 인코딩 해싱 암호화
목적 형태 변환 무결성 검증 기밀성 보호
되돌리기 가능 (누구나) 불가능 키 있으면 가능
키 필요 불필요 불필요 필요
대표 알고리즘 Base64, URL 인코딩 SHA-256, MD5 AES, RSA
비유 한국어→영어 번역 지문 채취 금고에 잠그기
보안 목적 아님 부분적 (무결성) 핵심 보안 기술
언제 쓰나 데이터 전송/저장 호환 비밀번호 저장, 서명 민감 데이터 보호

이거 섞어 쓰면 사고 난다

  • 비밀번호를 인코딩(Base64)으로 저장? → 3초 만에 원본 노출
  • 비밀번호를 암호화(AES)로 저장? → 키 탈취되면 전체 노출. 해싱이 맞음
  • 전송 데이터를 해싱으로 보호? → 원본 복원 불가. 암호화가 맞음

용도에 맞는 기술을 써야 해. 이걸 구분 못 하면 보안 사고의 원인이 된다.

주의사항 / 함정¶

삽질 포인트

1. 암호화 알고리즘 직접 만들지 마

자체 암호화 알고리즘은 100% 취약점이 있어. 수학자들이 수십 년 검증한 AES, RSA를 써. "우리만의 독자적 암호화"는 보안에서 최악의 선택이야.

2. 키 관리가 핵심이야

아무리 AES-256을 써도 키를 소스 코드에 하드코딩하면 끝이야. 키는 환경변수, 키 관리 서비스(AWS KMS 등)에 저장해야 해.

// ❌ 절대 이렇게 하지 마
String key = "my_secret_key_1234567890123456";

// ✅ 환경변수에서 로드
String key = System.getenv("ENCRYPTION_KEY");

3. ECB 모드 쓰지 마

AES에는 운영 모드가 있어. ECB(Electronic Codebook)는 같은 평문 블록이 항상 같은 암호문을 만들어서 패턴이 노출돼. CBC나 GCM 모드를 써야 해.

정리¶

개념 핵심
암호화 키로 잠그고 키로 여는 것. 기밀성 보호
대칭키 (AES) 같은 키로 암호화/복호화. 빠르지만 키 배포 문제
비대칭키 (RSA) 공개키/개인키 쌍. 키 배포 해결했지만 느림
TLS 방식 비대칭키로 대칭키 교환 → 대칭키로 통신. 둘의 장점 결합
핵심 비교 인코딩(형태 변환) vs 해싱(무결성) vs 암호화(기밀성)

한 문장 정리

암호화는 키를 가진 사람만 데이터를 읽을 수 있게 만드는 기술이야. 대칭키(빠름)와 비대칭키(안전한 키교환)를 합쳐 쓰는 게 실전이고, TLS가 그 대표야.

확인 문제¶

문제 1

대칭키 암호화와 비대칭키 암호화의 차이를 설명하고, 실전에서 왜 둘을 합쳐서 쓰는지 말해봐.

정답 보기

대칭키: 하나의 키로 암호화/복호화. 빠르지만 키를 어떻게 전달하나(키 배포 문제). 비대칭키: 공개키로 암호화, 개인키로 복호화. 키 배포 해결했지만 느림.

둘을 합쳐 쓰는 이유: 비대칭키로 대칭키를 안전하게 교환하고(느리지만 한 번만), 이후 대칭키로 실제 데이터를 통신한다(빠름). 이렇게 하면 보안(비대칭키)과 성능(대칭키) 둘 다 잡을 수 있어. TLS(HTTPS)가 바로 이 방식이야.

문제 2

비대칭키에서 "공개키로 암호화, 개인키로 복호화"와 "개인키로 서명, 공개키로 검증"의 차이를 설명해봐.

정답 보기

공개키로 암호화, 개인키로 복호화 = 기밀성 보호

  • 목적: 특정 사람만 데이터를 읽을 수 있게 하려고
  • 누구나 공개키로 암호화할 수 있지만, 개인키 가진 사람만 복호화 가능

개인키로 서명, 공개키로 검증 = 인증 + 무결성

  • 목적: "이 메시지를 내가 보냈고, 변조되지 않았다"를 증명하려고
  • 개인키 가진 사람만 서명할 수 있고, 누구나 공개키로 검증 가능

방향이 반대야. 기밀성이냐, 인증이냐에 따라 쓰는 방향이 달라.

문제 3

인코딩, 해싱, 암호화 중에서 비밀번호 저장에 적합한 것과 그 이유를 말해봐.

정답 보기

해싱이 적합하다.

  • 인코딩 (Base64 등): 누구나 디코딩 가능. DB 털리면 비밀번호 즉시 노출. 절대 안 됨.
  • 암호화 (AES 등): 키가 탈취되면 모든 비밀번호 한 번에 노출. 키 관리 리스크 큼.
  • 해싱 (SHA-256 + Salt 등): 원본 복원 자체가 불가능. DB가 털려도 비밀번호를 알 수 없음.

서버도 원본 비밀번호를 모르는 게 가장 안전한 설계야. 로그인 시에는 입력값을 해싱해서 저장된 해시와 비교하면 된다.
