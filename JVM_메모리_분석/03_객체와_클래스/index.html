03. 객체와 클래스¶

이 장의 목표: "Number of objects 5,231,251"과 "Number of classes 27,688"이 뭔지 이해한다

1. 클래스(Class)란 무엇인가¶

1.1 비유: 설계도와 건물¶

클래스 = 설계도 (붕어빵 틀)
객체   = 설계도로 만든 실제 건물 (붕어빵)

붕어빵 틀은 1개인데, 붕어빵은 100개 찍을 수 있다.

1.2 코드로 보기¶

// ====== 이게 클래스 (설계도) ======
public class User {
    String name;
    int age;

    public User(String name, int age) {
        this.name = name;
        this.age = age;
    }
}

// ====== 이게 객체 (인스턴스) ======
User user1 = new User("이준성", 25);   // 객체 1개 생성
User user2 = new User("김철수", 30);   // 객체 1개 더 생성
User user3 = new User("박영희", 28);   // 객체 1개 더 생성

// 클래스: 1개 (User)
// 객체:   3개 (user1, user2, user3)

1.3 메모리에서 보면¶

┌────── Metaspace ──────┐     ┌────────── Heap ──────────┐
│                       │     │                          │
│  User.class 설계도    │     │  user1 객체              │
│  ├ name: String       │     │  ├ name: "이준성"        │
│  ├ age: int           │     │  └ age: 25               │
│  └ 메서드 정보        │     │                          │
│                       │     │  user2 객체              │
│  (클래스는 1개)       │     │  ├ name: "김철수"        │
│                       │     │  └ age: 30               │
│                       │     │                          │
│                       │     │  user3 객체              │
│                       │     │  ├ name: "박영희"        │
│                       │     │  └ age: 28               │
│                       │     │                          │
│                       │     │  (객체는 3개)            │
└───────────────────────┘     └──────────────────────────┘

2. 우리 서버의 숫자¶

Number of classes:  27,688개
Number of objects:  5,231,251개

2.1 클래스 27,688개 — 이게 다 뭐야?¶

네가 만든 클래스만 있는 게 아니다. JVM에 로드된 모든 클래스가 포함된다.

클래스 27,688개의 구성

  1. Java 기본 클래스 (~수백 개) -- String, Integer, ArrayList, HashMap, Thread ...
  2. 프레임워크 클래스 (~수천 개) -- Spring: ApplicationContext, BeanFactory ... / MyBatis: SqlSession, SqlSessionFactory ... / eGovFramework: 각종 공통 클래스 ... / Tomcat: Servlet, Request, Response ...
  3. 라이브러리 클래스 (~수천 개) -- Jackson(JSON), Log4j2(로깅), Commons ...
  4. 우리가 만든 클래스 (~수백~수천 개) -- Controller, Service, VO, Mapper ...
  5. 동적 생성 클래스 (~수천 개) -- 프록시, 리플렉션, CGLIB ... / DelegatingClassLoader가 만든 것들!
  6. APM 에이전트 클래스 -- WhaTap, EXEM InterMax ...

2.2 정상 범위는?¶

일반적인 Spring 웹 애플리케이션:
  클래스 수: 10,000 ~ 20,000개 정도

우리 서버: 27,688개
  → 다소 많음
  → 왜? 클래스 중복 로드 (11,532개 중복) + 동적 생성 클래스 누적

2.3 객체 5,231,251개 — 이게 다 뭐야?¶

JVM 힙에 살아있는 모든 것이다.

523만 개 객체 = 힙 덤프 시점에 GC가 아직 안 치운 모든 것

포함되는 것들

  1. 문자열 (String, char[]) -- 가장 많음. SQL 쿼리문, URL, 설정값 등. "SELECT * FROM ..." 이런 것들이 전부 객체
  2. 컬렉션 내부 배열 (Object[], Node[]) -- HashMap 안의 버킷 배열, ArrayList 안의 내부 배열
  3. 우리 코드의 VO/DTO 객체 -- UserVO, ExamVO, CourseVO ...
  4. 프레임워크 내부 객체 -- Bean 정의, 핸들러 매핑, 인터셉터 ...
  5. 세션 객체 -- 접속 중인 사용자의 세션 데이터. MAT에서 StandardSession 2,989개 확인
  6. 래퍼 객체 -- Integer, Long, Boolean 등. int 10은 스택이지만 new Integer(10)은 힙
  7. 리플렉션/프록시 관련 객체 -- Method, Class, WeakReference ... 이게 누수의 주범

3. 객체 하나의 크기¶

3.1 Shallow Heap vs Retained Heap¶

이 개념은 MAT 분석에서 핵심이다.

Shallow Heap (얕은 힙):
  → 객체 자신만의 크기
  → 참조하는 다른 객체는 포함하지 않음

Retained Heap (유지 힙):
  → 이 객체가 GC되면 함께 해제되는 총 메모리
  → 이 객체만 참조하는 모든 하위 객체 포함

예시:
  User 객체
  ├ name → String "이준성" (40 byte)
  └ age = 25

  User의 Shallow Heap:  24 byte (객체 헤더 + 필드 참조)
  User의 Retained Heap: 64 byte (User 자신 + "이준성" String)

  만약 "이준성"을 다른 객체도 참조하고 있다면?
  → Retained Heap에 포함 안 됨 (User가 GC돼도 "이준성"은 살아남으니까)

비유

  • Shallow Heap = 내 몸무게
  • Retained Heap = 나 없으면 같이 사라지는 것들의 총 무게

예: 내가 사라지면

  • 내 전용 방도 비워짐
  • 나만 쓰던 물건도 버려짐
  • BUT 공용 화장실은 안 사라짐 (다른 사람도 쓰니까)

3.2 객체의 메모리 구성¶

Java 객체 하나는 최소한 이만큼 먹는다:

64-bit JVM 기준 (Compressed OOP 켜진 경우):

영역 크기 설명
Object Header 12 byte
- Mark Word 8 byte GC 정보, 해시코드, 락 상태
- Class Pointer 4 byte 이 객체가 어떤 클래스인지
Instance Data 가변 필드(변수)들의 실제 값. int: 4 byte, long: 8 byte, 참조: 4 byte (Compressed OOP)
Padding (패딩) 가변 8의 배수로 맞추기 위한 빈 공간. JVM은 8 byte 단위로 메모리 정렬 
예시: 빈 객체 (필드 없음)
  class Empty {}

  헤더: 12 byte
  데이터: 0 byte
  패딩: 4 byte (12 → 16으로 맞춤)
  합계: 16 byte

  → 아무것도 없는 객체도 16 byte를 먹는다!
  → 523만 개 x 최소 16 byte = 최소 83 MB

4. 우리 서버의 실제 MAT 데이터¶

MAT에서 확인된 상위 객체들:

클래스 객체 수 메모리
char[] 151,409 20.7 MB
String 150,538 3.6 MB
java.lang.ref.WeakReference 54,008 1.7 MB
DateTimeHelper$Day (WhaTap) 36,525 1.5 MB
java.lang.reflect.Method 23,512 2.1 MB
ConcurrentHashMap$Node 15,284 0.5 MB
java.lang.Class 14,458 60.1 MB
String[] 12,513 2.9 MB
HashMap$Node 12,288 0.4 MB
java.beans.MethodDescriptor 11,328 0.6 MB
StandardSession (Tomcat 세션) 2,989 0.3 MB

여기서 읽어야 할 것:

1. char[] + String이 가장 많다
   → 자바 8에서 String 내부는 char[] 배열
   → 문자열이 메모리의 상당 부분 차지 (정상)

2. java.lang.Class 14,458개가 60.1 MB
   → 클래스 메타데이터가 힙에도 참조로 존재
   → 전체의 17.74% 차지
   → 이게 MAT이 "Problem Suspect 1"로 찍은 것

3. DateTimeHelper$Day 36,525개
   → WhaTap APM이 100년치 날짜 데이터를 메모리에 올림
   → 36,525 = 365일 x 100년 + 윤년
   → 불필요한 메모리 사용

4. StandardSession 2,989개
   → 힙 덤프 시점에 접속 중인 세션
   → 약 3,000명이 세션을 갖고 있었다는 뜻

5. WeakReference 54,008개
   → 약한 참조 (GC가 수거 가능한 참조)
   → 리플렉션 캐시, 클래스로더 관련

5. 클래스 27,688개 vs 객체 523만 개¶

비율: 1개 클래스당 평균 189개 객체

하지만 분포는 균등하지 않다:

  String 클래스 → 150,538개 객체 (혼자 15만 개)
  char[] 클래스 → 151,409개 객체
  User 같은 VO → 수백~수천 개
  싱글톤 Bean → 1개

클래스가 많다 = 설계도가 많다 = 메타스페이스를 많이 쓴다
객체가 많다  = 실체가 많다  = 힙을 많이 쓴다

클래스는 한 번 로드되면 보통 안 사라짐
객체는 GC가 수시로 생성/제거함

클래스 27,688개 → 다소 많음 (중복 로드 의심)
객체 523만 개   → 힙 323MB면 적절 (객체당 평균 62 byte)

6. 핵심 정리¶

  • 설계도, 붕어빵 틀
  • Metaspace에 저장
  • 27,688개 = 다소 많음 (중복 로드 + 동적 생성)

객체 (Object):

  • 설계도로 만든 실체, 붕어빵
  • Heap에 저장
  • 5,231,251개 = 힙 323MB 대비 정상 범위

Shallow Heap:

  • 객체 자신의 크기만

Retained Heap:

  • 객체 + 이 객체만 참조하는 하위 객체 총합
  • MAT 분석에서 진짜 중요한 지표
