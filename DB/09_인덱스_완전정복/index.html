09. 인덱스 완전정복 -- 2.7억 건을 수 밀리초에 찾는 마법¶

1. 인덱스란?¶

1.1 비유: 책의 색인¶

비유: 책의 색인

1000페이지짜리 데이터베이스 교재에서 "트랜잭션"을 찾아야 한다.

방법 동작 시간
A: 풀스캔 1~1000페이지 한 장씩 넘기며 찾기 오래 걸림
B: 색인 "트랜잭션 ... p.42, p.157, p.891" → 바로 점프 수 초

DB의 인덱스가 정확히 이것이다. 컬럼 값 → 해당 행의 위치를 빠르게 찾는 자료구조.

1.2 DB에서의 인덱스¶

조건 동작 소요 시간
인덱스 없음 2.7억 건 전부 확인 27분
인덱스 있음 인덱스에서 위치 찾고 바로 점프 수 밀리초

2. B-Tree 인덱스 (가장 기본, 가장 중요)¶

2.1 B-Tree 구조¶

B-Tree = Balanced Tree (균형 트리)

왜 Balanced인가?

  • 모든 리프 노드의 깊이가 같다
  • 어떤 값을 찾든 동일한 횟수의 비교로 도달한다
  • "운 좋으면 빨리 찾고, 운 나쁘면 느리고" 같은 게 없다

구조 시각화:

                        [루트 노드]
                     [100 | 500 | 900]
                    /       |        \
               /            |            \
      [브랜치 노드]   [브랜치 노드]   [브랜치 노드]
      [20|50|80]      [200|350|450]   [600|750|850]
      / | | \          / | | \          / | | \
     /  | |  \        /  | |  \        /  | |  \
    리프 리프 리프    리프 리프 리프    리프 리프 리프
    [1..19][20..49]   [100..199]       [500..599]
    ↓      ↓          ↓                ↓
   실제   실제        실제             실제
   데이터  데이터     데이터            데이터
   위치    위치       위치              위치


  루트(Root):    트리의 최상단. 검색 시작점.
  브랜치(Branch): 중간 경로. 방향을 안내한다.
  리프(Leaf):    최하단. 실제 데이터 위치를 가리킨다.

2.7억 건이어도 빠른 이유:

트리 깊이 탐색 가능한 최대 행 수 (대략)
깊이 1 수백 건
깊이 2 수만 건
깊이 3 수백만 건
깊이 4 수억 건
깊이 5 수백억 건

인덱스의 위력

2.7억 건 → 트리 깊이 4~5 → 4~5번 비교만으로 원하는 행을 찾는다. 풀스캔: 2.7억 번 비교 vs 인덱스: 4~5번 비교. 이게 인덱스의 위력이다.

2.2 동작 원리¶

검색 (SELECT):

검색: 값 350 찾기

  1. 루트 [100 | 500 | 900] → 350은 100~500 사이 → 가운데 브랜치로
  2. 브랜치 [200 | 350 | 450] → 350 발견! 리프로
  3. 리프에서 실제 데이터 위치 → 디스크 해당 위치로 점프

총 비교 횟수: 3회 (풀스캔: 2.7억 회 vs 인덱스: 4~5회)

삽입 (INSERT):

삽입 (INSERT) 시 인덱스 동작

  1. 삽입할 위치를 검색으로 찾는다 (루트 → 브랜치 → 리프)
  2. 적절한 리프 노드에 값을 삽입 (빈자리가 있으면 그냥 넣음)
  3. 리프가 가득 차면? → 노드 분할(Split) 발생 → 부모도 가득 차면 연쇄 분할

이것이 INSERT 시 인덱스 유지 비용이다. 인덱스가 많을수록 INSERT마다 해야 할 작업이 늘어난다.

삭제 (DELETE):

삭제 (DELETE) 시 인덱스 동작

값 350 삭제

  1. 해당 리프에서 350을 "삭제됨"으로 마킹 → 실제로 지우는 게 아니라 마킹만 한다
  2. 나중에 정리(Merge) 작업이 필요할 수 있다 → 빈 노드가 많아지면 트리 재구성

DELETE가 많은 테이블의 인덱스는 시간이 지나면 비효율적이 된다. OPTIMIZE TABLE로 재구성 필요 (10장에서 자세히).

3. 인덱스 있을 때 vs 없을 때¶

3.1 성능 비교¶

연산 인덱스 없음 인덱스 있음
COUNT(*) 27분 수 초 (커버링 인덱스)
WHERE id = ? 27분 (최악) 수 밀리초
WHERE date > ? 27분 수 초 (범위 탐색)
ORDER BY col 정렬 추가 비용 이미 정렬됨
JOIN ON a.id = b.id 양쪽 풀스캔 한쪽은 인덱스 탐색

왜 이렇게 차이나는가

풀스캔 = O(n) → n이 2.7억이면 2.7억 번 읽기. 인덱스 = O(log n) → log(2.7억) = 28 (이진 탐색 기준) → B-Tree는 팬아웃이 크므로 실제로 4~5번. 27분 vs 수 밀리초. 이게 인덱스의 힘이다.

3.2 실행 계획 (EXPLAIN)¶

EXPLAIN이란?

"인덱스를 타는지 안 타는지" 어떻게 확인하는가? EXPLAIN을 쓴다. 쿼리 앞에 EXPLAIN만 붙이면 된다. 실제로 실행하지 않고, "이렇게 실행할 계획이다"를 보여준다.

기본 사용법:

EXPLAIN SELECT * FROM tb_lms_exam_stare_paper_backup WHERE exam_cd = 'EXAM001';

결과의 핵심 컬럼:

컬럼 의미
type 접근 방식 (가장 중요!)
key 실제 사용된 인덱스 이름 (NULL이면 인덱스 안 씀)
rows 검사할 예상 행 수 (적을수록 좋음)
Extra 추가 정보

type 컬럼 상세 (좋은 순서대로):

type 의미 성능 예시
system 1건짜리 테이블 최고 (거의 안 나옴)
const PK/UNIQUE로 1건 조회 최고 WHERE id = 1
eq_ref JOIN에서 PK/UNIQUE 매칭 매우좋음 JOIN ON a.id = b.id
ref 인덱스로 여러 건 조회 좋음 WHERE status = 'A'
range 인덱스 범위 탐색 양호 WHERE date > '...'
index 인덱스 풀스캔 나쁨 인덱스 전체를 읽음
ALL 테이블 풀스캔 최악 인덱스 안 탐

주의

type이 ALL이면 인덱스를 안 타고 있다는 뜻이다. 2.7억 건에 ALL이면 27분이다.

key 컬럼:

EXPLAIN SELECT * FROM tb_user WHERE user_id = 'USR001';

-- key: PRIMARY   → PK 인덱스를 사용했다
-- key: idx_name  → idx_name 인덱스를 사용했다
-- key: NULL      → 인덱스를 안 썼다 (풀스캔)

rows 컬럼:

rows 컬럼 해석

rows = DB가 이 쿼리를 위해 검사할 것으로 예상하는 행 수

rows 값 의미
1 1건만 보면 된다 (최고)
100 100건 검사 (양호)
270000000 2.7억 건 전부 (최악 = 풀스캔)

rows가 전체 행 수에 가까우면 인덱스가 없거나 안 타고 있는 것이다.

Extra 컬럼 주요 값:

Extra 컬럼 주요 값

Extra 값 의미
Using index 인덱스만으로 결과 반환 (커버링 인덱스, 최고)
Using where WHERE 필터링 적용
Using filesort 정렬을 위해 추가 작업 필요 (느릴 수 있음)
Using temporary 임시 테이블 사용 (느릴 수 있음)
Using index condition 인덱스 조건 푸시다운 (ICP)

주의

Using filesort + Using temporary가 같이 나오면 주의. 대량 데이터에서 정렬 + 임시 테이블 = 매우 느림.

실전 예시:

-- 인덱스 없는 상태
EXPLAIN SELECT COUNT(*) FROM tb_lms_exam_stare_paper_backup;

-- type: ALL
-- key: NULL
-- rows: 270000000
-- → 풀스캔. 2.7억 건 전부 읽는다. 27분.

-- 인덱스 있는 상태 (가정)
EXPLAIN SELECT COUNT(*) FROM tb_lms_exam_stare_paper_backup;

-- type: index
-- key: idx_exam_cd
-- rows: 270000000
-- Extra: Using index
-- → 인덱스 풀스캔이지만 데이터 페이지 안 읽음. 훨씬 빠름.

4. 인덱스 종류¶

4.1 단일 컬럼 인덱스¶

-- 하나의 컬럼에 인덱스
CREATE INDEX idx_exam_cd ON tb_exam (exam_cd);

-- exam_cd로 검색할 때 효과적
SELECT * FROM tb_exam WHERE exam_cd = 'EXAM001';  -- 인덱스 탐

4.2 복합 인덱스 (Composite Index)¶

-- 여러 컬럼을 조합한 인덱스
CREATE INDEX idx_exam_user ON tb_exam (exam_cd, user_no);

순서가 중요하다:

복합 인덱스의 최좌선 원칙 (Leftmost Prefix Rule)

인덱스: (exam_cd, user_no, reg_date)

이 인덱스를 탈 수 있는 쿼리:

  • WHERE exam_cd = 'E001' (첫 번째 컬럼)
  • WHERE exam_cd = 'E001' AND user_no = 'U001' (첫 번째 + 두 번째)
  • WHERE exam_cd = 'E001' AND user_no = 'U001' AND reg_date > '2024-01-01' (전부)

이 인덱스를 탈 수 없는 쿼리:

  • WHERE user_no = 'U001' (두 번째만)
  • WHERE reg_date > '2024-01-01' (세 번째만)
  • WHERE user_no = 'U001' AND reg_date > '2024-01-01' (두+세 번째)

왜? 전화번호부가 "성 - 이름 - 생년월일" 순으로 정렬되어 있다고 치자. "김" 씨를 찾는 건 쉽다 (첫 번째 기준). "김철수"를 찾는 것도 쉽다 (첫+두 번째 기준). 하지만 "이름이 철수"인 사람을 찾으려면? 전부 뒤져야 한다. 첫 번째 기준을 건너뛰면 정렬 순서를 활용할 수 없다. 이것이 최좌선 원칙이다. 인덱스의 왼쪽부터 순서대로 사용해야 한다.

(A, B)와 (B, A)는 완전히 다르다

인덱스 정렬 순서
(exam_cd, user_no) exam_cd로 먼저 정렬, 같은 exam_cd 안에서 user_no 정렬
(user_no, exam_cd) user_no로 먼저 정렬, 같은 user_no 안에서 exam_cd 정렬
  • WHERE exam_cd = ? → (exam_cd, user_no)가 탄다
  • WHERE user_no = ? → (user_no, exam_cd)가 탄다

순서를 잘못 만들면 인덱스가 있어도 안 탄다.

4.3 UNIQUE 인덱스¶

CREATE UNIQUE INDEX idx_email ON tb_user (email);

-- 해당 컬럼에 중복 값을 허용하지 않는다.
-- 인덱스 + 제약조건의 역할을 동시에 한다.
-- 검색 성능: 일반 인덱스와 동일
-- 추가 기능: INSERT/UPDATE 시 중복 체크

4.4 PRIMARY KEY (클러스터드 인덱스)¶

InnoDB에서 PRIMARY KEY는 특별하다

일반 인덱스 (세컨더리 인덱스):

  • 인덱스는 인덱스대로 정렬, 데이터는 데이터대로 저장
  • 인덱스에서 위치 찾고 → 데이터로 점프 (2단계)

PRIMARY KEY (클러스터드 인덱스):

  • 데이터 자체가 PK 순서대로 물리적으로 정렬되어 저장
  • PK로 찾으면 바로 데이터 (1단계)
  • 그래서 PK 조회가 가장 빠르다

세컨더리 인덱스 vs 클러스터드 인덱스

세컨더리 인덱스: 인덱스 순서와 데이터 순서가 다르다. 인덱스에서 위치 찾고 → 데이터 페이지로 점프 (2단계).

클러스터드 인덱스 (PK): 데이터 자체가 PK 순서대로 정렬. 찾으면 바로 데이터. 점프 불필요 (1단계).

InnoDB에서 PK가 없으면? UNIQUE NOT NULL 컬럼을 PK 대신 사용. 그것도 없으면 내부적으로 숨겨진 6바이트 Row ID 생성. 어떻게든 클러스터드 인덱스를 만든다.

4.5 FULLTEXT 인덱스¶

-- 전문 검색용 인덱스
CREATE FULLTEXT INDEX idx_content ON tb_board (content);

-- 사용법
SELECT * FROM tb_board WHERE MATCH(content) AGAINST('검색어');

-- 일반 LIKE '%검색어%'는 인덱스를 못 탄다 (앞에 %가 있으면)
-- FULLTEXT 인덱스는 단어 단위로 역인덱스를 만들어서 빠르게 검색
-- 단, 한글은 ngram 파서 설정이 필요할 수 있다

5. 인덱스를 걸어야 하는 경우¶

인덱스가 필요한 상황

  1. WHERE 절에 자주 쓰는 컬럼 → SELECT * FROM tb_exam WHERE exam_cd = ?; → exam_cd에 인덱스
  2. JOIN 조건 컬럼 → SELECT * FROM tb_a JOIN tb_b ON tb_a.user_no = tb_b.user_no; → tb_b.user_no에 인덱스 (드리븐 테이블 쪽)
  3. ORDER BY 컬럼 → SELECT * FROM tb_board ORDER BY reg_date DESC; → reg_date에 인덱스 (filesort 방지)
  4. GROUP BY 컬럼 → SELECT exam_cd, COUNT(*) FROM tb_exam GROUP BY exam_cd; → exam_cd에 인덱스
  5. 카디널리티가 높은 컬럼 → 카디널리티 = 고유값의 수. user_id: 100만 명 → 카디널리티 100만 → 인덱스 효과 높음. gender: M/F → 카디널리티 2 → 인덱스 효과 거의 없음

6. 인덱스를 걸면 안 되는 경우¶

인덱스가 오히려 해가 되는 상황

  1. INSERT/UPDATE가 매우 빈번한 테이블 → 인덱스가 많을수록 쓰기 성능 하락. 2.7억 건 테이블에 인덱스 5개면? INSERT 1건마다 인덱스 5개를 전부 업데이트해야 한다.
  2. 카디널리티가 극히 낮은 컬럼 → 성별 (M/F), 활성 여부 (Y/N), 상태 (A/B/C). 전체의 50%가 같은 값이면 인덱스로 걸러지는 게 없다. 인덱스 탔는데 결국 절반을 읽어야 하면 풀스캔보다 느릴 수 있다.
  3. 테이블이 매우 작을 때 → 100건짜리 테이블에 인덱스? 풀스캔이 더 빠르다. 인덱스 탐색 자체도 비용이다 (트리 순회). 적은 건수에서는 그냥 전부 읽는 게 낫다.
  4. 거의 조회하지 않는 컬럼 → WHERE에 안 쓰고, JOIN에 안 쓰고, 정렬에 안 쓰면 인덱스 만들어봤자 공간 낭비, 쓰기 성능만 하락.

인덱스 트레이드오프 원리

왜 인덱스를 안 거는 게 나을 수 있는지, 원리를 이해해야 한다. 인덱스 = 별도의 자료구조 = 유지 비용 있음. 읽기를 빠르게 하는 대신, 쓰기를 느리게 한다. 공짜가 아니다.

테이블 특성 인덱스 효과
읽기 많고 쓰기 적은 테이블 인덱스 유리
쓰기 많고 읽기 적은 테이블 인덱스 불리
대부분의 테이블 읽기가 훨씬 많으므로 인덱스 유리

우리 사례 (tb_lms_exam_stare_paper_backup): 쓰기만 미친 듯이 하고 읽기는 거의 안 하는 테이블. 인덱스 없었던 게 오히려 쓰기 성능에는 도움이었을 수 있다. 하지만 그래도 PK는 있어야 한다 (InnoDB 설계 원칙).

7. 인덱스의 비용¶

7.1 쓰기 성능 저하¶

INSERT 1건 실행 시 인덱스별 비용

인덱스 개수 동작
0개 데이터 페이지에 1건 추가
1개 데이터 페이지 + 인덱스 1개 업데이트
3개 데이터 페이지 + 인덱스 3개 업데이트
5개 데이터 페이지 + 인덱스 5개 업데이트

인덱스 하나 추가될 때마다: B-Tree에서 삽입 위치 찾기 → 리프 노드에 추가 → 필요시 노드 분할.

2.7억 건이 쌓인 이유를 생각해보자: 매 시험 제출마다 INSERT. 인덱스가 없었으니 INSERT는 빨랐을 것. 하지만 그 대가로 조회가 27분.

7.2 디스크 사용¶

인덱스의 디스크 비용

인덱스도 디스크 공간을 차지한다. information_schema.TABLES에서 DATA_LENGTH(데이터 크기)와 INDEX_LENGTH(인덱스 크기)로 확인 가능하다.

  • 인덱스 크기는 보통 데이터 크기의 10~30%
  • 43GB 테이블에 인덱스 3개면? 인덱스만 4~12GB 추가
  • /data 196GB 중 134GB 사용 중인 상황에서 43GB 테이블에 인덱스를 추가하면 디스크가 더 부족해질 수 있다

7.3 인덱스 생성 시간¶

대용량 테이블 인덱스 추가 비용

2.7억 건 테이블에 인덱스 추가:

ALTER TABLE tb_lms_exam_stare_paper_backup ADD INDEX idx_exam_cd (exam_cd);

소요 시간: 수 시간 (데이터 양과 서버 성능에 따라). 이 동안 테이블에 metadata lock이 걸리고, DDL이므로 다른 DML이 대기할 수 있으며, 서비스 영향이 있을 수 있다.

대안: pt-online-schema-change (Percona 도구)로 서비스 중 온라인 인덱스 추가 가능. 하지만 2.7억 건이면 이것도 수 시간.

교훈: 인덱스는 테이블이 작을 때 미리 설계해야 한다. 데이터가 쌓인 후에 추가하면 비용이 기하급수적으로 늘어난다.

8. 실전 사례¶

8.1 tb_lms_exam_stare_paper_backup 분석¶

실전 사례: 43GB 테이블의 교훈

테이블: tb_lms_exam_stare_paper_backup / 행 수: 2.7억 건 / 용량: 43GB / 인덱스: 없음 (PK도 없음)

증상: COUNT(*) 실행 시간: 27분. mysqldump 백업 시간: 20분+. 매일 00:00 백업이 14개월째 실패 (0바이트). /data 디스크 196GB 중 134GB 사용 (68%).

왜 인덱스가 없었나? 백업 테이블이라 "조회할 일 없다"고 판단. 그런데 mysqldump도, COUNT(*)도 결국 풀스캔이다. "조회 안 해도" 관리 작업(백업, 모니터링)에서 문제가 터진다.

왜 2.7억 건이 쌓였나? 시험 재채점 코드에서 무한 INSERT 발생. 한 번 재채점할 때마다 전체 응시자의 답안을 백업. 제어 로직 없이 무한 축적.

8.2 증상 치료 vs 원인 치료¶

증상 치료 vs 원인 치료

증상 치료 (인덱스): 인덱스를 추가하면 COUNT(*) 27분 → 수 초. mysqldump도 빨라질 수 있다. 하지만 데이터는 계속 쌓인다. 43GB → 50GB → 60GB → 결국 디스크 부족. 인덱스 자체도 수 GB 추가. 증상만 가렸을 뿐, 병은 진행 중.

원인 치료 (코드 수정): 재채점 시 무한 INSERT 로직을 수정. 이전 백업이 있으면 INSERT 하지 않는다. 데이터가 더 이상 폭증하지 않는다. 기존 43GB는 TRUNCATE로 정리. 이것이 근본 해결.

접근 효과
인덱스 추가 조회 빨라짐. 데이터 증가는 계속.
코드 수정 데이터 폭증 중단. 근본 원인 제거.
TRUNCATE 기존 데이터 정리. 43GB 회수.
전부 적용 근본 해결 + 재발 방지 + 공간 회수

"인덱스는 진통제다. 코드 수정이 수술이다. 진통제만 먹으면 병은 낫지 않는다."

9. 핵심 정리¶

09장 핵심 정리: 인덱스 완전정복

  1. 인덱스 = 책의 색인 → 특정 값을 빠르게 찾기 위한 자료구조. 없으면 풀스캔 (처음부터 끝까지).
  2. B-Tree: 2.7억 건도 4~5번 비교로 찾는다 → 균형 트리 (Balanced Tree). O(log n), 풀스캔 O(n)과 차원이 다름. 2.7억 건: 풀스캔 27분, 인덱스 수 밀리초.
  3. EXPLAIN으로 인덱스 사용 여부 확인 → type: ALL이면 풀스캔 (위험). key: NULL이면 인덱스 미사용. rows: 전체 행 수에 가까우면 풀스캔.
  4. 복합 인덱스는 순서가 생명 → 최좌선 원칙: 왼쪽부터 순서대로 사용해야 한다. (A,B) 인덱스에서 B만으로는 인덱스를 못 탄다.
  5. 인덱스는 공짜가 아니다 → 읽기 빨라지는 대신 쓰기 느려진다. 디스크 공간도 추가로 차지한다. 대용량 테이블에 나중에 추가하면 수 시간 소요.
  6. PK(클러스터드 인덱스)는 반드시 있어야 한다 → InnoDB는 PK 순서로 데이터를 물리적으로 정렬. PK 없는 테이블 = 설계 결함.
  7. 인덱스는 증상 치료, 코드 수정이 원인 치료 → 인덱스 추가해도 데이터 폭증은 멈추지 않는다. 근본 원인을 찾아서 코드를 고쳐야 한다.

"인덱스를 모르면 DB를 모르는 거다. 인덱스를 알면서 안 거는 건 설계다. 인덱스를 몰라서 안 건 건 사고다."

다음 장에서 배울 것: 10장에서는 InnoDB 스토리지 엔진의 내부 구조를 파헤친다. TRUNCATE가 왜 빠르고 DELETE가 왜 느린지, 43GB .ibd 파일이 왜 DELETE로는 안 줄어드는지, Buffer Pool이 뭔지, Redo/Undo 로그가 뭔지. 인덱스가 "어디에" 저장되고 "어떻게" 동작하는지의 물리적 기반을 이해하게 된다.
