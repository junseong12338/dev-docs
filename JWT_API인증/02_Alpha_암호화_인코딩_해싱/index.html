02. 암호화 vs 인코딩 vs 해싱 - Alpha¶

👹 "JWT 배우기 전에 이 3개 구분 못 하면 무조건 삽질한다. Base64가 암호화라고? 해싱이 복호화 가능하다고? 그런 소리 하면 불합격이야."

1. 핵심 개념 - "이 3개가 뭔데 왜 구분해야 해?"¶

비유로 문 열기¶

편지 비유

편지를 보낸다고 생각해봐.

인코딩 = 한국어 편지를 영어로 번역 → 영어 아는 사람은 누구나 읽을 수 있어 → 비밀이 아니야. 형식만 바꾼 거야.

암호화 = 편지를 비밀 암호로 변환 → 열쇠(키) 가진 사람만 읽을 수 있어 → 열쇠로 잠그고, 열쇠로 풀어.

해싱 = 편지를 분쇄기에 넣기 → 원본 복원 불가능 → "이 편지가 원본이랑 같은지"만 확인 가능

진짜 정의¶

구분 정의 목적 되돌릴 수 있어?
인코딩 데이터 형식 변환 전송/저장 호환성 ✅ 누구나 가능
암호화 키를 사용한 데이터 변환 기밀성 보장 ✅ 키 가진 사람만
해싱 단방향 데이터 변환 무결성 검증 ❌ 불가능

이거 헷갈리면 JWT 이해 못 해. 왜냐면 JWT 안에서 3개 다 쓰거든.

2. 인코딩 (Encoding) - "형식 변환일 뿐이야"¶

뭐야?¶

데이터를 다른 형식으로 바꾸는 것. 비밀이 아니야. 누구나 되돌릴 수 있어.

왜 필요해?¶

문제 상황:
  JSON 데이터 {"name": "김철수"} 를 URL로 보내고 싶어
  근데 URL에는 한글, 공백, 특수문자를 못 넣어

해결:
  URL 인코딩 → %7B%22name%22%3A%22%EA%B9%80...
  받는 쪽에서 디코딩하면 원본 복원

JWT에서 어디 쓰여?¶

Base64 인코딩이 JWT의 핵심이야.

JWT = Base64(Header) + "." + Base64(Payload) + "." + Signature

원본 JSON:
  {"alg": "HS256", "typ": "JWT"}

Base64 인코딩:
  eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9

되돌리기 (디코딩):
  eyJhbGci... → {"alg": "HS256", "typ": "JWT"}

여기서 중요한 거: Base64는 암호화가 아니야. 누구나 디코딩할 수 있어.

❌ "JWT는 암호화되어 있으니까 안전해"
✅ "JWT의 Header, Payload는 Base64 인코딩일 뿐. 누구나 읽을 수 있어."

이거 모르면 JWT에 비밀번호 넣는 실수를 해. 절대 하면 안 돼.

대표 인코딩들¶

인코딩 용도 예시
Base64 바이너리 → 텍스트 변환 JWT, 이미지 데이터 URL
URL Encoding URL에 특수문자 포함 공백 → %20
UTF-8 문자 → 바이트 변환 한글 인코딩

3. 암호화 (Encryption) - "키가 있어야 풀 수 있어"¶

뭐야?¶

키(열쇠)를 사용해서 데이터를 읽을 수 없게 변환하는 것. 키 없으면 못 풀어.

두 가지 종류¶

대칭키 vs 비대칭키 암호화

대칭키 암호화 (AES):

  • 잠그는 키 = 푸는 키 (같은 키 1개)
  • 택배 상자에 자물쇠 채우고, 같은 열쇠로 여는 것
  • 보내는 쪽: 키A로 잠금 → 받는 쪽: 키A로 열기

비대칭키 암호화 (RSA):

  • 잠그는 키 ≠ 푸는 키 (키 2개: 공개키, 비밀키)
  • 우체통: 누구나 넣을 수 있지만(공개키), 열쇠는 우체통 주인만 가지고 있음(비밀키)
  • 보내는 쪽: 공개키로 잠금 → 받는 쪽: 비밀키로 열기

우리 프로젝트에서 어디 쓰여?¶

AES-256 (대칭키 암호화) — LMS가 응답 데이터를 암호화해서 보낼 때.

LMS 서버:
  JSON 데이터 → AES-256 암호화(키: "MediOpi@85oO!!!!") → 암호문

NexClass 서버:
  암호문 → AES-256 복호화(키: "MediOpi@85oO!!!!") → JSON 데이터

같은 키로 잠그고, 같은 키로 풀어. → 대칭키.

인코딩이랑 뭐가 달라?¶

인코딩: 아무나 되돌릴 수 있어 (비밀 아님)
암호화: 키 가진 놈만 되돌릴 수 있어 (비밀)

인코딩 = 영어로 번역 (영어 아는 사람은 다 읽음)
암호화 = 금고에 넣기 (열쇠 가진 사람만 열 수 있음)

4. 해싱 (Hashing) - "원본 복원 불가능"¶

뭐야?¶

데이터를 고정 길이의 값으로 변환하는 것. 단방향이라 원본 복원이 불가능해.

비유¶

감자 → 감자칩

감자를 감자칩으로 만들 수 있어.
근데 감자칩을 다시 감자로 만들 수 있어?
→ 불가능.

해싱이 이거야.
데이터 → 해시값 (가능)
해시값 → 데이터 (불가능)

특징¶

1. 같은 입력 → 항상 같은 출력
   "hello" → 항상 "5d41402abc4b..."

2. 다른 입력 → 다른 출력
   "hello" → "5d41402abc4b..."
   "hello!" → "9c3bcd5c66ff..."  ← 완전 다름

3. 출력에서 입력 복원 불가능
   "5d41402abc4b..." → "hello" (X) 불가능

우리 프로젝트에서 어디 쓰여?¶

HMAC-SHA256 — JWT 서명할 때.

JWT 서명 과정:
  Header + Payload + SECRET_KEY → HMAC-SHA256 → 서명값

이 서명값으로 뭘 하냐?
→ "이 토큰이 위조되지 않았다"는 걸 증명

검증:
  서버가 같은 방식으로 서명을 다시 만들어서
  토큰에 붙어있는 서명이랑 비교
  → 같으면 = 위조 안 됨
  → 다르면 = 누가 건드렸네? 거부!

비밀번호 저장에도 쓰여¶

❌ DB에 비밀번호 그대로 저장: "1234"
✅ DB에 해시값 저장: "03ac674216f3e..."

로그인할 때:
  입력 "1234" → 해싱 → "03ac674216f3e..."
  DB에 저장된 해시값이랑 비교
  → 같으면 로그인 성공

DB가 털려도 해시값에서 원본 비밀번호 복원 불가능!

5. JWT에서 3개가 어떻게 쓰이는지¶

JWT = Base64(Header).Base64(Payload).Signature

      ─────────────  ─────────────  ─────────
      인코딩          인코딩          해싱(HMAC)
      (누구나 읽음)   (누구나 읽음)   (위조 방지)

JWT 토큰 구조

Header (인코딩)

  • Base64로 인코딩. 누구나 디코딩 가능.
  • "어떤 알고리즘 썼는지" 메타 정보

Payload (인코딩)

  • Base64로 인코딩. 누구나 디코딩 가능.
  • 사용자 정보, 만료시간 등 실제 데이터
  • 비밀번호 넣지 마! 누구나 읽을 수 있어!

Signature (해싱)

  • HMAC-SHA256(Header + Payload, SECRET_KEY)
  • 위조 방지용. 이게 있어서 토큰을 못 건드려.

LMS 응답에서는

  • AES-256 암호화 (별도)
  • JWT랑 다른 목적. 데이터 기밀성 보장.

역할 분담 정리¶

기술 JWT에서 하는 일 목적
Base64 인코딩 Header, Payload 형식 변환 URL/HTTP 전송 호환
HMAC-SHA256 해싱 Signature 생성 위조 방지
AES-256 암호화 LMS 응답 데이터 보호 기밀성 (JWT와 별개)

6. 주의사항 / 함정¶

함정 1: "Base64는 암호화다"¶

❌ Base64는 암호화가 아니야. 인코딩이야.
   키 없이 누구나 디코딩할 수 있어.
   jwt.io 가보면 토큰 붙여넣기만 해도 내용이 다 보여.

함정 2: "JWT는 안전하니까 뭐든 넣어도 돼"¶

❌ Payload는 누구나 읽을 수 있어.
   비밀번호, 주민번호, 카드번호 절대 넣지 마.
   넣어도 되는 것: 사용자 ID, 권한, 만료시간 정도.

함정 3: "해싱이랑 암호화랑 같은 거 아냐?"¶

❌ 완전 다른 거야.
   암호화: 잠그고 → 풀 수 있음 (양방향)
   해싱: 분쇄하면 → 복원 불가 (단방향)

함정 4: "HMAC이면 서명인데, 서명이면 암호화 아냐?"¶

❌ 서명 ≠ 암호화
   서명: "이거 내가 만든 거 맞아" 증명 (무결성)
   암호화: "이거 남이 못 읽게" 보호 (기밀성)

   JWT 서명은 "위조 안 됐다"를 보장하는 거지,
   "못 읽게 숨기는" 게 아니야.

7. 정리¶

구분 한 줄 정리
인코딩 형식 변환. 비밀 아님. 누구나 되돌림.
암호화 키로 잠금. 키 있는 놈만 풀 수 있음.
해싱 분쇄기. 원본 복원 불가. 같은지 비교만 가능.

이 챕터에서 반드시 기억할 것: JWT의 Header/Payload는 인코딩(누구나 읽음), Signature는 해싱(위조 방지), LMS 응답은 암호화(키로 복호화).

확인 문제 (5문제)¶

다음 문제를 풀어봐. 답 못 하면 위에서 다시 읽어.

Q1. Base64로 인코딩된 JWT Payload에 비밀번호를 넣으면 안 되는 이유는?

Q2. AES-256은 대칭키야 비대칭키야? 왜?

Q3. 해싱은 왜 "단방향"이라고 해? 암호화랑 뭐가 달라?

Q4. JWT 서명(Signature)이 해싱인데, 이걸로 뭘 보장하는 거야? "기밀성"이야 "무결성"이야?

Q5. NexClass ↔ LMS 통신에서 인코딩, 암호화, 해싱이 각각 어디서 쓰여? 3개 다 말해봐.

정답 보기

A1. Base64는 암호화가 아니라 인코딩이야. 키 없이 누구나 디코딩할 수 있어. jwt.io에 토큰 붙여넣으면 Payload 내용이 그대로 보여. 그래서 비밀번호 같은 민감 정보는 절대 넣으면 안 돼.

A2. 대칭키. 잠그는 키와 푸는 키가 같으니까. LMS와 NexClass가 "MediOpi@85oO!!!!" 라는 같은 키 1개를 공유해서, 한쪽이 암호화하면 다른 쪽이 같은 키로 복호화해.

A3. 해싱은 원본 복원이 불가능해서 단방향이야. 감자 → 감자칩은 되지만, 감자칩 → 감자는 안 되는 것처럼. 암호화는 키만 있으면 원본을 복원할 수 있는 양방향이야.

A4. "무결성"이야. "이 토큰이 위조/변조되지 않았다"는 걸 보장하는 거지, "이 토큰을 남이 못 읽게 숨긴다"(기밀성)는 게 아니야. Payload는 누구나 읽을 수 있으니까.

A5.

  • 인코딩: JWT의 Header와 Payload를 Base64로 인코딩 (형식 변환)
  • 해싱: JWT Signature를 HMAC-SHA256으로 생성 (위조 방지)
  • 암호화: LMS가 응답 데이터를 AES-256으로 암호화 (데이터 보호)
