07. JWT 보안 설계 - Delta¶

👹 "JWT 쓸 줄 아는 건 Lv3. JWT의 약점을 알고 대비하는 게 Lv4. 보안 설계 없이 JWT 쓰면 시한폭탄이야."

1. JWT의 약점들 - "완벽한 기술은 없어"¶

약점 1: 토큰 탈취¶

토큰 탈취 시나리오

NexClass → JWT → ???(공격자가 중간에서 훔침) → LMS

공격자가 토큰을 훔쳤어. 세션이면? 서버에서 세션 삭제하면 끝. JWT면? 서버가 아무것도 저장 안 하잖아. → 만료될 때까지 막을 방법이 없어!

이게 JWT의 가장 큰 약점이야.

약점 2: 토큰 크기¶

세션 ID: "abc123" (몇 바이트)
JWT: "eyJhbGci...SflKxwRJ..." (수백~수천 바이트)

매 요청마다 이걸 보내. 대역폭 낭비.
Payload에 많이 넣을수록 더 커짐.
→ 필요한 것만 넣어야 해.

약점 3: 서버에서 토큰 무효화 불가¶

세션:
  "이 사용자 차단!" → 서버에서 세션 삭제 → 즉시 차단

JWT:
  "이 토큰 차단!" → ...어떻게?
  → 서버가 토큰을 관리 안 하니까 개별 무효화가 안 돼
  → 만료될 때까지 기다리거나, 블랙리스트 운영해야 해

2. 만료 시간 설계 - "얼마로 해야 해?"¶

트레이드오프¶

만료 시간 짧으면 (5분):
  ✅ 탈취돼도 5분만 유효
  ❌ 5분마다 토큰 새로 발급해야 함
  ❌ 긴 작업 중에 자꾸 만료됨

만료 시간 길면 (24시간):
  ✅ 한 번 발급하면 하루 종일 사용
  ❌ 탈취되면 24시간 동안 무방비
  ❌ 보안 위험 큼

우리 프로젝트 설계¶

만료 시간: 30분 (1800초)

왜 30분?
  1. 동기화 작업 1회 = 보통 5~15분
  2. 넉넉하게 30분이면 충분
  3. 탈취돼도 30분 뒤 자동 무효화
  4. 매 동기화 시작마다 새 토큰 발급

더 안전하게?
  → 매 API 호출마다 토큰 새로 생성
  → 약간 비효율이지만 보안은 최고

실무에서의 일반적인 만료 시간¶

상황 만료 시간 이유
웹 로그인 (Access Token) 15~30분 짧게 유지 + Refresh Token으로 갱신
모바일 앱 1시간~24시간 사용자 편의
서버↔서버 API 30분~1시간 작업 단위에 맞춰서
일회성 인증 5분 이메일 인증 링크 등

3. 키 관리 - "SECRET_KEY를 어떻게 지켜?"¶

키 노출 = 게임 오버¶

SECRET_KEY가 노출되면?
  → 공격자가 유효한 JWT를 무한히 만들 수 있어
  → 만료 시간도 마음대로 설정 가능
  → 인증 시스템이 완전히 뚫림
  → 5만 명 개인정보 유출 가능

키 관리 방법 (안전한 순서)¶

Level 1 (최악): 코드에 하드코딩
  ❌ String secret = "KnuLmsSecretKey2026!@#0123456789";
  → 코드가 GitHub에 올라가면 키 노출

Level 2 (기본): 설정 파일 분리
  ✅ application.properties에 저장
  → .gitignore에 추가하면 코드에는 안 올라감
  → 우리 프로젝트가 이 방식

Level 3 (좋음): 환경변수
  ✅ 서버의 환경변수에 저장
  → 코드나 설정 파일에 키가 안 남음
  → export LMS_SECRET_KEY="KnuLmsSecretKey2026!@#0123456789"

Level 4 (최선): 비밀 관리 서비스
  ✅ AWS Secrets Manager, HashiCorp Vault 등
  → 키 자동 회전, 접근 제어, 감사 로그
  → 대기업/금융권에서 사용

우리 프로젝트 현재 방식¶

# application.properties
nexclass.api.key=knu-lms-2026                        ← APP_KEY (공개 가능)
nexclass.api.secret=KnuLmsSecretKey2026!@#0123456789  ← SECRET_KEY (비밀!)
lms.sync.aes-key=MediOpi@85oO!!!!                    ← AES KEY (비밀!)

Java에서:
@Value("${nexclass.api.secret}")
private String secretKey;  // 설정 파일에서 주입

→ 코드에 키가 직접 안 들어가.
→ 설정 파일만 잘 관리하면 돼.

4. 토큰 탈취 대응 - "훔쳐가면 어떡해?"¶

HTTPS 필수¶

HTTP (암호화 안 됨):
  NexClass ──JWT(평문)──→ 도청 가능 ──→ LMS
  공격자가 네트워크에서 JWT를 그대로 볼 수 있어!

HTTPS (암호화됨):
  NexClass ──JWT(TLS 암호화)──→ 도청 불가 ──→ LMS
  네트워크 도청해봤자 암호화된 쓰레기만 보여.

우리 프로젝트: https://www.knu10.cc ← HTTPS 사용 중 ✅

만료 시간 짧게¶

만료 30분 → 탈취돼도 30분만 유효
매 요청마다 새 토큰 → 탈취돼도 거의 즉시 만료

블랙리스트 (필요시)¶

토큰 탈취 발견 시:
  1. 해당 토큰의 jti(고유 ID)를 블랙리스트에 등록
  2. 서버가 검증 시 블랙리스트 확인
  3. 블랙리스트에 있으면 거부

단점: 서버가 블랙리스트를 저장해야 함
→ "서버가 아무것도 저장 안 한다"는 JWT 장점이 사라짐
→ 그래서 보통은 만료 시간으로 관리하는 게 일반적

5. Access Token + Refresh Token 패턴¶

웹 서비스에서 많이 쓰는 패턴 (참고)¶

Access Token + Refresh Token 패턴

Access Token:

  • 짧은 만료 (15분)
  • API 호출에 사용
  • 탈취돼도 15분만 유효

Refresh Token:

  • 긴 만료 (7일~30일)
  • Access Token 재발급에만 사용
  • 서버 DB에 저장 (무효화 가능)

흐름:

  1. Access Token으로 API 호출
  2. 만료됨 → Refresh Token으로 새 Access Token 발급
  3. Refresh Token도 만료 → 다시 로그인

우리 프로젝트에서는?¶

서버↔서버 통신이라 Refresh Token이 필요 없어.

왜?
  → 사람이 없으니까 "다시 로그인"이라는 개념이 없어.
  → 코드가 언제든 새 토큰을 만들 수 있어.
  → APP_KEY + SECRET_KEY만 있으면 토큰 무한 생성 가능.

  NexClass: "토큰 만료됐네? 새로 만들지 뭐."
  → generateToken() 호출하면 끝.

  브라우저 인증과 다른 점이 이거야.
  브라우저는 사용자가 다시 ID/PW 입력해야 하니까 Refresh Token이 필요하지만,
  서버는 키를 알고 있으니까 바로 새로 만들면 돼.

6. 트레이드오프 분석¶

설계 결정: 매 요청마다 새 토큰 vs 토큰 재사용¶

구분 방식 A 방식 B 방식 C (우리 선택)

7. 주의사항 / 함정¶

함정 1: "HTTPS 쓰니까 JWT 없어도 되지?"¶

❌ HTTPS는 전송 중 도청을 막는 거야.
   "누가 보냈는지"는 모르잖아.
   HTTPS + JWT = 전송 보호 + 신원 증명 = 완전체.

함정 2: "토큰 만료시간을 1년으로 하면 편하겠다"¶

❌ 토큰 탈취되면 1년 동안 무방비.
   편의 vs 보안의 트레이드오프에서 보안이 우선이야.
   서버↔서버는 토큰 재생성이 공짜니까 짧게 해.

함정 3: "알고리즘을 none으로 하면 서명 안 해도 되겠다"¶

❌ alg: "none"은 서명 없는 JWT.
   JWT 표준에 있지만 사용하면 보안 0.
   누구나 토큰을 만들 수 있어.
   절대 쓰지 마. 이건 유명한 JWT 취약점이야.

8. 정리¶

약점 대응
토큰 탈취 HTTPS + 짧은 만료시간
토큰 크기 Payload에 최소한만 넣기
무효화 불가 짧은 만료시간 + 필요시 블랙리스트
키 노출 설정 파일 분리 + 환경변수

이 챕터에서 반드시 기억할 것: - JWT는 완벽하지 않다. 만료 시간과 키 관리가 보안의 핵심. - 서버↔서버는 Refresh Token 없이 바로 재발급 가능. - HTTPS는 필수. JWT만으로는 전송 보안이 안 돼. - 설계 결정에는 항상 트레이드오프가 있다.

확인 문제 (5문제)¶

다음 문제를 풀어봐. 답 못 하면 위에서 다시 읽어.

Q1. JWT의 가장 큰 약점이 뭐야? 세션과 비교해서 설명해봐.

Q2. 만료 시간을 5분 vs 24시간으로 설정할 때 각각의 장단점을 말해봐.

Q3. SECRET_KEY를 코드에 하드코딩하면 안 되는 이유는? 대안 3가지를 말해봐.

Q4. 우리 프로젝트에서 Refresh Token이 필요 없는 이유는?

Q5. HTTPS가 있는데도 JWT가 필요한 이유는?

정답 보기

A1. 토큰 탈취 시 서버에서 개별 무효화가 안 돼. 세션은 서버에서 세션 삭제하면 즉시 차단 가능하지만, JWT는 서버가 아무것도 저장 안 하니까 만료될 때까지 기다려야 해.

A2. 5분: 탈취돼도 5분만 유효(장), 자주 재발급 필요(단). 24시간: 한 번 발급으로 오래 사용(장), 탈취되면 24시간 무방비(단).

A3. GitHub 등에 코드가 올라가면 키도 노출되니까. 대안: (1) 설정 파일 분리(application.properties), (2) 환경변수, (3) 비밀 관리 서비스(AWS Secrets Manager 등).

A4. 서버↔서버 통신이라 사람이 없어. 코드가 APP_KEY + SECRET_KEY를 알고 있으니까 토큰이 만료되면 바로 새로 만들면 끝. "다시 로그인" 개념이 없으니 Refresh Token도 필요 없어.

A5. HTTPS는 "전송 중 도청 방지"야. "누가 보냈는지(인증)"는 확인 안 해. JWT는 "나 NexClass야" 신원을 증명하는 거야. 둘 다 있어야 전송 보호 + 신원 증명이 완성됨.
