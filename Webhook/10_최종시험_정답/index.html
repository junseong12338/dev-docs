10. 최종시험 - 정답 및 해설¶

"정답 맞히는 건 시작이야. 왜 그게 정답인지 설명 못 하면 불합격."

이건 정답지가 아니야. 해설지야. 정답만 보고 넘어가면 Lv1에서 영원히 못 벗어나. 틀린 문제는 해당 챕터 다시 읽고 와.

객관식 정답 (20점)¶

Q1. API 호출에서 호출 코드를 짜는 건? (객관식, 2점)¶

정답: (B) 보내는 쪽

해설:

보기 왜 맞는가 / 틀렸는가
(A) 받는 쪽 받는 쪽은 응답만 한다
(B) 보내는 쪽 ✅ 호출(요청)하는 쪽이 코드를 짠다
(C) 양쪽 다 호출 코드는 보내는 쪽만 짠다
(D) 아무도 안 짜도 됨 코드 없이 HTTP 요청은 안 나간다

→ 01_API_호출이_뭐야.md 4절 참조

Q2. Webhook에서 보내는 코드를 짜는 건? (객관식, 2점)¶

정답: (B) 플랫폼 (Zoom 개발자)

해설:

보기 왜 맞는가 / 틀렸는가
(A) 우리 우리는 받는 코드만 짠다
(B) 플랫폼 ✅ Zoom 개발자가 Webhook 전송 시스템을 만들었다
(C) 양쪽 다 보내는 건 플랫폼, 받는 건 우리. 역할 분리
(D) LMS 개발자 LMS는 이 흐름에 관여 안 한다

→ 02_Webhook이_뭐야.md 2절 참조

Q3. NC → LMS 출결 전송은? (객관식, 2점)¶

정답: (D) Push이면서 API 호출

해설:

보기 왜 맞는가 / 틀렸는가
(A) Pull + Webhook 방향도 방식도 틀림
(B) Push + Webhook Push는 맞지만 Webhook은 아님. NC 코드에서 직접 호출하니까
(C) Pull + API 호출 NC가 LMS한테 보내는 거니까 Pull이 아니라 Push
(D) Push + API 호출 ✅ NC가 보내는 거(Push) + 코드에서 직접 호출(API 호출)

→ 03_Push와_Pull.md 4절 참조

Q4. Webhook 3단계 중 한 번만 하는 것은? (객관식, 2점)¶

정답: (C) URL 등록

해설: URL 등록은 Zoom Marketplace에서 딱 한 번. 이벤트 감지와 HTTP 전송은 이벤트 발생할 때마다 반복.

→ 04_Webhook_동작_원리.md 1절 참조

Q5. Webhook과 API 호출의 기술적 차이는? (객관식, 2점)¶

정답: (C) 기술적 차이 없음. 둘 다 HTTP 요청

해설: 이 문제가 핵심이야. 기술적으로 같아. 차이는 패턴(URL 등록 시스템 유무)이지 기술이 아니야.

보기 왜 틀렸는가
(A) Webhook도 HTTP 사용
(B) Webhook도 JSON 사용
(D) Webhook도 동기 응답(200 OK) 반환

→ 05_Webhook_vs_API_호출.md 1절 참조

Q6. Webhook 판별 핵심 질문은? (객관식, 2점)¶

정답: (B) "보내는 쪽에 URL 등록하는 설정 페이지가 있냐?"

해설: 이 한 가지 질문으로 Webhook vs API 호출을 판별할 수 있다.

→ 05_Webhook_vs_API_호출.md 4절 참조

Q7. 학생 입장/퇴장 이벤트 순서는? (객관식, 2점)¶

정답: (C) participant_joined → participant_left

해설: 들어가야(joined) 나갈 수(left) 있지. meeting.ended는 강의 전체 종료야.

→ 06_Zoom_Webhook_이벤트.md 1절 참조

Q8. Webhook 서명 검증 알고리즘은? (객관식, 2점)¶

정답: (C) HMAC-SHA256

해설:

보기 왜 틀렸는가
(A) AES-256 대칭키 암호화. 우리 프로젝트에서 LMS 데이터 복호화에 쓰지만 서명 검증은 아님
(B) RSA-2048 비대칭키. Zoom Webhook에서는 안 씀
(D) MD5 보안 취약. 현대에서 쓰면 안 됨

→ 07_Webhook_보안과_설계.md 2절 참조

Q9. NC가 응답 못 하면? (객관식, 2점)¶

정답: (B) Zoom이 자동으로 재시도 (최대 3회)

해설: Zoom은 약 5분, 15분, 30분 간격으로 3번 재시도한다. 이래서 멱등성이 필요한 거야.

→ 07_Webhook_보안과_설계.md 3절 참조

Q10. @RequestBody를 String으로 받는 이유? (객관식, 2점)¶

정답: (B) 서명 검증에 HTTP 요청 원문이 필요해서

해설: HMAC-SHA256 서명 검증은 요청 본문 원문(raw body)을 기반으로 계산. JSON 객체로 파싱하면 키 순서나 공백이 달라져서 서명이 안 맞을 수 있다.

→ 08_NexClass_실전_구현.md 3절 참조

단답형 정답 (24점)¶

Q11. API 호출의 3요소 (단답형, 3점)¶

Q25. "Webhook은 API 호출의 한 종류" 설명 (코드, 5점)¶

정답: Webhook도 HTTP 요청(POST)을 보내는 것이므로 기술적으로 API 호출이다. 다만, "플랫폼의 URL 등록 시스템을 통해 이벤트 발생 시 자동으로 호출되는" 특정 패턴이라서 별도 이름이 붙은 것이다. 덩크가 슛의 한 종류이듯, Webhook은 API 호출의 한 종류다.

채점 기준: "기술적으로 HTTP 요청" 2점 + "패턴/이름이 다른 것" 2점 + 비유/논리적 설명 1점

→ 05_Webhook_vs_API_호출.md 1절, 4절(Q4) 참조

서술형 정답 (21점)¶

Q26. Webhook vs API 호출 URL 등록 관점 (서술형, 4점)¶

정답:

Webhook은 플랫폼이 URL 등록 시스템(설정 페이지)을 제공하고, 거기에 URL을 등록하면 이벤트 발생 시 자동으로 호출해주는 패턴이다. 예를 들어 Zoom Marketplace에 NC 서버 URL을 등록하면, 학생이 입장할 때마다 Zoom이 등록된 URL로 알아서 POST 요청을 보낸다.

일반 API 호출은 이런 등록 시스템 없이, 개발자가 코드에 URL을 직접 작성해서 호출하는 것이다. 예를 들어 NC가 LMS한테 출결을 보낼 때, NC 코드에 http.post("https://lms-server/api/attendance", data)로 직접 짠다.

채점 기준: Webhook 설명(등록 시스템) 1점 + API 호출 설명(코드 직접) 1점 + 예시 각 1점

이건 불합격: "Webhook은 자동이고 API 호출은 수동" → 너무 모호함. URL 등록 시스템 언급 없으면 불합격.

→ 05_Webhook_vs_API_호출.md 2절 참조

Q27. Push ≠ Webhook 반박 (서술형, 4점)¶

정답:

NC가 강의 종료 후 LMS에 출결 데이터를 전송하는 것은 Push(NC가 보내는 것)이지만 Webhook이 아니라 API 호출이다.

이유: NC에는 URL 등록 시스템이 없고, NC 코드에서 LMS URL을 직접 호출한다. Webhook이 되려면 NC가 URL 등록 기능을 제공하고, LMS가 거기에 자기 URL을 등록하는 구조여야 한다.

따라서 Push는 방향(누가 보내냐)이고, Webhook/API 호출은 방식(어떻게 보내냐)이므로, Push라고 해서 무조건 Webhook이 되는 것은 아니다.

채점 기준: NC→LMS 예시 1점 + "URL 등록 시스템 없음" 근거 1점 + "Push=방향, Webhook=방식" 구분 2점

이건 불합격: "내부라서 Webhook이 아니다" → 내부/외부는 기준이 아님.

→ 03_Push와_Pull.md 4절, 05_Webhook_vs_API_호출.md 3절(Q2) 참조

Q28. NexClass 전체 흐름 (서술형, 5점)¶

정답:

  1. 사전 설정: Zoom Marketplace에서 NC 서버 URL을 Webhook 엔드포인트로 등록하고, participant_joined, participant_left, meeting.ended 이벤트를 구독한다.

  2. 학생 입장: 학생이 Zoom 강의실에 입장하면 Zoom이 participant_joined Webhook을 NC에 보낸다. NC는 서명 검증 후 JOIN_LOG 테이블에 입장 시간을 기록한다.

  3. 학생 퇴장: 학생이 퇴장하면 participant_left Webhook이 오고, JOIN_LOG에 퇴장 시간과 참여 시간을 기록한다.

  4. 강의 종료: 교수가 강의를 끝내면 meeting.ended Webhook이 온다. NC는 미퇴장 학생을 처리하고, 학생별 총 참여 시간을 합산해서 출결을 판정한다. 결과를 ATTENDANCE 테이블에 저장한다.

  5. LMS 전달: NC가 LMS한테 출결 결과를 API 호출로 전달한다. 이건 Webhook이 아니라 NC 코드에서 직접 호출하는 것이다.

채점 기준: 6개 키워드 각 0.5점(3점) + 흐름 정확성 2점

→ 08_NexClass_실전_구현.md 1절 참조

Q29. 서명 검증 필요성 (서술형, 4점)¶

정답:

Webhook 엔드포인트 URL(/webhook/zoom)은 인터넷에 공개된 URL이다. 이 URL을 아는 누구나 HTTP POST 요청을 보낼 수 있다.

서명 검증이 없으면, 해커가 가짜 participant_joined 데이터를 보내서 존재하지 않는 학생을 출석 처리시킬 수 있다. 또는 가짜 meeting.ended를 보내서 강의 중간에 출결 집계를 발동시킬 수 있다.

HMAC-SHA256 서명 검증을 하면, Secret Token을 모르는 사람은 올바른 서명값을 만들 수 없으므로 가짜 요청을 걸러낼 수 있다.

채점 기준: "URL 공개" 위험성 1점 + 구체적 공격 시나리오 2점 + HMAC-SHA256으로 방어 1점

→ 07_Webhook_보안과_설계.md 1~2절 참조

Q30. 주니어 개발자한테 설명 (서술형, 4점)¶

정답:

"자동으로 보내는 것"이 Webhook의 정의가 아니야. 자동이냐 수동이냐가 아니라, 어떤 구조로 보내느냐가 핵심이야.

Webhook은 플랫폼이 URL 등록 시스템을 제공하고, 거기에 URL을 등록해두면 이벤트 발생 시 자동으로 호출해주는 패턴이야. Zoom Marketplace에 URL 등록하는 설정 페이지가 있지? 그게 Webhook 시스템이야.

NC → LMS는 그런 등록 시스템 없이, NC 코드에서 http.post(lmsUrl, data) 직접 짠 거야. 우리 코드가 호출 타이밍과 대상을 전부 제어해. 자동으로 실행되는 건 맞지만, 그건 우리 코드의 로직이지 Webhook 시스템이 아니야.

채점 기준: "자동 ≠ Webhook" 1점 + URL 등록 시스템 설명 1.5점 + NC→LMS 구체 설명 1.5점

이건 불합격: "외부 서비스면 Webhook, 내부면 API 호출" → 내부/외부는 기준이 아님.

→ 05_Webhook_vs_API_호출.md 2~3절 참조
