### Q1. HTTP 상태코드 500의 의미 (객관식, 2점)

**정답**: **(D) 서버 내부 에러 (Internal Server Error)**

**해설**:

5xx 대역은 전부 **서버 쪽 잘못**이야. 클라이언트(우리)가 잘못한 게 아니야.

| 보기 | 왜 틀렸는가 |
|------|-------------|
| (A) 요청 형식 오류 | 그건 400 Bad Request. 클라이언트 잘못 |
| (B) 인증 실패 | 그건 401 Unauthorized. 클라이언트 잘못 |
| (C) 리소스 없음 | 그건 404 Not Found. URL 틀림 |
| **(D) 서버 내부 에러** | **정답. 서버가 요청 처리 중 터진 것** |

경상국립대 API가 500을 뱉으면? **경상국립대 서버 문제**야. 우리 요청은 정상이야. 이 구분 못 하면 디버깅할 때 삽질해.

**채점 기준**: 정답만 2점. 부분 점수 없음.

### Q2. GET과 POST의 차이 (단답형, 3점)

**정답**: GET은 데이터 **조회**(읽기)용이고 Body가 없다. POST는 데이터 **생성**(쓰기)용이고 Body에 데이터를 담아 보낸다.

**해설**:

| 구분 | GET | POST |
|------|-----|------|
| **목적** | 조회 (데이터 달라고) | 생성 (데이터 보내고 처리해달라고) |
| **Body** | 없음 (쿼리 파라미터로 전달) | 있음 (Body에 데이터 담음) |
| **멱등성** | 멱등 (몇 번 호출해도 같은 결과) | 비멱등 (호출할 때마다 새로 생성될 수 있음) |
| **캐시** | 가능 | 불가 |
| **우리 코드** | 경상국립대 API 전부 | 충남대, 부산대 일부 |

```java
// GET - 경상국립대
.uri(URI.create("https://api.gnu.ac.kr/api/hs/student?studentNo=174006"))
.GET()    // ← Body 없음. 파라미터는 URL에

// POST - 충남대
.header("Content-Type", "application/x-www-form-urlencoded")
.POST(HttpRequest.BodyPublishers.ofString(postData))  // ← Body에 데이터
```

**근본적 차이**: "GET은 가져오는 것, POST는 보내는 것"이 아니야. GET은 **서버 상태를 바꾸지 않는** 안전한 메서드이고, POST는 **서버 상태를 바꿀 수 있는** 메서드야. 이 차이를 모르면 REST API 설계할 때 망해.

**채점 기준**:
- "조회 vs 생성" 언급: 1점
- "Body 유무" 언급: 1점
- 멱등성 또는 안전성 차이까지 설명: 3점 (만점)

-> 01_HTTP_기초.md 2.1절 참조

### Q3. 401 vs 403 차이 (서술형, 5점)

**정답**: 401은 **인증(Authentication)** 실패, 403은 **인가(Authorization)** 실패다.

**해설**:

```
401 Unauthorized = "너 누구야?" (신원 확인 안 됨)
403 Forbidden    = "너 누군지는 알겠는데, 권한 없어" (접근 거부)
```

| 상태코드 | 의미 | 비유 | 해결 방법 |
|---------|------|------|-----------|
| **401** | 인증 안 됨 | 회사 건물에 사원증 없이 들어가려는 것 | 로그인해라 (사원증 보여줘) |
| **403** | 권한 없음 | 사원증은 있는데 서버실 출입 권한이 없는 것 | 권한 요청해라 (관리자한테 문의) |

경상국립대 API에서:
- `certification-key`를 안 보내면? **401** (너 누군지 모르겠어)
- `certification-key`를 보냈는데 만료된 키면? **401** (인증 실패)
- 인증은 됐는데 해당 API 접근 권한이 없으면? **403** (권한 없어)

**핵심 구분**: 인증(Authentication)은 "**누구**인지 확인하는 것", 인가(Authorization)는 "**뭘 할 수 있는지** 확인하는 것". 이 두 단어 헷갈리면 보안 설계할 때 사고 난다.

**채점 기준**:
- 인증/인가 구분만 함: 2점
- 각각의 의미와 예시: 4점
- 실무 상황까지 연결: 5점 (만점)

-> 01_HTTP_기초.md 3.1절 참조

### Q4. 서버가 완전히 다운되면 상태코드가 오는가? (단답형, 3점)

**정답**: **안 온다.** HTTP 응답 자체가 없고, Java에서는 `catch` 블록에서 `Exception`(ConnectException, IOException 등)이 발생한다.

**해설**:

```
500 에러 = 서버가 "나 에러 났어"라고 응답한 것 (응답은 옴)
서버 다운 = 서버가 아무 말도 못 함 (응답 자체가 없음)
```

| 상황 | HTTP 응답 | Java에서 |
|------|----------|----------|
| 서버 정상 + 처리 성공 | 200 | `response.body()` 사용 |
| 서버 정상 + 내부 에러 | 500 | `response.body()` 사용 (에러 JSON) |
| **서버 다운** | **없음** | **catch (ConnectException e)** |
| **네트워크 끊김** | **없음** | **catch (IOException e)** |
| **응답 시간 초과** | **없음** | **catch (HttpTimeoutException e)** |

```java
try {
    response = client.send(request, HttpResponse.BodyHandlers.ofString());
    // 여기까지 오면 서버가 응답은 한 거야 (200이든 500이든)
} catch (ConnectException e) {
    // 서버 다운. 연결 자체가 안 됨.
    // 상태코드? 없어. 응답 객체 자체가 null이야.
} catch (IOException e) {
    // 네트워크 문제. 상태코드 없음.
}
```

**"500이 오는 거 아니에요?"** 라고 답했으면 틀린 거야. 500도 **정상적인 HTTP 응답**이야. 서버가 "나 터졌어"라고 **말은 한** 거거든.

**채점 기준**:
- "안 온다"만 답함: 1점
- catch에서 Exception 발생 설명: 2점
- ConnectException 등 구체적 예외 언급: 3점 (만점)

### Q5. 중첩 JSON에서 특정 값 추출 (코드 작성, 5점)

**문제의 JSON**:
```json
{
  "status": "200",
  "data": {
    "subjectList": [
      {
        "studentInfo": [
          {"studentNo": "2021011711"},
          {"studentNo": "2024013560"}
        ]
      }
    ]
  }
}
```

**문제**: `"2024013560"`을 추출하는 Java 코드를 작성하시오.

**정답**:

```java
JSONParser parser = new JSONParser();
JSONObject jsonObject = (JSONObject) parser.parse(jsonString);

// 1단계: data 꺼내기 (JSONObject)
JSONObject dataObject = (JSONObject) jsonObject.get("data");

// 2단계: subjectList 꺼내기 (JSONArray)
JSONArray subjectList = (JSONArray) dataObject.get("subjectList");

// 3단계: 첫 번째 과목 꺼내기 (JSONObject) - 인덱스 0
JSONObject subject = (JSONObject) subjectList.get(0);

// 4단계: studentInfo 꺼내기 (JSONArray)
JSONArray studentInfo = (JSONArray) subject.get("studentInfo");

// 5단계: 두 번째 학생 꺼내기 (JSONObject) - 인덱스 1
JSONObject student = (JSONObject) studentInfo.get(1);

// 6단계: studentNo 꺼내기 (String)
String studentNo = (String) student.get("studentNo");  // "2024013560"
```

**해설**:

경로를 말로 하면:
```
전체(JSONObject)
  → "data"(JSONObject)
    → "subjectList"(JSONArray)
      → [0](JSONObject)
        → "studentInfo"(JSONArray)
          → [1](JSONObject)
            → "studentNo"(String) = "2024013560"
```

핵심은 **매 단계마다 타입이 뭔지 아는 것**이야:
- `{}` 보이면 `JSONObject` → `.get("키이름")`
- `[]` 보이면 `JSONArray` → `.get(인덱스)`
- `""` 보이면 `String`
- 숫자면 `Long`

두 번째 학생이니까 인덱스 **1**이야. 0부터 시작하니까.

**모범 답안 (instanceof 포함한 안전한 버전)**:

```java
JSONParser parser = new JSONParser();
Object parsedObject = parser.parse(jsonString);

if (parsedObject instanceof JSONObject) {
    JSONObject jsonObject = (JSONObject) parsedObject;

    if (jsonObject.get("data") instanceof JSONObject) {
        JSONObject dataObject = (JSONObject) jsonObject.get("data");

        if (dataObject.get("subjectList") instanceof JSONArray) {
            JSONArray subjectList = (JSONArray) dataObject.get("subjectList");
            JSONObject subject = (JSONObject) subjectList.get(0);

            if (subject.get("studentInfo") instanceof JSONArray) {
                JSONArray studentInfo = (JSONArray) subject.get("studentInfo");
                JSONObject student = (JSONObject) studentInfo.get(1);
                String studentNo = (String) student.get("studentNo");
            }
        }
    }
}
```

이거까지 쓰면 만점이야.

**채점 기준**:
- 경로는 맞는데 문법 오류: 2점
- 기본 버전 정답: 3점
- instanceof 체크 포함한 안전한 버전: 5점 (만점)

### Q6. JSONObject vs JSONArray 구분 (객관식, 2점)

**정답**: **(B) `{}`는 JSONObject, `[]`는 JSONArray**

**해설**:

| 구분 | JSONObject | JSONArray |
|------|-----------|-----------|
| **기호** | `{ }` | `[ ]` |
| **구조** | 키-값 쌍 (이름으로 접근) | 순서있는 목록 (인덱스로 접근) |
| **접근** | `.get("키이름")` | `.get(인덱스)` |

```json
{"name": "홍길동"}   ← JSONObject (중괄호)
["홍길동", "이순신"]  ← JSONArray (대괄호)
```

왜 틀렸는가:
| 보기 | 왜 틀렸는가 |
|------|-------------|
| (A) 둘 다 같다 | 완전히 다른 타입. 캐스팅하면 ClassCastException |
| **(B) `{}`=JSONObject, `[]`=JSONArray** | **정답** |
| (C) 반대 | 거꾸로 외운 거. 위험 |
| (D) 숫자가 있으면 JSONArray | 숫자와 관계없음. 기호로 구분 |

**채점 기준**: 정답만 2점. 부분 점수 없음.

-> 02_JSON_완전정복.md 4절 참조

### Q7. parser.parse() 반환타입을 JSONObject로 바로 받으면 위험한 이유 (서술형, 5점)

**정답**: `parser.parse()`의 반환 타입은 `Object`이며, 실제 파싱 결과가 JSONObject가 아닐 수 있기 때문이다. JSONArray, String, Long, null 등이 올 수 있고, JSONObject가 아닌 것을 강제 캐스팅하면 **ClassCastException**이 발생한다.

**해설**:

```java
// 위험한 코드
JSONObject jsonObject = (JSONObject) parser.parse(data);
// data가 "[1, 2, 3]" 이면? JSONArray가 반환됨
// → ClassCastException: JSONArray cannot be cast to JSONObject
```

`parser.parse()`가 `Object`를 반환하는 이유:

| 입력 문자열 | 파싱 결과 타입 |
|-------------|----------------|
| `{"name":"홍길동"}` | JSONObject |
| `[1, 2, 3]` | JSONArray |
| `"문자열"` | String |
| `123` | Long |
| `null` | null |

**파싱 전에는 뭐가 올지 모르잖아.** 그래서 모든 타입의 부모인 `Object`로 반환하는 거야. 이게 **다형성(Polymorphism)**이야.

안전한 패턴:
```java
Object parsedObject = parser.parse(data);  // 1. Object로 받아
if (parsedObject instanceof JSONObject) {  // 2. 타입 확인
    JSONObject jsonObject = (JSONObject) parsedObject;  // 3. 안전하게 캐스팅
}
```

**"에러 응답이 올 때만 문제 아닌가요?"** 라고 생각하면 틀린 거야. **어떤 상황에서 문제가 생길지 예측 못 하니까** 항상 방어적으로 코딩하는 거야. 경상국립대 학사연동에서 이거 안 해서 장애 터졌잖아.

**채점 기준**:
- "ClassCastException 발생"만 언급: 2점
- 반환 타입이 Object인 이유 설명: 3점
- 안전한 패턴(instanceof)까지 제시: 5점 (만점)

-> 02_JSON_완전정복.md 3.2절, 5절 참조

### Q8. {"data": ""} 에서 data의 타입 (단답형, 3점)

**정답**: **String** (빈 문자열 `""`)

**해설**:

```java
JSONObject jsonObject = (JSONObject) parser.parse("{\"data\": \"\"}");
Object data = jsonObject.get("data");

data instanceof String      // true  ← 빈 문자열이지만 String이야
data instanceof JSONObject  // false
data instanceof JSONArray   // false

// 따라서 이렇게 하면 터져:
JSONObject dataObj = (JSONObject) jsonObject.get("data");
// → ClassCastException: String cannot be cast to JSONObject
```

이게 경상국립대 API 에러 응답의 핵심이야:

```json
// 정상 응답: data가 JSONObject
{"status": "200", "data": {"subjectList": [...]}}

// 에러 응답: data가 String (빈 문자열)
{"status": "500", "data": ""}
```

**같은 키인데 타입이 달라진다.** `""` 보고 "null이겠지?" 또는 "빈 객체겠지?" 하면 틀린 거야. 쌍따옴표로 감싸져 있으면 **String**이야.

| JSON 값 | Java 타입 |
|---------|-----------|
| `"data": ""` | String (빈 문자열) |
| `"data": {}` | JSONObject (빈 객체) |
| `"data": []` | JSONArray (빈 배열) |
| `"data": null` | null |

**채점 기준**:
- "String" 정답: 2점
- 빈 문자열도 String임을 명확히 설명: 3점 (만점)

### Q9. RESTful URL 설계 원칙 (객관식, 2점)

**정답**: **(C) URL은 동사가 아니라 명사를 사용한다**

**해설**:

| 나쁜 설계 (동사) | 좋은 설계 (명사) | 이유 |
|------------------|------------------|------|
| `/getStudents` | `/students` (GET) | HTTP 메서드가 동사 역할을 하니까 |
| `/createUser` | `/users` (POST) | POST = 생성. URL에 동사 넣을 필요 없음 |
| `/deleteStudent/123` | `/students/123` (DELETE) | DELETE = 삭제. URL은 리소스만 표현 |

경상국립대 API도 이 원칙을 지키고 있어:
```
GET /api/hs/student          ← "student" = 명사
GET /api/hr/employee         ← "employee" = 명사
GET /api/hs/openSubjectStudents  ← 명사 (개설과목 수강생)
```

왜 동사를 안 쓰냐? **HTTP 메서드 자체가 동사**이기 때문이야. GET = 가져와, POST = 만들어, DELETE = 지워. URL에 또 동사를 넣으면 중복이야.

**채점 기준**: 정답만 2점. 부분 점수 없음.

### Q10. 경로 파라미터 vs 쿼리 파라미터 (단답형, 3점)

**정답**:
- **경로 파라미터 (Path Parameter)**: URL 경로 안에 포함. 리소스를 **식별**할 때 사용. 예: `/students/174006`
- **쿼리 파라미터 (Query Parameter)**: URL 뒤에 `?key=value`로 추가. **필터링/검색 조건**에 사용. 예: `/students?year=2026&semester=1`

**해설**:

```
경로 파라미터:
/api/hs/student/174006
                └─── 이 학생을 "식별"하는 값

쿼리 파라미터:
/api/hs/openSubjectStudents?year=2026&semester=1&page=1
                            └─── "조건"을 거는 값
```

경상국립대 API는 쿼리 파라미터 방식:
```java
String param = "?studentNo=" + sync.getStudentId();
// /api/hs/student?studentNo=174006
```

RESTful하게 하면 경로 파라미터가 더 맞아:
```
/api/hs/students/174006  ← 이게 더 RESTful
/api/hs/student?studentNo=174006  ← 현재 경상국립대 방식
```

| 구분 | 경로 파라미터 | 쿼리 파라미터 |
|------|-------------|-------------|
| **위치** | URL 경로 안 | URL 뒤 `?` 이후 |
| **용도** | 리소스 식별 (필수값) | 필터/검색/정렬/페이징 (선택값) |
| **예시** | `/students/{id}` | `/students?page=1&size=20` |
| **필수 여부** | 보통 필수 | 보통 선택 |

**채점 기준**:

### Q11. N+1 문제와 경상국립대 API의 관계 (서술형, 5점)

**정답**: 경상국립대 API는 수강생 학번 리스트를 1회 조회(1)한 후, 각 학생의 상세 정보를 1명씩 개별 호출(N)해야 하는 구조다. 수강생 500명이면 API를 501회 이상 호출해야 하며, 이것이 N+1 문제다. 호출 횟수가 많을수록 실패 확률이 높아지고, 소요 시간이 길어진다.

**해설**:

```
N+1 문제란:
  목록 조회 1회(1) + 상세 조회 N회(N) = N+1회 호출

경상국립대 학사연동 호출 흐름:
  1회: getKsnuSyncList() → 과목 정보 + 학번 리스트 수신
  N회: 학번 1개마다 getKsnuUserInfo() 개별 호출
  M회: 교수 1명마다 getKsnuProfInfo() 개별 호출

  과목 30개, 교수 30명, 학생 500명이면:
  1(과목 목록) + 30(교수 상세) + 500(학생 상세) = 531회
```

다른 대학교(충남대, 부산대)와 비교:

| 구분 | 경상국립대 | 다른 대학교 |
|------|-----------|-----------|
| 수강생 500명 조회 | API **531회 이상** | API **3~5회** |
| 소요 시간 | 수 분~수십 분 | 수 초 |
| 1건이라도 실패하면 | 해당 학생 **누락** | 전체 리스트 재조회 |

**현실에서 터진 사고**:
- 매 실행 시 랜덤으로 3~6명 수강취소 발생
- 원인: 500번 호출 중 일부가 경상국립대 서버 500 에러로 실패
- 실패한 학생 = 조회 안 됨 = suspend 상태 유지 = 수강취소 처리

이상적인 해결:
```
현재:  GET /api/hs/student?studentNo=174006  (500번 반복)
이상:  POST /api/hs/students/batch
       Body: {"studentNos": ["174006","174007",...]}  (1번에 끝)
```

**채점 기준**:
- N+1 개념만 설명: 2점
- 경상국립대 구조와 연결: 3점
- 실패 확률/시간/사고까지 연결: 5점 (만점)

### Q12. instanceof 연산자의 역할 (단답형, 3점)

**정답**: 변수 안에 들어있는 실제 객체가 **특정 타입인지 확인**하는 연산자. `true` 또는 `false`를 반환하며, 다운캐스팅 전에 **타입 안전성을 검증**하기 위해 사용한다.

**해설**:

```java
Object obj = parser.parse(data);

obj instanceof JSONObject   // obj가 JSONObject이면 true
obj instanceof JSONArray    // obj가 JSONArray이면 true
obj instanceof String       // obj가 String이면 true
null instanceof JSONObject  // null은 항상 false
```

왜 필요해? **다운캐스팅은 위험하니까.**

```java
// instanceof 없이 (도박)
JSONObject json = (JSONObject) obj;  // obj가 JSONArray면? 터짐

// instanceof 있으면 (안전)
if (obj instanceof JSONObject) {
    JSONObject json = (JSONObject) obj;  // 확인 후 캐스팅
}
```

**채점 기준**:
- "타입 확인" 언급: 1점
- true/false 반환 설명: 2점
- 다운캐스팅 안전성과 연결: 3점 (만점)

### Q13. ClassCastException 발생 조건 (객관식, 2점)

**정답**: **(B) 실제 타입과 다른 타입으로 강제 다운캐스팅할 때**

**해설**:

```java
Object obj = new JSONArray();           // 실제로는 JSONArray
JSONObject json = (JSONObject) obj;     // JSONObject로 강제 캐스팅
// → ClassCastException: JSONArray cannot be cast to JSONObject
```

왜 틀렸는가:

| 보기 | 왜 틀렸는가 |
|------|-------------|
| (A) 업캐스팅할 때 | 업캐스팅은 항상 안전. 자식 -> 부모는 자동 |
| **(B) 타입 불일치 다운캐스팅** | **정답** |
| (C) instanceof 사용할 때 | instanceof는 확인만 함. 예외 안 남 |
| (D) null 캐스팅할 때 | null은 어떤 타입으로든 캐스팅 가능 (예외 안 남) |

핵심: **컴파일 시점에 안 잡힌다.** 빌드 성공, 배포 성공, 실행하다가 특정 조건에서만 터진다. 이게 제일 위험한 버그야.

**정답**:

```java
JSONParser parser = new JSONParser();
Object parsedObject = parser.parse(response);  // 1. Object로 받기

if (parsedObject instanceof JSONObject) {      // 2. 타입 확인
    JSONObject jsonObject = (JSONObject) parsedObject;  // 3. 안전하게 캐스팅

3단계 패턴이야:
1. **Object로 받기** - `parser.parse()`의 반환을 Object로
2. **instanceof로 확인** - 실제 타입이 맞는지 체크
3. **캐스팅** - 확인 후에만 캐스팅

Object parsedObject = parser1.parse(userInfo);

        JSONObject userInfoObj = (JSONObject) jsonObject1.get("data");
        // ...
    }
}
```

**흔한 실수**:
```java
// 실수 1: Object로 안 받고 바로 캐스팅

// 실수 2: null 체크만 하고 타입 체크 안 함

    JSONObject data = (JSONObject) jsonObject.get("data");  // data가 ""이면 터짐!
}
```

### Q15. 다운캐스팅 vs 업캐스팅 (단답형, 3점)

**정답**:
- **업캐스팅 (자식 -> 부모)**: 자동. 캐스팅 연산자 불필요. 항상 안전.
- **다운캐스팅 (부모 -> 자식)**: 수동. `(타입)` 연산자 필수. ClassCastException 위험.

**해설**:

```java
// 업캐스팅 (자동, 안전)
JSONObject json = new JSONObject();
Object obj = json;            // 자식 → 부모. 캐스팅 연산자 필요 없음.
// JSONObject는 Object이기도 하니까. "강아지는 동물이다."

// 다운캐스팅 (수동, 위험)
Object obj = parser.parse(data);
JSONObject json = (JSONObject) obj;  // 부모 → 자식. (JSONObject) 필요.
// Object가 진짜 JSONObject인지 보장 못 하니까. "동물이 강아지인가? 고양이일 수도."
```

| 구분 | 업캐스팅 | 다운캐스팅 |
|------|---------|-----------|
| **방향** | 자식 -> 부모 | 부모 -> 자식 |
| **자동/수동** | 자동 | 수동 (명시적 캐스팅) |
| **안전성** | 항상 안전 | 위험 (ClassCastException 가능) |
| **비유** | 강아지는 동물이다 (당연) | 동물이 강아지인가? (확인 필요) |

**채점 기준**:
- 방향만 답함: 1점
- 자동/수동 차이 설명: 2점
- 안전성 차이까지 설명: 3점 (만점)

### Q16. rollbackFor = Exception.class + ClassCastException 발생 시 (서술형, 5점)

**정답**: **전체 롤백된다.** `@Transactional(rollbackFor = Exception.class)`는 Checked/Unchecked 모든 Exception에서 롤백한다. ClassCastException은 RuntimeException이므로 `rollbackFor` 없이도 기본적으로 롤백되지만, `rollbackFor = Exception.class`에 의해서도 롤백 대상이다. 메서드 안의 **모든 DB 변경이 취소**된다.

**해설**:

```
Exception 계층 구조:
  Exception
    ├── Checked Exception (IOException, ParseException...)
    └── RuntimeException (Unchecked)
          ├── NullPointerException
          ├── ClassCastException  ← 이놈
          └── IndexOutOfBoundsException

rollbackFor = Exception.class:
  → Exception의 모든 하위 타입에서 롤백
  → Checked든 Unchecked든 전부 포함
```

경상국립대 syncGsnu() 시나리오:
!!! danger "@Transactional(rollbackFor = Exception.class) 범위"
    - 과목 50개 INSERT 완료 (성공)
    - 교수 30명 INSERT 완료 (성공)
    - 수강생 200명 INSERT 완료 (성공)
    - 51번째 과목에서 ClassCastException 발생!

    -> **전체 ROLLBACK**

    -> 50개 과목, 30명 교수, 200명 수강생 전부 취소

**"ClassCastException만 롤백되나요?"** 아니. 메서드 전체가 하나의 트랜잭션이야. **원자성(Atomicity)에 의해 전부 성공하거나 전부 실패**해야 해. 중간까지만 성공하고 나머지는 실패하는 상태는 존재하지 않는다.

**채점 기준**:
- "롤백된다"만 답함: 2점
- Checked/Unchecked 구분 설명: 3점
- 원자성과 전체 롤백 시나리오 설명: 5점 (만점)

-> 05_Spring_Transactional.md 4절, 7.2절 참조

### Q17. ACID 원칙 중 Atomicity 설명 (단답형, 3점)

**정답**: **원자성(Atomicity)** - 트랜잭션 내 모든 작업이 **전부 성공하거나, 전부 실패**해야 한다. 중간 상태는 존재하지 않는다.

**해설**:

```
Atomicity (원자성): 전부 성공 or 전부 실패. 중간은 없다.
Consistency (일관성): 트랜잭션 전후로 데이터 규칙이 깨지면 안 된다.
Isolation (격리성): 동시 트랜잭션끼리 간섭하면 안 된다.
Durability (지속성): 커밋된 데이터는 영구 보존된다.
```

비유: 전기 스위치. ON 아니면 OFF. 절반만 켜진 상태는 없어.

은행 송금 예시:
```
A 계좌 -100만원 (성공)
B 계좌 +100만원 (실패)
→ 원자성에 의해 A 계좌 차감도 취소됨
→ 100만원 증발 방지
```

**채점 기준**:
- "전부 성공 or 전부 실패" 언급: 2점
- 비유 또는 예시 포함: 3점 (만점)

### Q18. private 메서드에 @Transactional이 안 먹히는 이유 (서술형, 5점)

**정답**: Spring의 `@Transactional`은 **프록시(Proxy) 패턴**으로 동작한다. Spring이 Service 객체 대신 프록시 객체를 생성하여, 메서드 호출을 가로채서 트랜잭션을 시작/커밋/롤백한다. 그런데 **private 메서드는 프록시가 오버라이드할 수 없기** 때문에 @Transactional이 적용되지 않는다.

**해설**:

```
@Transactional 동작 원리:
  Controller → [프록시 객체] → 실제 Service
                    │
                    ├─ 1. BEGIN TRANSACTION
                    ├─ 2. 실제 메서드 호출
                    ├─ 3-a. 정상 → COMMIT
                    └─ 3-b. 예외 → ROLLBACK
```

프록시는 **CGLIB 또는 JDK Dynamic Proxy**로 만들어지는데, 둘 다 **public 메서드만 가로챌 수 있어**:

```java
// 프록시가 가로챌 수 있는 것
@Transactional
public void doSomething() { ... }    // 가능 (public)

// 프록시가 가로챌 수 없는 것
@Transactional
private void doSomething() { ... }   // 불가능 (private은 오버라이드 불가)

@Transactional
protected void doSomething() { ... } // 기본적으로 불가능
```

같은 이유로 **같은 클래스 내부 호출**도 안 먹혀:
```java
public void outerMethod() {
    this.innerMethod();  // 프록시를 거치지 않고 직접 호출 → @Transactional 무시
}

@Transactional
public void innerMethod() { ... }
```

**해결 방법**: 별도 클래스로 분리하여 외부에서 호출하게 만든다.

**채점 기준**:
- "프록시" 언급: 2점
- 프록시 동작 원리 설명: 3점
- 내부 호출 문제까지 연결: 5점 (만점)

-> 05_Spring_Transactional.md 3.2절, 8.1절, 8.2절 참조

### Q19. 전파속성 REQUIRED vs REQUIRES_NEW (객관식, 2점)

**정답**: **(A) REQUIRED는 기존 트랜잭션에 참여하고, REQUIRES_NEW는 항상 새 트랜잭션을 만든다**

**해설**:

```
REQUIRED (기본값):
  "트랜잭션 있으면 합류, 없으면 내가 만들게"
  [──────── A의 트랜잭션 ────────────]
  [  A 작업  ] [  B 작업 (합류)  ] [커밋]
  → B가 터지면 A도 같이 롤백

REQUIRES_NEW:
  "난 독립적으로 할 거야"
  [──────── A의 트랜잭션 1 ──────────]
  [  A 작업  ]              [  A 계속  ]
               [── B 트랜잭션 2 ──]
  → B가 터져도 A는 별도 (try-catch 하면)
```

| 속성 | 기존 트랜잭션 있을 때 | 기존 트랜잭션 없을 때 | 롤백 범위 |
|------|---------------------|---------------------|-----------|
| REQUIRED | **참여** | 새로 생성 | 전체 롤백 |
| REQUIRES_NEW | **무시하고 새로 생성** | 새로 생성 | 자기 것만 롤백 |

충남대가 sync를 3번 나눠 호출한 것도 이 원리야. 스케줄러에 `@Transactional`이 없으니까, 각 `syncCnu()` 호출마다 REQUIRED에 의해 **새 트랜잭션**이 생성된다.

**채점 기준**: 정답만 2점. 부분 점수 없음.

### Q20. #{} vs ${} 차이 (단답형, 3점)

**정답**:
- `#{}` = **PreparedStatement 파라미터 바인딩**. 값이 `?`로 치환되고 별도 파라미터로 전달. **SQL Injection 방지**.
- `${}` = **문자열 직접 치환**. 값이 SQL에 그대로 삽입됨. **SQL Injection 위험**.

**해설**:

```sql
-- #{} 사용 (안전)
SELECT * FROM TB_USER WHERE USER_NO = #{userNo}
-- 실제 실행: SELECT * FROM TB_USER WHERE USER_NO = ?
-- 파라미터: '174006'

-- ${} 사용 (위험)
SELECT * FROM TB_USER WHERE USER_NO = ${userNo}
-- 실제 실행: SELECT * FROM TB_USER WHERE USER_NO = 174006
-- 값이 SQL에 그대로 들어감!
```

SQL Injection 공격 예시:
```
userNo = "174006; DROP TABLE TB_USER; --"

#{}: SELECT * FROM TB_USER WHERE USER_NO = ?
     → 파라미터로 전달되어 안전

${}: SELECT * FROM TB_USER WHERE USER_NO = 174006; DROP TABLE TB_USER; --
     → 테이블 삭제됨!
```

**언제 ${}를 쓰냐?** 테이블명이나 컬럼명을 동적으로 바꿀 때. 이때는 PreparedStatement의 `?`를 쓸 수 없으니까:
```sql

ORDER BY ${sortColumn}      -- 컬럼명도 ?로 못 바꿈
```
단, 이 경우에도 **화이트리스트 검증** 필수.

**채점 기준**:

- SQL Injection 위험성 언급: 3점 (만점)

### Q22. INSERT ON DUPLICATE KEY UPDATE 용도 (서술형, 5점)

**정답**: 데이터를 INSERT하되, PRIMARY KEY 또는 UNIQUE KEY가 **이미 존재하면 UPDATE**하는 MySQL 구문이다. "없으면 넣고, 있으면 갱신" (UPSERT) 패턴을 구현한다. 학사연동에서 수강생 정보를 동기화할 때, 신규 수강생은 INSERT하고 기존 수강생은 정보를 UPDATE하는 데 사용한다.

**해설**:

```sql
INSERT INTO TB_COURSE_USER (
    COURSE_CD, USER_NO, USER_NM, STATUS
) VALUES (
    'CS101', '174006', '홍길동', 'Y'
)
ON DUPLICATE KEY UPDATE
    USER_NM = VALUES(USER_NM),
    STATUS = VALUES(STATUS)
```

동작 흐름:
```
1. INSERT 시도
2. PK/UK 충돌 없으면 → INSERT 완료
3. PK/UK 충돌 있으면 → UPDATE 실행 (ON DUPLICATE KEY UPDATE 부분)
```

왜 이게 필요해?
```
학사연동 시나리오:
  1차 연동: 수강생 500명 INSERT (전부 신규)
  2차 연동: 수강생 502명 중 500명은 이미 존재, 2명 신규

  INSERT만 하면? → 500명 중복 에러
  SELECT 후 분기? → 502번 SELECT + INSERT/UPDATE = 비효율
  ON DUPLICATE KEY UPDATE? → 502번 INSERT 시도, 500번은 자동 UPDATE
```

**채점 기준**:
- "있으면 UPDATE, 없으면 INSERT" 설명: 2점
- UPSERT 패턴 설명: 3점
- 학사연동 적용 시나리오 설명: 5점 (만점)

### Q23. SQL 빈 줄이 문제되는 이유 (단답형, 3점)

**정답**: MyBatis(iBATIS)에서 SQL 쿼리 내부의 **빈 줄(empty line)은 쿼리를 끊어버릴 수 있다.** 빈 줄 이후의 SQL이 별도 구문으로 인식되거나, 파싱 에러가 발생한다.

**해설**:

```sql
-- 빈 줄 있는 SQL (문제 발생)
SELECT * FROM TB_USER

WHERE USER_NO = '174006'

-- MyBatis가 이걸 이렇게 인식할 수 있어:
-- 구문 1: SELECT * FROM TB_USER
-- 구문 2: WHERE USER_NO = '174006'  ← 문법 오류!
```

```sql
-- 올바른 SQL (빈 줄 없음)
SELECT * FROM TB_USER
WHERE USER_NO = '174006'
```

특히 Oracle + iBATIS 조합에서 이 문제가 자주 발생한다. 우리 프로젝트에서 **절대 SQL 내부에 빈 줄을 넣지 않는** 이유가 이거야.

**채점 기준**:
- "파싱 에러 발생" 언급: 2점
- 쿼리가 끊어지는 원리까지 설명: 3점 (만점)

### Q24. 지수 백오프의 재시도 간격 패턴 (단답형, 3점)

**정답**: **1초, 2초, 4초, 8초...** (2의 거듭제곱으로 증가). 재시도할 때마다 대기 시간을 **배로 늘리는** 패턴이다.

**해설**:

```
고정 간격 (Fixed):     1초 → 1초 → 1초 → 1초
지수 백오프 (Exponential Backoff): 1초 → 2초 → 4초 → 8초
```

```java
// 지수 백오프 구현
for (int retry = 0; retry < maxRetry; retry++) {
    try {
        return callApi();
    } catch (Exception e) {
        long waitTime = (long) Math.pow(2, retry) * 1000;  // 1s, 2s, 4s, 8s...
        Thread.sleep(waitTime);
    }
}
```

왜 지수 백오프를 쓰냐?
- 서버가 과부하 상태일 때 **고정 간격으로 계속 때리면** 서버가 더 죽어
- 간격을 늘려주면 서버가 **회복할 시간**을 줌
- AWS, Google Cloud 등 모든 대형 서비스가 이 패턴을 권장

우리 코드(경상국립대)는 **고정 1초 간격**:
```java
Thread.sleep(1000);  // 매번 1초. 지수 백오프 아님.
```
개선 여지가 있지만, 현재 구조에서는 이 정도로 충분해.

**채점 기준**:
- "1, 2, 4, 8초" 패턴 답함: 2점
- 배로 늘리는 이유 설명: 3점 (만점)

### Q25. 멱등성이 중요한 이유 (서술형, 5점)

**정답**: **멱등성(Idempotency)** 이란 같은 요청을 여러 번 보내도 결과가 동일한 성질이다. 재시도 시 **부작용(Side Effect)을 방지**하기 위해 필수적이다. 멱등하지 않은 요청을 재시도하면 데이터가 중복 생성되거나 잘못된 상태가 될 수 있다.

**해설**:

```
멱등한 요청 (GET):
  1번 호출: 학생 174006 조회 → 홍길동
  2번 호출: 학생 174006 조회 → 홍길동 (동일 결과)
  3번 호출: 학생 174006 조회 → 홍길동 (동일 결과)
  → 몇 번을 호출해도 결과가 같다. 재시도해도 안전.

멱등하지 않은 요청 (POST로 주문 생성):
  1번 호출: 주문 생성 → 주문번호 001 생성
  2번 호출: 주문 생성 → 주문번호 002 생성 (중복 주문!)
  3번 호출: 주문 생성 → 주문번호 003 생성 (중복 주문!)
  → 재시도하면 주문이 3개 생겨버림!
```

| HTTP 메서드 | 멱등성 | 재시도 안전성 |
|------------|--------|-------------|
| GET | 멱등 | 안전 |
| PUT | 멱등 | 안전 |
| DELETE | 멱등 | 안전 |
| POST | **비멱등** | **위험** |

경상국립대 학사연동은 전부 **GET**이니까 멱등해. 같은 학번으로 몇 번을 조회해도 같은 결과가 나와. 그래서 재시도 로직이 안전한 거야.

만약 POST로 수강 등록 API였다면? 재시도할 때마다 **수강 등록이 중복**될 수 있어. 이때는 멱등성 키(Idempotency Key)를 사용해서 중복 처리를 방지해야 해.

**채점 기준**:
- 멱등성 정의: 2점
- 재시도와 부작용 연결: 3점
- HTTP 메서드별 멱등성과 실무 예시: 5점 (만점)

### Q26. Thread.sleep(1000)이 톰캣에 미치는 영향 (코드 분석, 5점)

**문제 코드**:
```java
for (int retry = 0; retry < 3; retry++) {
    result = callApi();
    if (isSuccess(result)) break;
    Thread.sleep(1000);  // 이게 미치는 영향은?
}
```

**정답**: `Thread.sleep(1000)`은 **현재 스레드를 1초간 블로킹**한다. 톰캣은 스레드 풀 기반으로 요청을 처리하는데, sleep 중인 스레드는 **다른 요청을 처리할 수 없다.** 재시도가 3회이면 최대 3초간 스레드가 점유되며, 동시에 여러 학사연동이 실행되면 **톰캣 스레드 풀이 고갈**될 수 있다.

**해설**:

```
톰캣 스레드 풀 (기본 200개):
  [스레드1: 요청 처리 중]
  [스레드2: 요청 처리 중]
  [스레드3: Thread.sleep 중... 1초 대기... 아무것도 못 함]  ← 낭비!
  [스레드4: Thread.sleep 중... 1초 대기... 아무것도 못 함]  ← 낭비!
  ...
  [스레드200: 요청 처리 중]
```

경상국립대 학사연동의 최악 시나리오:
```
학생 500명 x 재시도 3회 x sleep 1초 = 최대 1,500초(25분) 스레드 점유

동시에 5개 과목 동기화 중이면?
→ 5 x 500명 x 3회 x 1초 = 7,500초 분량의 스레드 블로킹
→ 톰캣 스레드 풀 고갈 가능
→ 다른 사용자 요청 처리 불가 (사이트 먹통)
```

대안:
- **비동기 처리** (CompletableFuture, @Async)
- **별도 스레드풀** 사용 (메인 톰캣 풀과 분리)
- **메시지 큐** (RabbitMQ, Kafka)로 비동기 재시도

현재 코드에서는 **스케줄러에서 실행**하므로 사용자 요청 스레드와는 분리되어 있어. 하지만 스케줄러 스레드도 무한정 있는 건 아니니까 주의해야 해.

**채점 기준**:
- "스레드가 블로킹된다" 언급: 2점
- 톰캣 스레드 풀과 연결: 3점
- 스레드 고갈 시나리오 + 대안 제시: 5점 (만점)

### Q27. 경상국립대 수강취소 랜덤 발생 근본 원인 (서술형, 5점)

**정답**: 근본 원인은 **API 개별 호출 구조 + 서버 500 에러의 조합**이다. 수강생 조회가 1명씩 개별 API 호출 방식이기 때문에, 경상국립대 서버가 간헐적으로 500 에러를 반환하면 해당 학생의 정보를 가져오지 못한다. 학사연동 로직이 "전체 suspend 처리 후 API 성공한 학생만 복원"하는 구조이므로, API 실패한 학생은 **수강취소로 처리**된다.

**해설**:

전체 흐름:
```
1. 기존 수강생 전체를 suspend(정지) 처리
2. 경상국립대 API로 현재 수강생 목록 조회
3. API에서 확인된 학생만 suspend 해제 (복원)
4. API에서 확인 안 된 학생 = 수강취소 처리

문제 발생:
  학생 500명 중 API 3건 실패 (500 에러)
  → 3명은 suspend 해제 안 됨
  → 3명 수강취소 처리
  → 다음 날 연동에서 복구
  → 그 다음 날 또 다른 학생 실패
  → "매일 랜덤으로 3~6명 수강취소" 현상 발생
```

근본 원인 분석 (5 Whys):
```
Q1: 왜 수강취소가 발생하나?
A1: API 조회 실패한 학생이 suspend 상태로 남아서

Q2: 왜 API 조회가 실패하나?
A2: 경상국립대 서버가 간헐적으로 500 에러를 반환해서

Q3: 왜 한 명 실패가 수강취소로 이어지나?
A3: "전체 suspend → 확인된 학생만 복원" 구조이기 때문에

Q4: 왜 이런 구조가 됐나?
A4: 경상국립대 API가 학생 1명씩 개별 조회만 지원하기 때문에

Q5: 근본 원인은?
A5: N+1 구조의 API 설계 + 전체 suspend 방식의 동기화 로직
```

| 원인 레벨 | 내용 | 해결 가능 여부 |
|-----------|------|-------------|
| 표면 | 경상국립대 500 에러 | 우리가 해결 불가 (상대 서버) |
| 중간 | N+1 API 호출 구조 | 경상국립대 API 변경 요청 필요 |
| **근본** | **전체 suspend 방식** | **우리 코드에서 diff 방식으로 변경 가능** |

**채점 기준**:
- "API 500 에러" 원인만 언급: 2점
- 개별 호출 구조 + suspend 방식 연결: 3점
- 5 Whys 분석 수준으로 근본 원인 도달: 5점 (만점)

### Q28. 수정 전/후 코드 비교 분석 (코드 분석, 5점)

**문제**: 아래 수정 전/후 코드의 차이를 분석하시오.

**수정 전**:
```java
JSONObject jsonObject = (JSONObject) parser.parse(sjbRslt);
JSONObject dataObject = (JSONObject) jsonObject.get("data");
JSONArray sbjArray = (JSONArray) dataObject.get("subjectList");
```

**수정 후**:
```java
Object parsedObject = parser.parse(sjbRslt);
if (parsedObject instanceof JSONObject) {
    JSONObject jsonObject = (JSONObject) parsedObject;
    if (jsonObject.containsKey("data") && jsonObject.get("data") instanceof JSONObject) {
        JSONObject dataObject = (JSONObject) jsonObject.get("data");
        if (dataObject.get("subjectList") instanceof JSONArray) {
            JSONArray sbjArray = (JSONArray) dataObject.get("subjectList");
        }
    }
}
```

**정답**:

| 비교 항목 | 수정 전 | 수정 후 |
|-----------|---------|---------|
| **파싱 결과 수신** | JSONObject로 직접 캐스팅 | Object로 받음 |
| **타입 확인** | 없음 (기도 메타) | 매 단계 instanceof 체크 |
| **에러 응답 대응** | ClassCastException 발생 | 조건 불충족 시 안전하게 스킵 |
| **안전성** | 정상 응답에서만 동작 | 어떤 응답에서도 안 터짐 |

수정 전 코드의 문제:
```
경상국립대 에러 응답: {"status":"500","data":""}

수정 전:
  jsonObject.get("data") → String "" 반환
  (JSONObject) "" → ClassCastException!
  → @Transactional에 의해 전체 롤백
  → 이미 처리된 과목들도 전부 취소

수정 후:
  jsonObject.get("data") instanceof JSONObject → false
  → if 블록 진입 안 함
  → 에러 있는 데이터만 건너뛰고 나머지 계속 처리
```

**핵심 변경 3가지**:
1. `parser.parse()` 결과를 **Object로 수신** (다운캐스팅 위험 제거)
2. **instanceof 체크** 후 캐스팅 (타입 안전성 확보)
3. **containsKey 체크** 추가 (키 자체가 없는 경우 대비)

이 3줄 차이가 "매일 랜덤 수강취소" 장애와 "안정적 동기화"의 차이야.

**채점 기준**:
- 차이점 나열만 함: 2점
- 왜 위험한지 설명: 3점
- 실제 장애 시나리오와 연결: 5점 (만점)

동작 흐름:

```
retry=0 (1회차):
  ├─ API 호출
  ├─ 성공 (status 200 포함)? → break (반복 종료)
  └─ 실패? → "재시도 (1/3)" 출력 → 1초 대기 → 다음 반복

retry=1 (2회차):
  ├─ API 호출
  ├─ 성공? → break
  └─ 실패? → "재시도 (2/3)" 출력 → 1초 대기 → 다음 반복

retry=2 (3회차, 마지막):
  ├─ API 호출
  ├─ 성공? → break
  └─ 실패? → retry < 2 조건 불충족 → 로그/대기 없이 반복 종료
```

**핵심 포인트 분석**:

1. **성공 판단 기준**: `userInfo != null && userInfo.contains("\"status\":\"200\"")`
   - null 체크: 응답 자체가 없는 경우 방어
   - status 200 포함 확인: **JSON 파싱하지 않고 문자열 contains로 확인** (가벼움)

2. **`retry < 2` 조건**: 마지막 시도(retry=2)에서는 sleep 하지 않음
   - 어차피 마지막이니까 기다릴 필요 없음
   - 불필요한 1초 지연 제거

3. **3회 모두 실패 시**: for문이 정상 종료되고, userInfo에는 마지막 실패 응답이 남아있음
   - 이후 코드에서 이 실패 응답을 파싱하면 instanceof 체크에서 걸림
   - 해당 학생은 건너뛰어짐

**한계점**:
- 고정 1초 간격 (지수 백오프가 아님)
- 최대 3회 고정 (동적 조절 불가)
- 실패 학생 목록을 별도로 모아서 재처리하는 로직 없음

**채점 기준**:
- 기본 흐름만 설명: 2점
- 성공 판단 기준, retry < 2 조건 분석: 3점
- 한계점까지 분석: 5점 (만점)

### Q30. 근본적 해결 방안 제안 (서술형, 5점)

**정답**: 가장 근본적인 해결은 경상국립대에 **학생 목록 일괄 조회(Batch) API 신설을 요청**하는 것이다. 이를 통해 N+1 문제를 해소하고, API 호출 횟수를 수백 회에서 수 회로 줄일 수 있다.

**해설**:

해결 방안을 **우리가 할 수 있는 것**과 **경상국립대에 요청할 것**으로 나눠야 해:

**1단계: 우리가 할 수 있는 것 (즉시 적용 가능)**

| 방안 | 설명 | 효과 |
|------|------|------|
| **(a) diff 방식으로 변경** | 전체 suspend 대신 기존 데이터와 비교하여 변경분만 처리 | 실패한 학생이 수강취소되지 않음 |
| **(b) 실패 목록 별도 관리** | 실패한 학생 학번을 모아서 추가 재시도 | 누락 최소화 |
| **(c) 지수 백오프 적용** | 고정 1초 대신 1초, 2초, 4초로 증가 | 서버 부하 감소, 성공률 향상 |
| **(d) 병렬 처리** | CompletableFuture로 동시 호출 | 전체 소요 시간 단축 |

**2단계: 경상국립대에 요청할 것 (근본적 해결)**

```
요청 1: Batch 조회 API 신설
  현재: GET /api/hs/student?studentNo=174006  (1명씩)
  요청: POST /api/hs/students/batch
        Body: {"studentNos": ["174006","174007",...]}

요청 2: 수강생 상세 정보 포함 API
  현재: openSubjectStudents → 학번만 반환 → 학생 상세 따로 조회
  요청: openSubjectStudents?includeDetail=true → 상세 정보 포함 반환
```

효과 비교:

| 항목 | 현재 (N+1) | Batch API 적용 후 |
|------|-----------|-----------------|
| API 호출 횟수 | 500명 = **530회 이상** | **2~3회** |
| 소요 시간 | 수 분~수십 분 | **수 초** |
| 실패 확률 | 높음 (530번 중 1번만 실패해도) | **매우 낮음** |
| 수강취소 사고 | 매일 3~6명 | **0명** |

가장 현실적인 순서:
```
1순위: diff 방식 변경 (우리 코드만 수정, 즉시 가능)
       → 실패해도 수강취소 안 됨

2순위: 실패 목록 재시도 (우리 코드만 수정)
       → 누락 최소화

3순위: 경상국립대에 Batch API 요청 (협의 필요)
       → 근본적 해결. 하지만 시간 걸림
```

**"Batch API만 요청하면 되잖아요?"** 경상국립대가 언제 만들어줄지 모르잖아. 남에게 의존하는 해결책만 세우면 안 돼. **우리가 지금 당장 할 수 있는 것(1순위)부터 먼저 하고**, 근본적 해결(3순위)은 병행해야 해. 이게 실전 장애 대응이야.

**채점 기준**:
- "Batch API 요청" 하나만 답함: 2점
- 우리 측 개선 방안도 제시: 3점
- 우선순위와 현실적 판단까지: 5점 (만점)
