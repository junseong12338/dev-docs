### Q19. AuthenticInterceptor catch 블록 분석 (코드 분석, 5점)

```java
try {
    SecurityUtil.authorizationCheckRunner(request, response);
} catch(Exception e) {
    e.printStackTrace();
    throw e;
}
```

**정답:**

1. `SecurityUtil.authorizationCheckRunner()`를 호출한다 (인증/권한 체크)
2. 예외가 발생하면 `catch(Exception e)`에서 **모든 종류의 예외**를 잡는다
3. `e.printStackTrace()`로 예외의 전체 스택 트레이스를 **System.err(catalina.out)**에 출력한다
4. `throw e`로 잡은 예외를 **그대로 다시 던진다**
5. 결론: 이 catch 블록은 예외를 **처리하지 않는다**. 스택 트레이스를 찍고 다시 던지기만 한다. "구경만 하고 넘기는" 코드다.

!!! warning "불합격 기준"
    "예외를 처리한다"고 쓰면 0점. 이 코드는 예외를 처리하는 게 아니라 관찰하고 넘기는 것이다.

**출처: 03장**

### Q20. authorizationCheckRunner() 분석 (코드 분석, 5점)

```java
public static void authorizationCheckRunner(...) throws Exception {
    try {
        SecurityUtil.authorizationCheck(request, response);
    } catch (Exception ex) {
        throw ex;
    }
```

- catch에서 로깅, 변환, 복구 등 어떤 처리도 하지 않으므로 래퍼로서의 가치도 없다

### Q23. 예외별 분리 처리 (코드 분석, 5점)

**정답:**

수정 전: 모든 예외를 `Exception`으로 뭉뚱그려서 잡고, 전부 `e.printStackTrace()`로 출력한다. SessionBrokenException(정상 흐름)이든 NullPointerException(진짜 버그)이든 동일하게 처리한다.

수정 후: `SessionBrokenException`을 **먼저** 별도로 잡아서 로그 없이 그냥 던진다. 나머지 예외는 `Exception` catch에서 **Logger로** 기록한다.

SessionBrokenException을 별도로 잡는 이유:

1. **정상 흐름이다**: 세션 만료 → 로그인 리다이렉트는 에러가 아니라 보안 기능의 정상 작동
2. **빈도가 높다**: 하루 1,471회 발생. 매번 로그를 남기면 노이즈만 늘어난다
3. **진짜 문제가 묻힌다**: 정상 흐름의 로그에 파묻혀서 진짜 에러(NPE 등)를 발견하기 어렵다

Java의 catch는 위에서부터 매칭되므로, `SessionBrokenException`(하위 클래스)을 `Exception`(상위 클래스)보다 먼저 선언해야 한다.

**출처: 08장**

### Q24. 예외 계층과 catch 매칭 (코드 분석, 5점)

**정답:**

Java의 catch는 **위에서부터 순서대로 매칭**된다. 예외 객체가 해당 catch의 타입이거나 하위 타입이면 잡힌다.

`SessionBrokenException`은 `AuthorityException`의 하위 클래스이고, `AuthorityException`은 `MediopiaDefineException`의 하위 클래스이고, 이는 `Exception`의 하위 클래스다.

첫 번째 catch `catch(SessionBrokenException e)`에서:
- SessionBrokenException은 SessionBrokenException **그 자체**이므로 → **매칭된다. 여기서 잡힌다.**

두 번째 catch `catch(Exception e)`까지 내려갈 필요가 없다. 이미 첫 번째에서 잡혔으니까.

만약 `NullPointerException`이 발생하면:
- 첫 번째 catch: NullPointerException은 SessionBrokenException이 아니다 → **매칭 안 됨**
- 두 번째 catch: NullPointerException은 Exception의 하위 클래스다 → **매칭됨. 여기서 잡힌다.**

**출처: 06장**

### Q25. grep 명령어 분석 (코드 분석, 5점)

## 서술형 정답 (21점)

### Q26. HUB-LMS 세션 공유 (서술형, 4점)

**모범 답안:**

HUB와 LMS는 물리적으로 다른 서버다. 두 서버가 같은 Redis를 세션 저장소로 사용한다. 사용자가 HUB에서 로그인하면 세션 데이터(UserNo, UserType 등)가 Redis에 저장되고, 브라우저에 JSESSIONID 쿠키가 발급된다. 사용자가 LMS에 접속하면 브라우저가 같은 JSESSIONID 쿠키를 보내고, LMS는 이 ID로 Redis에서 세션을 조회하여 로그인된 사용자로 인식한다. Redis가 두 시스템 사이의 다리 역할을 하는 것이다.

!!! warning "불합격 기준"
    Redis의 역할을 빼고 "쿠키로 공유한다"만 쓰면 0점. 쿠키에는 ID만 있고, 실제 데이터는 Redis에 있다는 걸 빼먹으면 안 된다.

**출처: 05장**

### Q27. "세션 만료는 에러가 아니다" 근거 (서술형, 4점)

**모범 답안:**

1. **보안 기능의 정상 작동이다**: 세션 타임아웃은 무인 세션을 방지하기 위한 의도적인 보안 설계다. 2시간 동안 활동이 없으면 자동 로그아웃하는 것은 설계 의도대로 동작하는 것이다.
2. **일상적인 빈도로 발생한다**: 하루 1,471회 발생한다. 에러라면 이 빈도는 비정상이지만, 정상 흐름이라면 자연스러운 수치다 (사용자가 점심시간, 퇴근 후 세션 만료).
3. **후속 동작이 정상이다**: 세션 만료 후 로그인 페이지로 리다이렉트되고, 다시 로그인하면 정상 사용 가능하다. 데이터 손실이나 시스템 오류가 없다.

!!! warning "불합격 기준"
    근거가 2개 이하이거나, "당연히 정상이잖아요" 같은 근거 없는 답은 불합격.

**출처: 07장**

### Q28. WAS02 사고 연쇄 반응 (서술형, 5점)

**모범 답안:**

1. **e.printStackTrace()로 모든 예외의 스택 트레이스가 catalina.out에 출력됨**: SessionBrokenException 포함 모든 예외가 매번 35줄씩 기록됨. 하루 약 51,485줄의 불필요한 로그 발생.

2. **468일간 truncate 실패로 catalina.out이 4.4GB까지 성장**: WAS01은 logrotate가 정상 동작했지만, WAS02는 truncate가 안 되어 단일 파일이 무한히 커짐.

3. **로그 로테이션이 4.4GB 파일을 truncate 없이 복사**: 매일 4.4GB를 복사만 하고 원본을 비우지 않아서, 16일간 약 73GB의 로그 파일이 디스크에 축적됨.

4. **Acronis 백업이 매일 밤 73GB를 스캔하면서 buff/cache에 3.5GB 메모리 사용**: 파일 시스템 캐시가 메모리를 잡아먹어서 Free Memory가 149MB까지 감소.

5. **메모리 부족으로 Swap 81.6% 사용 → 서버 무응답**: Swap은 디스크 기반이라 메모리보다 수백 배 느림. Tomcat 쓰레드들이 Swap 대기하면서 응답 시간 폭증, 쓰레드 풀 고갈로 서버가 무응답 상태에 빠짐.

!!! warning "불합격 기준"
    단계 간 인과관계를 설명하지 않고 나열만 하면 3점 감점. "이렇게 되니까 → 이래서 → 이렇게 됐다"의 연쇄를 보여줘야 한다.

### Q29. Logger를 써야 하는 이유 (서술형, 4점)

**모범 답안:**

1. **로그 레벨 제어**: ERROR, WARN, INFO, DEBUG 등 레벨별로 on/off가 가능하다. 운영 환경에서는 WARN 이상만 남기고, 디버깅 시 DEBUG를 켤 수 있다. e.printStackTrace()는 무조건 출력되어 끌 수 없다.

2. **파일 로테이션**: Logger는 날짜별/크기별로 로그 파일을 자동 분리하고, 오래된 파일을 삭제한다. e.printStackTrace()는 catalina.out 하나에 무한히 쌓여서 디스크를 고갈시킨다.

3. **구조화된 포맷**: Logger는 타임스탬프, 쓰레드 이름, 클래스명을 포함한 일관된 포맷으로 출력한다. e.printStackTrace()는 날짜도 시간도 없는 원시 스택 트레이스만 출력해서 장애 분석이 어렵다.

**출처: 08장**

2. **catalina.out 파일 크기**: 단일 파일이 **1GB를 초과**하면 경고. 정상적으로 로테이션되고 있다면 하루치 이상 커지지 않아야 한다.
