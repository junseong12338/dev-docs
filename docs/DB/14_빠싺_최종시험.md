## Part 1: 기본 개념 (Lv1~Lv2) -- 15문제

### Q1. SQL 명령어 분류

다음 10개 명령어를 DDL / DML / DCL로 분류해라.

```
CREATE, SELECT, GRANT, DROP, INSERT,
REVOKE, ALTER, UPDATE, DELETE, TRUNCATE
```

??? success "정답 보기"
    **DDL (Data Definition Language)** -- 구조를 정의/변경/삭제: CREATE, DROP, ALTER, TRUNCATE

    **DML (Data Manipulation Language)** -- 데이터를 조작: SELECT, INSERT, UPDATE, DELETE

    **DCL (Data Control Language)** -- 권한을 제어: GRANT, REVOKE

    핵심 구분법:

    - DDL = 테이블(구조) 자체를 만들고/바꾸고/없앤다
    - DML = 테이블 안의 데이터(행)를 다룬다
    - DCL = 누가 뭘 할 수 있는지(권한)를 다룬다

    TRUNCATE가 DDL인 이유는 Q3에서 다룬다.

    참고: 01장, 02장

### Q2. DDL과 DML의 결정적 차이 1가지

DDL과 DML을 가르는 **가장 중요한** 차이 1가지를 말해라.

??? success "정답 보기"
    **Auto-Commit 여부.**

    - DDL: 실행 즉시 Auto-Commit → ROLLBACK 불가
    - DML: COMMIT 전까지 ROLLBACK 가능

    이게 왜 "결정적"인가:

    - DDL을 잘못 실행하면 되돌릴 수 없다
    - DML은 COMMIT 전에 "아차!" 하면 ROLLBACK 가능
    - 프로덕션에서 DDL 실행 = 돌이킬 수 없는 결정

    "문법이 다르다", "대상이 다르다"도 차이지만, 실전에서 사고를 내느냐 마느냐를 결정하는 건 Auto-Commit이다.

    참고: 02장, 04장

### Q3. TRUNCATE는 DDL인가 DML인가? 왜?

TRUNCATE TABLE은 데이터를 지우는 건데 왜 DML이 아니라 DDL인가?

??? success "정답 보기"
    **DDL이다.** 이유:

    TRUNCATE의 내부 동작: 1) 기존 .ibd 파일(데이터 파일) 삭제 → 2) 새 .ibd 파일 생성 → 테이블 구조를 "다시 만드는" 행위

    DELETE의 내부 동작: 1) 행을 하나씩 찾아서 → 2) 각 행에 "삭제됨" 마킹 → 데이터(행)를 "조작하는" 행위

    TRUNCATE는 "데이터를 하나씩 지우는" 게 아니라 "파일째 날리고 새로 만드는" 것이다. 내부적으로 DROP + CREATE에 가깝다.

    그래서:

    - Auto-Commit 된다 (ROLLBACK 불가)
    - Undo 로그 안 쓴다
    - 트리거 발동 안 한다
    - Auto Increment 초기화된다

    "데이터를 지우니까 DML 아닌가요?" → 행위의 결과가 아니라 내부 동작이 기준이다.

    참고: 03장

### Q4. Auto-Commit이란? DDL과의 관계

Auto-Commit이 뭔지 설명하고, DDL과 어떤 관계가 있는지 말해라.

??? success "정답 보기"
    Auto-Commit = 명령 실행 즉시 자동으로 COMMIT되는 것

    일반 DML: `BEGIN; INSERT ...; -- 아직 COMMIT 안 함. ROLLBACK 가능. COMMIT; ← 여기서 확정`

    DDL (CREATE, DROP, ALTER, TRUNCATE): `TRUNCATE TABLE tb_user; -- 실행 즉시 COMMIT됨. ROLLBACK 불가. 명시적 COMMIT 불필요. 이미 끝남.`

    DDL과의 관계:

    - DDL은 항상 Auto-Commit
    - DDL 실행 전에 열려있던 트랜잭션도 자동 COMMIT됨 (암묵적 커밋)
    - 이게 Q5의 함정이다

    참고: 04장

### Q5. 다음 코드의 결과를 예측해라

```sql
BEGIN;
INSERT INTO tb_test VALUES (1, 'A');
INSERT INTO tb_test VALUES (2, 'B');
TRUNCATE TABLE tb_test;
ROLLBACK;
SELECT * FROM tb_test;
```

결과는?

??? success "정답 보기"
    **결과: 빈 테이블 (0건)**

    ```
    BEGIN;
    INSERT (1, 'A');         ← 트랜잭션 안에서 INSERT
    INSERT (2, 'B');         ← 트랜잭션 안에서 INSERT
    TRUNCATE TABLE tb_test;  ← DDL! 여기서 3가지가 동시에 일어남:
                                1) 앞의 INSERT 2건 암묵적 COMMIT (확정)
                                2) TRUNCATE 실행 (전체 삭제)
                                3) TRUNCATE Auto-Commit (삭제 확정)
    ROLLBACK;                ← 돌릴 게 없다. 이미 다 COMMIT됨.
    SELECT * FROM tb_test;   ← 빈 테이블
    ```

    함정:

    - "ROLLBACK 하면 INSERT 2건이 살아나지 않을까?" → 안 살아남
    - TRUNCATE(DDL) 실행 시점에 앞의 DML이 암묵적 COMMIT되고, TRUNCATE 자체도 Auto-Commit이므로 전부 확정됨
    - ROLLBACK은 COMMIT 이후에는 무의미

    이게 DDL과 DML을 구분해야 하는 실전적 이유.

    참고: 03장, 04장

### Q6. DELETE vs TRUNCATE: 43GB 전체 삭제 비교

43GB(2.7억 건) 테이블을 전체 삭제할 때,
DELETE와 TRUNCATE 각각의 예상 소요시간과 이유를 말해라.

??? success "정답 보기"
    **DELETE FROM tb_backup;** -- 예상 시간: 수 시간 ~ 수십 시간

    1. 2.7억 행을 하나씩 찾아야 한다 (인덱스 없으면 풀스캔)
    2. 각 행마다 Undo 로그 기록 (약 43GB 추가)
    3. 각 행마다 Redo 로그 기록 (수 GB 추가)
    4. 각 행마다 Row Lock 획득
    5. 디스크 I/O 폭발
    6. 디스크 여유 53GB인데 Undo 43GB 필요 → Disk Full → 서버 죽음

    **TRUNCATE TABLE tb_backup;** -- 예상 시간: 2~3분

    1. OS에게 "43GB 파일 삭제해" 요청
    2. OS가 파일시스템에서 블록 해제 (파일 크기에 비례)
    3. 빈 .ibd 파일 새로 생성
    4. Undo/Redo 로그 거의 안 씀
    5. 추가 디스크 필요 없음

    결론: 43GB 전체 삭제에 DELETE를 쓰면 서버가 죽는다. TRUNCATE를 써야 한다.

    참고: 03장

### Q7. DROP과 TRUNCATE의 차이 3가지

DROP TABLE과 TRUNCATE TABLE의 차이를 3가지 이상 말해라.

??? success "정답 보기"
    | 항목 | TRUNCATE | DROP |
    |------|----------|------|
    | 테이블 존재 | 유지 (구조 살아있음) | 사라짐 (완전 소멸) |
    | 데이터 | 전부 삭제 | 전부 삭제 |
    | 인덱스 정의 | 유지 | 사라짐 |

    | Auto Increment | 초기화 (1부터) | - |
    | 이후 INSERT | 바로 가능 | 불가 (테이블 없음) |
    | 용도 | 데이터만 비우고 테이블은 계속 쓸 때 | 이 테이블 자체가 필요 없을 때 |

    핵심: TRUNCATE = 통장에서 돈 다 빼기 / DROP = 통장 해지

### Q8. ACID 4가지 설명

트랜잭션의 ACID 4가지를 각각 한 문장으로 설명해라.

??? success "정답 보기"
    - **A (Atomicity, 원자성)** → 트랜잭션 안의 작업은 전부 성공하거나 전부 실패한다. (중간 상태는 없다)
    - **C (Consistency, 일관성)** → 트랜잭션 전후로 DB는 항상 유효한 상태를 유지한다. (제약조건 위반 같은 건 허용 안 됨)
    - **I (Isolation, 격리성)** → 동시 실행 중인 트랜잭션들은 서로의 중간 결과를 볼 수 없다. (격리 수준에 따라 정도가 다름)
    - **D (Durability, 지속성)** → COMMIT된 데이터는 서버가 죽어도 보존된다. (Redo 로그가 이걸 보장함)

    외우지 마. 각각이 "없으면 뭐가 문제인지"를 생각해봐:

    - A 없으면: 이체에서 출금만 되고 입금은 안 됨
    - C 없으면: 잔액이 마이너스인 상태가 허용됨
    - I 없으면: 남이 수정 중인 값을 읽어서 잘못된 계산
    - D 없으면: COMMIT 했는데 서버 재시작하면 데이터 사라짐

    참고: 04장

### Q9. Undo 로그와 Redo 로그의 역할 차이

Undo 로그와 Redo 로그의 역할을 각각 설명해라.

??? success "정답 보기"
    **Undo 로그 -- "이전 상태로 되돌리기"**

    - 용도: ROLLBACK, MVCC 읽기
    - 기록: 변경 전 데이터 ("원래 이랬다")
    - 예: UPDATE로 score를 80→95로 바꿈 → Undo에 "원래 80이었다" 기록 → ROLLBACK 하면 Undo 보고 80으로 복원

    **Redo 로그 -- "변경 내용 다시 적용하기"**

    - 용도: 장애 복구 (Crash Recovery)
    - 기록: 변경 후 데이터 ("이렇게 바꿨다")
    - 예: UPDATE로 score를 80→95로 바꿈 → Redo에 "95로 바꿨다" 기록 → 서버 비정상 종료 후 재시작 시, COMMIT된 것만 Redo로 복원

    한 줄 요약:

    - Undo = "실수했으면 돌려놓을 수 있게" (ROLLBACK)
    - Redo = "서버 죽어도 데이터 안 잃게" (Crash Recovery)

    이게 TRUNCATE와 무슨 관계? TRUNCATE는 Undo 로그를 안 쓴다 → ROLLBACK 불가. DELETE는 Undo 로그를 쓴다 → 43GB면 Undo도 43GB → Disk Full

    참고: 04장, 10장

### Q10. WHERE 없는 DELETE의 문제점

`DELETE FROM tb_user;` (WHERE 없음)의 문제점 2가지를 말해라.

??? success "정답 보기"
    **문제 1: 전체 행을 하나씩 삭제 → 엄청나게 느리다**

    - Undo 로그: 전체 데이터만큼 생성
    - Redo 로그: 삭제 기록 전부 생성
    - Row Lock: 모든 행에 락
    - 대용량이면 서버 죽을 수 있다

    **문제 2: 디스크 공간이 반환되지 않는다**

    - DELETE는 행을 "삭제됨" 마킹만 한다
    - .ibd 파일 크기는 줄어들지 않는다
    - OPTIMIZE TABLE을 별도로 실행해야 공간 반환
    - 그것도 대용량이면 수 시간 걸린다

    그래서 전체 삭제가 목적이면 TRUNCATE를 써야 한다. WHERE 없는 DELETE = "가장 느리고 비효율적인 전체 삭제".

    참고: 03장

### Q11. 인덱스를 책의 색인에 비유해서 설명

인덱스가 뭔지 책의 색인에 비유해서 설명해라.

??? success "정답 보기"
    책에서 "트랜잭션"이라는 단어를 찾으려면:

    **색인 없는 책:** 1페이지부터 끝까지 한 장씩 넘기면서 찾는다. 500페이지면 최대 500페이지 확인. 이게 "풀스캔(Full Table Scan)"

    **색인 있는 책:** 뒤쪽 색인에서 "트랜잭션: p.42, p.108, p.256" 확인. 바로 해당 페이지로 이동. 3번만 확인하면 됨. 이게 "인덱스 스캔(Index Scan)"

    DB 인덱스도 같다:

    - 인덱스 = 정렬된 색인 (B-Tree 구조)
    - 인덱스 없이 검색 = 풀스캔 = 전체 다 읽기
    - 인덱스 있으면 = 색인 → 바로 이동

    2.7억 건에서 인덱스 없이 COUNT(*): 2.7억 행 전부 읽기 → 27분. 이게 우리 실제 사례.

    참고: 09장

### Q12. PK(Primary Key)의 2가지 특성

Primary Key가 가지는 2가지 필수 특성을 말해라.

??? success "정답 보기"
    1. **유일성 (Unique)** → 같은 값이 두 개 있을 수 없다. PK가 '1001'인 행이 2개면 "누가 진짜야?" 데이터 무결성의 기본.
    2. **NOT NULL** → 비어있을 수 없다. PK가 NULL이면 "이 행을 어떻게 특정해?" 식별 불가능한 행은 의미가 없다.

    추가로 알아야 할 것:

    - InnoDB에서 PK = Clustered Index (데이터 정렬 기준)
    - PK 순서대로 물리적으로 저장됨
    - PK가 없으면 InnoDB가 내부적으로 숨겨진 PK를 만듦 (비추천)

    참고: 09장

### Q13. FK가 있는 테이블에 TRUNCATE하면?

자식 테이블이 FK로 참조하고 있는 부모 테이블에 TRUNCATE하면 어떻게 되나?

??? success "정답 보기"
    에러 발생: `ERROR 1701 (42000): Cannot truncate a table referenced in a foreign key constraint`

    TRUNCATE는 FK 참조가 있으면 실행 거부된다.

    왜? TRUNCATE는 행 단위가 아니라 파일 단위 삭제 → 행 단위 FK 체크를 할 수 없다 → DB 무결성 보장 불가 → 거부

    해결 방법:

    1. 자식 테이블 먼저 TRUNCATE → 부모 TRUNCATE
    2. `SET FOREIGN_KEY_CHECKS = 0;` → TRUNCATE → `SET FOREIGN_KEY_CHECKS = 1;` (위험: FK 무결성 깨질 수 있음. 아는 상태에서만 쓸 것)
    3. DELETE 사용 (FK 체크하면서 삭제. 느리지만 안전)

    참고: 03장

### Q14. InnoDB와 MyISAM의 차이 3가지

InnoDB와 MyISAM의 주요 차이를 3가지 이상 말해라.

??? success "정답 보기"
    | 항목 | InnoDB | MyISAM |
    |------|--------|--------|
    | 트랜잭션 | 지원 | 미지원 |
    | 락 단위 | Row Lock (행 단위) | Table Lock (테이블) |
    | FK 지원 | 지원 | 미지원 |
    | Crash Recovery | 지원 (Redo 로그) | 미지원 |
    | MVCC | 지원 | 미지원 |
    | 동시성 | 높음 | 낮음 |
    | COUNT(*) | 느림 (MVCC 때문) | 빠름 (메타 저장) |

    MariaDB/MySQL 5.5부터 기본 엔진은 InnoDB. 실무에서 MyISAM 쓸 일은 거의 없다. 하지만 차이를 아는 건 InnoDB의 특성을 이해하는 데 필수.

    왜 InnoDB COUNT(*)가 느린가? MVCC 때문에 각 행이 "현재 트랜잭션에서 보이는지" 판단해야 함. 메타데이터에 정확한 행 수를 저장하지 않음. 2.7억 건 COUNT(*) = 27분이었던 이유.

    참고: 10장

### Q15. SELECT 실행 순서

다음 절이 실제로 실행되는 순서를 말해라.

```sql
SELECT, FROM, WHERE, GROUP BY, HAVING, ORDER BY
```

??? success "정답 보기"
    작성 순서: SELECT → FROM → WHERE → GROUP BY → HAVING → ORDER BY

    **실행 순서:** FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY

    1. FROM -- 어느 테이블에서?
    2. WHERE -- 어떤 조건으로 필터?
    3. GROUP BY -- 어떤 기준으로 그룹화?
    4. HAVING -- 그룹에 대한 조건?
    5. SELECT -- 어떤 컬럼을 가져올지?
    6. ORDER BY -- 어떤 순서로 정렬?

    왜 이 순서가 중요한가:

    - WHERE에서 SELECT의 별칭(alias)을 쓸 수 없는 이유: WHERE가 먼저 실행되니까
    - HAVING에서 집계함수를 쓸 수 있는 이유: GROUP BY 후에 실행되니까
    - EXPLAIN으로 실행 계획 볼 때 이 순서를 이해해야 최적화 가능

### Q16. 읽기 이상 현상 3가지

Dirty Read, Non-Repeatable Read, Phantom Read를 각각 한 문장으로 정의해라.

??? success "정답 보기"
    **Dirty Read (더티 리드)** → 다른 트랜잭션이 COMMIT하지 않은 데이터를 읽는 것. A가 수정 중인 값을 B가 읽었는데, A가 ROLLBACK하면? 잘못된 값을 읽은 거다.

    **Non-Repeatable Read (반복 불가능한 읽기)** → 같은 행을 두 번 읽었는데 값이 달라진 것. A가 읽고, B가 UPDATE+COMMIT, A가 다시 읽으면 값이 바뀌어 있다.

    **Phantom Read (팬텀 리드)** → 같은 조건으로 두 번 조회했는데 행 수가 달라진 것. A가 WHERE age > 20으로 조회 (5건), B가 INSERT+COMMIT, A가 다시 조회 (6건). 유령(phantom)처럼 행이 나타남.

    구분법:

    - Dirty Read: 아직 확정 안 된 값을 봄 (COMMIT 전)
    - Non-Repeatable Read: 같은 행인데 값이 바뀜 (UPDATE)
    - Phantom Read: 없던 행이 나타나거나 있던 행이 사라짐 (INSERT/DELETE)

    참고: 05장

### Q18. MVCC 설명

MVCC가 뭔지, 왜 필요한지 설명해라.

??? success "정답 보기"
    **MVCC = Multi-Version Concurrency Control (다중 버전 동시성 제어)**

    한 줄 정의: 데이터의 여러 버전을 유지해서, 읽기와 쓰기가 서로를 차단하지 않게 하는 기술

    원리:

    - UPDATE하면 원본은 Undo 로그에 보존, 새 값이 테이블에 기록
    - 각 트랜잭션은 자기 시작 시점의 "스냅샷"을 본다
    - A가 수정 중이어도 B는 A 시작 전 버전을 읽을 수 있다
    - 읽기 락 없이 일관된 읽기 가능

    왜 필요한가: MVCC 없으면 → 읽기도 락을 잡아야 함 → 동시성 떨어짐. MVCC 있으면 → 읽기는 락 없이 가능 → 동시성 높음.

    100명이 동시에 시험 결과를 조회하는데, 한 명이 UPDATE 중이라고 99명이 기다려야 한다면? → 서비스 불가능. 그래서 MVCC가 필수.

    참고: 05장

### Q19. --single-transaction의 원리

mysqldump의 `--single-transaction`이 락 없이 일관된 백업을 만드는 원리를 설명해라.

    1. 덤프 시작 시점에 REPEATABLE READ 트랜잭션을 연다
    2. MVCC 스냅샷이 생성된다 (이 시점의 데이터 버전)

    4. 읽기만 하므로 락을 잡지 않는다
    5. 다른 트랜잭션도 정상적으로 쓰기 가능

    이게 Q18(MVCC)과 직접 연결된다. MVCC를 이해해야 이 옵션이 왜 동작하는지 안다.

### Q20. 공유 락(S)과 배타 락(X)의 차이

Shared Lock(S)과 Exclusive Lock(X)의 차이를 설명하고 상호 호환성을 말해라.

??? success "정답 보기"
    **Shared Lock (S, 공유 락)** → "나 이 데이터 읽고 있어. 다른 사람도 읽어도 돼. 근데 수정은 안 돼." → `SELECT ... LOCK IN SHARE MODE`

    **Exclusive Lock (X, 배타 락)** → "나 이 데이터 수정하고 있어. 아무도 건드리지 마." → `UPDATE`, `DELETE`, `SELECT ... FOR UPDATE`

    호환성:

    | 요청 \ 보유 | S Lock | X Lock |
    |-------------|--------|--------|
    | S Lock | 허용 | 대기 |
    | X Lock | 대기 | 대기 |

    읽기+읽기 = 동시 가능 / 읽기+쓰기 = 대기 / 쓰기+쓰기 = 대기

    참고: 06장

### Q21. 메타데이터 락 설명

메타데이터 락(Metadata Lock)이 뭔지, DDL 실행 시 왜 필요한지 설명해라.

??? success "정답 보기"
    **메타데이터 락 = 테이블 구조(메타데이터)를 보호하는 락**

    왜 필요한가: A가 `SELECT * FROM tb_user WHERE name = '김철수';` 실행 중인데 B가 `ALTER TABLE tb_user DROP COLUMN name;` 실행하면? → A가 읽는 도중에 컬럼이 사라진다 → DB 정합성 붕괴

    그래서:

    - DML(SELECT, UPDATE 등) 실행 → 공유 메타데이터 락 획득 → "내가 이 테이블 구조 쓰고 있어. 구조 바꾸지 마."
    - DDL(ALTER, TRUNCATE 등) 실행 → 배타 메타데이터 락 필요 → "구조 바꿀 거야. 아무도 이 테이블 건드리지 마."

    DDL은 배타 메타데이터 락이 필요하므로: 누구라도 그 테이블에 공유 메타데이터 락을 잡고 있으면 대기. 이게 "Waiting for table metadata lock"의 정체.

    참고: 06장

### Q22. TRUNCATE의 metadata lock 대기 원인

TRUNCATE가 "Waiting for table metadata lock"에 걸리는 원인을 설명해라.

??? success "정답 보기"
    TRUNCATE = DDL = 배타 메타데이터 락 필요 → 그 테이블에 대해 열린 트랜잭션이 하나라도 있으면 대기

    **원인 시나리오:**

    1. WAS 커넥션 풀의 세션이 트랜잭션을 열었다 (BEGIN)
    2. 쿼리 실행 → 그 테이블에 공유 메타데이터 락 획득
    3. 비즈니스 로직 처리 후 커넥션을 풀에 반환
    4. 근데 COMMIT/ROLLBACK을 안 했다
    5. 커넥션 풀에서 Sleep 상태로 대기 (아무것도 안 함)
    6. 하지만 트랜잭션은 열려있고, 메타데이터 락도 잡고 있음

    → 이 상태에서 TRUNCATE 실행 → 배타 메타데이터 락 획득 시도 → Sleep 세션의 공유 락 때문에 대기 → 무한 대기 (Sleep 세션이 타임아웃되거나 KILL될 때까지)

    이게 우리가 실제로 겪은 상황이다. Sleep 2387초(40분) 세션이 범인이었다.

    참고: 06장, 13장

### Q23. Sleep 세션이 metadata lock을 잡는 이유

Sleep 상태인 세션이 어떻게 metadata lock을 잡고 있을 수 있나?

??? success "정답 보기"
    Sleep = "현재 실행 중인 쿼리가 없다" / Sleep 는 "아무것도 잡고 있지 않다"가 아니다.

    시간순으로 보면:

    - T1: 세션이 BEGIN (트랜잭션 시작)
    - T2: SELECT * FROM tb_backup; (메타데이터 락 획득)
    - T3: 쿼리 결과 반환 (쿼리 끝남 → Sleep으로 전환)
    - T4: ... (COMMIT 안 함)
    - T5: ... (계속 Sleep, 트랜잭션은 열린 상태)

    T3에서 쿼리는 끝났지만, 트랜잭션은 안 끝났다. 메타데이터 락은 트랜잭션이 끝날 때 해제된다. → Sleep인데도 락을 잡고 있는 것.

    WAS 커넥션 풀에서 흔히 발생:

    - 코드에서 COMMIT/ROLLBACK 누락
    - 예외 발생 시 finally에서 트랜잭션 정리 안 함
    - @Transactional 범위 밖에서 쿼리 실행 후 방치

    **핵심: Sleep은 "쿼리가 안 돌고 있다"이지 "락이 없다"가 아니다.**

### Q24. SHOW PROCESSLIST 위험 신호 State 3가지

SHOW PROCESSLIST에서 봤을 때 "위험하다"고 판단할 수 있는 State 3가지를 말해라.

### Q25. KILL과 KILL QUERY의 차이

KILL과 KILL QUERY의 차이를 설명해라.

??? success "정답 보기"
    **KILL [id]** → 해당 세션(커넥션) 자체를 종료. 커넥션 끊김. 실행 중이던 쿼리 중단 + 트랜잭션 ROLLBACK + 세션 종료. WAS 커넥션 풀이면 풀에서 제거 → 새 커넥션 생성

    **KILL QUERY [id]** → 해당 세션에서 현재 실행 중인 쿼리만 중단. 커넥션은 유지됨. 쿼리만 중단, 트랜잭션은 유지 (COMMIT도 ROLLBACK도 안 됨)

    언제 뭘 쓰나:

    - Sleep 세션이 metadata lock 잡고 있을 때 → KILL (세션 종료해야 트랜잭션 끝남)
    - 오래 걸리는 SELECT를 중단하고 싶을 때 → KILL QUERY
    - 확실하지 않으면 → KILL (세션 끊는 게 더 확실함)

    주의: Sleep 세션에 KILL QUERY 하면? 현재 쿼리가 없으므로 의미 없다. Sleep의 문제는 쿼리가 아니라 열린 트랜잭션이다. KILL로 세션을 끊어야 한다.

    참고: 07장

### Q26. WAS 커넥션 풀 세션을 KILL하면?

WAS 커넥션 풀의 세션을 DB에서 KILL하면 서비스에 영향이 있는가?

??? success "정답 보기"

    동작 과정:

    1. KILL 실행 → DB 측에서 커넥션 강제 종료

    5. 새 커넥션 생성 → 정상 작동

### Q29. B-Tree에서 2.7억 건 검색 시 비교 횟수

### Q30. 카디널리티 낮은 컬럼에 인덱스를 걸면 안 되는 이유

??? success "정답 보기"
    카디널리티 = 해당 컬럼의 고유한 값의 수. 성별: M, F → 카디널리티 2. 학번: 학생마다 다름 → 카디널리티 높음 (= 행 수)

    카디널리티 2인 컬럼에 인덱스를 걸면: WHERE gender = 'M' 조건으로 검색 시 → 전체 행의 약 50%를 가져와야 함 → 인덱스로 50% 위치를 찾고, 다시 원본 테이블에서 데이터 읽기 → 이 과정이 풀스캔보다 오히려 느릴 수 있음 (랜덤 I/O 발생)

    옵티마이저도 알고 있다: 카디널리티 낮은 인덱스는 무시하고 풀스캔 선택 → 인덱스 만들어놨는데 안 타는 상황

    **결론:** 인덱스는 "찾아야 할 범위가 좁을 때" 효과적. 전체의 50%를 찾아야 하면 인덱스 의미 없음. 인덱스가 효과적인 기준: 보통 전체의 10~15% 이하를 조회할 때.

    참고: 09장

### Q31. EXPLAIN에서 type=ALL의 의미

EXPLAIN 실행 결과에서 `type=ALL`이 의미하는 것은?

??? success "정답 보기"
    **type=ALL = Full Table Scan (풀스캔).** 인덱스를 하나도 사용하지 않고 테이블의 모든 행을 처음부터 끝까지 읽는다.

    EXPLAIN의 type 값 (좋은 순서): system > const > eq_ref > ref > range > index > ALL

    - const: PK나 Unique로 1건 특정 (최상)
    - eq_ref: 조인에서 PK/Unique로 1건 (좋음)
    - ref: 인덱스로 여러 건 (괜찮음)
    - range: 인덱스로 범위 검색 (괜찮음)
    - index: 인덱스 풀스캔 (아쉬움)
    - ALL: 테이블 풀스캔 (최악)

    ALL이 나오면: 해당 쿼리에 인덱스가 없거나, 있는데 옵티마이저가 안 쓰기로 결정했거나, WHERE 조건이 인덱스를 타지 못하는 형태거나.

    2.7억 건 테이블에서 type=ALL → 27분짜리 쿼리. 이게 우리 사례.

    참고: 09장

### Q32. 복합 인덱스 좌측 접두사 규칙

복합 인덱스 `(A, B, C)`에서 `WHERE B = 1 AND C = 2`로 조회하면 인덱스를 타는가?

??? success "정답 보기"
    **안 탄다.**

    이유: 복합 인덱스의 "좌측 접두사(Leftmost Prefix)" 규칙

    복합 인덱스 (A, B, C)가 동작하는 조건:

    - WHERE A = 1 → 인덱스 탐
    - WHERE A = 1 AND B = 2 → 인덱스 탐
    - WHERE A = 1 AND B = 2 AND C = 3 → 인덱스 탐 (최적)
    - WHERE B = 1 → 안 탐
    - WHERE B = 1 AND C = 2 → 안 탐
    - WHERE C = 3 → 안 탐

    왜? 인덱스는 (A, B, C) 순서로 정렬되어 있다. 전화번호부가 "성 → 이름 → 생년월일" 순이면, "이름이 철수인 사람"을 찾으려면 전부 뒤져야 한다. "성이 김씨"부터 시작해야 효율적으로 찾을 수 있다.

    참고: 09장

### Q33. DELETE 후 .ibd 파일 크기가 안 줄어드는 이유

2.7억 건을 DELETE한 후에도 .ibd 파일이 43GB인 이유를 설명해라.

??? success "정답 보기"
    **InnoDB의 DELETE는 "실제 삭제"가 아니라 "삭제 마킹"이다.**

    내부 동작:

    1. 행의 delete flag를 ON으로 설정
    2. 해당 공간은 "재사용 가능"으로 표시
    3. 하지만 .ibd 파일에서 물리적으로 제거되지는 않음
    4. 나중에 INSERT가 오면 그 공간을 재사용

    왜 파일 크기를 안 줄이나: 파일 중간의 빈 공간을 제거하려면 전체 파일을 재작성해야 함 → 이건 매우 비용이 큰 작업 → InnoDB는 자동으로 하지 않음

    공간을 실제로 줄이려면: `OPTIMIZE TABLE tb_backup;` → 테이블 재구성 (수 시간) / `ALTER TABLE tb_backup ENGINE=InnoDB;` → 동일 효과

    TRUNCATE는? 파일 자체를 삭제하고 새로 만든다 → 그래서 즉시 디스크 공간 반환

    이게 대용량 전체 삭제에 DELETE를 쓰면 안 되는 또 하나의 이유. 다 지워도 디스크 공간이 안 줄어든다.

### Q34. Buffer Pool 설명

Buffer Pool이 뭔지, 왜 필요한지 설명해라.

??? success "정답 보기"
    **Buffer Pool = InnoDB가 데이터를 메모리에 캐싱하는 영역**

    왜 필요한가: 디스크 읽기: ~10ms (밀리초) / 메모리 읽기: ~100ns (나노초) → 메모리가 디스크보다 약 100,000배 빠르다

    동작:

    1. SELECT 실행 → 먼저 Buffer Pool에서 찾는다
    2. 있으면 → 메모리에서 바로 반환 (빠름)
    3. 없으면 → 디스크에서 읽어서 Buffer Pool에 올리고 반환
    4. 자주 쓰는 데이터는 Buffer Pool에 계속 남아있음 (LRU)

    크기: 보통 전체 메모리의 60~80%를 Buffer Pool에 할당. 서버 메모리 8GB면 → Buffer Pool 5~6GB. Buffer Pool이 클수록 디스크 접근이 줄어 → 성능 향상.

    TRUNCATE 후 Buffer Pool: 해당 테이블의 캐시 페이지도 무효화됨. 다음 조회 시 디스크에서 새로 읽어야 함. 빈 테이블이므로 거의 영향 없음.

    참고: 10장

### Q40. find -ctime +3의 동작

`find $BACKUP_DIR -ctime +3 -exec rm -f {} \;`에서 3일 전 파일이 바로 안 지워지는 이유를 설명해라.

??? success "정답 보기"
    `-ctime +3` = "ctime이 3일(72시간)보다 오래된 파일"

    여기서 "3일"의 기준이 함정이다: `-ctime +3` → 3 x 24시간 = 72시간보다 오래된 것. 정확히 3일(72시간) 전 파일은 포함 안 됨. 3일 + a 되어야 포함됨.

    find의 시간 계산: `+3` = 4일 이상 전 (3을 초과해야 함) / `3` = 정확히 3일 전 (3.0 ~ 3.999...일) / `-3` = 3일 이내

    그래서:

    - 오늘 만든 파일: ctime 0일 → +3 미해당 (유지)
    - 1일 전: ctime 1일 → +3 미해당 (유지)
    - 2일 전: ctime 2일 → +3 미해당 (유지)
    - 3일 전: ctime 3일 → +3 미해당 **(유지!)** ← 여기가 함정
    - 4일 전: ctime 4일 → +3 해당 (삭제)

    실제로는 4일 이상 된 파일부터 지워진다. "3일 보관"이 아니라 "4일 보관"이 되는 셈.

    참고: 11장, 12장

TRUNCATE 실행 후 `table_rows=0`, `data_mb=0.02`가 의미하는 것은?

??? success "정답 보기"
    **table_rows = 0** → 테이블에 데이터가 0건. TRUNCATE가 정상적으로 모든 데이터를 삭제했다.

    **data_mb = 0.02 (약 20KB)** → 빈 .ibd 파일의 초기 크기. InnoDB는 빈 테이블이라도 최소한의 메타데이터를 가진 .ibd 파일을 유지. 테이블 정의, 세그먼트 헤더 등의 기본 구조. 이건 정상이다. 0.00이 되는 게 아니라 0.02가 정상.

    TRUNCATE 전: 43,010MB (43GB) → TRUNCATE 후: 0.02MB (20KB) → ~43GB 확보 → 디스크 72% → 29%

    참고: 08장, 13장

### Q43. RENAME TABLE 전략

RENAME TABLE 전략의 장점 3가지와 단점 2가지를 말해라.

??? success "정답 보기"
    RENAME TABLE 전략: 1) `RENAME TABLE tb_backup TO tb_backup_20260227;` (기존 → 보존) → 2) `CREATE TABLE tb_backup LIKE tb_backup_20260227;` (빈 테이블 생성) → 3) `TRUNCATE TABLE tb_backup_20260227;` (보존 테이블 삭제)

    **장점:**

    1. **서비스 무중단** → RENAME은 메타데이터만 변경 (밀리초). 서비스는 새 빈 테이블을 사용 → 정상 작동. TRUNCATE는 보존 테이블에 대해 실행 → 서비스 영향 없음
    2. **안전한 롤백 가능** → TRUNCATE 전까지 원본 데이터가 보존되어 있음. "아 잘못했다" → RENAME으로 원복 가능
    3. **작업 시간 분리** → RENAME (순간) + CREATE (순간) → 서비스 즉시 복구. TRUNCATE는 여유 있을 때 실행 (수 분). 긴급 조치와 후속 작업 분리 가능

    **단점:**

    1. **디스크 공간 일시적 증가** → TRUNCATE 전까지 원본 43GB가 그대로 존재. 디스크 여유가 부족하면 TRUNCATE도 못 할 수 있음
    2. **참조 관계 주의** → 다른 테이블이 FK로 참조하면 RENAME 실패. 뷰, 프로시저에서 테이블명 하드코딩되어 있으면 문제. 코드에서도 테이블명 직접 참조 확인 필요

    참고: 13장

### Q44. "서비스 복구 우선, 원인 분석 나중에"인 이유

장애 대응에서 "서비스 복구 우선, 원인 분석 나중에"인 이유를 말해라.

??? success "정답 보기"
    **이유 1: 사용자 피해의 시간 비례성.** 서비스 1분 죽어있으면 → 영향 받는 사용자 수 = 접속 시도한 사람 전원. 서비스 1시간 죽어있으면 → 민원, 업무 마비, 신뢰도 하락. 원인 분석에 2시간 쓰면 → 서비스 2시간 죽어있는 것. 대부분의 원인 분석은 서비스가 살아있어도 할 수 있다. → 먼저 살리고, 살아있는 상태에서 분석.

    **이유 2: 원인을 모르고도 복구할 수 있다.** 예: "DB가 느리다" → 문제 쿼리 KILL → 일시적 복구 / WAS 재시작 → 커넥션 풀 초기화. 이건 완벽한 해결이 아니지만, "서비스가 돌아가는 상태"는 확보했다. 완벽한 해결은 시간을 들여서.

    **이유 3: 판단력 저하 방지.** 장애 중에는 압박감, 시간 압박으로 판단력이 떨어진다. 그 상태에서 근본 원인을 찾으려 하면 실수할 확률이 높다. 임시 조치로 숨 돌릴 시간을 확보하고, 차분한 상태에서 분석하는 게 정확도가 높다.

    참고: 13장

### Q45. 백업 전략 설계

다음 조건에서 백업 전략을 설계해라.

```
DB 크기: 50GB
RPO: 1시간 (최대 1시간분의 데이터 손실 허용)
RTO: 4시간 (4시간 내 복구)
디스크 여유: 130GB
서비스: 24/7 무중단
```

??? success "정답 보기"
    **1. 풀 백업: 매일 1회 (새벽 3시).** `mysqldump --single-transaction --routines --triggers` → 24/7 서비스이므로 --single-transaction 필수 (락 없이 덤프). 예상 50GB → gzip 압축 → ~10GB. 디스크 130GB에 10GB x 7일 보관 = 70GB 사용 → 충분

    **2. 바이너리 로그: 항상 활성화.** RPO 1시간 → 풀 백업만으로는 최대 24시간 손실. 바이너리 로그로 풀 백업 이후의 변경분 보관. 풀 백업 + 바이너리 로그 = 임의 시점 복구(PITR) 가능. 바이너리 로그 보관: 최소 7일

    **3. 복구 절차 (RTO 4시간):** Step 1: 가장 최근 풀 백업 복원 (~2시간, 50GB) → Step 2: 바이너리 로그 적용 (풀 백업 이후 ~ 장애 시점) (~1시간) → Step 3: 검증 (~1시간) → 총 ~4시간 → RTO 충족

    **4. 검증:** 매주 1회 백업 복원 테스트 (별도 서버). 백업 파일 크기 모니터링 (0바이트 감지). 백업 성공/실패 알림.

    **5. 보관 정책:** 풀 백업: 7일 보관 (10GB x 7 = 70GB) / 바이너리 로그: 7일 보관 (~20GB) / 합계: ~90GB < 130GB → 여유 있음

    참고: 11장

### Q46. 데드락 시나리오

데드락이 발생하는 두 트랜잭션 시나리오를 작성하고, 방지법을 말해라.

??? success "정답 보기"
    **시나리오:**

    ```
    트랜잭션 A:
      BEGIN;
      UPDATE tb_student SET score = 95 WHERE id = 1;  -- id=1에 X Lock
      UPDATE tb_student SET score = 88 WHERE id = 2;  -- id=2에 X Lock 시도 → 대기!

    트랜잭션 B:
      BEGIN;
      UPDATE tb_student SET score = 77 WHERE id = 2;  -- id=2에 X Lock
      UPDATE tb_student SET score = 60 WHERE id = 1;  -- id=1에 X Lock 시도 → 대기!
    ```

    A: id=1 잠금, id=2 기다림 / B: id=2 잠금, id=1 기다림 → 서로 상대방을 기다림 → 영원히 끝나지 않음 → **데드락**

    InnoDB 대응: 데드락 감지 (deadlock detector) → 비용이 낮은 트랜잭션을 ROLLBACK (victim 선택) → `ERROR 1213: Deadlock found when trying to get lock`

    **방지법:**

    1. 접근 순서 통일: 항상 id 오름차순으로 접근 → A도 1→2, B도 1→2 → 데드락 불가
    2. 트랜잭션 짧게: 락 보유 시간 최소화
    3. 한 번에 필요한 락 다 잡기: SELECT ... FOR UPDATE
    4. 인덱스 사용: 행 단위 락 → 충돌 범위 최소화

    참고: 06장

### Q47. jQuery 이벤트 핸들러 중복 등록 수정법

jQuery 이벤트 핸들러 중복 등록(BUG-1) 문제의 수정 방법과 그 이유를 설명해라.

??? success "정답 보기"
    **수정 방법 1: .off().on() 패턴**

    ```javascript
    function initGradeReview() {
        $("#btnRegrade").off("click").on("click", function() {
            $.ajax({ url: "/exam/regrade", ... });
        });
    }
    ```

    .off()으로 기존 핸들러 제거 → .on()으로 새로 등록 → 항상 1개만 유지. 단점: off()이 해당 요소의 모든 click 핸들러를 지움.

    **수정 방법 2: 네임스페이스**

    ```javascript
    function initGradeReview() {
        $("#btnRegrade").off("click.regrade").on("click.regrade", function() {
            $.ajax({ url: "/exam/regrade", ... });
        });
    }
    ```

    .regrade 네임스페이스로 특정 핸들러만 제거/등록. 다른 click 핸들러에 영향 없음. 더 안전한 방법.

    **수정 방법 3: 이벤트 위임 + 플래그**

    ```javascript
    var isInitialized = false;
    function initGradeReview() {
        if (isInitialized) return;
        isInitialized = true;
        $(document).on("click", "#btnRegrade", function() { ... });
    }
    ```

    최초 1번만 등록. 동적으로 생성되는 요소에도 대응 가능.

    핵심은 "같은 핸들러가 중복 등록되지 않게" 하는 것. 방법 1: 제거 후 등록 (가장 단순) / 방법 2: 네임스페이스로 정밀 제어 (가장 안전) / 방법 3: 최초 1회만 등록 (구조적 해결)

    참고: 13장

### Q48. 루프 내 전체 백업 호출 수정법

루프 내 전체 백업 호출(BUG-2) 문제의 수정 방법과 폭발 계산을 설명해라.

??? success "정답 보기"
    **문제 코드:** `for (학생 100명) { backupAllStudentPapers(examCd); }` → INSERT: 100 x (100 x 50) = 500,000건

    **수정 코드 1:** `backupAllStudentPapers(examCd);` 를 루프 밖으로 이동 → INSERT: 1 x (100 x 50) = 5,000건

    **수정 코드 2:** 개별 백업으로 변경 → `backupStudentPaper(s.getNo(), examCd);` 를 루프 내에서 호출 → INSERT: 100 x 50 = 5,000건

    | 방식 | 계산 | INSERT |
    |------|------|--------|
    | 버그 (루프 내 전체) | 100 x 100 x 50 | 500,000 |
    | 수정1 (루프 밖 전체) | 1 x 100 x 50 | 5,000 |

### Q49. 연쇄 장애 각 단계

코드 버그 → 데이터 폭발 → 디스크 부족 → 백업 실패 연쇄의 각 단계를 한 문장씩 설명해라.

??? success "정답 보기"
    **1단계: 코드 버그** → jQuery 핸들러 중복 등록과 루프 내 전체 백업 호출로 인해, 재채점 1회에 정상의 500배 데이터가 INSERT된다.

    **2단계: 데이터 폭발** → 14개월간 삭제 정책 없이 축적된 결과, 백업 테이블이 2.7억 건 43GB까지 비정상 증가한다.

    **3단계: 디스크 부족** → 43GB 테이블이 전체 디스크의 86%를 차지하여 디스크 사용률이 72%에 도달하고 여유 공간이 53GB로 줄어든다.

    **4단계: 백업 실패** → mysqldump 예상 출력 73GB가 디스크 여유 53GB를 초과하여 매일 백업이 Disk Full로 실패하고, 14개월간 유효한 백업이 없는 상태가 된다.

    **연쇄의 핵심:** 각 단계가 다음 단계의 원인이 된다. 1단계를 안 잡으면 2단계는 필연적이고, 2단계를 안 잡으면 3단계는 시간문제이고, 3단계를 안 잡으면 4단계는 매일 반복된다. 어느 한 단계에서라도 감지했으면 여기까지 안 왔다.

    참고: 13장 (전체)

### Q50. 전체 학습 내용 한 문장 요약

이 전체 학습 내용(01~13장)을 한 문장으로 요약해봐.

??? success "정답 보기"
    정답은 하나가 아니다. 핵심을 담고 있으면 된다.

    예시 답안: "DB는 코드와 분리된 게 아니라 코드의 결과물이며, 코드 한 줄의 실수가 43GB 장애가 되기 전에 모니터링으로 감지하고, 원리를 이해하고, 대응할 수 있는 것이 DB를 '안다'는 것이다."

    - "코드가 DB를 만들고, DB가 서비스를 먹이고, 모니터링이 이 둘을 지킨다."
