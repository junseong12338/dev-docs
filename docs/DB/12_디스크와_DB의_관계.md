# 12. 디스크와 DB의 관계

---

## 1. DB 데이터는 어디에 저장되나?

DB가 마법처럼 데이터를 기억하는 거 아니다.
**전부 디스크 위의 파일이다.** 이걸 먼저 뼈에 새겨.

### 1.1 디렉토리 구조

```
MariaDB/MySQL 데이터 저장 위치 (우리 서버 기준):

/data/mysql/                        ← datadir (my.cnf에서 설정)
├── ibdata1                         ← InnoDB 시스템 테이블스페이스
├── ib_logfile0                     ← InnoDB Redo 로그 1
├── ib_logfile1                     ← InnoDB Redo 로그 2
├── mysql-bin.000001                ← 바이너리 로그 1
├── mysql-bin.000002                ← 바이너리 로그 2
├── mysql-bin.index                 ← 바이너리 로그 인덱스
├── mysql/                          ← mysql 시스템 DB
│   ├── user.frm                    ← 테이블 구조 정의
│   ├── user.ibd                    ← 테이블 데이터 (InnoDB)
│   └── ...
├── lxpknu10/                       ← 우리 DB 스키마
│   ├── tb_user.frm                 ← tb_user 테이블 구조
│   ├── tb_user.ibd                 ← tb_user 테이블 데이터
│   ├── tb_course.frm
│   ├── tb_course.ibd
│   ├── tb_lms_exam_stare_paper_backup.frm    ← 문제의 테이블 구조
│   ├── tb_lms_exam_stare_paper_backup.ibd    ← 문제의 테이블 데이터 (43GB!)
│   └── ...
└── performance_schema/
    └── ...
```

**파일별 역할:**

| 파일 | 역할 |
|------|------|
| `.frm` | 테이블 구조 정의 (컬럼, 타입, 인덱스). CREATE TABLE의 결과물. 수 KB 수준 |
| `.ibd` | InnoDB 테이블 데이터 + 인덱스. 실제 데이터가 여기 있다. innodb_file_per_table=ON일 때. 43GB짜리 .ibd가 문제의 원흉 |
| `ibdata1` | InnoDB 시스템 테이블스페이스. Undo 로그, 데이터 딕셔너리 등. innodb_file_per_table=OFF면 모든 데이터가 여기 들어감 |
| `ib_logfile0/1` | InnoDB Redo 로그. crash recovery에 사용. 10장에서 배운 것 |
| `mysql-bin.*` | 바이너리 로그 (Binary Log). 모든 데이터 변경 기록. 복제(Replication) 및 Point-in-Time Recovery에 사용 |

### 1.2 디스크 확인 명령어 3총사

```bash
# ─── 1. df -h: 파티션별 전체 사용량 ───
df -h

# ─── 2. du -sh: 디렉토리별 크기 ───
du -sh /data/mysql/lxpknu10/

# ─── 3. ls -lh: 파일별 크기 ───
ls -lh /data/mysql/lxpknu10/
```

**비유로 이해하기:**

!!! example "비유로 이해하기"
    | 비유 | 실제 | 확인 명령어 |
    |------|------|-----------|
    | 창고 | 디스크 | - |
    | 창고 안의 방 | 파티션 | `df -h` (각 방에 공간이 얼마나 남았지?) |
    | 방 안의 선반 | 디렉토리 | `du -sh` (이 선반에 상자가 얼마나 쌓여있지?) |
    | 선반 위의 상자 | 파일 | `ls -lh` (이 상자 크기가 얼마지?) |

## 2. df -h 읽는 법

### 2.1 실제 출력 예시

```
$ df -h

Filesystem                     Size  Used  Avail  Use%  Mounted on
/dev/mapper/VolGroup00-lv00     50G   12G    35G   26%   /
tmpfs                          7.8G     0   7.8G    0%   /dev/shm
/dev/sda1                      477M  130M   318M   30%   /boot
/dev/mapper/VolGroup01-lv01    196G  134G    53G   72%   /data
```

### 2.2 각 컬럼 의미

| 컬럼 | 의미 |
|------|------|
| Filesystem | 파일시스템 장치명. `/dev/mapper/...` = LVM (Logical Volume), `/dev/sda1` = 물리 디스크 첫 번째 파티션 |
| Size | 전체 크기. 196G = 196GB |
| Used | 사용 중인 크기. 134G = 134GB 사용 중 |
| Avail | 사용 가능한 크기. 53G = 53GB 여유. **주의: Size - Used ≠ Avail (예약 공간)** |
| Use% | 사용률. 72% = 위험 수준 진입 |
| Mounted on | 마운트 포인트 (이 디스크가 붙은 경로). `/data` = DB 데이터가 여기 있다 |

### 2.3 Size - Used ≠ Avail?

!!! note "196G - 134G = 62G인데 왜 Avail은 53G?"
    리눅스 파일시스템(ext4)은 전체의 약 5%를 예약한다. 196G x 5% = 약 10G 예약. 196G - 10G(예약) - 134G(사용) = 약 52G = 53G(반올림).

    **왜 예약해?** 디스크가 100% 차면 root도 파일 못 만듦. 예약 공간은 root만 사용 가능. 비상시 로그 파일 삭제 등 복구 작업용. `tune2fs -m` 옵션으로 예약 비율 조정 가능 (기본 5%).

### 2.4 Mount Point란

!!! note "Mount Point = 물리 디스크를 특정 경로에 연결"
    **비유:** USB를 컴퓨터에 꽂으면 D: 드라이브로 보이잖아. 리눅스에서는 USB를 /mnt/usb에 "마운트"한다.

    | 마운트 포인트 | 크기 | 용도 |
    |-------------|------|------|
    | / (루트) | 50GB | OS, 프로그램 |
    | /boot | 477MB | 부팅 파일 |
    | /data | 196GB | DB 데이터 전용 |

    **DB 데이터를 /data에 분리한 이유:** OS랑 DB 디스크를 분리하면 OS 디스크 가득 차도 DB 안전. DB 디스크 가득 차도 OS는 동작 → SSH 접속 가능 → 조치 가능. 만약 같은 디스크에 있으면? DB가 디스크 꽉 채움 → OS도 동작 불능 → SSH 접속도 안 됨 → 원격 복구 불가 → IDC 가야 함.

## 3. Disk Full이 발생하면?

### 3.1 연쇄 장애 시나리오

디스크가 가득 차면 하나만 안 되는 게 아니다. **연쇄적으로 죽는다.**

!!! danger "Disk Full 연쇄 장애 타임라인"
    | 시점 | 현상 |
    |------|------|
    | T+0초 | 바이너리 로그(binlog) 기록 실패. 에러: "Disk is full writing mysql-bin.XXXXXX". 모든 DML(INSERT, UPDATE, DELETE) 실패 시작 |
    | T+0초 | INSERT/UPDATE 쿼리 에러 반환. 에러: "No space left on device (Errcode: 28)". 사용자 화면에 에러 페이지 |
    | T+1분 | 쿼리 타임아웃 증가. 실패한 트랜잭션이 계속 재시도. DB 커넥션이 쌓이기 시작 |
    | T+5분 | 커넥션 풀 고갈. DB 커넥션 전부 사용 중 (반환 안 됨). 서비스 응답 불가 |
    | T+10분 | InnoDB 내부 장애. Redo 로그 기록 실패. SELECT는 되지만 INSERT/UPDATE/DELETE 전면 중단 |
    | T+30분 | 에러 로그 폭주. 디스크 가득 참 → 에러 로그도 기록 불가 → 진단 자체가 불가능 |
    | T+?? | mysqldump 실패. 0바이트 파일 생성. 우리 서버: 이 상태가 14개월 |

### 3.2 에러 메시지

이 에러를 보면 즉시 디스크 확인해야 한다.

!!! danger "에러 로그에서 볼 수 있는 메시지들"
    1. **바이너리 로그 관련:** `[ERROR] Disk is full writing './mysql-bin.000123' (Errcode: 28 "No space left on device")`
    2. **InnoDB 관련:** `[ERROR] InnoDB: Write to file ./ibdata1 failed at offset XXXXX, YYYY bytes should have been written, only 0 were written.`
    3. **일반 쿼리:** `ERROR 1021 (HY000): Disk full (/data/mysql/tmp/#sql_XXXX); waiting for someone to free some space...`
    4. **임시 테이블:** `ERROR 3 (HY000): Error writing file '/tmp/MYxxxxx' (Errcode: 28 - No space left on device)`

**Errcode: 28 = "No space left on device"**
이 번호를 기억해. 28이 보이면 디스크가 꽉 찬 거다.

### 3.3 실전: 229건의 Disk full 에러

!!! example "229건의 Disk full 에러 분석"
    **기간:** 2024-06-21 ~ 2026-02-27 (약 14개월, 616일) / **에러 수:** 229건

    **왜 매일 발생하지 않고 229건?** 디스크가 항상 100%는 아니었음. 일시적으로 여유가 생기면 binlog 기록 성공. 다시 차면 실패. 그래서 간헐적으로 229건.

    | 작업 | 필요 공간 | 결과 |
    |------|----------|------|
    | binlog | 수 MB 단위 기록 | 간헐적 성공/실패 |
    | mysqldump | 73GB 연속 기록 | 항상 실패 (53GB 여유로 절대 불가) |

## 4. 디스크 용량 관리

### 4.1 누가 디스크를 먹는가

디스크를 잡아먹는 주범 4가지:

| # | 항목 | 우리 서버 사용량 | 비율 |
|---|------|-----------------|------|
| 1 | DB 데이터 파일 (.ibd) (정상 ~7GB + 비정상 43GB) | ~90GB (43GB가 주범) | 67% |
| 2 | 바이너리 로그 (mysql-bin) | ~20GB (추정) | 15% |
| 3 | 백업 파일 (tar.gz) | 0 (전부 0바이트) | 0% |
| 4 | 로그 파일 (에러, 슬로우) | ~10GB (추정) | 7% |
| - | 기타 (OS, 임시파일 등) | ~14GB | 11% |
| - | **합계** | **134GB** | **100%** |

!!! danger "주범: .ibd 파일 하나 (43GB)"
    - 전체 사용량의 32%
    - 이거 하나 때문에 백업 불가
    - 이거 하나 때문에 디스크 경고
    - 이거 하나가 14개월의 재앙

### 4.2 용량 확인 방법

#### DB 레벨: information_schema로 확인

```sql
-- 테이블별 용량 확인 (08장에서 배운 것)
SELECT
    TABLE_NAME,
    ROUND(DATA_LENGTH / 1024 / 1024 / 1024, 2) AS data_gb,
    ROUND(INDEX_LENGTH / 1024 / 1024 / 1024, 2) AS index_gb,
    ROUND((DATA_LENGTH + INDEX_LENGTH) / 1024 / 1024 / 1024, 2) AS total_gb,
    TABLE_ROWS
FROM information_schema.TABLES
WHERE TABLE_SCHEMA = 'lxpknu10'
ORDER BY DATA_LENGTH DESC
LIMIT 10;

-- 결과 예시 (TRUNCATE 전):
-- +------------------------------------------+---------+----------+----------+-------------+
-- | TABLE_NAME                                | data_gb | index_gb | total_gb | TABLE_ROWS  |
-- +------------------------------------------+---------+----------+----------+-------------+
-- | tb_lms_exam_stare_paper_backup           |   43.21 |     0.00 |    43.21 | 276,490,000 |
-- | tb_lms_progress                           |    1.50 |     0.80 |     2.30 |   5,230,000 |
-- | tb_lms_exam_result                        |    0.70 |     0.30 |     1.00 |   2,100,000 |
-- | ...                                       |         |          |          |             |
-- +------------------------------------------+---------+----------+----------+-------------+

-- 스키마 전체 용량
SELECT
    TABLE_SCHEMA,
    ROUND(SUM(DATA_LENGTH + INDEX_LENGTH) / 1024 / 1024 / 1024, 2) AS total_gb,
    COUNT(*) AS table_count
FROM information_schema.TABLES
WHERE TABLE_SCHEMA = 'lxpknu10'
GROUP BY TABLE_SCHEMA;
```

#### OS 레벨: du, ls로 확인

```bash
# 스키마 디렉토리 전체 크기
du -sh /data/mysql/lxpknu10/
# → 90G    /data/mysql/lxpknu10/     (TRUNCATE 전)
# → 7.2G   /data/mysql/lxpknu10/     (TRUNCATE 후)

# 데이터 디렉토리 하위 디렉토리별 크기
du -sh /data/mysql/*/
# → 90G   /data/mysql/lxpknu10/
# → 15M   /data/mysql/mysql/
# → 680K  /data/mysql/performance_schema/

# 특정 파일 크기 확인
ls -lh /data/mysql/lxpknu10/tb_lms_exam_stare_paper_backup.ibd
# → -rw-rw---- 1 mysql mysql 43G ... tb_lms_exam_stare_paper_backup.ibd

# 가장 큰 파일 찾기
ls -lhS /data/mysql/lxpknu10/ | head -10
```

#### SQL로 확인하는 것과 OS로 확인하는 것의 차이

| information_schema | du / ls |
|--------------------|---------|
| DB가 인식하는 크기 | 실제 디스크 사용량 |
| 통계 기반 (추정치) | 파일시스템 기준 |
| ANALYZE TABLE 필요 | 실시간 정확값 |
| 원격 접속으로 가능 | SSH 접속 필요 |
| 테이블 단위 상세 | 파일 단위 |

!!! tip "두 가지가 크게 다르면?"
    - ibdata1에 데이터가 섞여 있거나
    - 파일 삭제 후 공간 미반환(DELETE vs TRUNCATE 차이)
    - 03장에서 배운 것: DELETE는 파일 크기 안 줄어듦!

### 4.3 용량 확보 방법

!!! warning "방법 1: 불필요한 데이터 정리"
    - **DROP TABLE**: 테이블 자체 삭제 → .frm + .ibd 즉시 삭제
    - **TRUNCATE TABLE**: 데이터 전부 삭제 → .ibd 재생성 (즉시 반환)
    - **DELETE**: 데이터 삭제하지만 .ibd 크기 안 줄어듦!

    03장 복습: DELETE는 디스크 공간을 반환하지 않는다.
    "마킹만 하고 빈자리로 남겨둠" → 43GB를 DELETE해도 43GB 그대로.
    OPTIMIZE TABLE 해야 공간 반환 (시간 오래 걸림). 그래서 TRUNCATE를 쓴 거다.

!!! tip "추가 용량 확보 방법"
    - **방법 2: 바이너리 로그 정리** -- 아래 5장에서 상세
    - **방법 3: 오래된 백업 파일 삭제** -- 우리 서버는 0바이트라서 해당 없음 (웃픈 상황)
    - **방법 4: 에러 로그 로테이션** -- 에러 로그가 수 GB까지 커질 수 있음. logrotate 설정 또는 수동 정리
    - **방법 5: 임시 파일 정리** -- /data/mysql/tmp/ 또는 /tmp/. 대용량 JOIN/SORT 시 임시 파일 생성됨

## 5. 바이너리 로그 관리

### 5.1 바이너리 로그란

!!! note "바이너리 로그 (Binary Log, binlog)"
    DB에서 발생한 모든 데이터 변경(INSERT, UPDATE, DELETE)을 기록하는 로그. 파일명: `mysql-bin.000001`, `mysql-bin.000002`, ... 위치: `/data/mysql/`

    | 용도 | 설명 |
    |------|------|
    | 복제 (Replication) | Master의 binlog를 Slave가 읽어서 동기화. Master-Slave 구조의 핵심 |
    | Point-in-Time Recovery (PITR) | 백업 시점 + binlog = 장애 직전까지 복구. RPO를 "수 분"으로 줄이는 핵심 |
    | 감사 (Audit) | 누가 언제 무슨 데이터를 변경했는지 추적 |

    **비유:** DB가 은행이면, binlog는 거래 내역서. 거래 내역서가 있으면 어느 시점으로든 잔액을 계산할 수 있다.

### 5.2 왜 디스크를 먹나?

!!! warning "binlog는 계속 쌓인다"
    매일 INSERT/UPDATE/DELETE가 발생하면 매일 binlog 파일이 생성됨. 크기: 일 수백 MB ~ 수 GB. 안 지우면? 무한히 증가.

    **예시:** mysql-bin.000001 ~ 000365, 각 500MB. 1년이면 약 180GB → 디스크 꽉 참.

### 5.3 자동 삭제 설정

```sql
-- 현재 설정 확인
SHOW VARIABLES LIKE 'expire_logs_days';

-- 설정 변경 (7일 보관)
SET GLOBAL expire_logs_days = 7;

-- my.cnf에 영구 설정
-- [mysqld]
-- expire_logs_days = 7

-- MariaDB 10.1.6+ / MySQL 8.0+
SHOW VARIABLES LIKE 'binlog_expire_logs_seconds';
-- → 초 단위로 설정 가능
-- → 604800 = 7일 (7 × 24 × 60 × 60)
```

### 5.4 수동 삭제

```sql
-- 특정 날짜 이전 binlog 삭제
PURGE BINARY LOGS BEFORE '2026-01-01 00:00:00';

-- 특정 파일 이전까지 삭제
PURGE BINARY LOGS TO 'mysql-bin.000100';

-- 현재 binlog 파일 목록 확인
SHOW BINARY LOGS;

-- 결과:
-- +------------------+-----------+
-- | Log_name         | File_size |
-- +------------------+-----------+
-- | mysql-bin.000095 | 536870912 |
-- | mysql-bin.000096 | 536870912 |
-- | mysql-bin.000097 | 268435456 |
-- +------------------+-----------+
```

**절대 하면 안 되는 것:**

!!! danger "절대 하면 안 되는 것"
    `rm -f /data/mysql/mysql-bin.*` -- 파일은 지워지지만 MariaDB는 모른다. mysql-bin.index 파일과 불일치. DB 재시작 시 에러. **반드시 PURGE 명령으로 삭제할 것!**

## 6. 모니터링

### 6.1 디스크 사용량 임계치

| 사용률 | 상태 및 조치 |
|--------|-------------|
| 0~60% | 정상. 모니터링만. |
| 60~70% | 주의. 증가 추이 확인. "며칠 후에 80% 넘을까?" |
| 70~80% | 경고. 용량 확보 계획 수립. 우리 서버가 72%였을 때 이미 늦었다. mysqldump가 73GB 필요했으니까. |
| 80~90% | 긴급. 즉시 용량 확보. binlog purge, 불필요 로그/테이블 정리 |
| 90~95% | 위험. 서비스 장애 임박. 즉시 조치. 새벽이라도 대응. |
| 95~100% | 장애. Disk Full 발생 중. INSERT/UPDATE 실패. 연쇄 장애 시나리오 진행 중 |

**임계치는 절대적인 게 아니다:**

!!! warning "임계치는 절대적인 게 아니다"
    단순히 "80%면 경고"가 아니라, "남은 공간으로 필요한 작업을 할 수 있느냐"가 핵심.

    | 사례 | 디스크 | 사용률 | 여유 | 상태 |
    |------|--------|-------|------|------|
    | Case A | 1TB | 80% | 200GB | 여유로움 |
    | Case B | 196GB | 72% | 53GB | 73GB 덤프 불가! |

    우리 서버는 72%인데 이미 백업 불가 상태였다. 퍼센트가 아니라 "절대 용량"과 "필요 용량"을 비교해야 한다.

### 6.2 백업 파일 크기 모니터링

```bash
# 간단한 모니터링 스크립트

#!/bin/bash
BACKUP_DIR="/backup/mysql"
TODAY=$(date +%Y%m%d)
LOG="/var/log/backup_monitor.log"

# 오늘 백업 파일 확인
BACKUP_FILE=$(ls -t $BACKUP_DIR/*.gz 2>/dev/null | head -1)

if [ -z "$BACKUP_FILE" ]; then
    echo "[$(date)] ALERT: 백업 파일 없음!" >> $LOG
elif [ ! -s "$BACKUP_FILE" ]; then
    echo "[$(date)] ALERT: 백업 파일 0바이트! $BACKUP_FILE" >> $LOG
else
    SIZE=$(ls -lh "$BACKUP_FILE" | awk '{print $5}')
    echo "[$(date)] OK: $BACKUP_FILE ($SIZE)" >> $LOG
fi
```

**이 스크립트가 14개월 전에 있었다면:**

!!! tip "이 스크립트가 14개월 전에 있었다면"
    `[2024-06-21] ALERT: 백업 파일 0바이트!` → 첫날에 발견 → 즉시 조치 → 14개월의 무방비 상태 없었을 것.

### 6.3 모니터링의 핵심

!!! note "모니터링 3원칙"
    1. **자동으로 돌아가야 한다**
        - 사람이 매일 확인하는 건 3일이면 까먹는다
        - crontab + 스크립트로 자동화

    2. **이상 시 알림이 와야 한다**
        - 로그만 쌓으면 안 봄 (우리 서버 증명)
        - 이메일, 슬랙, SMS 등으로 알림

    3. **알림에 대응 절차가 있어야 한다**
        - "디스크 80%입니다" 알림 → 뭘 해야 하는지 문서화
        - 대응 절차 없으면 알림 받아도 멍때림

## 8. 핵심 정리

!!! abstract "12장 핵심 정리"
    1. **DB 데이터는 디스크 위의 파일이다** -- .ibd = 데이터, .frm = 구조, mysql-bin = binlog. 마법이 아니다. 파일이다.
    2. **df -h로 "이 작업을 할 공간이 있는가"를 먼저 확인** -- 퍼센트가 아니라 절대 용량과 필요 용량을 비교. 72%라도 53GB 여유면 충분할 수 있고, 72%라도 73GB 필요하면 부족하다.
    3. **Disk Full = 연쇄 장애** -- binlog 실패 → DML 실패 → 커넥션 고갈 → 서비스 중단. 디스크 하나가 전부를 멈춘다.
    4. **디스크를 먹는 4대 주범** -- 데이터 파일, binlog, 백업 파일, 로그 파일. 하나만 관리 안 해도 디스크 터진다.
    5. **DELETE는 디스크 공간을 반환하지 않는다** -- TRUNCATE 또는 DROP으로 즉시 반환. 03장의 핵심이 여기서 다시 등장.
    6. **binlog는 반드시 자동 삭제 설정할 것** -- expire_logs_days 미설정 = 무한 증가. rm으로 삭제하면 안 됨 → PURGE 명령 사용.
    7. **모니터링 = 자동화 + 알림 + 대응 절차** -- 하나라도 빠지면 14개월 방치된다.
    8. **코드 → DB → 디스크 → 백업 → 복구** -- 이 체인 전체를 이해해야 한다. 한 곳이 끊기면 전부 끊긴다.
