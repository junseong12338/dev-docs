# 01. DB란 무엇인가

## 1. 가장 기초부터: 데이터를 어디에 저장하는가

### 1.1 파일에 저장하면 안 되나?

안 되는 건 아니다. 실제로 초창기에는 파일에 저장했다.

```
학생정보.txt:
홍길동,20,컴퓨터공학과
김철수,21,경영학과
이영희,22,국문학과
```

이게 10명이면 괜찮다. 근데 100만 명이면?

!!! danger "파일 저장의 5가지 문제"
    **문제 1: 검색**
    → "김철수" 찾으려면 100만 줄 다 읽어야 함
    → 시간: 파일 크기에 비례 (O(n))

    **문제 2: 동시 접근**
    → A가 파일 수정 중인데 B도 수정하면?
    → 둘 중 하나가 덮어씀 → 데이터 손실

    **문제 3: 무결성**
    → 나이에 "스물하나"를 넣어도 막을 수 없음
    → 데이터 형식 보장 불가

    **문제 4: 관계**
    → 학생의 수강 과목을 같이 저장하려면?
    → 파일 구조가 복잡해지고 중복 폭발

    **문제 5: 복구**
    → 파일이 깨지면? 반쯤 쓰다가 서버 죽으면?
    → 복구 불가

이 문제들을 해결하려고 만든 게 **데이터베이스(Database)**다.

### 1.2 데이터베이스의 정의

!!! note "정의"
    **데이터베이스(Database)** = 구조화된 데이터의 집합

    **DBMS(Database Management System)** = 데이터베이스를 관리하는 소프트웨어

파일과의 결정적 차이:

| 항목 | 파일 | 데이터베이스 |
|------|------|-------------|
| 검색 | 순차 탐색 (느림) | 인덱스 (빠름) |
| 동시 접근 | 충돌/덮어쓰기 | 락/트랜잭션으로 안전 |
| 무결성 | 보장 안 됨 | 타입/제약조건으로 보장 |
| 관계 표현 | 어려움 | JOIN으로 쉽게 |
| 복구 | 불가 | 트랜잭션 로그로 복구 |
| 보안 | 파일 권한만 | 사용자별 권한 관리 |

## 2. RDBMS (관계형 데이터베이스)

### 2.1 관계형이 뭐야?

데이터를 **테이블(표)** 형태로 저장하고, 테이블 간의 **관계(Relation)**를 정의하는 방식.

**tb_student (학생 테이블)**

| std_no | std_nm | dept_cd |
|--------|--------|---------|
| STD001 | 홍길동 | DEPT01 |
| STD002 | 김철수 | DEPT02 |
| STD003 | 이영희 | DEPT01 |

dept_cd로 연결 (관계) :material-arrow-down:

**tb_department (학과 테이블)**

| dept_cd | dept_nm |
|---------|---------|
| DEPT01 | 컴퓨터공학과 |
| DEPT02 | 경영학과 |

"관계형"의 핵심: **같은 데이터를 중복 저장하지 않고, 관계로 연결한다.**

!!! danger "나쁜 예 (중복)"
    홍길동, 컴퓨터공학과
    이영희, 컴퓨터공학과 -- "컴퓨터공학과" 중복

!!! tip "좋은 예 (관계)"
    학생: 홍길동 → DEPT01
    학생: 이영희 → DEPT01
    학과: DEPT01 → 컴퓨터공학과 -- 한 곳에만 저장

학과 이름이 바뀌면? 관계형은 tb_department 한 군데만 고치면 된다.
중복 저장했으면 100만 군데 다 고쳐야 한다.

### 2.2 핵심 용어

!!! note "핵심 용어 정리"
    **데이터베이스 (Database)** = 스키마(Schema)
    : 테이블들의 집합 / 우리: `lxpknu10`

    **테이블 (Table)** = 릴레이션(Relation)
    : 행(Row)과 열(Column)로 구성된 2차원 표 / 우리: `tb_lms_exam_stare_paper_backup`

    **행 (Row)** = 레코드(Record) = 튜플(Tuple)
    : 하나의 데이터 단위 / 우리 문제 테이블: 2.7억 행

    **열 (Column)** = 필드(Field) = 속성(Attribute)
    : 데이터의 항목 / 예: EXAM_CD, STD_NO, SCORE

    **기본키 (Primary Key, PK)**
    : 각 행을 유일하게 식별하는 열 / 중복 불가, NULL 불가

    **외래키 (Foreign Key, FK)**
    : 다른 테이블의 PK를 참조하는 열 / 테이블 간 관계를 만드는 핵심

### 2.3 주요 RDBMS 종류

| RDBMS | 라이선스 | 특징 |
|-------|----------|------|
| **MySQL** | 오픈소스 (GPL) | 가장 많이 사용, 오라클 소유 |
| **MariaDB** | 오픈소스 (GPL) | MySQL 포크, 호환, 우리가 사용 |
| **PostgreSQL** | 오픈소스 (BSD) | 표준 SQL 준수, 기능 풍부 |
| **Oracle DB** | 상용 (비쌈) | 대기업, 금융권 |
| **MS SQL Server** | 상용 | Microsoft, Windows 환경 |

### 2.4 MySQL vs MariaDB

!!! note "MySQL vs MariaDB 역사"
    **역사:**

    - 1995: MySQL 탄생 (오픈소스)
    - 2008: Sun Microsystems가 MySQL 인수
    - 2009: Oracle이 Sun 인수 → MySQL이 Oracle 소유
    - 2009: MySQL 창시자(Monty)가 "Oracle 믿을 수 없다" → MariaDB 포크

    **관계:**

    MariaDB는 MySQL의 포크(복사본에서 시작)
    → 대부분의 명령어, 설정, 도구가 동일
    → mysqldump, mysql 명령어 그대로 사용
    → 우리 서버도 MariaDB인데 mysqldump 쓴다

    **차이점:**

    MariaDB가 점점 독자 기능 추가 중 (Aria 엔진, 컬럼스토어 등)
    하지만 기본 사용법은 거의 동일

## 3. 테이블 구조 상세

### 3.1 CREATE TABLE 해부

```sql
CREATE TABLE tb_lms_exam_stare_paper (
    EXAM_CD       VARCHAR(30)    NOT NULL,     -- 시험코드
    EXAM_QSTN_SN  INT            NOT NULL,     -- 문항순번
    STD_NO        VARCHAR(20)    NOT NULL,     -- 학번
    SCORE         DECIMAL(5,2)   DEFAULT 0,    -- 점수
    ANSWER        TEXT,                         -- 답안
    REG_DTTM      VARCHAR(14),                 -- 등록일시
    MOD_DTTM      VARCHAR(14),                 -- 수정일시
    PRIMARY KEY (EXAM_CD, EXAM_QSTN_SN, STD_NO)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

하나씩 뜯어보자:

| 구문 | 의미 |
|------|------|
| `VARCHAR(30)` | 가변 길이 문자열, 최대 30자 |
| `INT` | 정수 (-21억 ~ +21억) |
| `DECIMAL(5,2)` | 소수점 포함 숫자 (최대 999.99) |
| `TEXT` | 긴 문자열 (최대 65,535자) |
| `NOT NULL` | NULL 불가 (반드시 값이 있어야 함) |
| `DEFAULT 0` | 값 안 넣으면 기본값 0 |
| `PRIMARY KEY` | 기본키 (유일 식별, 자동 인덱스) |
| `ENGINE=InnoDB` | 스토리지 엔진 (트랜잭션 지원) |

### 3.2 데이터 타입

| 타입 | 크기 | 용도 | 예시 |
|------|------|------|------|
| **TINYINT** | 1B | 0~255 또는 -128~127 | 상태값, 플래그 |
| **INT** | 4B | -21억~21억 | 카운터, ID |
| **BIGINT** | 8B | 매우 큰 정수 | 자동증가 PK |
| **VARCHAR(n)** | 가변 | 문자열 (최대 n자) | 이름, 코드 |
| **CHAR(n)** | 고정 n | 고정 길이 문자열 | 고정 코드 |
| **TEXT** | 최대 64KB | 긴 텍스트 | 답안, 본문 |
| **DECIMAL(p,s)** | 가변 | 정확한 소수 | 금액, 점수 |
| **DATETIME** | 8B | 날짜+시간 | 등록일시 |
| **TIMESTAMP** | 4B | 날짜+시간 (UTC) | 수정일시 |

**VARCHAR vs CHAR:**
```
VARCHAR(10)에 'ABC' 저장 → 3바이트 + 길이정보 = 4~5바이트
CHAR(10)에 'ABC' 저장    → 10바이트 (나머지 공백 패딩)
→ 길이가 다양하면 VARCHAR, 고정이면 CHAR
```

### 3.3 제약조건 (Constraints)

```sql
CREATE TABLE tb_order (
    order_id    BIGINT       AUTO_INCREMENT PRIMARY KEY,  -- 자동증가 PK
    customer_id VARCHAR(20)  NOT NULL,                    -- NULL 불가
    amount      DECIMAL(10,2) CHECK (amount >= 0),        -- 0 이상만 허용
    status      VARCHAR(10)  DEFAULT 'PENDING',           -- 기본값
    email       VARCHAR(100) UNIQUE,                      -- 중복 불가
    dept_cd     VARCHAR(10)  REFERENCES tb_dept(dept_cd), -- 외래키
    reg_dttm    DATETIME     DEFAULT NOW()
);
```

| 제약조건 | 의미 | 위반 시 |
|----------|------|---------|
| **PRIMARY KEY** | 유일 식별, NOT NULL + UNIQUE | INSERT 거부 |
| **NOT NULL** | NULL 불가 | INSERT/UPDATE 거부 |
| **UNIQUE** | 중복 불가 (NULL은 허용) | INSERT/UPDATE 거부 |
| **CHECK** | 조건 검사 | INSERT/UPDATE 거부 |
| **DEFAULT** | 기본값 | 값 안 주면 자동 설정 |
| **FOREIGN KEY** | 참조 무결성 | 부모 없는 값 거부 |

### 3.4 외래키(FK)와 참조 무결성

!!! note "FK 관계와 규칙"
    `tb_student.dept_cd` → `tb_department.dept_cd` (FK 관계)

    **규칙:**

    1. 학생의 dept_cd에 tb_department에 없는 값 넣기 → 거부
    2. tb_department에서 학생이 속한 학과 삭제 → 거부 (또는 CASCADE)

    **CASCADE 옵션:**

    | 옵션 | 동작 |
    |------|------|
    | `ON DELETE CASCADE` | 부모 삭제 시 자식도 자동 삭제 |
    | `ON DELETE SET NULL` | 부모 삭제 시 자식 FK를 NULL로 |
    | `ON DELETE RESTRICT` | 부모 삭제 거부 (기본값) |

!!! warning "TRUNCATE와 FK"
    FK가 걸려있는 테이블은 TRUNCATE 불가!
    → "Cannot truncate a table referenced in a foreign key constraint"
    → 자식 테이블 먼저 정리하거나 FK 해제 후 TRUNCATE

## 4. 스토리지 엔진

### 4.1 스토리지 엔진이란?

!!! note "스토리지 엔진이란?"
    DBMS가 데이터를 물리적으로 저장하고 읽는 방식을 결정하는 모듈.

    비유: 같은 창고(DBMS)인데 물건 정리 방법(엔진)이 다른 것.

    - 엔진 A: 물건마다 잠금장치 (InnoDB)
    - 엔진 B: 창고 전체 잠금장치 (MyISAM)

### 4.2 InnoDB vs MyISAM

| 항목 | InnoDB | MyISAM |
|------|--------|--------|
| **트랜잭션** | 지원 | 미지원 |
| **락 단위** | 행 레벨 | 테이블 레벨 |
| **FK 지원** | 지원 | 미지원 |
| **장애 복구** | Redo/Undo 로그 | 없음 |
| **동시 처리** | 높음 | 낮음 |
| **COUNT(*)** | 느림 (행 세기) | 빠름 (메타데이터) |
| **기본 엔진** | MariaDB 10.2+ | MariaDB 10.1 이하 |

**우리 서버**: InnoDB 사용 (MariaDB 기본)

### 4.3 InnoDB의 파일 구조

```
innodb_file_per_table = ON 일 때:

/data/mysql/lxpknu10/
├── tb_user.frm                    ← 테이블 구조 정의 (DDL)
├── tb_user.ibd                    ← 테이블 데이터 + 인덱스
├── tb_exam_stare_paper.frm
├── tb_exam_stare_paper.ibd        ← 이 파일이 데이터
├── tb_exam_stare_paper_backup.frm
└── tb_exam_stare_paper_backup.ibd ← 이놈이 43GB였음

공용 파일:
├── ibdata1                        ← 시스템 테이블스페이스
├── ib_logfile0                    ← Redo 로그
├── ib_logfile1                    ← Redo 로그
└── mysql-bin.XXXXXX               ← 바이너리 로그
```

!!! tip "핵심"
    | 명령 | 파일 레벨 동작 |
    |------|---------------|
    | `TRUNCATE` | .ibd 파일 삭제 후 빈 파일 재생성 |
    | `DROP` | .frm + .ibd 파일 모두 삭제 |
    | `DELETE` | .ibd 파일 안에서 행 마킹 (파일 크기 안 줄어듦) |

이게 왜 중요하냐면 — Part 03에서 DELETE/TRUNCATE/DROP 차이를 배울 때
"파일 레벨에서 뭐가 다른지"를 알아야 **왜** 속도 차이가 나는지 이해할 수 있다.

| 백업 | mysqldump 매일 00:00 | 14개월 실패 |

### 5.1 정상 vs 비정상

!!! warning "정상 vs 비정상"
    **정상**: 백업 테이블 수만~수십만 건, 수 GB 이하

    **비정상**: 백업 테이블 2.7억 건, 43GB

    - "왜 비정상이야?" → 코드 버그 (jQuery 핸들러 중복 + 루프 내 전체 백업)
    - "왜 14개월간 몰랐어?" → 백업 실패 모니터링 없었음
    - "왜 디스크가 찼어?" → 삭제 정책 없이 INSERT만 계속

## 6. 핵심 정리

!!! abstract "핵심 정리"
    - **DB** = 구조화된 데이터의 집합
    - **RDBMS** = 관계형 데이터베이스 관리 시스템
    - **테이블** = 행(데이터)과 열(항목)로 구성된 2차원 표
    - **PK** = 행을 유일하게 식별, **FK** = 테이블 간 관계
    - **InnoDB** = 트랜잭션 지원, 행 레벨 락, 장애 복구
    - **.ibd 파일** = 테이블의 실제 데이터 파일
    - **우리 서버**: MariaDB, InnoDB, lxpknu10 스키마
    - **문제**: 43GB 테이블 (코드 버그 → 데이터 폭발)
    - **다음 장에서 배울 것**: SQL 명령어 분류 (DDL/DML/DCL) → "왜 TRUNCATE는 Auto-Commit이고 DELETE는 아닌지"
