## 1. 장애 대응의 원칙

장애가 터졌을 때 "어떻게 고치지?"부터 생각하면 이미 늦었다.
순서가 있다. 순서 틀리면 장애가 더 커진다.

### 1.1 서비스 복구 우선 (임시 조치)

!!! danger "장애 터지면 첫 번째로 할 일"
    **하면 안 되는 것:** "왜 이런 일이 생겼지?" (원인 분석), "누가 이런 코드를 짰어?" (범인 찾기), "이거 완벽하게 고치려면..." (근본 수정)

    **해야 하는 것:** "지금 서비스 되고 있어? 안 되면 일단 살려"

    서비스가 죽어있는 1분 동안 교수님 30명이 전화한다. 완벽한 원인 분석 끝나고 고치면? 이미 학기가 끝났다.

임시 조치 예시:
- 디스크 가득 참 → 불필요한 로그 삭제, 용량 확보
- 특정 쿼리가 서버 잡아먹음 → KILL로 해당 세션 종료
- 서비스 응답 없음 → WAS 재시작
- 특정 기능에서만 문제 → 해당 기능 일시 비활성화

**핵심: 완벽하지 않아도 된다. 일단 서비스가 돌아가면 된다.**

### 1.2 근본 원인 분석 (나중에)

서비스 복구하고 나서, 숨 한 번 쉬고, 그때 원인을 파고든다.

!!! example "5 Whys 적용"
    - 왜 터졌어? → 디스크가 가득 찼다
    - 왜 디스크가 가득 찼어? → 43GB짜리 테이블이 하나 있다
    - 왜 43GB야? → 코드 버그로 데이터가 폭발했다
    - 왜 코드 버그가 14개월간 방치됐어? → 모니터링이 없었다

5 Whys. 표면에서 멈추면 같은 장애가 또 터진다.

### 1.3 재발 방지 (마지막)

!!! tip "재발 방지 = 시스템적으로 불가능하게 만드는 것"
    재발 방지는 "다음엔 조심하자"가 아니다.

    | 잘못된 재발 방지 | 올바른 재발 방지 |
    |----------------|----------------|
    | "개발자가 삭제 정책을 잊지 않으면 됩니다" | 배치 스케줄러가 90일 지난 데이터를 자동 삭제 |
    | "디스크 용량을 가끔 확인합니다" | 디스크 80% 넘으면 자동 알림 발송 |
    | "백업 로그를 가끔 확인합니다" | 백업 실패 시 즉시 알림 + 3일 연속 실패 시 경고 |

### 1.4 문서화 (항상)

!!! warning "문서화 안 하면 일어나는 일"
    - 3개월 뒤 같은 장애 → "이거 전에도 있었는데... 뭐였지?"
    - 담당자 퇴사 → "이 서버 뭐가 문제였는지 아는 사람?"
    - 인수인계 → "아 그냥 가끔 느려져요" (끝)

    **장애 보고서 = 미래의 너한테 보내는 편지.** "이런 일이 있었고, 이렇게 대응했고, 이렇게 방지했다"

## 2. 실전 시나리오 1: 43GB 테이블 발견

### 2.1 상황

!!! danger "2026-02-27 현장 상황"
    **보고 내용:**

    - "DB 백업이 매일 실패하고 있습니다"
    - "Disk is full 에러가 229건 누적됐습니다"
    - "최소 14개월간 백업 파일이 전부 0바이트입니다"

    **서버 상태:** /data: 196GB 중 134GB 사용 (72%), MariaDB 정상 구동 중, 서비스 운영 중 (학생 접속 중)

    **첫 번째 질문: 어디서부터 손 대야 하나?**

당장 서비스가 죽은 건 아니다. 하지만 72%면 언제 터져도 이상하지 않다.
백업이 14개월째 실패 중이라는 건, 14개월간 백업 없이 운영하고 있었다는 뜻이다.

**지금 터지면 14개월치 데이터가 날아간다. 이게 진짜 문제다.**

### 2.2 진단 과정

**Step 1: 디스크 상태 확인**

```bash
$ df -h /data
Filesystem      Size  Used Avail Use% Mounted on
/dev/sdb1       196G  134G   53G  72% /data
```

72%. 위험 구간이다.
mysqldump는 덤프 파일을 디스크에 쓴다. 53GB 여유로 50GB DB를 덤프하면?
덤프 도중에 디스크 꽉 차서 실패한다. 이게 14개월간 반복된 거다.

**Step 2: 테이블별 용량 확인**

```sql
SELECT table_name,
       table_rows,
       ROUND(data_length / 1024 / 1024 / 1024, 2) AS data_gb,
       ROUND(index_length / 1024 / 1024 / 1024, 2) AS index_gb
FROM information_schema.tables
WHERE table_schema = 'lxpknu10'
ORDER BY data_length DESC
LIMIT 10;
```

결과:

| table_name | table_rows | data_gb | index_gb |
|------------|-----------|---------|----------|
| tb_lms_exam_stare_paper_backup | 276,490,000 | 43.01 | 0.00 |
| tb_lms_exam_stare_paper | 2,340,000 | 3.82 | 0.45 |
| (나머지 테이블들) | ... | < 1 | ... |

43GB. 전체 DB의 86%가 테이블 **하나**에 있다.
2.7억 건. 인덱스 0.00GB. 인덱스가 없다는 건 이 테이블에 대한 모든 조회가 풀스캔이라는 뜻이다.

!!! question "여기서 멈추고 생각해봐"
    "backup" 테이블이 원본 테이블의 11배? 원본(paper): 2,340,000건 (3.82GB). 백업(backup): 276,490,000건 (43GB). **백업이 원본의 118배???**

    뭔가 심각하게 잘못된 거다. 정상적인 백업이라면 원본과 비슷하거나 약간 클 수는 있어도 118배가 될 수는 없다. → **코드를 봐야 한다.**

**Step 3: 슬로우 쿼리 확인**

!!! note "슬로우 쿼리 확인"
    mysqldump 실행 시간: 1,249초 (약 20분). tb_lms_exam_stare_paper_backup 풀스캔에 대부분 소비. 인덱스 없는 2.7억 건 테이블을 처음부터 끝까지 읽는 시간.

**Step 4: 백업 스크립트 분석**

```bash
# 매일 00:00 실행
mysqldump -uroot -pknu10*2024! --databases lxpknu10 > ${LXP}.sql
```

여기서 이미 문제 3개가 보인다:
1. 43GB 테이블을 필터링 없이 통째로 덤프
2. 비밀번호에 특수문자 `!` → 셸이 히스토리 확장으로 해석
3. 덤프 출력(73GB 예상) > 디스크 여유(53GB) → Disk Full

### 2.3 원인 분석 (코드 레벨)

DB만 봐서는 "왜 118배인지" 알 수 없다. 코드를 봐야 한다.

**BUG-1: jQuery 이벤트 핸들러 중복 등록**

!!! danger "BUG-1: jQuery 이벤트 핸들러 중복 등록"
    **증상:** 재채점 버튼 1번 클릭 → AJAX 요청이 N번 발생

    ```javascript
    function initGradeReview() {
        $("#btnRegrade").on("click", function() {
            $.ajax({ url: "/exam/regrade", ... });
        });
    }
    ```

    **문제:** `initGradeReview()`가 탭 전환마다 호출됨. 탭 3번 전환 → 핸들러 3개 등록 → 버튼 1번 클릭 → AJAX 3번 발생. 탭 10번 전환 → AJAX 10번 발생.

    **이게 왜 위험한가:** 교수가 시험 관리 화면에서 탭을 왔다갔다 하는 건 일상적인 행위. 5번만 왔다갔다 해도 재채점 1번에 AJAX 5번. 각 AJAX가 백업 INSERT를 유발하면? 데이터 5배.

**BUG-2: 루프 내 전체 백업 호출**

!!! danger "BUG-2: 루프 내 전체 백업 호출"
    ```java
    for (Student s : studentList) {           // 학생 100명
        backupAllStudentPapers(examCd);       // 전체 백업 호출!
    }
    ```

    `backupAllStudentPapers`가 뭘 하느냐: 해당 시험의 "모든 학생"의 "모든 문항" 답안을 백업 테이블에 INSERT.

    | 처리 단계 | 동작 |
    |----------|------|
    | 학생 1명 처리할 때 | 100명 전체 백업 호출 |
    | 학생 2명 처리할 때 | 100명 전체 백업 또 호출 |
    | ... | ... |
    | 학생 100명 처리할 때 | 100명 전체 백업 또 호출 |

    **결과:** 100명 x (100명 x 50문항) = 500,000건. 정상이었다면 100명 x 50문항 = 5,000건. **100배 폭발.**

**데이터 폭발 시뮬레이션**

BUG-1과 BUG-2가 합쳐지면?

!!! example "데이터 폭발 시뮬레이션 (BUG-1 + BUG-2 합산)"
    **조건:** 학생 100명, 문항 50개, 교수가 탭 5번 전환 후 재채점 (핸들러 5중 등록)

    | 시나리오 | 계산 | INSERT 건수 | 배수 |
    |---------|------|------------|------|
    | 정상 동작 | 100명 x 50문항 | 5,000건 | 1배 |
    | BUG-2만 | 100명 x (100명 x 50문항) | 500,000건 | 100배 |
    | BUG-1 + BUG-2 | 5회 x 100명 x (100명 x 50문항) | 2,500,000건 | **500배** |

    재채점 1회에 250만 건. 한 학기에 시험이 몇 번이야? 중간, 기말, 퀴즈... 과목이 몇 개야? 수십 개. 14개월이면? → **2.7억 건. 43GB. 숫자가 맞아 떨어진다.**

### 2.4 조치 계획 수립

진단이 끝나면 조치를 계획한다. 여기서 중요한 건 **순서**.

!!! abstract "조치 계획 -- 3단계"
    **Phase 1: 즉시 조치 (당일)** -- 목표: 백업 정상화 + 디스크 안전 구간 확보

    1. 43GB 테이블 처리 → 디스크 확보 (RENAME TABLE로 보존 → 빈 테이블 생성 → TRUNCATE)
    2. 백업 스크립트 수정 (--ignore-table, 비밀번호 따옴표 처리)

    **Phase 2: 코드 수정 (1주 이내)** -- 목표: 데이터 폭발 원인 제거

    1. BUG-1 수정: .off().on() 또는 이벤트 위임
    2. BUG-2 수정: 루프 밖으로 백업 호출 이동
    3. BUG-3 수정: 삭제 정책 추가 (90일 보관)
    4. BUG-4 수정: 하드코딩 테스트 값 제거

    **Phase 3: 후속 개선 (1개월 이내)** -- 목표: 재발 방지 시스템 구축

    1. BUG-5 수정: 백업 테이블에 인덱스 추가
    2. 디스크 모니터링 설정 (80% 알림)
    3. 백업 상태 모니터링 (실패 시 알림)
    4. 데이터 증가 추이 모니터링

왜 이 순서인가?

!!! tip "왜 이 순서인가?"
    **Phase 1을 먼저:** 디스크 72%는 "지금 당장" 위험하다. 백업 14개월 실패는 "지금 당장" 위험하다. 코드 수정은 테스트가 필요하다 = 시간이 필요하다. 디스크 확보는 코드 수정 없이 가능하다.

    **Phase 2를 그 다음:** 원인을 안 고치면 다시 쌓인다. Phase 1은 "증상 치료", Phase 2가 "원인 치료". 원인 치료 없이 증상만 잡으면 매달 TRUNCATE 해야 한다.

    **Phase 3를 마지막에:** 모니터링은 "예방"이다. 예방은 치료 후에 한다. 불 끄는 중에 화재경보기 설치하는 놈은 없다.

## 3. 실전 시나리오 2: TRUNCATE metadata lock

### 3.1 상황

!!! note "2026-02-27 TRUNCATE 실행 상황"
    **사전 조치 완료:**

    1. RENAME TABLE로 원본 보존 (tb_lms_exam_stare_paper_backup → tb_lms_exam_stare_paper_backup_2026_02_27)
    2. 빈 테이블 생성 → 서비스 정상 작동 확인

    이제 RENAME된 43GB 테이블을 TRUNCATE 할 차례. "TRUNCATE는 수 분이면 끝난다고 했잖아?"

### 3.2 진단 과정 (전체 타임라인)

!!! example "진단 타임라인"
    **T+0분:** TRUNCATE TABLE 실행. 예상 2~3분. 커서 깜빡... 결과가 안 나온다.

    **T+3분:** "좀 느린가? 43GB니까 좀 걸릴 수 있지." 기다린다.

    **T+6분:** "이건 좀 이상한데?" 새 터미널에서 `SHOW PROCESSLIST;`

    | Id | User | Host | db | Command | Time | State |
    |----|------|------|----|---------|------|-------|
    | 8468792 | root | local | lxpknu10 | Query | 362 | **Waiting for table metadata lock** |

    TRUNCATE 자체가 느린 게 아니다. 누군가가 metadata lock을 잡고 있어서 대기 중이다. 범인을 찾아야 한다.

    **T+7분:** `INNODB_TRX` 확인 → trx_id 129384, RUNNING, thread_id 8468780, trx_query: NULL. trx_query가 NULL이란 건 "트랜잭션은 열려있는데 현재 실행 중인 쿼리는 없다"는 뜻이다 (커넥션 풀에 반환된 상태, BEGIN 하고 COMMIT/ROLLBACK 안 한 채 방치).

    `SHOW PROCESSLIST WHERE Id = 8468780;` → Command: **Sleep**, Time: **2387초** (약 40분), User: lxpapp (WAS), Host: was01

    → WAS의 커넥션 풀 세션이 트랜잭션을 안 닫고 40분간 방치. 이 세션이 metadata lock을 잡고 있다. TRUNCATE(DDL)는 exclusive metadata lock이 필요하므로 대기.

    **T+8분:** KILL 판단. Sleep 상태(쿼리 미실행), Time 2387(40분간 유휴), WAS 커넥션 풀(KILL해도 풀이 재생성), RENAME 완료 테이블(서비스 무관). `KILL 8468780;` → Query OK.

### 3.3 조치

!!! warning "KILL 이후"
    **T+8분 (KILL 직후):** KILL 했더니 TRUNCATE 세션에도 영향 → `ERROR 1317 (70100): Query execution was interrupted`. 당황하지 마. TRUNCATE가 아직 완료 안 됐을 수 있다. 확인 → table_rows: 276490000, data_mb: 43010.00. 아직 안 지워졌다. 다시 해야 한다.

    **T+9분 (재실행):** 이번엔 방해꾼이 없다. `TRUNCATE TABLE tb_lms_exam_stare_paper_backup_2026_02_27;` → `Query OK, 0 rows affected (2 min 47.32 sec)`. 약 3분. 예상대로.

### 3.4 결과 확인

!!! success "결과 검증"
    **1. 테이블 확인**

    ```sql
    SELECT table_rows,
           ROUND(data_length/1024/1024, 2) AS data_mb
    FROM information_schema.tables
    WHERE table_name LIKE '%backup_2026%';
    ```

    | table_rows | data_mb |
    |-----------|---------|
    | 0 | 0.02 |

    0건, 0.02MB. 43GB → 0.02MB. 완료. (0.02MB는 빈 .ibd 파일의 초기 크기)

    **2. 디스크 확인**

    ```bash
    $ df -h /data
    Filesystem      Size  Used Avail Use% Mounted on
    /dev/sdb1       196G   54G  133G  29% /data
    ```

    72% → 29%. 80GB 확보. 디스크 안전.

    **3. 백업 테스트**

    ```bash
    $ mysqldump --single-transaction -uroot -p'knu10*2024!' \
      --ignore-table=lxpknu10.tb_lms_exam_stare_paper_backup_2026.. \
      --databases lxpknu10 > /data/backup/test_dump.sql
    ```

    정상 완료. 파일 크기 정상. **14개월 만에 백업 성공.**

## 4. 실전 시나리오 3: 금요일 저녁 장애 (가상)

### 4.1 상황

!!! danger "금요일 18:00 -- 장애 보고"
    - 전화 1: "학생인데요, 시험 점수가 0점으로 나와요"

    **추가 정보:**

    - 교수가 재채점을 3번 실행했음
    - 일부 학생 점수가 0점으로 초기화
    - 시스템 응답 시간이 점점 늘어남
    - df -h 확인: /data 95%

    **금요일 저녁이다. 인프라팀은 퇴근했다. 너 혼자서 대응해야 한다. 뭐부터 해?**

### 4.2 대응 계획

**Phase 1: 긴급 진단 (30분)**

!!! note "목표: 무슨 일이 일어나고 있는지 파악"
    **Step 1: 서비스 상태 확인 (5분)**

    - 웹사이트 접속 가능? 응답 시간은?
    - WAS 로그에 에러 있어?
    - DB 접속 가능?

    **Step 2: DB 상태 확인 (10분)**

    - SHOW PROCESSLIST; (지금 뭐가 돌아가고 있어?)
    - 슬로우 쿼리 있어?
    - 열린 트랜잭션 있어? (INNODB_TRX)

    **Step 3: 디스크 상태 확인 (5분)**

    - df -h (95%면 거의 꽉 찬 거다)
    - 뭐가 용량 잡아먹고 있어?
    - information_schema.tables로 테이블 용량 확인

    **Step 4: 데이터 정합성 확인 (10분)**

    - 0점인 학생 목록 확인
    - 재채점 전후 데이터 비교
    - 백업 테이블에 원본 데이터 있는지 확인

**Phase 2: 긴급 조치 (1시간)**

!!! warning "긴급 조치 우선순위"
    **우선순위 1: 디스크 확보 (95%는 빨간불)**

    - 불필요한 로그 파일 삭제
    - 백업 실패로 쌓인 0바이트 파일 정리
    - 가능하면 백업 테이블 TRUNCATE
    - 목표: 80% 이하로 낮추기

    **우선순위 2: 서비스 안정화**

    - 문제 쿼리 KILL
    - WAS 재시작 (커넥션 풀 초기화)
    - 재채점 기능 일시 비활성화 (더 이상 폭발 방지)

    **우선순위 3: 데이터 복구**

    - 백업 테이블에서 원본 점수 확인
    - 0점 학생의 정상 점수 복원
    - 복원 전 반드시 현재 상태 스냅샷

!!! danger "금요일 저녁 주의"
    금요일 저녁에 복잡한 데이터 복구는 위험하다. 디스크 확보 + 서비스 안정화만 하고, 데이터 복구는 월요일에 정밀하게.

**Phase 3: 원인 분석 (다음 주)**

!!! note "월요일에 차분하게"
    1. 재채점 코드 분석 → BUG-1, BUG-2 같은 패턴 있는지
    2. 0점 발생 원인 → 재채점 로직에서 점수 초기화 버그?
    3. 디스크 급증 원인 → 재채점 3번 x 데이터 폭발
    4. 타임라인 재구성 → 언제, 뭐가, 왜 발생했는지

**Phase 4: 재발 방지**

!!! tip "재발 방지 대책"
    1. 재채점 코드 버그 수정 (BUG-1, BUG-2 패턴)
    2. 재채점 실행 전 확인 다이얼로그 추가
    3. 디스크 모니터링 알림 (80%, 90% 단계별)
    4. 백업 테이블 자동 정리 스케줄러
    5. 재채점 로그 기록 (누가, 언제, 어떤 시험)

## 6. 장애 보고서 작성법

장애 보고서는 양식이 중요한 게 아니다.
**"읽는 사람이 이 보고서만 보고 상황을 완전히 이해할 수 있는가"**가 기준이다.

### 6.1 제목

!!! example "제목 형식"
    형식: `[장애] 날짜 - 한 줄 요약`

    - 좋은 예: [장애] 2026-02-27 - 43GB 백업 테이블로 인한 디스크 부족 및 백업 실패
    - 나쁜 예: [장애] DB 문제
    - 나쁜 예: [장애] 서버 이슈 해결

### 6.2 요약 (한 문장)

!!! example "요약 예시"
    "코드 버그로 시험 백업 테이블이 43GB(2.7억 건)까지 비정상 증가하여 디스크 사용률 72%에 도달했고, 14개월간 DB 백업이 실패한 상태였다."

    한 문장으로 "무엇이 문제였는지" 알 수 있어야 한다. 이 한 문장을 읽고 팀장이 심각성을 판단한다.

## 7. 교훈 총정리

!!! warning "교훈 1: 코드 버그 하나가 43GB 테이블을 만들 수 있다"
    jQuery 핸들러 중복 + 루프 내 전체 백업. 코드 몇 줄의 실수가 14개월에 걸쳐 2.7억 건을 만들었다. DB 문제라고 생각했는데 원인은 코드에 있었다. **"DB가 느려요" 할 때 코드부터 봐야 하는 이유.**

!!! warning "교훈 2: 14개월간 백업 실패 = 14개월간 백업 없이 운영"
    백업은 "돌리고 있으니까 안심"이 아니다. 백업이 "성공하고 있는지" 확인해야 한다. 0바이트 파일이 14개월간 쌓이고 있었는데 아무도 몰랐다. **백업 모니터링 없는 백업은 백업이 아니다.**

!!! warning "교훈 3: TRUNCATE는 부하 없지만 metadata lock은 있다"
    "TRUNCATE는 빠르다"고만 알면 당한다. DDL은 exclusive metadata lock이 필요하다. 누군가 그 테이블에 트랜잭션 열어놓으면 무한 대기. **TRUNCATE 전에 INNODB_TRX 확인하는 습관.**

!!! warning "교훈 4: SHOW PROCESSLIST는 DB 관리자의 CCTV"
    - "DB가 이상해요" → SHOW PROCESSLIST
    - "쿼리가 안 끝나요" → SHOW PROCESSLIST
    - "서버가 느려요" → SHOW PROCESSLIST
    - 뭐가 문제인지 모르겠으면 → SHOW PROCESSLIST

    **이것만 알아도 DB 장애의 절반은 원인을 찾을 수 있다.**

!!! warning "교훈 5: 모니터링 없으면 장애를 모른다"
    이 장애는 "터져서" 발견한 게 아니다. "우연히" 백업 상태를 확인하다 발견한 거다. 14개월간 모르고 있었다. **모니터링은 선택이 아니다. 없으면 장애를 방치하는 거다.**

!!! warning "교훈 6: 장애 대응의 순서가 결과를 결정한다"
    복구 → 원인 분석 → 재발 방지 → 문서화. 이 순서를 뒤집으면? 원인 분석하느라 서비스 3시간 죽이거나, 재발 방지에 1주일 쓰느라 복구를 안 하거나, 문서화 안 해서 3개월 뒤 같은 일 반복하거나.

!!! warning "교훈 7: 금요일 저녁에 복잡한 작업을 하지 마"
    금요일에는 임시 조치만. 근본 수정은 월요일에. 피곤한 상태에서 복잡한 데이터 복구하면 사고 친다. **"빨리 끝내고 퇴근하자"는 장애를 키우는 마인드.**
