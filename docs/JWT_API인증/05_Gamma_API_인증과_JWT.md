# 05. API 인증과 JWT - Gamma

---

> 👹 "JWT가 뭔지, 구조가 뭔지, 만들고 검증하는 건 알겠어.
> 근데 진짜 실전에서 외부 API 호출할 때 이게 어떻게 돌아가는지 모르면 소용없어."

---

## 1. API 인증이 뭐야? - "왜 아무나 못 부르게 해?"

### 비유

!!! example "회사 건물 비유"
    회사 건물에 들어가려면?

    **1단계: 출입증 발급 (= API 키 발급)**

    - "너 우리 회사 협력업체지? 출입증 줄게"
    - APP_KEY = 출입증 번호
    - SECRET_KEY = 출입증 비밀번호 (분실하면 큰일)

    **2단계: 출입증으로 게이트 통과 (= JWT로 API 호출)**

    - 매번 들어갈 때 출입증을 태그함
    - 게이트가 "이 사람 등록된 사람 맞네" 확인

    **3단계: 허가된 구역만 접근 (= API 권한 범위)**

    - 출입증에 "3층까지만" 적혀있으면 4층 못 감
    - JWT에 권한 정보가 있으면 그 범위만 접근 가능

### 진짜 정의

**API 인증(Authentication)**: API를 호출하는 쪽이 "나 누구야"를 증명하는 과정.

왜 필요해?

```
공개 API (날씨, 환율 등):
  → 아무나 호출 가능. 인증 없어도 됨.
  → 대신 호출 제한(Rate Limit) 정도만 걸어놓음.

비공개 API (학사 데이터, 개인정보 등):
  → 인가된 시스템만 호출 가능. 인증 필수.
  → 5만 명 개인정보를 아무나 가져가면? → 사고.
  → 그래서 JWT로 "너 진짜 NexClass야?" 확인하는 거야.
```

## 2. API 인증 방식들 - "JWT만 있는 게 아니야"

### 방식 비교

| 방식 | 원리 | 보안 | 복잡도 | 언제 써? |
|------|------|------|--------|---------|
| **API Key** | 키 하나를 쿼리/헤더에 첨부 | 낮음 | 쉬움 | 간단한 서비스, 내부 API |
| **Basic Auth** | ID:PW를 Base64로 인코딩해서 첨부 | 낮음 | 쉬움 | 간단한 인증 |
| **JWT** | 서명된 토큰으로 증명 | 높음 | 보통 | 서버↔서버, 확장 필요할 때 |
| **OAuth 2.0** | 인증 서버에서 토큰 발급 | 매우 높음 | 복잡 | 구글/카카오 로그인, 3rd party |

### API Key vs JWT

```
API Key 방식:
  GET /api/data?apiKey=abc123
  → 키가 URL에 노출됨. 로그에도 찍힘. 위험.
  → 키 하나로 모든 걸 해. 만료도 없고 권한 구분도 없어.

JWT 방식:
  GET /api/data
  Authorization: Bearer eyJhbGci...
  → 헤더에 넣어서 URL에 안 보임.
  → 만료 시간 있음. 시간 지나면 못 씀.
  → 서명 있음. 위조 불가능.
  → Payload에 권한 정보 넣을 수 있음.
```

### 우리 프로젝트는?

```
NexClass → LMS: JWT 방식

왜 API Key가 아니라 JWT?
1. 만료 기능 필요 (토큰 탈취해도 30분 지나면 못 씀)
2. 위조 방지 필요 (서명 검증)
3. 앱 식별 + 인증을 하나로 (Payload에 appKey 포함)
```

## 3. 서버 → 서버 인증의 특수성

### 브라우저 인증 vs 서버 인증

!!! note "브라우저 인증 vs 서버 인증"
    **브라우저 → 서버 (니가 해본 것):**

    - 사람이 ID/PW 입력
    - 쿠키/세션 기반
    - 브라우저가 쿠키 자동 관리

    **서버 → 서버 (지금 하는 것):**

    - 사람 없음. 코드가 자동으로 인증
    - JWT 기반
    - 코드에서 직접 토큰 생성 & 헤더 첨부

### 뭐가 다른데?

| 구분 | 브라우저 인증 | 서버 인증 |
|------|-------------|-----------|
| 인증 주체 | 사람 (ID/PW 입력) | 코드 (자동) |
| 인증 수단 | 세션/쿠키 | JWT 토큰 |
| 토큰 관리 | 브라우저가 자동 | 코드에서 직접 관리 |
| 사용자 경험 | 로그인 화면 | 없음 (사람이 안 봄) |
| 토큰 갱신 | 로그인 다시 | 코드가 자동 재발급 |

### 서버 인증에서 신경 써야 할 것

```
1. 토큰 만료 관리
   → 동기화 작업 중간에 만료되면?
   → 새 토큰 발급하고 재시도하는 로직 필요

2. 키 보안
   → SECRET_KEY가 코드에 하드코딩되면?
   → 설정 파일로 분리 + 환경변수 관리

3. 네트워크 오류 처리
   → LMS 서버가 안 받으면?
   → 재시도(retry) 로직 필요

4. 로깅
   → API 호출 성공/실패 기록
   → 문제 생겼을 때 추적 가능해야 함
```

## 4. 실전 API 호출 흐름 (NexClass → LMS)

### 전체 시퀀스

```mermaid
sequenceDiagram
    participant N as NexClass SyncService
    participant L as LMS Server
    N->>N: ① JWT 생성<br/>APP_KEY + SECRET_KEY<br/>Jwts.builder()...compact()
    N->>L: ② POST /vc/api/sync/term<br/>Authorization: Bearer {JWT}<br/>Body: {"pageNumber":0,"pageSize":1000}
    L->>L: ③ JWT 검증<br/>서명 / 만료 / 앱키 확인
    L->>L: ④ 데이터 조회<br/>DB에서 학기 조회
    L->>L: ⑤ AES 암호화<br/>JSON → 암호문
    L-->>N: ⑥ 암호화된 응답
    N->>N: ⑦ AES 복호화<br/>AES256Util.decrypt()
    N->>N: ⑧ JSON 파싱<br/>ObjectMapper → List&lt;Map&gt;
    N->>N: ⑨ DB 저장<br/>repository.save()
```

### 페이징은?

데이터가 많으면 한 번에 못 가져와. 나눠서 가져와야 해.

```
1페이지: POST /vc/api/sync/users?orgCd=ORG0000001
         Body: {"pageNumber": 0, "pageSize": 1000}
         → 1~1000번 사용자

2페이지: POST /vc/api/sync/users?orgCd=ORG0000001
         Body: {"pageNumber": 1, "pageSize": 1000}
         → 1001~2000번 사용자

...

52페이지: Body: {"pageNumber": 51, "pageSize": 1000}
         → 51001~52000번 사용자

각 페이지마다 JWT 첨부해야 해!
```

```
주의: pageNumber는 0부터 시작이야!
  0 = 1페이지
  1 = 2페이지
  ...
  51 = 52페이지
```

## 5. 에러 상황별 대응

### JWT 관련 에러

!!! danger "에러 대응 매트릭스"
    **401 (서명 불일치)**

    - SECRET_KEY가 맞는지 확인
    - LMS 쪽 키가 바뀌었을 수 있음
    - 운영팀에 확인 요청

    **401 (토큰 만료)**

    - 새 JWT 발급 후 재요청
    - 동기화가 오래 걸리면 요청마다 토큰 새로 생성

    **401 (앱키 미등록)**

    - APP_KEY가 LMS에 등록되어 있는지 확인
    - 오타 확인

    **400 (토큰 형식 오류)**

    - "Bearer " 빼먹었거나
    - 토큰 문자열이 잘렸거나
    - 인코딩 문제

    **500 (LMS 서버 오류)**

    - NexClass 문제 아님. LMS 서버 문제.
    - 잠시 후 재시도. 계속 실패하면 운영팀 연락.

    **연결 실패 (Connection Refused)**

    - LMS 서버가 꺼져있거나 네트워크 문제
    - 재시도 로직 + 로그 남기기

## 6. 주의사항 / 함정

### 함정 1: "토큰 한 번 만들어서 전역 변수에 저장해두면 편하잖아"

```
❌ 30분 지나면 만료돼.
   전역 변수에 넣어놓고 2시간 뒤에 쓰면 401.

✅ 요청할 때마다 새로 만들거나,
   만료 시간 체크해서 만료 직전에 갱신.

   우리 프로젝트:
   syncTerms() 안에서 매번 generateToken() 호출.
   안전하게.
```

### 함정 2: "페이징할 때 첫 페이지 토큰으로 52페이지까지 다 커버 되겠지"

```
△ 30분 안에 끝나면 괜찮아.
   근데 52페이지 돌리다 보면 시간 걸릴 수 있어.
   안전하게 페이지마다 토큰 확인하는 게 좋아.
```

### 함정 3: "SECRET_KEY를 로그에 찍어서 디버깅하자"

```
❌ 절대 금지.
   로그 파일이 노출되면 SECRET_KEY도 노출.
   디버깅할 때는 토큰 자체를 jwt.io에서 확인.
   SECRET_KEY는 절대 로그에 찍지 마.
```

## 7. 정리

| 질문 | 답 |
|------|-----|
| API 인증이 뭐야? | "나 누구야" 증명하는 과정 |
| 왜 JWT? | 만료+서명+확장성. API Key보다 안전. |
| 서버↔서버는 뭐가 특별해? | 사람 없이 코드가 자동 인증. 토큰 관리를 코드가 해야 함. |
| 에러나면? | 401이면 토큰 문제, 500이면 LMS 문제, 연결실패면 네트워크 문제. |

**이 챕터에서 반드시 기억할 것**:
- API 인증은 "너 누구야" 증명이고, JWT는 그 방법 중 하나
- 서버↔서버 통신에서는 코드가 토큰을 직접 관리해야 한다
- 매 요청마다 Authorization 헤더에 `Bearer {token}` 첨부
- 만료 관리, 에러 처리, 키 보안이 실전에서의 핵심

### 확인 문제 (4문제)

> 다음 문제를 풀어봐. 답 못 하면 위에서 다시 읽어.

**Q1.** API Key 방식 대신 JWT를 쓰는 이유 3가지를 말해봐.

**Q2.** 사용자 동기화(52페이지)를 돌리는 중에 JWT가 만료되면 어떻게 해야 해?

**Q3.** LMS에서 401이 왔어. 원인이 될 수 있는 것 3가지를 말해봐.

**Q4.** SECRET_KEY를 로그에 찍으면 안 되는 이유는? 대신 디버깅은 어떻게 해?

??? success "정답 보기"
    **A1.**

    1. 만료 기능: 토큰이 탈취돼도 시간 지나면 못 씀.
    2. 위조 방지: 서명이 있어서 토큰 변조 불가능.
    3. 데이터 포함: Payload에 앱키/권한 등 정보를 담을 수 있음.

    **A2.** 새 JWT를 발급해서 이어서 진행. 안전하게 하려면 매 페이지 요청마다 토큰을 새로 생성하거나, 요청 전에 만료 여부를 체크해서 만료됐으면 재발급.

    **A3.**

    1. 서명 불일치: SECRET_KEY가 맞지 않거나 토큰이 위조됨.
    2. 토큰 만료: exp 시각이 지남.
    3. 앱키 미등록: APP_KEY가 LMS에 등록되어 있지 않음.

    **A4.** 로그 파일이 노출되면 SECRET_KEY도 같이 노출돼서 공격자가 유효한 JWT를 맘대로 만들 수 있어. 디버깅은 JWT 토큰 문자열을 jwt.io에 붙여넣어서 Header/Payload를 확인하면 돼. SECRET_KEY는 로그에 절대 찍지 않는다.
