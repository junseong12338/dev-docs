## Section A: IP와 포트 (20점)

??? success "A1. (3점) WAS 서버 IP"
    - **공인 IP**: `49.247.45.181` — 인터넷에서 이 서버에 접근할 때 사용
    - **내부 IP**: `10.7.1.52` — 같은 학교 내부 네트워크(10.7.1.x)에서 사용
    - Nginx(WEB)가 NexClass에 요청 보낼 때: `http://10.7.1.52:8082` (내부 IP 사용)
    - 외부에서 직접 접근: 공인 IP 사용 가능하지만 **우리는 Nginx를 거쳐야 하므로** WEB 서버 IP를 통해 접근

    **채점**: 공인/내부 IP 각 1점, 사용처 설명 1점

??? success "A2. (3점) 포트의 본질"
    **포트의 본질**: 하나의 IP 주소(서버)에서 여러 프로그램이 동시에 돌아갈 때, **어떤 프로그램한테 데이터를 전달할지 구분하는 번호** (0~65535).

    **포트 없으면**: 서버 1대에 프로그램 1개만 실행 가능. LMS, NexClass, HUB를 돌리려면 서버가 3대 필요해. 포트가 있어서 `10.7.1.52:8080`(LMS), `:8082`(NexClass), `:8090`(HUB)로 구분 가능.

    **채점**: 본질 1점, 문제점 설명 2점

??? success "A3. (4점) hostname -I vs curl ifconfig.me"
    | 명령어 | 결과 | 이유 |
    |--------|------|------|
    | `hostname -I` | `10.7.1.52` (내부 IP) | 서버 자기 자신한테 "니 IP 뭐야?" 물어봄. 네트워크 인터페이스에 설정된 사설 IP 반환 |
    | `curl ifconfig.me` | `49.247.45.181` (공인 IP) | 외부 서비스한테 "내가 어디서 접속하는 것처럼 보여?" 물어봄. NAT를 거쳐서 나간 공인 IP 반환 |

    **핵심**: 같은 서버인데 안에서 보는 나(사설 IP) vs 밖에서 보는 나(공인 IP)가 다른 거야. **NAT(Network Address Translation)** 때문.

    **채점**: 각 명령어 결과 1점씩, 이유 설명 각 1점씩

??? success "A4. (4점) lsof 결과 없음"
    **의미**: 8082 포트에서 LISTEN하고 있는 프로세스가 **없다**. 즉, **NexClass Spring Boot가 실행되지 않고 있다**.

    **대응 방법**:
    ```bash
    cd /usr/local/nexclass
    nohup java -jar NexClass.jar &
    ```
    실행 후 다시 `sudo lsof -i :8082`로 LISTEN 상태 확인.

    **채점**: 의미 설명 2점, 대응 방법 2점

??? success "A5. (3점) HTTPS/HTTP 기본 포트"
    - HTTPS 기본 포트: **443**
    - HTTP 기본 포트: **80**
    - URL에 포트가 안 보이는 이유: 브라우저가 프로토콜에 따라 **기본 포트를 자동으로 붙여줌**. `https://nexclass.knu10.cc` = `https://nexclass.knu10.cc:443`. 기본 포트는 생략하는 게 관례.

    **채점**: 포트 번호 각 1점, 이유 1점

??? success "A6. (3점) Well-Known 포트 5개"
    | 포트 | 용도 |
    |:----:|------|
    | 80 | HTTP (웹 기본) |
    | 443 | HTTPS (암호화된 웹) |
    | 22 | SSH (원격 서버 접속) |
    | 3306 | MySQL/MariaDB (데이터베이스) |
    | 5432 | PostgreSQL (데이터베이스) |

    기타 허용: 21(FTP), 25(SMTP), 53(DNS), 8080(개발용 웹)

    **채점**: 올바른 포트-용도 쌍 하나당 0.6점 (5개 = 3점)

## Section B: DNS와 도메인 (20점)

??? success "A7. (4점) DNS 본질 + nslookup"
    **DNS 본질**: **도메인 이름을 IP 주소로 변환하는 시스템.** 사람이 읽을 수 있는 이름(nexclass.knu10.cc)을 컴퓨터가 이해하는 숫자(49.247.41.134)로 바꿔줌.

    **nslookup 결과**:
    ```
    Name:    nexclass.knu10.cc
    Address: 49.247.41.134
    ```

    **채점**: DNS 본질 2점, nslookup 결과 2점

??? success "A8. (4점) A vs CNAME"
    | 구분 | A 레코드 | CNAME 레코드 |
    |------|----------|-------------|
    | 매핑 | 도메인 → **IP** | 도메인 → **도메인** |
    | 예시 | nexclass.knu10.cc → 49.247.41.134 | www.knu10.cc → knu10.cc |

    **www.knu10.cc가 CNAME인 이유**: `knu10.cc`의 IP가 바뀌면 A 레코드 하나만 수정하면 됨. www는 CNAME으로 `knu10.cc`를 가리키니까 자동으로 따라감. 관리 편의성.

    **채점**: 차이 설명 2점, CNAME 이유 2점

??? success "A9. (4점) TTL 3600 삽질 시나리오"
    TTL 3600초 = **1시간** 동안 DNS 캐시가 유지됨.

    **최악의 시나리오**: 잘못된 IP로 A 레코드를 설정했다가 바로 고쳤어도, 전 세계 DNS 서버들이 **최대 1시간** 동안 잘못된 IP를 캐시에 보관해. 이 시간 동안 사용자들은 잘못된 IP로 접속 시도 → 서비스 접속 불가.

    **대응**: 긴급 변경이 예상되면 미리 TTL을 짧게(60~300초) 설정해둬야 해.

    **채점**: TTL 의미 1점, 최악 시나리오 설명 2점, 대응 1점

??? success "A10. (4점) Sectigo DCV CNAME"
    **뭐냐**: **Sectigo(인증기관)의 DCV(Domain Control Validation) 검증용 CNAME 레코드**야.

    **DCV 과정**: SSL 인증서 발급 시 CA(Sectigo)가 "이 도메인 니 거 맞아?" 확인해야 해 → "이 CNAME 추가해봐" → 추가하면 → "OK 니 도메인 맞네" → 인증서 발급.

    **남겨두는 이유**: 인증서 갱신할 때 또 쓰일 수 있어서.

    **채점**: DCV 설명 2점, CNAME 용도 1점, 남겨두는 이유 1점

??? success "A11. (4점) DNS 조회 과정"
    1. **브라우저 캐시**: 이전에 조회한 적 있으면 캐시에서 바로 반환
    2. **OS 캐시**: 브라우저 캐시 없으면 운영체제 DNS 캐시 확인
    3. **DNS Resolver** (ISP): OS에도 없으면 ISP의 DNS 서버에 질의
    4. **Root DNS**: `.cc` 담당 서버 주소 알려줌
    5. **TLD DNS (.cc)**: `knu10.cc` 담당 네임서버(Cloudflare) 주소 알려줌
    6. **Cloudflare**: `nexclass.knu10.cc`의 A 레코드 → `49.247.41.134` 응답
    7. **응답 캐시**: TTL 동안 각 단계에서 캐싱

    **채점**: 주요 단계 언급당 1점 (브라우저캐시, Resolver, Root/TLD, Cloudflare)

## Section C: HTTP/HTTPS와 SSL (20점)

??? success "A12. (3점) HTTP vs HTTPS 차이 3가지"
    | 구분 | HTTP | HTTPS |
    |------|------|-------|
    | **암호화** | 없음 (평문) | SSL/TLS 암호화 |
    | **포트** | 80 | 443 |
    | **보안** | 중간자 공격 취약 | 암호화로 보호 |

    **채점**: 차이 하나당 1점

??? success "A13. (4점) Zoom HTTPS 필수 이유"
    Zoom Webhook 데이터에 **회의 정보, 참가자 정보, 강의 주제** 등 민감한 데이터가 포함돼 있어. HTTP(평문)로 전송하면 네트워크 중간에서 **다 볼 수 있어**. HTTPS로 암호화해야 중간에 가로채도 해독 불가.

    추가로, IP 주소 직접 사용은 **SSL 인증서 적용이 어렵기 때문에** 도메인 기반으로 등록해야 함.

    **채점**: 데이터 보안 이유 2점, 구체적 예시 1점, 도메인 필요 이유 1점

??? success "A14. (3점) return 301 설명"
    ```nginx
    return 301 https://$host$request_uri;
    ```
    HTTP(80번 포트)로 들어온 요청을 **HTTPS(443)로 영구 리다이렉트**.
    - `301`: 영구 이동. 브라우저가 다음부터 바로 HTTPS로 접속
    - `$host`: 요청한 도메인 (nexclass.knu10.cc)
    - `$request_uri`: 요청 경로 (/webhook/zoom)
    - 결과: `http://nexclass.knu10.cc/webhook/zoom` → `https://nexclass.knu10.cc/webhook/zoom`

    **채점**: 리다이렉트 설명 1점, 변수 설명 1점, 301 의미 1점

??? success "A15. (4점) 인증서 파일 차이 + key 유출"
    | 파일 | 역할 | 공개 여부 |
    |------|------|:---------:|
    | `.crt.pem` | 인증서 (공개키 + 도메인 정보) | 공개 OK |
    | `.key.pem` | 개인키 (복호화용) | **절대 비공개** |

    **.key.pem 유출 시**: 누구나 nexclass.knu10.cc인 척 HTTPS 서버를 만들 수 있어. **중간자 공격(MITM)** 가능 → 사용자 데이터 가로채기/변조 가능. 즉시 인증서 폐기(revoke) + 새로 발급 필요.

    **채점**: 파일 차이 2점, 유출 영향 2점

??? success "A16. (3점) TLS 핸드셰이크"
    1. **ClientHello**: 브라우저 → 서버: "TLS로 통신하고 싶어" (지원하는 암호화 방식 목록)
    2. **ServerHello + 인증서**: 서버 → 브라우저: "OK, 내 인증서 여기 있어"
    3. **인증서 검증**: 브라우저가 CA 서명 확인, 도메인 일치, 유효기간 체크
    4. **키 교환**: 공개키로 암호화해서 대칭키 교환
    5. **암호화 통신 시작**: 교환한 대칭키로 이후 모든 데이터 암호화

    **채점**: 주요 단계 3개 이상 언급 시 3점

??? success "A17. (3점) 인증서 하나로 다중 서브도메인"
    **와일드카드 인증서** (`*.knu10.cc`)이기 때문. 와일드카드(`*`)는 모든 서브도메인(nexclass, lms, hub 등)을 커버해. 파일명에 `knu10.cc`만 있고 특정 서브도메인이 없는 이유가 이거야.

    **채점**: 와일드카드 개념 2점, 적용 설명 1점

## Section D: Nginx 리버스 프록시 (20점)

??? success "A18. (4점) 리버스 프록시 본질"
    **리버스 프록시**: 클라이언트와 백엔드 서버 사이에 서서, **클라이언트 대신 요청을 받아 백엔드로 전달**해주는 중간자. 클라이언트는 Nginx만 보이고 뒤에 뭐가 있는지 모름.

    **포워드 프록시와의 차이**:
    - **포워드 프록시**: 클라이언트 쪽에서 클라이언트를 숨김 ("내가 누군지 숨겨줘")
    - **리버스 프록시**: 서버 쪽에서 서버를 숨김 ("뒤에 뭐가 있는지 숨겨줘")

    **채점**: 리버스 프록시 설명 2점, 차이 설명 2점

??? success "A19. (4점) proxy 설정 해부"
    ```nginx
    proxy_pass http://10.7.1.52:8082;
    ```
    → 요청을 WAS 서버(내부 IP 10.7.1.52)의 8082 포트(NexClass)로 **HTTP**로 전달. 내부망이라 HTTPS 불필요.

    ```nginx
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    ```
    → 원래 클라이언트의 IP 주소를 `X-Forwarded-For` 헤더에 담아서 백엔드에 전달. 이거 없으면 NexClass 로그에 Nginx IP(10.7.1.139)만 찍힘.

    ```nginx
    proxy_set_header HOST $http_host;
    ```
    → 원래 요청의 도메인(nexclass.knu10.cc)을 `Host` 헤더에 담아서 백엔드에 전달. 백엔드가 어떤 도메인으로 들어온 요청인지 알 수 있게.

    **채점**: 각 줄 설명 1.3점씩 (반올림 4점)

??? success "A20. (4점) server_name 라우팅"
    HTTP 요청에는 `Host` 헤더에 도메인 이름이 들어있어. 4개 도메인이 같은 IP로 들어와도, Nginx가 이 **Host 헤더**를 보고 **server_name이 매칭되는 server 블록**으로 분기.

    | 요청 Host | server_name 매칭 | proxy_pass |
    |-----------|-----------------|------------|
    | www.knu10.cc | www.knu10.cc 블록 | :8080 |
    | lms.knu10.cc | lms.knu10.cc 블록 | :8081 |
    | nexclass.knu10.cc | nexclass.knu10.cc 블록 | :8082 |
    | hub.knu10.cc | hub.knu10.cc 블록 | :8090 |

    이걸 **가상 호스팅(Virtual Hosting)**이라고 해.

    **채점**: Host 헤더 메커니즘 2점, 매칭 테이블 1점, 가상 호스팅 언급 1점

??? success "A21. (4점) nginx -t 안 하면"
    **벌어지는 일**: 설정 파일에 문법 오류 → reload 시 Nginx가 새 설정 로드 실패 → 최악의 경우 Nginx 프로세스 죽음 → **LMS, HUB, NexClass 전부 외부 접속 불가**.

    **올바른 순서**:
    ```bash
    sudo nginx -t              # 1. 문법 검사 (필수!)
    # "test is successful" 확인
    sudo nginx -s reload       # 2. 검사 통과 후 리로드
    ```

    **채점**: 벌어지는 일 2점, 올바른 순서 2점

??? success "A22. (4점) SSL 종료"
    **SSL 종료(SSL Termination)**: Nginx가 HTTPS 요청을 받아서 **SSL 복호화**하고, 백엔드에는 **HTTP로 전달**하는 패턴.

    **왜 Nginx에서?**:
    1. **관리 편의**: SSL 인증서를 Nginx 한 곳에서만 관리. 백엔드마다 SSL 설정 불필요.
    2. **성능**: 백엔드가 암호화/복호화 연산 안 해도 됨. CPU 부하 감소.
    3. **유연성**: 백엔드 서버 추가/교체 시 SSL 재설정 불필요.
    4. **보안 OK**: Nginx ↔ 백엔드는 내부망(10.7.1.x)이라 평문이어도 안전.

    **채점**: SSL 종료 설명 2점, 이유 2점

## Section E: WEB/WAS 아키텍처와 실전 (20점)

??? success "A23. (4점) WEB/WAS 분리 이유 4가지"
    1. **보안**: WAS가 인터넷에 직접 노출 안 됨. WEB만 외부에 열림.
    2. **SSL 관리 일원화**: Nginx 한 곳에서만 SSL 관리.
    3. **독립적 관리/스케일링**: WEB/WAS 각각 독립적으로 재시작, 업데이트, 스케일 가능.
    4. **장애 격리**: NexClass 죽어도 LMS/HUB 정상. 앱끼리 영향 없음.

    **채점**: 이유 하나당 1점

??? success "A24. (4점) NexClass 장애 시 LMS/HUB 영향"
    **영향 안 받음.**

    **이유**: NexClass(:8082), LMS(:8080/:8081), HUB(:8090)는 각각 **독립 Java 프로세스**야. 포트도 다르고 프로세스도 별개. NexClass가 죽어도 다른 포트에서 돌아가는 LMS/HUB 프로세스에는 영향 없음. Nginx도 정상이라 LMS/HUB로의 요청은 문제없이 처리.

    **채점**: 결론 1점, 독립 프로세스 설명 2점, Nginx 정상 언급 1점

??? success "A25. (4점) DNS Only 설정 이유"
    **우리가 자체 Sectigo SSL 인증서를 Nginx에 직접 설치해서 쓰기 때문.**

    Proxied 모드: Cloudflare가 중간에서 SSL 처리하려 함 → 자체 인증서와 **충돌** 가능.
    DNS Only 모드: Cloudflare는 DNS 변환만 하고, SSL은 **우리 Nginx가 직접 처리**.

    | Proxied | DNS Only (우리) |
    |---------|----------------|
    | Cloudflare SSL | 자체 Sectigo SSL |
    | 서버 IP 숨김 | 서버 IP 노출 |
    | CDN/DDoS 방어 | 직접 관리 |

    **채점**: 자체 SSL 충돌 설명 2점, Proxied vs DNS Only 차이 2점

??? success "A26. (4점) 전체 요청 흐름"
    1. **DNS 조회**: `nexclass.knu10.cc` → Cloudflare → A 레코드 → `49.247.41.134` (WEB 서버 IP)
    2. **HTTPS 연결**: 브라우저/Zoom → `49.247.41.134:443` → TLS 핸드셰이크 (Sectigo 인증서 검증)
    3. **Nginx 처리**: SSL 복호화 → `server_name nexclass.knu10.cc` 매칭 → `location /` 매칭
    4. **리버스 프록시**: `proxy_pass http://10.7.1.52:8082` → WAS 서버의 NexClass로 HTTP 전달
    5. **NexClass 처리**: `/webhook/zoom` 엔드포인트가 요청 처리 → 응답 반환
    6. **역방향**: NexClass → Nginx(SSL 암호화) → 클라이언트(HTTPS)

    **채점**: DNS 1점, SSL/Nginx 1점, proxy_pass 1점, 전체 흐름 완결성 1점

??? success "A27. (4점) 실제 작업 2가지"
    1. **Cloudflare에 A 레코드 추가**: `nexclass` → `49.247.41.134` (DNS Only)
    2. **Nginx에 server 블록 추가**: nexclass.knu10.cc → proxy_pass :8082

    **나머지 안 해도 됐던 이유**:
    - **SSL 인증서**: 기존 **와일드카드 인증서** (`*.knu10.cc`)가 nexclass 서브도메인도 커버
    - **WAS 서버**: NexClass.jar이 이미 8082에서 실행 중
    - **방화벽**: 내부망 10.7.1.x 통신은 이미 열려있었음

    **채점**: 2가지 작업 각 1점, SSL 안 해도 되는 이유 1점, WAS 이미 동작 설명 1점

## 보너스 문제 (10점)

??? success "A28. (3점) 내부 HTTP 안전한 이유"
    Nginx와 NexClass는 **같은 내부망(10.7.1.x)**에 있어. 이 네트워크는 **학교 내부 네트워크**로, 외부 인터넷에서 접근 불가. 중간자가 이 구간에 끼어들 수 없으니까 HTTP(평문)여도 안전.

    HTTPS는 인터넷 구간(외부)에서만 필요하고, 내부 통신은 **암호화 오버헤드 없이 빠르게** 하는 게 더 효율적.

    **채점**: 내부망 설명 1점, 외부 접근 불가 1점, 효율성 언급 1점

??? success "A29. (3점) proxy_connect_timeout 1800"
    **의미**: Nginx가 백엔드 서버(NexClass)에 **연결을 시도하는 최대 대기 시간**이 1800초(30분).

    **기본값(60초)보다 큰 이유**: 화상강의 시스템이라 **긴 처리 시간이 필요할 수 있어**. Webhook 이벤트 처리, 대용량 데이터 동기화 등에서 60초 타임아웃이면 중간에 끊길 수 있음. 30분으로 여유를 줘서 안정적으로 처리.

    **채점**: 의미 1점, 기본값보다 큰 이유 2점

??? success "A30. (4점) 서버 1대에 전부 올리면"
    1. **보안 위험**: 웹 서버가 해킹되면 DB까지 직접 접근 가능. WEB/WAS 분리하면 WEB이 뚫려도 WAS/DB는 내부망 뒤에 있어서 바로 접근 불가.

    2. **장애 전파**: NexClass가 메모리 폭주하면 **같은 서버의 LMS, HUB, DB 전부** 영향 받음. CPU/메모리 자원을 다 잡아먹으니까.

    3. **독립 관리 불가**: Nginx 설정 변경하려면 서비스 전체 영향. Java 업데이트하려면 서비스 전체 영향. 하나 고치려면 전부 건드려야 해.

    4. **스케일링 불가**: 트래픽 급증 시 WEB만 추가하는 게 불가능. 서버 통째로 복제해야 함 → DB 동기화 문제 발생.

    5. **리소스 경합**: Nginx, Java 앱 4개, MariaDB가 CPU/메모리를 놓고 경쟁 → 성능 저하.

    **채점**: 문제점 하나당 1.3점 (3개 이상 = 4점)
