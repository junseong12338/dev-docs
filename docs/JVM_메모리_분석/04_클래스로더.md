# 04. 클래스로더 (ClassLoader)

**이 장의 목표**: "Number of class loaders 4,574"가 왜 비정상인지 이해한다

---

## 1. 클래스 로딩이란

### 1.1 기본 개념

Java 프로그램을 실행하면 .class 파일이 바로 메모리에 올라가는 게 아니다.
**필요할 때 그때그때 로딩**한다. 이걸 담당하는 게 ClassLoader(클래스로더)다.

```
User user = new User();
         ↑
이 순간에 "User.class가 메모리에 있나?" 확인
  → 있으면: 바로 사용
  → 없으면: ClassLoader가 User.class를 찾아서 메모리에 올림
```

### 1.2 클래스 로딩 과정 (3단계)

!!! note "클래스 로딩 3단계"
    **1단계: Loading (로딩)**

    - .class 파일을 찾아서 바이트 배열로 읽어옴
    - 파일 시스템, JAR, 네트워크 등에서

    **2단계: Linking (링킹)**

    - Verify: 바이트코드 검증 (올바른 .class인지)
    - Prepare: static 변수 메모리 할당
    - Resolve: 심볼릭 참조 → 실제 메모리 주소로 변환

    **3단계: Initialization (초기화)**

    - static 블록 실행, static 변수 값 할당

## 2. 클래스로더의 계층 구조

### 2.1 3개의 기본 클래스로더

Java는 클래스로더를 **계층 구조**로 운영한다.

``` mermaid
graph TB
    BOOT["<b>Bootstrap ClassLoader</b><br/>(최상위)<br/>역할: Java 핵심 클래스 로드<br/>대상: rt.jar (java.lang.*, java.util.* 등)<br/>언어: C/C++로 구현<br/>특징: Java 코드로 접근 불가 (null 반환)"]
    EXT["<b>Extension ClassLoader</b><br/>(확장 클래스로더)<br/>역할: Java 확장 라이브러리 로드<br/>대상: jre/lib/ext/ 폴더의 JAR 파일<br/>언어: Java로 구현"]
    APP["<b>Application ClassLoader</b><br/>(= System ClassLoader)<br/>역할: 우리가 만든 클래스 로드<br/>대상: classpath의 .class, JAR 파일<br/>언어: Java로 구현"]
    BOOT -->|부모| EXT
    EXT -->|부모| APP
```

### 2.2 위임 모델 (Parent Delegation Model)

클래스를 찾을 때 **반드시 부모에게 먼저 물어본다**.

```
User.class를 로드해야 할 때:

Application ClassLoader: "User.class 로드해야 함"
    ↓ "부모님, 혹시 있어요?"
Extension ClassLoader: "나한텐 없어"
    ↓ "부모님, 혹시 있어요?"
Bootstrap ClassLoader: "나한텐 없어"
    ↓ (최상위까지 없음, 다시 내려옴)
Extension ClassLoader: "나도 없어, 아들아 네가 찾아"
    ↓
Application ClassLoader: "내가 classpath에서 찾을게"
    → User.class 발견! 로드 완료!
```

```
왜 이렇게 하는가?

보안 때문이다.

만약 누군가 악의적으로 java.lang.String.class를 만들어서
classpath에 넣어도, Bootstrap ClassLoader가 먼저 진짜
String을 로드하기 때문에 위조된 String은 무시된다.
```

## 3. Tomcat의 클래스로더

### 3.1 웹 애플리케이션은 특별하다

Tomcat 같은 WAS(Web Application Server)는 여러 웹앱을 동시에 띄울 수 있다.
각 웹앱이 서로 간섭하면 안 되니까, **웹앱마다 별도 클래스로더**를 만든다.

``` mermaid
graph TB
    BOOT2["Bootstrap ClassLoader"]
    EXT2["Extension ClassLoader"]
    SYS["System ClassLoader"]
    COMMON["Common ClassLoader<br/>(Tomcat 공통)"]
    WEB1["WebApp CL #1<br/>(LMS 앱)"]
    WEB2["WebApp CL #2<br/>(다른 앱)"]
    BOOT2 --> EXT2
    EXT2 --> SYS
    SYS --> COMMON
    COMMON --> WEB1
    COMMON --> WEB2
```

!!! info "WebApp ClassLoader"
    - = ParallelWebappClassLoader (Tomcat 8.5+)
    - = 우리 앱의 .class, JAR 파일 담당
    - 웹앱마다 별도 ClassLoader 생성

### 3.2 정상적인 클래스로더 수

```
정상적인 웹 애플리케이션 1개를 띄우면:

  Bootstrap ClassLoader:        1개
  Extension ClassLoader:        1개
  System ClassLoader:           1개
  Common ClassLoader:           1개
  WebApp ClassLoader:           1개
  ─────────────────────────────────
  합계:                    약 5~10개

프레임워크가 추가하는 ClassLoader:
  Spring:                  몇 개
  CGLIB (프록시 생성):     몇 개
  ─────────────────────────────────
  합계:                    약 10~50개 (정상)
```

## 4. 우리 서버: 4,574개 — 왜 비정상인가

### 4.1 숫자 분해

우리 서버 ClassLoader 4,574개 구성:

| 종류 | 개수 | 상태 |
|------|------|------|
| 기본 ClassLoader (Boot+Ext+App) | ~5개 | 정상 |
| Tomcat WebApp ClassLoader | 2개 | 비정상 |
| Spring/프레임워크 | ~50개 | 정상 |
| sun.reflect.DelegatingClassLoader | 4,257개 | 비정상 |
| 기타 | ~260개 | - |

4,257개 / 4,574개 = 93%가 DelegatingClassLoader

### 4.2 DelegatingClassLoader가 뭔데?

```
JVM이 리플렉션 최적화를 위해 자동으로 만드는 ClassLoader

리플렉션 호출 흐름:
  method.invoke() 호출
  → 1~15번: 느린 방식(네이티브 코드)으로 실행
  → 16번째: "자주 쓰네, 최적화하자"
  → JVM이 전용 바이트코드 생성
  → 이 바이트코드를 로드할 ClassLoader 필요
  → DelegatingClassLoader 1개 생성!
```

!!! warning "리플렉션 인플레이션"
    **JVM 옵션:** `sun.reflect.inflationThreshold = 15` (기본값)

    **동작:**

    - `method.invoke()` 1~15번째 호출: NativeMethodAccessorImpl 사용 (JNI 호출, 느림)
    - `method.invoke()` 16번째 호출: "이 메서드 자주 쓰네!" → GeneratedMethodAccessor 바이트코드 동적 생성 → DelegatingClassLoader로 로드 → 이후부턴 빠르게 실행

    **문제:**

    - DelegatingClassLoader가 참조 유지되면 GC 못함
    - 계속 쌓임
    - 4,257개 누적!

### 4.3 누가 DelegatingClassLoader를 잡고 있나

```
GC Roots 분석 결과:

  java.util.TimerThread (Timer-0): 2,103개 (49%)
  ↑ RefreshableSqlSessionFactoryBean의 TimerTask가 만든 스레드
  ↑ 1초마다 126개 XML 파일 체크 → 리플렉션 사용 → ClassLoader 생성

  java.beans.ThreadGroupContext: 1,131개 (27%)
  ↑ JavaBeans 인트로스펙션 캐시

  Log4j2 AsyncLogger: 930개 (22%)
  ↑ 로깅 프레임워크의 비동기 처리

  whatap.agent: 40개 (1%)
  ↑ WhaTap APM 에이전트
```

``` mermaid
graph TB
    A["Timer-0 스레드 (살아있음)"] -->|참조| B["TimerTask 객체"]
    B -->|실행 시 리플렉션 사용| C["리플렉션 캐시"]
    C -->|참조| D["DelegatingClassLoader 2,103개"]
    D -->|"GC 불가 (Timer-0가 살아있으니까)"| E["메모리 누수"]
```

!!! tip "해결 방법"
    - interval=0 설정하면 Timer-0 안 만들어짐
    - 리플렉션 안 일어남
    - DelegatingClassLoader 안 생김
    - 누수 해결

### 4.4 ParallelWebappClassLoader 2개 문제

```
정상: 1개 (앱 1개 = ClassLoader 1개)
우리: 2개

원인:
  Tomcat Hot Deploy (재배포) 시 이전 ClassLoader가 해제 안 됨
  → 이전 ClassLoader가 메모리에 남아있음
  → 12.4 MB + 4.9 MB = 17.3 MB 낭비

비유:
  이사할 때 짐을 다 빼고 나가야 하는데
  이전 집에 짐이 그대로 남아있는 상태
```

## 5. 클래스로더 누수의 무서움

### 5.1 일반 객체 누수 vs 클래스로더 누수

```
일반 객체 누수:
  User 객체 1000개 안 치워짐
  → 수십 KB ~ 수 MB 정도
  → 그래도 힙 안에서 관리 가능

클래스로더 누수:
  ClassLoader 1개가 안 치워지면
  → 그 ClassLoader가 로드한 모든 클래스가 안 치워짐
  → 그 클래스들이 참조하는 모든 static 변수가 안 치워짐
  → 그 static 변수들이 참조하는 모든 객체가 안 치워짐
  → 연쇄적으로 수십 MB ~ 수백 MB 누수

  ClassLoader 누수 = 핵폭탄급
```

### 5.2 시간에 따른 누적

```
우리 서버:
  서버 재기동 후 5일 경과
  → DelegatingClassLoader 4,257개 누적

계산:
  1초마다 126개 XML 파일 체크
  → 각 파일마다 리플렉션 호출
  → 16번 넘으면 DelegatingClassLoader 생성
  → 시간이 지날수록 계속 누적

1일:  수백 개
5일:  4,257개
30일: 수만 개 (예상)
→ 결국 OutOfMemoryError로 서버 다운
```

## 6. 핵심 정리

!!! abstract "핵심 정리"
    **ClassLoader (클래스로더):**

    - .class 파일을 메모리에 올리는 담당자
    - 계층 구조 (Bootstrap → Extension → Application)
    - 위임 모델 (부모에게 먼저 물어봄)

    **정상 범위:** 10~50개 / **우리 서버:** 4,574개 (비정상)

    **원인:**

    - DelegatingClassLoader 4,257개 (93%)
    - 리플렉션 최적화를 위해 JVM이 자동 생성
    - Timer-0 스레드가 참조 잡아서 GC 불가
    - RefreshableSqlSessionFactoryBean의 interval=1000이 원흉

    **해결:** interval=0 설정

    **교훈:** ClassLoader 수가 비정상이면 → 리플렉션 남용 또는 클래스 동적 생성 의심 → 누가 참조 잡고 있는지 GC Root 추적

    **다음 장:** GC와 GC Roots -- "5,863개 GC Root가 뭘 의미하는지"
