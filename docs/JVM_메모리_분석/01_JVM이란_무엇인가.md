# 01. JVM이란 무엇인가

---

## 1. 가장 기초부터: 프로그램은 어떻게 실행되는가

### 1.1 컴퓨터가 이해하는 언어

컴퓨터는 **0과 1(기계어)**만 이해한다. 그게 전부다.

```
사람이 쓰는 코드:    int a = 10;
컴퓨터가 이해하는 것: 10111000 00001010 00000000 00000000 00000000
```

그래서 사람이 쓴 코드를 기계어로 **번역**해야 한다.
이 번역 방식이 크게 2가지가 있다.

### 1.2 컴파일 방식 vs 인터프리터 방식

```
[컴파일 방식] - C, C++, Go, Rust
소스코드 → (컴파일러) → 기계어 실행파일 → OS가 직접 실행
                         ↑
                    미리 번역 완료 (빠름)
                    BUT: Windows용, Linux용, Mac용 따로 만들어야 함

[인터프리터 방식] - Python, JavaScript
소스코드 → (인터프리터가 한 줄씩 읽으면서 실행)
                         ↑
                    실행할 때 번역 (느림)
                    BUT: 어디서든 실행 가능
```

### 1.3 Java의 선택: 둘 다 쓴다

Java는 "한 번 쓰면 어디서든 돌아간다 (Write Once, Run Anywhere)" 가 목표였다.
그래서 **컴파일 + 인터프리터 하이브리드** 방식을 선택했다.

```
Java 소스코드 (.java)
        ↓ javac (자바 컴파일러)
바이트코드 (.class)         ← 기계어가 아님! 중간 언어!
        ↓
       JVM (자바 가상 머신)  ← 이놈이 바이트코드를 기계어로 번역
        ↓
   OS가 실행
```

**핵심**: `.class` 파일(바이트코드)은 어떤 OS에서든 동일하다.
OS별로 다른 건 **JVM**이 처리한다.

## 2. JVM (Java Virtual Machine)

### 2.1 정의

**JVM = 자바 바이트코드를 실행하는 가상 컴퓨터**

"가상"이라는 건 물리적 하드웨어가 아니라 **소프트웨어로 만든 컴퓨터**라는 뜻이다.

```
실제 컴퓨터:  CPU + RAM + 디스크 (물리적)
JVM:         가상 CPU + 가상 메모리 + 가상 실행 엔진 (소프트웨어)
```

### 2.2 JVM이 하는 일

!!! note "JVM의 역할"
    1. **클래스 로딩 (Class Loading)** -- .class 파일을 메모리에 올린다
    2. **메모리 관리 (Memory Management)** -- 객체를 힙에 생성하고, 안 쓰는 건 GC가 치운다
    3. **바이트코드 실행 (Execution Engine)** -- 바이트코드를 기계어로 번역해서 실행한다
    4. **보안 관리 (Security)** -- 위험한 코드가 시스템에 접근 못하게 막는다
    5. **스레드 관리 (Thread Management)** -- 여러 작업을 동시에 실행한다

```
┌─────────────────────────────────────────────────────────────┐

│  └───────┬───────┘                                          │
│          ↓                                                  │

│  │  │  Heap  │ │ Stack  │ │Method  │ │  PC  │ │Native││    │
```

## 3. JDK / JRE / JVM 차이

이거 헷갈리는 사람 많다. 확실히 정리한다.

```
┌─────────────────────────────────────────────┐
│                   JDK                        │
│  (Java Development Kit - 개발 도구)          │
│                                             │
│  javac (컴파일러), jdb (디버거),             │
│  jar (패키징), javadoc (문서생성) ...        │
│                                             │
│  ┌─────────────────────────────────────┐    │
│  │              JRE                     │    │
│  │  (Java Runtime Environment)          │    │
│  │  = 자바 실행 환경                    │    │
│  │                                     │    │
│  │  Java API (라이브러리들)             │    │
│  │                                     │    │
│  │  ┌─────────────────────────────┐    │    │
│  │  │           JVM               │    │    │
│  │  │  = 자바 가상 머신            │    │    │
│  │  │  = 바이트코드 실행 엔진      │    │    │
│  │  └─────────────────────────────┘    │    │
│  └─────────────────────────────────────┘    │
└─────────────────────────────────────────────┘
```

```
JDK = JRE + 개발 도구 (컴파일러, 디버거 등)
JRE = JVM + 라이브러리
JVM = 바이트코드 실행 엔진
```

- **개발자**: JDK 설치 (코드를 짜고 컴파일해야 하니까)
- **사용자**: JRE만 있으면 됨 (실행만 하면 되니까)
- **서버**: JDK 또는 JRE (보통 JDK 설치)

## 4. OpenJDK vs Oracle JDK

우리 서버 JVM 버전이 이거였다:
```
openjdk version "1.8.0_392"
OpenJDK Runtime Environment (1.8.0_392-8u392-ga-1~20.04-b08)
OpenJDK 64-Bit Server VM (build 25.392-b08, mixed mode)
```

### 4.1 뭐가 다른가

| 구분 | Oracle JDK | OpenJDK |
|------|-----------|---------|
| 제작 | Oracle(오라클) 회사가 만듦 | 오픈소스 (무료) |
| 라이선스 | 상용 라이선스 (돈 내야 함, Java 11부터) | Oracle JDK와 코드 거의 동일 |
| 지원 | 기술 지원 제공 | 커뮤니티가 유지보수 |
| 특징 | 성능 최적화 일부 포함 | 대부분의 서버가 이걸 씀 |

### 4.2 버전 읽는 법

```
openjdk version "1.8.0_392"
                 │ │ │  │
                 │ │ │  └── 패치 번호 (392번째 보안/버그 패치)
                 │ │ └──── 마이너 버전
                 │ └────── 메이저 버전 (8)
                 └──────── 1. 접두사 (Java 8까지만 붙음)

즉: Java 8의 392번째 패치 버전
```

**참고**: Java 9부터는 `1.` 접두사가 사라짐
```
Java 8:  1.8.0_392
Java 11: 11.0.20
Java 17: 17.0.8
Java 21: 21.0.1
```

### 4.3 64-Bit Server VM

```
OpenJDK 64-Bit Server VM (build 25.392-b08, mixed mode)
         │         │                          │
         │         │                          └── 실행 모드
         │         │                              (인터프리터 + JIT 혼합)
         │         └── Server VM
         │             (서버 환경 최적화, 시작 느리지만 실행 빠름)
         └── 64비트 (메모리 4GB 이상 사용 가능)
```

**Client VM vs Server VM**:
```
Client VM: 빠른 시작, 적은 메모리 최적화 → 데스크톱 앱용
Server VM: 느린 시작, 장기 실행 최적화 → 서버용 (우리가 이거)
```

**mixed mode**:
```
인터프리터 모드:   바이트코드를 한 줄씩 번역 실행
JIT 컴파일 모드:   자주 쓰는 코드를 기계어로 미리 번역
mixed mode:       둘 다 사용 (기본값, 가장 효율적)
```

## 5. 우리 서버에 대입

```
JVM version: openjdk version "1.8.0_392"
             OpenJDK 64-Bit Server VM (build 25.392-b08, mixed mode)
```

이게 말해주는 것:

| 항목 | 값 | 의미 |
|------|-----|------|
| JDK 종류 | OpenJDK | 무료 오픈소스 JDK |
| Java 버전 | 8 (1.8.0) | 2014년 출시, LTS (장기 지원) |
| 패치 레벨 | 392 | 보안 패치 충분히 적용됨 |
| 아키텍처 | 64-bit | 4GB 이상 메모리 사용 가능 |
| VM 타입 | Server VM | 서버 환경 최적화 |
| 실행 모드 | mixed mode | 인터프리터 + JIT 혼합 |

### 5.1 Java 8인 이유

```
Java 8 (2014) → 2030년 12월까지 지원 (OpenJDK)
- 가장 많이 쓰이는 Java 버전
- Spring Framework 4.x, eGovFramework가 Java 8 기반
- 우리 LMS가 eGovFramework 기반이라 Java 8
```

### 5.2 현재 최신은?

```
Java 8   (2014) - LTS ← 우리 서버
Java 11  (2018) - LTS
Java 17  (2021) - LTS
Java 21  (2023) - LTS ← 현재 최신 LTS
```

## 6. 핵심 정리

!!! abstract "핵심 정리"
    **JVM = 자바 바이트코드를 실행하는 가상 컴퓨터**

    하는 일:

    1. .class 파일을 메모리에 올린다 (Class Loading)
    2. 객체를 힙에 만들고 관리한다 (Memory Management)
    3. 바이트코드를 기계어로 번역 실행한다 (Execution)
    4. 안 쓰는 객체를 치운다 (Garbage Collection)
