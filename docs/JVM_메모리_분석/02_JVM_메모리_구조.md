# 02. JVM 메모리 구조

**이 장의 목표**: "Used heap dump 323.4 MB"가 정확히 뭘 의미하는지 이해한다

---

## 1. 메모리(RAM)란 무엇인가

### 1.1 비유로 이해하기

```
컴퓨터를 사무실이라고 하면:

  하드디스크(SSD) = 서류 창고
  → 파일을 영구 보관하는 곳
  → 크지만 느림 (꺼내려면 창고까지 가야 함)

  RAM(메모리) = 책상
  → 지금 작업 중인 서류를 올려놓는 곳
  → 작지만 빠름 (바로 손이 닿음)

  CPU = 사람의 뇌
  → 실제로 계산/판단하는 곳
  → RAM에 있는 데이터만 처리 가능
```

**핵심**: 프로그램이 실행되려면 반드시 RAM(메모리)에 올라가야 한다.

### 1.2 단위

```
1 Byte     = 8 bit (영문 글자 1개)
1 KB       = 1,024 Byte (짧은 텍스트 파일)
1 MB       = 1,024 KB (사진 1장)
1 GB       = 1,024 MB (영화 1편)

우리 서버 힙: 323.4 MB = 약 3억 3천만 Byte
우리 서버 힙 최대 설정: 12 GB = 약 128억 Byte
```

## 3. Heap (힙) - 가장 중요

### 3.1 힙이란

**new 키워드로 생성된 모든 객체가 저장되는 공간**

```java
// 이 코드가 실행되면
User user = new User("이준성");
List<String> list = new ArrayList<>();
String name = new String("hello");
```

!!! example "힙에 이렇게 쌓인다"
    - User 객체
    - ArrayList 객체
    - String 객체
    - ...
    - (523만 개!)

### 3.2 힙의 내부 구조

``` mermaid
graph LR
    subgraph YOUNG["Young Generation (젊은 세대) - Minor GC로 정리"]
        EDEN["Eden<br/>새 객체 탄생"]
        S0["S0"]
        S1["S1"]
    end
    subgraph OLD["Old Generation (늙은 세대) - Full GC로 정리"]
        OLDOBJS["오래 살아남은 객체들<br/>Young에서 살아남아 승격"]
    end
    EDEN --> S0
    S0 --> S1
    S1 --> OLDOBJS
```

!!! tip "비유"
    - **Eden** = 신생아실 (객체가 태어나는 곳)
    - **S0/S1** = 유아반 (살아남은 객체가 잠시 머무는 곳)
    - **Old** = 어르신 마을 (오래 살아남은 객체가 사는 곳)

### 3.3 객체의 생애주기

```
1. 객체 생성 (new)
   → Eden 영역에 배치

2. Eden이 가득 참
   → Minor GC 발생
   → 살아있는 객체 → S0으로 이동
   → 죽은 객체 → 메모리 해제

3. 다음 Minor GC
   → Eden + S0의 살아있는 객체 → S1으로 이동
   → S0 비움

4. 이 과정 반복 (S0 ↔ S1 왔다갔다)
   → 일정 횟수(기본 15회) 살아남으면

5. Old Generation으로 승격
   → 여기서 오래 살다가

6. Old가 가득 차면
   → Full GC (Major GC) 발생
   → 서버가 잠깐 멈출 수 있음 (Stop-The-World)
```

### 3.4 우리 서버의 힙 설정

```
JVM 옵션:
  -Xms8192m    → 힙 초기 크기: 8 GB
  -Xmx12288m   → 힙 최대 크기: 12 GB

실제 사용량:
  Used heap dump: 323.4 MB

비유:
  12GB짜리 방을 빌렸는데 (Xmx)
  8GB만큼 가구를 미리 배치해놓고 (Xms)
  실제로 쓰고 있는 건 323MB (Used heap)

  → 방의 2.6%만 쓰고 있다
  → 전체 메모리 양 자체는 문제가 아니다
  → 문제는 "뭘로 채워져 있느냐" (다음 장에서)
```

## 4. Stack (스택) - 스레드마다 1개

### 4.1 스택이란

**메서드 호출 정보와 지역 변수가 저장되는 공간**

```java
public void methodA() {
    int x = 10;          // ← x는 스택에
    String name = "hi";  // ← name 참조는 스택에, "hi" 객체는 힙에
    methodB();           // ← methodB 호출 정보가 스택에 쌓임
}

public void methodB() {
    int y = 20;          // ← y는 스택에
}
```

```
┌────── Stack (LIFO: 후입선출) ──────┐
│                                    │
│  ┌──────────────────────────┐      │
│  │ methodB() 프레임          │ ← 나중에 들어옴, 먼저 나감
│  │   y = 20                 │      │
│  ├──────────────────────────┤      │
│  │ methodA() 프레임          │      │
│  │   x = 10                 │      │
│  │   name → (힙의 "hi" 참조)│      │
│  └──────────────────────────┘      │
│                                    │
└────────────────────────────────────┘
```

### 4.2 힙 vs 스택

| 구분 | Stack | Heap |
|------|-------|------|
| 공유 | 스레드마다 1개 | 모든 스레드가 공유 |
| 저장 내용 | 메서드 호출 정보 | 객체(인스턴스) 저장 |
| 대상 | 지역 변수 | new로 생성한 것들 |
| 정리 방식 | 자동 정리 | GC가 정리 |
| 속도 | 빠름 | 상대적으로 느림 |
| 크기 | 크기 작음 | 크기 큼 |
| 구조 | LIFO 구조 | 복잡한 구조 |

## 5. Metaspace (메타스페이스)

### 5.1 메타스페이스란

**클래스의 메타데이터(설계도 정보)가 저장되는 공간**

```
힙:        객체(인스턴스)가 사는 곳    → new User() 결과물
메타스페이스: 클래스 정보가 사는 곳     → User.class 설계도 자체
```

### 5.2 Java 7 vs Java 8 차이

```
Java 7 이하:
  ┌────── JVM 힙 ──────┐
  │  Young  │  Old      │
  │         │           │
  │         │  PermGen  │  ← 클래스 정보가 힙 안에 있었음
  │         │  (영구세대)│     크기 고정 → 넘치면 OutOfMemoryError
  └─────────────────────┘

Java 8 이상:
  ┌────── JVM 힙 ──────┐   ┌────── Native Memory ──────┐
  │  Young  │  Old      │   │  Metaspace                │
  │         │           │   │  (메타스페이스)             │
  │         │           │   │  크기 자동 확장             │
  └─────────────────────┘   └───────────────────────────┘

핵심 변화:
  PermGen (힙 내부, 고정 크기) → Metaspace (힙 외부, 자동 확장)
```

### 5.3 우리 서버 설정

```
-XX:MaxPermSize=4096m

주의: 이 옵션은 Java 8에서 무시됨!
Java 8은 PermGen이 없으니까.
→ Metaspace는 별도 옵션: -XX:MaxMetaspaceSize
→ 설정 안 하면 OS 메모리 한도까지 자동 확장
```

## 6. 다시 보는 323.4 MB

```
Used heap dump: 323.4 MB
```

이 숫자가 말해주는 것:

!!! note "323.4 MB의 의미"
    **"힙 덤프를 뜬 시점에 힙 영역에 살아있던 모든 객체의 총 크기가 323.4 MB다"**

    **포함:**

    - new로 생성된 모든 객체
    - 배열
    - 문자열
    - 컬렉션(List, Map, Set 등) 내부 데이터

    **불포함:**

    - 스택 (메서드 호출 정보)
    - 메타스페이스 (클래스 정보)
    - 네이티브 메모리
    - GC가 이미 회수한 객체

    **판단:**

    - 힙 최대 12GB 중 323MB 사용 = 2.6%
    - 힙 크기 자체는 여유 있음
    - 문제는 323MB 안에 "뭐가 들어있냐"
    - 불필요한 객체가 GC 안 되고 쌓여있으면 그게 누수

## 7. 핵심 정리

!!! abstract "핵심 정리"
    **JVM 메모리 = 힙 + 스택 + 메타스페이스 + 기타**

    **힙 (Heap):**

    - 객체가 사는 곳
    - Young(신생) + Old(노년) 세대
    - GC가 관리
    - 힙 덤프 = 이 영역의 스냅샷

    **스택 (Stack):**

    - 메서드 호출 + 지역 변수
    - 스레드마다 1개

    **메타스페이스 (Metaspace):**

    - 클래스 설계도 정보
    - Java 8부터 힙 밖으로 분리됨

    **Used heap 323.4 MB:**

    - = 힙 덤프 시점에 힙에 살아있던 객체 총량
    - = 12GB 중 2.6% 사용
    - = 양은 적지만, 안에 뭐가 있냐가 중요

    **다음 장:** 객체(Object)와 클래스(Class) -- "523만 개 객체"와 "27,688개 클래스"가 뭘 의미하는지
