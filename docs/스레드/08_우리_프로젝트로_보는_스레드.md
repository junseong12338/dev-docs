## 2. 문제의 시작: Bean 초기화

```
서버 기동 (Tomcat start)
  ↓
Spring Container 초기화
  ↓
context-sqlMap.xml 읽기
  ↓
Bean 생성: RefreshableSqlSessionFactoryBean
  ↓
afterPropertiesSet() 호출 (05장: Bean 생명주기 - 초기화 단계)
  ↓
super.afterPropertiesSet()  → SqlSessionFactory 빌드 (정상)
setRefreshable()            → 여기서 Timer 시작 (문제!)
```

## 3. Timer-0의 탄생과 동작

### 3.1 탄생

```java
// setRefreshable() 안에서
timer = new Timer(true);     // (02장) new → 스레드 생성 + start()
                              // (07장) true → 데몬 스레드
                              // 이름: Timer-0
```

### 3.2 상태 변화 (03장 복습)

```
Timer-0 스레드:
  NEW → start() → RUNNABLE → task.run() → TIMED_WAITING (1초)
                           → task.run() → TIMED_WAITING (1초)
                           → task.run() → TIMED_WAITING (1초)
                           → ... (서버 종료까지 무한 반복)
```

### 3.3 매 1초마다 하는 일

```
task.run()
  ↓
isModified()
  ↓
for (i = 0; i < 126; i++)           ← SQL XML 파일 126개 순회
  ↓
findModifiedResource(resource)
  ↓
resource.lastModified()              ← 리플렉션(Method.invoke) 발생
  ↓
리플렉션 16회 초과 시                 ← JVM inflationThreshold
  ↓
DelegatingClassLoader 생성           ← 새 ClassLoader가 메모리에 추가
  ↓
Timer-0 참조 체인에 연결             ← (07장) GC Root → GC 불가
```

## 4. 동기화 코드 분석 (04장 복습)

```java
// 우리 코드의 ReadWriteLock
private final ReentrantReadWriteLock rwl = new ReentrantReadWriteLock();
private final Lock r = rwl.readLock();
private final Lock w = rwl.writeLock();
```

!!! warning "누가 언제 락을 잡나"

    **Timer-0 스레드:**
    isModified() -> 파일 변경 감지 -> 변경 있으면 refresh() 호출 -> w.lock() (쓰기 락) -> SqlSessionFactory 재빌드 -> w.unlock()

    **http-nio 스레드들 (200개):**
    사용자 요청 처리 -> SQL 실행 필요 -> getParentObject() 호출 -> r.lock() (읽기 락) -> SqlSessionFactory 사용 -> r.unlock()

    **충돌 시나리오:**
    Timer-0이 refresh() 중 (w.lock) -> http 스레드가 getParentObject() 시도 -> r.lock() 대기 (BLOCKED 상태) -> refresh 끝나면 읽기 가능

    **이것도 성능 이슈:**
    refresh()가 오래 걸리면 -> 그 동안 모든 HTTP 요청이 BLOCKED -> 사용자한테는 "서버 느림"으로 체감

## 5. Bean 중복 = Timer 중복

```
MAT에서 확인한 것:
  Timer-0: RefreshableSqlSessionFactoryBean #1 (18.5MB)
  Timer-1: RefreshableSqlSessionFactoryBean #2 (16.7MB)

왜 2개?
  → context-sqlMap.xml이 2번 읽힘
  → Root Context + Servlet Context에서 각각 Bean 생성
  → Bean 2개 → Timer 2개 → 누수 속도 2배
```

## 7. 해결: 모든 개념의 연결

```
문제:
  Timer-0 (살아있는 스레드 = GC Root)
    → TimerTask (task)
      → RefreshableSqlSessionFactoryBean
        → mapperLocations → 리플렉션 → ClassLoader 누적

해결 방법 3가지와 각각의 원리:

방법 1: interval=0 설정
  → resetInterval()에서 timer.schedule() 호출 안 함
  → TimerTask 실행 안 됨
  → 리플렉션 안 발생
  → ClassLoader 안 생김
  → 하지만 Timer 객체 자체는 생성됨 (불필요한 리소스)

방법 2: SqlSessionFactoryBean으로 교체 (우리가 선택한 방법)
  → Timer 자체가 생성 안 됨
  → Timer-0 스레드 없음
  → GC Root 하나 줄어듦
  → 참조 체인 자체가 없음
  → 가장 근본적 해결

방법 3: Timer 대신 ScheduledExecutorService + 파일 변경 감시 개선
  → 리플렉션 없는 방식으로 파일 감시
  → 근데 운영에서 감시 기능 자체가 불필요
  → 오버엔지니어링
```

!!! tip "최선의 해결책"

    **"불필요한 기능은 끄는 게 아니라 제거한다"**

    - interval=0은 "끄는 것" -- Timer 객체 남아있음
    - SqlSessionFactoryBean 교체는 "제거" -- 깔끔함

    운영 서버에서 개발 편의 기능이 돌아가면 안 된다. 이번 사례가 그 증거다.

## 9. 핵심 정리

!!! abstract "핵심 정리"

    **이번 사례에서 사용된 스레드 개념:**

    - **01장**: 프로세스(Tomcat) 안에 스레드 여러 개
    - **02장**: new Timer(true) -> 내부에서 Thread 생성 + start()
    - **03장**: Timer-0은 RUNNABLE <-> TIMED_WAITING 무한 반복
    - **04장**: ReadWriteLock으로 refresh/getObject 동기화
    - **05장**: timer.schedule(task, 0, 1000) -> 1초마다 반복
    - **06장**: Tomcat은 스레드풀(200개)로 요청 처리
    - **07장**: Timer-0 = GC Root -> 참조 체인 GC 불가 -> 누적

    **결론:** 스레드 하나 잘못 만들면 서버가 죽는다. 개발 편의 기능이라고 가볍게 보면 안 된다. 운영 환경에 불필요한 스레드는 존재 자체가 위험이다.
