# 01. 스레드란 무엇인가

**이 장의 목표**: "스레드가 뭐야?"라고 물으면 3초 안에 대답할 수 있다

---

## 1. 프로세스부터 이해하자

### 1.1 프로세스란

프로그램을 실행하면 **프로세스**가 된다.

```
프로그램: 디스크에 저장된 파일 (chrome.exe, java.exe)
           → 아직 실행 안 됨, 그냥 파일

프로세스: 실행 중인 프로그램
           → 메모리에 올라가서 CPU가 처리 중
           → 운영체제가 관리

비유:
  프로그램 = 요리 레시피 (종이에 적혀 있음)
  프로세스 = 실제로 요리하고 있는 상태 (불 켜고 재료 손질 중)
```

### 1.2 프로세스의 구성

!!! note "프로세스 1개"

    - **코드(Code) 영역**: 실행할 코드 -- 프로그램 명령어가 저장됨
    - **데이터(Data) 영역**: 전역 변수 -- static 변수, 상수
    - **힙(Heap) 영역**: 동적 할당 -- new로 만든 객체
    - **스택(Stack) 영역**: 지역 변수 -- 메서드 호출 시 생기는 변수

### 1.3 프로세스끼리는 독립적

!!! note "프로세스끼리는 독립적"

    | 크롬 | 카카오톡 | IntelliJ |
    |------|---------|----------|
    | 메모리 A | 메모리 B | 메모리 C |
    | ↕ 불가 | ↕ 불가 | ↕ 불가 |

    - 각 프로세스는 자기 메모리만 쓴다
    - 크롬이 죽어도 카카오톡은 안 죽는다
    - 서로 직접 메모리 공유 불가

## 2. 스레드란

### 2.1 정의

**스레드(Thread)** = 프로세스 안에서 실행되는 작업 단위

```
프로세스 = 공장
스레드   = 공장 안의 직원

공장(프로세스) 1개 안에 직원(스레드)이 여러 명 있을 수 있다.
직원들은 같은 공장의 시설(메모리)을 함께 쓴다.
```

### 2.2 프로세스와 스레드 비교

!!! note "프로세스 vs 스레드"

    **프로세스:**

    - 독립적인 메모리 공간
    - 다른 프로세스와 메모리 공유 안 됨
    - 생성 비용 큼 (무거움)
    - 하나 죽어도 다른 프로세스 영향 없음

    **스레드:**

    - 프로세스 안에서 실행
    - 같은 프로세스의 메모리를 공유
    - 생성 비용 작음 (가벼움)
    - 하나 죽으면 같은 프로세스 내 다른 스레드에 영향

### 2.3 스레드가 공유하는 것 / 안 하는 것

!!! note "프로세스 메모리"

    **[공유하는 것]** -- 모든 스레드 공유

    - 코드 영역 (실행할 코드)
    - 데이터 영역 (static 변수)
    - 힙 영역 (new 객체)

    **[각자 가지는 것]**

    | 스택 #1 (스레드1) | 스택 #2 (스레드2) | 스택 #3 (스레드3) |
    |---|---|---|

    + 각 스레드마다 PC Register(실행 위치), Native Stack 별도

**핵심:**
- **힙은 공유** → 스레드 A가 만든 객체를 스레드 B도 접근 가능
- **스택은 별도** → 각 스레드가 호출한 메서드의 지역 변수는 자기만 접근

## 3. 왜 스레드를 쓰는가

### 3.1 동시에 여러 일을 하고 싶다

```
스레드 없이 (싱글 스레드):
  1. 파일 다운로드 (10초)
  2. 다운로드 끝날 때까지 화면 멈춤
  3. 다운로드 완료 후에야 다른 작업 가능

스레드 있으면 (멀티 스레드):
  메인 스레드:     화면 계속 응답
  다운로드 스레드:  파일 다운로드 (10초)
  → 화면 안 멈추고 다운로드도 됨
```

### 3.2 웹 서버에서의 스레드

```
Tomcat (우리 서버):

  사용자 A 요청 → 스레드 #1이 처리
  사용자 B 요청 → 스레드 #2가 처리
  사용자 C 요청 → 스레드 #3이 처리

  스레드가 1개뿐이면?
  → 사용자 A 처리 끝날 때까지 B, C는 대기
  → 현실에서 이러면 서비스 불가

  스레드가 여러 개면?
  → A, B, C 동시 처리
  → 이게 웹 서버가 동작하는 원리
```

### 3.3 메인 스레드

```
Java 프로그램을 실행하면:

  java MyApp
       ↓
  JVM이 프로세스 생성
       ↓
  main 스레드 자동 생성
       ↓
  public static void main(String[] args) 실행
       ↓
  이게 '메인 스레드'

모든 Java 프로그램은 최소 1개의 스레드(main)를 가진다.
추가 스레드는 코드에서 직접 만들어야 한다.
```

## 4. 실생활 비유로 정리

!!! example "실생활 비유"

    **회사(프로세스)**

    - 사장(메인 스레드): 프로그램 시작과 끝을 관리
    - 직원A(스레드1): DB 조회 담당
    - 직원B(스레드2): 파일 감시 담당
    - 직원C(스레드3): 로그 기록 담당

    **같은 사무실(힙 메모리)에서 일하니까:**

    - 같은 프린터(공유 자원) 사용 가능
    - 근데 동시에 프린터 쓰면 충돌(동기화 문제)

    **각자 책상(스택)은 따로 있으니까:**

    - 자기 메모 장은 남이 못 봄

## 5. 우리 서버의 스레드

```
LMS 서버 (Tomcat 프로세스) 안의 스레드들:

  main             → Tomcat 시작
  http-nio-*       → 사용자 요청 처리 (200개)
  Timer-0          → SQL XML 파일 감시 ← 문제의 원흉
  Timer-1          → SQL XML 파일 감시 (Bean 중복으로 2개)
  WhaTap-*         → APM 모니터링
  EXEM-*           → APM 모니터링
  GC Thread        → 가비지 컬렉션
  ...

Timer-0, Timer-1이 여기서 나온 거야.
이놈들이 1초마다 파일 감시하면서 메모리 누수를 일으킨 거.
```

## 6. 핵심 정리

!!! abstract "핵심 정리"

    - **프로세스**: 실행 중인 프로그램. 독립된 메모리 공간.
    - **스레드**: 프로세스 안의 실행 단위. 메모리를 공유함.
    - **왜 쓰는가**: 동시에 여러 일을 하려고
    - **위험한 점**: 공유 자원 접근 시 충돌 가능
    - **Java 최소 스레드**: main 1개
    - **우리 서버 문제**: Timer-0 스레드가 메모리 누수 원인

    **다음 장:** 스레드를 어떻게 만들고 실행하는지 -- Thread, Runnable, start()
