# 01. 메모리의 기초 중의 기초 - Alpha

---

## 1. 메모리가 뭐야? - "이게 뭐야?"

### 비유로 문 열기

책상을 생각해봐.

너 지금 공부하려면 어떻게 해? 책장에서 책을 꺼내서 **책상 위에 올려놓고** 펼쳐야 작업할 수 있잖아. 책장에 꽂혀 있는 상태로는 아무것도 못 해. 눈으로 볼 수도 없고, 펜으로 쓸 수도 없어.

!!! note "책장(디스크) vs 책상(메모리)"
    | 구분 | 책장 (디스크) | 책상 (메모리) |
    |------|-------------|-------------|
    | 역할 | 보관만 가능 | 여기서만 작업 가능 |
    | 용량 | 많이 저장 가능 | 공간 제한 있음 |
    | 속도 | 느림 | 빠름 |

    책장에서 책을 꺼내서 책상 위에 올려놓고 펼쳐야 작업할 수 있다.

비유는 여기까지. 진짜 이야기 하자.

### 본질

**메모리(RAM) = CPU가 직접 접근할 수 있는 작업 공간이다.**

컴퓨터에서 모든 작업은 CPU가 한다. 계산도, 비교도, 데이터 처리도 전부 CPU가 해. 근데 CPU가 일하려면 **데이터가 있어야** 하잖아. 그 데이터를 어디서 읽어오냐? 바로 **메모리**에서.

디스크(HDD/SSD)에도 데이터가 있지만, CPU가 디스크에서 직접 데이터를 읽어서 작업하지는 않아. 무조건 **디스크 -> 메모리로 복사** -> CPU가 메모리에서 읽어서 처리. 이 순서야.

!!! danger "데이터 흐름 (절대 법칙)"
    ```mermaid
    flowchart LR
        A["디스크<br/>(저장소)"] -->|복사| B["메모리(RAM)<br/>(작업 공간)"]
        B -->|읽기| C["CPU<br/>(처리기)"]
    ```
    CPU는 디스크에서 직접 못 읽어. 무조건 메모리를 거쳐야 해.

왜 이렇게 해야 하냐고? 속도 때문이야. 디스크는 느려. RAM은 빨라. CPU는 **미친 듯이 빨라**. CPU가 느린 디스크에서 직접 읽으면 CPU가 놀아. 그래서 빠른 메모리에 데이터를 올려놓고, CPU가 거기서 바로바로 읽는 거야.

## 2. 왜 메모리가 필요해? - "어떻게 돌아가?"

### "프로그램 실행한다" = 디스크에서 메모리로 올린다

이거 정확히 이해해야 해.

너 카카오톡 더블클릭하면 뭐가 일어나는지 알아? "카카오톡이 실행된다"고? 정확히 말하면 이런 일이 벌어져:

!!! note "프로그램 실행 과정 (단순화)"
    1. 디스크에 있는 KakaoTalk.exe 파일을 찾는다
    2. OS가 메모리(RAM) 공간을 확보한다 - "야, 이 프로그램 올릴 자리 만들어"
    3. 디스크에서 프로그램 코드를 메모리로 복사한다 (실행에 필요한 코드, 데이터 전부)
    4. CPU가 메모리에 올라간 코드를 한 줄씩 실행한다
    5. 실행 중에 필요한 데이터도 메모리에서 읽고 쓴다

    **결론: 메모리에 안 올라간 프로그램은 실행 불가. 끝.**

**프로그램 = 디스크에 저장된 파일** (그냥 데이터 덩어리)
**프로세스 = 메모리에 올라가서 실행 중인 프로그램** (살아있는 녀석)

이 차이 중요해. 02장에서 더 자세히 다루지만, 여기서 핵심만 짚으면:

- 디스크에 있는 건 그냥 "레시피(설계도)"야. 아무 일도 안 일어나.
- 메모리에 올라가는 순간 "요리 시작(실행)"이야.

### 서버에서 메모리가 특히 중요한 이유

PC는 혼자 쓰니까 메모리가 좀 부족해도 "좀 느려지네~" 하고 넘어가. 근데 서버는 달라.

!!! warning "서버 vs PC의 메모리 사용 차이"
    | 구분 | PC | 서버 |
    |------|-----|------|
    | 사용자 | 1명 (너) | 수십~수백 명 동시 |
    | 동시 부하 | 5~10개 프로그램 | 수백~수천 개 요청 |
    | 메모리 부족 시 | "좀 느려지네" → 프로그램 하나 끄면 됨 | 서비스 장애 → 전체 사용자 영향 → 접속 불가, 5xx 에러, 전화 폭주 |

    우리 LMS 서버? 학생 수백 명이 동시에 강의 듣고, 출석 체크하고, 과제 제출하고, 퀴즈 풀고 있어. 이 모든 요청이 전부 메모리를 먹어.

서버에서 메모리가 부족하면? 그게 바로 **장애**야. 그래서 서버 메모리를 이해하고 모니터링하는 게 중요한 거야.

## 3. RAM의 특성 - "진짜 정체가 뭐야?"

### RAM = Random Access Memory

이름부터 뜯어보자.

**Random Access**: 아무 위치나 바로 접근 가능하다는 뜻이야.

이게 무슨 말이냐면, 메모리 주소 0번이든 100만번이든 접근 속도가 **동일**하다는 거야. 카세트테이프처럼 처음부터 감아서 찾을 필요 없이, 원하는 위치를 바로 콕 찍어서 접근 가능.

!!! note "Random Access vs Sequential Access"
    **카세트테이프 (Sequential)**: 3번 곡 들으려면 1번 건너뛰고 2번 건너뛰고... 느려!

    **RAM (Random Access)**: 주소 #99832 읽으려면 바로 #99832 접근. 끝. 빨라!

    ```
    [ 0 ][ 1 ][ 2 ][ 3 ][ ... ][ 99k ][ ... ][ N ]  <-- 어디든 동일 시간
                                   ^
                                   |
                         바로 여기! (주소만 알면 즉시 접근)
    ```

### RAM의 핵심 특성 3가지

| 특성 | 설명 | 왜 중요해? |
|------|------|-----------|
| **빠르다** | 디스크보다 10만~100만 배 빠름 | CPU가 놀지 않고 일할 수 있음 |
| **휘발성** | 전원 끄면 데이터 전부 날아감 | 그래서 DB에 저장하는 거야 |
| **비싸다** | 같은 용량 대비 디스크보다 훨씬 비쌈 | 그래서 서버에 RAM 수백 GB 못 넣어 |

**휘발성(Volatile)**이 핵심이야. 전원 끄면 RAM에 있던 거 다 날아가. 그래서:

- 중요한 데이터는 **디스크(DB)에 저장**해야 해
- RAM은 순수하게 **작업 공간**이야. 영구 보관소가 아니야.
- 서버가 재시작되면 메모리에 있던 세션, 캐시 같은 거 전부 초기화돼

## 4. 디스크 vs RAM - "속도 차이가 얼마나 나는데?"

여기서 감이 안 오는 사람 많아. 숫자로 보여줄게.

### 속도 비교 (대략적 수치)

!!! abstract "저장 장치 속도 비교"
    | 장치 | 접근 시간 | 비유 |
    |------|----------|------|
    | CPU 레지스터 | ~0.3 ns | 눈 깜빡 (반사적) |
    | L1 캐시 | ~1 ns | 책상 위 메모지 확인 |
    | L2 캐시 | ~3-5 ns | 책상 서랍 열기 |
    | L3 캐시 | ~10-20 ns | 책상 옆 선반 |
    | RAM | ~50-100 ns | 방 안 책장에서 꺼내기 |
    | SSD | ~100,000 ns | 1층 창고 다녀오기 |
    | HDD | ~10,000,000 ns | 다른 건물 창고 다녀오기 |

    ※ ns = 나노초 = 10억분의 1초

    - RAM vs HDD 속도 차이: 약 **100,000배** (10만 배)
    - RAM vs SSD 속도 차이: 약 **1,000배**

이걸 사람 시간으로 변환하면 체감이 확 와.

### 사람 시간으로 변환 (RAM 접근 = 1초라고 치면)

!!! tip "RAM 접근이 1초라면, 다른 건 얼마나 걸릴까?"
    | 장치 | 시간 환산 | 비유 |
    |------|----------|------|
    | CPU 레지스터 | 0.003초 | 찰나 |
    | L1 캐시 | 0.01초 | 눈 깜빡 |
    | RAM | 1초 | 기준 |
    | SSD | 약 17분 | 편의점 다녀오기 |
    | HDD | 약 28시간 | 서울에서 부산 왕복 |

    CPU가 RAM 대신 HDD에서 데이터 읽으면? = **1초 걸릴 작업이 28시간** 걸린다는 뜻이야. 그래서 메모리에 올려놓고 작업하는 거야.

### 속도 vs 용량 vs 비용: 트레이드오프

세상에 공짜는 없어. 빠르면 비싸고 용량 적어. 느리면 싸고 용량 커.

| 장치 | 속도 | 용량 (서버 기준) | 가격 (GB당) | 영속성 |
|------|------|-----------------|------------|--------|
| RAM | 매우 빠름 | 16~256 GB | 비쌈 (~$5/GB) | 휘발성 (전원 끄면 삭제) |
| SSD | 빠름 | 256 GB ~ 수 TB | 중간 (~$0.1/GB) | 영구 저장 |
| HDD | 느림 | 1~20 TB | 쌈 (~$0.02/GB) | 영구 저장 |

이 트레이드오프 때문에 컴퓨터는 **계층 구조**로 메모리를 관리해. 바로 다음에 나올 내용이야.

## 5. 바이트 단위 체계 - "숫자를 읽을 줄 알아야지"

서버 모니터링할 때 "8.4Gi", "6.2Gi", "145Mi" 이런 숫자가 나와. 이거 뭔지 모르면 아무것도 못 읽어.

### 기본 단위: 바이트(Byte)

!!! note "바이트 단위 체계"
    | 단위 | 크기 | 설명 |
    |------|------|------|
    | 1 bit | 0 또는 1 | 컴퓨터가 이해하는 최소 단위 |
    | 1 Byte | 8 bits | 영문 글자 1개 저장할 수 있는 크기 |
    | 1 KB | 1,024 Bytes | 약 1천 바이트 |
    | 1 MB | 1,024 KB | 약 100만 바이트 |
    | 1 GB | 1,024 MB | 약 10억 바이트 |
    | 1 TB | 1,024 GB | 약 1조 바이트 |

    왜 1,000이 아니라 1,024? 컴퓨터는 2진수야. 2^10 = 1,024. 그래서 컴퓨터 세계에서 "킬로"는 1,024야.

### 실감나는 비교

!!! example "각 단위가 실제로 얼마나 큰지"
    | 단위 | 실감 비교 |
    |------|----------|
    | 1 B | 영문 글자 1개 ("A") |
    | 1 KB | 짧은 이메일 1통 (글자만) |
    | 1 MB | 고해상도 사진 1장 |
    | 1 GB | 영화 1편 (SD 화질) |
    | 1 TB | 영화 약 1,000편 |

    우리 LMS 서버 RAM: total: 15Gi = 약 15 GB = 영화 15편 분량의 작업 공간. 이 안에서 톰캣, JVM, OS 프로세스 다 돌아감.

### KB vs KiB — "이건 뭔 차이야?"

리눅스 `free -h` 출력에서 **Gi**, **Mi** 이런 단위가 나와. 이건 뭐냐면:

| 표기 | 의미 | 기준 |
|------|------|------|
| KB (Kilobyte) | 1,000 Bytes | 10진수 (SI 단위) |
| KiB (Kibibyte) | 1,024 Bytes | 2진수 (IEC 단위) |
| MB | 1,000,000 Bytes | 10진수 |
| MiB | 1,048,576 Bytes | 2진수 (1024 x 1024) |
| GB | 1,000,000,000 Bytes | 10진수 |
| GiB | 1,073,741,824 Bytes | 2진수 (1024^3) |

`free -h` 에서 `15Gi`라고 나오면 **15 GiB = 15 x 1,024^3 바이트**라는 뜻이야.

실무에서는 이 차이가 큰 문제가 되진 않지만, "Gi가 뭐야?"라는 질문에 대답 못 하면 기초가 없는 거야. 알고 넘어가.

## 6. 메모리 계층 구조 - "컴퓨터의 기억 창고 시스템"

### 비유: 회사의 서류 보관 체계

!!! note "회사 비유로 이해하는 메모리 계층"
    | 비유 | 실제 | 특징 |
    |------|------|------|
    | 머릿속 암기 | 레지스터/L1 캐시 | 초고속. 근데 많이 못 외움. |
    | 책상 위 서류 | L2/L3 캐시 | 손만 뻗으면 바로. 공간 한계. |
    | 사무실 캐비닛 | RAM | 일어나서 꺼내야 하지만 빠름. 적당한 용량. |
    | 지하 창고 | SSD / HDD | 엘리베이터 타고 내려가야 함. 느리지만 다 보관 가능. |

비유 끝. 본질 가자.

### 실제 메모리 계층 구조

!!! abstract "실제 메모리 계층 구조"
    빠름, 비쌈, 작음 ↔ 느림, 쌈, 큼

    | 계층 | 용량 | 속도 |
    |------|------|------|
    | CPU 레지스터 (수십 개) | 수십~수백 B | ~0.3ns |
    | L1 캐시 (코어당) | 32~64 KB | ~1ns |
    | L2 캐시 (코어당) | 256 KB~1 MB | ~3-5ns |
    | L3 캐시 (CPU 공유) | 수 MB~수십 MB | ~10-20ns |
    | RAM - 우리 서버: 16GB | 수 GB~수백 GB | ~50-100ns |
    | SSD | 수백 GB~수 TB | ~0.1ms |
    | HDD | 수 TB~수십 TB | ~10ms |

### 왜 이런 구조를 쓰냐?

간단해. **전부 레지스터 속도로 만들면 비용이 미친 듯이 비싸서**.

레지스터 수준의 속도를 16GB 만들면? 서버 한 대 가격이 수십억 원이야. 현실적으로 불가능해.

그래서 타협한 거야:
- 가장 자주 쓰는 데이터는 **빠르고 비싼** 저장소에
- 덜 쓰는 데이터는 **느리고 싼** 저장소에

이걸 **지역성(Locality)** 원리라고 해:

| 원리 | 설명 | 예시 |
|------|------|------|
| **시간적 지역성** | 최근 사용한 데이터는 또 쓸 확률 높음 | for 루프에서 변수 i를 계속 씀 |
| **공간적 지역성** | 근처 데이터도 같이 쓸 확률 높음 | 배열 순회하면 연속된 주소를 읽음 |

CPU는 이 원리를 이용해서, "이거 곧 또 쓸 것 같으니까 캐시에 올려놓자"라고 판단해. 이게 캐시가 효과적인 이유야.

## 7. 서버에서 이게 왜 중요해? - "코드로 보자"

### free 명령어 맛보기

나중에 03장에서 상세하게 다루지만, 맛만 보자. 리눅스 서버에서 `free -h` 치면 이런 게 나와:

```bash
$ free -h
              total        used        free      shared  buff/cache   available
Mem:           15Gi       8.4Gi       1.0Gi       0.0Ki       6.2Gi       6.9Gi
Swap:         4.0Gi       145Mi       3.9Gi
```

지금은 이것만 알면 돼:
- `total: 15Gi` = 이 서버에 RAM이 약 15 GB 있다
- `used: 8.4Gi` = 그 중 약 8.4 GB를 쓰고 있다
- 나머지는 03장에서 필드 하나하나 다 뜯어본다

### 서버 메모리 부족하면 벌어지는 일

!!! danger "서버 메모리 부족 시 단계별 증상"
    **[1단계] 느려짐**
    : 응답 시간 증가. 페이지 로딩 2초 → 5초 → 10초. 사용자: "뭐야 왜 이렇게 느려"

    **[2단계] Swap 사용 시작**
    : RAM 부족 → 디스크를 RAM처럼 사용 (Swap). 디스크는 RAM보다 1,000~100,000배 느림. 급격한 성능 저하.

    **[3단계] OOM Killer 발동**
    : Out Of Memory Killer. OS가 메모리 많이 먹는 프로세스를 강제 종료. 톰캣이 죽을 수 있음 = 서비스 장애.

    **[4단계] 서버 먹통**
    : SSH 접속도 안 됨. 재시작 필요. 학생들: "강의 안 들어가요!!"

이래서 서버 메모리를 이해하고 모니터링해야 하는 거야. "메모리 55%인데 괜찮아요?"라는 질문에 제대로 답하려면, 메모리가 뭔지부터 알아야 해. 그게 이 챕터야.

## 8. 주의사항 / 함정

### 함정 1: "메모리 = 저장공간"이라는 오해

!!! warning "함정 1: '메모리 = 저장공간'이라는 오해"
    "메모리가 부족해요" → "하드 용량이 부족한 거 아니에요?" **완전히 다른 거야.**

    | 구분 | 메모리 (RAM) | 저장공간 (디스크) |
    |------|-------------|-----------------|
    | 역할 | 실행 중인 프로그램의 작업 공간 | 파일, DB 데이터 보관 |
    | 영속성 | 전원 끄면 사라짐 | 전원 꺼도 유지됨 |
    | 용량 | 서버에서 보통 16~64 GB | 서버에서 보통 수백 GB ~ 수 TB |

    **"메모리 부족" ≠ "디스크 용량 부족"**. 이거 헷갈리면 서버 운영 못 해.

### 함정 2: "RAM 크면 무조건 좋다"는 오해

RAM이 크면 좋긴 해. 근데 무한정 크게 할 수 없어:
- **비용**: RAM은 비싸. 서버 한 대에 256GB 넣으면 비용이 어마어마해.
- **효율**: 특정 지점 이후에는 RAM을 늘려도 성능 향상이 없어.
- **GC 문제**: JVM 환경에서 힙을 너무 크게 잡으면 GC(Garbage Collection) 시간이 길어져서 오히려 느려질 수 있어. (이건 07장에서 다룸)

### 함정 3: "free 명령어에서 free가 0이면 위험하다"는 오해

!!! tip "함정 3: free가 0이면 위험하다는 오해"
    **틀림**: "free가 1.0Gi밖에 없어! 메모리 부족이야!"

    **진실**: 리눅스는 남는 메모리를 캐시로 활용해. free가 0에 가까운 건 "정상"일 수 있어. 진짜 봐야 하는 건 "available" 필드야.

    - 이건 03장에서 자세히 다룬다.
    - 지금은 **"free가 낮다고 무조건 위험한 건 아니다"** 이것만 기억.

## 9. 정리

### 핵심 요약 표

| 개념 | 한 줄 정리 |
|------|-----------|
| **메모리(RAM)** | CPU가 직접 접근하는 작업 공간. 빠르고, 휘발성이고, 비싸다. |
| **디스크(HDD/SSD)** | 데이터를 영구 저장하는 공간. 느리고, 비휘발성이고, 싸다. |
| **프로그램 실행** | 디스크에서 메모리로 올리는 것. 메모리에 없으면 실행 불가. |
| **메모리 계층 구조** | 레지스터 → 캐시 → RAM → SSD → HDD. 위로 갈수록 빠르고 비싸고 작다. |
| **바이트 단위** | B → KB(1024) → MB → GB → TB. Gi는 2진수 기준(GiB). |
| **서버 메모리** | 동시 사용자가 많아서 메모리 관리가 PC보다 훨씬 중요하다. |

### 이 챕터에서 반드시 기억할 것

!!! abstract "이 챕터에서 반드시 기억할 것"
    1. 메모리는 CPU의 작업 공간이다. 디스크와 다르다.
    2. 프로그램은 메모리에 올라가야 실행된다.
    3. RAM은 빠르고 휘발성이다.
    4. RAM과 디스크는 속도가 10만 배 차이난다.
    5. 서버는 메모리가 부족하면 장애가 난다.

### 확인 문제 (5문제)

> 다음 문제를 풀어봐. 답 못 하면 위에서 다시 읽어.

**Q1.** "프로그램을 실행한다"는 것을 메모리 관점에서 한 문장으로 설명해봐.

**Q2.** RAM이 디스크보다 빠른데 왜 모든 저장 장치를 RAM으로 안 만들어?

**Q3.** 서버에서 메모리가 부족해지면 OS가 최후의 수단으로 하는 행동은 뭐야?

**Q4.** `free -h` 출력에서 `15Gi`라는 숫자가 나왔다. "Gi"가 의미하는 것은? 일반적인 "GB"와 뭐가 달라?

**Q5.** 메모리 계층 구조에서 "위로 갈수록 빠른데 용량이 작은" 이유는 뭐야? 근본적인 이유 한 가지를 말해봐.

??? success "정답 보기"
    **A1.** 디스크에 있는 프로그램(코드와 데이터)을 메모리(RAM)로 복사해서 올리는 것. CPU는 메모리에 올라간 코드만 실행할 수 있다.

    **A2.** 비용 때문이야. RAM은 같은 용량 대비 디스크보다 훨씬 비싸. 그리고 휘발성이라 전원 끄면 데이터가 날아가기 때문에 영구 저장 용도로는 사용 불가. 속도-용량-비용의 트레이드오프.

    **A3.** OOM Killer (Out Of Memory Killer). 메모리를 많이 차지하는 프로세스를 OS가 강제 종료시킨다. 톰캣 같은 핵심 프로세스가 죽으면 서비스 장애로 이어진다.

    **A4.** Gi = GiB (기비바이트). 2진수 기준으로 1 GiB = 1,024^3 = 1,073,741,824 바이트. 일반 GB는 10진수 기준으로 1 GB = 1,000^3 = 1,000,000,000 바이트. 약 7% 차이가 있어. 리눅스 `free -h`는 2진수 기준(GiB)으로 표시한다.

    **A5.** 물리적으로 빠른 저장 장치(레지스터, SRAM 등)는 제조 비용이 비싸기 때문이야. 속도를 올리려면 트랜지스터 구조가 복잡해지고, 공간도 많이 차지하고, 발열도 커져서, 대용량으로 만드는 게 현실적으로 불가능해. 그래서 빠른 건 소량만, 느린 건 대량으로 배치하는 계층 구조를 쓰는 거야.
