# 02. 서버란 무엇인가 - Alpha

## 1. 서버가 뭐야? - "이게 뭐야?"

### 비유로 문 열기

식당을 생각해봐.

손님(클라이언트)이 들어와서 "된장찌개 주세요"라고 **요청**하면, 주방(서버)에서 된장찌개를 만들어서 **응답**으로 내보내잖아. 손님이 100명이 동시에 오면? 주방이 100명 치 주문을 처리해야 해. 주방 장비가 구려서 느리면? 손님들이 30분씩 기다리다가 "다른 데 가자" 하고 나가버려.

!!! example "서버의 비유: 식당"

    ```mermaid
    sequenceDiagram
        participant 손님 as 손님 (클라이언트)
        participant 주방 as 주방 (서버)
        손님->>주방: "된장찌개요" (요청)
        Note right of 주방: 주문 접수<br/>재료 꺼내기<br/>조리<br/>담기
        주방->>손님: 된장찌개 (응답)
    ```

    - 손님 1명이면? 여유 있어.
    - 손님 100명이면? 주방이 미쳐 돌아가.
    - 주방 냉장고(메모리)가 작으면? 재료 보관 못 해서 터져.

비유는 여기까지. 진짜 이야기 하자.

### 본질

**서버 = 요청(Request)을 받아서 응답(Response)하는 프로그램이 돌아가는 컴퓨터다.**

이 문장에서 핵심 세 개를 뽑아야 해:

1. **요청을 받는다**: 네트워크를 통해 외부에서 뭔가가 들어온다
2. **처리한다**: 받은 요청을 분석하고, DB 조회하고, 로직 돌리고
3. **응답을 보낸다**: 결과를 다시 네트워크를 통해 돌려준다

"서버"라는 단어를 들으면 거대한 컴퓨터를 떠올리는데, **본질은 소프트웨어(프로그램)**야. 네 노트북에서도 톰캣 깔아서 돌리면 그게 서버야. 근데 왜 실제 서비스에서는 전용 장비를 쓰느냐? 이유가 있어.

!!! note "서버의 본질"

    ```mermaid
    flowchart TB
        A["네트워크"] --> B
        subgraph B["서버 프로그램"]
            direction LR
            B1["요청 수신"] --> B2["처리"] --> B3["응답 전송"]
        end
        B --> C["서버 하드웨어 위에서 실행"]
    ```

    **핵심:** 서버 = "서빙(serve)하는 놈"
    클라이언트가 요청하면 서빙해주는 프로그램.
    그 프로그램이 돌아가는 컴퓨터도 통째로 "서버"라고 불러.

우리 LMS에서 학생이 "강의 목록 보여줘"하고 브라우저에서 URL을 치면:

1. 학생의 브라우저(클라이언트)가 HTTP 요청을 보냄
2. 우리 WAS01 서버의 톰캣(서버 프로그램)이 받음
3. 톰캣이 스프링 컨트롤러 호출 -> 서비스 -> DAO -> DB 조회
4. 결과를 JSP로 렌더링해서 HTML 응답을 보냄
5. 학생 브라우저에 강의 목록이 뜸

이 전체 과정이 **서버가 하는 일**이야.

## 2. 서버 vs PC - "뭐가 달라?"

### 겉으로 보면 그냥 컴퓨터잖아?

맞아. 서버도 CPU, RAM, 디스크 있고 OS 깔려있어. 근데 **목적이 다르면 설계가 달라**. 스포츠카와 트럭의 차이를 생각해봐. 둘 다 자동차인데, 스포츠카는 한 명이 빠르게, 트럭은 무거운 짐을 안정적으로.

!!! note "서버 vs PC 비교"

    | 항목 | PC (네 노트북) | 서버 (우리 WAS01) |
    |------|----------------|-------------------|
    | **CPU** | 4~8코어 (혼자 쓰니까) | 8~64코어 (동시 요청 수백 개 처리) |
    | **RAM** | 8~16 GB (프로그램 몇 개) | 16~256 GB (수백 동시 접속 처리) / 우리 WAS01: 15 GB |
    | **디스크** | 256GB~1TB SSD (개인 파일) | 수백 GB ~ 수 TB (로그, 데이터, 영상) |
    | **가동시간** | 쓸 때만 켬 (하루 8~12시간?) | 24시간 365일 (연중무휴, 전원 끄면 서비스 중단) |
    | **사용자 수** | 1명 (너) | 수십~수천 명 동시 |
    | **안정성** | 가끔 블루스크린 "재부팅하면 되지" | 죽으면 전체 서비스 중단 "재부팅하면 학생 수백 명 접속끊김" |
    | **모니터** | 있음 (화면 보고 작업) | 보통 없음 (SSH로 원격 접속) |
    | **OS** | Windows | Linux (Ubuntu, CentOS 등) |

### 항목별 왜 다른지

| 구분 | PC의 목적 | 서버의 목적 |
|------|-----------|-------------|
| 핵심 | 1명이 편하게 | 다수가 동시에 안정적으로 |
| CPU | 게임/편집 클럭 속도 중요 | 코어 수 중요 (동시 처리) |
| RAM | 프로그램 몇 개 돌리기 | 수백 세션 동시 유지 |
| 가동 | 필요할 때만 | 24/7/365 |
| 장애 | "에이 다시 켜야지" | "전화 폭주, 학생 수강 불가, 교수 항의" |

서버에서 RAM이 특히 중요한 이유:

!!! warning "서버 RAM이 특히 중요한 이유"

    - 학생 1명 접속 = 세션 1개 생성 = 메모리에 세션 데이터 유지
    - 학생 100명 접속 = 세션 100개 = 메모리 100배
    - 학생 500명 동시 접속 + 강의 재생 + 출석 체크 + 과제 제출 = 톰캣 스레드 수백 개 = 각 스레드마다 메모리 사용

    PC는 메모리 부족하면 네가 프로그램 하나 끄면 됨.
    서버는 메모리 부족하면 학생 수백 명이 동시에 피해 봄.
    그래서 서버는 메모리를 넉넉하게 잡고, 철저히 모니터링하는 거야.

### 서버실 이야기

서버는 보통 **서버실(데이터센터)**에 있어. 일반 방에서 돌리면 안 되냐고? 안 돼.

- **온도**: 서버는 열을 엄청 뿜어. 에어컨 없으면 과열돼서 죽어.
- **전원**: UPS(무정전 전원장치)가 있어야 정전 시에도 서비스 유지.
- **네트워크**: 일반 인터넷이 아니라 전용 회선으로 빠른 속도 보장.
- **보안**: 물리적 출입 통제.

우리 LMS 서버(WAS01)도 IDC(Internet Data Center)에 있어. 너 로컬에서 개발하고, 배포하면 그 IDC에 있는 서버에 올라가는 거야. SSH로 접속해서 관리하는 거고.

### 왜 리눅스를 쓰냐?

!!! tip "왜 서버에 리눅스를 쓰는가?"

    1. **무료** (라이선스 비용 0원) -- Windows Server 라이선스? 수백만 원. 서버 100대 운영하면? 비용 차이 어마어마
    2. **가볍다** -- GUI 없이 터미널만으로 운영 가능. GUI 없으면 그만큼 메모리/CPU 절약
    3. **안정적이다** -- 수개월~수년 재부팅 없이 가동 가능. 블루스크린? 그런 거 없어
    4. **원격 관리 편함** -- SSH로 어디서든 터미널 접속. 명령어 몇 줄이면 서버 상태 파악 가능
    5. **생태계** -- 톰캣, Nginx, MySQL, Redis 전부 리눅스에 최적화. 서버 소프트웨어 대부분이 리눅스 우선 지원

## 3. 프로세스란 무엇인가 - "이게 뭐야?"

### 프로그램 vs 프로세스 -- 이 차이를 확실히 해야 해

01장에서 잠깐 나왔는데, 여기서 완전히 파고든다.

!!! note "프로그램 vs 프로세스"

    **프로그램 (Program)**

    디스크에 저장된 실행 파일. 그냥 파일이야. 죽어있는 거야.

    - `C:\Program Files\KakaoTalk\KakaoTalk.exe` (디스크에 있음)
    - `/usr/local/tomcat/bin/bootstrap.jar` (디스크에 있음)
    - 아무 일도 안 일어남. 그냥 거기 놓여 있을 뿐.

    ---

    **프로세스 (Process)**

    프로그램이 메모리에 올라가서 실행 중인 상태. 살아있는 녀석.

    - KakaoTalk.exe를 더블클릭 -> 메모리에 올라감 -> 프로세스 탄생
    - bootstrap.jar 실행 -> 톰캣 프로세스 탄생 (PID 27154)
    - CPU 시간을 쓰고, 메모리를 차지하고, 네트워크를 쓰고.

비유 하나 더 해볼게.

!!! example "레시피 vs 요리 비유"

    **프로그램 = 레시피** (종이에 적힌 요리법)

    - 레시피가 있다고 음식이 나오냐? 안 나와.
    - 레시피를 보고 실제로 재료 꺼내서 조리해야 음식이 나와.

    **프로세스 = 실제 요리하고 있는 과정**

    - 재료(데이터) 쓰고 있고
    - 가스불(CPU) 쓰고 있고
    - 조리대(메모리) 차지하고 있고
    - 시간이 흐르면서 상태가 변하고 있어

    같은 레시피로 두 명이 동시에 요리할 수 있어? 당연하지! 같은 프로그램으로 프로세스 여러 개 만들 수 있어. 톰캣 인스턴스 2개 띄우면? 프로세스 2개. PID도 다르고.

비유 끝. 본질 가자.

### 프로세스의 정확한 정의

**프로세스 = OS가 관리하는 실행 단위. 독립된 메모리 공간과 자원을 할당받아 실행 중인 프로그램.**

핵심 속성:

| 속성 | 설명 |
|------|------|
| **독립된 메모리 공간** | 각 프로세스는 자기만의 메모리를 가짐. 다른 프로세스 메모리에 함부로 접근 불가. |
| **OS가 관리** | 프로세스 생성, 스케줄링, 종료 전부 OS가 담당. |
| **자원 할당** | CPU 시간, 메모리, 파일 핸들, 네트워크 소켓 등을 할당받음. |
| **고유 식별자** | PID(Process ID)로 구분. 중복 없음. |

프로세스가 가지는 것들:

!!! abstract "프로세스가 가지는 자원들 (PID: 27154)"

    **메모리 영역:**

    | 영역 | 내용 |
    |------|------|
    | 코드 영역 | 실행할 프로그램 코드 (기계어) |
    | 데이터 영역 | 전역 변수, static 변수 |
    | 힙 영역 | 동적으로 할당한 메모리 (new) |
    | 스택 영역 | 함수 호출 정보, 지역 변수 |

    **프로세스 속성:**

    | 속성 | 값 |
    |------|-----|
    | PID | 27154 |
    | PPID | 1 (부모 프로세스 ID) |
    | UID | tomcat (실행한 사용자) |
    | 상태 | Running / Sleeping / Zombie 등 |
    | 열린 파일들 | 로그 파일, 설정 파일 등 |
    | 네트워크 소켓 | 8080 포트 리스닝 등 |

## 4. PID, PPID, 부모-자식 관계 - "프로세스도 족보가 있다"

### PID (Process ID)

**PID = 프로세스의 주민등록번호.** OS가 프로세스를 구분하기 위해 붙이는 고유 번호야.

- 프로세스가 생성될 때 OS가 자동으로 할당
- 시스템에서 유일함 (같은 PID 두 개 없음)
- 프로세스가 죽으면 그 PID는 나중에 재사용될 수 있음
- 숫자로 되어 있음 (1, 27154, 1432 같은)

이 번호로 뭘 할 수 있냐:

```bash
kill 27154             # 이 프로세스 종료 (SIGTERM)
kill -9 27154          # 이 프로세스 강제 종료 (SIGKILL)
top -p 27154           # 이 프로세스만 실시간 모니터링
jstat -gc 27154        # 이 JVM의 GC 상태 확인
cat /proc/27154/status # 이 프로세스 상세 정보 조회
```

### 특수한 PID

| PID | 이름 | 역할 |
|-----|------|------|
| **PID 0** | swapper/scheduler | 커널 자체. ps에 안 나옴. |
| **PID 1** | systemd (또는 init) | 모든 프로세스의 조상. 시스템 부팅 시 OS가 가장 먼저 실행. |

### PPID (Parent Process ID)

**PPID = 부모 프로세스의 PID.** 이 프로세스를 만들어준 놈이 누구냐.

리눅스에서 프로세스는 **무에서 생기지 않아**. 반드시 기존 프로세스가 새 프로세스를 만들어(fork). 만들어준 놈이 부모, 만들어진 놈이 자식이야.

!!! note "프로세스 부모-자식 관계 (트리 구조)"

    ```mermaid
    graph TD
        A["PID 0 - 커널"] --> B["PID 1 - systemd<br/>(모든 프로세스의 조상)"]
        B --> C["PID 523 - sshd<br/>(SSH 데몬)"]
        B --> F["PID 27154 - java tomcat<br/>(톰캣 프로세스)"]
        B --> G["PID 892 - cron<br/>(스케줄러)"]
        B --> H["PID 1102 - mysqld<br/>(DB 프로세스)"]
        C --> D["PID 4821 - sshd: tideflo<br/>(내 SSH 세션)"]
        D --> E["PID 4822 - bash<br/>(내 쉘)"]
        E --> I["PID 5103 - ps -ef<br/>(내가 친 명령)"]
    ```

### 우리 톰캣의 PPID가 1인 이유

!!! tip "우리 톰캣의 PPID가 1인 이유"

    **원래 흐름:**
    `systemd(PID 1)` --> `startup.sh` --> `java (톰캣)`

    - startup.sh가 java를 실행하고 자기는 종료함
    - 부모(startup.sh)가 죽음
    - 고아가 된 java 프로세스를 PID 1(systemd)이 입양
    - PPID가 1이 됨

    **또는:**
    systemd가 직접 톰캣 서비스를 실행 (`systemctl start tomcat`) -> 처음부터 PPID가 1

    어느 쪽이든, 서버 프로세스의 PPID가 1인 건 정상이야.
    "이 프로세스는 시스템이 직접 관리하는 서비스 프로세스다"라는 뜻.

### 고아 프로세스와 좀비 프로세스

이건 알아두면 디버깅할 때 도움 돼.

| 종류 | 설명 | 위험도 |
|------|------|--------|
| **고아 프로세스** | 부모가 먼저 죽은 프로세스. PID 1이 입양. | 보통 무해. |
| **좀비 프로세스** | 자식이 죽었는데 부모가 종료 상태를 안 가져감(wait 안 함). 프로세스 테이블에 찌꺼기가 남음. | 많으면 PID 고갈 가능. |

!!! warning "좀비 프로세스 확인 방법"

    ```bash
    $ ps aux | grep Z
    ```

    STAT 컬럼에 "Z"가 보이면 좀비야.
    좀비가 몇 개면 괜찮아. 수백 개면? 문제야.
    좀비를 만드는 부모 프로세스를 찾아서 처리해야 해.

    좀비 자체는 메모리를 거의 안 먹어 (이미 죽었으니까).
    근데 PID 테이블의 슬롯을 차지하고 있어서 새 프로세스를 못 만들 수 있어.

## 5. `ps -ef` 출력 완전 해석 - "이 명령어 하나로 서버 상태 읽기"

### ps 명령어란?

**ps = Process Status.** 현재 실행 중인 프로세스 목록을 보여주는 명령어.

- `ps` = 내 프로세스만
- `-e` = 모든(Every) 프로세스
- `-f` = 풀(Full) 포맷 (상세 정보)

합치면: **"시스템에서 돌아가는 모든 프로세스를 상세하게 보여줘"**

서버에 SSH 접속해서 `ps -ef`를 치면 이런 식으로 나와:

```bash
$ ps -ef
UID        PID  PPID  C STIME TTY          TIME CMD
root         1     0  0 Mar09 ?        00:02:31 /sbin/init
root         2     0  0 Mar09 ?        00:00:01 [kthreadd]
root       523     1  0 Mar09 ?        00:00:12 /usr/sbin/sshd -D
tomcat   27154     1  0 Mar09 ?        00:45:54 /usr/lib/jvm/java-8-openjdk...
tideflo   4821   523  0 10:32 pts/0    00:00:00 -bash
tideflo   5103  4821  0 10:45 pts/0    00:00:00 ps -ef
```

### 각 컬럼 하나하나 뜯어보기

이거 대충 넘기면 안 돼. 서버에서 `ps -ef` 한 줄 보고 상황 파악하는 게 기본이야.

!!! note "ps -ef 컬럼 배치"

    ```
    UID    PID   PPID  C  STIME  TTY      TIME      CMD
    ------ ----- ----- -- ------ -------- --------- ------------------
    tomcat 27154 1     0  Mar09  ?        00:45:54  /usr/lib/jvm/...
    ```

### UID (User ID) - 누가 실행했냐

!!! note "UID = 이 프로세스를 실행한 사용자"

    | 값 | 의미 |
    |----|------|
    | root | 시스템 관리자. 모든 권한. |
    | tomcat | 톰캣 전용 사용자. |
    | tideflo | 너 (SSH 접속한 계정). |
    | mysql | DB 전용 사용자. |

    **왜 중요해?** 보안. 톰캣을 root로 실행하면 해킹당했을 때 시스템 전체가 털려. 그래서 전용 사용자(tomcat)를 만들어서 최소 권한으로 실행. 우리 서버에서 톰캣의 UID가 "tomcat"인 건 보안 설정이 된 거야.

    만약 UID가 "root"로 돼 있으면? "이거 root로 톰캣 돌리고 있는데 괜찮아?"라고 물어봐야 해. 1024 이하 포트(80, 443)를 쓰려면 root가 필요할 수 있지만, 보안상 권한 분리가 원칙이야.

---

### PID (Process ID) - 프로세스 고유 번호

!!! note "PID = 프로세스의 주민등록번호"

    우리 톰캣: **PID 27154**

    이 숫자로 할 수 있는 것:

    - `kill 27154` -- 톰캣 종료 (SIGTERM)
    - `kill -9 27154` -- 톰캣 강제 종료 (SIGKILL)
    - `cat /proc/27154/status` -- 이 프로세스 상세 정보 조회
    - `top -p 27154` -- 이 프로세스만 모니터링
    - `jstat -gc 27154` -- 이 JVM의 GC 상태 확인

    PID는 서버 관리의 기본이야. "톰캣 PID가 뭐야?"라는 질문에 바로 답할 수 있어야 해. PID는 프로세스가 시작될 때 OS가 부여하고, 프로세스가 죽으면 회수돼. 재시작하면 새 PID를 받아.

---

### PPID (Parent Process ID) - 부모가 누구냐

!!! note "PPID = 이 프로세스를 만든 부모 프로세스의 PID"

    우리 톰캣: **PPID 1**

    PPID = 1 -> systemd(init)가 부모. 시스템 서비스로 등록되어 실행 중이거나, 원래 부모가 죽어서 systemd에게 입양됨. 서버 프로세스에서 흔한 패턴. 정상이야.

    **비교:**

    | 프로세스 | PPID | 부모 |
    |----------|------|------|
    | 톰캣 | 1 | 시스템 서비스 |
    | 내 bash | 523 | sshd가 부모 |
    | ps -ef | 4821 | bash가 부모 |

    PPID를 추적하면 "이 프로세스가 어떻게 시작됐는지" 알 수 있어.

---

### C (CPU Utilization) - CPU 사용률

!!! note "C = CPU 사용률 (정수 %)"

    우리 톰캣: **C = 0**

    0이라고 CPU를 안 쓴다는 뜻이 아니야! ps -ef의 C 값은 "최근 스케줄링 기간 동안의 CPU 사용률"이야. 스냅샷이라서 정확하지 않아. 정확한 CPU 사용률을 보려면 `top` 명령어를 써야 해.

    근데 여기서 C가 99면? "야 이 프로세스 CPU 다 먹고 있어" -- 무한루프 돌거나 무거운 연산 중이거나. 확인해봐야 해.

---

### STIME (Start Time) - 언제 시작했냐

!!! note "STIME = 프로세스가 시작된 시간"

    우리 톰캣: **STIME = Mar09**

    | 표시 방식 | 의미 |
    |-----------|------|
    | "10:32" | 오늘 시작 (시:분) |
    | "Mar09" | 올해 시작 (월일) |
    | "2025" | 작년 이전 (연도) |

    Mar09 = 3월 9일에 시작됐다는 뜻. 오늘이 3월 11일이면? 이틀 전에 톰캣을 (재)시작한 거야.

    **왜 이게 중요해?** "톰캣 언제 재시작했어?" 물어보면 STIME 보면 돼. 갑자기 STIME이 오늘이면? "아 오늘 재시작됐구나." 의도하지 않은 재시작이면? OOM Killer에 죽었을 가능성. 서버 장애 원인 추적할 때 핵심 단서야.

---

### TTY (Terminal Type) - 어떤 터미널에서 실행됐냐

!!! note "TTY = 프로세스가 연결된 터미널"

    우리 톰캣: **TTY = ?**

    | 값 | 의미 |
    |----|------|
    | ? | 터미널 없음 (데몬 프로세스) |
    | pts/0 | 가상 터미널 0번 (SSH 접속 세션) |
    | pts/1 | 가상 터미널 1번 (다른 SSH 세션) |
    | tty1 | 물리 콘솔 (서버에 직접 모니터 연결) |

    **TTY가 ?인 이유:** 톰캣은 "데몬 프로세스"야. 뒤에서 조용히 돌아가는 놈. 누가 SSH 터미널에서 직접 실행한 게 아니라, 시스템 서비스로 백그라운드에서 돌고 있어. 그래서 연결된 터미널이 없어 -> ? 표시.

    데몬(Daemon)은 이 챕터 뒷부분(8절)에서 자세히 다룬다.

---

### TIME (Cumulative CPU Time) - CPU를 총 얼마나 썼냐

!!! note "TIME = 이 프로세스가 시작부터 지금까지 사용한 CPU 시간 합계"

    우리 톰캣: **TIME = 00:45:54**

    이건 "벽시계 시간(wall clock time)"이 아니야. 프로세스가 실제로 CPU를 점유해서 일한 시간만 합산한 거야.

    **해석:** Mar09에 시작 -> 지금까지 약 이틀. 이틀 = 48시간 = 2,880분 동안 돌았는데 CPU를 실제로 쓴 시간은 45분 54초.

    **이게 정상이야?** 정상. 서버 프로세스는 대부분의 시간을 "대기 상태"로 보내. 요청이 들어올 때만 CPU를 쓰고, 나머지는 잠자고 있어. 이틀 중 45분만 CPU를 썼다 = CPU를 아껴 쓰고 있다.

    이 값이 급격히 올라가면? CPU를 많이 쓰고 있다는 뜻. 부하 확인 필요. 무한루프, 대량 배치 처리, GC 폭주 등을 의심.

---

### CMD (Command) - 뭘 실행한 거냐

!!! note "CMD = 프로세스를 실행한 전체 명령어"

    이게 제일 긴 필드야. 여기에 모든 정보가 담겨 있어. 다음 절에서 우리 톰캣의 CMD를 완전히 뜯어본다.

    CMD에서 읽을 수 있는 것:

    - 어떤 프로그램인지 (java? python? nginx?)
    - JVM 옵션이 뭔지 (-Xms, -Xmx 등)
    - 설정 파일 경로가 어딘지
    - 어떤 인자(arguments)로 실행됐는지

    서버 관리자에게 CMD는 "이 프로세스의 신분증"이야.

### 정리: ps -ef 컬럼 한눈에

| 컬럼 | 뜻 | 왜 봐야 해? |
|------|-----|-------------|
| **UID** | 실행한 사용자 | 보안. root인지 전용 사용자인지 확인 |
| **PID** | 프로세스 고유 번호 | kill, jstat 등 명령어에 필요 |
| **PPID** | 부모 프로세스 번호 | 어떻게 시작됐는지 추적 |
| **C** | CPU 사용률 (%) | CPU 과다 사용 감지 (대략적) |
| **STIME** | 시작 시간 | 재시작 여부 확인, 장애 추적 |
| **TTY** | 연결된 터미널 | 데몬인지 대화형인지 구분 |
| **TIME** | 누적 CPU 시간 | CPU 사용량 추이 파악 |
| **CMD** | 실행 명령어 전체 | 프로세스 정체 파악, 옵션 확인 |

### CMD 부분 완전 분해

이제 긴 CMD를 쪼개서 하나하나 볼 거야.

#### 1) Java 실행 경로

```
/usr/lib/jvm/java-8-openjdk-amd64/jre/bin/java
```

!!! tip "이 경로에서 읽을 수 있는 정보"

    | 경로 부분 | 의미 |
    |-----------|------|
    | `/usr/lib/jvm/` | JVM이 설치된 기본 경로 |
    | `java-8` | Java 8 사용 중 |
    | `openjdk` | Oracle JDK가 아니라 OpenJDK (오픈소스 버전) |
    | `amd64` | 64비트 아키텍처 |
    | `/jre/bin/java` | JRE(Java Runtime Environment)의 java 실행파일 |

    즉: 이 서버는 **"64비트 OpenJDK 8"**로 톰캣을 돌리고 있다.

#### 2) 로깅 설정

```
-Djava.util.logging.config.file=/usr/local/tomcat/conf/logging.properties
-Djava.util.logging.manager=org.apache.juli.ClassLoaderLogManager
```

!!! note "로깅 설정 해석"

    `-D`는 "시스템 프로퍼티 설정"이야. Java에서 설정값을 넘기는 방법.

    - **logging.config.file** = 로깅 설정 파일 경로 -> `/usr/local/tomcat/conf/logging.properties` -> 톰캣의 기본 로깅 설정 파일
    - **logging.manager** = 로그 매니저 클래스 -> JULI (Java Util Logging Implementation) -> 톰캣 전용 로깅 시스템. 웹앱별로 로깅을 분리할 수 있게 해줌.

    실무에서: 로그가 어디에 남는지 알아야 장애 추적을 해. 이 설정이 로그 경로를 결정하는 거야.

#### 3) TLS 보안 설정

```
-Djdk.tls.ephemeralDHKeySize=2048
```

!!! note "TLS 보안 설정"

    TLS(HTTPS) 통신 시 사용하는 임시 DH 키의 크기를 2048비트로 설정. 기본값(1024비트)은 보안에 취약해서 2048 이상으로 올린 거야. HTTPS를 쓰는 서비스라면 거의 필수 설정.

#### 4) Java Agent (모니터링)

```
-javaagent:/home/jspd/jspd/lib/jspd.agent.jar
```

!!! note "Java Agent (모니터링) 해석"

    `-javaagent` = JVM에 에이전트를 심는 옵션

    - `jspd.agent.jar` -> JSPD(성능 모니터링 도구)의 에이전트
    - JVM 위에서 돌아가면서 성능 데이터를 수집하는 프로그램
    - 응답 시간, 메서드 호출 수, SQL 실행 시간 등을 모니터링
    - APM(Application Performance Management) 도구의 일종

    에이전트는 JVM이 시작할 때 같이 로드돼. 톰캣이 돌아가는 동안 백그라운드에서 계속 데이터를 수집해.

#### 5) 메모리 설정 (핵심 중의 핵심)

```
-Xms8192m -Xmx12288m -XX:MaxPermSize=4096m
```

!!! danger "JVM 메모리 설정 (제일 중요)"

    **`-Xms8192m`** -- Xms = 최소(initial) 힙 메모리. 8192m = 8192 MB = 8 GB. JVM이 시작할 때 바로 8 GB를 확보한다는 뜻.

    **`-Xmx12288m`** -- Xmx = 최대(maximum) 힙 메모리. 12288m = 12288 MB = 12 GB. JVM이 사용할 수 있는 힙 메모리의 상한선. 12 GB를 넘으면? OutOfMemoryError 발생.

    **`-XX:MaxPermSize=4096m`** -- MaxPermSize = PermGen(Permanent Generation) 최대 크기. 4096m = 4 GB. 클래스 메타데이터, 문자열 상수 풀 등을 저장하는 영역. (참고: Java 8부터 PermGen은 Metaspace로 대체됨. 이 옵션은 사실 Java 8에서는 무시돼. 레거시 설정이야.)

    | 항목 | 값 |
    |------|-----|
    | 서버 RAM | 15 GB |
    | JVM 힙 초기 (Xms) | 8 GB |
    | JVM 힙 최대 (Xmx) | 12 GB |
    | 나머지 | 3 GB (OS + 기타 프로세스 몫) |

    이 비율이 적절한지? -> 05~06장에서 깊이 다룬다.

#### 6) OOM 대응 설정

```
-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=/home/heapdump
```

!!! danger "힙 덤프 설정 (장애 대응의 생명줄)"

    **`+HeapDumpOnOutOfMemoryError`** -- OOM(OutOfMemoryError) 발생 시 힙 메모리 스냅샷을 떠라. `+` 기호 = 이 기능을 켜라 (`-` 기호 = 끄기)

    **`HeapDumpPath=/home/heapdump`** -- 힙 덤프 파일을 이 경로에 저장해라. OOM 터지면 `java_pid27154.hprof` 같은 파일이 생겨.

    **왜 중요해?**

    - OOM이 터지면 톰캣이 죽어. 서비스 장애.
    - 원인을 찾으려면 "그 순간 메모리에 뭐가 있었는지" 알아야 해.
    - 힙 덤프가 그 증거야. 이게 없으면 원인 추적 불가.
    - MAT(Memory Analyzer Tool)로 분석해.

    이 설정이 없는 서버는? OOM 터지면 증거가 없어. "왜 죽었지?" 미궁. **반드시 설정해야 하는 옵션이야.**

```
-Dorg.apache.catalina.security.SecurityListener.UMASK=0027

-Dcatalina.base=/usr/local/tomcat

```

!!! note "톰캣 관련 설정들"

    | 설정 | 설명 |
    |------|------|
    | `protocol.handler.pkgs` | 톰캣 내부 리소스 핸들러 패키지 지정 |
    | `SecurityListener.UMASK=0027` | 톰캣이 만드는 파일의 권한 설정. 0027 = 소유자 전체, 그룹 읽기+실행, 다른 사용자 접근 불가 |
    | `-classpath` | JVM이 클래스를 찾을 경로. bootstrap.jar(톰캣 부트스트랩), tomcat-juli.jar(톰캣 로깅 라이브러리) |
    | `catalina.base=/usr/local/tomcat` | 톰캣 설치 경로. base와 home이 같으면 단일 인스턴스. 다르면 여러 인스턴스를 하나의 톰캣 설치로 돌릴 수 있어 |
    | `catalina.home=/usr/local/tomcat` | 톰캣 홈 경로 |
    | `java.io.tmpdir=/usr/local/tomcat/temp` | 임시 파일 저장 경로. 파일 업로드 중간 저장 등에 사용 |

#### 8) 메인 클래스와 실행 인자

```
org.apache.catalina.startup.Bootstrap start
```

!!! note "메인 클래스 + 인자"

    **`org.apache.catalina.startup.Bootstrap`** -- 톰캣의 시작점. 이 클래스의 main() 메서드가 실행됨. 이놈이 톰캣 전체를 부팅하는 거야.

    **`start`** -- "시작해라"라는 인자. Bootstrap은 start, stop, configtest 등의 인자를 받아. start = 톰캣 서버 시작.

### 전체 요약: 이 한 줄에서 읽은 모든 것

!!! abstract "ps -ef | grep java 한 줄에서 읽은 정보 요약"

    | # | 항목 | 읽은 정보 |
    |---|------|-----------|
    | 1 | 누가 실행? | tomcat 사용자 (root 아님, 보안 OK) |
    | 2 | PID는? | 27154 |
    | 3 | 부모 프로세스? | systemd (PID 1, 시스템 서비스) |
    | 4 | 언제 시작? | 3월 9일 (이틀째 운영 중) |
    | 5 | 데몬인가? | 맞음 (TTY = ?) |
    | 6 | CPU 사용량? | 이틀간 45분 54초 (안정적) |
    | 7 | Java 버전? | OpenJDK 8, 64비트 |
    | 8 | 힙 메모리? | 초기 8GB, 최대 12GB |
    | 9 | 서버 RAM? | 15GB (12GB를 JVM에 할당 가능) |
    | 10 | OOM 대응? | 힙 덤프 설정 있음 (/home/heapdump) |
    | 11 | 모니터링? | JSPD 에이전트 설치됨 |
    | 12 | 톰캣 경로? | /usr/local/tomcat |

    이 한 줄만 읽을 줄 알아도 서버 상태 절반은 파악한 거야.

## 7. 톰캣 = 서블릿 컨테이너 = WAS - "톰캣이 뭘 하는 놈인지 정확히 알아야 해"

### 비유로 문 열기

식당에서 손님이 "비빔밥 주세요"라고 하면, 웨이터가 주방에 전달하고, 요리사가 만들잖아. 근데 웨이터만으로는 안 돼. 주방 시설이 있어야 해 -- 가스레인지, 냉장고, 식기 세척기. 이런 주방 시설을 관리하는 시스템이 **컨테이너**야.

비유 끝. 본질 가자.

### 톰캣의 정확한 정체

**톰캣(Apache Tomcat) = 서블릿 컨테이너 + 간단한 웹 서버**

| 용어 | 정확한 의미 |
|------|-------------|
| **서블릿(Servlet)** | Java로 만든 서버 프로그램. HTTP 요청을 받아서 처리하는 Java 클래스. |
| **서블릿 컨테이너** | 서블릿의 생명주기를 관리하는 환경. 서블릿 생성, 호출, 소멸을 담당. |
| **WAS** | Web Application Server. 웹 애플리케이션을 실행하는 서버. 톰캣이 여기 해당. |
| **웹 서버** | 정적 파일(HTML, CSS, JS, 이미지)을 제공. Nginx, Apache httpd 등. |

!!! example "웹 요청 처리 흐름 (우리 LMS 기준)"

    ```mermaid
    flowchart TD
        A["학생 브라우저"] -->|"HTTP 요청: GET /lms/course/list"| B["톰캣 (WAS01)"]
        B --> C["1. HTTP 요청 수신 (Coyote 커넥터)"]
        C --> D["2. 스레드 풀에서 스레드 하나 할당"]
        D --> E["3. 서블릿 컨테이너(Catalina)가<br/>적절한 서블릿/스프링 컨트롤러로 라우팅"]
        E --> F["4. 스프링 컨트롤러 → Service → DAO → DB 조회"]
        F --> G["5. JSP 렌더링 → HTML 생성"]
        G --> H["6. HTTP 응답 전송"]
        H --> I["학생 브라우저에 강의 목록 표시"]
    ```

### 톰캣 = java 프로세스

톰캣은 Java로 만들어졌어. 그래서 톰캣이 실행되면 `java` 프로세스가 생기는 거야.

`ps -ef | grep java`로 검색하면 톰캣 프로세스가 잡히는 이유가 이거야. 톰캣은 java 프로그램이니까.

```
톰캣 실행 = java 명령어 실행
         = java 프로세스 1개 생성
         = 이 프로세스 안에서 Spring이 돌아감
         = 이 프로세스가 메모리를 사용함
         = 이 메모리 사용량이 모니터링 대상
```

### 톰캣이 메모리를 쓰는 이유

이 시리즈가 "서버 메모리 완전정복"이니까, 톰캣이 왜 메모리를 먹는지 이해해야 해.

!!! warning "톰캣이 메모리를 사용하는 주요 요소"

    | # | 요소 | 설명 | 메모리 |
    |---|------|------|--------|
    | 1 | **스레드 풀** | 동시 요청 처리를 위해 스레드를 미리 만들어 놓음. 각 스레드마다 스택 메모리 사용 | 200개 x 1MB = 200MB |
    | 2 | **세션** | 로그인한 사용자마다 세션 객체 유지. 사용자 정보, 권한 정보 등 저장 | 사용자 500명 x 세션 크기 |
    | 3 | **웹 애플리케이션 (WAR)** | 스프링 빈, 서블릿, 필터 등 모든 Java 객체. 클래스 로딩 -> 메타스페이스 사용 | - |
    | 4 | **DB 커넥션 풀** | DB 연결을 미리 만들어 놓고 재사용. 각 커넥션이 메모리를 차지 | - |
    | 5 | **요청 처리 중 객체들** | VO 객체, List, Map, DB 결과셋. 이것들이 JVM 힙에 쌓임 | - |

    이 모든 게 "JVM 힙 메모리"에서 돌아가. 그래서 `-Xmx12288m` (12GB)을 설정한 거야.

## 8. 데몬 프로세스 - "TTY가 ? 인 이유"

### 데몬(Daemon)이 뭐야?

**데몬 = 백그라운드에서 계속 돌아가는 프로세스.** 사용자와 직접 상호작용하지 않고, 요청이 올 때까지 대기하다가 처리하는 놈이야.

이름의 유래가 재밌어. 그리스 신화의 "다이몬(daemon)" -- 보이지 않는 곳에서 일하는 존재. 딱 맞는 이름이야. 눈에 안 보이는 곳에서 묵묵히 일하니까.

!!! note "데몬 프로세스의 특징"

    1. **터미널(TTY)이 없다** -- ps -ef에서 TTY가 "?". 키보드 입력 안 받고, 화면 출력 안 함. 로그 파일에 기록할 뿐.
    2. **백그라운드에서 실행** -- 누가 SSH 접속해서 직접 실행하는 게 아님. 시스템 부팅 시 자동으로 시작되거나 `systemctl start tomcat` 같은 명령으로 시작.
    3. **부모가 PID 1 (init/systemd)** -- PPID가 1인 경우가 많음. 시스템이 직접 관리하는 프로세스.
    4. **계속 실행 중** -- 한 번 시작하면 종료 명령 전까지 계속 돌아감. 서버는 24/7 돌아야 하니까 당연히 데몬이어야 해.

### 데몬 vs 대화형 프로세스

!!! note "데몬 vs 대화형 프로세스 비교"

    | 구분 | 대화형 (Interactive) | 데몬 (Daemon) |
    |------|---------------------|---------------|
    | TTY | pts/0 | ? |
    | 실행 주체 | 사용자가 직접 실행 | 시스템이 자동 실행 |
    | 터미널 닫으면 | 같이 죽음 | 터미널과 무관하게 계속 실행 |
    | 예시 | ps -ef, vi, top | tomcat, sshd, mysql |

    너가 SSH로 접속해서 치는 명령어 = 대화형 / 톰캣, DB 서버, SSH 서버 = 데몬

    ps -ef 출력에서:
    ```
    tideflo  5103  4821  0 10:45 pts/0  00:00:00 ps -ef   <- 대화형
    tomcat  27154     1  0 Mar09 ?      00:45:54 java...  <- 데몬
    ```

### 리눅스에서 흔한 데몬들

| 데몬 | 이름 규칙 | 하는 일 |
|------|-----------|---------|
| **sshd** | SSH Daemon | SSH 원격 접속 처리 |
| **httpd** | HTTP Daemon | 웹 서버 (Apache) |
| **mysqld** | MySQL Daemon | DB 서버 |
| **crond** | Cron Daemon | 스케줄링 (정해진 시간에 작업 실행) |
| **syslogd** | System Log Daemon | 시스템 로그 관리 |

이름 끝에 **d**가 붙는 게 관례야. daemon의 d. 톰캣은 좀 다르게 java 프로세스로 보이지만, 본질은 데몬이야.

## 9. `ps aux` vs `ps -ef` - "뭐가 다른 거야?"

둘 다 프로세스 목록을 보여주는 건 같아. 근데 옵션 스타일이 달라.

### 옵션 스타일의 차이

| 구분 | ps -ef | ps aux |
|------|--------|--------|
| **옵션 스타일** | System V (유닉스 전통) | BSD (버클리 유닉스) |
| **하이픈(-)** | 있음 (ps **-e** **-f**) | 없음 (ps **a** **u** **x**) |
| **-e / a,x** | 모든 프로세스 | a(터미널 있는) + x(터미널 없는) = 전부 |
| **-f / u** | full format | user-oriented format |

### 출력 차이

```bash
# ps -ef 출력
UID        PID  PPID  C STIME TTY      TIME CMD
tomcat   27154     1  0 Mar09 ?    00:45:54 java ...

# ps aux 출력
USER       PID %CPU %MEM    VSZ   RSS TTY  STAT START   TIME COMMAND
tomcat   27154  0.0 58.3 17283456 8923456 ?  Sl  Mar09 45:54 java ...
```

### ps aux에만 있는 핵심 컬럼들

!!! tip "ps aux에만 있는 중요한 컬럼들"

    | 컬럼 | 설명 |
    |------|------|
    | **%CPU** | CPU 사용률 (ps -ef의 C보다 정확) |
    | **%MEM** | 물리 메모리 사용률 (이게 중요!) |
    | **VSZ** | Virtual Memory Size (가상 메모리 크기, KB) |
    | **RSS** | Resident Set Size (실제 물리 메모리 사용량, KB) |
    | **STAT** | 프로세스 상태: S=sleeping, R=running, Z=zombie, l=멀티스레드, s=세션 리더 |

    메모리 모니터링 관점에서: ps -ef는 메모리 정보가 없어. ps aux는 %MEM, VSZ, RSS를 보여줘. **메모리 확인하려면 ps aux가 더 유용해.**

    위 예시에서: %MEM = 58.3% (물리 메모리의 58.3%를 이 프로세스가 사용 중), RSS = 8923456 KB = 약 8.5 GB (실제 물리 메모리 사용량), VSZ = 17283456 KB = 약 16.5 GB (가상 메모리, 예약 포함)

    VSZ > RSS인 이유? -> 04장에서 상세히 다룬다 (committed vs resident)

### 그래서 뭘 쓰라고?

| 목적 | 추천 명령어 | 이유 |
|------|-------------|------|
| 프로세스 정체 파악 | `ps -ef \| grep java` | CMD가 잘 안 잘리고, PPID 보임 |
| 메모리/CPU 확인 | `ps aux \| grep java` | %MEM, RSS, VSZ 보임 |
| 종합 모니터링 | `top -p <PID>` | 실시간으로 변화 추적 가능 |

실무에서는 둘 다 써. 프로세스 정보 파악할 때는 보통 `ps -ef`를 먼저 치고, 메모리/CPU 확인할 때 `ps aux`를 쳐.

### grep 보충 설명

`ps -ef` 하면 프로세스가 수십~수백 개 쭉 나와. 그 중에서 java만 찾고 싶으면? `grep`으로 필터링해.

!!! note "grep = '이 문자열이 포함된 줄만 골라서 보여줘'"

    ```bash
    $ ps -ef | grep java
      ------   |  ----------
      전체목록  |  "java" 포함 줄만 필터
               |
               +-- 파이프(|): 왼쪽 결과를 오른쪽에 넘겨줘
    ```

    **자주 쓰는 패턴:**

    ```bash
    ps -ef | grep java                 # java 프로세스 찾기
    ps -ef | grep java | grep -v grep  # grep 자기 자신 제외
    ps -ef | grep catalina             # 톰캣만 확실하게 찾기
    ```

    `grep -v grep`을 왜 붙이냐: "grep java" 명령어 자체에 "java"라는 글자가 있으니까 grep 프로세스 자신도 결과에 나와. `-v`는 "이걸 제외해"라는 뜻.

## 10. 주의사항 / 함정

### 함정 1: "서버 = 하드웨어"라는 오해

!!! danger "함정 1"

    **X 흔한 오해:** "서버는 특별한 장비야"

    **O 진실:** 서버의 본질은 "소프트웨어(프로그램)"야. 네 노트북에서도 톰캣 돌리면 서버야. 다만 실서비스에서는 안정성, 성능 때문에 전용 장비를 쓰는 거야.

    "서버 = 서빙하는 프로그램 + 그걸 돌리는 컴퓨터" -- 이 두 가지를 문맥에 따라 구분해서 이해해야 해.

### 함정 2: "ps -ef의 TIME이 실행 시간"이라는 오해

!!! danger "함정 2"

    **X 흔한 오해:** "TIME이 00:45:54니까 톰캣이 45분 전에 시작된 거야"

    **O 진실:** TIME은 "벽시계 시간"이 아니라 "CPU 사용 시간 합계"야. 톰캣이 Mar09에 시작돼서 이틀 돌았는데, 그 이틀 중 CPU를 실제로 쓴 시간이 45분 54초라는 뜻. 시작 시간은 STIME 컬럼을 봐야 해.

### 함정 3: "java 프로세스 = 무조건 톰캣"이라는 단순화

!!! danger "함정 3"

    **X 오해:** java 프로세스는 무조건 톰캣이다

    **O 진실:** java 프로세스는 톰캣 말고도 있을 수 있어 -- Jenkins (CI/CD 서버), Elasticsearch (검색 엔진), 기타 Java 애플리케이션.

    **구분 방법:** CMD 부분에서 catalina, bootstrap 같은 톰캣 키워드가 있는지 확인. `ps -ef | grep catalina` <- 톰캣만 확실하게 찾기.

### 함정 4: "PermGen 설정이 Java 8에서도 작동한다"는 오해

!!! danger "함정 4"

    **X 흔한 오해:** "MaxPermSize=4096m이니까 PermGen이 4GB야"

    **O 진실:** Java 8부터 PermGen은 제거됐어. Metaspace로 대체됨. MaxPermSize 옵션은 Java 8에서 무시돼. JVM이 시작할 때 경고 메시지를 내보내:

    ```
    Java HotSpot(TM) 64-Bit Server VM warning:
    ignoring option MaxPermSize=4096m; support was removed in 8.0
    ```

    우리 서버에 이 옵션이 남아있는 건 레거시 설정이야. Java 7 이하에서 쓰던 걸 안 지운 것뿐. 동작에 문제는 없지만, 정리하면 더 깔끔해.

### 함정 5: "kill -9로 톰캣 종료하면 된다"는 안일함

!!! danger "함정 5"

    **X 안일한 태도:** "톰캣 안 죽네? kill -9 27154 하면 되지"

    **O 올바른 접근:** kill -9는 프로세스에게 정리할 시간을 안 줘. 진행 중인 DB 트랜잭션이 끊길 수 있고, 파일 쓰기 도중 깨질 수 있고, 세션 데이터 유실 가능.

    **올바른 순서:**

    1. `shutdown.sh` 실행 (톰캣 정상 종료)
    2. 안 죽으면 `kill 27154` (SIGTERM, 정상 종료 요청)
    3. 그래도 안 죽으면 `kill -9 27154` (최후의 수단)

| **데몬** | 백그라운드에서 계속 도는 프로세스. TTY가 ?. 서버는 대부분 데몬. |
| **톰캣** | 서블릿 컨테이너(WAS). Java 웹 앱의 실행 환경. java 프로세스로 돌아감. |
| **CMD** | 실행 명령어 전체. JVM 옵션, 메모리 설정, 경로 정보 다 있음. |

### 이 챕터에서 반드시 기억할 것

!!! abstract "이 챕터에서 반드시 기억할 것"

    1. 서버 = "요청 받고 응답하는 프로그램 + 그 프로그램이 도는 장비"
    2. 프로세스는 "실행 중인 프로그램". 메모리에 올라가야 프로세스.
    3. ps -ef 한 줄에서 UID, PID, PPID, STIME, TTY, TIME, CMD 읽기.
    4. 우리 톰캣: PID 27154, 힙 8-12GB, OpenJDK 8, 데몬 프로세스.
    5. 데몬 = 백그라운드 프로세스. TTY가 ?. 서버는 데몬으로 돌아야 해.
    6. TIME은 CPU 누적 사용 시간이지 실행 시간이 아니야.
    7. ps aux는 메모리(%MEM, RSS, VSZ) 정보를 추가로 보여줘.

### 확인 문제 (5문제)

> 다음 문제를 풀어봐. 답 못 하면 위에서 다시 읽어.

**Q1.** "프로그램"과 "프로세스"의 차이를 메모리 관점에서 한 문장으로 설명해봐.

**Q2.** 우리 톰캣의 `ps -ef` 출력에서 PPID가 1인 이유를 설명해봐. PID 1은 뭐야?

**Q3.** `ps -ef`에서 TIME이 `00:45:54`이고 STIME이 `Mar09`야. 이게 "톰캣이 45분 전에 시작됐다"는 뜻이 맞아? 틀리다면 TIME은 정확히 뭘 의미해?

**Q4.** 아래 ps -ef 출력에서 읽을 수 있는 정보를 최대한 많이 말해봐:
```
tomcat  27154  1  0 Mar09 ?  00:45:54 /usr/lib/jvm/java-8-openjdk-amd64/jre/bin/java ... -Xms8192m -Xmx12288m ...
```

**Q5.** `ps -ef`와 `ps aux`의 차이는 뭐야? 메모리 사용량을 확인하려면 어떤 걸 써야 해?

??? success "정답 보기"
    **A1.** 프로그램은 디스크에 저장된 파일(실행 코드)이고, 프로세스는 그 프로그램이 메모리(RAM)에 올라가서 CPU가 실행하고 있는 상태야. 메모리에 안 올라간 프로그램은 프로세스가 아니야. 같은 프로그램으로 프로세스를 여러 개 만들 수도 있어 (각각 독립된 메모리 공간, 각각 다른 PID).

    **A2.** PID 1은 systemd(또는 init)로, 리눅스에서 가장 먼저 시작되는 프로세스이자 모든 프로세스의 조상이야. 톰캣의 PPID가 1인 이유는 두 가지 가능성이 있어: (1) systemd가 직접 톰캣 서비스를 실행했거나(systemctl start tomcat), (2) 톰캣을 실행한 부모 프로세스(startup.sh 등)가 종료되면서 고아가 된 톰캣을 systemd가 입양한 거야. 어느 쪽이든 시스템 서비스로 관리되는 정상적인 상태야.

    **A3.** 틀려. TIME은 "벽시계 시간(경과 시간)"이 아니라 "프로세스가 시작부터 지금까지 실제로 CPU를 점유해서 사용한 누적 시간"이야. STIME이 Mar09니까 이틀 전에 시작됐고, 이틀(약 48시간) 동안 CPU를 실제로 쓴 시간이 45분 54초라는 뜻이야. 대부분의 시간은 대기 상태(sleep)로 요청을 기다리고 있었던 거야.

    **A4.** 읽을 수 있는 정보: (1) UID=tomcat: 전용 사용자로 실행 중, root 아님, 보안 OK. (2) PID=27154: 프로세스 고유 번호. (3) PPID=1: systemd가 부모, 시스템 서비스. (4) C=0: 현재 CPU 거의 안 쓰는 중. (5) STIME=Mar09: 3월 9일에 시작됨. (6) TTY=?: 데몬 프로세스, 터미널 없이 백그라운드 실행. (7) TIME=00:45:54: CPU 누적 사용 45분 54초. (8) CMD에서 Java 8 OpenJDK 64비트 사용 중. (9) 힙 초기 8GB(Xms), 최대 12GB(Xmx). (10) 서버 RAM 15GB 중 최대 12GB를 JVM에 할당 가능. (11) OOM 시 힙 덤프를 /home/heapdump에 저장하도록 설정됨.

    **A5.** ps -ef는 System V 스타일로 UID, PID, PPID, C, STIME, TTY, TIME, CMD를 보여줘. PPID와 전체 CMD 확인에 좋아. ps aux는 BSD 스타일로 %CPU, %MEM, VSZ, RSS, STAT 등 추가 컬럼을 보여줘. 메모리 사용량을 확인하려면 `ps aux`를 써야 해. %MEM은 물리 메모리 사용률, RSS는 실제 물리 메모리 사용량(KB), VSZ는 가상 메모리 크기(KB)를 보여주거든. ps -ef에는 메모리 관련 컬럼이 없어.
