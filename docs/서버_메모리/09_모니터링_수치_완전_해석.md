# 09. 모니터링 수치 완전 해석 - Omega

---

## 1. "메모리 55%입니다" - "이게 뭐야?"

### 비유로 문 열기

너 건강검진 받아봤어?

의사가 "혈압이 130이에요"라고 하면, 바로 이해돼? 130이 수축기인지 이완기인지, 측정 직후인지 안정 시인지, 너 나이 기준으로 높은 건지 낮은 건지... 맥락 없이 숫자만 던지면 의미가 없어.

서버 모니터링도 **똑같아.**

!!! example "주니어 vs 시니어"

    **관제팀:** "WAS01 메모리 55%입니다"

    **주니어:** "55%면... 반 좀 넘었네. 괜찮은 거 아닌가요?"

    **빠싺 시니어:**
    "뭐 기준 55%? used/total이야? RSS/total이야? buff/cache는 포함이야 불포함이야? available은 얼마야? OU 추이는 어때? Swap은 쓰고 있어? 야간 패턴은 횡보야 우상향이야?"

    이 차이가 주니어와 시니어의 차이야. 숫자 하나 보고 "괜찮네~"는 찍기야. 근거 대고 "이러이러해서 정상입니다"가 판단이야.

비유는 여기까지. 진짜 이야기 하자.

### 본질

**모니터링 수치는 "어떤 기준으로 계산했느냐"에 따라 전혀 다른 의미가 된다.**

"55%"라는 숫자를 보고 정상/이상을 판단하려면, 그 55%가 어디서 온 숫자인지부터 알아야 해. 그리고 그 숫자 하나만으론 부족해. 다른 지표들을 같이 봐야 비로소 판단이 가능해.

이게 이 챕터의 전부야.

## 2. 모니터링에서 보여주는 "메모리 %" 의 정체

### 2-1. 모니터링 툴마다 계산 방식이 다르다

이게 가장 먼저 알아야 할 핵심이야. **같은 서버를 봐도 툴마다 %가 다를 수 있어.**

!!! note "같은 서버, 같은 시점, 다른 수치"

    | 도구 | 수치 |
    |------|------|
    | InterMax (APM) | 메모리 사용률 55% |
    | Grafana 대시보드 | 메모리 사용률 62% |
    | top 명령어 | %MEM 55.0 |
    | free -h 직접 계산 | used/total = 56% |

    뭐야? 다 다른데? 어떤 게 "진짜" 수치야?

    답: **전부 맞아.** 계산 기준이 다른 거야. "진짜" 수치는 없어. "어떤 기준의" 수치인지만 있어.

왜 이런 일이 생기냐? 각 도구가 "메모리 사용률"을 계산하는 공식이 다르기 때문이야.

### 2-2. 대표적인 계산 방식 3가지

!!! note "모니터링 % 계산 방식 비교"

    **[방식 A] used / total * 100**

    free -h 출력에서 Mem 행의 used / total.
    핵심 문제: used에 buff/cache가 포함되느냐 안 되느냐.

    - CentOS 6 이전: used에 buff/cache 포함 → 수치가 높게 나옴
    - CentOS 7 이후: used에서 buff/cache 제외 → 좀 더 정확
    - 모니터링 에이전트가 /proc/meminfo를 직접 파싱하면 해석 방식에 따라 결과가 또 다를 수 있음

    우리 서버 예시: used 8.4Gi / total 15Gi = **56%**

    ---

    **[방식 B] (total - available) / total * 100**

    "실제로 못 쓰는 비율" 개념. available을 기준으로 계산하므로 더 현실적. Grafana 같은 대시보드에서 자주 사용.

    우리 서버 예시: (15Gi - 6.9Gi) / 15Gi = **54%**

    ---

    **[방식 C] 프로세스 RSS / total * 100**

    특정 프로세스 하나가 물리 RAM을 얼마나 차지하는지. top의 %MEM 컬럼이 이 방식.

    우리 서버 예시: RSS 8.4GB / total 15.3GB = **55%** (톰캣 프로세스 하나만의 비율)

### 2-3. buff/cache 포함 여부가 만드는 차이

이게 혼란의 원흉이야. 가장 많이 오해하는 부분.

!!! warning "buff/cache 포함 여부가 만드는 차이"

    우리 서버 free -h 출력: total 15Gi, used 8.4Gi, free 1.0Gi, buff/cache 6.2Gi, available 6.9Gi

    buff/cache 6.2Gi가 "사용 중"에 포함되느냐 안 되느냐에 따라:

    | 기준 | 계산 | 결과 |
    |------|------|------|
    | buff/cache 포함 | (8.4 + 6.2) / 15 | **97%** "서버 터진다!" |
    | buff/cache 불포함 | 8.4 / 15 | **56%** "절반 좀 넘은 수준" |

    같은 서버인데 97%와 56%. 어느 모니터링 에이전트냐에 따라 이 차이가 남.

    **그러면 뭘 봐야 하냐?**
    → % 숫자에 집착하지 말고 **available 절대값**을 봐.
    → available = 6.9Gi → 이게 진짜 여유 메모리야.

### 2-4. /proc/meminfo - 모든 수치의 원본

모든 모니터링 도구는 결국 `/proc/meminfo`에서 데이터를 읽어. 이걸 어떻게 조합하느냐가 차이를 만드는 거야.

```bash
$ cat /proc/meminfo | head -10
MemTotal:       16178032 kB      ← 물리 RAM 총량 (약 15.4Gi)
MemFree:         1048576 kB      ← 완전히 비어있는 RAM
MemAvailable:    7233536 kB      ← 실제 사용 가능 (free + 회수 가능 캐시)
Buffers:          204800 kB      ← 블록 디바이스 버퍼
Cached:          6348800 kB      ← 페이지 캐시
SwapCached:        20480 kB      ← Swap에서 다시 올라온 캐시
SwapTotal:       4194304 kB      ← Swap 총 크기
SwapFree:        4046848 kB      ← Swap 남은 양
...
```

!!! danger "같은 /proc/meminfo, 다른 결과"

    | 도구 | 계산 공식 | 결과 |
    |------|-----------|------|
    | 도구 A | MemTotal - MemFree | **93%** (무섭게 높아 보여!) |
    | 도구 B | MemTotal - MemFree - Buffers - Cached | **53%** (훨씬 낮아 보여) |
    | 도구 C | (MemTotal - MemAvailable) / MemTotal | **55%** |

    같은 /proc/meminfo인데 93%, 53%, 55%.
    공식이 다르면 결과가 다르다.
    **이래서 "55%입니다"에 "뭐 기준?"이라고 물어야 하는 거야.**

### 2-5. 핵심: "55%입니다"에 대한 올바른 반응

!!! tip "55%라는 숫자를 들으면 이렇게 해"

    1. **"뭐 기준 55%?"** → used/total? (total-available)/total? RSS/total? → buff/cache 포함이야 불포함이야?
    2. **"OS 전체 기준이야 JVM 기준이야?"** → 서버 전체 메모리 대비인지, JVM 힙 사용률인지 → 완전히 다른 의미야
    3. **"지금 이 순간만의 숫자야? 추이는?"** → 어제도 55%였어? 아니면 30%에서 올라온 거야? → 한 시점 스냅샷은 절반만 보여줘

    이 세 가지를 안 물어보고 "55%면 괜찮겠네~"는 **판단이 아니라 찍기야.**

## 3. 정상 vs 이상 판단하는 핵심 기준 3가지

모니터링 %가 뭐든 간에, **진짜 정상/이상을 판단하는 기준은 딱 3가지**야.

### 기준 1: available (free -h) - OS 레벨 여유

!!! note "기준 1: available"

    available = 새 프로세스가 Swap 없이 사용할 수 있는 예상 메모리 = free + 회수 가능한 buff/cache

    03장에서 배운 거야. available이 진짜 "OS 레벨의 여유 메모리"야. used도 아니고 free도 아니고 available.

    | 상태 | 기준 (우리 서버 15GB RAM 기준) |
    |------|------|
    | 안전 | available >= 전체의 20% (3GB 이상) |
    | 주의 | available 10%~20% (1.5GB ~ 3GB) |
    | 경고 | available < 10% (1.5GB 미만) |
    | 위험 | available < 5% (750MB 미만) |
    | 심각 | available < 500MB |

    **우리 서버 (누수 해결 후):** available = 6.9Gi / total = 15Gi = 46% → 안전. 여유 충분.

    왜 available이 첫 번째 기준이냐?
    → used가 높아도 available이 넉넉하면 OK. 캐시 때문일 수 있으니까.
    → used만 보고 공포에 빠지면 안 돼. **available이 진짜야.**

### 기준 2: Swap 사용량 - RAM 부족 징후

!!! note "기준 2: Swap 사용량"

    Swap = RAM이 부족할 때 디스크를 RAM처럼 쓰는 영역. 01장에서 배운 거야. 디스크는 RAM보다 1,000~100,000배 느려. Swap 쓴다 = 느린 디스크에서 데이터 읽고 쓴다 = 성능 폭락.

    | 상태 | 기준 |
    |------|------|
    | 최상 | Swap used = 0 |
    | 안전 | Swap 있지만 일정하게 유지 (증가 없음) |
    | 주의 | Swap 있고 완만하게 증가 중 |
    | 경고 | Swap 지속적으로 증가 (매일 늘어남) |
    | 위험 | Swap 빠르게 증가 or 전체의 50% 이상 |

    **핵심:** Swap "있다" 자체보다 **"증가 추세"**가 훨씬 중요해.

    - Swap이 200MB인데 한 달째 200MB? → 과거에 한번 밀렸던 흔적. 지금은 괜찮아.
    - Swap이 매일 100MB씩 늘어나? → 위험. RAM이 계속 부족해지고 있다는 뜻.

    **우리 서버:** Swap used = 145Mi / Swap total = 4.0Gi = 3.6% → 안전.

    vmstat으로 확인: `vmstat 1 5` → si/so가 0이면 현재 Swap 교환 없음

### 기준 3: OU (jstat Old Used) - JVM 누수 여부

!!! note "기준 3: OU (Old Used)"

    OU = JVM Old Generation에 있는 객체의 크기. 07장에서 배운 jstat -gc의 핵심 컬럼이야.

    왜 OU가 기준이냐? OS 레벨 메모리가 아무리 여유 있어도 JVM 힙 안에서 누수가 나면 결국 OOM 터져. OOM은 OS 메모리 부족이 아니라 JVM 힙 부족에서 터지는 거야.

    | 상태 | 기준 |
    |------|------|
    | 정상 | GC 후 OU 기준선이 일정 (톱니 패턴) |
    | 주의 | OU/OC > 50% 이상 지속 |
    | 경고 | GC 후 OU 기준선이 점점 상승 |
    | 위험 | OU/OC > 80% 지속 + Full GC 빈도 증가 |
    | 심각 | Full GC 후에도 OU 안 줄어듦 |

    **우리 서버 (누수 해결 후):** OU = 440MB / OC = 8GB = 5.4%, 기준선 430~450MB에서 일정 → 건강

    **우리 서버 (누수 있었을 때):** OU가 매일 수백MB씩 상승. RefreshableSqlSessionFactoryBean → Timer → GC Root → DelegatingClassLoader 5일 만에 4,574개 누적 → GC가 절대 회수 못 함 → 누수

### 3가지 기준 종합 표

!!! abstract "서버 건강 판단 3대 기준"

    | 기준 | 무엇을 보나 | 어디서 보나 | 왜 중요해 |
    |------|------------|------------|----------|
    | available | OS 여유 메모리 | free -h | OS 레벨 안전성 |
    | Swap 추세 | RAM 부족 징후 | free -h, vmstat | 성능 저하 예측 |
    | OU 추세 | JVM 누수 여부 | jstat -gc | OOM 예측 |

    - 이 세 개가 전부 안전하면 → **서버 건강**
    - 하나라도 경고면 → 추가 조사 필요
    - 둘 이상 위험이면 → 즉시 대응

    **우리 서버 현재:**
    available = 6.9Gi (46%) → 안전 / Swap = 145Mi (증가 없음) → 안전 / OU = 440MB/8GB (5.4%) → 안전
    → **3대 기준 전부 안전. "55%"라는 숫자에 속지 마.**

## 4. 정상 패턴 (톱니 패턴) - "이래야 정상이야"

### 4-1. 개념

정상적인 서버의 메모리 사용 패턴은 **톱니바퀴(Sawtooth) 모양**이야.

- 메모리 올라감 (요청 처리하면서 new로 객체 생성)
- GC 돌면서 내려감 (안 쓰는 객체 회수)
- 다시 올라감
- 다시 GC로 내려감

핵심은 **바닥(최저점, baseline)이 일정하게 유지**되는 거야.

### 4-2. 정상 패턴 ASCII 그래프

!!! tip "정상 패턴: 톱니 (Sawtooth Pattern)"

    ```
    OU
    (MB)

    700 |
        |         /\          /\          /\          /\
    600 |        /  \        /  \        /  \        /  \
        |       /    \      /    \      /    \      /    \
    500 |      /      \    /      \    /      \    /      \
        |     /        \  /        \  /        \  /        \
    400 |    /          \/          \/          \/          \
        |   / <-- 기준선(바닥) 약 400MB에서 일정!
    300 |  /
        | /
    200 |/
        +------+------+------+------+------+------+------+------→ 시간
               3h     6h     9h    12h    15h    18h    21h   24h
    ```

    **특징:**

    - 올라감 → GC → 내려감 → 올라감 → GC → 내려감 반복
    - 기준선(GC 후 최저점)이 ~400MB에서 일정하게 유지
    - 피크(최고점)도 ~650MB에서 비슷
    - 진폭(피크-바닥)이 일정
    - 하루 종일 이 패턴이면 → **완벽하게 정상**

### 4-3. 야간 횡보 패턴 (더 확실한 정상)

야간에 트래픽이 없으면 어떻게 될까? 새 객체가 안 만들어지니까 메모리가 올라가지도 내려가지도 않아. **수평선(횡보)**이 돼.

!!! tip "정상 패턴 변형: 주간 톱니 + 야간 횡보"

    ```
    OU
    (MB)

    700 |            /\     /\
        |           /  \   /  \
    600 |     /\   /    \ /    \
        |    /  \ /      V      \
    500 |   /    V                \
        |  /                      ─────────────────── ← 야간 횡보
    400 | /                       (트래픽 없음.       |
        |                          GC도 거의 안 돌아감)
    300 |                                             |
        +------+------+------+------+------+------+------+------→ 시간
         09:00 12:00 15:00 18:00 21:00 00:00 03:00 06:00 09:00
         ←──── 주간 (수강생 활동) ─────→←── 야간 (활동 없음) ──→
    ```

    야간에 수평선(횡보)이면:

    - 트래픽 없으니 새 객체 안 만들어짐
    - GC도 거의 안 돌아감
    - 메모리가 올라가지도 내려가지도 않음
    - **"트래픽이 없으면 메모리도 안정" = 아무 문제 없다는 확실한 증거**

### 4-4. 왜 톱니가 정상이냐

| 현상 | 원인 | 의미 |
|------|------|------|
| 올라감 | HTTP 요청 처리 → new로 VO, DTO, 임시 객체 생성 | JVM이 정상적으로 일하고 있음 |
| 내려감 (GC) | Eden이 차면 Minor GC → 죽은 객체 회수 | JVM이 정상적으로 청소하고 있음 |
| 바닥 일정 | GC 후 살아남은 장기 객체 크기가 일정 | 누수 없음. 정상적인 캐시/세션만 남아있음 |
| 피크 일정 | 트래픽 부하가 비슷 | 서비스 부하가 안정적 |
| 야간 횡보 | 트래픽 없음 → 객체 생성 없음 → GC 불필요 | 메모리 소비가 트래픽에 비례 = 정상 |

## 5. 이상 패턴 (우상향) - "이러면 문제야"

### 5-1. 개념

이상 패턴은 **기준선(baseline)이 점점 올라가는** 패턴이야.

- 메모리 올라감 → GC → **덜 내려감**
- 다시 올라감 → GC → **더 덜 내려감**
- GC가 치워도 치워도 바닥이 올라가 → GC Root가 잡고 놔주지 않는 놈이 있는 거야
- 결국 꼭대기에서 Full GC 연발 → 그래도 안 빠지면 → OOM

### 5-2. 이상 패턴 ASCII 그래프

!!! danger "이상 패턴: 우상향 (Memory Leak Pattern)"

    ```
    OU
    (MB)

    7500 |                                                  /\ ← OOM 임박
         |                                                 /  \
    6500 |                                           /\   /    \← FGC!
         |                                          /  \ /      \
    5500 |                                    /\   /    V
         |                                   /  \ /
    4500 |                             /\   /    V  ← GC 해도 안 내려감
         |                            /  \ /
    3500 |                      /\   /    V
         |                     /  \ /
    2500 |               /\   /    V        ← 바닥이 계속 올라감!
         |              /  \ /
    1500 |        /\   /    V
         |       /  \ /
     500 |  /\  /    V
         | /  \/
         +------+------+------+------+------+------+------+------→ 시간
          Day1   Day2   Day3   Day4   Day5   Day6   Day7   Day8
    ```

    **특징:**

    - GC가 돌아도 이전보다 덜 내려감 (기준선 상승)
    - 바닥이 점점 올라감: 500 → 1500 → 2500 → 3500 → ...
    - Full GC 빈도가 점점 잦아짐
    - Full GC 한 번에 걸리는 시간도 길어짐 (스캔할 객체가 많으니까)
    - 결국 OOM 또는 Full GC 무한 루프로 서비스 장애

    **이 패턴이 보이면: 메모리 누수 확정.** 코드 또는 설정 문제. 우리 서버에서 실제로 이 패턴이 나왔었어. (10장에서 상세 분석)

### 5-3. 야간에도 올라가는 패턴 (확정적 누수)

이게 가장 무서운 패턴이야. 사용자가 **0명인 새벽에도** 메모리가 올라가면 그건 100% 누수야.

!!! danger "이상 패턴 변형: 야간에도 우상향"

    ```
    모니터링 %

    80% |                                            /─────
        |                                           /  ← 야간인데
    70% |                         /────────────────/    올라감!
        |                        /   ← 사용자 없는 밤에도
    60% |       /───────────────/       메모리가 올라가면
        |      /                        그건 누수야.
    50% |─────/
        |
    40% |
        +------+------+------+------+------+------+------→ 시간
         Day1     Day1     Day2     Day2     Day3    Day3
         12:00    00:00    12:00    00:00    12:00   00:00
               야간       야간              야간
    ```

    왜 야간에 올라가냐? → 사용자 요청과 무관한 메모리 소비 = **누수**

    우리 서버 실제 사례: RefreshableSqlSessionFactoryBean이 1초마다 126개 파일 감시 → 매 감시마다 DelegatingClassLoader 생성 → Timer-0 스레드가 GC Root → 절대 회수 불가 → 사용자 0명이어도 Timer가 돌면서 메모리를 먹었어

### 5-4. 계단식 점프 패턴

!!! danger "이상 패턴 변형: 계단식 (Step Pattern)"

    ```
    모니터링 %

    70% |                                    ┌──────────
        |                                    |
    60% |                      ┌─────────────┘  ← 자정/03시에
        |                      |                   갑자기 점프
    50% |        ┌─────────────┘
        |        |  ← 특정 시각에 갑자기 올라감
    40% | ───────┘
        |
    30% |
        +------+------+------+------+------+------+------→ 시간
         Day1   Day2   Day3   Day4   Day5   Day6   Day7
    ```

    특정 시각에 반복적으로 뭔가가 실행돼서 메모리를 먹는 패턴 → 크론잡, 스케줄 작업, 서비스 재시작, 자동 배포 등

    **우리 서버 실제 사례:** test 앱이 Tomcat의 autoDeploy="true"로 설정되어 있어서 자정/03시 재기동할 때마다 test 앱이 재배포 → 매번 RefreshableSqlSessionFactoryBean + Timer 새로 생성 → 메모리가 계단식으로 점프

### 5-5. 정상 vs 이상 비교 한 눈에

!!! abstract "정상 vs 이상 비교 한 눈에"

    ```
    [정상: 톱니 패턴]              [이상: 우상향 패턴]

    OU                             OU
    |    /\    /\    /\            |              /\
    |   /  \  /  \  /  \          |            /  \  /\
    |  /    \/    \/    \         |          /    \/  \
    | /                  \        |        /          /\
    +────────────────→ 시간      |      /          /  \
                                  |    /          /    \
    바닥이 일정                    |  /          /
    = 누수 없음                    |/          /
    = GC가 잘 돌고 있음           +────────────────→ 시간

                                   바닥이 올라감
                                   = 누수
                                   = GC가 못 치우는 놈이 있음
    ```

    **기준선이 일정하냐 올라가냐. 이 한 가지가 정상/이상의 핵심 판단 기준이야.**

## 6. "메모리 55%인데 정상이에요?" 답하는 프레임워크

누군가 "메모리 55%인데 괜찮아요?"라고 물어봤을 때, **이 4단계**를 순서대로 확인해.

### Step 1: free -h로 available 확인

```bash
$ free -h
              total        used        free      shared  buff/cache   available
Mem:           15Gi       8.4Gi       1.0Gi       0.0Ki       6.2Gi       6.9Gi
Swap:         4.0Gi       145Mi       3.9Gi
```

!!! note "Step 1 체크포인트"

    - available = 6.9Gi
    - available / total = 6.9 / 15 = **46%**
    - → 46% 여유 → **안전** (기준: 20% 이상이면 안전)
    - 결론: OS 레벨에서는 여유 충분. 다음 Step으로.

### Step 2: jstat으로 OU 확인

```bash
$ jstat -gc $(pgrep -f catalina)
 S0C      S1C      S0U    S1U      EC        EU        OC         OU       ...
174592.0 174592.0  0.0   40277.8 1398272.0 1084498.1 8388608.0  451133.9  ...
```

!!! note "Step 2 체크포인트"

    - OU = 451134 KB = ~440MB / OC = 8388608 KB = 8GB
    - OU / OC = 440MB / 8GB = **5.4%** → Old Gen 5% 사용 → **아주 건강**
    - 추가 OU 추이 확인: `jstat -gc $(pgrep -f catalina) 5000 10`
    - → 440 → 445 → 438 → 442 → 440 → 441 → ... → 기준선 일정 → 톱니 패턴 → 정상
    - 결론: JVM 힙 누수 없음. 다음 Step으로.

### Step 3: Swap 사용량 확인

!!! note "Step 3 체크포인트"

    - Swap used = 145Mi / Swap total = 4.0Gi = **3.5%**
    - → Swap 거의 안 쓰고 있음
    - → 증가 추세인지 확인: 어제 같은 시간에도 145Mi였는지 비교
    - → 일정하면 과거 흔적일 뿐, 문제 없음
    - 결론: Swap 문제 없음. 다음 Step으로.

### Step 4: 야간 패턴 확인 (횡보 vs 우상향)

!!! note "Step 4 체크포인트"

    모니터링 대시보드(InterMax/Grafana)에서 24시간~72시간 그래프 확인

    Q: 야간(00시~06시) 메모리 그래프가?

    - **(A) 수평선 (횡보)** → 정상. 트래픽 없으니 메모리도 안정. 사용자 요청과 메모리가 비례 = 건강한 서버.
    - **(B) 계속 올라감 (우상향)** → 이상. 사용자 없는데 메모리 증가 = 누수. 즉시 원인 분석 필요.
    - **(C) 계단식 점프** → 이상. 특정 시각에 뭔가가 메모리를 먹음. 크론잡, 배포 스크립트, 자동 재시작 확인.

    **우리 서버 (누수 해결 후):** 야간 55%에서 수평 유지 → 정상. 결론: 누수 패턴 아님.

### 최종 판단

!!! abstract "최종 판단: '메모리 55%인데 정상이에요?'"

    | Step | 확인 항목 | 결과 |
    |------|----------|------|
    | Step 1 | available 46% | OK (20% 이상) |
    | Step 2 | OU 5.4%, 기준선 일정 | OK (톱니 패턴) |
    | Step 3 | Swap 3.5%, 증가 없음 | OK |
    | Step 4 | 야간 횡보 | OK (누수 패턴 아님) |

    **최종 답변:** "55%이지만, available 46%, OU 5.4%, Swap 증가 없음, 야간 횡보 확인. 4가지 기준 전부 충족. **정상입니다.**"

    이렇게 근거를 대고 말해야 하는 거야. "55%면 괜찮겠네~"가 아니라 **구체적 수치 + 판단 기준 + 결론**. 이게 시니어의 보고야.

## 7. 위험 신호 체크리스트

이건 외워. 이 중 하나라도 해당되면 조사해야 해.

!!! danger "위험 신호 체크리스트"

    | 레벨 | 신호 | 확인방법 | 의미 |
    |------|------|----------|------|
    | 주의 | available < 총 RAM의 20% | free -h | 여유 줄어듦 |
    | 주의 | Swap이 0이 아님 (존재) | free -h | 과거 압박 흔적 |
    | 경고 | available < 1GB | free -h | OS 여유 부족 |
    | 경고 | Swap이 지속적으로 증가 | free -h (시계열) | RAM 수요 > RAM 공급 |
    | 경고 | OU가 GC 후에도 기준선 상승 | jstat | 누수 의심 |
    | 경고 | FGC 횟수가 시간당 1회 이상 | jstat | Old 포화 접근 |
    | 위험 | Full GC 빈도 급증 (분당 1회 이상) | jstat | Old Gen 포화 임박 |
    | 위험 | Full GC 후에도 OU 안 줄어듦 | jstat | 누수 확정적 |
    | 위험 | OU/OC > 80% 지속 | jstat | OOM 임박 |
    | 심각 | available < 500MB | free -h | OOM Killer 발동 임박 |
    | 심각 | OU/OC > 90% + FGC 연속 발생 | jstat | OOM 초읽기 |
    | 심각 | Swap 전체의 50% 이상 사용 | free -h | 서비스 장애 |

### 위험 단계별 대응

!!! warning "위험 단계별 대응"

    - **[주의]** → 관찰 강화. 모니터링 주기 줄이기. 추이 기록 시작. *"지켜보고 있어야 해"*
    - **[경고]** → 원인 조사 시작. 담당자/팀에 보고. OU 추이 집중 모니터링. 힙덤프 준비. *"원인 찾아야 해"*
    - **[위험]** → 즉시 조치. 힙덤프 떠놓기. 재시작 타이밍 조율. 장애 대응 체계 가동. *"지금 당장 뭔가 해야 해"*
    - **[심각]** → 힙덤프 떠놓고 즉시 재시작. 트래픽 분산/차단. 재시작 후 사후 분석. *"이미 터지고 있거나 곧 터져"*

## 8. 모니터링 도구별 차이

### 8-1. 도구별 특성 비교

!!! note "도구별 특성 비교"

    **top / htop**

    - 보여주는 것: %MEM (RSS / 전체 RAM * 100), RES (물리 메모리 사용량), VIRT (가상 메모리 전체)
    - 장점: 프로세스 단위 실시간 확인. 어떤 프로세스가 먹는지 한눈에.
    - 단점: 히스토리(추이) 못 봄. 지금 이 순간만 보여줌.
    - 우리 서버: PID 123, VIRT 15.0g, RES 8.4g, %MEM 55.0 (톰캣)

    ---

    **free -h**

    - 보여주는 것: total, used, free, shared, buff/cache, available + Swap
    - 장점: OS 전체 메모리 상태를 한 번에 파악. **available이 핵심.**
    - 단점: 히스토리 없음. 프로세스별 분리 안 됨.

    ---

    **jstat -gc**

    - 보여주는 것: JVM 힙 영역별 용량(C)과 사용량(U), GC 횟수(YGC, FGC)와 소요시간(YGCT, FGCT)
    - 장점: JVM 내부의 진짜 상태를 볼 수 있음. **OU 추이가 누수 판단 핵심.**
    - 단점: JVM만 봄. OS 전체는 못 봄. 그래프 없이 숫자 나열.
    - 추이 확인: `jstat -gc <PID> 5000 20` (5초 간격 20회 = 100초간 관찰)

    ---

    **InterMax (우리 LMS 모니터링 APM)**

    - 보여주는 것: 서버 메모리 사용률 %, 시계열 그래프, 알람 설정, 응답 시간/TPS/에러율
    - 장점: **그래프로 추이를 볼 수 있음** (최대 장점). 야간 횡보/우상향/계단식 패턴 판단 가능.
    - 단점: % 계산 기준이 뭔지 정확히 확인 필요. buff/cache 포함 여부가 설정에 따라 다를 수 있음.

    ---

    **Grafana + Prometheus/node_exporter**

    - 보여주는 것: OS (total, used, available, cached, buffers) + JVM (JMX 연동 시)
    - 장점: 가장 유연. 계산 공식 커스터마이징. 시계열 + 알람 + 대시보드 공유.
    - 단점: 직접 구축/설정 필요. 러닝 커브.

### 8-2. 도구별 "메모리 %" 계산 비교 (같은 서버)

!!! danger "도구별 '메모리 %' 계산 비교 (같은 서버)"

    서버 상태: 15GB RAM, used 8.4GB, buff/cache 6.2GB, free 1.0GB, available 6.9GB, 톰캣 RSS 8.4GB

    | 도구 | 계산 공식 | 결과 |
    |------|----------|------|
    | top (%MEM) | RSS / total | **55%** |
    | free -h 기반 | used / total | **56%** |
    | Grafana (A) | (total - available) / total | **54%** |
    | Grafana (B) | (total - free) / total | **93%** |
    | 구형 도구 | (used + buff/cache) / total | **97%** |

    같은 서버인데 54%에서 97%까지 차이남.
    **이래서 "55%입니다"라는 말에 "뭐 기준 55%?"라고 물어야 하는 거야.**

### 8-3. 실전 도구 조합 전략

!!! tip "실전 모니터링 3단계 조합"

    **[1단계: 대시보드로 큰 그림 파악]**
    InterMax 또는 Grafana 시계열 그래프 → 톱니인지 우상향인지 야간 횡보인지 계단식인지 → "느낌"을 잡는 단계. 근데 느낌만으로는 부족해.

    **[2단계: CLI로 현재 상태 정밀 진단]**

    - `free -h` → available, Swap 확인
    - `jstat -gc $(pgrep -f catalina)` → OU, FGC 확인
    - `ps -eo pid,rss,comm --sort=-rss | head -5` → 상위 프로세스

    → **"근거"를 만드는 단계.** 숫자로 이야기해야 해.

    **[3단계: 의심 시 심층 분석]**

    - `jstat -gc <PID> 5000 20` → OU 추이 100초간 관찰
    - `jmap -histo <PID> | head -20` → 객체 수/크기 상위
    - `jcmd <PID> GC.heap_dump /tmp/dump.hprof` → 힙덤프
    - → MAT(Eclipse Memory Analyzer)로 분석 → Leak Suspects, Dominator Tree에서 누수 객체 식별

    대시보드만 보면 "느낌"만 알 수 있어. **CLI로 확인해야 "근거"가 생겨.** 근거 없는 판단은 판단이 아니야. 찍기야.

## 9. 실전: 알람이 울렸을 때 대응 순서

!!! warning "알람이 울렸을 때 대응 순서"

    **시나리오:** 오후 3시, 모니터링 알람 "WAS01 메모리 사용률 75% 초과"

    **[즉시 - 30초 이내]** `free -h`

    - available 확인: > 3GB → 급한 건 아님 / < 1GB → 위험
    - Swap 확인: 급증 중이면 위험 레벨 상향

    **[1분 이내]** `jstat -gc $(pgrep -f catalina)`

    - OU/OC 비율 계산: < 50% → JVM 넉넉 / > 80% → 힙 포화 임박

    **[5분 이내]** `jstat -gc $(pgrep -f catalina) 5000 12`

    - OU 추이 1분간 관찰: 올라가는 추세 → 누수 의심 / 톱니 패턴 → 일시적 부하

    **[15분 이내]** 대시보드에서 24~72시간 그래프 확인

    - 야간 횡보 + 주간만 올라감 → 트래픽 부하 (정상)
    - 며칠째 우상향 → 누수 (비정상)
    - 어제 배포 후부터 → 배포 코드 문제

    **[판단]**

    - A) available 넉넉 + OU 안정 + 톱니 패턴 → *"일시적 부하. 관찰 계속. 서비스 영향 없음."*
    - B) available 줄어드는 중 + OU 상승 + 우상향 → *"누수 의심. 힙덤프 떠놓기. 원인 분석 착수."*
    - C) Swap 급증 + Full GC 연발 + OU 90% 이상 → *"즉시 힙덤프 떠놓고 재시작. 장애 보고."*

## 10. 주의사항 / 함정

### 함정 1: "%만 보고 판단"

!!! danger "함정 1"

    **X 틀린 판단:** "메모리 55%니까 괜찮아" / "메모리 80%니까 위험해"

    **O 맞는 판단:** %는 입구일 뿐이야. available, OU, Swap, 추이까지 확인해야 진짜 판단이야.

    - 55%인데 OU가 우상향이면? → **위험**
    - 80%인데 buff/cache가 대부분이고 available 넉넉하면? → **안전**

### 함정 2: "한 시점만 보고 판단"

!!! danger "함정 2"

    **X 틀린 판단:** "지금 이 순간 available이 5GB니까 충분해"

    **O 맞는 판단:** 어제도 5GB였어? 아니면 10GB에서 5GB로 줄어든 거야? **추이(Trend)를 봐야 해.** 한 시점의 스냅샷은 절반만 보여줘.

    - 3일째 계속 5GB면 → 안정적이야.
    - 10GB → 7GB → 5GB 추세면 → 며칠 후 0GB 될 수 있어.

### 함정 3: "모니터링 올라갔으니 무조건 문제"

!!! danger "함정 3"

    **X 틀린 판단:** "34%에서 55%로 올라갔다! 문제 있다!"

    **O 맞는 판단:** JVM이 -Xms(8GB) < -Xmx(12GB)로 설정돼 있잖아. JVM이 힙을 점진적으로 확장하면서 → committed 증가 → 새 페이지에 접근 → RSS 증가 → 모니터링 % 증가. 이건 04장에서 배운 JVM의 정상 동작이야.

    **모니터링 %가 올라가는 것 자체가 문제가 아니야. "올라가고 나서 안정화되느냐, 계속 올라가느냐"가 핵심.**

### 함정 4: "GC 돌리면 모니터링 %가 내려간다"

!!! danger "함정 4"

    **X 틀린 기대:** "Full GC 돌았으니 모니터링 %가 내려가겠지?"

    **O 현실:** 08장 핵심이야. GC = JVM 힙 안에서 쓰레기 치우기 (JVM 레벨) / 모니터링 % = 프로세스가 점유한 물리 메모리 RSS 기준 (OS 레벨)

    GC가 힙 안에서 500MB 치워도 → JVM은 OS에 반환 안 해 → RSS 안 줄어듦 → **모니터링 % 안 내려감**

    이래서 모니터링 %만 보면 안 되고 **jstat의 OU를 같이 봐야 하는 거야.**

### 함정 5: "Swap이 0이 아니면 무조건 문제"

!!! danger "함정 5"

    **X 틀린 판단:** "Swap이 145MB나 쓰고 있어! 바로 문제다!"

    **O 맞는 판단:** Swap은 과거 어느 시점에 RAM이 부족해서 밀렸던 흔적일 수 있어. Linux는 vm.swappiness에 따라 약간의 Swap을 선제적으로 쓰기도 해.

    **핵심:** "지금도 계속 늘어나고 있느냐" = **추세가 중요.**

    - 145MB인데 한 달째 145MB → OK. 과거 흔적.
    - 145MB → 300MB → 500MB → 증가 중 → 경고. 원인 찾아.

## 11. 정리

### 핵심 요약 표

| 개념 | 한 줄 정리 |
|------|-----------|
| **모니터링 %** | 계산 기준(공식)에 따라 달라진다. "뭐 기준?"을 항상 물어야 한다. |
| **available** | 기준 1. 전체의 20% 이상이면 안전. 진짜 "OS 여유"야. |
| **Swap 추세** | 기준 2. 존재 자체보다 증가 추세가 중요. 추세 없으면 과거 흔적. |
| **OU 추세** | 기준 3. GC 후 기준선이 올라가면 누수. 일정하면 정상. |
| **정상 패턴** | 톱니(Sawtooth) 패턴. 바닥(기준선)이 일정. |
| **이상 패턴** | 우상향 패턴. 바닥이 계속 올라감. Full GC 빈도 증가. |
| **야간 횡보** | 가장 확실한 정상 신호. 트래픽 없는 시간에 메모리 안정. |
| **야간 상승** | 가장 확실한 누수 신호. 사용자 없는데 메모리 증가. |
| **계단식 점프** | 특정 시각의 스케줄 작업이나 재배포에 의한 이상. |
| **도구별 차이** | 같은 서버도 도구마다 %가 다를 수 있다. /proc/meminfo 해석 차이. |

### 판단 프레임워크 (외워)

!!! tip "판단 프레임워크 (외워)"

    "메모리 XX%입니다" → 이렇게 대응해:

    1. `free -h` → available 확인 (20% 이상이면 안전)
    2. `jstat -gc` → OU 확인 (기준선 일정이면 정상)
    3. `free -h` → Swap 추세 확인 (증가 없으면 OK)
    4. 대시보드 → 야간 패턴 확인 (횡보면 정상)

    4가지 전부 OK → **"정상입니다. 근거: [구체적 수치 나열]"**
    하나라도 이상 → **"조사 필요. [이 지표]가 [이런 상태]입니다."**

### 이 챕터에서 반드시 기억할 것

!!! danger "이 챕터에서 반드시 기억할 것"

    1. **모니터링 %는 계산 기준(공식)에 따라 다르다.** 숫자를 맹신하지 마. "뭐 기준 %?"를 물어.
    2. **정상/이상은 available, OU, Swap 세 가지로 판단한다.** % 하나로 판단하는 건 찍기야.
    3. **한 시점이 아니라 추이(Trend)를 봐야 한다.** 스냅샷은 절반만 보여줘. 패턴이 전체를 보여줘.
    4. **톱니 패턴 = 정상. 우상향 = 누수.** 야간 횡보 = 확실한 정상. 야간 상승 = 확실한 누수.
    5. "55%면 괜찮겠네~"는 주니어. **"available 46%, OU 5.4%, Swap 증가 없음, 야간 횡보. 4가지 기준 충족. 정상입니다."가 시니어.**

### 확인 문제 (5문제)

> 다음 문제를 풀어봐. 답 못 하면 위에서 다시 읽어.

**Q1.** 같은 서버를 보는데 InterMax는 55%, Grafana는 62%, top은 55.0%를 보여준다. 세 숫자가 다른 이유를 설명하고, 어떤 숫자를 기준으로 판단해야 하는지 말해봐.

**Q2.** 모니터링에서 "메모리 80%"라는 알람이 왔다. 이 서버가 진짜 위험한지 판단하려면 어떤 순서로 뭘 확인해야 해? 4단계로 말해봐.

**Q3.** 아래 두 야간 패턴 중 어떤 게 정상이야? 이유를 "기준선" 개념으로 설명해봐.
```
[패턴 A] 야간 00시~06시: 메모리가 수평선으로 55%에서 유지
[패턴 B] 야간 00시~06시: 메모리가 55%에서 61%로 서서히 증가
```

**Q4.** Swap이 500MB 사용 중이다. 이것만으로 "문제"라고 판단할 수 있어? 추가로 확인해야 할 것 2가지를 써봐.

**Q5.** Full GC가 Old Gen에서 1GB를 해제했는데, 모니터링 대시보드의 메모리 사용률 %가 안 내려간다. 왜 그런지, 어떤 챕터의 개념으로 설명할 수 있는지 써봐.

??? success "정답 보기"

    **A1.** 각 도구의 계산 공식이 다르기 때문이야. top의 %MEM은 "해당 프로세스의 RSS / 전체 RAM"으로 계산해서 55%. free -h 기반은 "used / total"로 계산해서 56% 비슷하지만 약간 다를 수 있어. Grafana는 설정된 쿼리에 따라 "(total - available) / total" 또는 다른 공식으로 계산해서 62%가 나올 수 있어. 중요한 건 %가 아니라 available 절대값(free -h), OU 추이(jstat), Swap 추세를 봐야 한다는 거야. % 숫자에 집착하지 말고 근본 지표를 봐.

    **A2.** Step 1: `free -h`로 available 확인. 전체의 20% 이상이면 즉각적 위험은 아님. Step 2: `jstat -gc`로 OU와 FGC 확인. OU/OC 80% 이상이면 힙 포화 임박. Step 3: Swap 사용량과 증가 추세 확인. 급증하고 있으면 위험. Step 4: 대시보드에서 24시간 그래프 확인. 톱니/횡보면 일시적 부하, 우상향이면 누수 의심.

    **A3.** 패턴 A가 정상이야. 야간에는 사용자 트래픽이 없으므로 새 객체가 안 만들어져. 메모리가 수평선(횡보)으로 유지되는 건 "트래픽과 메모리 소비가 비례" = 정상 동작이라는 뜻이야. 패턴 B는 사용자가 없는 야간에도 메모리 기준선이 올라가고 있으므로, 사용자 요청과 무관하게 메모리를 소비하는 뭔가가 있다는 뜻 = 메모리 누수 의심. 우리 서버의 RefreshableSqlSessionFactoryBean이 바로 이런 패턴이었어.

    **A4.** Swap 500MB만으로는 "문제"라고 판단할 수 없어. 추가 확인: (1) Swap 사용량의 시간별 추이. 며칠째 500MB로 일정하면 과거에 한번 밀렸던 흔적이고 지금은 괜찮아. 매일 100MB씩 늘어나면 RAM이 지속적으로 부족해지고 있으므로 위험. (2) `vmstat`의 si(swap in)/so(swap out) 수치. 0이면 현재 활발한 Swap 교환이 없으므로 과거 흔적. si/so가 지속적으로 0이 아니면 현재 진행형 문제.

    **A5.** GC는 JVM 힙 내부에서 쓰레기 객체를 치우는 작업이야 (JVM 레벨). GC가 1GB를 해제해도 JVM은 그 메모리 공간을 OS에 반환하지 않아. "나중에 새 객체 만들 때 재사용할 거니까 갖고 있을게." OS가 보는 RSS(Resident Set Size)는 줄어들지 않고, 모니터링 대시보드는 RSS 기준으로 %를 계산하니까 안 내려가는 거야. 이건 08장 "GC와 OS 페이지 관리는 다른 것이다"의 핵심 개념이야. GC(JVM 레벨)와 OS 페이지 관리(OS 레벨)는 완전히 다른 레벨의 작업이야.
