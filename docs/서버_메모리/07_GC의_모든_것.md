# 07. GC의 모든 것 - Delta

## 1. GC가 왜 필요한지 - "이게 뭐야?"

### 비유부터 가자

자취방. 너 자취 해봤으면 알 거야.

택배 상자, 빈 페트병, 다 먹은 컵라면, 과자 봉지.
매일 새로운 쓰레기가 나와. 치우지 않으면?
**방이 꽉 차서 새 짐을 놓을 데가 없어.**

GC(Garbage Collection)가 딱 그거야.
JVM 힙 안에 `new`로 만든 객체들이 쌓여.
어떤 건 아직 쓰고 있고, 어떤 건 더 이상 아무도 참조하지 않는 쓰레기야.
GC는 이 쓰레기 객체를 찾아서 메모리를 회수하는 **JVM의 자동 청소 시스템**이야.

비유는 여기까지. 정확한 정의:

```
GC (Garbage Collection)
= JVM이 힙(Heap) 메모리에서 더 이상 참조되지 않는 객체를 자동으로 탐지하고
  해당 메모리를 회수하는 메커니즘
```

### C/C++의 세계: 개발자가 직접 치우는 세상

Java 이전에는 어떻게 했을까? C/C++은 개발자가 **직접** 메모리를 할당하고 해제해야 해.

```c
// C언어 - 개발자가 직접 관리
char* name = (char*)malloc(100);   // 메모리 100바이트 할당
strcpy(name, "이준성");             // 데이터 넣기
// ... 사용 ...
free(name);                         // 반드시 직접 해제해야 해!
```

```cpp
// C++ - 마찬가지로 직접 관리
User* user = new User("이준성");   // 객체 생성 (메모리 할당)
// ... 사용 ...
delete user;                        // 반드시 직접 해제!
```

이게 왜 문제냐?

!!! danger "C/C++ 수동 메모리 관리의 지옥"

    **[버그 1] 메모리 누수 (Memory Leak)** → free()를 깜빡하면? 그 메모리는 영원히 회수 안 됨 → 프로그램이 오래 돌수록 메모리가 계속 쌓임 → 결국 메모리 부족으로 프로그램 죽음

    **[버그 2] Dangling Pointer (허상 포인터)** → free()한 메모리를 또 접근하면? 이미 해제된 곳을 읽음 → 엉뚱한 값 → 프로그램 폭주

    **[버그 3] Double Free (이중 해제)** → 같은 메모리를 두 번 free()하면? 힙 구조가 망가짐 → 프로그램 크래시

    **[결과]** C/C++ 버그의 상당수가 메모리 관리 실수에서 나와. 대규모 프로젝트에서 이걸 완벽하게 관리하는 건 거의 불가능.

### Java의 해답: GC에게 맡기자

Java는 이 문제를 어떻게 풀었냐? **개발자한테 해제를 안 시킨다.**

```java
// Java - GC가 알아서 치워줌
User user = new User("이준성");   // 객체 생성
// ... 사용 ...
user = null;                       // 참조 끊기 (또는 스코프 벗어남)
// free? delete? 필요 없어. GC가 알아서 치워줘.
```

!!! abstract "C/C++ vs Java 메모리 관리 비교"

    |  | C/C++ | Java |
    |------|-------|------|
    | 메모리 할당 | malloc() / new | new |
    | 메모리 해제 | free() / delete | GC가 자동으로 |
    | 누수 위험 | 매우 높음 | 낮음 (완전 0은 아님) |
    | Dangling Pointer | 발생 가능 | 불가능 (참조 추적) |
    | Double Free | 발생 가능 | 불가능 |
    | 개발자 부담 | 매우 높음 | 낮음 |
    | 대가 | 없음 (직접 관리) | STW 비용 (GC가 일할 때) |

    Java가 편하지만 공짜는 아니야. GC가 돌 때 "잠깐 다 멈춰!" 하는 비용(STW)을 감수해야 해. 이게 나중에 서비스 장애 원인이 될 수 있어.

### "쓰레기"의 정의

여기서 핵심. GC가 치우는 "쓰레기"란 뭐야?

```
쓰레기(Garbage)의 정확한 정의:
= 어떤 GC Root에서도 도달할 수 없는(Unreachable) 객체
```

"안 쓰는 객체"가 아니야. **"도달할 수 없는 객체"**야.
이 차이가 중요해. 다음 섹션에서 GC Root를 배우면 왜 이 정의가 중요한지 알게 돼.

GC가 없으면 어떻게 되냐?

!!! warning "GC가 없으면?"

    `new → new → new → new → ... → 힙 꽉 참` → `java.lang.OutOfMemoryError: Java heap space` → 서비스 다운 → 사용자: "LMS가 안 돼요"

    **GC가 있으면:** `new → new → [GC: 안 쓰는 거 치움] → new → new → ...` → 힙이 재활용돼서 계속 돌아감

    **우리 서버 (-Xmx12288m):** 12GB 힙에 new를 계속 하면 빠르면 몇 분 만에 꽉 찰 수 있어. GC가 계속 치워주니까 OU가 5% 수준으로 유지되는 거야.

## 2. GC Root란 - "누가 살아있는 놈이야?"

### GC Root가 뭐야?

GC는 "누가 쓰레기인지" 어떻게 판별해?
**GC Root에서 시작해서 참조 체인을 따라갈 수 있는 객체 = 살아있는 놈.**
따라갈 수 없는 객체 = 쓰레기.

GC Root는 "참조 추적의 시작점"이야. 이 시작점에서 출발해서 참조를 쭉 따라가.

### GC Root 4대 천왕

!!! note "GC Root 4대 천왕"

    **1. Stack 지역 변수 (Local Variables)** → 현재 실행 중인 메서드의 지역 변수, 파라미터. 메서드가 끝나면 스택에서 빠지고, 참조가 끊김. 가장 흔한 GC Root.

    **2. Static 변수 (Class Variables)** → 클래스에 붙어있는 변수. 클래스가 살아있는 한 참조 유지. 사실상 프로그램 끝날 때까지 살아있는 강력한 Root. **주의:** static 컬렉션에 객체를 넣고 안 빼면? 영원히 안 죽어. 이게 메모리 누수의 클래식한 원인 중 하나야.

    **3. JNI (Java Native Interface) 참조** → 네이티브 코드(C/C++)에서 잡고 있는 Java 객체 참조. JNI 코드에서 GlobalRef로 잡으면 GC Root가 됨. 서버 개발에서는 거의 볼 일 없어.

    **4. 실행 중인 스레드 (Active Thread)** → 살아있는 스레드 자체가 GC Root. 스레드 + 그 스레드의 스택에 있는 모든 지역 변수까지. **스레드가 안 죽으면 → 그 스레드가 참조하는 것도 안 죽어 → 스레드 누수 = 메모리 누수**

### GC Root가 왜 중요한지 - 우리 서버 실제 사례

우리 LMS 서버에서 실제로 문제됐던 거 기억나?
`RefreshableSqlSessionFactoryBean`이 **Timer 스레드**를 1초마다 만들었어.

Timer = 실행 중인 스레드 = **GC Root**.
GC Root가 잡고 있으면? GC가 못 치워.
Timer가 참조하는 DelegatingClassLoader → 그게 참조하는 클래스 메타데이터 → 전부 살아있음.
결과? Metaspace 메모리 누수.

!!! danger "우리 서버 메모리 누수와 GC Root의 관계"

    ```
    [GC Root: Timer 스레드]
          │
          ▼
    [TimerTask]
          │
          ▼
    [RefreshableSqlSessionFactoryBean]
          │
          ▼
    [DelegatingClassLoader]  ← 1초마다 새로 생성
          │
          ▼
    [클래스 메타데이터 in Metaspace]
    ```

    Timer 스레드가 살아있는 한 → 이 체인 전체가 살아있음 → GC가 절대 수거 못 함 → Metaspace 계속 증가 → 서비스 장애

이게 바로 GC Root를 이해해야 하는 이유야.
"왜 GC가 안 치워요?"에 대한 답은 거의 항상 "GC Root에서 도달 가능하니까"야.

### Reachability Analysis (도달 가능성 분석)

GC가 실제로 쓰레기를 구분하는 방식이야.

```
          GC Root (스택 변수)
            │
            ▼
        ┌───────┐      ┌───────┐      ┌───────┐
        │ Obj A │─────→│ Obj B │─────→│ Obj C │   ← 전부 Reachable = 살림
        └───────┘      └───────┘      └───────┘

        ┌───────┐      ┌───────┐
        │ Obj D │─────→│ Obj E │   ← GC Root에서 도달 불가 = 쓰레기
        └───────┘      └───────┘
               ↑              │
               └──────────────┘   ← 순환 참조해도 GC Root에서 안 닿으면 쓰레기

        ┌───────┐
        │ Obj F │   ← 아무도 참조 안 함 = 확정 쓰레기
        └───────┘
```

D와 E가 서로 참조하고 있어도 GC Root에서 도달 못 하면? **쓰레기.**
이전에 쓰던 "Reference Counting" 방식은 순환 참조를 못 잡아.
A→B, B→A 하면 둘 다 카운트 1이라 못 치워.
근데 Reachability Analysis는? GC Root에서 못 닿으면 끝이야. 순환이든 뭐든.

!!! abstract "Reference Counting vs Reachability Analysis"

    | Reference Counting | Reachability Analysis (Java가 쓰는 방식) |
    |---|---|
    | 객체마다 참조 수를 카운트 | GC Root에서 출발해서 도달 가능한지 확인 |
    | 카운트가 0이면 쓰레기 | 도달 불가능하면 쓰레기 |
    | 순환 참조 못 잡음 | 순환 참조도 잡음 |
    | Python이 이거 씀 (+순환 참조용 추가 GC 있음) | Java, C#이 이거 씀 |

## 3. Mark - Sweep - Compact

GC의 실제 동작은 3단계야:

!!! note "Mark - Sweep - Compact"

    ```
    [1단계: Mark (표시)]
    GC Root에서 출발 → 도달 가능한 객체에 "살아있음" 표시

    메모리: [*A][ D][*B][ E][*C][ F][*G][ H]
    *=Reachable(살림)  빈칸=Unreachable(쓰레기)
    이 단계가 비싸. 전체 참조 그래프를 순회해야 하니까.

    [2단계: Sweep (쓸어버림)]
    표시 없는 놈들 제거 → 메모리 해제

    메모리: [ A][  ][ B][  ][ C][  ][ G][  ]
    문제: 빈 공간이 여기저기 흩어져 있어! (Fragmentation/단편화)

    [3단계: Compact (압축)]
    살아남은 객체를 한쪽으로 밀어서 빈 공간 합침

    메모리: [ A][ B][ C][ G][      빈 공간 (연속)      ]
    → 연속된 빈 공간 확보! 새 객체 할당 가능
    ```

    Compact도 비싸. 객체를 물리적으로 옮기고, 그 객체를 참조하는 다른 객체들의 참조 주소도 전부 업데이트해야 해.

## 4. Minor GC (Young GC) - "빠르고 자주 돌아"

### Minor GC란?

06장에서 배운 JVM 힙 구조 기억해? 그걸 다시 꺼내와.

!!! note "JVM Heap 구조"

    ```
    ┌──────────────── Young Generation ────────────────┐  ┌─────────┐
    │ ┌──────────┐  ┌──────────┐  ┌──────────┐        │  │  Old    │
    │ │  Eden    │  │Survivor 0│  │Survivor 1│        │  │  Gen    │
    │ │ 새 객체  │  │  (From)  │  │  (To)    │        │  │ 오래된  │
    │ │ 여기 생성│  │ 살아남은 │  │  비어있음│        │  │ 객체들  │
    │ └──────────┘  └──────────┘  └──────────┘        │  │         │
    └──────────────────────────────────────────────────┘  └─────────┘
    ← Minor GC 대상 (Young Gen만) →          ← Full GC 대상 (전체) →
    ```

### Minor GC 발생 과정

!!! note "Minor GC 동작 과정 (상세)"

    **[STEP 1] Eden이 가득 참** → Minor GC 트리거!

    **[STEP 2] Eden + S0(From)에서 살아있는 객체를 S1(To)으로 복사** (age+1)

    **[STEP 3] Eden과 S0을 전부 비움** (한 방에 싹 날림)

    **[STEP 4] S0과 S1의 역할 교대** (From ↔ To 스왑). 다음 Minor GC 때는 S1이 From, S0이 To가 됨.

    **[STEP 5] age가 임계값(기본 15)에 도달한 객체 → Old Gen으로 승격.** 이걸 Promotion(승격)이라고 해.

### 왜 Minor GC는 빠를까?

핵심적인 관찰 하나: **대부분의 객체는 금방 죽어.**

이걸 **Weak Generational Hypothesis** (약한 세대 가설)라고 해:
> "대부분의 객체는 금방 죽고, 오래 사는 놈은 계속 오래 산다."

우리 LMS 서버 생각해봐. HTTP 요청 하나 처리하면서 뭘 만들어?

```
사용자가 LMS 과목 목록 조회:
  → HttpServletRequest 객체   (요청 끝나면 쓸모없음)
  → CourseVO 객체 10개        (응답 보내면 쓸모없음)
  → List<CourseVO>            (응답 보내면 쓸모없음)
  → String (JSON 변환용)      (응답 보내면 쓸모없음)
  → ...

요청 처리 시간: 100ms
이 객체들의 수명: 100ms

Eden에서 태어나서 Eden에서 죽어. Survivor까지도 안 가.
```

Eden의 90% 이상이 이런 단명 객체야.
Minor GC는 "살아남은 놈만 복사"하면 되니까, 10% 이하만 복사하면 끝.
나머지 90%는 Eden을 싹 비우면서 한 방에 날려.

```
Minor GC 속도가 빠른 이유:
1. 대상 영역이 작음 (Young Gen만, 전체 힙의 1/3 정도)
2. 대부분의 객체가 이미 죽어있음 (복사할 게 적음)
3. 복사 방식이라 Compact가 필요 없음 (To 영역에 순서대로 쌓음)

결과: 보통 수~수십 ms. 우리 서버에서 평균 48ms.
STW: 짧아서 사용자가 거의 못 느낌.
빈도: 자주 (수초~수분 단위). 이게 정상.
```

## 5. Full GC (Major GC) - "느리고 무섭다"

### Full GC란?

!!! danger "Full GC 상세"

    **발생 조건:** (1) Old Gen이 가득 참 (Promotion 받을 공간 부족) (2) Metaspace가 부족 (클래스 메타데이터 공간 부족) (3) System.gc() 호출 (4) Concurrent Mode Failure (CMS에서 동시 수행 중 Old Gen 부족)

    **대상:** 전체 Heap (Young Gen + Old Gen) + Metaspace

    **과정:** 1. Mark: 전체 힙에서 살아있는 객체 표시 → 2. Sweep: 쓰레기 제거 → 3. Compact: 살아남은 객체를 한쪽으로 밀어서 빈 공간 합침

    **왜 느림:** 우리 서버 -Xmx=12GB. 12GB 전체를 Mark-Sweep-Compact 해야 해. Old Gen(8GB)에는 오래 살아남은 객체들이 빽빽이 들어있어. 이걸 전부 스캔하고, 정리하고, 압축하는 데 수백 ms ~ 수 초.

    - **속도:** 느림 (수백 ms ~ 수 초, 힙 크기에 비례)
    - **빈도:** 드묾 (정상적이면 거의 안 일어남)
    - **STW:** 길다 (이게 문제)

### Full GC가 왜 무서운지

Full GC가 도는 동안 **모든 애플리케이션 스레드가 멈춰.**
LMS에 학생 100명이 동시에 접속해 있는 상황에서 Full GC가 2초 걸리면?

```
Full GC 2초 동안 벌어지는 일:

09:30:00.000  Full GC 시작 (STW!)
09:30:00.000  학생A: 과제 제출 클릭 → 요청 서버 도착 → 처리 멈춤
09:30:00.100  학생B: 강의실 입장 클릭 → 서버 응답 없음
09:30:00.300  학생C: 성적 조회 → 로딩 중...
09:30:00.500  학생D: "사이트 느린데요?" 슬랙 메시지
09:30:01.000  학생E: 타임아웃 에러 발생 (DB 커넥션 타임아웃)
09:30:01.500  학생F: "LMS 접속 안 돼요!" 문의 전화
09:30:02.000  Full GC 끝 (STW 해제)
09:30:02.001  밀려있던 요청 한꺼번에 처리 시작

2초. 겨우 2초.
근데 그 2초 동안 서버에 도착한 모든 요청이 멈춰있었어.
```

## 6. STW (Stop-The-World) 상세 - "왜 세상이 멈춰야 하는가"

### STW가 뭐야?

!!! note "Stop-The-World (STW)"

    ```
    시간 →  ──────────────────────────────────────────→

    App:    ████████████ ░░░░░░░░░ ████████████████████
    GC:     ............. █████████ ....................
                         ↑       ↑
                      STW 시작  STW 끝

    ████ = 애플리케이션 실행 중
    ░░░░ = 애플리케이션 정지 (STW)
    ████(GC) = GC 작업 중
    ```

### 왜 멈춰야 해?

GC가 객체 참조 그래프를 스캔하는 동안
애플리케이션이 계속 객체를 만들고 참조를 바꾸면?
스캔 결과가 뒤죽박죽이 돼.

비유하면 이래:

```
은행에서 금고 재고 조사를 한다고 생각해봐.

❌ 나쁜 방법:
  재고 조사 하는 동안 은행 영업을 계속 함
  → 직원이 금 개수를 세는 동안 다른 직원이 금을 꺼내감
  → 금 100개인데 세다 보니 98개?? 2개 어디갔지??
  → 결과를 신뢰할 수 없음

✅ GC의 방법:
  "잠깐 영업 멈춰! 아무도 금고 건드리지 마!" (STW)
  → 금 개수 정확히 셈 (Mark)
  → "재고 조사 끝, 영업 재개"
  → 결과를 신뢰할 수 있음
```

이게 STW의 본질이야.
**객체 참조 관계가 변하지 않는 상태에서 정확하게 스캔하기 위해** 모든 스레드를 멈추는 거야.

### STW 시간 비교

!!! abstract "STW 시간 비교 (실전)"

    |  | Minor GC STW | Full GC STW |
    |------|-------|------|
    | 시간 | 수 ms ~ 수십 ms | 수백 ms ~ 수 초 |
    | 체감 | 사용자 못 느낌 | "사이트 느린데요?" |
    | 빈도 | 자주 (초~분 단위) | 드묾 |
    | 위험도 | 낮음 | 높음 (서비스 장애 원인) |
    | **우리 서버 (WAS01)** | **평균 48ms** (901회 / 43.889초) | **평균 336ms** (6회 / 2.016초) |

STW가 길면 어떤 일이 벌어지는지 정리:

```
STW 100ms 미만:  사용자 체감 불가. 정상.
STW 100~500ms:   API 응답 느려짐. 민감한 사용자는 느낌.
STW 500ms~1초:   확실히 느림. "로딩 중..." 표시.
STW 1~3초:       사용자 불만. 타임아웃 가능.
STW 3초 이상:    서비스 장애 수준. 커넥션 풀 고갈 가능.
STW 10초 이상:   사실상 다운. 헬스체크 실패. 로드밸런서가 서버 빼버림.
```

## 7. GC 알고리즘들 - "다 같은 GC가 아니야"

GC 알고리즘은 여러 가지가 있어. 각각 철학이 달라.

### Serial GC

!!! note "Serial GC"

    **JVM 옵션:** `-XX:+UseSerialGC`

    GC를 단일 스레드(1개)로 수행. 가장 단순한 알고리즘. Young Gen: Copy(복사), Old Gen: Mark-Sweep-Compact.

    - **장점:** 구현 단순, 메모리 오버헤드 적음
    - **단점:** STW 길다 (혼자 하니까)
    - **용도:** 클라이언트 앱, 작은 힙 (< 100MB)
    - **서버: 절대 쓰지 마. 서버에서 싱글 스레드 GC는 자살행위.**

### Parallel GC (Throughput Collector)

!!! tip "Parallel GC (우리 서버)"

    **JVM 옵션:** `-XX:+UseParallelGC` (Java 8 기본값!)

    GC를 여러 스레드로 병렬 수행. "처리량(Throughput)" 우선 전략. STW는 있지만 병렬이라 Serial보다 빠름.

    - **장점:** 높은 처리량, 멀티코어 활용
    - **단점:** STW 자체는 여전히 있음 (길이가 줄었을 뿐)
    - **용도:** 배치 작업, 처리량 우선 서버

    **우리 WAS01 서버:** Java 8이므로 Parallel GC가 기본 적용. -Xms8192m -Xmx12288m 환경에서 잘 동작 중.

### CMS (Concurrent Mark Sweep)

!!! note "CMS (Concurrent Mark Sweep)"

    **JVM 옵션:** `-XX:+UseConcMarkSweepGC`

    "응답 시간(Latency)" 우선 전략. Mark와 Sweep의 대부분을 앱 스레드와 동시(Concurrent) 수행. STW를 최소화하려고 노력.

    **동작 (4단계):** Phase 1: Initial Mark (STW - 짧음) → Phase 2: Concurrent Mark (동시) → Phase 3: Remark (STW - 짧음) → Phase 4: Concurrent Sweep (동시)

    - **장점:** STW 매우 짧음
    - **단점:** Compact를 안 해 → 메모리 단편화 발생. 단편화 심해지면 → Concurrent Mode Failure → Full GC 강제 발동. CPU 많이 먹음.

    **상태:** Java 9에서 Deprecated, Java 14에서 제거. 지금은 안 씀. 역사적으로 알아두면 돼.

### G1 GC (Garbage First)

!!! note "G1 GC (Garbage First)"

    **JVM 옵션:** `-XX:+UseG1GC` (Java 9+ 기본값)

    **핵심 아이디어:** 힙을 고정된 Young/Old로 나누지 않고, 동일한 크기의 Region으로 쪼갠다. (E=Eden, S=Survivor, O=Old, H=Humongous)

    **특징:** Region 단위로 GC → 전체 힙을 안 건드려도 됨. 쓰레기가 많은 Region부터 수거 (Garbage First 이름의 유래). STW 시간 목표 설정 가능: `-XX:MaxGCPauseMillis=200` (기본 200ms).

    - **장점:** 예측 가능한 STW 시간. 대용량 힙(4GB 이상)에서 효율적. CMS의 단편화 문제 해결 (Region 단위 Compact).
    - **단점:** 작은 힙에서는 오히려 Parallel보다 나을 게 없을 수 있음. 메모리 오버헤드가 약간 있음 (Region 관리 비용).

    **우리 서버가 Java 11로 올리면 이걸 쓰게 됨.** -Xmx12GB 환경이면 G1GC가 확실히 유리.

### ZGC / Shenandoah (초저지연)

!!! note "ZGC / Shenandoah (미래형)"

    **ZGC:** `-XX:+UseZGC` (Java 15+). STW 10ms 미만 (힙 크기와 무관!). 16TB까지 지원. 거의 모든 작업을 동시(Concurrent) 수행. Oracle 주도 개발.

    **Shenandoah:** `-XX:+UseShenandoahGC` (Java 12+). ZGC와 비슷한 목표 (초저지연). Red Hat 주도 개발. Compact도 동시 수행 (이게 핵심).

    **둘 다 핵심:** STW를 극단적으로 줄임 (10ms 미만). 대규모 서비스에서 필요 (수백 GB 힙). 우리 서버(Java 8)에서는 못 씀. 알고만 있어.

### GC 알고리즘 종합 비교표

!!! abstract "GC 알고리즘 종합 비교표"

    | 알고리즘 | STW | 처리량 | 적합한 힙 | Java 버전 |
    |----------|-----|--------|-----------|-----------|
    | Serial | 길다 | 낮음 | < 100MB | 전부 |
    | **Parallel** ★ | 중간 | 높음 | ~ 수 GB | 8 기본 |
    | CMS | 짧음 | 중간 | ~ 수 GB | 9에서 제거 |
    | G1 | 짧음(예측) | 높음 | 4GB 이상 | 9+ 기본 |
    | ZGC | <10ms | 높음 | TB급 가능 | 15+ |
    | Shenandoah | <10ms | 높음 | GB~TB급 | 12+ |

    ★ = 우리 서버 (WAS01, Java 8)

    **정리:** Java 8 → Parallel (기본) / Java 11 → G1 (기본) / Java 17+ → ZGC/Shenandoah 고려 가능

    어떤 알고리즘을 쓰든 OU가 계속 올라가면 그건 누수야. 알고리즘은 "정리 방식"이지 "누수 치료약"이 아니야.

## 9. GC 로그 읽는 법

### GC 로그 활성화

우리 서버에서 GC 로그를 켜려면 JVM 옵션에 추가해야 해:

```bash
# Java 8 GC 로그 옵션
-verbose:gc
-XX:+PrintGCDetails
-XX:+PrintGCDateStamps
-XX:+PrintGCTimeStamps
-Xloggc:/var/log/tomcat/gc.log
```

### GC 로그 예시 (Minor GC)

```
2026-03-11T09:15:23.456+0900: 12345.678: [GC (Allocation Failure)
  [PSYoungGen: 3145728K->67584K(3407872K)]
  3596816K->518672K(11796480K), 0.0487123 secs]
  [Times: user=0.15 sys=0.01, real=0.05 secs]
```

이걸 뜯어보자:

!!! note "Minor GC 로그 해석"

    | 로그 항목 | 의미 |
    |-----------|------|
    | `2026-03-11T09:15:23.456+0900` | 발생 시각 |
    | `12345.678` | JVM 시작 후 경과 시간 (초) |
    | `[GC (Allocation Failure)` | GC 종류: Minor GC. 원인: Eden에 새 객체 못 넣음 |
    | `[PSYoungGen: 3145728K->67584K` | Young Gen: 3GB → 66MB (거의 다 죽었어!) |
    | `(3407872K)]` | Young Gen 전체 용량: 3.2GB |
    | `3596816K->518672K` | 전체 힙: 3.4GB → 506MB |
    | `(11796480K)` | 전체 힙 용량: 11.2GB |
    | `0.0487123 secs` | 소요 시간: 48.7ms (빠름!) |
    | `user=0.15 sys=0.01 real=0.05` | user > real: 멀티스레드 GC. 4코어 병렬 처리 |

### GC 로그 예시 (Full GC)

```
2026-03-11T14:30:45.789+0900: 54321.012: [Full GC (Ergonomics)
  [PSYoungGen: 67584K->0K(3407872K)]
  [ParOldGen: 8200000K->450000K(8388608K)]
  8267584K->450000K(11796480K),
  [Metaspace: 145000K->145000K(168688K)], 0.3360000 secs]
  [Times: user=1.20 sys=0.05, real=0.34 secs]
```

!!! note "Full GC 로그 해석"

    | 로그 항목 | 의미 |
    |-----------|------|
    | `[Full GC (Ergonomics)` | Full GC 발생! 원인: JVM 자체 판단 (자동 튜닝) |
    | `[PSYoungGen: 67584K->0K]` | Young Gen 완전 비움 |
    | `[ParOldGen: 8200000K->450000K (8388608K)]` | Old Gen: 8.0GB → 440MB (7.5GB를 치웠다!) |
    | `[Metaspace: 145000K->145000K (168688K)]` | Metaspace: 변화 없음 (클래스는 안 치워짐) |
    | `0.3360000 secs` | 336ms! 이 시간동안 STW |

    **이 Full GC는 건강한 거야.** Old Gen이 8GB까지 찼다가 440MB로 내려갔으니까. 누수라면? 치운 후에도 여전히 7~8GB일 거야.

## 10. 메모리 누수 시 GC 패턴 - "이 패턴이 보이면 위험하다"

### 정상 vs 누수 GC 패턴

!!! danger "OU(Old Used) 추이로 판단하는 서버 건강 상태"

    **[정상: 톱니(Sawtooth) 패턴]**

    ```
    OU
    2GB ┤
        │    /\      /\      /\      /\
    1GB ┤   /  \    /  \    /  \    /  \
        │  /    \  /    \  /    \  /    \
   440M ┤ /      \/      \/      \/      \  ← 바닥선(Baseline) 일정
        └──────────────────────────────────→ 시간
    ```

    올라갔다 (객체 생성) → GC 돌면 내려감 → 반복. **바닥선이 안정적 = 정상**

    **[누수: 우상향(Rising Baseline) 패턴]**

    ```
    OU
    8GB ┤                                          / ← OOM 임박!
        │                                        /
    6GB ┤                                /\    /
        │                          /\  /  \  /
    4GB ┤                    /\  /  \/    \/
        │              /\  /  \/
    2GB ┤        /\  /  \/
        │  /\  /  \/          ← 바닥선이 계속 올라감!
   440M ┤/  \/
        └──────────────────────────────────→ 시간
    ```

    GC가 돌아도 바닥선이 계속 올라감 = 뭔가 Old Gen에서 안 치워지는 놈이 있다 = **메모리 누수!**

### 누수 3대 증상

!!! danger "메모리 누수 3대 증상"

    **증상 1: OU 기준선(Baseline) 상승**

    GC 후의 OU가 이전 GC 후보다 높아짐

    | 시점 | GC 후 OU | 비고 |
    |------|----------|------|
    | 09:00 | 440 MB | GC 후 |
    | 11:00 | 480 MB | GC 후 (40MB 높아짐) |
    | 13:00 | 530 MB | GC 후 (50MB 더 높아짐) |
    | 15:00 | 600 MB | GC 후 (70MB 더 높아짐) |

    기준선이 440 → 480 → 530 → 600으로 계속 올라감 = **명백한 누수**

    **증상 2: Full GC 빈도 증가**

    Old Gen이 점점 차니까 Full GC가 점점 자주 발생

    | 월 | FGC 횟수 | 비고 |
    |----|----------|------|
    | 1월 | 2회 | |
    | 2월 | 5회 | |
    | 3월 | 15회 | 급증! |
    | 4월 | 50회 | 위험! |

    **증상 3: Full GC 해도 OU가 안 줄어듦**

    Full GC의 존재 이유: Old Gen 정리. 근데 정리해도 안 줄면?

    - Full GC 전 OU: 7.5 GB
    - Full GC 후 OU: 7.2 GB (겨우 300MB 줄음? Old Gen 8GB인데?)
    - 7.2GB가 전부 살아있는 객체
    - 누군가 계속 잡고 있어서 GC가 못 치움 = **전형적인 메모리 누수**

## 11. 주의사항 / 함정

### 함정 1: "GC 돌리면 모니터링 수치가 내려가겠지?"

!!! danger "함정 1"

    **❌ 틀린 생각:**
    System.gc() 호출 → 모니터링(free -h) used가 내려감

    **✅ 현실:**

    1. System.gc()는 "힌트"일 뿐. JVM이 무시할 수 있어.
    2. 설령 GC가 돌아도 "JVM 힙 안에서" 객체가 치워지는 거야.
    3. OS 모니터링(free -h)의 used는 안 내려가.
    → JVM이 OS한테 메모리를 돌려주지 않거든.
    → 이건 08장에서 자세히 다룰 거야. 핵심 내용이니까.

### 함정 2: "Full GC 횟수가 0이면 최고겠지?"

!!! danger "함정 2"

    **❌ 부분적으로만 맞아:**
    Full GC가 0이면 Old Gen이 넉넉하다는 뜻이긴 해.
    근데 Full GC가 안 일어난다고 누수가 없다는 보장은 아니야.
    OU가 서서히 차오르고 있는데 아직 Full GC 트리거까지 안 왔을 수도 있어.

    **✅ 올바른 판단:**
    Full GC 횟수 + OU 추이를 같이 봐야 해.
    FGC=0이어도 OU가 계속 올라가고 있으면 시한폭탄이야.

### 함정 3: "Minor GC가 1000번 넘으면 문제 아니야?"

!!! danger "함정 3"

    **❌ 아니야.** Minor GC는 자주 일어나는 게 정상이야.
    Young Gen은 작고, 새 객체가 계속 생기니까.
    우리 서버 901회? 서버 시작 후 며칠이면 이 정도는 당연해.

    **✅ 문제가 되는 건:**

    - Minor GC 시간이 비정상적으로 길어질 때 (보통 48ms인데 갑자기 500ms → Survivor가 꽉 차서)
    - Minor GC 후에도 Survivor가 계속 꽉 차서 Old Gen으로 넘어가는 경우 (Premature Promotion → Old Gen이 빨리 차게 됨)

### 함정 4: "GC 알고리즘 바꾸면 누수도 해결돼?"

!!! danger "함정 4"

    **❌ GC 알고리즘은 "정리 방식"이야. "근본 치료"가 아니야.**
    코드에 누수가 있으면 ZGC 써도 결국 OOM 터져.

    **✅ 순서:**

    1. 먼저 누수 있는지 확인 (OU 추이)
    2. 힙 사이즈 적절한지 확인 (-Xms, -Xmx)
    3. 코드 수준 누수 수정
    4. 그래도 STW가 문제면 그때 알고리즘 고려

### 함정 5: "Metaspace는 힙이겠지?"

!!! danger "함정 5"

    **❌ Metaspace는 힙 밖이야.** Native Memory에 있어.
    Java 8부터 PermGen이 Metaspace로 바뀌면서 힙 밖으로 나갔어.
    jstat -gc에서 MC/MU로 보이지만, -Xmx와는 별개야.
    -XX:MaxMetaspaceSize로 따로 제한 걸 수 있어.

    우리 서버 Metaspace: 142MB / 165MB = 86%
    높아 보이지만, 클래스 로딩이 끝나면 안정되는 게 보통이야.
    계속 올라가면? 클래스 로더 누수 (우리가 겪었던 바로 그거).

## 12. 정리

!!! abstract "GC 핵심 정리"

    - **GC** = JVM이 힙에서 안 쓰는 객체를 자동으로 치우는 것
    - **"쓰레기"** = 어떤 GC Root에서도 도달할 수 없는 객체
    - **GC Root 4대장**: Stack 변수, Static 변수, JNI, 실행 중 스레드 (Root에서 도달 불가 = 쓰레기 = 수거 대상)
    - **Mark(표시) → Sweep(제거) → Compact(압축)** 3단계

    |  | Minor GC | Full GC |
    |------|----------|---------|
    | 대상 | Young Gen | 전체 Heap |
    | 속도 | 빠름 (ms 단위) | 느림 (초 단위) |
    | 빈도 | 자주 | 드묾 |
    | STW | 짧음 | 길음 |
    | 트리거 | Eden 꽉 참 | Old Gen 부족 |

    **GC 알고리즘:**

    | 알고리즘 | 특징 |
    |----------|------|
    | Serial | 싱글 스레드. 서버에서 절대 안 씀 |
    | Parallel ★ | Java 8 기본. 처리량 우선 |
    | CMS | 응답시간 우선. Java 9에서 제거 |
    | G1 | Java 9+ 기본. Region 기반 |
    | ZGC | 초저지연 (<10ms). Java 15+ |

    **jstat -gc 핵심 컬럼:**

    - **OU** (Old Used): 누수 판단의 핵심. 계속 올라가면 누수.
    - **YGC/FGC**: GC 횟수. FGC 급증하면 위험 신호.
    - **YGCT/FGCT**: GC 소요 시간. 길어지면 STW 문제.

    **우리 서버(WAS01):**

    - OU = 440MB / 8GB = 5.3% → 매우 건강
    - YGC 901회, 평균 48ms → 정상
    - FGC 6회, 평균 336ms → 정상
    - 결론: **이 서버의 GC는 건강하다.**

    **누수 판단 핵심:**

    - OU 바닥선 안정 → 정상 (톱니 패턴)
    - OU 바닥선 우상향 → 누수
    - Full GC 후에도 OU 안 줄면 → 확정 누수

**이 챕터에서 반드시 기억할 것:**
1. "쓰레기" = GC Root에서 도달 불가능한 객체
2. OU가 계속 올라가면 누수, 톱니 패턴이면 정상
3. Full GC의 STW가 서비스 장애를 일으킨다
4. jstat -gc의 OU, YGC, FGC 세 개만 봐도 건강 판단 가능
5. GC 알고리즘은 "정리 방식"이지 "누수 치료약"이 아니다

## 확인 문제 (5문제)

> 다음 문제를 풀어봐. 답 못 하면 위에서 다시 읽어.

**Q1.** C/C++에서 `free()`를 까먹으면 무슨 일이 생겨? Java에서는 왜 이 문제가 없어? 대신 Java가 감수하는 비용은 뭐야?

**Q2.** GC Root 4가지를 말해봐. 그리고 우리 서버 메모리 누수 사건에서 문제가 된 GC Root 유형은 뭐였고 왜 문제였어?

**Q3.** 아래 OU 추이를 보고 정상/누수를 판단하고 근거를 말해봐:
```
시점    GC 후 OU
08:00   500 MB
10:00   520 MB
12:00   490 MB
14:00   530 MB
16:00   505 MB
18:00   480 MB
```

**Q4.** 우리 서버 jstat 출력에서 S0U=0, S1U=67,562KB이야. S0은 왜 비어있어? 만약 S0U와 S1U 둘 다 0이 아니면 뭘 의심해야 해?

**Q5.** 우리 서버의 GC 알고리즘은 뭐야? 왜 그거야? Java 11로 올리면 기본값이 뭐로 바뀌고, 그게 왜 우리 서버에 유리해?

??? success "정답 보기"

    **A1.** `free()`를 까먹으면 메모리 누수(Memory Leak)가 발생해. 그 메모리는 프로그램이 끝날 때까지 회수되지 않고 계속 쌓여. Java에서는 GC가 자동으로 참조되지 않는 객체를 탐지하고 회수하기 때문에 개발자가 직접 `free()`할 필요가 없어. 대신 GC가 돌 때 STW(Stop-The-World)가 발생해서 모든 애플리케이션 스레드가 일시 정지하는 비용을 감수해야 해. 특히 Full GC의 STW는 수백 ms~수 초 걸릴 수 있어서 서비스 장애 원인이 될 수 있어.

    **A2.** GC Root 4가지: Stack 지역변수, Static 변수, JNI 참조, 실행 중인 스레드. 우리 서버에서는 "실행 중인 스레드" 유형이 문제였어. RefreshableSqlSessionFactoryBean이 Timer 스레드를 1초마다 생성했는데, Timer는 실행 중인 스레드 = GC Root야. GC Root가 참조 체인으로 잡고 있는 DelegatingClassLoader와 그 클래스 메타데이터를 GC가 수거하지 못해서 Metaspace가 계속 증가하는 메모리 누수가 발생했어.

    **A3.** 정상이야. GC 후 OU가 500→520→490→530→505→480으로 올라갔다 내려갔다를 반복하고 있어. 핵심은 **바닥선(Baseline)**이야. 가장 낮은 값들을 보면 490→480으로 오히려 내려가고 있거든. 바닥선이 상승하지 않고 일정 범위(480~530MB)에서 왔다갔다 하는 건 GC가 정상적으로 Old Gen을 회수하고 있다는 뜻이야. 전형적인 톱니(Sawtooth) 패턴이야.

    **A4.** Survivor는 From과 To 두 역할을 번갈아 수행해. Minor GC 때 Eden + From에서 살아남은 객체를 To로 복사하고, 다음 GC 때 From/To 역할이 스왑돼. 그래서 **항상 하나는 비어있어야 정상**이야. S0U=0이면 S0이 현재 "To"(복사 대상), S1이 "From"(객체 있는 쪽)인 상태야. 만약 S0U와 S1U 둘 다 값이 있으면 Survivor 공간이 부족해서 정상적인 복사가 안 되고 있을 가능성이 있어. Premature Promotion(조기 승격)이 발생하고 있을 수 있으니 Old Gen으로의 승격이 과도한지 확인해야 해.

    **A5.** 우리 서버는 Java 8이므로 기본 GC 알고리즘은 **Parallel GC** (Throughput Collector)야. Java 8의 서버 환경 기본값이 Parallel GC이기 때문이야. Java 11로 올리면 기본값은 **G1GC** (Garbage First GC)로 바뀌어. G1GC가 우리 서버에 유리한 이유: (1) 힙을 Region 단위로 쪼개서 관리하므로 12GB 대용량 힙에 적합, (2) STW 시간 목표(`-XX:MaxGCPauseMillis`)를 설정할 수 있어서 Full GC의 긴 STW를 예측 가능하게 줄일 수 있음, (3) Compact를 Region 단위로 수행하므로 단편화 문제가 적음.
