# 04. 프로세스의 메모리 사용 원리 - Beta

---

## 1. 핵심 개념 - "이게 뭐야?"

### 호텔 비유로 문 연다

프로세스가 메모리를 쓰는 과정? 호텔 예약이랑 똑같아.

!!! example "호텔 비유"

    | 호텔 | 메모리 |
    |------|--------|
    | "방 10개 예약할게요" / "네, 잡아놨습니다" | **committed** (예약) -- OS가 가상 주소 공간 할당 |
    | 도착해서 5개 방만 문 열고 짐 풂 | **resident** (실사용) |
    | 나머지 5개: 예약은 돼 있지만 아무도 안 들어감 | committed - resident |
    | 실제로 사람이 있는 방 3개 | 진짜 사용 중인 데이터 |
    | 짐만 놓고 나간 방 2개 | GC 대상 (쓰레기 객체) |

비유는 여기까지. 진짜 얘기하자.

### 본질: 프로세스의 메모리 사용은 3단계다

프로세스가 메모리를 쓰려면 OS한테 먼저 허락을 받아야 해. 프로세스가 직접 RAM을 만지는 게 아니야. 항상 OS를 통해서 간접적으로 사용해.

!!! note "프로세스의 메모리 사용 3단계"

    **1단계: 예약 (committed)**

    - "OS야, 나 이만큼 쓸 수 있게 해줘"
    - OS: "OK, 가상 주소 공간 잡아줄게"
    - 아직 물리 RAM은 안 쓰임!

    **2단계: 실제 사용 (resident)**

    - 프로세스가 그 주소에 실제로 데이터를 쓰는 순간
    - OS가 물리 RAM 페이지를 할당 (page fault 처리)
    - 이제 진짜 RAM을 먹고 있는 상태

    **3단계: 사용 중 / 쓰레기**

    - resident 안에서도 진짜 쓰이는 데이터 vs 안 쓰이는 데이터
    - JVM 세계에서는 GC가 쓰레기를 구분해서 정리
    - OS 입장에서는 둘 다 "사용 중"으로 보임

이 3단계를 모르면 모니터링 수치를 봐도 "이게 정상인지 비정상인지" 판단을 못 해. 왜냐하면 committed, resident, 실제 사용량이 전부 다른 숫자인데, 뭘 보고 판단해야 하는지 모르니까.

## 2. 동작 원리 - "어떻게 돌아가?"

### 2-1. 가상 메모리 vs 물리 메모리

프로세스는 자기만의 독립된 메모리 공간을 갖고 있다고 생각해. 근데 이건 "가상"이야. 실제 RAM 주소가 아니라 OS가 만들어준 가짜 주소 공간이야.

!!! abstract "가상 메모리 vs 물리 메모리 매핑"

    | 프로세스 A가 보는 세상 (가상 메모리) | MMU 변환 | 실제 RAM (물리 메모리) |
    |--------------------------------------|----------|------------------------|
    | 0x0000 0000 | --> 매핑 | 0x3A00 0000 |
    | 0x0000 1000 | --> 매핑 | 0x1200 0000 |
    | 0x0000 2000 | --> 매핑 | 0x7F00 0000 |
    | 0x0000 3000 | 매핑 없음! | - |
    | 0x0000 4000 | 매핑 없음! | - |

    프로세스는 0x0000부터 순서대로 보이지만, 실제로는 RAM 여기저기에 흩어져 있음.

**왜 이렇게 하느냐?**

- **격리**: 프로세스 A가 프로세스 B의 메모리를 건드릴 수 없어. 각자 자기만의 가상 공간이니까.
- **효율**: 물리 RAM이 연속적이지 않아도 프로세스는 연속적으로 보임.
- **과다 할당 가능**: 가상 공간은 물리 RAM보다 더 크게 잡을 수 있어. 어차피 다 쓰지 않으니까.

### 2-2. committed (예약) - 계약서에 사인한 것

committed는 프로세스가 OS한테 "나 이만큼 쓸 수 있게 해줘"라고 요청해서 확보한 가상 메모리 양이야.

!!! warning "committed = 예약이지 실사용이 아니다"

    프로세스: "OS야, 나 12GB 쓸 거야"
    OS: "OK, 가상 주소 공간 12GB 잡아줄게"

    이 시점에서:

    - 가상 주소 공간: 12GB 확보됨
    - 물리 RAM 사용: **거의 0** -- 여기가 핵심!
    - swap 공간 예약: OS 정책에 따라 다름

    **committed = "쓸 수 있는 권리"이지 "쓰고 있는 양"이 아니야!**

JVM에서 `-Xmx12288m`이라고 설정하면? JVM이 OS한테 "나 힙으로 최대 12GB까지 쓸 수 있게 해줘"라고 요청해. OS가 "OK" 하면 12GB가 committed 되는 거야. 근데 JVM이 실제로 12GB를 다 쓰고 있는 건 아니야.

### 2-3. resident (RSS) - 실제로 물리 RAM에 올라간 것

resident는 가상 메모리 중에서 **실제로 물리 RAM에 올라가 있는** 부분이야. 프로세스가 어떤 메모리 주소에 처음 접근하면 그때 물리 RAM 페이지가 할당돼. 이걸 **page fault**라고 해.

!!! note "Page Fault 과정"

    1. 프로세스가 가상 주소 0x1234에 데이터 쓰려고 함
    2. MMU가 확인: "이 주소에 매핑된 물리 페이지 없는데?"
    3. Page Fault 발생! --> OS 커널에 알림
    4. OS: "OK, 물리 RAM에서 빈 페이지 하나 찾아서 매핑해줄게"
    5. 가상 주소 0x1234 <--> 물리 RAM 페이지 연결 완료
    6. 프로세스: 데이터 쓰기 성공

    --> 이 페이지가 이제 "resident" 상태가 된 거야

그래서 resident는 "프로세스가 실제로 손 댄 페이지의 합계"야.

### 2-4. 왜 committed > resident 인가?

이게 이 챕터의 핵심이야. 잘 봐.

!!! danger "왜 committed > resident 인가?"

    ```
    committed = 12GB (예약한 전체 크기)
    [████████████████████████████████░░░░░░░░░░░░░░░░░░░░░░]
     <---- resident = 8.4GB ------><-- 안 쓴 부분 3.6GB -->

     ████ = 물리 RAM에 올라가 있는 페이지 (resident)
     ░░░░ = 예약만 하고 아직 안 건드린 영역
    ```

    그리고 resident 안에서도:

    ```
    [████████████████████░░░░░░░░░░░░░]
     <-- 살아있는 객체 --><-- 쓰레기 -->
    ```

    GC가 정리하는 건 여기 안의 쓰레기!
    OS가 보는 resident 크기는 안 줄어!

**"방 10개 예약했지만 5개만 짐 풀었다"**

- 예약(committed): 10개 방 = 12GB
- 문 연 방(resident): 5개 방 = 8.4GB
- 실제 손님 있는 방: 3개 = 살아있는 객체
- 짐만 놓고 나간 방: 2개 = GC 대상 쓰레기 객체

### 2-5. 한 번 resident 된 페이지는 왜 안 내려가?

이거 진짜 중요해. 많이들 오해하는 부분이야.

"GC가 쓰레기 객체 치우면 메모리가 줄어들어야 하는 거 아니야?" 이렇게 생각하지? **틀렸어.**

resident가 안 줄어드는 이유 3가지:

!!! warning "resident가 안 줄어드는 이유"

    **이유 1: 프로세스가 반납 안 함**

    JVM이 GC로 쓰레기를 치워도, 그 메모리 영역을 OS한테 돌려주지 않아.
    왜? "나중에 또 쓸 거니까 갖고 있을게."
    호텔 비유: 방 비워도 체크아웃 안 함. "나중에 또 쓸 수도 있잖아."

    **이유 2: OS가 여유 있으면 굳이 안 뺏음**

    OS 입장에서 물리 RAM이 넉넉하면, 프로세스한테 "그 페이지 내놔"
    할 이유가 없어. 그냥 놔둬. 어차피 여유 있으니까.
    OS의 원칙: "굳이 건드릴 필요 없으면 안 건드린다."

    **이유 3: OS available 부족하면 그때 회수**

    다른 프로세스들이 메모리를 많이 요구해서 available이 부족해지면?
    OS가 최근에 안 쓴 페이지를 회수하거나 swap으로 보냄.
    이건 GC가 아니야! OS 레벨의 페이지 회수야. 완전 다른 메커니즘.

정리하면:

| 주체 | 하는 일 | 레벨 |
|------|---------|------|
| GC | JVM 힙 안에서 쓰레기 객체 정리 | JVM 내부 |
| OS 페이지 회수 | 물리 RAM 페이지를 swap으로 보내거나 회수 | OS 레벨 |

GC가 돌아도 OS가 보는 resident는 안 줄어들어. 이 두 개는 완전히 다른 세계의 일이야.

### 2-6. 전체 흐름 한 눈에 보기

!!! example "호텔 비유 전체 흐름 (프로세스 메모리)"

    ```mermaid
    flowchart TD
        A["[1] 예약 (committed)<br/>'방 10개 예약합니다'<br/>OS: '가상 주소 공간 할당'<br/>실제 RAM 사용: 0"]
        B["[2] 체크인 (resident)<br/>방 1~5에 짐 풂 = 물리 RAM 페이지 할당 (page fault)<br/>방 6~10은 빈 방 (아직 안 건드림)"]
        C["[3] 방 안 상태 (힙 내부)<br/>방 1,2,4: 손님 있음 (살아있는 객체)<br/>방 3,5: 짐만 놓고 나감 (쓰레기 객체)"]
        D["[4] GC 실행<br/>방 3,5의 쓰레기 치움 = 방 안은 깨끗해짐<br/>근데 체크아웃 안 함! = resident 안 줄어듦!"]
        E["[5] OS 입장<br/>RAM 여유 있음 = '뭐 그냥 놔둬'<br/>RAM 부족해짐 = '방 3,5 안 쓰잖아, 회수한다' (swap out)"]
        A --> B --> C --> D --> E
    ```

## 3. 코드/명령어로 보자

### 3-1. ps로 프로세스 메모리 확인

```bash
# ps -eo: 모든 프로세스를 지정한 포맷으로 출력
# pid: 프로세스 ID
# vsz: Virtual Memory Size (가상 메모리 = committed 포함, 단위: KB)
# rss: Resident Set Size (물리 RAM에 올라간 양, 단위: KB)
# comm: 프로세스 이름 (실행 파일명)
ps -eo pid,vsz,rss,comm | head -20
```

출력 예시:
```
  PID    VSZ   RSS COMMAND
    1   4500  3200 systemd
  123 15728640 8847360 java         ← 우리 톰캣 JVM
  456  125000  45000 nginx
```

여기서 java 프로세스를 보자:

!!! tip "VSZ vs RSS 해석"

    | 항목 | 값 | 의미 |
    |------|-----|------|
    | **VSZ** | 15,728,640 KB = 약 15GB | 가상 메모리 전체 크기. committed + 공유 라이브러리 등 포함. "이만큼 쓸 수 있게 해줘"라고 예약한 전체 크기 |
    | **RSS** | 8,847,360 KB = 약 8.4GB | 실제로 물리 RAM에 올라가 있는 양. "진짜로 손 댄 페이지" 합계 |
    | **차이** | 15GB - 8.4GB = 약 6.6GB | 예약만 하고 아직 안 건드린 영역 |

### 3-2. top에서 메모리 컬럼 읽기

```bash
# top 실행 후 보이는 프로세스 목록의 메모리 관련 컬럼
top
```

```
  PID USER      PR  NI    VIRT    RES    SHR S  %MEM
  123 tomcat    20   0   15.0g   8.4g  12345 S  55.0
```

| 컬럼 | 의미 | 세부 설명 |
|------|------|-----------|
| **VIRT** | Virtual Memory | ps의 VSZ와 같은 것. 가상 메모리 전체. |
| **RES** | Resident Memory | ps의 RSS와 같은 것. 물리 RAM에 올라간 양. |
| **SHR** | Shared Memory | 다른 프로세스와 공유하는 메모리 (라이브러리 등). |
| **%MEM** | RES / 전체 RAM | 물리 RAM 중 이 프로세스가 차지하는 비율. |

%MEM 계산 방법:
```
%MEM = (RES / 전체 물리 RAM) * 100
     = (8.4GB / 15.3GB) * 100
     = 약 55%

이 55%가 모니터링에서 "메모리 사용률 55%"로 보이는 숫자야!
```

### 3-3. VSZ와 RSS의 차이를 직접 확인

```bash
# 특정 프로세스(톰캣 JVM)의 VSZ, RSS 실시간 비교
# -p: 특정 PID만 필터
ps -p $(pgrep -f 'catalina') -o pid,vsz,rss,pcpu,pmem --no-headers
```

```bash
# /proc/[PID]/status에서 더 상세한 메모리 정보 확인
cat /proc/$(pgrep -f 'catalina')/status | grep -E "VmSize|VmRSS|VmSwap"
```

출력 예시:
```
VmSize:  15728640 kB    ← 가상 메모리 전체 (= VSZ)
VmRSS:    8847360 kB    ← 물리 RAM 사용량 (= RSS)
VmSwap:         0 kB    ← swap으로 밀린 양 (0이면 정상)
```

### 3-4. 메모리 매핑 상세 확인 (pmap)

```bash
# pmap: 프로세스의 메모리 매핑을 상세히 보여주는 도구
# -x: 확장 포맷 (RSS, Dirty 등 추가 정보)
pmap -x $(pgrep -f 'catalina') | tail -5
```

출력 예시:
```
Address           Kbytes     RSS   Dirty Mode  Mapping
...
total kB       15728640 8847360 8200000
```

- **Kbytes**: 가상 메모리 크기 (매핑된 전체)
- **RSS**: 물리 RAM에 올라간 양
- **Dirty**: 수정된 페이지 (디스크 원본과 다른 것)

## 4. 주의사항 / 함정

!!! danger "함정 1: VSZ가 크니까 메모리 부족이야!"

    **틀린 판단:** "VSZ가 15GB? 서버 RAM이 16GB인데 거의 다 쓰잖아!"

    **맞는 판단:** VSZ는 가상 메모리야. 실제 RAM 사용량이 아니야.
    RSS를 봐야 실제 물리 RAM 사용량을 알 수 있어.
    VSZ가 물리 RAM보다 큰 것도 가능해. 그게 가상 메모리의 존재 이유야.

!!! danger "함정 2: RSS가 안 줄어드니까 메모리 누수야!"

    **틀린 판단:** "GC 돌았는데 RSS가 안 줄어들어! 누수다!"

    **맞는 판단:** GC는 JVM 힙 내부에서 쓰레기를 치우는 거야.
    OS한테 메모리를 돌려주는 게 아니야.
    RSS가 안 줄어드는 건 정상이야.
    메모리 누수를 판단하려면 JVM 힙 내부의 살아있는 객체 크기를 봐야 해.
    jstat이나 힙덤프로 확인해야지 RSS로 판단하면 안 돼.

!!! danger "함정 3: committed = 실제 사용량이라고 착각"

    **틀린 이해:** "JVM이 12GB committed 했으니까 12GB 쓰고 있는 거야"

    **맞는 이해:** committed = "쓸 수 있는 권리"를 확보한 것.
    실제 사용량(resident) = "진짜 물리 RAM에 올라간 양".
    committed > resident가 정상이야.

    비유: 신용카드 한도 1000만원 = committed /
    이번 달 실제 사용액 300만원 = resident /
    한도가 1000만원이라고 1000만원 쓴 건 아니지?

!!! danger "함정 4: RSS = 내 프로세스만의 메모리라고 착각"

    **틀린 이해:** "RSS 8.4GB면 이 프로세스가 순수하게 8.4GB 쓰는 거지"

    **맞는 이해:** RSS에는 공유 메모리(SHR)가 포함돼 있어.
    예를 들어 libc.so 같은 공유 라이브러리는
    여러 프로세스가 같은 물리 페이지를 공유해.
    순수하게 이 프로세스만의 메모리를 알려면
    PSS(Proportional Set Size)를 봐야 해.
    근데 JVM의 경우 힙이 대부분이라 RSS ≈ 실제 사용량에 가까워.

!!! danger "함정 5: resident가 계속 올라가는 게 항상 비정상?"

    **틀린 판단:** "RSS가 3GB에서 8GB로 올라갔어! 뭔가 잘못된 거 아니야?"

    **맞는 판단:** JVM은 힙을 점진적으로 확장해. -Xms < -Xmx면
    시작할 때 -Xms만큼만 쓰다가 필요하면 늘려.
    GC 후에도 살아있는 객체가 많으면 힙을 키워.
    힙이 커지면 새 페이지에 접근 --> page fault --> RSS 증가.
    이건 JVM의 정상 동작이야. (06장에서 자세히 다룸)

## 5. 정리

### 핵심 요약 표

| 개념 | 의미 | 비유 | ps/top 컬럼 |
|------|------|------|-------------|
| **committed** | OS한테 확보한 가상 메모리 | 호텔 방 예약 | VSZ / VIRT |
| **resident (RSS)** | 실제 물리 RAM에 올라간 양 | 짐 푼 방 | RSS / RES |
| **사용 중 데이터** | 살아있는 객체 | 손님 있는 방 | jstat으로 확인 |
| **쓰레기 데이터** | GC 대상 죽은 객체 | 짐만 놓고 나간 방 | jstat으로 확인 |

### 크기 관계

```
항상 이 관계가 성립해:

  committed >= resident >= 살아있는 객체 크기

  예약한 양 >= 실제 RAM에 올린 양 >= 진짜 쓰고 있는 양
```

### 누가 뭘 관리하는가

| 주체 | 관리 대상 | 하는 일 |
|------|-----------|---------|
| **OS** | 물리 RAM 페이지 | 할당, 회수, swap 관리 |
| **JVM GC** | 힙 내부 객체 | 쓰레기 식별, 정리, 힙 공간 재사용 |
| **개발자** | 객체 생명주기 | 불필요한 참조 끊기, 누수 방지 |

### 한 줄 정리

> **committed는 예약, resident는 실사용, GC가 치워도 resident는 안 줄어든다. 이 세 가지 구분 못 하면 모니터링 수치 보는 의미가 없다.**

### 확인 문제 (5문제)

> 다음 문제를 풀어봐. 답 못 하면 위에서 다시 읽어.

**Q1.** 프로세스의 VSZ가 15GB이고 RSS가 8GB다. 서버 물리 RAM은 16GB다. 이 상황이 비정상인가?

**Q2.** JVM에서 GC가 돌아서 힙 사용량이 6GB에서 2GB로 줄었다. 이때 OS가 보는 RSS는 어떻게 변하는가?

**Q3.** committed와 resident의 차이를 호텔 비유를 쓰지 않고, 기술적으로 한 문장으로 설명해봐.

**Q4.** 프로세스의 RSS가 계속 증가하고 있다. 이것만으로 메모리 누수라고 판단할 수 있는가? 이유를 말해봐.

**Q5.** 다음 중 물리 RAM 사용량을 나타내는 것은?
- (A) VSZ
- (B) VIRT
- (C) RSS
- (D) committed

??? success "정답 보기"
    **A1.** 비정상이 아니다. VSZ는 가상 메모리 크기라서 물리 RAM보다 클 수 있다. 실제 물리 RAM 사용량은 RSS(8GB)이고, 이건 16GB 중 50%니까 정상 범위다. VSZ가 RAM보다 크다고 문제가 되는 게 아니다.

    **A2.** 거의 변하지 않는다. GC는 JVM 힙 내부에서 죽은 객체를 정리할 뿐, OS한테 물리 페이지를 반납하지 않는다. JVM은 그 공간을 나중에 재사용하기 위해 계속 들고 있다. 그래서 OS가 보는 RSS는 GC 전후로 거의 동일하다.

    **A3.** committed는 OS로부터 할당받은 가상 주소 공간의 크기이고, resident는 그 중에서 실제로 물리 RAM 페이지가 매핑되어 올라가 있는 부분의 크기다.

    **A4.** RSS 증가만으로는 메모리 누수라고 판단할 수 없다. JVM이 -Xms < -Xmx인 경우 힙을 점진적으로 확장하면서 RSS가 정상적으로 증가할 수 있다. 메모리 누수를 판단하려면 JVM 힙 내부에서 살아있는 객체의 크기가 GC 후에도 계속 증가하는지를 jstat이나 힙덤프로 확인해야 한다.

    **A5.** 정답: **(C) RSS**. VSZ(A)와 VIRT(B)는 가상 메모리 크기이고, committed(D)는 OS한테 확보한 가상 메모리 양이다. 실제 물리 RAM에 올라가 있는 양은 RSS(= RES)다.
