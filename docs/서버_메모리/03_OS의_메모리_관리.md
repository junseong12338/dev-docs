# 03. OS의 메모리 관리 - Beta

---

## 1. OS의 메모리 관리 철학 - "이게 뭐야?"

### 비유로 문 열기

냉장고를 생각해봐.

냉장고에 빈 공간이 많으면 효율적인 거야? 아니야. 냉장고 절반이 비어있으면 전기만 낭비하는 거잖아. 어차피 돌아가는 냉장고인데, 빈 공간 놔두는 것보다 식재료를 미리 넣어두는 게 나아.

리눅스도 똑같은 생각을 해.

!!! danger "리눅스의 메모리 관리 철학"

    **"남는 메모리는 낭비다."**

    비어있는 RAM은 아무 일도 안 하는 RAM이야.
    차라리 디스크 읽기 결과를 캐시해두면 나중에 또 읽을 때 디스크 안 가도 되잖아.
    그래서 남는 RAM은 전부 캐시로 채운다.

비유는 여기까지. 이 철학이 왜 중요하냐면, 이걸 모르면 `free -h` 결과를 보고 "free가 1GB밖에 없어! 위험해!"라고 오판하거든. 리눅스는 **의도적으로** 남는 메모리를 캐시로 채우기 때문에 free가 낮은 건 **정상**이야.

이게 이 챕터의 핵심 중 핵심이야. 이것만 제대로 이해해도 이 챕터 읽은 보람 있어.

## 2. 가상 메모리 - "프로세스마다 독립된 세계"

### 비유

아파트를 생각해봐. 101호, 201호, 301호 각각 "우리 집은 방 3개에 거실 1개"라고 말해. 전부 같은 구조를 가진 것 같지만, 실제로 건물에서 차지하는 물리적 위치는 다 다르잖아. 각 집은 자기 집 주소 체계(방1, 방2, 방3)를 쓰고, 건물 전체의 물리적 위치는 관리사무소(OS)가 알아서 관리해.

### 본질

**가상 메모리 = 각 프로세스에게 "너만의 독립된 메모리 주소 공간"을 주는 기술.**

!!! note "가상 메모리 구조"

    ```mermaid
    flowchart TD
        A["프로세스 A (톰캣/java)<br/>가상 주소: 0x0000~0xFFFF"] --> OS
        B["프로세스 B (MySQL)<br/>가상 주소: 0x0000~0xFFFF"] --> OS
        C["프로세스 C (nginx)<br/>가상 주소: 0x0000~0xFFFF"] --> OS
        OS["OS (메모리 관리자)<br/>가상 주소 → 물리 주소 변환 (매핑)"] --> RAM["물리 메모리 (RAM 16GB)<br/>A, A, B, C, A, B, B, C, ...<br/>실제로는 뒤섞여서 저장됨"]
    ```

    각 프로세스는 각자 0번부터 시작하는 것처럼 보이는 독립된 가상 주소 공간을 가짐.

### 왜 이렇게 하냐?

| 이유 | 설명 |
|------|------|
| **격리** | 프로세스 A가 프로세스 B의 메모리를 건드릴 수 없어. 톰캣이 MySQL 메모리를 못 건드려. 보안과 안정성. |
| **단순화** | 각 프로세스는 "나만의 0번부터 시작하는 메모리"가 있다고 생각하면 돼. 다른 프로세스 신경 안 써도 돼. |
| **효율** | 실제로 사용하는 만큼만 물리 메모리에 매핑. 4GB 예약했어도 1GB만 쓰면 물리 RAM은 1GB만 차지. |

이 세 번째 이유가 나중에 04장(committed vs resident)에서 중요해져. 지금은 "프로세스마다 독립 메모리, OS가 물리 매핑 관리" 이것만 기억해.

## 3. 페이지 - "메모리를 블록 단위로 관리"

### 비유

창고에 물건을 보관한다고 생각해봐. 물건마다 크기가 다르면 어디 넣을지 매번 고민해야 해. 근데 **규격 박스**를 쓰면? 모든 물건을 같은 크기 박스에 담고, 박스 단위로 선반에 정리하면 관리가 쉽잖아.

### 본질

**페이지(Page) = OS가 메모리를 관리하는 고정 크기 블록. 리눅스에서 기본 4KB.**

!!! note "페이지 기반 메모리 관리"

    | 가상 메모리 (프로세스 기준) | 매핑 | 물리 메모리 (RAM) |
    |---------------------------|------|-------------------|
    | 페이지 0 (4KB) | --> 매핑 | 프레임 3 |
    | 페이지 1 (4KB) | --> 매핑 | 프레임 7 |
    | 페이지 2 (4KB) | 아직 미매핑 | - |
    | 페이지 3 (4KB) | --> 매핑 | 프레임 5 |

    "페이지 2"는 아직 물리 메모리에 매핑 안 됨 -> 실제로 접근하면 그때서야 매핑 (Demand Paging)

### 왜 4KB 블록이야? 왜 바이트 단위가 아니야?

| 방식 | 문제 |
|------|------|
| 바이트 단위로 관리 | 매핑 테이블이 어마어마하게 커져. 16GB RAM이면 160억 개 항목 관리. 불가능. |
| 4KB 블록 단위로 관리 | 16GB / 4KB = 약 400만 개 항목. 관리 가능. |

4KB가 절대적인 건 아니야. 다른 크기도 있어 (Huge Pages = 2MB, 1GB). 근데 범용적으로 4KB가 가장 균형 잡힌 크기라서 리눅스 기본값이 4KB인 거야.

### 이게 왜 중요해?

나중에 `free -h`의 `buff/cache` 이해할 때, 그리고 Swap 이해할 때 "페이지 단위로 움직인다"는 걸 알아야 제대로 이해돼. OS가 메모리를 회수하거나 디스크로 내보낼 때 전부 **페이지 단위(4KB 블록 단위)**로 처리하거든.

## 4. free -h 출력 완전 해석 - "코드로 보자"

이 챕터의 핵심. 한 필드도 빠짐없이 간다.

### free -h가 뭐야?

`free` = 메모리 사용 현황을 보여주는 명령어
`-h` = human-readable. 사람이 읽기 쉬운 단위(Gi, Mi)로 표시

### 우리 서버 실제 출력

```bash
$ free -h
              total        used        free      shared  buff/cache   available
Mem:           15Gi       8.4Gi       1.0Gi       0.0Ki       6.2Gi       6.9Gi
Swap:         4.0Gi       145Mi       3.9Gi
```

이 두 줄에 서버 메모리의 모든 게 담겨있어. 하나씩 뜯어보자.

---

### 4.1 Mem 행 - 물리 메모리(RAM)

#### total: 15Gi

!!! note "total = 이 서버에 물리적으로 설치된 RAM 전체 용량"

    **15Gi** = 약 15 GiB = 약 16.1 GB (10진수 기준)

    이건 하드웨어로 결정되는 값이야. 서버에 RAM 16GB짜리를 꽂았으면 여기에 약 15Gi로 나와. (OS 커널이 일부를 미리 예약해서 정확히 16이 안 나와)

    이 값은 변하지 않아. 서버 전원 켜고 끄든 항상 같아. 바꾸려면? RAM을 물리적으로 추가/교체해야 해.

#### used: 8.4Gi

!!! note "used = 프로세스들이 실제로 사용 중인 메모리"

    **8.4Gi** = 약 8.4 GiB

    여기에 뭐가 포함되냐:

    - JVM(톰캣/Spring)이 사용 중인 메모리
    - OS 커널이 사용 중인 메모리
    - MySQL, 기타 프로세스가 사용 중인 메모리
    - buff/cache에 포함되지 않은 프로세스 전용 메모리

    계산: `used = total - free - buff/cache` -> `8.4 = 15 - 1.0 - 6.2 ≈ 7.8` (약간 반올림 차이)

    주의: used가 높다고 무조건 위험한 게 아니야. 프로세스가 정상적으로 사용 중인 거면 괜찮아. 문제는 used가 계속 올라가서 available이 바닥날 때야.

#### free: 1.0Gi

!!! warning "free = 아무도 안 쓰는, 완전히 비어있는 메모리"

    **1.0Gi** = 약 1 GiB

    "뭐? 15GB 중에 1GB만 남았어? 위험한 거 아니야?"

    **아니야. 이건 정상이야.** 리눅스는 남는 메모리를 놀리지 않아. 전부 캐시(buff/cache)로 활용해. 그래서 free가 0에 가까운 건 "리눅스가 일을 잘 하고 있다"는 뜻이야.

    free가 높다는 건? 서버가 방금 시작됐거나, 아직 캐시가 덜 쌓인 상태.

    **진짜 여유분은 free가 아니라 available을 봐야 해.**

#### shared: 0.0Ki

!!! note "shared = 여러 프로세스가 공유하는 메모리"

    **0.0Ki** = 거의 없음

    해당되는 것: tmpfs (메모리 기반 임시 파일시스템), 프로세스 간 공유 메모리 (shared memory, shmem)

    우리 서버에서는 거의 안 쓰고 있어서 0에 가까워. 이 값은 대부분의 경우 크게 신경 안 써도 돼.

#### buff/cache: 6.2Gi

!!! tip "buff/cache = 버퍼(Buffer) + 페이지 캐시(Page Cache)"

    **6.2Gi** = 약 6.2 GiB. 이게 이 챕터에서 가장 중요한 필드야.

    | 구성요소 | 설명 |
    |----------|------|
    | **Buffer (버퍼)** | 디스크의 메타데이터(파일 정보, 디렉토리 등)를 캐시해둔 것. 크기가 작음 (보통 수십~수백 MB) |
    | **Page Cache (페이지 캐시)** | 디스크에서 읽은 파일 내용을 RAM에 캐시. buff/cache의 대부분은 이거야. 같은 파일 다시 읽을 때 디스크 안 가도 됨 |

    **핵심:** 이건 프로세스가 "내 꺼!"라고 잡고 있는 메모리가 아니야. OS가 "어차피 남으니까 캐시해두자"라고 쓰는 메모리야.

    프로세스가 메모리를 더 달라고 하면? OS가 캐시를 버리고 그 공간을 프로세스에게 줘. 그래서 이건 **"반환 가능한" 메모리**야.

이게 제대로 이해돼야 해. 그림으로 다시 보자:

!!! abstract "RAM 15Gi의 실제 구성"

    ```
    total = 15Gi
    ┌──────────────┬─────────────────────┬──────┐
    │  used 8.4Gi  │  buff/cache 6.2Gi   │ free │
    │  (프로세스   │  (반환 가능!        │ 1.0Gi│
    │   전용)      │   프로세스가        │      │
    │              │   달라하면 줌)      │      │
    └──────────────┴─────────────────────┴──────┘
    ◄────── available = 6.9Gi (free + 반환 가능 cache) ──────►
    ```

#### available: 6.9Gi

!!! danger "available = 새 프로세스에게 실제로 줄 수 있는 메모리"

    **6.9Gi** = 약 6.9 GiB

    **이게 진짜 중요한 숫자야.**

    - `available ≈ free + 반환 가능한 buff/cache`
    - `6.9 ≈ 1.0 + 5.9 (cache 중 반환 가능 분)`
    - **"메모리 여유가 얼마나 있어?" 에 대한 답 = free가 아니라 available이야.**

    free를 보고 "1GB밖에 안 남았어!" 라고 당황하면? 틀려. available이 6.9Gi나 돼. 여유 충분해.

    - available이 1Gi 이하로 떨어지면? 그때서야 진짜 주의해야 해.
    - available = 0에 가까워지면? Swap 사용 시작 -> 성능 급락 -> 장애 가능

### 4.2 Swap 행 - 디스크를 RAM처럼

```bash
Swap:         4.0Gi       145Mi       3.9Gi
              total        used        free
```

#### Swap total: 4.0Gi

!!! note "Swap total = Swap 영역으로 설정된 디스크 공간"

    **4.0Gi** = 약 4 GiB

    Swap이 뭐냐: RAM이 부족할 때, 디스크의 일부를 RAM처럼 쓰는 공간. 물리 RAM 16GB + Swap 4GB = 총 20GB "처럼" 사용 가능. 근데! 디스크는 RAM보다 1,000~100,000배 느려!

    | 종류 | 속도 |
    |------|------|
    | RAM (빠름) | 1초에 작업 완료 |
    | Swap (느림) | 같은 작업에 수분 |

    Swap = "응급 예비 탱크". 쓰게 되면 살기는 하지만, 제대로 못 달린다.

#### Swap used: 145Mi

!!! warning "Swap used = 현재 실제로 Swap을 사용 중인 양"

    **145Mi** = 약 145 MiB

    Swap 사용 중이라는 건: 어느 시점에 RAM이 부족해서 일부 데이터를 디스크로 내보냈다. 지금 145MB 분량의 데이터가 디스크(Swap)에 있다.

    **145Mi면 위험한 거야?** 4Gi 중 145Mi = 약 3.5% 사용. 당장은 큰 문제 아니야. 근데 이 값이 계속 올라가면? 그건 경고 신호야. Swap이 절반(2Gi) 이상 사용되면 심각한 문제.

    **왜 위험하냐:** Swap 영역의 데이터에 접근하면 디스크 I/O 발생 -> 느려짐 -> 응답 시간 증가 -> "페이지 다 로딩되는 데 10초 걸려요" 이런 상황.

#### Swap free: 3.9Gi

!!! note "Swap free = Swap 중 사용 안 한 부분"

    **3.9Gi** = 4.0 - 0.145 ≈ 3.9. 이건 단순히 total - used야. Swap이 아직 3.9Gi 남았다 = 최악의 경우 여기까지 쓸 수 있다. 근데 Swap을 쓰는 것 자체가 이미 안 좋은 상황이야.

## 5. "free가 0인데 괜찮아?" - "어떻게 돌아가?"

이걸 이해 못 하는 사람이 정말 많아. 아주 자세히 설명할게.

### 시나리오로 이해하기

!!! example "시나리오: 서버 부팅 후 시간 경과"

    | 시점 | used | free | cache | available | 상태 |
    |------|------|------|-------|-----------|------|
    | **부팅 직후** | 2Gi | 12Gi | 1Gi | 13Gi | 서버 막 켰으니까 free가 엄청 많아. 캐시도 거의 없어. |
    | **1시간 후** | 5Gi | 6Gi | 4Gi | 10Gi | 프로세스들이 올라오고, 파일 읽기 결과가 캐시에 쌓이기 시작. |
    | **24시간 후** (우리 서버) | 8.4Gi | 1Gi | 6.2Gi | 6.9Gi | free가 1Gi밖에 안 남아! 위험한 거야? 아니. available이 6.9Gi나 돼. |
    | **정상 안정 상태** | 10Gi | 0.1Gi | 5Gi | 4.5Gi | free가 0.1Gi! 근데 괜찮아. available이 4.5Gi면 충분해. 리눅스가 남는 메모리를 전부 캐시로 쓰고 있을 뿐. |

### 핵심 공식

!!! danger "핵심 공식"

    - **X 잘못된 판단:** "여유 메모리 = free"
    - **O 올바른 판단:** "여유 메모리 = available"

    `available ≈ free + (반환 가능한 cache)`

    | available 비율 | 판단 |
    |----------------|------|
    | > total의 30% | 여유 있음 |
    | 20~30% | 주의 |
    | 10~20% | 경고 |
    | < 10% | 위험! 조치 필요 |

    우리 서버: available 6.9Gi / total 15Gi = **46%** -> 여유 있음

## 6. 페이지 캐시 (Page Cache) - "OS의 성능 비법"

### 왜 OS가 이렇게까지 캐싱을 하냐?

!!! example "페이지 캐시 동작 원리"

    **캐시 없을 때 - 매번 디스크 접근:**

    1. 프로세스: "config.xml 읽어줘" -> OS: 디스크에서 읽어옴 (느림) -> 프로세스에 전달
    2. 프로세스: "config.xml 또 읽어줘" -> OS: 디스크에서 또 읽어옴 (또 느림) -> 프로세스에 전달
    3. 프로세스: "config.xml 또또 읽어줘" -> OS: 또또 디스크... (계속 느림)

    ---

    **캐시 있을 때 - 두 번째부터 RAM에서 읽기:**

    1. 프로세스: "config.xml 읽어줘" -> OS: 디스크에서 읽어옴 (느림) + **RAM에 캐시해둠** -> 프로세스에 전달
    2. 프로세스: "config.xml 또 읽어줘" -> OS: **캐시에 있네! RAM에서 바로 읽음 (빠름!!)** -> 프로세스에 전달
    3. 프로세스: "config.xml 또또 읽어줘" -> OS: **캐시 히트! 초고속!**

    결과: 디스크 I/O 횟수 대폭 감소 -> 서버 성능 향상

### 캐시가 꽉 찼는데 프로세스가 메모리를 더 달라고 하면?

!!! tip "캐시 반환 (Reclaim) 과정"

    | 단계 | used | free | cache |
    |------|------|------|-------|
    | **Before** | 10Gi | 1Gi | 4Gi |
    | 프로세스: "2Gi 더 줘" / OS: "ㅇㅋ 캐시 2Gi 반환할게" | | | |
    | **After** | 12Gi | 1Gi | 2Gi |

    - 캐시를 줄여서 프로세스에게 메모리를 줌
    - 캐시가 줄면? 디스크 I/O가 늘어나서 좀 느려질 수 있어
    - 근데 프로세스가 죽는 것보다는 100만 배 나아

    캐시마저 반환해도 메모리가 부족하면? **Swap 사용 시작** -> 그래도 부족하면 -> **OOM Killer**

## 7. Swap 상세 - "최후의 방어선"

### Swap In / Swap Out

!!! warning "Swap 동작 메커니즘"

    ```mermaid
    flowchart LR
        A["RAM (빠름)"] -->|"Swap Out<br/>안 쓰는 거 내보내자"| B["Swap (디스크, 느림)"]
        B -->|"Swap In<br/>이거 다시 필요해!"| A
    ```

    **Swap Out** (RAM -> 디스크): RAM이 부족할 때, OS가 최근에 안 쓴 페이지(4KB 블록)를 RAM에서 꺼내서 디스크의 Swap 영역에 저장.

    **Swap In** (디스크 -> RAM): Swap에 내보낸 데이터가 다시 필요하면, 디스크에서 RAM으로 다시 가져옴.

    Swap In/Out이 빈번하면? **"Swap Thrashing (스래싱)"** -> 디스크 I/O 폭주 -> 서버 거의 멈춤. 이건 비상 상황이야.

### Swappiness

!!! note "swappiness = OS가 얼마나 적극적으로 Swap을 쓸지 결정하는 값"

    범위: 0 ~ 100

    | 값 | 의미 |
    |----|------|
    | 0 | Swap 거의 안 씀 (최대한 RAM에서 해결) |
    | 10 | 보수적. RAM이 정말 부족할 때만 Swap |
    | 60 | 기본값. 적당히 Swap 사용 (대부분 리눅스 기본) |
    | 100 | 적극적으로 Swap 사용 |

    **서버에서 권장하는 값:** 일반 서버: 10~30 / DB 서버: 1~10 / 우리 같은 WAS: 10~30

    ```bash
    $ cat /proc/sys/vm/swappiness
    60
    ```

### Swap이 사용되면 뭐가 문제냐

!!! danger "Swap 사용의 성능 영향"

    | 종류 | 접근 시간 |
    |------|-----------|
    | RAM | ~100 ns |
    | Swap (SSD) | ~100,000 ns |
    | Swap (HDD) | ~10,000,000 ns |

    차이: **1,000배 ~ 100,000배**

    **실제 영향:** 톰캣이 Swap에 있는 데이터를 읽으려 함 -> 디스크 I/O 발생 -> 쓰레드가 블로킹 -> 응답 지연 -> 다른 요청도 밀림 -> 전체적으로 느려짐

    우리 서버에서: Swap used 145Mi = 아직은 소량. 당장 문제 아님. 근데 이게 1Gi, 2Gi로 올라가면? 체감 성능 저하 시작. 3Gi 이상? 서비스 품질에 심각한 영향.

    **Swap 사용 = "RAM이 한때 부족했었다"는 증거.** 지금 RAM에 여유가 있어도 Swap used가 높으면, 과거에 메모리 압박이 있었다는 뜻이야.

## 8. 주의사항 / 함정

### 함정 1: free만 보고 "메모리 부족이다!" 판단

!!! danger "함정 1: free만 보고 판단하지 마"

    **X** "free가 1Gi밖에 안 남았어! 서버 위험해!"

    **O** available을 봐. 6.9Gi나 있잖아. 리눅스는 남는 메모리를 캐시로 쓰니까 free가 낮은 건 정상.

    **판단 순서:** 1. available 확인 (진짜 여유분) -> 2. Swap used 확인 (과거 부족 이력) -> 3. used 추이 확인 (계속 늘고 있나?)

    free만 보면 오진해. **available을 봐야 정확해.**

### 함정 2: "buff/cache가 너무 커! 뭔가 잘못된 거 아니야?"

!!! danger "함정 2: buff/cache가 크다고 비정상이 아니야"

    **X** "6.2Gi가 cache? 이거 비정상 아니야?"

    **O** 정상이야. 리눅스가 의도적으로 남는 RAM을 캐시로 채운 거야. 이 캐시 덕분에 파일 읽기가 빨라지는 거야. 프로세스가 메모리 더 필요하면 캐시를 반환하니까 걱정하지 마.

    오히려 cache가 0이면 그게 이상한 거야. "서버가 방금 켜졌나? 뭔가 캐시를 다 날린 건가?"

### 함정 3: "Swap이 있으니까 RAM이 부족해도 괜찮아"

!!! danger "함정 3: Swap은 대체재가 아니라 안전장치"

    **X** "Swap 4Gi 있으니까 RAM 부족해도 Swap이 커버해주겠지"

    **O** Swap은 "안전장치"야. "대체재"가 아니야. Swap = 응급실의 산소호흡기. 숨은 쉴 수 있지만 정상적인 생활은 불가능. Swap에 의존하는 서버 = 성능 바닥. Swap이 많이 사용되면 근본 원인(RAM 부족)을 해결해야 해.

    **근본 해결:** RAM 추가 (하드웨어) / 메모리를 많이 먹는 프로세스 원인 분석 (JVM 힙 크기 등) / 프로세스 수 줄이기, 애플리케이션 최적화

### 함정 4: used와 available의 합이 total이라는 오해

!!! danger "함정 4: used + available = total은 아니야"

    **X** "used + available = total 아니야?"

    **O** 아니야. 겹치는 부분이 있어.

    - `total = used + free + buff/cache` <-- 이건 맞아
    - `15 = 8.4 + 1.0 + 6.2 ≈ 15.6` (반올림 차이)

    `available ≈ free + (반환 가능한 cache 부분)` -- cache 중 일부는 반환 불가(Dirty Pages 등)할 수 있어서 정확히 free + cache = available은 아니야. 커널이 내부적으로 계산한 "실제 줄 수 있는 양"이야.

## 9. 전체 흐름 정리 - "큰 그림"

!!! abstract "OS 메모리 관리 전체 흐름 (한 눈에 보기)"

    **물리 RAM: 15Gi**

    | 영역 | 크기 | 상세 |
    |------|------|------|
    | **프로세스 사용 (used)** | 8.4Gi | JVM(톰캣/Spring): ~4~8Gi / OS 커널: 수백 MB / 기타 프로세스: 나머지 |

    | **free** | 1.0Gi | 완전히 비어있는 공간 |


## 10. 정리

### 핵심 요약 표

| 개념 | 한 줄 정리 |
|------|-----------|
| **OS 철학** | "남는 메모리는 낭비다" - 남는 RAM은 전부 캐시로 활용 |
| **가상 메모리** | 프로세스마다 독립된 주소 공간. OS가 물리 메모리에 매핑. |
| **페이지** | 메모리 관리 기본 단위. 리눅스 기본 4KB. |
| **total** | 물리 RAM 전체 용량. 하드웨어로 결정. |
| **used** | 프로세스가 사용 중인 메모리. |
| **free** | 완전히 비어있는 메모리. 낮아도 정상. |
| **buff/cache** | OS가 캐시로 활용 중인 메모리. 필요하면 반환 가능. |
| **available** | 진짜 여유분 = free + 반환 가능 cache. 이걸 봐야 해. |
| **Swap** | 디스크를 RAM처럼 쓰는 공간. 사용되면 경고 신호. |
| **Swap used** | 과거에 RAM이 부족했었다는 증거. |

### free -h 빠른 판단 가이드

!!! tip "free -h 빠른 판단 가이드"

    1. **available 확인** -> total의 30% 이상이면 일단 OK
    2. **Swap used 확인** -> 0이면 최고, 소량이면 주시, 많으면 경고
    3. **free가 낮아도 당황하지 마** -> 그건 캐시가 채우고 있는 거야
    4. **used가 계속 올라가는 추이면** -> 메모리 누수 의심

### 이 챕터에서 반드시 기억할 것

!!! abstract "이 챕터에서 반드시 기억할 것"
    1. free가 낮다고 위험한 거 아니야. available을 봐.
    2. buff/cache는 OS가 의도적으로 채운 캐시. 반환 가능.
    3. available = free + 반환 가능 cache = 진짜 여유분.
    4. Swap 사용 = 경고 신호. Swap은 대체재가 아니라 안전장치.
    5. "메모리 여유 있어?" 에 대한 답 = available을 total로 나눠.

### 확인 문제 (5문제)

> 다음 문제를 풀어봐. 답 못 하면 위에서 다시 읽어.

**Q1.** 아래 `free -h` 출력을 보고, 이 서버의 메모리 상태가 위험한지 판단해봐. 근거를 대.
```
              total        used        free      shared  buff/cache   available
Mem:           15Gi       8.4Gi       1.0Gi       0.0Ki       6.2Gi       6.9Gi
Swap:         4.0Gi       145Mi       3.9Gi
```

**Q2.** free가 0.1Gi이고 buff/cache가 8Gi인 서버가 있어. 이 서버 메모리 상태는 정상이야 비정상이야? available은 대략 얼마일 것 같아?

**Q3.** Swap used가 2Gi까지 올라간 서버가 있어. 이건 지금 RAM에 여유가 있어도 문제라고 봐야 하는 이유가 뭐야?

**Q4.** 리눅스가 남는 메모리를 캐시(buff/cache)로 활용하는 이유가 뭐야? 그냥 비워두면 안 돼?

**Q5.** 가상 메모리의 핵심 목적 3가지를 말해봐.

??? success "정답 보기"
    **A1.** 위험하지 않아. 근거: available이 6.9Gi로 total(15Gi)의 약 46%야. 30% 이상이면 여유 있는 상태. free가 1.0Gi밖에 안 되지만, buff/cache 6.2Gi 중 상당 부분이 반환 가능해서 available이 6.9Gi나 돼. Swap도 145Mi밖에 안 써서 당장 문제없어. 결론: 정상 운영 상태.

    **A2.** 정상이야. 리눅스가 남는 메모리를 캐시로 적극 활용하고 있는 건강한 상태야. available은 대략 free(0.1Gi) + 반환 가능 cache(대부분의 8Gi) = 약 7~8Gi 정도일 거야. free가 0에 가까운 건 리눅스가 메모리를 효율적으로 쓰고 있다는 뜻이지, 메모리가 부족하다는 뜻이 아니야.

    **A3.** Swap used가 2Gi라는 건 과거에 RAM이 심각하게 부족했었다는 증거야. 지금 RAM에 여유가 있어도, Swap에 내보낸 데이터가 아직 디스크에 있어. 그 데이터에 접근하면 디스크 I/O가 발생하고 느려져. 게다가 Swap In/Out이 반복되면(스래싱) 서버가 거의 멈출 수 있어. 근본 원인(왜 RAM이 부족했는지)을 분석해야 해.

    **A4.** 디스크 접근은 RAM보다 1,000~100,000배 느려. 남는 RAM을 비워두면 그냥 아무 일도 안 하는 낭비야. 차라리 디스크에서 읽은 파일 내용을 RAM에 캐시해두면, 같은 파일을 다시 읽을 때 디스크 안 가고 RAM에서 바로 읽을 수 있어서 엄청 빨라져. 프로세스가 메모리를 더 달라고 하면 캐시를 반환하면 되니까 손해도 없어. 리눅스의 메모리 관리 철학: "남는 메모리는 낭비다."

    **A5.** (1) 격리: 프로세스 간 메모리 접근 차단. 톰캣이 MySQL 메모리를 건드릴 수 없어 보안과 안정성 확보. (2) 단순화: 각 프로세스는 자기만의 독립 주소 공간(0번부터 시작)을 가지니까 다른 프로세스 신경 안 써도 돼. (3) 효율: 예약만 하고 실제 사용할 때만 물리 메모리에 매핑(Demand Paging)하니까, 4GB 예약해도 1GB만 쓰면 물리 RAM은 1GB만 차지.
