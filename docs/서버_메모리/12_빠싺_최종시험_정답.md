### Q1. 1KB는 몇 바이트인가? (객관식, 2점)

**정답**: **(C) 1,024 바이트**

**해설**:

| 보기 | 판정 | 이유 |
|------|------|------|
| (A) 100 바이트 | 틀림 | 10진법 착각. 컴퓨터는 2진법 기반이야. |
| (B) 1,000 바이트 | 틀림 | SI 단위(kB, 소문자 k)에서는 1,000이지만, 전통적인 컴퓨터 과학에서 1KB = 1,024. 이건 2^10이야. |
| (C) 1,024 바이트 | **정답** | 2^10 = 1,024. KB(대문자 K) = Kibibyte = 1,024 바이트. |
| (D) 1,048 바이트 | 틀림 | 아무 근거 없는 숫자. 함정. |

핵심은 **2의 거듭제곱**이야. 1KB = 2^10, 1MB = 2^20, 1GB = 2^30. 이거 외워.

-> 01장 "바이트 단위" 참조

### Q2. 메모리 계층 구조 순서 (객관식, 2점)

**정답**: **(B) 레지스터 -> 캐시 -> RAM -> 디스크**

**해설**:

| 보기 | 판정 | 이유 |
|------|------|------|
| (A) RAM -> 레지스터 -> 캐시 -> 디스크 | 틀림 | RAM이 CPU에 가장 가깝다고? 레지스터가 CPU **안에** 있어. |
| (B) 레지스터 -> 캐시 -> RAM -> 디스크 | **정답** | CPU 안(레지스터) -> CPU 옆(캐시) -> 메인보드(RAM) -> 저장장치(디스크) |
| (C) 캐시 -> 레지스터 -> RAM -> 디스크 | 틀림 | 레지스터가 캐시보다 더 가까워. 레지스터는 CPU 내부, 캐시는 CPU와 RAM 사이. |
| (D) 레지스터 -> RAM -> 캐시 -> 디스크 | 틀림 | 캐시가 RAM보다 더 멀다고? 캐시는 RAM 접근을 줄이려고 중간에 둔 거야. |

기억법: **가까울수록 빠르고 비싸고 작다**. 레지스터(가장 빠름, 수 바이트) -> 캐시(MB급) -> RAM(GB급) -> 디스크(TB급).

### Q3. PID 1인 프로세스 (객관식, 2점)

**정답**: **(B) init/systemd (최초 사용자 프로세스)**

**해설**:

| 보기 | 판정 | 이유 |
|------|------|------|
| (A) 커널 자체 | 틀림 | 커널은 프로세스가 아니야. 커널은 PID 개념 밖에 있어. 커널 스레드는 PID 2(kthreadd)부터. |
| (B) init/systemd | **정답** | 부팅 후 커널이 가장 먼저 실행하는 사용자 공간 프로세스. 모든 프로세스의 조상. |
| (C) bash 쉘 | 틀림 | 쉘은 사용자가 로그인할 때 생성되는 프로세스야. PID 1보다 한참 나중. |
| (D) Java 프로세스 | 틀림 | Java는 애플리케이션 프로세스. PID 1일 리가 없어. (Docker 컨테이너 안이면 예외적으로 가능하지만, 일반 서버에서는 아니야.) |

`ps -ef`에서 PID 1 = init 또는 systemd. 이건 **모든 프로세스의 부모**야. PPID를 추적하면 결국 1로 돌아간다.

### Q4. free -h의 available (객관식, 2점)

**정답**: **(C) 새 프로세스가 Swap 없이 사용할 수 있는 예상 메모리 (free + 회수 가능한 캐시/버퍼)**

**해설**:

| 보기 | 판정 | 이유 |
|------|------|------|
| (A) 현재 아무것도 사용하지 않는 물리 메모리 | 틀림 | 그건 `free` 컬럼이야. `available`이 아니야. |
| (B) Swap에서 사용 가능한 크기 | 틀림 | Swap 사용 가능 크기는 Swap 행의 free 컬럼이야. |
| (C) free + 회수 가능한 캐시/버퍼 | **정답** | OS는 캐시/버퍼를 필요하면 회수할 수 있어. 그래서 `available`은 `free`보다 항상 크거나 같아. |
| (D) total - used | 틀림 | 이건 대략적인 계산이지만 정확하지 않아. buff/cache가 빠져 있어. |

이게 이 시험에서 **가장 중요한 개념 중 하나**야. `free`가 0에 가까워도 `available`이 충분하면 메모리 부족이 아니야. Linux는 놀고 있는 메모리를 캐시/버퍼로 적극 활용하기 때문이야. 이걸 모르면 "free가 0이다! 메모리 부족이다!"라고 오판해.

### Q5. malloc과 물리 메모리 할당 (객관식, 2점)

**정답**: **(B) 아니다. 가상 주소 공간만 예약되고, 실제 접근 시 물리 메모리가 할당된다.**

**해설**:

| 보기 | 판정 | 이유 |
|------|------|------|
| (A) 즉시 1GB 물리 메모리 확보 | 틀림 | 이건 OS가 멍청하다고 가정하는 거야. 현대 OS는 **지연 할당(lazy allocation)** 을 해. |
| (B) 가상 주소 공간만 예약, 접근 시 할당 | **정답** | 이게 **committed vs resident**의 핵심이야. malloc은 가상 메모리를 예약(commit)하지만, 실제 물리 페이지 할당은 처음 접근하는 순간(page fault) 발생해. |
| (C) OS 설정에 따라 다르지만 일반적으로 즉시 | 틀림 | 반대야. 일반적으로 지연 할당이 기본. |
| (D) 부족하면 OOM Killer 발동 | 틀림 | malloc 시점이 아니라 실제로 물리 메모리가 고갈됐을 때 OOM Killer가 발동해. |

이게 VSZ(가상 크기)가 RSS(실제 물리 사용량)보다 큰 이유야. 예약은 많이 했지만 실제로 쓰는 건 그보다 적다.

### Q6. 힙 메모리와 관계없는 JVM 옵션 (객관식, 2점)

**정답**: **(C) `-XX:MaxMetaspaceSize=256m`**

**해설**:

| 보기 | 판정 | 이유 |
|------|------|------|
| (A) `-Xms512m` | 힙 관련 | 힙의 **초기 크기**. 시작할 때 이만큼 확보. |
| (B) `-Xmx2g` | 힙 관련 | 힙의 **최대 크기**. 여기까지 확장 가능. |
| (C) `-XX:MaxMetaspaceSize=256m` | **정답 (힙 아님)** | Metaspace는 **힙 밖**에 있어. 클래스 메타데이터를 저장하는 네이티브 메모리 영역이야. 힙이 아니야. |
| (D) `-Xmn256m` | 힙 관련 | Young Generation의 크기 설정. 힙의 일부. |

핵심: JVM 메모리 = 힙 + 힙 밖 영역(Metaspace, 스레드 스택, 네이티브 메모리 등). Metaspace는 힙이 아니야. `-Xmx`에 포함되지 않아. 이거 헷갈리면 메모리 계산 전부 틀려.

-> 05장 "JVM 옵션 해석" 참조

### Q7. 새 객체가 할당되는 영역 (객관식, 2점)

**정답**: **(C) Eden 영역**

**해설**:

| 보기 | 판정 | 이유 |
|------|------|------|
| (A) Old Generation | 틀림 | Old는 오래 살아남은 객체가 **승격(Promotion)** 되어 가는 곳이야. 새 객체가 바로 가는 곳이 아니야. (아주 큰 객체는 예외적으로 바로 Old로 갈 수 있지만, 그건 특수 케이스.) |
| (B) Survivor 영역 | 틀림 | Survivor는 Eden에서 GC를 살아남은 객체가 이동하는 곳이야. |
| (C) Eden 영역 | **정답** | 새 객체는 Eden에 생성 -> Minor GC 생존 시 Survivor로 -> 여러 번 생존하면 Old로 승격. |
| (D) Metaspace | 틀림 | Metaspace는 클래스 정보 저장소야. 객체 인스턴스가 아니야. |

객체 생명주기: `new Object()` -> **Eden** -> (Minor GC 생존) -> **Survivor** -> (여러 번 생존) -> **Old**

-> 06장 "Young/Old, Eden/Survivor" 참조

### Q8. Stop-The-World (객관식, 2점)

**정답**: **(B) GC가 실행되는 동안 해당 JVM의 모든 애플리케이션 스레드가 일시 정지된다**

**해설**:

| 보기 | 판정 | 이유 |
|------|------|------|
| (A) OS가 다른 모든 프로세스를 중지 | 틀림 | STW는 **해당 JVM 프로세스 안**의 얘기야. 다른 프로세스는 영향 없어. OS 레벨이 아니야. |
| (B) 해당 JVM의 모든 앱 스레드 일시 정지 | **정답** | GC 스레드만 동작하고, 나머지 애플리케이션 스레드는 전부 멈춰. 그래서 "Stop-The-World"야. |
| (C) 서버가 재시작 | 틀림 | GC 때문에 서버가 재시작하면 그건 장애야, GC가 아니야. |
| (D) 네트워크 요청을 못 받음 | 부분적으로 맞지만 틀림 | 직접적으로 네트워크를 차단하는 게 아니야. 앱 스레드가 멈추니까 **결과적으로** 요청 처리가 안 되는 거지, 네트워크 자체를 막는 건 아니야. |

STW 시간이 길어지면 사용자 입장에서 서비스가 멈춘 것처럼 보여. 그래서 GC 튜닝의 핵심은 **STW 시간 최소화**야.

-> 07장 "GC 원리, STW" 참조

### Q9. [함정 문제] GC 후 OS에 메모리 반환 (객관식, 2점)

**정답**: **(B) JVM 프로세스가 계속 점유하며, JVM이 나중에 재사용한다**

**해설**:

| 보기 | 판정 | 이유 |
|------|------|------|
| (A) OS에 즉시 반환 | **틀림 (함정!)** | 이게 가장 흔한 오해야. GC는 **JVM 내부**에서 객체를 해제하는 거지, OS에게 메모리를 돌려주는 게 아니야. JVM은 한번 확보한 힙 메모리를 보통 OS에 반환하지 않아. |
| (B) JVM이 계속 점유, 나중에 재사용 | **정답** | GC가 해제한 공간은 JVM 힙 안에서 "빈 공간"이 될 뿐이야. 새 객체 생성 시 그 빈 공간을 재사용해. OS 관점에서 프로세스 RSS는 줄어들지 않거나 아주 조금만 줄어. |
| (C) Swap으로 이동 | 틀림 | GC와 Swap은 관계없어. Swap은 OS가 물리 메모리 부족할 때 하는 거야. |
| (D) 디스크에 기록 후 해제 | 틀림 | GC는 디스크와 관계없어. 힙은 메모리에 있어. |

**이게 08장의 핵심이야.** GC(JVM 레벨)와 OS 페이지 관리(OS 레벨)는 **다른 레벨의 작업**이야. GC가 메모리를 "해제"해도 OS 관점에서 그 프로세스의 메모리 사용량은 거의 안 줄어. 이걸 모르면 "GC 했는데 왜 메모리 안 줄어요?"라고 당황하게 돼.

-> 08장 "GC != OS 페이지 회수" 참조

### Q10. 기준선 상승 패턴 (객관식, 2점)

**정답**: **(B) 메모리 누수(Memory Leak)**

**해설**:

| 보기 | 판정 | 이유 |
|------|------|------|
| (A) 정상적인 캐시 증가 | 틀림 | 캐시 증가라면 일정 수준에서 안정화돼야 해. **기준선이 계속 올라가면** 캐시가 아니야. |
| (B) 메모리 누수 | **정답** | GC 후에도 기준선이 올라간다 = GC가 회수하지 못하는 객체가 계속 쌓인다 = 누군가 참조를 놓지 않고 있다 = **누수**. |
| (C) 트래픽 증가에 의한 정상 동작 | 틀림 | 트래픽 증가면 GC 빈도가 늘지, GC 후 기준선이 올라가진 않아. GC 후에는 다시 내려와야 정상. |
| (D) GC 알고리즘 문제 | 틀림 | GC 알고리즘이 맞지 않으면 STW 시간이 길어지거나 비효율적일 수 있지만, 기준선 상승과는 직접 관련 없어. |

모니터링 판단의 핵심: **톱니 패턴(올라갔다 내려갔다)은 정상**, **기준선 상승(GC 후에도 점점 높아짐)은 누수 의심**.

-> 09장 "정상 vs 이상 판단" 참조

### Q11. RAM 풀네임과 핵심 특성 (단답형, 3점)

**정답**:
- 풀네임: **Random Access Memory** (1.5점)
- 핵심 특성: **휘발성(Volatile) - 전원이 꺼지면 저장된 데이터가 사라진다** (1.5점)

**해설**:

"Random Access"가 핵심이야. 어떤 위치든 동일한 시간에 접근 가능하다는 뜻이야. 디스크는 순차 접근이 빠르고 랜덤 접근이 느리지만, RAM은 어디를 읽든 속도가 같아.

그리고 **휘발성**. 전원 꺼지면 끝이야. 그래서 영구 저장이 필요한 데이터는 디스크에 써야 해. 서버가 재부팅되면 RAM에 있던 모든 프로세스 데이터는 사라진다.

**채점 기준**:
- "Random Access Memory" 정확히 쓰면 1.5점
- 휘발성(또는 전원 꺼지면 데이터 소멸) 언급하면 1.5점
- "Read And Memory" 같은 오답은 0점

-> 01장 "RAM이란" 참조

### Q12. PPID의 의미 (단답형, 3점)

**정답**: **Parent Process ID. 해당 프로세스를 생성(fork)한 부모 프로세스의 PID.**

**해설**:

모든 프로세스는 부모가 있어(PID 1 제외). 프로세스 A가 `fork()`로 프로세스 B를 만들면, B의 PPID는 A의 PID야.

예: Tomcat(PID 12345)이 있고 PPID가 1이면, systemd가 직접 Tomcat을 실행했다는 뜻이야. PPID를 추적하면 프로세스의 실행 계보를 알 수 있어.

**채점 기준**:
- "부모 프로세스의 PID" 의미 정확히 쓰면 3점
- "부모 프로세스" 까지만 쓰고 PID 언급 안 하면 2점
- "이전 프로세스" 같은 모호한 답은 1점

-> 02장 "프로세스, PID, ps -ef 해석" 참조

### Q13. free가 0이어도 메모리 부족 아닌 이유 (단답형, 3점)

**정답**: **Linux는 사용하지 않는 물리 메모리를 캐시/버퍼로 적극 활용하기 때문에 free가 0에 가까워도, 필요 시 캐시/버퍼를 회수하여 사용할 수 있다. 실제 사용 가능 메모리는 `available` 컬럼으로 봐야 한다.**

**해설**:

Linux의 메모리 관리 철학: **"놀고 있는 메모리는 낭비다."** 그래서 빈 메모리를 디스크 캐시(buff/cache)로 채워. 파일 접근 속도를 높이려고.

새 프로세스가 메모리를 요청하면? OS가 캐시 중 일부를 회수해서 줘. 그래서 `free`가 0이어도 `available`이 충분하면 정상이야.

**채점 기준**:
- 캐시/버퍼 회수 가능성 언급하면 2점
- available 컬럼 언급하면 1점 추가
- "free가 0이면 부족한 거 아니에요?" 같은 답은 0점

-> 03장 "free -h 해석, available" 참조

### Q14. VSZ와 RSS의 차이 (단답형, 3점)

**정답**: **VSZ(Virtual Size)는 프로세스가 예약한 가상 메모리 전체 크기이고, RSS(Resident Set Size)는 실제로 물리 메모리에 올라와 있는 크기이다.**

**해설**:

- **VSZ**: "이만큼 쓸 거야"라고 예약한 양. 실제로 다 쓰고 있는 건 아니야. committed 메모리.
- **RSS**: "지금 실제로 물리 RAM에 올라와 있는 양". resident 메모리.

VSZ >= RSS 항상 성립. 차이가 크면 "예약은 많이 했지만 실제로 쓰는 건 적다"는 뜻이야.

**채점 기준**:
- VSZ = 가상 메모리/예약 크기 의미 정확히 쓰면 1.5점
- RSS = 실제 물리 메모리 사용량 의미 정확히 쓰면 1.5점
- 하나만 맞으면 해당 점수만

### Q15. -Xms != -Xmx일 때 발생하는 현상 (단답형, 3점)

**정답**: **JVM이 힙 메모리를 -Xms 크기로 시작한 후, 필요에 따라 -Xmx까지 동적으로 확장(resize)하게 되며, 이 확장 과정에서 성능 오버헤드가 발생한다.**

**해설**:

`-Xms512m -Xmx2g`이면:
1. JVM 시작 시 힙 = 512MB
2. 객체가 많이 생성되면 힙이 부족해짐
3. JVM이 OS에게 추가 메모리 요청 (힙 확장, resize)
4. 확장할 때마다 비용 발생 (메모리 할당 + 기존 데이터 이동 가능성 + GC 트리거 가능)
5. 최대 2GB까지 확장 가능

문제는 **힙 확장이 공짜가 아니라는 거야**. 확장할 때 GC가 발생할 수도 있고, 일시적으로 성능이 떨어져. 그래서 실무에서는 `-Xms` = `-Xmx`로 맞추는 게 권장돼.

**채점 기준**:
- "힙 확장(resize) 발생" 언급하면 2점
- "성능 오버헤드/비용" 언급하면 1점 추가
- "에러가 난다" 같은 답은 0점 (에러가 아니라 확장이야)

### Q16. Minor GC와 Full GC 대상 영역 (단답형, 3점)

**정답**:
- **Minor GC**: Young Generation (Eden + Survivor 영역)
- **Full GC**: 힙 전체 (Young Generation + Old Generation), 경우에 따라 Metaspace 포함

**해설**:

| GC 종류 | 대상 영역 | 빈도 | STW 시간 |
|---------|-----------|------|----------|
| Minor GC | Young Generation | 자주 (초~분 단위) | 짧다 (보통 수~수십 ms) |
| Full GC | 힙 전체 (Young + Old) | 드물게 | 길다 (수백 ms ~ 초 단위) |

Minor GC는 Eden이 가득 차면 발생해. 빠르고 자주 일어나. Full GC는 Old 영역이 부족하거나 명시적 호출(`System.gc()`) 등에 의해 발생해. 느리고 STW가 길어서 서비스에 영향을 줘.

**채점 기준**:
- Minor GC 대상 정확히 쓰면 1.5점
- Full GC 대상 정확히 쓰면 1.5점
- "Minor = 작은 GC, Full = 큰 GC" 같은 답은 0점

-> 07장 "Minor/Full GC" 참조

### Q17. GC vs OS 페이지 관리 레벨 (단답형, 3점)

**정답**:
- **GC**: JVM(애플리케이션) 레벨. JVM 힙 안에서 객체 단위로 사용/해제를 관리한다.
- **OS 페이지 관리**: OS(커널) 레벨. 물리 메모리 페이지 단위로 프로세스에 할당/회수한다.

**해설**:

!!! note "GC vs OS 페이지 관리 - 레벨 비교"

    **JVM 레벨 (GC)**

    - 객체 단위 (new Object())
    - 힙 안에서 살아있는 객체 추적
    - 참조 없는 객체 해제
    - 해제해도 OS에 안 돌려줌

    **OS 레벨 (페이지 관리)**

    - 페이지 단위 (보통 4KB)
    - 물리 메모리 <-> 프로세스 매핑
    - 안 쓰는 페이지는 Swap으로 이동
    - 프로세스 종료 시 전체 회수

이 둘은 완전히 **다른 세계**야. GC가 열심히 해제해도 OS가 보기에는 "그 프로세스가 여전히 그 메모리를 쓰고 있다"로 보여.

**채점 기준**:
- GC = JVM/애플리케이션 레벨 명시하면 1.5점
- OS 페이지 = OS/커널 레벨 명시하면 1.5점
- "둘 다 메모리 관리" 같은 추상적 답은 0점

-> 08장 "GC와 OS 페이지 관리는 다른 것이다" 참조

### Q18. 정상 판단 핵심 기준 (단답형, 3점)

**정답**: 아래 3가지 중 **하나**를 정확히 쓰면 3점.

1. **톱니 패턴 (Sawtooth Pattern)**: 메모리가 올라갔다가 GC 후 내려오는 것이 반복되고, 기준선이 일정하면 정상.
2. **GC 후 기준선 안정**: GC 실행 후 메모리 사용량이 일정 수준으로 돌아오면 정상. 기준선이 계속 올라가면 이상.
3. **Full GC 빈도**: Full GC가 너무 자주(분 단위) 발생하지 않으면 정상. 빈번하면 힙 부족 또는 누수.

**해설**:

정상 판단의 핵심은 **"패턴"**이야. 한 시점의 수치가 아니라 **시간에 따른 변화**를 봐야 해.

- 톱니 패턴이면? -> 정상. 객체 생성/해제가 반복되는 건 당연한 거야.
- 기준선이 올라가면? -> 누수 의심. 해제 못 하는 객체가 쌓이고 있어.
- Full GC가 미친듯이 발생하면? -> 힙이 부족하거나 누수. 서비스에 STW 영향.

**채점 기준**:
- 위 3가지 중 하나를 정확히 쓰면 3점
- 키워드만 쓰고 설명이 부족하면 2점
- "메모리 사용률이 낮으면 정상"은 1점 (틀린 건 아닌데 핵심이 아니야)

-> 09장 "정상 vs 이상 판단, 위험 기준" 참조

### Q19. free -h 출력 해석 (코드/명령어 분석, 5점)

**정답**:

**(a)** 15GiB (약 16GB) -- (1점)

**(b)** 메모리 부족이 **아니다**. -- (2점)

근거: `free`는 312Mi로 적지만, `available`이 5.7Gi로 충분하다. Linux는 놀고 있는 메모리를 `buff/cache`(5.9Gi)로 사용하고 있고, 새 프로세스가 메모리를 요청하면 이 캐시를 회수해서 줄 수 있다. 메모리 부족 여부는 `free`가 아니라 `available`로 판단해야 한다.

**(c)** 반드시 문제는 아니다. 판단 기준: -- (2점)

- Swap 사용량이 **증가 추세**인지 확인. 1.2Gi가 고정되어 있으면 과거에 일시적으로 부족했던 흔적일 수 있고, 현재는 문제 없을 수 있다.
- Swap In/Out(si/so)이 활발한지 확인(`vmstat`). 활발하면 **현재** 물리 메모리가 부족해서 Swap을 쓰는 중이므로 문제.
- available 대비 Swap 비율 확인. available이 충분하면 현재 시점에서는 괜찮지만, Swap 사용 자체가 과거 메모리 압박의 흔적이므로 원인은 파악해야 해.

**해설**:

```
free -h 읽기 순서:
1. total → 물리 메모리 총량 (15Gi)
2. available → 실제 사용 가능량 (5.7Gi) ← 핵심!
3. free → 완전 빈 메모리 (312Mi) ← 이거만 보면 오판해
4. buff/cache → OS가 캐시로 쓰는 양 (5.9Gi) ← 회수 가능
5. Swap used → 과거 또는 현재 메모리 압박 흔적 (1.2Gi)
```

이 서버는 available이 5.7Gi로 총 메모리의 약 38%가 사용 가능해. 당장 문제 없어. 다만 Swap 1.2Gi 사용은 "언젠가 메모리가 부족했다"는 흔적이니까 원인은 찾아봐야 해.

**채점 기준**:
- (a) 15Gi 또는 약 16GB 쓰면 1점
- (b) "부족 아님" + available 근거 쓰면 2점. "부족 아님"만 쓰면 1점.
- (c) Swap 증가 추세 확인 또는 si/so 확인 언급하면 2점. "문제다/아니다"만 쓰면 0점.

### Q20. ps -ef JVM 옵션 해석 (코드/명령어 분석, 5점)

**정답**: (각 1점)

**(1)** `-Xms1g`: JVM **힙 메모리 초기 크기**를 1GB로 설정. JVM 시작 시 1GB의 힙을 확보한다.

**(2)** `-Xmx2g`: JVM **힙 메모리 최대 크기**를 2GB로 설정. 힙이 이 크기까지 확장될 수 있다. 이 이상은 OOM.

**(3)** `-XX:+UseG1GC`: **G1(Garbage-First) GC 알고리즘** 사용. 힙을 리전(Region) 단위로 나눠서 관리하며, STW 시간을 예측 가능하게 조절하는 것이 특징.

**(4)** `-XX:MaxMetaspaceSize=512m`: **Metaspace 최대 크기**를 512MB로 제한. 클래스 메타데이터 저장 영역의 상한선. 힙 밖 영역이다.

**(5)** `-Dserver.port=8080`: **JVM 시스템 프로퍼티** 설정. `server.port`라는 프로퍼티 값을 8080으로 지정. Spring Boot에서 HTTP 포트로 사용. 메모리 설정과는 관계없고, 애플리케이션 설정값이다.

**해설**:

`ps -ef`에서 java 프로세스를 보면 옵션이 줄줄이 붙어 있어. 이걸 하나하나 읽을 줄 알아야 해.

| 접두사 | 의미 |
|--------|------|
| `-Xms` | 힙 초기 크기 |
| `-Xmx` | 힙 최대 크기 |
| `-Xmn` | Young Generation 크기 |
| `-XX:+` | XX 옵션 활성화 |
| `-XX:-` | XX 옵션 비활성화 |
| `-XX:Key=Value` | XX 옵션 값 설정 |
| `-D` | 시스템 프로퍼티 설정 (메모리와 관계없는 경우 많음) |

여기서 `-Xms != -Xmx` (1g != 2g)라는 점 주목. 힙 확장이 발생할 수 있어. Q15에서 배운 내용이야.

**채점 기준**: 각 옵션당 1점. 핵심 의미가 맞으면 1점. 모호하면 0.5점.

### Q22. JVM 힙 구조 빈칸 (코드/명령어 분석, 5점)

**정답**: (각 1점)

- **(A)**: **Eden** -- 새 객체가 최초로 생성되는 영역
- **(B)**: **Survivor 0 (S0)** -- Minor GC 생존 객체가 이동하는 영역
- **(C)**: **Survivor 1 (S1)** -- Minor GC 생존 객체가 이동하는 영역 (S0과 번갈아 사용)
- **(D)**: **Old** -- 오래 살아남은 객체가 승격(Promotion)되어 저장되는 영역
- **(E)**: **Full** -- Old Generation 전체가 Full GC의 대상

**해설**:

```
Young Generation:
  Eden → 새 객체가 태어나는 곳
  Survivor 0(S0) ↔ Survivor 1(S1) → GC 생존자가 번갈아 이동

Old Generation:
  → Survivor에서 여러 번 살아남은(age threshold 초과) 객체가 승격(Promotion)
  → Full GC의 대상
```

객체 흐름: `new` -> **Eden** -> (Minor GC 생존) -> **S0 또는 S1** -> (여러 번 생존) -> **Old** -> (참조 끊기면) -> GC 대상

**채점 기준**: A~E 각 1점. 총 5점.

### Q23. RSS/VSZ/모니터링 55% 해석 (코드/명령어 분석, 5점)

**정답**:

**(a)** (2점)
VSZ 5.8GB - RSS 3.2GB = 2.6GB 차이는 **예약(committed)했지만 아직 실제 물리 메모리에 올리지 않은 가상 메모리 영역**이다. JVM이 `-Xmx4g`로 최대 4GB까지 힙을 예약할 수 있으므로, 가상 주소 공간은 크게 잡혀 있지만 실제로 물리 페이지가 할당된 건 3.2GB뿐이다.

**(b)** (1.5점)
"메모리 사용률 55%"는 **OS 전체 물리 메모리 기준**으로 계산됐을 가능성이 높다. 즉, (전체 프로세스의 used 메모리) / (total 물리 메모리) = 약 8.8GB / 16GB = 55%. 이건 JVM 하나가 아니라 서버 전체 프로세스의 메모리 사용률이다.

**(c)** (1.5점)
이 수치만으로 "메모리 부족"이라고 판단할 수 **없다**.

추가 확인 필요한 것 (하나 이상):
- **GC 로그/jstat**: Old 영역 사용률 추이. GC 후 기준선이 올라가는지.
- **`free -h`의 available**: 실제 사용 가능 메모리가 얼마인지.
- **시간 경과에 따른 추이**: 55%가 안정적인지, 계속 올라가는 중인지.

**해설**:

이 문제가 **이 시험의 핵심**이야. "55%"라는 숫자 하나만 보고 판단하면 안 돼. 알아야 할 것:

1. **55%가 뭘 기준으로 한 건지** (OS 전체? JVM 힙?)
2. **트렌드** (올라가는 중? 안정적?)
3. **GC 상태** (Old 영역은? Full GC 빈도는?)

RSS 3.2GB에 -Xmx 4GB면 아직 힙 확장 여유가 있어. 55%가 안정적이고 GC 후 기준선이 유지되면 **정상**이야.

**채점 기준**:
- (a) committed vs resident 개념으로 설명하면 2점. "모르겠다" 0점.
- (b) OS 전체 메모리 기준 언급하면 1.5점.
- (c) 추가 확인 항목 1가지 이상 쓰면 1.5점. "부족하다/아니다"만 쓰면 0점.

### Q24. LMS 서버 34% -> 55% 사례 (코드/명령어 분석, 5점)

**정답**:

**(a)** (1.5점)
34%에서 55%로의 증가가 "문제"라고 단정할 수 없는 이유: **JVM이 -Xms < -Xmx로 설정되어 있으면 힙이 확장되면서 자연스럽게 메모리 사용량이 올라가는 것일 수 있다.** 또는 애플리케이션의 정상적인 캐시/세션 증가에 의한 것일 수 있다. 수치 자체가 아니라 **패턴과 원인**이 중요하다.

**(b)** (2점, 각 1점)
확인해야 할 지표 2가지:
1. **jstat -gc로 Old 영역 사용률 추이**: GC 후 Old 사용량이 일정 수준으로 돌아오는지, 아니면 기준선이 계속 올라가는지.
2. **Full GC 빈도와 STW 시간**: Full GC가 급격히 늘었는지, STW 시간이 서비스에 영향을 주는 수준인지.

(다른 정답도 인정: free -h의 available 추이, Swap 사용 추이, 응답 시간 변화 등)

**(c)** (1.5점)
**메모리 누수(Memory Leak)**를 의심해야 한다. GC가 회수하지 못하는 객체(아직 참조가 남아있는 객체)가 계속 Old 영역에 쌓이고 있다는 뜻이다. 결국 Old가 가득 차면 Full GC가 반복되고, 그래도 공간 확보가 안 되면 OutOfMemoryError가 발생한다.

**해설**:

LMS 서버 사례에서 배운 핵심:

```
34% → 55% 자체는 숫자일 뿐이야. 중요한 건:
1. 왜 올라갔는가? (원인)
2. 올라가는 게 멈추는가? (안정화 여부)
3. GC가 제대로 동작하는가? (GC 후 회수 여부)
4. 서비스에 영향이 있는가? (응답 시간, STW)
```

**채점 기준**:
- (a) "수치만으로 판단 불가" + 이유 1개 쓰면 1.5점
- (b) 적절한 지표 2개 쓰면 2점, 1개면 1점
- (c) "메모리 누수" + OOM 가능성 언급하면 1.5점

### Q25. GC vs OS 페이지 관리 참/거짓 (코드/명령어 분석, 5점)

**정답**:

**(a)** "JVM에서 GC가 실행되면, 해제된 메모리는 OS에 즉시 반환된다." -- (2점)

**거짓.**

이유: GC는 **JVM 힙 내부**에서 객체를 해제하는 것이지, OS에게 메모리를 돌려주는 게 아니다. JVM은 한번 확보한 힙 메모리를 일반적으로 OS에 반환하지 않는다. GC 후 빈 공간은 JVM이 새 객체 생성 시 재사용한다. OS 관점에서 해당 프로세스의 RSS는 줄어들지 않거나 아주 미미하게 줄어든다.

(참고: 일부 GC 알고리즘(G1, ZGC 등)은 `-XX:MinHeapFreeRatio` 등 옵션으로 OS에 반환 가능하지만, **기본 동작이 아니고 "즉시 반환"은 틀린 표현**이다.)

**(b)** "OS는 프로세스가 사용하지 않는 물리 메모리 페이지를 자동으로 회수할 수 있다." -- (1.5점)

**참.**

OS는 메모리 압박 시 오래 접근하지 않은 페이지를 Swap으로 내보내거나, 캐시/버퍼를 회수할 수 있다. 이건 OS 레벨의 메모리 관리야. 프로세스가 명시적으로 반환하지 않아도 OS가 판단해서 회수한다.

**(c)** "GC가 하는 일과 OS의 페이지 회수가 하는 일은 동일한 레벨의 작업이다." -- (1.5점)

**거짓.**

이유: 완전히 다른 레벨이다.
- GC: **JVM(애플리케이션) 레벨**. 힙 안에서 **객체 단위**로 참조를 추적하고 해제.
- OS 페이지 관리: **OS(커널) 레벨**. **페이지 단위(4KB)**로 물리 메모리 할당/회수.

GC는 "이 객체가 더 이상 쓰이나?"를 판단하고, OS는 "이 페이지가 최근에 접근됐나?"를 판단한다. 판단 기준, 동작 단위, 동작 레벨 전부 다르다.

**해설**:

이 문제가 08장의 핵심을 전부 담고 있어.

!!! note "GC vs OS - 왜 헷갈리나"

    **헷갈리는 이유:** 둘 다 "메모리를 관리한다"고 하니까 같은 거라고 착각하는 거야.

    **다른 점:**

    - GC → JVM 안 → 객체 단위 → 참조 기반
    - OS → 커널 → 페이지 단위 → 접근 시간 기반

    **관계:**

    - GC가 해제해도 OS 관점에서 변화 없음
    - OS가 Swap으로 내려도 JVM은 모름
    - 서로 독립적으로 동작

**채점 기준**:
- (a) "거짓" + JVM 내부 해제/OS 반환 아님 설명하면 2점. "거짓"만 쓰면 0.5점.
- (b) "참" + OS 회수 메커니즘 설명하면 1.5점. "참"만 쓰면 0.5점.
- (c) "거짓" + 레벨 차이 설명하면 1.5점. "거짓"만 쓰면 0.5점.

-> 08장 "GC와 OS 페이지 관리는 다른 것이다" 참조

### Q26. "메모리"의 문맥별 의미 (서술형, 4점)

**정답**:

**문맥 1: "이 컴퓨터는 메모리가 16GB야"**
-> **물리 메모리(RAM)**. 메인보드에 장착된 RAM 모듈의 총 용량. 하드웨어 스펙. (1.5점)

**문맥 2: "이 프로세스가 메모리를 3GB 쓰고 있어"**
-> **해당 프로세스의 RSS(Resident Set Size)** 또는 넓게는 가상 메모리 사용량. 프로세스 단위의 메모리 사용. OS가 해당 프로세스에 할당한 물리 메모리 양. (1.5점)

**문맥 3: "JVM 힙 메모리를 2GB로 설정했어"**
-> **JVM 힙 영역의 크기** (-Xmx 설정값). JVM 프로세스 전체 메모리가 아니라 힙만. JVM은 힙 외에도 Metaspace, 스레드 스택, 네이티브 메모리 등을 사용하므로, "힙 2GB"라고 해서 프로세스 전체가 2GB인 건 아니다. (1점)

**해설**:

"메모리"라는 단어가 얼마나 모호한지 알겠어? 같은 단어인데 문맥마다 의미가 달라:

| 문맥 | 가리키는 것 | 레벨 |
|------|-------------|------|
| "컴퓨터 메모리 16GB" | RAM (물리 하드웨어) | 하드웨어 레벨 |
| "프로세스가 3GB 사용" | RSS/프로세스 메모리 | OS 레벨 |
| "JVM 힙 2GB" | 힙 영역 크기 | 애플리케이션(JVM) 레벨 |

서버 상태를 논의할 때 "메모리가 몇 %야"라고 하면, **뭘 기준으로 한 건지** 반드시 명확히 해야 해. 안 그러면 서로 다른 걸 보면서 "메모리 부족이다/아니다" 싸우게 돼.

**채점 기준**:
- 문맥 1: RAM/물리 메모리 명시 1.5점
- 문맥 2: 프로세스 단위/RSS 언급 1.5점
- 문맥 3: JVM 힙 영역 명시 + "프로세스 전체가 아님" 언급 1점
- 3개 다 "메모리입니다"로 퉁치면 0점

-> 01장 "메모리란", 04장 "프로세스 메모리", 06장 "JVM 힙" 참조

### Q27. [함정 문제] free가 0이면 메모리 부족? (서술형, 4점)

**정답**:

**틀릴 수 있는 이유** (2점):

Linux는 사용하지 않는 물리 메모리를 디스크 캐시(buff/cache)로 적극 활용한다. 따라서 시스템이 오래 가동되면 `free`는 자연스럽게 0에 가까워진다. 이건 메모리 부족이 아니라 **OS가 메모리를 효율적으로 사용하고 있다는 뜻**이다. 캐시/버퍼는 새 프로세스가 메모리를 요청하면 OS가 즉시 회수해서 줄 수 있다.

**올바른 판단 방법** (2점):

1. `free -h`에서 **`available` 컬럼**을 확인한다. available = free + 회수 가능한 캐시/버퍼. 이게 충분하면 메모리 부족이 아니다.
2. **Swap 사용량과 추이**를 확인한다. available이 부족하고 Swap In/Out이 활발하면 그때 비로소 "메모리 부족"이라고 판단할 수 있다.
3. `available`이 총 메모리의 10% 이하로 떨어지고, Swap 사용이 증가 추세이면 **메모리 부족**.

**해설**:

이건 **가장 흔한 오판** 중 하나야. 신입 개발자가 `free -h` 딱 보고 "free가 0이에요! 메모리 부족이에요!"라고 소리 지르는 상황.

```
판단 플로우:

free가 0에 가깝다
  → available 확인
    → available 충분 (총 메모리의 20% 이상)
      → 정상. 캐시가 잘 동작하고 있는 거야.
    → available 부족 (총 메모리의 10% 이하)
      → Swap 확인
        → Swap 미사용 또는 안정적
          → 주의 단계. 모니터링 강화.
        → Swap 증가 추세
          → 메모리 부족. 원인 분석 필요.
```

**채점 기준**:
- 캐시/버퍼 회수 가능성 설명하면 2점
- available 기반 판단 + Swap 확인 절차 쓰면 2점
- "free가 0이면 부족한 거 아닌가요?"에서 한 발짝도 못 벗어나면 0점
- 이거 틀리면 03장부터 다시 읽어

-> 03장 "free -h 해석, available의 진짜 의미" 참조

### Q28. -Xms == -Xmx 권장 이유 (서술형, 5점)

**정답**:

**(1) 힙 확장 비용 관점** (2점):

`-Xms != -Xmx`이면 JVM은 메모리가 부족할 때마다 OS에 추가 메모리를 요청해서 힙을 확장(resize)해야 한다. 이 확장 과정 자체가 비용이다:
- OS에 메모리 할당 요청 (시스템 콜)
- 새 메모리 영역 초기화
- 힙 확장 시 GC가 함께 트리거될 수 있음
- 확장 동안 애플리케이션 일시 정지 가능

`-Xms == -Xmx`로 설정하면 시작 시 전체 힙을 한 번에 확보하므로 **런타임 중 확장 비용이 0**이다.

**(2) GC 동작 예측 관점** (1.5점):

힙 크기가 변하면 GC의 동작도 변한다. Young/Old 비율, GC 트리거 임계값 등이 힙 크기에 따라 달라지기 때문이다. 힙 크기가 고정되면 GC 동작이 **일관적이고 예측 가능**해진다. 성능 테스트 결과를 프로덕션에 그대로 적용할 수 있다.

**(3) 모니터링 관점** (1.5점):

힙 크기가 고정되면 모니터링 수치 해석이 명확해진다. "전체 힙 대비 사용률 X%"가 항상 같은 의미를 가진다. 힙이 확장되는 중이면 분모(전체 힙)가 바뀌므로 사용률이 순간적으로 뛰었다 떨어졌다 하면서 **누수와 정상을 구분하기 어려워진다**.

**해설**:

```
-Xms512m -Xmx2g 일 때:

시간 →
힙크기: 512m → 768m → 1g → 1.2g → 1.5g → 2g
        ↑      ↑      ↑     ↑       ↑     ↑
       시작   확장1  확장2  확장3   확장4   최대

각 확장마다: OS 시스템콜 + GC 가능성 + 순간 정지

-Xms2g -Xmx2g 일 때:

시간 →
힙크기: 2g ──────────────────────────── 2g
        ↑                                ↑
       시작                            항상 동일

확장 없음. 깔끔.
```

실무에서는 거의 항상 `-Xms == -Xmx`로 맞춘다. 다르게 설정하는 건 개발 환경이나 메모리가 매우 제한적인 특수 상황뿐이야.

**채점 기준**:
- (1) 확장 비용 설명 + 구체적 내용 쓰면 2점. "비용이 든다"만 쓰면 1점.
- (2) GC 예측 가능성 설명하면 1.5점. "GC에 영향 준다"만 쓰면 0.5점.
- (3) 모니터링 해석 용이성 설명하면 1.5점. "모니터링이 편하다"만 쓰면 0.5점.

-> 06장 "-Xms/-Xmx, 힙 확장" 참조

### Q29. 모니터링 패턴 판단 (서술형, 4점)

**정답**:

**[패턴 A]** -- (2점)

**이상 (메모리 누수 의심).**

근거: 그래프가 24시간 동안 **지속적으로 우상향**하고 있다. 톱니 패턴(올라갔다 내려갔다)이 있긴 하지만, **기준선(GC 후 최저점)이 점점 올라가고 있다**. 20% -> 30% -> 40% -> 50% -> 60% -> 70% -> 80%으로 기준선이 상승. 이는 GC가 회수하지 못하는 객체가 계속 쌓이고 있다는 뜻이다. 전형적인 메모리 누수 패턴이며, 이대로 두면 결국 OOM이 발생한다.

**[패턴 B]** -- (2점)

**정상 (건강한 톱니 패턴).**

근거: 메모리 사용률이 약 40%~70% 사이에서 **주기적으로 올라갔다 내려갔다** 반복하고 있다. 핵심은 **GC 후 기준선이 일정** (약 40~50% 수준으로 매번 돌아옴)하다는 점이다. 이는 객체가 생성되고(사용률 상승) -> GC가 회수하고(사용률 하락) -> 다시 생성되는 정상적인 JVM 동작이다. 기준선이 올라가지 않으므로 누수가 없다.

**해설**:

```
모니터링 패턴 판단 핵심:

1. 기준선(GC 후 최저점)을 본다
   → 올라가는가? → 누수 의심
   → 일정한가?   → 정상

2. 톱니 패턴 자체는 정상이다
   → 올라갔다 내려가는 건 GC가 잘 동작한다는 뜻

3. 절대값이 아니라 패턴을 본다
   → 70%라도 안정적이면 정상
   → 40%라도 계속 올라가면 위험
```

**채점 기준**:
- 패턴 A: "누수/이상" 판단 + 기준선 상승 근거 쓰면 2점. 판단만 쓰면 1점.
- 패턴 B: "정상" 판단 + 기준선 안정 근거 쓰면 2점. 판단만 쓰면 1점.
- A와 B 판단이 반대면 각 0점.

-> 09장 "정상 vs 이상 판단" 참조

### Q30. 종합 판단 (서술형, 4점)

**정답**:

**빠싺 시니어로서의 대응** (최소 3가지 관점):

**관점 1: "55%"의 기준 확인** (1점)

추가 질문: "55%가 뭘 기준으로 한 거야? OS 전체 메모리? JVM 힙 사용률?"

- OS 전체 기준 55% (16GB 중 9GB)라면, 이건 모든 프로세스 합산이야. JVM 하나만의 문제가 아닐 수 있어.
- JVM 힙 기준 55%라면 의미가 다르지.
- 먼저 **기준을 명확히** 해야 판단이 가능해.

**관점 2: `free -h`의 free 300MB와 available 확인** (1점)

추가 질문: "available은 얼마야?"

- free가 300MB라고 했는데, Q13, Q27에서 배운 대로 **free만으로 판단하면 안 돼**.
- available이 4~5GB면 캐시/버퍼가 회수 가능하므로 당장 문제없어.
- available이 300MB 근처면 그때 비로소 주의.

**관점 3: 추이(Trend) 확인** (1점)

추가 질문: "55%가 안정적이야? 올라가는 중이야? 언제부터 이 수치야?"

- 안정적(한 달째 55% 유지): 정상. 서비스 특성상 그 정도 쓰는 거.
- 올라가는 중(어제 34%, 오늘 55%): 누수 또는 비정상 증가. 원인 파악 필요.
- **한 시점의 수치만으로는 절대 판단 불가.**

**추가 관점 (보너스):**

- **Swap 1GB 사용 원인**: "Swap이 언제부터 사용됐어? 증가 추세야?" -- Swap 사용 자체가 과거 메모리 압박의 흔적. 원인 필요.
- **jstat 확인**: "Old 영역 사용률은? Full GC 빈도는?" -- JVM 내부 상태를 봐야 실체를 알 수 있어.
- **서비스 영향**: "응답 시간은? 에러율은?" -- 메모리 수치보다 서비스 영향이 더 중요할 수 있어.

**종합 판단**:

이 보고는 **"데이터를 나열했을 뿐 분석하지 않았다."** 숫자만 나열하면 보고가 아니야. 55%라는 숫자, free 300MB라는 숫자, Swap 1GB라는 숫자를 **각각 해석하고, 추이와 맥락을 함께 봐야** 비로소 "문제다/아니다"를 판단할 수 있어.

**해설**:

이 문제가 01장부터 10장까지 **전부** 연결되는 종합 문제야.

```
01장: "메모리"가 뭘 가리키는지 (물리 메모리 16GB)
02장: 프로세스 관점 (어떤 프로세스가 얼마나 쓰는지)
03장: free -h 해석 (free 300MB ≠ 부족, available이 핵심)
04장: committed vs resident (RSS와 모니터링 %의 관계)
05장: JVM 옵션 확인 필요 (-Xms/-Xmx가 뭔지)
06장: 힙 구조/확장 여부
07장: GC 상태 (jstat 확인 필요)
08장: GC ≠ OS 메모리 관리 (다른 레벨)
09장: 정상 vs 이상 판단 (패턴으로 봐야 해)
10장: 실전 사례와 동일한 상황 (34%→55%)
```

**채점 기준**:
- 3가지 관점 이상 제시 + 각각 구체적 질문/판단 기준 쓰면 4점
- 2가지 관점이면 3점
- 1가지 관점이면 2점
- "문제 있다" 또는 "문제 없다"만 쓰면 0점
- **"추가 질문" 없이 단정하면 감점** -- 빠싺는 한 시점의 숫자로 단정하지 않아

-> 전 범위 (01~10장) 종합 참조
