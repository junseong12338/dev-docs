# 08. GC와 OS 페이지 관리는 다른 것이다 - Delta

---

## 1. 핵심 개념 - "이게 뭐야?"

이 챕터가 왜 따로 있냐?
**네가 헷갈렸으니까.** "GC 돌리면 모니터링 메모리 사용률이 내려가야 하는 거 아니에요?"

아니야. 절대 아니야. **완전히 다른 놈이야.**

비유부터 가자. 호텔을 상상해봐.

!!! example "호텔 비유"

    - **호텔** = 물리 RAM (15GB)
    - **방** = 메모리 페이지 (4KB 단위)
    - **투숙객** = 프로세스 (Tomcat, sshd, systemd 등)

    투숙객이 방을 잡았어. 방 안에 짐(Java 객체)을 쌓아놓고 생활 중.

    **GC = 방 안 청소부**

    - 방 안에 들어가서 쓰레기(안 쓰는 객체) 치움
    - 방 안이 깨끗해짐
    - 근데 방 자체는 그대로 투숙객 거야
    - 호텔 입장에선? "그 방 아직 사용 중" (used 그대로)

    **OS 페이지 관리 = 건물 관리인**

    - 투숙객이 퇴실했거나 오래 안 온 방의 열쇠 회수
    - 방 자체를 비움 → 다른 사람한테 줄 수 있음
    - 호텔 입장에선? "빈 방 늘었다" (available 증가)

    **핵심: 방 안 청소(GC)와 방 자체 회수(OS)는 다른 일이야**

비유는 여기까지. 정확한 정의:

```
GC (Garbage Collection)
= JVM 내부에서, 힙 메모리 안의 안 쓰는 Java 객체를 제거하는 것
= JVM 레벨의 작업
= OS는 이걸 모름

OS 페이지 관리 (Page Reclamation)
= 리눅스 커널이, 물리 RAM의 페이지를 관리하는 것
= OS 레벨의 작업
= JVM은 이걸 모름

둘은 서로 다른 레벨에서 독립적으로 동작한다.
```

## 2. 동작 원리 - "어떻게 돌아가?"

### 2-1. 두 레벨 비교 다이어그램

!!! note "두 레벨 비교 다이어그램"

    ```
    ┌───────────────────────────── 물리 RAM (15GB) ──────────────────┐
    │                                                                 │
    │  ┌─────────────────────────┐  ┌─────────┐  ┌─────────┐       │
    │  │     Tomcat (JVM)         │  │  sshd   │  │ systemd │  ...  │
    │  │                          │  │         │  │         │       │
    │  │  ┌──────── Heap ───────┐│  │         │  │         │       │
    │  │  │                     ││  │         │  │         │       │
    │  │  │  [객체A] [   ] [객체C]│  │         │  │         │       │
    │  │  │  [   ] [객체E] [   ] ││  │         │  │         │       │
    │  │  │  [객체G] [   ] [   ] ││  │         │  │         │       │
    │  │  │                     ││  │         │  │         │       │
    │  │  │  ↑ GC는 여기서 일해 ││  │         │  │         │       │
    │  │  │  [   ] = GC가 치운  ││  │         │  │         │       │
    │  │  │         빈 공간     ││  │         │  │         │       │
    │  │  └─────────────────────┘│  │         │  │         │       │
    │  │                          │  │         │  │         │       │
    │  └──────────┬───────────────┘  └────┬────┘  └────┬────┘       │
    │             │                       │             │            │
    │  ═══════════╪═══════════════════════╪═════════════╪════════════│
    │             │    OS 레벨 경계       │             │            │
    │  ═══════════╪═══════════════════════╪═════════════╪════════════│
    │             │                       │             │            │
    │  ┌──────────▼───────────────────────▼─────────────▼──────────┐│
    │  │                  물리 메모리 페이지들                       ││
    │  │  [P1][P2][P3][P4][P5][P6][P7][P8][P9][P10] ...            ││
    │  │   ↑ OS는 여기서 일해                                      ││
    │  │   페이지 단위로 할당/회수                                  ││
    │  └────────────────────────────────────────────────────────────┘│
    │                                                                 │
    └─────────────────────────────────────────────────────────────────┘
    ```

보여? **두 개가 완전히 다른 층에서 일하고 있어.**
GC는 JVM 힙 안에서만 놀아. OS는 물리 메모리 페이지 단위로 놀아.

### 2-2. GC가 하는 일 (상세)

!!! note "GC가 하는 일"

    - **누가:** JVM (Java Virtual Machine)
    - **뭘:** Java 객체 (new로 만든 것들) → VO, DTO, Service, 컬렉션, 문자열 등
    - **어디서:** JVM Heap 메모리 안에서만 → -Xms ~ -Xmx 범위의 가상 주소 공간
    - **언제:** Eden 꽉 차면 → Minor GC / Old Gen 부족하면 → Full GC
    - **결과:** 힙 안에서 쓰레기 객체 제거 → 빈 공간 확보 → 새 객체 할당 가능해짐
    - **모니터링:** jstat -gc의 OU (Old Used) 줄어듦, EU (Eden Used) 0에 가까워짐
    - **OS에 영향:** 없음! OS는 GC가 뭘 했는지 모름

### 2-3. OS 페이지 관리가 하는 일 (상세)

!!! note "OS 페이지 관리가 하는 일"

    - **누가:** 리눅스 커널 (mm 서브시스템)
    - **뭘:** 물리 RAM의 페이지 (4KB 단위) → 프로세스에 할당된 물리 메모리 프레임
    - **어디서:** 물리 RAM 전체 → 15GB 전체가 관리 대상
    - **언제:** available이 부족할 때 → kswapd가 백그라운드에서 회수 시도 → 더 급하면 direct reclaim (동기 회수)
    - **뭘 회수하나:**
        1. 페이지 캐시 (파일 읽기 캐시) → 가장 먼저 회수
        2. 비활성 페이지 → Swap으로 내보냄
        3. 프로세스 종료 시 → 해당 프로세스의 모든 페이지 회수
    - **결과:** 물리 RAM에서 빈 공간 확보 → free -h의 available 증가
    - **JVM에 영향:** JVM은 OS가 뭘 했는지 모름 (Swap out 되면 성능 저하는 느낄 수 있지만)

### 2-4. 서로 모르는 사이

이게 핵심이야. **GC는 OS를 모르고, OS는 GC를 모른다.**

!!! note "서로 모르는 사이"

    **GC 입장:**
    "나는 힙 안의 Java 객체만 관리해. 이 객체가 물리 RAM 어디에 있는지? 몰라. 관심 없어. 가상 주소만 다루니까. OS가 알아서 매핑해주겠지."

    **OS 입장:**
    "나는 물리 페이지만 관리해. 이 페이지 안에 Java 객체가 있는지 쓰레기가 있는지? 몰라. 프로세스가 쓰고 있다고 하면 쓰고 있는 거고, 안 쓰면 안 쓰는 거야. 근데 '안 쓴다'는 건 프로세스가 그 메모리를 munmap하거나 madvise(DONT_NEED)를 호출해야 아는 거야."

    **GC가 객체를 치워도, JVM은 OS에게 "이 페이지 돌려줄게"라고 말하지 않아. 힙 안에서 빈 공간으로 재사용할 뿐이야.**

### 2-5. 왜 모니터링 %가 안 내려가는가?

이걸 이해하면 네가 했던 질문에 완벽하게 답할 수 있어.

!!! danger "'GC 돌렸는데 모니터링 안 내려가요!' 의 진실"

    상황: 우리 서버 모니터링 55% 표시

    **[Step 1] GC가 객체를 치움**

    ```
    JVM Heap (8GB committed)
    ┌──────────────────────────────────────────────┐
    │ Before GC:                                    │
    │ [obj][obj][obj][obj][obj][obj][obj][obj]       │
    │                                                │
    │ After GC:                                     │
    │ [obj][   ][obj][   ][   ][obj][   ][   ]       │
    │       ↑        ↑    ↑        ↑    ↑            │
    │     빈 공간 (JVM 안에서 재사용 가능)           │
    └──────────────────────────────────────────────┘
    ```

    GC 결과: OU 줄어듦, 힙 안에 여유 공간 생김

    **[Step 2] OS는 뭘 보는가?**

    ```
    OS 입장에서 JVM 프로세스:
    ┌──────────────────────────────────────────────┐
    │ [P1][P2][P3][P4][P5][P6][P7][P8]...           │
    │  ↑ 이 페이지들 전부 "JVM이 쓰고 있음"         │
    │  ↑ GC 전이나 후나 JVM이 OS에게 반납 안 했음   │
    │  ↑ OS 입장에선 변한 게 없음                    │
    └──────────────────────────────────────────────┘
    ```

    OS 결과: used 변화 없음, available 변화 없음

    **[Step 3] 모니터링 도구가 보는 것**

    모니터링: used / total = 8.3GB / 15GB = 55% → GC 전이든 후든 55% 그대로

    **결론: 당연한 거야. 다른 레벨이니까.**

### 2-6. OS가 실제로 페이지를 회수하는 경우

그럼 OS는 언제 메모리를 회수해?

!!! note "OS가 페이지를 회수하는 조건"

    **1. available이 부족할 때**
    → 커널의 min_free_kbytes 임계치 이하로 떨어지면
    → kswapd가 깨어나서 비활성 페이지를 Swap으로 내보냄
    → 페이지 캐시를 해제함

    **2. 프로세스가 종료될 때**
    → 프로세스가 죽으면 해당 프로세스의 모든 페이지 회수
    → kill -9 tomcat 하면 JVM의 8GB 한 방에 회수됨

    **3. 프로세스가 명시적으로 반납할 때**
    → munmap() 시스템 콜로 가상 메모리 해제
    → madvise(MADV_DONTNEED)로 "이 페이지 안 쓸 거야" 알림
    → JVM은 보통 이걸 안 해 (힙을 재사용하니까)

    **우리 서버 상황:**
    available = 6.9GB → 넉넉함 → OS가 회수할 이유 없음
    → 그래서 used가 안 내려가는 거야 → **정상이야! 문제가 아니야!**

### 2-7. 전체 흐름 정리 다이어그램

!!! abstract "GC vs OS 페이지 관리 전체 비교"

    |  | GC (가비지 컬렉션) | OS 페이지 관리 |
    |------|----------|---------|
    | 담당자 | JVM | 리눅스 커널 |
    | 대상 | Java 객체 (힙 안의 쓰레기) | 물리 RAM 페이지 4KB (방 자체) |
    | 트리거 | Eden 꽉 참 (Minor) / Old 부족 (Full) | available 부족 / kswapd 깨어남 |
    | 결과 | 힙 안에 빈 공간 확보 (OU 줄어듦) | 물리 RAM 빈 공간 확보 (available 늘어남) |
    | 모니터링 | jstat -gc | free -h |
    | 상대방 인지 | OS를 모름 | GC를 모름 |
    | 비유 | 방 안 청소부 | 건물 관리인 |

## 3. 코드/명령어로 보자

### 3-1. GC가 일어났는지 확인하기

```bash
# JVM 힙 상태 확인 (GC 레벨)
$ jstat -gc $(pgrep -f tomcat)

# 출력에서 볼 것:
# OU (Old Used) → GC 후 줄어들었으면 GC가 일한 거
# YGC, FGC → GC 횟수 확인
```

### 3-2. OS 메모리 상태 확인하기

```bash
# OS 메모리 현황 (OS 레벨)
$ free -h
              total        used        free    shared  buff/cache   available
Mem:            15G        8.3G        1.2G       200M        5.7G        6.9G
Swap:          4.0G        145M        3.9G
```

### 3-3. 두 레벨을 동시에 보기

```bash
# 이 두 명령어를 같이 실행해봐
$ echo "=== OS Level ===" && free -h && echo "" && echo "=== JVM Level ===" && jstat -gc $(pgrep -f tomcat)
```

실제 시나리오:

```
=== GC 전 ===

[OS Level]
Mem: 15G total, 8.3G used, 6.9G available

[JVM Level]
OU = 440MB (Old Used)

=== GC 발생 후 (Minor GC) ===

[OS Level]
Mem: 15G total, 8.3G used, 6.9G available    ← 변화 없음!!!
                ^^^^                ^^^^

[JVM Level]
OU = 440MB (Old는 Minor GC 대상 아니니까 그대로)
EU = 100MB → 50MB로 줄어듦                    ← 여기가 변함
YGC = 1847 → 1848로 증가

=== 결론 ===
GC가 Eden을 청소했지만, OS 레벨 used는 8.3GB 그대로야.
GC는 JVM 안에서 일한 거고, OS는 그걸 모르니까.
```

### 3-4. OS가 실제로 메모리를 회수하는 걸 보려면?

```bash
# 프로세스를 죽이면 OS가 메모리 회수하는 걸 볼 수 있어
# (절대 운영 서버에서 하지 마. 이건 이해를 위한 설명이야)

# Before:
$ free -h
Mem: 15G total, 8.3G used, 6.9G available

# 만약 Tomcat(8GB 잡고 있는 놈)을 죽이면:
$ kill -9 $(pgrep -f tomcat)    # ← 절대 운영서버에서 하지 마!!!

# After:
$ free -h
Mem: 15G total, 1.5G used, 13G available
#                ↑ 확 줄어듦!

# 왜? 프로세스가 죽으면 OS가 그 프로세스의 모든 페이지를 회수하니까.
# 이게 OS 레벨 메모리 회수야. GC랑은 완전 다른 거야.
```

### 3-5. JVM이 OS에게 메모리를 돌려주는 특수한 경우

```
보통 JVM은 한번 잡은 힙 메모리를 OS에 안 돌려줘.
근데 예외가 있어:

1. G1GC + -XX:+ShrinkHeapInSteps (Java 12+)
   → Full GC 후 사용량이 적으면 힙을 줄이고 OS에 반환

2. ZGC + -XX:+ZUncommit (Java 13+)
   → 사용하지 않는 힙 페이지를 OS에 반환

3. -XX:MinHeapFreeRatio / -XX:MaxHeapFreeRatio
   → 힙 사용률이 낮으면 축소 가능 (Parallel GC)

우리 서버 (Java 8 + Parallel GC):
→ 기본적으로 힙 축소/반환 잘 안 일어남
→ -Xms=8GB로 최소 힙을 8GB 잡아놨으니 8GB 이하로 줄어들 이유가 없음
→ 그래서 OS used가 안 내려가는 거야
→ 정상 동작이야!
```

## 4. 주의사항 / 함정

### 함정 1: "GC 돌리면 모니터링 내려가야 하는 거 아니에요?"

!!! danger "함정 1"

    **❌ 아니야.**
    GC = JVM 안에서 객체 치우는 것 / 모니터링 = OS 레벨 used / total
    다른 레벨이야. 절대 1:1 대응 안 돼.

    **✅** GC 결과를 보려면 jstat -gc의 OU를 봐야 해.
    OS 모니터링(free -h)의 used를 보면 안 돼.

### 함정 2: "그럼 모니터링이 올라가는 건 무조건 JVM 힙 확장 때문이야?"

!!! danger "함정 2"

    **❌ 꼭 그런 건 아니야.** 여러 원인이 있어:

    1. JVM 힙 확장 (Xms→Xmx 사이에서 커질 때) ← 우리 경우
    2. 페이지 캐시 증가 (파일 I/O 많으면 캐시에 쌓임)
    3. 다른 프로세스 메모리 증가 (jspd, 모니터링 에이전트 등)
    4. Metaspace 증가 (클래스 로딩 많을 때)
    5. 실제 메모리 누수 (코드 버그)

    **✅** 원인을 특정하려면 프로세스별로 봐야 해:
    `ps -eo pid,rss,comm --sort=-rss | head -10`

### 함정 3: "available이 많으면 무조건 안전해?"

!!! danger "함정 3"

    **❌ 거의 맞지만, 함정이 있어:**
    available은 "OS가 회수할 수 있는 것 포함"이야.
    페이지 캐시를 포함하거든.
    근데 캐시가 자주 쓰이는 데이터면, 회수하면 I/O 성능이 떨어져.

    **✅** available이 2GB 이상이면 일반적으로 안전.
    근데 available이 줄어드는 추세인지도 같이 봐야 해.

### 함정 4: "JVM -Xms랑 -Xmx를 같게 하면 이런 혼란 없겠네?"

!!! warning "함정 4"

    **△ 반만 맞아.**
    -Xms = -Xmx로 하면 힙 확장이 안 일어나니까 "올라가는 현상"은 줄어들어.
    근데 다른 문제가 있어:

    - 시작부터 큰 메모리를 점유하니까 다른 프로세스에 영향
    - available이 처음부터 적어짐
    - 힙이 실제로 필요한 것보다 크면 메모리 낭비

    **✅** 실무에서는 -Xms = -Xmx로 맞추는 경우가 많아.
    왜? 힙 확장/축소 자체가 비용이고, 예측 가능성이 중요하니까.
    근데 그건 "메모리가 충분할 때" 이야기야.

### 함정 5: "free -h의 used가 높으면 위험한 거 아니야?"

!!! danger "함정 5"

    **❌ used가 높다고 위험한 게 아니야.**
    03장에서 배웠잖아. buff/cache도 used에 포함될 수 있어.
    봐야 할 건 used가 아니라 available이야.

    **✅ 판단 기준:**

    - available > 2GB → 여유 있음
    - available < 1GB → 주의
    - available < 500MB + Swap 증가 → 위험

    우리 서버: available = 6.9GB → 넉넉. 걱정 없어.

## 5. 정리

!!! abstract "GC vs OS 페이지 관리 핵심 정리"

    | GC | OS 페이지 관리 |
    |----|----------------|
    | JVM 내부 작업 | 커널 레벨 작업 |
    | Java 객체 대상 | 물리 페이지 대상 |
    | jstat -gc로 확인 | free -h로 확인 |
    | OU 줄어듦 = 성공 | available 늘어남 = 회수됨 |
    | OS를 모름 | GC를 모름 |

    **핵심 한 줄: GC가 객체를 치워도 OS used는 안 내려간다.**
    왜? 다른 레벨이니까. JVM은 치운 공간을 재사용하지 OS에 안 돌려줘.

    **모니터링 55%가 정상인 이유:**

    1. GC 레벨: OU 5% → 누수 없음
    2. OS 레벨: available 6.9GB → 여유 있음
    3. 55%는 JVM이 힙을 확장해서 페이지를 더 잡은 것뿐
    4. OS가 회수할 필요 없으니까(available 충분) used가 그대로인 것

**이 챕터에서 반드시 기억할 것:**
1. GC(JVM 레벨)와 OS 페이지 관리(커널 레벨)는 **완전히 다른 것**
2. GC가 객체를 치워도 OS 모니터링의 used는 **안 내려간다**
3. JVM은 한번 잡은 메모리를 OS에 보통 **돌려주지 않는다**
4. 모니터링 %를 볼 때는 **어느 레벨의 수치인지** 구분해야 한다

### 확인 문제 (5문제)

> 다음 문제를 풀어봐. 답 못 하면 위에서 다시 읽어.

**Q1.** GC가 힙에서 500MB 분량의 객체를 치웠다. free -h의 used는 얼마나 줄어들어?

**Q2.** 호텔 비유에서 GC는 뭐고, OS 페이지 관리는 뭐야? 둘의 차이를 비유로 설명해봐.

**Q3.** 우리 서버에서 모니터링이 42%→55%로 올라갔는데, OU는 440MB(5%)로 안정적이야. 이 상황을 GC 레벨과 OS 레벨로 나눠서 설명해봐.

**Q4.** OS가 물리 메모리 페이지를 회수하는 조건 3가지를 말해봐. 우리 서버에서는 왜 회수를 안 해?

**Q5.** 다음 중 올바른 설명을 모두 골라:
```
A. GC가 돌면 free -h의 available이 늘어난다
B. jstat -gc의 OU가 줄어들면 GC가 일한 것이다
C. JVM은 GC 후 빈 힙 공간을 OS에 바로 돌려준다
D. available이 6.9GB면 OS는 페이지 회수할 이유가 없다
E. OS는 JVM 힙 안의 객체가 쓰레기인지 아닌지 알 수 있다
```

??? success "정답 보기"
    **A1.** 안 줄어들어. 0이야. GC는 JVM 힙 안에서 객체를 치우는 것이고, OS는 그걸 모르니까. JVM이 OS에게 메모리를 반납하지 않는 한 free -h의 used는 변하지 않아.

    **A2.** GC = 방 안 청소부. 투숙객(JVM)의 방 안에 들어가서 쓰레기(안 쓰는 객체)를 치움. 근데 방 자체는 여전히 투숙객 거야. 호텔 관리인(OS) 입장에선 "그 방 아직 사용 중"이야.
    OS 페이지 관리 = 건물 관리인. 투숙객이 퇴실했거나 방을 오래 안 쓰면 열쇠를 회수해서 방 자체를 비움. 다른 사람한테 줄 수 있는 빈 방이 늘어남.
    차이: 청소부는 방 안 쓰레기를 치우고, 관리인은 방 자체를 회수해. 레벨이 다름.

    **A3.** GC 레벨: OU가 440MB(5%)로 안정적이므로 Old Gen에 누수 없음. GC가 정상적으로 쓰레기를 회수하고 있음. JVM 힙 안은 건강해.
    OS 레벨: 42%→55%로 올라간 건 JVM이 힙을 확장(-Xms 8GB에서 -Xmx 12GB 쪽으로)하면서 OS에 물리 페이지를 더 요청한 것. JVM이 더 많은 페이지를 점유하니까 OS used가 올라간 거야. available이 6.9GB로 충분하므로 문제없는 정상 동작이야.

    **A4.** OS가 물리 메모리 페이지를 회수하는 조건: (1) available이 부족할 때 (kswapd 임계치 이하로 떨어지면) (2) 프로세스가 종료될 때 (해당 프로세스의 모든 페이지 회수) (3) 프로세스가 명시적으로 반납할 때 (munmap, madvise 등). 우리 서버에서 회수 안 하는 이유: available이 6.9GB로 넉넉하기 때문에 커널이 페이지를 회수할 필요가 없어. 회수는 메모리가 부족할 때 하는 거지, 여유 있으면 안 해.

    **A5.** 올바른 것: **B, D**

    - A: 틀림. GC가 돌아도 free -h의 available은 변하지 않아. 다른 레벨이니까.
    - B: 맞음. OU가 줄어들면 Old Gen에서 객체가 수거된 거야.
    - C: 틀림. JVM은 보통 GC 후 빈 공간을 재사용하지 OS에 돌려주지 않아 (Java 8 + Parallel GC 기준).
    - D: 맞음. available이 충분하면 OS는 페이지 회수 동작을 할 이유가 없어.
    - E: 틀림. OS는 페이지 안의 내용물이 Java 객체인지 쓰레기인지 전혀 모름. 프로세스가 사용하는 페이지일 뿐이야.
