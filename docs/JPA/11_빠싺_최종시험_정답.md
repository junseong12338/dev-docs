## 객관식 정답 (10문제 × 2점 = 20점)

### Q1. 정답: **(B)**

Hibernate가 JPA 스펙을 구현한다.

> JPA는 **인터페이스(스펙)**이고, Hibernate는 그 스펙을 **구현한 라이브러리**야. Java에서 `interface`와 `class`의 관계랑 같아. JPA 스펙만 있으면 아무것도 못 해. Hibernate(또는 EclipseLink 등)가 있어야 실제로 동작해.
>
> (A) 상속 관계 아님. 스펙-구현 관계.
> (C) 동일하지 않음. JPA = 스펙, Hibernate = 구현체.
> (D) Hibernate는 JPA 없이도 독자적으로 동작 가능 (JPA 이전부터 존재).

📖 **Ch01 - JPA가 뭐야**

### Q2. 정답: **(B)**

JPA가 리플렉션으로 빈 객체를 먼저 생성하기 때문.

> JPA는 DB에서 데이터 가져올 때 `new Entity()`로 빈 객체 먼저 만들고, 리플렉션으로 필드값을 채워 넣어. 기본 생성자 없으면 객체를 못 만들어서 에러 터져.
>
> (A) Builder 패턴은 Lombok 편의 기능. JPA 필수 아님.
> (C) Lombok이 아니라 JPA가 요구하는 거야.
> (D) 엔티티는 스프링 빈이 아니야.

📖 **Ch02 - 엔티티 기초**

### Q3. 정답: **(B)**

엔티티의 PK가 null이거나 DB에 없을 때.

> `save()`는 내부적으로 PK를 확인해서 INSERT/UPDATE를 결정해. PK가 null이면 새 엔티티로 판단 → INSERT. PK가 있는데 DB에 존재하면 → UPDATE (merge). MyBatis의 INSERT/UPDATE를 하나로 합친 거야.
>
> (A) 모든 필드 null 여부가 아니라 PK만 봐.
> (C) @GeneratedValue 유무와 무관. PK 값으로 판단.
> (D) UPDATE도 실행해.

📖 **Ch03 - Repository와 CRUD**

### Q4. 정답: **(C)**

FK가 있는 쪽.

> 연관관계의 주인 = FK를 가진 테이블에 매핑된 엔티티. DB에서 FK가 있는 쪽이 관계를 관리하니까, JPA에서도 FK 있는 쪽이 주인이야. `@ManyToOne`이 붙는 쪽이 보통 FK를 가지고 있어.
>
> (A) @OneToMany는 주인이 아닌 쪽 (mappedBy로 표시).
> (B) PK가 아니라 FK 기준.
> (D) mappedBy가 붙은 쪽은 주인이 아닌 쪽이야.

📖 **Ch04 - 연관관계 기초**

### Q5. 정답: **(C)**

@Transactional 안에서 영속 상태 엔티티여야 한다.

> Dirty Checking은 영속성 컨텍스트가 엔티티의 스냅샷과 현재 상태를 비교해서 변경된 걸 자동 감지하는 거야. 이게 동작하려면: (1) 트랜잭션 안이어야 하고, (2) 엔티티가 영속 상태여야 해. 두 조건 다 필요.
>
> (A) flush()는 변경 감지 결과를 DB에 보내는 거지, 감지 자체를 시작하는 게 아님.
> (B) save() 안 불러도 변경 감지로 자동 반영돼. 이게 핵심.
> (D) @PreUpdate는 콜백이지 변경 감지 조건이 아님.

📖 **Ch05 - 영속성 컨텍스트**

### Q7. 정답: **(B)**

프록시 기반이라 this 호출은 프록시를 안 거쳐서.

> 스프링의 @Transactional은 AOP 프록시로 동작해. 외부에서 호출하면 프록시 → 실제 객체 순서로 가는데, 같은 클래스 내부에서 `this.methodB()` 하면 프록시를 건너뛰고 실제 객체의 메서드를 직접 호출해. 그래서 트랜잭션이 안 걸려.
>
> (A) 스프링이 금지하는 게 아니라 프록시 구조상 안 되는 거야.
> (C) 합류 여부는 전파(Propagation) 문제지, 내부 호출 문제가 아님.
> (D) readOnly랑 무관.

### Q8. 정답: **(C)**

FetchType.EAGER로 변경.

> EAGER는 N+1의 **주범**이야. 해결책이 아니라 원인이야. 연관 엔티티를 무조건 함께 가져오니까 불필요한 쿼리가 나가. LAZY로 바꾸고 필요할 때만 fetch join으로 가져오는 게 맞아.
>
> (A) fetch join — 1번 SQL로 JOIN해서 가져옴. 핵심 해결책.
> (B) @EntityGraph — fetch join의 어노테이션 버전.
> (D) @BatchSize — N번 쿼리를 batch_size만큼 묶어서 줄여줌.

### Q12. 정답:

**원시 타입 long은 null이 불가능해서, 새 엔티티(PK 미할당)와 기존 엔티티(PK=0)를 구분할 수 없다.**

### Q23. 정답:

**save()를 빼도 DB에 반영된다. 이유: Dirty Checking(변경 감지).**

> @Transactional 안에서 findById()로 가져온 엔티티는 **영속 상태**야. 영속 상태 엔티티의 필드를 변경하면, 트랜잭션 커밋 시점에 영속성 컨텍스트가 스냅샷과 비교해서 변경된 부분을 자동으로 UPDATE SQL로 보내. 이게 Dirty Checking이야.

> **save()를 빼도 되는 이유**: 이미 영속 상태라서 변경 감지가 동작하기 때문. save()는 불필요한 호출이지만 넣어도 에러는 안 나 (merge 동작).

📖 **Ch05** | 채점: Dirty Checking 언급 (2점) + 영속 상태 설명 (1점) + 빼도 되는 이유 (2점)

## 서술형 정답 (5문제 = 21점)

### Q26. (4점) "JPA 쓰면 SQL 몰라도 된다"가 틀린 이유

**모범 답안:**

1. **성능 튜닝에 SQL 필요**: N+1 문제 해결을 위한 fetch join, 벌크 연산 등은 JPQL을 알아야 하고, JPQL 자체가 SQL 기반이야. SQL 모르면 JPQL도 못 써.

2. **복잡한 쿼리는 네이티브 SQL 필수**: JPQL로 표현 불가능한 DB 함수, 복잡한 서브쿼리, 통계 쿼리는 `nativeQuery = true`로 직접 SQL을 써야 해.

3. **디버깅에 SQL 필요**: JPA가 생성하는 SQL을 읽고 이해해야 성능 문제나 잘못된 쿼리를 잡을 수 있어. SQL 모르면 로그 봐도 뭐가 문제인지 모름.

4. **DDL 설계는 SQL 영역**: 테이블 설계, 인덱스, 제약조건은 SQL/DB 지식이 필수. JPA가 DDL 자동 생성해주더라도 프로덕션에서는 직접 설계해야 해.

📖 **Ch01** | 채점: 이유 2가지 이상 (각 2점). 1가지만 정확하면 2점.

### Q27. (4점) 영속성 컨텍스트의 1차 캐시와 변경 감지를 MyBatis와 비교

**모범 답안:**

**1차 캐시:**
- **JPA**: 같은 트랜잭션 내에서 같은 PK로 조회하면 첫 번째만 SQL 실행, 두 번째부터는 1차 캐시에서 반환. SQL 횟수가 줄어듬.
- **MyBatis**: 캐시 없음. 같은 쿼리를 2번 호출하면 SQL도 2번 실행. 매번 DB에 다녀와야 해.

**변경 감지 (Dirty Checking):**
- **JPA**: 영속 상태 엔티티의 필드를 `setter`로 변경하면, 트랜잭션 커밋 시 스냅샷과 비교해서 변경된 부분만 자동으로 UPDATE SQL 생성. `repository.save()` 안 불러도 됨.
- **MyBatis**: 변경 감지 없음. 객체 값을 바꿔도 DB에 자동 반영 안 됨. 반드시 UPDATE SQL을 직접 작성하고 `mapper.update()`를 호출해야 해.

📖 **Ch05** | 채점: 1차 캐시 비교 (2점) + 변경 감지 비교 (2점). 비교 없이 JPA만 설명하면 절반.

### Q28. (4점) createLesson()에서 @Transactional이 필요한 이유

**모범 답안:**

createLesson()의 흐름은 3단계야:
1. **호스트 배정** — HostPool에서 가용 호스트를 찾아 HostAssignment에 배정 기록 INSERT
2. **Zoom 미팅 생성** — Zoom API 호출하여 meetingId, startUrl, joinUrl 획득
3. **강의 DB 저장** — Lesson 엔티티에 Zoom 결과 세팅 후 INSERT

이 3단계가 **하나의 원자적 작업**이어야 해:

- **Zoom 생성 실패 시**: 호스트가 배정된 상태로 남으면 그 호스트는 다른 강의에 사용 불가 (자원 점유). @Transactional이 있어야 전부 롤백되고, catch에서 `releaseHost()`로 호스트를 반환할 수 있어.
- **DB 저장 실패 시**: Zoom 미팅은 만들어졌는데 강의 데이터가 없으면 고아 미팅이 돼. 트랜잭션으로 DB 작업은 롤백하고, 별도로 Zoom 미팅 삭제 처리가 필요.

@Transactional 없이 각 단계가 독립적으로 실행되면 **중간에 실패했을 때 일관성이 깨져**. 전부 성공하거나 전부 실패해야 하는 게 트랜잭션의 핵심.

📖 **Ch07, Ch09** | 채점: 3단계 흐름 인지 (1점) + 실패 시 문제 설명 (2점) + 원자성/일관성 연결 (1점)

### Q29. (5점) N+1 문제 설명, MyBatis에서 없는 이유, JPA 해결 방법 2가지

**모범 답안:**

**N+1 문제란:**
목록 조회 1번(1) + 각 항목의 연관 데이터 조회 N번(N) = 총 N+1번 SQL이 실행되는 문제. 학기 100개 조회 후 각 학기의 과목에 접근하면 SQL 101번. 데이터 많으면 DB 성능이 심각하게 떨어짐.

**MyBatis에서 없는 이유:**
MyBatis는 SQL을 개발자가 직접 작성해. 학기와 과목을 한 번에 가져오고 싶으면 JOIN 쿼리를 직접 짜면 끝. 자동으로 N번 추가 쿼리가 나가는 구조 자체가 없어. JPA는 지연 로딩으로 연관 엔티티에 접근할 때 자동으로 쿼리를 생성하기 때문에 N+1이 발생하는 거야.

**JPA 해결 방법:**

1. **fetch join**: JPQL에서 `JOIN FETCH`를 써서 연관 엔티티를 한 방에 가져옴.
```java
@Query("SELECT t FROM Term t JOIN FETCH t.courses")
```
→ SQL 1번으로 학기 + 과목 다 가져옴. N+1번 → 1번.

2. **@EntityGraph**: 어노테이션 방식으로 fetch join과 동일한 효과.
```java
@EntityGraph(attributePaths = {"courses"})
List<Term> findAll();
```

(추가: @BatchSize로 N번 쿼리를 batch 크기만큼 묶어 줄이는 방법도 있음)

📖 **Ch08** | 채점: N+1 설명 (2점) + MyBatis 비교 (1점) + 해결 방법 2가지 (각 1점)

### Q30. (4점) String FK 설계의 장단점

**모범 답안:**

**장점:**

1. **UPSERT 단순화**: `save()`만 하면 끝. 연관 엔티티를 먼저 조회해서 세팅할 필요 없음. LMS 동기화 데이터를 그냥 String으로 넣으면 돼.

2. **N+1 문제 원천 차단**: 연관관계 자체가 없으니 지연 로딩, 즉시 로딩, N+1 같은 JPA 성능 함정이 아예 발생하지 않음.

3. **독립적 운영**: 각 엔티티가 독립적이라 한 테이블 변경이 다른 엔티티에 영향 안 줌. 동기화 순서도 자유로움.

**단점:**

1. **JOIN 불편**: `course.getTerm().getTermName()` 같은 객체 탐색 불가. 학기 정보가 필요하면 `termRepository.findById(course.getTermCd())`를 별도로 호출해야 함.

2. **타입 안전성 없음**: String이라서 잘못된 값이 들어가도 컴파일 타임에 못 잡음. 존재하지 않는 termCd를 넣어도 DB FK 제약이 없으면 그냥 저장됨.

3. **중복 조회 코드**: 여러 곳에서 연관 데이터가 필요하면 매번 `findById()` 호출 코드가 중복될 수 있음.

**NexClass에서 이 설계를 선택한 이유:**
LMS에서 가져와서 저장만 하는 동기화 데이터라 복잡한 JOIN이 필요 없고, 연관 데이터를 함께 조회하는 요구사항이 현재 없어서 **YAGNI** 원칙에 따라 String FK를 선택.
