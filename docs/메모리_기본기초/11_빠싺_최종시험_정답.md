### Q1. CPU가 RAM을 거치는 이유 (객관식, 2점)

**정답**: (B)

**해설**:

| 보기 | 왜 틀렸는가 (또는 왜 맞는가) |
|------|-------------------------------|
| (A) | HDD/SSD는 읽기 전용이 아니다. 당연히 쓰기도 된다. 이런 기본도 모르면 01부터 다시 읽어. |
| (B) | **정답.** CPU는 나노초(ns) 단위로 동작하고, HDD는 밀리초(ms) 단위다. 속도 차이가 10만~100만 배. 이 격차를 메우기 위해 중간에 빠른 매체(RAM, 캐시)를 계층적으로 둔다. 이게 메모리 계층 구조의 존재 이유야. |
| (C) | "법적으로 금지"? 그런 거 없다. OS가 관리 차원에서 직접 접근을 제한하는 거지, 법이 아니다. |
| (D) | RAM은 HDD/SSD보다 용량이 **훨씬 작다**. 정반대야. RAM이 비싸고 작으니까 계층 구조가 필요한 거다. |

핵심은 **속도 격차(speed gap)**다. 빠른 매체는 비싸고 작고, 느린 매체는 싸고 크다. 이 트레이드오프가 메모리 계층 구조의 근본 원리야.

### Q2. 가상 메모리의 핵심 이유 (객관식, 2점)

**정답**: (A)

**해설**:

| 보기 | 왜 틀렸는가 (또는 왜 맞는가) |
|------|-------------------------------|
| (A) | **정답.** 가상 메모리의 가장 핵심적인 목적은 **프로세스 격리**다. 각 프로세스가 독립된 주소 공간을 가지면, A 프로세스가 B 프로세스 메모리를 건드릴 수 없다. 물리 메모리보다 큰 주소 공간을 쓸 수 있는 건 부수 효과이지, 핵심 이유가 아니야. |
| (B) | 가상 메모리는 RAM 속도 자체를 바꾸지 않는다. 속도와 무관해. |
| (C) | CPU 캐시 적중률은 가상 메모리와 직접적인 관계가 없다. 지역성(locality)이 캐시 적중률을 결정하지, 가상 메모리가 결정하는 게 아니야. |
| (D) | Swap 쓰면 디스크 I/O가 오히려 **증가**한다. 향상이 아니라 성능 저하 요인이야. |

많은 사람이 "물리 메모리보다 큰 프로그램을 실행할 수 있다"를 핵심 이유로 아는데, 그건 **결과**이지 **목적**이 아니야. 가상 메모리의 근본 목적은 **보호와 격리**다.

### Q3. Stack과 Heap을 분리한 이유 (객관식, 2점)

**정답**: (B)

**해설**:

| 보기 | 왜 틀렸는가 (또는 왜 맞는가) |
|------|-------------------------------|
| (A) | Stack이 Heap보다 용량이 큰 게 아니다. 정반대. Stack은 보통 1~8MB, Heap은 GB 단위까지 간다. |
| (B) | **정답.** Stack은 LIFO 구조로 함수 호출/반환에 맞춰 자동 할당/해제된다. 크기가 컴파일 타임에 결정되니까 매우 빠르다. Heap은 런타임에 동적으로 크기가 결정되고, 수동(또는 GC) 관리가 필요하다. 전략이 근본적으로 다르니까 분리한 거야. |
| (C) | Stack도 Heap도 OS가 관리한다. 하드웨어가 Heap을 관리한다는 건 완전히 틀린 말이야. |
| (D) | Stack도 Heap도 읽기/쓰기 둘 다 된다. |

**근본**: 할당/해제 패턴이 다르면 최적화 전략이 다르다. Stack은 포인터 하나만 움직이면 되니까 O(1), Heap은 빈 공간 찾아야 하니까 느리다. 이 차이를 이해 못 하면 메모리 관리를 이해 못 한 거야.

### Q4. JVM의 Young/Old 구분 근거 (객관식, 2점)

**정답**: (B)

**해설**:

| 보기 | 왜 틀렸는가 (또는 왜 맞는가) |
|------|-------------------------------|
| (A) | "강한 참조 가설"이라는 건 없다. Strong Reference는 참조 타입(Strong/Soft/Weak/Phantom) 중 하나일 뿐이야. |
| (B) | **정답.** 약한 세대 가설(Weak Generational Hypothesis): "대부분의 객체는 금방 죽는다(young die young)." 이 관찰에 기반해서, 금방 죽는 놈들은 Young에서 빠르게 정리하고, 오래 살아남은 놈들만 Old로 보내는 전략을 쓴다. 이게 세대별 GC의 근거야. |
| (C) | 메모리 지역성 가설은 캐시 설계의 근거이지, JVM 세대 구분의 근거가 아니야. |
| (D) | "객체 불변 가설"이라는 건 GC와 무관하다. |

이 가설이 왜 중요하냐? 이걸 모르면 "왜 Eden이 있고, 왜 Survivor가 있고, 왜 Old로 Promotion 하는지" 전체 GC 전략의 이유를 설명 못 해. 가설을 모르면 GC를 모르는 거야.

### Q5. GC Root가 될 수 없는 것 (객관식, 2점) - 함정 문제

**정답**: (C)

**해설**:

| 보기 | 왜 틀렸는가 (또는 왜 맞는가) |
|------|-------------------------------|
| (A) | 스레드의 로컬 변수는 GC Root다. Stack Frame에 살아있으니까. |
| (B) | static 변수는 GC Root다. 클래스가 언로드되지 않는 한 계속 참조를 유지하니까. |
| (C) | **정답. Old Generation의 일반 객체는 GC Root가 아니다.** Old에 있다고 GC Root가 되는 게 아니야. GC Root는 "GC가 탐색을 시작하는 진입점"이지, "오래 살아남은 객체"가 아니다. 이거 헷갈리는 사람 많아. Old에 있어도 아무도 참조 안 하면 GC가 수거해. |
| (D) | JNI 참조는 GC Root다. 네이티브 코드에서 참조하고 있으니까 GC가 함부로 회수 못 해. |

**함정 포인트**: "오래 살아남았으니까 중요하겠지?" -> 아니야. GC Root는 **참조 탐색의 시작점**이다. 스레드 스택, static 변수, JNI 참조처럼 "외부에서 JVM 안을 가리키는 진입점"이야. 객체가 어디에 있느냐(Young/Old)와 GC Root인지 아닌지는 완전히 별개의 문제야.

### Q6. STW에 대한 틀린 설명 (객관식, 2점) - 함정 문제

**정답**: (B)

**해설**:

| 보기 | 왜 틀렸는가 (또는 왜 맞는가) |
|------|-------------------------------|
| (A) | 맞는 설명이다. STW는 GC 동작 중 애플리케이션 스레드가 멈추는 현상이야. |
| (B) | **틀린 설명이므로 정답.** Minor GC에서도 STW는 발생한다! 다만 시간이 매우 짧아서(수 ms) 체감 못 할 뿐이야. "Minor GC는 STW 없다"고 알고 있으면 큰 오해야. 모든 GC에는 STW가 있다. ZGC나 Shenandoah도 극히 짧은 STW는 존재해. |
| (C) | 맞는 설명이다. STW가 길어지면 그동안 요청 처리를 못 하니까 응답 지연이 발생한다. |
| (D) | 맞는 설명이다. G1 GC는 `-XX:MaxGCPauseMillis`로 STW 목표 시간을 설정할 수 있다. |

**이거 많이 틀려.** "Minor GC는 가볍다 -> STW 없다"로 점프하는 건 논리적 비약이야. 가볍다 =/= STW 없다. 짧을 뿐이지 있긴 있어.

### Q7. Java 메모리 누수의 정의 (객관식, 2점)

**정답**: (B)

**해설**:

| 보기 | 왜 틀렸는가 (또는 왜 맞는가) |
|------|-------------------------------|
| (A) | 객체를 많이 생성하는 것 자체는 누수가 아니다. 다 쓰고 GC가 회수하면 문제없어. |
| (B) | **정답.** 메모리 누수 = "쓰지 않는 객체인데 참조가 남아있어서 GC가 회수 못 하는 상태". 핵심은 **불필요한 참조 유지**야. C/C++에서는 free를 안 하는 거고, Java에서는 참조를 안 끊는 거야. |
| (C) | Heap 사이즈를 크게 설정하는 건 설정 문제이지 누수가 아니야. |
| (D) | OOM의 원인은 누수만이 아니다. 한 번에 큰 객체 할당, 설정 미스, 트래픽 폭증 등 다양해. |

**기억해**: 누수는 "메모리를 많이 쓰는 것"이 아니라 "안 쓰는 메모리를 못 돌려주는 것"이야. 이 구분 못 하면 누수를 진단할 수 없어.

### Q8. buff/cache가 높을 때 (객관식, 2점)

**정답**: (B)

**해설**:

| 보기 | 왜 틀렸는가 (또는 왜 맞는가) |
|------|-------------------------------|
| (A) | buff/cache 높다고 무조건 메모리 부족이 아니야. 이건 리눅스 초보가 가장 많이 하는 오해야. |
| (B) | **정답.** 리눅스는 유휴 메모리를 디스크 캐시로 활용한다. 이건 성능 최적화야. 애플리케이션이 메모리를 요청하면 OS가 캐시를 자동으로 회수해서 넘겨줘. 그래서 `free -h`에서 봐야 할 건 `free`가 아니라 `available`이야. |
| (C) | buff/cache는 회수 가능한 영역이다. "절대 회수 안 된다"는 완전히 틀린 말. |
| (D) | buff/cache와 Swap은 완전히 다른 개념이야. buff/cache는 RAM 위, Swap은 디스크 위. |

**리눅스 운영의 기본 중의 기본**: `free`가 0에 가까워도 `available`이 충분하면 괜찮다. `available`이 낮고 Swap을 활발히 쓰고 있으면 그때 걱정해.

### Q9. Xms와 Xmx를 동일하게 설정하는 이유 (객관식, 2점)

**정답**: (B)

**해설**:

| 보기 | 왜 틀렸는가 (또는 왜 맞는가) |
|------|-------------------------------|
| (A) | JVM 시작 속도와는 큰 관련이 없다. 메모리 예약과 실제 할당은 다르니까. |
| (B) | **정답.** Xms < Xmx이면, JVM이 힙을 동적으로 늘리거나 줄이는 과정에서 오버헤드가 발생한다. 특히 힙 확장 시 Full GC가 트리거될 수 있어. 운영 환경에서는 이런 예측 불가능한 GC를 없애기 위해 같은 값으로 고정하는 게 정석이야. |

| (D) | Metaspace는 `-XX:MaxMetaspaceSize`로 별도 설정한다. Xms/Xmx와 무관해. |

### Q10. OOM 발생 시 첫 행동 (객관식, 2점)

**정답**: (C)

**해설**:

| 보기 | 왜 틀렸는가 (또는 왜 맞는가) |
|------|-------------------------------|
| (A) | 원인 파악도 안 하고 코드부터 수정? 그건 도박이야. 원인이 코드가 아닐 수도 있어. |
| (B) | 힙 사이즈 늘리기는 임시 방편이지 해결이 아니야. 누수가 원인이면 4g든 8g든 결국 터져. |
| (C) | **정답.** 장애 대응의 철칙: (1) 증거 확보(힙덤프) -> (2) 영향도 파악 -> (3) 임시 조치 -> (4) 근본 원인 분석 -> (5) 영구 해결. 힙덤프 없이 재시작하면 증거가 사라져. |
| (D) | GC 알고리즘 변경은 원인 분석 후에나 고려하는 거야. 원인도 모르고 바꾸면 뭘 바꾸는 건지도 모르는 거잖아. |

**장애 대응의 핵심**: 증거 먼저, 조치 나중. 재시작하고 싶은 충동을 참고 힙덤프부터 떠. 증거 없으면 원인 분석 못 해.

### Q11. 메모리 계층 구조 순서 (단답형, 3점)

**정답**: 레지스터 -> CPU 캐시(L1/L2/L3) -> RAM(주기억장치) -> HDD/SSD(보조기억장치)

**해설**:

속도 순서대로:
- **레지스터**: ~0.3ns. CPU 내부. 가장 빠르고 가장 작다 (수십~수백 개).
- **CPU 캐시**: L1 ~1ns, L2 ~3-10ns, L3 ~10-20ns. CPU 근처. KB~MB 단위.
- **RAM**: ~100ns. 메인 메모리. GB 단위. 전원 꺼지면 날아간다(휘발성).
- **HDD/SSD**: HDD ~10ms, SSD ~0.1ms. 보조기억장치. TB 단위. 비휘발성.

위로 갈수록 빠르고/작고/비싸고, 아래로 갈수록 느리고/크고/싸다. 이 트레이드오프가 계층 구조의 존재 이유야.

**채점 기준**: 4단계 순서가 맞으면 3점. 순서 하나라도 틀리면 0점. "L1/L2/L3"를 별도로 쓰든 "CPU 캐시"로 묶든 상관없음.

### Q12. Page Fault 정의 (단답형, 3점)

**정답**: CPU가 접근하려는 페이지가 물리 메모리(RAM)에 없어서, 디스크(Swap 등)에서 해당 페이지를 RAM으로 가져와야 하는 상황.

**해설**:

Page Fault는 두 종류가 있다:
- **Minor Page Fault**: 페이지가 이미 RAM에 있지만 페이지 테이블에 매핑이 안 된 경우. 디스크 I/O 없이 해결.
- **Major Page Fault**: 페이지가 RAM에 없어서 디스크에서 읽어와야 하는 경우. 디스크 I/O 발생. 이게 성능에 치명적.

"물리 메모리에 없는 페이지에 접근"이라는 핵심만 들어가면 된다.

**채점 기준**: "접근하려는 페이지가 RAM에 없다"는 핵심이 있으면 3점. "에러가 발생한다"로만 쓰면 1점 (Page Fault는 에러가 아니라 정상적인 메커니즘이야).

### Q13. Stack Overflow 원인 2가지 (단답형, 3점)

**정답**:
1. **무한 재귀 호출** (종료 조건 없는 재귀 함수)
2. **너무 큰 지역 변수 할당** (거대한 배열을 Stack에 선언)

**해설**:

Stack은 크기가 제한되어 있다 (보통 1~8MB). 함수 호출마다 Stack Frame이 쌓이니까:
- 재귀가 끝없이 돌면 Stack Frame이 무한히 쌓여서 Stack 공간을 초과한다.
- 함수 하나에서 `int arr[10000000]` 같은 거대한 지역 변수를 선언하면 Stack Frame 하나가 Stack 전체보다 클 수 있다.

Java에서는 `StackOverflowError`, C에서는 Segmentation Fault로 터진다.

**채점 기준**: 2가지 모두 맞으면 3점, 1가지만 맞으면 1.5점. "메모리가 부족해서"처럼 모호하게 쓰면 0점.

### Q14. PermGen 대체 영역 (단답형, 3점)

**정답**: Metaspace, Heap 외부(Native Memory)에 존재한다.

**해설**:

- Java 7 이전: PermGen (Permanent Generation). Heap 내부에 존재. 고정 크기.
- Java 8 이후: Metaspace. Heap **외부**(Native Memory)에 존재. 기본적으로 OS가 허용하는 만큼 동적으로 증가.

왜 바꿨냐? PermGen은 크기가 고정이라 클래스가 많으면 `OutOfMemoryError: PermGen space`가 터졌다. 특히 동적 클래스 로딩이 많은 환경(JSP, 리플렉션 등)에서 문제가 잦았어. Metaspace는 Native Memory를 쓰니까 이 제약에서 벗어났다.

**채점 기준**: "Metaspace" + "Heap 외부(또는 Native Memory)" 둘 다 있어야 3점. "Metaspace"만 쓰면 2점.

### Q15. static 컬렉션의 누수 이유 (단답형, 3점)

**정답**: static 컬렉션은 클래스 수명과 같으므로 프로그램이 종료될 때까지 GC Root에서 도달 가능하며, 컬렉션에 추가된 객체들의 참조가 영원히 유지되어 GC가 회수할 수 없기 때문이다.

**해설**:

static 변수는 GC Root다. GC Root에서 도달 가능한 객체는 절대 회수 안 된다.

```
GC Root (static List)
  └─→ 객체1 (안 쓰지만 참조 유지)
  └─→ 객체2 (안 쓰지만 참조 유지)
  └─→ 객체3 (안 쓰지만 참조 유지)
  └─→ ... 계속 쌓임
```

add만 하고 remove를 안 하면, 컬렉션 크기가 무한히 증가한다. static이니까 프로그램 종료 전까지 절대 안 없어져. 이게 Java에서 가장 전형적인 메모리 누수 패턴이야.

**채점 기준**: "static이라 GC Root에서 도달 가능" + "참조가 영원히 유지" 두 핵심이 있으면 3점. "static이라 안 없어져서"만 쓰면 1.5점 (왜 안 없어지는지 원리를 안 쓴 거야).

-> 06_메모리_누수_패턴.md 참조

### Q19. EventManager 메모리 누수 (코드 분석, 5점)

**정답**:

**(1) 누수 지점**: `static final List<EventListener> listeners`에 익명 클래스 리스너가 추가되지만, 제거되지 않는 부분.

**(2) GC가 회수 못 하는 이유**:
- `listeners`는 `static`이므로 GC Root다.
- `UserSession.init()`에서 익명 클래스로 생성한 `EventListener`가 `listeners`에 추가된다.
- 익명 클래스는 외부 클래스(`UserSession`)에 대한 **암묵적 참조(implicit reference)**를 가진다.
- `UserSession`이 종료되어도, `listeners`가 `EventListener`를 참조하고, `EventListener`가 `UserSession`을 참조하므로 GC가 `UserSession`을 회수할 수 없다.
- 참조 체인: `GC Root → static listeners → EventListener → UserSession`

**(3) 해결 방법**:
- `removeListener()` 메서드를 추가하고, `UserSession` 종료 시 반드시 리스너를 제거한다.
- 또는 `WeakReference`를 사용하여 리스너를 약한 참조로 보관한다.
- 또는 익명 클래스 대신 `static` 내부 클래스를 사용하여 외부 클래스 참조를 끊는다.

**해설**:

이건 실무에서 가장 흔한 누수 패턴 중 하나야. Observer/Listener 패턴에서 **등록만 하고 해제를 안 하는 것**. 특히 익명 클래스의 암묵적 외부 참조를 모르면 왜 누수가 나는지 이해도 못 해.

**채점 기준**:
- 누수 지점 정확히 지목: 1점
- "static이라 GC Root" + "참조가 안 끊김" 설명: 2점
- 익명 클래스의 암묵적 외부 참조 언급: 1점
- 해결 방법 제시: 1점

### Q20. 참조 추적과 GC 회수 판단 (코드 분석, 5점)

**정답**:

**(1) 메서드 종료 직전 각 객체의 참조 상태**:

```
초기:  a → 객체A,  b → 객체B,  c → 객체C

a = b;   // a가 객체B를 가리킴. 객체A를 가리키는 변수가 없어짐.
         // a → 객체B,  b → 객체B,  c → 객체C

b = null; // b가 아무것도 안 가리킴.
          // a → 객체B,  b → null,  c → 객체C

c = a;   // c가 객체B를 가리킴. 객체C를 가리키는 변수가 없어짐.
         // a → 객체B,  b → null,  c → 객체B
```

**(2) GC가 회수 가능한 객체**: 객체A, 객체C

**(3) 이유**:
- **객체A**: `a = b;` 이후 아무 변수도 객체A를 참조하지 않는다. Unreachable. GC 회수 가능.
- **객체B**: `a`와 `c`가 모두 객체B를 참조하고 있다. Reachable. GC 회수 불가.
- **객체C**: `c = a;` 이후 아무 변수도 객체C를 참조하지 않는다. Unreachable. GC 회수 가능.

**해설**:

참조 변수와 실제 객체를 구분 못 하면 이 문제 못 푼다. `a = b`는 "a라는 변수가 b가 가리키는 객체를 가리키게 바꾼다"이지, "객체A를 객체B로 바꾼다"가 아니야. 변수는 **리모컨**이고, 객체는 **TV**야. 리모컨을 다른 TV에 연결해도 원래 TV는 그대로 있어 -- 그냥 조종할 리모컨이 없어진 것뿐.

**채점 기준**:
- 각 단계별 참조 상태 정확히 추적: 2점
- 회수 가능 객체 정확히 선택 (A, C): 1.5점
- Reachable/Unreachable 이유 설명: 1.5점

### Q21. JVM 메모리 영역 매핑 (코드 분석, 5점)

**정답**:

| 번호 | 요소 | 저장 위치 | 설명 |
|------|------|-----------|------|
| (1) | `GREETING` (static final String) | **Metaspace**(변수 참조) + **Heap**(String Pool의 "Hello" 객체) | static 변수의 메타 정보는 Metaspace, 문자열 리터럴 객체는 Heap의 String Pool에 저장 |
| (2) | `count` (인스턴스 변수) | **Heap** | Example 객체의 일부. 객체가 Heap에 있으니 인스턴스 변수도 Heap |
| (3) | `localVar` (지역 변수, int) | **Stack** | 메서드의 Stack Frame에 저장. 원시 타입이라 값 자체가 Stack에 |

- **실제 객체(TV)**: 항상 Heap에 있다.
- **static 정보**: Metaspace에 있다.
- **원시 타입 지역 변수**: 값 자체가 Stack에 있다 (참조가 아니라 값).

이 구분을 못 하면 "어디에 저장돼요?" 질문에 영원히 못 답해.

### Q22. free -h 출력 해석 (코드 분석, 5점)

**정답**:

**(1) 실제 사용 가능한 메모리**: `available` 값인 **1.8Gi**.
- `free`(256Mi)가 아니라 `available`(1.8Gi)을 봐야 한다. available은 free + 회수 가능한 buff/cache를 합산한 값이야.

**채점 기준**:
- (1) available 정확히 답변: 1점 (free라고 쓰면 0점)
- (2) 부족 판단 + 근거 2개: 2점
- (3) Swap 의미 설명: 2점

### Q23. ResourceHandler 누수 (코드 분석, 5점)

**정답**:

**(1) 누수 패턴 이름**: 리소스 미해제 (Resource Leak) / Close 누락

**(2) 누수 발생 상황**:
- `reader.readLine()` 도중 예외가 발생하면, `reader.close()`가 호출되지 않는다.
- 정상 흐름에서도 `close()`를 아예 호출하지 않고 있다.
- `BufferedReader`와 `FileReader`가 보유한 OS 파일 디스크립터와 내부 버퍼 메모리가 해제되지 않는다.
- 이런 호출이 반복되면 파일 디스크립터가 고갈되어 `Too many open files` 에러가 발생한다.

**(3) 수정된 코드**:

```java
public class ResourceHandler {
    public String readFile(String path) throws IOException {
        // try-with-resources: 블록 종료 시 자동으로 close() 호출
        try (BufferedReader reader = new BufferedReader(new FileReader(path))) {
            StringBuilder content = new StringBuilder();
            String line;
            while ((line = reader.readLine()) != null) {
                content.append(line);
            }
            return content.toString();
        }
        // 정상 종료든 예외 발생이든 reader.close()가 자동 호출됨
    }
}
```

**해설**:

Java 7 이후로는 `AutoCloseable`을 구현한 리소스는 반드시 try-with-resources로 써야 한다. `try-finally`로 직접 close하는 것보다 안전하고 코드도 깔끔해. 이 패턴을 모르면 파일, DB 커넥션, 네트워크 소켓 전부 누수 낼 수 있어.

**채점 기준**:
- 패턴 이름: 1점
- 누수 발생 상황 설명 (예외 시 close 안 됨): 2점
- try-with-resources로 수정: 2점 (try-finally도 정답 인정)

### Q24. JVM 옵션 해석 (코드 분석, 5점)

**정답**:

**(1) 각 옵션의 의미**:

| 옵션 | 의미 |
|------|------|
| `-Xms4g` | 초기 힙 크기 4GB. JVM 시작 시 확보하는 힙 메모리 양. |
| `-Xmx4g` | 최대 힙 크기 4GB. 힙이 이 이상 커질 수 없다. |
| `-XX:+UseG1GC` | G1 GC 알고리즘을 사용한다. |
| `-XX:MaxGCPauseMillis=200` | GC 일시정지(STW) 목표 시간을 200ms로 설정. G1이 이 목표를 맞추려고 노력한다 (보장은 아님). |
| `-XX:+HeapDumpOnOutOfMemoryError` | OOM 발생 시 자동으로 힙덤프를 생성한다. |
| `-XX:HeapDumpPath=/var/log/dumps/` | 힙덤프 파일 저장 경로를 `/var/log/dumps/`로 지정한다. |

**(2) Xms = Xmx 동일 설정 이유**:
힙 크기를 동적으로 조절하는 과정에서 발생하는 오버헤드를 방지하기 위해서. 힙 확장 시 Full GC가 트리거될 수 있고, 축소/확장 자체가 CPU를 소모한다. 운영 환경에서는 메모리를 미리 확보해두고 예측 가능하게 운영하는 게 정석이야.

**(3) G1 GC를 선택한 적절한 상황**:
힙 사이즈가 4GB 이상이고, 응답 지연 시간에 민감한 웹 애플리케이션 서버. G1은 힙을 Region으로 나눠서 부분적으로 수집하기 때문에 대용량 힙에서도 STW 시간을 일정 범위 내로 유지할 수 있다.

**채점 기준**:
- (1) 6개 옵션 의미: 각 0.5점 = 3점
- (2) Xms=Xmx 이유: 1점
- (3) G1 선택 상황 + 근거: 1점

### Q25. C 메모리 영역 매핑 (코드 분석, 5점)

**정답**:

| 번호 | 변수 | 메모리 영역 | 이유 |
|------|------|-------------|------|
| (1) | `globalVar = 100` | **Data** (초기화된 데이터 영역) | 초기화된 전역 변수. 프로그램 시작 시 값이 정해져 있으므로 Data 섹션. |
| (2) | `uninitGlobal` | **BSS** (미초기화 데이터 영역) | 초기화되지 않은 전역 변수. 0으로 초기화됨. BSS는 실행 파일 크기에 포함 안 됨. |
| (3) | `"constant"` (문자열 리터럴) | **Code(Text) / Read-only Data** | 문자열 리터럴은 읽기 전용 영역에 저장된다. 포인터 변수 `STR` 자체는 Data 영역. |
| (4) | `localVar` | **Stack** | 지역 변수. 함수 호출 시 Stack Frame에 할당, 반환 시 자동 해제. |
| (5) | `malloc` 할당 메모리 | **Heap** | 동적 할당. 포인터 변수 `heapVar`는 Stack, 실제 할당된 메모리는 Heap. |
| (6) | `staticVar` | **Data** | static 지역 변수는 함수 내부에 있어도 Data 영역에 저장된다. 함수 호출 간에 값이 유지되니까. |

**해설**:

이 문제에서 많이 틀리는 포인트:
- **(2)** Data와 BSS를 구분 못 하는 경우. 초기화 했으면 Data, 안 했으면 BSS. BSS가 왜 따로 있냐? 실행 파일에서 "이만큼 0으로 채울 영역이 필요해"라고 크기만 기록하면 되니까 파일 크기를 줄일 수 있어.
- **(3)** 문자열 리터럴과 포인터 변수를 구분 못 하는 경우. `STR`이라는 포인터는 Data, 가리키는 "constant"라는 문자열은 읽기 전용 영역.
- **(5)** `heapVar`와 `malloc` 결과를 구분 못 하는 경우. 포인터는 Stack, 가리키는 메모리는 Heap.
- **(6)** static 지역 변수를 Stack으로 착각하는 경우. "함수 안에 있으니까 Stack이겠지?" -> 아니야. static이 붙으면 Data야.

**채점 기준**: 6개 중 맞은 개수에 비례. (2)에서 "Data"라고만 쓰면 0.5점. (5)에서 포인터와 할당 영역을 구분하면 추가 0.5점.

### Q26. Thrashing 원인/증상/해결 (서술형, 4점)

**정답**:

**원인**: 물리 메모리가 부족한 상태에서 너무 많은 프로세스가 동시에 실행되면, 각 프로세스에 할당되는 물리 메모리(Resident Set)가 작아진다. 그러면 Page Fault가 빈번해지고, OS가 페이지를 Swap Out/In 하느라 대부분의 시간을 소비하게 된다. CPU는 실제 작업 대신 페이지 교체에만 매달리는 상태가 된다.

**증상**:
- CPU 사용률이 높은데(커널 모드) 실제 작업 처리량(throughput)은 극히 낮다
- 디스크 I/O가 폭증한다 (Swap 활동 급증)
- 시스템 전체가 극도로 느려진다
- 프로세스 응답 시간이 수십~수백 배로 늘어난다

**해결 방법**:
1. **물리 메모리 증설**: 근본 해결. 메모리가 부족하니 늘린다.
2. **동시 실행 프로세스 수 줄이기**: Working Set을 줄여서 각 프로세스에 충분한 메모리를 확보한다.
3. (추가) Swap 크기를 조절하거나, 메모리를 많이 쓰는 불필요한 프로세스를 종료한다.

**채점 기준**:
- 원인 설명 (Page Fault 빈발 + 실작업 못 함): 1.5점
- 증상 2가지 이상: 1점
- 해결 방법 2가지: 1.5점
- "메모리 부족해서 느려요"만 쓰면 1점 (Page Fault, Swap 교체 등 메커니즘 설명 없으면 깊이 부족)

### Q27. 객체의 일생 서술 (서술형, 5점)

**정답 (모범 답안)**:

1. **탄생 (Eden)**: `new`로 객체가 생성되면 Young Generation의 **Eden** 영역에 배치된다.

2. **첫 번째 시련 (Minor GC)**: Eden이 가득 차면 **Minor GC**가 발생한다. GC Root에서 도달 가능한(Reachable) 객체만 살아남고, 나머지는 즉시 회수된다. 살아남은 객체는 **Survivor** 영역(S0 또는 S1)으로 복사되고, **Age가 1**이 된다.

3. **반복 생존 (Survivor 왕복)**: 다음 Minor GC가 발생하면, Eden의 생존자 + 현재 Survivor의 생존자가 반대쪽 Survivor로 복사된다. 매번 생존할 때마다 **Age가 1씩 증가**한다. 한쪽 Survivor는 항상 비어있는 상태를 유지한다.

4. **승격 (Promotion)**: Age가 임계값(기본 15, `-XX:MaxTenuringThreshold`)에 도달하면, 객체는 **Old Generation**으로 **Promotion(승격)**된다. "이 객체는 오래 살아남았으니 Young에서 계속 검사하는 건 비효율적이다"라는 판단.

5. **최종 심판 (Major GC / Full GC)**: Old Generation이 가득 차면 **Major GC**(또는 Full GC)가 발생한다. Old 전체를 스캔해서 Unreachable 객체를 회수한다. 이때 **STW(Stop-The-World)**가 길게 발생할 수 있다.

6. **죽음**: GC Root에서 도달할 수 없는 상태(Unreachable)가 되면, GC가 해당 객체의 메모리를 회수한다. 객체의 일생이 끝난다.

```
new Object()
    ↓
  [Eden] ──Minor GC──→ [Survivor S0] ──Minor GC──→ [Survivor S1]
                              ↑                          ↓
                              └──────Minor GC─────────────┘
                                          │
                                    Age >= 임계값
                                          ↓
                                   [Old Generation]
                                          │
                                    Major/Full GC
                                          ↓
                                      회수 (죽음)
```

**채점 기준**:
- Eden 배치 설명: 0.5점
- Minor GC + Survivor 이동 설명: 1점
- Age 개념 설명: 1점
- Promotion 조건 + Old 이동: 1점
- Major GC / Full GC 설명: 1점
- 7개 필수 키워드 모두 포함: 0.5점
- "그냥 Young에서 Old로 가요"만 쓰면 1점. 과정 없이 결과만 쓰면 깊이 부족.

### Q28. GC 알고리즘 비교 (서술형, 4점)

**정답 (모범 답안)**:

| 구분 | Serial GC | Parallel GC | G1 GC |
|------|-----------|-------------|-------|
| **스레드** | 싱글 스레드 | 멀티 스레드 | 멀티 스레드 |
| **STW** | 길다 | 보통 | 예측 가능 (목표 설정) |
| **처리량** | 낮음 | 높음 | 보통~높음 |
| **적합 환경** | 소규모, 단일 CPU | 대용량 배치, 멀티 CPU | 대용량 힙, 응답 시간 중시 |

**Serial GC** (`-XX:+UseSerialGC`):
- **적합 환경**: 힙이 작고(수백 MB 이하), CPU가 1개인 환경. 임베디드, 클라이언트 애플리케이션.
- **왜**: 싱글 스레드로 동작하니까 멀티 스레드 오버헤드가 없다. 작은 힙에서는 오히려 이게 빠를 수 있어. CPU가 1개인데 멀티 스레드 GC 쓰면 컨텍스트 스위칭만 늘어나서 오히려 손해.

**Parallel GC** (`-XX:+UseParallelGC`):
- **적합 환경**: 배치 처리, 데이터 분석 등 **처리량(throughput)**이 중요하고 응답 지연은 덜 중요한 환경.
- **왜**: 여러 스레드가 동시에 GC를 하니까 전체 GC 시간은 줄어든다. 대신 STW 시간 자체는 길 수 있어. 하지만 배치 작업이면 잠깐 멈추는 건 상관없고, 전체 완료 시간이 중요하니까.

**G1 GC** (`-XX:+UseG1GC`):
- **적합 환경**: 힙이 4GB 이상이고, **응답 지연(latency)**에 민감한 웹 서버, API 서버.
- **왜**: 힙을 Region으로 나눠서 가비지가 많은 Region부터 우선 수집(Garbage First). `-XX:MaxGCPauseMillis`로 STW 목표 시간을 설정할 수 있어서 응답 시간을 예측 가능하게 유지할 수 있다. 대용량 힙을 통째로 수집하는 대신 부분 수집하니까 STW가 짧다.

**채점 기준**:
- 3개 GC 특성 비교: 1.5점
- 각각 적합한 환경 제시: 1.5점
- "왜" 그 환경에서 적합한지 근거: 1점
- 단순히 "Serial은 느리고 G1은 빠르다"만 쓰면 1점. "왜"가 없으면 이해 못 한 거야.

### Q29. OOM 장애 진단 5단계 (서술형, 4점)

**정답 (모범 답안)**:

**1단계: 증거 확보 (힙덤프 확보)**
- 도구: `-XX:+HeapDumpOnOutOfMemoryError`로 사전 설정된 힙덤프 파일 확인
- 또는 `jcmd {PID} GC.heap_dump /path/dump.hprof`로 수동 덤프
- 핵심: 재시작하기 **전에** 반드시 힙덤프를 확보해야 한다.

**2단계: 영향도 파악 + 임시 조치**
- 도구: 모니터링 대시보드, 로그 확인
- 행동: 서비스 영향 범위 파악 -> 필요시 해당 인스턴스 재시작 또는 트래픽 우회
- 핵심: 사용자 영향 최소화가 우선.

**3단계: GC 로그 분석**
- 도구: GC 로그 (`-Xlog:gc*`), `jstat -gcutil {PID}`
- 확인: Full GC 빈도, Old Generation 사용량 추이, GC 후에도 메모리가 안 줄어드는지
- 핵심: "GC가 열심히 뛰는데 메모리가 안 줄면" = 누수 의심

**4단계: 힙덤프 분석**
- 도구: Eclipse MAT (Memory Analyzer Tool), VisualVM
- 확인: Dominator Tree에서 메모리를 가장 많이 차지하는 객체 확인 -> Leak Suspects 리포트 -> 해당 객체의 GC Root까지의 참조 체인 추적
- 핵심: "누가 메모리를 먹고 있는지" + "왜 GC가 회수 못 하는지" 파악

**5단계: 근본 원인 수정 + 재발 방지**
- 행동: 코드 수정 (불필요한 참조 제거, 리소스 해제 등) -> 테스트 -> 배포
- 재발 방지: 모니터링 알림 설정, 코드 리뷰 기준 강화, 정기 힙 분석 스케줄 수립
- 핵심: 같은 장애가 다시 발생하면 안 된다.

**채점 기준**:
- 5단계 순서가 논리적: 1점
- 각 단계에 도구/명령어 포함: 1.5점
- "증거 먼저, 조치 나중" 원칙 반영: 0.5점
- 재발 방지 포함: 1점
- "재시작한다"만 쓰면 0점. 그건 대응이 아니라 도망이야.

### Q30. 메모리 전체 흐름 3문장 요약 (서술형, 4점)

**정답 (모범 답안)**:

물리 메모리(RAM)는 CPU 속도와 디스크 속도 격차를 메우기 위한 중간 계층이며, OS는 가상 메모리를 통해 각 프로세스에 독립된 주소 공간을 부여하고 페이징으로 물리 메모리와 매핑한다. 프로세스 내부에서는 Stack(자동 할당, 지역 변수)과 Heap(동적 할당, 객체)으로 메모리를 나눠 쓰며, JVM은 이 Heap을 다시 Young/Old/Metaspace로 세분화하여 세대별 GC 전략으로 메모리를 자동 관리한다. GC는 약한 세대 가설에 기반해 단명 객체를 Young에서 빠르게 수거하고, 장수 객체만 Old로 보내 Major GC로 정리함으로써 애플리케이션이 메모리 해제를 신경 쓰지 않게 해준다.

```
물리 메모리(RAM)
    ↓ OS가 가상화
가상 메모리 (프로세스별 독립 주소 공간)
    ↓ 프로세스 내부
Stack / Heap / Code / Data (프로세스 메모리 레이아웃)
    ↓ JVM이 Heap을 세분화
Young (Eden + Survivor) / Old / Metaspace
    ↓ GC가 자동 관리
Minor GC (Young) / Major GC (Old) → 메모리 회수
```

**채점 기준**:
- 5개 계층(물리메모리, 가상메모리, 프로세스 구조, JVM 메모리, GC) 모두 포함: 2점
- 각 계층 간 연결이 논리적: 1점
- 3문장 이내 제한 준수: 1점
- 계층 하나라도 빠지면 그만큼 감점. "JVM에 Young이랑 Old가 있어요"만 쓰면 1점. 전체 흐름을 연결 못 하면 부분 점수밖에 못 줘.
