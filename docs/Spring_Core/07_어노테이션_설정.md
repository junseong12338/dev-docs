# 07. 어노테이션 설정

**이 장의 목표**: @Component, @Service, @Autowired, @Resource가 뭔지 정확히 설명할 수 있다

---

## 1. 왜 어노테이션이 나왔나

### 1.1 XML의 한계

```
Bean이 10개일 때: XML 괜찮다
Bean이 100개일 때: XML 지옥

<bean id="userService" class="...UserServiceImpl" />
<bean id="boardService" class="...BoardServiceImpl" />
<bean id="examService" class="...ExamServiceImpl" />
<bean id="lectureService" class="...LectureServiceImpl" />
... (100개)

문제:
  → XML이 수백 줄이 된다
  → 클래스 만들 때마다 XML에도 등록해야 한다
  → 클래스와 XML이 따로 놀아서 싱크가 안 맞을 수 있다
  → 클래스 이름 바꾸면 XML도 바꿔야 한다
```

### 1.2 어노테이션의 등장

```
해결: 클래스에 직접 "나 Bean이야"라고 표시하면 되지 않나?

@Service  // ← "나 Bean이야. 등록해줘."
public class UserServiceImpl {
    // XML에 안 써도 된다!
}

Spring이 이 어노테이션을 발견하면 자동으로 Bean 등록.
XML 없이도 Bean 등록 가능.
```

## 2. Bean 등록 어노테이션 4형제

### 2.1 기본: @Component

```java
@Component
public class MyUtil {
    // Spring이 이 클래스를 Bean으로 등록한다
    // Bean 이름: "myUtil" (클래스명 첫글자 소문자)
}

// 이름 직접 지정 가능
@Component("customName")
public class MyUtil {
    // Bean 이름: "customName"
}
```

### 2.2 역할별 분화

!!! note "Bean 등록 어노테이션 4형제"

    **@Component**
    → 가장 기본. "나 Bean이야"
    → 범용적으로 사용

    **@Service**
    → @Component + "나 서비스 레이어야"
    → 비즈니스 로직 담당 클래스에 붙인다
    → 예: UserServiceImpl, ExamServiceImpl

    **@Repository**
    → @Component + "나 데이터 접근 레이어야"
    → DAO, Repository 클래스에 붙인다
    → 추가 기능: DB 예외를 Spring 예외로 변환

    **@Controller**
    → @Component + "나 웹 컨트롤러야"
    → HTTP 요청 처리 클래스에 붙인다
    → 추가 기능: @RequestMapping 등 사용 가능

### 2.3 4개의 관계

```
@Component ← 부모
  ├── @Service      ← 자식 (Service 전용)
  ├── @Repository   ← 자식 (DAO 전용)
  └── @Controller   ← 자식 (Controller 전용)

4개 다 Bean 등록한다는 건 같다.
차이는 "역할을 명시"하는 것.

@Component 하나만 써도 되지만,
@Service, @Repository, @Controller로 나누면:
  → 코드 읽을 때 "아 이건 서비스 레이어구나" 바로 알 수 있다
  → Spring이 역할별로 추가 기능을 제공할 수 있다
```

## 3. component-scan

### 3.1 어노테이션만으로는 부족하다

```
@Service를 붙였다고 끝이 아니다.
Spring한테 "이 패키지를 스캔해라"고 알려줘야 한다.

이걸 안 하면 Spring은 어노테이션이 붙은 클래스를 찾지 못한다.
```

### 3.2 설정

```xml
<!-- "이 패키지 하위를 스캔해서 어노테이션 붙은 클래스를 Bean으로 등록해라" -->
<context:component-scan base-package="egovframework.mediopia.lxp" />
```

```
동작:
  1. egovframework.mediopia.lxp 패키지와 모든 하위 패키지를 뒤진다
  2. @Component, @Service, @Repository, @Controller가 붙은 클래스를 찾는다
  3. 찾은 클래스를 Bean으로 자동 등록한다

예시:
  egovframework.mediopia.lxp.user.service.impl.UserServiceImpl
  → @Service가 붙어있으면 → Bean으로 등록 (이름: "userServiceImpl")
```

### 3.3 스캔 범위 주의

```
base-package가 넓으면:
  → 필요 없는 클래스도 Bean으로 등록될 수 있다
  → 시작 시간이 느려진다

base-package가 좁으면:
  → 필요한 클래스가 스캔 안 될 수 있다
  → "Bean을 찾을 수 없다" 에러

적절한 범위 설정이 중요하다.
```

## 4. 주입 어노테이션

### 4.1 @Resource (우리 프로젝트 주력)

```java
@Service
public class UserServiceImpl {

    @Resource(name = "sqlSession")
    private SqlSession sqlSession;
    // → "sqlSession"이라는 이름의 Bean을 찾아서 넣어줘

    @Resource(name = "userMapper")
    private UserMapper userMapper;
    // → "userMapper"라는 이름의 Bean을 찾아서 넣어줘
}
```

```
@Resource 규칙:
  1. name 지정하면 → 그 이름으로 Bean을 찾는다
  2. name 생략하면 → 필드명으로 Bean을 찾는다
  3. 이름으로 못 찾으면 → 타입으로 찾는다

소속: javax.annotation (Java 표준, JSR-250)
```

### 4.2 @Autowired

```java
@Service
public class UserServiceImpl {

    @Autowired
    private SqlSession sqlSession;
    // → SqlSession 타입의 Bean을 찾아서 넣어줘
}
```

```
@Autowired 규칙:
  1. 타입으로 Bean을 찾는다
  2. 같은 타입이 여러 개면 이름으로 구분
  3. @Qualifier("이름")로 명시 가능

소속: org.springframework.beans.factory.annotation (Spring 전용)
```

### 4.3 @Resource vs @Autowired 비교

!!! tip "@Resource vs @Autowired 비교"

    **@Resource:**

    - Java 표준 (javax.annotation)
    - 이름으로 먼저 찾는다
    - `@Resource(name = "sqlSession")`
    - 우리 프로젝트에서 주로 사용

    **@Autowired:**

    - Spring 전용
    - 타입으로 먼저 찾는다
    - `@Autowired` + `@Qualifier("sqlSession")`
    - Spring Boot 프로젝트에서 주로 사용

    뭘 쓰든 결과는 같다. Bean을 찾아서 주입한다.
    우리 프로젝트: @Resource 사용 (eGovFramework 컨벤션)

## 5. XML vs 어노테이션

### 5.1 비교

```
같은 Bean 등록을 두 가지 방식으로:

XML 방식:
  <bean id="userService" class="...UserServiceImpl">
      <property name="sqlSession" ref="sqlSession" />
  </bean>

어노테이션 방식:
  @Service("userService")
  public class UserServiceImpl {
      @Resource(name = "sqlSession")
      private SqlSession sqlSession;
  }

+ component-scan 필요:
  <context:component-scan base-package="..." />
```

### 5.2 각각 언제 쓰느냐

!!! note "XML vs 어노테이션 사용 기준"

    **XML이 적합한 경우:**

    - 라이브러리 클래스 (내가 수정 못 하는 코드)
    - 예: BasicDataSource, SqlSessionFactoryBean
    - 어노테이션을 붙일 수 없으니까 XML로 등록

    **어노테이션이 적합한 경우:**

    - 내가 만든 클래스
    - 예: UserServiceImpl, ExamController
    - 클래스에 직접 붙이면 되니까 간편

    **우리 프로젝트:**

    - DataSource, SqlSession → XML
    - Service, Controller → 어노테이션
    - 혼합 사용 (대부분의 프로젝트가 이렇다)

핵심 기준:
  "이 클래스의 소스코드를 수정할 수 있나?"
    → Yes → 어노테이션
    → No  → XML
```

## 6. 우리 프로젝트의 실제 패턴

```java
// Controller (어노테이션)
@Controller
public class ExamController {

    @Resource(name = "examService")
    private ExamService examService;

    @RequestMapping("/exam/list.do")
    public String list(Model model) {
        List<Exam> exams = examService.getList();
        model.addAttribute("exams", exams);
        return "exam/list";
    }
}

// Service (어노테이션)
@Service("examService")
public class ExamServiceImpl implements ExamService {

    @Resource(name = "examMapper")
    private ExamMapper examMapper;

    @Override
    public List<Exam> getList() {
        return examMapper.selectList();
    }
}

// Mapper는 component-scan + MapperConfigurer로 자동 등록
```

```
흐름:
  Controller (어노테이션 등록)
    → @Resource로 Service 주입
  Service (어노테이션 등록)
    → @Resource로 Mapper 주입
  Mapper (자동 스캔 등록)
    → SqlSession Bean 사용 (XML 등록)
  SqlSession (XML 등록)
    → DataSource Bean 주입 (XML 등록)

→ 위쪽(내 코드) = 어노테이션
→ 아래쪽(라이브러리) = XML
→ 혼합 사용
```

## 7. 핵심 정리

!!! abstract "핵심 정리"

    **Bean 등록 어노테이션:**
    @Component (범용) / @Service (서비스 레이어) / @Repository (데이터 접근 레이어) / @Controller (웹 컨트롤러)
    → 4개 다 Bean 등록. 역할만 다르다.

    **component-scan 필수:**
    `<context:component-scan base-package="..." />`
    → 이걸 안 하면 어노테이션을 찾지 못한다

    **주입 어노테이션:**
    @Resource: 이름으로 찾는다 (Java 표준)
    @Autowired: 타입으로 찾는다 (Spring 전용)

    **XML vs 어노테이션:**
    내 코드 → 어노테이션
    라이브러리 → XML
    → 혼합 사용이 일반적

    **다음 장:**
    Spring Context
    → Bean들이 사는 세상의 구조
