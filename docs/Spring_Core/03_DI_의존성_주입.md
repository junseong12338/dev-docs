# 03. DI - 의존성 주입

**이 장의 목표**: "DI가 뭐야?" "왜 직접 new 안 해?"에 정확히 대답할 수 있다

---

## 1. 의존성(Dependency)이 뭐야

### 1.1 정의

```
의존성 = A가 B 없이는 동작할 수 없는 관계

예시:
  자동차는 엔진이 없으면 못 달린다
  → 자동차는 엔진에 "의존"한다

코드:
  OrderService는 OrderRepository 없이 주문을 저장할 수 없다
  → OrderService는 OrderRepository에 "의존"한다
```

### 1.2 코드로 보면

```java
public class OrderService {
    private OrderRepository repository;  // ← 이게 의존성

    public void createOrder(Order order) {
        repository.save(order);  // repository 없으면 동작 불가
    }
}

OrderService → OrderRepository
"OrderService는 OrderRepository에 의존한다"
```

## 2. 의존성을 해결하는 3가지 방법

### 2.1 방법 1: 직접 생성 (new)

```java
public class OrderService {
    // 내가 직접 만든다
    private OrderRepository repository = new OrderRepository();
}
```

```
문제점:

  1. 강한 결합 (Tight Coupling)
     → OrderRepository를 다른 구현체로 바꾸려면?
     → OrderService 코드를 수정해야 한다
     → "OrderRepository"라는 구체 클래스에 묶여 있다

  2. 테스트 불가
     → 가짜(Mock) 객체로 바꿀 수 없다
     → 항상 진짜 DB에 연결해야 테스트 가능

  3. 중복 생성
     → 여러 Service에서 new OrderRepository() 하면
     → 같은 객체가 여러 개 생긴다
```

### 2.2 방법 2: 외부에서 받는다 (DI)

```java
public class OrderService {
    private OrderRepository repository;

    // 외부에서 넣어준다 (생성자로)
    public OrderService(OrderRepository repository) {
        this.repository = repository;
    }
}
```

```
장점:

  1. 느슨한 결합 (Loose Coupling)
     → OrderService는 "누가 넣어주든" 받기만 하면 된다
     → 구현체가 바뀌어도 OrderService 코드 수정 불필요

  2. 테스트 쉬움
     → 가짜 객체를 넣어줄 수 있다
     → new OrderService(mockRepository) → 테스트 가능

  3. 한 곳에서 관리
     → 객체 생성을 한 곳(Spring)에서 관리
```

### 2.3 방법 3: Spring이 자동으로 넣어준다

```
방법 2를 Spring이 자동으로 해주는 것 = DI (Dependency Injection)

내가 할 일:
  "나 이거 필요해"라고 선언만 한다

Spring이 할 일:
  선언된 타입의 Bean을 찾아서 넣어준다
```

## 3. DI (Dependency Injection) = 의존성 주입

### 3.1 정의

```
DI = 의존성 주입
   = 객체가 필요로 하는 의존성을 외부(Spring)에서 넣어주는 것

"주입(Injection)" = "넣어준다"

누가: Spring 컨테이너가
뭘:   Bean(객체)을
어디에: 다른 Bean이 필요로 하는 곳에
어떻게: setter 호출이나 생성자 호출로
```

### 3.2 DI의 3가지 방식

!!! note "DI 주입 방식 3가지"

    **1. 생성자 주입 (Constructor Injection)**
    → 생성자 파라미터로 받는다
    → 가장 권장되는 방식

    **2. Setter 주입 (Setter Injection)**
    → setter 메서드로 받는다
    → XML 설정에서 주로 사용 (우리 프로젝트)

    **3. 필드 주입 (Field Injection)**
    → @Autowired, @Resource로 필드에 직접 주입
    → 코드가 간단하지만 단점도 있다

## 4. 주입 방식 상세

### 4.1 생성자 주입

```java
public class OrderService {
    private final OrderRepository repository;

    // 생성자로 받는다
    public OrderService(OrderRepository repository) {
        this.repository = repository;
    }
}
```

```xml
<!-- XML 설정 -->
<bean id="orderRepository" class="com.example.OrderRepository" />

<bean id="orderService" class="com.example.OrderService">
    <constructor-arg ref="orderRepository" />
</bean>
```

```
Spring이 하는 일:
  OrderRepository repo = new OrderRepository();
  OrderService service = new OrderService(repo);  // 생성자로 주입

장점:
  - 불변(final) 가능 → 한번 넣으면 바꿀 수 없다 (안전)
  - 필수 의존성 보장 → 생성자에 있으니 null일 수 없다
  - 테스트 쉬움

단점:
  - 의존성 많으면 생성자 파라미터가 길어진다
```

### 4.2 Setter 주입

```java
public class SqlSessionFactoryBean {
    private DataSource dataSource;

    // setter로 받는다
    public void setDataSource(DataSource dataSource) {
        this.dataSource = dataSource;
    }
}
```

```xml
<!-- XML 설정 - 우리 프로젝트 방식 -->
<bean id="sqlSession" class="org.mybatis.spring.SqlSessionFactoryBean">
    <property name="dataSource" ref="dataSource" />
</bean>
```

```
Spring이 하는 일:
  SqlSessionFactoryBean sqlSession = new SqlSessionFactoryBean();
  sqlSession.setDataSource(dataSource);  // setter로 주입

핵심:
  <property name="dataSource"> → setDataSource() 호출
  <property name="url"> → setUrl() 호출
  <property name="interval"> → setInterval() 호출

  name → "set" + 첫글자 대문자 = setter 메서드명

장점:
  - 선택적 의존성에 적합 (있어도 되고 없어도 되는 것)
  - XML에서 직관적

단점:
  - null 가능성 (setter 호출 안 하면 null)
  - 불변 불가 (나중에 바꿀 수 있다)
```

### 4.3 필드 주입

```java
public class UserServiceImpl {
    @Resource(name = "sqlSession")
    private SqlSession sqlSession;

    // setter도 생성자도 없다
    // Spring이 리플렉션으로 직접 필드에 넣어준다
}
```

```
Spring이 하는 일:
  UserServiceImpl service = new UserServiceImpl();
  // 리플렉션으로 private 필드에 직접 접근해서 값을 넣는다
  service.sqlSession = sqlSession;  // (리플렉션)

장점:
  - 코드가 가장 간단

단점:
  - 테스트 어려움 (리플렉션 없이는 값을 넣을 수 없다)
  - 의존성이 숨겨진다 (외부에서 뭐가 필요한지 보이지 않는다)
  - 불변 불가
```

## 5. @Resource vs @Autowired

### 5.1 차이

!!! tip "@Resource vs @Autowired"

    **@Resource (javax.annotation):**
    → 이름(name)으로 Bean을 찾는다
    → `@Resource(name = "sqlSession")`
    → "sqlSession"이라는 이름의 Bean을 넣어줘
    → Java 표준 (JSR-250)

    **@Autowired (Spring):**
    → 타입(type)으로 Bean을 찾는다
    → `@Autowired`
    → "SqlSession 타입의 Bean을 찾아서 넣어줘"
    → Spring 전용

    우리 프로젝트: **@Resource를 주로 사용**

### 5.2 검색 순서

```
@Resource:
  1. name으로 찾는다 → @Resource(name = "sqlSession")
  2. name 없으면 필드명으로 찾는다 → private SqlSession sqlSession;
  3. 그래도 없으면 타입으로 찾는다

@Autowired:
  1. 타입으로 찾는다
  2. 같은 타입이 여러 개면 이름으로 구분
  3. @Qualifier("이름")로 명시 가능
```

## 6. DI가 없으면 생기는 문제 (실전)

### 6.1 우리 프로젝트 예시

```
context-sqlMap.xml에서:

<bean id="sqlSession" class="org.mybatis.spring.SqlSessionFactoryBean">
    <property name="dataSource" ref="dataSource" />
</bean>

만약 DI 없이 코드로 했다면:

// 모든 Service에서 이렇게 해야 한다
public class UserServiceImpl {
    private SqlSessionFactory sqlSession;

    public UserServiceImpl() {
        BasicDataSource ds = new BasicDataSource();
        ds.setUrl("jdbc:mariadb://krendb.sldb.iwinv.net:3306/...");
        ds.setUsername("...");
        ds.setPassword("...");

        SqlSessionFactoryBean factory = new SqlSessionFactoryBean();
        factory.setDataSource(ds);
        this.sqlSession = factory.getObject();
    }
}

// ExamServiceImpl에서도 똑같이
// BoardServiceImpl에서도 똑같이
// 100개 Service에서 전부 똑같이

문제:
  → DB 주소 바뀌면? 100군데 다 수정
  → 비밀번호 바뀌면? 100군데 다 수정
  → DataSource 객체가 100개 생성 → 메모리 낭비
  → Connection Pool이 100개 → DB 연결 폭발
```

### 6.2 DI로 해결

```xml
<!-- 한 곳에서 정의 -->
<bean id="dataSource" class="org.apache.commons.dbcp2.BasicDataSource">
    <property name="url" value="jdbc:mariadb://..." />
</bean>

<bean id="sqlSession" class="org.mybatis.spring.SqlSessionFactoryBean">
    <property name="dataSource" ref="dataSource" />
</bean>
```

```java
// 모든 Service에서 그냥 가져다 쓴다
public class UserServiceImpl {
    @Resource(name = "sqlSession")
    private SqlSession sqlSession;
    // 끝. DB 주소? 비밀번호? 몰라도 된다. Spring이 알아서 넣어준다.
}

public class ExamServiceImpl {
    @Resource(name = "sqlSession")
    private SqlSession sqlSession;
    // 같은 sqlSession Bean이 주입된다 (같은 객체)
}
```

```
결과:
  → DB 주소 바뀌면? XML 한 곳만 수정
  → DataSource 객체 1개만 생성 → 메모리 절약
  → Connection Pool 1개 → DB 연결 관리 용이
  → Service는 비즈니스 로직에만 집중
```

## 7. DI의 본질

!!! note "DI의 본질"

    **DI의 본질 = 관심사의 분리 (Separation of Concerns)**

    **Service의 관심사:** 비즈니스 로직
    → "주문을 어떻게 처리할까"
    → "성적을 어떻게 계산할까"

    **Service가 신경 안 써도 되는 것:**
    → DB 연결을 어떻게 만들까
    → 객체를 어떻게 생성할까
    → 어떤 구현체를 쓸까

    이 "신경 안 써도 되는 것"을 Spring이 대신 해주는 게 DI.

    코드로 보면:
    "나 sqlSession 필요해" (선언)
    "어떻게 만드는지는 몰라, 그냥 줘" (Spring이 주입)
    → 깔끔.

## 8. 핵심 정리

!!! abstract "핵심 정리"

    **의존성** = A가 B 없이 못 돌아가는 관계
    **주입** = 외부에서 넣어주는 것
    **DI** = 의존성 주입 = Spring이 Bean을 알아서 넣어준다

    **주입 방식 3가지:**

    - 생성자 주입: constructor-arg (가장 권장)
    - Setter 주입: property (XML에서 주로 사용)
    - 필드 주입: @Resource, @Autowired (간편하지만 단점 있음)

    **@Resource:** 이름으로 찾는다 (Java 표준)
    **@Autowired:** 타입으로 찾는다 (Spring 전용)

    **DI의 본질:** 관심사의 분리
    → 객체 생성/관리는 Spring이
    → 비즈니스 로직은 내가

    **다음 장:**
    XML 설정 완전정복
    → 우리 프로젝트의 XML 설정을 한 줄도 빠짐없이 읽는다
