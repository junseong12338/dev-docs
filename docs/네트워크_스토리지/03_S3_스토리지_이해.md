## 스토리지의 3가지 유형

!!! note "스토리지의 3가지 유형"

    **1. 블록 스토리지 (Block Storage)**

    - HDD/SSD 같은 디스크, 서버에 직접 연결, 빠름, 비쌈
    - 예: AWS EBS, 네 PC의 C드라이브

    **2. 파일 스토리지 (File Storage)**

    - 폴더/파일 구조 (우리가 아는 탐색기), NAS, 네트워크 드라이브
    - 예: NFS, SMB, VOD 서버의 /voddata/

    **3. 오브젝트 스토리지 (Object Storage) -- 오늘 이거!**

    - 파일을 "오브젝트"로 저장, 폴더 개념 없음 (키-값 구조), HTTP API로 접근, 싸고 확장성 좋음
    - 예: AWS S3, GSCDN Storage

## 오브젝트 스토리지의 핵심 개념

### Bucket = 최상위 컨테이너

```
storage.gscdn.com (스토리지 서비스)
  ├── knu9          ← Bucket (네가 본 "폴더")
  ├── other-bucket
  └── another-bucket
```

> **"폴더가 있는데 bucket이라고 부르는 거임?"** ← 네가 한 질문
>
> 👹 좋은 질문이야. 답은: **진짜 폴더가 아니야.**

### 폴더가 아닌 이유

```
일반 파일 시스템:
  /knu9/tideflo/week1_backup/abc.mp4
  → knu9 폴더 → tideflo 폴더 → week1_backup 폴더 → abc.mp4 파일
  → 폴더를 하나씩 타고 들어감

오브젝트 스토리지:
  Bucket: knu9
  Key: "tideflo/week1_backup/abc.mp4"  ← 이게 그냥 "이름"이야
  → 폴더 구조가 아니라, 슬래시(/)가 포함된 하나의 문자열
  → WinSCP가 /를 보고 폴더처럼 보여주는 것뿐
```

| 구분 | 파일 스토리지 | 오브젝트 스토리지 |
|------|-------------|-------------------|
| 구조 | 폴더 > 파일 (트리) | Bucket > Key (플랫) |
| 접근 | 경로 탐색 | HTTP API (GET/PUT) |
| 메타데이터 | 이름, 날짜 정도 | 무제한 커스텀 |
| 수정 | 일부 수정 가능 | 덮어쓰기만 가능 |
| 용도 | 일반 파일 | 대용량, 정적 콘텐츠 |

## S3 API 동작 원리

### 기본 HTTP 메서드

```
PUT    /knu9/tideflo/week1_backup/abc.mp4  → 업로드 (파일 올리기)
GET    /knu9/tideflo/week1_backup/abc.mp4  → 다운로드 (파일 받기)
DELETE /knu9/tideflo/week1_backup/abc.mp4  → 삭제
HEAD   /knu9/tideflo/week1_backup/abc.mp4  → 메타데이터만 확인
GET    /knu9?list-type=2                   → 목록 조회
```

> 👹 **핵심: S3는 결국 HTTP 요청이야.**
> 브라우저가 웹페이지 요청하는 것과 같은 원리.
> 차이점은 **인증 헤더**가 붙는다는 것.

---

## 인증 방식 비교 (오늘의 삽질 원인)

### 방식 1: API Token (Swift 방식)
```
curl -X PUT \
  -H "X-Auth-Token: gAAAAABh6RIA..." \   ← 토큰 하나로 인증
  -T file.mp4 \
  https://storage.gscdn.com/v1/AUTH_.../knu9/file.mp4
```
- 토큰 하나만 보내면 끝
- **문제: 토큰 만료되면 끝.** 오늘 401 에러 원인
- **문제: 읽기 전용 토큰이면 PUT 불가.** 오늘 겪은 문제

### 방식 2: Access Key + Secret Key (S3 방식)
```python
# boto3가 내부적으로 하는 것:
# 1. 요청 내용을 정해진 형식으로 정렬 (Canonical Request)
# 2. Secret Key로 서명 생성 (HMAC-SHA256)
# 3. Authorization 헤더에 서명 첨부

s3 = boto3.client('s3',
    endpoint_url='https://storage.gscdn.com',
    aws_access_key_id='a1a59c...',      # 누구인지
    aws_secret_access_key='edd018...',   # 서명 생성용 비밀키
)
```
- 매 요청마다 서명 새로 생성 → **만료 없음**
- boto3, WinSCP가 자동으로 서명 처리

### 👹 그래서 왜 Cyberduck은 안 됐는데?

```
문제 1: S3 프로토콜 선택
  Cyberduck → GET / (루트 목록 조회)
  → storage.gscdn.com은 Swift 기반
  → Swift가 / 요청을 처리 못 함
  → "Invalid path: /" 에러

문제 2: Swift 프로토콜 선택
  Cyberduck → Keystone 인증 시도
  → 제공받은 건 S3 Key인데 Keystone 인증에 넣음
  → 인증 방식 불일치
  → 로그인 실패

문제 3: WinSCP S3 프로토콜
  WinSCP → virtual-hosted style 시도
  → mediopia.storage.gscdn.com 으로 접속
  → 이런 도메인은 없음
  → 호스트 찾을 수 없음
```

## Virtual-Hosted vs Path Style

> 이게 WinSCP에서 `mediopia.storage.gscdn.com` 에러난 이유야.

```
Virtual-Hosted Style (AWS 기본):
  https://knu9.storage.gscdn.com/tideflo/file.mp4
  → bucket 이름이 서브도메인으로 감
  → DNS에 knu9.storage.gscdn.com이 등록돼 있어야 함
  → GSCDN은 이거 안 됨 ❌

Path Style (GSCDN 방식):
  https://storage.gscdn.com/knu9/tideflo/file.mp4
  → bucket 이름이 URL 경로에 포함
  → DNS 추가 등록 불필요
  → GSCDN은 이걸 씀 ✅
```

```python
# boto3에서 path style 강제 설정
config=Config(s3={'addressing_style': 'path'})
```

> 👹 **이거 모르면 S3 호환 스토리지에서 계속 삽질해.**
> AWS가 아닌 S3 호환 서비스 (GSCDN, MinIO, Ceph 등)는
> 대부분 **path style**만 지원해.

## OpenStack Swift vs AWS S3

> 👹 "storage.gscdn.com이 뭐 기반이야?" → "Swift요" → "그게 뭔데?"

| AWS S3 | OpenStack Swift |
|--------|----------------|
| Amazon이 만듦 | OpenStack 재단이 만듦 |
| 상용 서비스 | 오픈소스 |
| S3 API | Swift API |
| Access Key 인증 | Keystone 토큰 인증 |
| Bucket | Container |
| 글로벌 표준 | 프라이빗 클라우드에 많음 |

GSCDN (GS네오텍):
  내부: OpenStack Swift 기반
  외부: S3 호환 API 제공 (양쪽 다 지원)
  → Swift 토큰으로도 되고, S3 Key로도 됨
  → 근데 토큰이 읽기 전용이라 PUT은 S3 Key로만 가능했음
```
