07. HTTP Live Streaming - 웹 스트리밍의 표준 - Beta¶

1. HLS란 - "이게 뭐야?"¶

유튜브에서 영상 볼 때, 파일 전체를 다운로드하고 나서 재생하나? 아니지. 누르자마자 바로 재생되지.

그게 스트리밍이고, 그 스트리밍을 구현하는 가장 대표적인 프로토콜이 HLS야.

HLS의 정체

HLS (HTTP Live Streaming) = Apple이 2009년에 만든 HTTP 기반 스트리밍 프로토콜. 영상을 작은 조각(세그먼트)으로 잘라서, 일반 웹 서버(HTTP)를 통해 전달하는 방식이야.

비유하자면, 영화 필름 한 통을 통째로 보내는 게 아니라, 4초짜리 짧은 클립을 연달아 보내서 이어붙여 재생하는 거야.

비유는 여기까지. 구조를 보자.

2. 왜 HTTP 기반인가 - "이게 왜 중요해?"¶

HLS 이전에는 RTMP(Real-Time Messaging Protocol)라는 게 있었어. Adobe Flash가 쓰던 프로토콜인데, 이게 문제가 있었어.

문제 RTMP HLS (HTTP 기반)
방화벽 1935번 포트 사용, 기업 방화벽에서 차단 80/443 포트(HTTP/HTTPS), 어디서든 통과
CDN 전용 스트리밍 CDN 필요 일반 CDN 그대로 사용 가능
캐싱 불가 HTTP 캐싱 그대로 활용
SSL 별도 설정 HTTPS 그대로 사용
모바일 Flash 필요 (iOS 불가) 네이티브 지원

이게 HLS가 이긴 결정적 이유야

특별한 서버, 특별한 포트, 특별한 CDN이 필요 없어. 기존 웹 인프라 그대로 쓸 수 있으니까 도입 비용이 거의 없어. Nginx든 Apache든 S3든, HTTP 파일을 서빙할 수 있으면 HLS 스트리밍이 가능해.

3. HLS 동작 원리 - "어떻게 돌아가?"¶

HLS의 핵심은 2가지 파일이야.

.m3u8 파일 (플레이리스트)¶

.m3u8 = 재생 목록

텍스트 파일이야. 어떤 세그먼트를 어떤 순서로 재생할지 적혀있어. 플레이어는 이 파일을 읽고, 거기 적힌 세그먼트를 순서대로 다운로드해서 재생해.

#EXTM3U
#EXT-X-VERSION:3
#EXT-X-TARGETDURATION:4
#EXT-X-MEDIA-SEQUENCE:0
#EXTINF:4.000,
segment_000.ts
#EXTINF:4.000,
segment_001.ts
#EXTINF:4.000,
segment_002.ts
#EXTINF:4.000,
segment_003.ts
#EXTINF:3.500,
segment_004.ts
#EXT-X-ENDLIST

각 줄의 의미:

태그 의미
#EXTM3U "이 파일은 m3u8 플레이리스트야"
#EXT-X-VERSION:3 HLS 프로토콜 버전
#EXT-X-TARGETDURATION:4 세그먼트 최대 길이: 4초
#EXT-X-MEDIA-SEQUENCE:0 첫 번째 세그먼트 번호
#EXTINF:4.000, 다음 세그먼트의 재생 시간: 4초
segment_000.ts 실제 영상 조각 파일
#EXT-X-ENDLIST 재생 목록 끝 (VOD에만 있음, 라이브에는 없어)

.ts 파일 (세그먼트)¶

.ts = 실제 영상 조각

MPEG-2 Transport Stream 형식의 영상/오디오 데이터 파일. 각 세그먼트는 독립적으로 재생 가능한 영상 조각이야.

전체 흐름¶

sequenceDiagram
    participant P as 플레이어 (브라우저)
    participant S as 웹 서버 / CDN

    P->>S: 1. playlist.m3u8 요청
    S-->>P: 2. 플레이리스트 반환 (세그먼트 목록)
    P->>S: 3. segment_000.ts 다운로드
    Note over P: 세그먼트 0 재생 시작
    P->>S: 4. segment_001.ts 다운로드 (미리)
    Note over P: 세그먼트 0 재생 중 + 세그먼트 1 버퍼링
    P->>S: 5. segment_002.ts 다운로드 (미리)
    Note over P: 세그먼트 1 재생 중 + 세그먼트 2 버퍼링

핵심 포인트

플레이어는 현재 세그먼트를 재생하면서 다음 세그먼트를 미리 다운로드해. 이게 버퍼링이야. 세그먼트를 미리 확보해두니까, 네트워크가 잠깐 느려져도 끊기지 않아.

4. 세그먼트 개념 - "왜 4초야?"¶

세그먼트 길이의 의미¶

세그먼트 길이는 트레이드오프야.

세그먼트 길이 장점 단점
짧게 (2초) 지연 시간(latency) 짧음, 화질 전환 빠름 HTTP 요청 많아짐, 오버헤드 증가
표준 (4~6초) 균형 잡힌 성능 -
길게 (10초) HTTP 요청 적음, 인코딩 효율 좋음 지연 시간 길어짐, 화질 전환 느림

세그먼트 길이와 GOP의 관계

04장에서 배운 거 기억나? 세그먼트 경계에 반드시 I-프레임(키프레임)이 와야 해. 세그먼트 4초 + 30fps면 GOP = 120이어야 해. 안 맞추면 세그먼트 시작이 깨져.

Apple의 권장 설정¶

Apple HLS 가이드라인

Apple은 6초 세그먼트를 권장해. 하지만 실전에서는 4초를 많이 써. 이유는 ABR(다음 챕터에서 배워) 전환 속도가 더 빠르기 때문이야.

세그먼트와 탐색 (Seek)¶

사용자가 영상에서 특정 시간으로 점프하면 어떻게 될까?

RTMP는 죽었나?

재생용으로는 죽었어 (Flash 종료). 하지만 송출(ingest) 용도로는 아직 살아있어. OBS에서 유튜브/트위치로 라이브 방송할 때 RTMP로 보내고, 플랫폼이 HLS/DASH로 변환해서 시청자에게 뿌려.

실전에서 대부분 HLS

DRM이 Widevine/PlayReady 필수가 아니라면, HLS가 가장 안전한 선택이야. iOS 네이티브 지원 + 가장 넓은 호환성 + 기존 CDN 사용 가능.

6. 브라우저 호환성 - "hls.js가 뭐야?"¶

Safari vs 나머지 브라우저¶

이거 모르면 삽질해

Safari는 HLS를 네이티브 지원한다. <video> 태그에 m3u8 URL만 넣으면 재생돼. 하지만 Chrome, Firefox, Edge는 네이티브로 HLS를 지원 안 해. JavaScript 라이브러리가 필요해.

브라우저 HLS 네이티브 필요한 것
Safari (macOS/iOS) 지원 없음, 그냥 됨
Chrome 미지원 hls.js 필요
Firefox 미지원 hls.js 필요
Edge 미지원 hls.js 필요

hls.js¶

hls.js = 브라우저에서 HLS를 재생하게 해주는 JavaScript 라이브러리

Video.js 공식 HLS 플러그인이기도 해. MSE(Media Source Extensions) API를 사용해서 m3u8 파싱 → ts 다운로드 → 디코딩을 JavaScript로 처리해.

<!-- hls.js 기본 사용법 -->
<video id="video" controls></video>

<script src="https://cdn.jsdelivr.net/npm/hls.js@latest"></script>
<script>
  var video = document.getElementById('video');
  var videoSrc = 'https://cdn.example.com/video/playlist.m3u8';

  // hls.js 지원 여부 확인
  if (Hls.isSupported()) {
    // Chrome, Firefox, Edge 등
    var hls = new Hls();
    hls.loadSource(videoSrc);       // m3u8 URL 로드
    hls.attachMedia(video);         // video 태그에 연결
    hls.on(Hls.Events.MANIFEST_PARSED, function() {
      video.play();                 // 매니페스트 파싱 완료 후 재생
    });
  } else if (video.canPlayType('application/vnd.apple.mpegurl')) {
    // Safari (네이티브 지원)
    video.src = videoSrc;           // 그냥 URL만 넣으면 됨
    video.addEventListener('loadedmetadata', function() {
      video.play();
    });
  }
</script>

실전에서 hls.js 설정

hls.js는 ABR(적응적 비트레이트)도 자동으로 처리해줘. 네트워크 상태에 따라 화질을 자동 전환하는 로직이 내장돼 있어. 이건 다음 챕터(08)에서 자세히 다뤄.

7. 정리¶

항목 핵심
HLS Apple이 만든 HTTP 기반 스트리밍 프로토콜
HTTP 기반 장점 방화벽 통과, CDN 호환, 캐싱 가능, 별도 서버 불필요
.m3u8 텍스트 기반 재생 목록, 세그먼트 순서와 정보 기록
.ts 실제 영상/오디오 데이터가 담긴 세그먼트 파일
세그먼트 보통 4~6초 단위, GOP와 맞춰야 함
DASH HLS의 경쟁자, Widevine DRM 필요 시 선택
RTMP 재생용으로는 사망, 송출용으로만 생존
hls.js Safari 외 브라우저에서 HLS 재생을 가능하게 하는 JS 라이브러리

한 줄 정리

HLS는 영상을 4초짜리 .ts 조각으로 잘라서 .m3u8 재생 목록과 함께 HTTP로 전달하는 프로토콜이고, Safari 외에는 hls.js가 필요하다.

확인 문제¶

다음 문제를 풀어봐. 답 못 하면 위에서 다시 읽어.

Q1. HLS가 HTTP 기반이라서 얻는 장점 3가지를 말해봐.

Q2. .m3u8 파일과 .ts 파일의 역할을 각각 한 줄로 설명해봐.

Q3. HLS 세그먼트 길이를 2초로 줄이면 어떤 장점과 단점이 생겨?

Q4. Chrome에서 <video src="playlist.m3u8">만 넣으면 재생이 되나? 왜?

Q5. RTMP가 재생 프로토콜로 사망한 이유는 뭐고, 아직 어디서 쓰여?

정답 보기 **A1.** (1) 80/443 포트를 사용해서 **기업 방화벽을 통과**할 수 있다 (2) 일반 CDN을 **그대로 사용** 가능하다 (3) HTTP **캐싱**을 활용할 수 있어서 서버 부하가 줄어든다. (HTTPS 그대로 사용, 별도 서버 불필요 등도 정답) **A2.** .m3u8 = 어떤 세그먼트를 어떤 순서로 재생할지 적힌 **텍스트 기반 재생 목록**. .ts = 실제 영상과 오디오 데이터가 담긴 **영상 조각 파일**. **A3.** 장점: 지연 시간(latency)이 줄어들고 화질 전환(ABR)이 빨라진다. 단점: HTTP 요청 횟수가 2배로 늘어나서 서버 부하와 네트워크 오버헤드가 증가한다. **A4.** 안 된다. Chrome은 HLS를 **네이티브로 지원하지 않기** 때문이다. hls.js 같은 JavaScript 라이브러리를 사용해서 MSE API를 통해 재생해야 한다. Safari만 네이티브 지원한다. **A5.** RTMP는 Adobe Flash에 의존했는데, Flash가 **2020년에 지원 종료**되면서 재생 프로토콜로서 사망했다. 하지만 **라이브 송출(ingest)** 용도로는 아직 쓰인다. OBS에서 유튜브/트위치로 방송할 때 RTMP로 보내고, 플랫폼이 HLS로 변환해서 시청자에게 전달한다.
