03. 두 가지 영상 전달 방식의 차이 - Alpha¶

1. "이게 뭐야?" - 영화 한 편을 보는 두 가지 방법¶

영화 한 편을 본다고 해봐. 방법이 두 가지야.

음식 배달 비유

방법 A - 마트 장보기 (다운로드): 마트에 가서 재료를 전부 사온 다음에 요리 시작. 재료가 다 올 때까지 기다려야 하지만, 한번 사오면 내 맘대로 쓸 수 있어.

방법 B - 뷔페 (스트리밍): 접시에 조금씩 담아오면서 바로 먹기. 기다릴 필요 없이 바로 먹기 시작하고, 더 필요하면 또 가져오면 돼.

비유는 여기까지. 기술적으로 정확히 뭐가 다른지 보자.

정확한 정의

다운로드: 영상 파일 전체를 클라이언트(PC/스마트폰)에 저장한 후 재생하는 방식

스트리밍: 영상을 작은 조각으로 나눠서 실시간으로 전달받으면서 재생하는 방식

2. 프로그레시브 다운로드 - 전체 파일 받으면서 재생¶

"다운로드"라고 하면 "파일 전부 다 받을 때까지 기다려야 하는 거 아냐?" 라고 생각할 수 있어. 근데 현실에서는 프로그레시브 다운로드라는 게 있어.

정확한 정의

프로그레시브 다운로드 = 파일을 처음부터 순서대로 받으면서, 받은 부분까지 재생을 시작하는 방식

파일 전체를 다 받을 때까지 안 기다려도 돼. 앞부분이 도착하면 바로 재생 시작. 하지만 본질은 "파일 다운로드"야. 서버에서 파일 하나를 통째로 보내는 거지.

sequenceDiagram
    participant 브라우저
    participant 서버

    브라우저->>서버: GET /video.mp4 (파일 전체 요청)
    서버-->>브라우저: 파일 데이터 (처음부터 순서대로 전송)
    Note over 브라우저: 받은 부분부터 재생 시작
    서버-->>브라우저: 파일 데이터 (계속 전송...)
    서버-->>브라우저: 파일 데이터 (전송 완료)
    Note over 브라우저: 파일 전체가 로컬에 저장됨

프로그레시브 다운로드의 특징¶

항목 설명
전송 방식 HTTP로 파일 하나를 통째로 전송
서버 부담 낮음 (그냥 파일 보내면 됨)
화질 변환 불가능 (하나의 고정된 화질)
저장 파일 전체가 클라이언트에 저장됨
예시 웹서버에 mp4 올려놓고 <video src="video.mp4">

프로그레시브 다운로드는 스트리밍이 아니야

"받으면서 재생하니까 스트리밍 아냐?" -- 아니야. 보이는 건 비슷해도 내부 동작이 완전히 달라. 프로그레시브 다운로드는 파일 전체를 결국 다 받는 거야. 진짜 스트리밍은 필요한 조각만 받는 거야.

3. 진짜 스트리밍 (HLS/DASH) - 조각내서 실시간 전달¶

정확한 정의

적응적 스트리밍(Adaptive Streaming) = 영상을 작은 조각(segment)으로 나누고, 클라이언트 네트워크 상황에 따라 적절한 화질의 조각을 실시간으로 전달하는 방식

대표적인 프로토콜: HLS (HTTP Live Streaming), DASH (Dynamic Adaptive Streaming over HTTP)

동작 원리¶

sequenceDiagram
    participant 플레이어
    participant 서버

    플레이어->>서버: 플레이리스트 요청 (manifest.m3u8)
    서버-->>플레이어: 플레이리스트 (조각 목록 + 화질 목록)

    플레이어->>서버: 조각 1 요청 (1080p, 10초분)
    서버-->>플레이어: segment_001.ts
    Note over 플레이어: 조각 1 재생 시작

    Note over 플레이어: 네트워크 느려짐 감지
    플레이어->>서버: 조각 2 요청 (720p로 낮춤, 10초분)
    서버-->>플레이어: segment_002.ts
    Note over 플레이어: 조각 2 재생 (화질 낮아짐)

    Note over 플레이어: 네트워크 회복
    플레이어->>서버: 조각 3 요청 (다시 1080p)
    서버-->>플레이어: segment_003.ts

HLS vs DASH¶

항목 HLS DASH
만든 곳 Apple MPEG (국제 표준)
플레이리스트 .m3u8 .mpd
세그먼트 컨테이너 .ts (MPEG-TS) .m4s (fMP4)
브라우저 지원 Safari 네이티브, 나머지는 hls.js 대부분 dash.js 필요
점유율 모바일/iOS 강세 표준이지만 HLS가 더 많이 쓰임

기억할 것

실무에서는 HLS가 사실상 표준이야. Apple이 만들었지만 hls.js 라이브러리 덕분에 거의 모든 브라우저에서 재생 가능해. 이 시리즈에서도 HLS 중심으로 다룰 거야.

4. 각 방식의 장단점 비교¶

항목 프로그레시브 다운로드 스트리밍 (HLS/DASH)
구현 난이도 매우 쉬움 (파일 올리면 끝) 복잡함 (인코딩 + 세그먼트 + 플레이리스트)
서버 비용 낮음 (정적 파일 서빙) 상대적으로 높음 (인코딩 서버 필요)
화질 적응 불가능 (고정 화질) 가능 (네트워크에 따라 자동 전환)
탐색(Seek) 받은 데이터 범위 내에서만 아무 위치나 즉시 이동 가능
파일 보호 어려움 (파일 URL 노출) 상대적으로 쉬움 (조각이라 전체 파일 없음)
대역폭 낭비 있음 (안 본 부분도 받음) 적음 (보는 부분만 받음)
적합한 상황 짧은 영상, 소규모 서비스 긴 영상, 대규모 서비스, 라이브

판단 기준

영상이 짧고(5분 이하) 사용자가 적으면? → 프로그레시브 다운로드로 충분해. 영상이 길고 사용자가 많으면? → 무조건 스트리밍. 라이브 방송? → 스트리밍만 가능.

5. 앞으로 감기, 뒤로 감기 동작 차이¶

"1시간짜리 영상에서 45분 지점으로 건너뛰기" -- 이거 두 방식에서 완전히 다르게 동작해.

프로그레시브 다운로드에서 탐색¶

graph LR
    subgraph "프로그레시브 다운로드"
        A["0분~20분<br/>이미 받은 구간"] -->|탐색 가능| B["20분 지점으로 이동"]
        C["20분~60분<br/>아직 안 받은 구간"] -->|탐색 불가| D["45분 지점?<br/>아직 데이터가 없음"]
    end

프로그레시브 다운로드의 한계

아직 안 받은 구간으로는 이동할 수 없어. 정확히 말하면, HTTP Range 요청을 지원하는 서버라면 건너뛸 수는 있어. 하지만 해당 지점부터 다시 받기 시작하고, 중간에 건너뛴 부분은 다시 받을 수도 있어. 효율이 안 좋지.

스트리밍에서 탐색¶

graph LR
    subgraph "스트리밍 (HLS)"
        E["45분 지점으로 이동 요청"]
        E --> F["플레이리스트에서<br/>45분 근처 조각 번호 찾기"]
        F --> G["해당 조각만 요청<br/>segment_270.ts"]
        G --> H["즉시 재생 시작"]
    end

스트리밍의 강점

플레이리스트에 각 조각의 시간 정보가 다 적혀있어. 45분 지점이면? 플레이리스트에서 해당 시간의 조각 번호를 찾아서 그 조각만 요청하면 돼. 앞의 44분 데이터는 받을 필요가 없어.

비교 정리¶

동작 프로그레시브 다운로드 스트리밍 (HLS)
뒤로 감기 (이미 본 부분) 이미 받은 데이터에서 재생 해당 조각 다시 요청 or 캐시
앞으로 감기 (안 본 부분) Range 요청으로 가능하지만 비효율적 해당 조각만 바로 요청 (효율적)
아무 위치 점프 느리거나 불가능 즉시 가능

6. 캐시 저장 방식¶

영상 데이터는 받으면 어딘가에 임시로 저장돼. 어디에? 어떻게?

프로그레시브 다운로드의 캐시¶

항목 설명
저장 위치 브라우저 디스크 캐시 (또는 메모리 캐시)
저장 단위 파일 전체 (하나의 큰 덩어리)
캐시 키 URL (/video/lecture01.mp4)
재방문 시 캐시에 남아있으면 다시 안 받아도 됨
용량 파일 전체 크기만큼 차지

스트리밍의 캐시¶

항목 설명
저장 위치 브라우저 메모리 (RAM) + 일부 디스크
저장 단위 세그먼트 단위 (10초짜리 조각)
버퍼 관리 현재 재생 위치 근처 조각만 유지, 지나간 조각은 버림
재방문 시 다시 받아야 함 (조각이 휘발성)
용량 재생 중인 구간 근처만 (전체 파일보다 훨씬 적음) 
graph TB
    subgraph "프로그레시브 다운로드 캐시"
        PD["전체 파일 1개<br/>lecture.mp4 (2GB)"]
        PD --> DISK["디스크 캐시에 2GB 저장"]
    end

    subgraph "스트리밍 캐시 (버퍼)"
        S1["조각 47<br/>(현재 재생 중)"]
        S2["조각 48~50<br/>(미리 받아둠)"]
        S3["조각 44~46<br/>(이미 재생, 곧 삭제)"]
        S1 --> RAM["RAM에 수십 MB만 유지"]
        S2 --> RAM
        S3 -.->|삭제| GONE["버려짐"]
    end

핵심 차이

프로그레시브 다운로드는 파일 전체를 캐시에 저장해. 2GB 영상이면 2GB 먹어. 스트리밍은 현재 재생 위치 근처 조각만 메모리에 들고 있어. 수십 MB 수준. 지나간 조각은 버리고 앞으로 나올 조각을 미리 받아두는 슬라이딩 윈도우 방식이야.

그래서 스트리밍이 메모리에 유리해

2시간짜리 영화를 스트리밍으로 보면 RAM 수십 MB만 쓰지만, 프로그레시브 다운로드로 보면 디스크에 수 GB가 쌓여. 모바일에서는 이 차이가 크게 느껴져.

7. 정리¶

한 줄 정리

프로그레시브 다운로드 = 파일 통째로 보내기. 스트리밍 = 조각내서 필요한 것만 보내기. 규모가 커지면 스트리밍이 답이다.

이 챕터에서 반드시 기억할 것¶

개념 한 줄 설명
프로그레시브 다운로드 파일 전체를 받으면서 앞부분부터 재생. 스트리밍이 아님
HLS Apple이 만든 적응적 스트리밍. 영상을 조각내서 전달. 사실상 표준
DASH 국제 표준 적응적 스트리밍. HLS보다 점유율 낮음
적응적 비트레이트 네트워크 상황에 따라 화질을 자동으로 올리거나 낮추는 것
세그먼트 스트리밍에서 영상을 나눈 작은 조각 (보통 2~10초 단위)
플레이리스트 조각들의 목록과 정보를 담은 파일 (.m3u8, .mpd)

확인 문제¶

다음 문제를 풀어봐. 답 못 하면 위에서 다시 읽어.

Q1. 프로그레시브 다운로드와 스트리밍의 근본적인 차이는? "받으면서 재생한다"가 스트리밍의 정의가 아닌 이유를 설명해봐.

Q2. 1시간짜리 강의 영상에서 45분 지점으로 점프할 때, 프로그레시브 다운로드와 HLS 스트리밍 각각 어떻게 동작해?

Q3. HLS와 DASH 중 실무에서 더 많이 쓰이는 건 뭐야? 이유는?

Q4. 스트리밍이 프로그레시브 다운로드보다 메모리(RAM)를 적게 쓰는 이유는?

Q5. 어떤 상황에서 프로그레시브 다운로드가 스트리밍보다 나은 선택이 될 수 있어?

정답 보기

A1. 프로그레시브 다운로드도 "받으면서 재생"하지만, 본질은 파일 전체를 결국 다 받는 거야. 스트리밍은 영상을 작은 조각으로 나눠서 필요한 조각만 받아. 화질 적응도 가능하고, 안 본 부분의 데이터를 낭비하지 않아.

A2. 프로그레시브 다운로드: 45분 지점까지 데이터가 아직 안 도착했으면 기다려야 해. HTTP Range 요청을 쓸 수 있지만, 서버가 해당 지점부터 다시 보내야 해서 비효율적. HLS: 플레이리스트에서 45분 지점의 조각 번호를 찾아서 그 조각만 바로 요청. 앞의 44분 데이터는 받을 필요 없이 즉시 재생 시작.

A3. HLS가 더 많이 쓰여. Apple이 만들었지만 hls.js 라이브러리 덕분에 거의 모든 브라우저에서 재생 가능하고, iOS/Safari에서 네이티브 지원되니까 모바일 커버리지가 높아.

A4. 프로그레시브 다운로드는 파일 전체를 디스크/메모리에 저장해. 2GB 영상이면 2GB 차지. 스트리밍은 현재 재생 위치 근처 조각만 메모리에 들고 있고, 지나간 조각은 버려. 수십 MB 수준만 유지하는 슬라이딩 윈도우 방식이야.

A5. 영상이 짧고(5분 이하), 사용자가 적고, 구현을 빠르게 해야 할 때. 서버에 mp4 파일 올려놓고 <video> 태그로 걸면 끝이니까 인코딩 파이프라인이나 세그먼트 생성 과정이 필요 없어. 비용과 복잡도 면에서 유리해.
