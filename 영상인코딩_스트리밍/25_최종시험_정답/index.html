각 문제의 채점 기준을 참고해서 부분 점수를 매겨. 자기 자신에게 관대하게 채점하면 의미 없어. 80점 이상 합격. 못 넘기면 틀린 문제의 관련 챕터 다시 읽고 재시험.

객관식 (Q1~Q10, 각 2점)¶

Q1. 컨테이너가 아닌 것 (객관식, 2점)¶

정답: C) H.264

해설: H.264는 코덱(압축 알고리즘)이지 컨테이너가 아니다. MP4, MKV, WebM은 모두 데이터를 담는 컨테이너 포맷이다. 컨테이너는 상자, 코덱은 상자 안의 내용물을 압축하는 방식. 이 차이를 못 구분하면 영상 관련 작업에서 계속 삽질한다.

  • A) MP4: 컨테이너 맞음. MPEG 국제 표준
  • B) MKV: 컨테이너 맞음. Matroska 오픈소스
  • C) H.264: 코덱. 컨테이너가 아님
  • D) WebM: 컨테이너 맞음. Google이 만든 웹 최적화 포맷

채점 기준: 정답만 맞으면 2점. 오답 0점.

-> 01장 참조

Q2. 프로그레시브 다운로드 vs HLS (객관식, 2점)¶

정답: C) 프로그레시브 다운로드도 적응적 비트레이트(ABR)를 지원한다

해설: 프로그레시브 다운로드는 하나의 고정된 파일을 통째로 받는 방식이다. 파일이 하나니까 네트워크 상황에 따라 화질을 바꿀 수 없다. ABR은 여러 해상도의 세그먼트를 준비해놓고 클라이언트가 대역폭에 맞는 세그먼트를 선택하는 기술인데, 이건 HLS/DASH 같은 스트리밍 프로토콜에서만 가능하다.

  • A) 맞는 설명. 프로그레시브 다운로드는 결국 파일 전체를 받는다
  • B) 맞는 설명. HLS는 필요한 조각만 받는다
  • C) 틀린 설명. ABR은 스트리밍 전용 기술이다
  • D) 맞는 설명. HLS 재생 흐름의 첫 단계

채점 기준: 정답만 맞으면 2점. 오답 0점.

-> 03장, 08장 참조

Q3. 가장 용량이 작은 프레임 타입 (객관식, 2점)¶

정답: C) B-프레임

해설: B-프레임은 앞쪽과 뒤쪽 프레임을 모두 참조해서 차이점만 저장하기 때문에 가장 작다. I-프레임은 완전한 사진이라 가장 크고, P-프레임은 이전 프레임만 참조해서 중간 크기.

  • A) I-프레임: 가장 크다. 독립적인 완전한 이미지
  • B) P-프레임: 중간. 이전 프레임과의 차이만 저장
  • C) B-프레임: 가장 작다. 양방향 참조
  • D) 크기는 프레임 타입에 따라 확실히 다르다

채점 기준: 정답만 맞으면 2점. 오답 0점.

-> 04장 참조

Q4. 1080p + 1Mbps 인코딩 결과 (객관식, 2점)¶

정답: B) 블록 노이즈(깍두기 현상)가 발생한다

해설: 1080p는 약 200만 픽셀인데, 1Mbps로는 이 픽셀들을 표현할 데이터가 부족하다. 인코더가 디테일을 뭉개버려서 블록 단위로 깨지는 블록 노이즈가 발생한다. 해상도만 높다고 화질이 좋은 게 아니야. 비트레이트가 해상도에 맞게 충분해야 한다.

  • A) 해상도가 자동으로 바뀌지 않는다. 지정한 대로 출력된다
  • B) 정답. 데이터 부족으로 블록 노이즈 발생
  • C) 인코딩 자체는 성공한다. 결과물이 나쁠 뿐
  • D) 비트레이트를 낮추면 파일은 작아진다

채점 기준: 정답만 맞으면 2점. 오답 0점.

-> 06장 참조

Q5. HLS 표준 오디오 코덱 (객관식, 2점)¶

정답: B) AAC

해설: HLS(HTTP Live Streaming) 스펙에서 오디오 코덱으로 AAC를 공식 표준으로 지정하고 있다. MP3도 일부 지원하지만 비표준이고, Opus와 FLAC은 HLS에서 지원하지 않는다.

  • A) MP3: HLS에서 비표준. 일부 호환은 되지만 공식이 아님
  • B) AAC: HLS 공식 표준 오디오 코덱
  • C) Opus: WebM/DASH에서 쓰임. HLS 비지원
  • D) FLAC: 무손실 코덱. 스트리밍에 부적합

채점 기준: 정답만 맞으면 2점. 오답 0점.

Q6. -c copy 옵션 (객관식, 2점)¶

정답: C) 스트림을 디코딩/인코딩 없이 컨테이너만 바꿔서 복사한다

해설: -c copy는 비디오/오디오 스트림을 디코딩하지 않고 그대로 복사한다. 코덱 변환 없이 컨테이너만 바꾸는 거라서 매우 빠르고 품질 손실이 없다. 이걸 리먹싱이라고 한다.

  • A) 파일 자체를 복사하는 게 아니라 스트림을 복사하는 거다
  • B) 재인코딩은 -c:v libx264 같은 옵션일 때 일어난다
  • C) 정답. 디코딩/인코딩 건너뛰고 스트림 그대로 복사
  • D) 코덱 정보 출력은 ffprobe의 역할이다

채점 기준: 정답만 맞으면 2점. 오답 0점.

Q7. #EXT-X-ENDLIST 태그 (객관식, 2점)¶

정답: B) VOD 플레이리스트에만 있고, 라이브에는 없다

해설: #EXT-X-ENDLIST는 "이 플레이리스트의 끝"을 나타낸다. VOD는 영상이 이미 완성됐으니까 끝이 있지만, 라이브 스트리밍은 아직 진행 중이니까 끝 태그가 없다. 라이브가 종료되면 그때 추가된다.

  • A) 라이브 플레이리스트에는 없다. 필수가 아님
  • B) 정답. VOD 전용 태그
  • C) 세그먼트의 끝이 아니라 플레이리스트의 끝이다
  • D) 미디어 플레이리스트에 사용된다. 마스터 플레이리스트 아님

채점 기준: 정답만 맞으면 2점. 오답 0점.

Q8. CDN 캐시 미스 동작 (객관식, 2점)¶

정답: B) 오리진 서버에서 원본을 가져와서 캐싱 후 반환한다

해설: CDN 엣지 서버에 요청한 파일이 캐시에 없으면(캐시 미스), 오리진 서버(Origin)로 요청을 보내서 원본을 가져온다. 가져온 파일을 엣지에 캐싱한 다음 사용자에게 반환한다. 다음 번 같은 요청은 캐시에서 바로 반환된다(캐시 히트).

  • A) 404가 아니라 오리진으로 풀백한다
  • B) 정답. Origin에서 가져와서 캐싱 후 반환
  • C) 기본 동작은 오리진 서버로 요청이다
  • D) 브라우저 캐시와 CDN 캐시는 다른 레이어다

채점 기준: 정답만 맞으면 2점. 오답 0점.

Q9. ABR 화질 전환 주체 (객관식, 2점)¶

정답: C) 클라이언트(HLS.js 등 플레이어)가 대역폭을 측정해서 전환

해설: ABR에서 화질 전환을 결정하는 건 클라이언트 측 플레이어다. 세그먼트를 다운로드하면서 실제 대역폭을 측정하고, 다음 세그먼트를 어떤 해상도로 요청할지 결정한다. 서버는 그냥 요청받은 파일을 보내줄 뿐이야.

  • A) 서버는 파일을 보내줄 뿐, 화질 전환 결정을 하지 않는다
  • B) CDN도 파일을 캐싱해서 전달할 뿐이다
  • C) 정답. 클라이언트가 대역폭 측정 + 화질 결정
  • D) 자동 전환이 ABR의 핵심이다

채점 기준: 정답만 맞으면 2점. 오답 0점.

Q10. 비동기 인코딩 이유 (객관식, 2점)¶

정답: B) 인코딩이 수 분~수십 분 걸리므로 사용자 응답을 블로킹하지 않기 위해

해설: 1시간짜리 영상을 4해상도로 인코딩하면 수십 분 걸릴 수 있다. 업로드 API에서 인코딩까지 동기적으로 처리하면 사용자는 그 시간 동안 응답을 기다려야 한다. 비동기로 처리하면 "업로드 완료" 응답을 바로 보내고, 인코딩은 백그라운드에서 돌린다.

  • A) 비동기와 품질은 관련 없다
  • B) 정답. 사용자 경험을 위한 비동기 처리
  • C) FFmpeg은 동기적으로도 잘 동작한다
  • D) S3 업로드는 동기/비동기 모두 가능하다

채점 기준: 정답만 맞으면 2점. 오답 0점.

단답형 (Q11~Q18, 각 3점)¶

Q11. 코덱 확인 도구 (단답형, 3점)¶

정답: ffprobe

해설: ffprobe는 FFmpeg 패키지에 포함된 미디어 파일 분석 도구다. 파일의 코덱, 해상도, 비트레이트, 프레임레이트, 스트림 정보를 확인할 수 있다.

ffprobe -v quiet -print_format json -show_streams input.mp4

채점 기준:

  • ffprobe 이름 정확히 적으면: 3점
  • "ffmpeg으로 확인"이라고 쓰면: 1점 (ffmpeg은 변환 도구, 분석은 ffprobe)
  • MediaInfo 등 다른 도구를 적으면: 1점 (가능은 하지만 FFmpeg 패키지 표준 도구가 아님)

-> 01장, 09장 참조

Q12. 비압축 1초 크기 (단답형, 3점)¶

정답: 약 180MB (1920 x 1080 x 3 x 30 = 186,624,000 바이트)

해설: 한 프레임 = 1920 x 1080 x 3바이트(RGB) = 6,220,800바이트 = 약 6MB. 30fps이므로 1초 = 6MB x 30 = 약 180MB.

채점 기준:

  • 180MB(또는 약 186MB)와 계산 과정 모두 적으면: 3점
  • 답만 맞으면: 2점
  • 계산 과정은 있지만 답이 틀리면: 1점
  • 완전 틀리면: 0점

-> 01장, 02장 참조

Q13. 키프레임 간격 (단답형, 3점)¶

정답: 4초 (120프레임 / 30fps = 4초)

해설: GOP 120은 120프레임마다 I-프레임이 하나씩 들어간다는 뜻이다. 30fps에서 120프레임 = 4초. 이 값은 HLS 세그먼트 길이(4초)와 맞추기 위한 설정이다.

채점 기준:

  • 4초 + 계산 근거: 3점
  • 4초만 적으면: 2점
  • 틀리면: 0점

-> 04장 참조

Q14. CRF + maxrate 이유 (단답형, 3점)¶

정답: CRF는 복잡한 장면에서 비트레이트가 무제한으로 올라갈 수 있으므로, maxrate로 상한선을 걸어서 스트리밍 시 대역폭 초과를 방지하기 위해서다.

해설: CRF는 품질을 유지하기 위해 필요한 만큼 비트레이트를 쓴다. 복잡한 장면에서 비트레이트가 폭발하면 사용자의 네트워크 대역폭을 초과해서 버퍼링이 걸린다. -maxrate로 비트레이트 상한선을 설정하면 품질도 챙기면서 대역폭 폭발도 방지할 수 있다.

채점 기준:

  • "비트레이트 폭발 방지" + "스트리밍 대역폭" 키워드 포함: 3점
  • "비트레이트 제한" 정도만 적으면: 2점
  • 너무 모호하면: 1점

-> 06장 참조

Q15. scale -2 vs -1 (단답형, 3점)¶

정답: -1은 종횡비를 유지하면서 자동 계산하지만 결과가 홀수일 수 있다. -2는 종횡비 유지 + 짝수를 보장한다. H.264는 해상도가 짝수여야 하므로 -2가 안전하다.

채점 기준:

  • "짝수 보장"과 "H.264 짝수 요구" 모두 언급: 3점
  • "짝수 보장"만 언급: 2점
  • 부분적 이해: 1점

-> 10장 참조

채점 기준:

  • 두 가지 모두 정확: 3점
  • 하나만 맞으면: 2점
  • 부분적 이해: 1점

Q17. DB 폴링의 단점 (단답형, 3점)¶

정답: 폴링 주기만큼 지연이 발생하고, 작업이 없을 때도 계속 DB를 조회하므로 불필요한 부하가 생긴다.

해설: DB 폴링은 주기적으로(예: 5초마다) DB를 조회해서 새 작업이 있는지 확인한다. 작업이 없어도 쿼리가 계속 실행되니까 DB에 부하를 주고, 폴링 주기만큼 작업 시작이 지연된다.

채점 기준:

  • "폴링 부하"와 "지연" 둘 다 언급: 3점
  • 하나만 언급: 2점
  • 모호한 답변: 1점

-> 19장 참조

Q18. 영상 상태 전이 (단답형, 3점)¶

정답: UPLOADED -> ENCODING -> COMPLETED (정상 흐름)

해설: 업로드 완료 시 UPLOADED, 워커가 인코딩 시작하면 ENCODING, 인코딩 성공하면 COMPLETED. 실패 시에는 ENCODING -> FAILED로 전이되고, 재시도 시 FAILED -> ENCODING으로 돌아간다.

채점 기준:

  • 3단계 순서 정확: 3점
  • 2단계만 적거나 순서 하나 틀리면: 2점
  • FAILED까지 포함하면 보너스 없지만 감점도 없음

-> 23장 참조

코드 작성/분석 (Q19~Q25, 각 5점)¶

Q19. 720p 변환 명령어 (코드 작성, 5점)¶

정답:

ffmpeg -i input.mp4 \
  -c:v libx264 \
  -preset medium \
  -crf 23 \
  -vf "scale=1280:720" \
  -c:a aac \
  -b:a 128k \
  output_720p.mp4

채점 기준:

항목 배점
-i input.mp4 입력 지정 0.5점
-c:v libx264 비디오 코덱 1점
-preset medium 프리셋 0.5점
-crf 23 품질 1점
-vf "scale=1280:720" 해상도 1점
-c:a aac -b:a 128k 오디오 1점
  • 옵션 순서는 상관없음
  • scale=1280:-2도 정답 인정
  • 출력 파일명이 다르면 감점 없음

-> 09장, 10장 참조

Q20. -c:v copy + -vf 에러 (코드 분석, 5점)¶

정답: -c:v copy는 비디오 스트림을 디코딩하지 않고 그대로 복사하는 옵션이다. -vf "scale=..."은 디코딩된 프레임에 필터를 적용하는 옵션이다. 디코딩을 하지 않으면 필터를 적용할 프레임이 없으므로 두 옵션이 충돌해서 에러가 발생한다.

채점 기준:

항목 배점
-c:v copy가 디코딩을 건너뛴다는 점 언급 2점
-vf는 디코딩된 프레임에 적용된다는 점 언급 2점
충돌 원인을 논리적으로 연결 1점

-> 10장 참조

Q21. m3u8 HLS 표준 위반 (코드 분석, 5점)¶

정답: #EXT-X-TARGETDURATION:4인데 #EXTINF:6.200000인 세그먼트(segment_002.ts)가 있다. EXTINF 값(6.2초)이 TARGETDURATION(4초)을 초과하므로 HLS 표준 위반이다. 일부 플레이어(특히 iOS Safari)에서 재생 오류가 발생하거나 재생 자체가 안 될 수 있다.

채점 기준:

항목 배점
segment_002.ts의 6.2초가 문제라고 지적 2점
TARGETDURATION(4초) 초과가 원인이라고 설명 2점
재생 오류 가능성 언급 1점
  • TARGETDURATION을 7로 올려야 한다고 답해도 부분 점수 인정 (2점)

-> 11장 참조

Q22. HLS 인코딩 명령어 (코드 작성, 5점)¶

정답:

ffmpeg -i input.mp4 \
  -c:v libx264 \
  -b:v 2500k \
  -maxrate 2800k \
  -bufsize 5600k \
  -vf "scale=1280:720" \
  -c:a aac -b:a 128k \
  -g 120 -keyint_min 120 -sc_threshold 0 \
  -hls_time 4 \
  -hls_playlist_type vod \
  -hls_segment_filename "seg_%03d.ts" \
  720p.m3u8

채점 기준:

항목 배점
코덱 + 비트레이트 (-c:v libx264 -b:v 2500k -maxrate 2800k) 1점
GOP 고정 (-g 120 -keyint_min 120) 1점
HLS 옵션 (-hls_time 4 -hls_playlist_type vod) 1점
세그먼트 파일명 + 출력 1점
오디오 + 해상도 (-c:a aac -b:a 128k, -vf scale) 1점
  • -bufsize 값은 maxrate의 1~2배 범위면 정답 인정
  • -sc_threshold 0 없어도 감점 없음 (필수는 아님)

-> 10장, 11장 참조

Q23. ABR 해상도 선택 (코드 분석, 5점)¶

정답: 480p (854x480) 스트림을 선택한다.

해설: 플레이어는 사용자의 대역폭(2Mbps = 2,000,000bps) 이하인 스트림 중 가장 높은 BANDWIDTH를 선택한다. 각 스트림의 BANDWIDTH를 보면:

  • 360p: 500,000 (2Mbps 이하)
  • 480p: 1,400,000 (2Mbps 이하)
  • 720p: 2,800,000 (2Mbps 초과, 선택 불가)
  • 1080p: 5,000,000 (2Mbps 초과, 선택 불가)

2Mbps 이하이면서 가장 높은 BANDWIDTH는 1,400,000 (480p)이다.

채점 기준:

항목 배점
480p 정답 2점
"대역폭 이하에서 가장 높은 것" 선택 원리 설명 2점
각 스트림 BANDWIDTH와 비교 과정 1점
  • 720p라고 답하면: 0점 (2.8Mbps > 2Mbps라서 대역폭 초과)
  • 360p라고 답하면: 1점 ("이하" 조건은 맞지만 가장 높은 것을 선택해야 함)

-> 08장, 11장 참조

Q24. ffprobe 분석 + 인코딩 주의점 (코드 분석, 5점)¶

정답:

주의해야 할 점:

  1. 오디오 코덱이 AC3(돌비 디지털)이다: HLS는 AAC만 표준 지원한다. -c:a copy를 쓰면 안 되고, 반드시 -c:a aac으로 재인코딩해야 한다.

  2. 비디오 코덱이 HEVC(H.265)이다: 원본이 H.265인데 HLS 호환성을 위해 H.264로 트랜스코딩해야 할 수 있다. H.265 HLS도 가능하지만 브라우저 호환성이 제한적이다.

  3. 4K 60fps 원본이라 인코딩 시간이 매우 길다: 4해상도로 인코딩하면 상당한 서버 리소스가 필요하다. -preset을 fast 이상으로 올려서 시간을 단축할 수 있지만 압축 효율이 떨어진다.

  4. 오디오가 6채널(5.1 서라운드)이다: 웹 스트리밍은 보통 스테레오(2ch)로 다운믹스해야 한다 (-ac 2).

채점 기준:

항목 배점
AC3 → AAC 변환 필요 지적 2점
나머지 주의점 중 1개 이상 지적 2점
근거와 함께 설명 1점
  • 2가지 이상 지적하면 충분. 4가지 중 아무 2개 조합이든 인정

-> 05장, 09장 참조

Q25. 강의 영상 인코딩 설정 (코드 분석, 5점)¶

정답: 설정 B가 적절하다.

해설:

강의 영상은 음성 위주이고 화면 변화가 적다.

비교 항목 설정 A 설정 B 판단
CRF 18 (거의 원본급) 23 (기본값) 강의 영상은 23이면 충분. 18은 과잉 품질
maxrate 8Mbps 5Mbps 화면 변화 적은 강의에 8Mbps는 낭비
오디오 비트레이트 320kbps 128kbps 음성 위주에 320kbps는 과잉. 128kbps로 차이 못 느낌

설정 A는 고품질 음악 영상이나 영화에 적합한 설정이다. 강의 영상에 쓰면 파일 크기만 불필요하게 커지고, 대역폭과 스토리지 비용만 낭비된다. 화질 차이는 사람이 거의 못 느낀다.

채점 기준:

항목 배점
설정 B 선택 1점
CRF 18 vs 23 비교 근거 1.5점
오디오 320k vs 128k 비교 근거 1.5점
"강의 영상 특성"과 연결한 설명 1점
  • 설정 A를 선택하면: 합리적 근거가 있어도 최대 2점

-> 05장, 06장 참조

서술형 (Q26~Q30, 각 4~5점)¶

Q26. 트랜스코딩 10번 반복 (서술형, 4점)¶

정답:

트랜스코딩을 10번 반복하면 화질이 점점 나빠진다.

원인은 손실 압축의 누적이다. 트랜스코딩은 "디코딩 -> 인코딩"을 거치는데, 매번 인코딩할 때 손실 압축이 적용된다. 첫 번째 인코딩에서 인간이 인지 못하는 정보를 제거했는데, 두 번째 인코딩에서 다시 정보를 제거하면 이번에는 원래 유지되던 정보까지 추가로 날아간다. 이게 10번 반복되면 블록 노이즈, 색상 열화, 디테일 손실이 심각해진다.

비유하면 JPEG 이미지를 저장 -> 열기 -> 저장을 10번 반복하면 화질이 처참해지는 것과 같은 원리다.

채점 기준:

항목 배점
"화질이 나빠진다" 결론 1점
손실 압축 누적이 원인임을 설명 1.5점
"디코딩 -> 인코딩" 과정에서 정보 손실이 반복됨 1.5점

-> 02장, 04장 참조

Q27. HLS vs 프로그레시브 다운로드 (서술형, 4점)¶

정답:

HLS가 유리한 상황 3가지:

  1. 긴 영상 + 탐색(seek)이 빈번할 때: 1시간 강의에서 45분 지점으로 점프하면 HLS는 해당 세그먼트만 요청. 프로그레시브 다운로드는 해당 지점까지 데이터가 없으면 기다려야 한다.

  2. 사용자 네트워크 환경이 다양할 때: HLS는 ABR로 네트워크 상황에 따라 화질을 자동 전환한다. 프로그레시브 다운로드는 고정 화질이라 느린 네트워크에서 버퍼링이 걸린다.

  3. 동시 접속자가 많을 때: HLS 세그먼트는 CDN 캐시가 잘 먹는다. 프로그레시브 다운로드는 파일 전체를 보내야 해서 대역폭 낭비가 크다.

프로그레시브 다운로드가 나은 상황 1가지:

짧은 영상(5분 이하), 소규모 사용자, 빠르게 구현해야 할 때. 서버에 mp4 올려놓고 <video> 태그로 걸면 끝이라 인코딩 파이프라인이 필요 없다.

채점 기준:

항목 배점
HLS 유리한 상황 3가지 (각 0.75점) 2.25점
프로그레시브 다운로드가 나은 상황 1가지 1점
구체적 근거와 함께 설명 0.75점
  • 2가지만 적으면 HLS 부분 최대 1.5점

-> 03장, 08장 참조

Q28. GOP 크기와 HLS 세그먼트 (서술형, 5점)¶

정답:

HLS 세그먼트는 키프레임(I-프레임) 경계에서만 잘린다. 세그먼트 길이가 4초이고 30fps이면, 4초 간격으로 정확히 I-프레임이 위치해야 세그먼트가 깔끔하게 4초씩 잘린다. 30fps x 4초 = 120프레임. 따라서 GOP 크기를 120으로 설정한다.

GOP가 맞지 않을 때 (예: GOP 90):

GOP 90이면 3초(90/30)마다 I-프레임이 삽입된다. -hls_time 4로 4초 세그먼트를 만들려고 하면, 4초 지점에 I-프레임이 없으니까 FFmpeg이 가장 가까운 I-프레임(3초 또는 6초 지점)에서 자른다. 결과적으로:

  • 세그먼트 길이가 3초, 6초 등으로 불규칙해진다
  • #EXT-X-TARGETDURATION과 실제 세그먼트 길이가 불일치할 수 있다
  • 일부 플레이어에서 버퍼링이나 재생 오류가 발생한다

채점 기준:

항목 배점
"키프레임 경계에서 세그먼트가 잘린다" 설명 1.5점
120 = 30fps x 4초 계산 근거 1점
GOP 불일치 시 문제점 설명 2점
구체적 예시(GOP 90 → 3초/6초 불규칙) 0.5점

-> 04장, 11장 참조

Q29. CDN 유무 판단 (서술형, 4점)¶

정답:

CDN 없이 가능한 상황:

  • 사용자가 소수(수십 명 이하)이고, 한 지역에 집중되어 있을 때
  • Object Storage가 사용자와 같은 리전에 있어서 물리적 거리가 가까울 때
  • 초기 서비스 런칭 단계에서 트래픽이 적고 비용을 아껴야 할 때

CDN 없이 불가능한(또는 매우 비효율적인) 상황:

  • 동시 접속자가 수백~수천 명 이상일 때. Origin에 부하 집중되어 응답 속도 저하
  • 사용자가 전국 또는 글로벌에 분산되어 있을 때. 물리적 거리로 인한 지연

CDN 도입 시 고려 사항:

  1. 캐시 TTL 설정: VOD 콘텐츠(m3u8, ts)는 변경되지 않으므로 긴 TTL 설정 가능. 하지만 영상을 교체하면 캐시 무효화(purge)가 필요하다.

  2. 비용: CDN은 트래픽(전송량) 기반 과금이다. 영상 스트리밍은 트래픽이 크므로 비용이 꽤 나올 수 있다. 트래픽 예측과 비용 시뮬레이션이 필요하다.

채점 기준:

항목 배점
CDN 없이 가능한 상황 서술 1점
CDN 없이 불가능한 상황 서술 1점
CDN 도입 시 고려 사항 2가지 2점
  • 고려 사항 1개만 적으면: 1점

-> 17장, 20장 참조

Q30. 인코딩 워커 장애 복구 (서술형, 4점)¶

정답:

감지 방법:

  • 타임아웃 기반: 일정 시간(예: 30분) 이상 ENCODING 상태가 유지되면 장애로 판단한다. 스케줄러(또는 cron)가 주기적으로 "ENCODING 상태이면서 updated_at이 30분 이상 지난 레코드"를 조회한다.

  • Heartbeat 기반: 워커가 인코딩 중에 주기적으로(예: 30초마다) DB에 heartbeat(타임스탬프 갱신)를 보낸다. heartbeat가 일정 시간 끊기면 장애로 판단한다.

복구 방법:

  1. 해당 레코드의 상태를 FAILED로 변경한다.
  2. 재시도 횟수(retry_count)를 확인해서 한도 내면 다시 큐에 넣는다.
  3. 재시도 한도를 초과하면 관리자에게 알림을 보내고 수동 확인 대기 상태로 둔다.
  4. 임시 파일(인코딩 중간 결과물)이 있으면 정리(cleanup) 한다.

채점 기준:

항목 배점
감지 방법 (타임아웃 또는 heartbeat) 1.5점
FAILED 상태 변경 + 재시도 큐 등록 1.5점
재시도 한도, 알림, 임시 파일 정리 중 1개 이상 1점
  • 감지 방법 없이 복구만 적으면: 최대 2점
