09. FFmpeg 기초 - Gamma¶

1. FFmpeg이 뭐야?¶

영상을 변환하고, 자르고, 합치고, 스트리밍하는 만능 커맨드라인 도구다.

GUI 없이 터미널에서 영상 작업을 전부 할 수 있어. 유튜브, 넷플릭스, VLC 같은 서비스들도 내부적으로 FFmpeg을 쓰거나 FFmpeg 기반 라이브러리를 활용한다.

핵심 정의

FFmpeg = 입력 파일을 받아서, 원하는 형태로 가공해서, 출력 파일로 내보내는 도구

  • 비디오/오디오 인코딩, 디코딩, 트랜스코딩
  • 포맷 변환 (MP4 → AVI, MKV → MP4 등)
  • 스트리밍 생성 (HLS, DASH)
  • 필터 적용 (리사이즈, 크롭, 워터마크)

2. 설치¶

sudo apt update
sudo apt install ffmpeg

# EPEL + RPM Fusion 활성화 후
sudo yum install ffmpeg ffmpeg-devel

brew install ffmpeg

1. https://ffmpeg.org/download.html 에서 빌드 다운로드
2. 압축 해제 후 bin 폴더를 PATH에 추가
3. cmd에서 ffmpeg -version 확인

설치 확인:

# 버전 확인
ffmpeg -version

# 지원 코덱 목록
ffmpeg -codecs

# 지원 포맷 목록
ffmpeg -formats

3. FFmpeg의 기본 구조 - 파이프라인¶

FFmpeg의 핵심 구조는 Input → Demux → Decode → Filter → Encode → Mux → Output 이다.

graph LR
    A[Input 파일] --> B[Demuxer<br/>컨테이너 분해]
    B --> C[Decoder<br/>코덱 디코딩]
    C --> D[Filter<br/>가공/변환]
    D --> E[Encoder<br/>코덱 인코딩]
    E --> F[Muxer<br/>컨테이너 조립]
    F --> G[Output 파일]

각 단계가 뭘 하는지 뜯어보자:

단계 역할 예시
Demuxer 컨테이너(MP4, MKV) 껍데기를 벗겨서 비디오/오디오 스트림 분리 MP4에서 H.264 비디오 + AAC 오디오 꺼냄
Decoder 압축된 스트림을 원본 프레임으로 복원 H.264 → 원본 YUV 프레임
Filter 프레임 단위로 가공 (리사이즈, 크롭 등) 1920x1080 → 1280x720
Encoder 가공된 프레임을 다시 압축 YUV 프레임 → H.264
Muxer 압축된 스트림을 컨테이너에 담기 H.264 + AAC → MP4

이거 중요

코덱 변환 없이 컨테이너만 바꿀 때는 Decode/Encode를 건너뛸 수 있어. 이걸 스트림 복사(-c copy)라고 하고, 엄청 빠르다.

4. 기본 명령어 패턴¶

FFmpeg의 명령어는 이 패턴을 벗어나지 않는다:

ffmpeg [글로벌옵션] [입력옵션] -i 입력파일 [출력옵션] 출력파일

가장 단순한 변환¶

# MP4 → AVI 변환 (코덱 자동 선택)
ffmpeg -i input.mp4 output.avi

이러면 FFmpeg이 알아서 출력 포맷에 맞는 코덱을 선택한다. 근데 실무에서 이렇게 쓰면 안 돼. 코덱을 명시적으로 지정해야 예측 가능한 결과가 나온다.

코덱 지정 변환¶

# H.264 비디오 + AAC 오디오로 변환
ffmpeg -i input.mp4 -c:v libx264 -c:a aac output.mp4
옵션 의미
-i input.mp4 입력 파일 지정
-c:v libx264 비디오 코덱 = H.264
-c:a aac 오디오 코덱 = AAC

스트림 복사 (재인코딩 없이 컨테이너만 변환)¶

# MKV → MP4 (코덱 변환 없음, 컨테이너만 변경)
ffmpeg -i input.mkv -c copy output.mp4

스트림 복사 vs 재인코딩

  • -c copy: 코덱 그대로, 컨테이너만 바꿈. 초 단위로 끝남
  • -c:v libx264: 디코딩 → 인코딩. 분~시간 단위 소요

스트림 복사가 가능한 조건: 원본 코덱이 대상 컨테이너에서 지원되어야 함

5. ffmpeg vs ffprobe vs ffplay¶

FFmpeg 패키지에는 3개의 실행 파일이 들어있다. 각각 용도가 완전 다르다.

도구 역할 언제 쓰나
ffmpeg 변환/인코딩/스트리밍 생성 영상을 가공할 때
ffprobe 파일 정보 분석 영상을 분석할 때
ffplay 재생 영상을 확인할 때

ffprobe - 파일 정보 뜯어보기¶

# 기본 정보 출력
ffprobe input.mp4

# JSON 포맷으로 상세 정보
ffprobe -v quiet -print_format json -show_streams -show_format input.mp4

출력 예시:

{
  "streams": [
    {
      "codec_name": "h264",
      "width": 1920,
      "height": 1080,
      "r_frame_rate": "30/1",
      "bit_rate": "4500000"
    },
    {
      "codec_name": "aac",
      "sample_rate": "44100",
      "channels": 2,
      "bit_rate": "128000"
    }
  ],
  "format": {
    "duration": "3600.000000",
    "size": "2025000000",
    "bit_rate": "4500000"
  }
}

ffprobe를 쓰는 이유

인코딩 전에 원본 파일 정보를 알아야 올바른 인코딩 옵션을 결정할 수 있다. 해상도, 코덱, 비트레이트, 프레임레이트를 확인하지 않고 인코딩하면 삽질한다.

ffplay - 빠른 재생 확인¶

# 기본 재생
ffplay input.mp4

# 특정 시점부터 재생
ffplay -ss 00:01:30 input.mp4

# 루프 재생
ffplay -loop 0 input.mp4

서버 환경(GUI 없음)에서는 ffplay를 쓸 수 없다. 로컬 개발 환경에서 결과물 확인용으로만 쓴다.

6. 자주 쓰는 글로벌 옵션¶

옵션 의미 예시
-y 출력 파일 덮어쓰기 (확인 없이) ffmpeg -y -i in.mp4 out.mp4
-n 출력 파일 이미 있으면 중단 ffmpeg -n -i in.mp4 out.mp4
-v quiet 로그 출력 최소화 ffmpeg -v quiet -i in.mp4 out.mp4
-v error 에러만 출력 자동화 스크립트에서 유용
-hide_banner 빌드 정보 숨김 출력 깔끔하게 할 때

실무에서 -y 사용 주의

-y는 기존 파일을 확인 없이 덮어쓴다. 자동화 스크립트에서 의도치 않게 원본을 날릴 수 있으니, 입력/출력 경로를 반드시 분리해라.

7. 주의사항 / 함정¶

함정 1: 코덱 미지정¶

# 이렇게 하면 FFmpeg이 알아서 코덱 선택
ffmpeg -i input.mp4 output.avi

문제: FFmpeg 버전, 빌드 옵션에 따라 선택되는 코덱이 다르다. 서버 A에서 돌리면 되는데 서버 B에서 안 되는 상황이 생겨. 코덱은 항상 명시해라.

함정 2: 입출력 순서¶

# 틀린 순서 - 이러면 에러 나거나 의도치 않은 결과
ffmpeg output.mp4 -i input.mp4 -c:v libx264

# 맞는 순서 - 입력 먼저, 출력 마지막
ffmpeg -i input.mp4 -c:v libx264 output.mp4

FFmpeg은 옵션의 위치가 의미를 결정한다. -i 앞에 오는 옵션은 입력 옵션, -i와 출력 파일 사이의 옵션은 출력 옵션이다.

함정 3: 파일 확장자 함정¶

# .ts 확장자 = MPEG-TS (Transport Stream)
# TypeScript의 .ts와 완전히 다름
ffmpeg -i input.mp4 -c copy output.ts

HLS에서 쓰는 .ts는 MPEG Transport Stream이다. TypeScript 아니야.

8. 정리¶

항목 내용
FFmpeg 영상/오디오 변환, 인코딩, 스트리밍 생성 도구
기본 파이프라인 Input → Demux → Decode → Filter → Encode → Mux → Output
기본 명령어 ffmpeg -i 입력 [옵션] 출력
ffprobe 파일 정보 분석 (코덱, 해상도, 비트레이트)
ffplay 간단한 재생 확인
-c copy 재인코딩 없이 스트림 복사 (빠름)
-c:v / -c:a 비디오/오디오 코덱 명시 지정

한 줄 정리

FFmpeg = 영상 파이프라인의 스위스 아미 나이프. 코덱은 항상 명시하고, 변환 전에 ffprobe로 원본부터 확인해라.

확인 문제¶

다음 문제를 풀어봐. 답 못 하면 위에서 다시 읽어.

Q1. FFmpeg의 처리 파이프라인 6단계를 순서대로 나열해봐.

Q2. -c copy 옵션을 썼을 때 건너뛰는 단계는 뭐고, 왜 빠른지 설명해봐.

Q3. ffmpeg, ffprobe, ffplay 중에서 서버 환경(GUI 없음)에서 원본 영상의 해상도와 코덱을 확인하려면 뭘 써야 하고, 어떤 명령어를 치는지 적어봐.

Q4. 아래 명령어에서 문제점이 뭔지 찾아봐. 

ffmpeg -i input.mkv output.mp4

정답 보기

A1. Demux(분해) → Decode(디코딩) → Filter(필터) → Encode(인코딩) → Mux(조립) → Output(출력). 정확히는 Input → Demux → Decode → Filter → Encode → Mux → Output 7단계라고도 볼 수 있지만, 핵심 처리 단계는 Demux → Decode → Filter → Encode → Mux 5단계다.

A2. Decode(디코딩)와 Encode(인코딩) 단계를 건너뛴다. 압축된 스트림을 풀었다가 다시 압축하는 과정이 없으니 CPU 연산이 거의 없고, 단순히 스트림 데이터를 컨테이너 간에 옮기기만 하니까 빠르다.

A3. ffprobe를 써야 한다. ffplay는 GUI가 필요하고, ffmpeg은 변환 도구다. 

ffprobe -v quiet -print_format json -show_streams input.mp4

A4. 코덱이 명시되지 않았다. FFmpeg이 출력 포맷에 따라 자동으로 코덱을 선택하는데, 버전이나 빌드 옵션에 따라 결과가 달라질 수 있다. -c:v libx264 -c:a aac 처럼 코덱을 명시해야 한다.
