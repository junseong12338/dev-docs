2. 상태 코드 체계¶

실전에서 쓰는 상태 코드를 정밀하게 설계하면 어디서 문제가 발생했는지 바로 알 수 있어.

상태 코드 이름 설명 비고
11 UPLOAD_COMPLETE 원본 업로드 완료, 인코딩 대기 워커가 이 상태를 찾아감
16 ENCODING_START 워커가 작업 시작 원본 다운로드 중
20 ENCODING_PROGRESS FFmpeg 인코딩 진행 중 진행률 업데이트
39 ENCODING_DONE 인코딩 완료, S3 업로드 중 결과물 전송 중
40 TRANSCODE_COMPLETE 전체 완료. 재생 가능. 최종 성공 상태

43 ENCODE_FAIL_UPLOAD 결과물 S3 업로드 실패 스토리지 문제

왜 실패 코드를 세분화하나?

"실패"라고만 기록하면 뭐가 문제인지 모르잖아. 다운로드 실패면 S3 연결을 확인해야 하고, FFmpeg 실패면 원본 파일을 확인해야 하고, 업로드 실패면 버킷 권한을 확인해야 해. 실패 유형별로 코드를 나눠야 자동 대응이 가능해져.

3. 워커 스케줄링 - @Scheduled 폴링¶

워커 코드 구조¶

@Component
public class EncodingWorker {

    private final EncodingJobRepository jobRepository;
    private final S3StorageService storageService;
    private final FfmpegService ffmpegService;

    // 이전 작업 완료 후 5초 대기 → 다음 폴링
    @Scheduled(fixedDelay = 5000)
    public void pollAndProcess() {
        // 1. 대기 중인 작업 선점 (원자적 UPDATE)
        EncodingJob job = jobRepository.findAndClaimWaitingJob(getWorkerId());

        if (job == null) {
            return; // 할 일 없음
        }

        try {
            processEncoding(job);
        } catch (Exception e) {
            handleFailure(job, e);
        }
    }
}

작업 선점 쿼리¶

-- 원자적 선점: UPDATE가 성공하면 이 워커가 선점한 것
UPDATE TB_ENCODING_JOB
SET STATUS = 16,
    WORKER_ID = #{workerId},
    STARTED_AT = NOW()
WHERE STATUS = 11
ORDER BY CREATED_AT ASC
LIMIT 1;

fixedDelay vs fixedRate

  • fixedDelay = 5000: 이전 작업이 끝난 후 5초 대기. 인코딩은 오래 걸리니까 이게 맞아.
  • fixedRate = 5000: 이전 작업 시작 후 5초마다 실행. 인코딩이 5초 안에 안 끝나면 작업이 겹쳐.

인코딩 워커는 반드시 fixedDelay를 써야 해.

4. FFmpeg 프로세스 실행 - ProcessBuilder¶

왜 ProcessBuilder야?¶

FFmpeg는 외부 프로세스야. Java 코드 안에서 직접 실행되는 게 아니라, 운영체제의 별도 프로세스로 실행돼. Java에서 외부 프로세스를 실행하려면 ProcessBuilder를 써야 해.

FFmpeg 실행 코드¶

@Service
public class FfmpegService {

    // FFmpeg 실행 경로 (서버에 설치된 경로)
    private final String ffmpegPath = "/usr/bin/ffmpeg";

    public FfmpegResult transcode(Path inputFile, Path outputDir,
                                  String resolution, String bitrate) throws Exception {

        // 출력 파일 경로
        Path outputFile = outputDir.resolve("stream.m3u8");

        // FFmpeg 명령어 구성
        List<String> command = List.of(
            ffmpegPath,
            "-i", inputFile.toString(),          // 입력 파일
            "-c:v", "libx264",                   // 비디오 코덱
            "-crf", "23",                        // 화질 기준
            "-preset", "fast",                   // 인코딩 속도
            "-vf", "scale=-2:" + getHeight(resolution), // 해상도 조정
            "-b:v", bitrate,                     // 비트레이트 상한
            "-maxrate", bitrate,                 // 최대 비트레이트
            "-bufsize", calculateBufsize(bitrate), // 버퍼 크기
            "-c:a", "aac",                       // 오디오 코덱
            "-b:a", "128k",                      // 오디오 비트레이트
            "-f", "hls",                         // HLS 출력
            "-hls_time", "10",                   // 세그먼트 길이 (초)
            "-hls_list_size", "0",               // 플레이리스트에 모든 세그먼트 포함
            "-hls_segment_filename",
                outputDir.resolve("seg_%03d.ts").toString(), // 세그먼트 파일명
            outputFile.toString()                // 출력 플레이리스트
        );

        // ProcessBuilder로 FFmpeg 실행
        ProcessBuilder pb = new ProcessBuilder(command);
        pb.redirectErrorStream(true); // stderr을 stdout으로 합침

        Process process = pb.start();

        // FFmpeg 출력 읽기 (진행률 파싱용)
        String lastOutput = readProcessOutput(process);

        // 프로세스 종료 대기
        int exitCode = process.waitFor();

        if (exitCode != 0) {
            throw new EncodingException("FFmpeg 실패. exitCode=" + exitCode
                + ", output=" + lastOutput);
        }

        return new FfmpegResult(outputDir, exitCode);
    }

    // 해상도 문자열 → 높이 변환
    private String getHeight(String resolution) {
        return switch (resolution) {
            case "1080p" -> "1080";
            case "720p" -> "720";
            case "480p" -> "480";
            case "360p" -> "360";
            default -> "720";
        };
    }
}

redirectErrorStream(true) 필수

FFmpeg는 진행 상황을 stderr로 출력해 (stdout이 아니야). redirectErrorStream(true)를 안 하면 진행률을 못 읽어. 그리고 stderr 버퍼가 가득 차면 FFmpeg가 멈춰버려 (데드락).

5. 진행률 업데이트 흐름¶

FFmpeg 출력 파싱¶

FFmpeg가 인코딩 중 출력하는 라인 예시:

frame= 3000 fps= 60 q=23.0 size=   30720kB time=00:02:00.00 bitrate=2097.2kbits/s speed=2.0x

// FFmpeg 출력에서 진행률 파싱
private String readProcessOutput(Process process) throws IOException {
    BufferedReader reader = new BufferedReader(
        new InputStreamReader(process.getInputStream())
    );

    String line;
    String lastLine = "";
    int lastReportedProgress = 0;

    while ((line = reader.readLine()) != null) {
        lastLine = line;

        // time= 패턴 파싱
        int progress = parseTimeToProgress(line, totalDurationSeconds);

        // 5% 이상 변화 시에만 DB 업데이트
        if (progress - lastReportedProgress >= 5) {
            jobRepository.updateProgress(currentJobId, progress);
            lastReportedProgress = progress;
        }
    }

    return lastLine;
}

// time=00:02:00.00 형태에서 진행률 계산
private int parseTimeToProgress(String line, double totalSeconds) {
    Pattern pattern = Pattern.compile("time=(\\d{2}):(\\d{2}):(\\d{2})\\.(\\d{2})");
    Matcher matcher = pattern.matcher(line);

    if (matcher.find()) {
        double currentSeconds =
            Integer.parseInt(matcher.group(1)) * 3600 +
            Integer.parseInt(matcher.group(2)) * 60 +
            Integer.parseInt(matcher.group(3));

        return (int) Math.min((currentSeconds / totalSeconds) * 100, 100);
    }
    return -1; // 파싱 실패
}

진행률 업데이트 전체 흐름¶

sequenceDiagram
    participant FFmpeg as FFmpeg 프로세스
    participant Worker as Java 워커
    participant DB as 데이터베이스
    participant Client as 프론트엔드

    loop FFmpeg 출력 라인마다
        FFmpeg->>Worker: stderr 출력 (time=00:01:30...)
        Worker->>Worker: 진행률 계산 (50%)
        alt 5% 이상 변화
            Worker->>DB: UPDATE progress = 50
        end
    end

    loop 프론트엔드 폴링
        Client->>DB: 진행률 조회 (3초마다)
        DB-->>Client: progress = 50
        Client->>Client: 프로그레스바 업데이트
    end

6. S3 업로드 후 DB URL 갱신¶

인코딩이 끝나면 결과물(.m3u8 + .ts 파일들)을 S3에 올리고, DB에 재생 URL을 저장해야 해.

private void uploadAndFinalize(EncodingJob job, Path outputDir) throws Exception {
    // 1. 상태: S3 업로드 중
    job.setStatus(39); // ENCODING_DONE, 업로드 중
    jobRepository.save(job);

    // 2. HLS 결과물 S3 업로드
    String videoId = String.valueOf(job.getVideoId());
    String resolution = job.getResolution();
    String keyPrefix = String.format("videos/%s/%s/", videoId, resolution);

    // 디렉토리 내 모든 파일 업로드 (.m3u8 + .ts)
    try (Stream<Path> files = Files.list(outputDir)) {
        files.forEach(file -> {
            String key = keyPrefix + file.getFileName().toString();
            storageService.uploadFile(key, file);
        });
    }

    // 3. DB에 재생 URL 저장
    String playlistUrl = String.format("https://{CDN_DOMAIN}/%sstream.m3u8",
        keyPrefix); // 실제 CDN 도메인으로 교체 필요
    job.setOutputPath(playlistUrl);
    job.setStatus(40); // TRANSCODE_COMPLETE
    job.setCompletedAt(LocalDateTime.now());
    job.setProgress(100);
    jobRepository.save(job);

    // 4. 임시 파일 정리
    cleanupTempFiles(outputDir);
}

URL은 CDN 도메인으로

DB에 저장하는 URL은 S3 직접 URL이 아니라 CDN 도메인 URL이어야 해. 사용자가 S3에 직접 접근하면 CDN 캐시를 안 타서 느리고, S3 이그레스 비용도 나와.

S3 직접 URL (사용 금지):
https://my-bucket.s3.ap-northeast-2.amazonaws.com/videos/123/1080p/stream.m3u8

CDN URL (이걸 DB에 저장):
https://cdn.example.com/videos/123/1080p/stream.m3u8

7. 전체 코드 흐름 - 업로드부터 완료까지¶

통합 처리 메서드¶

private void processEncoding(EncodingJob job) {
    Path tempDir = null;

    try {
        // ── 1단계: 임시 디렉토리 생성 ──
        tempDir = Files.createTempDirectory("encoding_" + job.getJobId());
        Path inputFile = tempDir.resolve("input.mp4");
        Path outputDir = tempDir.resolve("output");
        Files.createDirectories(outputDir);

        // ── 2단계: S3에서 원본 다운로드 ──
        job.setStatus(16); // ENCODING_START
        jobRepository.save(job);

        storageService.downloadFile(job.getSourcePath(), inputFile);

        // ── 3단계: FFmpeg 인코딩 실행 ──
        job.setStatus(20); // ENCODING_PROGRESS
        jobRepository.save(job);

        ffmpegService.transcode(inputFile, outputDir,
            job.getResolution(), job.getBitrate());

        // ── 4단계: 결과물 S3 업로드 + DB 갱신 ──
        uploadAndFinalize(job, outputDir);

    } catch (Exception e) {
        handleFailure(job, e);
    } finally {
        // ── 5단계: 임시 파일 정리 (성공/실패 모두) ──
        if (tempDir != null) {
            cleanupTempFiles(tempDir);
        }
    }
}

실패 처리¶

private void handleFailure(EncodingJob job, Exception e) {
    // 실패 유형별 상태 코드 분류
    int failureStatus;
    if (e instanceof S3DownloadException) {
        failureStatus = 41; // 다운로드 실패
    } else if (e instanceof EncodingException) {
        failureStatus = 42; // FFmpeg 실패
    } else if (e instanceof S3UploadException) {
        failureStatus = 43; // 업로드 실패
    } else {
        failureStatus = 49; // 알 수 없는 실패
    }

    job.setStatus(failureStatus);
    job.setErrorMessage(truncate(e.getMessage(), 2000)); // 메시지 길이 제한
    job.setRetryCount(job.getRetryCount() + 1);
    jobRepository.save(job);

    log.error("인코딩 실패. jobId={}, status={}, error={}",
        job.getJobId(), failureStatus, e.getMessage(), e);
}

임시 파일 정리¶

private void cleanupTempFiles(Path directory) {
    try {
        // 디렉토리와 하위 파일 전부 삭제
        Files.walk(directory)
            .sorted(Comparator.reverseOrder()) // 하위 → 상위 순서로 삭제
            .forEach(path -> {
                try {
                    Files.deleteIfExists(path);
                } catch (IOException e) {
                    log.warn("임시 파일 삭제 실패: {}", path, e);
                }
            });
    } catch (IOException e) {
        log.warn("임시 디렉토리 정리 실패: {}", directory, e);
    }
}

임시 파일 정리는 finally에서

인코딩 성공이든 실패든 임시 파일은 반드시 정리해야 해. 안 하면 디스크가 꽉 차서 다음 인코딩이 다 실패해. finally 블록에서 정리하는 게 철칙이야.

8. 정리¶

항목 핵심
상태 코드 실패도 유형별로 세분화 (41/42/43/49). 어디서 터졌는지 바로 파악.
워커 폴링 @Scheduled + fixedDelay. 작업 끝나고 대기.
FFmpeg 실행 ProcessBuilder. redirectErrorStream 필수.
진행률 FFmpeg stderr 파싱, 5% 단위 DB 업데이트.
URL 저장 CDN 도메인 URL로 저장. S3 직접 URL 금지.
임시 파일 finally에서 반드시 정리. 디스크 풀 방지.

한 줄 정리

트랜스코딩 시스템의 실전 핵심은 상태 추적의 정밀함과 실패 복구의 견고함이다. 성공 경로보다 실패 경로를 더 꼼꼼히 설계해야 한다.

확인 문제¶

다음 문제를 풀어봐. 답 못 하면 위에서 다시 읽어.

Q1. 상태 코드 42와 43의 차이는? 각각 어떤 확인이 필요해?

Q2. @Scheduled(fixedDelay = 5000) 대신 fixedRate = 5000을 쓰면 뭐가 문제야?

Q3. FFmpeg 실행 시 redirectErrorStream(true)를 안 하면 생기는 문제 2가지는?

Q4. DB에 저장하는 재생 URL을 S3 직접 URL 대신 CDN URL로 해야 하는 이유 2가지는?

Q5. 임시 파일 정리를 try 블록 안(성공 시)에만 하면 뭐가 문제야?

정답 보기

A1. 42는 FFmpeg 인코딩 자체가 실패한 거야 (코덱 불일치, 손상된 파일 등). 원본 파일을 확인해야 해. 43은 인코딩은 성공했지만 결과물을 S3에 올리는 게 실패한 거야. 스토리지 연결, 버킷 권한, 디스크 용량을 확인해야 해.

A2. fixedRate는 이전 작업 시작 후 5초마다 새 작업을 가져와. 인코딩은 몇 분~몇 십 분 걸리는데, 5초마다 새 작업을 가져가면 작업이 겹쳐서 메모리/CPU가 폭주해. fixedDelay는 이전 작업이 끝난 후 5초 대기하니까 안전해.

A3. (1) FFmpeg는 진행 상황을 stderr로 출력해. redirectErrorStream 안 하면 진행률을 파싱할 수 없어. (2) stderr 버퍼가 가득 차면 FFmpeg 프로세스가 블로킹되어 멈춰버려 (데드락 발생).

A4. (1) S3 직접 접근은 CDN 캐시를 안 타서 느려. 사용자와 S3 서버 사이 거리만큼 지연이 생겨. (2) S3 직접 접근은 이그레스 비용이 발생해. CDN을 통하면 S3에서 CDN으로 한 번만 가져오고, 이후는 CDN 엣지에서 서빙하니까 비용이 줄어.

A5. 인코딩이 실패했을 때 임시 파일이 안 지워져. 실패가 반복되면 임시 파일이 계속 쌓여서 디스크가 꽉 차고, 그러면 다음 인코딩도 전부 실패하는 연쇄 장애가 발생해. finally에서 정리해야 성공/실패 모두 정리가 보장돼.
