11. HLS 세그먼트 구조 - Gamma¶

1. HLS 파일 구성 - "뭘로 이루어져 있어?"¶

HLS(HTTP Live Streaming)는 두 종류의 파일로 구성된다:

파일 확장자 역할
플레이리스트 .m3u8 세그먼트 목록과 재생 순서를 담은 텍스트 파일
세그먼트 .ts 실제 영상/오디오 데이터가 담긴 조각 파일 
graph TD
    A[원본 영상<br/>input.mp4] -->|FFmpeg 인코딩| B[playlist.m3u8<br/>세그먼트 목록]
    A -->|FFmpeg 인코딩| C[segment_000.ts]
    A -->|FFmpeg 인코딩| D[segment_001.ts]
    A -->|FFmpeg 인코딩| E[segment_002.ts]
    A -->|FFmpeg 인코딩| F[...]
    B --> C
    B --> D
    B --> E

플레이어는 .m3u8 파일을 먼저 받아서 세그먼트 목록을 확인하고, 그 목록에 적힌 .ts 파일을 순서대로 다운로드하면서 재생한다.

왜 쪼개는 거야?

통째로 다운로드하면 재생 시작까지 오래 걸리고, 중간 탐색(시킹)도 힘들다. 작은 조각으로 쪼개면 첫 번째 조각만 받아도 바로 재생 시작 가능하고, HTTP 캐시도 먹고, CDN 배포도 쉬워진다.

2. .ts 파일이란¶

.ts = MPEG Transport Stream. 원래 방송(DVB, ATSC)에서 쓰던 컨테이너 포맷이다.

왜 하필 TS인가?¶

특성 설명
자기 완결적 각 세그먼트가 독립적으로 디코딩 가능. 앞 세그먼트 없어도 됨
에러 내성 패킷 단위(188바이트) 구조라 일부 손실돼도 나머지 재생 가능
방송 표준 수십 년간 검증된 포맷

각 .ts 파일은 보통 2~10초 분량의 영상을 담고 있다. 기본값은 HLS 표준에서 6초를 권장하지만, 실무에서는 4초를 많이 쓴다.

fMP4 (Fragmented MP4)

HLS v7부터 .ts 대신 .m4s (Fragmented MP4) 세그먼트도 지원한다. DASH와 호환되고, 오버헤드가 적어서 점점 많이 쓰이는 추세다. 근데 레거시 호환성 때문에 아직 .ts가 주류야.

3. .m3u8 파일 해부¶

.m3u8은 그냥 UTF-8 인코딩된 텍스트 파일이다. 열어보면 사람이 읽을 수 있어.

미디어 플레이리스트 (단일 해상도)¶

#EXTM3U
#EXT-X-VERSION:3
#EXT-X-TARGETDURATION:5
#EXT-X-MEDIA-SEQUENCE:0
#EXT-X-PLAYLIST-TYPE:VOD
#EXTINF:4.000000,
segment_000.ts
#EXTINF:4.000000,
segment_001.ts
#EXTINF:4.000000,
segment_002.ts
#EXTINF:4.000000,
segment_003.ts
#EXTINF:3.520000,
segment_004.ts
#EXT-X-ENDLIST

한 줄씩 뜯어보자:

태그 의미
#EXTM3U "이 파일은 M3U 플레이리스트야" - 파일 시작을 알리는 매직 넘버
#EXT-X-VERSION:3 HLS 프로토콜 버전. 3이 가장 범용적
#EXT-X-TARGETDURATION:5 세그먼트 최대 길이(초). 실제 세그먼트 중 가장 긴 것의 올림값
#EXT-X-MEDIA-SEQUENCE:0 첫 번째 세그먼트의 시퀀스 번호
#EXT-X-PLAYLIST-TYPE:VOD VOD(전체 영상) or EVENT(라이브)
#EXTINF:4.000000, 바로 다음 줄 세그먼트의 재생 시간(초)
segment_000.ts 실제 세그먼트 파일 경로 (상대 또는 절대 URL)
#EXT-X-ENDLIST "플레이리스트 끝" - VOD에만 있음. 라이브에는 없음

마스터 플레이리스트 (멀티 해상도)¶

#EXTM3U
#EXT-X-VERSION:3
#EXT-X-STREAM-INF:BANDWIDTH=800000,RESOLUTION=640x360
360p/playlist.m3u8
#EXT-X-STREAM-INF:BANDWIDTH=1400000,RESOLUTION=854x480
480p/playlist.m3u8
#EXT-X-STREAM-INF:BANDWIDTH=2800000,RESOLUTION=1280x720
720p/playlist.m3u8
#EXT-X-STREAM-INF:BANDWIDTH=5000000,RESOLUTION=1920x1080
1080p/playlist.m3u8
태그 의미
#EXT-X-STREAM-INF "다음 줄은 variant 스트림의 URL이야"
BANDWIDTH=800000 이 스트림에 필요한 최소 대역폭(bps)
RESOLUTION=640x360 해상도

미디어 vs 마스터 플레이리스트 구분

  • 미디어 플레이리스트: #EXTINF가 있고, 세그먼트(.ts) 파일을 가리킨다
  • 마스터 플레이리스트: #EXT-X-STREAM-INF가 있고, 미디어 플레이리스트(.m3u8)를 가리킨다

플레이어는 마스터 플레이리스트를 먼저 받고 → 대역폭에 맞는 미디어 플레이리스트를 선택 → 세그먼트를 다운로드한다.

4. 핵심 태그 상세¶

#EXT-X-TARGETDURATION¶

#EXT-X-TARGETDURATION:5
  • 모든 세그먼트의 최대 길이를 의미
  • 실제 세그먼트 중 가장 긴 것의 올림(ceil)값이어야 한다
  • 예: 세그먼트가 4.0초, 4.0초, 3.5초면 → #EXT-X-TARGETDURATION:4

이 값이 틀리면

일부 플레이어에서 재생이 안 되거나, 버퍼링이 발생한다. FFmpeg이 자동으로 설정해주지만, 수동으로 .m3u8을 편집할 때 실수하면 안 돼.

#EXTINF¶

#EXTINF:4.000000,
segment_000.ts
  • 바로 다음 줄 세그먼트의 정확한 재생 시간(초)
  • 쉼표(,) 뒤에 제목을 넣을 수 있지만 보통 비워둔다
  • 이 값이 #EXT-X-TARGETDURATION을 초과하면 안 된다

#EXT-X-MEDIA-SEQUENCE¶

#EXT-X-MEDIA-SEQUENCE:0
  • 플레이리스트의 첫 번째 세그먼트가 전체 스트림에서 몇 번째인지
  • VOD에서는 보통 0
  • 라이브 스트리밍에서는 이 값이 계속 증가한다 (오래된 세그먼트가 제거되니까)

#EXT-X-PLAYLIST-TYPE¶

#EXT-X-PLAYLIST-TYPE:VOD
값 의미 #EXT-X-ENDLIST
VOD 녹화된 영상. 플레이리스트가 변하지 않음 있음
EVENT 라이브 이벤트. 세그먼트가 추가만 됨 (삭제 안 됨) 라이브 중에는 없음, 끝나면 추가
(생략) 라이브. 세그먼트가 추가/삭제됨 (슬라이딩 윈도우) 없음

5. 재생 흐름¶

플레이어가 HLS를 재생하는 전체 흐름이다:

sequenceDiagram
    participant P as 플레이어
    participant S as 서버

    P->>S: GET master.m3u8
    S-->>P: 마스터 플레이리스트 반환
    Note over P: 대역폭 기반으로<br/>해상도 선택
    P->>S: GET 720p/playlist.m3u8
    S-->>P: 미디어 플레이리스트 반환
    P->>S: GET 720p/segment_000.ts
    S-->>P: 세그먼트 데이터
    Note over P: 재생 시작
    P->>S: GET 720p/segment_001.ts
    S-->>P: 세그먼트 데이터
    P->>S: GET 720p/segment_002.ts
    S-->>P: 세그먼트 데이터

세그먼트 길이와 지연시간

  • 세그먼트가 짧을수록 (2초): 시킹이 정확하고, 화질 전환이 빠르다. 근데 HTTP 요청이 많아진다.
  • 세그먼트가 길수록 (10초): HTTP 요청이 적다. 근데 시킹 정확도가 떨어지고, 화질 전환이 느리다.
  • 4~6초가 실무에서 가장 많이 쓰이는 구간이다.

6. FFmpeg으로 HLS 생성하기¶

# 기본 HLS 생성
ffmpeg -i input.mp4 \
  -c:v libx264 \
  -c:a aac \
  -hls_time 4 \                          # 세그먼트 길이 4초
  -hls_playlist_type vod \               # VOD 모드
  -hls_segment_filename "segment_%03d.ts" \  # 세그먼트 파일명 패턴
  playlist.m3u8                          # 출력 플레이리스트
옵션 의미
-hls_time 4 각 세그먼트의 목표 길이 (초)
-hls_playlist_type vod VOD용 플레이리스트 생성
-hls_segment_filename 세그먼트 파일 이름 패턴 (%03d = 001, 002, ...)

hls_time은 '목표'지 '정확한 값'이 아니다

FFmpeg은 키프레임(IDR 프레임) 경계에서 세그먼트를 자른다. -hls_time 4로 설정해도 키프레임 간격에 따라 실제 세그먼트 길이는 3.8초~5.2초 등으로 변할 수 있다. 정확하게 4초로 자르려면 -g(GOP 크기)와 -keyint_min을 맞춰야 한다. 이건 12장(인코딩 프리셋)에서 자세히 다룬다.

8. 주의사항 / 함정¶

함정 1: TARGETDURATION과 실제 세그먼트 불일치¶

#EXT-X-TARGETDURATION:4
#EXTINF:6.000000,
segment_000.ts

EXTINF(6초)가 TARGETDURATION(4초)를 초과한다. 이러면 일부 플레이어(특히 iOS Safari)에서 재생 오류가 난다.

함정 2: 상대 경로 vs 절대 경로¶

# 상대 경로 (플레이리스트와 같은 위치 기준)
segment_000.ts

# 절대 URL
https://cdn.example.com/video/segment_000.ts

CDN을 쓸 때 경로가 꼬이면 404가 뜬다. 마스터 플레이리스트에서 미디어 플레이리스트를 가리키는 경로, 미디어 플레이리스트에서 세그먼트를 가리키는 경로 둘 다 확인해야 한다.

함정 3: .ts와 TypeScript 충돌¶

웹 프로젝트에서 .ts 확장자가 TypeScript로 인식될 수 있다. 웹서버 MIME 타입 설정에서 .ts를 video/mp2t로 명시해야 한다.

# nginx 설정 예시
types {
    video/mp2t ts;
    application/vnd.apple.mpegurl m3u8;
}

9. 정리¶

항목 내용
.m3u8 UTF-8 텍스트 파일. 세그먼트 목록과 메타데이터
.ts MPEG Transport Stream. 2~10초 분량의 영상 조각
마스터 플레이리스트 여러 해상도의 미디어 플레이리스트를 가리킴 (#EXT-X-STREAM-INF)
미디어 플레이리스트 세그먼트 목록을 담음 (#EXTINF)
TARGETDURATION 세그먼트 최대 길이. 실제 최대값의 올림
ENDLIST VOD에만 있음. 라이브에는 없음

한 줄 정리

HLS = 마스터 플레이리스트가 미디어 플레이리스트를 가리키고, 미디어 플레이리스트가 .ts 세그먼트를 가리키는 계층 구조다.

확인 문제¶

다음 문제를 풀어봐. 답 못 하면 위에서 다시 읽어.

Q1. 마스터 플레이리스트와 미디어 플레이리스트의 차이를 설명하고, 각각에 들어있는 핵심 태그를 하나씩 말해봐.

Q2. #EXT-X-TARGETDURATION:4인데 세그먼트 중 하나가 5.2초면 어떤 문제가 생기는지 설명해봐.

Q3. VOD용 플레이리스트에는 있고, 라이브 스트리밍 플레이리스트에는 없는 태그가 뭔지, 왜 없는지 설명해봐.

Q4. -hls_time 4로 설정했는데 실제 세그먼트 길이가 정확히 4초가 아닌 이유를 설명해봐.

정답 보기

A1. 마스터 플레이리스트는 여러 해상도의 미디어 플레이리스트 URL을 담고 있다. 핵심 태그는 #EXT-X-STREAM-INF (BANDWIDTH, RESOLUTION 포함). 미디어 플레이리스트는 세그먼트(.ts) 파일의 목록과 각 세그먼트의 재생 시간을 담고 있다. 핵심 태그는 #EXTINF.

A2. #EXTINF 값이 #EXT-X-TARGETDURATION을 초과하면 HLS 표준 위반이다. 일부 플레이어(특히 iOS Safari 같은 엄격한 플레이어)에서 재생 오류가 발생하거나 재생 자체가 안 될 수 있다.

A3. #EXT-X-ENDLIST 태그. VOD는 영상이 끝나있으니까 "여기가 끝"이라고 명시할 수 있지만, 라이브 스트리밍은 아직 끝나지 않았으니까 이 태그가 없다. 라이브가 종료되면 그때 추가된다.

A4. FFmpeg은 키프레임(IDR 프레임) 경계에서만 세그먼트를 자를 수 있다. 키프레임이 정확히 4초마다 있지 않으면, 가장 가까운 키프레임 위치에서 잘리기 때문에 실제 길이가 달라진다. 정확히 4초로 자르려면 GOP 크기(-g)와 최소 키프레임 간격(-keyint_min)을 프레임레이트 * 세그먼트 시간으로 맞춰야 한다.
