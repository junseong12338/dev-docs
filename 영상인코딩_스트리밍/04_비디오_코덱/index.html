04. H.264/H.265 - 영상 압축의 핵심 - Beta¶

1. H.264(AVC)란 - "이게 뭐야?"¶

영상 압축을 택배 포장이라고 생각해봐.

1080p 영상 1초는 원래 약 1.5Gbps의 데이터야. 1시간짜리 영화면 675GB. 이걸 그대로 보내면 인터넷 회선이 터져.

H.264는 이 짐을 진공 포장기로 압축하는 거야. 675GB를 2~4GB로 줄여. 비유는 여기까지고, 진짜 얘기 하자.

H.264의 정체

H.264, 정식 명칭 AVC(Advanced Video Coding)는 ITU-T와 ISO/IEC가 공동 개발한 영상 압축 표준이다. 2003년에 발표됐고, 20년 넘게 지구상에서 가장 널리 쓰이는 비디오 코덱이다.

왜 표준이 됐냐? 이유는 3가지야.

이유 설명
압축 효율 이전 세대(MPEG-2) 대비 50% 더 작은 파일 크기에 동일 화질
범용 호환성 모든 브라우저, 모든 모바일, 모든 TV가 지원
유연한 프로파일 저사양 모바일부터 4K 방송까지 커버

기억할 것

H.264가 표준인 이유 = "어디서든 돌아간다". 코덱은 효율도 중요하지만 호환성이 더 중요하다.

2. 프레임 타입 - "영상 압축의 핵심 트릭"¶

영상은 초당 24~60장의 프레임(사진)으로 구성돼. 근데 연속된 프레임은 대부분 비슷해.

뉴스 앵커가 말하는 장면을 생각해봐. 배경은 안 변하고, 입만 움직여. 이걸 매 프레임마다 통째로 저장하면 미친 거지. 그래서 H.264는 프레임을 3가지 타입으로 나눈다.

I-프레임 (Intra Frame, 키프레임)¶

I-프레임 = 완전한 사진

다른 프레임 참조 없이 독립적으로 화면 전체를 저장하는 프레임. 사진 한 장이라고 생각하면 돼. 혼자서도 온전한 이미지를 보여줄 수 있어.

  • 용량이 가장 크다 (전체 화면 정보 포함)
  • 탐색(seek) 시 반드시 I-프레임부터 시작해야 한다
  • 영상의 "기준점" 역할

P-프레임 (Predicted Frame)¶

P-프레임 = 변한 부분만 저장

이전 프레임과의 차이점만 저장하는 프레임. "앵커 배경은 그대로고, 입 모양만 바뀌었어" → 입 모양 변화만 기록.

  • I-프레임보다 훨씬 작다 (보통 1/3~1/10)
  • 이전 프레임에 의존한다 (이전 프레임이 깨지면 같이 깨져)

B-프레임 (Bidirectional Frame)¶

B-프레임 = 앞뒤 모두 참조

앞쪽 프레임과 뒤쪽 프레임 모두 참조해서 차이점만 저장. "5번 프레임이 3번이랑 7번 사이 중간값이네" → 차이만 기록.

  • 가장 작다 (압축 효율 최고)
  • 앞뒤 프레임 모두에 의존
  • 디코딩 순서가 표시 순서와 다르다 (이거 중요해, 나중에 나와)

    I1[I-프레임<br/>완전한 사진<br/>크기: 큼] --> P1[P-프레임<br/>이전과 차이<br/>크기: 중간]
    P1 --> B1[B-프레임<br/>앞뒤 참조<br/>크기: 작음]
    B1 --> P2[P-프레임<br/>이전과 차이<br/>크기: 중간]
    P2 --> B2[B-프레임<br/>앞뒤 참조<br/>크기: 작음]
    B2 --> I2[I-프레임<br/>새 기준점<br/>크기: 큼]

주의사항

I-프레임이 손상되면 다음 I-프레임까지 전부 깨진다. P, B 프레임은 I-프레임 없이는 아무것도 못 보여줘.

3. GOP (Group of Pictures) - "프레임 묶음"¶

GOP는 하나의 I-프레임으로 시작하는 프레임 그룹이야.

택배로 비유하면, I-프레임은 새 상자이고, P/B-프레임은 그 상자 안에 차곡차곡 넣는 물건들이야. 다음 I-프레임이 나오면 새 상자 시작.

비유는 여기까지. 진짜 구조를 보자.

GOP 크기 = 30이면:
I B B P B B P B B P B B P B B P B B P B B P B B P B B P B B I (다음 GOP)
|<----------------------- 30 프레임 ----------------------->|

GOP 크기의 의미¶

GOP 크기 장점 단점
짧은 GOP (30~60) 탐색 정확도 높음, 에러 복구 빠름 파일 크기 증가 (I-프레임이 많으니까)
긴 GOP (120~250) 파일 크기 작음, 압축 효율 좋음 탐색 부정확, 에러 전파 범위 넓음

HLS 스트리밍에서 GOP

HLS 세그먼트 길이(보통 4~6초)에 맞춰서 GOP 크기를 설정해야 해. 30fps 영상에 4초 세그먼트면 GOP = 120 (30fps x 4초). 세그먼트 경계에 I-프레임이 와야 깔끔하게 잘린다.

keyint와 min-keyint¶

# GOP 크기 = 120 (30fps 기준 4초마다 I-프레임)
ffmpeg -i input.mp4 -g 120 -keyint_min 120 output.mp4

# -g: GOP 크기 (최대 키프레임 간격)
# -keyint_min: 최소 키프레임 간격 (이것보다 가까이 I-프레임 안 넣음)

장면 전환(Scene Change)

영화에서 장면이 확 바뀌면 인코더가 자동으로 I-프레임을 삽입해. 이건 scenecut 옵션으로 제어하는데, HLS에서는 세그먼트 정렬을 위해 고정 GOP를 쓰는 경우가 많아.

4. H.265(HEVC) - "H.264의 후계자"¶

H.265의 정체

H.265, 정식 명칭 HEVC(High Efficiency Video Coding). 2013년에 발표됐고, H.264 대비 동일 화질에서 파일 크기 약 50% 감소.

어떻게 50%나 줄였냐? 핵심은 블록 크기야.

항목 H.264 (AVC) H.265 (HEVC)
최대 블록 크기 16x16 (매크로블록) 64x64 (CTU)
분할 방식 고정 크기 분할 쿼드트리 재귀 분할
움직임 예측 정수/반정수 펠 더 정밀한 움직임 벡터
인트라 예측 방향 9개 35개
루프 필터 디블록킹 디블록킹 + SAO

간단히 말하면, H.264는 화면을 16x16 블록으로 잘라서 처리하는데, H.265는 64x64 블록으로 시작해서 필요한 만큼 쪼개는 거야.

하늘같이 단순한 영역은 64x64 통째로, 얼굴같이 복잡한 영역은 4x4까지 세밀하게. 불필요한 데이터를 더 줄일 수 있으니까 효율이 올라가는 거지.

graph TD
    subgraph H.264
        A[16x16 매크로블록] --> B[고정 크기로 분할]
        B --> C[단순한 영역도 16x16]
    end

    subgraph H.265
        D[64x64 CTU] --> E{영역이 복잡한가?}
        E -->|단순| F[64x64 그대로]
        E -->|복잡| G[32x32로 분할]
        G --> H{더 복잡한가?}
        H -->|예| I[16x16, 8x8, 4x4까지]
        H -->|아니오| J[32x32 유지]
    end

그럼 왜 아직도 H.264를 써?¶

H.265의 치명적 약점

라이선스 비용과 호환성 문제 때문이야.

비교 H.264 H.265
브라우저 지원 모든 브라우저 Safari, Edge (Chrome은 하드웨어 가속 시만)
라이선스 비교적 단순 3개 특허 풀, 복잡하고 비쌈
인코딩 속도 빠름 H.264 대비 2~10배 느림
하드웨어 디코딩 거의 모든 기기 최신 기기만
스트리밍 호환 어디서든 재생 일부 환경에서 재생 불가

실전 판단

유튜브, 넷플릭스는 H.265를 쓴다. 하드웨어 지원이 되는 기기에만 H.265 스트림을 내려보내고, 안 되면 H.264로 폴백해. 네가 만드는 서비스가 범용 웹 서비스면 H.264가 안전하고, 모바일 앱 전용이면 H.265 고려해볼 만하다.

5. 프로파일과 레벨 - "같은 코덱인데 등급이 있다"¶

H.264는 하나의 코덱이지만, 성능과 기능에 따라 등급이 나뉜다. 이걸 프로파일(Profile)과 레벨(Level)이라고 해.

프로파일 (Profile) - "어떤 기능을 쓸 수 있나"¶

프로파일 특징 용도
Baseline B-프레임 없음, CABAC 없음 저사양 모바일, 화상통화, 실시간 통신
Main B-프레임 지원, CABAC 지원 TV 방송, 일반 스트리밍
High 8x8 변환, 양자화 스케일링 블루레이, 고화질 스트리밍, 유튜브

CABAC이 뭐냐면

Context-Adaptive Binary Arithmetic Coding. 엔트로피 코딩 방식 중 하나야. 쉽게 말해 데이터를 더 효율적으로 압축하는 알고리즘인데, 연산량이 많아서 Baseline에서는 뺐어. Baseline이 CAVLC(더 단순한 방식)를 쓰는 이유는 실시간 인코딩에서 속도가 중요하기 때문이야.

레벨 (Level) - "얼마나 큰 영상을 처리할 수 있나"¶

레벨은 해상도, 비트레이트, 프레임레이트의 상한선을 정의해.

레벨 최대 해상도 최대 비트레이트 용도
3.0 720x576 (SD) 10 Mbps SD 방송
3.1 1280x720 (HD) 14 Mbps 720p 스트리밍
4.0 2048x1024 20 Mbps 1080p 스트리밍
4.1 2048x1024 50 Mbps 고화질 1080p
5.1 4096x2160 (4K) 300 Mbps 4K 영상 
# H.264 High 프로파일, 레벨 4.1로 인코딩
ffmpeg -i input.mp4 \
  -c:v libx264 \
  -profile:v high \
  -level 4.1 \
  output.mp4

# -profile:v high: High 프로파일 사용 (B-프레임, CABAC 활성화)
# -level 4.1: 1080p@30fps, 최대 50Mbps까지 허용

프로파일/레벨 잘못 설정하면

High 프로파일로 인코딩한 영상을 Baseline만 지원하는 저사양 기기에서 재생하면 디코딩 실패한다. 타겟 기기가 뭔지 먼저 확인하고 프로파일을 정해야 해.

6. 코덱 선택 기준 - "그래서 뭘 써야 해?"¶

코덱 선택은 트레이드오프야. 완벽한 선택은 없어.

기준 H.264 선택 H.265 선택
대상 불특정 다수 웹 사용자 특정 앱/기기 사용자

대역폭 네트워크 여유 있음 대역폭 비용 절감 필수

현실적인 선택

대부분의 웹 서비스에서는 H.264 High 프로파일이 정답이야. H.265는 "추가 최적화"로 생각해. H.264로 먼저 서비스하고, 필요하면 H.265를 추가하는 게 현실적이야.

AV1이라는 녀석도 있다¶

차세대 코덱 AV1

Google, Mozilla, Amazon 등이 만든 로열티 프리 코덱. H.265와 비슷한 효율인데 라이선스 비용이 없어. YouTube, Netflix가 적극 도입 중이지만, 아직 인코딩 속도가 매우 느리고 하드웨어 디코딩 지원이 제한적이야.

7. 정리¶

항목 핵심
H.264 가장 호환성 좋은 코덱, 어디서든 돌아간다
I-프레임 독립적인 완전한 사진, 탐색의 기준점
P-프레임 이전 프레임과의 차이만 저장
B-프레임 앞뒤 프레임 모두 참조, 가장 작음
GOP I-프레임으로 시작하는 프레임 묶음, HLS 세그먼트와 맞춰야 함
H.265 H.264 대비 50% 효율적이지만 호환성/비용 문제
프로파일 Baseline(저사양) < Main(일반) < High(고화질)
레벨 해상도/비트레이트/프레임레이트의 상한선

한 줄 정리

H.264는 "어디서든 재생되는" 코덱이고, I/P/B 프레임으로 "변한 부분만 저장"해서 영상을 압축한다. H.265는 더 효율적이지만 호환성 비용을 치러야 한다.

확인 문제¶

다음 문제를 풀어봐. 답 못 하면 위에서 다시 읽어.

Q1. I-프레임, P-프레임, B-프레임 중에서 용량이 가장 작은 건 뭐고, 왜 작아?

Q2. GOP 크기가 120이고 영상이 30fps일 때, 몇 초마다 I-프레임(키프레임)이 들어가?

Q3. H.265가 H.264보다 압축 효율이 50% 좋은데, 왜 아직도 H.264를 쓰는 서비스가 더 많아?

Q4. H.264 Baseline 프로파일에서 B-프레임을 쓸 수 있어, 없어? 그리고 왜 그런 거야?

Q5. HLS 스트리밍에서 세그먼트 길이가 4초이고 fps가 30일 때, GOP 크기는 얼마로 설정해야 해?

정답 보기 **A1.** B-프레임이 가장 작다. 앞쪽과 뒤쪽 프레임을 **모두 참조**해서 차이점만 저장하기 때문에, 한 방향만 참조하는 P-프레임보다 더 적은 데이터로 표현할 수 있다. **A2.** 4초마다. GOP 120 / 30fps = 4초. 즉 4초에 한 번 I-프레임이 삽입된다. **A3.** 크게 3가지 이유: (1) 브라우저 호환성이 H.264가 압도적으로 좋다 (2) H.265는 라이선스 비용이 복잡하고 비싸다 (3) 인코딩 속도가 H.264 대비 2~10배 느려서 서버 비용이 올라간다. **A4.** 쓸 수 없다. Baseline 프로파일은 B-프레임과 CABAC을 지원하지 않는다. 이유는 Baseline이 **실시간 통신(화상회의 등)** 용도인데, B-프레임 디코딩은 미래 프레임을 기다려야 하므로 **지연(latency)**이 생기기 때문이다. **A5.** 120. 30fps x 4초 = 120. 세그먼트 경계에 I-프레임이 정확히 위치해야 HLS 세그먼트가 깔끔하게 잘린다.
