06. 화질을 결정하는 두 가지 숫자 - Beta¶

1. 해상도란 - "이게 뭐야?"¶

TV 매장에서 "4K"라고 써있는 거 본 적 있지? 그게 해상도야.

해상도 = 화면을 구성하는 점(픽셀)의 개수. 점이 많을수록 그림이 세밀해져.

비유는 여기까지. 진짜 얘기 하자.

해상도의 정의

해상도는 가로 픽셀 x 세로 픽셀로 표현한다. 1920x1080이면 가로 1920개, 세로 1080개, 총 2,073,600개의 점으로 화면을 그린다는 뜻이야.

주요 해상도 정리¶

명칭 해상도 (가로 x 세로) 총 픽셀 수 용도
360p 640 x 360 230,400 모바일 저화질, 데이터 절약
480p (SD) 854 x 480 409,920 옛날 DVD급, 저속 네트워크
720p (HD) 1280 x 720 921,600 일반 스트리밍, 강의 영상
1080p (Full HD) 1920 x 1080 2,073,600 고화질 스트리밍, 유튜브 기본
1440p (2K) 2560 x 1440 3,686,400 게임, 고화질 모니터
2160p (4K) 3840 x 2160 8,294,400 넷플릭스 프리미엄, 방송

p가 뭐냐면

360p의 p는 progressive(프로그레시브)의 약자야. 화면을 위에서 아래로 한 번에 그린다는 뜻이야. 예전 TV에는 i(interlaced)도 있었는데 (1080i), 이건 홀수 줄/짝수 줄을 번갈아 그리는 방식이야. 요즘은 거의 다 p를 쓴다.

해상도가 높으면 뭐가 달라?¶

graph LR
    A[360p<br/>23만 픽셀] -->|4배| B[720p<br/>92만 픽셀]
    B -->|2.25배| C[1080p<br/>207만 픽셀]
    C -->|4배| D[4K<br/>829만 픽셀]

핵심 함정

해상도가 2배(360p -> 720p)가 되면 픽셀 수는 4배가 돼. 가로 2배 x 세로 2배 = 4배. 이건 비트레이트, 저장 공간, 인코딩 시간 전부에 영향을 미쳐.

2. 비트레이트란 - "이게 뭐야?"¶

해상도가 "화면의 크기"라면, 비트레이트는 "화면의 밀도"야.

같은 1080p 영상이라도 비트레이트가 낮으면 깍두기 현상(블록 노이즈)이 생기고, 높으면 선명해져.

비트레이트의 정의

비트레이트 = 초당 사용하는 데이터의 양. 단위는 bps(bits per second). 5Mbps면 매초 5메가비트의 데이터를 사용한다는 뜻이야.

택배로 비유하면:

  • 해상도 = 상자 크기 (큰 상자 = 큰 그림)
  • 비트레이트 = 상자 안 내용물 밀도 (꽉 찬 상자 = 선명한 그림)

상자가 크면(해상도 높으면) 내용물(비트레이트)도 많아야 빈틈이 없어.

비트레이트 체감
1 Mbps 720p에서 겨우 볼 만한 수준, 움직임 많으면 깨짐
3 Mbps 720p에서 깨끗, 1080p에서 약간 뭉개짐
5 Mbps 1080p에서 양호
8 Mbps 1080p에서 깨끗
15 Mbps 1080p에서 매우 선명, 4K에서 최소 수준
25+ Mbps 4K에서 양호

3. 해상도와 비트레이트 관계 - "둘은 짝꿍이야"¶

이게 진짜 중요해. 해상도만 올린다고 화질이 좋아지는 게 아니야.

480p 영상을 1080p로 업스케일하면 어떻게 될까? 해상도 숫자는 1080p가 되는데, 원본 정보가 480p밖에 없으니까 뿌옇게 번져 보여. 없는 정보를 만들어낼 수는 없으니까.

반대로, 1080p 영상인데 비트레이트를 500kbps로 낮추면? 픽셀은 200만 개가 있는데 데이터가 부족해서 블록 노이즈가 생겨. 피자 도우는 큰데 토핑이 없는 거야.

이거 진짜 많이 실수해

"1080p니까 화질 좋겠지" → 아니야. 1080p + 1Mbps면 720p + 3Mbps보다 화질이 더 나빠. 해상도에 맞는 충분한 비트레이트를 줘야 의미가 있어.

4. CBR vs VBR - "비트레이트를 어떻게 배분하나"¶

비트레이트를 매 초마다 고정할 수도 있고, 상황에 따라 변동시킬 수도 있어.

CBR (Constant Bitrate) - 고정 비트레이트¶

CBR

영상 전체에서 비트레이트를 일정하게 유지. 어두운 화면이든 폭발 장면이든 같은 데이터 양.

# CBR 5Mbps로 인코딩
ffmpeg -i input.mp4 \
  -c:v libx264 \
  -b:v 5M \           # 비디오 비트레이트: 5Mbps
  -maxrate 5M \       # 최대 비트레이트도 5Mbps로 제한
  -bufsize 10M \      # 버퍼 크기 (보통 maxrate의 2배)
  output.mp4

VBR (Variable Bitrate) - 가변 비트레이트¶

VBR

장면 복잡도에 따라 비트레이트를 유동적으로 조절. 단순한 장면은 적게, 복잡한 장면은 많이.

# VBR - 평균 5Mbps, 최대 8Mbps
ffmpeg -i input.mp4 \
  -c:v libx264 \
  -b:v 5M \           # 목표 평균 비트레이트: 5Mbps
  -maxrate 8M \       # 최대 비트레이트: 8Mbps까지 허용
  -bufsize 10M \      # 버퍼 크기
  output.mp4

비교¶

항목 CBR VBR
비트레이트 항상 고정 장면에 따라 변동
파일 크기 예측 가능 예측 어려움
화질 단순한 장면에서 낭비, 복잡한 장면에서 부족 전체적으로 균일한 화질
스트리밍 대역폭 예측 쉬움 대역폭 변동 있음
용도 라이브 스트리밍, 대역폭 제한 환경 VOD, 파일 다운로드

    subgraph CBR

    end

    subgraph VBR

    end

언제 뭘 쓰냐

  • 라이브 방송: CBR (네트워크 대역폭 예측이 중요)
  • VOD 스트리밍: VBR (화질 균일성이 중요)
  • HLS 멀티 비트레이트: 각 해상도별로 VBR + maxrate 제한

5. CRF (Constant Rate Factor) - "품질 기준 인코딩"¶

CBR/VBR은 비트레이트를 기준으로 인코딩해. 근데 더 똑똑한 방법이 있어.

CRF = 품질을 기준으로 비트레이트를 자동 결정

"5Mbps로 인코딩해" 대신 "이 정도 품질로 인코딩해"라고 지정하는 방식이야. 인코더가 알아서 장면마다 필요한 만큼 비트레이트를 배분해.

CRF 값의 의미¶

H.264(libx264) 기준:

CRF 값 품질 파일 크기 용도
0 무손실 (lossless) 매우 큼 마스터 보관용
15~17 거의 원본급 큼 고품질 보관, 편집용
18~20 매우 좋음 중간~큼 고화질 스트리밍
21~23 좋음 (기본값 23) 중간 일반 스트리밍, 대부분의 용도
24~27 보통 작음 모바일, 저대역폭
28~32 낮음 매우 작음 미리보기, 썸네일
51 최저 최소 쓸 일 없음 
# CRF 23으로 인코딩 (기본값, 대부분의 상황에 적합)
ffmpeg -i input.mp4 \
  -c:v libx264 \
  -crf 23 \            # 품질 기준: 23 (낮을수록 고화질, 파일 커짐)
  output.mp4

CRF의 함정

CRF는 비트레이트를 제한하지 않아. 복잡한 장면에서 비트레이트가 폭발할 수 있어. 스트리밍용이면 반드시 maxrate로 상한선을 걸어야 해.

CRF + maxrate (스트리밍용 정석)¶

# CRF 기반이지만 최대 비트레이트 제한
ffmpeg -i input.mp4 \
  -c:v libx264 \
  -crf 23 \            # 품질 기준: 23
  -maxrate 5M \        # 최대 5Mbps 제한 (이보다 높아지지 않음)
  -bufsize 10M \       # 버퍼 크기 (maxrate의 약 2배)
  output.mp4

이게 실전에서 가장 많이 쓰는 조합

CRF + maxrate = "이 정도 품질로 인코딩하되, 비트레이트는 이 이상 넘기지 마" 품질도 챙기고 대역폭 폭발도 방지하는 가장 현실적인 방법이야.

CBR vs VBR vs CRF 한눈에 비교¶

방식 기준 장점 단점 용도
CBR 고정 비트레이트 대역폭 예측 가능 화질 불균일 라이브 스트리밍
VBR 평균 비트레이트 화질 균일 파일 크기 예측 어려움 VOD
CRF 목표 품질 최적 품질/크기 비율 비트레이트 예측 불가 로컬 인코딩
CRF+maxrate 품질 + 상한선 품질 + 대역폭 제어 CRF보다 약간 품질 타협 HLS 스트리밍

6. 실전 비트레이트 가이드 - "그래서 얼마로 해?"¶

해상도별 권장 비트레이트 (H.264 기준)¶

해상도 프레임레이트 권장 비트레이트 (VBR) 최대 비트레이트 CRF 참고
360p 30fps 400~700 kbps 1 Mbps 28~30
480p 30fps 700~1,200 kbps 2 Mbps 26~28
720p 30fps 1,500~3,000 kbps 4 Mbps 23~25
720p 60fps 2,500~4,500 kbps 6 Mbps 23~25
1080p 30fps 3,000~6,000 kbps 8 Mbps 21~23
1080p 60fps 4,500~9,000 kbps 12 Mbps 21~23
1440p 30fps 6,000~13,000 kbps 18 Mbps 20~22
2160p (4K) 30fps 13,000~34,000 kbps 40 Mbps 18~20
2160p (4K) 60fps 20,000~50,000 kbps 60 Mbps 18~20

이 표는 참고용이야

영상 콘텐츠에 따라 달라져. 강의(화면 변화 적음)는 낮은 쪽, 게임/스포츠(움직임 많음)는 높은 쪽으로 잡아야 해.

HLS 멀티 비트레이트 인코딩 실전 예시¶

# 1080p
ffmpeg -i input.mp4 \
  -c:v libx264 -crf 21 -maxrate 5M -bufsize 10M \
  -vf scale=1920:1080 \
  -c:a aac -b:a 128k \
  output_1080p.mp4

# 720p
ffmpeg -i input.mp4 \
  -c:v libx264 -crf 23 -maxrate 3M -bufsize 6M \
  -vf scale=1280:720 \
  -c:a aac -b:a 128k \
  output_720p.mp4

# 480p
ffmpeg -i input.mp4 \
  -c:v libx264 -crf 25 -maxrate 1.5M -bufsize 3M \
  -vf scale=854:480 \
  -c:a aac -b:a 96k \
  output_480p.mp4

# 360p
ffmpeg -i input.mp4 \
  -c:v libx264 -crf 28 -maxrate 800k -bufsize 1.6M \
  -vf scale=640:360 \
  -c:a aac -b:a 64k \
  output_360p.mp4

해상도 줄일 때 비율 주의

-vf scale=1280:720처럼 가로:세로 비율(16:9)을 맞춰야 해. 비율이 안 맞으면 영상이 찌그러져. 확실하지 않으면 -vf scale=-2:720으로 세로만 지정하고 가로는 자동 계산시켜.

7. 정리¶

항목 핵심
해상도 가로 x 세로 픽셀 수. 해상도 2배 = 픽셀 4배
비트레이트 초당 데이터 양. 해상도에 맞는 비트레이트가 있어야 화질 유지
해상도-비트레이트 관계 해상도만 올리고 비트레이트 안 올리면 오히려 화질 나빠짐
CBR 고정 비트레이트, 라이브 스트리밍용
VBR 가변 비트레이트, VOD용
CRF 품질 기준 인코딩, 가장 효율적
CRF+maxrate 스트리밍 실전 정석, 품질 + 대역폭 제어

한 줄 정리

화질 = 해상도 x 비트레이트. 둘 다 적절해야 의미가 있고, 스트리밍에서는 CRF + maxrate 조합이 정석이다.

확인 문제¶

다음 문제를 풀어봐. 답 못 하면 위에서 다시 읽어.

Q1. 1080p 영상을 1Mbps로 인코딩하면 어떤 현상이 생겨? 그리고 왜 생겨?

Q2. 라이브 스트리밍에서 CBR을 쓰는 이유는 뭐야? VBR이 화질 더 좋은데 왜 안 써?

Q3. CRF 18과 CRF 28로 같은 영상을 인코딩하면, 파일 크기는 어느 쪽이 더 크고 왜 그래?

Q4. CRF만 써서 HLS 스트리밍용 영상을 인코딩하면 어떤 문제가 생길 수 있어?

Q5. 해상도가 720p에서 1080p로 올라가면, 픽셀 수는 약 몇 배 증가해?

정답 보기 **A1.** 블록 노이즈(깍두기 현상)가 생긴다. 1080p는 약 200만 픽셀인데 1Mbps로는 이 픽셀들을 표현할 데이터가 부족해서, 인코더가 디테일을 뭉개버리기 때문이다. 차라리 720p에 3Mbps가 화질이 더 좋다. **A2.** 라이브 스트리밍은 **네트워크 대역폭 예측이 중요**하기 때문이다. VBR은 복잡한 장면에서 비트레이트가 급증할 수 있는데, 그 순간 네트워크 대역폭을 초과하면 버퍼링이 걸린다. CBR은 항상 일정한 데이터를 보내니까 대역폭 계획이 쉽다. **A3.** CRF 18이 훨씬 크다. CRF 값이 **낮을수록 고화질**이고, 고화질일수록 더 많은 데이터(높은 비트레이트)를 사용하기 때문이다. CRF 6 차이면 파일 크기가 대략 2~4배 차이 난다. **A4.** 비트레이트 폭발 문제가 생길 수 있다. CRF는 품질 유지를 위해 복잡한 장면에서 비트레이트를 무제한으로 올리는데, 스트리밍에서는 사용자의 네트워크 대역폭을 초과하면 버퍼링이 걸린다. 그래서 반드시 **-maxrate로 상한선**을 걸어야 한다. **A5.** 약 2.25배. 720p는 921,600 픽셀(1280x720), 1080p는 2,073,600 픽셀(1920x1080). 2,073,600 / 921,600 = 약 2.25배.
