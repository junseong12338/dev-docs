22. 트러블슈팅 - Omega 실전¶

1. 인코딩 실패 - 원인별 대응¶

인코딩이 실패하면 에러 메시지부터 읽어. "안 돼요"만 가지고는 아무것도 못 해.

원인 1: 코덱 불일치¶

Unknown decoder 'h264_qsv'
Decoder h264_cuvid not found
원인 해결
FFmpeg에 해당 코덱이 컴파일 안 됨 ffmpeg -codecs로 지원 코덱 확인
하드웨어 가속 코덱인데 GPU가 없음 소프트웨어 코덱(libx264)으로 변경
입력 파일의 코덱을 디코딩 못 함 ffprobe로 원본 코덱 확인 후 대응 
# 지원 코덱 확인
ffmpeg -codecs | grep h264

# 원본 파일 코덱 확인
ffprobe -v quiet -print_format json -show_streams input.mp4

원인 2: 메모리 부족 (OOM)¶

Cannot allocate memory
Killed (signal 9)
원인 해결
동시 인코딩 수가 너무 많음 워커 동시 실행 수 제한
4K 이상 고해상도 인코딩 서버 메모리 확인, swap 활성화
Java 프로세스 + FFmpeg 메모리 합산 초과 FFmpeg 전용 메모리 확보 
# FFmpeg 프로세스 메모리 사용량 확인
ps aux | grep ffmpeg

# 시스템 메모리 상태 확인
free -h

# 서버 OOM Killer 로그 확인
dmesg | grep -i "out of memory"

OOM Killer가 FFmpeg를 죽이면

리눅스 OOM Killer는 메모리가 부족하면 프로세스를 강제 종료해. Java 워커는 살아있는데 FFmpeg만 죽으면 process.waitFor()가 비정상 종료 코드를 반환해. exitCode가 137(128+9)이면 OOM Killer가 죽인 거야.

원인 3: 디스크 부족¶

No space left on device
Error writing output file
원인 해결
임시 파일이 안 지워짐 임시 디렉토리 정리 스크립트 추가
인코딩 결과물 크기 예측 실패 충분한 여유 공간 확보 (원본 x 3 이상)
/tmp 파티션 용량 부족 임시 디렉토리를 별도 파티션으로 설정 
# 디스크 사용량 확인
df -h

# 대용량 파일/디렉토리 찾기
du -sh /tmp/encoding_* | sort -rh | head -20

# 오래된 임시 파일 정리 (24시간 이상 된 것)
find /tmp/encoding_* -mtime +1 -delete

디스크 모니터링 필수

인코딩 서버는 디스크 사용률 80% 알림을 걸어놔. 인코딩 결과물은 원본 파일의 1~3배 크기가 될 수 있어. 5GB 원본이면 15GB 이상 여유 공간이 필요해.

2. 화질 이슈¶

블러 (흐릿함)¶

증상 원인 해결
텍스트가 뭉개져 보임 CRF 값이 너무 높음 CRF 20~23으로 낮추기
전체적으로 흐릿함 비트레이트 부족 해상도별 적정 비트레이트 확인
특정 장면만 흐림 빠른 움직임에 비트레이트 부족 -maxrate와 -bufsize 증가

끊김 (Stuttering)¶

증상 원인 해결
재생 중 자주 버퍼링 비트레이트가 사용자 대역폭 초과 ABR(다중 해상도) 적용
특정 구간에서 멈춤 해당 세그먼트 파일 손상 해당 세그먼트 재인코딩
시작할 때 오래 걸림 GOP가 너무 큼 -g 30 (1초 단위 키프레임)

색상 깨짐¶

증상 원인 해결
색이 바랜 느낌 색공간(color space) 변환 문제 -colorspace bt709 -color_primaries bt709 -color_trc bt709 추가
녹색/보라색으로 깨짐 픽셀 포맷 불일치 -pix_fmt yuv420p 명시
HDR 영상이 이상함 HDR → SDR 변환 필요 tonemap 필터 적용 
# 색상 문제 해결 - 표준 색공간 강제
ffmpeg -i input.mp4 \
  -c:v libx264 -crf 23 \
  -pix_fmt yuv420p \
  -colorspace bt709 \
  -color_primaries bt709 \
  -color_trc bt709 \
  output.mp4

yuv420p 지정은 습관으로

-pix_fmt yuv420p를 안 넣으면 원본 픽셀 포맷을 그대로 쓰는데, yuv444p나 yuv422p 같은 포맷은 대부분의 브라우저/디바이스에서 재생 안 돼. 항상 yuv420p로 강제하는 게 안전해.

3. 성능 튜닝¶

하드웨어 가속 (GPU 인코딩)¶

가속 방식 GPU FFmpeg 인코더 속도 향상 화질 (vs libx264)
NVENC NVIDIA h264_nvenc 5~10배 약간 떨어짐
QSV Intel (내장 GPU) h264_qsv 3~5배 비슷
VAAPI Intel/AMD (Linux) h264_vaapi 3~5배 비슷
VideoToolbox macOS h264_videotoolbox 3~5배 비슷 
# NVIDIA GPU 가속 인코딩
ffmpeg -hwaccel cuda -i input.mp4 \
  -c:v h264_nvenc \
  -preset p4 \
  -b:v 4500k \
  -c:a aac -b:a 128k \
  output.mp4

# Intel QSV 가속 인코딩
ffmpeg -hwaccel qsv -i input.mp4 \
  -c:v h264_qsv \
  -preset medium \
  -b:v 4500k \
  output.mp4

GPU 인코딩의 트레이드오프

GPU 인코딩은 속도는 빠르지만 같은 비트레이트에서 화질이 약간 떨어져. CPU(libx264)의 압축 효율을 못 따라가. 대량 인코딩에서 속도가 중요하면 GPU, 화질이 중요하면 CPU를 써.

멀티스레드 활용¶

# CPU 스레드 수 지정 (기본값: 자동 감지)
ffmpeg -i input.mp4 -c:v libx264 -threads 4 output.mp4

# 스레드 0 = 자동 (코어 수만큼)
ffmpeg -i input.mp4 -c:v libx264 -threads 0 output.mp4
설정 적합한 상황
-threads 0 (자동) 서버에서 FFmpeg 하나만 실행할 때
-threads 4 (제한) 동시에 여러 FFmpeg를 실행할 때 (코어 분배)

동시 인코딩 시 코어 분배

8코어 서버에서 동시 2개 인코딩한다면 각각 -threads 4로 제한하는 게 좋아. 자동(0)으로 하면 두 프로세스가 8코어를 놓고 경쟁해서 오히려 느려져.

인코딩 속도 최적화 팁¶

팁 효과
임시 디렉토리를 SSD에 배치 I/O 병목 해소. HDD 대비 2~3배 빠름.
네트워크 스토리지 대신 로컬 디스크 사용 S3에서 다운로드 후 로컬에서 작업
불필요한 필터 제거 필터 하나당 처리 시간 증가
2-pass 대신 CRF 사용 2-pass는 2번 돌려야 해서 시간 2배

4. HLS 재생 안 되는 경우¶

CORS 에러¶

Access to XMLHttpRequest at 'https://cdn.example.com/stream.m3u8'
from origin 'https://www.example.com' has been blocked by CORS policy

가장 흔한 HLS 재생 실패 원인

브라우저에서 HLS.js로 영상을 재생할 때, .m3u8과 .ts 파일을 XMLHttpRequest로 가져와. 이때 Origin이 다르면 CORS 에러가 나서 재생이 안 돼.

# CDN 또는 S3 버킷 정책에 CORS 헤더 추가
# Nginx 설정 예시
location /videos/ {
    add_header Access-Control-Allow-Origin "*";
    add_header Access-Control-Allow-Methods "GET, HEAD, OPTIONS";
    add_header Access-Control-Allow-Headers "Range";
    add_header Access-Control-Expose-Headers "Content-Length, Content-Range";
}

// S3 버킷 CORS 설정 (JSON)
{
    "CORSRules": [
        {
            "AllowedOrigins": ["https://www.example.com"],
            "AllowedMethods": ["GET", "HEAD"],
            "AllowedHeaders": ["*"],
            "ExposeHeaders": ["Content-Length", "Content-Range"],
            "MaxAgeSeconds": 86400
        }
    ]
}

MIME Type 오류¶

파일 올바른 Content-Type 잘못된 경우
.m3u8 application/vnd.apple.mpegurl application/octet-stream → 재생 안 됨
.ts video/mp2t application/octet-stream → 일부 플레이어 실패
.mp4 video/mp4 보통 자동 인식 
# Nginx MIME type 설정
types {
    application/vnd.apple.mpegurl  m3u8;
    video/mp2t                     ts;
}

Mixed Content 에러¶

Mixed Content: The page at 'https://...' was loaded over HTTPS,
but requested an insecure resource 'http://cdn.example.com/...'

사이트가 HTTPS인데 CDN URL이 HTTP면 브라우저가 차단해. CDN도 HTTPS 설정 필수.

기타 재생 실패¶

증상 원인 해결
404 Not Found .ts 세그먼트 경로가 .m3u8에 적힌 것과 다름 .m3u8 내용 확인, 상대 경로 사용
첫 화면 검은색 첫 세그먼트에 키프레임 없음 -force_key_frames "expr:gte(t,n_forced*2)"
특정 해상도만 안 됨 해당 해상도 .m3u8이 없거나 세그먼트 누락 master.m3u8과 각 해상도 .m3u8 확인
iOS에서만 안 됨 HLS 규격 불일치 Apple HLS Authoring Specification 확인

5. FFmpeg 에러 메시지 읽는 법¶

에러 메시지 구조¶

FFmpeg 에러는 보통 이런 구조야:

[component @ 0x7f...] Error message
[output_format @ 0x7f...] Error message
Last message repeated N times
Error while ... : error code

자주 보는 에러 메시지 사전¶

에러 메시지 의미 대응
No such file or directory 입력/출력 경로가 잘못됨 경로 확인, 디렉토리 존재 확인
Invalid data found when processing input 입력 파일이 손상됨 ffprobe로 파일 검증, 재업로드 요청
Discarded N corrupted packets 파일 일부가 손상됨 원본 확인, 무시 가능한 수준인지 판단
Error while decoding stream 디코딩 실패 입력 코덱 확인, 디코더 설치 확인
Discarding packet with unset pts 타임스탬프 없는 패킷 -fflags +genpts 옵션 추가
Too many packets buffered 버퍼 오버플로 -max_muxing_queue_size 1024 추가
Could not find codec parameters 코덱 정보를 못 읽음 파일 손상 또는 지원 안 되는 포맷
Discarded N corrupted frames 프레임 손상 원본 품질 문제, 재업로드 권유

FFmpeg 디버깅 순서¶

graph TD
    A[FFmpeg 에러 발생] --> B[에러 메시지 읽기]
    B --> C{파일 문제?}
    C -->|Yes| D[ffprobe로 원본 확인]
    C -->|No| E{코덱 문제?}
    D --> D1{파일 정상?}
    D1 -->|No| D2[사용자에게 재업로드 요청]
    D1 -->|Yes| E
    E -->|Yes| F[ffmpeg -codecs 로 지원 확인]
    E -->|No| G{리소스 문제?}
    F --> F1[코덱 설치 또는 대체 코덱 사용]
    G -->|메모리| H[동시 인코딩 수 줄이기]
    G -->|디스크| I[임시 파일 정리]
    G -->|No| J[FFmpeg 로그 레벨 올려서 재실행]

    style A fill:#e74c3c,color:#fff
    style J fill:#9b59b6,color:#fff
# 디버깅용 - 로그 레벨 올리기
ffmpeg -v debug -i input.mp4 -c:v libx264 output.mp4

# 원본 파일 검증
ffprobe -v error -show_entries stream=codec_name,width,height,duration \
  -of default=noprint_wrappers=1 input.mp4

# 파일 무결성 검사 (디코딩만 하고 출력은 안 함)
ffmpeg -v error -i input.mp4 -f null - 2>error_log.txt

디버깅의 첫 번째 원칙

에러 메시지를 정확히 읽어. "인코딩 안 돼요"는 정보가 아니야. FFmpeg가 뱉은 에러 메시지의 첫 번째 에러가 진짜 원인이야. 마지막 에러는 보통 첫 번째 에러의 결과일 뿐이야.

7. 정리¶

항목 핵심
인코딩 실패 코덱 확인 → 메모리 확인 → 디스크 확인 순서로 디버깅
화질 이슈 CRF 조정, yuv420p 강제, 색공간 명시
성능 GPU 가속은 속도, CPU는 화질. 동시 인코딩 시 코어 분배.
HLS 재생 CORS, MIME Type, HTTPS 3가지 먼저 확인
FFmpeg 에러 첫 번째 에러가 진짜 원인. 로그 레벨 올려서 디버깅.

한 줄 정리

트러블슈팅의 핵심은 에러 메시지를 읽는 것이다. 대부분의 문제는 FFmpeg가 이미 답을 알려주고 있어. 그걸 읽기만 하면 돼.

확인 문제¶

다음 문제를 풀어봐. 답 못 하면 위에서 다시 읽어.

Q1. FFmpeg 프로세스가 exitCode 137로 종료됐다. 뭐가 원인이고 어떻게 확인해?

Q2. 인코딩된 영상의 색이 바래 보인다. 어떤 FFmpeg 옵션을 확인해야 해?

Q3. HLS 영상이 브라우저에서 재생 안 되고 콘솔에 "CORS policy" 에러가 뜬다. 뭘 설정해야 해?

Q4. 8코어 서버에서 FFmpeg를 동시에 3개 돌릴 때, -threads 옵션을 어떻게 설정하는 게 좋아?

Q5. FFmpeg 에러 메시지가 여러 줄인데, 어느 줄을 먼저 봐야 해? 이유는?

정답 보기

A1. exitCode 137 = 128 + 9(SIGKILL). 리눅스 OOM Killer가 메모리 부족으로 프로세스를 강제 종료한 거야. dmesg | grep -i "out of memory"로 OOM 로그를 확인하고, 동시 인코딩 수를 줄이거나 서버 메모리를 증설해야 해.

A2. 색공간(color space) 설정을 확인해야 해. -colorspace bt709 -color_primaries bt709 -color_trc bt709를 명시하고, 반드시 -pix_fmt yuv420p도 지정해. 원본이 다른 색공간이면 변환 과정에서 색이 바래 보일 수 있어.

A3. CDN 또는 S3 버킷에 CORS 헤더를 설정해야 해. Access-Control-Allow-Origin에 프론트엔드 도메인을 허용하고, Access-Control-Allow-Methods에 GET, HEAD를 포함해야 해. HLS.js는 XMLHttpRequest로 .m3u8과 .ts를 가져오기 때문에 CORS가 필수야.

A4. 8코어를 3개 프로세스에 분배하니까 각각 -threads 2 정도로 제한하는 게 좋아 (6코어 사용 + 2코어는 OS/Java 워커용). -threads 0(자동)으로 하면 3개 프로세스가 8코어를 놓고 경쟁해서 컨텍스트 스위칭 오버헤드 때문에 오히려 느려져.

A5. 첫 번째 에러 줄을 먼저 봐야 해. FFmpeg는 첫 에러 발생 후 연쇄적으로 다른 에러가 쏟아지는 경우가 많아. 마지막 줄의 에러는 보통 첫 번째 에러의 결과일 뿐이야. 근본 원인은 항상 가장 먼저 나온 에러에 있어.
