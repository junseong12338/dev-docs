15. 청크 업로드 - Gamma¶

1. 대용량 파일 업로드의 문제¶

강의 영상 하나가 2GB라고 해보자. 이걸 일반적인 HTTP POST로 한 번에 올리면?

문제 설명
타임아웃 서버/프록시의 요청 타임아웃 초과. Nginx 기본 60초면 2GB 못 올림
메모리 폭발 서버가 2GB를 메모리에 올려야 함. 동시 업로드 3개면 6GB
네트워크 끊김 1.9GB 올리다 네트워크 끊기면? 처음부터 다시
진행률 표시 불가 사용자는 "지금 몇 %인지" 알 수 없음
웹서버 제한 Nginx client_max_body_size, Tomcat maxPostSize 등

실무에서 실제로 터지는 시나리오

  • 강사가 2시간짜리 강의 영상(3GB)을 업로드
  • 99% 올라간 시점에 네트워크 순단 발생
  • 처음부터 다시 업로드
  • 강사가 화가 나서 전화함

이걸 방지하는 게 청크 업로드다.

2. 청크 업로드 원리¶

핵심은 단순하다: 큰 파일을 작은 조각으로 쪼개서 순서대로 보내고, 서버에서 합친다.

sequenceDiagram
    participant C as 클라이언트
    participant S as 서버

    C->>C: 2GB 파일을 5MB씩 쪼갬<br/>(400개 청크)

    C->>S: POST /upload/init<br/>(파일명, 크기, 청크 수)
    S-->>C: uploadId: abc123

    C->>S: PUT /upload/abc123/chunk/0<br/>(5MB 데이터)
    S-->>C: 200 OK

    C->>S: PUT /upload/abc123/chunk/1<br/>(5MB 데이터)
    S-->>C: 200 OK

    Note over C,S: ... 398개 더 ...

    C->>S: PUT /upload/abc123/chunk/399<br/>(마지막 청크)
    S-->>C: 200 OK

    C->>S: POST /upload/abc123/complete
    S->>S: 400개 청크를 하나로 합침
    S-->>C: 200 OK (업로드 완료)

3단계 흐름¶

단계 요청 서버 동작
1. 초기화 파일 메타데이터 전송 uploadId 발급, 임시 디렉토리 생성
2. 청크 전송 각 청크를 순서대로 전송 임시 디렉토리에 청크 파일 저장
3. 완료 완료 요청 청크들을 하나로 합치고, 임시 파일 삭제

3. multipart vs chunk 차이¶

이 두 개를 헷갈리는 사람이 많다.

Multipart Upload (multipart/form-data)¶

POST /upload HTTP/1.1
Content-Type: multipart/form-data; boundary=----abc123

------abc123
Content-Disposition: form-data; name="title"

강의 제목
------abc123
Content-Disposition: form-data; name="file"; filename="lecture.mp4"
Content-Type: video/mp4

(2GB 바이너리 데이터 전체)
------abc123--
  • 하나의 HTTP 요청에 파일 전체 + 메타데이터를 같이 보내는 방식
  • 파일이 작으면 문제없지만, 대용량이면 앞에서 말한 문제가 전부 발생
  • HTML <form>의 기본 파일 업로드 방식

Chunk Upload¶

PUT /upload/abc123/chunk/0
Content-Type: application/octet-stream
Content-Range: bytes 0-5242879/2147483648

(5MB 바이너리 데이터)
  • 여러 개의 HTTP 요청으로 파일을 쪼개서 보냄
  • 각 요청은 작으니까 타임아웃/메모리 문제 없음
  • 실패한 청크만 재전송 가능
항목 Multipart Chunk
HTTP 요청 수 1개 N개 (청크 수만큼)
파일 전송 단위 파일 전체 5~10MB 조각
실패 시 처음부터 다시 실패한 청크만 다시
진행률 브라우저 기본 (부정확) 청크 단위로 정확
메모리 사용 파일 전체 크기 청크 크기만큼
적합한 크기 ~100MB 100MB~

multipart ≠ chunk

"multipart upload"는 하나의 요청에 여러 파트를 담는 것이고, "chunk upload"는 하나의 파일을 여러 요청으로 나눠 보내는 것이다. 이름이 비슷해서 헷갈리는데, 완전히 다른 개념이야.

4. 클라이언트 측 구현 원리¶

JavaScript (브라우저)에서 파일 쪼개기¶

// File API의 slice() 메서드로 파일을 쪼갠다
const CHUNK_SIZE = 5 * 1024 * 1024; // 5MB

async function uploadFile(file) {
  // 1단계: 업로드 초기화
  const totalChunks = Math.ceil(file.size / CHUNK_SIZE);
  const initRes = await fetch('/api/upload/init', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({
      fileName: file.name,
      fileSize: file.size,
      totalChunks: totalChunks
    })
  });
  const { uploadId } = await initRes.json();

  // 2단계: 청크별 전송
  for (let i = 0; i < totalChunks; i++) {
    const start = i * CHUNK_SIZE;                        // 시작 바이트
    const end = Math.min(start + CHUNK_SIZE, file.size); // 끝 바이트
    const chunk = file.slice(start, end);                // File.slice()로 자름

    await fetch(`/api/upload/${uploadId}/chunk/${i}`, {
      method: 'PUT',
      headers: {
        'Content-Type': 'application/octet-stream',
        'Content-Range': `bytes ${start}-${end - 1}/${file.size}`
      },
      body: chunk  // 5MB 조각
    });

    // 진행률 업데이트
    const progress = Math.round(((i + 1) / totalChunks) * 100);
    console.log(`${progress}% 완료`);
  }

  // 3단계: 완료 요청
  await fetch(`/api/upload/${uploadId}/complete`, {
    method: 'POST'
  });
}

File.slice()가 핵심

File.slice(start, end)는 파일을 메모리에 전부 올리지 않고, 해당 범위만 읽어서 Blob으로 반환한다. 2GB 파일을 slice해도 메모리에는 5MB만 올라간다.

6. 재시도 / 이어받기 구현 원리¶

재시도 (Retry)¶

특정 청크 전송이 실패하면 그 청크만 다시 보낸다.

async function uploadChunkWithRetry(uploadId, index, chunk, maxRetries = 3) {
  for (let attempt = 0; attempt < maxRetries; attempt++) {
    try {
      const res = await fetch(`/api/upload/${uploadId}/chunk/${index}`, {
        method: 'PUT',
        body: chunk
      });
      if (res.ok) return true;
    } catch (err) {
      console.log(`청크 ${index} 실패 (${attempt + 1}/${maxRetries})`);
      // 지수 백오프: 1초 → 2초 → 4초
      await new Promise(r => setTimeout(r, 1000 * Math.pow(2, attempt)));
    }
  }
  throw new Error(`청크 ${index} 업로드 실패 (${maxRetries}회 시도)`);
}

지수 백오프 (Exponential Backoff)

재시도 간격을 1초, 2초, 4초, 8초... 로 점점 늘린다. 서버가 과부하 상태일 때 빠르게 재시도하면 상황을 악화시키니까.

이어받기 (Resume)¶

브라우저를 닫았다가 다시 열어도 이어서 업로드하는 기능.

sequenceDiagram
    participant C as 클라이언트
    participant S as 서버

    C->>S: GET /upload/abc123/status
    S-->>C: {uploadedChunks: [0, 1, 2, 3]}<br/>4번까지 완료

    Note over C: 5번 청크부터 이어서 전송

    C->>S: PUT /upload/abc123/chunk/4
    S-->>C: 200 OK
    C->>S: PUT /upload/abc123/chunk/5
    S-->>C: 200 OK

구현 핵심:

  1. 서버에 업로드 상태 조회 API 필요 (어느 청크까지 받았는지)
  2. 클라이언트에 uploadId를 로컬 저장 (localStorage, IndexedDB)
  3. 재접속 시 저장된 uploadId로 상태 조회 → 이어서 전송
// 이어받기 구현 핵심
async function resumeUpload(file) {
  // 이전 업로드 정보 확인
  const savedUpload = localStorage.getItem(`upload_${file.name}_${file.size}`);

  let uploadId;
  let startChunk = 0;

  if (savedUpload) {
    uploadId = JSON.parse(savedUpload).uploadId;
    // 서버에 상태 확인
    const statusRes = await fetch(`/api/upload/${uploadId}/status`);
    const status = await statusRes.json();
    startChunk = status.uploadedChunks.length;  // 완료된 다음 청크부터
    console.log(`${startChunk}번 청크부터 이어서 업로드`);
  } else {
    // 새 업로드
    const initRes = await fetch('/api/upload/init', { /* ... */ });
    uploadId = (await initRes.json()).uploadId;
    localStorage.setItem(`upload_${file.name}_${file.size}`,
      JSON.stringify({ uploadId }));
  }

  // startChunk부터 전송
  const totalChunks = Math.ceil(file.size / CHUNK_SIZE);
  for (let i = startChunk; i < totalChunks; i++) {
    // ... 청크 전송 ...
  }
}

7. 청크 크기 결정¶

청크 크기 장점 단점
1MB 재시도 비용 적음 HTTP 요청 수 많음, 오버헤드 큼
5MB 균형 잡힘 -
10MB HTTP 요청 적음 재시도 시 10MB 다시 전송
50MB 요청 최소 청크 단위 실패 시 부담 큼

실무 권장: 5~10MB

  • AWS S3 Multipart Upload: 최소 5MB (마지막 청크 제외)
  • NHN Cloud Object Storage: 5MB 권장
  • 대부분의 CDN/스토리지가 5MB 기준

네트워크 환경이 불안정한 곳(모바일)이면 작게(2~5MB), 안정적인 곳(사내망)이면 크게(10~20MB) 잡으면 된다.

8. 주의사항 / 함정¶

함정 1: 임시 파일 정리 안 함¶

업로드가 중간에 중단된 임시 청크 파일이 서버에 쌓인다. 정리 안 하면 디스크가 찬다.

# 24시간 이상 지난 임시 파일 정리 (cron)
find /tmp/uploads -type d -mtime +1 -exec rm -rf {} +

임시 파일 관리 필수

사용자가 업로드를 시작하고 브라우저를 닫는 경우가 흔하다. 서버에 완료되지 않은 청크들이 수백 GB 쌓일 수 있다. TTL(Time To Live) 정책을 반드시 적용해라.

함정 2: 청크 순서 보장¶

네트워크 상태에 따라 청크가 순서 없이 도착할 수 있다. 서버에서 청크 번호를 기준으로 저장하고, 합칠 때 번호 순서대로 합쳐야 한다.

함정 3: 파일 무결성 검증 안 함¶

전송 중 데이터가 손상될 수 있다. 각 청크의 해시(MD5, SHA-256)를 같이 보내서 서버에서 검증해야 한다.

// 청크 전송 시 해시 포함
const hashBuffer = await crypto.subtle.digest('SHA-256', await chunk.arrayBuffer());
const hashHex = Array.from(new Uint8Array(hashBuffer))
  .map(b => b.toString(16).padStart(2, '0')).join('');

await fetch(`/api/upload/${uploadId}/chunk/${i}`, {
  method: 'PUT',
  headers: {
    'X-Chunk-Hash': hashHex  // 서버에서 검증
  },
  body: chunk
});

함정 4: 동시 업로드 제한 없음¶

사용자가 10개 파일을 동시에 업로드하면 서버가 뻗는다. 동시 업로드 수를 제한해야 한다.

9. 정리¶

항목 내용
청크 업로드 대용량 파일을 작은 조각으로 나눠서 순차 전송
3단계 초기화(uploadId 발급) → 청크 전송 → 완료(합치기)
multipart와 차이 multipart는 1요청에 전체, chunk는 N요청에 분할
재시도 실패한 청크만 다시 전송 (지수 백오프)
이어받기 uploadId 저장 + 서버 상태 조회 → 이어서 전송
청크 크기 5~10MB 권장
임시 파일 TTL 정책으로 자동 정리 필수

한 줄 정리

대용량 파일은 쪼개서 보내고, 실패하면 그 조각만 다시 보내고, 중단되면 이어서 보낸다. 서버에 쌓이는 임시 파일 정리는 반드시 해라.

확인 문제¶

다음 문제를 풀어봐. 답 못 하면 위에서 다시 읽어.

Q1. multipart upload와 chunk upload의 차이를 HTTP 요청 수, 실패 시 대응, 메모리 사용 3가지 관점에서 비교해봐.

Q2. 3GB 파일을 5MB 청크로 나누면 청크가 몇 개 생기는지 계산하고, 마지막 청크의 크기는 얼마인지 말해봐.

Q3. 이어받기(Resume) 기능을 구현하려면 클라이언트와 서버 각각에서 뭘 저장/제공해야 하는지 설명해봐.

Q4. 업로드가 중간에 중단된 임시 청크 파일을 관리하지 않으면 어떤 문제가 생기고, 어떻게 해결하는지 설명해봐.

Q5. 청크 크기를 1MB로 잡으면 생기는 문제와, 50MB로 잡으면 생기는 문제를 각각 설명해봐.

정답 보기

A1. Multipart: HTTP 요청 1개, 실패 시 파일 전체 재전송, 파일 전체가 메모리에 올라감. Chunk: HTTP 요청 N개(청크 수만큼), 실패 시 해당 청크만 재전송, 청크 크기(5~10MB)만 메모리에 올라감.

A2. 3GB = 3 * 1024MB = 3072MB. 3072MB / 5MB = 614.4 → 올림하면 615개. 마지막 청크: 3072MB - (614 * 5MB) = 3072 - 3070 = 2MB.

A3. 클라이언트: uploadId를 로컬(localStorage 등)에 저장해야 재접속 시 이전 업로드를 식별 가능. 서버: 업로드 상태 조회 API(GET /upload/{uploadId}/status)를 제공해서, 어느 청크까지 수신 완료했는지 반환해야 한다. 클라이언트는 이 정보를 바탕으로 완료된 다음 청크부터 전송을 이어간다.

A4. 사용자가 업로드를 시작하고 브라우저를 닫거나, 업로드 중 오류로 중단되면 서버에 완료되지 않은 청크 파일이 쌓인다. 정리하지 않으면 디스크 용량이 고갈된다. 해결: TTL(Time To Live) 정책을 적용해서 일정 시간(예: 24시간) 이상 지난 임시 파일을 cron 등으로 자동 삭제한다.

A5. 1MB: 3GB 파일이면 청크가 3072개. HTTP 요청이 3072번이라 요청 오버헤드(헤더, 연결 수립)가 크고, 서버에 임시 파일이 3072개 생긴다. 50MB: 재시도 시 50MB를 다시 보내야 하니 비용이 크고, 네트워크 불안정한 환경에서 하나의 청크 전송이 실패할 확률이 높아진다.
