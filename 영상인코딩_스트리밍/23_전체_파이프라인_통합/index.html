23. 업로드 -> 인코딩 -> 저장 -> CDN -> 재생 - Zeta¶

1. 전체 파이프라인 한눈에 보기¶

지금까지 22개 챕터에서 조각조각 배웠어. 이제 전부 연결한다. 영상 하나가 사용자 브라우저에서 재생되기까지 어떤 과정을 거치는지, 어떤 기술이 쓰이는지, 처음부터 끝까지 한 번에 본다.

sequenceDiagram
    participant 사용자 as 사용자 (브라우저)
    participant 웹서버 as 웹 서버 (API)
    participant 큐 as 작업 큐
    participant 워커 as 인코딩 워커
    participant 스토리지 as Object Storage (S3)
    participant CDN as CDN
    participant DB as DB

    사용자->>웹서버: 1. 영상 파일 업로드 (Chunk Upload)
    웹서버->>웹서버: 2. 청크 수신 + 병합 → 임시 파일 저장
    웹서버->>DB: 3. 영상 레코드 생성 (상태: UPLOADED)
    웹서버->>큐: 4. 인코딩 작업 등록
    웹서버-->>사용자: 5. 업로드 완료 응답

    큐->>워커: 6. 작업 할당
    워커->>DB: 7. 상태 변경 (ENCODING)
    워커->>워커: 8. FFmpeg 인코딩 (4해상도 HLS)
    워커->>스토리지: 9. m3u8 + ts 파일 업로드
    워커->>DB: 10. 상태 변경 (COMPLETED) + URL 저장
    워커->>웹서버: 11. 인코딩 완료 콜백

    사용자->>웹서버: 12. 영상 재생 요청
    웹서버->>DB: 13. 영상 URL 조회
    웹서버-->>사용자: 14. master.m3u8 URL 반환
    사용자->>CDN: 15. master.m3u8 요청
    CDN->>스토리지: 16. 캐시 미스 시 Origin에서 가져옴
    CDN-->>사용자: 17. 플레이리스트 반환
    사용자->>CDN: 18. 세그먼트(.ts) 요청 (ABR 기반)
    CDN-->>사용자: 19. 세그먼트 반환 → HLS.js 재생

이게 전체 그림이야. 단계 하나하나 뜯어본다.

2. 각 단계별 기술 매핑¶

단계 1-2: 업로드¶

사용되는 기술

  • Multipart / Chunk Upload: 대용량 파일을 잘게 쪼개서 전송
  • 서버 로컬 저장: 청크를 수신하면서 임시 디렉토리에 병합
  • 진행률 표시: 클라이언트에서 업로드 %를 보여주려면 청크 단위 응답 필요
graph LR
    A[사용자 브라우저] -->|청크 1 POST| B[웹서버]
    A -->|청크 2 POST| B
    A -->|청크 3 POST| B
    B --> C[임시 파일 병합]
    C --> D[/tmp/upload/video_raw.mp4]
항목 설명
왜 청크 업로드? 수 GB 파일을 한 번에 보내면 타임아웃, 메모리 초과 발생. 챕터 15 참조
청크 크기 보통 5~10MB. 너무 작으면 HTTP 오버헤드, 너무 크면 재전송 비용 증가
실패 복구 청크 단위라서 실패한 청크만 재전송하면 됨

단계 3-5: 작업 등록¶

사용되는 기술

  • DB 상태 관리: 영상 상태를 UPLOADED → ENCODING → COMPLETED / FAILED로 추적
  • 작업 큐 (MQ / DB 폴링 / Redis): 인코딩 작업을 비동기로 처리하기 위한 큐잉
stateDiagram-v2
    [*] --> UPLOADED: 업로드 완료
    UPLOADED --> ENCODING: 워커가 작업 수령
    ENCODING --> COMPLETED: 인코딩 성공
    ENCODING --> FAILED: 인코딩 실패
    FAILED --> ENCODING: 재시도
    COMPLETED --> [*]
큐잉 방식 장점 단점 적합한 상황
MQ (RabbitMQ 등) 확실한 전달 보장, 재시도 내장 인프라 추가 필요 대규모 시스템
DB 폴링 추가 인프라 없음, 구현 단순 폴링 부하, 지연 발생 소규모 시스템
Redis 큐 빠름, 가볍다 영속성 약함 중간 규모

단계 6-10: 인코딩¶

사용되는 기술

  • FFmpeg: 인코딩 엔진. H.264 + AAC로 HLS 출력
  • 멀티 해상도: 1080p, 720p, 480p, 360p 4단계
  • HLS 세그먼트: m3u8(플레이리스트) + ts(세그먼트) 생성
  • S3 호환 Object Storage: 인코딩 결과물 업로드
# 실제 인코딩 시 이런 명령어가 4번 실행된다 (해상도별)
ffmpeg -i /tmp/upload/video_raw.mp4 \
  -c:v libx264 -preset medium \
  -crf 23 -maxrate 5M -bufsize 10M \
  -vf "scale=1920:1080" \
  -c:a aac -b:a 128k -ar 48000 -ac 2 \
  -g 120 -keyint_min 120 -sc_threshold 0 \
  -hls_time 4 -hls_playlist_type vod \
  -hls_segment_filename "1080p/segment_%03d.ts" \
  1080p/playlist.m3u8
인코딩 설정 값 근거 (챕터)
코덱 H.264 (libx264) 범용 호환성 최강 (04장)
오디오 AAC, 48kHz, 128kbps HLS 표준 오디오 (05장)
CRF + maxrate 품질 기반 + 상한선 스트리밍 정석 (06장)
GOP 120 (30fps x 4초) 세그먼트 경계 정렬 (04장, 11장)
세그먼트 4초 실무 최적 구간 (11장)
프리셋 medium 속도/품질 균형 (10장)

단계 9: 저장¶

사용되는 기술

  • S3 호환 Object Storage: 인코딩 결과물(m3u8 + ts) 저장
  • 버킷 구조: 영상 ID별 디렉토리로 정리

단계 15-19: CDN + 재생¶

사용되는 기술

  • CDN: 전 세계 엣지 서버에 캐싱하여 사용자 가까운 곳에서 전달
  • HLS.js: 브라우저에서 HLS 재생을 가능하게 하는 JavaScript 라이브러리
  • ABR (Adaptive Bitrate): 네트워크 상황에 따라 자동 화질 전환
graph LR
    A[브라우저<br/>HLS.js] -->|master.m3u8 요청| B[CDN 엣지 서버]
    B -->|캐시 히트| A
    B -->|캐시 미스| C[Origin<br/>Object Storage]
    C -->|원본 반환| B
    B -->|캐시 저장 + 반환| A

    A -->|네트워크 빠름| D[1080p 세그먼트 요청]
    A -->|네트워크 느림| E[480p 세그먼트 요청]
항목 설명
CDN 캐시 .m3u8과 .ts 파일은 정적 파일이라 캐싱 효율 최고
TTL 설정 VOD는 캐시 오래 유지 가능. 라이브는 짧게
HLS.js 역할 m3u8 파싱, 세그먼트 요청, ABR 제어, 버퍼 관리 전부 담당
postMessage iframe 안에서 재생할 때 부모 페이지와 상태 통신에 사용

프론트엔드에서 URL 흐름¶

sequenceDiagram
    participant 페이지 as 웹 페이지
    participant API as REST API
    participant 플레이어 as HLS.js 플레이어

    페이지->>API: GET /api/videos/{id}
    API-->>페이지: { hlsUrl: "https://cdn.../master.m3u8", status: "COMPLETED" }

postMessage의 역할

LMS처럼 영상 플레이어가 iframe 안에 있을 때, 부모 페이지가 재생 상태(재생 중, 일시정지, 현재 시간, 완료)를 알아야 할 때 postMessage로 통신한다. 학습 진도 추적, 이어보기 기능 등에 필수적이야.

5. "이 시스템을 다르게 설계한다면?"¶

지금까지 본 파이프라인이 유일한 정답은 아니야. 다른 선택지가 있고, 각각 트레이드오프가 있다. 이걸 분석할 수 있어야 진짜 설계를 이해한 거야.

대안 1: 서버리스 인코딩¶

항목 현재 (워커 패턴) 대안 (서버리스)
구조 상시 가동 워커 서버 AWS Lambda / Cloud Functions
장점 제어 용이, 디버깅 쉬움 유휴 비용 0, 자동 스케일링
단점 유휴 시에도 서버 비용 발생 실행 시간 제한 (15분), 메모리 제한
적합 영상이 자주, 많이 올라오는 서비스 영상이 가끔 올라오는 서비스

서버리스의 한계

1시간짜리 4K 영상을 4해상도로 인코딩하면 수십 분 걸린다. Lambda의 15분 제한에 안 맞아. 짧은 영상이나 해상도 하나씩 분할 처리하면 가능하지만, 복잡도가 올라간다.

대안 2: 클라우드 인코딩 서비스¶

항목 현재 (자체 FFmpeg) 대안 (AWS MediaConvert 등)
구조 직접 FFmpeg 명령어 관리 API 호출로 인코딩 요청
장점 완전한 제어, 비용 예측 가능 운영 부담 없음, 자동 스케일링
단점 서버 관리, 장애 대응 직접 비용 높음, 세부 제어 어려움
적합 인코딩 옵션 커스텀이 중요한 경우 빠르게 서비스 런칭, 운영 인력 부족

대안 3: DASH 대신 HLS¶

항목 현재 (HLS) 대안 (DASH)
표준 Apple 자체 표준 MPEG 국제 표준
브라우저 Safari 네이티브 + hls.js dash.js 필요
DRM FairPlay (Apple) Widevine, PlayReady
선택 이유 iOS 네이티브 지원, 모바일 커버리지 DRM 다양성, 표준 준수

현실적인 판단

"왜 HLS야?" → iOS Safari에서 네이티브로 재생돼. DASH는 안 돼. 모바일 사용자가 많은 서비스에서 HLS를 버리기 어렵다. 최근에는 CMAF(Common Media Application Format)으로 HLS와 DASH를 하나의 세그먼트 포맷으로 통합하는 흐름도 있어.

대안 4: Object Storage 직접 서빙 vs CDN¶

항목 CDN 사용 Object Storage 직접
지연시간 사용자 가까운 엣지에서 응답 리전 하나에서 응답
비용 CDN 트래픽 비용 추가 스토리지 아웃바운드만
적합 사용자가 전국/글로벌 사용자가 한 지역에 집중

CDN 없이 되는 거 아냐?

된다. 사용자 10명이면 Object Storage 직접 서빙해도 문제없어. 근데 사용자가 수백, 수천 명이 동시에 영상을 보면? Origin 서버에 부하가 집중되고, 응답 속도가 느려진다. CDN은 "나중에 추가"가 쉬운 레이어라서, 처음에 빼고 시작해도 괜찮아. 트래픽 늘면 그때 넣어.

6. 정리¶

한 줄 정리

영상 파이프라인 = 업로드(청크) → 큐잉(비동기) → 인코딩(FFmpeg/HLS) → 저장(S3) → 배포(CDN) → 재생(HLS.js/ABR). 각 단계는 독립적이지만 데이터 흐름으로 연결되어 있고, 상태 관리(DB)가 전체를 관통한다.

이 챕터에서 반드시 기억할 것¶

개념 한 줄 설명
전체 파이프라인 업로드 → 큐잉 → 인코딩 → 저장 → CDN → 재생
상태 관리 UPLOADED → ENCODING → COMPLETED/FAILED로 추적
데이터 형태 변화 원본 mp4 → m3u8 + ts → CDN URL → 브라우저 프레임
비동기 인코딩 업로드와 인코딩은 분리. 큐로 연결
각 단계의 독립성 인코딩 방식 바꿔도 저장/배포는 영향 없음
트레이드오프 완벽한 설계는 없다. 상황에 맞는 선택지가 있을 뿐

확인 문제¶

다음 문제를 풀어봐. 답 못 하면 위에서 다시 읽어.

Q1. 사용자가 영상을 업로드한 후 바로 재생할 수 없는 이유를 파이프라인 관점에서 설명해봐.

Q2. 인코딩 워커가 죽었을 때, DB 상태가 ENCODING인 채로 멈춰있으면 어떻게 복구해야 해?

Q3. CDN 없이 Object Storage에서 직접 서빙하면 어떤 문제가 생길 수 있는지, 그리고 언제 CDN 없이도 괜찮은지 말해봐.

Q4. 이 파이프라인에서 HLS 대신 프로그레시브 다운로드를 쓰면 어떤 단계가 필요 없어지고, 어떤 문제가 생기는지 설명해봐.

Q5. "인코딩 서버를 자체 운영하는 것"과 "클라우드 인코딩 서비스를 쓰는 것"의 트레이드오프를 최소 3가지 기준으로 비교해봐.

정답 보기

A1. 업로드 완료 후 인코딩 단계를 거쳐야 하기 때문이다. 원본 파일은 HLS 세그먼트로 변환되지 않은 상태라서 스트리밍 재생이 불가능하다. 인코딩이 완료되어 m3u8 + ts 파일이 Object Storage에 올라가고, DB 상태가 COMPLETED로 바뀌어야 비로소 재생 URL을 사용자에게 내려줄 수 있다.

A2. 타임아웃 기반 복구가 필요하다. 일정 시간(예: 30분) 이상 ENCODING 상태가 유지되면 FAILED로 변경하고 재시도 큐에 넣는다. 또는 워커가 주기적으로 heartbeat를 보내서, heartbeat가 끊기면 다른 워커가 작업을 인수하는 방식도 있다.

A3. 동시 접속자가 많아지면 Origin(Object Storage) 서버에 부하가 집중되어 응답 속도가 느려지고 버퍼링이 발생한다. 또한 사용자가 지리적으로 분산되어 있으면 물리적 거리 때문에 지연시간이 커진다. 사용자가 소수이고 한 지역에 집중되어 있으면 CDN 없이도 괜찮다.

A4. 프로그레시브 다운로드를 쓰면 인코딩 단계에서 HLS 세그먼트 생성이 필요 없어진다. 단일 mp4 파일만 만들면 된다. 하지만 (1) ABR(적응적 화질 전환)이 불가능하고 (2) 대역폭 낭비가 생기고(안 본 부분도 받음) (3) 탐색(seek)이 비효율적이고 (4) 영상 파일 URL이 그대로 노출되어 보안이 약해진다.

A5. (1) 비용: 자체 운영은 서버 유지 비용이 고정적이지만 예측 가능. 클라우드 서비스는 사용량 비례인데 대량이면 비쌀 수 있음. (2) 제어: 자체 운영은 FFmpeg 옵션을 완전히 제어 가능. 클라우드 서비스는 제공하는 옵션 범위 내에서만 가능. (3) 운영 부담: 자체 운영은 장애 대응, 스케일링을 직접 해야 함. 클라우드 서비스는 관리형이라 운영 부담이 적음. (4) 스케일링: 자체 운영은 서버 추가/제거를 직접 해야 함. 클라우드 서비스는 자동 스케일링.
