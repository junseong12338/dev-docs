9. 최종시험 - 정답 및 해설¶

정답 맞히는 건 시작이야

정답만 보고 넘어가면 Lv1에서 영원히 못 벗어나. 왜 그게 정답인지, 왜 나머지가 틀렸는지까지 이해해야 해. 틀린 문제는 해당 챕터 다시 읽고 와.

Part 1. 객관식 정답 (Q1~Q10)¶

Q1. 메모리 계산 (객관식, 2점)¶

정답: (B) 2GB

해설: Total = Used + buff/cache + Free. 제로섬이야. 세 개 합치면 반드시 Total이 돼.

16 = 6 + 8 + Free → Free = 2GB

보기 설명
(A) 0GB 16 - 6 - 8 = 2니까 0이 아니야. 계산을 안 한 거지.
(B) 2GB 정답. Total - Used - buff/cache = Free.
(C) 8GB buff/cache 값을 free로 착각한 거야.
(D) 10GB Total - Used만 한 거지. buff/cache를 빼먹었어.

-> [Ch01] OS 메모리 구조 참조

Q2. available 컬럼 (객관식, 2점)¶

정답: (C) available

해설: free만 보면 "메모리 부족이다!" 하고 오판해. buff/cache는 회수 가능하니까, 실제 여유는 available이 알려줘. available = Free + 회수 가능한 buff/cache.

보기 설명
(A) free 가장 흔한 실수. free만 보고 메모리 부족이라고 착각하는 거야. buff/cache가 회수 가능하다는 걸 모르는 사람이 하는 판단이지.
(B) total 물리 RAM 총량이야. 여유를 판단하는 값이 아니야.
(C) available 정답. OS가 "실제로 쓸 수 있는 메모리"를 계산한 값.
(D) buff/cache 캐시 크기만 보고 뭘 판단해? 이건 판단 지표가 아니라 구성 요소야.

Q3. buff/cache 60% 상태 판단 (객관식, 2점)¶

정답: (B) 정상. OS가 빈 메모리를 캐시로 활용하는 건 설계된 동작이다.

해설: Linux OS는 빈 메모리를 놀리지 않아. "쓸 데 없으면 캐시로 쓰자"가 설계 철학이야. buff/cache가 크다는 건 OS가 효율적으로 메모리를 활용하고 있다는 뜻이지.

보기 설명
(A) 비정상, 즉시 회수 전형적인 함정. buff/cache가 크면 불안해하는 건 OS 동작을 모르는 거야.
(B) 정상 정답. OS 설계 의도대로 작동 중.
(C) 메모리 누수 buff/cache는 커널이 관리하는 캐시야. 누수가 아니야. 누수는 Used가 계속 증가하는 패턴이지.
(D) 수동으로 비우기 echo 3 > /proc/sys/vm/drop_caches 같은 짓 하지 마. 오히려 성능 떨어져.

Q4. Swap 발생 조건 (객관식, 2점)¶

정답: (C) available이 부족해서 물리 메모리만으로 감당이 안 될 때 Swap이 시작된다.

해설: Swap은 Free가 0이라서 발생하는 게 아니야. OS는 Free가 줄면 먼저 buff/cache를 회수해. 그래도 모자라면, 즉 available까지 바닥나면 그때 Swap이 발생해.

보기 설명
(A) Free가 0이면 함정. Free가 0이어도 buff/cache 회수하면 돼. available이 핵심이야.
(B) buff/cache가 가득 차면 buff/cache가 크다는 건 정상 동작이야. Swap 조건과 무관해.
(C) available 부족 정답. 물리 메모리(Free + 회수 가능 캐시)로 감당 안 될 때 Swap 시작.
(D) Used가 50% 넘으면 그런 기준은 없어. Used 비율과 Swap 발생은 직접적 관계가 아니야.

Q5. vm.swappiness=0의 의미 (객관식, 2점)¶

정답: (C) 가능한 한 Swap을 피하고, 극한 상황에서만 Swap한다.

해설: vm.swappiness=0이어도 Swap은 발생할 수 있어. OOM(Out Of Memory)을 피하기 위해 커널이 최후의 수단으로 Swap을 쓰거든. "완전 비활성화"가 아니야.

보기 설명
(A) 절대 안 한다 함정. 0이어도 극한 상황에선 Swap 한다. "절대"란 건 없어.
(B) 완전 비활성화 A와 같은 말이야. Swap을 완전 끄려면 swapoff 명령을 써야 해.
(C) 가능한 한 피한다 정답. buff/cache 회수를 극한까지 하고, 진짜 안 되면 그때 Swap.
(D) 바로 Swap 정반대야. 이건 swappiness가 높을 때의 행동이지.

Q6. vm.swappiness 기본값 (객관식, 2점)¶

보기 설명
(A) 0 운영 서버에서 권장하는 값이지, 기본값이 아니야.
(B) 10 Ubuntu 데스크톱 등 일부 배포판에서 쓰기도 하지만 커널 기본값은 아니야.

(D) 60 정답. Linux 커널 기본값.

Q7. Swap이 안 줄어드는 이유 (객관식, 2점)¶

정답: (B) Swap에 나간 데이터를 다시 RAM으로 가져오는 건 비용이 크기 때문에 OS가 적극적으로 하지 않는다.

해설: Swap-in(디스크 → RAM)은 디스크 I/O가 발생해. OS 입장에서는 "지금 안 쓰는 데이터인데 굳이 RAM으로 가져올 필요 있어?"라고 판단하는 거야. 비용 대비 이득이 없으니까 안 해.

보기 설명
(A) 커널 버그 버그가 아니라 의도된 설계야.
(B) 비용이 크니까 안 해 정답. Swap-in은 디스크 I/O 비용 발생. OS는 비용 대비 판단을 해.
(C) 포맷해야 비워짐 Swap은 파일시스템이 아니야. 포맷 개념이 아니지.
(D) 프로세스 종료 시만 프로세스 종료 시 해당 프로세스의 Swap은 해제되지만, 그게 유일한 방법은 아니야. swapoff로도 가능해.

Q8. 백업 중 첫 번째 변화 (객관식, 2점)¶

정답: (C) buff/cache 증가, Free 감소

해설: 디스크에서 파일을 읽으면 OS가 읽은 데이터를 자동으로 buff/cache에 저장해. 이 공간은 Free에서 가져오니까 제로섬으로 Free가 줄어.

보기 설명
(A) Used 증가 백업 프로세스 자체의 메모리는 미미해. 대량 변화는 캐시에서 일어나.
(B) Swap 증가 Swap은 마지막 단계야. 첫 번째 변화가 아니야.
(C) buff/cache 증가, Free 감소 정답. 디스크 읽기 → OS가 캐시 → Free에서 공간 확보.
(D) available 증가 buff/cache가 늘어도 그건 회수 가능하니까 available은 크게 안 변해. 증가하진 않아.

Q9. Swap이 위험한 근본 이유 (객관식, 2점)¶

정답: (B) 디스크는 RAM보다 수천~수만 배 느려서, Swap I/O가 시스템 전체를 느리게 만든다.

해설: RAM 접근 속도는 나노초(ns) 단위, 디스크(HDD)는 밀리초(ms) 단위야. 수만 배 차이. Swap이 발생하면 메모리 접근이 디스크 I/O로 바뀌니까, 시스템 전체가 느려져.

보기 설명
(A) 시스템이 꺼진다 Swap이 가득 차도 바로 꺼지진 않아. OOM Killer가 먼저 프로세스를 죽여.
(B) 디스크가 느리니까 정답. RAM vs 디스크 속도 차이가 근본 원인.
(C) 데이터 손상 Swap 자체가 데이터를 손상시키진 않아.
(D) CPU 사용률 급증 Swap의 직접적 영향은 I/O wait이지, CPU 사용률이 아니야.

-> [Ch03] Swap이란 참조

Q10. 계단식 Swap 증가 패턴 (객관식, 2점)¶

정답: (B) 주기적인 작업이 반복적으로 메모리를 압박하고, Swap은 비가역적이라 누적된다.

해설: 계단식 = 주기적 원인 + 비가역성의 조합이야. 매일 백업이 돌면서 Swap을 조금씩 발생시키는데, 한번 올라간 Swap은 안 줄어드니까 누적되는 거지.

보기 설명
(A) 대규모 할당 한 번 한 번이면 계단이 아니라 점프 한 번이겠지.
(B) 주기적 작업 + 비가역성 정답. 반복 원인 + Swap 누적 = 계단식 패턴.
(C) 메모리 누수 누수면 직선 상승이지, 계단식이 아니야.
(D) Swap 파티션 손상 파티션 손상이면 아예 Swap을 못 쓰지, 계단식으로 쌓이지 않아.

Q12. available 공식 (단답형, 3점)¶

정답: available = Free + 회수 가능한 buff/cache

해설: "회수 가능한"이 핵심이야. buff/cache 전부가 아니야. 커널이 반드시 유지해야 하는 캐시가 일부 있어서 available은 Free + buff/cache보다 약간 작아.

이건 불합격

  • available = Free + buff/cache (회수 가능한이 빠지면 부분 점수 2점)
  • available = Total - Used (개념 자체를 모르는 거야)

Q13. buff/cache 회수 시점 (단답형, 3점)¶

정답: 프로그램이 메모리를 요청했는데 Free가 부족할 때, OS가 buff/cache를 회수해서 Free를 확보한다.

해설: OS는 buff/cache를 "있으면 좋고, 없어도 되는" 것으로 취급해. 프로그램이 메모리를 필요로 하면 캐시보다 프로그램이 우선이니까 캐시를 밀어내고 공간을 줘.

이건 불합격

  • "일정 시간이 지나면" (시간 기반이 아니야)
  • "수동으로 비울 때" (자동으로 해)

Q15. Swap 강제 비우기 명령어 (단답형, 3점)¶

정답:

swapoff -a && swapon -a

해설: swapoff -a로 Swap을 끄면 Swap에 있던 데이터가 RAM으로 돌아와. 그다음 swapon -a로 다시 켜. 단, RAM에 여유가 있어야 해. 없으면 OOM 발생할 수 있어.

이건 불합격

  • swapoff 만 쓰고 swapon을 안 한 경우 (Swap이 꺼진 채로 운영하면 안 돼)
  • echo 3 > /proc/sys/vm/drop_caches (이건 buff/cache 비우는 거지, Swap이 아니야)

Q21. 백업 시나리오 분석 (코드/분석, 5점)¶

정답:

Swap이 0.2GB로 그대로인 이유: Swap의 비가역성 때문. 한번 Swap에 나간 데이터는 OS가 적극적으로 RAM으로 안 가져와. 백업이 끝나서 메모리 압박이 줄어도, Swap-in은 비용이 크니까 "굳이 안 해". 해당 데이터에 다시 접근할 때만 RAM으로 돌아와.

buff/cache가 9GB인 이유: 백업 중 100GB 파일을 읽으면서 OS가 디스크 데이터를 캐시에 올렸어. 백업 끝나도 캐시를 자동으로 비우지 않아. "나중에 또 읽을 수 있으니 일단 놔두자"가 OS 전략이야. 누군가 메모리를 요청하면 그때 회수해.

채점 기준:

항목 배점
Swap 비가역성 설명 2점
buff/cache가 안 줄어드는 이유 2점
OS의 판단 기준(비용 대비 이득) 언급 1점

이건 불합격

  • "버그다" (설계된 동작이야)
  • "시간이 지나면 줄어든다" (자동으로 안 줄어)

Q22. 두 서버 건강 상태 비교 (코드/분석, 5점)¶

정답: 둘 다 건강해. 거의 동일한 상태야.

서버 X는 Free가 10GB로 많고 buff/cache가 2GB로 적어 → 부팅 직후이거나 디스크 작업이 거의 없었던 상태.

서버 Y는 Free가 1GB로 적지만 buff/cache가 11GB로 큼 → 디스크 작업이 많아서 OS가 캐시를 많이 잡은 상태. 하지만 available이 11GB니까 실제 여유는 충분해.

핵심: available이 각각 11.5GB, 11GB로 거의 같아. free가 아니라 available로 판단해야 해. Free가 작다고 불안해하면 안 돼.

채점 기준:

항목 배점
둘 다 건강하다고 판단 2점
available 기준으로 판단 근거 제시 2점
buff/cache의 의미 설명 1점

이건 불합격

  • "서버 Y가 위험하다" (Free만 보고 판단한 거야. available을 봐)
  • "서버 X가 더 건강하다" (available 차이가 0.5GB야. 유의미한 차이 아니야)

Q23. WAS02 장애 분석 (코드/분석, 5점)¶

정답:

(a) 매일 새벽 백업 실행 시마다 Swap이 약 0.3~0.4GB씩 증가하는 계단식 누적 패턴. 백업이 끝나도 Swap은 줄어들지 않고 유지돼.

(b) 근본 원인:

  1. 매일 새벽 백업이 대량 디스크 읽기 수행
  2. 디스크 데이터가 buff/cache로 올라오면서 Free 고갈
  3. Free가 부족해지면 Swap 발생
  4. Swap은 비가역적이라 백업 후에도 안 줄어듦
  5. 이게 매일 누적 → Swap이 1.6GB까지 쌓임
  6. Swap에 나간 WAS 데이터에 접근할 때마다 디스크 I/O 발생 → 응답 지연

(c) 해결 방안:

  • vm.swappiness를 1로 낮춰서 Swap 발생을 최소화
  • 백업 시간을 WAS 부하가 적은 시간으로 조정하거나, 백업 서버를 분리

채점 기준:

항목 배점
(a) 계단식 누적 패턴 식별 1점
(b) 인과관계 체인 (백업→캐시→Free 고갈→Swap→누적→지연) 2점
(c) 해결 방안 2가지 + 각각의 이유 2점

이건 불합격

  • "메모리를 늘린다" (근본 원인을 안 건드리면 똑같이 반복돼)
  • "Swap을 끈다" (Swap 없으면 OOM Killer가 WAS를 죽일 수 있어)

Q24. swappiness 차이 분석 (코드/분석, 5점)¶

정답:

vm.swappiness는 OS가 "메모리 부족할 때 캐시를 회수할 건지 Swap을 할 건지"의 비율을 결정해.

  • 서버 A (swappiness=60): 메모리 압박 시 캐시 회수와 Swap을 비교적 균등하게 수행. "캐시 회수가 귀찮으면 Swap해도 돼"라는 태도. 그래서 available이 3GB여도 Swap을 0.5GB나 함.
  • 서버 B (swappiness=1): 메모리 압박 시 캐시 회수를 극단적으로 선호. "Swap은 최후의 수단"이라는 태도. buff/cache를 끝까지 회수하고 나서야 Swap을 해. 그래서 Swap이 0.1GB밖에 안 됨.

핵심: swappiness 값이 높으면 OS가 Swap에 대해 관대해지고, 낮으면 Swap을 최대한 피하려고 캐시를 먼저 희생시켜. 같은 available이어도 Swap을 얼마나 적극적으로 쓰느냐의 차이.

채점 기준:

항목 배점
swappiness의 역할 설명 (캐시 회수 vs Swap 비율) 2점
두 서버의 행동 차이 설명 2점
같은 available인데 왜 다른지 연결 1점

이건 불합격

  • "swappiness가 높으면 Swap을 많이 한다" 한 줄로 끝내면 부분 점수 1점. 왜, 어떤 메커니즘인지가 빠져 있어.

Q25. Swap 발생 예측 (코드/분석, 5점)¶

정답: 곧 Swap이 발생할 가능성이 매우 높다.

판단 근거:

  1. available이 0.8GB밖에 안 됨: 8GB 서버에서 0.8GB면 10%야. 여유가 거의 없어.
  2. Free가 0.1GB: 사실상 바닥. 새 메모리 요청이 오면 즉시 캐시 회수 시작.
  3. buff/cache가 1.1GB: 회수할 캐시도 별로 없어. 다 회수해도 겨우 1GB 정도.
  4. Used가 6.8GB: Total 8GB 중 85% 점유. 프로그램이 조금만 더 메모리 요청하면 바로 한계.
  5. 현재 Swap이 0GB: 아직 안 터졌지만, available 0.8GB면 터지기 직전.

결론: 프로세스가 0.8GB 이상 추가 메모리를 요청하면 Swap 발생. 시간문제야.

채점 기준:

항목 배점
Swap 발생 가능성 높다고 판단 1점
available 기반 판단 근거 2점
Used/Free/buff/cache 종합 분석 2점

이건 불합격

  • "Swap이 0이니까 안전하다" (현재 상태만 보면 안 돼. 추세와 여유를 봐야지)
  • "Free가 0.1이니까 위험하다" (Free만으로 판단하면 안 돼. available을 봐)

Part 4. 서술형 정답 (Q26~Q30)¶

Q26. buff/cache가 많으면 메모리 부족인가? (서술형, 4점)¶

정답: 틀리다. buff/cache가 많다고 메모리 부족이 아니야.

모범 답안:

buff/cache는 OS가 디스크 읽기 성능을 올리려고 빈 메모리를 활용해서 만든 캐시야. 존재 이유가 "안 쓰는 메모리를 놀리지 말자"인 거지. 그래서 서버가 오래 켜져 있을수록 buff/cache가 커지는 건 정상이야.

핵심은 buff/cache가 회수 가능하다는 거야. 프로그램이 메모리를 요청하면 OS가 캐시를 밀어내고 그 공간을 줘. 프로그램이 캐시보다 우선이니까. 그래서 "실제 여유"를 보려면 free가 아니라 available(= Free + 회수 가능한 buff/cache)을 봐야 해.

buff/cache가 많은데 available도 큰 경우: 건강한 상태. buff/cache가 적은데 Used가 커서 available이 작은 경우: 이게 진짜 위험.

채점 기준:

항목 배점
틀리다고 답 + buff/cache의 존재 이유(성능 캐시) 1점
회수 가능하다는 메커니즘 설명 1점
available로 판단해야 한다는 결론 1점
건강한 상태 vs 위험한 상태 구분 1점

이건 불합격

  • "buff/cache가 많으면 위험할 수 있다" (위험한 게 아니야)
  • 회수 가능하다는 설명 없이 "괜찮다"만 쓴 경우 (왜 괜찮은지를 써야지)

-> [Ch02] buff/cache란 참조

Q27. Swap의 비가역성 (서술형, 4점)¶

정답:

비가역적이라는 뜻: 한번 물리 메모리에서 Swap 영역(디스크)으로 밀려난 데이터는, 메모리 여유가 생겨도 자동으로 RAM에 돌아오지 않는다. 올라간 Swap 사용량이 자연스럽게 내려오지 않는다는 뜻.

OS가 적극적으로 해제하지 않는 이유: Swap-in(디스크 → RAM)은 디스크 I/O가 발생해. OS 입장에서 "지금 아무도 안 쓰는 데이터인데 디스크 I/O 비용을 들여서 RAM으로 가져올 이유가 있어?"라고 판단하는 거야. 비용 대비 이득이 없으니까 안 해. 해당 데이터에 다시 접근하는 프로세스가 있을 때만 그때 Swap-in이 발생해.

운영에 미치는 영향: Swap이 누적된다. 주기적 작업(백업 등)이 Swap을 조금씩 발생시키면, 시간이 지날수록 Swap 사용량이 계단식으로 올라가. Swap에 나간 데이터에 접근할 때마다 디스크 I/O가 발생하니까 시스템 전체가 느려져. Swap을 비우려면 swapoff -a && swapon -a를 수동으로 해야 해.

채점 기준:

항목 배점
비가역성의 정확한 의미 1점
OS가 안 해제하는 이유 (비용 대비 판단) 1점
운영 영향 (누적, 성능 저하) 1점
해결 방법 (swapoff/swapon) 언급 1점

이건 불합격

  • "Swap은 안 줄어든다" 한 줄로 끝 (왜 안 줄어드는지가 핵심이야)
  • "커널 버그다" (설계된 동작이야)

-> [Ch05] Swap의 비가역성 참조

Q28. 새벽 백업과 WAS 성능 저하 인과관계 (서술형, 5점)¶

정답:

전체 인과관계:

  1. 디스크 스캔: 새벽 백업이 대량의 파일을 순차적으로 읽음
  2. buff/cache 증가: OS가 읽은 디스크 데이터를 자동으로 캐시에 올림. 이 공간은 Free에서 가져옴
  3. Free 고갈: 캐시가 Free를 계속 잡아먹으면서 Free가 바닥남
  4. available 감소: Free가 없고, 회수할 캐시도 한계에 도달하면 available이 급감
  5. Swap 발생: available이 부족해져서 OS가 물리 메모리의 데이터를 디스크(Swap)로 밀어냄. 이때 WAS의 메모리 데이터도 Swap에 나갈 수 있음
  6. 성능 저하: WAS가 Swap에 나간 데이터에 접근할 때마다 디스크 I/O 발생. RAM 접근(나노초)이 디스크 접근(밀리초)으로 바뀌면서 응답 시간이 수천 배 느려짐

추가 악화 요인: Swap은 비가역적이라 백업이 끝나도 Swap이 안 줄어듦. 매일 백업이 반복되면 Swap이 계단식으로 누적되고, 시간이 지날수록 WAS 성능이 점진적으로 악화됨.

채점 기준:

항목 배점
디스크 스캔 → buff/cache 증가 1점
buff/cache → Free 고갈 → available 감소 1점
available 부족 → Swap 발생 1점
Swap → 디스크 I/O → 성능 저하 1점
비가역성으로 인한 누적 효과 1점

이건 불합격

  • 중간 단계를 생략하고 "백업하면 Swap 생겨서 느려진다"로 끝내면 2점 이하
  • 인과관계 순서가 틀린 경우 (예: Swap → Free 고갈 순서)

Q29. swappiness 60 vs 1의 행동 차이 (서술형, 4점)¶

정답:

둘 다 available이 1GB로 메모리가 부족한 상황. 여기서 추가 메모리 요청이 오면:

swappiness=60 서버: OS가 "캐시 회수도 하고, Swap도 적당히 섞어서 하자"라고 판단해. buff/cache 회수와 Swap을 비교적 균등하게 병행함. 결과적으로 Swap이 상대적으로 많이 발생.

swappiness=1 서버: OS가 "Swap은 최후의 수단. 캐시를 끝까지 회수하자"라고 판단해. buff/cache를 극한까지 밀어내고, 그래도 안 되면 그때서야 Swap을 함. 결과적으로 Swap 발생이 최소화됨. 대신 캐시가 거의 사라져서 디스크 읽기 성능이 떨어질 수 있음.

트레이드오프: swappiness를 낮추면 Swap은 줄어들지만, 캐시가 많이 회수되니까 디스크 읽기 성능이 떨어질 수 있어. 하지만 Swap의 성능 패널티(수천 배 느림)에 비하면 캐시 손실이 훨씬 나은 선택이야. 그래서 운영 서버에서는 swappiness를 낮추는 거야.

채점 기준:

항목 배점
swappiness=60의 행동 (캐시 회수 + Swap 병행) 1점
swappiness=1의 행동 (캐시 우선 회수, Swap 최소화) 1점
두 접근의 트레이드오프 (캐시 손실 vs Swap 손실) 1점
왜 운영 서버에서 낮은 값을 권장하는지 결론 1점

이건 불합격

  • "60은 Swap 많이 하고, 1은 적게 한다" 한 줄 (메커니즘 설명이 없어)
  • 트레이드오프 언급 없이 "낮을수록 좋다" (장단점이 있어)

Q30. Swap 폭증 예방 방안 (서술형, 4점)¶

정답 (예시, 4가지 이상):

1. vm.swappiness 값을 1~10으로 낮추기 왜: 기본값 60은 너무 공격적으로 Swap을 사용해. 1로 설정하면 OS가 buff/cache를 최대한 회수하고 Swap을 최후의 수단으로만 사용하게 돼. /etc/sysctl.conf에 써서 영구 적용.

2. 메모리 모니터링 체계 구축 왜: available이 일정 수준 이하로 떨어지면 알림을 받아야 해. Swap이 터진 후에 알면 이미 늦어. free -h를 주기적으로 기록하거나 Grafana 같은 도구로 대시보드 구성.

3. 백업 작업을 WAS 서버가 아닌 별도 서버에서 수행 왜: 백업이 대량 디스크 읽기를 하면 buff/cache가 Free를 잡아먹고, 결국 Swap까지 이어져. 백업 서버를 분리하면 WAS 메모리에 영향을 안 줘.

4. WAS 메모리(힙) 크기를 서버 메모리 대비 적정 비율로 설정 왜: 16GB 서버에 힙을 14GB 주면 OS용 메모리가 2GB밖에 안 남아. buff/cache와 OS 프로세스가 쓸 공간이 부족해져서 Swap이 발생해. 일반적으로 전체의 60~70% 정도로 잡아.

5. 정기적으로 Swap 사용량 체크하고 누적 시 swapoff/swapon으로 리셋 왜: Swap은 비가역적이라 누적돼. 주기적으로 확인하고, Swap이 쌓이면 여유 있는 시간에 swapoff -a && swapon -a로 비워줘야 해.

채점 기준:

항목 배점
4가지 이상 방안 제시 2점
각 방안마다 "왜" 이유 포함 2점

이건 불합격

  • "메모리를 늘린다" 한 가지만 쓴 경우 (돈으로 해결하는 건 방안이 아니야)
  • 이유 없이 방안만 나열한 경우 (왜 그렇게 해야 하는지가 빠져 있어)
