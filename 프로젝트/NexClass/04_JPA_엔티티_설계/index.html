04. JPA 엔티티 설계¶

난이도: Beta | 전제 지식: Java 클래스, DB 기초 (테이블, 컬럼, PK) | 예상 학습 시간: 60분

이게 뭐야?¶

JPA 엔티티는 Java 객체와 DB 테이블을 매핑하는 클래스야. @Entity가 붙은 Java 클래스 하나 = DB 테이블 하나. 클래스의 필드 하나 = 테이블의 컬럼 하나.

NexClass에는 11개 엔티티(= 11개 테이블)가 있다. 각각의 역할, 관계, 설계 이유를 분석한다.

어떻게 돌아가?¶

JPA는 Entity 클래스를 보고 DB 테이블과 자동으로 매핑한다.

graph LR
    subgraph "Java 세계"
        E["Lesson lesson = new Lesson();<br/>lesson.setLessonCd('L001');<br/>lesson.setLessonName('OOP');"]
    end

    subgraph "JPA (Hibernate)"
        H["INSERT INTO TB_NEXCLASS_LESSON<br/>(LESSON_CD, LESSON_NAME)<br/>VALUES ('L001', 'OOP')"]
    end

    subgraph "DB 세계"
        D["TB_NEXCLASS_LESSON<br/>LESSON_CD | LESSON_NAME<br/>L001 | OOP"]
    end

    E -->|"lessonRepository.save(lesson)"| H
    H -->|"SQL 실행"| D

개발자는 SQL을 안 써도 되고, Java 객체만 다루면 JPA가 알아서 SQL을 생성해.

왜 이렇게 테이블을 나눴어?¶

SRP (Single Responsibility Principle) 관점¶

하나의 테이블은 하나의 변경 이유만 가져야 한다

Lesson 테이블에 출석 데이터까지 넣으면? 강의 정보가 바뀔 때도, 출석 데이터가 바뀔 때도 같은 테이블을 건드려야 해. 변경 이유가 2개 -> SRP 위반.

테이블 그룹 테이블 변경 이유
핵심 업무 Lesson 강의 정보 변경 시
핵심 업무 Attendance 출결 데이터 변경 시
핵심 업무 Recording 녹화 데이터 변경 시
인프라 HostPool 호스트 계정 추가/제거 시
인프라 HostAssignment 호스트 배정/반환 시
학사 동기화 Term, Course, User, CourseUser LMS 데이터 변경 시
이력 관리 SyncLog 동기화 실행 시
연결 LessonUser Registrant 등록/취소 시

코드로 보자 - 핵심 엔티티 상세 분석¶

Lesson 엔티티 (가장 중요)¶

Lesson은 NexClass의 핵심 엔티티야. LMS 강의 정보 + Zoom 미팅 정보를 합쳐서 저장해.

@Entity                                    // JPA 엔티티 선언
@Table(name = "TB_NEXCLASS_LESSON")        // 매핑할 테이블명
@Getter @Setter                            // Lombok: getter/setter 자동 생성
@Builder                                   // Lombok: 빌더 패턴
@NoArgsConstructor                         // JPA 필수: 기본 생성자
@AllArgsConstructor                        // Builder에 필요
public class Lesson {

    // ===== LMS에서 받는 필드 =====

    @Id                                    // 이 필드가 PK
    @Column(name = "LESSON_CD", length = 50)  // DB 컬럼 매핑
    private String lessonCd;               // 강의 코드 (PK, 자연키)

    @Column(name = "COURSE_CD", length = 50)
    private String courseCd;               // 교과목 코드

    @Column(name = "LESSON_NAME", length = 200)
    private String lessonName;             // 강의명

    @Column(name = "LESSON_START_DATE")
    private LocalDateTime lessonStartDate; // 시작 일시

    @Column(name = "LESSON_END_DATE")
    private LocalDateTime lessonEndDate;   // 종료 일시

    @Column(name = "LESSON_TOTAL_TIME")
    private Integer lessonTotalTime;       // 총 강의 시간 (분 단위)

    @Column(name = "LESSON_PROF_CD", length = 50)
    private String lessonProfCd;           // 교수 코드

    // ===== NexClass가 자체 생성하는 필드 =====

    @Column(name = "ZOOM_MEETING_ID", length = 50)
    private String zoomMeetingId;          // Zoom 미팅 ID (Zoom API 응답)

    @Column(name = "ZOOM_START_URL", length = 2000)
    private String zoomStartUrl;           // 호스트 입장 URL

    @Column(name = "ZOOM_JOIN_URL", length = 2000)
    private String zoomJoinUrl;            // 참가자 입장 URL

    @Column(name = "HOST_EMAIL", length = 100)
    private String hostEmail;              // 배정된 호스트 이메일

    @Column(name = "SYNC_STATUS", length = 20)
    private String syncStatus;             // 동기화 상태 (PENDING/SUCCESS/FAIL)

    // ===== 자동 관리 필드 =====

    @Column(name = "CREATED_AT", updatable = false)  // 수정 시 변경 안 함
    private LocalDateTime createdAt;

    @Column(name = "UPDATED_AT")
    private LocalDateTime updatedAt;

    @PrePersist                            // INSERT 직전에 실행
    protected void onCreate() {
        createdAt = LocalDateTime.now();   // 생성 시각 자동 설정
        updatedAt = LocalDateTime.now();
        if (syncStatus == null) {
            syncStatus = "PENDING";        // 기본값 설정
        }
    }

    @PreUpdate                             // UPDATE 직전에 실행
    protected void onUpdate() {
        updatedAt = LocalDateTime.now();   // 수정 시각 자동 갱신
    }
}

Lesson 필드를 두 그룹으로 나누는 이유

LMS에서 받는 필드: lessonCd, courseCd, lessonName, ... (LMS가 API 요청에 담아 보냄)

NexClass가 만드는 필드: zoomMeetingId, zoomStartUrl, hostEmail, ... (Zoom API 호출 결과)

하나의 테이블에 두 출처의 데이터가 합쳐져 있어. 강의를 "생성"하면 LMS 데이터가 먼저 저장되고, Zoom 미팅이 만들어지면 Zoom 데이터가 채워지는 거야.

Attendance 엔티티 (출석 기록)¶

@Entity
@Table(name = "TB_NEXCLASS_ATTENDANCE")
@Getter @Setter @NoArgsConstructor @AllArgsConstructor @Builder
public class Attendance {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)  // AUTO_INCREMENT
    @Column(name = "ATTEND_ID")
    private Long attendId;              // PK: 자동 증가 (대리키)

    @Column(name = "LESSON_CD", length = 50)
    private String lessonCd;            // 어떤 강의의 출석인지 (FK 역할)

    @Column(name = "USER_CD", length = 50)
    private String userCd;              // 학생 코드 (null이면 외부 참가자)

    @Column(name = "USER_NAME", length = 100)
    private String userName;            // Zoom에서 표시된 이름

    @Column(name = "USER_EMAIL", length = 200)
    private String userEmail;           // Zoom 참가 이메일

    @Column(name = "JOIN_TIME")
    private LocalDateTime joinTime;     // 입장 시간

    @Column(name = "LEAVE_TIME")
    private LocalDateTime leaveTime;    // 퇴장 시간

    @Column(name = "DURATION")
    private Integer duration;           // 참여 시간 (초 단위)

    @Column(name = "EXTERNAL_PARTICIPANT", length = 1)
    private String externalParticipant; // 외부 참가자? (Y/N)

    @Column(name = "ATTEND_STATUS", length = 10)
    private String attendStatus;        // 출결 판정 (PRESENT/ABSENT)

    @Column(name = "PROCESSED_AT")
    private LocalDateTime processedAt;  // Webhook 처리 시각

    @PrePersist
    protected void onCreate() {
        this.createdAt = LocalDateTime.now();
        if (this.externalParticipant == null) {
            this.externalParticipant = "N";  // 기본값: 내부 참가자
        }
    }
}

같은 학생의 여러 행이 생기는 이유

학생이 수업 도중 입퇴장을 반복하면? Zoom Reports API는 각 세션을 별도의 행으로 반환해.

김개발 입장 09:00 → 퇴장 09:30 → Attendance 1행
김개발 재입장 09:35 → 퇴장 10:00 → Attendance 2행

그래서 Attendance는 학생 단위가 아니라 세션 단위로 저장돼. API 응답 시 같은 userCd로 그룹핑해서 historyList로 묶어 반환해.

왜 테이블 2개로 분리했어?

HostPool: 호스트 계정 자체의 정보. 거의 안 바뀜 (계정 추가/삭제 시에만 변경).

HostAssignment: 배정 이력. 강의가 생성/삭제될 때마다 변경.

변경 주기가 다르면 테이블을 분리한다. 이게 SRP의 DB 적용이야.

만약 하나로 합쳤다면? HostPool에 현재 배정된 강의 코드 컬럼을 넣으면... 동시에 여러 시간대에 같은 호스트를 배정하는 게 불가능해져. HostAssignment를 별도 테이블로 두면 같은 호스트가 시간대별로 여러 강의에 배정될 수 있어.

@Entity와 @Table¶

@Entity                                // "이 클래스는 JPA 엔티티야" 선언
@Table(name = "TB_NEXCLASS_LESSON")    // "TB_NEXCLASS_LESSON 테이블에 매핑해" 지정
public class Lesson { ... }

@Table을 안 쓰면?

클래스 이름이 테이블 이름이 돼. Lesson 클래스 -> LESSON 테이블. NexClass는 TB_NEXCLASS_ 접두어를 쓰니까 @Table을 명시해야 해.

@Id와 @GeneratedValue¶

@Id
@Column(name = "LESSON_CD", length = 50)
private String lessonCd;  // LMS가 보내는 강의 코드가 곧 PK

비즈니스 의미가 있는 값을 PK로 사용. LMS가 보내는 lessonCd가 고유하니까 그걸 그대로 PK로.

@Id
@GeneratedValue(strategy = GenerationType.IDENTITY)  // AUTO_INCREMENT
@Column(name = "ATTEND_ID")
private Long attendId;  // 의미 없는 숫자가 PK

비즈니스 의미 없는 자동 증가 숫자를 PK로 사용. 출석 기록에는 자연스러운 PK가 없으니까.

NexClass에서의 사용 패턴:

PK 유형 사용하는 엔티티 이유
자연키 Lesson, HostPool, Term, Course, User 외부에서 제공하는 코드가 이미 고유
대리키 Attendance, Recording, HostAssignment, CourseUser, SyncLog, LessonUser 자연스러운 고유 값이 없거나, 같은 조합이 여러 행 가능

@Column(
    name = "LESSON_NAME",       // DB 컬럼명
    length = 200,               // VARCHAR(200)
    updatable = false,          // UPDATE 시 이 컬럼 제외
    nullable = false,           // NOT NULL 제약
    columnDefinition = "TEXT"   // 직접 DDL 지정
)

왜 @PrePersist를 쓰냐?

매번 save() 호출 전에 lesson.setCreatedAt(LocalDateTime.now()) 쓰는 거 까먹을 수 있잖아. @PrePersist를 쓰면 JPA가 알아서 호출해줘. 실수 방지.

관계 설계 분석¶

NexClass의 특이한 점: @ManyToOne을 안 쓴다¶

이게 핵심이야

보통 JPA 교과서에서는 @ManyToOne, @OneToMany로 엔티티 간 관계를 매핑한다고 가르쳐.

근데 NexClass는 관계 어노테이션을 하나도 안 써. 대신 FK 역할을 하는 String 필드로 연결해.

@Entity
public class Attendance {
    @ManyToOne
    @JoinColumn(name = "LESSON_CD")
    private Lesson lesson;  // 객체 참조로 관계 표현
}

// 사용: attendance.getLesson().getLessonName()

@Entity
public class Attendance {
    @Column(name = "LESSON_CD")
    private String lessonCd;  // 문자열로 관계 표현
}

// 사용: lessonRepository.findById(attendance.getLessonCd())

왜 이렇게 했어?

관점 @ManyToOne 방식 ID 참조 방식 (NexClass)
편의성 객체 그래프 탐색 편리 별도 조회 필요
성능 N+1 문제 발생 가능 필요할 때만 조회
복잡도 양방향 관계, cascade 관리 단순
결합도 엔티티 간 강결합 느슨한 결합
동기화 JPA가 관계 관리 직접 관리

NexClass의 선택 이유

  1. 테이블이 11개뿐이야. 복잡한 객체 그래프가 필요 없어
  2. 조인이 거의 없어. 대부분 단일 테이블 조회
  3. LMS 호환이 최우선. 엔티티 관계보다 API 응답 형식이 중요
  4. N+1 문제를 원천 차단. 연관관계가 없으니 지연 로딩 이슈도 없어

이게 YAGNI야. 필요 없는 복잡함을 추가하지 마.

Cascade Delete 전략¶

강의를 삭제하면 관련 데이터도 삭제해야 해. NexClass는 JPA Cascade가 아닌 서비스 레벨에서 직접 삭제한다.

// LessonService.java - deleteLesson()
@Transactional
public void deleteLesson(String lessonCd) {
    // 1. Zoom 미팅 삭제
    zoomService.deleteMeeting(lesson.getZoomMeetingId());

    // 2. 호스트 반환
    hostPoolService.releaseHost(lessonCd);

    // 3. 관련 데이터 삭제 (수동 Cascade)
    attendanceRepository.deleteAll(attendanceRepository.findByLessonCd(lessonCd));
    recordingRepository.deleteAll(recordingRepository.findByLessonCd(lessonCd));
    lessonUserRepository.deleteAll(lessonUserRepository.findByLessonCd(lessonCd));

    // 4. 강의 삭제
    lessonRepository.delete(lesson);
}

왜 JPA Cascade를 안 써?

// 이렇게 할 수도 있었어:
@OneToMany(mappedBy = "lesson", cascade = CascadeType.ALL, orphanRemoval = true)
private List<Attendance> attendances;

안 쓴 이유:

  1. 순서 제어: Zoom 미팅 삭제 -> 호스트 반환 -> DB 삭제. 이 순서가 중요한데, JPA Cascade는 순서를 보장하지 않아
  2. 외부 시스템 연동: Zoom API 호출이 끼어있어서, DB 삭제만으로 끝나지 않아
  3. 에러 처리: Zoom 삭제 실패해도 DB 삭제는 진행해야 해. Cascade는 이런 부분 제어가 어려워
  4. 명시성: 코드를 보면 뭐가 삭제되는지 바로 보여. Cascade는 암묵적이라 놓치기 쉬워

함정¶

JPA 엔티티 설계 시 흔한 함정

함정 1: @NoArgsConstructor 빼먹기

JPA는 리플렉션으로 엔티티를 생성하는데, 기본 생성자가 없으면 에러. Lombok의 @NoArgsConstructor를 반드시 넣어야 해.

함정 2: updatable = false 빼먹기

@Column(name = "CREATED_AT", updatable = false)  // 이거 없으면?
private LocalDateTime createdAt;

updatable = false가 없으면 UPDATE 시 createdAt도 같이 바뀔 수 있어. 생성 시각은 절대 안 바뀌어야 하니까 반드시 설정해.

함정 3: @PrePersist에서 null 체크 안 하기

@PrePersist
protected void onCreate() {
    if (syncStatus == null) {        // null 체크 필수!
        syncStatus = "PENDING";
    }
}

이미 값이 설정되어 있는데 덮어쓰면 안 돼.

함정 4: length 설정 안 하기

@Column(name = "ZOOM_START_URL")  // length 안 쓰면 기본 VARCHAR(255)
private String zoomStartUrl;       // Zoom URL은 255자를 넘을 수 있어!

Zoom URL은 매우 길어서 length = 2000으로 설정해야 해. 기본 255로 두면 데이터 잘림.

정리¶

항목 내용
총 엔티티 수 11개
PK 전략 자연키 5개 (Lesson, HostPool, Term, Course, User) + 대리키 6개
관계 매핑 @ManyToOne 미사용, ID 참조 방식 (결합도 낮춤)
Cascade JPA Cascade 미사용, 서비스 레벨에서 직접 삭제
생명주기 관리 @PrePersist (INSERT 전), @PreUpdate (UPDATE 전)
설계 원칙 SRP (변경 이유별 테이블 분리), YAGNI (불필요한 관계 매핑 제거)

한줄 정리

NexClass의 11개 엔티티는 변경 이유(SRP)에 따라 분리되었고, 불필요한 연관관계 매핑을 제거(YAGNI)하여 단순하고 유지보수하기 쉬운 구조를 유지한다.

확인문제¶

Q1. NexClass에서 @ManyToOne 대신 ID 참조를 사용한 이유 3가지를 말해봐.

  1. N+1 문제 원천 차단: 연관관계가 없으니 지연 로딩(Lazy Loading) 이슈가 발생할 수 없다

  2. 느슨한 결합: 엔티티가 서로 독립적이라, 하나를 수정해도 다른 엔티티에 영향이 없다. 예를 들어 Attendance 엔티티를 수정해도 Lesson 엔티티는 전혀 영향받지 않아

  3. 단순함: 양방향 관계 관리, cascade 설정, fetch 전략 등 복잡한 JPA 설정이 필요 없다. 테이블 11개에 대부분 단일 테이블 조회인 이 프로젝트에서 연관관계 매핑은 오버엔지니어링이야

Q2. HostPool과 HostAssignment를 왜 분리했어? 하나로 합치면 안 돼?

변경 주기가 다르기 때문.

  • HostPool: 호스트 계정 자체의 정보. 계정을 추가하거나 비활성화할 때만 변경. 거의 안 바뀜
  • HostAssignment: 호스트 배정 이력. 강의가 생성/삭제될 때마다 INSERT/DELETE. 자주 바뀜

하나로 합치면? HostPool에 "현재 배정 강의" 컬럼을 넣으면, 같은 호스트가 시간대별로 여러 강의에 배정되는 것이 불가능해져. HostAssignment를 별도로 두면 시간대 기반 배정이 가능하고, 이력도 남길 수 있어.

이게 SRP를 DB에 적용한 거야 - 변경 이유가 다르면 분리한다.

Q3. Lesson 엔티티의 @PrePersist에서 syncStatus 기본값을 'PENDING'으로 설정하는 이유는?

강의가 생성될 때 Zoom 미팅 생성이 아직 안 끝났을 수 있어.

흐름: 1. LMS 요청 수신 -> Lesson 엔티티 생성 (syncStatus = PENDING) 2. Zoom 미팅 생성 시도 3. 성공 -> syncStatus = SUCCESS 4. 실패 -> syncStatus = FAIL

@PrePersist에서 PENDING을 기본값으로 넣으면: - 어떤 경로로 save()가 호출되든 빈 상태로 저장되는 것을 방지 - 개발자가 syncStatus 설정을 까먹어도 최소한 PENDING이 들어감 - DB에서 syncStatus가 null인 행이 생기지 않음

Q4. Recording 엔티티에서 recordUrl 컬럼의 length가 500인 이유는?

Zoom의 share_url은 매우 길어. 기본 URL + 패스코드 토큰까지 합치면 200~400자 정도.

기본값 VARCHAR(255)를 쓰면 URL이 잘릴 수 있어. 그러면 학생이 녹화를 재생할 수 없게 돼.

500으로 넉넉하게 잡되, 2000까지 가지 않은 건 녹화 URL이 Zoom Start URL(호스트 입장 URL)만큼 길지 않기 때문. Lesson의 zoomStartUrl은 2000인데, 이건 OAuth 토큰이 포함되어서 정말 길거든.

Q5. JPA Cascade 대신 서비스 레벨에서 직접 삭제하는 이유를 설명해봐.

강의 삭제 시 처리 순서: 1. Zoom 미팅 삭제 (외부 API 호출) 2. 호스트 반환 (HostAssignment 삭제) 3. 출석/녹화/수강생 데이터 삭제 4. 강의 삭제

JPA Cascade로는 이걸 못 해:

  • 순서 보장 불가: Cascade는 JPA가 알아서 순서를 정해. Zoom API를 먼저 호출해야 하는데 이걸 보장 못 해
  • 외부 시스템 연동 불가: Cascade는 DB 삭제만 해. Zoom API 호출은 못 해
  • 에러 처리 불가: Zoom 삭제 실패해도 DB 삭제는 진행해야 하는데, Cascade는 이런 분기가 불가능
  • 명시성: 코드에 삭제 대상이 명확히 보여. Cascade는 암묵적이라 "이것도 삭제되나?"를 엔티티 정의를 봐야 알 수 있어
