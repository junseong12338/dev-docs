30. 배포와 인프라¶

이 챕터에서 배우는 것¶

지금까지 29개 챕터에 걸쳐 NexClass의 코드를 분석했다. 근데 코드만 잘 짰다고 끝이야? 아니야. 돌아가는 서버에 올라가야 비로소 의미가 있다.

이 챕터에서는 NexClass가 어떻게 서버에 배포되고, 외부와 어떻게 통신하는지 다룬다.

주제 핵심
개발 환경 localhost:8082 + ngrok
프로덕션 환경 iwinv 서버 + Nginx + SSL
배포 방식 JAR 패키징 → 서버 전송 → nohup 실행
환경별 설정 application.properties 관리

개발 환경 구성¶

localhost:8082¶

NexClass 개발 환경은 단순하다.

# 로컬에서 Spring Boot 실행
mvn spring-boot:run
# 또는 IDE에서 NexClassApplication.java 실행

기본 포트는 8082다.

# application.properties
server.port = 8082

로컬에서 잘 돌아간다. API도 호출된다. 대시보드도 보인다.

근데 문제가 하나 있다.

Zoom Webhook은 public URL로만 보낼 수 있다. localhost:8082는 네 컴퓨터에서만 접근 가능한 주소야. Zoom 서버가 네 컴퓨터에 직접 접근할 방법이 없다.

ngrok -- 왜 필요한가?¶

Zoom Webhook의 요구사항¶

Zoom이 meeting.ended나 recording.completed 이벤트를 보내려면, 인터넷에서 접근 가능한 URL이 필요하다.

Zoom 서버 → ??? → localhost:8082/webhook/zoom

이 ???를 채워주는 게 ngrok이다.

ngrok이 뭐야?¶

ngrok은 로컬 서버를 인터넷에 노출시켜주는 터널링 서비스다.

쉽게 말해서, Zoom이 https://xxx.ngrok-free.app/webhook/zoom으로 요청을 보내면, ngrok이 그걸 받아서 네 컴퓨터의 localhost:8082/webhook/zoom으로 전달해준다.

ngrok 동작 원리¶

sequenceDiagram
    participant Z as Zoom 서버
    participant N as ngrok 서버 (클라우드)
    participant A as ngrok 에이전트 (로컬)
    participant S as NexClass (localhost:8082)

    Note over A,N: 1. ngrok 에이전트가 ngrok 서버에 터널 연결
    A->>N: TCP 터널 수립 (Outbound)
    N-->>A: 공개 URL 할당 (https://xxx.ngrok-free.app)

    Note over Z,S: 2. Zoom Webhook 수신 흐름
    Z->>N: POST https://xxx.ngrok-free.app/webhook/zoom
    N->>A: 터널을 통해 요청 전달
    A->>S: localhost:8082/webhook/zoom으로 포워딩
    S-->>A: 200 OK 응답
    A-->>N: 응답 전달
    N-->>Z: 200 OK

핵심은 이거다: ngrok 에이전트가 먼저 ngrok 서버에 연결한다 (Outbound). 그래서 방화벽이 있어도 동작해. 외부에서 들어오는 게 아니라 내부에서 나가는 연결이니까.

ngrok 실행 방법¶

# 기본 실행 (임시 URL 발급)
ngrok http 8082

# 고정 도메인 사용 (NexClass 실제 설정)
ngrok http --url=zena-tasteful-guillermo.ngrok-free.app 8082

주의사항

ngrok 무료 플랜의 임시 URL은 재시작할 때마다 바뀐다. 그때마다 Zoom Marketplace에서 Webhook URL을 변경해야 한다. NexClass는 ngrok 유료 플랜의 고정 도메인을 사용해서 이 문제를 해결했다.

ngrok의 한계¶

ngrok은 어디까지나 개발/테스트용이다.

항목 ngrok 프로덕션
안정성 ngrok 서버 장애 시 불통 직접 관리 가능
속도 추가 홉(hop) 발생 직접 연결
보안 제3자 서버 경유 직접 SSL 관리
비용 무료/유료 플랜 서버 비용만

JAR 배포 방식¶

Spring Boot의 실행 가능 JAR¶

Spring Boot는 Fat JAR(Uber JAR)를 만든다. 의존성 라이브러리, 설정 파일, 임베디드 톰캣까지 전부 하나의 JAR 파일에 포함된다.

<!-- pom.xml에서 JAR 이름 설정 -->
<build>
    <finalName>NexClass</finalName>
    <!-- ... -->
</build>

이 설정 덕분에 빌드하면 NexClass.jar라는 이름으로 생성된다.

배포 과정¶

flowchart LR
    A["mvn package<br/>(로컬 빌드)"] --> B["NexClass.jar<br/>생성"]
    B --> C["scp로 서버 전송"]
    C --> D["기존 프로세스 종료"]
    D --> E["nohup java -jar<br/>NexClass.jar &"]
    E --> F["동작 확인<br/>jps -l"]

실제 명령어¶

# 로컬에서 JAR 빌드 (테스트 스킵)
mvn clean package -DskipTests
# 결과물: target/NexClass.jar

# SCP로 서버에 JAR 전송
scp target/NexClass.jar {USER}@49.247.41.134:/usr/local/nexclass/

# 실행 중인 NexClass 프로세스 찾기
jps -l | grep NexClass
# 출력: 12345 NexClass.jar

# 프로세스 종료
kill 12345

# nohup으로 백그라운드 실행
cd /usr/local/nexclass
nohup java -jar NexClass.jar &

# 프로세스 확인
jps -l
# 또는
sudo lsof -i :8082

# 헬스 체크
curl http://localhost:8082/health
# {"status":"UP","message":"NexClass is healthy!"}

nohup이 뭐야?¶

nohup은 SSH 세션이 끊어져도 프로세스를 유지시켜주는 명령어다.

# nohup 없이 실행하면
java -jar NexClass.jar &
# → SSH 접속 끊으면 프로세스도 죽는다

# nohup으로 실행하면
nohup java -jar NexClass.jar &
# → SSH 끊어도 계속 돌아간다
# → 출력은 nohup.out 파일에 저장된다

왜 systemd 서비스로 안 만들었나?

systemd 서비스로 등록하면 자동 재시작, 로그 관리 등 장점이 많다. 하지만 NexClass는 아직 프로덕션 정식 오픈 전이고, 빠른 배포/재시작이 우선이다. 정식 운영 시에는 systemd 유닛 파일을 만들어야 한다. 이것도 트레이드오프야. 편의성(nohup) vs 안정성(systemd).

Nginx 리버스 프록시¶

리버스 프록시가 뭐야?¶

리버스 프록시는 클라이언트와 애플리케이션 서버 사이에서 요청을 중계하는 서버다.

왜 필요하냐고?

이유 설명
포트 숨기기 클라이언트가 :8082를 알 필요 없다. :443(HTTPS)으로 접근
SSL 처리 Nginx가 SSL 인증서를 관리. 애플리케이션은 HTTP만 처리
정적 파일 CSS, JS, 이미지를 Nginx가 직접 서빙 (성능 향상)
로드 밸런싱 서버가 여러 대면 요청을 분배 (현재는 1대라 해당 없음) 
sequenceDiagram
    participant C as 클라이언트
    participant N as Nginx (:443)
    participant A as NexClass (:8082)

    C->>N: HTTPS 요청 (nexclass.knu10.cc)
    Note over N: SSL 복호화 (HTTPS → HTTP)
    N->>A: HTTP 요청 (localhost:8082)
    A-->>N: HTTP 응답
    Note over N: SSL 암호화 (HTTP → HTTPS)
    N-->>C: HTTPS 응답

Nginx 설정¶

# /etc/nginx/sites-available/nexclass
server {
    listen 80;
    server_name nexclass.knu10.cc;

    # HTTP → HTTPS 리다이렉트
    return 301 https://$server_name$request_uri;
}

server {
    listen 443 ssl;
    server_name nexclass.knu10.cc;

    # SSL 인증서 (Let's Encrypt)
    ssl_certificate /etc/letsencrypt/live/nexclass.knu10.cc/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/nexclass.knu10.cc/privkey.pem;

    # 리버스 프록시 -- NexClass로 전달
    location / {
        proxy_pass http://localhost:8082;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}

각 설정이 하는 일¶

설정 역할
listen 80 HTTP(평문) 요청 수신
return 301 https://... HTTP를 HTTPS로 강제 리다이렉트
listen 443 ssl HTTPS 요청 수신
ssl_certificate SSL 인증서 경로 (Let's Encrypt)
proxy_pass 요청을 localhost:8082로 전달
X-Real-IP 원래 클라이언트 IP를 헤더에 담아서 전달
X-Forwarded-Proto 원래 프로토콜(HTTPS)을 헤더에 담아서 전달

proxy_set_header가 왜 중요해?

Nginx가 중간에 있으면 NexClass 입장에서 모든 요청이 localhost에서 온 것처럼 보인다. X-Real-IP 헤더가 없으면 로그에 클라이언트 IP 대신 127.0.0.1만 찍힌다. X-Forwarded-Proto가 없으면 Spring Boot가 HTTP로 착각해서 리다이렉트 URL이 깨질 수 있다.

SSL/TLS (Let's Encrypt)¶

SSL이 왜 필요해?¶

Zoom Webhook은 HTTPS URL만 허용한다. HTTP로는 Webhook을 보내주지 않아.

그리고 API 통신에 JWT, AES 암호화 키가 오가는데, HTTP(평문)면 네트워크 중간에서 그대로 읽힌다.

Let's Encrypt¶

Let's Encrypt는 무료 SSL 인증서를 발급해주는 인증 기관이다.

# certbot으로 Let's Encrypt 인증서 발급
sudo certbot certonly --nginx -d nexclass.knu10.cc

# 인증서 위치
# /etc/letsencrypt/live/nexclass.knu10.cc/fullchain.pem  (인증서)
# /etc/letsencrypt/live/nexclass.knu10.cc/privkey.pem    (개인키)

인증서 갱신

Let's Encrypt 인증서는 90일마다 갱신해야 한다. certbot은 자동 갱신 크론을 설정해준다. 

# 자동 갱신 확인
sudo certbot renew --dry-run

NexClass 실제 상황¶

NexClass 서버의 SSL 상황을 정리하면 이렇다.

항목 상태
knu10.cc 메인 도메인 GoGetSSL 인증서 (hub, lms, www 커버)
nexclass.knu10.cc Let's Encrypt 발급 완료
포트 80/443 iwinv 네트워크 프록시가 앞단 관리
포트 8082 외부 접근 차단됨

포트 8082가 외부에서 차단되어 있기 때문에, 반드시 Nginx를 거쳐서 접근해야 한다. 이것 자체가 보안 계층이 하나 더 있는 셈이야.

문제: 개발 vs 프로덕션 설정이 다르다¶

# 개발 환경

# 프로덕션 환경

Spring Boot Profile 전략¶

Spring Boot는 Profile로 환경별 설정을 분리할 수 있다.

src/main/resources/
    application.properties           # 공통 설정
    application-dev.properties       # 개발 전용
    application-prod.properties      # 프로덕션 전용

# 개발 환경 실행
java -jar NexClass.jar --spring.profiles.active=dev

# 프로덕션 환경 실행
java -jar NexClass.jar --spring.profiles.active=prod

NexClass 현재 상황

NexClass는 현재 Profile 분리를 하지 않고 단일 application.properties를 사용한다. 서버 배포 시 직접 설정 파일을 수정하는 방식이다. 프로덕션 정식 운영 시에는 Profile 분리가 필요하다.

설정 파일에서 절대 하면 안 되는 것¶

보안 사고 예방

  • application.properties에 비밀번호를 하드코딩하고 Git에 커밋하지 마라
  • .gitignore에 반드시 추가하거나, 환경 변수로 관리해라
  • NexClass의 jwt.secret, zoom.client-secret, lms.sync.aes-key 등은 전부 민감 정보다

환경 변수를 사용하는 방법:

# application.properties에서 환경 변수 참조
jwt.secret=${JWT_SECRET}
zoom.client-secret=${ZOOM_CLIENT_SECRET}

# 서버에서 환경 변수 설정 후 실행
export JWT_SECRET=NexClassSecretKeyForJwt...
export ZOOM_CLIENT_SECRET=IbR1dk1ZxDRX...
nohup java -jar NexClass.jar &

확인 문제¶

1. Zoom Webhook 개발 시 ngrok이 필요한 이유는?

Zoom Webhook은 public URL(인터넷에서 접근 가능한 주소)로만 이벤트를 전송하기 때문이다.

localhost:8082는 개발자 컴퓨터에서만 접근 가능한 주소다. Zoom 서버는 인터넷에 있으므로 localhost에 접근할 수 없다. ngrok은 인터넷에서 접근 가능한 URL을 만들어서 로컬 서버로 터널링해준다.

2. Nginx 리버스 프록시에서 proxy_set_header X-Real-IP 설정이 없으면 어떤 문제가 발생하나?

NexClass 애플리케이션에서 모든 요청의 클라이언트 IP가 127.0.0.1(localhost)로 기록된다.

Nginx가 요청을 중계하면, NexClass 입장에서 요청 출처는 Nginx(같은 서버의 localhost)다. 실제 클라이언트 IP를 알려면 Nginx가 X-Real-IP 헤더에 원본 IP를 담아서 전달해야 한다. 이 정보가 없으면 로그 분석, 보안 감사, 접근 제어에 문제가 생긴다.

3. NexClass가 nohup 대신 systemd 서비스를 사용하면 어떤 장점이 있나?

자동 재시작, 로그 관리, 부팅 시 자동 실행 등 운영 안정성이 크게 향상된다.

  • 프로세스가 죽으면 systemd가 자동으로 재시작해준다 (Restart=always)
  • journalctl로 로그를 체계적으로 관리할 수 있다
  • 서버 재부팅 시 자동으로 애플리케이션이 시작된다 (enable)
  • 현재 nohup 방식은 프로세스가 죽으면 수동으로 재시작해야 하고, 서버 재부팅 시에도 수동 실행이 필요하다
4. application.properties에 비밀번호를 하드코딩하고 Git에 커밋하면 어떤 보안 문제가 발생하나?

Git 히스토리에 비밀번호가 영구 보존되어, 나중에 파일을 삭제해도 과거 커밋에서 복원할 수 있다.

  • 저장소가 public이면 전 세계에 비밀번호가 공개된다
  • private 저장소라도 접근 권한이 있는 모든 사람이 비밀번호를 볼 수 있다
  • git filter-branch나 BFG로 히스토리를 정리해야 하는데, 이미 clone한 사람의 로컬에는 남아 있다
  • 해결: 환경 변수(${JWT_SECRET})나 외부 설정 서버를 사용해라
5. 포트 8082가 외부에서 차단된 것이 보안에 어떤 이점을 주는가?

애플리케이션에 직접 접근할 수 없으므로, 반드시 Nginx를 거쳐야 한다.

  • Nginx의 SSL 처리를 우회할 수 없다 (평문 HTTP로 직접 접근 불가)
  • Nginx의 요청 필터링, 속도 제한 등 보안 기능이 항상 적용된다
  • 공격 표면(attack surface)이 줄어든다 (외부에서 보이는 건 80/443 포트뿐)
  • 만약 8082가 열려 있으면 http://49.247.41.134:8082로 SSL 없이 직접 접근이 가능해진다
