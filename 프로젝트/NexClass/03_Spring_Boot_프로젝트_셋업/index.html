03. Spring Boot 프로젝트 셋업¶

난이도: Alpha | 전제 지식: Spring Boot 기초 | 예상 학습 시간: 35분

이게 뭐야?¶

application.properties는 Spring Boot 애플리케이션의 모든 설정이 모여있는 파일이야. 데이터베이스 연결 정보, 보안 설정, 외부 API 키, 스케줄러 설정 등 애플리케이션이 동작하는 데 필요한 모든 값이 여기에 있다.

이 파일 하나를 완전히 이해하면, 프로젝트가 어떤 외부 시스템과 어떻게 연결되는지 전체 그림이 보여.

어떻게 돌아가?¶

Spring Boot는 애플리케이션 시작 시 application.properties를 자동으로 읽어서, @Value 어노테이션이 붙은 필드에 값을 주입한다.

graph LR
    subgraph "application.properties"
        P1["jwt.secret = {JWT_SECRET_KEY}"]
        P2["zoom.client-id = {ZOOM_CLIENT_ID}"]
        P3["spring.datasource.url = jdbc:mariadb://..."]
    end

    subgraph "Java 코드"
        J1["@Value('${jwt.secret}')<br/>private String secretKey;"]
        J2["@Value('${zoom.client-id}')<br/>private String clientId;"]
        J3["자동 설정<br/>DataSource Bean 생성"]
    end

    P1 -->|"값 주입"| J1
    P2 -->|"값 주입"| J2
    P3 -->|"자동 설정"| J3

@Value vs 자동 설정

직접 주입 (@Value): 개발자가 명시적으로 @Value("${key}") 써서 가져옴

자동 설정 (Auto Configuration): spring.datasource.* 같은 표준 프로퍼티는 Spring Boot가 알아서 Bean으로 만들어줌. 코드에서 @Value를 안 써도 DataSource가 자동 생성됨

코드로 보자¶

NexClass의 application.properties를 섹션별로 완전 분석한다.

1. 서버 기본 설정¶

spring.application.name = NexClass    # 애플리케이션 이름 (로그, 모니터링에서 사용)
server.port = 8082                    # 톰캣 포트 (기본값 8080이 아닌 8082)

왜 8082야?

기본 포트 8080은 다른 서비스가 이미 쓰고 있을 수 있어. 같은 서버에 LMS(8080)가 이미 돌고 있다면, NexClass는 다른 포트를 써야 하잖아. 포트 충돌은 서버 운영에서 자주 발생하는 문제야.

2. JWT 설정¶

jwt.secret = {JWT_SECRET_KEY}         # JWT 토큰 서명에 사용하는 비밀키
jwt.expiration = 300000               # 토큰 유효 시간 (밀리초, 300000ms = 5분)

코드에서의 사용:

// JwtProvider.java
@Value("${jwt.secret}")       // application.properties에서 값 주입
private String secretKey;     // {JWT_SECRET_KEY} 값이 들어옴

@Value("${jwt.expiration}")
private long expiration;      // 300000 (5분)

public String createToken(String subject) {
    Date now = new Date();
    Date expireDate = new Date(now.getTime() + expiration);  // 현재 + 5분

    // secretKey로 HMAC-SHA 키 생성
    SecretKey key = Keys.hmacShaKeyFor(secretKey.getBytes(StandardCharsets.UTF_8));

    return Jwts.builder()
            .subject(subject)      // 토큰 주인 (userCd 또는 apiKey)
            .issuedAt(now)         // 발급 시간
            .expiration(expireDate) // 만료 시간
            .signWith(key)         // 서명
            .compact();            // 토큰 문자열 생성
}

JWT 비밀키 보안

jwt.secret은 절대 외부에 노출되면 안 되는 값이야. 이 키를 알면 누구나 유효한 JWT 토큰을 만들 수 있어 = 인증 우회.

프로덕션에서는 환경 변수나 Vault 같은 시크릿 관리 도구를 써야 해. 소스코드에 하드코딩하면 Git에 올라가는 순간 끝이야.

왜 5분이야?

토큰 유효 시간이 길면 탈취 시 위험하고, 짧으면 매번 재발급해야 해.

NexClass의 경우:

  • LMS가 API 호출 전에 토큰을 발급받음
  • 하나의 작업(강의 생성 등)은 보통 몇 초면 끝남
  • 5분이면 충분하고, 탈취되어도 5분 후 만료

3. API 인증 키 설정¶

nexclass.api.key = {API_KEY}          # LMS를 식별하는 API 키
nexclass.api.secret = {API_SECRET}    # API 키에 대한 비밀 키 (JWT 서명 검증용)

코드에서의 사용:

// AuthService.java
@Value("${nexclass.api.key}")
private String apiKey;         // LMS가 보낸 issuer(iss)와 비교

@Value("${nexclass.api.secret}")
private String apiSecret;     // LMS JWT 서명 검증에 사용

private TokenResponse authenticateByLmsToken(String requestToken) {
    // 1. apiSecret으로 서명 키 생성
    SecretKey key = Keys.hmacShaKeyFor(apiSecret.getBytes(StandardCharsets.UTF_8));

    // 2. LMS가 보낸 JWT를 검증
    Claims claims = Jwts.parser()
            .verifyWith(key)
            .build()
            .parseSignedClaims(requestToken)
            .getPayload();

    // 3. issuer가 apiKey와 일치하는지 확인
    String issuer = claims.get("iss", String.class);
    if (!apiKey.equals(issuer)) {
        throw new IllegalArgumentException("Invalid API Key");
    }

    // 4. NexClass 자체 JWT 토큰 발급
    String userId = claims.get("uId", String.class);
    String token = jwtProvider.createToken(userId != null ? userId : apiKey);
    // ...
}

인증 흐름 이해

LMS가 자기만의 JWT를 만듦 (apiSecret으로 서명, apiKey를 issuer로)
   ↓
NexClass에 토큰 요청 (requestToken에 LMS JWT 담아서)
   ↓
NexClass가 apiSecret으로 서명 검증 + apiKey로 issuer 확인
   ↓
검증 통과하면 NexClass 자체 JWT 발급

두 시스템이 같은 키(apiSecret)를 공유하고 있어야 이게 돌아가. 이걸 대칭키 기반 인증이라고 해.

4. Zoom API 설정¶

zoom.account-id = {ZOOM_ACCOUNT_ID}       # Zoom 계정 ID
zoom.client-id = {ZOOM_CLIENT_ID}         # Zoom 앱 Client ID
zoom.client-secret = {ZOOM_CLIENT_SECRET} # Zoom 앱 Client Secret
zoom.base-url = https://api.zoom.us/v2    # Zoom REST API 기본 URL
zoom.oauth-url = https://zoom.us/oauth/token  # Zoom OAuth 토큰 발급 URL

코드에서의 사용:

// ZoomService.java
@Value("${zoom.account-id}")
private String accountId;        // S2S OAuth에서 account_id 파라미터

@Value("${zoom.client-id}")
private String clientId;         // Basic Auth 헤더에 사용

@Value("${zoom.client-secret}")
private String clientSecret;     // Basic Auth 헤더에 사용

public String getAccessToken() {
    // client_id:client_secret을 Base64 인코딩
    String credentials = clientId + ":" + clientSecret;
    String encodedCredentials = Base64.getEncoder()
            .encodeToString(credentials.getBytes(StandardCharsets.UTF_8));

    HttpHeaders headers = new HttpHeaders();
    headers.set("Authorization", "Basic " + encodedCredentials);

    // grant_type: account_credentials (S2S OAuth 방식)
    MultiValueMap<String, String> body = new LinkedMultiValueMap<>();
    body.add("grant_type", "account_credentials");
    body.add("account_id", accountId);
    // ...
}

S2S OAuth가 뭐야?

Server-to-Server OAuth. 사용자 로그인 없이 서버 간 직접 인증하는 방식이야.

일반 OAuth는 "사용자가 로그인 -> 동의 -> 토큰 발급" 순서인데, S2S는 "서버가 client_id + client_secret으로 직접 토큰 요청".

왜 S2S? NexClass는 서버야. 사람이 로그인할 필요가 없어.

5. LMS 동기화 설정¶

lms.sync.base-url = https://lms.knu10.cc  # LMS API 서버 주소
lms.sync.org-cd = ORG0000001              # 조직 코드 (LMS 식별용)
lms.sync.aes-key = {AES_KEY}              # AES-256 복호화 키 (LMS 응답 해독)
lms.sync.page-size = 1000                 # 페이징 단위 (한 번에 1000건)

코드에서의 사용:

// LmsSyncService.java
@Value("${lms.sync.base-url}")
private String baseUrl;       // LMS API 호출 시 기본 URL

@Value("${lms.sync.aes-key}")
private String aesKey;        // LMS 응답 복호화 키

private Map<String, Object> callLmsApi(String type, Integer pageSize, Integer pageNumber) {
    // ...
    // LMS 응답은 AES-256으로 암호화되어 있음
    AES256Util aes = new AES256Util(aesKey);
    String json = aes.decrypt(encrypted);  // 암호문 -> JSON 평문
    return objectMapper.readValue(json, new TypeReference<>() {});
}

왜 LMS 응답이 암호화되어 있어?

LMS API 응답에는 학생 개인정보(이름, 학번, 이메일)가 들어있어. 네트워크를 통해 전송되는 데이터가 탈취되면 개인정보 유출이야. 그래서 LMS가 AES-256으로 암호화해서 보내고, NexClass가 같은 키로 복호화하는 거야.

6. MariaDB 설정¶

# JDBC 연결 URL (호스트, 포트, 데이터베이스명, 옵션)
spring.datasource.url = jdbc:mariadb://{DB_HOST}:3306/lxpknu10?allowMultiQueries=true&useUnicode=true

spring.datasource.username = {DB_USERNAME}     # DB 접속 계정
spring.datasource.password = {DB_PASSWORD}     # DB 접속 비밀번호
spring.datasource.driver-class-name = org.mariadb.jdbc.Driver  # JDBC 드라이버
spring.datasource.hikari.max-lifetime = 28000  # 커넥션 최대 생존 시간 (밀리초)

max-lifetime이 왜 28초야?

MariaDB의 기본 wait_timeout이 28800초 (8시간)인데, HikariCP의 기본 maxLifetime은 30분이야.

문제는 MariaDB가 타임아웃 전에 커넥션을 끊는 경우가 있어. 그러면 HikariCP가 이미 죽은 커넥션을 쓰려다 에러가 나.

28000(28초)으로 짧게 설정하면 커넥션을 자주 갱신하니까 안전해. 대신 커넥션 생성 비용이 늘어나는 트레이드오프가 있어.

이건 실제로 HikariPool 경고 로그가 발생해서 추가한 설정이야.

JDBC URL 옵션 분석

옵션 의미
allowMultiQueries=true 하나의 Statement에서 여러 SQL 실행 허용
useUnicode=true 유니코드 문자(한글 등) 지원

7. JPA 설정¶

spring.jpa.hibernate.ddl-auto = none           # DDL 자동 생성 안 함 (테이블 직접 관리)
spring.jpa.show-sql = true                     # 실행되는 SQL을 콘솔에 출력
spring.jpa.properties.hibernate.format_sql = true       # SQL을 보기 좋게 포맷팅
spring.jpa.properties.hibernate.dialect = org.hibernate.dialect.MariaDBDialect  # MariaDB 전용 SQL 생성

ddl-auto 옵션 비교:

값 의미 사용 시기
none 아무것도 안 함 프로덕션 (테이블을 직접 관리)
validate 엔티티와 테이블 일치 검증만 프로덕션 권장
update 변경사항 자동 반영 개발 환경
create 매번 테이블 새로 생성 테스트
create-drop 시작 시 생성, 종료 시 삭제 테스트

ddl-auto = update를 프로덕션에서 쓰면?

절대 쓰지 마. Hibernate가 ALTER TABLE을 자동 실행하는데:

  1. 컬럼 삭제는 안 해 (추가만 함) -> 쓰레기 컬럼 누적
  2. 인덱스 관리가 제멋대로
  3. 실수로 엔티티 필드 이름 바꾸면 데이터 손실

프로덕션에서는 none 또는 validate만 쓰고, DDL 변경은 DBA가 직접 SQL로 실행해야 해.

8. Zoom Webhook 설정¶

zoom.webhook.secret-token = {WEBHOOK_SECRET_TOKEN}  # Webhook 서명 검증 토큰

코드에서의 사용:

// WebhookController.java
@Value("${zoom.webhook.secret-token}")
private String secretToken;    // Zoom이 Webhook 보낼 때 서명에 사용하는 토큰

// HMAC-SHA256 서명 검증: Zoom이 보낸 서명과 우리가 만든 서명 비교
private boolean verifySignature(String rawBody, String timestamp, String zoomSignature) {
    String message = "v0:" + timestamp + ":" + rawBody;  // Zoom 서명 규칙
    String expectedSignature = "v0=" + hmacSha256(message, secretToken);
    return expectedSignature.equals(zoomSignature);  // 일치하면 진짜 Zoom이 보낸 거
}

왜 서명 검증을 해?

아무나 POST /webhook/zoom에 가짜 데이터를 보낼 수 있잖아. "미팅 끝났다"는 가짜 Webhook을 보내면 출결이 엉망이 돼.

서명 검증이 있으면:

  1. Zoom만 아는 secretToken으로 서명
  2. NexClass가 같은 secretToken으로 검증
  3. 서명이 안 맞으면 -> 401 Unauthorized 응답, 무시

Spring Boot 자동 설정이 뭘 해주는지¶

application.properties에 spring.datasource.*를 설정하면, Spring Boot가 자동으로:

자동 설정 뭘 해주냐면
DataSource HikariCP 커넥션 풀 + MariaDB 드라이버로 DataSource Bean 생성
EntityManagerFactory JPA EntityManager 자동 구성
TransactionManager @Transactional 어노테이션 활성화
내장 톰캣 server.port에 맞는 포트로 웹서버 자동 시작
Jackson @RequestBody/@ResponseBody JSON 직렬화/역직렬화
Thymeleaf src/main/resources/templates/ 경로 자동 인식

자동 설정의 원리

Spring Boot는 spring-boot-autoconfigure JAR 안에 수백 개의 AutoConfiguration 클래스가 있어. 클래스패스에 어떤 라이브러리가 있는지 감지하고, 조건에 맞는 Bean을 자동 생성해.

예: spring-boot-starter-data-jpa가 있으면 → JpaRepositoriesAutoConfiguration 활성화 → @Repository 인터페이스에 대한 구현체 자동 생성

이게 "Convention over Configuration" (설정보다 관례) 철학이야.

프로파일 관리 (dev vs prod)¶

NexClass는 현재 단일 properties 파일을 쓰고 있어. 프로덕션과 개발 환경이 분리되어 있지 않다.

이건 아직 해결 안 된 부분이야

원래는 이렇게 분리해야 해:

application.properties           # 공통 설정
application-dev.properties       # 개발 환경 (로컬 DB, 디버그 로그)
application-prod.properties      # 운영 환경 (운영 DB, 에러 로그만)

실행 시 프로파일 지정:

# 개발 환경
java -jar NexClass.jar --spring.profiles.active=dev

# 운영 환경
java -jar NexClass.jar --spring.profiles.active=prod

분리해야 하는 설정값들:

함정¶

설정 관련 흔한 함정들

함정 1: 비밀 키를 소스코드에 하드코딩

application.properties가 Git에 올라가면 모든 비밀 키가 노출돼. .gitignore에 추가하거나, 환경 변수로 관리해야 해.

# 이렇게 하면 환경 변수에서 읽음
jwt.secret = ${JWT_SECRET_KEY}
zoom.client-secret = ${ZOOM_CLIENT_SECRET}

함정 2: show-sql=true를 프로덕션에서 사용

모든 SQL이 로그에 찍히면 성능 저하 + 로그 폭발. 개발 환경에서만 쓰고, 프로덕션에서는 반드시 false로.

함정 3: DB 커넥션 풀 설정 무시

기본 설정으로 놔두면 트래픽 몰릴 때 커넥션 고갈돼. HikariCP의 max-lifetime, maximum-pool-size를 환경에 맞게 튜닝해야 해.

함정 4: cron 표현식 실수

0 0 3 * * *를 0 3 * * *로 쓰면 매 분 3초마다 실행돼. Spring의 cron은 6자리(초 분 시 일 월 요일)이고, Unix cron은 5자리(분 시 일 월 요일). 헷갈리면 큰일 나.

한줄 정리

application.properties는 애플리케이션의 모든 외부 의존성 연결 정보와 동작 설정이 모여있는 파일이야. 이 파일 하나를 읽으면 프로젝트가 어떤 시스템과 어떻게 연결되는지 전체 그림이 보인다.

확인문제¶

Q1. jwt.expiration이 300000인 이유를 보안 관점에서 설명해봐.

300000밀리초 = 5분.

토큰 유효 시간이 짧을수록 보안에 유리해:

  • 토큰이 탈취되어도 5분 후 만료 -> 피해 최소화
  • 하나의 API 호출은 보통 수 초면 끝남 -> 5분이면 충분

반대로 유효 시간이 길면(예: 24시간):

  • 탈취 시 하루 종일 악용 가능
  • 하지만 매번 재발급 안 해도 되니 편리

트레이드오프: 보안 vs 편의성. NexClass는 서버 간 통신이라 재발급이 어렵지 않으니 5분으로 짧게 설정한 거야.

Q2. spring.jpa.hibernate.ddl-auto=none인 이유는?

NexClass는 운영 시스템이야. 테이블 구조 변경은 DBA가 직접 SQL로 실행해야 해.

update를 쓰면:

  • Hibernate가 자동으로 ALTER TABLE 실행 -> 예상 못 한 DDL 변경
  • 컬럼 삭제는 안 해줌 -> 쓰레기 컬럼 누적
  • 인덱스 관리가 제멋대로 -> 성능 문제

none이면 Hibernate가 테이블에 아무것도 안 해. Entity와 테이블이 안 맞으면 런타임 에러가 나지만, 그게 차라리 낫지. 의도하지 않은 DDL 변경보다 에러가 낫다.

Q3. Zoom S2S OAuth에서 client_id와 client_secret을 Base64로 인코딩하는 이유는?

HTTP Basic Authentication 표준 규격이야.

Authorization: Basic base64(client_id:client_secret)

RFC 7617에 정의된 방식:

  1. client_id:client_secret 형식으로 합침
  2. Base64로 인코딩
  3. Authorization: Basic {인코딩된 값} 헤더에 넣음

Base64는 암호화가 아니야 (누구나 디코딩 가능). 단순히 특수문자를 안전하게 HTTP 헤더에 넣기 위한 인코딩이야. 실제 보안은 HTTPS가 담당해.

Q4. LMS 응답을 AES-256으로 암호화하는 이유는?

LMS API 응답에는 학생 개인정보가 포함돼 (이름, 학번, 이메일).

HTTPS만으로도 전송 중 암호화는 되지만, 이중 암호화(Defense in Depth)의 일환이야:

  1. HTTPS: 전송 계층 암호화 (네트워크 도청 방지)
  2. AES-256: 애플리케이션 계층 암호화 (중간자 공격, 로그 노출 방지)

특히 서버 로그에 응답이 찍히더라도, AES로 암호화되어 있으면 평문 개인정보가 로그에 안 남아.

Q5. nexclass.dummy-email-domain의 역할은?

Zoom Registrant API는 참가자 등록 시 이메일이 필수야. 근데 학생들은 Zoom 계정이 없어.

그래서 userCd@nexclass.local 형식의 더미 이메일을 만들어 Zoom에 등록하는 거야.

나중에 출결 처리할 때 이 더미 이메일의 @ 앞부분(userCd)을 추출해서 학생을 식별해.

등록: "USER001@nexclass.local" → Zoom에 Registrant 등록
매칭: Zoom Reports API에서 "USER001@nexclass.local" → @ 앞 = "USER001" → DB에서 학생 찾기

nexclass.local은 실제로 존재하지 않는 도메인이야. 메일이 발송될 일이 없으니까 아무 도메인이나 써도 돼.
