17. 비동기 처리와 @Async¶

난이도: Delta | 선수 지식: 15_Webhook_수신과_검증, Spring Boot 기초, 스레드 개념

이 챕터에서 배우는 것¶

학습 목표

  • 동기 vs 비동기의 차이를 본질적으로 이해한다
  • Spring @Async가 어떻게 동작하는지 안다
  • NexClass에서 @Async를 쓰는 3가지 곳과 각각의 이유를 설명할 수 있다
  • @Async의 치명적 함정들을 알고 피할 수 있다
  • 스레드 풀 설정의 트레이드오프를 이해한다

1. 동기 vs 비동기¶

동기 (Synchronous) -- 순서대로, 기다림¶

비유: 음식점에서 음식 주문

동기: 카운터에서 주문하고, 음식 나올 때까지 그 자리에서 기다린다. 내 뒤에 줄 선 사람들도 다 기다려야 해. 비동기: 주문하고 진동벨 받아서 자리에 앉는다. 음식 나오면 벨이 울려서 가져온다. 뒤에 줄 선 사람들은 바로 주문할 수 있어.

sequenceDiagram
    participant C as 호출자
    participant A as 작업 A (3초)
    participant B as 작업 B (2초)

    rect rgb(255, 230, 230)
        Note over C,B: 동기 처리 - 총 5초
        C->>A: 작업 A 시작
        Note over C: 3초 대기...
        A-->>C: 작업 A 완료
        C->>B: 작업 B 시작
        Note over C: 2초 대기...
        B-->>C: 작업 B 완료
    end

    rect rgb(230, 255, 230)
        Note over C,B: 비동기 처리 - 총 3초
        C->>A: 작업 A 시작 (별도 스레드)
        C->>B: 작업 B 시작 (별도 스레드)
        Note over C: 다른 일 가능
        B-->>C: 작업 B 완료 (2초)
        A-->>C: 작업 A 완료 (3초)
    end

NexClass에서 비동기가 필요한 이유¶

NexClass의 Webhook 처리를 보면:

작업 소요 시간 동기로 하면?
Webhook 수신 → JSON 파싱 → 서명 검증 ~50ms 문제없음
Zoom Reports API 호출 3~10초 Zoom Webhook 타임아웃
출결 데이터 가공 + DB 저장 ~1초 더 지연
LMS Push 통보 ~2초 더더 지연
합계 6~13초 Zoom이 재시도 시작

Zoom은 Webhook 응답을 수 초 안에 받아야 해. 6~13초나 기다리면 Zoom이 "전달 실패"로 판단하고 재시도해. 그래서 무거운 처리는 비동기로 백그라운드에서 돌려야 해.

2. Spring @Async 어노테이션¶

어떻게 동작해?¶

// 이 메서드를 호출하면 별도 스레드에서 실행된다
@Async
public void processAttendance(String meetingId, String uuid) {
    // ... 무거운 작업 (API 호출, DB 저장 등)
}

@Async를 붙이면 Spring이 이 메서드를 호출한 스레드와 다른 스레드에서 실행해. 호출한 쪽은 기다리지 않고 즉시 리턴돼.

sequenceDiagram
    participant T1 as 메인 스레드<br/>(HTTP 요청 처리)
    participant SP as Spring AOP<br/>(프록시)
    participant T2 as 비동기 스레드<br/>(스레드 풀)

    T1->>SP: processAttendance() 호출
    SP->>T2: 새 스레드에서 실행
    SP-->>T1: 즉시 리턴 (void)
    Note over T1: 200 OK 응답 가능

    Note over T2: Reports API 호출 중...
    Note over T2: DB 저장 중...
    Note over T2: LMS Push 중...
    T2->>T2: 완료

@EnableAsync -- 비동기 활성화¶

@Async를 쓰려면 반드시 어딘가에 @EnableAsync가 있어야 해:

// NexClassApplication.java
@EnableAsync       // ← 이게 없으면 @Async가 무시돼!
@EnableScheduling
@SpringBootApplication
public class NexClassApplication {
    public static void main(String[] args) {
        SpringApplication.run(NexClassApplication.class, args);
    }
}

@EnableAsync 빼먹으면?

@Async 어노테이션이 붙어있어도 @EnableAsync가 없으면 그냥 동기로 실행돼. 에러도 안 나고, 경고도 안 나고, 조용히 동기로 동작해. 가장 발견하기 어려운 버그 중 하나야.

3. NexClass에서 @Async를 쓰는 곳¶

NexClass에서 @Async를 쓰는 곳은 3군데야. 각각 이유가 다르다.

3.1 processAttendance() -- 출결 처리¶

// AttendanceService.java
@Async
public void processAttendance(String meetingId, String uuid) {
    log.info("[ATTEND] ===== 출결 처리 시작 =====");

    // (1) 트랜잭션 내에서 출결 저장
    String[] result = saveAttendance(meetingId, uuid);
    if (result == null) return;

    // (2) 트랜잭션 커밋 후 LMS Push
    lmsSyncService.pushAttendanceCompleted(result[0], result[1]);
    log.info("[ATTEND] ===== 출결 처리 완료 =====");
}

왜 @Async?

  • Zoom Reports API 호출이 3~10초 걸려
  • 참여자 데이터 가공 + DB 저장에 1초 이상
  • LMS Push에 2초 이상
  • 이걸 다 기다리면 Zoom Webhook 타임아웃 → 재시도 → 중복 처리

호출 지점: WebhookController.handleMeetingEnded()

// WebhookController.java
private void handleMeetingEnded(Map<String, Object> object) {
    String meetingId = String.valueOf(object.get("id"));
    String uuid = (String) object.get("uuid");
    // @Async 메서드 호출 → 즉시 리턴
    attendanceService.processAttendance(meetingId, uuid);
    // 여기 도달하면 processAttendance는 별도 스레드에서 실행 중
}

3.2 processRecording() -- 녹화 처리¶

// AttendanceService.java
@Async
@Transactional
public void processRecording(String meetingId, Map<String, Object> object) {
    log.info("[RECORD] ===== 녹화 처리 시작 =====");

    // Lesson 조회 → Recordings API 호출 → DB 저장
    Optional<Lesson> lessonOpt = lessonRepository.findByZoomMeetingId(meetingId);
    // ... 녹화 파일 처리 로직 ...
}

왜 @Async?

  • Zoom Recordings API 호출로 share_url을 가져오는 데 시간 소요
  • MP4 파일 필터링, 중복 체크, Duration 계산 등 가공 작업 있음
  • 출결과 같은 이유: Webhook 응답은 즉시 보내야 하니까

3.3 registerCourseStudentsAsync() -- 수강생 Zoom 등록¶

// RegistrantService.java
@Async
public void registerCourseStudentsAsync(String lessonCd, String courseCd, String zoomMeetingId) {
    List<CourseUser> students = courseUserRepository.findByCourseCd(courseCd);
    log.info("[Async] Registrant 등록 시작: lessonCd={}, 수강생={}명", lessonCd, students.size());

    int successCount = 0;
    int failCount = 0;

    for (CourseUser student : students) {
        try {
            // 각 학생마다 Zoom Registrant API 호출 (학생 수만큼 API 호출)
            String email = student.getUserCd() + "@" + dummyEmailDomain;
            Map<String, Object> result = zoomService.registerParticipant(
                    zoomMeetingId, email, userName, userNo);
            // joinUrl 저장...
            successCount++;
        } catch (Exception e) {
            failCount++;
        }
    }
    log.info("[Async] Registrant 등록 완료: 성공={}, 실패={}", successCount, failCount);
}

왜 @Async?

이 메서드는 학생 수만큼 Zoom API를 호출해. 학생이 30명이면 API 30번 호출.

시나리오 동기 비동기
학생 30명, API 1회 0.5초 15초 대기 후 강의 생성 응답 강의 생성 응답 즉시 (0.1초)
학생 100명, API 1회 0.5초 50초 대기 (타임아웃 확정) 강의 생성 응답 즉시

이 문제의 실제 해결 히스토리

NexClass 초기에는 registerCourseStudentsAsync가 동기였어. 강의 생성 API가 15초 이상 걸리면서 LMS에서 타임아웃이 났지. 이걸 @Async로 분리해서 해결한 거야. 실제로 겪은 문제야.

4. 스레드 풀 설정¶

기본 스레드 풀¶

Spring @Async의 기본 스레드 풀은 SimpleAsyncTaskExecutor야. 이 녀석은 요청마다 새 스레드를 생성해.

기본 스레드 풀의 문제

SimpleAsyncTaskExecutor는 스레드 풀이 아니야. 매 호출마다 새 스레드를 만들어. 스레드 재사용이 없어.

Webhook이 동시에 100개 들어오면? 스레드 100개 생성. 1,000개면? 스레드 1,000개. 메모리 부족으로 서버가 죽을 수 있어.

커스텀 스레드 풀 설정 (권장)¶

프로덕션에서는 반드시 스레드 풀을 직접 설정해야 해:

// AsyncConfig.java (NexClass에는 아직 미구현, 프로덕션 시 추가 권장)
@Configuration
@EnableAsync
public class AsyncConfig implements AsyncConfigurer {

    @Override
    public Executor getAsyncExecutor() {
        ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
        executor.setCorePoolSize(5);       // 기본 스레드 수
        executor.setMaxPoolSize(10);       // 최대 스레드 수
        executor.setQueueCapacity(25);     // 대기 큐 크기
        executor.setThreadNamePrefix("nexclass-async-");  // 스레드 이름 접두사
        executor.initialize();
        return executor;
    }
}
설정 의미 NexClass 적정값 이유
corePoolSize 항상 유지되는 스레드 수 5 동시 Webhook 처리 5개면 충분
maxPoolSize 큐 가득 찰 때 최대 스레드 수 10 피크 시간 대비
queueCapacity 대기 작업 큐 크기 25 일시적 폭주 대비

스레드 풀 동작 원리:

flowchart TD
    A["@Async 메서드 호출"] --> B{"코어 스레드<br/>여유 있어?"}
    B -->|있음| C["코어 스레드에서 실행"]
    B -->|없음| D{"큐에 자리 있어?"}
    D -->|있음| E["큐에 대기"]
    D -->|없음| F{"최대 스레드<br/>도달?"}
    F -->|아니오| G["새 스레드 생성해서 실행"]
    F -->|예| H["RejectedExecutionException!"]

5. @Async의 함정¶

함정 1: 같은 클래스 내 호출 -- 가장 흔한 실수¶

이건 진짜 많이 틀려

@Service
public class MyService {

    // ❌ 같은 클래스의 @Async 메서드를 직접 호출 → 비동기 동작 안 함!
    public void doWork() {
        this.asyncMethod();  // this = 실제 객체, 프록시 아님
    }

    @Async
    public void asyncMethod() {
        // 이 메서드는 동기로 실행된다!
    }
}

왜 안 돼?

Spring @Async는 AOP 프록시로 동작해. 외부에서 호출할 때만 프록시를 거쳐가.

sequenceDiagram
    participant 외부 as 외부 (Controller)
    participant P as Spring 프록시
    participant S as 실제 Service 객체

    rect rgb(230, 255, 230)
        Note over 외부,S: ✅ 외부에서 호출 - @Async 동작
        외부->>P: asyncMethod()
        P->>P: 새 스레드 생성
        P->>S: asyncMethod() (새 스레드)
        P-->>외부: 즉시 리턴
    end

    rect rgb(255, 230, 230)
        Note over 외부,S: ❌ 같은 클래스에서 호출 - @Async 무시
        S->>S: this.asyncMethod()
        Note over S: 프록시를 안 거쳐서<br/>동기로 실행됨
    end

NexClass는 어떻게 했는지 봐:

// WebhookController → AttendanceService 호출 (다른 클래스 = ✅)
attendanceService.processAttendance(meetingId, uuid);

// LessonService → RegistrantService 호출 (다른 클래스 = ✅)
registrantService.registerCourseStudentsAsync(lessonCd, courseCd, zoomMeetingId);

전부 다른 클래스에서 호출하고 있어. 같은 클래스 내 호출이 아니라서 @Async가 정상 동작해.

해결 방법

같은 클래스 내에서 @Async 메서드를 호출해야 한다면:

  1. 별도 클래스로 분리 (NexClass가 쓴 방법 - 가장 깔끔)
  2. ApplicationContext에서 자기 자신의 프록시를 주입받아 호출
  3. @Lazy + @Autowired 자기 참조

1번이 정답이야. 나머지는 우회책이고 코드가 더러워져.

함정 2: 반환값 처리¶

// ❌ @Async + void가 아닌 반환값 → 호출자가 결과를 받을 수 없음
@Async
public String processData() {
    return "result";  // 호출자는 이 값을 받을 수 없어!
}

// 호출 측
String result = service.processData();  // result = null (!!!)

@Async 메서드가 즉시 리턴하니까, 반환값을 받을 수 없어. 결과가 필요하면 CompletableFuture를 써야 해:

// ✅ CompletableFuture로 비동기 결과 받기
@Async
public CompletableFuture<String> processData() {
    String result = heavyWork();
    return CompletableFuture.completedFuture(result);
}

// 호출 측
CompletableFuture<String> future = service.processData();
String result = future.get();  // 결과 나올 때까지 대기 (blocking)

NexClass는 왜 void만 쓸까?

NexClass의 @Async 메서드는 전부 void야.

  • processAttendance(): 결과를 기다릴 필요 없음 (Webhook은 즉시 200 OK)
  • processRecording(): 동일
  • registerCourseStudentsAsync(): 강의 생성 응답에 등록 결과 불필요

비동기의 목적이 "결과를 나중에 받자"가 아니라 "응답을 빨리 보내자" 이니까.

함정 3: 예외 처리¶

@Async
public void processAttendance(String meetingId, String uuid) {
    // 이 메서드에서 예외가 발생하면?
    // 호출자(WebhookController)는 이미 200 OK를 보냈어.
    // 예외가 터져도 아무도 모른다!
}

@Async 메서드에서 예외가 발생하면 호출자에게 전파되지 않아. 별도 스레드에서 실행 중이니까.

NexClass는 이걸 try-catch + 로그로 처리해:

// WebhookController에서
try {
    attendanceService.processAttendance(meetingId, uuid);
} catch (Exception e) {
    // 이 catch는 @Async 메서드 내부 예외를 잡지 못해!
    // processAttendance 호출 자체의 예외만 잡음 (스레드 생성 실패 등)
    log.error("출결 처리 실패: {}", e.getMessage());
}

// AttendanceService 내부에서
@Async
public void processAttendance(String meetingId, String uuid) {
    // 내부에서 자체적으로 예외 처리
    try {
        reportData = zoomService.getReportParticipants(meetingId);
    } catch (Exception e) {
        log.error("[ATTEND] Reports API 호출 실패: meetingId={}, error={}",
                meetingId, e.getMessage(), e);
        return;  // 실패하면 여기서 종료
    }
}

비동기 예외 처리 전략

전략 설명 NexClass 적용
내부 try-catch 메서드 안에서 직접 처리 O (현재 방식)
AsyncUncaughtExceptionHandler 글로벌 예외 핸들러 X (프로덕션 시 추가 권장)
CompletableFuture.exceptionally() 비동기 결과의 예외 처리 X (void라 불필요)

함정 4: @Transactional과 @Async 조합¶

// AttendanceService.java
@Async
@Transactional  // 이 조합이 동작할까?
public void processRecording(String meetingId, Map<String, Object> object) {
    // ...
}

@Async와 @Transactional을 같이 쓸 수 있어. 하지만 주의점이 있어:

  • @Async 메서드는 별도 스레드에서 실행돼
  • @Transactional은 해당 스레드의 트랜잭션을 관리해
  • 호출자의 트랜잭션과는 완전히 별개야
flowchart LR
    subgraph "메인 스레드"
        A["WebhookController"] --> B["return 200 OK"]
        A1["트랜잭션 없음"]
    end

    subgraph "비동기 스레드"
        C["processRecording()"] --> D["DB 저장"]
        C1["별도 트랜잭션 시작"]
        D --> E["트랜잭션 커밋"]
    end

    A -->|"@Async 호출"| C

NexClass의 processAttendance()를 보면 더 재미있는 패턴이 있어:

@Async
public void processAttendance(String meetingId, String uuid) {
    // 1단계: 트랜잭션 내에서 출결 저장
    String[] result = saveAttendance(meetingId, uuid);  // @Transactional 메서드

    // 2단계: 트랜잭션 커밋 후 LMS Push
    lmsSyncService.pushAttendanceCompleted(result[0], result[1]);
}

@Transactional
public String[] saveAttendance(String meetingId, String uuid) {
    // DB 저장 로직...
}

saveAttendance()가 끝나면 트랜잭션이 커밋돼. 그 후에 pushAttendanceCompleted()를 호출해. 왜 이렇게 분리했을까?

LMS가 Push를 받으면 NexClass DB에서 출결 데이터를 조회해. saveAttendance()의 트랜잭션이 아직 커밋 안 된 상태에서 Push를 보내면, LMS가 조회할 때 데이터가 아직 안 보여. 그래서 커밋 후에 Push를 보내야 해.

7. 주의사항¶

프로덕션에서 반드시 체크할 것

1. 스레드 풀 설정을 반드시 해라

기본 SimpleAsyncTaskExecutor는 스레드를 무한정 생성해. 동시 Webhook이 많이 들어오면 OOM(Out of Memory)으로 서버 다운.

2. 비동기 메서드 실패 시 데이터 유실 가능

@Async로 호출한 메서드가 실패해도 Webhook은 이미 200 OK를 보냈어. Zoom은 성공으로 판단하고 재시도 안 해. 그래서 비동기 메서드 내부에서 확실한 예외 처리 + 로그가 필수야.

3. 서버 재시작 시 진행 중인 작업 유실

메모리에 있는 스레드 풀의 작업은 서버 재시작하면 사라져. 중요한 작업이면 메시지 큐(RabbitMQ, Kafka 등)를 써야 하는데, NexClass 규모에서는 오버엔지니어링이야.

4. @Async 메서드에서 Request 스코프 객체 접근 불가

@Async 메서드는 HTTP 요청 스레드와 다른 스레드에서 실행돼. HttpServletRequest나 @RequestScope 빈에 접근하면 에러 나. 필요한 데이터는 파라미터로 넘겨야 해.

@EnableAsync @Async 활성화 (없으면 동기로 동작)
같은 클래스 호출 @Async 무시됨 (AOP 프록시 우회)
스레드 풀 기본은 위험, 반드시 커스텀 설정
예외 처리 호출자에게 전파 안 됨, 내부에서 처리 필수

9. 확인문제¶

문제 1: 왜 비동기?¶

NexClass에서 processAttendance()가 동기로 실행된다면 구체적으로 어떤 시나리오에서 문제가 발생하나?
  1. Zoom이 meeting.ended Webhook을 보낸다.
  2. processAttendance()가 동기로 실행된다.
  3. Zoom Reports API 호출에 5초, DB 저장에 1초, LMS Push에 2초 → 총 8초.
  4. Zoom은 Webhook 응답을 수 초 안에 기대하는데 8초 후에 200 OK를 받는다.
  5. Zoom은 이를 타임아웃으로 판단하고 같은 이벤트를 재시도한다.
  6. 재시도된 이벤트가 또 8초 걸리고, 그 사이 Zoom이 또 재시도.
  7. 최악의 경우 출결 데이터가 중복 저장되거나, Zoom이 Webhook URL을 비활성화.

문제 2: 같은 클래스 호출¶

다음 코드에서 asyncMethod()는 비동기로 실행될까? 이유와 함께 설명해봐. 
@Service
public class AttendanceService {
    @Async
    public void asyncMethod() { ... }

    public void caller() {
        this.asyncMethod();  // 이게 비동기로 동작할까?
    }
}

비동기로 동작하지 않는다.

this.asyncMethod()는 실제 객체의 메서드를 직접 호출하는 거야. Spring @Async는 AOP 프록시를 통해 동작하는데, 같은 클래스 내에서 this로 호출하면 프록시를 거치지 않고 실제 객체의 메서드를 바로 호출해.

프록시가 개입하지 않으니까 @Async 어노테이션이 무시되고, 일반 동기 메서드처럼 실행된다.

해결: asyncMethod()를 별도 서비스 클래스로 분리하고, 그 서비스를 주입받아서 호출해야 한다. NexClass의 RegistrantService 분리가 바로 이 패턴이야.

문제 3: @EnableAsync¶

@EnableAsync를 빼먹으면 어떻게 되나? 에러가 나? 경고라도 나?

에러도 안 나고, 경고도 안 나. 그냥 조용히 동기로 실행돼. 이게 이 문제의 무서운 점이야. 테스트 환경에서는 정상처럼 보이거든 (동기든 비동기든 결과는 같으니까).

프로덕션에서 Webhook 타임아웃이 발생해야 비로소 "어? 왜 비동기가 안 되지?" 하고 알게 돼. 로그에 스레드 이름을 보면 알 수 있어:

  • @Async 동작 시: [nexclass-async-1], [task-1] 등 별도 스레드 이름
  • @Async 미동작 시: [http-nio-8082-exec-1] 등 HTTP 요청 처리 스레드 이름

문제 4: 비동기 예외¶

@Async 메서드에서 RuntimeException이 발생하면, WebhookController의 try-catch에서 잡을 수 있어?

잡을 수 없다.

@Async 메서드는 별도 스레드에서 실행돼. WebhookController의 try-catch는 HTTP 요청 처리 스레드에서 동작해. 서로 다른 스레드의 예외는 전파되지 않아.

try {
    attendanceService.processAttendance(meetingId, uuid);
    // 이 시점에서 processAttendance는 아직 시작도 안 했을 수 있어
    // (스레드 풀에 작업이 등록된 것뿐)
} catch (Exception e) {
    // processAttendance 내부의 예외는 여기서 안 잡혀
    // 여기서 잡히는 건 스레드 풀에 작업 등록 자체가 실패한 경우뿐
}

그래서 @Async 메서드 내부에서 try-catch를 해야 하고, NexClass의 processAttendance()가 바로 그렇게 하고 있어.

문제 5: 트레이드오프¶

NexClass에서 @Async 대신 메시지 큐(RabbitMQ, Kafka)를 쓸 수도 있었어. @Async를 선택한 이유와 트레이드오프를 분석해봐.

@Async 선택 이유:

  1. 구현 단순: 어노테이션 하나면 됨. 인프라 추가 불필요.
  2. 운영 비용 낮음: RabbitMQ/Kafka 서버를 별도로 운영할 필요 없음.
  3. NexClass 규모에 적합: 동시 Webhook이 수십 개 수준. 수천 개가 아님.

@Async의 단점 (메시지 큐 대비):

  1. 서버 재시작 시 작업 유실: 메모리 큐라서 재시작하면 진행 중인 작업 사라짐. 메시지 큐는 디스크에 저장.
  2. 재시도 불가: 실패한 작업을 자동으로 재시도할 수 없음. 메시지 큐는 DLQ(Dead Letter Queue) 지원.
  3. 모니터링 어려움: 대기 중인 작업, 실패한 작업을 추적하기 어려움.
  4. 스케일아웃 불가: 서버가 여러 대면 작업 분산이 안 됨.

결론: NexClass는 학교 내부 시스템이고, 동시 사용량이 적고, 서버 1대로 운영해. 이 규모에서 메시지 큐는 오버엔지니어링이야. YAGNI 원칙. 하지만 사용량이 10배 이상 늘어나면 메시지 큐를 고려해야 해.
