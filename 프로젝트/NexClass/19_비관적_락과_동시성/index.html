19. 비관적 락과 동시성¶

난이도: Delta | 선수 지식: 18_호스트_풀_동적_할당, @Transactional, SQL 기초

이 챕터에서 배우는 것¶

학습 목표

  • 동시성 문제(Race Condition)가 왜 생기는지 본질적으로 이해한다
  • 낙관적 락 vs 비관적 락의 트레이드오프를 분석할 수 있다
  • NexClass가 비관적 락을 선택한 이유를 설명할 수 있다
  • SELECT FOR UPDATE가 DB에서 어떻게 동작하는지 안다
  • 데드락의 발생 원리와 방지 전략을 이해한다

1. 동시성 문제가 뭐야?¶

Race Condition -- 경쟁 상태¶

핵심 비유: 콘서트 좌석 예매

BTS 콘서트 A열 7번 좌석이 1개 남았어. 김개발과 홍길동이 동시에 "예매" 버튼을 누른다.

  1. 김개발: "A열 7번 비어있나?" → DB 조회 → "비어있네!" → 예매 진행
  2. 홍길동: "A열 7번 비어있나?" → DB 조회 → "비어있네!" → 예매 진행

둘 다 "비어있다"는 결과를 봤어. 둘 다 예매에 성공해. 좌석 1개에 2명이 예매됨. 이게 Race Condition이야.

NexClass에서의 Race Condition¶

호스트 풀에서도 똑같은 문제가 발생할 수 있어:

sequenceDiagram
    participant A as 강의 생성 요청 A
    participant DB as DB (HostAssignment)
    participant B as 강의 생성 요청 B

    Note over A,B: kren5@만 가용한 상태

    A->>DB: SELECT 가용 호스트 → kren5@ 발견
    B->>DB: SELECT 가용 호스트 → kren5@ 발견

    Note over A,B: 둘 다 kren5@가 비어있다고 판단!

    A->>DB: INSERT 배정 (kren5@ → 강의A)
    B->>DB: INSERT 배정 (kren5@ → 강의B)

    Note over DB: kren5@가 2개 강의에<br/>동시 배정됨!

이 문제의 근본 원인은 "조회"와 "삽입" 사이에 시간 간격이 있기 때문이야. 그 사이에 다른 트랜잭션이 끼어들 수 있어.

왜 @Transactional만으로는 안 돼?¶

@Transactional
public String assignHost(LocalDateTime startTime, LocalDateTime endTime, String lessonCd) {
    // 이 메서드가 @Transactional이지만...
    List<String> assignedEmails = hostAssignmentRepository.findAssignedHostEmails(startTime, endTime);
    // ↑ 여기서 조회하고

    // ... 필터링 ...

    hostAssignmentRepository.save(assignment);
    // ↑ 여기서 저장할 때까지 시간 간격 존재
}

@Transactional은 원자성(Atomicity)과 일관성(Consistency)을 보장하지만, 격리성(Isolation) 레벨에 따라 다른 트랜잭션이 동시에 같은 데이터를 읽을 수 있어.

기본 격리 수준(READ COMMITTED)에서는:

  • 트랜잭션 A가 "kren5@ 가용" 읽음 (아직 커밋 안 한 상태)
  • 트랜잭션 B가 "kren5@ 가용" 읽음 (A가 INSERT 안 했으니까 보여)
  • 둘 다 kren5@에 배정 → 중복!

2. 해결 방법 -- 락(Lock)¶

락이 뭐야?¶

비유: 화장실 잠금장치

화장실(공유 자원)에 들어가면 문을 잠가. (락 획득) 다른 사람은 문이 잠겨있으면 기다려야 해. (블로킹) 볼일 끝나고 나오면 잠금을 풀어. (락 해제) 그 다음 사람이 들어감.

DB에서의 락도 같은 원리야. 특정 데이터에 락을 걸면 다른 트랜잭션은 그 데이터에 접근할 수 없어 (또는 제한적으로만 가능).

두 가지 접근법¶

구분 낙관적 락 (Optimistic) 비관적 락 (Pessimistic)
철학 "충돌 안 나겠지?" "충돌 날 거야!"
방식 충돌 감지 (사후 처리) 충돌 예방 (사전 차단)
구현 버전 번호 체크 DB 행 잠금
충돌 시 예외 발생 → 재시도 대기 → 순서대로 처리

3. 낙관적 락 (Optimistic Locking)¶

동작 원리¶

"충돌이 잘 안 나겠지?"라는 낙관적인 가정 하에, 일단 작업하고 커밋할 때 충돌 여부를 확인해.

sequenceDiagram
    participant A as 트랜잭션 A
    participant DB as DB
    participant B as 트랜잭션 B

    A->>DB: SELECT kren5 (version=1)
    B->>DB: SELECT kren5 (version=1)

    A->>DB: UPDATE SET version=2<br/>WHERE version=1
    Note over DB: version 1→2 성공

    B->>DB: UPDATE SET version=2<br/>WHERE version=1
    Note over DB: version이 이미 2!<br/>WHERE 조건 불일치
    DB-->>B: OptimisticLockException!
    Note over B: 재시도 필요

JPA에서의 구현¶

@Entity
public class HostPool {
    @Id
    private String hostEmail;

    @Version  // ← 이 어노테이션이 핵심
    private Long version;

    // ...
}

@Version 필드가 있으면 JPA가 자동으로:

  1. UPDATE 시 WHERE version = ? 조건 추가
  2. 조건에 맞는 행이 없으면 OptimisticLockException 발생

낙관적 락의 한계¶

// 낙관적 락으로 호스트 배정한다면?
@Transactional
public String assignHost(...) {
    List<HostPool> available = findAvailable();  // 가용 호스트 조회
    HostPool host = available.get(0);

    // HostPool에 @Version이 있다고 가정
    // 근데... HostPool을 UPDATE하는 게 아니라
    // HostAssignment를 INSERT하는 거잖아!
    // @Version 체크가 불가능해.
}

낙관적 락이 NexClass 호스트 풀에 안 맞는 이유

낙관적 락은 같은 행을 UPDATE할 때 동작해. 호스트 배정은 HostPool을 UPDATE하는 게 아니라 HostAssignment에 INSERT하는 거야. INSERT에는 버전 충돌 체크가 적용 안 돼.

무리하게 적용하려면:

  • HostPool에 currentAssignmentCount 같은 컬럼을 추가하고
  • 배정 시 이 값을 UPDATE하면서 @Version 체크

하지만 이건 불필요하게 복잡해지고, HostPool의 책임이 늘어나. SRP 위반.

4. 비관적 락 (Pessimistic Locking)¶

동작 원리¶

"충돌 날 거야!"라는 비관적인 가정 하에, 데이터를 읽을 때부터 잠가버려.

sequenceDiagram
    participant A as 트랜잭션 A
    participant DB as DB
    participant B as 트랜잭션 B

    A->>DB: SELECT ... FOR UPDATE<br/>(락 획득)
    Note over DB: kren5 행 잠김!

    B->>DB: SELECT ... FOR UPDATE<br/>(락 시도)
    Note over B: 대기 중...<br/>(A가 끝날 때까지)

    A->>DB: INSERT 배정 (kren5 → 강의A)
    A->>DB: COMMIT (락 해제)

    Note over B: 락 획득!
    B->>DB: 다시 조회 → kren5 이미 배정됨
    B->>DB: kren6 선택 → INSERT 배정
    B->>DB: COMMIT (락 해제)

핵심: SELECT 시점에 행을 잠그니까 다른 트랜잭션이 같은 데이터를 동시에 읽을 수 없어.

JPA에서의 구현¶

@Repository
public interface HostPoolRepository extends JpaRepository<HostPool, String> {

    // 비관적 락: SELECT ... FOR UPDATE
    @Lock(LockModeType.PESSIMISTIC_WRITE)
    @Query("SELECT h FROM HostPool h WHERE h.hostStatus = :status")
    List<HostPool> findByHostStatusWithLock(@Param("status") String status);
}

@Lock(LockModeType.PESSIMISTIC_WRITE)를 붙이면 JPA가 실행하는 SQL이 달라져:

-- @Lock 없이 (일반 조회)
SELECT * FROM TB_NEXCLASS_HOST_POOL WHERE HOST_STATUS = 'ACTIVE';

-- @Lock(PESSIMISTIC_WRITE) 추가 시
SELECT * FROM TB_NEXCLASS_HOST_POOL WHERE HOST_STATUS = 'ACTIVE'
FOR UPDATE;  -- ← 이 부분이 추가됨

SELECT FOR UPDATE가 DB에서 하는 일¶

flowchart TD
    A["SELECT ... FOR UPDATE 실행"] --> B["해당 행들에 배타적 락(X Lock) 설정"]
    B --> C{"다른 트랜잭션이<br/>같은 행 접근?"}
    C -->|"일반 SELECT"| D["읽기 가능<br/>(락 모드에 따라 다름)"]
    C -->|"SELECT FOR UPDATE"| E["대기<br/>(락 해제까지 블로킹)"]
    C -->|"UPDATE / DELETE"| F["대기<br/>(락 해제까지 블로킹)"]

배타적 락(Exclusive Lock, X Lock):

  • 같은 행에 대해 다른 트랜잭션의 쓰기를 차단
  • MariaDB(InnoDB)에서는 다른 트랜잭션의 SELECT FOR UPDATE도 차단
  • 일반 SELECT는 읽기 가능 (MVCC에 의해 스냅샷 읽기)
  • 트랜잭션 COMMIT 또는 ROLLBACK 시 자동 해제

락 모드 종류¶

JPA 락 모드 SQL 설명
PESSIMISTIC_READ SELECT ... FOR SHARE 공유 락. 다른 트랜잭션도 읽기 가능, 쓰기 불가
PESSIMISTIC_WRITE SELECT ... FOR UPDATE 배타 락. 다른 트랜잭션 읽기/쓰기 모두 불가
PESSIMISTIC_FORCE_INCREMENT FOR UPDATE + 버전 증가 배타 락 + @Version 자동 증가

NexClass는 PESSIMISTIC_WRITE를 써. 호스트 배정은 반드시 한 트랜잭션만 처리해야 하니까.

5. NexClass에서 비관적 락을 선택한 이유¶

트레이드오프 분석¶

이건 진짜 중요한 설계 결정이야. 왜 비관적 락인지 근거를 대야 해.

기준 낙관적 락 비관적 락 NexClass에서는?
충돌 빈도 충돌이 드물 때 유리 충돌이 잦을 때 유리 같은 시간대 강의 생성 = 충돌 가능성 높음
실패 처리 예외 발생 → 재시도 로직 필요 대기 → 자동 순서 처리 재시도 로직 복잡도 회피
데이터 구조 UPDATE 대상 필요 SELECT 대상이면 충분 INSERT 기반이라 낙관적 락 부적합
풀 크기 대규모에서 유리 소규모에서 유리 20개 호스트 = 소규모
성능 락 대기 없음 락 대기 있음 대기 시간 미미 (수 ms)
구현 복잡도 재시도 로직 필요 쿼리에 FOR UPDATE 추가 단순한 게 낫다

결정적 이유 3가지¶

1. INSERT 기반 작업

호스트 배정은 HostAssignment에 INSERT하는 거야. 낙관적 락의 @Version은 UPDATE에서 동작해. INSERT에는 버전 체크가 안 돼. 비관적 락은 SELECT 시점에 잠그니까 INSERT든 UPDATE든 상관없어.

2. 호스트 풀이 작다 (20개)

풀이 작으면 충돌 가능성이 높아. 20개 중 마지막 1개만 남았는데 동시에 2개 요청이 오면? 낙관적 락이면 하나가 실패하고 재시도해야 해. 근데 재시도해도 가용 호스트가 없으면 또 실패. 비관적 락이면 하나가 기다렸다가 정상 처리되거나, 없으면 깔끔하게 에러.

3. 재시도 로직의 복잡도

낙관적 락은 OptimisticLockException 발생 시 재시도 로직을 구현해야 해:

// ❌ 낙관적 락 + 재시도 (복잡해짐)
@Retryable(value = OptimisticLockException.class, maxAttempts = 3)
public String assignHost(...) {
    // 재시도 시 가용 호스트가 바뀌어 있을 수 있어
    // 재시도 횟수 제한, 백오프 전략 등 고려 필요
}

// ✅ 비관적 락 (단순)
public String assignHost(...) {
    // 락 획득까지 대기 → 순서대로 처리
    // 재시도 로직 불필요
}

6. NexClass의 현재 구현 상태¶

현재 코드 분석

현재 NexClass의 HostPoolService.assignHost()를 보면:

@Transactional
public String assignHost(LocalDateTime startTime, LocalDateTime endTime, String lessonCd) {
    List<HostPool> activeHosts = hostPoolRepository.findByHostStatus("ACTIVE");
    List<String> assignedEmails = hostAssignmentRepository.findAssignedHostEmails(startTime, endTime);
    // ...
}

findByHostStatus()에 @Lock 어노테이션이 없어. 비관적 락이 적용되지 않은 상태야.

CLAUDE.md에는 "비관적 락"이라고 적혀있지만, 실제 코드에는 아직 락이 구현되지 않았어. 이건 프로덕션 배포 전에 반드시 추가해야 할 사항이야.

비관적 락을 적용한다면 이런 코드가 될 거야:

// HostPoolRepository.java - 비관적 락 적용 버전
@Repository
public interface HostPoolRepository extends JpaRepository<HostPool, String> {

    // 기존: 락 없는 조회
    List<HostPool> findByHostStatus(String hostStatus);

    // 추가: 비관적 락이 걸린 조회
    @Lock(LockModeType.PESSIMISTIC_WRITE)
    @Query("SELECT h FROM HostPool h WHERE h.hostStatus = :status")
    List<HostPool> findByHostStatusForUpdate(@Param("status") String status);
}

// HostPoolService.java - 비관적 락 적용 버전
@Transactional
public String assignHost(LocalDateTime startTime, LocalDateTime endTime, String lessonCd) {
    // 비관적 락으로 호스트 조회 (다른 트랜잭션 대기)
    List<HostPool> activeHosts = hostPoolRepository.findByHostStatusForUpdate("ACTIVE");

    // 이 시점에서 다른 assignHost 호출은 블로킹됨
    List<String> assignedEmails = hostAssignmentRepository
            .findAssignedHostEmails(startTime, endTime);

    // ... 나머지는 동일 ...
}

비관적 락 적용 전후 비교¶

sequenceDiagram
    participant A as 요청 A
    participant DB as DB
    participant B as 요청 B

    A->>DB: SELECT ACTIVE 호스트 (락 없음)
    B->>DB: SELECT ACTIVE 호스트 (락 없음)
    Note over A,B: 둘 다 kren5@ 가용으로 보임
    A->>DB: INSERT kren5 → 강의A
    B->>DB: INSERT kren5 → 강의B
    Note over DB: kren5 중복 배정!
end
sequenceDiagram
    participant A as 요청 A
    participant DB as DB
    participant B as 요청 B

    A->>DB: SELECT ... FOR UPDATE (락 획득)
    B->>DB: SELECT ... FOR UPDATE (대기...)
    Note over B: A가 끝날 때까지 블로킹

    A->>DB: kren5@ 가용 확인
    A->>DB: INSERT kren5 → 강의A
    A->>DB: COMMIT (락 해제)

    Note over B: 락 획득!
    B->>DB: SELECT 다시 실행
    B->>DB: kren5@ 이미 배정됨 → kren6@ 선택
    B->>DB: INSERT kren6 → 강의B
    B->>DB: COMMIT
end

7. 데드락 (Deadlock)¶

데드락이 뭐야?¶

비유: 좁은 골목 교착

차 A가 북쪽에서 내려오고, 차 B가 남쪽에서 올라와. 골목이 좁아서 둘 다 비켜줘야 하는데, 둘 다 안 비켜. A: "B가 비켜야 내가 갈 수 있어" B: "A가 비켜야 내가 갈 수 있어" → 영원히 못 움직임. 이게 데드락.

DB에서의 데드락:

sequenceDiagram
    participant A as 트랜잭션 A
    participant DB as DB
    participant B as 트랜잭션 B

    A->>DB: LOCK 테이블1 (성공)
    B->>DB: LOCK 테이블2 (성공)

    A->>DB: LOCK 테이블2 (B가 잠금 → 대기)
    B->>DB: LOCK 테이블1 (A가 잠금 → 대기)

    Note over A,B: 둘 다 상대방을 기다림<br/>= 데드락!

NexClass에서 데드락 가능성¶

NexClass의 호스트 배정은 한 방향으로만 락을 잡아:

  1. HostPool SELECT FOR UPDATE (락 획득)
  2. HostAssignment INSERT (새 행 추가)
  3. COMMIT (락 해제)

여러 테이블에 교차로 락을 잡지 않아. 그래서 데드락 가능성이 매우 낮아.

데드락 방지 원칙

1. 락 순서를 항상 같게 유지: 모든 트랜잭션이 테이블A → 테이블B 순서로 락을 잡으면 데드락이 안 생겨. 2. 락 범위를 최소화: 필요한 행만 잠그고, 트랜잭션을 짧게 유지. 3. 타임아웃 설정: 일정 시간 이상 락을 못 잡으면 포기하고 롤백.

데드락이 발생하면?¶

MariaDB(InnoDB)는 데드락을 자동 감지해. 데드락이 감지되면 하나의 트랜잭션을 자동으로 롤백시켜 (비용이 적은 쪽을 선택).

// 데드락 발생 시 JPA에서는 이런 예외가 올 수 있어
try {
    hostPoolService.assignHost(startTime, endTime, lessonCd);
} catch (PessimisticLockingFailureException e) {
    // 데드락 또는 락 타임아웃
    log.error("호스트 배정 실패 (락 충돌): {}", e.getMessage());
    throw new RuntimeException("호스트 배정 중 충돌이 발생했습니다. 다시 시도해주세요.");
}

8. 비관적 락의 단점 -- 성능 vs 안전성¶

비관적 락은 안전성을 성능으로 사는 거야. 무조건 좋은 게 아니야.

성능 영향¶

flowchart LR
    subgraph "비관적 락 없음"
        A1["요청 A: 5ms"] --> B1["요청 B: 5ms"]
        Note1["병렬 처리, 총 5ms"]
    end

    subgraph "비관적 락 있음"
        A2["요청 A: 5ms (락 보유)"] --> C2["요청 B: 5ms 대기 + 5ms 처리"]
        Note2["직렬 처리, 총 10ms"]
    end
항목 락 없음 비관적 락
동시 처리량 높음 낮음 (직렬화)
응답 시간 일정 대기 시간 추가 가능
데이터 정합성 보장 안 됨 보장됨
구현 복잡도 단순 약간 복잡 (데드락 고려)

NexClass에서 비관적 락이 적절한 이유¶

규모가 핵심이야

항목 NexClass 대규모 서비스 (쿠팡, 인터파크)
동시 요청 수 수 건 수천~수만 건
풀 크기 20개 수만 좌석
락 보유 시간 ~10ms ~10ms
대기 시간 영향 무시 가능 치명적
적합한 락 비관적 낙관적 + 재시도

NexClass는 학교 시스템이야. 동시에 강의를 생성하는 경우가 많아야 2~3건이지. 락 대기 시간이 10ms 더 걸려도 전혀 문제 안 돼.

반면 인터파크 콘서트 예매처럼 동시에 수만 명이 접근하는 경우에 비관적 락을 쓰면? 수만 명이 줄 서서 기다려야 해. 이때는 낙관적 락 + 재시도가 더 적합해.

9. 대안 기술 비교¶

비관적 락 외에도 동시성을 해결하는 방법이 있어:

기술 설명 적합한 상황 NexClass에서는?
비관적 락 DB 행 잠금 소규모, 충돌 빈번 현재 선택
낙관적 락 버전 번호 체크 대규모, 충돌 드문 부적합 (INSERT 기반)
분산 락 (Redis) Redis SETNX 다중 서버 환경 오버엔지니어링
UNIQUE 제약조건 DB 레벨 중복 방지 단순 중복 방지 보조적으로 사용 가능
큐 기반 직렬화 요청을 큐에 넣고 순차 처리 비동기 허용 시 오버엔지니어링

UNIQUE 제약조건으로 보완

비관적 락과 별도로, DB 레벨에서 안전장치를 추가할 수 있어:

-- 같은 호스트가 같은 시간에 중복 배정되는 걸 DB에서 원천 차단
-- (시간 겹침은 UNIQUE로 못 잡지만, 정확히 같은 배정은 방지)
ALTER TABLE TB_NEXCLASS_HOST_ASSIGNMENT
ADD UNIQUE INDEX UK_HOST_LESSON (HOST_EMAIL, LESSON_CD);

이건 "같은 호스트가 같은 강의에 2번 배정"되는 걸 막아줘. 비관적 락이 어떤 이유로 실패해도 DB가 최종 방어선 역할을 해.

10. 정리¶

개념 한 줄 요약
Race Condition 조회와 삽입 사이에 다른 트랜잭션이 끼어드는 문제
낙관적 락 충돌 감지 후 재시도 (UPDATE + @Version)
비관적 락 충돌 사전 차단 (SELECT FOR UPDATE)
SELECT FOR UPDATE 조회한 행에 배타적 락 → 다른 트랜잭션 대기
데드락 두 트랜잭션이 서로의 락을 기다리는 교착 상태
NexClass 선택 비관적 락 -- 풀 크기 소규모, INSERT 기반, 재시도 불필요

11. 확인문제¶

문제 1: Race Condition 시나리오¶

월요일 9~10시에 가용 호스트가 kren5@ 1개만 남은 상태야. 두 교수가 동시에 같은 시간에 강의를 생성하면, 락이 없을 때 구체적으로 어떤 일이 벌어지나? 단계별로 설명해봐.

시간순 시나리오:

  1. 트랜잭션 A: findByHostStatus("ACTIVE") → [kren1~kren19] 조회
  2. 트랜잭션 B: findByHostStatus("ACTIVE") → [kren1~kren19] 조회 (같은 시점)
  3. 트랜잭션 A: findAssignedHostEmails(9시, 10시) → [kren1~kren4, kren6~kren19] 배정됨 → 가용: [kren5]
  4. 트랜잭션 B: findAssignedHostEmails(9시, 10시) → [kren1~kren4, kren6~kren19] 배정됨 → 가용: [kren5] (A의 INSERT가 아직 커밋 안 됨)
  5. 트랜잭션 A: INSERT INTO HostAssignment (kren5, 강의A, 9시, 10시)
  6. 트랜잭션 B: INSERT INTO HostAssignment (kren5, 강의B, 9시, 10시)
  7. 트랜잭션 A: COMMIT
  8. 트랜잭션 B: COMMIT

결과: kren5@가 강의A와 강의B에 동시 배정됨. Zoom에서 같은 호스트로 2개 미팅이 돌아가면 한쪽이 실패하거나 둘 다 불안정해짐.

문제 2: 낙관적 vs 비관적¶

인터파크에서 BTS 콘서트 좌석 예매 시스템을 만든다면 낙관적 락과 비관적 락 중 뭘 쓰겠어? 이유는?

낙관적 락이 적절하다.

이유:

  1. 동시 사용자가 수만 명: 비관적 락을 쓰면 수만 명이 순서대로 대기. 한 명당 50ms만 걸려도 10,000명이면 500초(8분). 사용자 이탈.

  2. 좌석이 수만 개로 충돌 확률이 낮음: 1만 좌석에 1만 명이 동시 접근해도 같은 좌석에 2명이 겹칠 확률은 상대적으로 낮아. 대부분 첫 시도에 성공.

  3. 재시도 비용이 낮음: 충돌 발생 시 "이 좌석은 이미 예매되었습니다. 다른 좌석을 선택하세요" 메시지로 유도. 재시도라기보다 다른 좌석 선택.

  4. 확장성: 낙관적 락은 DB 락을 안 잡으니까 읽기 성능에 영향이 없어. 캐시와 조합하기도 좋아.

NexClass(20개 호스트, 동시 2~3건)와는 상황이 완전히 다른 거야. 규모에 따라 전략이 달라진다. 이걸 아는 게 설계야.

문제 3: SELECT FOR UPDATE¶

SELECT FOR UPDATE가 걸린 상태에서 다른 트랜잭션이 같은 테이블에 일반 SELECT를 하면 어떻게 되나? MariaDB(InnoDB) 기준.

일반 SELECT는 정상 실행된다. 블로킹되지 않아.

MariaDB의 InnoDB는 MVCC(Multi-Version Concurrency Control) 을 사용해. 일반 SELECT는 트랜잭션 시작 시점의 스냅샷을 읽어. FOR UPDATE 락이 걸려있어도 스냅샷 읽기는 영향을 안 받아.

단, 해당 트랜잭션이 커밋하기 전의 데이터(기존 스냅샷)를 읽게 돼. 즉, 트랜잭션 A가 kren5를 배정하고 아직 커밋 안 했으면, 일반 SELECT로는 "kren5 미배정"으로 보여. (커밋 후에야 "배정됨"으로 보임)

이게 바로 READ COMMITTED 격리 수준의 동작이야.

문제 4: 데드락¶

NexClass의 호스트 배정에서 데드락이 발생하기 어려운 이유를 설명해봐.

데드락은 두 트랜잭션이 서로 다른 자원의 락을 교차로 기다릴 때 발생해.

NexClass의 assignHost()는:

  1. HostPool을 SELECT FOR UPDATE (락 획득)
  2. HostAssignment를 조회 (락 없음)
  3. HostAssignment에 INSERT (새 행이라 기존 행의 락과 무관)
  4. COMMIT (락 해제)

모든 트랜잭션이 같은 순서로 같은 테이블에만 락을 잡아. 트랜잭션 A가 HostPool 락 잡고 있으면, 트랜잭션 B는 HostPool 락을 기다려. B가 A의 락을 기다리지만, A는 B의 어떤 자원도 기다리지 않아. (A는 COMMIT만 하면 돼)

단방향 대기이기 때문에 데드락이 발생하지 않아. 데드락은 순환 대기(Circular Wait) 가 있어야 하는데, 여기엔 순환이 없어.

문제 5: 설계 결정¶

NexClass의 호스트 풀이 20개가 아니라 2,000개이고, 동시 강의 생성 요청이 초당 100건이라면 비관적 락 대신 뭘 쓰겠어? 구체적인 대안을 설계해봐.

이 규모에서는 비관적 락이 병목이 된다. 초당 100건이 직렬화되면 응답 시간이 급증해.

대안 1: 낙관적 락 + 재시도

  • HostAssignment에 INSERT 후 UNIQUE 제약조건으로 충돌 감지
  • 충돌 시 다른 호스트로 재시도 (최대 3회)
  • 대부분 1회에 성공 (2,000개 풀이니까 충돌 확률 낮음)

대안 2: Redis 분산 락

  • 호스트별로 Redis 키를 만들어서 SETNX (Set if Not Exists)
  • 락 획득 성공 → 배정, 실패 → 다른 호스트 시도
  • DB 부하를 Redis로 분산

대안 3: 큐 기반

  • 배정 요청을 메시지 큐(Kafka)에 넣고
  • 전용 컨슈머가 순차 처리
  • 응답은 비동기 (콜백 또는 폴링)

내 선택: 대안 1 (낙관적 락 + 재시도). 이유: 인프라 추가 없이 코드 레벨에서 해결. 2,000개 풀이면 충돌 확률이 낮아서 재시도 빈도도 낮음. Redis나 Kafka를 추가하면 운영 복잡도가 급증하는데, 이 정도 규모에서는 과도해.
