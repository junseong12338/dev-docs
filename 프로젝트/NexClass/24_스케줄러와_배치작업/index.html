24. 스케줄러와 배치작업¶

핵심 개념¶

이 챕터의 핵심

NexClass에는 사람이 실행하지 않아도 자동으로 돌아가는 작업이 있다. 매일 새벽 3시에 LMS 데이터를 동기화하고, 활성 강의의 Registrant를 갱신한다. 이걸 스케줄러(Scheduler)라고 한다. Spring의 @Scheduled 어노테이션으로 구현한다.

Spring @Scheduled가 뭐야?¶

@Scheduled는 Spring이 제공하는 작업 예약 기능이다. "이 메서드를 특정 시간/주기에 자동으로 실행해줘"라고 선언하면 Spring이 알아서 실행한다.

// 매일 새벽 3시에 실행
@Scheduled(cron = "0 0 3 * * *")
public void dailySync() {
    // 이 메서드가 새벽 3시에 자동 호출됨
}

@Scheduled 사용 조건

  1. @EnableScheduling 설정이 있어야 한다
  2. 메서드가 Spring Bean 안에 있어야 한다 (@Component, @Service 등)
  3. 메서드는 void 반환, 파라미터 없음이어야 한다
  4. 메서드는 public이어야 한다

Cron 표현식 기초¶

Cron 표현식은 실행 시점을 정의하는 문자열이다.

초  분  시  일  월  요일
 0   0  3   *   *    *
위치 의미 범위 예시의 값 설명
1 초(second) 0-59 0 0초에
2 분(minute) 0-59 0 0분에
3 시(hour) 0-23 3 3시에
4 일(day) 1-31 * 매일
5 월(month) 1-12 * 매월
6 요일(weekday) 0-7 * 매요일

자주 쓰는 Cron 예시¶

표현식 의미
0 0 3 * * * 매일 새벽 3시
0 */10 * * * * 10분마다
0 0 * * * * 매 시간 정각
0 0 9 * * MON-FRI 평일 오전 9시
0 30 1 1 * * 매월 1일 새벽 1시 30분
0 0 0 * * * 매일 자정

특수 문자 의미

문자 의미 예시
* 모든 값 매 시간, 매일
*/n n마다 */10 = 10마다
- 범위 MON-FRI = 월~금
, 나열 1,15 = 1일과 15일

SyncScheduler 코드 분석¶

전체 구조¶

@Slf4j
@Component                          // Spring Bean으로 등록
@RequiredArgsConstructor            // 생성자 주입
public class SyncScheduler {

    private final LmsSyncService lmsSyncService;     // LMS 동기화
    private final RegistrantService registrantService; // Registrant 동기화
    private final LessonRepository lessonRepository;   // 활성 강의 조회
}

@Component vs @Service

스케줄러는 @Component를 쓴다. @Service가 아니야. 왜? 스케줄러는 비즈니스 로직이 아니라 인프라 계층이기 때문이다.

  • @Service: 비즈니스 로직 담당 (AttendanceService, LmsSyncService)
  • @Component: 범용 컴포넌트 (스케줄러, 유틸리티)

기능적 차이는 없다. 둘 다 Spring Bean이다. 의도를 표현하는 거야.

dailySync() - 매일 새벽 3시 전체 동기화¶

/**
 * 매일 새벽 3시 전체 동기화
 * 1. 학사 데이터 (term → course → user → courseUser)
 * 2. 활성 강의 Registrant 동기화 (수강 변동 반영)
 */
@Scheduled(cron = "${nexclass.scheduler.sync-cron}")
public void dailySync() {
    log.info("===== [스케줄러] 일일 동기화 시작 =====");
    long start = System.currentTimeMillis();

    // 1단계: 학사 데이터 동기화
    try {
        lmsSyncService.syncAll();
    } catch (Exception e) {
        log.error("[스케줄러] 학사 데이터 동기화 실패: {}", e.getMessage());
    }

    // 2단계: 활성 강의 Registrant 동기화
    try {
        syncRegistrants();
    } catch (Exception e) {
        log.error("[스케줄러] Registrant 동기화 실패: {}", e.getMessage());
    }

    long elapsed = (System.currentTimeMillis() - start) / 1000;
    log.info("===== [스케줄러] 일일 동기화 완료 ({}초) =====", elapsed);
}

왜 각 단계를 try-catch로 감쌌나?

1단계가 실패해도 2단계는 실행되어야 한다.

학사 동기화가 실패해도 Registrant 동기화는 별개 작업이다. try-catch 없이 예외가 전파되면 1단계 실패 시 2단계가 아예 실행 안 된다.

❌ try-catch 없는 경우:
syncAll() 실패 → 예외 전파 → dailySync() 종료 → syncRegistrants() 미실행

✅ try-catch 있는 경우:
syncAll() 실패 → 로그 남기고 계속 → syncRegistrants() 정상 실행

이건 부분 실패 허용(Partial Failure Tolerance) 패턴이야.

cron 값을 하드코딩 안 하고 ${nexclass.scheduler.sync-cron}으로 외부화한 이유

application.properties에서 관리하면:

  1. 배포 없이 변경 가능 (properties만 수정)
  2. 환경별 다른 스케줄 설정 가능 (개발: 10분마다, 운영: 새벽 3시)
  3. 코드 변경 없이 스케줄 비활성화 가능 (주석 처리)
# application.properties
nexclass.scheduler.sync-cron=0 0 3 * * *

syncRegistrants() - Registrant 동기화¶

학사 데이터 동기화 후 수강 변동 반영이 필요하다. 학생이 수강 추가/취소하면 Zoom Registrant도 갱신해야 한다.

private void syncRegistrants() {
    // 현재 활성인 강의 목록 조회
    List<Lesson> activeLessons =
        lessonRepository.findActiveLessons(LocalDateTime.now());
    log.info("[스케줄러] Registrant 동기화 대상: {}개 강의",
             activeLessons.size());

    int successCount = 0;
    int failCount = 0;

    // 각 활성 강의에 대해 Registrant 동기화
    for (Lesson lesson : activeLessons) {
        try {
            registrantService.syncRegistrantsForLesson(lesson);
            successCount++;
        } catch (Exception e) {
            failCount++;
            log.error("[스케줄러] Registrant 동기화 실패: lessonCd={}, error={}",
                    lesson.getLessonCd(), e.getMessage());
        }
    }

    log.info("[스케줄러] Registrant 동기화 결과: 성공={}, 실패={}",
             successCount, failCount);
}

개별 강의 실패가 전체를 멈추지 않는다

강의 A의 Registrant 동기화가 실패해도 강의 B, C, D는 정상 처리된다.

for 루프 안에서 try-catch 하니까 하나가 터져도 다음으로 넘어간다. 마지막에 성공/실패 카운트를 로그로 남겨서 어디가 문제인지 알 수 있다.

@Scheduled 실행 방식 비교¶

Spring @Scheduled는 3가지 실행 방식을 제공한다.

fixedDelay vs fixedRate vs cron¶

@Scheduled(fixedDelay = 60000)  // 60초
public void task() { ... }

이전 실행이 끝난 후 60초 뒤에 다음 실행. 작업 시간이 30초면 실제 주기는 90초 (30 + 60).

|--작업(30초)--|----대기(60초)----|--작업(30초)--|----대기(60초)----|

용도: 작업이 겹치면 안 되는 경우. 작업 완료를 보장한 후 다음 실행.

@Scheduled(fixedRate = 60000)  // 60초
public void task() { ... }

이전 실행 시작 후 60초 뒤에 다음 실행. 작업 시간이 30초면 실제 대기 시간은 30초 (60 - 30).

|--작업(30초)--|--대기(30초)--|--작업(30초)--|--대기(30초)--|

주의: 작업 시간이 fixedRate보다 길면 밀린다 (큐잉됨).

용도: 일정한 주기로 실행해야 하는 경우. 모니터링, 상태 체크.

@Scheduled(cron = "0 0 3 * * *")  // 매일 3시
public void task() { ... }

특정 시각에 실행. 가장 유연한 방식.

용도: "매일 새벽 3시", "매주 월요일 9시" 같은 정확한 시각 지정.

NexClass에서의 선택¶

작업 방식 이유
일일 동기화 cron "매일 새벽 3시"라는 정확한 시각

왜 새벽 3시야?

  1. 사용자 없는 시간대: 새벽에는 수업이 없다
  2. 서버 부하 최소: 다른 작업과 충돌 안 함
  3. 업무 시작 전 완료: 오전 수업 전에 최신 데이터 확보
  4. 관행: 배치 작업은 전통적으로 새벽에 돌린다

@EnableScheduling 설정¶

@Scheduled가 동작하려면 Spring에게 "스케줄링 기능을 켜줘"라고 알려야 한다.

@SpringBootApplication
@EnableScheduling        // 이게 없으면 @Scheduled가 무시된다!
@EnableAsync             // @Async도 같이 활성화
public class NexClassApplication {
    public static void main(String[] args) {
        SpringApplication.run(NexClassApplication.class, args);
    }
}

@EnableScheduling 안 붙이면?

@Scheduled 어노테이션이 있어도 아무 일도 안 일어난다. 에러도 안 난다. 그냥 조용히 무시된다.

"스케줄러 안 도는데 왜 에러가 안 나지?"라는 질문의 답이 대부분 @EnableScheduling 누락이다.

스케줄러 실패 시 처리¶

현재 NexClass의 전략¶

// 각 단계를 try-catch로 감싸서 에러 로깅
try {
    lmsSyncService.syncAll();
} catch (Exception e) {
    log.error("[스케줄러] 학사 데이터 동기화 실패: {}", e.getMessage());
    // 재시도 없음 → 다음 실행 때 자동 재시도 (24시간 후)
}
항목 현재 전략 설명
에러 처리 로그 기록 log.error()로 남김
재시도 없음 다음 스케줄 실행 시 자동 재시도
알림 없음 관리자에게 별도 알림 없음
SyncLog 기록됨 FAILED 상태 + 에러 메시지

재시도가 없는 건 괜찮나?

현재 NexClass에서는 괜찮다.

  1. 동기화는 매일 돌아간다. 오늘 실패해도 내일 다시 시도.
  2. SyncLog에 FAILED가 기록되니 관리자가 대시보드에서 확인 가능.
  3. 수동 동기화(SyncController)로 즉시 재실행 가능.

프로덕션 레벨에서는 재시도 + 알림 추가를 고려해야 한다.

프로덕션에서 개선할 수 있는 것들¶

// ❌ 현재: 단순 로그
log.error("[스케줄러] 실패: {}", e.getMessage());

// ✅ 개선 1: 재시도 (최대 3회, 지수 백오프)
@Retryable(maxAttempts = 3, backoff = @Backoff(delay = 5000, multiplier = 2))
public void syncAllWithRetry() { ... }

// ✅ 개선 2: 알림 (Slack, 이메일)
catch (Exception e) {
    log.error("실패: {}", e.getMessage());
    alertService.sendSlack("[NexClass] 동기화 실패: " + e.getMessage());
}

스케줄러와 서버 재시작의 관계¶

서버 재시작하면 스케줄러는 어떻게 되나?

스케줄러는 서버 메모리에서 동작한다. DB에 스케줄 정보를 저장하지 않는다.

상황 결과
서버 정상 가동 @Scheduled 정상 실행
서버 중지 스케줄러 중지. 놓친 작업은 실행 안 됨
서버 재시작 스케줄러 다시 시작. 다음 스케줄부터 실행 
서버 시작 (오후 2시) → 스케줄러 등록 (새벽 3시 실행 대기)
새벽 3시 → dailySync() 실행
서버 중지 (오전 11시)
서버 재시작 (오후 1시) → 스케줄러 다시 등록 → 새벽 3시 대기
↑ 오전 11시~오후 1시 사이에 스케줄된 작업이 있었다면 "놓침"

놓친 스케줄은 어떡해?

Spring @Scheduled는 놓친 작업을 보완 실행(catch-up) 하지 않는다.

새벽 3시에 서버가 꺼져 있었다면? 그 날의 동기화는 안 된다. 다음 날 새벽 3시에 다시 실행된다.

급하면 관리자가 수동으로 SyncController API를 호출하면 된다.

이걸 해결하려면 Quartz 같은 외부 스케줄러를 쓰거나, 서버 시작 시 "마지막 동기화가 24시간 이상 전이면 즉시 실행" 로직을 추가할 수 있다.

기본 스레드 풀이 1개라는 거에 주의

Spring @Scheduled의 기본 스레드 풀 크기는 1이다. 스케줄러가 여러 개 있어도 동시에 1개만 실행된다.

NexClass는 현재 스케줄러가 1개(dailySync)라서 괜찮다. 스케줄러가 추가되면 스레드 풀 크기를 늘려야 한다.

# application.properties
spring.task.scheduling.pool.size=2

스케줄러 vs 이벤트 기반 처리¶

NexClass에는 두 가지 자동 처리 방식이 있다.

구분 스케줄러 이벤트 기반
트리거 시간 (매일 3시) 이벤트 (Zoom 웹훅)
예시 dailySync() processAttendance()
주기 고정 (cron 표현식) 실시간 (이벤트 발생 시)
장점 예측 가능, 배치 처리 적합 즉시 반응, 실시간
단점 지연 (최대 24시간) 이벤트 놓칠 수 있음
용도 대량 데이터 동기화 개별 이벤트 처리

두 방식을 같이 쓰는 이유

출결 처리: 이벤트 기반이 적합하다. 미팅 끝나면 바로 처리해야 하니까. 학사 데이터 동기화: 스케줄러가 적합하다. 5만 건 데이터를 실시간으로 동기화할 필요 없다.

"모든 걸 하나의 방식으로 해결하려 하지 마. 상황에 맞는 도구를 써야 한다."

주의사항¶

스케줄러 개발/운영 시 주의할 것들

1. @EnableScheduling 확인

빠뜨려도 에러가 안 난다. 그냥 스케줄러가 안 도는 거야. "왜 안 돌아가지?"의 첫 번째 체크 포인트.

2. 서버 다운 시 놓침

서버가 꺼져 있으면 스케줄 실행이 안 된다. 중요한 작업이면 서버 시작 시 보완 로직을 추가해야 한다.

3. 긴 작업은 타임아웃 고려

동기화가 30분 걸리면? 다음 스케줄과 겹칠 수 있다. fixedDelay를 쓰거나, 실행 중 플래그를 둬서 중복 실행을 방지해야 한다.

4. 로컬 개발 시 스케줄러 비활성화

개발 중에 새벽 3시 동기화가 돌면 불필요한 API 호출이 발생한다. 개발 환경에서는 cron 값을 비활성화하거나 프로파일로 분기.

# application-dev.properties
nexclass.scheduler.sync-cron=-  # 비활성화

5. 스레드 풀 크기

스케줄러 추가 시 기본 스레드 풀(1개)로는 부족할 수 있다. 작업이 밀리거나 지연될 수 있으니 풀 크기를 늘려야 한다.

정리¶

이 챕터에서 기억해야 할 것

  1. @Scheduled + @EnableScheduling = Spring 스케줄링
  2. Cron 표현식: 초 분 시 일 월 요일 - 0 0 3 * * * = 매일 새벽 3시
  3. fixedDelay: 이전 완료 후 N초 뒤 / fixedRate: 이전 시작 후 N초 뒤 / cron: 특정 시각
  4. 부분 실패 허용: 각 단계를 try-catch로 감싸서 하나 실패해도 나머지 실행
  5. 서버 중지 = 스케줄 놓침: 보완 로직 또는 수동 트리거 필요
  6. cron 값 외부화: application.properties에서 관리 → 배포 없이 변경
  7. 기본 스레드 풀 1개: 스케줄러 추가 시 풀 크기 설정 확인

확인문제¶

1. Cron 표현식 '0 0 3 * * '를 한국어로 해석해봐. 그리고 '0 /10 * * * *'는?

0 0 3 * * *: 매일 새벽 3시 0분 0초에 실행. - 0초, 0분, 3시, 매일(), 매월(), 매요일(*)

0 */10 * * * *: 10분마다 실행. - 0초, 10분 간격(/10), 매시(), 매일(), 매월(), 매요일(*) - 즉, 00분, 10분, 20분, 30분, 40분, 50분에 실행

2. dailySync()에서 syncAll()과 syncRegistrants()를 각각 try-catch로 감싼 이유는?

부분 실패 허용(Partial Failure Tolerance) 패턴이다.

try-catch 없이 예외가 전파되면 syncAll() 실패 시 syncRegistrants()가 아예 실행되지 않는다.

두 작업은 독립적이다. 학사 데이터 동기화가 실패해도 Registrant 동기화는 (기존 데이터로라도) 실행될 수 있다.

각각 try-catch로 감싸면 하나가 실패해도 나머지는 정상 실행되고, 실패한 작업은 에러 로그로 기록된다.

3. fixedDelay와 fixedRate의 차이를 설명하고, NexClass가 cron을 선택한 이유는?

fixedDelay: 이전 실행이 끝난 후 N밀리초 뒤에 다음 실행. 작업 시간 30초, delay 60초면 → 실제 주기 90초.

fixedRate: 이전 실행이 시작한 후 N밀리초 뒤에 다음 실행. 작업 시간 30초, rate 60초면 → 실제 대기 30초.

NexClass가 cron을 선택한 이유: "매일 새벽 3시"라는 정확한 시각에 실행해야 하기 때문이다. fixedDelay/fixedRate는 "N초마다"라서 서버 시작 시간에 따라 실행 시각이 달라진다.

4. 서버가 새벽 2시에 다운되고 오전 8시에 재시작되면, 새벽 3시 스케줄은 어떻게 되나?

실행되지 않는다.

Spring @Scheduled는 서버 메모리에서 동작한다. 서버가 꺼져 있으면 스케줄러도 꺼져 있다. 놓친 스케줄을 보완 실행(catch-up)하는 기능이 없다.

오전 8시에 재시작하면 스케줄러가 다시 등록되고, 다음 날 새벽 3시에 실행된다.

그 날의 동기화가 필요하면 관리자가 수동으로 POST /api/sync/all을 호출해야 한다.

5. @EnableScheduling 없이 @Scheduled를 붙이면 어떻게 되나?

아무 일도 안 일어난다. 에러도 안 난다.

@EnableScheduling이 없으면 Spring이 스케줄러 인프라를 초기화하지 않는다. @Scheduled 어노테이션은 그냥 무시된다. 메서드가 존재하지만 아무도 호출하지 않는 거야.

이게 위험한 이유는 실패가 조용하기 때문이다. 에러 로그도 안 남고, 그냥 안 돌아갈 뿐이야. 스케줄러가 안 도는 것 같으면 첫 번째로 확인할 것이 @EnableScheduling이다.
