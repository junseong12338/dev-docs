12. Zoom Server-to-Server OAuth 연동¶

핵심 개념: OAuth 간단 복습¶

OAuth는 "내 대신 API 써도 돼"라는 권한 위임 프로토콜이야.

일상에서 비유하면:

"구글 캘린더 앱이 내 구글 계정의 일정을 읽어도 된다"를 허락하는 과정

이때 내 구글 비밀번호를 캘린더 앱에 직접 주지 않는다. 대신 "이 앱이 일정 읽기만 할 수 있는 토큰"을 발급받아서 쓴다. 이게 OAuth의 핵심이야.

일반 OAuth 흐름 (Authorization Code Grant)¶

sequenceDiagram
    participant U as 사용자 (브라우저)
    participant APP as 앱 서버
    participant AUTH as Zoom 인증 서버

    U->>AUTH: ① 로그인 + "이 앱에 권한 줄래?" 동의
    AUTH-->>APP: ② Authorization Code 전달
    APP->>AUTH: ③ Code + Client Secret → 토큰 요청
    AUTH-->>APP: ④ Access Token 발급
    APP->>APP: ⑤ 토큰으로 Zoom API 호출

핵심: 사용자가 직접 동의해야 토큰이 발급된다.

동작 원리: S2S OAuth는 뭐가 다른데?¶

Server-to-Server (S2S) OAuth¶

"사용자? 사용자 없어. 서버끼리 직접 통신하는 거야."

NexClass는 학생이 Zoom에 로그인해서 동의하는 게 아니야. NexClass 서버가 Zoom 서버에 직접 요청을 보내서 미팅을 만들고, 참가자를 등록하고, 출석을 조회한다. 사용자 개입이 전혀 없다.

구분 일반 OAuth S2S OAuth
사용자 동의 필요 (브라우저 로그인) 불필요
Grant Type authorization_code account_credentials
인증 방식 Client ID + Secret + 사용자 Code Client ID + Secret + Account ID
토큰 범위 사용자 개인 리소스 계정 전체 리소스
용도 사용자별 기능 서버 자동화, 배치 작업

S2S OAuth의 파워

S2S 토큰은 Zoom 계정 전체에 대한 권한을 가진다. 특정 사용자가 아니라 관리자 레벨이야. 모든 미팅을 만들고, 모든 참가자 데이터를 조회할 수 있다. 그래서 Client Secret이 유출되면 계정 전체가 위험해진다.

Account Credentials Grant 흐름¶

sequenceDiagram
    participant NC as NexClass 서버
    participant ZA as Zoom OAuth 서버<br/>(zoom.us/oauth/token)
    participant ZP as Zoom API 서버<br/>(api.zoom.us/v2)

    Note over NC,ZP: ① 토큰 발급

    NC->>NC: Client ID + Client Secret<br/>→ Base64 인코딩
    NC->>ZA: POST /oauth/token<br/>Authorization: Basic {Base64(id:secret)}<br/>grant_type=account_credentials<br/>account_id={ZOOM_ACCOUNT_ID}
    ZA->>ZA: Client ID/Secret 검증<br/>Account ID 확인<br/>앱 권한(Scope) 확인
    ZA-->>NC: 200 OK<br/>{"access_token": "eyJ...", "expires_in": 3600}

    Note over NC,ZP: ② API 호출

    NC->>ZP: POST /v2/users/{email}/meetings<br/>Authorization: Bearer {access_token}
    ZP->>ZP: 토큰 검증 → 권한 확인
    ZP-->>NC: 201 Created<br/>{"id": 12345, "join_url": "..."}

3가지 정보만 있으면 된다:

  1. Client ID -- Zoom Marketplace에서 앱 만들 때 발급
  2. Client Secret -- 같이 발급 (절대 노출 금지)
  3. Account ID -- Zoom 계정 식별자

코드 분석: ZoomService.getAccessToken()¶

@Slf4j
@Service
public class ZoomService {

    @Value("${zoom.account-id}")
    private String accountId;       // Zoom 계정 ID: {ZOOM_ACCOUNT_ID}

    @Value("${zoom.client-id}")
    private String clientId;        // 앱 Client ID: {ZOOM_CLIENT_ID}

    @Value("${zoom.client-secret}")
    private String clientSecret;    // 앱 Client Secret: {ZOOM_CLIENT_SECRET}

    @Value("${zoom.oauth-url}")
    private String oauthUrl;        // https://zoom.us/oauth/token

    // ── 토큰 캐싱 ──
    private String accessToken;         // 현재 유효한 토큰
    private LocalDateTime tokenExpiry;  // 만료 시각

    /**
     * Zoom OAuth 액세스 토큰 발급
     * 캐싱된 토큰이 유효하면 재사용, 아니면 새로 발급
     */
    public String getAccessToken() {

        // ① 캐싱 확인: 토큰이 아직 유효하면 그대로 반환
        if (accessToken != null && tokenExpiry != null
                && LocalDateTime.now().isBefore(tokenExpiry)) {
            log.debug("Using cached Zoom access token");
            return accessToken;
        }

        log.info("Requesting new Zoom access token");

        // ② Basic Auth 헤더: client_id:client_secret → Base64 인코딩
        String credentials = clientId + ":" + clientSecret;
        String encodedCredentials = Base64.getEncoder()
            .encodeToString(credentials.getBytes(StandardCharsets.UTF_8));

        HttpHeaders headers = new HttpHeaders();
        headers.setContentType(MediaType.APPLICATION_FORM_URLENCODED);
        headers.set("Authorization", "Basic " + encodedCredentials);

        // ③ 요청 바디: grant_type + account_id
        MultiValueMap<String, String> body = new LinkedMultiValueMap<>();
        body.add("grant_type", "account_credentials");  // S2S 전용 grant type
        body.add("account_id", accountId);

        HttpEntity<MultiValueMap<String, String>> request =
            new HttpEntity<>(body, headers);

        try {
            // ④ POST 요청 → 토큰 응답
            ResponseEntity<Map> response =
                restTemplate.postForEntity(oauthUrl, request, Map.class);
            Map<String, Object> responseBody = response.getBody();

            if (responseBody != null
                    && responseBody.containsKey("access_token")) {

                accessToken = (String) responseBody.get("access_token");
                int expiresIn = (Integer) responseBody.get("expires_in");

                // ⑤ 만료 5분 전에 갱신하도록 설정
                tokenExpiry = LocalDateTime.now()
                    .plusSeconds(expiresIn - 300);

                log.info("Zoom access token obtained successfully");
                return accessToken;
            }
        } catch (Exception e) {
            log.error("Failed to get Zoom access token: {}",
                e.getMessage());
            throw new RuntimeException(
                "Zoom 토큰 발급 실패: " + e.getMessage());
        }

        throw new RuntimeException("Zoom 토큰 발급 실패: 응답 없음");
    }
}

토큰 발급 요청 상세¶

POST https://zoom.us/oauth/token HTTP/1.1
Content-Type: application/x-www-form-urlencoded
Authorization: Basic Q1Y4bnZKQ1FJMmpRNVZyaFoza3c6SGJSMW...

grant_type=account_credentials&account_id={ZOOM_ACCOUNT_ID}

{
    "access_token": "eyJzdiI6IjAwMDAwMSIsImFsZyI6IkhTNT...",
    "token_type": "bearer",
    "expires_in": 3600,
    "scope": "meeting:write:admin meeting:read:admin ..."
}

원본: {ZOOM_CLIENT_ID}:{ZOOM_CLIENT_SECRET}
Base64: Q1Y4bnZKQ1FJMmpRNVZyaFoza3c6SGJSMW...

이건 HTTP Basic 인증의 표준이야.
"아이디:비밀번호"를 Base64로 인코딩해서 헤더에 넣는다.
Base64는 암호화가 아니라 인코딩이니까 HTTPS 없으면 노출된다.

토큰 캐싱 전략¶

왜 매번 새로 안 받나?¶

Zoom OAuth 토큰의 유효시간은 3600초(1시간)이야. 수업 하나 생성하는 데 API를 3~4번 호출하는데, 매번 토큰을 새로 발급받으면:

  1. 불필요한 네트워크 요청 (매번 zoom.us에 POST)
  2. API Rate Limit 소모 (Zoom은 분당 요청 수를 제한한다)
  3. 응답 시간 증가 (토큰 발급에 200~500ms 추가)

캐싱 로직 상세¶

// 유효 기간: 3600초 (1시간)
// 캐싱 만료: 3600 - 300 = 3300초 (55분)

if (accessToken != null && tokenExpiry != null
        && LocalDateTime.now().isBefore(tokenExpiry)) {
    return accessToken;  // 캐시 히트: 기존 토큰 반환
}
// 캐시 미스: 새 토큰 발급

왜 5분 전에 갱신하나?

시간 → ─────────────────────────────────────────────>
       |← 토큰 유효 (3600초) →|
       |← 캐시 사용 (3300초) →|← 갱신 구간 (300초) →|
       ^발급                  ^캐시 만료            ^토큰 만료

토큰이 딱 만료되는 시점에 API를 호출하면 요청 도중에 만료돼서 실패할 수 있다. 5분의 여유를 두면 만료 직전에 안전하게 새 토큰을 받을 수 있다. 이걸 early refresh 패턴이라고 해.

현재 캐싱의 한계

이 캐싱은 인스턴스 변수에 저장한다. 서버가 여러 대면? 서버 A가 발급한 토큰을 서버 B는 모른다.

NexClass는 서버 1대라서 문제없지만, 다중 서버 환경에서는 Redis 같은 외부 캐시를 써야 한다.

Zoom API 인증 헤더 구성¶

토큰을 발급받은 후 실제 API 호출은 이렇게 한다:

// 모든 Zoom API 호출에 공통으로 사용되는 패턴
String token = getAccessToken();  // 캐시 or 새 발급

HttpHeaders headers = new HttpHeaders();
headers.setContentType(MediaType.APPLICATION_JSON);
headers.setBearerAuth(token);  // "Authorization: Bearer {token}" 헤더 설정

setBearerAuth(token)은 내부적으로 이렇게 동작한다:

Authorization: Bearer eyJzdiI6IjAwMDAwMSIsImFsZyI6IkhTNTEyIiwidHlwIjoiSldU...
인증 단계 헤더 형식 용도
토큰 발급 Authorization: Basic {Base64(id:secret)} Zoom에게 "나 이 앱이야" 증명
API 호출 Authorization: Bearer {access_token} Zoom에게 "나 인증된 앱이야" 증명

S2S OAuth의 보안 고려사항¶

Client Secret 유출 시 영향¶

유출 정보 영향 범위
Client ID만 위험 낮음 (ID만으로는 토큰 발급 불가)
Client Secret만 위험 낮음 (Account ID도 필요)
Client ID + Secret + Account ID 치명적: 전체 Zoom 계정 제어 가능

3가지가 모두 유출되면 공격자가 미팅 생성, 참가자 조회, 녹화 다운로드까지 전부 할 수 있다.

보안 체크리스트¶

항목 NexClass 현재 상태 프로덕션 권장
Client Secret 저장 application.properties 평문 환경 변수 또는 Vault
Account ID 저장 application.properties 평문 환경 변수
토큰 로깅 access_token 로깅 안 함 로깅 금지 유지
HTTPS zoom.us/api.zoom.us (강제 HTTPS) 유지
앱 권한 범위 필요한 Scope만 설정 정기 검토

application.properties를 Git에 올리면 끝이야

zoom.client-secret, zoom.account-id 같은 값이 GitHub에 올라가면 봇이 자동으로 스캔해서 악용한다. 실제로 일어나는 일이야. .gitignore에 application.properties를 반드시 추가하거나, 민감 정보는 환경 변수로 분리해야 한다.

주의사항¶

S2S OAuth 운영 시 알아야 할 것

1. 토큰 유효시간은 Zoom이 정한다

현재 3600초(1시간)인데 Zoom이 언제든 바꿀 수 있다. expires_in 값을 하드코딩하지 말고 응답에서 동적으로 읽어야 한다. NexClass는 이걸 제대로 하고 있다.

2. Rate Limit에 걸릴 수 있다

Zoom API는 분당/일당 요청 수를 제한한다. 토큰 발급도 Rate Limit에 포함되니까 캐싱이 중요하다.

3. S2S 앱은 Zoom Marketplace에서 활성화해야 한다

관리자가 Zoom Marketplace에서 앱을 비활성화하면 토큰 발급이 즉시 거부된다. 서버 에러 로그에 401이 뜨면 앱 상태부터 확인해라.

정리¶

개념 한 줄 요약
OAuth "내 대신 API 써도 돼"라는 권한 위임 프로토콜
S2S OAuth 사용자 동의 없이 서버끼리 직접 인증하는 방식
Account Credentials Grant S2S 전용 grant type, Client ID + Secret + Account ID로 토큰 발급
Basic Auth id:secret을 Base64 인코딩하여 Authorization 헤더에 넣는 방식
Bearer Token 발급받은 access_token을 API 호출 시 헤더에 넣는 방식
토큰 캐싱 만료 5분 전까지 기존 토큰 재사용 (early refresh 패턴)

확인 문제¶

1. 일반 OAuth(Authorization Code Grant)와 S2S OAuth(Account Credentials Grant)의 핵심 차이는?

사용자 동의 여부다.

  • 일반 OAuth: 사용자가 브라우저에서 로그인하고 "이 앱에 권한 줄래?" 동의해야 한다. 토큰은 해당 사용자의 리소스에만 접근 가능하다.
  • S2S OAuth: 사용자 개입 없이 서버가 Client ID + Secret + Account ID로 직접 토큰을 발급받는다. 토큰은 Zoom 계정 전체 리소스에 접근 가능하다.

NexClass는 학생/교수의 동의를 받을 수 없는 구조(LMS에서 자동으로 미팅 생성)라서 S2S가 유일한 선택이다.

2. ZoomService에서 토큰 만료 5분 전에 갱신하도록 설정한 이유는?

토큰이 만료되는 시점에 API를 호출하면 요청 도중 만료될 수 있기 때문이다.

tokenExpiry = LocalDateTime.now().plusSeconds(expiresIn - 300);

토큰 유효시간이 3600초(1시간)면, 3300초(55분) 후에 캐시를 무효화한다. 이 5분(300초)의 여유 시간 동안 새 토큰을 발급받으면 기존 토큰이 만료되기 전에 안전하게 교체할 수 있다. 이걸 early refresh 패턴이라고 한다.

3. 토큰 발급 시 Basic Auth를 쓰고, API 호출 시 Bearer Auth를 쓴다. 왜 다를까?

각 단계에서 증명하는 것이 다르기 때문이다.

  • Basic Auth (토큰 발급): "나는 이 앱이다"를 증명한다. Client ID와 Secret을 Base64로 인코딩하여 전달한다. Zoom 인증 서버가 이 앱이 유효한지 확인한다.
  • Bearer Auth (API 호출): "나는 이미 인증된 앱이다"를 증명한다. 발급받은 access_token을 전달한다. Zoom API 서버가 이 토큰이 유효한지만 확인한다.

두 단계를 분리하는 이유는 Client Secret을 매 API 호출마다 노출하지 않기 위해서다. 토큰 발급 시 한 번만 Secret을 보내고, 이후에는 토큰만 사용한다.

4. 현재 토큰 캐싱이 인스턴스 변수에 저장되어 있다. 서버가 3대로 늘어나면 어떤 문제가 생기나?

서버 간 토큰이 공유되지 않아 불필요한 토큰 발급이 발생한다.

  • 서버 A가 토큰을 발급받아 캐싱해도 서버 B, C는 모른다
  • 서버 B, C도 각각 새 토큰을 발급받는다 (총 3번 발급)
  • Rate Limit 소모가 3배 증가한다
  • 해결: Redis, Memcached 같은 외부 캐시에 토큰을 저장하면 모든 서버가 같은 토큰을 공유할 수 있다

NexClass는 서버 1대 운영이라 현재 문제가 없지만, 스케일 아웃 시 반드시 개선해야 할 부분이다.
