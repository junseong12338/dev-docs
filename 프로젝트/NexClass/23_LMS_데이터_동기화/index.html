23. LMS 데이터 동기화¶

핵심 개념¶

이 챕터의 핵심

NexClass는 독립 시스템이 아니다. LMS(LXP-KNU10)의 학사 데이터가 있어야 동작한다. 학기, 과목, 사용자, 수강생 정보가 NexClass DB에 없으면 강의 생성도, 출결 매칭도 불가능하다. 그래서 LMS API를 호출해서 NexClass DB에 동기화(Sync)한다.

동기화를 이해하려면 "왜 NexClass에 LMS 데이터가 필요한가?"부터 시작해야 한다.

왜 동기화가 필요한가?¶

NexClass가 하는 일을 생각해봐.

NexClass 기능 필요한 LMS 데이터
강의 생성 어떤 과목(Course)의 강의인지
수강생 Registrant 등록 이 과목의 수강생(CourseUser) 목록
출결 이메일 매칭 학생 정보(User.userCd)
호스트 배정 어떤 학기(Term)에 속하는지
LMS Push LMS의 과목 코드(courseLmsCd)

LMS DB 직접 접근하면 안 돼?

안 된다. 시스템 분리 원칙 때문이다.

LMS와 NexClass는 독립적인 시스템이다. LMS DB에 직접 접속하면:

  1. LMS DB 스키마 변경 시 NexClass가 깨진다
  2. LMS DB 장애 시 NexClass도 같이 죽는다
  3. 두 시스템의 결합도(coupling)가 극도로 높아진다

API를 통한 동기화는 느슨한 결합(loose coupling)을 유지한다. LMS 내부가 어떻게 바뀌든 API 스펙만 같으면 NexClass는 영향 없다.

동기화 대상과 순서¶

순서가 중요하다¶

graph TD
    A["1. Term<br/>(학기)"] --> B["2. Course<br/>(과목)"]
    B --> C["3. User<br/>(사용자)"]
    C --> D["4. CourseUser<br/>(수강생 매핑)"]

    A -.->|"Course가 Term을 참조"| B
    B -.->|"CourseUser가 Course를 참조"| D
    C -.->|"CourseUser가 User를 참조"| D

    style A fill:#ff6b6b,color:#fff
    style B fill:#ffd43b,color:#333
    style C fill:#ffd43b,color:#333
    style D fill:#51cf66,color:#fff

왜 이 순서를 바꾸면 안 되나?

참조 무결성(Referential Integrity) 때문이다.

Course 테이블에는 termCd가 있다 (어떤 학기 소속인지). Term을 먼저 넣지 않고 Course를 넣으면? 참조할 Term이 없다.

CourseUser는 courseCd와 userCd를 둘 다 참조한다. Course와 User가 먼저 있어야 CourseUser를 넣을 수 있다.

Term(학기) → Course(과목이 학기를 참조) → User(사용자)
                                          → CourseUser(과목+사용자 매핑)

데이터 규모¶

동기화 대상 건수 페이징
Term (학기) ~16건 불필요
Course (과목) ~351건 불필요
User (사용자) ~51,683건 필요
CourseUser (수강생) 대량 필요

User와 CourseUser는 건수가 많아서 페이징 처리가 필수다. 한 번에 5만 건 가져오면 메모리 터진다.

LmsSyncService 코드 분석¶

전체 동기화 엔트리 포인트¶

@Slf4j
@Service
@RequiredArgsConstructor
public class LmsSyncService {

    private final TermRepository termRepository;
    private final CourseRepository courseRepository;
    private final UserRepository userRepository;
    private final CourseUserRepository courseUserRepository;
    private final SyncLogRepository syncLogRepository;

    private final RestClient restClient = RestClient.create();
    private final ObjectMapper objectMapper = new ObjectMapper();

    // 설정값 (application.properties에서 주입)
    @Value("${lms.sync.base-url}")
    private String baseUrl;              // LMS API 베이스 URL

    @Value("${lms.sync.org-cd}")
    private String orgCd;                // 기관 코드

    @Value("${nexclass.api.key}")
    private String apiKey;               // JWT issuer (API Key)

    @Value("${nexclass.api.secret}")
    private String apiSecret;            // JWT 서명 키

    @Value("${lms.sync.aes-key}")
    private String aesKey;               // AES-256 복호화 키

    @Value("${lms.sync.page-size}")
    private int pageSize;                // 페이징 크기

    // 순서 중요! FK 의존성 순서대로 동기화
    public void syncAll() {
        log.info("=== LMS 전체 동기화 시작 ===");
        syncTerms();        // 1. 학기 먼저
        syncCourses();      // 2. 과목 (학기 참조)
        syncUsers();        // 3. 사용자
        syncCourseUsers();  // 4. 수강생 매핑 (과목+사용자 참조)
        log.info("=== LMS 전체 동기화 완료 ===");
    }
}

JWT 토큰 생성¶

LMS API를 호출하려면 인증이 필요하다. JWT 토큰을 만들어서 Authorization 헤더에 넣는다.

private String generateJwtToken() {
    // LMS의 JwtServerManager와 동일한 방식으로 키 생성
    SecretKey key = new SecretKeySpec(
        apiSecret.getBytes(), "HmacSHA256");

    return Jwts.builder()
            .issuer(apiKey)           // LMS가 issuer로 APP_KEY 검증
            .issuedAt(new Date())     // 발급 시간
            .expiration(new Date(
                System.currentTimeMillis() + 3600000))  // 1시간 유효
            .signWith(key)            // HMAC-SHA256 서명
            .compact();
}

Header.Payload.Signature

Payload:
{
    "iss": "{API_KEY}",     ← LMS가 이걸로 "누구인지" 확인
    "iat": 1711065600,      ← 언제 발급했는지
    "exp": 1711069200       ← 언제 만료되는지

LMS는 issuer를 보고 "아, NexClass에서 온 요청이구나"를 판단한다. apiSecret으로 서명을 검증해서 위조 여부를 확인한다.

LMS API 호출 + AES 복호화¶

이게 동기화의 핵심 메서드다. 모든 동기화 메서드가 이걸 호출한다.

private Map<String, Object> callLmsApi(String type,
                                        Integer pageSize,
                                        Integer pageNumber) {
    String token = generateJwtToken();

    // 요청 바디 구성 (페이징 파라미터)
    Map<String, Object> requestBody;
    if (pageSize != null) {
        requestBody = Map.of("page", Map.of(
                "pageSize", pageSize,
                "pageNumber", pageNumber
        ));
    } else {
        requestBody = Map.of();  // 페이징 불필요한 경우
    }

    // LMS API 호출 (POST)
    String url = baseUrl + "/vc/api/sync/" + type;
    log.info("LMS API 호출: {}", url);

    String encrypted = restClient.post()
            .uri(url)
            .contentType(MediaType.APPLICATION_JSON)
            .header("Authorization", "Bearer " + token)
            .body(requestBody)
            .retrieve()
            .onStatus(status -> status.isError(), (request, response) -> {
                // 에러 응답도 암호화되어 있으므로 복호화 시도
                String errorBody = new String(
                    response.getBody().readAllBytes(),
                    StandardCharsets.UTF_8);
                try {
                    AES256Util aes = new AES256Util(aesKey);
                    String decrypted = aes.decrypt(errorBody);
                    log.error("LMS 에러 응답 (복호화): {}", decrypted);
                    throw new RuntimeException("LMS 에러: " + decrypted);
                } catch (GeneralSecurityException e) {
                    log.error("LMS 에러 응답 (원문): {}", errorBody);
                    throw new RuntimeException("LMS 에러: " + errorBody);
                }
            })
            .body(String.class);  // 암호화된 문자열로 받기

    // AES-256 복호화 → JSON 문자열 → Map 변환
    try {
        AES256Util aes = new AES256Util(aesKey);
        String json = aes.decrypt(encrypted);

        return objectMapper.readValue(json,
            new TypeReference<Map<String, Object>>() {});
    } catch (GeneralSecurityException e) {
        throw new RuntimeException("AES 복호화 실패: " + e.getMessage(), e);
    } catch (Exception e) {
        throw new RuntimeException("JSON 파싱 실패: " + e.getMessage(), e);
    }
}

왜 응답이 암호화되어 있어?

학사 데이터는 민감 정보다. 학생 이름, 학번, 이메일 등이 포함되어 있다. HTTPS만으로는 부족하다고 판단해서 이중 보안을 적용했다.

  1. HTTPS: 전송 구간 암호화
  2. AES-256: 응답 본문 자체를 암호화

LMS가 보내는 모든 응답은 AES-256으로 암호화되어 있고, NexClass가 aesKey로 복호화해야 실제 JSON을 볼 수 있다.

public void syncTerms() {

            (List<Map<String, Object>>) result.get("list");

        for (Map<String, Object> item : list) {
            Term term = Term.builder()
                    .termCd((String) item.get("termCd"))
                    .termName((String) item.get("termName"))
                    .termYear((String) item.get("termYear"))
                    .termNo((String) item.get("termNo"))
                    .curriCd((String) item.get("curriCd"))
                    .orgCd((String) item.get("orgCd"))
                    .serviceStart((String) item.get("serviceStart"))
                    .serviceEnd((String) item.get("serviceEnd"))
                    .useYn((String) item.get("useYn"))
                    .syncedAt(LocalDateTime.now())
                    .build();

    }
}

Upsert 전략¶

Upsert가 뭐야?

Upsert = Update + Insert

  • 해당 PK가 이미 있으면 → UPDATE (기존 데이터 갱신)
  • 해당 PK가 없으면 → INSERT (새 데이터 삽입)

JPA의 save() 메서드는 기본적으로 Upsert 동작을 한다. 엔티티에 PK가 설정되어 있으면 해당 PK로 조회해서 있으면 UPDATE, 없으면 INSERT를 수행한다.

LMS 데이터: termCd="2026-1"
NexClass DB: termCd="2026-1" 없음
→ INSERT INTO TB_NEXCLASS_TERM (TERM_CD, ...) VALUES ('2026-1', ...)

LMS 데이터: termCd="2025-2", termName="2025년 2학기(수정)"
NexClass DB: termCd="2025-2" 있음
→ UPDATE TB_NEXCLASS_TERM SET TERM_NAME='2025년 2학기(수정)' WHERE TERM_CD='2025-2'

사용자 동기화 (syncUsers) - 페이징 처리¶

5만 건 이상이라 페이징 필수.

public void syncUsers() {
    SyncLog syncLog = saveSyncLog("users", "PROCESSING");

    try {
        int totalCount = 0;
        int pageNumber = 0;

        // 페이징 루프
        while (true) {
            Map<String, Object> result =
                callLmsApi("users", pageSize, pageNumber);
            List<Map<String, Object>> list =
                (List<Map<String, Object>>) result.get("list");

            // 데이터 없으면 종료
            if (list == null || list.isEmpty()) break;

            for (Map<String, Object> item : list) {
                User user = User.builder()
                        .userCd((String) item.get("userCd"))
                        .userName((String) item.get("userName"))
                        .userNo((String) item.get("userNo"))
                        .userType((String) item.get("userType"))
                        .userEmail((String) item.get("userEmail"))
                        .deptCd((String) item.get("deptCd"))
                        .orgCd((String) item.get("orgCd"))
                        .enrolledYn((String) item.get("enrolledYn"))
                        .syncedAt(LocalDateTime.now())
                        .build();
                userRepository.save(user);
                totalCount++;
            }

            // 페이징 종료 조건: 마지막 페이지 도달
            Map<String, Object> page =
                (Map<String, Object>) result.get("page");
            int pageCount = (int) page.get("pageCount");
            pageNumber++;

            if (pageNumber >= pageCount) break;

            log.info("사용자 동기화 진행중: {}/{} 페이지",
                     pageNumber, pageCount);
        }

        completeSyncLog(syncLog, totalCount);
        log.info("사용자 동기화 완료: {}건", totalCount);

    } catch (Exception e) {
        failSyncLog(syncLog, e.getMessage());
        log.error("사용자 동기화 실패: {}", e.getMessage());
    }
}

수강생 동기화 - 중복 처리 (Upsert 수동 구현)¶

수강생 매핑(CourseUser)은 단순 save()로는 안 된다. PK가 Auto Increment라서 같은 userCd + courseCd 조합이 중복 삽입될 수 있다.

public void syncCourseUsers() {
    SyncLog syncLog = saveSyncLog("crsUsers", "PROCESSING");

    try {
        int totalCount = 0;
        int pageNumber = 0;

        while (true) {
            Map<String, Object> result =
                callLmsApi("crsUsers", pageSize, pageNumber);
            List<Map<String, Object>> list =
                (List<Map<String, Object>>) result.get("list");

            if (list == null || list.isEmpty()) break;

            for (Map<String, Object> item : list) {
                String userCd = (String) item.get("userCd");
                String courseCd = (String) item.get("courseCd");

                // 기존 데이터 확인
                Optional<CourseUser> existing =
                    courseUserRepository.findByUserCdAndCourseCd(
                        userCd, courseCd);

                if (existing.isPresent()) {
                    // UPDATE: 기존 데이터 갱신
                    CourseUser cu = existing.get();
                    cu.setUserType((String) item.get("userType"));
                    cu.setSyncedAt(LocalDateTime.now());
                    courseUserRepository.save(cu);
                } else {
                    // INSERT: 새 데이터 삽입
                    CourseUser courseUser = CourseUser.builder()
                            .courseCd(courseCd)
                            .userCd(userCd)
                            .userType((String) item.get("userType"))
                            .orgCd((String) item.get("orgCd"))
                            .syncedAt(LocalDateTime.now())
                            .build();
                    courseUserRepository.save(courseUser);
                }
                totalCount++;
            }

            Map<String, Object> page =
                (Map<String, Object>) result.get("page");
            int pageCount = (int) page.get("pageCount");
            pageNumber++;
            if (pageNumber >= pageCount) break;
        }

        completeSyncLog(syncLog, totalCount);
    } catch (Exception e) {
        failSyncLog(syncLog, e.getMessage());
    }
}

왜 수강생만 수동 Upsert야?

Term, Course, User: PK가 의미 있는 값 (termCd, courseCd, userCd). 같은 PK로 save() 하면 JPA가 자동으로 UPDATE 한다.

CourseUser: PK가 Auto Increment (id). 같은 userCd + courseCd 조합이어도 save() 하면 새 행이 INSERT 된다. 그래서 findByUserCdAndCourseCd()로 먼저 확인하고, 있으면 UPDATE, 없으면 INSERT.

이 문제가 실제로 발생했다 (2026-03-20). Duplicate Entry 에러가 터져서 이 방식으로 수정했다.

Push 실패하면 출결이 유실되나?

아니다. Push는 "출결 처리 끝났으니 가져가라"는 알림일 뿐이다.

LMS는 이 알림을 받고 GET /v2/api/lessons/{lessonCd}/attends를 호출해서 출결 데이터를 직접 가져간다.

Push가 실패해도 출결 데이터는 NexClass DB에 이미 커밋되어 있다. LMS가 나중에 수동으로 조회하면 가져갈 수 있다.

SyncLog: 동기화 이력 기록¶

// 시작 기록
private SyncLog saveSyncLog(String type, String status) {
    SyncLog syncLog = SyncLog.builder()
            .syncType(type)
            .syncStatus(status)          // "PROCESSING"
            .startedAt(LocalDateTime.now())
            .build();
    return syncLogRepository.save(syncLog);
}

// 성공 기록
private void completeSyncLog(SyncLog syncLog, int count) {
    syncLog.setSyncStatus("COMPLETED");
    syncLog.setRecordCount(count);       // 처리 건수
    syncLog.setCompletedAt(LocalDateTime.now());
    syncLogRepository.save(syncLog);
}

// 실패 기록
private void failSyncLog(SyncLog syncLog, String errorMessage) {
    syncLog.setSyncStatus("FAILED");
    syncLog.setErrorMessage(errorMessage); // 에러 메시지
    syncLog.setCompletedAt(LocalDateTime.now());
    syncLogRepository.save(syncLog);
}

주의사항¶

동기화 시 주의할 것들

1. 동기화 순서 변경 금지

syncAll() 안의 호출 순서를 바꾸면 참조 무결성 위반이 발생한다. 반드시 Term → Course → User → CourseUser 순서.

2. AES 키 불일치

NexClass의 aesKey와 LMS의 암호화 키가 다르면 복호화 실패. "JSON 파싱 실패"로 보이지만 실제 원인은 키 불일치다.

3. JWT 만료

토큰 유효 시간이 1시간. 동기화가 1시간 넘게 걸리면 후반부 API 호출에서 401 에러가 난다. 현실적으로 전체 동기화는 수 분이면 끝나니까 괜찮다.

4. CourseUser 중복 체크

findByUserCdAndCourseCd()를 안 하고 그냥 save() 하면 같은 수강생이 여러 행으로 들어간다. 실제로 발생했던 버그다.

정리¶

이 챕터에서 기억해야 할 것

  1. 동기화 순서: Term → Course → User → CourseUser (참조 무결성)
  2. 인증: JWT 토큰 (apiKey + apiSecret)
  3. 보안: AES-256 암호화된 응답을 복호화
  4. Upsert: PK 기반 자동(Term/Course/User) vs 수동(CourseUser)
  5. 페이징: User/CourseUser는 대량이라 페이징 필수
  6. SyncLog: 모든 동기화 이력 기록 (모니터링/디버깅/감사)
  7. 개별 API: 필요한 것만 빠르게 동기화 가능

확인문제¶

1. 동기화 순서가 Term → Course → User → CourseUser인 이유는?

참조 무결성(Referential Integrity) 때문이다.

  • Course에는 termCd가 있다 → Term이 먼저 있어야 함
  • CourseUser에는 courseCd와 userCd가 있다 → Course와 User가 먼저 있어야 함

순서를 바꾸면 참조할 데이터가 없어서 FK 제약 위반 또는 데이터 불일치가 발생한다.

2. LMS API 응답이 AES-256으로 암호화되어 있는 이유는?

이중 보안이다.

  1. HTTPS: 전송 구간 암호화 (네트워크 도청 방지)
  2. AES-256: 응답 본문 암호화 (서버 측 데이터 보호)

학사 데이터에는 학생 이름, 학번, 이메일 등 민감한 개인정보가 포함되어 있다. HTTPS가 뚫리더라도 AES-256 암호화가 한 번 더 보호한다.

3. CourseUser 동기화에서 findByUserCdAndCourseCd()를 쓰는 이유는? Term이나 Course처럼 그냥 save()하면 안 되나?

Term, Course, User의 PK는 의미 있는 값 (termCd, courseCd, userCd). 같은 PK로 save() 하면 JPA가 기존 행을 UPDATE한다.

CourseUser의 PK는 Auto Increment (id). 같은 userCd + courseCd 조합이어도 save() 하면 새 id로 INSERT된다. 중복 데이터가 생기는 거야.

그래서 findByUserCdAndCourseCd()로 먼저 확인하고, 있으면 기존 엔티티를 UPDATE, 없으면 새로 INSERT하는 수동 Upsert를 한다.

4. Push 실패하면 출결 데이터가 유실되나? 이유를 설명해봐

유실되지 않는다.

Push는 LMS에 "출결 처리 끝났으니 가져가라"는 알림이다. 출결 데이터 자체는 saveAttendance()에서 이미 NexClass DB에 커밋되어 있다.

Push가 실패해도 LMS가 나중에 수동으로 GET /v2/api/lessons/{lessonCd}/attends를 호출하면 데이터를 가져갈 수 있다.

이게 트랜잭션 분리의 효과다. 데이터 저장과 알림을 분리했기 때문에 알림 실패가 데이터 유실로 이어지지 않는다.

5. 전체 동기화(syncAll) 대신 개별 동기화 API를 따로 만든 이유는?

전체 동기화는 5만 건 이상이라 시간이 오래 걸린다.

"학기 정보만 바뀌었는데 5만 건 사용자를 다시 동기화할 필요 없다."

개별 API를 제공하면: 1. 필요한 것만 빠르게 동기화 가능 2. 대시보드에서 관리자가 선택적으로 실행 가능 3. 특정 동기화가 실패했을 때 해당 것만 재실행 가능

이건 책임 분리 원칙이기도 하다. 각 동기화는 독립적으로 실행될 수 있어야 한다.
