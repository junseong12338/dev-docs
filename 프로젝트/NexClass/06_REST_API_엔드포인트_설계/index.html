06. REST API 엔드포인트 설계¶

이게 뭐야?¶

REST API는 HTTP 프로토콜 위에서 자원(Resource)을 주고받는 규칙이야.

  • 자원: 강의, 출석, 녹화, 호스트 -- 이런 것들
  • 행위: 조회, 생성, 수정, 삭제 -- HTTP 메서드로 표현
  • 표현: JSON 형식으로 데이터를 주고받음

핵심 한줄

URL은 "뭘" 나타내고, HTTP 메서드는 "어떻게" 할지를 나타낸다. URL에 동사를 넣지 마.

어떻게 돌아가?¶

NexClass의 API 설계 철학¶

NexClass는 일반적인 REST API가 아니야. VirtualCampus라는 기존 솔루션의 API를 100% 호환해야 해.

sequenceDiagram
    participant LMS as LMS (기존 시스템)
    participant NC as NexClass (우리)
    participant VC as VirtualCampus (기존 솔루션)

    Note over LMS, VC: Before: LMS → VirtualCampus
    LMS->>VC: POST /v2/api/lessons
    VC-->>LMS: { status: 200, message: "OK", datas: {...} }

    Note over LMS, NC: After: LMS → NexClass (드롭인 교체)
    LMS->>NC: POST /v2/api/lessons (같은 URL)
    NC-->>LMS: { status: 200, message: "OK", datas: {...} } (같은 응답)

제약 조건

LMS 코드를 수정할 수 없어. LMS가 이미 VirtualCampus API를 호출하는 코드가 박혀있거든. 그래서 NexClass는 URL 경로, 파라미터 이름, 응답 구조 전부 VirtualCampus와 똑같이 맞춰야 해.

전체 엔드포인트 목록¶

NexClass의 API는 크게 4개 컨트롤러로 나뉘어.

1. AuthController -- 인증 API¶

HTTP 경로 목적 요청 응답
POST /v2/api/auth/tokens JWT 토큰 발급 TokenRequest { status, message, datas: { accessToken } }

2. LessonController -- 강의 API (핵심)¶

HTTP 경로 목적 인증
GET /v2/api/lessons/hosts/statistics/available-count 가용 호스트 수 조회 JWT
POST /v2/api/lessons 강의 생성 JWT
GET /v2/api/lessons/{lessonCd} 강의 단건 조회 JWT
PATCH /v2/api/lessons/{lessonCd} 강의 수정 JWT
DELETE /v2/api/lessons/{lessonCd} 강의 삭제 JWT
GET /v2/api/lessons/{lessonCd}/host-url 호스트 입장 URL JWT
GET /v2/api/lessons/{lessonCd}/join-url 참가자 입장 URL JWT
GET /v2/api/lessons/{lessonCd}/attends 출석 기록 조회 JWT
GET /v2/api/lessons/{lessonCd}/records 녹화 목록 조회 JWT
PATCH /v2/api/lessons/{lessonCd}/records/{recordCd} 녹화 설정 수정 JWT

3. SyncController -- 동기화 API¶

HTTP 경로 목적 인증
POST /api/sync/all 전체 동기화 실행 -
POST /api/sync/terms 학기 동기화 -
POST /api/sync/courses 과목 동기화 -
POST /api/sync/users 사용자 동기화 -
POST /api/sync/course-users 수강생 동기화 -

4. WebhookController -- Zoom 웹훅 수신¶

HTTP 경로 목적 인증
POST /webhook/zoom Zoom 이벤트 수신 HMAC-SHA256 서명 검증

5. DashboardController -- 관리자 대시보드¶

HTTP 경로 목적 인증
GET /dashboard 메인 대시보드 세션
GET/POST /dashboard/login 로그인 -
GET /dashboard/logout 로그아웃 세션
GET /dashboard/lessons 강의 관리 세션
GET /dashboard/lessons/{lessonCd}/attendance 출결 현황 세션
GET /dashboard/recordings 녹화 관리 세션
GET /dashboard/hosts 호스트 풀 세션
GET /dashboard/sync 동기화 관리 세션

URL 설계 규칙¶

자원 중심 URL 설계¶

/v2/api/lessons                          → 강의 컬렉션
/v2/api/lessons/{lessonCd}               → 특정 강의 1건
/v2/api/lessons/{lessonCd}/attends       → 특정 강의의 출석 기록
/v2/api/lessons/{lessonCd}/records       → 특정 강의의 녹화 목록
/v2/api/lessons/{lessonCd}/records/{recordCd}  → 특정 녹화 1건
/v2/api/lessons/{lessonCd}/host-url      → 특정 강의의 호스트 URL
/v2/api/lessons/{lessonCd}/join-url      → 특정 강의의 참가자 URL

URL 계층 구조 읽는 법

/v2/api/lessons/{lessonCd}/attends를 해석하면:

  • /v2/api → API 버전 2
  • /lessons → 강의 자원
  • /{lessonCd} → 특정 강의 (변수)
  • /attends → 그 강의의 출석 기록 (하위 자원)

부모 → 자식 관계가 URL 경로에 그대로 드러나. 강의가 있어야 출석이 있으니까.

HTTP 메서드 선택 기준¶

왜 메서드를 구분하냐?¶

같은 URL /v2/api/lessons/{lessonCd}에 대해 HTTP 메서드로 행위를 구분하는 거야.

HTTP 메서드 행위 NexClass 예시 멱등성 안전성
GET 조회 강의 정보 가져오기 O O
POST 생성 새 강의 만들기 X X
PATCH 부분 수정 강의명만 바꾸기 X X
DELETE 삭제 강의 삭제하기 O X
PUT 전체 교체 (NexClass 미사용) O X

멱등성(Idempotent)이 뭔데?

같은 요청을 여러 번 보내도 결과가 같다는 뜻이야.

  • GET: 10번 조회해도 데이터 안 바뀜 → 멱등
  • DELETE: 1번 삭제든 10번 삭제든 결과는 "삭제됨" → 멱등
  • POST: 10번 보내면 10개 생김 → 비멱등

NexClass의 deleteLesson() 코드를 보면 이 원칙을 지키고 있어: 

public void deleteLesson(String lessonCd) {
    Optional<Lesson> optional = lessonRepository.findById(lessonCd);
    if (optional.isEmpty()) {
        log.info("이미 삭제된 강의 (멱등 처리): lessonCd={}", lessonCd);
        return;  // 이미 없어도 에러 안 냄 → 멱등!
    }
    // ...실제 삭제 로직
}

Controller 코드의 공통 패턴¶

buildVcResponse() -- 응답 표준화 헬퍼¶

LessonController 안에 반복되는 응답 구조를 헬퍼 메서드로 뽑아냈어.

// 범용 응답 생성
private Map<String, Object> buildVcResponse(Object datas) {
    Map<String, Object> result = new HashMap<>();
    result.put("status", 200);
    result.put("message", "OK");
    result.put("datas", datas);     // 뭐든 들어갈 수 있음
    return result;
}

// 강의 전용 응답 생성 (serviceSyncStatus 포함)
private Map<String, Object> buildVcLessonResponse(Lesson lesson) {
    Map<String, Object> datas = new HashMap<>();
    datas.put("lessonCd", lesson.getLessonCd());
    datas.put("lessonName", lesson.getLessonName());
    // ... 나머지 필드들 ...
    datas.put("serviceSyncStatus", "DONE");  // LMS가 이걸 확인함!

    return buildVcResponse(datas);
}

왜 buildVcLessonResponse()를 따로 만들었어?

강의 관련 API는 항상 serviceSyncStatus: "DONE"을 포함해야 해. LMS가 이 값을 보고 "NexClass에서 Zoom 미팅 생성이 완료됐구나"라고 판단하거든. 빼먹으면 LMS가 동기화 실패로 처리해.

SecurityContext에서 사용자 정보 추출¶

// join-url API에서 JWT 토큰의 subject(userCd)를 꺼내는 코드
@GetMapping("/{lessonCd}/join-url")
public Map<String, Object> getJoinUrl(@PathVariable String lessonCd) {
    // JwtAuthenticationFilter가 이미 토큰을 검증하고
    // SecurityContext에 userCd를 넣어놨어
    String userCd = SecurityContextHolder.getContext()
            .getAuthentication().getName();

    String joinUrl = lessonService.getJoinUrl(lessonCd, userCd);
    return buildVcResponse(joinUrl);
}

함정¶

함정 1: @RequestMapping 경로 중첩

@RequestMapping("/v2/api/lessons")  // 클래스 레벨
public class LessonController {

    @GetMapping("/{lessonCd}")      // 메서드 레벨
    // 최종 경로: /v2/api/lessons/{lessonCd}
}
클래스 레벨의 @RequestMapping과 메서드 레벨의 @GetMapping이 합쳐져서 최종 경로가 된다. 이거 모르면 "왜 404 나지?" 하게 돼.

함정 2: @RequestParam 필수 vs 선택

// 기본값: required = true → 안 보내면 400 에러!
@RequestParam String startDate

// required = false → 안 보내도 null로 들어옴
@RequestParam(required = false) String excludedLessonCds

// defaultValue → 안 보내면 기본값 사용
@RequestParam(defaultValue = "false") boolean forceSync

함정 3: POST에 @RequestBody 빼먹으면 null

// ❌ @RequestBody 없으면 lesson이 빈 객체로 들어와
@PostMapping
public Map<String, Object> createLesson(Lesson lesson) { ... }

// ✅ @RequestBody 있어야 JSON → 객체 변환
@PostMapping
public Map<String, Object> createLesson(@RequestBody Lesson lesson) { ... }

함정 4: Map 반환 -- 타입 안전성 없음

NexClass는 VirtualCampus 호환을 위해 Map<String, Object>를 반환하고 있어. 이건 VirtualCampus 스펙을 맞추기 위한 타협이야.

실제 프로덕션에서는 ApiResponse<T> 같은 타입이 있는 제네릭 DTO를 쓰는 게 맞아. 키 이름 오타 못 잡고, 타입 변환 에러를 컴파일 타임에 못 잡거든.

핵심 요약표¶

개념 설명
REST URL = 자원(명사), HTTP 메서드 = 행위(동사)

한줄 정리¶

URL은 자원을 가리키고, HTTP 메서드가 행위를 결정한다. NexClass는 VirtualCampus 호환을 위해 경로와 응답 구조를 100% 맞췄다.

Q2. NexClass에서 강의 수정에 PUT이 아니라 PATCH를 쓰는 이유는?

PUT은 전체 교체, PATCH는 부분 수정이야.

강의 수정할 때 이름만 바꾸고 싶은데 PUT을 쓰면 나머지 모든 필드도 보내야 해. 안 보내면 null로 덮어써지거든.

PATCH를 쓰면 바꾸고 싶은 필드만 보내면 돼: 

{ "lessonName": "수정된 이름" }
나머지 필드는 기존 값이 유지돼.

이건 08장(PATCH 메서드와 부분수정)에서 더 자세히 다뤄.

Q3. @RestController와 @Controller의 차이를 NexClass 코드로 설명해봐.
  • @RestController = @Controller + @ResponseBody
  • 반환값이 자동으로 JSON으로 직렬화돼

NexClass에서: - LessonController(@RestController): Map<String, Object> 반환 → JSON으로 변환 - DashboardController(@Controller): "dashboard/main" 반환 → Thymeleaf 템플릿 이름으로 해석

같은 /dashboard/lessons 경로를 @RestController에서 반환하면 문자열 "dashboard/lessons"가 JSON으로 나가버려.

Q4. NexClass의 API 경로가 /v2/api/lessons로 시작하는 이유는?

VirtualCampus 호환 때문이야.

LMS 코드에 이미 POST /v2/api/lessons 같은 URL이 하드코딩되어 있어. LMS 코드를 수정할 수 없으니까, NexClass가 VirtualCampus와 동일한 경로를 쓰는 수밖에 없어.

/v2는 VirtualCampus의 API 버전을 의미해. NexClass가 만든 경로가 아니라 기존 솔루션의 경로를 그대로 가져온 거야. 이걸 드롭인 교체라고 해 -- API URL만 바꾸면 LMS 입장에서는 뭐가 바뀌었는지 모르게 만드는 거지.

Q5. deleteLesson()에서 '이미 삭제된 강의'를 에러 없이 처리하는 이유는?

멱등성(Idempotency) 때문이야.

Optional<Lesson> optional = lessonRepository.findById(lessonCd);
if (optional.isEmpty()) {
    log.info("이미 삭제된 강의 (멱등 처리): lessonCd={}", lessonCd);
    return;  // 에러 안 냄
}

DELETE 메서드는 HTTP 스펙상 멱등해야 해. "삭제해줘" 요청을 2번 보내도 결과는 "삭제된 상태"로 같아야 한다는 거지.

만약 두 번째 요청에서 404를 던지면? LMS가 "삭제 실패!"라고 판단해서 재시도 루프에 빠질 수 있어. 네트워크 불안정으로 같은 요청이 중복으로 올 수 있거든.
