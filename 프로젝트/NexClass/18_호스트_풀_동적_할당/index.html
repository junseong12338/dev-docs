18. 호스트 풀 동적 할당¶

난이도: Delta | 선수 지식: JPA 엔티티, Repository 패턴, @Transactional

이 챕터에서 배우는 것¶

학습 목표

  • 호스트 풀이 왜 필요한지, 어떤 문제를 해결하는지 이해한다
  • HostPool + HostAssignment 테이블 분리 설계의 이유를 안다
  • assignHost() 코드를 한 줄씩 분석할 수 있다
  • 시간대 기반 가용성 체크의 SQL 쿼리를 이해한다
  • 호스트 부족 시 처리 전략을 설명할 수 있다

1. 호스트 풀이 뭐야?¶

Zoom 호스트 계정 = 유한한 자원¶

핵심 비유

호스트 풀은 렌터카 회사야. 차(Zoom 계정)는 21대밖에 없어. 손님(강의)이 오면 빈 차를 배정하고, 다 쓰면 반납받아. 같은 시간에 같은 차를 두 손님에게 줄 수는 없잖아.

NexClass가 연동하는 Zoom 계정 구조:

구분 범위 비고
Zoom 라이선스 총 21개 kren@~kren19@knu10.or.kr + 소유자
소유자 계정 kren@knu10.or.kr 풀에서 제외 (관리용)
호스트 풀 20개 (kren1@~kren19@) 강의에 배정 가능한 계정

Zoom에서 미팅을 만들려면 호스트 계정이 필요해. 교수 1명이 강의 1개를 열면 호스트 계정 1개를 차지하는 거야.

왜 호스트 풀이 필요해?¶

호스트 풀 없이 호스트를 어떻게 배정할까?

// 강의마다 호스트를 하드코딩?
String host = "kren1@knu10.or.kr";  // 다른 강의도 kren1을 쓰면?
같은 시간에 2개 강의가 kren1을 쓰면 충돌. 두 번째 강의의 미팅 생성이 실패해.

// 1번부터 순서대로?
int nextHost = lastUsedHost + 1;
월요일 1교시 강의가 kren1, 2교시 강의가 kren2... 근데 1교시 강의가 끝나면 kren1은 쓸 수 있잖아. 시간대를 고려하지 않으면 호스트가 금방 바닥나.

// 해당 시간대에 비어 있는 호스트를 찾아서 배정
// 월요일 1교시(9~10시): kren1 사용 중
// 월요일 2교시(10~11시): kren1 반납됨 → 재사용 가능!
같은 호스트가 시간대가 겹치지 않으면 여러 강의에 배정될 수 있어. 이게 핵심이야.

2. 테이블 설계 -- 왜 2개로 분리했나?¶

HostPool (고정) + HostAssignment (동적)¶

erDiagram
    TB_NEXCLASS_HOST_POOL ||--o{ TB_NEXCLASS_HOST_ASSIGNMENT : "1:N"

    TB_NEXCLASS_HOST_POOL {
        varchar HOST_EMAIL PK "kren1@knu10.or.kr"
        varchar HOST_NAME "호스트 1"
        varchar HOST_STATUS "ACTIVE / INACTIVE"
        datetime CREATED_AT ""
        datetime UPDATED_AT ""
    }

    TB_NEXCLASS_HOST_ASSIGNMENT {
        bigint ASSIGNMENT_ID PK "AUTO_INCREMENT"
        varchar HOST_EMAIL FK "어떤 호스트를"
        varchar LESSON_CD "어떤 강의에"
        datetime START_TIME "시작 시각"
        datetime END_TIME "종료 시각"
        datetime CREATED_AT ""
    }

왜 2개로 분리했을까?

설계 장점 단점
통합 (1개 테이블) 단순 호스트 상태와 배정 이력이 뒤섞임
분리 (2개 테이블) 역할 명확, 이력 관리 용이 조인 필요

SRP (Single Responsibility Principle) 적용

  • HostPool: "어떤 호스트가 존재하는가?" (마스터 데이터, 거의 안 바뀜)
  • HostAssignment: "어떤 호스트가 어떤 강의에 언제 배정되었는가?" (트랜잭션 데이터, 자주 바뀜)

변경 이유가 다르니까 분리한 거야. 호스트 추가/삭제는 HostPool만, 배정/반환은 HostAssignment만 건드려.

HostPool 엔티티¶

@Entity
@Table(name = "TB_NEXCLASS_HOST_POOL")
public class HostPool {

    @Id
    @Column(name = "HOST_EMAIL", length = 100)
    private String hostEmail;       // PK = 이메일 (kren1@knu10.or.kr)

    @Column(name = "HOST_NAME", length = 200)
    private String hostName;        // 호스트 이름 (관리용)

    @Column(name = "HOST_STATUS", length = 20)
    private String hostStatus;      // ACTIVE / INACTIVE

    @Column(name = "CREATED_AT", updatable = false)
    private LocalDateTime createdAt;

    @Column(name = "UPDATED_AT")
    private LocalDateTime updatedAt;

    @PrePersist
    protected void onCreate() {
        createdAt = LocalDateTime.now();
        updatedAt = LocalDateTime.now();
        if (hostStatus == null) {
            hostStatus = "ACTIVE";  // 기본값: 활성
        }
    }

    @PreUpdate
    protected void onUpdate() {
        updatedAt = LocalDateTime.now();
    }
}

설계 포인트

  • PK가 이메일: 자연키(Natural Key) 사용. Zoom 계정 자체가 고유하니까 별도 ID 불필요.
  • hostStatus: ACTIVE만 배정 대상. 특정 계정에 문제 생기면 INACTIVE로 바꿔서 풀에서 제외.
  • @PrePersist/@PreUpdate: JPA 라이프사이클 콜백. 수동으로 시간 안 넣어도 됨.

HostAssignment 엔티티¶

@Entity
@Table(name = "TB_NEXCLASS_HOST_ASSIGNMENT")
public class HostAssignment {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    @Column(name = "ASSIGNMENT_ID")
    private Long assignmentId;       // 대리키(Surrogate Key), AUTO_INCREMENT

    @Column(name = "HOST_EMAIL", length = 100)
    private String hostEmail;        // 어떤 호스트를 (FK 역할)

    @Column(name = "LESSON_CD", length = 50)
    private String lessonCd;         // 어떤 강의에

    @Column(name = "START_TIME")
    private LocalDateTime startTime; // 시작 시각

    @Column(name = "END_TIME")
    private LocalDateTime endTime;   // 종료 시각

    @Column(name = "CREATED_AT", updatable = false)
    private LocalDateTime createdAt;
}

설계 포인트

  • 대리키 assignmentId: 배정은 여러 건 생길 수 있으니까 AUTO_INCREMENT.
  • startTime/endTime: 시간대 겹침 체크의 핵심 컬럼.
  • @ManyToOne 안 쓴 이유: JPA 연관관계 매핑을 안 했어. 왜? HostPool과 HostAssignment의 관계가 단순하고, 조인보다 별도 쿼리가 더 효율적인 경우에는 굳이 연관관계를 걸 필요 없어.

3. HostPoolService.assignHost() 코드 분석¶

이 메서드가 호스트 풀의 핵심이야. 한 줄씩 뜯어보자.

@Transactional
public String assignHost(LocalDateTime startTime, LocalDateTime endTime, String lessonCd) {
    log.info("호스트 배정 요청: lessonCd={}, {} ~ {}", lessonCd, startTime, endTime);

    // 1단계: ACTIVE 호스트 전체 조회
    List<HostPool> activeHosts = hostPoolRepository.findByHostStatus("ACTIVE");

    // 2단계: 해당 시간대에 이미 배정된 호스트 이메일 조회
    List<String> assignedEmails = hostAssignmentRepository
            .findAssignedHostEmails(startTime, endTime);

    // 3단계: 가용 호스트 = 전체 - 배정된 호스트
    List<HostPool> availableHosts = activeHosts.stream()
            .filter(host -> !assignedEmails.contains(host.getHostEmail()))
            .toList();

    // 4단계: 가용 호스트 없으면 에러
    if (availableHosts.isEmpty()) {
        log.error("가용 호스트 없음: {} ~ {} (전체: {}, 배정: {})",
                startTime, endTime, activeHosts.size(), assignedEmails.size());
        throw new RuntimeException("가용 호스트가 없습니다.");
    }

    // 5단계: 첫 번째 호스트 선택
    HostPool selectedHost = availableHosts.get(0);

    // 6단계: 배정 기록 저장
    HostAssignment assignment = HostAssignment.builder()
            .hostEmail(selectedHost.getHostEmail())
            .lessonCd(lessonCd)
            .startTime(startTime)
            .endTime(endTime)
            .build();
    hostAssignmentRepository.save(assignment);

    log.info("호스트 배정 완료: {} -> lessonCd={}", selectedHost.getHostEmail(), lessonCd);

    // 7단계: 배정된 호스트 이메일 반환
    return selectedHost.getHostEmail();
}

// 예: 월요일 9~10시에 배정된 호스트가 kren1, kren3이면
// 결과: ["kren1@knu10.or.kr", "kren3@knu10.or.kr"]

이 쿼리가 핵심이야. 다음 섹션에서 자세히 다룬다.

        .toList();
// 전체 20개 - 배정 2개 = 가용 18개

가용 호스트 중 첫 번째를 선택해. 이건 설계 결정이야.

다른 선택 전략은?

전략 설명 장점 단점
첫 번째 (현재) 항상 리스트 첫 번째 단순, 예측 가능 특정 호스트에 편중 가능
랜덤 랜덤 선택 부하 분산 예측 불가
최소 사용 가장 적게 쓴 호스트 균등 분산 쿼리 추가 필요
라운드 로빈 순서대로 돌아가며 균등, 예측 가능 상태 관리 필요

NexClass 규모(20개 호스트, 하루 수십 강의)에서는 첫 번째 선택으로 충분해. 호스트 수천 개에 강의 수만 개면 다른 전략이 필요하겠지.

4. 시간대 기반 가용성 체크¶

시간 겹침 쿼리 -- 핵심 SQL¶

// HostAssignmentRepository.java
@Query("SELECT a.hostEmail FROM HostAssignment a " +
       "WHERE a.startTime < :endTime AND a.endTime > :startTime")
List<String> findAssignedHostEmails(
        @Param("startTime") LocalDateTime startTime,
        @Param("endTime") LocalDateTime endTime);

이 쿼리의 조건: 기존 시작 < 새 종료 AND 기존 종료 > 새 시작

왜 이 조건이 겹침을 의미할까?

두 시간 구간이 겹치는 모든 경우를 생각해보자:

gantt
    title 시간 겹침 판단
    dateFormat HH:mm

    기존 (9:30~10:30) :crit, c2, 09:30, 10:30

    기존 (8:00~11:00) :crit, c4, 08:00, 11:00

    section "Case 5: 안 겹침"
    기존 (7:00~9:00) :done, c5, 07:00, 09:00

    기존 (10:00~11:00) :done, c6, 10:00, 11:00
Case 기존 새 (9~10) 조건 충족? 겹침?
1 8:30~9:30 8:30 < 10:00 AND 9:30 > 9:00 O O
2 9:30~10:30 9:30 < 10:00 AND 10:30 > 9:00 O O
3 9:15~9:45 9:15 < 10:00 AND 9:45 > 9:00 O O
4 8:00~11:00 8:00 < 10:00 AND 11:00 > 9:00 O O
5 7:00~9:00 7:00 < 10:00 AND 9:00 > 9:00 X (9=9) X
6 10:00~11:00 10:00 < 10:00 AND 11:00 > 9:00 X (10=10) X

시간 겹침 판단 공식

두 구간 [A_start, A_end]와 [B_start, B_end]가 겹치는 조건: A_start < B_end AND A_end > B_start

이건 외우지 말고 이해해. "겹치지 않는 경우"의 여집합이야. 겹치지 않으려면: A가 B 전에 끝나거나 (A_end <= B_start) A가 B 후에 시작하거나 (A_start >= B_end). 이걸 뒤집으면 위 조건이 나와.

5. 호스트 해제 (releaseHost)¶

강의를 삭제하면 호스트를 반납해야 해:

@Transactional
public void releaseHost(String lessonCd) {
    log.info("호스트 반환 요청: lessonCd={}", lessonCd);
    hostAssignmentRepository.deleteByLessonCd(lessonCd);
    log.info("호스트 반환 완료: lessonCd={}", lessonCd);
}

단순해. HostAssignment 테이블에서 해당 강의 코드의 배정 기록을 삭제하면 끝이야.

배정 기록이 사라지면 → 해당 시간대의 findAssignedHostEmails() 결과에서 빠지고 → 다른 강의에 배정 가능해져.

sequenceDiagram
    participant LC as LessonController
    participant HS as HostPoolService
    participant DB as HostAssignment 테이블

    LC->>HS: releaseHost("LESSON_001")
    HS->>DB: DELETE FROM TB_NEXCLASS_HOST_ASSIGNMENT<br/>WHERE LESSON_CD = 'LESSON_001'

HostPool 테이블은 안 건드려

호스트 반환 시 HostPool의 hostStatus를 바꾸지 않아. HostPool은 호스트 목록 (마스터 데이터)이고, HostAssignment가 배정 현황 (트랜잭션 데이터)이니까. 배정 기록만 지우면 자동으로 가용 상태가 돼.

6. 소유자 계정 제외¶

Zoom 라이선스 21개:
  kren@knu10.or.kr   ← 소유자 (Zoom 관리자)
  kren1@knu10.or.kr  ← 풀 대상
  kren2@knu10.or.kr  ← 풀 대상
  ...
  kren19@knu10.or.kr ← 풀 대상

소유자 계정(kren@)을 풀에서 제외하는 이유:

  1. 관리 안정성: 소유자 계정으로 Zoom 설정 변경, 라이선스 관리 등을 하는데, 이 계정이 미팅에 사용 중이면 관리 작업에 지장
  2. 기능 제한: 소유자 계정이 미팅 호스트일 때 특정 관리 기능이 제한될 수 있어
  3. 리스크 분산: 소유자 계정에 문제 생기면 전체 Zoom 환경에 영향, 미팅 호스트로 쓰지 않으면 리스크 격리

구현 방식: DB의 HostPool 테이블에 kren@를 아예 안 넣거나, hostStatus = 'INACTIVE'로 설정.

전략 설명 적합한 상황

가용 호스트 수 조회 API¶

관리자가 "지금 몇 개 남았어?" 확인할 수 있는 메서드:

public int getAvailableCount(LocalDateTime startTime, LocalDateTime endTime) {
    // 1. ACTIVE 호스트 전체 수
    List<HostPool> activeHosts = hostPoolRepository.findByHostStatus("ACTIVE");

    // 2. 해당 시간대에 이미 배정된 호스트 수
    List<String> assignedEmails = hostAssignmentRepository
            .findAssignedHostEmails(startTime, endTime);

    // 3. 가용 호스트 수 = 전체 - 배정
    int availableCount = activeHosts.size() - assignedEmails.size();

    log.info("가용 호스트 수: {} (전체: {}, 배정: {})",
            availableCount, activeHosts.size(), assignedEmails.size());

    return availableCount;
}

9. 주의사항¶

호스트 풀 운영 시 주의점

1. 동시성 문제 (Race Condition)

두 강의가 동시에 호스트를 배정받으려 하면? 같은 호스트를 중복 배정할 수 있어. 이 문제는 다음 챕터(19. 비관적 락과 동시성)에서 깊이 다룬다.

2. 호스트 풀 고갈 모니터링

20개 호스트 중 18개가 배정되면 위험 수준이야. 대시보드에서 가용 호스트 수를 모니터링하고, 임계값 이하면 알림을 보내야 해.

3. 시간대 데이터 정합성

HostAssignment의 startTime/endTime이 실제 강의 시간과 안 맞으면? 호스트가 이미 끝난 강의에 묶여있을 수 있어. 강의 시간 변경 시 HostAssignment도 같이 업데이트해야 한다.

4. 호스트 비활성화 시 기존 배정 처리

kren5를 INACTIVE로 바꿨는데, 이미 내일 강의에 배정되어 있으면? 기존 배정은 유지하되 새 배정은 안 되게 해야 해. 현재 구현은 이 케이스를 별도 처리하지 않아.

10. 정리¶

개념 한 줄 요약
호스트 풀 Zoom 계정이라는 유한 자원을 동적으로 관리하는 시스템
테이블 분리 HostPool(마스터) + HostAssignment(트랜잭션) = SRP 적용
시간대 겹침 A.start < B.end AND A.end > B.start 조건으로 판단
assignHost() 전체 조회 → 겹침 조회 → 필터링 → 첫 번째 선택 → 배정 저장
releaseHost() 배정 기록 삭제 → 자동으로 가용 복귀
소유자 제외 관리 안정성 + 리스크 분산

11. 확인문제¶

문제 1: 테이블 분리 이유¶

HostPool과 HostAssignment를 하나의 테이블로 합쳤다면 어떤 문제가 생기나? 최소 2가지.

  1. 상태와 이력의 혼재: "이 호스트가 활성(ACTIVE)인가?"와 "이 호스트가 언제 어떤 강의에 배정되었는가?"가 한 테이블에 섞인다. 한 호스트가 여러 강의에 배정되면 같은 호스트 이메일로 여러 행이 생기는데, hostStatus같은 마스터 정보가 중복 저장된다.

  2. 호스트 삭제/비활성화 어려움: 통합 테이블에서 호스트를 INACTIVE로 바꾸려면 해당 호스트의 모든 배정 행을 업데이트해야 한다. 분리되어 있으면 HostPool의 한 행만 바꾸면 됨.

  3. 쿼리 복잡도 증가: 가용 호스트를 찾을 때 "ACTIVE이면서 해당 시간대에 배정 안 된" 조건을 한 테이블 내 셀프 조인으로 처리해야 해서 쿼리가 복잡해진다.

문제 2: 시간 겹침 쿼리¶

월요일 9시~10시에 강의를 만들려 한다. HostAssignment에 다음 데이터가 있을 때 kren1은 가용한가? kren2는? 
| HOST_EMAIL | START_TIME     | END_TIME       |
|------------|----------------|----------------|
| kren1@     | 월요일 08:00   | 월요일 09:00   |
| kren2@     | 월요일 09:30   | 월요일 10:30   |

kren1@: 가용하다. 조건: 08:00 < 10:00 (TRUE) AND 09:00 > 09:00 (FALSE). 두 번째 조건이 FALSE라 AND 결과가 FALSE. 겹침 아님. (kren1의 종료 시각 9:00과 새 강의 시작 9:00이 정확히 같은 건 겹침이 아니야. > 이지 >=가 아니니까.)

kren2@: 가용하지 않다. 조건: 09:30 < 10:00 (TRUE) AND 10:30 > 09:00 (TRUE). 둘 다 TRUE라 AND 결과가 TRUE. 겹침이야.

문제 3: 호스트 선택 전략¶

availableHosts.get(0) 대신 랜덤 선택을 했다면, 어떤 장단점이 있을까?

장점:

  • 호스트별 사용 빈도가 더 균등해진다. get(0)은 DB 정렬 순서에 따라 특정 호스트(예: kren1)가 항상 먼저 선택되어 편중될 수 있어.
  • Zoom API Rate Limit이 호스트별로 적용된다면, 분산 효과가 있다.

단점:

  • 같은 입력에 대해 결과가 달라져서 디버깅이 어렵다. "왜 이 강의에 kren7이 배정됐지?" 추적이 힘들어.
  • 테스트가 비결정적(non-deterministic)이 됨. 테스트 시 시드(seed)를 고정해야 해.

NexClass 규모에서는 get(0)으로 충분해. 호스트 20개에 시간대별로 분산되니까 실제로 편중이 심하지 않아.

문제 4: 호스트 부족 시나리오¶

20개 호스트가 전부 월요일 1교시(9~10시)에 배정되어 있다. 21번째 강의가 같은 시간에 생성 요청되면 어떻게 되나? 이 상황을 개선할 방안을 제안해봐.

현재 구현에서는 RuntimeException("가용 호스트가 없습니다.")가 발생하고, 강의 생성이 실패한다.

개선 방안:

  1. 사전 가용성 체크: 강의 생성 API 호출 전에 getAvailableCount()로 가용 여부를 확인하는 API를 제공. LMS에서 미리 체크하고 사용자에게 "해당 시간대는 호스트가 부족합니다" 안내.

  2. 시간대 제안: "9~10시는 불가하지만 10~11시에는 3개 가용합니다" 같은 대안 시간대를 응답에 포함.

  3. 대기 큐: 요청을 큐에 넣고, 호스트가 반납되면 자동 배정. 단, 강의 시작 시간까지 배정이 안 되면 실패 처리.

  4. 에러 메시지 상세화: "가용 호스트 없음" 대신 "월요일 9~10시에 20/20 호스트 사용 중. 10~11시에는 5개 가용" 같이 상세 정보 제공.

문제 5: 설계 결정¶

HostPool의 PK를 이메일(자연키)로 쓴 이유와, AUTO_INCREMENT(대리키)로 바꾸면 생기는 변화를 설명해봐.

자연키(이메일) 사용 이유:

  • Zoom 계정 이메일은 이미 고유하고 절대 바뀌지 않아. 별도 ID를 만들 이유가 없어.
  • HostAssignment에서 HOST_EMAIL을 FK로 쓰면 조인 없이도 어떤 호스트인지 바로 알 수 있어. 가독성이 좋아.
  • 테이블에 데이터가 20건뿐이라 성능 차이가 없어.

대리키로 바꾸면:

  • HostPool에 HOST_ID BIGINT AUTO_INCREMENT PK 추가.
  • HostAssignment의 HOST_EMAIL이 HOST_ID로 바뀜.
  • 쿼리 결과에서 "kren5@"가 보이던 게 "5"로 바뀌어서 디버깅이 불편해짐.
  • 호스트 이메일이 바뀔 가능성은? Zoom 계정 이메일은 바뀌지 않아. 바뀔 일이 없는 데이터에 대리키를 쓰는 건 불필요한 추상화야.
  • 데이터가 수백만 건이면 INT PK가 VARCHAR PK보다 조인 성능이 좋지만, 20건 테이블에서는 의미 없어.
