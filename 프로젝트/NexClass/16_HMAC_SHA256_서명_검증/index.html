16. HMAC-SHA256 서명 검증¶

난이도: Gamma | 선수 지식: 15_Webhook_수신과_검증, 해시 함수 기초 개념

이 챕터에서 배우는 것¶

학습 목표

  • HMAC이 뭔지, 일반 해시와 뭐가 다른지 이해한다
  • SHA-256 해시 함수의 특성을 안다
  • Zoom의 서명 생성/검증 방식을 코드 레벨로 분석한다
  • 서명 검증이 왜 중요한지 -- 없으면 뭐가 터지는지 안다
  • 타임스탬프 검증과 리플레이 공격 방지를 이해한다

1. 해시 함수 기초¶

해시가 뭐야?¶

핵심 비유

해시 함수는 고기 분쇄기야. 어떤 크기의 고기를 넣어도 같은 크기의 다진 고기가 나와. 그리고 다진 고기로는 절대 원래 고기를 복원할 수 없어. 이게 핵심이야.

해시 함수의 3가지 특성:

특성 설명 비유
고정 길이 출력 입력 크기 상관없이 출력은 항상 같은 길이 1KB든 1GB든 결과는 256비트
일방향성 해시값에서 원본을 역추적 불가능 다진 고기 → 원래 고기 복원 불가
충돌 저항성 같은 해시값을 내는 다른 입력을 찾기 극도로 어려움 사실상 지문처럼 유일 
입력: "Hello"      → SHA-256 → 2cf24dba5fb0a30e26e83b2ac5b9e29e1b161e5c...
입력: "Hello!"     → SHA-256 → 334d016f755cd6dc58c53a86e183882f8ec14f52...
입력: "hello"      → SHA-256 → 2cf24dba5fb0a30e26e83b2ac5b9e29e1b161e5c... (다름!)

한 글자만 바뀌어도 해시값이 완전히 달라져. 이걸 눈사태 효과(Avalanche Effect) 라고 해.

SHA-256이란¶

SHA-256 = Secure Hash Algorithm, 256-bit

  • 256비트 = 64개의 16진수 문자
  • 출력 예시: a1b2c3d4e5f6... (64글자)
  • 현재 가장 널리 쓰이는 해시 알고리즘
  • 비트코인도 SHA-256을 쓴다

2. HMAC이 뭐야?¶

일반 해시의 한계¶

일반 해시로 메시지 무결성을 검증한다고 쳐보자:

메시지: "미팅 종료" → SHA-256 → "abc123..."
해시값을 같이 보냄: { message: "미팅 종료", hash: "abc123..." }

이거 안전해 보여? 전혀 아니야.

공격자가 이렇게 하면 끝이야:

가짜 메시지: "미팅 종료 (가짜)" → SHA-256 → "xyz789..."
조작된 요청: { message: "미팅 종료 (가짜)", hash: "xyz789..." }

해시 알고리즘은 공개되어 있어. 누구나 아무 데이터의 SHA-256을 만들 수 있어. 그래서 일반 해시만으로는 "이 메시지를 누가 만들었는지" 증명할 수 없어.

HMAC = Hash + Secret Key¶

HMAC (Hash-based Message Authentication Code) 은 해시에 비밀 키를 더한 거야.

HMAC-SHA256("미팅 종료", "비밀키abc") → "7f8a2b..."

비밀 키를 모르는 사람은 이 HMAC 값을 만들 수 없어. 비밀 키를 아는 사람만 올바른 HMAC을 생성할 수 있다.

flowchart LR
    subgraph "일반 해시 (SHA-256)"
        A1["데이터"] --> B1["SHA-256"] --> C1["해시값"]
        D1["누구나 만들 수 있음"] -.-> B1
    end

    subgraph "HMAC-SHA256"
        A2["데이터"] --> B2["HMAC-SHA256"]
        E2["비밀 키"] --> B2
        B2 --> C2["HMAC 값"]
        D2["비밀 키 아는 사람만<br/>만들 수 있음"] -.-> B2
    end

HMAC이 보장하는 2가지¶

보장 항목 설명
무결성 (Integrity) 데이터가 중간에 변조되지 않았음
인증 (Authentication) 비밀 키를 가진 사람이 만들었음

일반 해시는 무결성만 부분적으로 보장해. HMAC은 무결성 + 인증 둘 다 보장한다.

3. Zoom의 서명 방식¶

Zoom이 서명을 만드는 과정¶

Zoom이 Webhook을 보낼 때, 이런 과정으로 서명을 만들어:

flowchart TD
    A["v0:{timestamp}:{rawBody}"] --> B["HMAC-SHA256<br/>(비밀 키: {WEBHOOK_SECRET_TOKEN})"]
    B --> C["해시값: 7f8a2b3c..."]
    C --> D["서명: v0=7f8a2b3c..."]
    D --> E["x-zm-signature 헤더에 담아서 전송"]

1단계: 메시지 구성

message = "v0:" + timestamp + ":" + rawBody
  • v0: 서명 버전 (현재 v0)
  • timestamp: Zoom이 요청을 보낸 시각 (Unix timestamp)
  • rawBody: HTTP 요청 본문 전체 (JSON 문자열)

2단계: HMAC-SHA256 해싱

hash = HMAC-SHA256(message, secretToken)

3단계: 서명 포맷팅

signature = "v0=" + hash

4단계: HTTP 헤더에 담아서 전송

x-zm-signature: v0=7f8a2b3c4d5e6f...
x-zm-request-timestamp: 1711234567

NexClass가 검증하는 과정¶

NexClass는 같은 과정을 반복해서, 결과가 같은지 비교해:

sequenceDiagram
    participant Z as Zoom
    participant N as NexClass

    Note over Z: 서명 생성
    Z->>Z: message = "v0:" + timestamp + ":" + rawBody
    Z->>Z: signature = "v0=" + HMAC(message, secret)
    Z->>N: POST /webhook/zoom<br/>x-zm-signature: v0=abc...<br/>x-zm-request-timestamp: 17112...<br/>Body: {"event":"meeting.ended"...}

    Note over N: 서명 검증
    N->>N: message = "v0:" + timestamp + ":" + rawBody
    N->>N: expected = "v0=" + HMAC(message, secret)
    N->>N: expected == zoomSignature?
    alt 같으면
        N->>N: 검증 성공 → 이벤트 처리
    else 다르면
        N->>N: 검증 실패 → 401 Unauthorized
    end

핵심은 이거야: Zoom과 NexClass가 같은 비밀 키를 갖고 있으니까, 같은 입력으로 HMAC을 만들면 같은 결과가 나와야 해. 다르면 누군가 중간에 데이터를 조작한 거야.

4. NexClass 검증 코드 분석¶

verifySignature() 메서드¶

private boolean verifySignature(String rawBody, String timestamp, String zoomSignature) {
    // 방어 코드: 헤더가 없으면 검증 실패
    if (timestamp == null || zoomSignature == null) {
        return false;
    }

    // Zoom과 동일한 방식으로 메시지 구성
    String message = "v0:" + timestamp + ":" + rawBody;

    // 같은 비밀 키로 HMAC-SHA256 생성
    String expectedSignature = "v0=" + hmacSha256(message, secretToken);

    // Zoom이 보낸 서명과 우리가 만든 서명 비교
    return expectedSignature.equals(zoomSignature);
}

단계별로 뜯어보자:

if (timestamp == null || zoomSignature == null) {
    return false;
}

헤더가 아예 없는 요청은 Zoom이 보낸 게 아니야. 바로 실패 처리.

String message = "v0:" + timestamp + ":" + rawBody;

Zoom이 서명 만들 때 쓴 것과 완전히 동일한 형식이어야 해. v0:1711234567:{"event":"meeting.ended",...}

String expectedSignature = "v0=" + hmacSha256(message, secretToken);

같은 메시지 + 같은 비밀 키 = 같은 HMAC 값이 나와야 해.

return expectedSignature.equals(zoomSignature);

우리가 만든 서명과 Zoom이 보낸 서명이 같으면 검증 성공.

hmacSha256() 메서드 -- 핵심 암호화 로직¶

private String hmacSha256(String data, String key) {
    try {
        // (1) HMAC-SHA256 알고리즘 인스턴스 생성
        Mac mac = Mac.getInstance("HmacSHA256");

        // (2) 비밀 키를 바이트 배열로 변환
        SecretKeySpec secretKey = new SecretKeySpec(
                key.getBytes(StandardCharsets.UTF_8), "HmacSHA256");

        // (3) 키로 Mac 초기화
        mac.init(secretKey);

        // (4) 데이터를 해싱 → 바이트 배열 반환
        byte[] hash = mac.doFinal(data.getBytes(StandardCharsets.UTF_8));

        // (5) 바이트 배열 → 16진수 문자열 변환
        StringBuilder hexString = new StringBuilder();
        for (byte b : hash) {
            String hex = Integer.toHexString(0xff & b);
            if (hex.length() == 1) hexString.append('0');  // 한 자리면 앞에 0 붙임
            hexString.append(hex);
        }
        return hexString.toString();
    } catch (Exception e) {
        throw new RuntimeException("HMAC-SHA256 생성 실패: " + e.getMessage());
    }
}

바이트 → 16진수 변환이 왜 필요해?

mac.doFinal()은 바이트 배열을 반환해. 바이트값은 -128~127 범위인데, 이걸 사람이 읽을 수 있는 문자열로 바꿔야 비교할 수 있어. 바이트 0x0f → 문자열 "0f", 바이트 0xa3 → 문자열 "a3". 0xff & b는 부호 없는 값으로 변환하는 비트마스크야 (Java의 byte는 signed니까).

UTF-8 인코딩을 명시하는 이유¶

key.getBytes(StandardCharsets.UTF_8)
data.getBytes(StandardCharsets.UTF_8)

인코딩을 명시하지 않으면?

// ❌ 시스템 기본 인코딩 사용 (Windows: MS949, Linux: UTF-8)
key.getBytes()  // 운영체제에 따라 결과가 달라짐!

// ✅ UTF-8 명시 (어디서든 같은 결과)
key.getBytes(StandardCharsets.UTF_8)
개발 환경(Windows)과 서버(Linux)에서 기본 인코딩이 달라서, 같은 코드인데 서명 검증이 실패하는 경우가 실무에서 종종 있어. 반드시 인코딩을 명시해야 한다.

5. 서명 검증이 왜 중요한가?¶

서명 검증 없으면 어떻게 되는지¶

sequenceDiagram
    participant 공격자 as 공격자
    participant N as NexClass
    participant DB as DB

    rect rgb(255, 230, 230)
        Note over 공격자,DB: ❌ 서명 검증 없는 경우
        공격자->>N: POST /webhook/zoom<br/>{"event":"meeting.ended",<br/>"payload":{"object":{"id":"가짜미팅ID"}}}
        N->>N: 서명 검증 안 함
        N->>DB: 가짜 출결 데이터 저장!
        Note over DB: 데이터 오염
    end

서명 검증이 없으면, Webhook URL을 아는 사람이라면 누구나 가짜 이벤트를 보낼 수 있어:

  1. 가짜 출결: 학생이 수업 안 들어도 출석으로 처리
  2. 가짜 녹화: 존재하지 않는 녹화 URL을 DB에 주입
  3. 서비스 교란: 대량의 가짜 이벤트로 서버 부하 유발

서명 검증은 "이 요청이 진짜 Zoom이 보낸 거야" 라는 걸 증명하는 유일한 방법이야.

어떤 공격을 막아주나?¶

공격 유형 설명 HMAC 검증으로 방지
위조 (Forgery) 공격자가 가짜 Webhook 전송 비밀 키 없이 유효한 서명 생성 불가
변조 (Tampering) 전송 중 데이터 수정 rawBody 1바이트라도 바뀌면 서명 불일치
중간자 (MITM) 중간에서 가로채서 수정 HTTPS + HMAC 이중 방어

6. 타임스탬프 검증과 리플레이 공격¶

리플레이 공격이 뭐야?¶

리플레이 공격 시나리오

공격자가 Zoom의 정상적인 Webhook 요청을 도청했다고 쳐보자. 서명도 유효하고, rawBody도 원본 그대로야. 공격자가 이 요청을 나중에 다시 전송하면?

서명 검증만으로는 이걸 막을 수 없어. 서명이 유효하니까.

sequenceDiagram
    participant Z as Zoom
    participant 공격자 as 공격자
    participant N as NexClass

    Z->>N: 정상 Webhook (서명 유효)
    N-->>Z: 200 OK

    Note over 공격자: 이 요청을 도청해서 저장

    공격자->>N: 같은 요청 재전송 (5분 후)
    Note over N: 서명은 유효한데...<br/>타임스탬프가 5분 전이야!
    N->>N: 타임스탬프 검증 → 거부

타임스탬프 검증 원리¶

x-zm-request-timestamp 헤더에는 Zoom이 요청을 보낸 시각이 들어있어. 이 값이 현재 시각과 너무 차이나면 리플레이 공격으로 판단하고 거부하는 거야.

// 리플레이 공격 방지를 위한 타임스탬프 검증 (NexClass에는 아직 미구현)
// 구현한다면 이렇게 될 거야:
private boolean isTimestampValid(String timestamp) {
    long requestTime = Long.parseLong(timestamp);
    long currentTime = System.currentTimeMillis() / 1000;  // 초 단위
    long diff = Math.abs(currentTime - requestTime);

    // 5분(300초) 이상 차이나면 거부
    return diff <= 300;
}

NexClass의 현재 상태

현재 NexClass의 verifySignature()는 타임스탬프의 존재 여부만 확인하고, 시간 차이 검증은 하지 않아. 프로덕션 배포 시에는 타임스탬프 검증을 추가하는 게 좋아. 하지만 서버 시계가 정확하지 않으면 정상 요청도 거부될 수 있으니까, 허용 범위를 넉넉하게 (5~10분) 잡아야 해.

타임스탬프가 서명에 포함되는 이유¶

message = "v0:" + timestamp + ":" + rawBody

타임스탬프가 서명 메시지에 포함되어 있어. 이 말은:

  • 공격자가 타임스탬프를 현재 시각으로 바꾸면? → rawBody는 같아도 message가 바뀌니까 서명이 달라져 → 검증 실패
  • 공격자가 타임스탬프를 안 바꾸면? → 서명은 유효하지만 타임스탬프 검증에서 거부

둘 다 막혀. 이게 타임스탬프를 서명에 포함시키는 이유야.

7. CRC 검증과 HMAC 검증의 차이¶

이 두 가지는 목적이 완전히 다른 검증 메커니즘이야. 헷갈리면 안 돼.

구분 CRC 검증 HMAC 서명 검증
목적 URL 소유권 증명 메시지 위조/변조 방지
언제 Webhook URL 등록 시 (1회 + 주기적) 모든 이벤트 수신 시 (매번)
방향 Zoom → NexClass → Zoom (Challenge-Response) Zoom이 생성, NexClass가 검증 (단방향)
입력 plainToken (Zoom이 보낸 랜덤 토큰) v0:{timestamp}:{rawBody}
비밀 키 secretToken (동일) secretToken (동일)
결과 encryptedToken을 JSON으로 응답 헤더의 서명과 비교
실패 시 URL 등록 실패 401 Unauthorized

흐름 비교¶

Zoom: "plainToken 줄 테니까 암호화해서 돌려봐"
NexClass: "encryptedToken 여기"
Zoom: "맞네, 너 진짜 이 URL 주인이구나"

Zoom: "이벤트 데이터 보낸다. 서명은 헤더에 있어"
NexClass: "같은 방식으로 서명 만들어볼게... 맞네, 진짜 Zoom이 보낸 거구나"

왜 둘 다 필요해?¶

  • CRC만 있으면: URL 등록은 되지만, 이후 이벤트가 진짜 Zoom이 보낸 건지 검증할 수 없어
  • HMAC만 있으면: URL 등록 자체가 안 돼 (Zoom이 CRC를 요구하니까)
  • 둘 다 있으면: URL 소유권도 증명하고, 매 이벤트의 진위도 확인할 수 있어

8. 실전에서 자주 발생하는 문제¶

서명 검증 관련 실무 함정

1. JSON 직렬화 순서 문제

같은 JSON이라도 키 순서가 다르면 다른 문자열이야.

{"event":"meeting.ended","payload":{}}
{"payload":{},"event":"meeting.ended"}

이 둘은 의미는 같지만 HMAC은 완전히 다르다. 그래서 rawBody를 객체로 변환하지 않고 문자열 그대로 써야 해.

2. 프록시/게이트웨이가 Body를 수정하는 경우

nginx나 API Gateway가 요청 본문을 재인코딩하면 서명이 깨져. 프록시 설정에서 Body를 그대로 전달하도록 해야 해.

3. 인코딩 불일치

개발 환경(Windows, MS949)과 서버(Linux, UTF-8)에서 getBytes() 결과가 달라질 수 있어. 반드시 StandardCharsets.UTF_8을 명시해야 해.

9. 보안 강도 -- HMAC-SHA256은 얼마나 안전한가?¶

256비트 보안

SHA-256은 2^256 가지 가능한 출력을 가져. 이 숫자가 얼마나 큰지 감이 안 올 거야.

  • 2^256 ≈ 1.16 x 10^77
  • 관측 가능한 우주의 원자 수: ≈ 10^80
  • 모든 컴퓨터를 동원해서 무차별 대입해도 우주 수명보다 오래 걸려

비밀 키를 모르는 상태에서 유효한 HMAC을 만들어내는 건 사실상 불가능해.

10. 정리¶

개념 한 줄 요약
SHA-256 어떤 입력이든 256비트 고정 길이 해시값으로 변환, 역추적 불가
HMAC 해시 + 비밀 키 → 무결성과 인증을 동시에 보장
Zoom 서명 v0={timestamp}:{body} → HMAC-SHA256 → x-zm-signature 헤더
NexClass 검증 같은 방식으로 서명 생성 → Zoom이 보낸 서명과 비교
리플레이 공격 유효한 요청을 도청해서 재전송 → 타임스탬프 검증으로 방지
CRC vs HMAC CRC는 URL 소유권(1회), HMAC은 매 이벤트 인증(매번)

11. 확인문제¶

문제 1: 일반 해시 vs HMAC¶

SHA-256(data)와 HMAC-SHA256(data, key)의 결정적 차이는 뭐야? 왜 Webhook 검증에 일반 SHA-256을 쓰면 안 돼?

일반 SHA-256은 누구나 만들 수 있어. SHA-256 알고리즘은 공개되어 있으니까. 공격자가 가짜 Webhook 데이터를 만들고, 그 데이터의 SHA-256 해시를 같이 보내면 "해시가 일치하니까 유효하다"고 판단해버려.

HMAC-SHA256은 비밀 키가 있어야 만들 수 있어. 비밀 키를 모르는 공격자는 유효한 HMAC을 생성할 수 없다. 그래서 "이 HMAC이 맞으면 비밀 키를 아는 사람(= Zoom)이 만든 것"이라고 확신할 수 있어.

핵심: 일반 해시는 무결성만 (데이터가 깨지지 않았는지), HMAC은 무결성 + 인증 (누가 만들었는지)을 보장한다.

문제 2: 서명 메시지 형식¶

Zoom의 서명 메시지 형식이 v0:{timestamp}:{rawBody}야. 여기서 timestamp를 빼면 어떤 보안 문제가 생겨?

타임스탬프가 없으면 리플레이 공격에 취약해진다.

공격자가 정상적인 Webhook 요청(서명 포함)을 도청하면, 그 요청을 언제든 다시 보낼 수 있어. rawBody와 서명이 여전히 유효하니까.

타임스탬프가 서명에 포함되면:

  • 공격자가 타임스탬프를 현재 시각으로 바꾸면 → 메시지가 바뀌어서 서명 불일치
  • 타임스탬프를 안 바꾸면 → 서명은 유효하지만 시간 차이가 너무 커서 거부

둘 다 막힌다.

문제 3: rawBody 문자열¶

서명 검증에서 rawBody를 Map<String, Object>로 받은 뒤 다시 JSON 문자열로 변환하면 안 되는 이유를 구체적으로 설명해봐.

JSON 직렬화는 비결정적(non-deterministic) 이야. 같은 데이터라도 직렬화할 때마다:

  1. 키 순서가 달라질 수 있어: {"a":1,"b":2} vs {"b":2,"a":1}
  2. 공백/줄바꿈이 달라질 수 있어: {"a": 1} vs {"a":1}
  3. 숫자 표현이 달라질 수 있어: 1.0 vs 1
  4. 이스케이프가 달라질 수 있어: \/ vs /

HMAC은 1바이트라도 다르면 결과가 완전히 달라져 (눈사태 효과). Zoom이 {"event":"meeting.ended"} 이 문자열로 서명을 만들었는데, 우리가 Map → JSON 변환 시 {"event" : "meeting.ended"} (공백 추가)가 되면 서명이 불일치해.

문제 4: 보안 시나리오¶

공격자가 NexClass의 Webhook URL(POST /webhook/zoom)을 알고 있다. 서명 검증이 있으면 공격자가 할 수 있는 것과 없는 것을 구분해봐.

할 수 있는 것:

  • Webhook URL에 대량의 요청을 보내서 서버 부하 유발 (DoS). 단, 서명 검증 실패로 실제 처리는 안 됨. 서명 검증 자체의 CPU 비용은 미미하지만, 네트워크 대역폭은 소모될 수 있어.
  • 서명 검증 실패 로그를 대량 생성하여 디스크 공간 고갈 유발 가능.

할 수 없는 것:

  • 가짜 출결 데이터 주입: 비밀 키 없이 유효한 서명 생성 불가
  • 가짜 녹화 데이터 주입: 동일한 이유
  • 정상 요청의 데이터 변조: rawBody 1바이트라도 바꾸면 서명 불일치
  • 비밀 키 추출: HMAC 출력으로부터 키를 역산하는 건 계산적으로 불가능

문제 5: 코드 분석¶

hmacSha256() 메서드에서 0xff & b는 왜 필요해? 이걸 빼면 어떤 문제가 생기나?

Java의 byte는 signed 타입이야. 범위가 -128 ~ 127이지. 바이트 값이 0xA3 (163)이면 Java에서는 -93으로 해석돼.

Integer.toHexString(-93)은 "ffffff a3" (32비트 기준)을 반환해. 원하는 건 "a3"인데.

0xff & b는 비트마스크야. 하위 8비트만 남기고 나머지를 0으로 만들어서 unsigned로 변환한다.

byte b = (byte) 0xA3;     // Java에서는 -93
int unsigned = 0xff & b;  // 163
Integer.toHexString(163); // "a3" ← 올바른 결과

이걸 빼면 음수 바이트 값에 대해 해시 문자열이 완전히 달라져서 Zoom의 서명과 일치하지 않게 된다.
