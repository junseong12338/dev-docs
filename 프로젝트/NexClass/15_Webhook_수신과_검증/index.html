15. Webhook 수신과 검증¶

난이도: Gamma | 선수 지식: REST API, HTTP 프로토콜, Spring Boot Controller

이 챕터에서 배우는 것¶

학습 목표

  • Webhook이 뭔지, 왜 쓰는지 이해한다
  • Zoom이 보내는 3가지 Webhook 이벤트를 구분한다
  • WebhookController의 전체 흐름을 코드 레벨로 분석한다
  • CRC 검증(URL 소유권 증명)의 원리를 이해한다
  • "즉시 200 OK → 비동기 처리" 패턴이 왜 필수인지 안다

1. Webhook이 뭐야?¶

Push vs Pull -- 두 가지 데이터 수신 방식¶

핵심 비유

Pull (폴링): 네가 5분마다 우체통을 확인하러 가는 것. 편지 없어도 가야 해. Push (Webhook): 우체부가 편지 오면 초인종 눌러주는 것. 올 때만 대응하면 돼.

방식 설명 장점 단점
Pull (폴링) 주기적으로 "새 데이터 있어?" 물어봄 구현 단순 리소스 낭비, 실시간성 떨어짐
Push (Webhook) 이벤트 발생하면 상대가 알려줌 실시간, 효율적 수신 서버 필요, 보안 검증 필수

NexClass에서 Webhook을 쓰는 이유가 뭘까?

Zoom에서 미팅이 끝나면 그걸 즉시 알아야 출결 처리를 할 수 있어. 5분마다 "미팅 끝났어?" 하고 물어보는 건 말이 안 되잖아. 미팅이 수백 개일 수도 있고, 언제 끝날지도 모르는데.

그래서 Zoom한테 말하는 거야: "미팅 끝나면 이 URL로 알려줘."

Webhook의 동작 원리¶

sequenceDiagram
    participant Z as Zoom 서버
    participant N as NexClass<br/>(WebhookController)
    participant DB as NexClass DB

    Note over Z,N: 사전 설정: Zoom에 Webhook URL 등록
    Z->>N: POST /webhook/zoom (이벤트 데이터)
    N->>N: 서명 검증 (진짜 Zoom이야?)
    N-->>Z: 200 OK (즉시 응답)
    N->>N: 비동기 처리 시작
    N->>DB: 출결/녹화 데이터 저장

Webhook은 결국 HTTP POST 요청이야. Zoom이 우리 서버로 POST 요청을 보내는 거지. 그러니까 우리 입장에서는 API 엔드포인트를 하나 만들어서 대기하고 있으면 돼.

2. Zoom Webhook 이벤트 종류¶

NexClass가 처리하는 이벤트는 딱 3가지야:

이벤트 언제 발생 NexClass 처리
endpoint.url_validation Webhook URL 등록/검증 시 CRC 응답 반환 (URL 소유권 증명)
meeting.ended Zoom 미팅 종료 시 출결 처리 (Reports API → DB)
recording.completed 녹화 인코딩 완료 시 녹화 URL 저장 (share_url → DB)

중요한 포인트

recording.completed는 미팅 종료 직후가 아니야. Zoom이 녹화 파일을 인코딩하는 데 수 분에서 수십 분이 걸려. 그래서 meeting.ended와 recording.completed는 별개의 시점에 도착한다.

이벤트 흐름 타임라인¶

sequenceDiagram
    participant 교수 as 교수
    participant Z as Zoom
    participant N as NexClass

    교수->>Z: 미팅 종료 버튼 클릭
    Z->>N: meeting.ended (즉시)
    N->>N: 출결 처리 시작

    Note over Z: 녹화 인코딩 중... (수 분)

    Z->>N: recording.completed (수 분 후)
    N->>N: 녹화 URL 저장

3. WebhookController 코드 상세 분석¶

3.1 클래스 선언부¶

@Slf4j                              // 로그 자동 생성 (Lombok)
@RestController                     // REST API 컨트롤러 (JSON 응답)
@RequestMapping("/webhook/zoom")    // 기본 경로: /webhook/zoom
@RequiredArgsConstructor            // final 필드 생성자 주입 (Lombok)
public class WebhookController {

    // application.properties에서 주입
    @Value("${zoom.webhook.secret-token}")
    private String secretToken;     // Zoom Webhook 비밀 토큰: {WEBHOOK_SECRET_TOKEN}

    // 출결 + 녹화 처리를 위임할 서비스
    private final AttendanceService attendanceService;
}

@Value로 비밀 토큰 주입

secretToken은 코드에 직접 쓰면 안 돼. application.properties에서 가져온다. 왜? 코드에 비밀 값을 하드코딩하면 Git에 올라가니까. 설정 파일은 .gitignore 처리하거나 환경변수로 관리해야 해.

3.2 메인 핸들러 메서드 -- handleWebhook()¶

여기가 핵심이야. Zoom에서 오는 모든 이벤트가 이 하나의 메서드를 통과해:

@PostMapping
public ResponseEntity<?> handleWebhook(
        @RequestBody String rawBody,           // (1) 요청 본문 - 문자열 그대로
        @RequestHeader(value = "x-zm-request-timestamp", required = false)
            String timestamp,                  // (2) Zoom이 보낸 타임스탬프
        @RequestHeader(value = "x-zm-signature", required = false)
            String zoomSignature) {            // (3) Zoom이 만든 서명

왜 @RequestBody String rawBody를 쓸까?

보통은 @RequestBody WebhookDTO dto 이렇게 객체로 받잖아. 근데 여기선 문자열 그대로 받아.

이유가 있어:

  1. 서명 검증: HMAC-SHA256은 원본 문자열이 필요해. 객체로 변환하면 원본이 깨져
  2. 유연성: Zoom 이벤트마다 JSON 구조가 다르니까 하나의 DTO로 못 받아

절대 하면 안 되는 실수

// ❌ 이렇게 하면 서명 검증이 깨진다
@RequestBody Map<String, Object> body  // 객체로 변환하면 원본 문자열 사라짐

// ✅ 원본 문자열로 받아야 서명 검증 가능
@RequestBody String rawBody

3.3 처리 순서 -- 4단계 파이프라인¶

flowchart TD
    A["1. JSON 파싱"] --> B{"event 확인"}
    B -->|endpoint.url_validation| C["2. CRC 검증<br/>(서명 검증 건너뜀)"]
    B -->|그 외 이벤트| D["3. 서명 검증<br/>(HMAC-SHA256)"]
    D -->|실패| E["401 Unauthorized"]
    D -->|성공| F{"4. 이벤트 라우팅"}
    F -->|meeting.ended| G["출결 처리<br/>(@Async)"]
    F -->|recording.completed| H["녹화 처리<br/>(@Async)"]
    F -->|기타| I["무시 (로그만)"]
    C --> J["200 OK + JSON 응답"]
    G --> K["200 OK (즉시)"]
    H --> K
    I --> K

코드로 보면:

// 1단계: JSON 파싱
Map<String, Object> body;
try {
    ObjectMapper mapper = new ObjectMapper();
    body = mapper.readValue(rawBody, Map.class);  // 문자열 → Map 변환
} catch (Exception e) {
    log.error("[WEBHOOK] JSON 파싱 실패: {}", e.getMessage());
    return ResponseEntity.badRequest().build();   // 400 Bad Request
}

String event = (String) body.get("event");        // 이벤트 타입 추출

// 2단계: CRC 검증 (endpoint.url_validation만 해당)
// ★ 서명 검증보다 먼저 처리! CRC는 서명 없이 들어온다
if ("endpoint.url_validation".equals(event)) {
    return handleCrcValidation(body);
}

// 3단계: 서명 검증 (HMAC-SHA256)
if (!verifySignature(rawBody, timestamp, zoomSignature)) {
    log.warn("Webhook 서명 검증 실패");
    return ResponseEntity.status(HttpStatus.UNAUTHORIZED).build();  // 401
}

// 4단계: 이벤트 라우팅
Map<String, Object> payload = (Map<String, Object>) body.get("payload");
Map<String, Object> object = (Map<String, Object>) payload.get("object");

switch (event) {
    case "meeting.ended":
        handleMeetingEnded(object);         // 비동기 처리 (@Async)
        break;
    case "recording.completed":
        handleRecordingCompleted(object);   // 비동기 처리 (@Async)
        break;
    default:
        log.info("[WEBHOOK] 처리하지 않는 이벤트: {}", event);
}

return ResponseEntity.ok().build();  // 200 OK (즉시 반환)

순서가 중요해

CRC 검증을 서명 검증 앞에 두는 이유가 있어. endpoint.url_validation 이벤트는 Zoom이 URL 등록 시 보내는 건데, 이때는 서명 헤더가 없을 수 있어. 서명 검증을 먼저 하면 CRC 요청이 401로 거부당해.

4. CRC 검증 -- URL 소유권 증명¶

왜 CRC 검증이 필요해?¶

Zoom에 Webhook URL을 등록할 때, Zoom이 이렇게 물어봐:

"야, 네가 https://nexclass.knu10.cc/webhook/zoom 이 URL의 진짜 주인 맞아? 증명해봐."

이걸 CRC (Challenge-Response Check) 라고 해. Zoom이 랜덤 토큰(plainToken)을 보내면, 우리가 비밀 키로 암호화해서 돌려보내는 거야. 비밀 키를 아는 사람만 올바른 암호화 값을 만들 수 있으니까.

sequenceDiagram
    participant Z as Zoom
    participant N as NexClass

    Note over Z: Webhook URL 등록 시도
    Z->>N: POST /webhook/zoom<br/>event: "endpoint.url_validation"<br/>payload.plainToken: "abc123..."
    N->>N: HMAC-SHA256(plainToken, secretToken)<br/>→ encryptedToken 생성
    N-->>Z: 200 OK<br/>{ plainToken: "abc123...",<br/>  encryptedToken: "7f8a2b..." }
    Note over Z: 검증 성공 → URL 등록 완료

CRC 검증 코드 분석¶

private ResponseEntity<?> handleCrcValidation(Map<String, Object> body) {
    // Zoom이 보낸 payload에서 plainToken 추출
    Map<String, Object> payload = (Map<String, Object>) body.get("payload");
    String plainToken = (String) payload.get("plainToken");

    log.info("CRC 검증 요청: plainToken={}", plainToken);

    // 비밀 키로 HMAC-SHA256 해싱 → encryptedToken 생성
    String encryptedToken = hmacSha256(plainToken, secretToken);

    // Zoom이 기대하는 응답 형식으로 반환
    Map<String, Object> response = new HashMap<>();
    response.put("plainToken", plainToken);       // 원본 토큰 (그대로 돌려줌)
    response.put("encryptedToken", encryptedToken); // 암호화된 토큰

    return ResponseEntity.ok(response);
    // 응답 JSON: { "plainToken": "abc123...", "encryptedToken": "7f8a2b..." }
}

CRC vs HMAC 서명 검증의 차이

구분 CRC 검증 HMAC 서명 검증
목적 URL 소유권 증명 메시지 위조 방지
언제 Webhook URL 등록/재검증 시 모든 이벤트 수신 시
방향 Zoom → NexClass → Zoom (왕복) Zoom이 만들고 NexClass가 검증 (단방향)
빈도 1회 (등록 시) + 주기적 재검증 매 이벤트마다

5. 이벤트 라우팅 -- meeting.ended & recording.completed¶

meeting.ended 처리¶

private void handleMeetingEnded(Map<String, Object> object) {
    // Zoom Webhook payload에서 미팅 정보 추출
    String meetingId = String.valueOf(object.get("id"));   // 미팅 ID
    String uuid = (String) object.get("uuid");             // 미팅 고유 UUID
    String topic = (String) object.get("topic");           // 미팅 제목

    log.info("미팅 종료: meetingId={}, uuid={}, topic={}", meetingId, uuid, topic);

    try {
        // AttendanceService에 출결 처리 위임 (@Async 메서드)
        attendanceService.processAttendance(meetingId, uuid);
    } catch (Exception e) {
        log.error("출결 처리 실패: meetingId={}, error={}", meetingId, e.getMessage());
    }
}

recording.completed 처리¶

private void handleRecordingCompleted(Map<String, Object> object) {
    String meetingId = String.valueOf(object.get("id"));
    String topic = (String) object.get("topic");

    log.info("녹화 완료: meetingId={}, topic={}", meetingId, topic);

    try {
        // 녹화 처리 위임 - object 전체를 넘김 (recording_files 포함)
        attendanceService.processRecording(meetingId, object);
    } catch (Exception e) {
        log.error("녹화 처리 실패: meetingId={}, error={}", meetingId, e.getMessage());
    }
}

object 전체를 넘기는 이유

meeting.ended는 meetingId와 uuid만 있으면 Reports API로 데이터를 가져올 수 있어. 하지만 recording.completed는 Webhook payload 자체에 녹화 파일 목록(recording_files)이 들어 있어. 그래서 object 전체를 넘기는 거야.

6. 즉시 200 OK + 비동기 처리 패턴¶

이 패턴이 Webhook 처리의 생명선이야.

왜 즉시 200 OK를 보내야 해?¶

sequenceDiagram
    participant Z as Zoom
    participant N as NexClass
    participant API as Zoom Reports API
    participant DB as NexClass DB

    rect rgb(255, 230, 230)
        Note over Z,DB: ❌ 동기 처리 (잘못된 방식)
        Z->>N: meeting.ended
        N->>API: Reports API 호출 (3~5초)
        API-->>N: 참여자 데이터
        N->>DB: 출결 저장 (1초)
        N-->>Z: 200 OK (6초 후)
        Note over Z: 타임아웃! 재시도 시작
    end

    rect rgb(230, 255, 230)
        Note over Z,DB: ✅ 비동기 처리 (올바른 방식)
        Z->>N: meeting.ended
        N-->>Z: 200 OK (즉시, 100ms 이내)
        Note over N,DB: 백그라운드에서 처리
        N->>API: Reports API 호출
        API-->>N: 참여자 데이터
        N->>DB: 출결 저장
    end

Zoom은 Webhook 응답을 일정 시간 안에 받아야 해. 보통 3~10초. 그 안에 200 OK를 못 받으면 "전달 실패"로 판단하고 재시도해.

출결 처리는 Reports API 호출 → 데이터 가공 → DB 저장까지 수 초에서 수십 초가 걸려. 이걸 다 끝내고 200 OK를 보내면 타임아웃이 날 수 있어.

그래서 패턴은 이래:

  1. Webhook 수신
  2. 즉시 200 OK 반환 (Zoom한테 "잘 받았어" 신호)
  3. 비동기로 실제 처리 (@Async 메서드 호출)

코드에서 이 패턴이 어떻게 구현되어 있는지¶

// 4. 이벤트 라우팅 -- 이 시점에서 @Async 메서드 호출
switch (event) {
    case "meeting.ended":
        handleMeetingEnded(object);    // 내부에서 @Async 메서드 호출
        break;
    case "recording.completed":
        handleRecordingCompleted(object); // 내부에서 @Async 메서드 호출
        break;
}

// handleMeetingEnded/handleRecordingCompleted가 리턴되면 즉시 여기로 옴
// @Async 메서드는 별도 스레드에서 실행 중
return ResponseEntity.ok().build();  // ← 200 OK 즉시 반환

attendanceService.processAttendance()는 @Async 어노테이션이 붙어있어서 별도 스레드에서 실행돼. 그래서 handleMeetingEnded()는 processAttendance()가 끝나길 기다리지 않고 바로 리턴해.

7. Zoom Webhook 재시도 정책¶

Zoom의 재시도 메커니즘

Zoom은 Webhook 전송 실패 시 자동으로 재시도한다. 재시도 간격은 점점 늘어나고 (지수 백오프), 최종적으로 실패하면 Zoom Marketplace에서 알림이 뜬다.

이게 의미하는 건: 우리 서버가 일시적으로 다운돼도 데이터를 잃지 않는다는 거야. Zoom이 알아서 재시도해주니까.

단, 재시도에 대비한 멱등성(Idempotency) 처리가 필요해. 같은 이벤트가 2번 와도 출결이 2번 저장되면 안 되잖아.

NexClass의 녹화 처리에서 멱등성을 어떻게 보장하는지 봐:

// AttendanceService.processRecording() 내부
String recordCd = (String) file.get("id");
if (recordingRepository.findByRecordCd(recordCd).isPresent()) {
    log.info("[RECORD] 이미 저장된 녹화 스킵: recordCd={}", recordCd);
    continue;  // 중복이면 건너뛰기
}

recordCd로 이미 저장했는지 확인하고, 있으면 건너뛴다. Zoom이 3번 재시도해도 녹화 데이터는 1건만 저장돼.

9. 주의사항¶

실무에서 자주 하는 실수

1. 200 OK를 늦게 보내는 것

Webhook 처리 로직이 길어지면 200 OK가 늦어지고, Zoom이 재시도해서 중복 이벤트가 온다. 반드시 비동기 처리 패턴을 써야 해.

2. 서명 검증 안 하는 것

서명 검증 없이 Webhook을 받으면, 아무나 POST 요청 보내서 가짜 출결 데이터를 넣을 수 있어. https://nexclass.knu10.cc/webhook/zoom으로 가짜 meeting.ended 이벤트를 보내면? 서명 검증 없으면 그대로 처리돼버려.

3. rawBody를 객체로 먼저 변환하는 것

서명 검증에는 원본 문자열이 필요해. 객체로 변환하면 JSON 직렬화 순서, 공백 등이 바뀌어서 서명이 달라져.

4. CRC 검증에서 서명 체크하는 것

endpoint.url_validation은 서명 헤더 없이 올 수 있어. 서명 검증보다 앞에서 처리해야 한다.

10. 정리¶

개념 한 줄 요약

재시도 정책 Zoom이 실패 시 알아서 재시도 → 멱등성 처리 필수

11. 확인문제¶

문제 1: Webhook vs Polling¶

NexClass가 Zoom 이벤트를 폴링 방식으로 처리한다면 어떤 문제가 생기나? 최소 3가지.
  1. 리소스 낭비: 미팅이 안 끝났는데도 주기적으로 API를 호출해야 한다. Zoom API Rate Limit에 걸릴 수 있다.
  2. 실시간성 부족: 폴링 주기가 5분이면, 미팅 종료 후 최대 5분이 지나야 출결 처리가 시작된다. 교수/학생 입장에서 "왜 출결이 안 잡혀요?" 문의가 폭주한다.
  3. 확장성 문제: 강의가 1,000개면 1,000개를 주기적으로 체크해야 한다. Webhook은 이벤트가 발생한 것만 처리하면 되니까 O(이벤트 수)지만, 폴링은 O(전체 강의 수)야.
  4. 복잡도 증가: "이미 처리한 미팅인지" 상태 관리가 필요하다. Webhook은 이벤트 자체가 "새로운 일이 일어났다"는 의미니까 상태 관리가 단순해.

문제 2: CRC 검증 순서¶

endpoint.url_validation 처리를 HMAC 서명 검증 뒤로 옮기면 어떻게 되나?

CRC 검증 요청은 서명 헤더(x-zm-signature)가 없거나 다른 형식일 수 있다. 서명 검증을 먼저 하면 verifySignature() 메서드에서 timestamp == null || zoomSignature == null 체크에 걸려서 401 Unauthorized를 반환한다. 결과: Zoom은 "이 URL은 유효하지 않다"고 판단하고 Webhook URL 등록이 실패한다. 그래서 CRC 검증은 반드시 서명 검증 앞에 있어야 해.

문제 3: rawBody를 문자열로 받는 이유¶

@RequestBody String rawBody 대신 @RequestBody Map<String, Object> body로 받으면 어떤 문제가 생기나?

HMAC-SHA256 서명 검증은 원본 HTTP 요청 본문의 정확한 바이트열이 필요해. Map으로 역직렬화하면:

  1. JSON 키 순서가 바뀔 수 있다
  2. 공백, 줄바꿈이 달라질 수 있다
  3. 숫자 형식이 바뀔 수 있다 (1.0 → 1)

이 중 하나라도 달라지면 HMAC 해시값이 완전히 달라져서 서명 검증이 100% 실패한다. 그래서 서명 검증용으로는 반드시 String rawBody로 원본 그대로 받아야 해.

문제 4: 200 OK 지연¶

Zoom Webhook에 3초 안에 200 OK를 보내지 못하면 어떻게 되나? NexClass 입장에서 어떤 문제가 생길 수 있어?
  1. Zoom은 응답 타임아웃으로 판단하고 같은 이벤트를 재시도한다.
  2. 재시도된 이벤트를 또 처리하면 출결이 중복 저장될 수 있다.
  3. 반복적으로 타임아웃이 나면 Zoom이 해당 Webhook URL을 비활성화할 수 있다.
  4. 그래서 "즉시 200 OK → 비동기 처리" 패턴이 필수이고, 멱등성(같은 이벤트 2번 와도 결과가 같음) 처리도 해야 한다.

문제 5: 이벤트 라우팅¶

Zoom이 meeting.started 이벤트를 보냈다면 NexClass는 어떻게 처리하나? 코드를 보고 설명해봐.

switch 문의 default 분기에 걸린다: 

default:
    log.info("[WEBHOOK] 처리하지 않는 이벤트: {}", event);
로그만 남기고 아무 처리 없이 200 OK를 반환한다. 왜 에러를 반환하지 않을까? Zoom한테 "이 이벤트 모르겠어"라고 에러를 보내면 Zoom이 계속 재시도하니까. 처리할 필요 없는 이벤트도 200 OK로 응답해서 "잘 받았어, 재시도 안 해도 돼"라고 알려줘야 해.
