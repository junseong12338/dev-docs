09. JWT 인증 시스템¶

이게 뭐야?¶

API를 아무나 호출하면 안 되잖아. "너 누구야? 진짜 LMS에서 온 거 맞아?" 이걸 확인하는 게 인증(Authentication)이야.

JWT(JSON Web Token)는 JSON 형태의 토큰으로 인증하는 방식이야. 세션 없이, DB 조회 없이, 토큰 자체에 정보가 다 들어있어.

핵심 한줄

JWT = 디지털 출입증. 발급받은 토큰을 매 요청마다 보여주면 된다. 서버는 토큰의 서명만 검증하면 끝.

어떻게 돌아가?¶

JWT 구조: Header.Payload.Signature¶

JWT는 점(.)으로 구분된 3개 파트로 이루어져 있어.

eyJhbGciOiJIUzI1NiJ9.eyJzdWIiOiJTVFUwMDEiLCJpYXQiOjE3MTExNjAwMDAsImV4cCI6MTcxMTE2MDMwMH0.Kx5_abc123...

graph LR
    subgraph "Header"
        H["{ alg: 'HS256' }<br/>서명 알고리즘"]
    end
    subgraph "Payload"
        P["{ sub: 'STU001',<br/>  iat: 1711160000,<br/>  exp: 1711160300 }<br/>담긴 정보 (Claims)"]
    end
    subgraph "Signature"
        S["HMAC-SHA256(<br/>  Header + Payload,<br/>  비밀키<br/>)<br/>위변조 방지 서명"]
    end

    H --- P --- S
파트 내용 인코딩 역할
Header 알고리즘 정보 Base64URL "이 토큰은 HS256으로 서명했어"
Payload 클레임 (데이터) Base64URL "주인은 STU001이고, 5분 후 만료"
Signature 서명값 HMAC-SHA256 "이 토큰은 위조 안 됐어"

Base64 인코딩 != 암호화

Header와 Payload는 Base64로 인코딩된 거지 암호화가 아니야. 누구나 디코딩해서 내용을 볼 수 있어. 비밀번호 같은 민감한 정보를 Payload에 넣으면 안 돼.

Signature만 비밀키로 만들어지기 때문에, 내용을 볼 수는 있지만 위조는 못 해.

NexClass에서 JWT를 왜 쓰나?¶

sequenceDiagram
    participant LMS
    participant NC as NexClass
    participant DB as MariaDB

    LMS->>NC: POST /v2/api/auth/tokens<br/>{ grantType, requestToken }

    LMS->>NC: POST /v2/api/lessons<br/>Authorization: Bearer eyJ...
    NC->>NC: 토큰 검증 (DB 조회 안 함!)
    NC->>DB: INSERT INTO TB_NEXCLASS_LESSON
    NC-->>LMS: { status: 200, datas: {...} }

    Note over LMS, NC: 3단계: 5분 후 토큰 만료
    LMS->>NC: GET /v2/api/lessons/L001<br/>Authorization: Bearer eyJ... (만료된 토큰)
    NC->>NC: 토큰 만료 → 인증 실패
    NC-->>LMS: 401 Unauthorized
방식 세션 기반 JWT 기반 (NexClass)
상태 저장 서버 메모리에 세션 저장 서버에 아무것도 저장 안 함
확인 방법 세션 ID로 DB 조회 토큰 서명만 검증
확장성 서버 여러 대면 세션 공유 필요 비밀키만 공유하면 됨
만료 서버에서 세션 삭제 토큰 자체에 만료시간 포함

코드로 보자¶

JwtProvider -- 토큰 생성/검증/추출¶

NexClass의 JWT 핵심 클래스. 딱 3가지 역할만 해.

@Component
public class JwtProvider {

    @Value("${jwt.secret}")        // application.properties에서 비밀키 주입
    private String secretKey;       // {JWT_SECRET_KEY}

    @Value("${jwt.expiration}")    // 만료시간 (밀리초)
    private long expiration;        // 300000 = 5분

1. 토큰 생성 (createToken)¶

public String createToken(String subject) {
    Date now = new Date();
    Date expireDate = new Date(now.getTime() + expiration);

    // 비밀키로 SecretKey 객체 생성
    SecretKey key = Keys.hmacShaKeyFor(
            secretKey.getBytes(StandardCharsets.UTF_8));

    return Jwts.builder()
            .subject(subject)       // 토큰 주인 (userCd 또는 apiKey)
            .issuedAt(now)          // 발급 시각
            .expiration(expireDate) // 만료 시각
            .signWith(key)          // 서명 (HMAC-SHA256)
            .compact();             // 최종 JWT 문자열 반환
}

subject에 뭐가 들어가?

  • LMS 토큰 방식: userId (학번/교번) → 학생별 식별 가능
  • API Key 직접 방식: apiKey → API 호출자 식별

getJoinUrl()에서 SecurityContextHolder.getContext().getAuthentication().getName()으로 이 subject를 꺼내서 학생별 고유 URL을 제공해.

2. 토큰 검증 (validateToken)¶

public boolean validateToken(String token) {
    try {
        SecretKey key = Keys.hmacShaKeyFor(
                secretKey.getBytes(StandardCharsets.UTF_8));

        Jwts.parser()
            .verifyWith(key)        // 같은 비밀키로 서명 검증
            .build()
            .parseSignedClaims(token);  // 파싱 + 검증 동시 수행

        return true;  // 검증 성공

    } catch (Exception e) {
        return false;  // 서명 불일치, 만료, 형식 오류 등
    }
}

validateToken()이 잡는 예외들

예외 원인
ExpiredJwtException 토큰 만료 (5분 지남)
SignatureException 서명 불일치 (비밀키 다름 = 위조)
MalformedJwtException 토큰 형식 오류 (점 3개 아님)
UnsupportedJwtException 지원 안 되는 JWT 유형

이 모든 예외를 catch(Exception)으로 잡아서 false 반환. 왜 실패했는지는 중요하지 않고, 유효하냐 아니냐만 판단하면 되니까.

3. Subject 추출 (getSubject)¶

public String getSubject(String token) {
    SecretKey key = Keys.hmacShaKeyFor(
            secretKey.getBytes(StandardCharsets.UTF_8));

    return Jwts.parser()
            .verifyWith(key)
            .build()
            .parseSignedClaims(token)
            .getPayload()
            .getSubject();  // Payload의 sub 필드 반환
}

JwtAuthenticationFilter -- 요청 가로채기¶

모든 HTTP 요청이 Controller에 도달하기 전에 이 필터를 먼저 거쳐.

@Component
@RequiredArgsConstructor
public class JwtAuthenticationFilter extends OncePerRequestFilter {

    private final JwtProvider jwtProvider;

    @Override
    protected void doFilterInternal(HttpServletRequest request,
                                    HttpServletResponse response,
                                    FilterChain filterChain)
            throws ServletException, IOException {

        // 1. Authorization 헤더 꺼내기
        String header = request.getHeader("Authorization");

        // 2. 토큰 없거나 Bearer 형식 아니면 → 그냥 통과
        if (header == null || !header.startsWith("Bearer ")) {
            filterChain.doFilter(request, response);
            return;
        }

        // 3. "Bearer " 잘라내고 토큰만 추출
        String token = header.substring(7);
        // "Bearer eyJhbGciOiJ..." → "eyJhbGciOiJ..."

        // 4. 토큰 유효하면 → SecurityContext에 사용자 정보 저장
        if (jwtProvider.validateToken(token)) {
            String userCd = jwtProvider.getSubject(token);

            UsernamePasswordAuthenticationToken authentication =
                    new UsernamePasswordAuthenticationToken(
                            userCd,           // principal (사용자 식별)
                            null,             // credentials (비밀번호 -- 불필요)
                            Collections.emptyList()  // 권한 목록
                    );

            SecurityContextHolder.getContext()
                    .setAuthentication(authentication);
        }

        // 5. 다음 필터로 넘김 (항상 실행)
        filterChain.doFilter(request, response);
    }
}

토큰이 무효해도 왜 에러를 안 던지고 그냥 넘겨?

SecurityConfig에서 경로별로 인증 필요 여부를 설정하거든.

  • /v2/api/auth/tokens → 인증 불필요 (토큰 발급 API니까)
  • /webhook/zoom → 인증 불필요 (Zoom이 호출하니까)
  • /v2/api/lessons/** → 인증 필요

필터는 "토큰 있으면 검증해서 SecurityContext에 넣어"까지만 하고, "인증 안 된 요청을 거부"하는 건 SecurityConfig의 역할이야. 역할 분리.

방식 1: LMS 토큰 방식 (VirtualCampus 호환)¶

private TokenResponse authenticateByLmsToken(String requestToken) {
    try {
        // 1. LMS가 보낸 JWT를 NexClass의 apiSecret으로 검증
        SecretKey key = Keys.hmacShaKeyFor(
                apiSecret.getBytes(StandardCharsets.UTF_8));

        Claims claims = Jwts.parser()
                .verifyWith(key)
                .build()
                .parseSignedClaims(requestToken)
                .getPayload();

        // 2. issuer(iss) 확인 -- apiKey와 일치해야 함
        String issuer = claims.get("iss", String.class);
        if (!apiKey.equals(issuer)) {
            throw new IllegalArgumentException("Invalid API Key");
        }

        // 3. userId(uId) 추출
        String userId = claims.get("uId", String.class);

        // 4. NexClass 자체 JWT 발급
        String token = jwtProvider.createToken(
                userId != null ? userId : apiKey);

        return TokenResponse.builder()
                .accessToken(token)
                .tokenType("Bearer")
                .expiresIn(300)  // 5분
                .build();

    } catch (Exception e) {
        throw new IllegalArgumentException("Invalid API Key");
    }
}

왜 LMS 토큰을 받아서 NexClass 토큰을 다시 발급해?

LMS 토큰과 NexClass 토큰은 비밀키가 다르고, 용도가 달라.

  • LMS 토큰: LMS가 만든 거. "나 LMS야, 이 학생은 STU001이야"
  • NexClass 토큰: NexClass가 만든 거. "인증 확인됨, STU001에게 5분 유효한 출입증 발급"

NexClass API를 호출할 때는 NexClass가 발급한 토큰을 써야 해. JwtAuthenticationFilter가 NexClass 비밀키로 검증하니까.

방식 2: API Key 직접 방식¶

private TokenResponse authenticateByApiKey(TokenRequest request) {
    // 1. API Key 검증
    if (!apiKey.equals(request.getApiKey())) {
        throw new IllegalArgumentException("Invalid API Key");
    }

    // 2. Secret Key 검증
    if (!apiSecret.equals(request.getSecretKey())) {
        throw new IllegalArgumentException("Invalid Secret Key");
    }

    // 3. 토큰 생성
    String token = jwtProvider.createToken(request.getApiKey());

    // 4. 응답 반환
    return TokenResponse.builder()
            .accessToken(token)
            .tokenType("Bearer")
            .expiresIn(300)
            .build();
}

Bearer 토큰 형식¶

HTTP 요청에서 JWT를 보내는 표준 방식이야.

GET /v2/api/lessons/L001 HTTP/1.1
Host: nexclass.knu10.cc
Authorization: Bearer eyJhbGciOiJIUzI1NiJ9.eyJzdWIiOiJTVFUwMDEi...
구성요소 값 설명
헤더 이름 Authorization HTTP 표준 인증 헤더
스킴 Bearer "이 토큰을 가진 사람(bearer)에게 접근 허용"
토큰 eyJhbG... JWT 문자열 
// JwtAuthenticationFilter에서 Bearer 처리
String header = request.getHeader("Authorization");
// header = "Bearer eyJhbGciOiJIUzI1NiJ9..."

if (header == null || !header.startsWith("Bearer ")) {
    // Bearer 형식 아니면 → 인증 없이 통과
    filterChain.doFilter(request, response);
    return;
}

String token = header.substring(7);  // "Bearer " 7글자 잘라냄
// token = "eyJhbGciOiJIUzI1NiJ9..."

'Bearer ' 뒤에 공백이 포함된 7글자

"Bearer " -- B, e, a, r, e, r, (공백) = 7글자. substring(7)로 정확히 토큰 부분만 추출.

토큰 만료 시간 -- 왜 5분?¶

// application.properties
jwt.expiration=300000  // 밀리초 = 300초 = 5분

// TokenResponse
.expiresIn(300)  // 초 단위로 클라이언트에 알려줌

5분이 적절한 이유

너무 길면 보안 위험, 너무 짧으면 사용성 문제.

시간 장점 단점
1분 보안 최고 API 호출 중 만료될 수 있음
5분 보안과 사용성 균형 NexClass 선택
30분 편함 토큰 탈취 시 30분간 악용 가능
24시간 매우 편함 보안 위험 매우 높음

NexClass에서 5분인 이유: - LMS → NexClass API 호출은 요청 직전에 토큰을 발급받아 - 한번 발급받고 여러 API를 호출하는 시나리오 - 강의 생성 + Zoom 미팅 생성 + Registrant 등록까지 보통 2-3초 - 5분이면 충분하고, 토큰 탈취되어도 5분 안에 만료

apiKey + apiSecret -> JWT 발급 흐름¶

설정값¶

# application.properties (실제 값은 마스킹)
nexclass.api.key={API_KEY}
nexclass.api.secret={API_SECRET}
jwt.secret={JWT_SECRET_KEY}
jwt.expiration=300000
설정 용도 길이 요구사항
nexclass.api.key API 호출자 식별 제한 없음
nexclass.api.secret API 호출자 검증 + LMS 토큰 서명키 256bit(32바이트) 이상
jwt.secret NexClass 토큰 서명키 256bit(32바이트) 이상
jwt.expiration 토큰 유효시간 (밀리초) -

비밀키 길이 요구사항

HMAC-SHA256은 최소 256비트(32바이트) 비밀키를 요구해.

SecretKey key = Keys.hmacShaKeyFor(
        secretKey.getBytes(StandardCharsets.UTF_8));

비밀키가 32바이트 미만이면 WeakKeyException이 터져. 영문 32자 이상으로 설정해야 해.

함정¶

함정 1: JWT는 암호화가 아니야

JWT의 Header와 Payload는 Base64로 인코딩된 거야. 누구나 디코딩 가능.

# Payload 부분을 디코딩하면
echo "eyJzdWIiOiJTVFUwMDEiLCJpYXQiOjE3MTExNjAwMDB9" | base64 -d
# → {"sub":"STU001","iat":1711160000}

Signature 덕분에 위조는 못 하지만, 내용은 볼 수 있어. 비밀번호, 주민번호 같은 민감 정보를 Payload에 넣으면 안 돼.

함정 2: 토큰 무효화가 안 돼

JWT는 Stateless(서버에 상태 저장 안 함)야. 한번 발급한 토큰은 만료 전까지 유효해.

"이 토큰을 무효화해줘"를 하려면? - 블랙리스트를 DB/Redis에 저장해야 해 → JWT의 Stateless 장점을 잃어 - 비밀키를 바꾸면? → 모든 토큰이 무효화됨 (너무 과격)

NexClass는 만료시간 5분으로 이 문제를 완화해. 최악의 경우 5분만 버티면 자동 만료.

함정 3: validateToken() 에서 시간 기반 검증

// 서버 시간이 틀리면?
Date expireDate = new Date(now.getTime() + expiration);
JWT의 만료 검증은 서버의 시스템 시각에 의존해. 서버 시간이 5분 빠르면 → 토큰이 즉시 만료 서버 시간이 5분 느리면 → 만료된 토큰도 유효 판정

운영 서버의 NTP(시간 동기화)가 잘 되어 있어야 해.

함정 4: OncePerRequestFilter의 의미

public class JwtAuthenticationFilter extends OncePerRequestFilter {
OncePerRequestFilter = 하나의 HTTP 요청에 딱 한 번만 실행되는 필터.

왜 이게 중요해? Spring에서 필터가 중복 실행되는 경우가 있거든 (forward, include 등). 일반 Filter를 쓰면 같은 요청에서 토큰 검증이 2번 될 수 있어. OncePerRequestFilter가 이걸 방지해줘.

함정 5: SecurityContext 미설정 시 동작

if (jwtProvider.validateToken(token)) {
    // 유효하면 SecurityContext에 저장
    SecurityContextHolder.getContext().setAuthentication(authentication);
}
// 유효하지 않으면? → SecurityContext에 아무것도 안 넣음
// → 인증 필요한 경로면 SecurityConfig에서 403/401 반환
// → 인증 불필요한 경로면 그냥 통과
filterChain.doFilter(request, response);
필터는 "이 사람 인증됐어"를 SecurityContext에 기록할 뿐이야. "인증 안 된 사람 거부"는 SecurityConfig의 책임.

한줄 정리¶

JWT는 서버에 상태를 저장하지 않는 인증 방식. JwtProvider가 토큰을 만들고, JwtAuthenticationFilter가 매 요청마다 검증하고, AuthService가 LMS 인증과 토큰 발급을 관리한다.

확인문제¶

Q1. JWT의 3개 파트 이름과 각각의 역할을 설명해봐.
  • Header: 서명 알고리즘 정보 ({"alg": "HS256"}). 이 토큰이 어떤 방식으로 서명됐는지 알려줘.
  • Payload: 클레임(데이터) ({"sub": "STU001", "exp": 1711160300}). 토큰의 주인, 만료시간 등 실제 정보가 담겨있어.
  • Signature: 위변조 방지 서명. Header + Payload를 비밀키로 HMAC-SHA256 해시한 값. 이 서명이 있어서 토큰 내용을 바꾸면 검증이 실패해.
Q2. JwtAuthenticationFilter가 토큰이 무효해도 에러를 던지지 않는 이유는?

필터의 역할과 SecurityConfig의 역할이 분리되어 있기 때문.

  • 필터: "토큰 있으면 검증해서 SecurityContext에 넣어." (인증 정보 설정만)
  • SecurityConfig: "이 경로는 인증 필요, 이 경로는 불필요." (접근 제어)

/v2/api/auth/tokens(토큰 발급)나 /webhook/zoom(Zoom 웹훅) 같은 경로는 인증 없이도 접근 가능해야 해. 필터가 무조건 에러를 던지면 이런 경로도 접근 못 하게 되거든.

Q3. NexClass에 비밀키가 2개인 이유는?

서로 다른 주체가 만든 토큰을 검증하기 위해.

  • apiSecret (nexclass.api.secret): LMS가 만든 JWT를 NexClass가 검증하는 데 사용. LMS와 NexClass가 같은 키를 공유해야 서명 검증이 가능.
  • jwt.secret: NexClass가 만든 JWT를 NexClass 자신이 검증하는 데 사용. 이건 NexClass 내부에서만 사용하는 키.

LMS 토큰은 "나 LMS야"를 증명하는 거고, NexClass 토큰은 "인증 완료된 사용자야"를 증명하는 거야. 용도가 다르니까 키도 달라야 해.

Q4. 토큰 만료시간을 24시간으로 늘리면 어떤 문제가 생겨?

토큰 탈취 시 24시간 동안 악용 가능.

JWT는 한번 발급하면 서버에서 무효화할 수 없어 (Stateless). 누군가 토큰을 탈취하면 만료될 때까지 해당 사용자 행세를 할 수 있어.

  • 5분: 최악의 경우 5분간 악용 → 피해 제한적
  • 24시간: 최악의 경우 24시간 악용 → 강의 삭제, 출결 조작 등 심각한 피해

NexClass에서 LMS는 API 호출 직전에 토큰을 발급받으니까 5분이면 충분해.

Q5. Bearer eyJhbG...에서 'Bearer ' 앞 7자를 잘라내는 이유를 코드로 설명해봐. 
String header = request.getHeader("Authorization");
// header = "Bearer eyJhbGciOiJIUzI1NiJ9..."

String token = header.substring(7);
// token = "eyJhbGciOiJIUzI1NiJ9..."

HTTP 표준에서 Authorization 헤더 형식이 {스킴} {토큰}이야. Bearer 스킴 + 공백 = 7글자. JwtProvider의 validateToken()과 getSubject()는 순수 JWT 문자열만 받아. "Bearer " 접두어가 붙은 채로 넘기면 파싱 에러가 나.

그래서 substring(7)로 "Bearer " 부분을 잘라내고 순수 토큰만 추출하는 거야.
