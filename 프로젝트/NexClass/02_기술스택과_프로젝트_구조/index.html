02. 기술스택과 프로젝트 구조¶

난이도: Alpha | 전제 지식: Java 기본 문법, 웹 개발 기초 | 예상 학습 시간: 40분

이게 뭐야?¶

프로젝트에 어떤 기술을 왜 선택했는지, 그리고 코드가 어떤 구조로 정리되어 있는지 분석한다.

기술 선택에는 항상 이유가 있어. "남들이 쓰니까"는 이유가 아니야. 각 기술이 이 프로젝트에서 어떤 문제를 해결하는지 알아야 한다.

각 기술 선택의 이유¶

Spring Boot 4.0.1¶

왜 Spring Boot야?

한 줄 답: REST API 서버를 빠르게 만들 수 있는 가장 검증된 프레임워크

장점 이 프로젝트에서의 의미
자동 설정 (Auto Configuration) DB 연결, 보안 설정 등을 최소 설정으로 사용
내장 톰캣 JAR 하나로 서버 실행 가능 (배포 간단)
방대한 생태계 Security, JPA, Thymeleaf 등 필요한 것 전부 있음
검증된 안정성 대학교 운영 시스템이라 안정성이 최우선

트레이드오프

대안: Express.js (Node.js), FastAPI (Python), Ktor (Kotlin)

근데 이 프로젝트에서 Spring Boot를 선택한 진짜 이유가 뭐냐면: LMS가 이미 Spring 기반이야. 같은 기술 스택을 쓰면 팀 내 기술 공유가 쉽고, LMS 코드를 참고하기도 편해. 기술 선택은 "뭐가 좋냐"보다 "팀과 환경에 뭐가 맞냐"가 더 중요하다.

Java 17¶

이유 설명
LTS (장기 지원) 운영 시스템에는 LTS 버전을 쓰는 게 정석
Records, Sealed Classes 최신 문법 활용 가능
Spring Boot 4.0 호환 Spring Boot 4.0이 Java 17 이상을 요구함

JPA (Hibernate)¶

왜 JPA야? MyBatis 안 써?

LMS(LXP-KNU10)는 MyBatis를 써. 근데 NexClass는 JPA를 선택했어.

비교 항목 MyBatis JPA
SQL 작성 직접 작성 (완전 제어) 자동 생성 (편리)
학습 비용 낮음 (SQL만 알면 됨) 높음 (영속성 컨텍스트 등)
단순 CRUD 생산성 보통 높음 (인터페이스만 만들면 끝)
복잡한 쿼리 강함 약함 (JPQL, QueryDSL 필요)
테이블 수 많을 때 유리 적을 때 유리

선택 이유

NexClass는 테이블 11개, 대부분 단순 CRUD야. 복잡한 JOIN이 거의 없고, 조건 검색도 단순해. 이런 경우 JPA가 생산성이 압도적으로 높아.

MyBatis였으면 11개 테이블 x (Mapper XML + VO + DAO) = 33개 파일 이상이 필요했을 거야. JPA는? 11개 Entity + 11개 Repository 인터페이스 = 22개 파일, 그것도 대부분 몇 줄짜리.

MariaDB 11¶

이유 설명
MySQL 호환 LMS가 MySQL 계열 DB를 쓰므로 호환성 보장
오픈소스 라이선스 비용 없음
경량 Oracle 같은 무거운 DB가 필요 없는 규모
기존 인프라 활용 이미 서버에 MariaDB가 설치되어 있음

JWT (jjwt 0.12.3)¶

왜 JWT야?

NexClass는 REST API 서버야. 세션 기반 인증은 REST의 Stateless 원칙에 안 맞아.

근데 여기서 재밌는 게 있어:

인증 방식 사용처
JWT API 요청 (/v2/api/**)
세션 대시보드 (/dashboard/**)

두 가지를 동시에 쓴다. API는 JWT, 대시보드는 세션. 왜? API는 LMS 같은 서버가 호출하니까 JWT가 맞고, 대시보드는 관리자가 브라우저에서 쓰니까 세션이 편해.

이게 트레이드오프야. "하나만 쓰면 되지"가 아니라, 용도에 따라 최적의 방식을 선택하는 거야.

Thymeleaf + Bootstrap 5¶

이유 설명
SSR (서버 사이드 렌더링) 대시보드는 별도 프론트엔드 프로젝트 없이 빠르게 구현
Spring Boot 통합 Spring Boot와 가장 잘 어울리는 템플릿 엔진
Bootstrap CDN 별도 빌드 도구(webpack 등) 없이 CDN으로 바로 사용

트레이드오프

대안: React, Vue.js (SPA)

SPA를 쓰면 UX가 더 좋겠지만, 별도 프론트엔드 프로젝트가 필요하고 빌드/배포 파이프라인도 복잡해져. 대시보드는 관리자 1-2명만 쓰는 내부 도구야. SPA까지 쓸 필요가 없어. YAGNI (You Ain't Gonna Need It) - 지금 필요 없는 건 만들지 마.

Apache HttpClient5¶

// pom.xml에서
// RestTemplate이 PATCH 메서드를 지원하려면 HttpClient5가 필요
private final RestTemplate restTemplate = new RestTemplate(
    new org.springframework.http.client.HttpComponentsClientHttpRequestFactory()
);

왜 이게 필요해?

Java의 기본 HttpURLConnection은 PATCH 메서드를 지원하지 않아. Zoom API에서 미팅 수정 시 PATCH를 쓰는데, 이걸 지원하려면 HttpClient5가 필요해.

이거 모르면 "PATCH 요청이 안 돼요" 하면서 삽질하게 돼.

Lombok¶

@Getter          // getter 자동 생성
@Setter          // setter 자동 생성
@Builder          // 빌더 패턴 자동 생성
@NoArgsConstructor // 기본 생성자
@AllArgsConstructor // 전체 필드 생성자
@RequiredArgsConstructor // final 필드 생성자 (DI용)
@Slf4j            // Logger 자동 생성

Lombok이 없으면?

11개 Entity 클래스에 Getter/Setter를 전부 직접 쓴다고 생각해봐. 필드가 평균 10개씩이면 getter 10개 + setter 10개 = 220줄의 보일러플레이트 코드. Lombok 쓰면 어노테이션 3줄로 끝.

<dependency>
    <groupId>io.jsonwebtoken</groupId>
    <artifactId>jjwt-api</artifactId>      <!-- API (컴파일 타임) -->
    <version>0.12.3</version>
</dependency>
<dependency>
    <groupId>io.jsonwebtoken</groupId>
    <artifactId>jjwt-impl</artifactId>     <!-- 구현체 (런타임만) -->
    <version>0.12.3</version>
    <scope>runtime</scope>
</dependency>
<dependency>
    <groupId>io.jsonwebtoken</groupId>
    <artifactId>jjwt-jackson</artifactId>  <!-- JSON 처리 (런타임만) -->
    <version>0.12.3</version>
    <scope>runtime</scope>
</dependency>

API-Implementation 분리 패턴이야. 컴파일 타임에는 API(인터페이스)만 의존하고, 런타임에는 구현체가 자동 로딩돼. 이게 바로 DIP (의존성 역전 원칙)의 실전 적용이야.

각 계층의 역할¶

계층 역할 규칙
Controller HTTP 요청/응답 처리 비즈니스 로직을 넣지 않는다
Service 비즈니스 로직 Controller를 직접 호출하지 않는다
Repository DB 접근 비즈니스 로직을 넣지 않는다
Entity DB 테이블 매핑 비즈니스 로직을 최소화한다

계층 간 의존성 방향

위에서 아래로만 의존해야 한다.

  • Controller -> Service -> Repository (O)
  • Repository -> Service (X)
  • Service -> Controller (X)

이 규칙을 깨면 순환 참조가 발생하고, 코드가 스파게티가 돼.

주목할 점

ZoomService가 가장 많이 참조돼. LessonService, AttendanceService, RegistrantService가 모두 ZoomService를 쓴다.

이건 ZoomService가 외부 API 연동의 단일 접점 역할을 하기 때문이야. Zoom API 호출이 필요한 곳이 여러 곳인데, 각각 직접 호출하면 토큰 관리, 에러 처리가 중복돼. ZoomService 하나로 모아놓으면 토큰 캐싱, 재시도 로직 같은 걸 한 곳에서만 관리하면 돼.

각 패키지 상세 역할¶

controller/ - 요청의 입구¶

HTTP 요청을 받아서 적절한 Service에 위임하고, 응답을 만들어서 반환한다. 비즈니스 로직을 절대 넣지 않는다.

@RestController
@RequestMapping("/v2/api/lessons")    // 이 컨트롤러가 담당하는 URL
@RequiredArgsConstructor              // 생성자 주입 (DI)
public class LessonController {

    private final LessonService lessonService;  // Service에 위임

    @PostMapping
    public Map<String, Object> createLesson(@RequestBody Lesson lesson) {
        Lesson created = lessonService.createLesson(lesson);  // 위임만 한다
        return buildVcLessonResponse(created);  // 응답 형식 변환
    }
}

JPA Repository 인터페이스. JpaRepository를 상속하면 기본 CRUD가 자동으로 생긴다.

// 이 인터페이스만 만들면 save(), findById(), delete() 등이 자동 생성됨!
public interface LessonRepository extends JpaRepository<Lesson, String> {
    // 커스텀 쿼리: Zoom Meeting ID로 강의 찾기
    Optional<Lesson> findByZoomMeetingId(String zoomMeetingId);

    // 커스텀 쿼리: 활성 강의 조회
    @Query("SELECT l FROM Lesson l WHERE l.lessonEndDate > :now")
    List<Lesson> findActiveLessons(@Param("now") LocalDateTime now);
}

MyBatis였으면 Mapper XML에 SELECT문 쓰고, ResultMap 정의하고... JPA는 인터페이스만 만들면 끝.

    <plugins>

            <configuration>
                <annotationProcessorPaths>

            <artifactId>spring-boot-maven-plugin</artifactId>
            <configuration>
                <excludes>

왜 Lombok을 JAR에서 제외해?

Lombok은 컴파일 타임에만 필요해. 컴파일할 때 Getter/Setter 코드를 생성하고, 런타임에는 이미 생성된 코드가 있으니 Lombok이 필요 없어. JAR 파일 크기도 줄일 수 있고.

함정¶

흔한 실수들

실수 1: Controller에 비즈니스 로직 넣기

// 이러면 안 돼
@PostMapping
public Map<String, Object> createLesson(@RequestBody Lesson lesson) {
    // Zoom API 호출을 Controller에서 직접? SRP 위반!
    Map<String, Object> zoomResult = zoomService.createMeeting(...);
    lesson.setZoomMeetingId(String.valueOf(zoomResult.get("id")));
    lessonRepository.save(lesson);
    // ...
}

Controller는 요청을 받고 응답을 보내는 것만 해야 해. 비즈니스 로직은 Service로.

실수 2: Service에서 HttpServletRequest 직접 접근

Service는 HTTP에 대해 모르는 게 맞아. HTTP 관련 처리는 Controller/Filter에서 해.

실수 3: Entity를 API 응답에 직접 반환

Entity에는 DB 내부 정보가 있을 수 있어. 응답은 DTO나 Map으로 변환해서 보내야 해. NexClass에서는 Map<String, Object>를 많이 쓰는데, 이것도 일종의 트레이드오프야. DTO를 만들면 타입 안전하지만, VirtualCampus 응답 형식을 맞추려면 Map이 더 유연해.

DB MariaDB 11 (MySQL 호환, 오픈소스)
인증 JWT (API) + Session (대시보드)
템플릿 Thymeleaf + Bootstrap 5 CDN (대시보드용)
아키텍처 레이어드 (Controller -> Service -> Repository)
패키지 역할별 분리 (controller, service, entity, repository, security, config, dto, exception, util)

한줄 정리

NexClass는 레이어드 아키텍처를 따르는 Spring Boot 프로젝트로, 각 기술은 "이 프로젝트에서 어떤 문제를 해결하는가"를 기준으로 선택했다.

확인문제¶

Q1. NexClass에서 JPA를 선택한 이유를 MyBatis와 비교해서 설명해봐.

NexClass는 테이블 11개, 대부분 단순 CRUD 구성이야. 복잡한 JOIN이나 동적 쿼리가 거의 없어.

이런 경우 JPA의 장점:

  • Repository 인터페이스만 만들면 기본 CRUD 자동 생성
  • Entity 클래스가 곧 테이블 매핑 (별도 XML 불필요)
  • 22개 파일로 끝 (Entity 11 + Repository 11)

MyBatis였다면:

  • VO 11개 + Mapper 인터페이스 11개 + Mapper XML 11개 = 33개 파일 이상
  • 각 XML에 INSERT, SELECT, UPDATE, DELETE 쿼리를 직접 작성

하지만 복잡한 쿼리가 많거나, DBA가 SQL을 직접 관리해야 하는 환경이면 MyBatis가 더 나을 수 있어. 기술 선택에 "항상 정답"은 없다.

Q2. 레이어드 아키텍처에서 계층 간 의존성 방향이 왜 중요해?

의존성은 위에서 아래로만 흘러야 한다 (Controller -> Service -> Repository).

역방향 의존이 생기면:

  1. 순환 참조: A가 B를 참조하고, B가 A를 참조하면 스프링이 빈을 생성할 수 없어
  2. 변경 전파: 하위 계층이 상위 계층을 알면, 하위 계층을 수정할 때 상위 계층도 영향받아
  3. 테스트 어려움: 하위 계층을 독립적으로 테스트할 수 없어

핵심 원칙: 상위 계층은 하위 계층을 알지만, 하위 계층은 상위 계층을 모른다.

Q3. JWT와 세션 인증을 동시에 쓰는 이유는?

API 요청 (/v2/api/**): LMS 같은 서버가 호출한다. 서버 간 통신에는 세션이 맞지 않아 (Stateless 원칙). JWT가 적합.

대시보드 (/dashboard/**): 관리자가 브라우저에서 접속한다. 브라우저는 쿠키로 세션을 유지하는 게 자연스럽고 편해.

하나로 통일할 수도 있지만, 용도에 맞는 최적의 방식을 선택한 거야. 이게 트레이드오프 분석의 결과.

Q4. 왜 Apache HttpClient5가 필요해?

Java의 기본 HTTP 클라이언트 (HttpURLConnection)은 PATCH 메서드를 지원하지 않아.

Zoom API에서 미팅 수정(updateMeeting)은 PATCH 메서드를 사용하는데, HttpClient5가 없으면 RestTemplate으로 PATCH 요청을 보낼 수 없어.

spring-boot-starter-webmvc에 내장된 HTTP 클라이언트만으로는 부족해서, httpclient5를 별도 의존성으로 추가하고 HttpComponentsClientHttpRequestFactory를 RestTemplate에 설정한 거야.

Q5. ZoomService가 여러 Service에서 참조되는 이유를 SRP 관점에서 설명해봐.

ZoomService는 Zoom API 호출이라는 단일 책임만 가져. 토큰 관리, API 호출, 에러 처리를 한 곳에서 담당해.

이걸 각 Service가 직접 했다면:

  • LessonService에서 토큰 발급 + 미팅 생성 API 호출
  • AttendanceService에서 토큰 발급 + Reports API 호출
  • RegistrantService에서 토큰 발급 + Registrant API 호출

토큰 캐싱 로직이 3곳에 중복되고, Zoom API 변경 시 3곳을 전부 수정해야 해. DRY 원칙 위반.

ZoomService로 모으면 Zoom 관련 변경이 한 곳에서만 발생해.
