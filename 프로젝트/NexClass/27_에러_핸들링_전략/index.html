27. 에러 핸들링 전략¶

핵심개념¶

에러가 터졌을 때 사용자한테 "500 Internal Server Error" 던져놓고 끝내면, 그건 시스템이 아니라 쓰레기야.

에러 핸들링은 선택이 아니야

  • 사용자(LMS)가 에러 원인을 알아야 스스로 대처할 수 있어
  • 운영자가 에러 원인을 알아야 빠르게 수정할 수 있어
  • 에러 응답 형식이 일관적이어야 클라이언트가 파싱할 수 있어

에러 핸들링의 목표는 세 가지야:

  1. 일관된 응답 형식 - 어떤 에러든 같은 JSON 구조
  2. 적절한 정보 노출 - 디버깅에 필요한 건 주고, 보안 민감한 건 숨기고
  3. 중앙 집중 관리 - 에러 처리 로직을 한 곳에서 관리

동작원리¶

Spring 에러 처리 메커니즘¶

graph TD
    A[클라이언트 요청] --> B[DispatcherServlet]
    B --> C[Controller]
    C -->|정상| D[200 OK 응답]
    C -->|예외 발생| E{예외 타입?}

    E -->|IllegalArgumentException| F["@ExceptionHandler<br/>handleIllegalArgument()"]
    E -->|SecurityException| G["@ExceptionHandler<br/>handleSecurityException()"]
    E -->|기타 Exception| H["@ExceptionHandler<br/>handleException()"]

    F --> I[400 Bad Request]
    G --> J[401 Unauthorized]
    H --> K[500 Internal Server Error]

    I --> L[ErrorResponse JSON]
    J --> L
    K --> L

@RestControllerAdvice가 하는 일¶

Spring의 @RestControllerAdvice는 모든 Controller에서 발생하는 예외를 가로채서 한 곳에서 처리해.

@ControllerAdvice vs @RestControllerAdvice

어노테이션 반환 타입 용도
@ControllerAdvice 뷰 이름 (HTML) 웹 페이지 에러 화면
@RestControllerAdvice JSON/XML API 에러 응답 (NexClass)

@RestControllerAdvice = @ControllerAdvice + @ResponseBody

코드 분석¶

GlobalExceptionHandler¶

@Slf4j
@RestControllerAdvice  // 모든 Controller 예외를 여기서 처리
public class GlobalExceptionHandler {

    // === 1. 일반 예외 (최후의 방어선) ===
    @ExceptionHandler(Exception.class)
    public ResponseEntity<ErrorResponse> handleException(
            Exception e,
            HttpServletRequest request) {

        // 운영자용: 전체 스택트레이스 로그
        log.error("[500] {} {} - {}",
            request.getMethod(),
            request.getRequestURI(),
            e.getMessage(), e);  // e를 넘기면 스택트레이스 출력

        // 클라이언트용: 최소 정보만 응답
        ErrorResponse error = ErrorResponse.of(
            HttpStatus.INTERNAL_SERVER_ERROR.value(),  // 500
            "Internal Server Error",
            e.getMessage(),
            request.getRequestURI()
        );

        return ResponseEntity
            .status(HttpStatus.INTERNAL_SERVER_ERROR)
            .body(error);
    }

    // === 2. 잘못된 요청 예외 ===
    @ExceptionHandler(IllegalArgumentException.class)
    public ResponseEntity<ErrorResponse> handleIllegalArgument(
            IllegalArgumentException e,
            HttpServletRequest request) {

        // 400은 클라이언트 실수 → 로그 레벨 낮춤
        ErrorResponse error = ErrorResponse.of(
            HttpStatus.BAD_REQUEST.value(),  // 400
            "Bad Request",
            e.getMessage(),
            request.getRequestURI()
        );

        return ResponseEntity
            .status(HttpStatus.BAD_REQUEST)
            .body(error);
    }

    // === 3. 인증 실패 예외 ===
    @ExceptionHandler(SecurityException.class)
    public ResponseEntity<ErrorResponse> handleSecurityException(
            SecurityException e,
            HttpServletRequest request) {

        ErrorResponse error = ErrorResponse.of(
            HttpStatus.UNAUTHORIZED.value(),  // 401
            "Unauthorized",
            e.getMessage(),
            request.getRequestURI()
        );

        return ResponseEntity
            .status(HttpStatus.UNAUTHORIZED)
            .body(error);
    }
}

ErrorResponse DTO¶

@Getter
@Builder
@NoArgsConstructor
@AllArgsConstructor
public class ErrorResponse {

    private LocalDateTime timestamp;  // 에러 발생 시각
    private int status;               // HTTP 상태 코드
    private String error;             // 에러 유형
    private String message;           // 에러 상세 메시지
    private String path;              // 요청 경로

    // 팩토리 메서드 (빌더 패턴 활용)
    public static ErrorResponse of(
            int status, String error,
            String message, String path) {
        return ErrorResponse.builder()
            .timestamp(LocalDateTime.now())
            .status(status)
            .error(error)
            .message(message)
            .path(path)
            .build();
    }
}

비즈니스 예외¶

클라이언트가 잘못된 요청을 보냈거나, 비즈니스 규칙에 위반되는 경우.

// LessonService - 강의 조회 실패
public Lesson getLesson(String lessonCd) {
    return lessonRepository.findById(lessonCd)
        .orElseThrow(() -> new RuntimeException(
            "강의를 찾을 수 없습니다: " + lessonCd));
    // → GlobalExceptionHandler.handleException()이 잡음
    // → 500 응답 (사실 404가 더 적절)
}

// HostPoolService - 가용 호스트 없음
if (availableHosts.isEmpty()) {
    throw new RuntimeException("가용 호스트가 없습니다.");
    // → 500 응답 (사실 비즈니스 예외라 409가 더 적절)
}

현재 코드의 한계

비즈니스 예외를 RuntimeException으로 던지고 있어. 이러면 모든 비즈니스 예외가 500 응답이 돼.

개선 방법: 커스텀 예외 클래스를 만들어서 HTTP 상태 코드를 구분해야 해.

// 이렇게 개선할 수 있어
public class LessonNotFoundException extends RuntimeException { ... }
public class HostNotAvailableException extends RuntimeException { ... }

// GlobalExceptionHandler에 추가
@ExceptionHandler(LessonNotFoundException.class)
→ 404 Not Found

@ExceptionHandler(HostNotAvailableException.class)
→ 409 Conflict

인증 예외¶

JWT 검증 실패 시 처리.

// JwtAuthenticationFilter - 토큰 검증
if (jwtProvider.validateToken(token)) {
    String userCd = jwtProvider.getSubject(token);
    // SecurityContext에 저장
} else {
    // 검증 실패 → 아무것도 안 함
    // SecurityConfig에서 permitAll/authenticated 설정으로 제어
}

JWT 실패는 Filter 레벨에서 처리돼

JwtAuthenticationFilter에서 토큰이 유효하지 않으면 SecurityContext에 인증 정보를 안 넣어. 그러면 Spring Security가 자동으로 401 응답을 보내. GlobalExceptionHandler까지 안 와.

로깅 전략¶

뭘 로그에 남기고, 뭘 응답에 담는지¶

정보 로그 (서버) 응답 (클라이언트) 이유
스택트레이스 O X 보안 (내부 구조 노출 방지)
에러 메시지 O O 디버깅에 필요
요청 경로 O O 어떤 API에서 발생했는지
요청 메서드 O X 로그 검색용
타임스탬프 O O 시점 파악
요청 바디 선택적 X 민감 데이터 포함 가능 
// 로그: 스택트레이스 포함 (e를 마지막 인자로)
log.error("[500] {} {} - {}",
    request.getMethod(),    // GET, POST 등
    request.getRequestURI(),// /v2/api/lessons
    e.getMessage(),         // 에러 메시지
    e);                     // 스택트레이스 전체

// 응답: 최소 정보만
ErrorResponse.of(
    500,
    "Internal Server Error",  // 일반적 에러 유형
    e.getMessage(),           // 구체적 메시지
    request.getRequestURI()   // 요청 경로
);

로그 레벨 사용 기준

레벨 용도 NexClass 예시
ERROR 즉시 조치 필요 DB 연결 실패, API 500 에러
WARN 주의 필요 (잠재적 문제) 매칭 실패, 데이터 누락
INFO 정상 동작 기록 API 호출, 처리 완료
DEBUG 디버깅용 상세 정보 변수 값, 중간 상태

에러 전파 vs 에러 처리¶

에러를 잡았을 때, 여기서 처리할 것인가 vs 위로 올릴 것인가?

판단 기준¶

기준 여기서 처리 (catch + 대응) 위로 전파 (throw/rethrow)
복구 가능? 대안이 있으면 처리 대안 없으면 전파
호출자가 알아야 해? 몰라도 되면 처리 알아야 하면 전파
비즈니스 영향? 부분 실패 허용이면 처리 전체 실패면 전파

주의사항¶

RuntimeException 남용

현재 코드는 거의 모든 예외를 RuntimeException으로 던져. 이러면 모든 비즈니스 에러가 500 응답이 돼.

HTTP 상태 코드를 제대로 쓰려면 커스텀 예외 클래스가 필요해.

에러 메시지에 민감 정보 노출

e.getMessage()를 그대로 응답에 담고 있어. DB 에러면 테이블명, 컬럼명이 노출될 수 있어.

프로덕션에서는 500 에러의 message를 일반적인 메시지로 교체해야 해.

// 현재: 내부 정보 노출 위험
ErrorResponse.of(500, "Internal Server Error",
    e.getMessage(), ...);

// 개선: 일반적 메시지로 교체
ErrorResponse.of(500, "Internal Server Error",
    "서버 내부 오류가 발생했습니다. 관리자에게 문의하세요.",
    ...);

정리¶

항목 내용
중앙 처리 @RestControllerAdvice + @ExceptionHandler
응답 형식 ErrorResponse (timestamp, status, error, message, path)
예외 분류 비즈니스 / 인증 / 외부 API / 시스템
로깅 ERROR (스택트레이스 포함), 응답에는 최소 정보
전파 기준 복구 가능하면 처리, 불가능하면 전파
한계 RuntimeException 남용, 커스텀 예외 미분리, 민감 정보 노출 위험
