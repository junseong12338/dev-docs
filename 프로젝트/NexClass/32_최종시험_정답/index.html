32. 최종시험 - 정답 및 해설¶

해설 원칙

정답 맞히는 건 시작이야. 왜 그게 정답인지 설명 못 하면 불합격. 이건 정답지가 아니야. 해설지야.

Part 1: 객관식 해설 (20점)¶

Q1. NexClass 프로젝트의 핵심 목표 (객관식, 2점)¶

정답: (B) LMS의 VirtualCampus API를 호환하는 자체 화상강의 솔루션으로 드롭인 교체한다

해설:

NexClass의 존재 이유는 드롭인 교체다. LMS 코드는 한 줄도 안 건드리고, API URL만 바꾸면 기존 VirtualCampus 대신 NexClass가 동작한다. 17개 API 엔드포인트를 100% 호환하는 게 핵심이야.

  • (A) 범용 화상회의 플랫폼이 아니다. KNU LMS 전용 솔루션이야.
  • (C) LMS 전체를 재개발하는 게 아니다. LMS는 그대로 두고 화상강의 부분만 교체한다.
  • (D) Zoom SDK를 내장하는 게 아니다. Zoom S2S OAuth API를 서버 간 통신으로 호출한다.

채점 기준: 정답만 인정. 부분 점수 없음.

Q2. PATCH 메서드를 위한 의존성 (객관식, 2점)¶

Q20. HMAC-SHA256 서명 생성 (코드 작성, 5점)¶

정답:

String message = "v0:" + timestamp + ":" + rawBody;
String expectedSignature = "v0=" + hmacSha256(message, secretToken);

해설:

Zoom Webhook 서명 검증의 전체 구조:

  1. Zoom이 x-zm-request-timestamp(타임스탬프)와 x-zm-signature(서명)를 헤더에 보낸다
  2. NexClass는 "v0:" + timestamp + ":" + rawBody를 secretToken으로 HMAC-SHA256 해시한다
  3. "v0=" + 해시값을 Zoom이 보낸 서명과 비교한다
  4. 일치하면 Zoom이 보낸 진짜 요청이다
private boolean verifySignature(String rawBody, String timestamp, String zoomSignature) {
    String message = "v0:" + timestamp + ":" + rawBody;
    String expectedSignature = "v0=" + hmacSha256(message, secretToken);
    return expectedSignature.equals(zoomSignature);
}

v0: 접두사는 Zoom의 서명 프로토콜 버전이다. 향후 v1:으로 바뀔 수 있다.

채점 기준: message 구성 3점 (구분자 : 누락 시 1점 감점, v0: 누락 시 2점 감점). expectedSignature 구성 2점 (v0= 접두사 누락 시 1점 감점).

Q21. 이메일 매칭 버그 (코드 분석, 5점)¶

정답: equals()를 사용하여 대소문자를 구분하고 있다. equalsIgnoreCase()로 변경해야 한다.

해설:

버그가 있는 코드:

if (lu.getUserCd().equals(emailPrefix)) {  // 대소문자 구분!

수정된 코드 (실제 NexClass 코드):

if (lu.getUserCd().equalsIgnoreCase(emailPrefix)) {  // 대소문자 무시

Zoom은 이메일을 소문자로 변환하기도 하고 원본 대소문자를 유지하기도 한다. 예를 들어 LMS에서 User001로 등록하면 Zoom에서 user001@nexclass.local로 반환될 수 있다. equals()를 쓰면 "User001" != "user001"이 되어 학생을 찾지 못한다.

이건 실제로 NexClass 개발 과정에서 발견된 버그다.

채점 기준: 버그 식별("대소문자 구분 문제") 3점, 수정 방법("equalsIgnoreCase") 2점. 버그를 찾지 못하면 0점.

Q22. 호스트 배정 5단계 (코드 분석, 5점)¶

정답:

  1. ACTIVE 상태인 호스트 전체를 조회한다 (hostPoolRepository.findByHostStatus("ACTIVE"))
  2. 해당 시간대에 이미 배정된 호스트 이메일 목록을 조회한다 (hostAssignmentRepository.findAssignedHostEmails(startTime, endTime))
  3. 전체 호스트에서 배정된 호스트를 제외하여 가용 호스트를 필터링한다 (stream + filter)
  4. 가용 호스트가 없으면 RuntimeException을 던진다
  5. 가용 호스트의 첫 번째를 선택하고 HostAssignment에 배정 기록을 저장한다

해설:

public String assignHost(LocalDateTime startTime, LocalDateTime endTime, String lessonCd) {
    // 1. ACTIVE 호스트 전체
    List<HostPool> activeHosts = hostPoolRepository.findByHostStatus("ACTIVE");

    // 2. 시간대 중복 호스트
    List<String> assignedEmails = hostAssignmentRepository
        .findAssignedHostEmails(startTime, endTime);

    // 3. 가용 호스트 = 전체 - 배정
    List<HostPool> availableHosts = activeHosts.stream()
        .filter(host -> !assignedEmails.contains(host.getHostEmail()))
        .toList();

    // 4. 가용 호스트 없으면 에러
    if (availableHosts.isEmpty()) {
        throw new RuntimeException("가용 호스트가 없습니다.");
    }

    // 5. 첫 번째 선택 + 배정 기록 저장
    HostPool selectedHost = availableHosts.get(0);
    // ... assignment 저장
    return selectedHost.getHostEmail();
}

핵심은 시간대 기반 배정이라는 점이다. 같은 호스트라도 시간이 안 겹치면 여러 강의에 배정 가능하다. 테이블 2개(HOST_POOL 고정 + HOST_ASSIGNMENT 동적)로 분리한 것도 이 로직을 위해서다.

채점 기준: 각 단계 1점. 순서가 틀리면 감점. 핵심 키워드 누락 시 해당 단계 0점.

Q24. 녹화 URL 형식과 share_url vs play_url (코드 분석, 5점)¶

정답:

share_url + ?pwd= 형식으로 저장하는 이유: 학생이 녹화 영상에 접근할 때 비밀번호 입력 없이 바로 재생할 수 있게 하기 위해서다. share_url에 ?pwd=playPasscode를 붙이면 Zoom이 패스코드를 URL 파라미터로 인식하여 자동 인증한다.

play_url vs share_url 차이:

항목 play_url share_url
접근 방식 Zoom 계정 로그인 또는 API 토큰 필요 공개 링크 (누구나 접근 가능)
인증 JWT Bearer 토큰 패스코드 (URL 파라미터로 전달 가능)
용도 API에서 직접 다운로드 브라우저에서 재생
LMS 연동 사용하지 않음 LMS에서 학생에게 제공하는 URL

NexClass는 share_url을 사용한다. 학생이 LMS에서 "다시보기" 버튼을 클릭하면 share_url?pwd=xxx로 이동하여 Zoom 계정 없이도 녹화 영상을 볼 수 있다.

해설:

// 실제 NexClass 코드
String recordUrl;
if (shareUrl != null && playPasscode != null) {
    recordUrl = shareUrl + "?pwd=" + playPasscode;  // 패스코드 포함 URL
} else if (shareUrl != null) {
    recordUrl = shareUrl;                            // 패스코드 없이
} else {
    recordUrl = (String) file.get("play_url");       // 폴백
}

playPasscode는 recording_play_passcode에서 가져온다. 이건 URL 인코딩된 패스코드로, 일반 password와는 다르다.

채점 기준: 비밀번호 없이 재생 가능하다는 이유 2점, play_url과 share_url 차이 설명 3점. "share_url이 공개 링크"라는 핵심이 없으면 차이 설명 1점.

Q25. 동기화 순서의 필요성 (코드 분석, 5점)¶

정답:

동기화 순서가 term -> course -> user -> courseUser여야 하는 이유는 FK(외래 키) 의존성 때문이다.

  • Course는 Term의 term_cd를 FK로 참조한다
  • CourseUser는 Course의 course_cd와 User의 user_cd를 FK로 참조한다

순서를 courseUser -> user -> course -> term으로 바꾸면:

  1. CourseUser를 먼저 저장하려고 할 때, 참조하는 Course와 User가 아직 DB에 없다
  2. FK 제약 조건 위반으로 INSERT가 실패한다 (데이터 무결성 에러)
  3. Course를 Term보다 먼저 저장해도 같은 문제가 발생한다

해설:

이건 관계형 데이터베이스의 기본이야. FK가 참조하는 부모 레코드가 먼저 존재해야 자식 레코드를 INSERT할 수 있다. 이 순서를 위상 정렬(Topological Sort)이라고도 한다.

Term (부모)
  └→ Course (Term의 자식)
       └→ CourseUser (Course + User의 자식)

User (부모)
  └→ CourseUser (Course + User의 자식)

NexClass는 UPSERT(있으면 UPDATE, 없으면 INSERT) 방식을 사용하므로, 이미 존재하는 레코드는 UPDATE되고 새 레코드만 INSERT된다. 하지만 순서는 여전히 중요하다.

채점 기준: FK 의존성 언급 3점, 역순 시 발생하는 문제(FK 위반 에러) 2점. "순서가 중요하다"만 적고 이유가 없으면 1점.

Q26. 전체 출결 흐름 (서술형, 5점)¶

모범 답안:

  1. 교수가 Zoom 미팅을 종료한다
  2. Zoom 서버가 meeting.ended 웹훅을 NexClass의 /webhook/zoom 엔드포인트로 전송한다
  3. WebhookController가 요청을 수신하고, HMAC-SHA256 서명을 검증하여 Zoom이 보낸 진짜 요청인지 확인한다
  4. 서명 검증 통과 후 즉시 200 OK를 응답하고, handleMeetingEnded()를 호출한다
  5. AttendanceService.processAttendance()가 @Async로 백그라운드에서 실행된다
  6. ZoomService.getReportParticipants()로 Zoom Reports API를 호출하여 실제 참여자 데이터(이메일, 입장/퇴장 시각, duration)를 조회한다
  7. 각 참여자에 대해 이메일 기반 매칭을 수행한다: 참여자의 user_email에서 @ 앞부분을 추출하고, LessonUser 테이블의 userCd와 대소문자 무시 비교한다
  8. 호스트(교수)는 출결 저장에서 제외한다 (hostEmail 비교)
  9. 매칭된 각 세션을 개별 Attendance 엔티티로 저장한다 (같은 학생이 입퇴장을 반복하면 각 세션이 별도 레코드)
  10. saveAttendance() 트랜잭션이 커밋된 후, LmsSyncService.pushAttendanceCompleted()가 LMS에 출결 완료를 Push한다 (POST /vc/api/sync/webhook)
  11. LMS의 VcSyncApiController가 Push를 수신하고, NexClass API를 호출하여 출결 데이터를 조회한 후, LMS 자체 기준으로 최종 출결 판정을 한다

채점 기준:

요소 배점
WebhookController + HMAC 서명 검증 1점
AttendanceService + @Async 백그라운드 처리 1점
Zoom Reports API + 이메일 매칭 1점
트랜잭션 커밋 후 LMS Push 1점
LMS 수신 + 최종 판정 1점

Q27. 대시보드 인증 설계 트레이드오프 (서술형, 5점)¶

모범 답안:

컨트롤러 레벨 세션 체크를 선택한 이유:

Spring Security의 UserDetailsService를 사용하려면 UserDetailsService 구현체, PasswordEncoder Bean, 폼 로그인 설정, 권한(Role) 체계 등을 전부 구현해야 한다. 관리자 1명만 사용하는 대시보드에 이 인프라를 구축하는 것은 오버엔지니어링이다.

1. 장점:

  • 구현 단순성: session.getAttribute("admin") 한 줄로 인증을 처리할 수 있다. 별도의 인증 인프라 코드가 필요 없다.
  • JWT와의 충돌 방지: Spring Security의 폼 로그인을 활성화하면 JWT 필터 체인과 설정이 복잡해진다. 컨트롤러 레벨 인증은 SecurityConfig를 건드리지 않으므로 JWT 인증에 영향을 주지 않는다.
  • 빠른 개발: 세션 체크 코드 몇 줄로 구현 완료. 개발 속도가 빠르다.

2. 단점:

  • 보안 일관성 부족: Spring Security의 보안 기능(CSRF, 세션 고정 공격 방어 등)을 대시보드에서 사용할 수 없다.
  • 유지보수 위험: 새 대시보드 엔드포인트를 추가할 때마다 세션 체크 코드를 빠뜨릴 수 있다. 실수하면 인증 없이 접근 가능한 페이지가 생긴다.
  • 확장성 제한: 관리자가 여러 명이 되거나, 역할(Role) 기반 인가가 필요하면 직접 구현해야 한다.

3. 이 설계를 뒤집어야 하는 상황:

  • 관리자가 2명 이상으로 늘어나서 역할 구분이 필요할 때
  • 대시보드가 외부 인터넷에 노출되어 보안 강화가 필요할 때
  • 감사(Audit) 로그나 로그인 이력 관리가 요구될 때

채점 기준:

요소 배점
장점 2가지 이상 2점
단점 2가지 이상 2점
뒤집을 상황 1점

불합격 기준: 트레이드오프 분석 없이 "단순해서"만 적으면 1점.

Q29. 트랜잭션 분리 이유 (서술형, 4점)¶

모범 답안:

분리한 이유: saveAttendance()와 pushAttendanceCompleted()를 같은 트랜잭션에 넣으면, LMS Push가 완료될 때까지 트랜잭션이 커밋되지 않는다. 문제는 LMS Push 후 LMS가 NexClass API를 콜백 호출하여 출결 데이터를 조회한다는 것이다.

분리하지 않으면 발생하는 문제:

  1. saveAttendance()가 출결 데이터를 DB에 저장한다 (아직 커밋 안 됨)
  2. 같은 트랜잭션 안에서 pushAttendanceCompleted()가 LMS에 Push를 보낸다
  3. LMS가 Push를 받고 즉시 NexClass API를 호출하여 출결 데이터를 조회한다
  4. 아직 트랜잭션이 커밋되지 않았으므로 LMS는 출결 데이터를 찾을 수 없다 (REPEATABLE READ 격리 수준에서 다른 트랜잭션의 미커밋 데이터는 보이지 않는다)
  5. LMS는 "출결 데이터 없음"으로 처리하여 출결이 누락된다

해결: saveAttendance()를 별도 @Transactional 메서드로 분리하여 먼저 커밋시킨 후, pushAttendanceCompleted()를 호출한다. 이렇게 하면 LMS 콜백 시점에 출결 데이터가 이미 DB에 커밋되어 있다.

@Async
public void processAttendance(String meetingId, String uuid) {
    // 1단계: 트랜잭션 내에서 출결 저장 + 커밋
    String[] result = saveAttendance(meetingId, uuid);

    // 2단계: 커밋 후 LMS Push (LMS 콜백 시 데이터 조회 가능)
    lmsSyncService.pushAttendanceCompleted(lessonCd, lessonProfCd);
}

채점 기준:

요소 배점
LMS 콜백 시 미커밋 데이터 문제 설명 2점
분리하지 않았을 때의 구체적 문제(출결 누락) 1점
해결 방법(별도 @Transactional + 커밋 후 Push) 1점

불합격 기준: "트랜잭션 분리가 좋으니까"만 적으면 0점. 이유 없는 답은 답이 아니야.

Q30. ngrok vs Nginx 비교 (서술형, 3점)¶

모범 답안:

개발 환경에서 ngrok을 사용하는 이유: 로컬 개발 서버(localhost:8082)에는 공인 IP와 도메인이 없다. Zoom Webhook은 인터넷에서 접근 가능한 public URL이 필요하다. ngrok은 추가 인프라 없이 터널링으로 즉시 public URL을 제공하므로 개발 시 빠르게 Webhook을 테스트할 수 있다.

프로덕션에서 Nginx를 사용하는 이유: 프로덕션 서버는 공인 IP와 자체 도메인(nexclass.knu10.cc)이 있다. Nginx는 SSL 처리, 리버스 프록시, 정적 파일 서빙, 요청 필터링 등 프로덕션 수준의 기능을 제공한다.

ngrok을 프로덕션에서 계속 사용하면 안 되는 이유:

  1. 안정성 문제: ngrok 클라우드 서버에 장애가 나면 NexClass 전체가 불통이 된다. 제3자 서비스에 대한 의존성이 생긴다. Nginx는 같은 서버에서 돌아가므로 외부 의존성이 없다.

  2. 성능 저하: 모든 요청이 ngrok 서버를 경유하므로 추가 네트워크 홉이 발생한다. LMS API 호출, Webhook 수신 등 실시간성이 중요한 통신에서 지연이 생긴다.

  3. 보안 위험: 트래픽이 ngrok의 제3자 서버를 경유한다. JWT, API 키 등 민감한 데이터가 ngrok 인프라를 거치게 된다. 자체 SSL + Nginx 구성이면 트래픽이 직접 서버로 들어온다.

채점 기준:

요소 배점
ngrok 사용 이유 (개발 환경) 1점
ngrok을 프로덕션에서 쓰면 안 되는 이유 2가지 이상 2점

불합격 기준: "ngrok은 개발용이니까"만 적으면 0점. 왜 개발 전용인지 구체적 이유가 있어야 한다.
