28. 전체 플로우 통합¶

핵심개념¶

지금까지 각 기능을 쪼개서 봤어. 이제 전체 시스템을 하나의 흐름으로 연결한다.

개별 기능 이해 vs 전체 흐름 이해

각 Service가 뭘 하는지 아는 건 Lv2야. 전체 요청이 어떤 경로로 흘러가고, 어떤 컴포넌트가 언제 호출되고, 에러가 나면 어디서 잡히는지 전체 그림을 그릴 수 있어야 Lv4야.

NexClass 시스템의 4대 흐름:

Flow 트리거 종착점
수업 생성 LMS API 요청 Zoom 미팅 생성 + 학생 등록
수업 진행 + 출석 Zoom meeting.ended 웹훅 LMS 출결 반영
녹화 Zoom recording.completed 웹훅 학생 다시보기 가능
데이터 동기화 스케줄러/수동 실행 NexClass DB에 학사 데이터 반영

Flow 1: 수업 생성 전체 흐름¶

LMS에서 화상강의를 만들면 벌어지는 일.

sequenceDiagram
    participant LMS as LMS (LXP-KNU10)
    participant Auth as AuthController
    participant JWT as JwtAuthenticationFilter
    participant LC as LessonController
    participant LS as LessonService
    participant HPS as HostPoolService
    participant ZS as ZoomService
    participant RS as RegistrantService
    participant DB as NexClass DB
    participant Zoom as Zoom Platform

    Note over LMS: 교수가 화상강의 생성 버튼 클릭

    LMS->>Auth: POST /v2/api/auth/token
    Note over LMS,Auth: { apiKey, apiSecret }
    Auth-->>LMS: { token: "eyJ..." }

    LMS->>JWT: POST /v2/api/lessons
    Note over LMS,JWT: Authorization: Bearer {token}
    JWT->>JWT: 토큰 검증 + userCd 추출
    JWT->>LC: 인증된 요청 전달

    LC->>LS: createLesson(lesson)

    LS->>HPS: assignHost(startTime, endTime, lessonCd)
    HPS->>DB: 가용 호스트 조회 (ACTIVE + 시간대 미배정)
    DB-->>HPS: 호스트 목록
    HPS->>DB: HOST_ASSIGNMENT 저장
    HPS-->>LS: hostEmail 반환

    LS->>ZS: createMeeting(name, startTime, duration, hostEmail)
    ZS->>Zoom: POST /v2/users/{hostEmail}/meetings
    Zoom-->>ZS: { id, start_url, join_url, password }
    ZS-->>LS: Zoom 미팅 정보

    LS->>DB: LESSON 저장 (meetingId, joinUrl 등)

    LS->>RS: registerCourseStudentsAsync(lessonCd, courseCd, meetingId)
    Note over RS: @Async 비동기 실행
    RS->>DB: COURSE_USER에서 수강생 조회
    RS->>Zoom: POST /v2/meetings/{meetingId}/registrants (학생별)
    Zoom-->>RS: { join_url: "학생별 고유 URL" }
    RS->>DB: LESSON_USER 저장 (userCd + joinUrl)

    LS-->>LC: Lesson 엔티티
    LC-->>LMS: VcApiResponse (joinUrl, startUrl 등)

컴포넌트 매핑¶

단계 Controller Service Entity/Table
인증 AuthController AuthService + JwtProvider -
강의 생성 LessonController LessonService TB_NEXCLASS_LESSON
호스트 배정 - HostPoolService TB_NEXCLASS_HOST_POOL, HOST_ASSIGNMENT
Zoom 미팅 - ZoomService - (외부 API)
학생 등록 - RegistrantService TB_NEXCLASS_LESSON_USER

Flow 2: 수업 진행 + 출석 전체 흐름¶

교수가 미팅을 종료하면 벌어지는 일.

sequenceDiagram
    participant Prof as 교수
    participant Student as 학생
    participant Zoom as Zoom Platform
    participant WC as WebhookController
    participant AS as AttendanceService
    participant ZS as ZoomService
    participant DB as NexClass DB
    participant Sync as LmsSyncService
    participant LMS as LMS

    Note over Prof,Student: === 수업 진행 중 ===
    Prof->>Zoom: 미팅 시작 (start_url)
    Student->>Zoom: 미팅 참여 (join_url - 학생별 고유)
    Note over Zoom: 학생 입퇴장 기록 축적

    Prof->>Zoom: 미팅 종료

    Note over Zoom,WC: === Webhook 수신 ===
    Zoom->>WC: POST /webhook/zoom
    Note over Zoom,WC: event: "meeting.ended"<br/>x-zm-signature: HMAC-SHA256
    WC->>WC: 서명 검증 (HMAC-SHA256)
    WC-->>Zoom: 200 OK (즉시 응답)

    Note over WC,AS: === @Async 비동기 처리 ===
    WC->>AS: processAttendance(meetingId, uuid)

    AS->>DB: meetingId로 Lesson 조회
    DB-->>AS: Lesson (lessonCd, hostEmail, lessonProfCd)

    AS->>ZS: getReportParticipants(meetingId)
    ZS->>Zoom: GET /v2/report/meetings/{meetingId}/participants
    Zoom-->>ZS: 참여자 목록 (name, email, duration, join/leave)
    ZS-->>AS: 참여자 데이터

    AS->>DB: LessonUser 조회 (등록 학생 목록)

    loop 각 참여자
        AS->>AS: 이메일 매칭 (userCd@nexclass.local → userCd)
        AS->>AS: 호스트 스킵 (hostEmail과 비교)
        AS->>AS: 출결 판정 (duration/lessonTotalTime >= 99%)
        AS->>DB: Attendance 저장 (세션별 개별)
    end

    Note over AS,DB: @Transactional 커밋 완료

    AS->>Sync: pushAttendanceCompleted(lessonCd, lessonProfCd)
    Sync->>Sync: JWT 토큰 생성
    Sync->>LMS: POST /vc/api/sync/webhook
    Note over Sync,LMS: { eventType, datas: { lessonCd } }
    LMS-->>Sync: 200 OK

    Note over LMS: === LMS 콜백 ===
    LMS->>AS: GET /v2/api/lessons/{lessonCd}/attends
    Note over LMS,AS: Authorization: Bearer {JWT}
    AS->>DB: Attendance 조회 (lessonCd)
    AS-->>LMS: VcApiAttendVO (historyList 포함)
    LMS->>LMS: 자체 출결 판정 + DB 저장

컴포넌트 매핑¶

단계 Controller Service Entity/Table
Webhook 수신 WebhookController - -
참여자 조회 - ZoomService - (Zoom API)

Flow 4: 데이터 동기화 흐름¶

LMS 학사 데이터를 NexClass DB로 동기화.

sequenceDiagram
    participant Dash as 대시보드/스케줄러
    participant DC as DashboardController
    participant Sync as LmsSyncService
    participant LMS as LMS API
    participant DB as NexClass DB

    Dash->>DC: POST /dashboard/sync/all
    DC->>Sync: syncAll()

    Note over Sync: === 순서 중요: FK 의존성 ===

    Sync->>Sync: 1. JWT 토큰 생성
    Sync->>LMS: POST /vc/api/sync/term
    Note over Sync,LMS: Authorization: Bearer {JWT}
    LMS-->>Sync: AES-256 암호화된 응답
    Sync->>Sync: AES-256 복호화 → JSON
    Sync->>DB: Term UPSERT (~16건)

    Sync->>LMS: POST /vc/api/sync/crs
    LMS-->>Sync: AES-256 암호화
    Sync->>Sync: 복호화
    Sync->>DB: Course UPSERT (~351건)

    loop 페이지별 (pageSize=1000)
        Sync->>LMS: POST /vc/api/sync/users
        LMS-->>Sync: AES-256 암호화
        Sync->>Sync: 복호화
        Sync->>DB: User UPSERT
    end
    Note over DB: ~51,683건 완료

    loop 페이지별
        Sync->>LMS: POST /vc/api/sync/crsUsers
        LMS-->>Sync: AES-256 암호화
        Sync->>Sync: 복호화
        Sync->>DB: CourseUser UPSERT (중복 체크)
    end

    Sync-->>DC: 완료
    DC-->>Dash: { success: true }

에러 처리 (Error Handling)¶

Flow 에러 발생 지점 처리 방식 영향
수업 생성 Zoom API 실패 catch → syncStatus=FAIL + 호스트 반환 강의 생성은 되지만 Zoom 미팅 없음
출석 처리 Reports API 실패 catch → return null (처리 중단) 출결 데이터 미저장
출석 처리 LMS Push 실패 catch → 로그 (데이터는 보존) LMS에 미통보
녹화 처리 Recordings API 실패 catch → 웹훅 데이터 사용 (fallback) play_url로 대체
데이터 동기화 AES 복호화 실패 throw → 해당 동기화 중단 나머지 동기화는 독립 실행

주의사항¶

비동기 처리의 함정

Flow 2, 3에서 @Async를 쓰고 있어. WebhookController는 즉시 200 OK를 반환하고, 출결/녹화 처리는 백그라운드에서 돌아.

문제: 비동기 메서드 안에서 에러가 나면 호출자는 모른다. 에러가 로그에만 남고, Zoom은 이미 200 OK를 받았으니 재전송 안 해. → 로그 모니터링이 필수.

트랜잭션 경계 주의

processAttendance()는 @Async라 별도 스레드에서 실행돼. saveAttendance()는 @Transactional이야.

@Async + @Transactional이 같은 메서드에 있으면 트랜잭션이 제대로 안 걸릴 수 있어. (프록시 문제)

NexClass는 이걸 메서드를 분리해서 해결했어:

  • processAttendance() - @Async (비동기 진입점)
  • saveAttendance() - @Transactional (트랜잭션 단위)

확인문제¶

문제 1¶

수업 생성 Flow에서 Zoom 미팅 생성이 실패하면 어떻게 되는지, 호스트 배정부터 끝까지 추적해봐.

힌트

LessonService.createLesson()의 catch 블록을 봐. syncStatus는 뭐로 설정되고, 호스트는 어떻게 되고, DB에는 뭐가 저장돼?

문제 2¶

출석 처리 Flow에서 @Async를 쓰는 이유가 뭐야? 만약 동기로 처리하면 어떤 문제가 생겨?

힌트

Zoom Webhook은 응답 시간에 민감해. 3초 안에 응답 안 하면 Zoom이 재전송해. Reports API 호출 + DB 저장 + LMS Push까지 3초 안에 될까?

문제 3¶

데이터 동기화에서 순서가 Term -> Course -> User -> CourseUser인 이유를 FK 의존성 관점에서 설명해봐.

힌트

CourseUser 테이블이 Course와 User를 참조해. Course가 없는 상태에서 CourseUser를 넣으면 어떻게 돼?

문제 4¶

NexClass 시스템에는 3가지 인증 방식(JWT, 세션, HMAC 서명)이 공존해. 각각 어떤 Flow에서 왜 사용되는지 설명해봐.

힌트

통신 주체가 누구인지 생각해봐. 서버 간? 브라우저? 외부 플랫폼?

문제 5¶

녹화 처리 Flow에서 Recordings API 호출이 실패해도 녹화를 저장할 수 있는 이유가 뭐야? 이 fallback 전략의 장단점은?

힌트

웹훅 payload에도 녹화 파일 정보가 들어있어. share_url은 API에서만 얻을 수 있지만, play_url은 웹훅에도 있어. play_url은 비밀번호 입력이 필요하다는 단점이 있어.
