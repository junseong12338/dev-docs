14. Zoom Registrant 시스템¶

핵심 개념: Registrant가 뭐야?¶

Registrant는 미팅에 사전 등록된 참가자야.

일반 Zoom 미팅은 join_url만 있으면 누구나 들어올 수 있다. URL을 공유하면 외부인도 입장 가능하지. 하지만 Registrant를 사용하면 등록된 사람만 입장할 수 있고, 등록된 각 사람에게 고유한 joinUrl이 발급된다.

구분 일반 미팅 Registrant 미팅
입장 방식 공통 join_url 학생별 고유 joinUrl
접근 제어 URL 아는 사람 전부 등록된 사람만
참가자 식별 입력한 이름에 의존 (거짓 가능) 등록 정보(이메일)로 확정
출석 추적 이름 매칭 (부정확) registrant_id로 정확한 매칭

동작 원리: NexClass에서 Registrant를 왜 쓰나?¶

핵심 문제: 학생을 어떻게 구분할 것인가?¶

Zoom 미팅에 30명이 참가했다. 출석을 찍어야 해. 그런데 이 사람이 우리 학교 김개발 학생인지 어떻게 아냐?

Zoom 참가자 이름: "김개발"
DB 학생 이름: "김개발"

문제:
- 같은 이름이 2명이면?
- 학생이 이름을 "ㅋㅋ"로 바꾸면?
- 외부인이 "김개발"로 들어오면?

등록 이메일: USR001@nexclass.local
Zoom 참가자 이메일: USR001@nexclass.local

이메일에서 userCd 추출: USR001
→ DB에서 학생 확정

결과: 위조 불가능, 동명이인 문제 없음

Registrant 방식이 유일하게 신뢰할 수 있는 학생 식별 방법이야.

이메일 포맷: userCd@nexclass.local¶

String email = student.getUserCd() + "@" + dummyEmailDomain;
// 예: USR001@nexclass.local

왜 이런 이상한 이메일 형식을 쓰냐?

이유 설명
실제 이메일이 없다 학생들에게 Zoom 계정이 없어. 실제 이메일로 등록하면 Zoom 초대 메일이 가는데 의미 없잖아
userCd를 심는다 이메일의 @ 앞 부분 = userCd. 나중에 출석 매칭할 때 이메일에서 userCd를 추출한다
nexclass.local .local 도메인은 실제 메일 서버가 없어서 Zoom 초대 메일이 발송 안 됨
고유성 보장 userCd는 LMS에서 유일하니까 이메일도 유일해짐

이게 왜 천재적이냐면

Zoom Registrant API는 이메일을 필수 파라미터로 요구한다. 그런데 학생에게 Zoom 계정도 이메일도 없어. 해결: 가짜 이메일을 만들되 userCd를 인코딩해서 넣는다. 나중에 Reports API로 참여자 조회하면 이메일이 나오고, 거기서 userCd를 역추출하면 끝이야.

코드 분석: RegistrantService¶

registerCourseStudentsAsync() -- 비동기 대량 등록¶

@Slf4j
@Service
@RequiredArgsConstructor
public class RegistrantService {

    private final ZoomService zoomService;
    private final CourseUserRepository courseUserRepository;
    private final UserRepository userRepository;
    private final LessonUserRepository lessonUserRepository;

    @Value("${nexclass.dummy-email-domain}")
    private String dummyEmailDomain;   // nexclass.local

    /**
     * 수강생 전원을 Zoom Registrant로 비동기 등록
     * @param lessonCd      강의 코드
     * @param courseCd      과목 코드 (수강생 목록 조회용)
     * @param zoomMeetingId Zoom 미팅 ID
     */
    @Async  // 별도 스레드에서 실행 → API 응답 지연 없음
    public void registerCourseStudentsAsync(String lessonCd,
            String courseCd, String zoomMeetingId) {

        // ① 해당 과목의 수강생 전원 조회
        List<CourseUser> students =
            courseUserRepository.findByCourseCd(courseCd);
        log.info("[Async] Registrant 등록 시작: lessonCd={}, 수강생={}명",
            lessonCd, students.size());

        int successCount = 0;
        int failCount = 0;

        // ② 각 수강생을 Zoom에 등록
        for (CourseUser student : students) {
            try {
                // 이메일: userCd@nexclass.local
                String email = student.getUserCd() + "@" + dummyEmailDomain;

                // 사용자 정보 조회 (이름, 학번)
                User user = userRepository.findById(
                    student.getUserCd()).orElse(null);
                String userName = user != null && user.getUserName() != null
                    ? user.getUserName() : student.getUserCd();
                String userNo = user != null && user.getUserNo() != null
                    ? user.getUserNo() : student.getUserCd();

                // ③ Zoom Registrant API 호출
                Map<String, Object> result = zoomService.registerParticipant(
                    zoomMeetingId, email, userName, userNo);

                // ④ 개인 joinUrl 추출 + 한국어 설정
                String joinUrl = (String) result.get("join_url");
                if (joinUrl != null && !joinUrl.contains("uiLang")) {
                    joinUrl += (joinUrl.contains("?") ? "&" : "?")
                        + "uiLang=ko-KO";
                }

                // ⑤ LessonUser 테이블에 저장
                LessonUser lessonUser = LessonUser.builder()
                    .lessonCd(lessonCd)
                    .userCd(student.getUserCd())
                    .registrantId((String) result.get("registrant_id"))
                    .joinUrl(joinUrl)
                    .build();
                lessonUserRepository.save(lessonUser);

                successCount++;
            } catch (Exception e) {
                failCount++;
                log.warn("[Async] Registrant 등록 실패: userCd={}, error={}",
                    student.getUserCd(), e.getMessage());
            }
        }
        log.info("[Async] Registrant 등록 완료: lessonCd={}, 성공={}, 실패={}",
            lessonCd, successCount, failCount);
    }
}

@Async가 왜 필요한가?¶

수강생이 30명이면 Zoom API를 30번 호출해야 한다. 각 호출에 200~500ms 걸리면?

30명 x 300ms = 약 9초

9초 동안 LMS가 응답을 기다리게 하면 안 돼. 그래서 비동기로 처리한다.

@Async 주의사항

@Async는 같은 클래스 내부에서 호출하면 동작하지 않는다. 스프링 AOP 프록시가 적용되어야 하는데, 자기 자신의 메서드를 호출하면 프록시를 거치지 않기 때문이야.

그래서 NexClass는 RegistrantService를 별도 서비스 클래스로 분리했어. LessonService에서 registrantService.registerCourseStudentsAsync()를 호출하면 프록시를 통해 @Async가 정상 동작한다.

syncRegistrantsForLesson() -- 수강생 변동 시 재동기화¶

강의를 만든 이후에 수강생이 변동될 수 있다 (수강 추가/취소). 이때 Registrant를 동기화해야 해.

@Transactional
public void syncRegistrantsForLesson(Lesson lesson) {
    String lessonCd = lesson.getLessonCd();
    String courseCd = lesson.getCourseCd();
    String zoomMeetingId = lesson.getZoomMeetingId();

    // 1. 현재 수강생 (LMS 동기화 후 최신)
    List<CourseUser> courseUsers =
        courseUserRepository.findByCourseCd(courseCd);
    Set<String> courseUserCds = courseUsers.stream()
        .map(CourseUser::getUserCd).collect(Collectors.toSet());

    // 2. 기존 Registrant (강의 생성 시 등록된 학생)
    List<LessonUser> lessonUsers =
        lessonUserRepository.findByLessonCd(lessonCd);
    Map<String, LessonUser> lessonUserMap = lessonUsers.stream()
        .collect(Collectors.toMap(LessonUser::getUserCd, lu -> lu));

    // 3. 추가 대상: CourseUser에 있는데 LessonUser에 없는 학생
    Set<String> toAdd = new HashSet<>(courseUserCds);
    toAdd.removeAll(lessonUserMap.keySet());

    // 4. 제거 대상: LessonUser에 있는데 CourseUser에 없는 학생
    Set<String> toRemove = new HashSet<>(lessonUserMap.keySet());
    toRemove.removeAll(courseUserCds);

    // 5. 새 수강생 → Zoom 등록 + DB 저장
    for (String userCd : toAdd) {
        // Zoom registerParticipant() 호출
        // LessonUser 저장
    }

    // 6. 수강 취소 → Zoom 취소 + DB 삭제
    for (String userCd : toRemove) {
        // Zoom cancelRegistrant() 호출
        // LessonUser 삭제
    }
}

Registrant 취소 (cancelRegistrant)¶

// ZoomService.java
public void cancelRegistrant(String meetingId, String registrantId) {
    String token = getAccessToken();

    HttpHeaders headers = new HttpHeaders();
    headers.setContentType(MediaType.APPLICATION_JSON);
    headers.setBearerAuth(token);

    // action: "cancel" → 등록 취소
    Map<String, Object> body = new HashMap<>();
    body.put("action", "cancel");
    body.put("registrants", new Map[]{Map.of("id", registrantId)});

    HttpEntity<Map<String, Object>> request =
        new HttpEntity<>(body, headers);
    String url = baseUrl + "/meetings/" + meetingId + "/registrants/status";

    // PUT으로 상태 변경
    restTemplate.exchange(url, HttpMethod.PUT, request, Void.class);
}

cancel vs deny vs 삭제

Zoom Registrant 상태 변경 API는 action 파라미터로 제어한다:

  • approve: 승인 (auto_approve가 true면 자동이라 안 씀)
  • cancel: 취소 (NexClass가 사용하는 것)
  • deny: 거부

cancel된 학생은 해당 미팅의 고유 joinUrl로 접속해도 입장이 차단된다.

대소문자 주의

equalsIgnoreCase()를 쓴다. 왜? Zoom이 이메일을 반환할 때 대소문자를 바꿀 수 있거든. 등록 시 USR001@nexclass.local이었는데 Reports API에서 usr001@nexclass.local로 올 수 있다. 대소문자 무시 비교를 안 하면 매칭 실패 → 출석 누락이야.

주의사항¶

Registrant 운영 시 주의할 점

1. Registrant API Rate Limit

수강생 등록은 학생 1명당 API 1회 호출이다. 100명짜리 과목이면 100번 호출. Rate Limit에 걸리기 쉽다. @Async로 백그라운드 처리하면서 실패 시 로그만 남기고 넘어가는 이유다.

2. 실패한 등록은 자동 재시도 안 된다

현재 코드에서 등록 실패한 학생은 로그만 남긴다. 그 학생은 개인 joinUrl이 없어서 fallback(공통 join_url)으로 입장한다. 출석 매칭이 안 될 수 있으므로, 실패 시 알림/재시도 메커니즘이 필요하다.

3. 수강생 동기화 타이밍

LMS에서 수강 변동이 일어난 후 syncRegistrantsForLesson()이 실행되어야 새 학생이 Registrant로 등록된다. 동기화 전에 수업이 시작되면 새 학생은 입장 못 하거나 출석 매칭이 안 된다.

4. joinUrl에 uiLang 추가

if (joinUrl != null && !joinUrl.contains("uiLang")) {
    joinUrl += (joinUrl.contains("?") ? "&" : "?") + "uiLang=ko-KO";
}

Zoom 기본 언어가 영어라서 한국어 UI로 강제 전환한다. 이미 uiLang이 포함된 URL에는 중복 추가하지 않는다.

확인 문제¶

1. NexClass에서 이메일 형식을 userCd@nexclass.local로 만든 이유는?

Zoom Registrant API가 이메일을 필수로 요구하지만, 학생에게 Zoom 계정이 없기 때문이다.

해결 전략: 1. 이메일 @ 앞에 userCd를 넣어서 나중에 출석 매칭 시 역추출한다 2. .local 도메인은 실제 메일 서버가 없어서 Zoom 초대 메일이 발송되지 않는다 3. userCd는 LMS에서 유일하므로 이메일도 자동으로 유일해진다

이 설계로 "학생에게 Zoom 계정 없이도 개별 식별 가능한 Registrant 등록"이 가능해졌다.

2. registerCourseStudentsAsync()에 @Async를 붙인 이유와, @Async가 동작하려면 어떤 조건이 필요한가?

이유: 수강생 30명이면 Zoom API를 30번 호출해야 하고, 각 호출에 200~500ms 소요된다. 동기 처리 시 최대 15초까지 LMS 응답이 지연된다. @Async로 백그라운드 처리하면 LMS는 즉시 응답을 받는다.

동작 조건: 1. @EnableAsync 설정이 있어야 한다 (Spring Boot에서 활성화) 2. 다른 클래스에서 호출해야 한다 -- 같은 클래스 내부 호출은 프록시를 거치지 않아 동기로 실행된다 3. 그래서 RegistrantService를 별도 클래스로 분리하고 LessonService에서 호출한다

3. syncRegistrantsForLesson에서 '추가 대상'과 '제거 대상'을 어떻게 계산하는가?

집합 연산(차집합)으로 계산한다.

CourseUser (현재 수강생): {A, B, C, D, E}
LessonUser (기존 등록):  {A, B, C, F}

추가 대상 = CourseUser - LessonUser = {D, E}
  → 새로 수강 신청한 학생 → Zoom 등록 + DB 저장

제거 대상 = LessonUser - CourseUser = {F}
  → 수강 취소한 학생 → Zoom 취소 + DB 삭제

Java 코드에서는 Set.removeAll()로 차집합을 계산한다: 

Set<String> toAdd = new HashSet<>(courseUserCds);
toAdd.removeAll(lessonUserMap.keySet());  // 차집합

4. 출석 매칭에서 equalsIgnoreCase()를 사용하는 이유는?

Zoom API가 이메일의 대소문자를 변경할 수 있기 때문이다.

등록 시: USR001@nexclass.local (대문자) Reports API 반환: usr001@nexclass.local (소문자로 변환될 수 있음)

equals()를 쓰면 대소문자가 다르다고 매칭 실패 → 출석 누락. equalsIgnoreCase()를 쓰면 대소문자를 무시하고 매칭 성공.

이메일은 RFC 5321에 의해 로컬 파트(@ 앞)의 대소문자를 구분하지만, 실무에서는 대부분의 시스템이 대소문자를 무시한다. 방어적 프로그래밍으로 equalsIgnoreCase()가 올바른 선택이다.

5. Registrant 등록이 실패한 학생은 어떻게 되나? 현재 설계의 한계는?

현재 동작: 실패한 학생은 로그만 남기고 넘어간다 (failCount++, log.warn()). 해당 학생은 LessonUser 테이블에 저장되지 않으므로 개인 joinUrl이 없다.

수업 입장 시: LessonService.getJoinUrl()에서 LessonUser를 못 찾으면 공통 join_url에 uname 파라미터를 붙여서 반환한다 (fallback). 입장은 가능하지만 Registrant가 아니라 일반 참가자로 입장한다.

출석 매칭 시: 이메일 매칭이 안 되므로 외부 참가자(externalParticipant: "Y")로 처리된다. LMS 출결에 반영되지 않아 출석 누락이 발생할 수 있다.

현재 한계: - 실패 시 자동 재시도 메커니즘이 없다 - 관리자에게 실패 알림이 가지 않는다 (로그만 있음) - 개선 방안: 실패 학생 목록을 DB에 저장하고, 스케줄러로 재시도하거나 대시보드에서 수동 재시도 기능 제공
