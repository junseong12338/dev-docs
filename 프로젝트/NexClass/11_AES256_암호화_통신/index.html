11. AES-256 암호화 통신¶

핵심 개념: 대칭키 암호화가 뭐냐¶

암호화는 크게 두 가지야:

구분 대칭키 (Symmetric) 비대칭키 (Asymmetric)
키 개수 1개 (암호화 = 복호화 같은 키) 2개 (공개키 + 개인키)
속도 빠르다 느리다 (100~1000배)
대표 알고리즘 AES, DES, 3DES RSA, ECC
핵심 문제 키 교환을 어떻게 하냐 느리다
용도 데이터 암호화 키 교환, 전자서명

비유

대칭키는 같은 열쇠로 잠그고 여는 자물쇠야. 나도 열쇠 있고, 상대방도 같은 열쇠 있으면 둘 다 열 수 있어. 문제는? 열쇠를 어떻게 안전하게 전달하냐는 거야.

AES가 뭐가 특별해?¶

AES(Advanced Encryption Standard)는 미국 정부가 공인한 표준 대칭키 알고리즘이다.

  • 2001년 NIST가 선정 (DES 후계자)
  • 블록 크기: 128비트 (16바이트)
  • 키 길이: 128 / 192 / 256비트

AES-256은 키 길이가 256비트(32바이트)짜리 AES야. 현존하는 대칭키 암호 중 가장 강력한 등급이다.

AES-256이면 안전한 거 아니야?

알고리즘이 강해도 키 관리가 엉망이면 의미 없어. 키를 application.properties에 평문으로 박아두면 서버 털리는 순간 암호화고 뭐고 다 끝이야. AES-256의 보안성은 알고리즘이 아니라 키 관리에 달려있다.

동작 원리: NexClass에서 AES를 왜 쓰나?¶

LMS와 NexClass 사이의 통신 보호¶

sequenceDiagram
    participant NC as NexClass
    participant LMS as LMS (KNU10)

    Note over NC,LMS: NexClass가 학사 데이터를 요청하는 흐름

    NC->>NC: ① JWT 토큰 생성<br/>(apiKey + apiSecret으로 서명)
    NC->>LMS: ② POST /vc/api/sync/term<br/>Authorization: Bearer {JWT}
    LMS->>LMS: ③ JWT 검증 (apiKey, apiSecret 확인)
    LMS->>LMS: ④ DB에서 학기 데이터 조회
    LMS->>LMS: ⑤ JSON을 AES-256으로 암호화<br/>→ Base64 인코딩
    LMS-->>NC: ⑥ 암호화된 문자열 반환
    NC->>NC: ⑦ AES-256 복호화<br/>→ Base64 디코딩 → AES 복호화
    NC->>NC: ⑧ JSON 파싱 → Entity → DB 저장

왜 암호화가 필요한가?

LMS API 응답에는 학생 개인정보(이름, 학번, 이메일)가 포함돼 있다. 네트워크를 통해 평문으로 전송하면 중간에 가로채기(MITM) 당할 수 있어.

"HTTPS 쓰면 되지 않아?"라고 물을 수 있는데, 맞아. HTTPS는 전송 구간을 암호화한다. 하지만 LMS 쪽에서 애플리케이션 레벨 암호화를 추가로 적용한 거야. 이건 이중 보안이지.

계층 암호화 보호 범위
전송 계층 HTTPS (TLS) 클라이언트 ↔ 서버 간 통신
애플리케이션 계층 AES-256 데이터 자체를 암호화 (서버 내부에서도 보호)

코드 분석: AES256Util.java¶

NexClass의 AES 복호화 유틸리티를 줄 단위로 뜯어보자.

public class AES256Util {

    // IV (Initialization Vector): 초기화 벡터
    private final String iv;

    // AES 암호화에 사용할 키 스펙
    private final SecretKeySpec keySpec;

    /**
     * 생성자: 키 문자열로 IV와 KeySpec 초기화
     * @param key -- AES 키 (application.properties의 lms.sync.aes-key)
     */
    public AES256Util(String key) {
        // IV = 키의 앞 16바이트 (AES 블록 크기 = 128비트 = 16바이트)
        this.iv = key.substring(0, 16);

        // 키를 16바이트로 잘라서 사용 (AES-128 수준)
        byte[] keyBytes = new byte[16];
        byte[] b = key.getBytes(StandardCharsets.UTF_8);
        int len = Math.min(b.length, keyBytes.length);
        System.arraycopy(b, 0, keyBytes, 0, len);

        // "AES" 알고리즘용 SecretKeySpec 생성
        this.keySpec = new SecretKeySpec(keyBytes, "AES");
    }

    /**
     * AES-256 복호화
     * LMS 응답(Base64 인코딩된 AES 암호문)을 평문 JSON으로 변환
     */
    public String decrypt(String encrypted) throws GeneralSecurityException {
        // AES/CBC/PKCS5Padding 방식 Cipher 생성
        Cipher cipher = Cipher.getInstance("AES/CBC/PKCS5Padding");

        // 복호화 모드로 초기화 (키 + IV)
        cipher.init(Cipher.DECRYPT_MODE, keySpec,
            new IvParameterSpec(iv.getBytes(StandardCharsets.UTF_8)));

        // Base64 디코딩 → 바이너리 암호문
        byte[] decoded = Base64.getDecoder().decode(encrypted);

        // AES 복호화
        byte[] decrypted = cipher.doFinal(decoded);

        // 바이트 배열 → 문자열 (UTF-8)
        return new String(decrypted, StandardCharsets.UTF_8);
    }
}

CBC 모드가 뭐야?¶

AES는 블록 암호야. 데이터를 16바이트(128비트) 블록으로 쪼개서 암호화한다. 그런데 각 블록을 독립적으로 암호화하면(ECB 모드) 같은 평문 블록은 같은 암호문이 나와서 패턴이 드러나.

CBC(Cipher Block Chaining)는 이전 암호문 블록을 다음 블록 암호화에 섞는 방식이야. 그래서 같은 평문이라도 매번 다른 암호문이 나온다.

flowchart LR
    subgraph "CBC 모드 암호화 흐름"
        IV["IV<br/>(초기화 벡터)"] --> XOR1["XOR"]
        P1["평문 블록 1"] --> XOR1
        XOR1 --> ENC1["AES 암호화"]
        ENC1 --> C1["암호문 블록 1"]

        C1 --> XOR2["XOR"]
        P2["평문 블록 2"] --> XOR2
        XOR2 --> ENC2["AES 암호화"]
        ENC2 --> C2["암호문 블록 2"]

        C2 --> XOR3["XOR"]
        P3["평문 블록 3"] --> XOR3
        XOR3 --> ENC3["AES 암호화"]
        ENC3 --> C3["암호문 블록 3"]
    end
용어 설명
IV (Initialization Vector) 첫 번째 블록에 XOR할 랜덤 값. 같은 키+같은 평문이라도 IV가 다르면 다른 암호문이 된다
XOR 비트 연산. 같으면 0, 다르면 1
PKCS5Padding 마지막 블록이 16바이트 미만이면 패딩(채우기)하는 방식

IV를 키에서 추출하는 것 -- 이게 안전한가?¶

this.iv = key.substring(0, 16);

솔직히 말하면 이건 좋은 방식이 아니야

원래 IV는 매 암호화마다 랜덤으로 생성해야 한다. 키에서 IV를 고정으로 추출하면 같은 평문 → 같은 암호문이 되어 CBC 모드의 장점이 사라진다.

하지만 NexClass에서는 LMS와의 호환성 때문에 이렇게 한다. LMS 쪽 AES256Util도 같은 방식으로 IV를 생성하니까, 양쪽이 동일한 IV를 써야 복호화가 된다.

이건 LMS가 정한 규격이라 NexClass가 맞춘 거야. "왜 이렇게 했냐"가 아니라 "왜 이렇게 해야만 했냐"를 이해해야 해.

키 길이 -- AES-256이라면서 16바이트?¶

byte[] keyBytes = new byte[16];  // 16바이트 = 128비트

이름은 AES-256인데 실제로는 AES-128이야

키를 16바이트(128비트)로 잘라서 쓰기 때문에 실제 암호화 강도는 AES-128이다. AES-256으로 하려면 키가 32바이트(256비트)여야 해.

이것도 LMS 호환 때문이야. LMS의 AES256Util이 이 방식으로 구현되어 있어서 NexClass도 동일하게 맞춘 거다. 클래스 이름이 AES256이지만 실제 키 길이는 128비트.

AES-128도 현재 기술로는 충분히 안전하다. 깨진 적 없어. 다만 이름과 실제가 다른 건 알고 있어야 해.

실전 사용: LmsSyncService에서 AES 복호화¶

AES256Util이 실제로 어떻게 쓰이는지 보자.

// LmsSyncService.java -- callLmsApi() 메서드 내부

// ① LMS API 호출 → 암호화된 문자열 받기
String encrypted = restClient.post()
    .uri(url)
    .header("Authorization", "Bearer " + token)
    .body(requestBody)
    .retrieve()
    .body(String.class);
// encrypted = "dGhpcyBpcyBhbiBlbmNyeXB0ZWQg..." (Base64 인코딩된 AES 암호문)

// ② AES 복호화
AES256Util aes = new AES256Util(aesKey);  // aesKey = "{AES_KEY}"
String json = aes.decrypt(encrypted);
// json = '{"list": [{"termCd": "TERM001", "termName": "2026-1학기", ...}]}'

// ③ JSON 파싱
Map<String, Object> result = objectMapper.readValue(json,
    new TypeReference<Map<String, Object>>() {});

대칭키 vs 비대칭키: 언제 뭘 쓰나?¶

기준 대칭키 (AES) 비대칭키 (RSA)
속도 빠르다 느리다 (100~1000배)
키 관리 양쪽이 같은 키 공유 필요 공개키는 공개, 개인키만 비밀
키 교환 안전한 채널 필요 (문제!) 공개키 배포로 해결
데이터 크기 제한 없음 키 크기에 의존 (보통 작은 데이터만)
NexClass 용도 LMS 응답 데이터 암호화 사용 안 함

실무에서는 둘을 섞어 쓴다 (하이브리드 암호화):

  1. RSA로 AES 키를 안전하게 교환
  2. AES로 실제 데이터를 암호화

HTTPS가 정확히 이 방식이야. TLS 핸드셰이크에서 RSA/ECDHE로 세션 키를 교환하고, 이후 통신은 AES로 한다.

NexClass-LMS 사이에서는? 키를 application.properties에 직접 박아뒀으니까 키 교환 문제가 없어. 배포할 때 양쪽 서버에 같은 키를 넣는 거지. 이건 사전 공유 키(Pre-Shared Key) 방식이야.

주의사항¶

실무에서 AES 쓸 때 반드시 지킬 것

1. IV 재사용 금지 (원칙적으로)

같은 키 + 같은 IV로 다른 평문을 암호화하면 XOR 분석으로 평문 추출이 가능하다. NexClass는 LMS 호환 때문에 고정 IV를 쓰지만, 새로 설계할 때는 절대 이러면 안 된다.

2. 키를 소스코드에 하드코딩하지 마라

# 현재 방식 (개발 편의)
lms.sync.aes-key={AES_KEY}

프로덕션에서는 환경 변수, Vault, AWS KMS 등으로 키를 관리해야 한다.

3. 에러 메시지로 암호화 정보를 노출하지 마라

"AES 복호화 실패: Invalid padding" 같은 에러 메시지는 Padding Oracle Attack의 단서가 된다. 에러 메시지는 일반적인 "복호화 실패"로만 반환해야 한다.

4. ECB 모드 절대 쓰지 마라

Cipher.getInstance("AES") 이렇게만 쓰면 기본이 ECB야. 반드시 "AES/CBC/PKCS5Padding" 또는 "AES/GCM/NoPadding"을 명시해라.

정리¶

개념 한 줄 요약
AES 미국 표준 대칭키 블록 암호 알고리즘
AES-256 256비트(32바이트) 키를 사용하는 AES (가장 강력한 등급)
CBC 모드 이전 블록 암호문을 다음 블록에 XOR하여 패턴을 숨기는 방식

PKCS5Padding 마지막 블록을 16바이트로 채우는 패딩 방식
Base64 바이너리를 텍스트로 변환하는 인코딩 (암호화 아님!)
NexClass 용도 LMS API 응답(학생 개인정보 포함)을 AES로 암호화하여 전송

확인 문제¶

1. NexClass에서 LMS API 응답을 AES로 암호화하는 이유는? HTTPS로 충분하지 않나?

HTTPS는 전송 구간만 보호하고, AES는 데이터 자체를 보호하는 것이다.

HTTPS(TLS)는 클라이언트~서버 사이의 네트워크 구간을 암호화한다. 하지만 서버 내부에서 로그에 찍히거나, 프록시를 거치거나, 응답이 캐싱되면 평문이 노출될 수 있다.

애플리케이션 레벨 AES 암호화는 데이터 자체를 보호하므로 네트워크 밖에서도 암호문 상태를 유지한다. 이중 방어(Defense in Depth)다.

2. AES256Util에서 IV를 키의 앞 16바이트로 추출하는 것의 문제점과, 그럼에도 이렇게 한 이유는?

문제점: IV가 고정이면 같은 키+같은 평문에 대해 항상 같은 암호문이 나온다. CBC 모드의 핵심 장점(같은 평문이라도 IV가 다르면 다른 암호문)이 사라진다. 이론적으로 암호문 패턴 분석이 가능해진다.

이렇게 한 이유: LMS의 기존 AES256Util이 이 방식으로 구현되어 있다. NexClass는 LMS의 암호문을 복호화해야 하므로 동일한 키+IV 추출 방식을 사용할 수밖에 없다. 호환성 제약에 의한 트레이드오프다.

3. CBC와 ECB 모드의 차이를 설명하고, ECB가 위험한 이유는?

ECB (Electronic Codebook): 각 블록을 독립적으로 암호화한다. 같은 평문 블록은 항상 같은 암호문 블록이 된다. 이미지를 ECB로 암호화하면 원본의 윤곽이 그대로 보인다 (유명한 Linux 펭귄 예시).

CBC (Cipher Block Chaining): 이전 블록의 암호문을 다음 블록의 평문에 XOR한 후 암호화한다. 첫 블록은 IV와 XOR한다. 같은 평문이라도 위치에 따라 다른 암호문이 나온다.

ECB가 위험한 이유는 패턴이 보존되기 때문이다. 같은 데이터가 반복되면 암호문에서도 그 반복이 드러나서 공격자가 구조를 유추할 수 있다.

4. 대칭키 암호화의 가장 큰 약점은 무엇이며, NexClass는 이를 어떻게 해결했나?

가장 큰 약점: 키 교환 문제. 암호화와 복호화에 같은 키를 써야 하므로, 양쪽이 안전하게 키를 공유해야 한다. 네트워크로 키를 보내면 그 키 자체가 탈취될 수 있다.

NexClass의 해결 방식: 사전 공유 키(Pre-Shared Key). application.properties에 키를 직접 넣어두고, 배포 시 양쪽 서버에 동일한 키를 설정한다. 키가 네트워크를 통해 전송되지 않으므로 키 교환 문제가 발생하지 않는다. 단, 서버가 해킹되면 키가 노출되므로 프로덕션에서는 Vault 등의 비밀 관리 도구를 사용해야 한다.
