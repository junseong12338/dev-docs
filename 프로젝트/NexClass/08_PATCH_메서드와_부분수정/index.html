08. PATCH 메서드와 부분수정¶

이게 뭐야?¶

강의 이름만 바꾸고 싶은데, 강의 시작시간, 종료시간, 교수자코드, 주차, 차시... 나머지 필드 10개를 전부 다시 보내야 한다면?

그게 PUT이야. 전체 교체. 안 보낸 필드는 null로 덮어써짐.

PATCH는 바꿀 필드만 보내면 돼. 나머지는 기존값 유지.

핵심 한줄

PUT = 전체 교체, PATCH = 부분 수정. 바꿀 것만 보내고 나머지는 건드리지 마.

어떻게 돌아가?¶

PUT vs PATCH 비교¶

graph LR
    subgraph "PUT (전체 교체)"
        direction TB
        PUT_REQ["요청: { name: '수정됨',<br/>startDate: '2026-03-23T09:00',<br/>endDate: '2026-03-23T10:00',<br/>totalTime: 60,<br/>profCd: 'PROF001',<br/>... 모든 필드 }"]
        PUT_DB["DB: 요청 데이터로<br/>전체 교체"]
    end

    subgraph "PATCH (부분 수정)"
        direction TB
        PATCH_REQ["요청: { name: '수정됨' }<br/>(이것만 보냄)"]
        PATCH_DB["DB: name만 바뀜<br/>나머지는 기존값 유지"]
    end
구분 PUT PATCH
의미 자원 전체 교체 자원 일부 수정
보내는 데이터 모든 필드 수정할 필드만
안 보낸 필드 null/기본값으로 덮어씀 기존값 유지
멱등성 멱등 비멱등 (상황에 따라)
NexClass 사용 안 씀 강의 수정, 녹화 설정

구체적인 차이¶

PUT /v2/api/lessons/LESSON_001

{
    "lessonName": "수정된 강의명",
    "courseCd": "CS101",
    "lessonWeek": 3,
    "lessonOrder": 1,
    "lessonStartDate": "2026-03-23T09:00:00",
    "lessonEndDate": "2026-03-23T10:00:00",
    "lessonTotalTime": 60,
    "lessonProfCd": "PROF001",
    "lessonType": "COURSE",
    "lmsOrgCd": "ORG001"
}
강의명 하나 바꾸려고 필드 10개를 다 보내야 해. 하나라도 빼먹으면 null로 덮어써짐.

PATCH /v2/api/lessons/LESSON_001

{
    "lessonName": "수정된 강의명"
}
바꿀 것만 보내면 끝. 나머지 필드는 DB에 있는 기존값 그대로 유지.

코드로 보자¶

NexClass에서 PATCH를 쓰는 곳¶

NexClass에서 PATCH를 쓰는 API는 2개야:

API 경로 목적
강의 수정 PATCH /v2/api/lessons/{lessonCd} 강의 정보 부분 수정
녹화 설정 PATCH /v2/api/lessons/{lessonCd}/records/{recordCd} 녹화 공개/비공개 변경

1. 강의 수정 -- updateLesson()¶

Controller (요청 받는 곳)

// LessonController.java
@PatchMapping("/{lessonCd}")
public Map<String, Object> updateLesson(
        @PathVariable String lessonCd,
        @RequestBody Lesson lesson) {
    Lesson updated = lessonService.updateLesson(lessonCd, lesson);
    return buildVcLessonResponse(updated);
}

Service (핵심 로직)

// LessonService.java
@Transactional
public Lesson updateLesson(String lessonCd, Lesson updateData) {
    log.info("강의 수정 요청: lessonCd={}", lessonCd);

    // 1단계: 기존 데이터 조회
    Lesson lesson = getLesson(lessonCd);

    // 2단계: null이 아닌 필드만 업데이트 (PATCH 핵심!)
    if (updateData.getLessonName() != null) {
        lesson.setLessonName(updateData.getLessonName());
    }
    if (updateData.getLessonSubject() != null) {
        lesson.setLessonSubject(updateData.getLessonSubject());
    }
    if (updateData.getLessonStartDate() != null) {
        lesson.setLessonStartDate(updateData.getLessonStartDate());
    }
    if (updateData.getLessonEndDate() != null) {
        lesson.setLessonEndDate(updateData.getLessonEndDate());
    }
    if (updateData.getLessonTotalTime() != null) {
        lesson.setLessonTotalTime(updateData.getLessonTotalTime());
    }

    // 3단계: Zoom 미팅도 같이 수정
    if (lesson.getZoomMeetingId() != null) {
        try {
            zoomService.updateMeeting(
                    lesson.getZoomMeetingId(),
                    lesson.getLessonName(),
                    lesson.getLessonStartDate() != null
                        ? lesson.getLessonStartDate()
                            .format(DateTimeFormatter.ISO_LOCAL_DATE_TIME)
                        : null,
                    lesson.getLessonTotalTime() != null
                        ? lesson.getLessonTotalTime() : 0
            );
            lesson.setSyncStatus("SUCCESS");
        } catch (Exception e) {
            log.error("Zoom 미팅 수정 실패: {}", e.getMessage());
            lesson.setSyncStatus("FAIL");
            lesson.setSyncErrorMsg(e.getMessage());
        }
    }

    // 4단계: DB 저장
    return lessonRepository.save(lesson);
}

PATCH 로직의 핵심 패턴

if (updateData.getXxx() != null) {
    lesson.setXxx(updateData.getXxx());
}
이 패턴이 전부야. 클라이언트가 보낸 필드(null이 아닌 것)만 기존 엔티티에 덮어쓰는 것. 안 보낸 필드는 JSON 파싱 시 null이 되니까 if문에서 걸러짐.

2. 녹화 설정 수정 -- updateRecording()¶

Controller

// LessonController.java
@PatchMapping("/{lessonCd}/records/{recordCd}")
public Map<String, Object> updateRecording(
        @PathVariable String lessonCd,
        @PathVariable String recordCd,
        @RequestBody Map<String, String> body) {

    // LMS는 recordViewYn으로 보냄, 호환을 위해 viewYn도 체크
    String viewYn = body.get("recordViewYn");
    if (viewYn == null) viewYn = body.get("viewYn");

    Recording updated = lessonService.updateRecording(lessonCd, recordCd, viewYn);
    // ...응답 생성
}

Service

// LessonService.java
@Transactional
public Recording updateRecording(String lessonCd, String recordCd, String viewYn) {
    log.info("녹화 설정 수정: lessonCd={}, recordCd={}, viewYn={}",
             lessonCd, recordCd, viewYn);

    // 1. 녹화 조회
    Recording recording = recordingRepository.findByRecordCd(recordCd)
            .orElseThrow(() -> new RuntimeException(
                "녹화를 찾을 수 없습니다: " + recordCd));

    // 2. 소속 강의 확인 (보안 체크)
    if (!recording.getLessonCd().equals(lessonCd)) {
        throw new RuntimeException("강의 코드가 일치하지 않습니다.");
    }

    // 3. viewYn만 수정
    recording.setViewYn(viewYn);

    // 4. 저장
    return recordingRepository.save(recording);
}

녹화 수정에서 소속 강의 확인이 왜 필요해?

URL이 /v2/api/lessons/{lessonCd}/records/{recordCd}잖아. 누군가 lessonCd를 다른 강의 코드로 바꿔서 요청하면? 다른 강의의 녹화 설정을 몰래 바꿀 수 있어.

그래서 recordCd로 녹화를 찾은 다음, 그 녹화의 lessonCd가 URL의 lessonCd와 일치하는지 확인하는 거야. 일치 안 하면 에러.

HttpClient5로 PATCH 지원한 이유¶

Spring RestTemplate의 PATCH 문제¶

NexClass가 Zoom API를 호출할 때도 PATCH를 써야 해. 근데 문제가 있어.

Spring RestTemplate은 PATCH를 제대로 지원 안 한다

Java의 기본 HttpURLConnection은 GET, POST, PUT, DELETE, HEAD, OPTIONS, TRACE만 지원해. PATCH가 없어.

RestTemplate으로 PATCH 요청을 보내면: 

java.net.ProtocolException: Invalid HTTP method: PATCH
이 에러가 터져.

해결 방법: Apache HttpClient5¶

그래서 NexClass는 HTTP 클라이언트를 Apache HttpClient5로 교체했어.

<!-- pom.xml -->
<dependency>
    <groupId>org.apache.httpcomponents.client5</groupId>
    <artifactId>httpclient5</artifactId>
</dependency>

Apache HttpClient5는 PATCH 메서드를 완벽히 지원하거든.

graph LR
    subgraph "RestTemplate 기본"
        RT[RestTemplate] --> HUC[HttpURLConnection]
        HUC -->|PATCH| FAIL["ProtocolException!"]
    end

    subgraph "HttpClient5 교체 후"
        RT2[RestTemplate] --> HC5[Apache HttpClient5]
        HC5 -->|PATCH| OK["정상 동작"]
    end

어디서 PATCH가 필요해?¶

NexClass → Zoom API 호출 시:

Zoom API HTTP 메서드 용도
미팅 수정 PATCH 미팅 제목, 시간 변경
미팅 생성 POST 새 미팅 생성
미팅 삭제 DELETE 미팅 삭제
미팅 참가자 조회 GET 참가자 목록

Zoom API가 미팅 수정에 PATCH를 쓰니까, NexClass도 PATCH를 보낼 수 있어야 해. RestTemplate의 기본 HTTP 구현체가 PATCH를 못 보내니까 Apache HttpClient5로 교체한 거야.

부분 수정 패턴 심화¶

null 체크 방식의 한계¶

현재 NexClass가 쓰는 방식:

if (updateData.getLessonName() != null) {
    lesson.setLessonName(updateData.getLessonName());
}

이 방식의 문제가 뭘까?

"의도적으로 null로 만들고 싶을 때"를 구분 못 해.

  • 클라이언트가 "lessonSubject": null을 보내면? → "null로 비워줘"라는 의미
  • 클라이언트가 lessonSubject 필드를 안 보내면? → "기존값 유지해줘"라는 의미

둘 다 Java에서는 updateData.getLessonSubject() == null이야. 구분이 안 돼.

더 나은 방법들¶

// Controller에서 Map으로 받으면 "키 존재 여부"로 구분 가능
@PatchMapping("/{lessonCd}")
public Map<String, Object> updateLesson(
        @PathVariable String lessonCd,
        @RequestBody Map<String, Object> updates) {

    if (updates.containsKey("lessonName")) {
        // 키가 있으면 → 값을 바꾸겠다는 의미 (null 포함)
        lesson.setLessonName((String) updates.get("lessonName"));
    }
    // 키가 없으면 → 건드리지 않음
}

// Request DTO에 Optional 사용 (Jackson 설정 필요)
public class LessonUpdateRequest {
    private Optional<String> lessonName;      // 있으면 변경
    private Optional<String> lessonSubject;   // 없으면 기존 유지
}

// null = 기존값 유지, 값 있으면 = 변경
// "의도적으로 null로 만들기"는 지원 안 함
// → NexClass에서는 이게 필요 없으니까 충분
if (updateData.getLessonName() != null) {
    lesson.setLessonName(updateData.getLessonName());
}

NexClass에서 현재 방식이 충분한 이유

LMS가 보내는 PATCH 요청에서 "이 필드를 null로 비워줘"라는 케이스가 없어. 강의명, 시작시간, 종료시간은 항상 값이 있어야 하거든. 그래서 "null이면 기존값 유지" 패턴만으로 충분해. YAGNI(You Ain't Gonna Need It).

PATCH와 Zoom 미팅 동기화¶

강의 수정 시 DB만 바꾸면 안 돼. Zoom 미팅도 같이 바꿔야 해.

sequenceDiagram
    participant LMS
    participant NC as NexClass
    participant DB as MariaDB
    participant Zoom

    LMS->>NC: PATCH /v2/api/lessons/L001<br/>{ "lessonName": "수정됨", "lessonTotalTime": 90 }

    NC->>DB: SELECT * FROM TB_NEXCLASS_LESSON WHERE LESSON_CD = 'L001'
    DB-->>NC: 기존 Lesson 데이터

    Note over NC: 필드 업데이트<br/>lessonName: "원래이름" → "수정됨"<br/>lessonTotalTime: 60 → 90

    NC->>Zoom: PATCH /meetings/{meetingId}<br/>{ "topic": "수정됨", "duration": 90 }

    alt Zoom 성공
        Zoom-->>NC: 200 OK
        NC->>NC: syncStatus = "SUCCESS"
    else Zoom 실패
        Zoom-->>NC: 에러
        NC->>NC: syncStatus = "FAIL"
        NC->>NC: syncErrorMsg = 에러메시지
    end

    NC->>DB: UPDATE TB_NEXCLASS_LESSON SET ...
    NC-->>LMS: { status: 200, datas: { serviceSyncStatus: "DONE" } }

Zoom 수정 실패해도 DB는 저장돼

코드를 보면 Zoom API 실패가 catch 블록에서 처리되고, syncStatus만 FAIL로 바뀔 뿐 DB 저장은 진행돼.

} catch (Exception e) {
    log.error("Zoom 미팅 수정 실패: {}", e.getMessage());
    lesson.setSyncStatus("FAIL");
    lesson.setSyncErrorMsg(e.getMessage());
    // return도 throw도 안 함 → 아래 save()로 진행
}
return lessonRepository.save(lesson);  // FAIL이어도 저장

이건 의도적인 설계야. Zoom이 일시적으로 안 되더라도 DB에는 변경 사항을 기록해두고, 나중에 재동기화할 수 있게 하는 거지. syncStatus로 동기화 실패 여부를 추적할 수 있어.

함정¶

함정 1: PATCH인데 전체 데이터를 보내는 LMS

// LMS가 보내는 PATCH 요청 (실제)
{
    "lessonCd": "LESSON_001",
    "courseCd": "CS101",
    "lessonName": "수정된 이름",
    "lessonSubject": "수정된 주제",
    "lessonWeek": 3,
    "lessonOrder": 1,
    "lessonStartDate": "2026-03-23T09:00:00",
    "lessonEndDate": "2026-03-23T10:00:00",
    "lessonTotalTime": 90,
    "lessonProfCd": "PROF001"
}
LMS가 PATCH를 쓰면서도 전체 필드를 다 보내. 이건 LMS 구현이 PUT처럼 동작하는 거야. 그래도 NexClass 쪽에서 null 체크를 해두면, 혹시 LMS가 일부 필드만 보내는 경우도 대응 가능해.

함정 2: @RequestBody에 Entity를 직접 받는 것

// NexClass 현재 코드
@PatchMapping("/{lessonCd}")
public Map<String, Object> updateLesson(
        @PathVariable String lessonCd,
        @RequestBody Lesson lesson) { ... }
Entity를 직접 @RequestBody로 받으면 위험해: - 클라이언트가 zoomMeetingId, syncStatus, hostEmail 같은 내부 필드를 보내면? - updateData에 해당 값이 들어오고, null 체크 로직이 있으면 덮어써질 수 있어

더 안전한 방법은 수정 가능한 필드만 가진 DTO를 만드는 거야: 

// 이게 더 안전해
public class LessonUpdateRequest {
    private String lessonName;
    private String lessonSubject;
    private LocalDateTime lessonStartDate;
    private LocalDateTime lessonEndDate;
    private Integer lessonTotalTime;
    // zoomMeetingId, syncStatus 같은 내부 필드는 없음!
}

함정 3: Integer vs int

// ❌ int는 null이 안 돼 → 기본값 0이 들어와
if (updateData.lessonTotalTime != 0) {  // 0분 강의는 수정 안 되는 버그!
    lesson.setLessonTotalTime(updateData.lessonTotalTime);
}

// ✅ Integer는 null 가능 → null 체크로 "안 보냄" 판단 가능
if (updateData.getLessonTotalTime() != null) {
    lesson.setLessonTotalTime(updateData.getLessonTotalTime());
}
PATCH 로직에서 원시 타입(int, boolean)을 쓰면 "안 보냄"과 "0/false를 보냄"을 구분 못 해. 그래서 NexClass의 Lesson 엔티티는 Integer lessonTotalTime으로 Wrapper 타입을 써.

정리¶

핵심 요약표¶

개념 설명
PUT 전체 교체 -- 모든 필드를 보내야 함
PATCH 부분 수정 -- 수정할 필드만 보내면 됨
null 체크 패턴 if (값 != null) → 보낸 필드만 업데이트
Wrapper 타입 Integer, Boolean 사용 → null로 "미전송" 표현
HttpClient5 RestTemplate의 PATCH 미지원 문제 해결
동기화 DB 수정 + Zoom API 수정을 같이 해야 함

한줄 정리¶

PATCH는 "보낸 것만 바꾸고 나머지는 건드리지 마." null 체크로 구현하고, Wrapper 타입을 써야 원시 타입과 "미전송"을 구분할 수 있다.

확인문제¶

Q1. PUT과 PATCH의 가장 큰 차이를 한 문장으로 설명해봐.

PUT은 자원 전체를 교체하고, PATCH는 자원의 일부만 수정한다.

PUT으로 강의 수정하면 안 보낸 필드가 null로 덮어써지고, PATCH로 수정하면 안 보낸 필드는 기존값이 유지돼.

Q2. NexClass의 updateLesson()에서 if (updateData.getLessonName() != null) 패턴이 하는 일은?

클라이언트가 보낸 필드(null이 아닌 것)만 기존 엔티티에 덮어쓰고, 안 보낸 필드(null인 것)는 기존값을 유지하는 것.

JSON에서 필드를 안 보내면 Jackson이 해당 필드를 null로 놔두거든. 그래서 null이면 "이 필드는 수정 안 하겠다"라고 해석하는 거야.

// 요청: { "lessonName": "수정됨" }
// → updateData.getLessonName() = "수정됨" (보냄 → 업데이트)
// → updateData.getLessonSubject() = null (안 보냄 → 건드리지 않음)
Q3. Spring RestTemplate에서 PATCH가 안 되는 이유와 해결 방법은?

원인: Java의 기본 HTTP 구현체(HttpURLConnection)가 PATCH 메서드를 지원하지 않아. GET, POST, PUT, DELETE, HEAD, OPTIONS, TRACE만 허용하고 PATCH는 ProtocolException을 던져.

해결: HTTP 구현체를 Apache HttpClient5로 교체. 

<dependency>
    <groupId>org.apache.httpcomponents.client5</groupId>
    <artifactId>httpclient5</artifactId>
</dependency>
Apache HttpClient5는 PATCH를 완전히 지원하니까, RestTemplate의 백엔드로 쓰면 PATCH 요청이 정상 동작해.

Q4. PATCH 수정에서 Entity를 직접 @RequestBody로 받으면 어떤 보안 문제가 생길 수 있어?

클라이언트가 내부 전용 필드를 조작할 수 있어.

Lesson 엔티티에는 zoomMeetingId, syncStatus, hostEmail 같은 서버 내부에서만 관리하는 필드가 있어. Entity를 직접 @RequestBody로 받으면 클라이언트가 이 필드들을 요청에 포함시킬 수 있어:

{
    "lessonName": "수정됨",
    "syncStatus": "SUCCESS",
    "hostEmail": "hacker@evil.com"
}

현재 NexClass 코드에서는 updateLesson() 메서드가 syncStatus를 직접 세팅하는 로직이 있어서 덮어써지지만, 다른 필드에 대해서는 보호가 약해. 수정 가능한 필드만 가진 DTO를 별도로 만드는 게 안전해.

Q5. Integer 대신 int를 쓰면 PATCH 로직에서 어떤 문제가 생겨?

"필드를 안 보냄"과 "0을 보냄"을 구분할 수 없어.

  • int 타입은 null이 불가능하고, JSON에서 안 보내면 기본값 0이 들어와
  • Integer 타입은 null이 가능하고, JSON에서 안 보내면 null이 들어와

예시: 

// int lessonTotalTime인 경우
// 요청: { "lessonName": "수정" }  → lessonTotalTime = 0 (기본값)
// 요청: { "lessonTotalTime": 0 }  → lessonTotalTime = 0 (의도적 0)
// 두 경우가 구분 안 돼!

// Integer lessonTotalTime인 경우
// 요청: { "lessonName": "수정" }  → lessonTotalTime = null (미전송)
// 요청: { "lessonTotalTime": 0 }  → lessonTotalTime = 0 (의도적 0)
// null과 0으로 구분 가능!
