22. 녹화 관리 시스템¶

핵심 개념¶

이 챕터의 핵심

NexClass 녹화 시스템은 Zoom 클라우드 녹화를 기반으로 한다. 미팅이 끝나면 Zoom이 녹화 파일을 인코딩하고, 완료되면 recording.completed 웹훅을 보낸다. NexClass는 이 웹훅을 받아서 녹화 URL을 DB에 저장한다. 학생은 비밀번호 입력 없이 바로 녹화를 볼 수 있다. 그게 핵심이야.

녹화 처리 전체 흐름¶

sequenceDiagram
    participant Prof as 교수 (Zoom)
    participant Zoom as Zoom Platform
    participant WH as WebhookController
    participant AS as AttendanceService
    participant ZS as ZoomService
    participant DB as NexClass DB
    participant Stu as 학생

    Prof->>Zoom: 미팅 종료
    Note over Zoom: 클라우드 녹화 인코딩 (수 분 소요)
    Zoom->>WH: POST /api/webhook (recording.completed)
    WH->>WH: HMAC-SHA256 서명 검증
    WH-->>Zoom: 200 OK

    Note over WH,AS: @Async 백그라운드 처리

    WH->>AS: processRecording(meetingId, object)
    AS->>DB: meetingId로 Lesson 조회
    AS->>ZS: getMeetingRecordings(meetingId)
    ZS->>Zoom: GET /v2/meetings/{id}/recordings
    Zoom-->>ZS: share_url, recording_play_passcode, password
    ZS-->>AS: 녹화 데이터

    loop 각 녹화 파일
        AS->>AS: MP4만 필터 (오디오/채팅 제외)
        AS->>DB: 중복 체크 (recordCd)
        AS->>AS: share_url + ?pwd=passcode 결합
        AS->>DB: Recording 엔티티 저장
    end

    Stu->>DB: 녹화 URL 요청
    DB-->>Stu: share_url?pwd=passcode (비밀번호 없이 재생)

출결(meeting.ended)과 녹화(recording.completed)는 별개 이벤트

미팅이 끝나면 meeting.ended가 즉시 온다. 근데 recording.completed는 Zoom이 녹화 파일을 인코딩하는 시간이 필요해서 수 분~수십 분 후에 온다.

그래서 두 이벤트를 각각 처리하는 거야. 출결 처리 끝났는데 녹화가 아직 안 왔을 수 있다. 정상이야.

processRecording() 코드 분석¶

엔트리 포인트¶

@Async
@Transactional
public void processRecording(String meetingId, Map<String, Object> object) {
    log.info("[RECORD] ===== 녹화 처리 시작 =====");

    // 1. meetingId로 Lesson 찾기
    Optional<Lesson> lessonOpt = lessonRepository.findByZoomMeetingId(meetingId);
    if (lessonOpt.isEmpty()) {
        log.warn("[RECORD] 매칭되는 강의 없음: meetingId={}", meetingId);
        return;  // NexClass에 등록 안 된 미팅이면 무시
    }
    String lessonCd = lessonOpt.get().getLessonCd();

    // 2. 웹훅에서 녹화 파일 목록 추출
    List<Map<String, Object>> recordingFiles =
        (List<Map<String, Object>>) object.get("recording_files");
    if (recordingFiles == null || recordingFiles.isEmpty()) {
        log.info("[RECORD] 녹화 파일 없음: meetingId={}", meetingId);
        return;
    }
    log.info("[RECORD] 녹화 파일 {}개", recordingFiles.size());

    // ...이하 녹화 저장 로직
}

여기서는 @Async + @Transactional이 같이 붙어 있다

출결 처리에서는 분리했는데 녹화에서는 왜 같이?

출결은 트랜잭션 커밋 후 LMS Push가 필요해서 분리했다. 녹화는 Push가 없다. DB 저장만 하면 끝이야. 트랜잭션 분리할 이유가 없으니까 같이 붙여도 된다.

Zoom Recordings API 호출¶

웹훅 데이터만으로는 정보가 부족하다. Recordings API를 추가 호출해야 한다.

// Zoom Recordings API 호출 - share_url과 패스코드 가져오기
Map<String, Object> recordingsData = null;
try {
    recordingsData = zoomService.getMeetingRecordings(meetingId);
    log.info("[RECORD] Recordings API 호출 성공");
} catch (Exception e) {
    log.warn("[RECORD] Recordings API 호출 실패, 웹훅 데이터 사용: {}",
             e.getMessage());
}

// 패스코드 추출
String password = (String) object.get("password");  // 웹훅 기본 비밀번호
String shareUrl = null;
String playPasscode = null;

if (recordingsData != null) {
    // API 응답에서 더 정확한 데이터 가져오기
    if (recordingsData.get("password") != null) {
        password = (String) recordingsData.get("password");
    }

    // recording_play_passcode: URL에 붙이는 인코딩된 패스코드
    if (recordingsData.get("recording_play_passcode") != null) {
        playPasscode = (String) recordingsData.get("recording_play_passcode");
    }

    // share_url: 최상위에 있음 (개별 파일이 아님!)
    if (recordingsData.get("share_url") != null) {
        shareUrl = (String) recordingsData.get("share_url");
    }
}

웹훅 데이터 vs Recordings API 데이터

데이터 웹훅 Recordings API
recording_files O O
password (평문) O O
share_url X O
recording_play_passcode X O

share_url과 recording_play_passcode는 Recordings API에서만 얻을 수 있다. 웹훅만으로는 비밀번호 없이 재생하는 URL을 만들 수 없어. 그래서 API를 한 번 더 호출하는 거야.

MP4 필터링과 중복 체크¶

for (Map<String, Object> file : recordingFiles) {

    // ── MP4(영상)만 저장. 오디오/채팅/타임라인 제외 ──
    String fileType = (String) file.get("file_type");
    String recordingType = (String) file.get("recording_type");
    if (fileType != null && !"MP4".equalsIgnoreCase(fileType)) {
        log.info("[RECORD] 비영상 파일 스킵: file_type={}, recording_type={}",
                 fileType, recordingType);
        continue;  // MP4 아니면 건너뜀
    }

    // ── 중복 저장 방지 ──
    String recordCd = (String) file.get("id");
    if (recordingRepository.findByRecordCd(recordCd).isPresent()) {
        log.info("[RECORD] 이미 저장된 녹화 스킵: recordCd={}", recordCd);
        continue;  // 이미 있으면 건너뜀
    }

    // ...이하 저장 로직
}

왜 MP4만 저장하나?

Zoom 클라우드 녹화는 여러 파일을 생성한다.

file_type 내용 저장 여부
MP4 영상 (화면 + 스피커) 저장
M4A 오디오만 스킵
CHAT 채팅 로그 스킵
TIMELINE 타임라인 메타데이터 스킵

학생이 다시보기하는 건 영상이다. 오디오만 따로 필요한 경우는 없다. 채팅 로그는 필요하면 나중에 추가할 수 있지만, 지금은 YAGNI.

중복 저장은 왜 막아야 하나?

Zoom이 웹훅을 재전송할 수 있다. 네트워크 문제로 200 OK를 못 받으면 같은 웹훅을 다시 보낸다.

recordCd (Zoom 녹화 파일 ID)로 중복 체크하지 않으면 같은 녹화가 여러 번 저장된다.

이걸 멱등성(Idempotency) 처리라고 한다. 같은 요청을 여러 번 처리해도 결과가 같아야 한다는 원칙이야.

녹화 시간 계산¶

// 녹화 시간 계산 (recording_start → recording_end)
Integer duration = null;
String recordingStart = (String) file.get("recording_start");
String recordingEnd = (String) file.get("recording_end");

if (recordingStart != null && recordingEnd != null) {
    try {
        LocalDateTime start = LocalDateTime.parse(recordingStart,
            DateTimeFormatter.ISO_DATE_TIME);
        LocalDateTime end = LocalDateTime.parse(recordingEnd,
            DateTimeFormatter.ISO_DATE_TIME);
        duration = (int) java.time.Duration.between(start, end).getSeconds();
    } catch (Exception e) {
        log.warn("[RECORD] 녹화 시간 파싱 실패: start={}, end={}",
                 recordingStart, recordingEnd);
    }
}

왜 recording_start/recording_end에서 직접 계산하나?

웹훅 데이터에 duration 필드가 없는 경우가 있다. 있어도 단위가 다를 수 있다 (분 vs 초). 직접 계산하면 단위 혼동 없이 초 단위로 통일할 수 있다.

playUrlWithPasscode 자동 구성¶

이게 이 챕터에서 가장 중요한 부분이야.

// share_url + 패스코드 토큰 결합 → 비밀번호 입력 없이 재생
String recordUrl;
if (shareUrl != null && playPasscode != null) {
    // 최선: share_url + ?pwd=인코딩된패스코드
    recordUrl = shareUrl + "?pwd=" + playPasscode;
} else if (shareUrl != null) {
    // 차선: share_url만 (비밀번호 입력 필요)
    recordUrl = shareUrl;
} else {
    // 최후: play_url (인증 필요할 수 있음)
    recordUrl = (String) file.get("play_url");
}

https://zoom.us/rec/share/xxxxxxxxxxxx?pwd=EnCoDeDpAsScOdE

학생이 이 URL을 클릭하면 비밀번호 입력 없이 바로 녹화가 재생된다. recording_play_passcode는 URL에 사용할 수 있도록 인코딩된 버전이다. 평문 password와는 다르다.

https://zoom.us/rec/share/xxxxxxxxxxxx

비밀번호 입력 화면이 나온다. 학생이 비밀번호를 알아야 한다. UX가 안 좋다.

https://zoom.us/rec/play/xxxxx

Zoom 인증이 필요할 수 있다. 학생이 Zoom 계정이 없으면 못 본다. 최후의 수단이야.

왜 3단계 폴백이 필요하나?

Recordings API가 실패할 수 있다. 실패하면 shareUrl과 playPasscode가 null이다.

그래도 웹훅 데이터에는 play_url이 있다. "최선이 안 되면 차선, 차선도 안 되면 최후의 수단" - 이게 방어적 프로그래밍이야.

Recording 엔티티 저장¶

// recordPassword에 playPasscode 저장 (URL용 인코딩된 패스코드)
String storedPassword = playPasscode != null ? playPasscode : password;

Recording recording = Recording.builder()
        .recordCd(recordCd)          // Zoom 녹화 파일 ID (중복 체크용)
        .lessonCd(lessonCd)          // 어떤 강의의 녹화인지
        .recordUrl(recordUrl)        // share_url?pwd=passcode (또는 폴백 URL)
        .recordPassword(storedPassword)  // 인코딩된 패스코드 (또는 평문 비밀번호)
        .recordDuration(duration)    // 녹화 시간 (초)
        .recordSize(file.get("file_size") != null
            ? ((Number) file.get("file_size")).longValue() : null)  // 파일 크기
        .build();

recordingRepository.save(recording);

Recording 엔티티 설계¶

@Entity
@Table(name = "TB_NEXCLASS_RECORDING")
@Builder
public class Recording {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    @Column(name = "RECORD_ID")
    private Long recordId;            // DB PK (Auto Increment)

    @Column(name = "RECORD_CD", length = 100)
    private String recordCd;          // Zoom 녹화 파일 ID (중복 체크용)

    @Column(name = "LESSON_CD", length = 50)
    private String lessonCd;          // 강의 코드

    @Column(name = "RECORD_URL", length = 500)
    private String recordUrl;         // 재생 URL (패스코드 포함)

    @Column(name = "RECORD_PASSWORD", length = 500)
    private String recordPassword;    // 인코딩된 패스코드 또는 평문 비밀번호

    @Column(name = "RECORD_DURATION")
    private Integer recordDuration;   // 녹화 시간 (초)

    @Column(name = "RECORD_SIZE")
    private Long recordSize;          // 파일 크기 (bytes)

    @Column(name = "VIEW_YN", length = 1)
    private String viewYn;            // 녹화 가시성 (Y: 공개, N: 비공개)

    @Column(name = "CREATED_AT")
    private LocalDateTime createdAt;  // 생성 시점

    @PrePersist
    protected void onCreate() {
        this.createdAt = LocalDateTime.now();
        if (this.viewYn == null) {
            this.viewYn = "Y";        // 기본값: 공개
        }
    }
}

필드별 설계 의도¶

필드 설계 의도
recordId DB 내부 PK. Auto Increment
recordCd Zoom 녹화 파일 ID. 중복 체크의 핵심
recordUrl share_url?pwd=passcode. 이게 학생에게 보이는 URL
recordPassword recording_play_passcode 저장. URL에 이미 포함되어 있지만 별도 보관
recordDuration recording_start ~ recording_end 계산 결과 (초)
recordSize 파일 크기. 관리용 정보
viewYn 교수가 녹화 공개/비공개 제어. 기본 "Y"

viewYn: 녹화 가시성 제어¶

@PrePersist

viewYn vs recordViewYn

NexClass DB에서는 viewYn이라는 컬럼명을 쓴다. 근데 LMS API 응답에서는 recordViewYn이라는 필드명으로 보낸다.

위치 필드명 설명
NexClass DB VIEW_YN 컬럼명
NexClass Entity viewYn Java 필드명
LMS API 응답 recordViewYn VcApiLectureRecordVO 필드명

LMS가 recordViewYn으로 요청하니까 API 응답 변환할 때 매핑해야 한다.

녹화 가시성 흐름을 정리하면 이렇다.

flowchart LR
    A["녹화 저장<br/>viewYn = 'Y' (기본)"] --> B{교수가 비공개 설정?}
    B -->|Yes| C["viewYn = 'N'"]
    B -->|No| D["viewYn = 'Y' 유지"]

    C --> E["학생 조회 시<br/>recordViewYn = 'N'<br/>→ 녹화 안 보임"]
    D --> F["학생 조회 시<br/>recordViewYn = 'Y'<br/>→ 녹화 보임"]

RecordingRepository¶

public interface RecordingRepository extends JpaRepository<Recording, Long> {

    // 강의별 녹화 목록 조회
    List<Recording> findByLessonCd(String lessonCd);

메서드 용도
findByLessonCd() LMS가 특정 강의의 녹화 목록 요청할 때
findByRecordCd() 웹훅 재전송 시 중복 체크

password vs recording_play_passcode¶

이 차이를 명확히 알아야 한다.

구분 password recording_play_passcode
형태 평문 (abc123) 인코딩 (EnCoDeDpAsScOdE)
용도 사용자가 직접 입력 URL 파라미터로 사용
URL 사용 불가 ?pwd= 뒤에 붙임
보안 사람이 읽을 수 있음 URL-safe 인코딩 
평문 비밀번호: abc123
URL용 패스코드: EnCoDeDpAsScOdE

share_url만: https://zoom.us/rec/share/xxx
  → 비밀번호 입력 화면 뜸 → "abc123" 입력해야 함

share_url + pwd: https://zoom.us/rec/share/xxx?pwd=EnCoDeDpAsScOdE
  → 비밀번호 입력 없이 바로 재생!

학생 UX가 갈리는 포인트

share_url만 저장하면 학생이 녹화를 볼 때마다 비밀번호를 입력해야 한다. 비밀번호를 모르면? 교수한테 물어봐야 한다. 매번.

share_url?pwd=passcode를 저장하면 클릭 한 번으로 재생된다. 이 한 줄의 차이가 UX를 결정한다.

주의사항¶

실수하기 쉬운 포인트들

1. share_url은 개별 파일이 아닌 최상위에 있다

{
    "share_url": "여기!",       // 최상위
    "recording_files": [
        { "id": "...", "play_url": "개별 URL" }  // 파일 레벨
    ]
}

share_url을 recording_files 안에서 찾으면 null이다. Zoom API 응답 구조를 잘 봐야 한다.

2. Recordings API 호출 실패 시 폴백

API가 실패해도 웹훅 데이터의 play_url로 폴백한다. 완전히 실패하는 것보다 불완전하더라도 저장하는 게 낫다. 나중에 수동으로 URL을 업데이트할 수 있으니까.

3. MP4 외 파일 타입 무시

Zoom은 MP4, M4A, CHAT, TIMELINE 등 여러 파일을 만든다. 전부 저장하면 DB가 불필요하게 커진다. 학생이 보는 건 영상(MP4)뿐이다.

4. 녹화 인코딩 시간

meeting.ended와 recording.completed는 시간 차이가 있다. 1시간 수업이면 인코딩에 5-15분 걸릴 수 있다. "미팅 끝났는데 녹화가 없어요"라는 문의가 올 수 있다. 정상이야.

정리¶

이 챕터에서 기억해야 할 것

  1. recording.completed 웹훅이 트리거 (meeting.ended 아님)
  2. Recordings API 추가 호출 - share_url과 playPasscode는 웹훅에 없음
  3. playUrlWithPasscode = share_url?pwd=recording_play_passcode
  4. MP4만 저장 - 오디오, 채팅, 타임라인은 스킵
  5. 중복 체크 - recordCd로 멱등성 보장
  6. viewYn - 기본 "Y" (공개), LMS API에서는 recordViewYn으로 매핑
  7. 3단계 폴백 - share_url+pwd > share_url > play_url

확인문제¶

1. recording.completed 웹훅을 받은 후 Recordings API를 추가로 호출하는 이유는?

웹훅 데이터에는 share_url과 recording_play_passcode가 포함되어 있지 않다. 이 두 값은 Recordings API에서만 얻을 수 있다.

share_url + ?pwd=recording_play_passcode를 조합해야 학생이 비밀번호 입력 없이 바로 녹화를 볼 수 있다. 이게 없으면 매번 비밀번호를 입력해야 하는 UX가 된다.

2. MP4만 저장하고 M4A(오디오), CHAT(채팅)을 저장하지 않는 이유는?

YAGNI (You Ain't Gonna Need It) 원칙이다.

학생이 다시보기하는 건 영상이다. 오디오만 따로 듣는 경우는 거의 없고, 채팅 로그를 별도로 보는 기능도 없다. 지금 필요 없는 데이터를 저장하면 DB 용량만 낭비된다.

나중에 채팅 기록이 필요해지면 그때 추가하면 된다.

3. 중복 녹화 방지를 recordCd로 하는데, 이게 멱등성과 무슨 관계야?

멱등성(Idempotency): 같은 요청을 여러 번 처리해도 결과가 같다는 원칙.

Zoom이 웹훅을 재전송하면 같은 recording.completed 이벤트가 두 번 온다. recordCd로 중복 체크하지 않으면 같은 녹화가 2개 저장된다.

if (recordingRepository.findByRecordCd(recordCd).isPresent()) {
    continue;  // 이미 있으면 스킵 → 멱등성 보장
}

한 번 처리하든 열 번 처리하든 결과는 녹화 1개. 이게 멱등성이야.

4. password와 recording_play_passcode의 차이는 뭐고, 왜 recording_play_passcode를 우선 저장하나?

password는 평문 비밀번호다. 사람이 직접 입력할 때 쓴다. recording_play_passcode는 URL용 인코딩된 패스코드다. ?pwd= 파라미터로 URL에 붙일 수 있다.

recording_play_passcode를 우선 저장하는 이유는 이걸 URL에 붙이면 비밀번호 입력 없이 바로 재생할 수 있기 때문이다. 평문 password는 URL 파라미터로 쓸 수 없다.

String storedPassword = playPasscode != null ? playPasscode : password;

playPasscode가 있으면 그걸 쓰고, 없으면 password를 폴백으로 저장한다.

5. Recordings API가 실패했을 때의 폴백 전략을 순서대로 설명해봐

3단계 폴백이다.

1단계 (최선): share_url + ?pwd=recording_play_passcode Recordings API 성공 시. 비밀번호 없이 바로 재생.

2단계 (차선): share_url만 recording_play_passcode가 null인 경우. 비밀번호 입력 필요.

3단계 (최후): 웹훅의 play_url Recordings API 자체가 실패한 경우. Zoom 인증 필요할 수 있음.

방어적 프로그래밍이야. API 하나 실패했다고 전체를 포기하지 않는다. 불완전하더라도 최대한 저장하고, 나중에 수동으로 보완할 수 있게 한다.
