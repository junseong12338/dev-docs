24. Webhook과 프록시 - Omega¶

1. 상황 설정¶

Webhook은 일반적인 HTTP 요청의 정반대야. 보통은 우리가 외부 서버에 요청을 보내지? Webhook은 외부 서버(Zoom)가 우리한테 요청을 보내는 거야. 이걸 "역방향 API 호출"이라고 생각하면 돼.

이 챕터에서는 Zoom에서 보낸 Webhook이 NexClass의 WebhookController까지 도달하는 전체 과정을 추적한다. 프록시, SSL, 터널, 서명 검증까지 전부 엮여있어.

왜 이걸 알아야 하냐면

Webhook이 안 오면 출결 처리가 안 돼. 녹화 URL도 안 저장돼. 학생 성적에 직결되는 거야. "Webhook이 왜 안 오지?" 디버깅하려면 전체 경로를 완벽하게 이해해야 해.

2. Webhook의 본질: 역방향 API 호출¶

일반 API vs Webhook¶

graph LR
    subgraph "일반 API 호출"
        A1[NexClass] -->|"요청"| B1[Zoom API]
        B1 -->|"응답"| A1
    end

    subgraph "Webhook (역방향)"
        A2[Zoom] -->|"요청 (이벤트 알림)"| B2[NexClass]
        B2 -->|"응답 (200 OK)"| A2
    end
항목 일반 API 호출 Webhook
누가 요청하나 우리 (NexClass) 상대방 (Zoom)
누가 서버인가 Zoom이 서버 우리가 서버
언제 발생하나 우리가 원할 때 이벤트 발생 시
URL 누가 제공 Zoom이 API URL 제공 우리가 Webhook URL 제공
HTTPS 필수 아님 (권장) 필수 (Zoom 정책)

핵심 차이

일반 API는 우리가 "데이터 주세요"라고 물어보는 거야 (Pull). Webhook은 Zoom이 "이벤트 발생했어요, 알려드립니다"라고 먼저 알려주는 거야 (Push). 클라이언트-서버 역할이 뒤바뀌는 거지.

HTTPS가 필수인 이유¶

Zoom은 Webhook 엔드포인트로 HTTPS URL만 허용한다. HTTP는 거부해.

이유 설명
데이터 보호 Webhook 페이로드에 미팅 정보, 참여자 정보가 포함됨
위변조 방지 중간에서 Webhook 데이터를 바꿀 수 없도록
인증 보장 SSL 인증서로 수신 서버의 신원 확인
업계 표준 Stripe, GitHub, Slack 등 모든 주요 서비스가 HTTPS 필수

이게 프록시와 연결되는 지점

우리 서버에 직접 HTTPS를 설정할 수 없으니까 (iwinv 프록시 미설정 + 포트 8082 차단), ngrok이 HTTPS를 대신 제공하는 거야. 8장(SSL/TLS 기초)과 9장(SSL과 프록시)에서 배운 "SSL 종료" 개념이 바로 이거.

3. 현재 Webhook 전체 흐름¶

meeting.ended: 출결 처리¶

교수가 Zoom 미팅을 종료하면 자동으로 출결 처리가 시작된다.

sequenceDiagram
    participant ZOOM as Zoom 서버
    participant NGROK_C as ngrok 클라우드
    participant NGROK_A as ngrok 에이전트<br/>(서버 내부)
    participant WC as WebhookController
    participant AS as AttendanceService
    participant ZOOM_API as Zoom Reports API
    participant DB as MariaDB
    participant LMS as LMS 서버

    Note over ZOOM: 교수가 미팅 종료

    ZOOM->>NGROK_C: 1. POST https://abc-xyz.ngrok-free.dev/api/webhook<br/>Body: {event: "meeting.ended", ...}<br/>Header: x-zm-signature (HMAC-SHA256)

    Note over NGROK_C: 2. SSL 종료 (HTTPS → HTTP)

    NGROK_C->>NGROK_A: 3. 터널 전달
    NGROK_A->>WC: 4. localhost:8082/api/webhook

    Note over WC: 5. HMAC-SHA256 서명 검증<br/>Zoom Secret Token으로 재계산 → 비교

    WC->>AS: 6. processAttendance(meetingId)

    AS->>ZOOM_API: 7. GET /v2/report/meetings/{meetingId}/participants
    ZOOM_API-->>AS: 8. 참여자 목록 (이메일, 입장/퇴장, duration)

    Note over AS: 9. 이메일 매칭<br/>20230001@nexclass.local → userCd: 20230001<br/>호스트(교수) 스킵

    AS->>DB: 10. Attendance 저장 (세션별 개별)

    AS->>LMS: 11. POST /vc/api/sync/webhook<br/>{eventType: "lessons.attends.processing-completed",<br/>datas: {lessonCd, lessonProfCd}}

    Note over LMS: 12. LMS가 NexClass API 호출 → 출결 데이터 조회

단계별 상세 설명¶

1단계: Zoom이 Webhook 전송

Zoom 서버가 미팅 종료 이벤트를 감지하면, Zoom Marketplace에 등록된 Webhook URL로 POST 요청을 보낸다.

{
  "event": "meeting.ended",
  "payload": {
    "account_id": "{ZOOM_ACCOUNT_ID}",
    "object": {
      "id": 12345678901,
      "topic": "2026-1학기 자료구조 3주차",
      "host_id": "abc123",
      "host_email": "host01@example.or.kr",
      "start_time": "2026-03-19T09:00:00Z",
      "end_time": "2026-03-19T10:00:00Z"
    }
  }
}

헤더에는 서명이 포함돼:

x-zm-request-timestamp: 1742342400
x-zm-signature: v0=a1b2c3d4e5f6...

2~4단계: ngrok 경유

22장에서 설명한 것과 동일한 경로. Zoom -> ngrok 클라우드 -> 터널 -> ngrok 에이전트 -> localhost:8082.

5단계: HMAC-SHA256 서명 검증

이게 보안의 핵심이야. "이 Webhook이 진짜 Zoom에서 보낸 거 맞아?"를 검증하는 거지.

검증 과정:
1. 수신한 타임스탬프 + 요청 본문으로 메시지 생성
   message = "v0:{timestamp}:{requestBody}"

2. Zoom Secret Token으로 HMAC-SHA256 해시 계산
   expectedSignature = HMAC-SHA256(secretToken, message)

3. 수신한 서명과 비교
   received: "v0=a1b2c3d4e5f6..."
   expected: "v0=a1b2c3d4e5f6..."
   → 일치하면 Zoom이 보낸 게 맞음

서명 검증 안 하면

아무나 /api/webhook에 가짜 Webhook을 보낼 수 있어. 존재하지 않는 미팅의 가짜 출결 데이터가 DB에 들어가. 서명 검증은 생략 불가야.

7~8단계: Zoom Reports API 호출

Webhook 자체에는 참여자 상세 정보가 없어. 그래서 meeting.ended 이벤트를 받으면 NexClass가 Zoom Reports API를 호출해서 참여자 목록을 가져온다.

GET https://api.zoom.us/v2/report/meetings/12345678901/participants
Authorization: Bearer {Zoom S2S OAuth Token}

여기서 NexClass가 클라이언트

Webhook은 Zoom이 클라이언트고 우리가 서버였지? 근데 Reports API 호출은 우리가 클라이언트고 Zoom이 서버야. 한 흐름 안에서 클라이언트-서버 역할이 뒤바뀌는 거지.

9단계: 이메일 매칭

Zoom Reports API가 반환하는 참여자 이메일로 학생을 식별한다.

Zoom 이메일 매칭 규칙 userCd
20230001@nexclass.local @ 앞부분 추출 + 대소문자 무시 20230001
20230002@nexclass.local @ 앞부분 추출 + 대소문자 무시 20230002
host01@example.or.kr 호스트 이메일 → 스킵 (저장 안 함)

11단계: LMS Push

출결 처리가 끝나면 LMS에 "출결 데이터 준비됐어"라고 알린다. LMS가 다시 NexClass API를 호출해서 데이터를 가져가는 구조야.

share_url의 의미

Zoom Recordings API가 반환하는 share_url에는 패스코드 토큰이 포함돼 있어. 학생이 이 URL로 접속하면 비밀번호 입력 없이 바로 녹화 영상을 볼 수 있어.

문제 ngrok이 해결하는 방법
HTTPS 필수 ngrok이 SSL 인증서 제공
포트 8082 차단 아웃바운드 터널로 우회
고정 URL 필요 ngrok이 공개 URL 제공
iwinv 프록시 미설정 iwinv를 완전히 우회

ngrok의 문제¶

문제 영향
URL 변경 ngrok 재시작하면 URL 바뀜 → Zoom Marketplace 재설정 필요
프로세스 의존 ngrok 프로세스 죽으면 Webhook 수신 불가 → 출결 처리 실패
속도 ngrok 클라우드를 경유하니까 직접 연결보다 느림
신뢰성 ngrok 서비스 장애 시 우리 서버도 접근 불가
프로덕션 부적합 무료 플랜 제한 + 안정성 보장 못 함

5. Cloudflare Tunnel로 전환하면¶

ngrok의 문제를 해결하는 대안이 Cloudflare Tunnel이야. 16장에서 원리를 배웠으니, 여기서는 NexClass에 적용했을 때 뭐가 달라지는지 보자.

비교¶

항목 ngrok Cloudflare Tunnel
URL 세션마다 변경 (무료) nexclass.knu10.cc 고정
SSL ngrok 자체 인증서 Cloudflare Universal SSL
프로세스 포그라운드 (터미널 유지) 백그라운드 서비스 (systemd)
비용 무료/유료 무료
Zoom 설정 URL 바뀔 때마다 재설정 한 번이면 끝
디버깅 ngrok inspect (4040 포트) Cloudflare 대시보드
안정성 프로세스 의존 systemd 자동 재시작

6. Webhook 디버깅¶

ngrok inspect¶

ngrok은 http://127.0.0.1:4040에 디버깅 대시보드를 제공한다.

확인 가능한 정보 설명
수신된 요청 Zoom이 보낸 Webhook 페이로드 전체
요청 헤더 x-zm-signature, Content-Type 등
응답 코드 200, 500 등 우리 서버의 응답
응답 시간 처리에 걸린 시간
Replay 같은 요청을 다시 보내기 (디버깅용) 
# 서버에서 ngrok 실행 중일 때
# 브라우저에서 접속 (서버 내부에서만 가능)
http://127.0.0.1:4040

# 또는 SSH 포트 포워딩으로 로컬에서 접속
ssh -L 4040:localhost:4040 ubuntu@xxx.xxx.xx.xxx
# 그 다음 로컬 브라우저에서 http://localhost:4040

# ngrok API로 최근 요청 확인
curl http://127.0.0.1:4040/api/requests/http

7. 정리¶

Webhook 흐름 요약¶

이벤트 트리거 처리 결과
meeting.ended 교수 미팅 종료 Reports API -> 이메일 매칭 -> 출결 저장 DB 저장 + LMS Push
recording.completed 녹화 인코딩 완료 Recordings API -> share_url 추출 DB 저장

한 줄 정리

Webhook = 외부 서버(Zoom)가 우리한테 보내는 역방향 API 호출. HTTPS가 필수라서 ngrok이 SSL을 대신 제공하고, 도착하면 HMAC-SHA256으로 진짜 Zoom이 보낸 건지 검증한다.

이 챕터에서 반드시 기억할 것

  1. Webhook은 역방향이야. Zoom이 클라이언트, 우리가 서버.
  2. HTTPS 필수. Zoom이 HTTP 엔드포인트 거부한다.
  3. HMAC-SHA256 서명 검증으로 위조 요청을 막는다.
  4. meeting.ended Webhook 자체에는 참여자 상세가 없어. 별도로 Reports API를 호출해야 해.
  5. ngrok URL이 바뀌면 Zoom Marketplace에서 재설정 필수. Cloudflare Tunnel이면 이 문제가 사라져.

확인 문제¶

다음 문제를 풀어봐. 답 못 하면 위에서 다시 읽어.

Q1. 일반 API 호출과 Webhook의 핵심 차이는 뭐야? "누가 요청하는지"를 기준으로 설명해봐.

Q2. Zoom이 HTTP(HTTPS가 아닌) Webhook URL을 거부하는 이유 2가지를 말해봐.

Q3. HMAC-SHA256 서명 검증에서, NexClass가 수신한 서명과 비교하기 위해 직접 계산하는 데 필요한 재료 3가지는 뭐야?

Q4. meeting.ended Webhook을 받은 후 NexClass가 Zoom Reports API를 호출한다. 이때 NexClass는 클라이언트야, 서버야? Webhook 수신 시점과 역할이 어떻게 바뀌나?

Q5. ngrok을 Cloudflare Tunnel로 교체하면 Zoom Webhook 운영에서 뭐가 달라져?

정답 보기

A1. 일반 API 호출은 우리(NexClass)가 외부 서버(Zoom)에 요청을 보내는 거고, Webhook은 반대로 외부 서버(Zoom)가 우리한테 요청을 보내는 거야. 일반 API는 Pull 방식(필요할 때 가져옴), Webhook은 Push 방식(이벤트 발생 시 알려줌).

A2. (1) 데이터 보호: Webhook 페이로드에 미팅 정보, 참여자 이메일 등 민감 정보가 포함돼서 암호화 없이 전송하면 중간에 탈취 가능. (2) 위변조 방지: HTTP는 암호화가 없어서 중간에서 Webhook 데이터를 변조할 수 있음. HTTPS면 TLS 암호화로 보호됨.

A3. (1) 수신한 타임스탬프 (x-zm-request-timestamp 헤더). (2) 요청 본문 (request body 전체). (3) Zoom Secret Token (Zoom Marketplace에서 발급받은 비밀 키). 이 세 가지로 v0:{timestamp}:{body} 메시지를 만들고 HMAC-SHA256 해시를 계산한다.

A4. Webhook 수신 시점에는 NexClass가 서버 (Zoom이 보낸 요청을 받는 쪽). Reports API 호출 시점에는 NexClass가 클라이언트 (Zoom API에 요청을 보내는 쪽). 하나의 처리 흐름 안에서 서버 -> 클라이언트로 역할이 바뀐다.

A5. (1) URL이 nexclass.knu10.cc/api/webhook으로 고정돼서, ngrok 재시작 시 Zoom Marketplace 재설정이 불필요. (2) cloudflared가 systemd 서비스로 돌기 때문에 프로세스가 죽어도 자동 재시작 → 안정성 향상. (3) 터미널을 열어두고 관리할 필요가 없음.
