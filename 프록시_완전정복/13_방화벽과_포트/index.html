13. 방화벽과 포트 - Gamma¶

1. 핵심 개념 - "방화벽이 뭔데?"¶

건물에 경비원이 있다고 생각해봐. 방문자가 오면 경비원이 "누구세요? 어디 가세요?" 물어보고, 허가된 사람만 통과시키고 나머지는 돌려보내. 방화벽(Firewall)이 정확히 그 역할이야.

네트워크에서 오가는 패킷(데이터)을 보고, 규칙에 따라 통과시키거나 차단하는 시스템이야.

방화벽의 본질

방화벽은 네트워크 트래픽을 필터링하는 시스템이다. "이 IP에서 이 포트로 오는 트래픽은 허용, 나머지는 차단" 같은 규칙을 설정해서 원치 않는 접근을 막는다.

방화벽 vs 프록시 - 헷갈리니까 정리하자¶

이 둘은 근본적으로 하는 일이 달라:

항목 방화벽 (Firewall) 프록시 (Proxy)
하는 일 트래픽 통과 여부 결정 트래픽을 대신 전달
비유 경비원 (출입 통제) 비서 (대리 업무)
판단 기준 IP, 포트, 프로토콜 URL, 도메인, 컨텐츠
데이터 변경 안 함 (통과/차단만) 헤더 추가/수정 가능
위치 네트워크 경계 클라이언트-서버 사이

한 줄 구분

방화벽 = "이 놈 들여보낼까 말까?" / 프록시 = "이 놈 대신 전달해줄까?"

2. 인바운드 vs 아웃바운드 - "들어오는 것 vs 나가는 것"¶

방화벽 규칙은 크게 두 방향이야:

graph LR
    A["외부 인터넷"] -->|"인바운드 (들어오는 것)"| B["서버"]
    B -->|"아웃바운드 (나가는 것)"| A
방향 설명 예시
인바운드 (Inbound) 외부에서 서버로 들어오는 트래픽 브라우저가 웹페이지 요청, SSH 접속
아웃바운드 (Outbound) 서버에서 외부로 나가는 트래픽 서버가 API 호출, 패키지 다운로드

대부분의 서버는 인바운드를 제한하고 아웃바운드는 거의 안 막아

왜냐? 서버가 외부로 나가는 건 대부분 정상적인 작업이야 (패키지 업데이트, API 호출 등). 근데 외부에서 서버로 들어오는 건 공격일 수 있으니까 엄격하게 관리하는 거야.

이게 14장(터널링)에서 나오는 핵심 개념이야. 인바운드가 막혀있으니까, 아웃바운드(서버 안에서 밖으로)로 터널을 뚫어서 우회하는 거야.

3. 포트 - "어떤 문으로 들어오는 건데?"¶

서버에 포트가 65,535개 있어. 각 포트는 건물의 출입문 번호라고 생각하면 돼.

대표적인 포트¶

포트 프로토콜 용도
22 SSH 원격 서버 접속 (터미널)
80 HTTP 웹 (암호화 안 됨)
443 HTTPS 웹 (암호화)
3306 MySQL/MariaDB 데이터베이스
5432 PostgreSQL 데이터베이스
8080 HTTP (비표준) 톰캣, 개발 서버
8082 HTTP (비표준) NexClass Spring Boot

0~1023 = Well-Known Ports

0~1023번까지는 "잘 알려진 포트"야. 국제 표준으로 정해져 있어. 80=HTTP, 443=HTTPS, 22=SSH. 1024 이상은 자유롭게 쓸 수 있고, 그래서 Spring Boot 기본 포트가 8080이고, 우리 NexClass는 8082를 쓰는 거야.

포트와 방화벽의 관계¶

방화벽 규칙은 포트 단위로 설정하는 경우가 많아:

"80번 포트 인바운드 → 허용"  = 웹 서버 접근 허용
"22번 포트 인바운드 → 허용"  = SSH 접속 허용
"3306번 포트 인바운드 → 차단" = DB 외부 접근 차단
"8082번 포트 인바운드 → ???"  = NexClass 접근 ???

4. 리눅스 방화벽 도구 - "실제로 어떻게 설정해?"¶

ufw (Uncomplicated Firewall) - 쉬운 버전¶

Ubuntu에서 기본 제공하는 방화벽 도구야. iptables의 프론트엔드인데, 훨씬 간단해.

# 방화벽 상태 확인
sudo ufw status

# 방화벽 활성화
sudo ufw enable

# 포트 열기
sudo ufw allow 80/tcp       # HTTP
sudo ufw allow 443/tcp      # HTTPS
sudo ufw allow 22/tcp       # SSH (이거 안 열면 SSH 접속 끊겨!)
sudo ufw allow 8082/tcp     # NexClass

# 포트 닫기
sudo ufw deny 3306/tcp      # DB 외부 접근 차단

# 특정 IP만 허용
sudo ufw allow from {허용할_IP} to any port 22    # 특정 IP만 SSH 허용 (실제 IP로 교체)

# 규칙 삭제
sudo ufw delete allow 8082/tcp

# 상태 + 번호 확인
sudo ufw status numbered

SSH(22번) 닫으면 서버 접속 자체가 안 돼

방화벽 설정할 때 22번 포트 닫으면 그 순간부터 서버에 접속을 못 해. 물리적으로 서버 앞에 가서 직접 조작해야 하는 상황이 발생하니까 22번은 절대 건드리지 마. 원격 서버 관리의 생명줄이야.

iptables - 어려운 버전¶

ufw가 내부적으로 쓰는 도구야. 직접 쓸 일은 많지 않지만 알아는 둬야 해.

# 현재 규칙 확인
sudo iptables -L -n -v

# 80번 포트 인바운드 허용
sudo iptables -A INPUT -p tcp --dport 80 -j ACCEPT

# 8082번 포트 인바운드 차단
sudo iptables -A INPUT -p tcp --dport 8082 -j DROP

# 모든 아웃바운드 허용
sudo iptables -A OUTPUT -j ACCEPT

ufw vs iptables

ufw는 iptables의 래퍼(wrapper)야. 결국 같은 일을 해. 특별한 이유 없으면 ufw 써. iptables 직접 만지다가 실수하면 서버 먹통 되기 딱 좋아.

5. 클라우드 방화벽 - "서버 안 말고 밖에도 방화벽이 있어"¶

여기서 많은 사람이 헷갈려. 방화벽은 서버 안에만 있는 게 아니야.

클라우드(AWS, iwinv 등)에서 호스팅하면, 네트워크 레벨 방화벽이 서버 앞에 또 있어:

graph LR
    A["외부 인터넷"] --> B["클라우드 방화벽<br>(보안 그룹 / 네트워크 ACL)"]
    B --> C["서버 방화벽<br>(ufw / iptables)"]
    C --> D["서비스<br>(nginx, Spring Boot)"]
계층 이름 위치
1단계 클라우드 방화벽 (보안 그룹) 클라우드 인프라 (서버 밖)
2단계 서버 방화벽 (ufw/iptables) 서버 OS 내부
3단계 어플리케이션 방화벽 (WAF 등) 어플리케이션 레벨

두 방화벽 다 열어야 통과됨

클라우드 보안 그룹에서 8082를 열어도, 서버 ufw에서 8082를 안 열면 안 돼. 반대도 마찬가지야. 양쪽 다 허용해야 트래픽이 통과해.

AWS vs iwinv 비교¶

항목 AWS iwinv
방화벽 이름 Security Group 보안 그룹 / 네트워크 설정
설정 방법 AWS 콘솔에서 UI로 iwinv 콘솔 또는 고객센터 요청
기본 상태 모든 인바운드 차단, 아웃바운드 허용 비슷 (80/443은 프록시 경유)

6. 실전 - "포트 8082가 왜 안 되는 거야?"¶

NexClass가 localhost:8082에서 돌아가. 서버 안에서는 잘 돼:

# 서버 안에서 접속 → 성공
curl http://localhost:8082/health
# {"status":"UP"}

근데 외부에서 http://xxx.xxx.xx.xxx:8082로 접속하면 안 돼. 왜?

graph TD
    A["외부 브라우저<br>xxx.xxx.xx.xxx:8082"] --> B{"iwinv 네트워크"}
    B -->|"포트 8082?<br>허용 안 됨"| C["차단"]
    B -->|"포트 80/443?<br>iwinv 프록시 경유"| D["iwinv 앞단 프록시"]
    D --> E["서버"]
포트 외부 접근 이유
22 (SSH) 가능 서버 관리 필수 → 열려있음
80 (HTTP) iwinv 프록시 경유 iwinv가 앞단에서 관리
443 (HTTPS) iwinv 프록시 경유 iwinv가 앞단에서 관리
8082 차단 iwinv가 비표준 포트 외부 노출 안 함

"그러면 방화벽 열면 되지 않아?"¶

안 돼. 두 가지 이유야:

이유 1: iwinv 네트워크 레벨에서 막힘

서버 안에서 ufw allow 8082 해봐야 소용없어. iwinv 네트워크가 먼저 막으니까. 서버 방화벽까지 도달을 못 해.

이유 2: 방화벽 뚫려도 HTTPS가 안 됨

Zoom Webhook은 HTTPS만 허용해. 8082 포트를 외부에 열어도 HTTP야. HTTPS로 만들려면 SSL 인증서가 필요하고, 8082 포트에 직접 SSL을 붙이는 건 nginx 없이는 번거로워.

정리

포트 8082 외부 접속이 안 되는 건 iwinv 네트워크가 차단하기 때문이야. 서버 방화벽 문제가 아니라 클라우드 방화벽 문제. 그래서 ngrok 같은 아웃바운드 터널로 우회하는 거야.

7. 주의사항 / 함정¶

함정 1: SSH 포트 닫기

방화벽 만지다가 22번 포트 닫으면 서버 접속이 끊겨. 원격 서버 관리할 때 절대, 절대 22번 안 건드려.

함정 2: 클라우드 방화벽 무시

서버 안에서 ufw allow 8082 했는데 안 된다고 삽질하는 경우가 많아. 클라우드 네트워크 방화벽도 열어야 해. 두 겹이야.

함정 3: 방화벽 = 보안 끝?

방화벽은 보안의 한 레이어일 뿐이야. 80번 포트 열어놨으면 거기로 들어오는 공격은 방화벽이 못 막아. SQL 인젝션, XSS 같은 건 어플리케이션 레벨에서 막아야 해.

함정 4: 방화벽 열면 HTTPS도 된다?

아니야. 방화벽은 트래픽 통과 여부만 결정해. HTTPS는 SSL 인증서가 있어야 동작하는 거야. 포트 열어봤자 SSL 인증서 없으면 HTTPS 안 돼.

8. 정리¶

항목 내용
방화벽이란? 네트워크 트래픽을 규칙에 따라 허용/차단하는 시스템
인바운드 외부 → 서버 (보통 엄격히 제한)
아웃바운드 서버 → 외부 (보통 대부분 허용)
방화벽 vs 프록시 방화벽 = 통과 여부, 프록시 = 대신 전달
서버 방화벽 ufw, iptables (서버 OS 내부)
클라우드 방화벽 보안 그룹 (서버 밖, 네트워크 레벨)
양쪽 다 열어야 클라우드 + 서버 방화벽 둘 다 허용해야 통과
우리 상황 iwinv 네트워크가 8082 차단 → ngrok으로 우회

한 줄 정리

방화벽은 "통과시킬까 말까"를 결정하는 경비원. 서버 안과 밖에 각각 있고, 양쪽 다 열어야 트래픽이 통과한다.

확인 문제¶

다음 문제를 풀어봐. 답 못 하면 위에서 다시 읽어.

Q1. 방화벽과 프록시의 근본적인 차이가 뭐야?

Q2. 인바운드와 아웃바운드 트래픽 중 보통 어느 쪽을 더 엄격하게 제한해? 왜?

Q3. 서버에서 sudo ufw allow 8082/tcp를 실행했는데도 외부에서 8082 포트 접속이 안 돼. 원인이 뭘 수 있어?

Q4. 포트 8082 외부 접속이 된다고 해도, Zoom Webhook이 사용할 수 없는 이유가 뭐야?

Q5. 방화벽을 열면 HTTPS가 가능해지는가? 이유를 설명해봐.

정답 보기

A1. 방화벽은 트래픽의 통과 여부를 결정하는 거고(허용/차단), 프록시는 트래픽을 대신 전달하는 거야. 방화벽은 경비원(출입 통제), 프록시는 비서(대리 업무). 방화벽은 데이터를 변경하지 않고 통과/차단만 하지만, 프록시는 헤더를 추가하거나 수정할 수 있어.

A2. 인바운드를 더 엄격하게 제한해. 외부에서 서버로 들어오는 트래픽은 공격일 수 있으니까. 아웃바운드(서버 → 외부)는 패키지 업데이트, API 호출 등 정상적인 작업이 대부분이라 거의 안 막아. 이 차이가 터널링이 가능한 이유야.

A3. 클라우드(iwinv) 네트워크 레벨 방화벽이 8082를 차단하고 있을 가능성이 높아. 서버 안의 ufw는 서버 OS 방화벽이고, 그 앞에 클라우드 방화벽이 또 있어. 양쪽 다 열어야 외부 트래픽이 통과해.

A4. Zoom Webhook은 HTTPS만 허용해. 8082 포트를 외부에 열어도 HTTP(평문)로 서비스되니까 Zoom이 거부해. HTTPS로 만들려면 SSL 인증서가 필요하고, 인증서를 8082 포트에 적용하는 별도 작업이 필요해.

A5. 안 돼. 방화벽은 트래픽 통과 여부만 결정하는 거지, HTTPS 기능을 제공하는 게 아니야. HTTPS는 SSL/TLS 인증서가 있어야 동작해. 방화벽이 443 포트를 열어줘도 인증서 없으면 HTTPS 연결이 성립 안 돼.
