6. 리버스 프록시 - Beta¶

1. 핵심 개념 - "이게 뭐야?"¶

이거 왜 알아야 하냐면, NexClass가 지금 겪고 있는 문제가 바로 리버스 프록시 문제야. hub.knu10.cc는 되는데 nexclass.knu10.cc는 안 되는 이유? 리버스 프록시 설정이 안 돼 있어서야.

비유: 호텔 프론트 데스크¶

호텔 생각해봐. 손님(클라이언트)이 호텔에 오면 방(서버)에 직접 가는 게 아니라, 프론트 데스크(리버스 프록시)를 거쳐.

  • 손님은 "301호 고객 만나러 왔어요"라고 프론트에 말함
  • 프론트가 301호에 연결해줌
  • 손님은 301호가 건물 어디 있는지 몰라도 됨
  • 프론트가 보안 검사도 하고, 안내도 하고, 기록도 남김

비유는 여기까지. 진짜 정의 가자.

리버스 프록시 정의

서버 측에 위치하여, 클라이언트의 요청을 받아서 뒤에 있는 실제 서버(백엔드)로 전달하는 중간 서버. 클라이언트는 리버스 프록시만 보고, 실제 백엔드 서버의 존재를 모른다.

포워드 vs 리버스 -- 이 차이가 핵심¶

항목 포워드 프록시 리버스 프록시
위치 클라이언트 측 서버 측
누구를 대신하나 클라이언트를 대신 서버를 대신
누가 모르나 서버가 클라이언트를 모름 클라이언트가 서버를 모름
누가 설정하나 클라이언트/네트워크 관리자 서버 관리자
용도 접속 제어, IP 숨기기 보안, 로드밸런싱, SSL 처리
보호 대상 클라이언트 서버

가장 흔한 혼동

포워드 프록시는 클라이언트를 숨기고, 리버스 프록시는 서버를 숨긴다. 방향이 반대야. 이름에 "리버스"가 붙은 이유가 이거다.

2. 동작 원리 - "어떻게 돌아가?"¶

기본 흐름¶

sequenceDiagram
    participant C as 클라이언트 (브라우저)
    participant R as 리버스 프록시 (nginx)
    participant S as 백엔드 서버 (Spring Boot :8082)

    C->>R: 1. nexclass.knu10.cc 접속 (포트 443)
    Note over R: 2. 도메인 확인 + 라우팅 결정
    R->>S: 3. localhost:8082로 전달
    S-->>R: 4. 응답 반환
    R-->>C: 5. 클라이언트에게 전달

클라이언트가 보는 건 오직 리버스 프록시뿐이야. 뒤에 Spring Boot가 8082 포트에서 돌고 있다는 사실을 클라이언트는 전혀 모름.

왜 필요한가? 4가지 핵심 이유¶

graph TB
    R[리버스 프록시]
    R --> A[보안: 서버 IP 숨기기]
    R --> B[SSL 처리: HTTPS 한 곳에서]
    R --> C[로드밸런싱: 트래픽 분산]
    R --> D[가상호스트: 도메인별 라우팅]

1. 보안 -- 서버를 숨긴다

클라이언트가 직접 백엔드 서버에 접속하면, 서버 IP가 노출돼. 공격 대상이 되는 거지. 리버스 프록시를 앞에 두면 클라이언트는 프록시 IP만 알고, 백엔드 서버 IP는 몰라.

2. SSL 처리 -- 한 곳에서 관리

서버가 10대면 10대 전부 인증서 설치해야 해? 리버스 프록시에서 HTTPS 처리하고 백엔드에는 HTTP로 보내면 인증서 관리가 한 곳에서 끝나.

3. 로드밸런싱 -- 트래픽 분산

백엔드 서버가 여러 대면, 리버스 프록시가 "이번 요청은 1번 서버, 다음 요청은 2번 서버"로 분산시켜줌.

4. 가상호스트 -- 도메인별 라우팅

같은 IP인데 hub.knu10.cc와 lms.knu10.cc를 다른 서비스로 보내야 한다? 리버스 프록시가 도메인 이름 보고 알맞은 백엔드로 라우팅해줌.

3. 코드/설정으로 보자¶

대표적인 리버스 프록시들¶

이름 특징 용도
nginx 가볍고 빠름, 설정 간결 웹 서버 + 리버스 프록시
Apache httpd 역사 깊음, 모듈 풍부 전통적인 웹 서버
HAProxy 로드밸런싱 특화 대규모 트래픽 분산
Cloudflare CDN + 프록시 글로벌 서비스
iwinv 프록시 호스팅사 내장 프록시 호스팅 서비스
ngrok 터널링 + 리버스 프록시 개발/테스트

nginx 리버스 프록시 설정¶

이건 우리 NexClass 서버에 실제로 설정돼 있는 구조야.

# /etc/nginx/sites-available/nexclass
server {
    listen 80;                              # 80 포트로 들어온 요청을
    server_name nexclass.knu10.cc;          # 이 도메인이면

    location / {
        proxy_pass http://localhost:8082;   # Spring Boot (8082)로 전달
        proxy_set_header Host $host;        # 원래 도메인 정보 전달
        proxy_set_header X-Real-IP $remote_addr;  # 클라이언트 IP 전달
    }
}

이 설정이 의미하는 건:

  1. nexclass.knu10.cc로 요청이 들어오면
  2. nginx가 받아서
  3. localhost:8082 (Spring Boot)로 전달

클라이언트는 nginx하고만 통신하고, Spring Boot는 외부에 노출 안 돼.

iwinv 프록시 구조 -- 실전 상황¶

graph LR
    A[클라이언트] -->|"hub.knu10.cc"| B[iwinv 프록시]
    A -->|"lms.knu10.cc"| B
    A -->|"nexclass.knu10.cc"| B
    B -->|"라우팅 설정됨"| C[HUB 서비스]
    B -->|"라우팅 설정됨"| D[LMS 서비스]
    B -->|"라우팅 안 됨!"| E["??? (연결 실패)"]

iwinv가 앞단에서 리버스 프록시 역할을 하고 있어. 포트 80/443으로 들어온 요청을 도메인별로 내부 서비스에 라우팅하는 거지.

근데 문제가 뭐냐면:

  • hub.knu10.cc -- iwinv 프록시에 라우팅 설정됨 -- 정상 동작
  • lms.knu10.cc -- iwinv 프록시에 라우팅 설정됨 -- 정상 동작
  • nexclass.knu10.cc -- iwinv 프록시에 라우팅 미설정 -- 접속 불가

서버에서 8082 포트에 Spring Boot가 잘 돌고 있어도, 앞단 리버스 프록시가 그 포트로 안 보내주면 외부에서는 접속이 안 돼. 이게 리버스 프록시의 역할을 몸으로 체험한 거야.

ngrok -- 임시 리버스 프록시¶

sequenceDiagram
    participant C as 클라이언트 / Zoom
    participant N as ngrok 서버
    participant L as localhost:8082

    C->>N: https://abc-xyz.ngrok-free.dev
    Note over N: SSL 종료 + 터널링
    N->>L: HTTP로 localhost:8082 전달
    L-->>N: 응답
    N-->>C: HTTPS로 응답 전달

ngrok도 리버스 프록시야. 외부 요청을 받아서 내 로컬 서버로 전달해주잖아. SSL도 ngrok이 처리해주고.

4. 주의사항/함정¶

리버스 프록시 없으면 생기는 문제들¶

직접 노출의 위험

  • 서버 IP가 노출됨 -- DDoS 공격 대상
  • 포트가 직접 열림 -- 포트 스캐닝 취약
  • SSL을 각 서버마다 설정해야 함 -- 관리 지옥
  • 도메인별 라우팅 불가 -- 서버 하나에 서비스 하나

"포트 열면 되는 거 아니야?"¶

맞아, 8082 포트를 방화벽에서 열면 외부에서 직접 접속은 가능해. 근데:

항목 포트 직접 개방 리버스 프록시 경유
URL http://xxx.xxx.xx.xxx:8082 https://nexclass.knu10.cc
SSL 직접 설정 필요 프록시가 처리
보안 서버 IP + 포트 노출 서버 IP 숨김
도메인 도메인 연결 별도 프록시가 라우팅
실무 개발 테스트용 프로덕션 표준

핵심

프로덕션 환경에서 백엔드 포트를 직접 노출하는 건 안 해. 반드시 리버스 프록시를 거쳐야 한다. iwinv가 8082 포트를 외부 차단한 것도 이런 이유야.

다중 프록시 구조¶

실제 환경에서는 프록시가 한 겹이 아닐 수 있어.

graph LR
    A[클라이언트] --> B[CDN / Cloudflare]
    B --> C[iwinv 프록시]
    C --> D[nginx]
    D --> E[Spring Boot :8082]

프록시가 여러 겹이면 "진짜 클라이언트 IP가 뭐야?"가 복잡해져. 이건 나중에 프록시 헤더 챕터에서 다룬다.

5. 정리¶

항목 내용
정의 서버 측에서 클라이언트 요청을 받아 백엔드로 전달하는 중간 서버
위치 서버와 클라이언트 사이, 서버 쪽
주요 용도 보안, SSL 처리, 로드밸런싱, 가상호스트
누가 숨겨지나 서버가 클라이언트로부터 숨겨짐
대표 소프트웨어 nginx, Apache, HAProxy
NexClass 현황 iwinv 프록시에 nexclass 라우팅 미설정 -- ngrok으로 우회 중

한 줄 정리

리버스 프록시 = 서버 앞에 서서 클라이언트 요청을 대신 받는 문지기. 클라이언트는 실제 서버를 모른다.

확인 문제¶

다음 문제를 풀어봐. 답 못 하면 위에서 다시 읽어.

Q1. 포워드 프록시는 누구를 숨기고, 리버스 프록시는 누구를 숨기나?

Q2. hub.knu10.cc는 접속되는데 nexclass.knu10.cc는 안 되는 이유를 리버스 프록시 관점에서 설명해봐.

Q3. 리버스 프록시를 쓰면 백엔드 서버 포트(8082)를 외부에 직접 열 필요가 없는 이유는?

Q4. ngrok은 왜 리버스 프록시라고 할 수 있어?

Q5. 리버스 프록시가 필요한 이유 4가지를 말해봐.

정답 보기

A1. 포워드 프록시는 클라이언트를 숨긴다 (서버가 클라이언트를 모름). 리버스 프록시는 서버를 숨긴다 (클라이언트가 실제 서버를 모름).

A2. iwinv가 앞단에서 리버스 프록시 역할을 하는데, hub.knu10.cc에 대한 라우팅 규칙은 설정돼 있고, nexclass.knu10.cc에 대한 라우팅 규칙은 설정되지 않았다. 그래서 nexclass로 요청이 와도 iwinv 프록시가 어디로 보내야 할지 몰라서 접속이 안 되는 거다.

A3. 리버스 프록시가 외부 요청을 받아서 내부 백엔드 서버로 전달해주기 때문. 클라이언트는 프록시(80/443 포트)하고만 통신하고, 프록시가 내부적으로 localhost:8082에 전달한다. 백엔드 포트가 외부에 열릴 필요가 없다.

A4. ngrok은 외부 요청(https://abc-xyz.ngrok-free.dev)을 받아서 내부 서버(localhost:8082)로 전달해주는 구조다. 클라이언트는 ngrok 서버하고만 통신하고, 실제 백엔드 서버의 위치를 모른다. 리버스 프록시의 정의에 정확히 부합한다.

A5. (1) 보안 -- 서버 IP를 숨겨서 직접 공격을 방지 (2) SSL 처리 -- 프록시에서 HTTPS를 처리하고 백엔드에는 HTTP로 전달, 인증서 관리 일원화 (3) 로드밸런싱 -- 여러 백엔드 서버에 트래픽 분산 (4) 가상호스트 -- 같은 IP에서 도메인별로 다른 서비스로 라우팅
