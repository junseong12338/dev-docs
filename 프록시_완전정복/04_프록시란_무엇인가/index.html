04. 프록시란 무엇인가 - Alpha¶

이거 왜 알아야 하냐면¶

지금까지 IP, 포트, HTTP, DNS를 배웠어. 클라이언트가 서버에게 요청을 보내고 응답을 받는 흐름. 근데 현실 세계에서는 클라이언트와 서버가 직접 대화하는 경우가 거의 없어. 그 사이에 뭔가가 껴있어. 그게 프록시야. 우리 NexClass가 접속 안 되는 이유, ngrok을 쓰는 이유, iwinv가 앞단에서 뭔가 하는 이유 — 전부 프록시와 관련돼 있어. 이번 챕터에서 프록시의 본질을 잡으면, 나머지 챕터 전부가 "아 그래서 그랬구나"로 연결돼.

1. 핵심 개념 - "이게 뭐야?"¶

프록시 = 대리인¶

식당 비유로 가자.

  • 프록시 없는 세계: 손님이 직접 주방에 가서 주문하고, 직접 음식 가져온다.
  • 프록시 있는 세계: 손님이 웨이터에게 주문하고, 웨이터가 주방에 전달하고, 웨이터가 음식 가져다준다.

여기서 웨이터가 프록시야.

정확한 정의

프록시(Proxy)는 클라이언트와 서버 사이에서 요청과 응답을 중계하는 중간 서버야. "대리(Proxy)"라는 뜻 그대로, 클라이언트를 대신해서 서버에 요청하거나, 서버를 대신해서 클라이언트에 응답한다.

프록시의 본질¶

비유는 이해 돕기용이고, 진짜 핵심은 이거야:

프록시의 본질 한 줄

"중간에서 요청을 가로채서, 자기가 원하는 처리를 한 뒤, 대신 전달한다."

이 한 줄이 전부야. 프록시가 "자기가 원하는 처리"로 뭘 하느냐에 따라 기능이 달라지는 거지.

프록시가 하는 "처리" 기능 이름 비유
요청을 적절한 서버로 보내기 라우팅 웨이터가 한식 주문은 한식 주방으로, 양식은 양식 주방으로
자주 요청되는 걸 저장해두기 캐싱 웨이터가 물은 주방 안 거치고 직접 가져다줌
여러 서버에 골고루 나눠주기 로드밸런싱 웨이터가 바쁜 주방 대신 한가한 주방으로
위험한 요청 걸러내기 보안/필터링 웨이터가 수상한 손님 출입 차단
암호화 처리해주기 SSL 종료 웨이터가 봉투 뜯어서 주방에 전달
요청자 정보 숨기기 익명화 웨이터가 "누가 주문했는지"를 주방에 안 알려줌

2. 동작 원리 - "어떻게 돌아가?"¶

프록시 없는 세계 vs 있는 세계¶

sequenceDiagram
    participant C as 클라이언트
    participant S as 서버

    C->>S: 요청 (직접)
    S->>C: 응답 (직접)
    Note over C,S: 단순하지만, 서버가 직접 노출됨
sequenceDiagram
    participant C as 클라이언트
    participant P as 프록시
    participant S as 서버

    C->>P: 요청
    Note over P: 요청 분석/처리/변환
    P->>S: 요청 전달 (필요시 변환)
    S->>P: 응답
    Note over P: 응답 분석/처리/변환
    P->>C: 응답 전달

프록시가 없으면 클라이언트가 서버 IP와 포트를 직접 알아야 해. 프록시가 있으면 클라이언트는 프록시만 알면 돼. 서버가 어디있는지, 몇 번 포트인지 몰라도 돼.

프록시가 왜 생겨났는지¶

프록시는 처음부터 있던 게 아니야. 인터넷이 커지면서 직접 연결만으로는 해결 못 하는 문제가 생겼어.

문제 프록시가 해결하는 방법
서버 직접 노출 → 공격 위험 프록시 뒤에 서버를 숨겨
트래픽 폭주 → 서버 다운 여러 서버로 분산(로드밸런싱)
같은 데이터 반복 요청 → 느림 프록시가 캐시해서 바로 응답
하나의 IP에 여러 서비스 도메인별로 다른 서비스로 라우팅
내부 서버가 외부 접근 불가 프록시가 외부 요청을 대신 전달

우리 서버 실제 상황

iwinv 서버에 hub.knu10.cc, lms.knu10.cc, nexclass.knu10.cc 서비스가 전부 돌고 있어. IP는 하나(xxx.xxx.xx.xxx)인데 서비스는 셋이야. 어떻게 구분하냐? iwinv 앞단 프록시가 도메인 이름을 보고 적절한 서비스로 라우팅해줘. (nexclass는 이 라우팅 설정이 안 돼있어서 접속 불가)

두 가지 방향¶

프록시는 어디에 위치하느냐에 따라 크게 두 종류로 나뉘어:

graph LR
    subgraph 포워드 프록시
        C1[클라이언트] --> FP[프록시]
        FP --> S1[서버]
    end

    subgraph 리버스 프록시
        C2[클라이언트] --> RP[프록시]
        RP --> S2[서버]
    end
구분 포워드 프록시 리버스 프록시
위치 클라이언트 쪽 서버 쪽
누구 편? 클라이언트를 대리 서버를 대리
클라이언트가 아는 것 프록시 주소 프록시 주소 (서버 모름)
서버가 아는 것 프록시 IP (클라이언트 모름) 클라이언트 IP (프록시가 전달)
대표 예시 회사 방화벽, VPN nginx, iwinv 앞단 프록시
주요 목적 클라이언트 보호/제어 서버 보호/분산

우리한테 중요한 건 리버스 프록시

NexClass 입장에서는 리버스 프록시가 핵심이야. iwinv 앞단 프록시, nginx, ngrok 전부 리버스 프록시 역할이거든. 포워드 프록시는 개념 이해용이고, 나머지 챕터 대부분은 리버스 프록시를 다룬다. 각각의 상세는 05장(포워드), 06장(리버스)에서 깊이 파고든다.

3. 코드/설정으로 보자¶

iwinv가 왜 프록시를 두는지¶

우리 서버의 네트워크 구조를 보자:

graph TD
    A[브라우저] -->|"hub.knu10.cc"| B[iwinv 앞단 프록시]
    A -->|"lms.knu10.cc"| B
    A -->|"nexclass.knu10.cc"| B
    B -->|"hub → 내부서비스A"| C[HUB 서비스]
    B -->|"lms → 내부서비스B"| D[LMS 서비스]
    B -->|"nexclass → ???"| E[설정 없음!]
    F[NexClass Spring Boot :8082] -.->|"연결 안 됨"| B

    style E fill:#f66,stroke:#333,color:#fff
    style F fill:#ff9,stroke:#333

iwinv가 프록시를 두는 이유:

  1. 하나의 공인 IP로 여러 서비스 운영 — 도메인별 라우팅
  2. 포트 80/443만 외부에 개방 — 나머지 포트 보안 차단
  3. SSL 인증서 관리 — GoGetSSL 인증서를 프록시에서 처리

nexclass가 안 되는 진짜 이유

  1. DNS는 정상 (nexclass.knu10.cc → xxx.xxx.xx.xxx)
  2. iwinv 앞단 프록시에 도착
  3. 프록시가 "nexclass.knu10.cc? 이거 어디로 보내야 하지?" → 라우팅 규칙 없음
  4. 접속 실패

해결하려면: iwinv에 "nexclass.knu10.cc 요청은 localhost:8082로 보내줘"라고 프록시 라우팅 설정을 요청해야 해. 또는 nginx를 앞단에 직접 세우거나, ngrok으로 우회하거나.

ngrok도 프록시야¶

ngrok은 터널링 + 리버스 프록시 역할을 한다:

sequenceDiagram
    participant Z as Zoom Webhook
    participant N as ngrok 서버
    participant L as localhost:8082

    Z->>N: POST https://abc-xyz.ngrok-free.dev/webhook
    Note over N: ngrok이 요청을 중계 (프록시!)
    N->>L: POST http://localhost:8082/webhook
    L->>N: 200 OK
    N->>Z: 200 OK

외부(Zoom)에서 내부(localhost:8082)로 직접 접근할 수 없으니까, ngrok이 중간에서 대리(프록시)해주는 거야. 프록시의 본질 — "중간에서 요청을 가로채서 대신 전달" — 그대로지?

4. 주의사항/함정¶

함정 1: 프록시 = VPN?

아니야. VPN은 모든 트래픽을 암호화해서 터널로 보내는 거고, 프록시는 특정 요청을 중계하는 거야. VPN은 네트워크 계층, 프록시는 애플리케이션 계층에서 동작해. 겹치는 부분이 있지만 같은 건 아니야. 이건 19장에서 정확히 비교한다.

함정 2: 프록시를 거치면 느려지지 않나?

중간에 하나가 더 끼니까 이론적으로는 맞아. 하지만 실제로는:

  • 캐싱으로 더 빨라지는 경우가 더 많아
  • 로드밸런싱으로 서버 과부하를 막아서 전체적으로 빨라져
  • 지리적으로 가까운 프록시(CDN)가 응답하면 훨씬 빨라

프록시의 오버헤드(추가 지연)는 보통 밀리초(ms) 수준이고, 얻는 이점이 훨씬 커.

함정 3: 프록시가 다 해주니까 서버 보안 안 해도 되지?

절대 아니야. Defense in Depth(깊이 있는 방어)가 원칙이야. 프록시가 뚫리면? 서버가 바로 노출돼. 프록시에서 보안 처리 + 서버에서도 보안 처리. 둘 다 해야 해.

함정 4: 프록시 vs 게이트웨이 vs 로드밸런서

이 용어들이 혼용되는 경우가 많은데:

용어 핵심 역할 겹치는 부분
프록시 요청 중계 라우팅, 캐싱, 보안
게이트웨이 네트워크 경계의 문 프록시 + 프로토콜 변환
로드밸런서 트래픽 분산 프록시의 한 기능

실무에서는 이 세 가지를 하나의 소프트웨어(nginx, HAProxy 등)가 다 해. 그래서 경계가 모호해.

5. 정리¶

개념 핵심 우리 프로젝트에서
프록시 중간에서 요청을 가로채서 대신 전달 iwinv 앞단, nginx, ngrok
포워드 프록시 클라이언트를 대리 회사 방화벽 뒤에서 인터넷 접속
리버스 프록시 서버를 대리 iwinv가 도메인별로 서비스 분기
프록시 핵심 기능 라우팅, 캐싱, 로드밸런싱, 보안 nexclass 라우팅 미설정이 원인 
graph TD
    A[프록시의 본질] --> B[중간에서 가로챈다]
    B --> C[라우팅]
    B --> D[캐싱]
    B --> E[로드밸런싱]
    B --> F[보안/필터링]
    B --> G[SSL 처리]
    B --> H[익명화]

한 줄 정리

프록시는 클라이언트와 서버 사이에서 요청을 가로채고, 처리하고, 대신 전달하는 중간 서버다. 인터넷의 웨이터.

이 챕터에서 반드시 기억할 것

  1. 프록시의 본질 = "중간에서 가로채서 대신 전달"
  2. 포워드 프록시는 클라이언트 편, 리버스 프록시는 서버 편
  3. 프록시가 하는 일: 라우팅, 캐싱, 로드밸런싱, 보안, SSL
  4. 우리 nexclass가 접속 안 되는 이유 = 프록시 라우팅 미설정
  5. ngrok도 프록시의 한 형태야 (터널링 + 리버스 프록시)

확인 문제¶

다음 문제를 풀어봐. 답 못 하면 위에서 다시 읽어.

Q1. 프록시의 본질을 한 문장으로 정의해봐.

Q2. 포워드 프록시와 리버스 프록시의 핵심 차이를 "누구 편이냐"로 설명해봐.

Q3. iwinv 서버에 hub.knu10.cc, lms.knu10.cc, nexclass.knu10.cc 세 서비스가 있다. IP는 하나인데 서비스가 셋이야. 프록시가 이 세 서비스를 어떻게 구분해서 라우팅하는지 설명해봐.

Q4. ngrok이 프록시라고 할 수 있는 이유를 "프록시의 본질"에 비춰서 설명해봐.

Q5. "프록시가 있으면 서버 보안은 안 해도 된다"는 주장이 왜 틀렸는지 설명해봐.

정답 보기

A1. 프록시는 클라이언트와 서버 사이에서 요청을 가로채고, 자기가 원하는 처리를 한 뒤, 대신 전달하는 중간 서버다.

A2. 포워드 프록시는 클라이언트 편이야. 클라이언트를 대리해서 서버에 요청을 보내줘 (서버는 클라이언트가 누군지 모름). 리버스 프록시는 서버 편이야. 서버를 대리해서 클라이언트에게 응답해줘 (클라이언트는 뒷단 서버가 어디있는지 모름).

A3. HTTP 요청의 Host 헤더를 본다. 브라우저가 hub.knu10.cc로 요청하면 Host: hub.knu10.cc가 헤더에 들어있고, nexclass.knu10.cc로 요청하면 Host: nexclass.knu10.cc가 들어있어. 프록시가 이 Host 값을 읽고, 미리 설정된 라우팅 규칙에 따라 해당 서비스로 전달한다. 이걸 가상 호스트(Virtual Host) 기반 라우팅이라고 해.

A4. ngrok은 외부(Zoom)의 요청을 받아서, 내부(localhost:8082)로 대신 전달한다. 외부 요청을 가로채서, 내부로 대신 전달하는 거야. 프록시의 본질("중간에서 가로채서 대신 전달")에 정확히 부합해. 추가로 HTTPS 처리(SSL 종료)도 해주니까 리버스 프록시 + 터널링을 겸하고 있어.

A5. Defense in Depth(깊이 있는 방어) 원칙 때문이야. 프록시가 뚫리면 서버가 바로 노출돼. 프록시 보안은 첫 번째 방어선이지 유일한 방어선이 아니야. 서버 자체에서도 인증(JWT), 인가(권한 체크), 입력 검증, SQL Injection 방어 등을 독립적으로 해야 해. 한 겹이 무너져도 다음 겹이 방어하는 구조가 되어야 안전하다.
