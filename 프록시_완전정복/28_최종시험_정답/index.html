28. 최종시험 - 정답 및 해설¶

해설지 사용법

"정답 맞히는 건 시작이야. 왜 그게 정답인지 설명 못 하면 불합격."

이건 정답지가 아니야. 해설지야. 정답만 보고 넘어가면 Lv1에서 영원히 못 벗어나. 틀린 문제는 해당 챕터 다시 읽고 와.

Part 1: 객관식 정답 (10문제, 각 2점 = 20점)¶

Q1. 포트(Port) 정의 (객관식, 2점)¶

정답: (C) 하나의 IP 주소에서 어떤 프로그램에 데이터를 전달할지 구분하는 숫자다

해설:

보기 설명
(A) 물리적 주소는 MAC 주소야. 포트는 논리적 숫자.
(B) 0~1023은 Well-Known 포트일 뿐이야. 전체 범위는 0~65535. NexClass는 8082를 쓰잖아.

(D) 포트 80은 HTTP, SSH는 22번이야. 기본 포트 헷갈리면 Ch01 다시 읽어.

Q5. SSL/TLS 오답 찾기 (객관식, 2점)¶

(B) 맞아. SSL 인증서는 서버 신원 보증(CA가 발급) + 공개키 포함.
(C) 맞아. SSL 3.0 이후 TLS 1.0, 1.1, 1.2, 1.3으로 발전. 현재는 TLS를 쓰지만 관습적으로 SSL이라고 부르기도 해.
(D) 정답(틀린 것). SSL/TLS는 전송 구간(Transport Layer)만 암호화해. 브라우저↔서버 사이의 데이터 전송만. DB 내부 데이터 암호화는 SSL과 무관해. 그건 별도의 저장소 암호화(Encryption at Rest)야.

이거 자주 혼동해

SSL = "데이터가 이동할 때" 보호. DB 암호화 = "데이터가 저장될 때" 보호. 완전히 다른 레이어야.

Q16. Cloudflare Tunnel vs ngrok 운영 차이 (단답형, 3점)¶

Q17. DNS만으로 접속 안 되는 이유 (단답형, 3점)¶

정답: DNS는 도메인→IP 변환만 해줄 뿐이고, 해당 IP의 80/443 포트에서 요청을 받아주는 프록시(또는 웹 서버)가 Host 헤더 기반으로 해당 도메인을 라우팅하는 설정이 필요하다. iwinv 앞단 프록시에 nexclass.knu10.cc 라우팅이 설정되어 있지 않으면 요청이 전달되지 않는다.

해설: DNS는 "주소록"이야. "nexclass.knu10.cc = xxx.xxx.xx.xxx"라는 걸 알려줄 뿐이지, 그 IP에 도착한 후 누가 그 요청을 받아서 처리해줄 건지는 DNS 관할이 아니야. iwinv 앞단 프록시가 "nexclass.knu10.cc"라는 Host를 알아야 뒤로 전달해줌.

채점 기준:

  • "프록시/웹 서버에 라우팅 설정 필요" 언급: 2점
  • "iwinv 앞단 프록시"나 "Host 헤더" 구체적 언급: 1점
  • "방화벽 설정 필요" 정도만 쓴 경우: 1점 (부분적으로 맞지만 핵심이 아님)

Q18. WAF 역할과 일반 방화벽 차이 (단답형, 3점)¶

정답: WAF는 HTTP 요청의 내용(본문, 헤더, URL 파라미터)을 검사하여 SQL Injection, XSS 등 애플리케이션 레벨 공격을 탐지/차단한다. 일반 방화벽과의 가장 큰 차이는 동작 계층이다 --- 일반 방화벽은 3-4계층(IP/포트 기반), WAF는 7계층(HTTP 내용 기반).

해설: 일반 방화벽은 "이 IP/포트에서 오는 트래픽을 허용/차단"만 해. 패킷 안에 뭐가 들어있는지는 안 봐. WAF는 "이 HTTP 요청에 ' OR 1=1 -- 같은 SQL Injection 패턴이 있네? 차단!" 이런 걸 해. 동작하는 레이어가 근본적으로 달라.

채점 기준:

  • WAF 역할(애플리케이션 공격 탐지/차단) 설명: 1.5점
  • 일반 방화벽과의 차이(동작 계층 또는 검사 대상) 설명: 1.5점
  • "보안 장치" 같은 일반적인 답변: 0.5점

Q20. nginx 설정 오류 찾기 (코드 분석, 5점)¶

정답:

오류 1: server_name 누락 server_name 지시문이 없어서 모든 도메인의 요청을 이 서버 블록이 받게 된다. 다른 도메인(hub.knu10.cc, lms.knu10.cc) 요청까지 처리하게 될 수 있어.

수정: proxy_pass http://localhost:8082; (URI 경로 제거) 또는 location /api/로 통일

해설: nginx의 proxy_pass에 URI 경로가 포함되면 location의 매칭 부분이 잘려나가고 proxy_pass의 URI로 대체돼. 이 치환 규칙을 모르면 경로가 꼬여. 그리고 server_name 없으면 default server로 동작해서 의도치 않은 요청까지 받아.

정답 (모범 답안):

장점 1: 백엔드 서버 부하 감소 SSL 핸드셰이크와 암호화/복호화는 CPU를 많이 먹어. 이걸 앞단 프록시에서 처리하면 백엔드 Spring Boot는 비즈니스 로직에만 집중할 수 있다.

장점 2: 인증서 관리 단순화 인증서를 앞단 한 곳에서만 관리하면 돼. 백엔드 서버가 여러 대여도 인증서 갱신은 프록시에서만 하면 된다.

단점 1: 프록시~백엔드 구간 평문 노출 SSL 종료 후 백엔드까지는 HTTP 평문이야. 내부 네트워크가 뚫리면 데이터가 노출된다. 완화: 내부 구간도 TLS 적용(내부 mTLS) 또는 신뢰할 수 있는 격리된 네트워크에서만 운영.

단점 2: 프록시가 단일 장애점(SPOF)이 될 수 있음 모든 SSL 처리가 프록시에 집중되므로, 프록시가 죽으면 전체 서비스가 중단된다. 완화: 프록시 이중화(Active-Standby 또는 Active-Active).

채점 기준:

  • 장점 2가지: 각 0.5점 (1점)
  • 단점 2가지: 각 0.5점 (1점)
  • 완화 방법 각각: 각 0.5점 (1점)
  • 전체 논리 흐름: 1점
  • 장점/단점만 쓰고 완화 없으면: -1점

Q28. DNS vs Tunnel 택배 비유 + 기술 설명 (서술형, 4점)¶

정답 (모범 답안):

택배 비유: DNS A레코드 설정은 집 주소를 주소록에 등록하는 것이야. "nexclass.knu10.cc = xxx.xxx.xx.xxx"라고 등록하면 택배 기사(브라우저)가 주소를 보고 찾아올 수 있어. 하지만 집 앞에 자물쇠 걸린 대문(방화벽)이 있으면 택배를 전달할 수 없어. 주소를 알아도 문이 안 열리면 소용없는 거야.

Cloudflare Tunnel은 집 안에서 밖으로 파이프를 내보내는 것이야. 대문이 잠겨 있어도 상관없어. 집 안에서 Cloudflare 창고(클라우드)까지 파이프를 연결해두면, 택배 기사가 Cloudflare 창고에 맡기면 파이프를 타고 집 안으로 들어온다. 주소록에는 "Cloudflare 창고"가 등록돼 있어.

기술적 설명: DNS A레코드는 도메인을 서버의 공인 IP에 직접 매핑한다. 요청이 그 IP의 80/443 포트에 도달하려면 방화벽과 프록시가 해당 포트를 열고 라우팅해줘야 한다 (인바운드 방식). Cloudflare Tunnel은 서버 내부의 cloudflared가 Cloudflare 클라우드로 아웃바운드 연결을 먼저 열고, DNS는 Cloudflare의 프록시 IP를 가리킨다. 외부 요청은 Cloudflare → 터널 → 서버 순서로 전달되므로 서버의 인바운드 포트를 열 필요가 없다.

채점 기준:

  • 택배 비유 적절: 1.5점
  • 기술적 설명 정확: 1.5점
  • 인바운드/아웃바운드 차이 언급: 1점
  • 비유만 쓰고 기술 설명 없으면: 1.5점 (질문에 "반드시 기술적 설명 추가"라고 했어)
  • 기술 설명만 쓰고 비유 없으면: 2점

Q29. NexClass 접속 문제 해결 방안 3가지 + 트레이드오프 (서술형, 5점)¶

정답 (모범 답안):

방안 1: iwinv에 정식 프록시 + SSL 요청

  • 장점: hub/lms와 동일한 검증된 구조. 고정 도메인. 외부 의존 최소화. 안정적.
  • 단점: iwinv 대응 속도에 의존 (요청 후 며칠~몇 주). 직접 제어 불가.
  • 선택 시점: 시간 여유가 있고, 장기 프로덕션 운영이 목표일 때. 최선의 선택.

방안 2: Cloudflare Tunnel

  • 장점: iwinv 요청 없이 즉시 설정 가능. 고정 도메인. SSL + WAF + CDN 자동 제공. systemd 서비스로 자동 복구.
  • 단점: Cloudflare 외부 서비스에 의존. 무료 플랜 기능 제한. DNS를 Cloudflare로 관리해야 함. iwinv 앞단 프록시와 경로 충돌 가능.
  • 선택 시점: iwinv 응답이 느리고, 빠르게 프로덕션 수준의 서비스가 필요할 때. 차선의 선택.

방안 3: ngrok (현재 상태 유지)

  • 장점: 5분 안에 설정 완료. 개발/테스트 환경에서 편리. 별도 설정 불필요.
  • 단점: 무료 플랜 URL 변경. 터미널 의존. 트래픽 제한. 프로덕션 부적합. Zoom Webhook URL 수시 변경 필요.
  • 선택 시점: 개발 단계에서 빠르게 테스트만 하면 될 때. 임시 방편일 뿐.

트레이드오프 핵심: 안정성과 제어권 vs 설정 속도와 편의성의 트레이드오프. 방안 1은 안정적이지만 느리고, 방안 3은 빠르지만 불안정하고, 방안 2는 중간이다.

채점 기준:

  • 방안 3가지 제시: 각 0.5점 (1.5점)
  • 각 방안의 장점 + 단점: 각 0.5점 (1.5점)
  • 각 방안의 선택 시점(트레이드오프): 각 ~0.33점 (1점)
  • 전체 트레이드오프 비교/분석: 1점
  • 방안 2가지만 쓴 경우: 최대 3점
  • 장단점 없이 방안만 나열: 1.5점

불합격 기준

"iwinv에 요청한다", "ngrok 쓴다" 같은 한 줄 답변 3개는 1점. 트레이드오프 분석이 없으면 설계 판단을 안 한 거야.

Q30. NexClass 아키텍처 개선안 - 보안/성능/가용성 (서술형, 4점)¶

정답 (모범 답안):

보안 개선: Cloudflare 또는 iwinv 앞단에서 WAF 활성화 현재 ngrok 경유 시 WAF가 없어서 SQL Injection, XSS 등 애플리케이션 레벨 공격에 무방비다. Cloudflare Tunnel로 전환하면 Cloudflare의 WAF가 자동 적용되어 악성 요청을 앱 서버에 도달하기 전에 차단할 수 있다. 또는 iwinv 정식 설정 시 nginx에서 기본적인 요청 필터링 규칙을 추가할 수 있다.

성능 개선: CDN 캐싱 도입 (정적 리소스) 현재 모든 요청이 ngrok → Spring Boot까지 도달해야 해. 정적 파일(CSS, JS, 이미지)까지 전부. Cloudflare CDN을 도입하면 정적 리소스는 엣지 서버에서 캐시로 제공되어 원본 서버 부하가 줄고 응답 속도가 빨라진다. ngrok 경유 레이턴시(~50-100ms 추가)도 제거된다.

가용성 개선: ngrok 탈출 + systemd 서비스 등록 현재 최대 약점은 ngrok 프로세스 의존이야. 터미널 닫히면 서비스 전체가 중단되고 URL도 바뀐다. Cloudflare Tunnel의 cloudflared를 systemd 서비스로 등록하면, 서버 재시작 시 자동으로 터널이 복구되고, 고정 도메인이 유지되어 Zoom Webhook URL을 다시 설정할 필요도 없다.

채점 기준:

  • 보안 개선안 + 근거: ~1.33점
  • 성능 개선안 + 근거: ~1.33점
  • 가용성 개선안 + 근거: ~1.33점
  • 각 개선안이 "왜 현재보다 나은지" 근거 명확: 포함된 점수
  • 개선안만 쓰고 근거 없으면: 각 0.5점
  • 3가지 관점 중 2가지만 답변: 최대 2.66점

불합격 기준

"보안 강화한다, 성능 올린다, 안정성 높인다" 같은 추상적 답변은 0점. 구체적인 어떻게(방법)와 왜(근거)가 있어야 해.
