25. "접속 안 돼요" 디버깅 - Omega¶

1. 상황 설정¶

"접속 안 돼요"는 개발 인생에서 가장 많이 듣는 신고야. 근데 이 한 마디 안에 원인이 최소 7가지는 숨어있어. DNS 문제일 수도 있고, 방화벽일 수도 있고, SSL일 수도 있고, 앱이 죽었을 수도 있어.

이 챕터에서는 위에서 아래로 계층별로 진단하는 체계적인 방법을 배운다. 찍기가 아니라 과학적 디버깅이야.

찍기 vs 체계적 디버깅

  • 찍기: "혹시 이거 아닐까?" → 랜덤으로 이것저것 만져봄 → 시간 낭비
  • 체계적: "7계층 위에서부터 순서대로 확인" → 원인 빠르게 특정 → 해결

2. 전체 진단 흐름¶

진단 순서: 위에서 아래로¶

왜 위에서 아래로 하냐? 바깥쪽(DNS)이 안 되면 안쪽(앱)을 확인해봐야 소용없거든. 건물 주소가 틀리면 엘리베이터가 잘 작동하는지 확인할 이유가 없는 거야.

3. 계층별 진단 명령어¶

1단계: DNS 확인¶

질문: 도메인이 IP 주소로 변환되나?

nslookup nexclass.knu10.cc

정상 결과: 

Server:   8.8.8.8
Address:  8.8.8.8#53

Non-authoritative answer:
Name:    nexclass.knu10.cc
Address: xxx.xxx.xx.xxx

실패 결과: 

** server can't find nexclass.knu10.cc: NXDOMAIN


dig nexclass.knu10.cc +short

정상: xxx.xxx.xx.xxx

실패: 아무것도 안 나옴

결과 의미 조치
IP 반환됨 DNS 정상 다음 단계로
NXDOMAIN 도메인 자체가 등록 안 됨 DNS 레코드 확인 (Cloudflare 대시보드)
SERVFAIL DNS 서버 오류 네임서버 설정 확인
잘못된 IP A 레코드 잘못 설정됨 Cloudflare에서 A 레코드 수정

우리 상황

nexclass.knu10.cc의 DNS는 정상이야. Cloudflare에 A 레코드가 설정돼 있어서 xxx.xxx.xx.xxx로 잘 변환돼. DNS는 문제가 아니야.

2단계: 네트워크 도달 확인¶

질문: 해당 IP의 서버까지 패킷이 도달하나?

curl -v https://nexclass.knu10.cc 2>&1

이 명령 하나로 DNS, TCP 연결, SSL, HTTP 응답을 전부 확인 가능.

성공 시: 

*   Trying xxx.xxx.xx.xxx:443...
* Connected to nexclass.knu10.cc (xxx.xxx.xx.xxx) port 443
* SSL connection using TLS 1.3
< HTTP/1.1 200 OK

타임아웃 시: 

*   Trying xxx.xxx.xx.xxx:443...
* connect to xxx.xxx.xx.xxx port 443 failed: Connection timed out

연결 거부 시: 

*   Trying xxx.xxx.xx.xxx:443...
* connect to xxx.xxx.xx.xxx port 443 failed: Connection refused


ping xxx.xxx.xx.xxx

주의: ping(ICMP)이 막혀있어도 HTTP는 될 수 있어. ping 실패 = 서버 죽음은 아니야.

traceroute nexclass.knu10.cc

# Windows
tracert nexclass.knu10.cc

패킷이 어디까지 가다가 멈추는지 확인 가능.

결과 의미
Connection timed out 방화벽이 패킷을 DROP (무시)
Connection refused 서버에 도달했지만 해당 포트에 서비스 없음
200 OK 정상 연결

3단계: 프록시 확인¶

질문: iwinv 프록시가 nexclass.knu10.cc 요청을 서버로 라우팅하나?

이건 직접 명령어로 확인하기 어려워. iwinv 측 설정이니까.

# 간접 확인: 같은 IP의 다른 도메인이 되는지 비교
curl -v https://hub.knu10.cc 2>&1 | head -20    # 되면 iwinv 프록시 자체는 정상
curl -v https://nexclass.knu10.cc 2>&1 | head -20  # 안 되면 라우팅 미설정
hub.knu10.cc nexclass.knu10.cc 결론
200 OK timeout/거부 iwinv 프록시에 nexclass 라우팅 미설정
200 OK 200 OK 프록시 정상, 다른 문제
timeout timeout iwinv 네트워크 전체 문제

우리 서버의 현재 상태

hub.knu10.cc는 200 OK, nexclass.knu10.cc는 타임아웃. 결론: iwinv 프록시에 nexclass 라우팅이 설정 안 됨. 22장에서 분석한 그 문제.

4단계: SSL 확인¶

질문: SSL 인증서가 유효한가?

# SSL 인증서 상세 확인
openssl s_client -connect nexclass.knu10.cc:443 -servername nexclass.knu10.cc 2>/dev/null | openssl x509 -noout -text | head -30

확인할 항목:

항목 확인 내용 명령어 일부
도메인 일치 인증서 CN/SAN에 도메인 포함 Subject:, Subject Alternative Name:
유효기간 만료 안 됐는지 Not Before:, Not After:
발급자 신뢰할 수 있는 CA인지 Issuer: 
# 인증서 만료일만 빠르게 확인
echo | openssl s_client -connect nexclass.knu10.cc:443 -servername nexclass.knu10.cc 2>/dev/null | openssl x509 -noout -dates

정상: 

notBefore=Jan 15 00:00:00 2026 GMT
notAfter=Jan 15 23:59:59 2027 GMT

우리 상황

GoGetSSL 인증서가 knu10.cc, hub, lms, www만 커버한다. nexclass는 미포함. Let's Encrypt 인증서가 /etc/letsencrypt/live/nexclass.knu10.cc/에 발급돼 있지만 미사용 상태.

5단계: nginx 확인¶

질문: nginx가 정상 동작하나?

# nginx 설정 문법 검사
sudo nginx -t

정상: 

nginx: the configuration file /etc/nginx/nginx.conf syntax is ok
nginx: configuration file /etc/nginx/nginx.conf test is successful

에러: 

nginx: [emerg] unknown directive "proxypass" in /etc/nginx/sites-enabled/nexclass:5
nginx: configuration file /etc/nginx/nginx.conf test failed


# nginx 프로세스 확인
ps aux | grep nginx
# 결과에 master process, worker process가 보여야 정상

# nginx 에러 로그 확인
sudo tail -50 /var/log/nginx/error.log

# nginx 접근 로그 확인 (요청이 nginx까지 오는지)
sudo tail -50 /var/log/nginx/access.log
확인 정상 비정상
nginx -t syntax is ok test failed
프로세스 master + worker 존재 프로세스 없음
access.log 요청 기록 있음 기록 없음 (트래픽 안 옴)
error.log 에러 없음 연결 오류 등

access.log에 기록이 없다면

nginx 자체 문제가 아니라, 트래픽이 nginx까지 안 오는 거야. 앞단(iwinv 프록시)에서 막힌 거지. 22장에서 설명한 "무용지물" 상황.

6단계: 앱 확인¶

질문: Spring Boot가 살아있나?

# 헬스 체크 (서버 내부에서)
curl http://localhost:8082/health
# 정상: {"status":"UP"}

# Java 프로세스 확인
jps -l
# 결과에 NexClass.jar이 보여야 함

# 또는
ps aux | grep java

# 포트 확인
sudo lsof -i :8082
# 결과: java ... TCP *:8082 (LISTEN)
확인 정상 비정상
/health {"status":"UP"} Connection refused
jps -l NexClass.jar 보임 안 보임
lsof -i :8082 LISTEN 상태 결과 없음

앱이 죽었으면 재시작:

# 재시작
cd /usr/local/nexclass
nohup java -jar NexClass.jar &

# 로그 확인
tail -f /home/ubuntu/nohup.out

앱이 떴는데 응답이 이상하면

/health은 되는데 다른 API가 안 되면 코드 레벨 문제야. Spring Boot 로그(/home/ubuntu/nohup.out)를 확인해야 해.

7단계: 방화벽 확인¶

질문: OS 레벨 방화벽이 막고 있나?

# ufw 상태 확인
sudo ufw status

# iptables 규칙 확인
sudo iptables -L -n

# 특정 포트 허용 (필요 시)
sudo ufw allow 8082/tcp

서버 방화벽 vs iwinv 방화벽

서버의 ufw/iptables를 열어도 iwinv 네트워크 레벨에서 막으면 소용없어. 이건 22장에서 설명한 포트 8082 차단 문제야. 계층이 다르다는 걸 인식해야 해.

4. 에러 메시지별 원인 분석¶

브라우저나 curl에서 나오는 에러 메시지가 어떤 계층의 문제인지 바로 파악하는 표야.

에러 메시지 의미 문제 계층 확인 방법
ERR_NAME_NOT_RESOLVED 도메인을 IP로 못 바꿈 1단계: DNS nslookup
ERR_CONNECTION_TIMED_OUT 서버에 도달 못 함 (패킷 DROP) 2~3단계: 방화벽/프록시 curl -v, 포트 확인
ERR_CONNECTION_REFUSED 서버에 도달했지만 포트에 서비스 없음 6단계: 앱 lsof -i, jps -l
ERR_SSL_PROTOCOL_ERROR SSL 핸드셰이크 실패 4단계: SSL openssl s_client
ERR_CERT_COMMON_NAME_INVALID 인증서 도메인 불일치 4단계: SSL 인증서 SAN 확인
502 Bad Gateway 프록시/nginx는 살았지만 백엔드 죽음 6단계: 앱 curl localhost:8082
403 Forbidden 접근 거부됨 3단계: 프록시 또는 앱 프록시 설정 + 앱 로그
504 Gateway Timeout 백엔드 응답 시간 초과 6단계: 앱 앱 로그, DB 연결 확인

5. 실전 사례: "nexclass.knu10.cc 접속 안 됨" 디버깅 재현¶

실제로 우리가 겪은 문제를 처음부터 디버깅하는 과정을 재현해보자.

신고 내용¶

"nexclass.knu10.cc에 접속하면 페이지가 안 뜹니다."

디버깅 시작¶

1단계: DNS 확인

$ nslookup nexclass.knu10.cc
Name:    nexclass.knu10.cc
Address: xxx.xxx.xx.xxx

DNS 정상

IP가 정상 반환됐어. DNS 문제는 아니야.

2단계: 네트워크 도달 확인

$ curl -v https://nexclass.knu10.cc 2>&1
*   Trying xxx.xxx.xx.xxx:443...
* connect to xxx.xxx.xx.xxx port 443 failed: Connection timed out
* Failed to connect to nexclass.knu10.cc port 443: Connection timed out

타임아웃!

IP까지의 443 포트 연결이 타임아웃. 서버에 도달 못 했거나, 도달했지만 응답 안 함.

3단계: 같은 IP의 다른 도메인으로 비교

$ curl -v https://hub.knu10.cc 2>&1 | head -10
*   Trying xxx.xxx.xx.xxx:443...
* Connected to hub.knu10.cc (xxx.xxx.xx.xxx) port 443
* SSL connection using TLS 1.3
< HTTP/1.1 200 OK

hub은 되고 nexclass만 안 된다

같은 IP인데 hub은 200, nexclass는 타임아웃. 서버나 네트워크 자체의 문제가 아니야. 특정 도메인만 안 되는 거지.

결론 도출: 같은 IP에서 hub은 되고 nexclass만 안 되면, 가능성은 두 가지:

  1. iwinv 프록시에 nexclass 라우팅이 없다
  2. SSL 인증서에 nexclass가 포함 안 됐다

4단계: SSL 인증서 확인

# hub의 인증서 확인 (성공하는 도메인)
$ echo | openssl s_client -connect hub.knu10.cc:443 -servername hub.knu10.cc 2>/dev/null | openssl x509 -noout -subject -ext subjectAltName

subject=CN = knu10.cc
X509v3 Subject Alternative Name:
    DNS:knu10.cc, DNS:hub.knu10.cc, DNS:lms.knu10.cc, DNS:www.knu10.cc

nexclass.knu10.cc가 없어

GoGetSSL 인증서의 SAN(Subject Alternative Name)에 nexclass가 포함 안 됐어. hub, lms, www만 있어.

5~6단계: 서버 내부 확인 (SSH 접속 후)

# nginx 설정 확인
$ sudo nginx -t
nginx: the configuration file /etc/nginx/nginx.conf syntax is ok

# Spring Boot 확인
$ curl http://localhost:8082/health
{"status":"UP"}

# 포트 확인
$ sudo lsof -i :8082
java  12345 ... TCP *:8082 (LISTEN)

서버 내부는 전부 정상

nginx 설정 OK, Spring Boot 살아있음, 포트 리스닝 중. 문제는 서버 안이 아니라 바깥(iwinv 프록시)이야.

6. 정리¶

진단 순서 요약¶

단계 확인 대상 핵심 명령어 정상 기준
1 DNS nslookup IP 반환
2 네트워크 curl -v Connected
3 프록시 다른 도메인 비교 동일 응답
4 SSL openssl s_client 인증서 유효 + 도메인 매칭
5 nginx nginx -t, 로그 syntax ok
6 앱 curl localhost:포트 응답 반환
7 방화벽 ufw status 포트 열림

에러 -> 계층 빠른 매핑¶

에러 계층
ERR_NAME_NOT_RESOLVED DNS
ERR_CONNECTION_TIMED_OUT 방화벽/프록시
ERR_CONNECTION_REFUSED 앱 죽음
502 Bad Gateway 앱 죽음 (nginx는 살아있음)
SSL 에러 인증서

한 줄 정리

"접속 안 돼요" 디버깅은 DNS -> 네트워크 -> 프록시 -> SSL -> nginx -> 앱 -> 방화벽 순서로, 바깥에서 안쪽으로 좁혀가는 거야.

이 챕터에서 반드시 기억할 것

  1. 진단은 위에서 아래로 (바깥에서 안쪽). DNS부터 시작해.
  2. curl -v가 가장 강력한 진단 도구. 한 번에 DNS + TCP + SSL + HTTP 전부 확인.
  3. 에러 메시지가 계층을 알려준다. TIMED_OUT = 방화벽/프록시, REFUSED = 앱 죽음.
  4. 같은 IP의 다른 도메인이 되는지 비교하면 프록시 문제 특정 가능.
  5. 서버 내부가 전부 정상이면 문제는 바깥(프록시/방화벽)에 있다.

확인 문제¶

다음 문제를 풀어봐. 답 못 하면 위에서 다시 읽어.

Q1. "접속 안 돼요" 디버깅은 왜 DNS(바깥)부터 시작하나? 앱(안쪽)부터 확인하면 안 되는 이유는?

Q2. curl -v https://nexclass.knu10.cc를 쳤더니 Connection timed out이 나왔다. 이게 의미하는 건 뭐고, 다음으로 뭘 확인해야 해?

Q3. ERR_CONNECTION_REFUSED와 ERR_CONNECTION_TIMED_OUT의 차이를 설명하고, 각각 어떤 계층의 문제인지 말해봐.

Q4. 502 Bad Gateway가 뜨면 nginx가 문제야, Spring Boot가 문제야? 왜?

Q5. hub.knu10.cc는 접속되고 nexclass.knu10.cc는 안 된다. 둘 다 같은 서버 IP를 가리킨다. 이 정보만으로 어떤 결론을 내릴 수 있어?

정답 보기

A1. 바깥쪽(DNS, 네트워크)이 안 되면 안쪽(nginx, Spring Boot)을 확인해봐야 의미가 없기 때문이야. DNS에서 IP를 못 받으면 패킷이 서버까지 가지도 않는데, Spring Boot가 살아있는지 확인할 이유가 없지. 바깥부터 확인해서 문제 범위를 빠르게 좁혀가는 게 효율적이야.

A2. Connection timed out은 TCP 연결 요청(SYN 패킷)을 보냈는데 응답(SYN-ACK)이 안 왔다는 뜻이야. 서버에 도달 못 했거나, 도달했지만 방화벽이 패킷을 DROP한 거지. 다음으로 확인할 건: (1) 같은 IP의 다른 도메인(hub.knu10.cc)이 되는지 비교, (2) 직접 IP:포트로 접근 시도. 다른 도메인이 되면 iwinv 프록시의 라우팅 문제.

A3. TIMED_OUT은 패킷을 보냈는데 아무 응답이 없는 거야. 방화벽이 패킷을 DROP(무시)하면 이렇게 돼. 문제 계층: 방화벽 또는 프록시. REFUSED는 서버에 도달은 했지만, 해당 포트에서 연결을 거부한 거야. 서비스(앱)가 안 돌고 있을 때 이렇게 돼. 문제 계층: 애플리케이션.

A4. Spring Boot가 문제야. 502 Bad Gateway는 "프록시(nginx)가 백엔드(Spring Boot)에 요청을 전달하려고 했는데, 백엔드가 응답하지 않는다"는 뜻이야. nginx 자체는 살아있으니까 요청을 받을 수 있었고, proxy_pass로 전달하려 했지만 Spring Boot가 죽었거나 응답을 못 한 거지. 확인: curl http://localhost:8082/health.

A5. DNS는 정상이고, 서버 자체도 정상이라는 결론을 내릴 수 있어. 같은 IP로 hub은 되니까 서버가 살아있고 네트워크도 정상이야. 차이점은 iwinv 프록시의 도메인별 라우팅 설정이야. hub은 라우팅 규칙이 있고 nexclass는 없어서, iwinv 프록시가 nexclass 요청을 서버로 전달하지 않는 것. SSL 인증서에도 nexclass가 미포함일 가능성이 높아.
