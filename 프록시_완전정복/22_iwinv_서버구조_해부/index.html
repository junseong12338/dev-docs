22. iwinv 서버구조 해부 - Omega¶

1. 상황 설정¶

여기서부터 실전이야. Alpha~Delta에서 배운 프록시, DNS, 방화벽, 터널링, SSL 개념이 진짜 우리 서버에서 어떻게 얽혀 돌아가는지 해부한다.

"nexclass.knu10.cc에 접속이 안 돼요"라는 신고가 들어왔을 때, 왜 안 되는지 30초 안에 파악하려면 이 구조를 머릿속에 완벽하게 그릴 수 있어야 해.

이 챕터에서 다루는 건 이론이 아니야

실제 운영 중인 NexClass 서버의 네트워크 아키텍처를 뜯어본다. 배운 개념이 현실에서 어떻게 조합되는지 보는 거야.

2. 전체 구조 분석¶

3단 구조: iwinv 프록시 -> nginx -> Spring Boot¶

우리 서버는 요청이 코드에 도달하기까지 3개의 계층을 통과한다. 이걸 "3단 프록시 체인"이라고 생각하면 돼.

graph TD
    subgraph 인터넷
        USER[사용자 브라우저]
    end

    subgraph iwinv 호스팅 네트워크
        PROXY["iwinv 앞단 프록시<br/>(80/443 수신, SSL 처리)"]
        FW["방화벽<br/>(포트 제어)"]
    end

    subgraph 서버 내부 - xxx.xxx.xx.xxx
        NGINX["nginx<br/>(server_name 기반 라우팅)"]
        SPRING["Spring Boot<br/>(localhost:8082)"]
    end

    USER -->|"https://nexclass.knu10.cc"| PROXY
    PROXY -->|"hub/lms만 라우팅"| FW
    FW -->|"허용된 포트만 통과"| NGINX
    NGINX -->|"proxy_pass localhost:8082"| SPRING

    style PROXY fill:#ff6b6b,color:#fff
    style NGINX fill:#4ecdc4,color:#fff
    style SPRING fill:#45b7d1,color:#fff

각 계층이 뭘 하는지 하나씩 뜯어보자.

1층: iwinv 앞단 프록시 (최외곽)¶

iwinv 프록시의 정체

iwinv는 호스팅 업체야. 우리 서버가 물리적으로 iwinv 데이터센터에 있고, iwinv는 자기 네트워크 앞단에 리버스 프록시를 운영한다. 이게 6장(리버스 프록시)에서 배운 그거야.

이 프록시가 하는 일:

역할 설명 상태
포트 80/443 수신 인터넷에서 오는 HTTP/HTTPS 트래픽을 받음 동작 중
SSL 처리 GoGetSSL 인증서로 HTTPS 암호화/복호화 hub, lms, www만 커버
도메인별 라우팅 요청의 Host 헤더 보고 어떤 서버로 보낼지 결정 nexclass 미설정
포트 제어 80/443 외 포트는 외부 접근 차단 8082 차단됨

현재 라우팅 상태:

graph LR
    subgraph iwinv 앞단 프록시
        R{도메인 라우팅}
    end

    R -->|"hub.knu10.cc"| HUB["HUB 서버 ✅"]
    R -->|"lms.knu10.cc"| LMS["LMS 서버 ✅"]
    R -->|"www.knu10.cc"| WWW["웹 서버 ✅"]
    R -->|"nexclass.knu10.cc"| NONE["??? ❌ 미설정"]

    style NONE fill:#ff0000,color:#fff

핵심 문제

iwinv 프록시가 nexclass.knu10.cc 요청을 받아도 어디로 보내야 하는지 모른다. 라우팅 규칙이 없으니까. hub, lms는 누군가가 iwinv한테 "이 도메인은 이 서버로 보내줘"라고 설정을 요청했기 때문에 되는 거야.

2층: nginx (서버 내부 프록시)¶

nginx의 역할

서버 안에 설치된 리버스 프록시. nexclass.knu10.cc 요청을 받으면 localhost:8082(Spring Boot)로 전달하는 게 역할이야.

nginx 설정은 이렇게 돼 있어:

# /etc/nginx/sites-available/nexclass
server {
    listen 80;                          # 80번 포트에서 대기
    server_name nexclass.knu10.cc;      # 이 도메인 요청만 처리

    location / {
        proxy_pass http://localhost:8082;  # Spring Boot로 전달
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}

이 설정이 의미하는 건:

설정 의미
listen 80 80번 포트에서 대기
server_name nexclass.knu10.cc 이 도메인으로 온 요청만 처리
proxy_pass http://localhost:8082 Spring Boot로 전달
X-Real-IP, X-Forwarded-For 원래 클라이언트 IP 정보 보존

하지만 무용지물

nginx 설정은 완벽해. 문제는 트래픽이 nginx까지 안 온다는 거야. iwinv 프록시가 nexclass.knu10.cc 트래픽을 서버로 안 보내주니까, nginx가 아무리 잘 설정돼 있어도 처리할 요청이 없어.

이건 마치 레스토랑 주방을 완벽하게 세팅해놓고, 건물 입구를 안 열어놓은 거야. 손님이 건물에 못 들어오니까 주방이 놀고 있는 거지.

3층: Spring Boot (애플리케이션)¶

Spring Boot의 역할

실제 비즈니스 로직을 처리하는 애플리케이션. localhost:8082에서 대기하면서 요청을 처리한다.

3. 왜 hub/lms는 되고 nexclass만 안 되나¶

병목 지점¶

문제의 본질

DNS는 됐다. 서버 안의 nginx도 됐다. Spring Boot도 돌고 있다. 단 하나, iwinv 프록시의 라우팅 규칙만 없다. 이 한 곳이 막혀서 전체가 안 되는 거야.

방법 설명 필요한 작업

Cloudflare Tunnel ngrok 대안. 고정 URL + 무료 SSL cloudflared 설정

핵심 깨달기

"서버에 nginx 설정했는데 왜 안 되나요?"라는 질문의 답이 여기 있어. nginx는 서버 안의 2층이야. 1층(iwinv 프록시)이 문을 안 열어주면 2층에 아무리 완벽한 설정이 있어도 쓸모없어.

이건 "내 가게 내부를 리모델링했는데 왜 손님이 안 와요?"라고 묻는 것과 같아. 건물 입구가 잠겨있기 때문이야.

4. 현재 우회 경로: ngrok¶

iwinv 프록시를 못 뚫으니까, 완전히 다른 경로로 외부 접근을 만들었어. 그게 ngrok이야.

    subgraph "정상 경로 (미동작)"
        A1[브라우저] -->|"HTTPS"| B1[iwinv 프록시]
        B1 -->|"❌"| C1[nginx]
        C1 --> D1[Spring Boot]

    subgraph "ngrok 우회 경로 (현재 사용)"
        A2[브라우저] -->|"HTTPS"| B2[ngrok 클라우드]
        B2 -->|"터널"| C2["ngrok 에이전트<br/>(서버 내부)"]
        C2 -->|"localhost:8082"| D2[Spring Boot]
    end

항목 정상 경로 ngrok 우회 경로
SSL iwinv GoGetSSL ngrok 자체 인증서
도메인 nexclass.knu10.cc abc-xyz.ngrok-free.dev
프록시 경유 iwinv 프록시 ngrok 클라우드
nginx 경유 경유함 경유 안 함
안정성 항상 동작 (설정되면) 프로세스 죽으면 끝

ngrok의 한계

  • URL이 세션마다 바뀜 (무료 플랜)
  • 프로세스가 죽으면 접속 불가
  • Zoom Webhook URL을 매번 재설정해야 함
  • 프로덕션용이 아님. 임시방편.

5. 포트 8082 차단 문제¶

"그냥 포트 8082를 외부에 열면 안 돼?"라고 생각할 수 있어. 안 돼. 왜 안 되는지 보자.

포트가 차단된 이유¶

graph LR
    A[외부에서<br/>xxx.xxx.xx.xxx:8082] -->|"TCP 패킷"| B["iwinv 네트워크<br/>방화벽"]
    B -->|"포트 8082?<br/>허용 목록에 없음"| C["❌ DROP<br/>(타임아웃)"]

    style C fill:#ff0000,color:#fff

iwinv는 80과 443 포트만 외부에 열어둔다. 보안 정책이야. 이건 호스팅사가 결정하는 거라서, 서버 안에서 ufw allow 8082 같은 걸 해봐야 소용없어. iwinv 네트워크 레벨에서 이미 막혀있거든.

검증 방법¶

# 서버 안에서는 된다
curl http://localhost:8082/health
# 결과: {"status":"UP"}

# 외부에서 직접 포트 접근은 안 된다
curl -v http://xxx.xxx.xx.xxx:8082/health
# 결과: Connection timed out (방화벽이 패킷을 DROP)

# ngrok URL로는 된다
curl https://abc-xyz.ngrok-free.dev/health
# 결과: {"status":"UP"}

5. 정리¶

3단 구조 요약¶

계층 무엇 역할 현재 상태
1층 iwinv 앞단 프록시 SSL + 도메인 라우팅 nexclass 미설정
2층 nginx server_name 기반 프록시 설정 완료, 트래픽 없음
3층 Spring Boot API + 대시보드 처리 정상 동작 중
우회 ngrok iwinv 프록시 우회 터널 임시 운영 중

문제 원인 한 줄 정리¶

한 줄 정리

iwinv 프록시에 nexclass.knu10.cc 라우팅 규칙이 없다. DNS도 됐고, nginx도 됐고, Spring Boot도 돌고 있는데, 1층(iwinv 프록시)이 문을 안 열어서 전체가 안 되는 거야.

이 챕터에서 반드시 기억할 것

  1. 서버 아키텍처는 iwinv 프록시 -> nginx -> Spring Boot 3단 구조다.
  2. iwinv 프록시가 도메인별 라우팅을 제어한다. 여기에 등록 안 되면 끝이야.
  3. "nginx 설정했는데 왜 안 돼?"의 답: iwinv 프록시가 트래픽을 안 보내줘서.
  4. 포트 8082는 iwinv 네트워크 레벨에서 차단됐다. 서버 방화벽이 아니야.
  5. ngrok은 iwinv 프록시를 완전히 우회하는 별도 경로다.

확인 문제¶

다음 문제를 풀어봐. 답 못 하면 위에서 다시 읽어.

Q1. 우리 서버의 3단 구조를 순서대로 말하고, 각 계층이 하는 핵심 역할 하나씩만 말해봐.

Q2. hub.knu10.cc는 접속되고 nexclass.knu10.cc는 안 되는 이유가 뭐야? "DNS 문제"라고 답하면 틀린 거야.

Q3. nginx 설정이 완벽해도 외부 접속이 안 될 수 있다. 왜?

Q4. 서버에서 ufw allow 8082를 실행해도 외부에서 8082 포트 접근이 안 된다. 왜?

Q5. ngrok이 iwinv 프록시를 "우회"한다는 게 정확히 무슨 뜻이야? 어떤 경로를 타는지 설명해봐.

정답 보기

A1. 1층: iwinv 앞단 프록시 (SSL 처리 + 도메인별 라우팅). 2층: nginx (server_name 기반으로 요청을 내부 포트로 전달). 3층: Spring Boot (비즈니스 로직 처리, localhost:8082에서 대기).

A2. iwinv 프록시의 라우팅 테이블에 hub.knu10.cc는 등록돼 있지만, nexclass.knu10.cc는 등록돼 있지 않기 때문이야. DNS는 둘 다 정상이고, 서버 내부 설정(nginx, Spring Boot)도 둘 다 돼 있어. 차이점은 오직 iwinv 프록시의 라우팅 설정 유무.

A3. nginx는 서버 내부(2층)에서 동작한다. 외부 트래픽이 서버에 도달하려면 먼저 1층(iwinv 프록시)을 통과해야 하는데, iwinv 프록시가 해당 도메인의 라우팅 규칙을 갖고 있지 않으면 트래픽이 서버까지 오지 않는다. nginx에 아무리 완벽한 설정이 있어도 처리할 요청 자체가 없는 거야.

A4. ufw는 서버 내부의 소프트웨어 방화벽이야. 하지만 포트 8082 차단은 iwinv 네트워크 레벨(서버 바깥)에서 일어난다. 패킷이 서버에 도달하기 전에 iwinv 인프라에서 이미 DROP 되기 때문에, 서버 내부의 방화벽 설정을 바꿔도 효과가 없어.

A5. ngrok은 서버 내부에서 ngrok 클라우드로 아웃바운드 터널을 생성한다. 외부 사용자는 ngrok 클라우드(abc-xyz.ngrok-free.dev)에 접속하고, ngrok 클라우드가 이미 열려 있는 터널을 통해 서버 내부의 localhost:8082로 트래픽을 전달한다. iwinv 프록시를 아예 거치지 않는 완전히 별도의 경로야. iwinv의 80/443 포트나 도메인 라우팅과 무관하게 동작한다.
