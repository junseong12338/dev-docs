02. 클라이언트 서버와 HTTP - Alpha¶

이거 왜 알아야 하냐면¶

프록시는 클라이언트와 서버 사이에 끼어드는 놈이야. 그런데 클라이언트가 뭐고 서버가 뭔지, 이 둘이 어떻게 대화하는지(HTTP) 모르면 "사이에 끼어든다"는 말 자체가 이해가 안 돼. 식당에서 손님과 주방 사이에 웨이터가 있는 건데, 손님이 뭐고 주방이 뭔지 모르면 웨이터가 뭘 하는지 설명이 안 되잖아.

1. 핵심 개념 - "이게 뭐야?"¶

클라이언트와 서버¶

식당을 생각해봐.

역할 식당 비유 인터넷에서
클라이언트 손님 (주문하는 쪽) 브라우저, 앱, curl 등 (요청하는 쪽)
서버 주방 (만들어주는 쪽) Spring Boot, nginx 등 (응답하는 쪽)
요청 "볶음밥 하나요" GET /dashboard/login
응답 볶음밥 나옴 200 OK + HTML

정확한 정의

  • 클라이언트(Client): 서비스를 요청하는 쪽. 브라우저가 대표적이지만, 다른 서버가 API를 호출할 때 그 서버도 클라이언트야.
  • 서버(Server): 요청을 받아서 응답하는 쪽. 항상 켜져있고, 요청이 오면 처리해서 결과를 돌려준다.

흔한 오해: 클라이언트 = 브라우저?

아니야. 클라이언트는 역할이지 특정 프로그램이 아니야. LMS가 NexClass API를 호출할 때, LMS 서버가 클라이언트 역할을 해. 같은 프로그램이 상황에 따라 클라이언트도 되고 서버도 돼.

우리 NexClass 프로젝트에서의 역할:

graph LR
    A[브라우저] -->|요청| B[NexClass 서버]
    B -->|응답| A
    C[LMS 서버] -->|API 호출| B
    B -->|API 응답| C
    B -->|API 호출| D[Zoom API]
    D -->|API 응답| B

여기서 NexClass는:

  • 브라우저/LMS가 요청할 때 → 서버 역할
  • Zoom API를 호출할 때 → 클라이언트 역할

HTTP란¶

클라이언트와 서버가 대화하려면 규칙이 필요해. 그 규칙이 HTTP(HyperText Transfer Protocol)야.

정확한 정의

HTTP는 웹에서 클라이언트와 서버가 데이터를 주고받는 통신 규약. "이런 형식으로 요청 보내, 이런 형식으로 응답 받아" 라는 약속이야.

HTTP의 핵심 특성:

특성 의미 비유
요청-응답 항상 클라이언트가 먼저 요청, 서버가 응답 손님이 먼저 주문, 주방이 요리
비연결성 응답 보내면 연결 끊음 (기본적으로) 음식 나오면 웨이터 다른 테이블로
무상태 이전 요청을 기억 못 함 매번 "처음 뵙겠습니다"

무상태(Stateless)가 왜 중요하냐

서버가 이전 요청을 기억 못 하니까, "나 로그인한 사람이야"를 매 요청마다 증명해야 해. 그래서 JWT 토큰이나 세션 쿠키가 필요한 거야. NexClass에서 API 호출 시 JWT를 매번 헤더에 넣는 이유가 이거야.

2. 동작 원리 - "어떻게 돌아가?"¶

HTTP 요청 (Request)¶

손님이 주문서를 쓴다고 생각해봐. 주문서에는 정해진 형식이 있어.

GET /v2/api/lessons/L001/attends HTTP/1.1
Host: nexclass.knu10.cc
Authorization: Bearer eyJhbGciOiJIUzI1...
Accept: application/json

이걸 분해하면:

부분 예시 식당 비유
메서드(Method) GET 주문 종류 (먹기/포장/취소)
경로(Path) /v2/api/lessons/L001/attends 메뉴 이름
HTTP 버전 HTTP/1.1 주문서 양식 버전
헤더(Headers) Host, Authorization 등 특이사항 메모 (알레르기, VIP 등)
바디(Body) (GET은 보통 없음) 커스텀 요구사항 상세

HTTP 메서드¶

메서드 의미 식당 비유 NexClass 예시
GET 조회 (달라고 요청) "메뉴판 보여주세요" 강의 목록 조회
POST 생성 (새로 만들어달라고) "볶음밥 하나 주문이요" 강의 생성
PUT 전체 수정 "볶음밥을 짬뽕으로 바꿔주세요" 강의 정보 전체 수정
PATCH 부분 수정 "볶음밥에 계란 추가요" 강의 일부 정보 수정
DELETE 삭제 "주문 취소요" 강의 삭제

GET vs POST 핵심 차이

  • GET: 데이터를 URL에 붙여서 보냄 (?key=value). 바디 없음. 브라우저 주소창에 보여.
  • POST: 데이터를 바디에 넣어서 보냄. URL에 안 보임. 비밀번호 같은 민감 정보는 반드시 POST.

HTTP 응답 (Response)¶

서버가 요청을 처리하고 돌려주는 결과야. 식당에서 음식이 나오는 거지.

HTTP/1.1 200 OK
Content-Type: application/json
Date: Wed, 19 Mar 2026 09:00:00 GMT

{
  "status": 200,
  "message": "success",
  "datas": [
    {
      "userCd": "2024001",
      "lessonTime": 3600,
      "lessonStartDate": "2026-03-19 09:00:00"
    }
  ]
}
부분 예시 의미
상태 라인 HTTP/1.1 200 OK 처리 결과 요약
응답 헤더 Content-Type: application/json 응답 부가 정보
응답 바디 JSON 데이터 실제 데이터

HTTP 상태 코드¶

이거 외워. 디버깅할 때 상태 코드만 보면 뭐가 문제인지 80%는 감 잡을 수 있어.

코드 의미 식당 비유 디버깅 포인트
200 OK, 성공 음식 나왔어 정상
201 Created, 생성됨 새 메뉴 등록됨 POST 성공
301 Moved Permanently 가게 이사함 URL 변경됨, 리다이렉트
302 Found (임시 이동) 오늘만 옆 가게에서 임시 리다이렉트
400 Bad Request 주문서를 이상하게 씀 요청 형식 오류
401 Unauthorized 회원증 안 가져옴 인증 안 됨 (JWT 없거나 만료)
403 Forbidden 회원인데 VIP 전용 권한 없음
404 Not Found 그런 메뉴 없음 URL이 틀렸거나 리소스 없음
500 Internal Server Error 주방 폭발 서버 코드 에러 (로그 확인)
502 Bad Gateway 웨이터가 주방 문 못 열어 프록시가 백엔드 연결 실패
503 Service Unavailable 가게 임시 휴업 서버 과부하 또는 점검 중
504 Gateway Timeout 웨이터가 주방 기다리다 포기 프록시가 백엔드 응답 대기 초과

502와 504는 프록시 관련 에러야

502 Bad Gateway, 504 Gateway Timeout이 뜨면 서버 코드 문제가 아니라 프록시/네트워크 문제일 확률이 높아. 이건 나중에 프록시 챕터에서 다시 나온다.

3. 코드/설정으로 보자¶

실전: 브라우저에서 nexclass.knu10.cc 접속 시¶

브라우저에서 https://nexclass.knu10.cc/dashboard/login을 치면 실제로 이런 HTTP 요청이 나가:

sequenceDiagram
    participant B as 브라우저
    participant DNS as DNS 서버
    participant P as 프록시/네트워크
    participant S as NexClass (8082)

    B->>DNS: nexclass.knu10.cc의 IP가 뭐야?
    DNS-->>B: xxx.xxx.xx.xxx
    B->>P: GET /dashboard/login (xxx.xxx.xx.xxx:443)
    Note over P: 여기서 막힘! (프록시 미설정)
    P--xB: 연결 실패 또는 타임아웃

왜 접속이 안 되는지 HTTP 관점에서 보면

  1. 브라우저가 HTTPS 요청을 보냄 → 포트 443
  2. iwinv 앞단 프록시가 nexclass.knu10.cc에 대한 라우팅 규칙이 없음
  3. 요청이 Spring Boot(8082)까지 도달하지 못함
  4. 브라우저에 연결 실패 또는 타임아웃 표시

반면 hub.knu10.cc는 iwinv 프록시에 라우팅 설정이 되어있어서, 포트 443 → 내부 포트로 정상 전달.

curl로 HTTP 요청 직접 보내기¶

브라우저 없이도 HTTP 요청을 보낼 수 있어. curl이라는 도구로:

# NexClass 헬스체크 API 호출
curl -v http://localhost:8082/health

# -v 옵션: 요청/응답 헤더 전부 보여줌 (verbose)
# 결과:
# > GET /health HTTP/1.1        ← 이게 요청
# > Host: localhost:8082
# >
# < HTTP/1.1 200 OK             ← 이게 응답
# < Content-Type: application/json
# <
# {"status":"UP"}               ← 이게 바디

# NexClass 인증 API 호출
curl -v -X POST http://localhost:8082/v2/api/auth/token \
  -H "Content-Type: application/json" \
  -d '{"clientKey":"LMS_KEY","clientSecret":"LMS_SECRET"}'

# -X POST: 메서드 지정
# -H: 헤더 추가
# -d: 바디 데이터

# 인증이 필요한 API 호출
curl -v http://localhost:8082/v2/api/lessons \
  -H "Authorization: Bearer eyJhbGciOiJIUzI1NiJ9..."

# Authorization 헤더에 JWT 토큰을 넣어서 보냄
# 토큰 없으면 401 Unauthorized 응답

curl은 디버깅의 핵심 도구

브라우저에서 안 될 때, curl로 직접 요청 보내면 어디서 막히는지 정확히 알 수 있어. 프록시 문제인지, 서버 문제인지, 인증 문제인지 구분할 수 있거든.

HTTP 헤더 핵심¶

요청/응답 헤더 중에서 자주 나오는 것들:

헤더 방향 역할 예시
Host 요청 어떤 도메인으로 요청했는지 nexclass.knu10.cc
Content-Type 양방향 데이터 형식 application/json
Authorization 요청 인증 정보 Bearer eyJ...
User-Agent 요청 클라이언트 종류 Mozilla/5.0...
X-Forwarded-For 요청 원래 클라이언트 IP (프록시 경유 시) xxx.xxx.xxx.xxx
Set-Cookie 응답 쿠키 설정 JSESSIONID=abc123

X-Forwarded-For가 왜 중요하냐

프록시를 거치면 서버 입장에서 요청자 IP가 프록시 IP로 바뀌어. 원래 클라이언트 IP를 알려면 X-Forwarded-For 헤더를 봐야 해. 이건 10장(프록시 헤더)에서 깊이 다룬다.

4. 주의사항/함정¶

함정 1: HTTPS와 HTTP는 다른 포트야

http://nexclass.knu10.cc → 포트 80 https://nexclass.knu10.cc → 포트 443

이 두 개는 완전히 다른 문이야. HTTP로 요청했는데 HTTPS만 열려있으면 접속 안 돼. 반대도 마찬가지.

함정 2: 200 OK인데 데이터가 이상한 경우

HTTP 200은 "서버가 정상적으로 응답했다"는 뜻이지, "데이터가 올바르다"는 뜻이 아니야. 상태코드 200인데 바디에 에러 메시지가 들어있을 수 있어.

HTTP/1.1 200 OK

{
  "status": 200,
  "message": "success",
  "datas": []
}

위 응답은 200 OK지만, datas가 빈 배열이야. "성공했는데 데이터가 없다"는 거지. API 설계에 따라 이게 정상일 수도, 문제일 수도 있어.

함정 3: 302 리다이렉트 루프

로그인 안 했을 때 /dashboard 접속 → 302로 /dashboard/login으로 보냄 → 로그인 페이지에서도 무슨 조건 때문에 또 302 → 무한 루프. 브라우저가 "리다이렉트가 너무 많습니다" 에러를 뱉어. Spring Security 설정할 때 자주 겪는 문제야.

함정 4: Content-Type 불일치

요청할 때 Content-Type: application/json이라고 해놓고 바디에 폼 데이터(key=value)를 보내면 서버가 파싱을 못 해. 400 Bad Request가 뜨거나, 파싱은 되는데 값이 전부 null이야. 이거 의외로 자주 실수해.

5. 정리¶

개념 핵심 우리 프로젝트에서
클라이언트 요청하는 쪽 브라우저, LMS 서버
서버 응답하는 쪽 NexClass Spring Boot
HTTP 웹 통신 규약 모든 API 호출의 기반
요청 메서드 + URL + 헤더 + 바디 GET /v2/api/lessons
응답 상태코드 + 헤더 + 바디 200 OK + JSON
상태 코드 처리 결과 숫자 200(성공), 401(인증실패), 502(프록시에러)

한 줄 정리

HTTP는 클라이언트가 서버에게 "이거 줘"라고 요청하고, 서버가 "여기 있어"라고 응답하는 규칙이다.

이 챕터에서 반드시 기억할 것

  1. 클라이언트와 서버는 역할이지 고정된 게 아니다. NexClass도 때로는 클라이언트야.
  2. HTTP는 무상태(Stateless)라서 매 요청마다 인증 정보를 보내야 한다.
  3. 상태 코드 200/400/401/403/404/500/502/504는 외워.
  4. curl -v로 HTTP 요청/응답을 직접 볼 수 있다.
  5. 502/504는 프록시 문제 신호.

확인 문제¶

다음 문제를 풀어봐. 답 못 하면 위에서 다시 읽어.

Q1. NexClass 서버가 Zoom API를 호출할 때, NexClass는 클라이언트야 서버야? 이유를 말해봐.

Q2. HTTP 상태 코드 502 Bad Gateway가 뜨면, 문제가 있는 곳은 어디일 가능성이 가장 높아? (a) 브라우저, (b) 프록시와 백엔드 사이, (c) 서버 코드의 NullPointerException, (d) DNS 서버

Q3. 다음 curl 명령어가 하는 일을 한 문장으로 설명해봐. 

curl -X POST http://localhost:8082/v2/api/auth/token \
  -H "Content-Type: application/json" \
  -d '{"clientKey":"KEY","clientSecret":"SECRET"}'

Q4. HTTP가 "무상태(Stateless)"라는 게 실제 개발에서 어떤 문제를 만들고, NexClass에서는 이걸 어떻게 해결하고 있어?

정답 보기

A1. NexClass는 클라이언트 역할이야. Zoom API에 요청을 보내는 쪽이니까. 클라이언트/서버는 고정된 게 아니라 역할이야. NexClass는 브라우저/LMS의 요청을 받을 때는 서버, Zoom API를 호출할 때는 클라이언트.

A2. (b) 프록시와 백엔드 사이. 502 Bad Gateway는 프록시(게이트웨이)가 백엔드 서버에 연결을 시도했지만 실패했거나, 잘못된 응답을 받았다는 뜻이야. 서버가 죽어있거나, 포트가 틀렸거나, 방화벽이 막고 있을 때 발생.

A3. localhost의 8082 포트에 있는 NexClass 인증 API(/v2/api/auth/token)에 JSON 형식의 인증 정보(clientKey, clientSecret)를 담아 POST 요청을 보낸다.

A4. 무상태는 서버가 이전 요청을 기억 못 한다는 뜻이야. 그래서 로그인 후에도 "나 인증된 사용자야"를 매 요청마다 증명해야 해. NexClass에서는 API 호출 시 JWT 토큰을 Authorization 헤더에 넣어서 해결하고, 대시보드에서는 세션 쿠키로 해결한다.
