11. nginx 기초 - Gamma¶

1. 핵심 개념 - "nginx가 뭔데?"¶

nginx(엔진엑스)는 웹 서버이자 리버스 프록시야.

"웹 서버"라는 말이 좀 헷갈리지? 단순하게 생각해. 브라우저가 "이 페이지 줘"라고 요청하면, 그 요청을 받아서 파일이나 응답을 돌려주는 프로그램이 웹 서버야. 그리고 nginx는 그 일을 하는 프로그램 중 하나고.

nginx의 본질

nginx는 이벤트 기반 비동기 아키텍처로 동작하는 웹 서버다. 하나의 프로세스가 수천 개의 동시 연결을 처리할 수 있어. Apache가 요청 하나당 쓰레드 하나를 만드는 것과 근본적으로 다르다.

Apache vs nginx - 뭐가 다른데?¶

식당으로 비유하면 이래:

  • Apache = 손님 한 명당 웨이터 한 명. 손님이 100명이면 웨이터 100명 필요해. 손님이 메뉴 고르는 동안 웨이터는 옆에서 멍때리고 있어. (프로세스/쓰레드 모델)
  • nginx = 웨이터 한 명이 홀 전체 관리. 손님 A한테 메뉴판 주고 → 손님 B 주문 받고 → 손님 C 음식 가져다주고. 쉬는 시간 없이 다음 할 일로 넘어가. (이벤트 루프 모델)
항목 Apache nginx
아키텍처 프로세스/쓰레드 기반 이벤트 기반 비동기
동시 연결 연결당 쓰레드 → 메모리 많이 먹음 하나의 워커가 수천 연결 처리
정적 파일 느림 빠름 (2~3배)
동적 컨텐츠 mod_php 등 내장 가능 자체 처리 안 함 → 백엔드로 프록시
설정 .htaccess 지원 (디렉토리별 설정) 지원 안 함 (중앙 설정만)
리버스 프록시 가능하지만 태생이 아님 태생이 리버스 프록시
점유율 (2026) 줄어드는 중 1위

비유는 비유일 뿐

식당 비유는 이해 돕기용이야. 진짜 핵심은 이벤트 루프라는 개념이다. Node.js가 싱글 쓰레드로 동시성을 처리하는 것과 같은 원리야. "한 놈이 모든 걸 처리한다"가 아니라 "I/O 대기 시간에 다른 일을 한다"가 정확한 설명이다.

2. 설치 - "어떻게 깔아?"¶

Ubuntu/Debian 기준:

# 패키지 목록 업데이트
sudo apt update

# nginx 설치
sudo apt install nginx

# 설치 확인
nginx -v
# nginx version: nginx/1.x.x

# 실행 상태 확인
sudo systemctl status nginx

설치하면 기본적으로 자동 시작돼. 브라우저에서 서버 IP 접속하면 "Welcome to nginx!" 페이지가 뜨면 성공이야.

3. 설정 파일 구조 - "어디에 뭐가 있어?"¶

nginx 설정은 트리 구조야. 이걸 모르면 설정 파일 찾다가 하루 다 간다.

graph TD
    A["/etc/nginx/nginx.conf<br>메인 설정 파일"] --> B["include /etc/nginx/conf.d/*.conf"]
    A --> C["include /etc/nginx/sites-enabled/*"]
    D["/etc/nginx/sites-available/<br>사용 가능한 사이트 설정"] -->|"심볼릭 링크"| C
    C --> E["sites-enabled/default<br>기본 사이트"]
    C --> F["sites-enabled/nexclass<br>우리가 추가한 설정"]

각 파일/디렉토리 역할¶

경로 역할
/etc/nginx/nginx.conf 메인 설정. 전역 설정 + include로 하위 설정 불러옴
/etc/nginx/sites-available/ 사용 가능한 사이트 설정 파일 모음. 여기 있다고 활성화는 아님
/etc/nginx/sites-enabled/ 활성화된 사이트. sites-available의 심볼릭 링크
/etc/nginx/conf.d/ 추가 설정. .conf 확장자 파일 자동 로드

sites-available vs sites-enabled 왜 분리해?

사이트 설정을 활성화/비활성화할 때 파일을 지우지 않고 심볼릭 링크만 걸거나 끊으면 되니까. 설정 파일 자체는 sites-available에 보관하고, 활성화할 때만 sites-enabled에 링크를 건다.

# 사이트 활성화: sites-available → sites-enabled 심볼릭 링크
sudo ln -s /etc/nginx/sites-available/nexclass /etc/nginx/sites-enabled/

# 사이트 비활성화: 링크만 삭제 (원본 안 건드림)
sudo rm /etc/nginx/sites-enabled/nexclass

nginx.conf 기본 구조¶

# 전역 설정 (Global context)
user www-data;              # nginx를 어떤 사용자 권한으로 실행할지
worker_processes auto;      # 워커 프로세스 수 (auto = CPU 코어 수)
pid /run/nginx.pid;         # PID 파일 위치

# 이벤트 설정
events {
    worker_connections 1024;  # 워커당 최대 동시 연결 수
}

# HTTP 설정
http {
    # 공통 설정
    include /etc/nginx/mime.types;      # 파일 확장자 → Content-Type 매핑
    default_type application/octet-stream;

    # 로그
    access_log /var/log/nginx/access.log;
    error_log /var/log/nginx/error.log;

    # 하위 설정 파일 포함
    include /etc/nginx/conf.d/*.conf;
    include /etc/nginx/sites-enabled/*;
}

4. 핵심 디렉티브 - "이것만 알면 돼"¶

nginx 설정의 핵심 개념은 디렉티브(directive)야. "명령어"라고 생각하면 돼.

설정 문법의 3가지 규칙¶

# 규칙 1: 단순 디렉티브 = 이름 + 값 + 세미콜론
listen 80;
server_name nexclass.knu10.cc;

# 규칙 2: 블록 디렉티브 = 이름 + 중괄호
server {
    # 이 안이 하나의 "서버 블록"
}

# 규칙 3: 블록 안에 블록 (중첩)
server {
    location / {
        # server 블록 안의 location 블록
    }
}

세미콜론 빼먹으면 nginx 안 뜬다

99%의 nginx 설정 에러가 세미콜론 누락이야. 에러 나면 제일 먼저 세미콜론부터 확인해.

핵심 디렉티브 6개¶

server - 가상 호스트 정의¶

server {
    # 이 블록 하나가 하나의 "사이트"
    # 도메인별로 다른 server 블록을 만들 수 있어
}

listen - 어떤 포트로 받을지¶

server {
    listen 80;         # HTTP 80번 포트
    listen 443 ssl;    # HTTPS 443번 포트
    listen 8080;       # 커스텀 포트
}

server_name - 어떤 도메인의 요청인지¶

server {
    listen 80;
    server_name nexclass.knu10.cc;    # 이 도메인으로 온 요청만 처리
}

server {
    listen 80;
    server_name hub.knu10.cc;         # 이 도메인은 다른 server 블록이 처리
}

같은 포트, 다른 도메인

80번 포트로 오는 요청이라도 server_name이 다르면 다른 server 블록이 처리해. 이게 가상 호스트(Virtual Host)야. 하나의 서버에서 여러 도메인을 서비스할 수 있는 원리.

location - URL 경로별 처리¶

server {
    listen 80;
    server_name nexclass.knu10.cc;

    # / 아래 모든 경로
    location / {
        proxy_pass http://localhost:8082;
    }

    # /api/ 로 시작하는 경로만
    location /api/ {
        proxy_pass http://localhost:8082;
    }

    # 정적 파일 (이미지, CSS, JS)
    location /static/ {
        root /var/www/nexclass;
    }
}

root - 정적 파일 위치¶

location /static/ {
    root /var/www/nexclass;
    # /static/logo.png 요청 → /var/www/nexclass/static/logo.png 파일 반환
}

proxy_pass - 어디로 전달할지 (리버스 프록시의 핵심)¶

location / {
    proxy_pass http://localhost:8082;
    # 이 요청을 localhost:8082로 전달해라
    # = Spring Boot(NexClass)로 보내라
}

5. nginx 명령어 - "시작/중지/재시작"¶

명령어 설명
sudo systemctl start nginx nginx 시작
sudo systemctl stop nginx nginx 중지
sudo systemctl restart nginx nginx 재시작 (연결 끊김)
sudo systemctl reload nginx 설정만 다시 읽기 (연결 안 끊김)
sudo nginx -t 설정 파일 문법 검사
sudo systemctl status nginx 실행 상태 확인
sudo systemctl enable nginx 부팅 시 자동 시작 등록

설정 바꾸고 바로 restart 하지 마

무조건 nginx -t로 문법 검사 먼저 해. 문법 에러 있는데 restart 하면 nginx가 죽어버려. 그러면 서버 전체가 먹통이야.

# 올바른 순서
sudo nginx -t                    # 1. 문법 검사
# nginx: the configuration file /etc/nginx/nginx.conf syntax is ok
# nginx: configuration file /etc/nginx/nginx.conf test is successful

sudo systemctl reload nginx      # 2. 문법 OK면 reload (restart 아님!)

reload와 restart의 차이:

명령 동작 기존 연결
restart nginx 프로세스 종료 → 재시작 끊김
reload 새 설정으로 워커 교체 유지

운영 중인 서버에서는 무조건 reload 써. restart는 순간적으로 서비스가 끊기니까.

6. 실전 - "우리 서버에는 어떻게 돼 있어?"¶

NexClass가 돌아가는 iwinv 서버(xxx.xxx.xx.xxx)에 nginx가 이미 설치돼 있어.

# 서버 접속 후 확인
nginx -v
# nginx version: nginx/1.x.x

# 설정 확인
cat /etc/nginx/sites-available/nexclass

우리 서버의 nginx 설정은 대략 이렇게 생겼어:

server {
    listen 80;
    server_name nexclass.knu10.cc;

    location / {
        proxy_pass http://localhost:8082;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }
}

이 설정이 하는 말:

  1. 80번 포트로 들어오는 요청 중
  2. nexclass.knu10.cc 도메인으로 온 것만
  3. Spring Boot(localhost:8082)로 전달해라

그런데 이 설정이 있어도 외부에서 접속이 안 돼. 왜?

nginx 설정이 있어도 안 되는 이유

iwinv 앞단 프록시가 80/443 포트를 관리하고 있어. 외부 요청이 nginx까지 도달을 못 하는 거야. hub.knu10.cc는 iwinv 프록시에 라우팅 설정이 되어 있으니까 되고, nexclass.knu10.cc는 설정이 안 되어 있으니까 안 되는 거지.

graph LR
    A[브라우저] -->|nexclass.knu10.cc| B[iwinv 앞단 프록시]
    B -->|"nexclass? 모르는데?"| C["차단 (라우팅 미설정)"]
    B -->|"hub? 알지"| D[nginx :80]
    D --> E[Spring Boot :8082]

이 문제는 12장(nginx 리버스 프록시)에서 더 깊이 파고, 14장(터널링)에서 우회 방법을 배운다.

7. 주의사항 / 함정¶

함정 1: 세미콜론

nginx 설정에서 세미콜론 하나 빼먹으면 전체가 안 돌아가. nginx -t 에러 메시지를 꼭 읽어.

함정 2: sites-enabled에 직접 파일 만들기

sites-available에 파일 만들고 심볼릭 링크를 거는 게 정석이야. sites-enabled에 직접 파일 만들면 관리가 개판 돼.

함정 3: default 설정 안 지우기

/etc/nginx/sites-enabled/default가 남아있으면 의도치 않게 기본 설정이 적용될 수 있어. 커스텀 설정이 제대로 안 먹으면 default부터 확인해.

함정 4: restart vs reload

운영 서버에서 restart하면 순간 다운타임이 생겨. reload를 써. 특히 트래픽 많을 때 restart 치면 사용자들이 에러 봐.

8. 정리¶

항목 내용
nginx란? 이벤트 기반 비동기 웹 서버 + 리버스 프록시
vs Apache nginx = 이벤트 루프 (효율적), Apache = 쓰레드/프로세스 (직관적)
메인 설정 /etc/nginx/nginx.conf
사이트 설정 sites-available/ → sites-enabled/ (심볼릭 링크)
핵심 디렉티브 server, listen, server_name, location, root, proxy_pass
문법 검사 sudo nginx -t (필수!)
설정 반영 sudo systemctl reload nginx (restart 아님!)

한 줄 정리

nginx = 요청을 받아서 적절한 곳에 전달하는 교통경찰. 설정 바꾸면 nginx -t → reload.

확인 문제¶

다음 문제를 풀어봐. 답 못 하면 위에서 다시 읽어.

Q1. nginx와 Apache의 가장 근본적인 아키텍처 차이가 뭐야?

Q2. nginx 설정을 수정한 후 올바른 적용 순서 2단계를 말해봐.

Q3. sites-available과 sites-enabled를 왜 분리해놨어?

Q4. 아래 설정에서 server_name의 역할이 뭐야? 

server {
    listen 80;
    server_name nexclass.knu10.cc;
    location / {
        proxy_pass http://localhost:8082;
    }
}

Q5. 우리 서버에 nginx 설정이 있는데도 nexclass.knu10.cc 접속이 안 되는 이유가 뭐야?

정답 보기

A1. Apache는 요청마다 프로세스/쓰레드를 생성하는 모델이고, nginx는 이벤트 기반 비동기 모델이야. nginx는 하나의 워커 프로세스가 이벤트 루프를 돌면서 수천 개의 동시 연결을 처리해. I/O 대기 시간에 다른 요청을 처리하니까 메모리 효율이 훨씬 좋아.

A2. (1) sudo nginx -t로 설정 문법 검사 → (2) sudo systemctl reload nginx로 설정 반영. restart가 아니라 reload야. restart는 기존 연결이 끊기니까.

A3. 사이트 설정을 활성화/비활성화할 때 파일 자체를 지우지 않고 심볼릭 링크만 걸거나 끊으면 되니까. 설정 원본은 sites-available에 보관하고, 활성화할 때만 sites-enabled에 링크를 거는 구조야.

A4. 같은 80번 포트로 들어오는 요청이라도 어떤 도메인으로 온 요청인지 구분하는 역할이야. nexclass.knu10.cc로 온 요청만 이 server 블록이 처리하고, 다른 도메인은 다른 server 블록이 처리해. 이게 가상 호스트의 원리야.

A5. iwinv 앞단 프록시가 80/443 포트를 관리하고 있어서, 외부 요청이 nginx까지 도달을 못 해. iwinv 프록시에 nexclass.knu10.cc 라우팅 설정이 안 되어 있기 때문이야. hub.knu10.cc나 lms.knu10.cc는 iwinv 프록시에 설정이 되어 있으니까 되는 거고.
