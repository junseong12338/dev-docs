21. 프록시 설계 판단 - Delta¶

1. 핵심 개념 - "설계란 선택이야"¶

이 챕터의 핵심 원칙

"방법이 하나밖에 없어요" = 분석 안 한 거야.

모든 설계 결정에는 최소 3가지 선택지가 있어. 각각 장점이 있고, 단점이 있어. 그 중에서 왜 이걸 골랐는지, 언제 이 선택을 뒤집어야 하는지까지 대답할 수 있어야 설계했다고 할 수 있어.

이 챕터는 프록시/네트워크 설계에서 실제로 만나는 3가지 상황에 대해, 선택지를 나열하고, 트레이드오프를 분석하고, 판단 근거를 정리하는 거야.

"이렇게 하면 돼요"가 아니라 "왜 이렇게 했는지, 다른 방법은 뭐였는지, 뒤집을 상황은 뭔지"를 다룬다.

2. 상황 1: "서버를 인터넷에 공개해야 해"¶

선택지 3가지¶

방법: 서버 방화벽에서 포트(예: 8082)를 외부에 직접 개방
구조: 사용자 → xxx.xxx.xx.xxx:8082 → Spring Boot
장점 단점
가장 단순 (설정 1줄) 서버 IP 직접 노출
지연 시간 최소 (중간 경유 없음) 포트 스캔/DDoS에 무방비
디버깅 쉬움 (경로 단순) SSL 직접 관리해야 함
가상호스트(도메인 분기) 불가

언제 쓰나: 개발/테스트 환경에서 빠르게 확인할 때. 프로덕션에서는 절대 안 쓴다.

방법: nginx가 80/443으로 받아서 내부 포트로 전달
구조: 사용자 → nginx(:443) → Spring Boot(:8082)
장점 단점
SSL 종료 (nginx에서 인증서 관리) nginx 설정/관리 필요
가상호스트 (도메인별 서비스 분기) 하나의 장애 지점(SPOF) 추가
보안 헤더 일괄 추가 인바운드 포트(80, 443) 열어야 함
로드밸런싱 가능 SSL 인증서 갱신 관리 필요
서버 포트 직접 노출 안 함

언제 쓰나: 인바운드 포트를 열 수 있고, 여러 서비스를 도메인으로 분기해야 할 때. 가장 표준적인 프로덕션 패턴.

방법: 서버 내부에서 터널 클라이언트가 외부로 아웃바운드 연결
구조: 사용자 → 터널 서비스 → (터널) → Spring Boot(:8082)
장점 단점
인바운드 포트 안 열어도 됨 터널 서비스에 의존 (장애 시 접속 불가)
SSL 자동 제공 ngrok: URL 랜덤 변경 (재시작마다)
방화벽/NAT 우회 지연 시간 추가 (터널 경유)
설정 간단 (ngrok http 8082) 대역폭 제한 (무료 티어)

언제 쓰나: 인바운드 포트를 열 수 없는 환경. iwinv처럼 호스팅 업체가 포트를 차단하거나, 개발 환경에서 빠르게 외부 공개할 때.

트레이드오프 비교 테이블¶

기준 A. 포트 직접 B. 리버스 프록시 C. 터널
보안 최하 (IP+포트 노출) 높음 (포트 숨김, SSL) 높음 (IP 숨김, SSL 자동)
복잡도 최저 중간 (nginx 설정) 낮음 (명령어 1줄)
성능 최고 (직접 연결) 높음 (프록시 1단) 중간 (터널 경유 지연)
유연성 최저 (단일 서비스) 최고 (가상호스트, LB) 낮음 (단일 터널)
안정성 서버만 의존 nginx + 서버 터널 서비스 + 서버
프로덕션 적합도 부적합 최적 임시/개발용

판단 근거¶

NexClass 현재 상황에서의 선택

상황: iwinv가 포트 8082 외부 차단. 인바운드 포트를 열 수 없음.

  • A(포트 직접 개방) → 불가. iwinv가 차단함.
  • B(리버스 프록시) → nginx가 설치돼 있고 설정도 됐는데, iwinv 앞단 프록시가 nexclass.knu10.cc를 라우팅하지 않아서 단독으로는 불가.
  • C(터널) → ngrok으로 현재 사용 중. 인바운드 불필요. 개발 단계에 적합.

프로덕션 최적 답: iwinv에 앞단 프록시 설정 요청 + Cloudflare Tunnel 조합 (B+C 하이브리드).

선택을 뒤집을 상황¶

현재 선택 뒤집을 상황
C(터널) iwinv가 앞단 프록시 설정 완료하면 → B(리버스 프록시)로 전환
ngrok URL 변경이 더 이상 수용 불가면 → Cloudflare Tunnel로 전환
B(nginx) 단독 서비스가 글로벌해지면 → CDN(Cloudflare Proxied) 추가

3. 상황 2: "HTTPS가 필요해"¶

왜 이 상황이 생기나¶

  • Zoom Webhook은 HTTPS 엔드포인트만 허용
  • 브라우저가 HTTP를 경고 표시하고 기능 제한 (쿠키, 위치정보 등)
  • 보안 표준 (전송 중 암호화)

선택지 3가지¶

방법: Spring Boot에 PKCS12/JKS 인증서 넣고 HTTPS 활성화
구조: 사용자 → Spring Boot(:8443, HTTPS)
장점 단점
경유 서버 없음 (직접 종료) 인증서 갱신을 서버에서 직접 해야 함
구조 단순 여러 서비스마다 각각 인증서 관리
인증서 설정이 프레임워크마다 다름
포트 443을 서버가 직접 점유

언제 쓰나: 서비스가 1개고, 앞에 프록시 없이 직접 노출할 때. 마이크로서비스 환경에서는 거의 안 써.

방법: nginx에서 SSL 종료, 내부는 HTTP로 통신
구조: 사용자 →(HTTPS)→ nginx(:443) →(HTTP)→ Spring Boot(:8082)
장점 단점
인증서를 nginx 한 곳에서 관리 nginx-서버 구간은 HTTP (내부 네트워크)
certbot 자동 갱신 간편 nginx 설정 필요
서버는 HTTP만 처리 → 단순 nginx 장애 = 전체 먹통
여러 서비스 인증서 한 곳에서 관리

언제 쓰나: 가장 일반적인 프로덕션 패턴. 서비스가 여러 개고, 인증서 관리를 중앙화하고 싶을 때.

방법: ngrok/cloudflared가 알아서 SSL 제공
구조: 사용자 →(HTTPS)→ ngrok/CF →(터널)→ Spring Boot(:8082)
장점 단점
인증서 관리 제로 터널 서비스에 완전 의존
설정 제로 (자동) 인증서 커스터마이징 불가
인바운드 포트 불필요 지연 시간 추가
Let's Encrypt 갱신 걱정 없음 서비스 장애 시 HTTPS도 먹통

언제 쓰나: 인바운드 포트를 열 수 없거나, SSL 설정에 시간을 쓰고 싶지 않을 때. 개발/임시 환경에 최적.

트레이드오프 비교 테이블¶

기준 A. 서버 직접 B. 프록시 SSL 종료 C. 터널 자동 SSL
인증서 관리 직접 (서버마다) 중앙 (nginx) 자동 (서비스)
갱신 수동 or certbot certbot 자동 완전 자동
성능 SSL 연산 서버 부담 SSL 연산 nginx 부담 SSL 연산 터널 서비스 부담
유연성 낮음 높음 (SNI, 다중 인증서) 낮음
내부 통신 HTTPS (암호화됨) HTTP (내부망 평문) HTTP (터널 내부)
프로덕션 적합도 가능하지만 비추 최적 임시용

B 선택 시 주의: 내부 구간 HTTP

nginx에서 SSL을 종료하면, nginx → Spring Boot 구간은 HTTP(평문)야. 같은 서버 안에서의 localhost 통신이라 보통 괜찮지만, 서버가 물리적으로 분리돼 있으면 이 구간도 HTTPS로 해야 해. 이걸 End-to-End Encryption이라고 해.

4. 상황 3: "여러 서비스를 하나의 IP에서 운영해야 해"¶

NexClass가 정확히 이 상황이야¶

iwinv 서버 하나(IP: xxx.xxx.xx.xxx)에서: - hub.knu10.cc → HUB 시스템 - lms.knu10.cc → LMS 시스템 - nexclass.knu10.cc → NexClass (화상강의)

전부 같은 IP인데 도메인이 다르잖아. 어떻게 분기해?

선택지 2가지¶

hub.knu10.cc     → xxx.xxx.xx.xxx:8080
lms.knu10.cc     → xxx.xxx.xx.xxx:8081
nexclass.knu10.cc → xxx.xxx.xx.xxx:8082
장점 단점
설정 간단 (포트만 다르게) 사용자가 포트 번호 기억해야 함
프록시 불필요 URL이 hub.knu10.cc:8080으로 지저분
포트마다 방화벽 열어야 함
SSL 인증서 서비스마다 따로 관리
표준 포트(80, 443) 아니면 일부 네트워크에서 차단됨

언제 쓰나: 개발 환경에서 빠르게 테스트할 때만. 프로덕션에서는 절대 안 써.

hub.knu10.cc     → nginx(:443) → localhost:8080
lms.knu10.cc     → nginx(:443) → localhost:8081
nexclass.knu10.cc → nginx(:443) → localhost:8082
장점 단점
사용자는 도메인만 기억 (포트 필요 없음) nginx 설정 필요
표준 포트(443)만 열면 됨 nginx 장애 = 전체 먹통
SSL 인증서 한 곳에서 관리 설정 복잡도 증가
URL 깔끔 (hub.knu10.cc)
내부 포트 외부 노출 안 됨

언제 쓰나: 프로덕션 표준. 같은 IP에서 여러 서비스를 도메인으로 분기해야 할 때.

가상호스트 동작 원리¶

sequenceDiagram
    participant U as 사용자
    participant N as nginx (:443)
    participant H as HUB (:8080)
    participant L as LMS (:8081)
    participant X as NexClass (:8082)

    U->>N: Host: hub.knu10.cc
    N->>H: proxy_pass localhost:8080

    U->>N: Host: lms.knu10.cc
    N->>L: proxy_pass localhost:8081

    U->>N: Host: nexclass.knu10.cc
    N->>X: proxy_pass localhost:8082

핵심: HTTP Host 헤더

사용자가 hub.knu10.cc로 접속하면 HTTP 요청에 Host: hub.knu10.cc 헤더가 포함돼. nginx는 이 헤더를 보고 어떤 server 블록으로 라우팅할지 결정해. 같은 IP, 같은 포트(443)인데 도메인으로 분기할 수 있는 이유가 이거야.

nginx 가상호스트 설정 예시¶

# HUB 서비스
server {
    listen 443 ssl;
    server_name hub.knu10.cc;
    ssl_certificate /etc/ssl/hub.crt;
    ssl_certificate_key /etc/ssl/hub.key;
    location / {
        proxy_pass http://localhost:8080;
    }
}

# LMS 서비스
server {
    listen 443 ssl;
    server_name lms.knu10.cc;
    ssl_certificate /etc/ssl/lms.crt;
    ssl_certificate_key /etc/ssl/lms.key;
    location / {
        proxy_pass http://localhost:8081;
    }
}

# NexClass 서비스
server {
    listen 443 ssl;
    server_name nexclass.knu10.cc;
    ssl_certificate /etc/ssl/nexclass.crt;
    ssl_certificate_key /etc/ssl/nexclass.key;
    location / {
        proxy_pass http://localhost:8082;
    }
}

5. 종합 판단 프레임워크¶

설계 판단 체크리스트

어떤 프록시/네트워크 구조를 선택하든, 이 질문에 전부 답할 수 있어야 해:

  1. 선택지를 최소 3개 나열했나? → 못 하면 분석 부족
  2. 각 선택지의 장점을 최소 2개 말할 수 있나? → 못 하면 이해 부족
  3. 각 선택지의 단점을 최소 2개 말할 수 있나? → 못 찾으면 분석 부족
  4. 왜 이걸 골랐는지 정량적 근거가 있나? → "느낌적으로"는 근거 아니야
  5. 이 선택을 뒤집을 상황은? → 대답 못 하면 설계 이해 못 한 거야

NexClass 최적 아키텍처 선택¶

상황 선택 근거
서버 공개 B+C 하이브리드 (nginx + Cloudflare Tunnel) iwinv 인바운드 차단 → 터널 필수, 다중 서비스 → nginx 필요
HTTPS C (터널 자동 SSL) + B (nginx SSL 종료) 현재: ngrok 자동 SSL. 프로덕션: nginx + certbot
다중 서비스 B (가상호스트) hub/lms/nexclass 도메인 분기 필수

최종 프로덕션 아키텍처¶

graph TD
    U[사용자/Zoom Webhook] -->|HTTPS| IW[iwinv 앞단 프록시 :443]
    IW -->|nexclass.knu10.cc| NX[nginx]
    NX -->|proxy_pass| SB[Spring Boot :8082]

    IW -->|hub.knu10.cc| HUB[HUB 서비스]
    IW -->|lms.knu10.cc| LMS[LMS 서비스]

    style IW fill:#f90,color:#fff
    style NX fill:#4a9,color:#fff
    style SB fill:#69b,color:#fff

이 구조의 의미

iwinv 앞단 프록시가 첫 번째 리버스 프록시로서 SSL 종료 + 도메인 라우팅을 하고, nginx가 두 번째 리버스 프록시로서 로컬 포트로 전달해. 프록시가 2단으로 쌓이는 구조야. hub.knu10.cc, lms.knu10.cc는 이미 이렇게 동작하고 있어. nexclass.knu10.cc만 iwinv에 라우팅 요청이 안 된 상태인 거야.

대안: Cloudflare Tunnel 경유 (iwinv 설정 불가 시)¶

graph TD
    U[사용자/Zoom] -->|HTTPS| CF[Cloudflare 엣지]
    CF -->|cloudflared 터널| NX[nginx :80]
    NX -->|proxy_pass| SB[Spring Boot :8082]

    style CF fill:#f90,color:#fff
    style NX fill:#4a9,color:#fff
    style SB fill:#69b,color:#fff
선택 조건 이 선택을 뒤집을 상황
Cloudflare Tunnel iwinv 앞단 프록시 설정이 안 될 때 iwinv가 설정 완료하면 → 직접 라우팅으로 전환 (터널 불필요)
ngrok 개발/테스트 단계 프로덕션 전환 시 → Cloudflare Tunnel 또는 iwinv 직접 라우팅

6. 주의사항 / 함정¶

함정 1: 하나만 고르면 된다는 생각

실전에서는 조합이야. CDN + 터널 + 리버스 프록시를 동시에 쓰는 게 흔해. "프록시 하나면 되지 않나?"가 아니라 "각각 뭘 해결하는지"를 보고 조합해.

함정 2: 개발 환경 구조를 프로덕션에 그대로 쓰기

ngrok은 개발용이야. URL 랜덤 변경, 대역폭 제한, 안정성 보장 없음. "개발에서 잘 되니까 프로덕션에서도 ngrok 쓰자"는 사고야.

함정 3: 프록시 2단 이상 쌓으면 디버깅 지옥

Cloudflare → iwinv 프록시 → nginx → Spring Boot. 4단 경유. "요청이 어디서 막힌 거야?"를 찾으려면 각 구간별로 로그를 봐야 해. 구조가 복잡해질수록 X-Forwarded-For, X-Real-IP 같은 프록시 헤더 관리가 중요해져.

함정 4: 선택 근거 없이 남들 따라하기

"다른 회사가 Cloudflare 쓰니까 우리도 쓰자"는 근거가 아니야. 우리 상황(iwinv 인바운드 차단, 서비스 규모, 예산, 요구사항)에 맞는 선택을 해야 해. 남의 답을 베끼는 건 설계가 아니야.

7. 정리¶

상황 추천 선택 핵심 판단 기준
서버 인터넷 공개 인바운드 가능 → 리버스 프록시, 불가능 → 터널 인바운드 포트 개방 가능 여부
HTTPS 필요 프록시 SSL 종료가 표준, 터널은 임시 인증서 관리 부담 vs 의존성
다중 서비스 리버스 프록시 + 가상호스트 도메인 분기 필요 여부
보안 강화 CDN/프록시 (Cloudflare) 추가 공격 방어 필요 수준

이 챕터에서 반드시 기억할 것

  1. "방법이 하나밖에 없어요"는 분석 안 한 거야. 최소 3가지 선택지를 나열해.
  2. 각 선택지의 장점 2개 + 단점 2개 이상 말할 수 있어야 해.
  3. 왜 이걸 골랐는지 근거를 대야 해. "느낌"은 근거 아니야.
  4. 이 선택을 뒤집을 상황까지 생각해야 설계한 거야.
  5. 실전에서는 하나만 고르는 게 아니라 조합이야.

확인 문제¶

다음 문제를 풀어봐. 답 못 하면 위에서 다시 읽어.

Q1. NexClass 서버의 인바운드 포트가 차단된 상황에서 웹 서비스를 공개하려면 어떤 선택지가 있어? 최소 2가지 말하고, 각각의 장단점을 비교해봐.

Q2. "HTTPS가 필요하다"는 요구사항에 대해 3가지 방법(서버 직접, 프록시 SSL 종료, 터널 자동 SSL)의 트레이드오프를 설명하고, NexClass에 가장 적합한 걸 근거와 함께 골라봐.

Q3. hub.knu10.cc, lms.knu10.cc, nexclass.knu10.cc를 하나의 서버 IP에서 운영하려면, "포트 번호 분기"와 "가상호스트 분기" 중 왜 가상호스트가 맞는지 근거 3가지를 대봐.

Q4. 현재 ngrok을 쓰고 있는 NexClass가 "이 선택을 뒤집을 상황"은 뭐야? 뒤집으면 뭘로 바꿔야 해?

Q5. 사용자 → Cloudflare → iwinv 프록시 → nginx → Spring Boot 구조의 장점 2가지와 단점 2가지를 말해봐.

정답 보기

A1. (1) Cloudflare Tunnel: 서버에서 cloudflared로 아웃바운드 연결, 인바운드 불필요, SSL 자동, URL 고정. 단점: Cloudflare에 의존, 지연 추가. (2) ngrok 터널: 명령어 1줄로 즉시 공개, SSL 자동. 단점: URL 랜덤 변경, 터미널 유지 필수, 무료 티어 제한. (3) iwinv에 앞단 프록시 설정 요청: 가장 정석이지만 업체 대응 필요, 시간 소요.

A2. 서버 직접: 인증서를 서버마다 관리해야 하고 갱신도 직접 — NexClass 단독이면 가능하지만 비효율. 프록시 SSL 종료: nginx에서 중앙 관리, certbot 자동 갱신 — 프로덕션 최적이지만 현재는 인바운드 차단돼서 외부 접근 불가. 터널 자동 SSL: 설정 제로, 인바운드 불필요 — 현재 상황에 가장 적합. 프로덕션에서는 iwinv 설정 완료 후 프록시 SSL 종료로 전환이 목표.

A3. (1) 사용자 편의 — 포트 번호 기억 안 해도 됨. URL이 nexclass.knu10.cc로 깔끔. (2) 보안 — 표준 포트(443)만 열면 됨. 내부 포트(8080, 8081, 8082) 외부 노출 안 됨. (3) SSL 관리 — nginx 한 곳에서 인증서 관리. 포트 분기면 서비스마다 각각 인증서 설정 필요.

A4. 뒤집을 상황: (1) iwinv가 nexclass.knu10.cc 앞단 프록시 설정을 완료하면 ngrok 불필요 → nginx 리버스 프록시로 전환. (2) 프로덕션 배포 시 URL 안정성/대역폭 필요하면 → Cloudflare Tunnel로 전환. (3) Zoom Webhook URL 변경이 수용 불가능해지면 → 고정 URL이 가능한 방법(Cloudflare Tunnel, iwinv 설정)으로 전환.

A5. 장점: (1) 보안 극대화 — Cloudflare DDoS 방어 + IP 숨김, iwinv 내부 포트 차단, nginx 보안 헤더, 4단 방어. (2) 유연성 — 각 계층이 독립적이라 하나만 변경 가능 (예: Cloudflare 빼고 iwinv 직접 가능). 단점: (1) 디버깅 어려움 — 요청이 4단을 거치니까 "어디서 막힌 거야?"를 찾으려면 각 구간 로그를 다 봐야 함. (2) 장애 포인트 증가 — Cloudflare 장애, iwinv 장애, nginx 장애 중 하나만 터져도 전체 먹통. 단일 장애 지점(SPOF)이 3개.
