Part 2 답안¶

A6.¶

방법 1: Thread 클래스 상속 (class MyThread extends Thread)
방법 2: Runnable 인터페이스 구현 (class MyTask implements Runnable)

Runnable이 더 좋다.
이유: Java는 단일 상속만 가능하므로, Thread를 상속하면 다른 클래스를 상속 못 한다.
Runnable은 인터페이스이므로 다른 클래스 상속과 함께 사용 가능하다.
또한 작업(Runnable)과 실행(Thread)을 분리하는 것이 설계적으로 더 깔끔하다.

A7.¶

run(): 현재 스레드에서 run() 메서드를 일반 메서드처럼 호출한다. 새 스레드 생성 없음.
start(): JVM에 새 스레드 생성을 요청하고, OS가 새 스레드를 할당하고,
         새 스레드의 스택 메모리를 할당하고, 그 새 스레드에서 run()을 실행한다.
         start()를 호출한 스레드는 기다리지 않고 바로 다음 코드를 실행한다.

A8.¶

IllegalThreadStateException이 발생한다.
한 번 start()된 스레드는 TERMINATED 상태가 되면 다시 시작할 수 없다.
새로운 작업을 실행하려면 새 Thread 객체를 만들어야 한다.

A9.¶

"Timer-0"
Timer 내부에서 "Timer-" + 일련번호 형식으로 자동 이름이 붙는다.
두 번째 Timer는 "Timer-1"이 된다.

Part 3 답안¶

A10.¶

1. NEW: 생성됨 (start() 전)
2. RUNNABLE: 실행 가능 (CPU 할당 대기 또는 실행 중)
3. BLOCKED: 락(synchronized) 대기
4. WAITING: 무기한 대기 (notify/unpark 필요)
5. TIMED_WAITING: 시간 제한 대기 (일정 시간 후 자동 복귀)
6. TERMINATED: 종료됨 (재시작 불가)

A11.¶

"실행 중 (Running)"과 "실행 대기 (Ready)"
Java에서는 이 둘을 구분하지 않고 둘 다 RUNNABLE로 표현한다.
실제로 CPU를 받아서 실행 중인지, CPU를 기다리는 중인지는 OS만 안다.

A12.¶

BLOCKED: synchronized 락을 기다리는 상태.
         다른 스레드가 synchronized 블록을 빠져나오면 락을 획득하고 진행.

WAITING: 특정 조건(notify, unpark, join 완료)을 기다리는 상태.
         다른 스레드가 명시적으로 깨워줘야 진행.

A13.¶

1. Thread.sleep(millis) - 지정 시간만큼 대기
2. Object.wait(timeout) - 지정 시간 또는 notify까지 대기
3. Thread.join(millis) - 대상 스레드 종료 또는 지정 시간까지 대기
4. timer.schedule(task, delay, period) - Timer의 반복 대기
5. LockSupport.parkNanos() - 지정 시간만큼 대기

A14.¶

불가능. IllegalThreadStateException이 발생한다.
TERMINATED 상태의 스레드는 다시 살릴 수 없다.

Part 4 답안¶

A15.¶

Race Condition(경쟁 조건): 두 개 이상의 스레드가 동시에 같은 공유 자원에
접근하고, 그 결과가 스레드 실행 순서(타이밍)에 따라 달라지는 상황.

위험한 이유:
1. 항상 발생하는 게 아니라 타이밍에 따라 발생 → 재현 어려움
2. 테스트에서는 안 터지고 운영에서 터질 수 있음
3. 데이터 손상, 계산 오류 등 치명적 버그 발생
4. 찾기 극도로 어려움 (디버거 붙이면 타이밍 바뀌어서 안 터짐)

A16.¶

synchronized 메서드: 메서드 전체를 잠금. 범위가 넓음.
synchronized 블록: 필요한 부분만 잠금. 범위가 좁음.

블록이 더 효율적이다.
메서드 전체를 잠그면 동기화 불필요한 코드까지 잠기므로
다른 스레드가 불필요하게 대기하게 된다.
블록을 사용하면 꼭 필요한 부분만 잠가서 대기 시간을 줄인다.

A17.¶

읽기가 쓰기보다 훨씬 많을 때.

synchronized는 읽기든 쓰기든 무조건 한 스레드만 접근 가능하다.
ReadWriteLock은 읽기끼리는 동시 접근을 허용한다.

예: 읽기 90%, 쓰기 10%인 경우
  synchronized: 읽기끼리도 대기 → 병목
  ReadWriteLock: 읽기끼리는 동시 → 9배 빠를 수 있음

A18.¶

데드락(Deadlock): 두 개 이상의 스레드가 서로 상대방이 가진 자원(락)을 기다리며
영원히 진행하지 못하는 상태.

발생 조건 (4가지 동시 충족 시):
1. 상호 배제: 자원을 한 번에 하나만 사용 가능
2. 점유 대기: 자원을 가진 채로 다른 자원 대기
3. 비선점: 다른 스레드의 자원을 강제로 뺏을 수 없음
4. 순환 대기: A→B→A 형태의 순환 대기 발생

A19.¶

1. 락 획득 순서 통일: 모든 스레드가 lock1 → lock2 순서로 획득.
   역순 획득을 방지하면 순환 대기 조건이 깨짐.

2. 타임아웃 사용: tryLock(timeout)으로 일정 시간 안에 못 얻으면 포기.
   영원히 기다리지 않으므로 데드락 회피.

3. 중첩 락 최소화: synchronized 안에 synchronized 넣지 않기.
   중첩이 없으면 순환 가능성 자체가 낮아짐.

A20.¶

1개. TimerThread라는 스레드.
이 스레드는 while(true) 무한 루프를 돌면서:
1. TaskQueue에서 가장 빠른 작업을 꺼냄
2. 실행 시간이 됐으면 task.run() 실행
3. 안 됐으면 그 시간까지 wait()
4. 반복 작업이면 다시 큐에 넣음
이걸 cancel()이 호출되거나 JVM이 종료될 때까지 반복한다.

A21.¶

task: 실행할 작업 (TimerTask 객체)
0: 최초 실행까지 대기 시간 (밀리초). 0 = 즉시 시작
1000: 반복 간격 (밀리초). 1000 = 1초마다 반복

A22.¶

while (true) {
    큐에서 가장 빠른 작업 꺼냄
    실행 시간까지 남은 시간 계산
    남은 시간 > 0이면 queue.wait(남은시간)
    남은 시간 <= 0이면 task.run() 실행
}
핵심: while(true)로 무한 루프 → cancel() 전까지 스레드 안 죽음

A23.¶

1. TaskQueue를 비운다 (예약된 작업 전부 제거)
2. TimerThread의 while 루프를 종료시킨다
3. Timer 스레드가 TERMINATED 상태가 된다
4. Timer 스레드가 GC Root에서 제거됨 → 참조 체인 끊김 → GC 가능

Part 6 답안¶

A24.¶

1. 스레드 생성 비용: OS에 스레드 생성을 요청하는 건 비싼 작업.
   1000번 생성/소멸하면 그 비용이 누적됨.
2. 메모리 낭비: 스레드 1개당 스택 메모리 ~512KB.
   1000개면 500MB가 스택만으로 소모.
3. OS 부하: 스레드가 너무 많으면 OS의 컨텍스트 스위칭 비용이 폭증.
   실제 처리보다 스레드 전환에 CPU를 더 쓰게 됨.
4. 제어 불가: 동시 스레드 수 제한 없이 무한 생성 → OOM 위험.

A25.¶

동시에 요청을 처리하는 스레드가 최대 200개라는 뜻이다.
200개가 모두 사용 중일 때 새 요청이 오면:
  → acceptCount(대기 큐) 크기만큼 큐에서 대기한다.
  → 대기 큐도 가득 차면 요청을 거부한다 (503 에러 또는 연결 거부).

A26.¶

1. 스레드 수:
   Timer는 스레드 1개. 작업 하나가 오래 걸리면 다음 작업이 밀린다.
   ScheduledExecutorService는 스레드 여러 개 가능. 병렬 처리 가능.

2. 예외 처리:
   Timer는 작업에서 예외가 터지면 Timer 전체가 죽는다 (모든 예약 작업 중단).
   ScheduledExecutorService는 예외가 터져도 다른 작업에 영향 없다.

Part 7 답안¶

A27.¶

일반 스레드: JVM은 모든 일반 스레드가 종료될 때까지 종료되지 않는다.
             일반 스레드가 1개라도 살아있으면 JVM은 계속 돌아간다.

데몬 스레드: 모든 일반 스레드가 종료되면 데몬 스레드는 자동 종료된다.
             JVM 종료를 막지 않는다. 보조 역할.

예: Tomcat의 HTTP 스레드(일반)가 살아있는 한, Timer-0(데몬)도 계속 살아있다.

A28.¶

데몬 스레드는 "JVM 종료 시" 같이 죽는 거지,
"JVM이 돌아가는 동안" 알아서 죽는 게 아니다.

우리 서버: Timer-0은 데몬 스레드(new Timer(true))지만,
Tomcat이 돌아가는 동안(= 일반 스레드 살아있는 동안)에는 안 죽는다.
5일 동안 살아있으면서 DelegatingClassLoader 4,257개를 누적시켰다.

데몬이라고 안심하면 안 된다.
서버가 돌아가는 동안에는 일반 스레드와 동일하게 메모리를 잡아먹는다.

A29.¶

스레드가 살아있다는 건 아직 실행할 코드가 남아있다는 뜻이다.
실행할 코드가 참조하는 객체를 아직 사용할 수 있다.
그래서 GC가 "이 객체 아직 쓸 수도 있겠구나"라고 판단해서 못 치운다.

GC는 "확실히 안 쓰는 것"만 치운다.
스레드가 살아있으면 "확실히 안 쓴다"고 보장 못 하므로 GC 대상에서 제외.

A30.¶

1. 스레드가 오래 살아있음 (무한 루프, Timer 등)
2. 그 스레드가 참조하는 객체 체인이 시간에 따라 계속 커짐
3. 스레드가 GC Root이므로 체인 전체가 GC 불가

3가지가 동시에 충족되면 메모리 누수.
하나라도 빠지면 누수 아님:
  - 스레드가 금방 죽으면 → 참조 끊김 → GC 가능
  - 체인이 안 커지면 → 메모리 일정 유지 (정상)
  - GC Root가 아니면 → GC가 알아서 정리

Part 8 답안¶

A31.¶

1. 서버 시작 → Spring이 context-sqlMap.xml 읽음
2. RefreshableSqlSessionFactoryBean 생성 → afterPropertiesSet() 호출
3. setRefreshable()에서 new Timer(true) → Timer-0 스레드 탄생
4. timer.schedule(task, 0, 1000) → 1초마다 task.run() 실행 시작
5. task.run() → isModified() → mapperLocations 126개 XML의 lastModified() 호출
6. lastModified() 내부에서 리플렉션(Method.invoke) 발생
7. 리플렉션 호출 16회 초과 시 JVM이 바이트코드 생성 → DelegatingClassLoader 생성
8. Timer-0(GC Root) → TimerTask → 리플렉션 캐시 → DelegatingClassLoader 참조 체인 연결
9. Timer-0이 살아있으므로 체인 전체 GC 불가
10. 1초마다 반복 → 5일간 4,257개 누적 → 매일 메모리 ~10% 증가

A32.¶

그 객체가 GC Root까지 참조 체인으로 연결되어 있다는 뜻.
= GC 대상이 아니다.
= GC가 치울 수 없다.
= 메모리에 계속 남아있다.

결과가 안 나왔다면 = GC Root에 연결 안 됨 = GC 가능 = 정상.

A33.¶

interval=0일 때:
  → timer.schedule() 호출 안 함 → 감시 안 함 → 누수 방지됨
  → 하지만 Timer 객체 자체는 생성됨 (new Timer(true))
  → Timer-0 스레드가 생성됨 (비록 할 일 없이 대기하지만)
  → 불필요한 리소스(스레드 1개 + Timer 객체)가 남아있음
  → 코드에 의미 없는 기능이 남아있어 혼란을 줄 수 있음

SqlSessionFactoryBean 교체:
  → Timer 객체 자체가 생성 안 됨
  → Timer-0 스레드 없음
  → 불필요한 코드/리소스 완전 제거
  → 가장 깔끔하고 근본적인 해결

A34.¶

Bean 2개 → Timer 2개 (Timer-0, Timer-1)
→ 1초마다 감시하는 스레드가 2개
→ DelegatingClassLoader 생성 속도가 2배
→ 메모리 누수 속도가 2배
→ 실제로 18.5MB + 16.7MB = 35.2MB를 두 Bean이 차지

Bean 중복은 component-scan 범위가 Root Context와 Servlet Context에서
겹치기 때문에 발생한다.

A35.¶

1. 배포 직후: Timer-0 스레드가 존재하지 않는지 확인
   (jstack 또는 MAT의 Thread Overview에서 확인)

2. 1일 후: 메모리 사용량이 증가하지 않는지 확인
   (APM 대시보드의 메모리 그래프가 평평한지)

3. 7일 후: ClassLoader 수가 정상 범위(10~50개)를 유지하는지 확인
   (jcmd VM.classloader_stats 또는 APM 메트릭)

7일간 메모리 그래프가 일정하고 ClassLoader 수가 정상이면 해결 확정.
