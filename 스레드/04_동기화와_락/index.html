04. 동기화와 락¶

이 장의 목표: 두 스레드가 동시에 같은 자원에 접근하면 왜 문제인지, 어떻게 막는지 이해한다

1. 문제 상황: 경쟁 조건 (Race Condition)¶

1.1 예시¶

class Counter {
    int count = 0;

    void increment() {
        count++;  // 이거 한 줄인데 사실 3단계야
    }
}

count++는 사실 이렇게 동작한다:

1단계: count 값을 읽는다 (READ)    → count = 0
2단계: 1을 더한다 (ADD)            → 0 + 1 = 1
3단계: 결과를 저장한다 (WRITE)      → count = 1

1.2 두 스레드가 동시에 count++하면?¶

기대: count = 2 (1씩 두 번 더했으니까)

실제로 벌어질 수 있는 일:

시간  │  스레드A          │  스레드B          │  count
──────┼───────────────────┼───────────────────┼────────
  1   │  READ count=0     │                   │    0
  2   │                   │  READ count=0     │    0
  3   │  ADD 0+1=1        │                   │    0
  4   │                   │  ADD 0+1=1        │    0
  5   │  WRITE count=1    │                   │    1
  6   │                   │  WRITE count=1    │    1
──────┼───────────────────┼───────────────────┼────────
결과  │                   │                   │    1 ← 2여야 하는데!

두 스레드가 동시에 READ해서 둘 다 0을 읽었다.
둘 다 0+1=1을 저장했다.
결과: 2가 아니라 1.
1번 증가가 "씹힌" 거다.

이걸 경쟁 조건(Race Condition)이라고 한다.

1.3 왜 위험한가¶

이 버그의 특징:
  - 항상 발생하는 게 아님 (타이밍에 따라 다름)
  - 테스트에서는 안 터지고 운영에서 터짐
  - 재현이 어려움
  - 찾기 극도로 어려움

→ 멀티스레드 버그는 최악의 버그

2. 해결: synchronized¶

2.1 synchronized 메서드¶

class Counter {
    int count = 0;

    synchronized void increment() {  // ← 한 번에 하나의 스레드만 실행 가능
        count++;
    }
}

synchronized가 하는 일:
  스레드A가 increment() 진입 → 문 잠금 (락 획득)
  스레드B가 increment() 진입 시도 → 문 잠겨있음 → 대기 (BLOCKED)
  스레드A가 increment() 완료 → 문 열림 (락 해제)
  스레드B가 increment() 진입 → 문 잠금
  ...

→ 절대 동시에 실행 안 됨
→ Race Condition 방지

2.2 synchronized 블록¶

class Counter {
    int count = 0;
    final Object lock = new Object();

    void increment() {
        synchronized(lock) {     // ← 이 블록 안만 동기화
            count++;
        }
        // 여기는 동기화 밖 → 여러 스레드 동시 실행 가능
    }
}

메서드 전체 vs 블록:

synchronized 메서드:  메서드 전체를 잠금 → 범위 넓음
synchronized 블록:    필요한 부분만 잠금 → 범위 좁음 → 더 효율적

비유:
  메서드 전체 잠금 = 화장실 전체를 혼자 차지
  블록만 잠금     = 변기만 혼자 쓰고, 세면대는 공유

3. Lock 인터페이스 (더 세밀한 제어)¶

3.1 ReentrantLock¶

import java.util.concurrent.locks.Lock;
import java.util.concurrent.locks.ReentrantLock;

class Counter {
    int count = 0;
    Lock lock = new ReentrantLock();

    void increment() {
        lock.lock();       // 잠금
        try {
            count++;
        } finally {
            lock.unlock();  // 반드시 해제 (finally에서!)
        }
    }
}

3.2 synchronized vs Lock¶

synchronized vs Lock (ReentrantLock)

synchronized:

  • 블록 끝나면 자동 해제
  • 간단함
  • try/finally 안 써도 됨
  • 타임아웃 불가

Lock:

  • 수동으로 unlock() 해야 함
  • finally에서 반드시 unlock() (안 하면 영원히 잠김)
  • tryLock(timeout) 가능
  • 읽기/쓰기 분리 가능 (ReentrantReadWriteLock)

3.3 ReentrantReadWriteLock¶

// 우리 코드에 있는 것!
private final ReentrantReadWriteLock rwl = new ReentrantReadWriteLock();
private final Lock r = rwl.readLock();   // 읽기 락
private final Lock w = rwl.writeLock();  // 쓰기 락

읽기 락 (r.lock()):
  - 여러 스레드가 동시에 읽기 가능
  - 쓰기 중이면 대기

쓰기 락 (w.lock()):
  - 쓰기 중에는 다른 모든 접근 차단
  - 읽기든 쓰기든 다 대기

비유:
  도서관 책:
  - 읽기: 여러 명이 동시에 읽을 수 있음
  - 수정: 한 명만 수정 가능, 수정 중엔 아무도 못 읽음

4. 데드락 (Deadlock)¶

4.1 정의¶

두 스레드가 서로 상대방이 가진 락을 기다리면서 영원히 진행 못 하는 상태

4.2 예시¶

스레드A: lock1 획득 → lock2 대기 중...
스레드B: lock2 획득 → lock1 대기 중...

→ A는 B가 lock2 놓기를 기다림
→ B는 A가 lock1 놓기를 기다림
→ 둘 다 영원히 대기
→ 데드락!

// 데드락 예시 코드
Object lock1 = new Object();
Object lock2 = new Object();

// 스레드 A
new Thread(() -> {
    synchronized(lock1) {          // lock1 획득
        Thread.sleep(100);
        synchronized(lock2) { }    // lock2 대기... 영원히
    }
}).start();

// 스레드 B
new Thread(() -> {
    synchronized(lock2) {          // lock2 획득
        Thread.sleep(100);
        synchronized(lock1) { }    // lock1 대기... 영원히
    }
}).start();

4.3 데드락 방지¶

데드락 방지 방법

  1. 락 획득 순서 통일 -- 모든 스레드가 lock1 -> lock2 순서로 획득 -- 역순 방지
  2. 타임아웃 사용 -- tryLock(1, TimeUnit.SECONDS) -- 1초 안에 못 얻으면 포기
  3. 중첩 락 최소화 -- synchronized 안에 synchronized 넣지 않기

5. 우리 프로젝트의 동기화¶

// RefreshableSqlSessionFactoryBean.java

private final ReentrantReadWriteLock rwl = new ReentrantReadWriteLock();
private final Lock r = rwl.readLock();
private final Lock w = rwl.writeLock();

// SQL 매퍼 새로고침 (쓰기)
public void refresh() {
    w.lock();          // 쓰기 락 획득
    try {
        super.afterPropertiesSet();  // SqlSessionFactory 재빌드
    } finally {
        w.unlock();    // 쓰기 락 해제
    }
}

// SqlSessionFactory 가져오기 (읽기)
private Object getParentObject() {
    r.lock();          // 읽기 락 획득
    try {
        return super.getObject();
    } finally {
        r.unlock();    // 읽기 락 해제
    }
}

왜 이렇게 했나:

Timer-0 스레드:  1초마다 파일 체크 → 변경 감지 시 refresh() 호출 (쓰기)
HTTP 스레드들:   사용자 요청 처리 시 getParentObject() 호출 (읽기)

쓰기 중에 읽으면?
  → 반쯤 만들어진 SqlSessionFactory를 읽을 수 있음
  → 에러 발생 가능

ReadWriteLock으로:
  → 읽기끼리는 동시 가능 (성능)
  → 쓰기 중에는 읽기 차단 (안전)

6. 핵심 정리¶

핵심 정리

경쟁 조건(Race Condition): 여러 스레드가 동시에 같은 자원 접근 -- 결과 꼬임

해결 방법:

  • synchronized: 간단, 자동 해제, 세밀한 제어 불가
  • Lock: 수동 해제 필요, 타임아웃/읽기쓰기 분리 가능

데드락: 두 스레드가 서로의 락을 기다림 -- 영원히 멈춤. 방지: 락 순서 통일, 타임아웃, 중첩 최소화

우리 코드: ReentrantReadWriteLock 사용 -- refresh(쓰기)와 getObject(읽기) 분리
