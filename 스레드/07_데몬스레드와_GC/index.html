07. 데몬 스레드와 GC¶

이 장의 목표: 데몬 스레드가 뭔지, 스레드가 왜 GC Root인지, 왜 메모리 누수 원인이 되는지 이해한다

1. 데몬 스레드 vs 일반 스레드¶

1.1 정의¶

일반 스레드 vs 데몬 스레드

일반 스레드 (User Thread):

  • JVM은 일반 스레드가 모두 끝날 때까지 종료 안 함
  • 기본값

데몬 스레드 (Daemon Thread):

  • 일반 스레드가 모두 끝나면 자동 종료됨
  • 보조 역할 (GC, 모니터링 등)
  • JVM 종료를 막지 않음

1.2 설정 방법¶

Thread t = new Thread(() -> {
    while (true) {
        System.out.println("실행 중");
    }
});

t.setDaemon(true);  // 데몬 스레드로 설정 (start() 전에!)
t.start();

1.3 비유¶

일반 스레드 = 정직원
  → 정직원이 다 퇴근해야 회사(JVM) 문 닫음

데몬 스레드 = 야간 경비원
  → 정직원 다 퇴근하면 경비원도 자동 퇴근
  → 경비원 혼자 남아있다고 회사가 안 닫히는 건 아님

1.4 JVM 종료 조건¶

JVM이 종료되는 조건:
  → 모든 일반 스레드(User Thread)가 종료됨
  → 데몬 스레드만 남으면 JVM 종료
  → 데몬 스레드는 강제 종료됨 (finally도 실행 안 될 수 있음!)

2. 우리 코드의 데몬 스레드¶

timer = new Timer(true);  // true = 데몬 스레드

Timer-0은 데몬 스레드다.
  → JVM 종료 시 자동으로 같이 죽는다.
  → 서버(Tomcat)가 정상 종료하면 같이 죽는다.

그런데 문제는:
  → 서버가 돌아가는 동안에는 죽지 않는다!
  → Tomcat의 HTTP 스레드(일반 스레드)가 살아있으니까
  → JVM이 종료되지 않으니까
  → Timer-0(데몬)도 안 죽는다
  → 5일, 10일, 30일... 계속 살아있다

데몬이면 알아서 죽지 않나요?

아니. 데몬은 "JVM 종료 시" 같이 죽는 거지, "JVM이 돌아가는 동안" 알아서 죽는 게 아니야.

서버가 돌아가고 있으면 = JVM 안 죽음 = 데몬도 안 죽음

데몬 스레드 != 안전한 스레드. 데몬 스레드도 메모리 누수 원인이 될 수 있다.

3. 스레드와 GC Root¶

3.1 GC Root 종류 (복습)¶

GC가 객체를 치울지 말지 판단하는 기준:
  → GC Root에서 참조 체인으로 연결되어 있으면 → 안 치움
  → 연결 안 되어 있으면 → 치움

GC Root 종류:
  1. 활성 스레드 (Thread)             ← 핵심!
  2. static 변수
  3. JNI 참조
  4. synchronized에 사용된 모니터 객체
  5. 시스템 클래스 (Bootstrap ClassLoader가 로드한 것)

3.2 왜 활성 스레드가 GC Root인가¶

스레드가 살아있다는 건:
  → 실행할 코드가 아직 남아있다는 뜻
  → 그 코드가 참조하는 객체를 아직 쓸 수도 있다는 뜻
  → 그러니까 GC가 못 치움

비유:
  직원이 아직 회사에 있으면
  → 직원 책상 위의 서류를 치울 수 없음
  → 직원이 쓸 수도 있으니까
  → 직원이 퇴사해야 서류 정리 가능

3.3 스레드 → 참조 체인¶

활성 스레드가 GC Root일 때:

스레드의 스택 프레임 안에 있는 모든 지역 변수
  → 그 변수들이 가리키는 객체
    → 그 객체들의 필드가 가리키는 객체
      → 그 객체들의 필드가 가리키는 객체
        → ... (체인)

이 체인에 연결된 모든 것이 GC 불가

4. 메모리 누수가 발생하는 패턴¶

4.1 패턴: 살아있는 스레드 + 커지는 참조¶

메모리 누수 패턴

조건:

  1. 스레드가 오래 살아있음 (무한 루프, Timer 등)
  2. 그 스레드가 참조하는 객체 체인이 계속 커짐
  3. GC가 치울 수 없음 (스레드 = GC Root)

결과: 메모리 사용량이 시간에 비례하여 증가 -- 결국 OutOfMemoryError

우리 서버:

  1. Timer-0 살아있음 (while(true)) -- 충족
  2. 리플렉션 캐시 -> ClassLoader 계속 생성 -- 충족
  3. Timer-0이 GC Root -> 체인 GC 불가 -- 충족

3가지 조건 모두 충족 -- 메모리 누수!

4.2 정상적인 경우¶

스레드가 살아있어도 참조 체인이 안 커지면 문제없다.

예: Tomcat HTTP 스레드
  → 요청 처리 → 응답 보냄 → 지역 변수 사라짐
  → 다음 요청 처리 → 이전 객체는 참조 끊김 → GC 가능
  → 스레드는 살아있지만 메모리 누적 없음

예: 우리 Timer-0 (문제)
  → 1초마다 실행 → 리플렉션 캐시에 ClassLoader 추가
  → 추가된 것은 참조 체인에 연결 → GC 불가
  → 스레드 살아있고 참조 체인도 계속 커짐 → 누적!

5. 스레드 관련 메모리 누수 방지 원칙¶

스레드 메모리 누수 방지 원칙

  1. 불필요한 장수 스레드 만들지 않기 -- 필요 없으면 스레드 안 만드는 게 최선
  2. 스레드가 참조하는 객체 체인 최소화 -- 필요 없는 참조는 null 처리 -- 큰 컬렉션 참조 주의
  3. ThreadLocal 사용 시 반드시 remove() -- ThreadLocal은 스레드별 변수 저장 -- 스레드풀에서 재사용되면 이전 값이 남아있음 -- remove() 안 하면 누수
  4. Timer 대신 ScheduledExecutorService 사용 -- 예외 안전, 스레드 관리 용이
  5. 운영 환경에서 개발용 기능 끄기 -- 우리 케이스: SQL 자동 감지 = 개발 편의 기능 -- 운영에서는 꺼야 함

6. ThreadLocal — 추가 개념¶

// ThreadLocal: 스레드별로 독립된 변수를 가질 수 있음
ThreadLocal<String> userName = new ThreadLocal<>();

// 스레드 A에서
userName.set("홍길동");

// 스레드 B에서
userName.set("이순신");

// 스레드 A에서 get() → "홍길동" (B의 값이 아님)
// 스레드 B에서 get() → "이순신" (A의 값이 아님)

ThreadLocal 누수 패턴:

스레드풀 사용 시:
  스레드 #1: userName.set("홍길동") → 요청 처리 → 응답 → 반납
  스레드 #1: 다음 요청 → userName.get() → "홍길동" ← 이전 값이 남아있음!

  → remove() 안 하면 이전 사용자 데이터가 다른 요청에 노출
  → 보안 문제 + 메모리 누수
  → 반드시 finally에서 userName.remove() 호출

7. 핵심 정리¶

핵심 정리

데몬 스레드: 일반 스레드 다 죽으면 자동 종료. JVM 종료를 막지 않음. 하지만 JVM이 돌아가는 동안에는 안 죽음! 데몬이라고 안전한 게 아니다.

스레드와 GC: 활성 스레드 = GC Root. 스레드가 참조하는 모든 체인 = GC 불가. 참조 체인이 계속 커지면 = 메모리 누수.

우리 서버: Timer-0 (데몬, 살아있음) -> TimerTask -> 리플렉션 캐시 -> DelegatingClassLoader 4,257개 (GC 불가, 누적)

방지 원칙: 불필요한 장수 스레드 금지 / 참조 체인 최소화 / ThreadLocal은 반드시 remove() / 운영에서 개발용 기능 끄기
