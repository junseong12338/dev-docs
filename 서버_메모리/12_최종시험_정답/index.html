12. 최종시험 - 정답 및 해설¶

"정답 맞히는 건 시작이야. 왜 그게 정답인지 설명 못 하면 불합격."

이건 정답지가 아니야. 해설지야. 정답만 보고 넘어가면 Lv1에서 영원히 못 벗어나. 틀린 문제는 해당 챕터 다시 읽고 와.

PART 1: 객관식 정답 및 해설¶

Q1. 1KB는 몇 바이트인가? (객관식, 2점)¶

정답: (C) 1,024 바이트

해설:

보기 판정 이유
(A) 100 바이트 틀림 10진법 착각. 컴퓨터는 2진법 기반이야.
(B) 1,000 바이트 틀림 SI 단위(kB, 소문자 k)에서는 1,000이지만, 전통적인 컴퓨터 과학에서 1KB = 1,024. 이건 2^10이야.
(C) 1,024 바이트 정답 2^10 = 1,024. KB(대문자 K) = Kibibyte = 1,024 바이트.
(D) 1,048 바이트 틀림 아무 근거 없는 숫자. 함정.

핵심은 2의 거듭제곱이야. 1KB = 2^10, 1MB = 2^20, 1GB = 2^30. 이거 외워.

-> 01장 "바이트 단위" 참조

Q2. 메모리 계층 구조 순서 (객관식, 2점)¶

정답: (B) 레지스터 -> 캐시 -> RAM -> 디스크

해설:

보기 판정 이유
(A) RAM -> 레지스터 -> 캐시 -> 디스크 틀림 RAM이 CPU에 가장 가깝다고? 레지스터가 CPU 안에 있어.
(B) 레지스터 -> 캐시 -> RAM -> 디스크 정답 CPU 안(레지스터) -> CPU 옆(캐시) -> 메인보드(RAM) -> 저장장치(디스크)
(C) 캐시 -> 레지스터 -> RAM -> 디스크 틀림 레지스터가 캐시보다 더 가까워. 레지스터는 CPU 내부, 캐시는 CPU와 RAM 사이.
(D) 레지스터 -> RAM -> 캐시 -> 디스크 틀림 캐시가 RAM보다 더 멀다고? 캐시는 RAM 접근을 줄이려고 중간에 둔 거야.

기억법: 가까울수록 빠르고 비싸고 작다. 레지스터(가장 빠름, 수 바이트) -> 캐시(MB급) -> RAM(GB급) -> 디스크(TB급).

-> 01장 "메모리 계층 구조" 참조

Q3. PID 1인 프로세스 (객관식, 2점)¶

정답: (B) init/systemd (최초 사용자 프로세스)

해설:

보기 판정 이유
(A) 커널 자체 틀림 커널은 프로세스가 아니야. 커널은 PID 개념 밖에 있어. 커널 스레드는 PID 2(kthreadd)부터.
(B) init/systemd 정답 부팅 후 커널이 가장 먼저 실행하는 사용자 공간 프로세스. 모든 프로세스의 조상.
(C) bash 쉘 틀림 쉘은 사용자가 로그인할 때 생성되는 프로세스야. PID 1보다 한참 나중.
(D) Java 프로세스 틀림 Java는 애플리케이션 프로세스. PID 1일 리가 없어. (Docker 컨테이너 안이면 예외적으로 가능하지만, 일반 서버에서는 아니야.)

Q14. VSZ와 RSS의 차이 (단답형, 3점)¶

Q15. -Xms != -Xmx일 때 발생하는 현상 (단답형, 3점)¶

Q16. Minor GC와 Full GC 대상 영역 (단답형, 3점)¶

정답: - Minor GC: Young Generation (Eden + Survivor 영역) - Full GC: 힙 전체 (Young Generation + Old Generation), 경우에 따라 Metaspace 포함

해설:

GC 종류 대상 영역 빈도 STW 시간
Minor GC Young Generation 자주 (초~분 단위) 짧다 (보통 수~수십 ms)
Full GC 힙 전체 (Young + Old) 드물게 길다 (수백 ms ~ 초 단위)

Minor GC는 Eden이 가득 차면 발생해. 빠르고 자주 일어나. Full GC는 Old 영역이 부족하거나 명시적 호출(System.gc()) 등에 의해 발생해. 느리고 STW가 길어서 서비스에 영향을 줘.

채점 기준: - Minor GC 대상 정확히 쓰면 1.5점 - Full GC 대상 정확히 쓰면 1.5점 - "Minor = 작은 GC, Full = 큰 GC" 같은 답은 0점

Q17. GC vs OS 페이지 관리 레벨 (단답형, 3점)¶

정답: - GC: JVM(애플리케이션) 레벨. JVM 힙 안에서 객체 단위로 사용/해제를 관리한다. - OS 페이지 관리: OS(커널) 레벨. 물리 메모리 페이지 단위로 프로세스에 할당/회수한다.

해설:

GC vs OS 페이지 관리 - 레벨 비교

JVM 레벨 (GC)

  • 객체 단위 (new Object())
  • 힙 안에서 살아있는 객체 추적
  • 참조 없는 객체 해제
  • 해제해도 OS에 안 돌려줌

OS 레벨 (페이지 관리)

  • 페이지 단위 (보통 4KB)
  • 물리 메모리 <-> 프로세스 매핑
  • 안 쓰는 페이지는 Swap으로 이동
  • 프로세스 종료 시 전체 회수

이 둘은 완전히 다른 세계야. GC가 열심히 해제해도 OS가 보기에는 "그 프로세스가 여전히 그 메모리를 쓰고 있다"로 보여.

채점 기준: - GC = JVM/애플리케이션 레벨 명시하면 1.5점 - OS 페이지 = OS/커널 레벨 명시하면 1.5점 - "둘 다 메모리 관리" 같은 추상적 답은 0점

-> 08장 "GC와 OS 페이지 관리는 다른 것이다" 참조

Q18. 정상 판단 핵심 기준 (단답형, 3점)¶

정답: 아래 3가지 중 하나를 정확히 쓰면 3점.

  1. 톱니 패턴 (Sawtooth Pattern): 메모리가 올라갔다가 GC 후 내려오는 것이 반복되고, 기준선이 일정하면 정상.
  2. GC 후 기준선 안정: GC 실행 후 메모리 사용량이 일정 수준으로 돌아오면 정상. 기준선이 계속 올라가면 이상.
  3. Full GC 빈도: Full GC가 너무 자주(분 단위) 발생하지 않으면 정상. 빈번하면 힙 부족 또는 누수.

해설:

정상 판단의 핵심은 "패턴"이야. 한 시점의 수치가 아니라 시간에 따른 변화를 봐야 해.

  • 톱니 패턴이면? -> 정상. 객체 생성/해제가 반복되는 건 당연한 거야.
  • 기준선이 올라가면? -> 누수 의심. 해제 못 하는 객체가 쌓이고 있어.
  • Full GC가 미친듯이 발생하면? -> 힙이 부족하거나 누수. 서비스에 STW 영향.

채점 기준: - 위 3가지 중 하나를 정확히 쓰면 3점 - 키워드만 쓰고 설명이 부족하면 2점 - "메모리 사용률이 낮으면 정상"은 1점 (틀린 건 아닌데 핵심이 아니야)

(a) 15GiB (약 16GB) -- (1점)

(b) 메모리 부족이 아니다. -- (2점)

채점 기준: - (a) 15Gi 또는 약 16GB 쓰면 1점 - (b) "부족 아님" + available 근거 쓰면 2점. "부족 아님"만 쓰면 1점. - (c) Swap 증가 추세 확인 또는 si/so 확인 언급하면 2점. "문제다/아니다"만 쓰면 0점.

Q22. JVM 힙 구조 빈칸 (코드/명령어 분석, 5점)¶

정답: (각 1점)

  • (A): Eden -- 새 객체가 최초로 생성되는 영역
  • (B): Survivor 0 (S0) -- Minor GC 생존 객체가 이동하는 영역
  • (C): Survivor 1 (S1) -- Minor GC 생존 객체가 이동하는 영역 (S0과 번갈아 사용)
  • (D): Old -- 오래 살아남은 객체가 승격(Promotion)되어 저장되는 영역
  • (E): Full -- Old Generation 전체가 Full GC의 대상

해설:

Young Generation:
  Eden → 새 객체가 태어나는 곳
  Survivor 0(S0) ↔ Survivor 1(S1) → GC 생존자가 번갈아 이동

Old Generation:
  → Survivor에서 여러 번 살아남은(age threshold 초과) 객체가 승격(Promotion)
  → Full GC의 대상

객체 흐름: new -> Eden -> (Minor GC 생존) -> S0 또는 S1 -> (여러 번 생존) -> Old -> (참조 끊기면) -> GC 대상

채점 기준: A~E 각 1점. 총 5점.

Q25. GC vs OS 페이지 관리 참/거짓 (코드/명령어 분석, 5점)¶

정답:

(a) "JVM에서 GC가 실행되면, 해제된 메모리는 OS에 즉시 반환된다." -- (2점)

거짓.

이유: GC는 JVM 힙 내부에서 객체를 해제하는 것이지, OS에게 메모리를 돌려주는 게 아니다. JVM은 한번 확보한 힙 메모리를 일반적으로 OS에 반환하지 않는다. GC 후 빈 공간은 JVM이 새 객체 생성 시 재사용한다. OS 관점에서 해당 프로세스의 RSS는 줄어들지 않거나 아주 미미하게 줄어든다.

(참고: 일부 GC 알고리즘(G1, ZGC 등)은 -XX:MinHeapFreeRatio 등 옵션으로 OS에 반환 가능하지만, 기본 동작이 아니고 "즉시 반환"은 틀린 표현이다.)

(b) "OS는 프로세스가 사용하지 않는 물리 메모리 페이지를 자동으로 회수할 수 있다." -- (1.5점)

참.

OS는 메모리 압박 시 오래 접근하지 않은 페이지를 Swap으로 내보내거나, 캐시/버퍼를 회수할 수 있다. 이건 OS 레벨의 메모리 관리야. 프로세스가 명시적으로 반환하지 않아도 OS가 판단해서 회수한다.

(c) "GC가 하는 일과 OS의 페이지 회수가 하는 일은 동일한 레벨의 작업이다." -- (1.5점)

거짓.

이유: 완전히 다른 레벨이다. - GC: JVM(애플리케이션) 레벨. 힙 안에서 객체 단위로 참조를 추적하고 해제. - OS 페이지 관리: OS(커널) 레벨. 페이지 단위(4KB)로 물리 메모리 할당/회수.

GC는 "이 객체가 더 이상 쓰이나?"를 판단하고, OS는 "이 페이지가 최근에 접근됐나?"를 판단한다. 판단 기준, 동작 단위, 동작 레벨 전부 다르다.

해설:

이 문제가 08장의 핵심을 전부 담고 있어.

GC vs OS - 왜 헷갈리나

헷갈리는 이유: 둘 다 "메모리를 관리한다"고 하니까 같은 거라고 착각하는 거야.

다른 점:

  • GC → JVM 안 → 객체 단위 → 참조 기반
  • OS → 커널 → 페이지 단위 → 접근 시간 기반

관계:

  • GC가 해제해도 OS 관점에서 변화 없음
  • OS가 Swap으로 내려도 JVM은 모름
  • 서로 독립적으로 동작

채점 기준: - (a) "거짓" + JVM 내부 해제/OS 반환 아님 설명하면 2점. "거짓"만 쓰면 0.5점. - (b) "참" + OS 회수 메커니즘 설명하면 1.5점. "참"만 쓰면 0.5점. - (c) "거짓" + 레벨 차이 설명하면 1.5점. "거짓"만 쓰면 0.5점.

-> 08장 "GC와 OS 페이지 관리는 다른 것이다" 참조

PART 4: 서술형 정답 및 해설¶

Q26. "메모리"의 문맥별 의미 (서술형, 4점)¶

정답:

문맥 1: "이 컴퓨터는 메모리가 16GB야" -> 물리 메모리(RAM). 메인보드에 장착된 RAM 모듈의 총 용량. 하드웨어 스펙. (1.5점)

문맥 2: "이 프로세스가 메모리를 3GB 쓰고 있어" -> 해당 프로세스의 RSS(Resident Set Size) 또는 넓게는 가상 메모리 사용량. 프로세스 단위의 메모리 사용. OS가 해당 프로세스에 할당한 물리 메모리 양. (1.5점)

문맥 3: "JVM 힙 메모리를 2GB로 설정했어" -> JVM 힙 영역의 크기 (-Xmx 설정값). JVM 프로세스 전체 메모리가 아니라 힙만. JVM은 힙 외에도 Metaspace, 스레드 스택, 네이티브 메모리 등을 사용하므로, "힙 2GB"라고 해서 프로세스 전체가 2GB인 건 아니다. (1점)

해설:

"메모리"라는 단어가 얼마나 모호한지 알겠어? 같은 단어인데 문맥마다 의미가 달라:

문맥 가리키는 것 레벨
"컴퓨터 메모리 16GB" RAM (물리 하드웨어) 하드웨어 레벨
"프로세스가 3GB 사용" RSS/프로세스 메모리 OS 레벨
"JVM 힙 2GB" 힙 영역 크기 애플리케이션(JVM) 레벨

서버 상태를 논의할 때 "메모리가 몇 %야"라고 하면, 뭘 기준으로 한 건지 반드시 명확히 해야 해. 안 그러면 서로 다른 걸 보면서 "메모리 부족이다/아니다" 싸우게 돼.

채점 기준: - 문맥 1: RAM/물리 메모리 명시 1.5점 - 문맥 2: 프로세스 단위/RSS 언급 1.5점 - 문맥 3: JVM 힙 영역 명시 + "프로세스 전체가 아님" 언급 1점 - 3개 다 "메모리입니다"로 퉁치면 0점

-> 01장 "메모리란", 04장 "프로세스 메모리", 06장 "JVM 힙" 참조

Q27. [함정 문제] free가 0이면 메모리 부족? (서술형, 4점)¶

정답:

틀릴 수 있는 이유 (2점):

Linux는 사용하지 않는 물리 메모리를 디스크 캐시(buff/cache)로 적극 활용한다. 따라서 시스템이 오래 가동되면 free는 자연스럽게 0에 가까워진다. 이건 메모리 부족이 아니라 OS가 메모리를 효율적으로 사용하고 있다는 뜻이다. 캐시/버퍼는 새 프로세스가 메모리를 요청하면 OS가 즉시 회수해서 줄 수 있다.

올바른 판단 방법 (2점):

  1. free -h에서 available 컬럼을 확인한다. available = free + 회수 가능한 캐시/버퍼. 이게 충분하면 메모리 부족이 아니다.
  2. Swap 사용량과 추이를 확인한다. available이 부족하고 Swap In/Out이 활발하면 그때 비로소 "메모리 부족"이라고 판단할 수 있다.
  3. available이 총 메모리의 10% 이하로 떨어지고, Swap 사용이 증가 추세이면 메모리 부족.

해설:

이건 가장 흔한 오판 중 하나야. 신입 개발자가 free -h 딱 보고 "free가 0이에요! 메모리 부족이에요!"라고 소리 지르는 상황.

판단 플로우:

free가 0에 가깝다
  → available 확인
    → available 충분 (총 메모리의 20% 이상)
      → 정상. 캐시가 잘 동작하고 있는 거야.
    → available 부족 (총 메모리의 10% 이하)
      → Swap 확인
        → Swap 미사용 또는 안정적
          → 주의 단계. 모니터링 강화.
        → Swap 증가 추세
          → 메모리 부족. 원인 분석 필요.

채점 기준: - 캐시/버퍼 회수 가능성 설명하면 2점 - available 기반 판단 + Swap 확인 절차 쓰면 2점 - "free가 0이면 부족한 거 아닌가요?"에서 한 발짝도 못 벗어나면 0점 - 이거 틀리면 03장부터 다시 읽어

-> 03장 "free -h 해석, available의 진짜 의미" 참조

Q28. -Xms == -Xmx 권장 이유 (서술형, 5점)¶

정답:

(1) 힙 확장 비용 관점 (2점):

-Xms != -Xmx이면 JVM은 메모리가 부족할 때마다 OS에 추가 메모리를 요청해서 힙을 확장(resize)해야 한다. 이 확장 과정 자체가 비용이다: - OS에 메모리 할당 요청 (시스템 콜) - 새 메모리 영역 초기화 - 힙 확장 시 GC가 함께 트리거될 수 있음 - 확장 동안 애플리케이션 일시 정지 가능

-Xms == -Xmx로 설정하면 시작 시 전체 힙을 한 번에 확보하므로 런타임 중 확장 비용이 0이다.

(2) GC 동작 예측 관점 (1.5점):

힙 크기가 변하면 GC의 동작도 변한다. Young/Old 비율, GC 트리거 임계값 등이 힙 크기에 따라 달라지기 때문이다. 힙 크기가 고정되면 GC 동작이 일관적이고 예측 가능해진다. 성능 테스트 결과를 프로덕션에 그대로 적용할 수 있다.

(3) 모니터링 관점 (1.5점):

힙 크기가 고정되면 모니터링 수치 해석이 명확해진다. "전체 힙 대비 사용률 X%"가 항상 같은 의미를 가진다. 힙이 확장되는 중이면 분모(전체 힙)가 바뀌므로 사용률이 순간적으로 뛰었다 떨어졌다 하면서 누수와 정상을 구분하기 어려워진다.

해설:

-Xms512m -Xmx2g 일 때:

시간 →
힙크기: 512m → 768m → 1g → 1.2g → 1.5g → 2g
        ↑      ↑      ↑     ↑       ↑     ↑
       시작   확장1  확장2  확장3   확장4   최대

각 확장마다: OS 시스템콜 + GC 가능성 + 순간 정지

-Xms2g -Xmx2g 일 때:

시간 →
힙크기: 2g ──────────────────────────── 2g
        ↑                                ↑
       시작                            항상 동일

확장 없음. 깔끔.

실무에서는 거의 항상 -Xms == -Xmx로 맞춘다. 다르게 설정하는 건 개발 환경이나 메모리가 매우 제한적인 특수 상황뿐이야.

채점 기준: - (1) 확장 비용 설명 + 구체적 내용 쓰면 2점. "비용이 든다"만 쓰면 1점. - (2) GC 예측 가능성 설명하면 1.5점. "GC에 영향 준다"만 쓰면 0.5점. - (3) 모니터링 해석 용이성 설명하면 1.5점. "모니터링이 편하다"만 쓰면 0.5점.

Q29. 모니터링 패턴 판단 (서술형, 4점)¶

정답:

[패턴 A] -- (2점)

이상 (메모리 누수 의심).

근거: 그래프가 24시간 동안 지속적으로 우상향하고 있다. 톱니 패턴(올라갔다 내려갔다)이 있긴 하지만, 기준선(GC 후 최저점)이 점점 올라가고 있다. 20% -> 30% -> 40% -> 50% -> 60% -> 70% -> 80%으로 기준선이 상승. 이는 GC가 회수하지 못하는 객체가 계속 쌓이고 있다는 뜻이다. 전형적인 메모리 누수 패턴이며, 이대로 두면 결국 OOM이 발생한다.

[패턴 B] -- (2점)

정상 (건강한 톱니 패턴).

근거: 메모리 사용률이 약 40%~70% 사이에서 주기적으로 올라갔다 내려갔다 반복하고 있다. 핵심은 GC 후 기준선이 일정 (약 40~50% 수준으로 매번 돌아옴)하다는 점이다. 이는 객체가 생성되고(사용률 상승) -> GC가 회수하고(사용률 하락) -> 다시 생성되는 정상적인 JVM 동작이다. 기준선이 올라가지 않으므로 누수가 없다.

해설:

모니터링 패턴 판단 핵심:

1. 기준선(GC 후 최저점)을 본다
   → 올라가는가? → 누수 의심
   → 일정한가?   → 정상

2. 톱니 패턴 자체는 정상이다
   → 올라갔다 내려가는 건 GC가 잘 동작한다는 뜻

3. 절대값이 아니라 패턴을 본다
   → 70%라도 안정적이면 정상
   → 40%라도 계속 올라가면 위험

채점 기준: - 패턴 A: "누수/이상" 판단 + 기준선 상승 근거 쓰면 2점. 판단만 쓰면 1점. - 패턴 B: "정상" 판단 + 기준선 안정 근거 쓰면 2점. 판단만 쓰면 1점. - A와 B 판단이 반대면 각 0점.

-> 09장 "정상 vs 이상 판단" 참조

Q30. 종합 판단 (서술형, 4점)¶

정답:

시니어로서의 대응 (최소 3가지 관점):

관점 1: "55%"의 기준 확인 (1점)

추가 질문: "55%가 뭘 기준으로 한 거야? OS 전체 메모리? JVM 힙 사용률?"

  • OS 전체 기준 55% (16GB 중 9GB)라면, 이건 모든 프로세스 합산이야. JVM 하나만의 문제가 아닐 수 있어.
  • JVM 힙 기준 55%라면 의미가 다르지.
  • 먼저 기준을 명확히 해야 판단이 가능해.

관점 2: free -h의 free 300MB와 available 확인 (1점)

추가 질문: "available은 얼마야?"

  • free가 300MB라고 했는데, Q13, Q27에서 배운 대로 free만으로 판단하면 안 돼.
  • available이 4~5GB면 캐시/버퍼가 회수 가능하므로 당장 문제없어.
  • available이 300MB 근처면 그때 비로소 주의.

관점 3: 추이(Trend) 확인 (1점)

추가 질문: "55%가 안정적이야? 올라가는 중이야? 언제부터 이 수치야?"

  • 안정적(한 달째 55% 유지): 정상. 서비스 특성상 그 정도 쓰는 거.
  • 올라가는 중(어제 34%, 오늘 55%): 누수 또는 비정상 증가. 원인 파악 필요.
  • 한 시점의 수치만으로는 절대 판단 불가.

추가 관점 (보너스):

  • Swap 1GB 사용 원인: "Swap이 언제부터 사용됐어? 증가 추세야?" -- Swap 사용 자체가 과거 메모리 압박의 흔적. 원인 필요.
  • jstat 확인: "Old 영역 사용률은? Full GC 빈도는?" -- JVM 내부 상태를 봐야 실체를 알 수 있어.
  • 서비스 영향: "응답 시간은? 에러율은?" -- 메모리 수치보다 서비스 영향이 더 중요할 수 있어.

종합 판단:

이 보고는 "데이터를 나열했을 뿐 분석하지 않았다." 숫자만 나열하면 보고가 아니야. 55%라는 숫자, free 300MB라는 숫자, Swap 1GB라는 숫자를 각각 해석하고, 추이와 맥락을 함께 봐야 비로소 "문제다/아니다"를 판단할 수 있어.

해설:

이 문제가 01장부터 10장까지 전부 연결되는 종합 문제야.

01장: "메모리"가 뭘 가리키는지 (물리 메모리 16GB)
02장: 프로세스 관점 (어떤 프로세스가 얼마나 쓰는지)
03장: free -h 해석 (free 300MB ≠ 부족, available이 핵심)
04장: committed vs resident (RSS와 모니터링 %의 관계)
05장: JVM 옵션 확인 필요 (-Xms/-Xmx가 뭔지)
06장: 힙 구조/확장 여부
07장: GC 상태 (jstat 확인 필요)
08장: GC ≠ OS 메모리 관리 (다른 레벨)
09장: 정상 vs 이상 판단 (패턴으로 봐야 해)
10장: 실전 사례와 동일한 상황 (34%→55%)

채점 기준: - 3가지 관점 이상 제시 + 각각 구체적 질문/판단 기준 쓰면 4점 - 2가지 관점이면 3점 - 1가지 관점이면 2점 - "문제 있다" 또는 "문제 없다"만 쓰면 0점 - "추가 질문" 없이 단정하면 감점 -- 는 한 시점의 숫자로 단정하지 않아
