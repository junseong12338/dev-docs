10. 실전: 우리 서버 사례 분석 - Omega¶

1. 핵심 개념 - "이게 뭐야?"¶

이 챕터는 이론이 아니야. 진짜 일어났던 일이야.

우리 LMS 서버에서 메모리가 매일 올라가서 밤새 분석하고, 원인 찾고, 고치고, 또 올라가서 또 분석하고, 또 고치고... 그 전 과정을 전부 기록한 거야.

01장부터 09장까지 배운 모든 개념이 여기서 전부 쓰여. 이론으로 배운 게 실전에서 어떻게 적용되는지, 이 챕터가 증명해.

우리 서버 스펙

항목 값
RAM 15GB
Java Java 8 (Parallel GC 기본)
WAS Tomcat
Framework Spring (eGovFramework 기반)
APM InterMax (jspd 에이전트)
JVM 옵션 -Xms8192m (최소 힙: 8GB), -Xmx12288m (최대 힙: 12GB)
문제 모니터링 메모리 사용률이 계속 올라감
기간 2026년 3월 초 ~ 3월 11일

2. 동작 원리 - "어떻게 돌아가?"¶

여기서는 "원리"가 아니라 실제 사건의 타임라인이야. 3번의 문제가 있었고, 각각 원인과 해결이 달랐어.

2-1. 전체 타임라인¶

사건 타임라인

3/1~3/5 모니터링: 메모리 매일 +10% 상승 → "아 이거 뭐야..."

3/6 [1차 문제 해결] 원인: RefreshableSqlSessionFactoryBean → 조치: SqlSessionFactoryBean으로 교체 → 결과: ROOT 앱 누수 해결

3/7~3/8 모니터링: 여전히 상승 중 → "엥? 고쳤는데 왜 또?"

3/9 [2차 문제 해결] 원인: webapps/test에 같은 누수 코드 방치 → 조치: test 앱을 webapps 바깥으로 이동 → 결과: 진짜 누수 완전 해결

3/10~3/11 모니터링: 42% → 55% 상승, 야간 횡보 → "아직도 올라가는데...?"

3/11 [3차 분석] 분석: 힙덤프 + jstat + free -h 종합 → 결론: 누수 아님. JVM 힙 확장 (정상 동작) → "아, 이건 정상이구나!"

2-2. 1차 문제 (3/6 해결) - 진짜 누수¶

현상¶

모니터링 그래프:

Memory %
100│
 90│
 80│                                          /
 70│                                        /
 60│                                      /
 50│                                    /

  0│────/────\────/─────────\/────────────→ 시간
      3/1   3/2   3/3   3/4   3/5   3/6

매일 약 10% 상승. 밤에도 안 멈춤.
→ 09장에서 배운 "야간에도 상승 = 누수" 판단 기준에 해당!

분석 과정¶

# Step 1: PID 확인
$ ps -ef | grep java | grep tomcat
root 12345 ... -Xms8192m -Xmx12288m ...

# Step 2: 힙덤프
$ jcmd 12345 GC.heap_dump /tmp/heap_20260306.hprof

# Step 3: MAT(Eclipse Memory Analyzer)로 분석
# → Leak Suspects Report
# → Dominator Tree

MAT 분석 결과¶

MAT Leak Suspects Report

Problem Suspect 1:

"java.util.Timer" 인스턴스가 다수 발견. 각 Timer가 "DelegatingClassLoader"를 참조. DelegatingClassLoader가 대량의 클래스 메타데이터를 잡고 있음.

Dominator Tree: 

java.util.Timer (스레드)
  └─ DelegatingClassLoader
       └─ [대량의 클래스 메타데이터]
            └─ mybatis 관련 클래스들

Timer 스레드 개수: 수천 개 (정상이면 0~1개)

원인¶

1차 문제 원인

범인: RefreshableSqlSessionFactoryBean (interval="1000" - 1초마다 SQL 매핑 파일 변경 감시)

뭘 하는 놈이냐: → eGovFramework에서 제공하는 "개발 편의" 빈. → SQL 매핑 파일(mapper.xml)이 수정되면 자동으로 리로드. → 개발 환경에서 서버 재시작 없이 SQL 수정 반영하려고 쓰는 거.

문제: → 1초마다 Timer 스레드를 생성 → 각 Timer가 새 ClassLoader(DelegatingClassLoader)를 만듦 → Timer = 실행 중인 스레드 = GC Root (07장!) → GC Root가 참조하고 있으니까 GC가 수거 불가! → 1초마다 수거 불가 객체가 쌓임 = 메모리 누수

GC Root (Active Thread)
  ├─ Timer-1 → ClassLoader-1 → [classes]
  ├─ Timer-2 → ClassLoader-2 → [classes]
  ├─ Timer-3 → ClassLoader-3 → [classes]
  ├─ ...
  └─ Timer-N → ClassLoader-N → [classes]

전부 GC Root에서 도달 가능
→ GC가 하나도 수거 못 함
→ 1초마다 쌓임 = 누수

조치¶

<!-- ❌ Before: 운영 환경에서 절대 쓰면 안 되는 설정 -->
<bean id="sqlSession" class="org.egovframework.rte.psl.dataaccess.
    EgovSqlSessionFactoryBean">
<!-- 실제로는 RefreshableSqlSessionFactoryBean 이었음 -->
<!-- interval="1000" → 1초마다 갱신 체크 -->

<!-- ✅ After: 운영 환경용 설정 -->
<bean id="sqlSession" class="org.mybatis.spring.SqlSessionFactoryBean">
<!-- SqlSessionFactoryBean: 감시 안 함. Timer 안 만듦. 안전. -->

조치 결과:
→ Timer 스레드 생성 중단
→ DelegatingClassLoader 무한 생성 중단
→ GC가 기존 쓰레기 수거 가능해짐
→ ROOT 앱의 메모리 누수 해결

여기서 쓴 개념들: - 07장: GC Root (Active Thread), Reachability Analysis - 07장: GC가 수거 못 하는 이유 (GC Root에서 도달 가능)

진짜 누수는 다 잡았어. 근데 모니터링이 또 올라가.

모니터링 그래프:

Memory %

   │                               ────

      3/9                3/10              3/11

42% → 55%. 누수 잡았는데도 올라감.
"아직도 누수가 있나...?"

분석 과정 (이번엔 다르게 접근)¶

이번엔 바로 힙덤프부터 뜨지 않았어. 순서대로 판단했어. 09장에서 배운 체크리스트대로:

[1단계: 현재 상태 파악]

# OS 레벨
$ free -h

Mem:           15G    8.3G   1.2G    200M     5.7G        6.9G

판단:
  used = 8.3GB → 55% 맞아
  available = 6.9GB → 넉넉! 위험 아님
  Swap = 145MB → 정상
  → OS 레벨에선 문제없어

# JVM 레벨
$ jstat -gc $(pgrep -f tomcat)
 S0C      S1C      S0U    S1U      EC        EU        OC         OU        MC       MU      CCSC    CCSU     YGC   YGCT    FGC   FGCT     GCT
174592.0 174592.0  0.0   40277.8 1398272.0 1084498.1 8388608.0  451133.9  155648.0 143567.2 18944.0 16861.5  1847  22.577   4    1.462   24.039

판단:
  OU = 451,134 KB = ~440MB
  OC = 8,388,608 KB = 8GB
  OU/OC = 440MB / 8GB = 5.4%
  → Old Gen 거의 비어있음. 누수 증거 전혀 없음!!!

  YGC = 1,847회, YGCT = 22.577초
  → 1회 평균 = 12ms. 정상.

  FGC = 4회, FGCT = 1.462초
  → 1회 평균 = 365ms. 4번밖에 안 일어남. 정상.

[2단계: 추이 분석]

──────────────
09:00    440

→ 톱니 패턴! 440MB 근처에서 안정적.
→ 누수 없음 확정.

[3단계: 그럼 왜 55%야?]

여기서 04장, 06장, 08장의 개념이 전부 연결돼:

'55%인데 왜 정상인가?' 분석

[04장 개념: committed vs resident] JVM 시작 시 8GB 예약(committed). 바로 전부 물리 RAM에 올리진 않음. 사용하면서 필요한 만큼 물리 RAM에 매핑 (Resident). 시간이 지나면 한 번이라도 쓴 페이지는 물리 RAM에 올라가 있음 → committed 영역 대부분이 Resident로 올라감.

[06장 개념: 힙 확장] -Xms(8GB) ≠ -Xmx(12GB) → JVM이 필요하면 힙을 확장할 수 있어. 확장할 때마다 OS에 물리 페이지를 추가 요청 → OS used가 올라감.

[08장 개념: GC vs OS 페이지 관리] GC가 힙 안에서 객체를 치워도 → JVM은 OS에 페이지를 돌려주지 않음 → OS used는 그대로 → 모니터링 %도 그대로.

[종합] 42% → 55%는:

  • JVM이 힙을 8GB에서 좀 더 확장하면서 물리 페이지를 더 잡음
  • OS used가 올라감 → 모니터링 %가 올라감
  • 하지만 OU는 5%밖에 안 됨 → 힙 안은 텅텅 빔
  • available 6.9GB → OS 여유 충분 / Swap 145MB → 정상

결론: 정상 동작이야. 문제가 아니야.

검증: 힙덤프도 한 번 더 확인¶

  → Timer 스레드 폭증 없음 (1차, 2차 조치가 효과 있었다는 증거)

4. 주의사항 / 함정¶

함정 1: "원인 하나 찾았으니 끝이겠지"¶

함정 1

❌ 1차 문제 고치고 "해결했다!" 생각했어. 근데 2차 문제가 있었어. 같은 JVM 프로세스 안에 다른 앱(test/)이 같은 누수 코드를 갖고 있었어.

✅ 교훈: 원인을 하나 찾았으면, "같은 원인이 다른 곳에도 있는가?" 반드시 확인. Tomcat은 webapps/ 안에 여러 앱을 자동 배포하니까, 모든 앱을 확인해야 해.

함정 2: "모니터링이 올라가면 무조건 누수"¶

함정 2

❌ 3차 분석에서 증명했어. 42% → 55% 올라갔지만 누수가 아니었어. JVM 힙 확장(정상 동작)이었어.

✅ 교훈: 모니터링 %만 보고 판단하지 마. OU 추이 + available + Swap 세 개를 같이 봐. 다 정상이면? 정상인 거야.

함정 3: "개발 편의 도구를 운영에 그대로 올리기"¶

함정 3

❌ RefreshableSqlSessionFactoryBean은 "개발 편의" 도구야. 이걸 운영 서버에 그대로 올렸어. 그게 누수 원인이 됐어.

✅ 교훈: 개발 환경과 운영 환경의 설정은 반드시 분리해야 해. profile별 설정 분리: application-dev.xml / application-prod.xml. 개발 편의 도구는 운영에서 절대 사용하면 안 돼.

함정 4: "-Xms와 -Xmx를 다르게 설정하면 혼란 생긴다"¶

함정 4

△ 우리 서버: -Xms=8GB, -Xmx=12GB. 이 차이(4GB) 때문에 JVM이 힙을 확장할 수 있고, 확장하면 OS used가 올라가고, 그걸 보고 "누수인가?" 혼란이 생겼어.

✅ 교훈: -Xms = -Xmx로 맞추면 이런 혼란을 줄일 수 있어. 대신 시작부터 큰 메모리를 잡으니까 available이 줄어드는 건 감수해야 해. 트레이드오프야.

함정 5: "서버 재시작하면 다 해결되겠지"¶

함정 5

❌ 재시작하면 일시적으로 메모리가 낮아져. 근데 누수 코드가 그대로 있으면? 시간이 지나면 또 똑같이 올라가. 재시작은 "임시 조치"일 뿐 "근본 해결"이 아니야.

✅ 교훈: 재시작 = 증상 완화. 코드 수정 = 근본 해결. 둘 다 해야 해. 순서는: 재시작(임시) → 원인 분석 → 코드 수정(근본)

우리 서버 사례 핵심 정리

챕터 사용한 개념

  • OU 5% (누수 없음)

  2. 원인 하나 찾았으면 "같은 원인이 다른 곳에도 있는가?" 확인해
  3. 개발 편의 도구를 운영에 올리지 마. 환경별 설정 분리 필수

이 챕터에서 반드시 기억할 것: 1. 01~09장의 이론은 전부 실전에서 쓰인다. 이론 따로 실전 따로가 아니야. 2. "55%인데 왜 정상인가"를 OU, available, Swap, 추이 네 가지로 설명할 수 있어야 해. 3. RefreshableSqlSessionFactoryBean(Timer 무한생성) 패턴은 실제 누수의 대표 사례야. 4. 하나의 JVM(Tomcat) 안에 여러 앱이 있을 수 있다. 전부 확인해.

Q3. 3차 분석에서 "55%인데 정상"이라고 판단한 근거 4가지를 말해봐.

Q4. 아래 상황이 우리 서버라고 치자. 1차, 2차, 3차 중 어느 상황에 해당해? 근거도 대:

jstat -gc: OU = 420MB, OC = 8GB, FGC = 3
OU 추이: 420 → 425 → 418 → 430 → 415 (톱니)
모니터링: 48%

Q5. 이 사례에서 배운 걸 바탕으로, "서버 메모리가 올라가고 있습니다" 라는 보고를 받았을 때 네가 확인해야 할 순서를 5단계로 적어봐. 각 단계에서 사용할 명령어도 같이.

정답 보기

A1. RefreshableSqlSessionFactoryBean은 interval=1000 설정으로 인해 1초마다 Timer 스레드를 생성해. Timer 스레드는 실행 중인 스레드이고, 실행 중인 스레드는 GC Root 4대장 중 하나야. GC Root에서 도달 가능한 객체는 GC가 수거할 수 없어. 각 Timer가 DelegatingClassLoader를 참조하고, ClassLoader가 대량의 클래스 메타데이터를 잡고 있으니까, GC Root(Timer) → ClassLoader → 클래스 메타데이터 전체가 수거 불가능해져. 1초마다 이런 체인이 생기니까 메모리가 계속 차오르는 거야.

A2. ROOT 앱의 누수 코드는 고쳤지만, webapps/test 폴더에 같은 RefreshableSqlSessionFactoryBean 설정이 남아있었어. Tomcat의 autoDeploy 메커니즘이 핵심이야. server.xml에 autoDeploy="true"가 설정되어 있으면, Tomcat은 webapps/ 폴더 안의 모든 폴더를 자동으로 별도의 웹 애플리케이션으로 배포해. test/ 폴더가 webapps/ 안에 있었기 때문에 Tomcat이 자동으로 test 앱을 로딩했고, test 앱은 자체적인 Spring Context를 만들면서 RefreshableSqlSessionFactoryBean도 따로 초기화한 거야. 하나의 Tomcat 프로세스(하나의 JVM) 안에 ROOT 앱과 test 앱이 동시에 떠 있었고, ROOT만 고치면 test의 누수는 그대로 남아있는 거야.

A3. "55%인데 정상" 판단 근거 4가지: 1. OU = 440MB / OC = 8GB = 5% → Old Gen이 거의 비어있음. GC가 정상적으로 수거하고 있고 누수 증거 없음. 2. available = 6.9GB → OS 레벨에서 여유 메모리 충분. 4GB 이상이면 안전. 3. Swap = 145MB → 500MB 미만이므로 정상. 물리 RAM이 부족하지 않다는 증거. 4. 야간 횡보 → 모니터링 수치가 밤에 더 이상 올라가지 않고 멈춤. 누수라면 야간에도 올라가야 함. 추가로 힙덤프 MAT 분석에서도 Leak Suspects가 없었음.

A4. 3차 상황에 해당 (정상 동작). 근거: - OU/OC = 420MB/8GB = 5.3% → 안전 (50% 미만) - OU 추이: 420→425→418→430→415 → 톱니 패턴 → 누수 아님 - available = 6.5GB → 안전 (4GB 이상) - Swap = 200MB → 안전 (500MB 미만) - FGC = 3회 → 정상 - 모니터링 48% → 표면적 수치일 뿐, 내용물이 전부 안전 이건 JVM 힙 확장으로 인한 정상적인 메모리 사용 증가. 3차 분석과 동일한 패턴.

A5. 메모리 상승 보고 시 확인 순서: 1. OS 상태 확인: free -h → available과 Swap 확인. available < 1GB이거나 Swap이 급증하면 긴급. 2. JVM 상태 확인: jstat -gc $(pgrep -f tomcat) → OU/OC 비율과 FGC 횟수 확인. OU > 80%면 누수 의심. 3. OU 추이 확인: jstat -gc $(pgrep -f tomcat) 5000 10 → 톱니인지 우상향인지 판단. 우상향이면 누수. 4. 프로세스 확인: ps -eo pid,rss,comm --sort=-rss | head -10 → 어떤 프로세스가 메모리를 먹고 있는지. 예상치 못한 놈이 있으면 원인 후보. 5. 힙덤프 분석 (누수 의심 시만): jcmd <PID> GC.heap_dump /tmp/dump.hprof → MAT로 Leak Suspects, Dominator Tree 확인. 누수 객체와 참조 체인 특정.

1~3번까지 정상이면 3차 사례처럼 "정상 동작"으로 판단하고 보고. 이상 징후 있으면 4~5번으로 원인 특정.
